# 리액트의 탄생과 기본 철학

---

## React는 왜 탄생했는가

### jQuery 시대의 한계

2010년대, 페이스북 같은 대규모 웹앱에서 **jQuery식 명령형 DOM 조작**은 한계에 부딪혔다.

```javascript
// 할 일 하나를 완료 처리할 때 — DOM 조작 4개를 수동으로 관리
function completeTodo(id) {
  document.getElementById(`todo-${id}`).style.textDecoration = "line-through";
  const count = document.getElementById("done-count");
  count.innerText = parseInt(count.innerText) + 1;
  const remain = document.getElementById("remain-count");
  remain.innerText = parseInt(remain.innerText) - 1;
  if (parseInt(remain.innerText) === 0) {
    document.getElementById("congrats").style.display = "block";
  }
}
```

하나의 동작에 DOM 조작 4개. 하나라도 빠뜨리면 **화면과 데이터가 불일치**한다.

| 문제 | 설명 |
|------|------|
| **상태 추적 불가** | 데이터가 DOM 안에 흩어져 있어 현재 상태 파악이 어렵다 |
| **동기화 지옥** | 데이터 하나가 바뀌면 관련 DOM을 전부 수동으로 찾아 업데이트해야 한다 |
| **코드 스파게티** | UI가 복잡해질수록 `getElementById`, `innerHTML` 조작이 기하급수적으로 늘어난다 |

### 2013년 — Facebook의 해법: React

> **핵심 아이디어:** "상태(state)가 바뀌면 UI를 자동으로 다시 그려준다."
> 개발자는 **현재 상태에 맞는 UI가 어떤 모습이어야 하는지**만 선언하면 된다.

**설계 원칙 4가지:** 선언형 · 컴포넌트 · 상태 · Virtual DOM

## React의 4대 철학

### (1) 선언형(Declarative) UI

**"무엇(What)을 보여줄지"를 선언하고, "어떻게(How)" DOM을 조작할지는 React가 담당.**

```javascript
// 명령형 — DOM 상태를 직접 추적하고 조작
const btn = document.querySelector('#btn');
const panel = document.querySelector('#panel');
btn.addEventListener('click', () => {
  panel.style.display = (panel.style.display === 'none') ? 'block' : 'none';
});
```

```jsx
// 선언형 — 상태(open)만 관리하면 React가 UI를 일관되게 맞춤
function Toggle() {
  const [open, setOpen] = useState(false);
  return (
    <>
      <button onClick={() => setOpen(o => !o)}>{open ? "닫기" : "열기"}</button>
      {open && <div>내용</div>}
    </>
  );
}
```

**왜 유리한가?**
- 상태만 바꾸면 관련 UI가 **자동으로** 전부 갱신된다
- DOM 업데이트 하나를 빠뜨려 화면이 꼬이는 일이 없다
- 코드가 **의도(What)**를 표현하므로 가독성이 높다

> 추상화 비용이 있지만, 대규모 앱에서 **개발·유지 효율과 버그 감소 이점**이 훨씬 크다.

---

### (2) 컴포넌트(Component)

**UI를 독립 기능 단위로 나눈 재사용 가능한 함수.**

효과: 응집도↑, 결합도↓, 테스트·합성·교체 용이, 팀 병렬 작업 유리.

```jsx
function Button({ kind = "primary", children }) {
  const className = kind === "primary" ? "btn-primary" : "btn-ghost";
  return <button className={className}>{children}</button>;
}

function App() {
  return (
    <div>
      <Button>좋아요</Button>
      <Button kind="ghost">공유하기</Button>
    </div>
  );
}
```

> "컴포넌트 = 파일 1개"가 아니다. 핵심은 **"역할 단위"**다.

---

### (3) 상태(State)

**시간에 따라 변할 수 있는 데이터이며, UI가 반드시 따르는 진실의 원천(source of truth).**

핵심 원칙: `UI = f(state)` — UI는 상태의 함수다.

| 구분 | 설명 |
|------|------|
| **로컬 상태** | `useState`로 관리하는 컴포넌트 내부 데이터 (기본) |
| **전역/공유 상태** | 필요시 Context, Redux, Zustand 등으로 관리 |
| **배치 업데이트** | 여러 상태 변경을 묶어 렌더 횟수를 줄이는 최적화 |

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <>
      <p>{count}번 클릭됨</p>
      {/* 비권장: 같은 렌더의 count를 재사용하므로 +1만 반영될 수 있음 */}
      <button onClick={() => setCount(count + 1)}>+1 (기본)</button>

      {/* 권장: 함수형 업데이트 → 직전 상태를 안전하게 참조 */}
      <button onClick={() => {
        setCount(c => c + 1);
        setCount(c => c + 1); // 동일 이벤트 내에서도 +2 보장
      }}>
        +2 (함수형 업데이트)
      </button>
    </>
  );
}
```

**왜 함수형 업데이트인가?**
`setCount(count + 1)`을 두 번 호출하면 같은 렌더의 `count`(예: 0)를 두 번 참조해 **+1만 반영**된다. 함수형 업데이트 `setCount(c => c + 1)`은 최신 값을 `c`로 받아 안전하게 누적한다.

> 상태 = 전역 상태가 아니다. **로컬 상태가 기본**이며, 필요한 경우에만 전역 도구를 도입한다.

### (4) Virtual DOM (VDOM)

**UI를 JS 객체(설계도)로 구성하고, 이전과 새 설계도를 비교(diff)한 뒤 변경된 부분만 실제 DOM에 패치하는 전략.**

```
(이전 VDOM)  vs  (새 VDOM)
          │  diff
          ▼
    최소 변경(패치 리스트)
          │
          ▼
    실제 DOM에 적용
```

```jsx
// React — 설계도 변경만 선언하면 React가 최소 패치 처리
const root = ReactDOM.createRoot(document.getElementById("root"));
root.render(<h1>Hello!</h1>);
// 상태/props가 바뀌면 VDOM diff 후 필요한 부분만 갱신
```

**핵심 포인트:**
- VDOM이 DOM을 **대체**하는 게 아니다. 최종 출력은 실제 DOM이며, VDOM은 **비교·계산용** 중간 레이어
- VDOM이 항상 빠른 건 아니다. 진짜 장점은 **"충분히 빠르면서 선언적 프로그래밍을 가능하게 해주는 것"**

## JSX와 렌더 파이프라인

### JSX는 React.createElement의 문법 설탕

브라우저는 JSX를 모른다. Babel/SWC가 빌드 시 변환한다:

```javascript
// 개발자가 작성하는 JSX
const el = <h1 className="title">Hello</h1>;

// Babel이 변환한 결과
const el = React.createElement("h1", { className: "title" }, "Hello");

// 결과는 동일한 JS 객체 (= Virtual DOM 객체)
// { type: "h1", props: { className: "title", children: "Hello" } }
```

**중첩된 JSX:**

```jsx
// JSX
<div>
  <h1>제목</h1>
  <p>내용</p>
</div>

// 변환 결과
React.createElement("div", null,
  React.createElement("h1", null, "제목"),
  React.createElement("p", null, "내용")
);
```

**JSX에서 `{}`** — JavaScript 표현식을 삽입하는 구멍:

```jsx
const name = "홍길동";
<h1>Hello, {name.toUpperCase()}!</h1>
// 변수, 함수 호출, 삼항 연산자 등 표현식(expression)이면 가능
// if문, for문 같은 문(statement)은 직접 넣을 수 없다
```

### 전체 렌더 파이프라인

```
JSX (사람 친화 문법)
  └─ Babel/SWC (트랜스파일)
      └─ React.createElement() → React Element (JS 객체)
            └─ Virtual DOM (가상 트리)
                  └─ Diff/Reconciliation (차이 계산)
                        └─ Fiber (우선순위·스케줄링)
                              └─ Render Phase (VDOM 계산 — 중단 가능)
                                    └─ Commit Phase (실제 DOM 패치 — 원자적)
```

## ReactDOM — 설계도를 실물로 바꾸는 브리지

React(설계/상태) ↔ 브라우저 DOM(실물 화면) 사이의 **브리지(다리)**.

### createRoot vs legacy render

| | `createRoot` (React 18+) | `render` (레거시) |
|--|--------------------------|-------------------|
| **API** | `ReactDOM.createRoot(container).render(ui)` | `ReactDOM.render(ui, container)` |
| **동시성** | Concurrent Features 지원 | 미지원 |
| **배치** | 모든 곳에서 자동 배치 | 이벤트 핸들러 안에서만 |
| **권장** | 현재 표준 | 더 이상 권장하지 않음 |

```jsx
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<h1>Hello, React!</h1>);
```

## Fiber — React의 렌더링 엔진 (React 16+)

### 왜 필요한가

과거(React 15 이하)의 Stack Reconciler는 렌더링이 **블로킹 단일 작업**이었다. 한번 시작하면 끝날 때까지 멈출 수 없어 프레임 드롭이 발생했다.

```
[ Stack Reconciler ] 렌더 시작 ────────────────── 렌더 끝
                              ↑ 사용자 입력 무시

[ Fiber Reconciler ] 렌더 시작 ── 청크1 ── (급한 일) ── 청크2 ── 렌더 끝
                              ↑ 중간에 사용자 입력 먼저 처리
```

### 2단계 렌더링

| | Render Phase | Commit Phase |
|--|-------------|-------------|
| **하는 일** | VDOM 계산, diff | 실제 DOM 패치, 이펙트 실행 |
| **중단 가능?** | O (Interruptible) | X (Non-interruptible) |
| **부수효과?** | 없음 — 순수 계산만 | 있음 — DOM 변경, useEffect 등 |

### 우선순위 시스템 (Lanes)

React 18에서 도입된 **Lanes** 모델:

| 우선순위 | 예시 | 처리 |
|----------|------|------|
| **긴급 (Sync)** | 사용자 타이핑, 클릭 | 즉시 처리 |
| **높음 (Transition)** | 페이지 전환, 검색 결과 | 급한 일 뒤에 처리 |
| **낮음 (Deferred)** | 오프스크린 렌더링 | 여유 있을 때 처리 |

```jsx
import { startTransition } from 'react';

function handleSearch(input) {
  setInputValue(input);            // 긴급: 입력값 즉시 반영
  startTransition(() => {
    setSearchResults(filterData(input)); // 낮은 우선순위: 여유 생기면 렌더
  });
}
```

### Automatic Batching (React 18)

React 18부터 **어디서든** 자동 배치:

```jsx
// React 17 — setTimeout 안에서는 배치 안 됨
setTimeout(() => {
  setCount(c => c + 1);  // 렌더 1번
  setFlag(f => !f);      // 렌더 1번 → 총 2번
}, 1000);

// React 18 — 자동 배치
setTimeout(() => {
  setCount(c => c + 1);  // 렌더 X
  setFlag(f => !f);      // 렌더 X → 총 1번만
}, 1000);
```

## 면접 핵심 Q&A

### React 기본 & 철학

**Q. 왜 React를 쓰나요? jQuery 대비 장점은?**
> 대규모 UI에서 상태를 선언적으로 모델링하고, VDOM diff로 최소 패치하여 **일관성·성능·협업**을 동시에 확보한다. jQuery는 DOM을 직접 조작하므로 규모가 커지면 상태-UI 불일치가 빈번해진다. React는 상태만 바꾸면 UI가 자동으로 따라온다.

**Q. 선언형 프로그래밍이 뭔가요?**
> "무엇을 보여줄지"만 기술하고, "어떻게 DOM을 조작할지"는 React가 처리한다. 개발자는 `상태 → UI 매핑 규칙`만 작성하면 되므로 코드량이 줄고, 의도가 명확해지고, 버그 표면적이 줄어든다.
> ```jsx
> {open && <Panel />}  // "open이면 패널을 보여줘"라는 규칙만 작성
> ```

**Q. 단방향 데이터 흐름(One-way data flow)이란?**
> 데이터는 **부모 → 자식** 방향으로만 props를 통해 전달된다. 자식이 부모 상태를 바꾸려면 부모가 내려준 **콜백 함수**를 호출해야 한다. 변경의 출처를 추적하기 쉽고 디버깅이 수월하다.
> ```jsx
> function Parent() {
>   const [value, setValue] = useState("");
>   return <Child value={value} onChange={setValue} />;
> }
> function Child({ value, onChange }) {
>   return <input value={value} onChange={e => onChange(e.target.value)} />;
> }
> ```

**Q. 언제 React가 과한가요?**
> 상태 변화가 적고 수명 짧은 정적/소규모 페이지. 단순 마케팅 페이지, 블로그 등은 React 없이도 충분하다. React의 진가는 **복잡한 상태 변화가 빈번한 인터랙티브 UI**에서 나온다.

### JSX & Babel

**Q. JSX가 뭔가요?**
> `React.createElement()`의 **문법 설탕**. Babel이 빌드 시점에 JSX를 `createElement` 호출로 변환하며, 그 결과는 실제 DOM이 아닌 **Virtual DOM 객체**다.

**Q. JSX 없이 React를 쓸 수 있나요?**
> 가능하다. `React.createElement("div", { className: "box" }, "Hello")`처럼 직접 호출하면 된다. 하지만 실무에서 JSX 없이 쓰는 경우는 거의 없다.

**Q. `class` 대신 `className`을 쓰는 이유?**
> JSX는 JavaScript이므로 예약어 `class`와 충돌한다. DOM의 `className` 프로퍼티명을 그대로 사용한다. 마찬가지로 `for` 대신 `htmlFor`.

**Q. JSX에서 `{}`의 역할은?**
> JavaScript **표현식(expression)**을 삽입하는 구멍. 변수, 함수 호출, 삼항 연산자 등 값을 반환하는 것이면 가능하다. `if`문이나 `for`문 같은 **문(statement)**은 직접 넣을 수 없다.

**Q. Babel과 SWC의 차이는?**
> 둘 다 JSX/최신 문법을 브라우저 호환 코드로 변환하는 **트랜스파일러**. Babel은 JS 기반, SWC는 Rust 기반으로 **빌드 속도가 훨씬 빠르다**. Next.js 12+는 SWC가 기본.

---

### Virtual DOM & 렌더링

**Q. Virtual DOM이 뭔가요? 왜 쓰나요?**
> UI 구조를 **JS 객체 트리**로 표현한 것. 상태가 바뀌면 새 VDOM을 만들고, 이전 VDOM과 **비교(diffing)**해서 변경된 부분만 실제 DOM에 반영한다. 직접 DOM을 조작하는 것보다 **일관성 있고 예측 가능**한 업데이트가 가능하다.

**Q. Virtual DOM이 항상 더 빠른가요?**
> **아니다.** 단순한 DOM 조작은 직접 하는 게 더 빠를 수 있다. VDOM의 진짜 장점은 속도가 아니라 **"충분히 빠르면서 선언적 프로그래밍을 가능하게 해주는 것"**이다.

**Q. Reconciliation(재조정)이란?**
> 이전 VDOM 트리와 새 VDOM 트리를 비교해 최소한의 변경을 계산하는 알고리즘.
> - **같은 타입** → 속성(props)만 업데이트
> - **다른 타입** → 기존 트리를 버리고 새로 생성
> - **리스트 항목** → `key`를 기준으로 식별하고 재배치

**Q. key는 왜 필요한가요? index를 key로 쓰면 안 되는 이유?**
> React가 리스트의 각 항목을 **고유하게 식별**하기 위해 필요하다. `index`를 쓰면 항목이 추가/삭제/재정렬될 때 어떤 항목이 바뀐 건지 정확히 모르므로 **불필요한 리렌더링**이나 **입력값이 엉뚱한 항목에 남는 버그**가 생긴다.
> ```jsx
> // ❌ {items.map((item, i) => <Item key={i} data={item} />)}
> // ✅ {items.map(item => <Item key={item.id} data={item} />)}
> ```

**Q. 배치 업데이트(Batching)란?**
> 여러 `setState`를 묶어 **렌더를 한 번만** 일으키는 최적화. React 18부터는 `setTimeout`, `Promise` 안에서도 자동 배치(Automatic Batching) 적용.

---

### Fiber & React 18

**Q. Fiber가 뭔가요?**
> React 16부터 도입된 렌더링 엔진. 렌더링을 잘게 쪼개 **중단·재개** 가능하고, **우선순위**를 부여해 급한 작업(사용자 입력)을 먼저 처리한다. 이전 Stack Reconciler는 한번 시작하면 끝날 때까지 멈출 수 없었다.

**Q. Render Phase와 Commit Phase의 차이?**
> Render Phase — VDOM 계산/diff. **중단 가능**. 부수효과 없이 순수 계산만.
> Commit Phase — 실제 DOM 패치, 이펙트 실행. **중단 불가**. 짧고 원자적.

**Q. React 18의 Concurrent Features?**
> Fiber 위에 구축된 기능들. 렌더링을 중단하고 더 급한 업데이트를 먼저 처리할 수 있다.
> - `startTransition`: 낮은 우선순위 업데이트 표시
> - `useTransition`: Transition 상태 + pending 플래그
> - `useDeferredValue`: 값의 업데이트를 지연
> - **Automatic Batching**: 어디서든 setState 묶어 렌더 최소화

**Q. createRoot와 render의 차이?**
> `ReactDOM.render()`는 레거시 API로 동기적 렌더링만 지원. `ReactDOM.createRoot()`는 React 18의 Concurrent 기능을 활성화하며, 자동 배치와 우선순위 스케줄링이 작동한다.

**Q. React Element와 실제 DOM의 차이?**
> React Element는 `{ type: "div", props: { ... } }` 형태의 **가벼운 JS 객체(설계도)**다. ReactDOM이 이 설계도를 기반으로 실제 DOM을 생성/패치한다. 불변(immutable)이고 매 렌더마다 새로 생성된다.

### 컴포넌트 & Props & State

**Q. 함수 컴포넌트와 클래스 컴포넌트의 차이?**
> 함수 컴포넌트는 **Hooks**로 상태/생명주기를 관리. 클래스 컴포넌트는 `this.state`, `componentDidMount` 등을 사용. React 팀은 **함수 컴포넌트 + Hooks**를 권장하며, 새 코드에서 클래스를 쓸 이유는 거의 없다.

**Q. Props와 State의 차이?**
> | | Props | State |
> |--|-------|-------|
> | **방향** | 부모 → 자식 전달 | 컴포넌트 내부 소유 |
> | **변경** | 읽기 전용 | `setState`로 변경 가능 |
> | **목적** | 외부 설정값 | 시간에 따라 바뀌는 내부 데이터 |

**Q. `setState`는 왜 비동기적으로 동작하나요?**
> 성능 최적화를 위해 상태 변경을 **큐에 모아서(배치)** 한 번에 처리한다. `setState` 직후 상태를 읽으면 아직 이전 값이다.
> ```jsx
> setCount(count + 1);
> console.log(count); // 아직 이전 값!
> setCount(c => c + 1); // 함수형 업데이트로 최신 값 참조
> ```

**Q. 함수형 업데이트는 언제 써야 하나요?**
> **이전 상태값에 기반해서** 새 값을 계산할 때는 항상 함수형 업데이트가 안전하다. 같은 이벤트에서 여러 번 호출하거나, 비동기 콜백 안에서 클로저 문제를 방지한다.

## 용어 글로서리

### React 핵심

| 용어 | 정의 |
|------|------|
| **JSX** | `React.createElement()`의 문법 설탕. `{}`로 JS 표현식 삽입 |
| **Virtual DOM** | UI를 JS 객체 트리로 표현한 설계도. diff 후 최소 패치로 실제 DOM 업데이트 |
| **Reconciliation** | 이전/새 VDOM을 비교해 최소 변경을 계산하는 알고리즘 |
| **Component** | UI를 독립 기능 단위로 나눈 재사용 가능한 함수. "역할" 단위 |
| **Props** | 부모 → 자식으로 전달되는 읽기 전용 입력값 |
| **State** | 컴포넌트 내부의 변할 수 있는 데이터. UI의 source of truth |
| **Batching** | 여러 setState를 묶어 렌더 횟수를 줄이는 최적화 |
| **key** | 리스트 항목의 고유 식별자. 추가/삭제/재정렬 추적용 |
| **Declarative** | "무엇을" 선언하면 "어떻게"는 React가 처리. `UI = f(state)` |
| **One-way Data Flow** | 데이터가 부모 → 자식 한 방향으로만 흐르는 구조 |
| **React Element** | `createElement`가 반환하는 JS 객체(설계도). 불변, 매 렌더마다 새로 생성 |
| **ReactDOM** | React ↔ 브라우저 DOM 사이의 브리지 |
| **createRoot** | React 18+ 루트 생성 API. Concurrent Features 활성화 |

### Fiber & 동시성

| 용어 | 정의 |
|------|------|
| **Fiber** | React 16+ 렌더링 엔진. 중단·재개 가능한 우선순위 기반 스케줄링 |
| **Render Phase** | VDOM 계산/diff. 중단 가능. 순수 계산만 |
| **Commit Phase** | 실제 DOM 패치/이펙트 실행. 중단 불가. 원자적 |
| **Lanes** | React 18의 우선순위 모델. 업데이트에 긴급/높음/낮음 레인 부여 |
| **startTransition** | 낮은 우선순위 업데이트를 표시하는 API |
| **Automatic Batching** | React 18에서 어디서든 setState를 묶어 렌더 1회로 최소화 |

### 빌드 & 도구

| 용어 | 정의 |
|------|------|
| **Babel** | JS 기반 트랜스파일러. JSX/최신 문법 → 브라우저 호환 코드 |
| **SWC** | Rust 기반 트랜스파일러. Babel보다 빠름. Next.js 12+ 기본 |
| **Vite** | ES 모듈 기반 빌드 도구. 빠른 개발 서버 |
| **HMR** | 코드 수정 시 전체 새로고침 없이 변경된 모듈만 교체 |
