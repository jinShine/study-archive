## 47강. React Router 기초 -- SPA의 한계를 넘는 라우팅 시스템

### 도입

리액트 앱은 SPA(Single Page Application) 방식으로 동작한다. 단 하나의 HTML 파일에서 자바스크립트가 컴포넌트를 교체하기 때문에 페이지 전환이 빠르지만, **URL이 변하지 않아 공유가 불가능하고 뒤로 가기가 작동하지 않는** 치명적 한계가 있다. React Router는 화면의 변화와 URL의 변화를 하나로 동기화해 주는 라우팅 라이브러리이다.

### 개념 설명

#### SPA vs MPA

| 구분 | MPA (전통적 웹) | SPA (리액트) |
|------|:---:|:---:|
| 페이지 전환 | 서버에 새 HTML 요청 (깜빡임) | JS로 컴포넌트만 교체 (빠름) |
| URL 관리 | 서버가 자동 처리 | **라우터 라이브러리 필요** |
| 상태 유지 | 새로고침마다 초기화 | 전환 시에도 State 유지 |

#### React Router v7의 세 가지 모드

1. **프레임워크 모드**: 빌드 도구와 서버까지 포함한 풀 프레임워크 (고도의 SSR)
2. **데이터 모드**: JS 객체 형태로 경로를 미리 정의하여 로딩 성능 극대화 (대규모 프로젝트 표준)
3. **선언적 모드**: `<Routes>`, `<Route>` 컴포넌트로 직관적으로 설계 (학습 시작점)

#### 핵심 컴포넌트

- `BrowserRouter` -- 앱 전체의 URL 변화를 감시하는 센서. History API를 활용해 새로고침 없이 주소를 변경
- `Routes` -- 현재 URL과 일치하는 **단 하나의 Route**만 렌더링하는 필터
- `Route` -- 특정 경로(path)와 컴포넌트(element)를 연결하는 표지판

### 코드 예제

```jsx
// main.jsx
import React from 'react';
import ReactDOM from 'react-dom/client';
import { BrowserRouter } from 'react-router'; // v7 단일 패키지
import App from './App';

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <BrowserRouter>
      <App />
    </BrowserRouter>
  </React.StrictMode>
);
```

```jsx
// App.jsx
import { Routes, Route } from 'react-router';
import InventoryPage from './pages/InventoryPage';
import AnalyticsPage from './pages/AnalyticsPage';
import LogPage from './pages/LogPage';

function App() {
  return (
    <Routes>
      <Route index element={<InventoryPage />} />
      <Route path="analytics" element={<AnalyticsPage />} />
      <Route path="logs" element={<LogPage />} />
    </Routes>
  );
}

export default App;
```

### 코드 해설

1. `BrowserRouter`가 App 전체를 감싸면, 하위 어떤 컴포넌트에서든 주소 정보에 접근 가능하다.
2. `Routes`는 현재 브라우저 주소와 매칭되는 **단 하나의 Route**만 렌더링한다.
3. `index` 속성은 부모 경로(`/`)에 들어왔을 때 기본으로 보여줄 컴포넌트를 지정한다.
4. `<a>` 태그를 쓰면 전체 페이지가 새로고침되어 React State가 초기화되므로, SPA에서는 반드시 `Link` 컴포넌트를 사용해야 한다.

### 실무 비유

BrowserRouter는 건물의 **중앙 안내 시스템**, Routes는 현재 층에서 올바른 방을 찾아주는 **층별 안내판**, Route는 각 방에 붙은 **호수 표지판**이다.

### 핵심 포인트

- React Router v7은 `react-router` 단일 패키지로 설치한다 (`-dom` 불필요)
- `BrowserRouter`는 앱의 최상위(main.jsx)에서 감싼다
- `<a>` 태그 대신 `Link` 컴포넌트를 사용해야 SPA의 이점을 유지할 수 있다
- `index` 속성은 부모 경로의 기본 화면을 지정한다

## 48강. 중첩 라우팅과 레이아웃 -- Outlet을 활용한 레이아웃 설계

### 도입

실제 서비스에서는 상단 바, 사이드바 등 **공통 요소는 고정**된 채 중앙 콘텐츠만 바뀌어야 한다. 페이지를 이동할 때마다 전체 화면이 교체되면 사용자 경험이 떨어진다. 중첩 라우팅(Nested Routes)과 Outlet을 사용하면 공통 레이아웃을 유지하면서 콘텐츠 영역만 부드럽게 교체할 수 있다.

### 개념 설명

- **중첩 라우팅**: 부모 Route 안에 자식 Route를 배치하는 구조. 부모 컴포넌트는 항상 렌더링되고, 자식은 URL에 따라 교체된다.
- **Outlet**: 부모 레이아웃 컴포넌트 내에서 자식 컴포넌트가 렌더링될 **빈 자리(예약석)**를 지정하는 컴포넌트
- **NavLink**: Link의 확장판. 현재 URL과 일치하면 `isActive` 속성이 true가 되어 활성화 스타일을 자동 적용할 수 있다

### 코드 예제

```jsx
// Layout.jsx
import { Outlet } from 'react-router';
import Sidebar from './Sidebar';

export default function Layout() {
  return (
    <div style={{ display: 'flex', height: '100vh' }}>
      {/* Sidebar는 페이지 이동 시에도 다시 그려지지 않고 고정 */}
      <Sidebar />
      <main style={{ flex: 1, padding: '40px', overflowY: 'auto' }}>
        {/* 현재 주소와 일치하는 자식 컴포넌트가 이 자리에 렌더링 */}
        <Outlet />
      </main>
    </div>
  );
}
```

```jsx
// Sidebar.jsx
import { NavLink } from 'react-router';

export default function Sidebar() {
  const getStyle = ({ isActive }) => ({
    display: 'block',
    padding: '12px 20px',
    backgroundColor: isActive ? '#2563eb' : 'transparent',
    color: isActive ? 'white' : '#475569',
    textDecoration: 'none',
    borderRadius: '8px',
  });

  return (
    <nav style={{ width: '260px', padding: '20px', borderRight: '1px solid #e2e8f0' }}>
      <NavLink to="/" style={getStyle}>인벤토리 관리</NavLink>
      <NavLink to="/analytics" style={getStyle}>데이터 분석</NavLink>
    </nav>
  );
}
```

```jsx
// App.jsx
import { Routes, Route } from 'react-router';
import Layout from './components/Layout';
import InventoryPage from './pages/InventoryPage';
import AnalyticsPage from './pages/AnalyticsPage';

export default function App() {
  return (
    <Routes>
      {/* 부모 Route: Layout이 항상 렌더링 */}
      <Route path="/" element={<Layout />}>
        {/* 자식 Route: Outlet 자리에 교체됨 */}
        <Route index element={<InventoryPage />} />
        <Route path="analytics" element={<AnalyticsPage />} />
      </Route>
    </Routes>
  );
}
```

### 코드 해설

1. `<Route path="/" element={<Layout />}>` -- 최상위 부모로 Layout을 지정한다.
2. 자식 Route들(`index`, `analytics`)은 Layout의 `<Outlet />` 위치에 렌더링된다.
3. 페이지를 이동해도 Sidebar와 Header는 **메모리상에 그대로 유지**되고, Outlet 자리만 교체된다.
4. NavLink의 `isActive` 콜백으로 현재 활성 메뉴에 자동으로 하이라이트 스타일을 적용한다.

### 실무 비유

미술관에 비유하면, Layout은 건물의 **벽과 조명과 복도**(변하지 않는 것), Outlet은 벽에 걸린 **액자 속 그림**(변하는 것)이다. 그림을 바꾸려고 미술관 전체를 철거하고 다시 짓는 것(`<a>` 태그)은 비효율적이다.

### 핵심 포인트

- 부모 Route에 공통 레이아웃 컴포넌트를 지정하고, `<Outlet />`으로 자식 예약석을 만든다
- NavLink의 `isActive`로 현재 메뉴를 시각적으로 강조한다
- 중첩 구조에서 공통 요소는 리렌더링 없이 메모리에 유지된다

## 49강. 동적 라우팅 useParams -- 데이터 상세 페이지 구현

### 도입

상품이 만 개라면 만 개의 상세 페이지 컴포넌트를 만들어야 할까? 동적 라우팅(Dynamic Routing)은 경로에 **변수 자리(:파라미터)**를 선언하여, 단 하나의 컴포넌트로 수만 개의 서로 다른 상세 페이지를 처리할 수 있게 해 준다.

### 개념 설명

- 경로에 콜론(`:`)을 붙이면 **변수 자리**가 된다: `path="inventory/:productId"`
- `useParams()` 훅은 현재 URL에서 변수 자리의 실제 값을 **객체 형태**로 반환한다
- **중요**: useParams로 가져온 모든 값은 항상 **문자열(String)**이다. 숫자 비교 시 반드시 `Number()`로 변환해야 한다

### 코드 예제

```jsx
// App.jsx
import { Routes, Route } from 'react-router';
import Layout from './components/Layout';
import InventoryPage from './pages/InventoryPage';
import ProductDetail from './pages/ProductDetail';

export default function App() {
  return (
    <Routes>
      <Route element={<Layout />}>
        <Route index element={<InventoryPage />} />
        {/* :productId는 변수 자리 -- 어떤 값이든 매칭됨 */}
        <Route path="inventory/:productId" element={<ProductDetail />} />
      </Route>
    </Routes>
  );
}
```

```jsx
// ProductDetail.jsx
import { useParams } from 'react-router';

export default function ProductDetail() {
  // URL에서 동적 파라미터 추출: /inventory/101 -> { productId: "101" }
  const { productId } = useParams();

  return (
    <div>
      <h2>PRODUCT ANALYSIS</h2>
      <p>현재 조회 중인 상품 코드: #{productId}</p>
      {/* productId를 key로 서버에서 상세 데이터를 불러오는 로직 배치 */}
    </div>
  );
}
```

### 코드 해설

1. `path="inventory/:productId"` -- `/inventory/1`, `/inventory/999`, `/inventory/hello` 등 어떤 값이 와도 이 Route로 매칭된다.
2. `useParams()`는 `{ productId: "101" }` 형태의 객체를 반환한다.
3. **타입 함정**: `productId`는 숫자 101이 아니라 **문자열 "101"**이다. DB의 숫자형 ID와 비교할 때 `Number(productId)` 변환이 필수이다.

### 실무 비유

호텔의 **전자 키카드 시스템**과 같다. 모든 객실의 문은 동일한 구조(ProductDetail)이지만, 키카드(productId)에 따라 열리는 방(데이터)이 달라진다.

### 핵심 포인트

- 경로에 `:변수명`으로 동적 파라미터를 선언한다
- `useParams()`로 URL에서 실제 값을 객체로 추출한다
- **반환값은 항상 문자열** -- 숫자 연산 시 타입 변환 필수

## 50강. 쿼리스트링 useSearchParams -- 데이터 필터링 구현

### 도입

경로 파라미터(`:id`)가 **어떤 리소스에 접근할지** 결정한다면, 쿼리스트링(`?key=value`)은 **한 페이지 내에서의 세밀한 옵션**(필터, 정렬, 페이지네이션 등)을 결정한다. useSearchParams는 쿼리스트링을 useState처럼 읽고 쓸 수 있게 해 주는 훅이다.

### 개념 설명

- 쿼리스트링: URL 경로 뒤의 `?category=electronics&sort=price` 부분
- `useSearchParams()`는 `[searchParams, setSearchParams]` 튜플을 반환한다
  - `searchParams.get('key')` -- 특정 키의 값을 읽는다 (없으면 null)
  - `setSearchParams({ key: value })` -- 페이지 새로고침 없이 쿼리스트링을 변경한다
- useState와의 차이: 데이터가 **메모리가 아닌 브라우저 주소창**에 저장된다 -> 새로고침해도 유지, URL 공유 가능

### 코드 예제

```jsx
// InventoryPage.jsx
import { useSearchParams } from 'react-router';

export default function InventoryPage() {
  const [searchParams, setSearchParams] = useSearchParams();

  // 주소창에서 'category' 값 읽기 (기본값: 'all')
  const category = searchParams.get('category') || 'all';

  // 버튼 클릭 시 주소창의 쿼리스트링만 변경 (페이지 이동 없음)
  const handleFilter = (type) => {
    setSearchParams({ category: type });
  };

  return (
    <div>
      <h2>INVENTORY MANAGEMENT</h2>

      <div>
        <button onClick={() => handleFilter('all')}>전체</button>
        <button onClick={() => handleFilter('electronics')}>전자기기</button>
        <button onClick={() => handleFilter('furniture')}>가구</button>
      </div>

      <p>현재 필터: <strong>{category.toUpperCase()}</strong></p>
    </div>
  );
}
```

### 코드 해설

1. `searchParams.get('category')` -- URL이 `/?category=furniture`이면 `"furniture"`를 반환한다.
2. `setSearchParams({ category: 'electronics' })` -- URL이 `/?category=electronics`로 바뀌지만 페이지가 새로고침되지 않는다.
3. 브라우저를 **새로고침해도** 쿼리스트링이 URL에 남아있으므로 필터 상태가 유지된다.
4. 이 URL을 그대로 공유하면 상대방도 **동일한 필터 상태**를 볼 수 있다.

### 실무 비유

식당의 **주문서**에 비유할 수 있다. 경로 파라미터가 "어떤 식당에 갈지"를 결정한다면, 쿼리스트링은 "맵기 조절, 사이드 추가" 같은 **세부 옵션**이다. 주문서를 친구에게 보내면 같은 메뉴를 주문할 수 있다.

### 핵심 포인트

- `useSearchParams`는 URL 기반의 useState이다 -- 새로고침 유지, 공유 가능
- `searchParams.get('key')`로 읽고, `setSearchParams({ key: value })`로 쓴다
- SEO 이점: 검색 로봇이 각 필터 결과를 개별 페이지로 인식하여 검색 유입을 늘린다

## 51강. useNavigate 조건부 이동 -- 프로그래밍 방식의 페이지 전환

### 도입

Link와 NavLink는 사용자가 **직접 클릭**해야 이동한다. 하지만 실무에서는 로그인 성공 후 대시보드로 자동 이동, 권한 없는 사용자를 로그인 페이지로 강제 이동 등 **코드가 스스로 페이지를 이동시키는** 상황이 빈번하다. useNavigate는 자바스크립트 로직으로 라우팅을 제어하는 훅이다.

### 개념 설명

- `useNavigate()` -- 호출하면 navigate 함수를 반환한다
- `navigate('/path')` -- 지정한 경로로 이동 (히스토리 스택에 추가)
- `navigate('/path', { replace: true })` -- 현재 기록을 **덮어써서** 뒤로 가기를 방지
- `navigate(-1)` -- 브라우저의 뒤로 가기와 동일하게 이전 페이지로 이동
- `navigate(-2)` -- 두 페이지 전으로 이동

### 코드 예제

```jsx
// LoginPage.jsx
import { useNavigate } from 'react-router';

export default function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = async () => {
    console.log("로그인 프로세스 시작...");

    // 서버 인증 시뮬레이션
    await new Promise((resolve) => setTimeout(resolve, 1000));

    // 성공 시 대시보드로 이동, replace로 로그인 페이지 기록 제거
    navigate('/', { replace: true });
  };

  return (
    <div>
      <h2>SYSTEM ACCESS</h2>
      <button onClick={handleLogin}>LOGIN TO DASHBOARD</button>
    </div>
  );
}
```

```jsx
// Dashboard.jsx
import { useNavigate } from 'react-router';

export default function Dashboard() {
  const navigate = useNavigate();

  return (
    <div>
      <h1>Main Dashboard</h1>
      <button onClick={() => navigate(-1)}>뒤로 가기</button>
    </div>
  );
}
```

### 코드 해설

1. `navigate('/', { replace: true })` -- 로그인 성공 후 대시보드로 이동하되, 로그인 페이지 기록을 히스토리에서 제거한다. 이로써 대시보드에서 뒤로 가기를 눌러도 로그인 창이 다시 나타나지 않는다.
2. `navigate(-1)` -- 히스토리 스택에서 한 단계 뒤로 이동한다. 사용자가 어디서 왔는지 기억하지 않아도 된다.
3. `async/await`와 결합하면 비동기 작업(API 호출 등) 완료 후 안전하게 이동할 수 있다.

### 실무 비유

- **Link** = 에스컬레이터: 사용자가 직접 발을 디뎌 이동하는 능동적 수단
- **useNavigate** = 엘리베이터 자동 제어: 시스템이 상황을 판단하여 사용자를 최적의 위치로 보내는 명령

### 핵심 포인트

- Link는 사용자 클릭용, useNavigate는 시스템 로직용이다
- `replace: true`는 히스토리를 덮어써서 보안과 UX를 동시에 잡는다
- `navigate(-1)`은 주소를 몰라도 이전 페이지로 돌아갈 수 있는 편의 기능이다

## 52강. 데이터 라우터 아키텍처 -- createBrowserRouter의 현대적 설계

### 도입

기존의 BrowserRouter + JSX 방식은 직관적이지만, 페이지 이동 시 데이터를 미리 확보할 수 없어 "텅 빈 화면 + 로딩 스피너" 문제가 발생한다. 데이터 라우터(Data Router)는 모든 경로를 **자바스크립트 객체**로 중앙 관리하여, 렌더링 전 데이터 로딩(loader), 에러 처리(errorElement) 등 고급 기능을 지원하는 v7의 표준 아키텍처이다.

### 개념 설명

- `createBrowserRouter(routeConfig[])` -- JS 객체 배열로 전체 라우팅 지도를 미리 생성
- `RouterProvider` -- 생성된 router 객체를 앱에 주입하는 컴포넌트. **BrowserRouter를 대체**한다
- 데이터 라우터를 사용하면 main.jsx에서 BrowserRouter 감싸기가 불필요해진다

### 코드 예제

```jsx
// App.jsx
import { createBrowserRouter, RouterProvider } from 'react-router';
import Layout from './components/Layout';
import InventoryPage from './pages/InventoryPage';
import AnalyticsPage from './pages/AnalyticsPage';

// 객체 배열로 전체 경로를 중앙 관리
const router = createBrowserRouter([
  {
    path: "/",
    element: <Layout />,
    children: [
      { index: true, element: <InventoryPage /> },
      { path: "analytics", element: <AnalyticsPage /> },
    ],
  },
]);

export default function App() {
  return <RouterProvider router={router} />;
}
```

```jsx
// main.jsx -- BrowserRouter 제거
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    {/* BrowserRouter로 감싸지 않음. RouterProvider가 그 역할을 대신함 */}
    <App />
  </React.StrictMode>
);
```

### 코드 해설

1. `createBrowserRouter`는 전체 경로를 하나의 객체 트리로 파악한다. 이를 통해 사용자가 도착하기 전에 필요한 데이터를 미리 로딩하는 `loader` 기능을 사용할 준비가 된다.
2. `RouterProvider`가 router 객체를 받아 앱에 주입하므로, 기존의 BrowserRouter 래핑이 불필요하다.
3. 중첩 구조도 `children` 배열로 직관적으로 표현한다.

### 실무 비유

기존 JSX 방식이 **종이 지도**를 들고 건물을 직접 찾아가는 것이라면, 데이터 라우터는 **중앙 관제실의 디지털 GPS** 시스템이다. 전체 경로를 한눈에 파악하고, 사용자가 도착하기 전에 필요한 물자(데이터)를 미리 보내둘 수 있다.

### 핵심 포인트

- `createBrowserRouter`로 객체 기반 라우팅을 구성한다
- `RouterProvider`가 BrowserRouter를 완전히 대체한다
- 이 구조가 loader, errorElement 등 고급 기능의 전제 조건이다

## 53강. 데이터 로더 최적화 -- 렌더링 전 데이터를 미리 가져오기

### 도입

기존의 useEffect 방식은 컴포넌트가 화면에 나타난 **뒤에야** 데이터를 가져오므로, 사용자는 빈 화면이나 로딩 스피너를 마주해야 한다. 데이터 로더(loader)는 **컴포넌트가 렌더링되기 전에** 라우터가 데이터를 미리 확보하는 프리페칭(Pre-fetching) 기술이다.

### 개념 설명

- **loader 함수**: 컴포넌트 외부에서 정의하는 async 함수. 라우터가 페이지 이동 시 컴포넌트를 그리기 전에 먼저 실행한다
- **useLoaderData()**: loader가 반환한 데이터를 컴포넌트 내에서 즉시 꺼내 쓰는 훅. useState, useEffect 없이 데이터가 이미 준비되어 있다
- **동작 원리**: 링크 클릭 -> 라우터가 loader 실행 -> 데이터 완료 시까지 현재 페이지 유지 -> 데이터 준비 완료 시 새 페이지 한꺼번에 렌더링

### 코드 예제

```jsx
// InventoryPage.jsx
import { useLoaderData } from 'react-router';

// 컴포넌트 외부에서 정의 -- 라우터가 미리 실행할 수 있어야 함
export const inventoryLoader = async () => {
  const response = await fetch('https://api.sampleapis.com/coffee/hot');
  if (!response.ok) {
    throw new Error("데이터를 가져오는 데 실패했습니다.");
  }
  return response.json();
};

export default function InventoryPage() {
  // loader가 반환한 데이터를 즉시 사용 (이미 준비 완료)
  const products = useLoaderData();

  return (
    <div>
      <h2>REAL-TIME INVENTORY</h2>
      <div>
        {products.slice(0, 6).map(item => (
          <div key={item.id}>
            <h3>{item.title}</h3>
            <p>ID: #{item.id}</p>
          </div>
        ))}
      </div>
    </div>
  );
}
```

```jsx
// App.jsx
import { createBrowserRouter, RouterProvider } from 'react-router';
import Layout from './components/Layout';
import InventoryPage, { inventoryLoader } from './pages/InventoryPage';

const router = createBrowserRouter([
  {
    path: "/",
    element: <Layout />,
    children: [
      {
        index: true,
        element: <InventoryPage />,
        loader: inventoryLoader, // 경로에 loader 등록
      },
    ],
  },
]);

export default function App() {
  return <RouterProvider router={router} />;
}
```

### 코드 해설

1. `inventoryLoader`는 컴포넌트 **외부**에서 export된 async 함수이다. 라우터가 미리 가로채서 실행할 수 있어야 하기 때문이다.
2. 라우터 설정에서 `loader: inventoryLoader`로 특정 경로에 등록한다.
3. 사용자가 해당 경로로 이동하면, 라우터가 **먼저** loader를 실행하고, 데이터가 준비될 때까지 현재 페이지를 유지한다.
4. 데이터가 완료되면 `useLoaderData()`를 통해 컴포넌트에서 즉시 사용할 수 있다. useState, useEffect가 불필요하다.

### 실무 비유

기존 방식(useEffect)이 식당에 도착해 자리에 앉은 **다음** 주문을 시작하는 것이라면, 데이터 로더는 집에서 출발할 때 **이미 메뉴를 주문**해두는 시스템이다. 식당 문을 여는 순간 테이블에는 이미 음식이 놓여 있다.

### 핵심 포인트

- loader 함수는 컴포넌트 외부에서 정의하고, 라우터 설정에 등록한다
- `useLoaderData()`로 프리페칭된 데이터를 즉시 사용한다
- 데이터 준비 전까지 이전 화면을 유지하므로, 빈 화면이나 로딩 스피너가 사라진다
- loader에서 throw한 에러는 errorElement가 포착한다

## 54강. 에러 바운더리와 리다이렉트 -- 안정적인 라우팅을 위한 방어 코드

### 도입

데이터 로더가 API 장애로 실패하거나, 권한 없는 사용자가 보호된 페이지에 접근할 때 앱이 하얗게 죽으면 안 된다. 에러 바운더리(Error Boundary)는 사고를 격리하여 앱 전체의 크래시를 방지하고, 리다이렉트(redirect)는 부적절한 접근을 렌더링 전에 차단하는 방어 메커니즘이다.

### 개념 설명

- **errorElement**: 특정 Route에서 에러가 발생했을 때 해당 컴포넌트 대신 렌더링할 에러 전용 화면
- **useRouteError()**: 라우터가 포착한 에러 정보(status, message 등)를 객체로 반환하는 훅
- **redirect(path)**: loader 내부에서 사용하며, 컴포넌트가 **렌더링되기 전에** 즉시 다른 경로로 이동시키는 함수
  - useNavigate와의 차이: useNavigate는 컴포넌트 **내부**에서 실행되지만, redirect는 **loader 단계**에서 실행되어 화면을 그리기 시도조차 하지 않는다

### 코드 예제

```jsx
// ErrorPage.jsx
import { useRouteError, Link } from 'react-router';

export default function ErrorPage() {
  const error = useRouteError();
  console.error("시스템 장애 감지:", error);

  return (
    <div style={{ textAlign: 'center', padding: '50px' }}>
      <h1>System Interrupted</h1>
      <p>에러 상태: {error.status || "Unknown Error"}</p>
      <Link to="/">BACK TO DASHBOARD</Link>
    </div>
  );
}
```

```jsx
// App.jsx
import { createBrowserRouter, RouterProvider, redirect } from 'react-router';
import Layout from './components/Layout';
import ErrorPage from './pages/ErrorPage';
import InventoryPage, { inventoryLoader } from './pages/InventoryPage';

// 보안 로직이 포함된 보호 로더
const protectedLoader = async () => {
  const isUserAuthenticated = false; // 실무: 세션/토큰 확인

  if (!isUserAuthenticated) {
    // 렌더링 시도 전에 즉시 로그인 페이지로 방향 전환
    return redirect('/login');
  }
  return inventoryLoader();
};

const router = createBrowserRouter([
  {
    path: "/",
    element: <Layout />,
    errorElement: <ErrorPage />, // 최상위에서 모든 하위 에러를 포착
    children: [
      {
        index: true,
        element: <InventoryPage />,
        loader: protectedLoader,
      },
    ],
  },
]);

export default function App() {
  return <RouterProvider router={router} />;
}
```

### 코드 해설

1. `errorElement: <ErrorPage />`를 최상위 Route에 배치하면, 하위 어떤 Route에서 에러가 발생해도 앱 전체가 죽지 않고 ErrorPage가 대신 렌더링된다.
2. `useRouteError()`로 에러 객체의 status, message 등을 읽어 사용자에게 의미 있는 안내를 제공한다.
3. `redirect('/login')`은 loader 단계에서 실행되므로, 권한 없는 사용자의 화면에는 보호된 페이지가 **한 프레임도 보이지 않는다**.
4. errorElement는 하위 Route에도 개별 설정할 수 있어, 경로별 맞춤 에러 화면이 가능하다.

### 실무 비유

- **에러 바운더리** = 빌딩의 **방화셔터**. 50층에서 화재가 나도 해당 층만 차단하고, 다른 층은 정상 운영한다.
- **redirect** = 출입 통제 시스템. 신분증이 없으면 건물 로비에서 즉시 안내 데스크로 돌려보낸다.

### 핵심 포인트

- `errorElement`를 최상위에 배치하여 앱 전체의 크래시를 방지한다
- `useRouteError()`로 에러 정보를 읽어 사용자 친화적인 안내를 제공한다
- `redirect`는 loader 단계에서 작동하므로 useNavigate보다 빠르고 안전하다
- 대규모 서비스에서 API 장애는 불가피하므로, 에러 바운더리는 **필수** 구현 사항이다

## 면접 핵심 Q&A

**Q1. SPA에서 React Router가 필요한 이유는?**
> SPA는 단일 HTML에서 JS로 컴포넌트만 교체하므로, URL이 변하지 않아 페이지 공유가 불가능하고 브라우저의 뒤로 가기가 작동하지 않는다. React Router는 History API를 활용하여 URL 변화와 화면 변화를 동기화함으로써 이 문제를 해결한다.

**Q2. Outlet의 역할과 중첩 라우팅의 장점은?**
> Outlet은 부모 레이아웃 컴포넌트에서 자식 컴포넌트가 렌더링될 "예약석"을 지정하는 컴포넌트이다. 중첩 라우팅을 통해 사이드바, 헤더 등 공통 요소는 메모리에 유지한 채 콘텐츠 영역만 교체할 수 있어, 불필요한 리렌더링을 방지하고 부드러운 페이지 전환 UX를 제공한다.

**Q3. useParams로 가져온 값이 항상 문자열인 이유와 주의점은?**
> useParams는 URL 주소창의 텍스트를 그대로 읽어오기 때문에 모든 값이 문자열(String)이다. DB의 숫자형 ID와 비교할 때 `Number()`로 타입 변환하지 않으면, `"101" === 101`이 false가 되어 데이터가 있는데도 찾지 못하는 버그가 발생한다.

**Q4. useSearchParams와 useState의 차이는?**
> useState는 데이터를 컴포넌트 메모리에 저장하므로 새로고침 시 초기화되고 URL 공유가 불가능하다. useSearchParams는 데이터를 브라우저 주소창(`?key=value`)에 저장하므로 새로고침해도 유지되고, URL을 공유하면 상대방도 동일한 상태를 볼 수 있다. SEO 측면에서도 검색 로봇이 각 필터를 개별 페이지로 인식하는 이점이 있다.

**Q5. createBrowserRouter가 BrowserRouter를 대체하는 이유는?**
> createBrowserRouter는 객체 기반으로 전체 경로를 미리 파악하므로, 컴포넌트가 렌더링되기 전에 데이터를 로딩(loader)하거나 에러를 처리(errorElement)하는 고급 기능을 지원할 수 있다. JSX 기반의 BrowserRouter는 컴포넌트가 렌더링된 후에야 데이터 패칭을 시작하므로 "텅 빈 화면 + 로딩 스피너" 문제가 발생한다.

**Q6. loader와 useEffect 방식의 데이터 패칭 차이는?**
> useEffect는 컴포넌트가 화면에 마운트된 후 실행되므로 "렌더 -> 로딩 -> 데이터 표시" 순서를 거쳐 빈 화면이 발생한다. loader는 라우터가 컴포넌트를 렌더링하기 전에 데이터를 미리 확보하므로 "데이터 확보 -> 완성된 화면 렌더링"이라는 워터폴 없는 흐름을 만든다. 이전 페이지를 유지하다가 새 화면을 한꺼번에 보여주어 UX가 크게 향상된다.

**Q7. redirect와 useNavigate의 차이는?**
> useNavigate는 컴포넌트 내부에서 실행되므로 컴포넌트가 일단 렌더링된 후 이동이 발생한다. redirect는 loader 단계(렌더링 전)에서 실행되므로 보호된 페이지가 화면에 한 프레임도 노출되지 않는다. 인증이 필요한 페이지 보호에는 redirect가 더 안전하다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| SPA (Single Page Application) | 단일 HTML에서 JS로 컴포넌트만 교체하여 페이지를 전환하는 웹 애플리케이션 방식 |
| BrowserRouter | 브라우저의 History API를 활용하여 URL 변화를 감시하는 최상위 라우팅 컴포넌트 |
| Routes / Route | 현재 URL과 매칭되는 단 하나의 경로를 찾아 해당 컴포넌트를 렌더링하는 필터/표지판 |
| Outlet | 부모 레이아웃에서 자식 컴포넌트가 렌더링될 예약석을 지정하는 컴포넌트 |
| NavLink | Link의 확장판. 현재 URL과 일치 시 isActive를 제공하여 활성화 스타일 적용 가능 |
| 동적 파라미터 (`:param`) | 경로에 콜론으로 선언하는 변수 자리. 어떤 값이든 매칭됨 |
| useParams | URL의 동적 파라미터 값을 객체로 반환하는 훅 |
| useSearchParams | URL의 쿼리스트링(`?key=value`)을 읽고 쓸 수 있는 훅 |
| useNavigate | 자바스크립트 로직으로 프로그래밍 방식의 페이지 이동을 수행하는 훅 |
| replace | 히스토리 스택의 현재 항목을 덮어써서 뒤로 가기를 방지하는 옵션 |
| createBrowserRouter | JS 객체 배열로 전체 라우팅을 중앙 관리하는 데이터 라우터 생성 함수 |
| RouterProvider | createBrowserRouter로 생성한 router를 앱에 주입하는 컴포넌트 |
| loader | 컴포넌트 렌더링 전에 라우터가 미리 실행하는 데이터 프리페칭 함수 |
| useLoaderData | loader가 반환한 데이터를 컴포넌트에서 즉시 꺼내 쓰는 훅 |
| errorElement | 에러 발생 시 일반 컴포넌트를 대체할 전용 에러 화면 컴포넌트 |
| useRouteError | 라우터가 포착한 에러 정보를 객체로 반환하는 훅 |
| redirect | loader 단계에서 렌더링 전 즉시 다른 경로로 이동시키는 함수 |

## 자가 점검 체크리스트

- [ ] SPA에서 URL 동기화가 필요한 이유를 설명할 수 있다
- [ ] BrowserRouter, Routes, Route의 관계를 이해하고 기본 라우팅을 구성할 수 있다
- [ ] `<a>` 태그 대신 Link/NavLink를 써야 하는 이유를 논리적으로 설명할 수 있다
- [ ] 중첩 라우팅으로 공통 레이아웃을 유지하면서 콘텐츠만 교체하는 구조를 설계할 수 있다
- [ ] NavLink의 isActive 콜백을 활용하여 현재 메뉴를 시각적으로 강조할 수 있다
- [ ] `:파라미터`와 useParams를 사용하여 동적 상세 페이지를 구현할 수 있다
- [ ] useParams의 반환값이 항상 문자열임을 인지하고 타입 변환을 적용할 수 있다
- [ ] useSearchParams로 필터 상태를 URL에 저장하고, 새로고침해도 유지됨을 확인했다
- [ ] useNavigate와 replace 옵션을 활용하여 조건부 페이지 이동을 구현할 수 있다
- [ ] createBrowserRouter로 객체 기반 데이터 라우터 구조를 설계할 수 있다
- [ ] loader 함수로 렌더링 전 데이터를 프리페칭하고, useLoaderData로 즉시 사용할 수 있다
- [ ] errorElement와 useRouteError로 에러 화면을 구현하여 앱 크래시를 방지할 수 있다
- [ ] redirect로 인증되지 않은 사용자를 렌더링 전에 차단할 수 있다
