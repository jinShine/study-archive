📖 상세 개념 가이드 (The Core Manual)
지난 시간 우리는 Immer라는 마법사를 통해 복잡한 중첩 객체를 우아하게 다루는 법을 익혔습니다. 이제 리액트 컴포넌트 안에서라면 그 어떤 데이터 구조도 두렵지 않으실 겁니다.

하지만 프로젝트가 커지면 아주 당혹스러운 장벽에 부딪힙니다. 바로 리액트의 통제권이 전혀 미치지 않는 '메마른 땅'에서 스토어의 상태를 읽거나 수정해야 하는 상황이죠. 오늘은 이 단절의 고통과 보안 통신의 기초 개념을 아주 기초부터 정밀하게 파헤쳐 보겠습니다.

1. 배경: 전용 멤버십 클럽과 외부 전문가의 비유
리액트 컴포넌트들은 마치 아주 호화로운 '전용 멤버십 클럽' 안에 모여 있는 정회원들과 같습니다.

클럽 내부 (React World):useStore라는 마이크(Hook)만 들면 누구와도 자유롭게 대화할 수 있는 특권이 주어집니다.
클럽 외부 (Vanilla JS World): 클럽 밖에서 일하는 배달원이나 수리공 같은 외부 전문가들(API 로직, 유틸리티 함수, 일반 .ts 파일)은 클럽 안의 정보가 필요해도 마이크를 쓸 수 없습니다.
고통의 핵심: 리액트의 규칙상 "마이크(Hook)는 오직 정회원(컴포넌트)만 사용할 수 있다"는 원칙 때문입니다. 밖에서 마이크를 켜는 순간, 리액트 보안 요원은 "당신은 회원이 아니군요!"라며 에러를 던집니다.
2. 기초 개념: 웹 보안의 삼총사 (Interceptor, Authorization, Bearer)
리액트 밖에서 상태가 가장 절실한 순간은 바로 API 통신 시점입니다. 이 과정을 이해하기 위해 세 가지 핵심 개념을 짚고 넘어가야 합니다.

인터셉터(Interceptor) - '중간 검문소': 우리가 편지를 우체통에 넣기 전, 중간 검문소에서 내용을 확인하고 인증 도장을 찍어주는 보안 요원입니다. 모든 요청마다 일일이 인증 정보를 적는 노가다를 방지하기 위해 사용합니다.
헤더(Headers) - '봉투 겉면 정보': 편지 내용(Body)과는 별개로, 이 편지를 누가 보냈는지, 어떤 자격이 있는지 봉투 겉면에 적는 '메타 데이터'입니다. 보안 정보는 주로 Authorization이라는 칸에 적습니다.
Bearer 토큰 - '열쇠 소지자': 'Bearer'는 사전적으로 '소지한 사람'이라는 뜻입니다. 즉, "이 토큰을 손에 쥐고 있는 사람에게 이 정보에 접근할 수 있는 권한을 주어라"라고 서버에 요청하는 일종의 신분증 제시 방법입니다. 보통 Authorization: Bearer [토큰값] 형태로 작성합니다.
/* [Concept Code]: 일반적인 보안 헤더의 구조 */
{
  headers: {
    // "이 토큰을 소지한(Bearer) 나에게 접근 권한을 허가하라"는 뜻
    Authorization: "Bearer eyJhbGciOiJIUzI1Ni..."
  }
}
💻 실습: 고통의 현장 (API 인터셉터 단절 상황)
우리는 이제 axios 검문소(Interceptor)에서 우리 스토어에 저장된 '신분증(Token)'을 꺼내 봉투 겉면(Header)에 붙이려고 합니다. 하지만 이곳은 리액트 클럽 밖입니다.

/* [File Path]: src/api/http.ts
   [Copyright]: © nhcodingstudio 소유
   [Status]: ❌ 실행 시 런타임 에러 발생 (Invalid Hook Call)
*/

import axios from 'axios';
import { useAuthStore } from '../store/useAuthStore'; // 1. 스토어 마이크를 가져옴

const http = axios.create({
  baseURL: '<https://api.example.com>',
});

/**
 * [🚨 절망적인 고통의 코드]
 * 인터셉터는 일반 자바스크립트 함수입니다. (리액트 컴포넌트가 아님)
 */
http.interceptors.request.use((config) => {

  // ERROR: 훅(Hook)은 오직 리액트 컴포넌트 내부에서만 호출 가능합니다!
  // 리액트 엔진은 여기서 "Invalid hook call" 에러를 내뱉으며 앱을 멈춥니다.
  const token = useAuthStore((state) => state.token);

  if (token) {
    // 스토어에서 토큰을 가져올 수 없으니, 헤더에 신분증을 붙일 방법이 없습니다.
    config.headers.Authorization = `Bearer ${token}`;
  }

  return config;
});

export default http;

🔍 정밀 분석: 왜 이 코드가 아키텍처를 파괴할까요?
1. 리액트 생명주기와의 단절
리액트 훅(useAuthStore)은 리액트의 렌더링 사이클 안에서만 살아 숨 쉽니다. 일반 자바스크립트 파일인 .ts 파일은 리액트가 화면을 그리는 것과 상관없이 독립적으로 존재하므로, 훅을 통해 데이터를 수혈받을 통로가 완전히 차단됩니다.

2. 타입 안전성의 붕괴 (The 'Any' Temptation)
이 단절을 해결하겠다고 많은 개발자가 window.token = ... 같은 전역 변수에 값을 복사해두는 편법을 씁니다. 하지만 이 순간, 우리가 정성껏 만든 타입 시스템은 무너지고 any라는 이름의 시한폭탄이 우리 코드 곳곳에 심어지게 됩니다.

3. 기형적인 설계의 탄생
리액트 밖에서 상태를 수정할 방법이 없으니, 리액트 컴포넌트 어딘가에 "외부 메시지를 대신 받아줄 대리인 컴포넌트"를 세워두는 복잡하고 기형적인 구조를 만들게 됩니다. 이는 로직을 파편화시키고 버그가 발생했을 때 추적을 불가능하게 만드는 원흉이 됩니다.

✅ 아키텍트의 체크리스트: 단절의 증상들
런타임 에러: Invalid hook call. Hooks can only be called inside of the body of a function component. 문구를 보게 됨.
중복 로직: 각 API 호출 시마다 컴포넌트 안에서 토큰을 꺼내 인자로 넘겨줘야 하는 번거로움 발생.
테스트 어려움: 순수 로직 파일이 리액트 환경에 의존하게 되어 유닛 테스트 작성이 매우 고달파짐.
