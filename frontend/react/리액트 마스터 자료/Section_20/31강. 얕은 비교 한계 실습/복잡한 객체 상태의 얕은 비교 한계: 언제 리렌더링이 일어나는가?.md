📖 심층 배경지식 가이드 (The Core Background)
지난 시간 우리는 커스텀 로거를 통해 스토어 내부의 데이터 흐름을 투명하게 관찰하는 법을 배웠습니다. 이제 블랙박스를 손에 넣었으니 모든 게 순조로울 것 같았죠?

그런데 로거를 켜두고 앱을 구동하다 보면 아주 기묘한 현상을 목격하게 됩니다. 분명히 유저의 '이름'만 바꿨는데, 유저의 이름과 전혀 상관없는 다른 컴포넌트들까지 우르르 다시 그려지는 현상입니다. 데이터는 평면적으로 잘 관리하고 있는데 왜 이런 불필요한 움직임이 생길까요? 오늘은 리액트 성능 최적화의 최대 난제인 얕은 비교(Shallow Comparison)의 한계와 그로 인한 리렌더링 지옥을 파헤쳐 보겠습니다.

1. 자바스크립트의 메모리 모델: 참조(Reference) vs 값(Value)
자바스크립트에서 객체를 다룰 때 가장 먼저 이해해야 하는 것은 데이터가 메모리에 저장되는 방식입니다.

원시 타입 (Primitive Types): String, Number, Boolean 등은 값 그 자체가 변수에 저장됩니다. 내용이 같으면 같은 것으로 판명됩니다.
참조 타입 (Reference Types): Object, Array 등은 데이터가 담긴 메모리의 주소(참조값)가 변수에 저장됩니다.
핵심 개념: 자바스크립트에서 houseA = { }와 houseB = { }를 생성하면, 두 객체의 내부 프로퍼티가 100% 일치하더라도 houseA === houseB의 결과는 항상 false입니다. 두 객체는 메모리라는 거대한 땅 위에 서로 다른 좌표에 지어진 '똑같이 생긴 집'일 뿐이기 때문입니다.

2. Zustand의 판단 근거: Strict Equality (===)
Zustand는 상태가 변했는지 판단하여 컴포넌트를 다시 그릴지 결정할 때, 기본적으로 엄격한 동등성 비교(Strict Equality)를 수행합니다.

판단 수식: Selector(이전 상태) === Selector(다음 상태)
의미: Selector가 반환한 결과물의 메모리 주소가 바뀌었는가?
📖 상세 개념 가이드 (The Problem Manual)
1. Selector의 배신: 객체 리터럴의 함정
우리가 실무에서 편리함을 위해 자주 사용하는 방식이 사실 성능의 치명적인 구멍이 됩니다.

/* [Pain Case]: 객체 리터럴 형태로 여러 값을 한꺼번에 반환하는 Selector */
// 파일 위치: src/components/UserProfile.tsx

const { name, role } = useAuthStore((state) => ({
  name: state.user?.name,
  role: state.user?.role
}));

🔍 상세 분석 (Why it fails?)

새로운 객체 생성: 위 코드의 (state) => ({ ... }) 부분은 Selector 함수가 실행될 때마다 중괄호 { }를 통해 새로운 객체 리터럴을 메모리에 생성합니다.
무조건적 불일치: 스토어 내부의 어떤 상태(예: 알림 개수)가 변해서 Selector가 재실행되면, 비록 name과 role의 값은 이전과 같더라도 새로 만들어진 객체의 주소는 이전 렌더링 시점의 주소와 다릅니다.
오판: Zustand 엔진은 주소가 달라졌으므로 "데이터가 변했다!"라고 오판하고, 해당 컴포넌트를 강제로 리렌더링합니다.
2. 실무 시나리오: 실시간 대시보드의 비극
상황: 왼쪽 사이드바(유저 정보 표기)와 중앙 차트(1초마다 주식 데이터 업데이트)가 하나의 스토어에 공존합니다.
현상: 중앙의 차트가 업데이트될 때마다, 데이터가 고정된 왼쪽 사이드바가 1초마다 깜빡이며 다시 그려집니다.
원인: 사이드바가 위와 같은 객체 리터럴 방식의 Selector를 사용하고 있기 때문입니다. 차트 배열이 변할 때마다 사이드바는 껍데기(객체 참조)가 바뀌었다는 이유로 불필요하게 렌더링됩니다.
⚖️ 리렌더링 현상 분석 리스트
Zustand에서 발생하는 불필요한 리렌더링의 흐름을 정리해 드립니다.

상태 업데이트 발생: 스토어의 특정 조각(예: chartData)이 변경됨.
모든 Selector 호출: 해당 스토어를 구독 중인 모든 컴포넌트의 Selector 함수가 일제히 실행됨.
참조값 비교: 각 Selector의 반환값에 대해 이전 값 === 다음 값 비교 수행.
오류 발생 지점: 객체 리터럴 { }이나 배열 리터럴 [ ]을 반환하는 Selector는 항상 새로운 참조를 생성하여 비교 결과가 무조건 false가 됨.
결과: 데이터 내용의 변화 여부와 상관없이 관련 컴포넌트들이 모두 다시 그려짐 (성능 저하).
💡 아키텍트의 통찰
여기서 핵심은 "Zustand는 죄가 없다"는 것입니다. Zustand는 약속된 대로 메모리 주소를 체크했을 뿐이고, 매번 새로운 집(객체)을 지어서 건네준 것은 바로 우리의 코드였으니까요.

시니어 개발자는 단순히 기능을 구현하는 것을 넘어, 자신이 짠 Selector 한 줄이 리액트 렌더링 트리에 어떤 파급 효과를 주는지 정확히 예측할 수 있어야 합니다. 자바스크립트의 얕은 비교는 매우 빠르고 효율적이지만, 이를 오용하면 앱 전체를 무겁게 만드는 '유령 리렌더링'의 주범이 됩니다.

🧪 테스트 가이드: 고통을 증명하는 법
로거를 켜둔 상태에서 다음을 시도해 보세요.

현상 1 - 리렌더링 폭주: 유저 정보와 상관없는 count를 증가시켰음에도, 콘솔에는 🚨 [렌더링 발생] 로그가 끝도 없이 찍히게 됩니다.
현상 2 - 무한 루프 에러: 리액트는 "데이터가 변했네? 다시 그려!" → "또 새로운 객체네? 다시 그려!"를 반복하다가 결국 Maximum update depth exceeded 에러를 뿜으며 앱을 중단시킵니다.
결론: Selector에서 { }를 매번 새로 만드는 행위는 성능 저하를 넘어 시스템의 크래시(Crash)를 유발하는 치명적인 범인임이 증명되었습니다.
