# 20. Zustand 상태관리 완전 정복

## 개요

이 섹션은 리액트 전역 상태 관리의 핵심 라이브러리인 **Zustand**를 기초 철학부터 고급 최적화 패턴까지 완전 정복하는 것을 목표로 한다. Context API의 구조적 한계(전염성 리렌더링, Provider 지옥)를 분석한 뒤, Zustand가 이를 어떻게 해결하는지를 단계적으로 학습한다.

**학습 범위:**
- Context API vs Zustand 비교 및 철학적 차이
- Zustand 핵심 API(`create`, `set`, `get`)와 `useSyncExternalStore` 메커니즘
- 슬라이스 패턴 / `StateCreator` 제네릭 / 비동기 액션 설계
- useReducer 대비 Zustand의 선언적 간결함
- Selector 기반 렌더링 최적화 (Strict Equality)
- Persist 미들웨어를 통한 상태 영속성과 하이드레이션
- Immer 결합(Proxy 엔진)으로 중첩 객체 불변성 자동 관리
- Vanilla API(`createStore`)를 통한 리액트 외부 상태 접근
- Provider 지옥 탈출과 Flat State 아키텍처 설계
- 커스텀 미들웨어 제작 (고차 함수 패턴)
- Shallow 비교 최적화(`useShallow`) 및 커스텀 동등성 함수
- 상태 스냅샷과 시간 여행(Undo/Redo) 디버깅
- 실무 기술 선택 기준 (하이브리드 모델)

---

## 강의별 핵심 요약

### 16강. 상태 관리의 세대교체 -- Context API vs Zustand

- **Context API의 한계**: Provider 내부의 어떤 값이든 하나만 바뀌면 구독 중인 모든 하위 컴포넌트가 리렌더링되는 "전염성 리렌더링" 문제가 발생한다. Context는 상태 관리 도구가 아닌 '의존성 주입 도구'이다.
- **Zustand의 핵심 철학**: 리액트 컴포넌트 트리 **외부**에 독립적인 중앙 창고(Store)를 두고, 셀렉터(Selector) 패턴으로 필요한 데이터만 구독한다. 사과를 꺼낸다고 포도를 기다리는 사람이 리렌더링될 일이 없다.
- **코드 간결성**: Context API는 인터페이스 + 컨텍스트 생성 + Provider + 커스텀 훅 등 4단계와 `useCallback`/`useMemo` 최적화가 필요하지만, Zustand는 `create` 함수 하나로 끝난다.
- **전략적 선택 기준**: 빈번한 업데이트가 있는 비즈니스 데이터(사용자 정보, 장바구니)는 Zustand, 거의 바뀌지 않는 정적 설정(다크 모드, 다국어)은 Context API가 적합하다.
- 설치: `npm install zustand` (의존성 없는 초경량 라이브러리)

### 17강. Zustand 엔진의 삼총사 -- create, set, get

- **create**: 스토어를 생성하는 공장 함수. 리액트 컴포넌트 트리 외부의 순수 JS 영역에 상태를 정의하며, 커스텀 훅을 반환한다.
- **set (얕은 병합)**: `set({ username: '새값' })` -- 변경할 속성만 전달하면 나머지는 자동 보존된다. 내부적으로 `Object.assign` 또는 스프레드 연산자와 유사하게 동작한다. 단순한 값 교체에 유리하다.
- **set (함수형 업데이트)**: `set((state) => ({ points: state.points + 10 }))` -- 클로저를 활용해 항상 최신 상태를 참조하므로, 비동기 처리나 짧은 시간 내 연속 업데이트에서 데이터 유실(Race Condition)을 방지한다. 대규모 시스템에서 권장되는 방식.
- **get**: `get().points` -- 리렌더링을 유발하지 않고 현재 스토어의 스냅샷을 읽는다. 액션 내부에서 조건 분기, 로그 기록 등 연산 전용으로 유용하다.
- **useSyncExternalStore**: Zustand가 리액트와 동기화하는 내부 메커니즘. `subscribe`(리스너 등록) + `getSnapshot`(최신 값 가져오기)으로 외부 상태 변경 시 티어링(Tearing) 없이 안전한 렌더링을 보장한다.

### 18강. Zustand 슬라이스 패턴 설계

- **슬라이스 패턴**: 거대한 스토어를 도메인별(예: 화장품, 의류)로 분리하여 유지보수 독립성을 확보하는 아키텍처 전략이다. 백화점의 층별 관리와 같다.
- **StateCreator 제네릭**: `StateCreator<전체스토어, [], [], 현재슬라이스>` 형태로, 각 슬라이스가 전체 스토어 구조를 인지(다른 슬라이스 참조 가능)하면서도 자신의 영역만 책임진다. 빈 배열 `[]`은 미들웨어를 사용하지 않을 때의 예약석이다.
- **import type**: 타입만 가져올 때 반드시 `import type`을 사용해야 런타임에 존재하지 않는 값을 찾는 `SyntaxError`를 방지할 수 있다. 브라우저가 실행될 때 코드가 생성되지 않는 순수 타입 임포트.
- **통합 스토어**: `create<T>()((...a) => ({ ...createSliceA(...a), ...createSliceB(...a) }))` -- `(...a)` Rest 인자로 `set`, `get`, `api` 도구를 통째로 받아 각 슬라이스에 배분하고, 스프레드로 하나의 중앙 통제실 객체로 병합한다.
- **확장성**: 새 도메인이 추가되면 `types.ts`에 인터페이스를 추가하고 새 슬라이스 파일만 만들면 기존 코드를 거의 건드리지 않는다.

### 19강. [Pain] useReducer의 장황함

- **보일러플레이트의 늪**: 하나의 액션을 추가하려면 Action Type 유니온 타입 정의 -> Reducer의 switch-case 수정 -> UI의 dispatch 호출부 수정, 최소 3곳을 동시에 고쳐야 한다.
- **스프레드 지옥**: 상태가 깊어지면 `...state`를 수없이 중첩해야 하는 불변성 유지의 고통이 따른다. 중괄호 하나 위치 실수로 전체 데이터가 유실될 수 있다.
- **dispatch 우체통 방식**: 직접 함수를 실행하는 것이 아니라 `dispatch({ type: '...' })` 객체를 생성하여 간접적으로 상태를 변경해야 하며, 이는 인지 부하를 높인다.
- **useReducer의 적합한 용도**: 특정 컴포넌트 내부에서만 쓰이는 복잡한 로컬 상태가 상태 머신처럼 동작할 때, 또는 외부 라이브러리 없이 리액트 내장 기능만으로 엄격한 로직 분리가 필요할 때.

### 20강. [Solution] Zustand의 간결함

- **선언적 상태 변경**: 함수를 정의하는 행위 자체가 액션이며, 그 내부의 `set` 호출이 리듀서 역할을 동시에 수행한다. Action Type과 switch-case가 완전히 사라진다. `increase`라는 함수 이름이 곧 `type: 'INCREMENT'` 역할.
- **자동 불변성(Shallow Merge)**: `set({ message: nextMessage })` 한 줄이면 나머지 상태는 엔진이 자동 보존한다. "어떻게 복사할까"가 아닌 "무엇을 바꿀까"에 집중하게 된다.
- **타입 추론 극대화**: 함수 매개변수에 타입만 명시하면 payload 타입 추론이 완벽하게 작동하여, Action 유니온 타입이 불필요하다.
- **인지 부하 감소**: `onClick={increase}` 처럼 액션 함수를 이벤트 핸들러에 직접 연결하므로 코드의 인과관계가 명확하다. 스토어 정의부 한 곳에 함수만 추가하면 모든 준비가 끝난다.

### 21강. Selector 최적화

- **선택적 구독(Selective Subscription)**: `useStore((state) => state.bears)` -- 전체 스토어가 아닌 특정 필드만 감시하여, 해당 값이 변할 때만 리렌더링이 발생한다. 거대한 물류센터에서 '사과 가격만 알림' 구독하는 것과 같다.
- **엄격한 비교(Strict Equality)**: Zustand는 셀렉터 반환값을 이전 값과 `===`로 비교하여, 동일하면 리렌더링 신호를 보내지 않는다. 상태 변경 -> 모든 셀렉터 재실행 -> 결과값 비교 -> 다를 때만 알림의 4단계 플로우.
- **렌더링 격리**: 동일한 스토어를 공유하더라도 성능 관점에서는 개별 `useState`를 쓰는 것처럼 완벽하게 분리된다. 곰 컴포넌트와 물고기 컴포넌트가 서로 영향을 주지 않는다.
- **Best Practice**: `const state = useStore()` 처럼 통째로 가져오는 습관을 버리고, 반드시 `(state) => state.part` 형태의 셀렉터를 사용하여 "연비"를 챙겨야 한다.

### 22강. [Pain] 수동 localStorage 관리의 고통

- **휘발성 메모리**: Zustand 스토어는 브라우저 새로고침 시 모든 데이터가 초기화된다. 칠판에 적은 내용을 누군가 매번 지워버리는 것과 같다.
- **수동 관리의 3대 고통**: (1) `JSON.parse`가 `any`를 반환하여 타입 안전성 실종 -- 과거에 저장된 데이터 형식이 현재 코드와 달라졌을 때 런타임 폭탄, (2) 모든 상태마다 `useState` 지연 초기화 + `useEffect` + `JSON.stringify` 동기화 로직 필요, (3) SSR 환경에서 `typeof window !== 'undefined'` 방어 코드 필수 -- `localStorage`는 브라우저에만 존재.
- **주객전도**: 비즈니스 로직보다 저장 로직이 더 길어지는 역전 현상이 발생한다.
- **성능 악영향**: 데이터가 커질수록 `JSON.stringify` 연산 부담이 브라우저 성능을 갉아먹는다.

### 23강. Zustand Persist 미들웨어

- **persist 미들웨어**: `create<T>()(persist((set) => ({...}), { name: 'storage-key' }))` -- 커링 문법으로 미들웨어를 래핑하며, 단 한 줄의 설정으로 상태 영속성을 확보한다.
- **동작 원리 4단계**: (1) persist가 스토어 생성 로직을 감싸 상태 변경을 가로챔, (2) 새 상태를 `JSON.stringify`로 직렬화, (3) `name` 설정값을 key로 `localStorage`에 즉시 저장, (4) 페이지 새로고침 시 `name`으로 창고를 조회하여 `JSON.parse`로 복원(하이드레이션).
- **하이드레이션(Hydration)**: 비어있는 스토어에 과거 데이터를 채워넣는 복구 과정. 사용자는 깜빡임 없이 이전 화면을 그대로 이어본다.
- **저장소 유연성**: 설정값 하나만 바꾸면 `localStorage`에서 `sessionStorage`로 즉시 변경 가능하다.
- **타입 안전성**: 제네릭 기반으로 완벽한 타입 추론을 지원하여 22강의 `any` 공포를 해소한다. SSR 하이드레이션도 자동 지원.

### 24강. Zustand와 Immer 결합

- **Immer 미들웨어**: `create<T>()(immer((set) => ({...})))` -- `set` 함수의 인자가 Proxy 기반의 `draft` 객체로 변환되어 직접 대입(`=`)이 가능해진다.
- **Proxy 엔진 원리**: 원본 데이터를 감싸는 '대리인(비서)' 객체. `draft.count = 1`이라고 명령하면 실제 데이터를 바꾸지 않고 비서가 수정 요청을 수첩에 기록(Interception)한 뒤, 작업 종료 시 변경 부분만 새것으로 갈아 끼운 불변 객체를 생성하여 반환한다.
- **스프레드 지옥 탈출**: `draft.user.profile.settings.theme = 'dark'` 한 줄로, 4단계 깊이의 중첩 객체를 안전하게 수정할 수 있다. 데이터가 아무리 깊어져도 코드 복잡도가 일정하게 유지된다.
- **타입스크립트 공조**: 경로 탐색 도중에도 타입 정보가 완벽히 유지되어 `draft.user.` 까지만 입력해도 자동 완성이 정확하게 작동한다.
- **네이밍 컨벤션**: 인자를 `state` 대신 `draft`로 명명하여 "이 코드는 Proxy 환경에서 안전하게 수정된다"는 의도를 동료에게 전달한다.

### 25강. [Pain] 리액트 밖에서 상태 접근

- **훅 규칙의 벽**: 리액트 훅(`useStore`)은 오직 리액트 컴포넌트 내부에서만 호출 가능하다. Axios 인터셉터, 유틸 함수 등 일반 `.ts` 파일에서는 `Invalid hook call` 에러가 발생한다.
- **실무 시나리오**: API 요청 시 인터셉터(Interceptor, 중간 검문소)에서 토큰을 헤더에 자동 주입하거나, 401 응답 시 자동 로그아웃 처리가 필요한데, 인터셉터는 리액트 밖이다.
- **잘못된 해결책**: `window.token = ...` 같은 전역 변수 편법은 타입 시스템을 파괴하고 `any` 시한폭탄을 심는다. 리액트 컴포넌트에 "대리인 컴포넌트"를 세우는 기형적 설계도 로직을 파편화한다.
- **웹 보안 삼총사**: 인터셉터(검문소), Headers(봉투 겉면 메타데이터), Bearer 토큰(`Authorization: Bearer [토큰값]` -- 소지한 사람에게 접근 권한을 허가하는 신분증 제시 방식).

### 26강. [Solution] Zustand Vanilla API

- **createStore (`zustand/vanilla`)**: 리액트 훅이 아닌 순수 JS Store API 객체를 반환한다. 리액트 의존성이 전혀 없으므로 어떤 JS 환경에서도 동작하며, 리액트 컴포넌트 로드 전에도 메모리에 존재할 수 있다.
- **getState()**: 구독/리렌더링 없이 호출 시점의 최신 상태 스냅샷을 반환한다. 인터셉터는 요청마다 매번 실행되므로, `getState()`도 매번 최신 토큰을 안정적으로 가져온다. 25강의 `Invalid hook call` 에러를 완벽 해결.
- **setState()**: 리액트 외부에서 상태를 직접 변경하며, 이 변경은 해당 상태를 구독 중인 리액트 컴포넌트에 즉시 반영된다. 401 응답 시 자동 로그아웃, 외부 시스템에서 상태 클린업 등에 활용.
- **useStore 브릿지**: `export const useAuth = () => useStore(authStore)` -- 바닐라 엔진을 리액트 훅으로 변환하여, 하나의 엔진으로 Vanilla와 React 두 세계를 동시에 지배하는 구조.

### 27강. Context 지옥

- **Provider 지옥**: 프로젝트가 커지면 Auth, Theme, Cart, Notification, Modal, Language, Search 등 10개 이상의 Provider가 중첩되어 피라미드 형태의 코드가 만들어지며, 닫는 태그 위치 하나만 틀려도 앱 전체가 깨진다.
- **순서 의존성**: `CartProvider`가 `AuthProvider`의 정보를 참조하면, 반드시 Auth가 Cart를 감싸야 하며 순서가 바뀌면 에러가 발생한다. 이 순서를 결정하는 것조차 설계의 짐.
- **테스트 격리 실패**: 최하위 컴포넌트 하나를 테스트하려 해도 10개의 Provider를 Mock으로 세워야 하므로, 유닛 테스트가 통합 테스트 수준으로 무거워진다.
- **RSC와의 충돌**: Context Provider는 본질적으로 클라이언트 컴포넌트이므로, 트리 상단을 Provider로 도배하면 그 아래 모든 컴포넌트가 서버 컴포넌트 최적화 기회를 잃는다.

### 28강. Flat State 아키텍처

- **외부 저장소(External Store)**: Zustand 스토어는 리액트 트리 밖에 독립적으로 존재하므로, Provider로 감싸는 수직적 계층 구조가 불필요하다. 컴포넌트는 자신이 트리의 어디에 있든 외부 저장소에 직접 핫라인을 연결한다.
- **App.tsx 다이어트**: 수십 줄의 Provider 계단이 사라지고 `<DashboardLayout />`만 남는 극단적 간결함을 달성한다. 트리 상단의 불필요한 연산을 제거하여 반응 속도를 극대화한다.
- **RSC 친화력**: `App.tsx`를 Provider로 감쌀 필요가 없으므로, 최상위 구조를 순수 서버 컴포넌트(Pure Server Component)로 유지할 수 있어 번들 크기 최적화에 유리하다.
- **낮은 결합도**: 컴포넌트를 프로젝트의 어떤 위치로 옮기더라도, import 경로만 맞으면 수정 없이 즉시 동작하는 모듈형 아키텍처. 파일 의존성 테스트로 검증 가능.

### 30강. 커스텀 미들웨어

- **미들웨어 원리**: 상태 변경 명령(`set`)이 스토어에 도달하기 전 중간에서 가로채는(Intercept) 레이어. 원본 `set`을 감싸는 Wrapper 함수를 만드는 것이 핵심이다. 건물 입구의 보안 게이트와 같다.
- **커스텀 로거 제작 -- 5단계 흐름**: (1) 명령 포착(loggedSet에 도착) -> (2) 과거 기록(`get()`으로 Prev 스냅샷) -> (3) 명령 이행(원본 `set(...)` 실행, 동기적) -> (4) 미래 기록(`get()`으로 Next 스냅샷) -> (5) 콘솔 출력 및 종료.
- **타입 시스템(T, Mps, Mcs)**: `StateCreator<T, Mps, Mcs>` 제네릭 체인으로, T는 상태 데이터 타입, Mps는 이미 적용된 미들웨어 정보, Mcs는 현재 추가하는 미들웨어 정보. 여러 미들웨어를 중첩해도 타입 추론이 깨지지 않는 안전한 스토어를 보장한다.
- **고차 함수(HOF) 구조**: 원본 스토어 조리법(함수)을 인자로 받아, 로깅 기능이 추가된 새로운 조리법(함수)을 반환한다. 원본 코드를 수정하지 않고 기능을 덧붙이는 우아한 방식.
- **확장 가능성**: 단순 콘솔 출력을 넘어 Sentry, 로그 서버 등 외부 모니터링 서비스로 데이터를 자동 전송하도록 확장 가능하다.

### 31강. 얕은 비교 한계

- **참조 vs 값**: JS에서 원시 타입(String, Number)은 값 자체로 비교되지만, 참조 타입(Object, Array)은 내부 프로퍼티가 100% 동일해도 서로 다른 메모리 주소에 생성되므로 `===` 비교 시 항상 `false`이다.
- **객체 리터럴 셀렉터의 함정**: `useStore((state) => ({ name: state.user?.name }))` -- 실행될 때마다 `{}` 리터럴이 새로운 객체를 생성하므로, 내용이 같아도 참조가 달라 Zustand가 "데이터가 변했다"고 오판하여 매번 리렌더링이 발생한다.
- **실무 시나리오**: 실시간 대시보드에서 차트 데이터가 1초마다 변할 때, 데이터가 고정된 사이드바까지 함께 깜빡이며 다시 그려지는 비극. 차트 배열이 변할 때마다 사이드바의 껍데기(객체 참조)가 바뀌었다고 판단.
- **최악의 결과**: `Maximum update depth exceeded` -- "데이터 변했네? 다시 그려!" -> "또 새 객체네? 다시 그려!"를 반복하다 무한 루프 에러로 앱이 크래시된다. Zustand는 죄가 없고, 매번 새 객체를 건넨 우리 코드가 원인.

### 32강. Shallow 최적화

- **셀렉터 쪼개기(Primitive Selection)**: 각 원시 값을 개별 훅으로 구독하면 값 자체로 비교되어 가장 확실하고 신뢰도 높은 최적화가 된다. 코드가 조금 길어지는 단점.
- **useShallow**: `useShallow((state) => ({ name, role }))` -- 셀렉터가 반환한 새 객체를 한 꺼풀 벗겨 내부의 1단계 속성들을 이전 값과 대조하여, 내용이 같으면 "리액트에게 알리지 마!"라고 명령한다. import 경로: `zustand/react/shallow`. 가독성과 성능의 균형이 가장 우수하며 실무에서 가장 많이 쓰인다.
- **커스텀 동등성 함수**: 훅의 두 번째 인자로 `(prev, next) => boolean` 함수를 전달하여 리렌더링 조건을 100% 개발자가 통제한다. "시간 차이 60초 미만이면 동일 취급" 같은 비즈니스 로직 기반의 임계값 비교에 유용.
- **전략 선택 가이드**: 원시값 -> 기본 비교(Strict Equality), 1단계 객체 -> `useShallow`, 깊은 중첩 -> Deep Equal(lodash `isEqual` 등)/커스텀 함수, 비즈니스 조건 -> 커스텀 동등성 함수.

### 33강. 상태 스냅샷

- **Past/Future Stack**: 사용자가 행동할 때마다 현재 상태를 `past` 배열에 Push하고, Undo 시 현재 값을 `future`로 이동시키는 두 개의 스택 구조로 시간 여행을 구현한다. 디지털 사진첩과 같다.
- **불변성 활용**: Zustand의 `set`이 기존 상태를 수정하지 않고 새 객체를 반환하는 특성 덕분에, 과거 상태 객체를 배열에 담아두는 것만으로 완벽한 기록 시스템이 된다.
- **메모리 최적화**: `.slice(-50)`으로 최근 50개의 스냅샷만 유지하여 메모리 누수를 방지한다.
- **타임라인 분기**: 새로운 액션 발생 시 `future: []`로 초기화하여, Undo 이후 새 분기가 시작되면 이전 미래 기록을 끊고 타임라인이 다시 선형적으로 흐르게 만든다.
- **방어적 UI**: `past.length === 0`일 때 Undo 버튼 비활성화, `future.length === 0`일 때 Redo 비활성화로 불가능한 동작을 원천 차단.

### 34강. Context API vs Zustand 기술 선택 기준

- **기준 1 -- 렌더링 퍼포먼스**: Context는 값 하나만 바뀌어도 모든 구독 컴포넌트가 리렌더링(아파트 전체 방송), Zustand는 셀렉터로 필요한 컴포넌트만 업데이트(전용 호출기). Context는 상태 관리 도구가 아닌 의존성 주입 장치.
- **기준 2 -- 개발자 경험(DX)**: Context는 Provider를 겹겹이 중첩해야 하는 마트료시카 구조, Zustand는 Provider 없이 훅 한 줄로 어디서든 호출하는 평면 구조. "설정보다 규약"을 선호하는 현대적 프론트엔드에 적합.
- **기준 3 -- 생태계와 확장성**: Context는 persist, logging, 시간 여행 등 모든 기능을 직접 구현해야 하는 순정차, Zustand는 검증된 미들웨어 생태계(persist, immer, devtools)를 갖춘 풀옵션차.
- **하이브리드 모델(황금률)**: Context API는 다크 모드/언어 설정 등 정적 설정값, Zustand는 장바구니/사용자 인증/실시간 데이터 등 변화가 잦은 비즈니스 데이터에 사용한다.
- **실무 꿀팁**: 전역 상태가 5개 이상이면 Zustand 필수, 번들 사이즈가 극도로 중요한 초경량 프로젝트만 Context API 고려, 단위 테스트가 중요하면 Provider 없는 Zustand가 더 간편.

## 면접 대비 Q&A

**Q1. Zustand가 Context API보다 리렌더링 성능이 좋은 이유는?**

A: Context API는 Provider의 value가 변경되면 해당 Context를 구독하는 **모든** 하위 컴포넌트가 리렌더링된다. 반면 Zustand는 셀렉터(Selector) 패턴으로 컴포넌트가 스토어의 특정 조각만 구독하며, 해당 조각의 반환값을 `===`(Strict Equality)로 비교하여 **실제 값이 변한 컴포넌트만** 리렌더링한다. 내부적으로는 리액트 18의 `useSyncExternalStore`를 활용하여 외부 상태와 렌더링 사이클을 안전하게 동기화하며, 티어링(Tearing) 현상도 방지한다.

**Q2. Zustand의 `set` 함수에서 얕은 병합과 함수형 업데이트의 차이는?**

A: 얕은 병합(`set({ key: value })`)은 변경할 속성만 전달하면 나머지를 자동 보존하는 간편한 방식이다. 함수형 업데이트(`set((state) => ({ key: state.key + 1 }))`)는 클로저를 통해 항상 최신 상태를 참조하므로, 비동기 처리나 빠른 연속 업데이트에서 데이터 유실(Race Condition)을 방지한다. 불변성이 중요한 대규모 시스템에서는 함수형 업데이트가 권장된다.

**Q3. 슬라이스 패턴이란 무엇이며 왜 사용하는가?**

A: 슬라이스 패턴은 거대한 스토어를 도메인(기능 단위)별로 분리하여 각자 독립적인 파일에서 관리하는 아키텍처이다. `StateCreator<전체스토어, [], [], 현재슬라이스>` 제네릭을 사용해 각 슬라이스가 전체 스토어 타입을 인지하면서도 자기 영역만 구현하고, 최종적으로 `(...a)` Rest 인자와 스프레드로 합쳐진다. 이를 통해 관심사 분리(SoC), 독립적 테스트, 팀 단위 병렬 개발이 가능해지며, 새 도메인 추가 시 기존 코드를 거의 건드리지 않는다.

**Q4. Zustand의 Vanilla API(`createStore`)는 언제 사용하는가?**

A: 리액트 훅 규칙 상 컴포넌트 외부(Axios 인터셉터, 유틸 함수, 일반 `.ts` 파일)에서는 `useStore` 훅을 호출할 수 없다. `createStore`(`zustand/vanilla`)로 생성한 스토어는 순수 JS 객체이므로 `store.getState()`로 어디서든 상태를 읽고, `store.setState()`로 상태를 변경할 수 있다. 대표 사례: API 인터셉터에서 Bearer 토큰 주입, 401 응답 시 자동 로그아웃 처리. `useStore(authStore)`로 바닐라 엔진을 리액트 훅에 연결하면, 하나의 엔진으로 두 세계를 동시에 지배한다.

**Q5. Persist 미들웨어의 동작 원리와 하이드레이션을 설명하라.**

A: Persist 미들웨어는 스토어의 `set`을 감싸서, 상태가 변경될 때마다 자동으로 `JSON.stringify`하여 `localStorage`에 저장한다. 페이지 새로고침 시에는 설정된 name key로 `localStorage`를 조회하고 `JSON.parse`로 복원하여 스토어를 이전 데이터로 채우는데, 이 복구 과정을 하이드레이션(Hydration)이라 한다. 제네릭 기반 타입 추론으로 `any` 위험 없이 타입 안전성을 유지하며, 설정값 하나만 바꾸면 `sessionStorage` 등 다른 저장소로 즉시 전환 가능하다.

**Q6. 셀렉터에서 객체 리터럴을 반환하면 왜 불필요한 리렌더링이 발생하는가?**

A: JS에서 `{}` 리터럴은 실행될 때마다 새로운 메모리 주소에 객체를 생성한다. Zustand는 기본적으로 `===`(참조 비교)로 셀렉터 반환값의 변화를 판단하므로, 내용이 동일해도 새 객체라는 이유로 "데이터가 변했다"고 오판하여 불필요한 리렌더링을 유발한다. 최악의 경우 `Maximum update depth exceeded` 무한 루프 에러로 앱이 크래시된다. 해결책으로는 원시값 개별 구독, `useShallow`, 또는 커스텀 동등성 함수를 사용한다.

**Q7. Immer 미들웨어를 Zustand와 결합하면 어떤 이점이 있는가?**

A: Immer는 Proxy 엔진을 통해 `draft` 객체를 제공한다. 개발자가 `draft.user.settings.theme = 'dark'`처럼 직접 대입하면, Proxy가 이를 가로채서(Interception) 변경 요청을 기록한 뒤, 변경 사항만 반영된 새 불변 객체를 자동 생성한다. 이로써 4단계 이상 깊은 중첩 객체에서 `...state`를 겹겹이 쓰는 "스프레드 지옥"을 탈출하고, 데이터가 아무리 깊어져도 코드 복잡도가 일정하게 유지된다. 타입스크립트 경로 탐색 자동 완성도 완벽히 작동한다.

**Q8. useShallow와 커스텀 동등성 함수의 차이와 사용 시기는?**

A: `useShallow`(`zustand/react/shallow`)는 셀렉터 반환 객체의 1단계 속성들을 이전 값과 얕은 비교하여, 내용이 같으면 리렌더링을 차단한다. 객체 구조 분해 할당의 편리함을 유지하면서 참조 불일치를 해결하며, 대부분의 실무 케이스에 적합하다. 커스텀 동등성 함수(`(prev, next) => boolean`)는 깊은 중첩 객체 비교나, "시간 차이 60초 미만이면 동일 취급" 같은 비즈니스 로직 기반의 정교한 렌더링 통제가 필요할 때 사용한다.

**Q9. 실무에서 Context API와 Zustand를 어떻게 병행하는가?**

A: **하이브리드 모델**이 권장된다. Context API는 다크 모드, 다국어 설정 등 변화가 거의 없는 정적 설정값에 사용하고, Zustand는 장바구니, 사용자 인증, 실시간 데이터 등 변화가 잦고 성능이 민감한 비즈니스 데이터에 사용한다. 전역 상태가 5개 이상이면 Zustand 도입을 강력히 권장하며, 번들 사이즈가 1KB라도 아쉬운 초경량 프로젝트에서만 Context API 단독 사용을 고려한다. 단위 테스트가 중요하다면 Provider 없이 훅만 테스트할 수 있는 Zustand가 유리하다.

**Q10. 커스텀 미들웨어의 구조와 원리를 설명하라.**

A: Zustand 미들웨어는 고차 함수(HOF) 구조를 갖는다. 원본 스토어 생성 함수(`StateCreator`)를 인자로 받아, 기능이 추가된 새로운 생성 함수를 반환한다. 핵심은 원본 `set` 함수를 가로채는 것으로, Wrapper 함수(`loggedSet`) 안에서 변경 전 상태를 `get()`으로 스냅샷 찍고, 원본 `set(...args)`을 실행한 뒤(동기적), 다시 `get()`으로 변경 후 상태를 찍어 대조한다. `StateCreator<T, Mps, Mcs>` 제네릭 체인으로 여러 미들웨어를 중첩해도 타입이 깨지지 않으며, `import type`으로 런타임 에러를 방지한다.

## 용어 글로서리

| 용어 | 설명 |
|---|---|
| **Store (스토어)** | 리액트 트리 외부에 존재하는 전역 상태 저장소. `create` 또는 `createStore`로 생성 |
| **Selector (셀렉터)** | 스토어에서 필요한 데이터 조각만 골라내는 함수. `(state) => state.field` 형태. 선택적 구독의 핵심 |
| **Shallow Merge (얕은 병합)** | `set`에 변경할 속성만 전달하면 나머지를 자동 보존하는 Zustand의 기본 업데이트 방식 |
| **Functional Update (함수형 업데이트)** | `set((state) => ({...}))` 형태로 클로저를 통해 최신 상태를 참조하는 안전한 업데이트 방식 |
| **Slice Pattern (슬라이스 패턴)** | 거대한 스토어를 도메인별로 분리하여 독립적으로 관리하는 아키텍처 전략 |
| **StateCreator** | 슬라이스 생성 시 전체 스토어 타입을 인지하면서 자기 영역을 정의하는 Zustand 제네릭 타입 |
| **Persist (영속성)** | 프로그램 종료/새로고침 후에도 데이터가 사라지지 않고 유지되는 특성 |
| **Hydration (하이드레이션)** | 비어있는 스토어에 저장소의 과거 데이터를 채워넣는 복구 과정 |
| **Serialization (직렬화)** | 객체를 문자열로 변환하는 과정 (`JSON.stringify`) |
| **Deserialization (역직렬화)** | 문자열을 객체로 복원하는 과정 (`JSON.parse`) |
| **Immer / Draft** | Proxy 기반으로 불변성을 자동 관리하는 라이브러리. `draft` 객체에 직접 대입하면 안전한 새 객체 생성 |
| **Proxy (프록시)** | JS의 대리인 객체. 속성 접근/수정을 가로채서(Interception) 중간 로직을 삽입할 수 있는 메타 프로그래밍 도구 |
| **Vanilla API** | `zustand/vanilla`의 `createStore`로 생성한 리액트 비의존 Store API 객체 |
| **getState()** | Vanilla API에서 리렌더링 없이 현재 상태 스냅샷을 즉시 반환하는 메서드 |
| **setState()** | Vanilla API에서 리액트 외부에서 상태를 직접 변경하는 메서드. 구독 컴포넌트에 즉시 반영 |
| **useStore** | 바닐라 스토어를 리액트 훅으로 변환하는 브릿지 함수 |
| **useShallow** | 셀렉터 반환 객체의 1단계 속성을 얕은 비교하여 불필요한 리렌더링을 차단하는 유틸. `zustand/react/shallow` |
| **Strict Equality (===)** | Zustand의 기본 비교 전략. 메모리 주소(참조)가 같은지 확인 |
| **Middleware (미들웨어)** | `set` 호출을 가로채서 추가 로직(로깅, 저장, 검증 등)을 삽입하는 레이어 |
| **HOF (고차 함수)** | 함수를 인자로 받거나 반환하는 함수. 미들웨어의 핵심 구조 |
| **Tearing (티어링)** | Concurrent Rendering 중 같은 상태를 참조하는 컴포넌트가 서로 다른 값을 표시하는 현상 |
| **useSyncExternalStore** | 외부 상태와 리액트 렌더링 사이클을 안전하게 동기화하는 리액트 18+ 전용 훅 |
| **Boilerplate (보일러플레이트)** | 기능에 비해 지나치게 장황하고 반복적인 코드 |
| **Provider Hell** | Context API 사용 시 수십 개의 Provider가 중첩되어 가독성과 성능이 저하되는 현상 |
| **Flat State Architecture** | Provider 중첩 없이 모든 스토어가 수평적으로 존재하는 Zustand 기반 아키텍처 |
| **External Store (외부 저장소)** | 리액트 컴포넌트 트리 밖에 독립적으로 존재하는 데이터 센터. Zustand 스토어의 본질 |
| **Bearer Token** | `Authorization: Bearer [토큰]` 형태로 서버에 인증 정보를 전달하는 표준 방식 |
| **Interceptor (인터셉터)** | API 요청/응답을 가로채서 인증 도장을 찍거나 에러를 처리하는 중간 검문소 |
| **Time Travel (시간 여행)** | Past/Future 스택으로 상태를 기록하여 Undo/Redo를 구현하는 디버깅 기법 |
| **Snapshot (스냅샷)** | 특정 시점의 상태를 사진 찍듯 캡처하여 보관하는 기록 |
| **import type** | 런타임에 코드가 생성되지 않는 순수 타입 임포트. 빌드 에러 방지에 필수 |

## 자가 점검 체크리스트

- [ ] Context API의 전염성 리렌더링 문제와 Zustand가 이를 해결하는 원리(셀렉터 + `===` 비교 + `useSyncExternalStore`)를 설명할 수 있다.
- [ ] `create`, `set`(얕은 병합/함수형), `get`의 역할 차이를 코드로 작성할 수 있다.
- [ ] 슬라이스 패턴으로 스토어를 도메인별 분리하고 `StateCreator` 제네릭으로 타입을 연결하며, `import type`으로 런타임 에러를 방지할 수 있다.
- [ ] useReducer의 보일러플레이트(Action Type -> Reducer -> Dispatch 3곳 수정)와 Zustand의 선언적 방식(함수 하나가 곧 액션+리듀서)의 차이를 비교 설명할 수 있다.
- [ ] 셀렉터를 사용하여 필요한 데이터만 구독하고, 콘솔 로그로 렌더링 격리를 검증할 수 있다.
- [ ] `persist` 미들웨어를 적용하여 새로고침 후에도 상태가 유지되는 스토어를 구현하고, 하이드레이션 과정을 설명할 수 있다.
- [ ] Immer 미들웨어를 결합하여 깊은 중첩 객체를 `draft` 대입으로 안전하게 수정하고, Proxy 엔진의 Interception 원리를 설명할 수 있다.
- [ ] `createStore` (Vanilla API)로 Axios 인터셉터에서 `getState()`로 토큰을 읽고, `setState()`로 401 시 자동 로그아웃을 구현할 수 있다.
- [ ] Provider 지옥의 문제점(순서 의존성, 테스트 격리 실패)과 Flat State 아키텍처의 이점을 RSC 관점에서 설명할 수 있다.
- [ ] 커스텀 미들웨어(로거)를 고차 함수 구조로 직접 작성하고, `T/Mps/Mcs` 타입 체인의 역할을 설명할 수 있다.
- [ ] 객체 리터럴 셀렉터의 참조 불일치 문제를 인지하고, `useShallow` 또는 커스텀 동등성 함수로 해결할 수 있다.
- [ ] Past/Future 스택 기반의 Undo/Redo 시스템(상태 스냅샷)을 설계하고, 메모리 최적화(`.slice(-50)`)와 타임라인 분기(`future: []`)를 적용할 수 있다.
- [ ] 프로젝트 상황에 맞게 Context API와 Zustand를 하이브리드로 조합하는 3가지 기준(렌더링 퍼포먼스, DX, 생태계)을 제시할 수 있다.
