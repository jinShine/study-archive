## 37강. useLayoutEffect -- 시각적 결함 없는 DOM 제어

### 도입

`useEffect`는 화면이 **이미 그려진 후**에 실행된다. 로직이 완벽해도 초기값이 먼저 Paint되고 나서 보정값이 적용되면 사용자는 **깜빡임(Flicker/Jank)** 을 목격한다. `useLayoutEffect`는 이 문제를 근본적으로 해결하기 위해 탄생했다.

### 개념 설명

#### 리액트 렌더링 파이프라인

```
Render 단계       Commit 단계             Paint 단계
(가상 DOM 계산) -> (실제 DOM 반영) ------> (브라우저 픽셀 렌더링)
                         |                      |
                  useLayoutEffect 실행    useEffect 실행
                  (Paint 차단, 동기)      (Paint 이후, 비동기)
```

| 구분 | useEffect | useLayoutEffect |
| ---- | --------- | --------------- |
| 실행 시점 | Paint **이후** (비동기) | Commit 직후, Paint **이전** (동기) |
| 사용자 인지 | 초기값이 잠깐 보임 (깜빡임 가능) | 보정 완료된 최종 결과만 보임 |
| 브라우저 차단 | 차단하지 않음 | **차단함** -- 로직이 끝날 때까지 Paint 대기 |
| 비유 | 막이 오른 뒤 스태프가 소품 교체 | 막이 오르기 전 무대 세팅 완료 |

#### 깜빡임 발생 흐름 (useEffect 사용 시)

1. **Paint 완료** -- 브라우저가 초기값(예: `0`)을 화면에 그림
2. **useEffect 실행** -- 새 값 계산
3. **재렌더링** -- 화면 다시 그림
4. 결과: 사용자 눈에 `0 -> 새 값`으로 변하는 과정이 노출

#### 깜빡임 해결 흐름 (useLayoutEffect 사용 시)

1. **Commit 완료** -- DOM에 반영되었지만 아직 화면에는 안 보임
2. **useLayoutEffect 실행** -- 값 계산 및 보정 (Paint 차단 중)
3. **Paint** -- 보정이 끝난 최종 결과만 화면에 표시
4. 결과: 사용자는 깜빡임 없이 완벽한 화면만 봄

### 코드 예제

#### 기본 신택스

```jsx
import { useLayoutEffect } from 'react';

useLayoutEffect(() => {
  // 실행 로직: 실제 DOM 수치 측정 또는 스타일 동기 수정

  return () => {
    // 클린업: 다음 실행 전이나 언마운트 시 정리
  };
}, [/* 의존성 배열 */]);
```

#### 실제 DOM 너비 측정

```jsx
import { useState, useLayoutEffect, useRef } from 'react';

export default function BasicLayoutEffect() {
  const elementRef = useRef(null);
  const [width, setWidth] = useState(0);

  useLayoutEffect(() => {
    if (elementRef.current) {
      // 브라우저 레이아웃 엔진에게 실제 수치를 직접 질의
      const rect = elementRef.current.getBoundingClientRect();
      setWidth(rect.width);
    }
  }, []); // 마운트 시 1회 측정

  return (
    <div ref={elementRef} style={{ width: '50%', background: 'lightgrey', padding: '20px' }}>
      내 실제 너비는 <b>{width}px</b> 입니다.
    </div>
  );
}
```

#### 팝업 정중앙 위치 보정

```jsx
import { useState, useLayoutEffect, useRef } from 'react';

export default function PopupCenterSystem() {
  const [isOpen, setIsOpen] = useState(false);
  const [offset, setOffset] = useState(0);
  const popupRef = useRef(null);

  useLayoutEffect(() => {
    if (isOpen && popupRef.current) {
      // Paint 직전에 팝업의 실제 높이를 가져와 보정값 계산
      const height = popupRef.current.offsetHeight;
      setOffset(height / 2);
    }
  }, [isOpen]);

  return (
    <div style={{ textAlign: 'center', padding: '50px' }}>
      <button onClick={() => setIsOpen(!isOpen)}>팝업 토글</button>
      {isOpen && (
        <div
          ref={popupRef}
          style={{
            position: 'fixed',
            top: `calc(50% - ${offset}px)`,
            left: '50%',
            transform: 'translateX(-50%)',
            padding: '40px',
            backgroundColor: 'white',
            boxShadow: '0 10px 30px rgba(0,0,0,0.3)',
          }}
        >
          <p>깜빡임 없는 완벽한 정중앙 팝업</p>
        </div>
      )}
    </div>
  );
}
```

### 코드 해설

- **`getBoundingClientRect()`** -- 브라우저 레이아웃 엔진이 계산한 실제 물리적 수치(너비, 높이, 좌표)를 반환한다. 가상 DOM이 아닌 **실제 DOM**의 값이다.
- **`useLayoutEffect` 내부 setState** -- 일반적으로 이펙트 안에서 setState는 리렌더링을 유발하지만, `useLayoutEffect`에서는 리액트가 Paint 전에 상태 업데이트를 **일괄 처리(batching)** 한다. 사용자는 불완전한 화면을 보지 못하고 보정된 최종 결과물만 본다.
- **팝업 보정 공식** -- `top: calc(50% - offset)` 에서 offset은 팝업 높이의 절반. 뷰포트 중앙(50%)에서 팝업 높이의 절반만큼 위로 올려 시각적 정중앙을 달성한다.

### 실무 비유

> 연극 공연에서 **useEffect**는 막이 올라 관객이 보는 도중에 스태프가 소품을 교체하는 것이다. **useLayoutEffect**는 막이 오르기 전에 모든 세팅을 끝내놓아 관객은 처음부터 완벽한 장면만 보게 되는 것이다.

### 핵심 포인트

| 항목 | 설명 |
| ---- | ---- |
| 사용 권장 | 레이아웃 측정, 팝업/툴팁 위치 보정, 애니메이션 시작점 설정 |
| 사용 지양 | API 호출, 로그 기록, 단순 상태 변경 등 레이아웃과 무관한 작업 |
| 주의사항 | 내부 로직이 무거우면 **Paint가 차단**되어 화면이 멈춘 것처럼 보임 |
| 기본 원칙 | 먼저 `useEffect`로 시도하고, 깜빡임이 발생할 때만 `useLayoutEffect`로 전환 |

## 38강. useId -- 하이드레이션을 지키는 고유 ID 전략

### 도입

서버 사이드 렌더링(SSR) 환경에서 `Math.random()`이나 카운터 변수로 ID를 생성하면 서버와 클라이언트에서 값이 달라져 **하이드레이션 불일치(Hydration Mismatch)** 오류가 발생한다. `useId`는 컴포넌트의 트리 내 위치를 기반으로 양쪽에서 동일한 ID를 보장하는 React 18 내장 훅이다.

### 개념 설명

#### 기존 방식의 문제

```
서버: Math.random() -> 0.7234   (HTML 생성)
클라이언트: Math.random() -> 0.1592  (하이드레이션)
-> ID 불일치 -> React 경고 및 UI 깨짐
```

#### useId의 해결 원리

| 핵심 | 설명 |
| ---- | ---- |
| ID 발급 기준 | 컴포넌트의 **트리 내 절대적 위치** (랜덤 아님) |
| 일관성 보장 | 서버와 클라이언트에서 같은 위치 = 같은 ID |
| 출력 형태 | `:r1:`, `:r2:` 등 콜론이 포함된 문자열 |
| 콜론 포함 이유 | CSS 선택자에서 특수 문자 취급 -> ID를 CSS 스타일링에 직접 사용하는 나쁜 습관 방지 |

#### 주요 용도

- **웹 접근성(a11y)** -- `<label htmlFor>` 과 `<input id>` 연결
- **SSR 안전** -- 서버/클라이언트 ID 동기화
- **다중 인스턴스** -- 같은 컴포넌트를 여러 번 렌더링해도 충돌 없음

### 코드 예제

#### 단일 ID -- 웹 접근성 연결

```jsx
import { useId } from 'react';

export default function AccessibilityInput({ label }) {
  // 컴포넌트 인스턴스 전용 고유 ID (예: ":r1:")
  const id = useId();

  return (
    <div style={{ marginBottom: '20px' }}>
      <label htmlFor={id} style={{ display: 'block', fontWeight: 'bold' }}>
        {label}
      </label>
      <input
        id={id}
        type="text"
        placeholder="내용을 입력해주세요"
        style={{ padding: '10px', width: '100%' }}
      />
    </div>
  );
}
```

#### 접미사(Suffix) 전략 -- 멀티 ID 관리

```jsx
import { useId } from 'react';

export default function RegistrationForm() {
  // 하나의 루트 ID 발급
  const baseId = useId();

  return (
    <form style={{ padding: '30px', border: '1px solid #eee' }}>
      <h2>회원 가입 정보</h2>

      <section>
        <label htmlFor={baseId + '-lastName'}>성</label>
        <input id={baseId + '-lastName'} type="text" />
      </section>

      <section>
        <label htmlFor={baseId + '-firstName'}>이름</label>
        <input id={baseId + '-firstName'} type="text" />
      </section>
    </form>
  );
}
```

### 코드 해설

- **`useId()` 호출** -- 컴포넌트 인스턴스마다 고유한 문자열 반환. 같은 컴포넌트를 2번 렌더링하면 서로 다른 ID가 생성된다.
- **`htmlFor` + `id` 연결** -- 라벨을 클릭하면 연결된 입력창에 자동 포커스. 스크린 리더가 라벨과 입력창의 관계를 파악하여 접근성 향상.
- **접미사 패턴** -- `useId()`를 여러 번 호출하는 대신, 하나의 `baseId`에 `-lastName`, `-firstName` 등을 붙여 파생. 루트가 유일하므로 파생 ID도 전체 페이지에서 유일함이 보장된다.

### 실무 비유

> `useId`는 건물의 **호실 번호 체계**와 같다. `Math.random()`은 매번 주사위를 굴려 번호를 정하는 것이라 관리실(서버)과 입주자(클라이언트)가 서로 다른 번호를 가지게 된다. 반면 `useId`는 건물 구조(트리 위치)에 따라 자동 부여되므로 누가 확인해도 같은 번호다.

### 핵심 포인트

| 항목 | 설명 |
| ---- | ---- |
| 사용 대상 | `htmlFor`/`id` 연결, `aria-describedby` 등 접근성 속성 |
| 사용 금지 | 리스트의 `key`, CSS 선택자, 데이터 식별자 |
| 접미사 패턴 | `baseId + '-suffix'`로 하나의 ID에서 여러 ID 파생 |
| SSR 안전 | 서버와 클라이언트에서 100% 동일한 값 보장 |
| 콜론 포함 | 의도적 설계 -- CSS 직접 사용 방지, 관심사 분리 유도 |

## 면접 핵심 Q&A

**Q1. useEffect와 useLayoutEffect의 차이를 설명하세요.**

> `useEffect`는 브라우저가 화면을 Paint한 **이후** 비동기로 실행되므로 사용자가 초기값을 잠깐 볼 수 있다. `useLayoutEffect`는 DOM Commit 직후 Paint **이전**에 동기로 실행되어 보정이 끝난 최종 결과만 화면에 표시된다. 다만 동기 실행이므로 내부 로직이 무거우면 Paint가 차단되어 UI가 멈출 수 있다.

**Q2. useLayoutEffect를 모든 곳에 쓰면 안 되는 이유는?**

> `useLayoutEffect`는 브라우저의 Paint를 **차단(Blocking)** 한다. 내부에서 무거운 연산이나 네트워크 요청을 수행하면 화면 렌더링이 지연되어 사용자 경험이 오히려 나빠진다. 기본적으로 `useEffect`를 사용하고, 깜빡임이 실제로 발생하는 경우에만 `useLayoutEffect`로 전환하는 것이 올바른 전략이다.

**Q3. useId는 왜 Math.random()보다 안전한가요?**

> `Math.random()`은 호출 시점마다 다른 값을 반환하므로 SSR에서 서버가 생성한 ID와 클라이언트 하이드레이션 시 생성한 ID가 달라져 불일치 오류가 발생한다. `useId`는 컴포넌트의 **트리 내 위치**를 기반으로 ID를 생성하므로 서버와 클라이언트에서 동일한 값이 보장된다.

**Q4. useId가 생성하는 ID에 콜론(:)이 포함된 이유는?**

> CSS 선택자에서 콜론은 특수 문자로 취급되어 일반적인 방법으로는 스타일을 적용하기 어렵다. 이는 리액트 팀이 의도적으로 설계한 것으로, 개발자가 ID를 CSS 선택자로 직접 사용하는 나쁜 습관을 방지하고 **관심사 분리**(ID는 접근성 용도, 스타일은 class 사용)를 유도하기 위함이다.

**Q5. 하나의 컴포넌트에서 여러 ID가 필요할 때 useId를 여러 번 호출해야 하나요?**

> 아니다. **접미사(Suffix) 패턴**을 사용한다. `useId()`를 한 번 호출하여 `baseId`를 받고, `baseId + '-email'`, `baseId + '-password'`처럼 접미사를 붙여 파생 ID를 만든다. 루트 ID가 유일하므로 파생 ID도 전체 페이지에서 충돌하지 않는다.

## 용어 글로서리

| 용어 | 설명 |
| ---- | ---- |
| Flicker (깜빡임) | 초기값이 먼저 화면에 표시된 후 보정값으로 바뀌면서 사용자에게 번쩍임이 보이는 현상 |
| Jank | 프레임 드롭이나 레이아웃 재계산으로 인해 UI가 끊기거나 덜컹거리는 현상 |
| Paint | 브라우저가 DOM 변경사항을 실제 픽셀로 화면에 그리는 단계 |
| Commit | React가 가상 DOM 비교 결과를 실제 DOM에 반영하는 단계 (아직 화면에는 안 보임) |
| 동기 실행 | 해당 코드가 완료될 때까지 다음 단계(Paint)가 대기하는 실행 방식 |
| 하이드레이션 (Hydration) | 서버가 생성한 HTML에 React가 이벤트 핸들러를 연결하여 상호작용 가능하게 만드는 과정 |
| 하이드레이션 불일치 | 서버 HTML과 클라이언트 렌더링 결과가 달라 React가 경고를 발생시키는 상황 |
| 접근성 (a11y) | 장애를 가진 사용자도 웹 콘텐츠를 동등하게 이용할 수 있도록 보장하는 설계 원칙 |
| `getBoundingClientRect()` | 요소의 실제 크기와 뷰포트 대비 위치를 반환하는 브라우저 API |
| Suffix Pattern | 하나의 base ID에 접미사를 붙여 여러 고유 ID를 파생하는 기법 |
