# 37강. useLayoutEffect 배경 -- 시각적 결함과 탈출구

## 도입

리액트의 깊은 내부 동작 원리를 탐구하는 시간이다. 지금까지 우리는 `useEffect`라는 도구를 통해 대부분의 비동기 작업을 처리해 왔다.

하지만 실무의 정교한 UI 세계에서는 0.1초의 찰나에 발생하는 '시각적 결함(Jank)'이 서비스의 퀄리티를 결정짓기도 한다. 왜 `useEffect`만으로는 부족한 순간이 오는지, 그리고 리액트가 숨겨둔 비장의 카드인 `useLayoutEffect`가 어떻게 화면 깜빡임을 해결하는지 그 공학적 배경을 파헤쳐 본다.

## 개념 설명

### 화면 깜빡임(Flicker)이란?

로직은 완벽한데 화면이 '번쩍'하며 요소가 순간적으로 이동하는 현상이다. 이는 `useEffect`의 비동기적 특성 때문에 발생한다.

1. **1단계 [Paint 완료]**: 브라우저가 화면에 초기값(예: 0)을 이미 그려버린다.
2. **2단계 [useEffect 실행]**: 화면이 그려진 직후, 뒤늦게 로직이 실행되어 새로운 값을 계산한다.
3. **3단계 [재렌더링]**: 값이 바뀌었으므로 리액트가 화면을 다시 그린다.
4. **결과**: 사용자 눈에는 '0' → '새 숫자'로 변하는 과정이 노출되어 '깜빡임'으로 인식된다.

### 리액트의 렌더링 파이프라인

이 문제를 근본적으로 해결하려면 리액트 엔진이 화면을 업데이트하는 순서를 명확히 알아야 한다.

- **Render 단계**: 컴포넌트 함수를 실행해 바뀔 내용을 계산하는 가상 DOM 작업
- **Commit 단계**: 계산된 내용을 실제 DOM에 반영하는 단계 (아직 화면엔 안 보임)
- **Paint 단계**: 브라우저가 변경된 DOM을 바탕으로 실제 픽셀을 화면에 그리는 단계

핵심 차이점:

- `useEffect`: Paint 단계가 완전히 끝난 후에 실행된다. (사용자가 이미 화면을 본 후 실행)
- `useLayoutEffect`: Commit 직후, Paint 직전에 동기적으로 실행된다. (사용자가 보기 전 실행)

## 실무 비유

이 두 훅의 결정적인 차이를 연극 무대에 비유하면 매우 명확해진다.

- **useEffect (비동기)**: 연극의 막이 오르고 관객이 보고 있는데 스태프가 나와 소품을 바꾸는 것. 관객은 어수선한 과정을 모두 보게 된다(깜빡임 발생).
- **useLayoutEffect (동기)**: 막이 오르기 전, 무대 뒤에서 모든 세팅을 완벽히 끝내는 것. 관객은 처음부터 완벽한 장면만 보게 된다(깜빡임 없음).

## 핵심 포인트

- `useLayoutEffect`는 브라우저의 페인팅을 **차단(Blocking)**한다. 이 훅 내부의 로직이 너무 무거우면 사용자는 화면이 멈춘 것 같은 답답함을 느끼게 된다.
- **사용 권장**: 레이아웃 측정, 팝업 위치 보정, 애니메이션 시작점 설정 등 시각적 연속성이 필수일 때
- **사용 지양**: 데이터 통신(API 호출), 로그 기록, 단순 상태 변경 등 레이아웃과 무관한 작업

- [ ] `useLayoutEffect`는 브라우저가 화면을 그리기 전에 실행되어 깜빡임을 원천 차단한다는 점을 파악했는가?
