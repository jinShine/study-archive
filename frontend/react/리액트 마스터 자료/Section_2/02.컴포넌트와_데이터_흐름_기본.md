# 컴포넌트와 데이터 흐름 기본

---

## 컴포넌트 — UI를 이루는 최소 단위

### 컴포넌트의 정의와 핵심 규칙

**컴포넌트 = JSX를 반환하는 함수.** React에서는 함수가 JSX(설계도)를 반환하면 그것이 곧 컴포넌트다.

**핵심 규칙:**

| 규칙 | 설명 |
|------|------|
| **대문자 시작** | `<Hello />`는 사용자 정의 컴포넌트, `<div>`는 브라우저 태그 |
| **하나의 루트 반환** | React가 서브트리 단위로 비교·교체하기 때문 |
| **순수 함수처럼** | 같은 props → 같은 결과를 반환해야 예측 가능 |

### 대문자/소문자와 Babel 변환

```jsx
<div />     // → React.createElement("div", ...)     ← 문자열 (브라우저 태그)
<Hello />   // → React.createElement(Hello, ...)    ← 식별자 (함수 참조)
```

소문자는 **"DOM 요소 타입 문자열"**, 대문자는 **"코드 상의 함수 참조"**로 변환된다. 컴포넌트가 import 안 되었으면 식별자를 못 찾아 에러가 난다.

### 가장 기본적인 컴포넌트

```jsx
// 1) 정의
function Hello() {
  return <h1>Hello, React!</h1>;
}

// 2) 사용
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<Hello />);
```

**내부 동작:**

```
<Hello />
  → React.createElement(Hello, null)     // Babel 변환
  → Hello() 호출 → <h1>...</h1> 반환      // React Element 생성
  → VDOM diff → 실제 DOM 패치             // ReactDOM 처리
```

### Fragment — 불필요한 div 없이 여러 요소 반환

```jsx
function Card() {
  return (
    <>
      <h2>제목</h2>
      <p>설명 내용</p>
    </>
  );
}
```

- `<></>` (짧은 Fragment): DOM에 추가 노드를 만들지 않고 자식만 묶는 래퍼
- **key가 필요하면** `<React.Fragment key={...}>` 긴 문법을 사용

```
Card()
└─ Fragment <>
   ├─ <h2>제목</h2>
   └─ <p>설명 내용</p>
(최상위 하나의 루트 OK, 불필요한 <div> 추가 없음)
```

### 컴포지션(합성) — 작은 조각으로 큰 UI 만들기

```jsx
function Emoji() {
  return <div className="emoji" aria-label="dev">🧑‍💻</div>;
}
function Title() {
  return <h1>Welcome</h1>;
}
function Header() {
  return (
    <header className="header">
      <Emoji />
      <Title />
    </header>
  );
}

root.render(<Header />);
```

```
Header
└─ <header.header>
   ├─ <Emoji>  → <div.emoji>🧑‍💻</div>
   └─ <Title>  → <h1>Welcome</h1>
```

각각 역할이 분리된 작은 컴포넌트를 조합해 상위 UI 구조를 만든다. 재사용·테스트·교체가 쉽다.

## Props — 컴포넌트의 외부 입력값

### Props의 정의와 호출 모델

**Props(Properties)** = 컴포넌트에 전달되는 **입력값 묶음(객체)**.

```jsx
// 이 셋은 같은 일을 한다:
<Greeting name="Yongsu" age={20} />                          // JSX
React.createElement(Greeting, { name: "Yongsu", age: 20 })  // Babel 변환
Greeting({ name: "Yongsu", age: 20 })                       // 함수 관점
```

속성들이 하나의 객체로 수집되어 **첫 번째 인자(props)**로 전달된다.

### 기본 사용

```jsx
function Greeting(props) {
  console.log(props); // { name: "Yongsu" }
  return <h1>Hello, {props.name}!</h1>;
}

root.render(<Greeting name="Yongsu" />);
```

### JSX 속성 이름 규칙 — HTML과의 차이

```jsx
// HTML                                    // JSX
<label for="email" class="label">          <label htmlFor="email" className="label">
<button onclick="alert('hi')">            <button onClick={() => alert('hi')}>
```

| HTML | JSX | 이유 |
|------|-----|------|
| `class` | `className` | JS 예약어 `class`와 충돌 |
| `for` | `htmlFor` | JS 예약어 `for`와 충돌 |
| `onclick="..."` | `onClick={fn}` | 문자열이 아니라 **함수 참조** 전달 |

### 따옴표 vs 중괄호

```jsx
<Greeting name="Yongsu" />        // 문자열 리터럴
<Greeting name={userName} />      // 변수 (표현식 평가)
<Greeting age={20 + 1} />         // 연산 결과 (21)
<Greeting isAdmin={true} />       // 불리언
```

### 구조 분해와 기본값

```jsx
function Greeting({ name = "Guest" }) {
  return <h1>Hello, {name}!</h1>;
}

root.render(<Greeting />);            // "Hello, Guest!"
root.render(<Greeting name="Kim" />); // "Hello, Kim!"
```

### Props는 읽기 전용 — 파생 값은 새 변수로

```jsx
function Price({ amount }) {
  // amount = amount * 1.1;  // ❌ props 직접 수정 금지!
  const withTax = amount * 1.1;   // ✅ 파생 값은 새 변수
  return <span>{withTax}원</span>;
}
```

부모가 내려준 값은 **읽기 전용**. 변경이 필요하면 부모가 새 값을 내려줘야 한다.

### Rest/Spread — 속성 위임 패턴

```jsx
function Input({ label, ...inputProps }) {
  return (
    <label>
      {label}
      <input {...inputProps} />
    </label>
  );
}

root.render(
  <Input label="이름" type="text" placeholder="홍길동" required />
);
```

```
props = { label, type, placeholder, required }
         │
         ├─ label → 분리해서 텍스트로 사용
         └─ inputProps = { type, placeholder, required } → <input>에 스프레드
```

`label`만 꺼내고 나머지를 `<input>`에 일괄 전달. 폼 컴포넌트에서 자주 쓰는 패턴이다.

## 단방향 데이터 흐름과 children

### 단방향 데이터 흐름 (부모 → 자식)

```jsx
function Child({ user }) {
  return <h2>Welcome, {user}</h2>;
}
function Parent() {
  return <Child user="Yongsu" />;
}
```

```
Parent (데이터 소유자)
   │ props.user
   ▼
Child → 화면: "Welcome, Yongsu"

(자식이 부모 데이터를 바꾸려면? → 콜백 함수 / 상태 리프트업)
```

### children — 태그 안쪽 콘텐츠 전달

```jsx
function Panel({ children }) {
  return <div className="panel">{children}</div>;
}

root.render(
  <Panel>
    <p>공지사항입니다.</p>
    <button>확인</button>
  </Panel>
);
```

열고 닫는 태그 사이의 모든 콘텐츠가 `props.children`에 담긴다. 문자열, 요소, 배열 뭐든 가능.

```
Panel({ children: [<p/>, <button/>] })
→ <div.panel>
    ├─ <p>공지사항입니다.</p>
    └─ <button>확인</button>
  </div>
```

이 패턴으로 **레이아웃/컨테이너 컴포넌트**를 쉽게 추상화할 수 있다.

## 리스트 렌더링 — map, filter, key

### map 기본 — 배열을 JSX로 변환

```jsx
const fruits = ["사과", "바나나", "체리"];

function FruitList() {
  return (
    <ul>
      {fruits.map((fruit, index) => (
        <li key={index}>{fruit}</li>
      ))}
    </ul>
  );
}
```

`map`은 각 원소를 JSX 요소로 변환해 **배열을 반환**한다. JSX의 `{}`는 표현식 슬롯이므로 배열을 그대로 렌더할 수 있다.

**화살표 함수 블록 주의:**

```jsx
// ✅ 즉시 반환 (소괄호)
{fruits.map(fruit => (<li>{fruit}</li>))}

// ❌ 블록인데 return 없음 → undefined, 렌더링 안 됨
{fruits.map(fruit => { <li>{fruit}</li> })}

// ✅ 블록이면 return 필수
{fruits.map(fruit => {
  return <li>{fruit}</li>;
})}
```

### filter + map 체이닝 — 조건부 리스트

```jsx
const products = [
  { id: 1, name: "Keyboard", price: 30000 },
  { id: 2, name: "Mouse",    price: 15000 },
  { id: 3, name: "Monitor",  price: 220000 },
];

function CheapProducts() {
  return (
    <ul>
      {products
        .filter(p => p.price < 20000)
        .map(p => (
          <li key={p.id}>
            {p.name} — {p.price.toLocaleString()}원
          </li>
        ))}
    </ul>
  );
}
```

```
products
  │ filter(price < 20000)
  ▼
[Mouse(15000)]
  │ map(→ <li key=id>…)
  ▼
<ul>
  <li key=2>Mouse — 15,000원</li>
</ul>
```

`filter`로 원하는 데이터만 추출하고, `map`으로 JSX 변환. **원본 배열은 변경되지 않는다**(불변성 유지).

### key — 리스트의 식별자

```jsx
const users = [
  { id: 1, name: "Kim" },
  { id: 2, name: "Lee" },
  { id: 3, name: "Park" },
];

function UserList() {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

**key의 diff 동작:**

```
이전: [ id:1(Kim), id:2(Lee), id:3(Park) ]
이후: [ id:1(Kim), id:3(Park), id:4(Choi) ]

key 비교:
- id:1 → 재사용
- id:2 → 제거
- id:3 → 재사용
- id:4 → 신규 마운트
```

| key 전략 | 적합한 경우 | 주의점 |
|----------|-------------|--------|
| **고유 ID** (`user.id`) | 항상 권장 | 안정적 diff, 상태 보존 |
| **index** | 정적 리스트, 재정렬 없는 경우 | 순서 변경 시 버그 가능 |
| **랜덤 값** | 사용 금지 | 매 렌더마다 key가 바뀌어 전부 재마운트 |

## 면접 핵심 Q&A

### 컴포넌트

**Q. 컴포넌트 이름은 왜 대문자로 시작해야 하나요?**
> Babel이 JSX를 변환할 때 **소문자는 문자열**(브라우저 태그), **대문자는 식별자**(함수 참조)로 처리하기 때문이다. `<hello />`는 `createElement("hello", ...)`로 변환되어 의도와 다른 결과가 나온다.

**Q. Fragment가 뭔가요? 왜 쓰나요?**
> DOM에 추가 노드를 만들지 않고 여러 자식을 묶는 래퍼다. `<div>`로 감싸면 불필요한 DOM 노드가 생기므로 `<>...</>`를 사용한다. 리스트에서 key가 필요하면 `<React.Fragment key={...}>` 긴 문법을 쓴다.

**Q. 컴포넌트가 하나의 루트를 반환해야 하는 이유?**
> React의 Reconciliation은 **서브트리 단위**로 비교·교체한다. 루트가 여러 개면 비교 단위가 흐려져 성능과 정확성에 불리하다. Fragment를 쓰면 DOM 노드 없이 하나의 루트를 만족시킬 수 있다.

### Props & 데이터 흐름

**Q. Props와 State의 차이는?**
> | | Props | State |
> |--|-------|-------|
> | **방향** | 부모 → 자식 전달 | 컴포넌트 내부 소유 |
> | **변경** | 읽기 전용 (자식이 수정 불가) | `setState`로 변경 가능 |
> | **목적** | 외부 설정값 | 시간에 따라 바뀌는 내부 데이터 |

**Q. Props는 왜 읽기 전용인가요?**
> 단방향 데이터 흐름을 보장하기 위해서다. props를 자식이 마음대로 바꾸면 데이터 변경의 출처를 추적할 수 없고, 부모-자식 간 상태 불일치가 생긴다. 변경이 필요하면 **부모가 새 값을 내려주거나**, 부모가 내려준 **콜백을 호출**해야 한다.

**Q. Rest/Spread 패턴은 언제 쓰나요?**
> 컴포넌트가 일부 props만 사용하고 나머지를 하위 DOM 요소에 **위임**할 때 쓴다. 폼 입력 래퍼(`Input`, `Select` 등)에서 `type`, `placeholder`, `required` 같은 속성을 일일이 전달하지 않고 `{...inputProps}`로 일괄 전달할 수 있다.

**Q. children이 뭔가요?**
> 컴포넌트의 **열고 닫는 태그 사이**에 들어가는 모든 콘텐츠가 `props.children`으로 전달되는 특수 props다. 문자열, JSX 요소, 배열 등 어떤 타입이든 가능하다. 레이아웃/컨테이너 컴포넌트를 만들 때 핵심적으로 사용된다.
>
> ```jsx
> <Panel>
>   <p>이 내용이 props.children</p>
> </Panel>
> ```

**Q. 단방향 데이터 흐름이란?**
> 데이터가 **부모 → 자식** 한 방향으로만 props를 통해 흐르는 구조다. 자식이 부모 상태를 바꾸려면 부모가 내려준 **콜백 함수**를 호출해야 한다. 이 구조 덕분에 데이터 변경의 출처를 추적하기 쉽고 디버깅이 수월하다.

### 리스트 렌더링 & key

**Q. key는 왜 필요한가요?**
> React가 리스트의 각 항목을 **고유하게 식별**하기 위해 필요하다. key가 없으면 항목 추가/삭제/재정렬 시 React가 어떤 항목이 바뀐 건지 모르기 때문에 불필요한 리렌더링이 발생하거나 **입력값이 엉뚱한 항목에 남는** 버그가 생긴다.

**Q. index를 key로 쓰면 안 되는 이유?**
> 정적 리스트(순서 변경/삽입/삭제 없음)에서는 괜찮지만, **순서가 바뀌는 리스트**에서 index를 쓰면 key가 항목이 아닌 위치에 묶이게 되어 React가 잘못된 항목을 재사용한다. **고유 ID**를 key로 쓰는 게 항상 안전하다.

**Q. JSX에서 map 쓸 때 흔한 실수는?**
> 화살표 함수에서 **중괄호 블록 `{}`을 쓰면서 `return`을 빠뜨리는 것**이다. `() => (<li>...</li>)`는 즉시 반환이지만, `() => { <li>...</li> }`는 `return`이 없어 `undefined`가 되어 아무것도 렌더되지 않는다.

## 용어 글로서리

| 용어 | 정의 |
|------|------|
| **Component** | JSX를 반환하는 함수. 대문자 시작, 하나의 루트 반환이 규칙 |
| **Fragment** | DOM에 추가 노드 없이 자식을 묶는 래퍼. `<>...</>` 또는 `<React.Fragment>` |
| **Props** | 컴포넌트에 전달되는 입력값 객체. 읽기 전용 |
| **children** | 태그 안쪽 콘텐츠가 통째로 담기는 특수 props |
| **Rest/Spread** | `...rest`로 나머지를 모으고, `{...obj}`로 펼치는 패턴 |
| **key** | 리스트 항목의 고유 식별자. diff 정확도와 상태 보존에 직결 |
| **Composition (합성)** | 작은 컴포넌트를 조합해 큰 UI를 만드는 패턴 |
| **단방향 데이터 흐름** | 데이터가 부모 → 자식 한 방향으로만 흐르는 구조 |
| **Lifting State Up** | 자식의 변경 의도를 부모로 올려 부모가 상태를 소유하게 하는 패턴 |
| **Single Source of Truth** | 데이터의 진실이 한 곳에만 존재하는 원칙 |
| **StrictMode** | 개발 진단용 래퍼. 렌더 함수를 이중 호출해 부수효과를 점검 |

## 자가 점검 체크리스트

- [ ] `<Hello />`가 함수 호출 → React Element → DOM 반영 흐름임을 설명할 수 있다
- [ ] 대문자/소문자 인식 차이와 하나의 루트 규칙을 말할 수 있다
- [ ] Fragment의 짧은/긴 문법 차이를 안다 (key 가능 여부)
- [ ] `className` / `htmlFor` / `onClick` 등 JSX 속성 규칙을 구분한다
- [ ] props 읽기 전용 원칙과 구조 분해 기본값을 활용한다
- [ ] Rest/Spread 패턴으로 속성 위임을 할 수 있다
- [ ] children에 문자열·요소·배열이 모두 들어갈 수 있음을 안다
- [ ] 리스트는 반드시 key를 제공한다 (가능하면 id)
- [ ] index를 key로 쓰면 안 되는 이유를 설명할 수 있다
- [ ] JSX 안에서 map/filter 체이닝과 즉시 반환 패턴을 올바르게 쓴다
- [ ] 부모가 데이터의 Single Source of Truth를 가진다는 것을 이해한다
