# 🟩 Mission Y — 재사용 가능한 데이터 패칭 훅 만들기: useFetchProducts

## 📝 설명

상품 목록, 상세, 추천, 검색 결과 등 여러 화면에서 공통적으로 "목록 불러오기" 로직이 반복됩니다. 컴포넌트마다 `axios.get`, 로딩/에러 상태, 요청 취소(`AbortController`), 에러 정규화를 매번 구현하면 중복과 레이스 컨디션(이전 요청 응답이 뒤늦게 도착해 화면을 덮는 문제)이 잦아집니다.

이번 미션의 목표는:

- 데이터를 가져오는 로직을 커스텀 훅 `useFetchProducts`로 캡슐화하여 한 번 구현 → 여러 곳 재사용.
- 훅 내부에서 `AbortController`로 의존성 변경·언마운트 시 이전 요청을 즉시 취소.
- 에러 정규화(메시지 표준화)로 사용자에게 일관된 오류 안내 제공.
- 호출부는 `{ data, loading, error, refetch }`만 받아 UI 렌더링에만 집중.

### 실행 흐름 (사용자 시나리오)

1. 사용자가 "전자제품" 카테고리를 선택하고 검색어를 입력한다.
2. `params`가 바뀌면 `useFetchProducts`가 이전 요청을 취소하고 새 요청을 보낸다.
3. 네트워크 상황에 따라 성공/실패/타임아웃이 발생할 수 있으며, 훅은 이를 로딩/에러/데이터로 구조화해 반환한다.
4. 컴포넌트는 상태에 따라 "불러오는 중...", "에러 메시지", "상품 목록"을 예측 가능하게 렌더링한다.
5. 사용자가 "다시 시도"를 누르면, 현재 파라미터로 `refetch()`가 동일 요청을 재수행한다.

## ✅ 요구사항 (체크리스트)

### 핵심 기능

- `useFetchProducts(params)` 훅 구현 (`params`: `{ keyword?, category?, sort? }`).
- 훅은 `{ data, loading, error, refetch }` 반환.
- 마운트 및 `params` 변경 시 요청하고, cleanup에서 `abort()`로 이전 요청 취소.
- axios 인스턴스(`apiClient`) 사용(공통 `baseURL`/`timeout`/`headers`).
- 에러 정규화 함수로 메시지 표준화(취소/타임아웃/네트워크/404/5xx/기타).
- `refetch()`는 현재 `params`로 재요청.

### 품질/안전

- `useEffect`를 `async`로 만들지 말고, 내부에서 `async` 함수/IIFE 사용.
- 취소(`ERR_CANCELED`) 시 `setState` 호출 회피.
- 의존성 배열에는 effect 내부에서 실제로 참조하는 값만 포함.

### 사용 예

- `ProductListHooked`에서 훅 사용, 필터/검색/정렬 변경 시 자동 갱신.
- 에러 시 정규화 메시지와 다시 시도(`refetch`) 버튼 제공.

## 🧱 Base Code

### `src/services/apiClient.js`

```jsx
import axios from "axios";

export const apiClient = axios.create({
  baseURL: "https://fakestoreapi.com",
  timeout: 8000,
  headers: { "Content-Type": "application/json" },
});
```

### `src/services/normalizeAxiosError.js`

```jsx
export function normalizeAxiosError(err) {
  if (err.code === "ERR_CANCELED") return "요청이 취소되었습니다.";
  if (err.code === "ECONNABORTED") return "요청 시간이 초과되었습니다.";
  if (!err.response)               return "인터넷 연결을 확인하세요.";
  const { status } = err.response;
  if (status === 404) return "데이터를 찾을 수 없습니다.";
  if (status >= 500)  return "서버에 문제가 발생했습니다. 잠시 후 다시 시도해 주세요.";
  return "알 수 없는 오류가 발생했습니다.";
}
```

> 전 상태 문제: 각 컴포넌트에서 axios/취소/에러/로딩이 중복 구현, 빠른 파라미터 변경 시 레이스 컨디션, 에러 메시지 비일관.

## 📌 미션의 의의 (DevTools 관찰 포인트 · 안정성 포인트)

- **캡슐화**: 취소/에러/로딩/재시도를 훅 내부로 숨겨 호출부 단순화.
- **레이스 방지**: `params` 변경 시 cleanup → `abort` → 새 요청 순서를 보장.
- **UX 일관**: 에러 정규화로 메시지 표준화 → 사용자 신뢰성 향상.
- **확장성**: 같은 패턴으로 `useFetchItem`, `useFetchUsers` 등 손쉽게 확장.

### DevTools 체크

- Network: 필터 변경 시 이전 요청이 `(canceled)`로 종료되는지 확인.
- Console: 언마운트 후 `setState` 경고 없음.
- Performance: 잦은 파라미터 변경에도 불필요한 렌더/중복 요청 최소화.
