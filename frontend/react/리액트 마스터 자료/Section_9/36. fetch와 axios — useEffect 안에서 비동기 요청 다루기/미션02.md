# 🟩 HTTP 요청과 useEffect: fetch vs axios 제대로 쓰기

## 📝 설명

사내 쇼핑몰 대시보드 팀에 합류했습니다. 홈 화면에서 최근 게시물 5개를 외부 API(JSONPlaceholder)에서 받아와 목록으로 보여줘야 합니다. 초기 렌더 직후 한 번만 데이터를 불러오고, 사용자가 다른 화면으로 이동하거나 검색 필터를 바꿀 때는 진행 중인 요청을 안전하게 취소해야 합니다. 또한 로딩/에러/데이터 상태를 명확히 노출해 퍼블리셔·QA가 상태 변화를 시각적으로 확인할 수 있어야 합니다.

이 업무는 단순히 "데이터만 띄우는" 수준을 넘어, 다음의 실무 핵심 역량을 요구합니다:

- **렌더링 단계와 네트워크 단계의 분리**: `useEffect` 시점에 요청을 보내 렌더 블로킹을 피한다.
- **안전한 비동기 처리**: effect를 `async`로 만들지 않고, 내부 `async` + cleanup으로 자원 누수/레이스 컨디션을 방지한다.
- **요청 취소(`AbortController`)**: 언마운트·의존성 변경 시 오래된 응답이 최신 UI를 덮어쓰는 문제를 근본 차단한다.
- **상태 3종 세트(loading/error/data)**: UI/UX 일관성 및 디버깅 가시성 확보.
- **fetch vs axios 비교 운용**: 팀/프로젝트 선호에 맞추어 동일 요구사항을 두 방식 모두로 구현할 수 있어야 한다.

### 대상 API

- 목록: `https://jsonplaceholder.typicode.com/posts?_limit=5`
- (옵션) 검색: `https://jsonplaceholder.typicode.com/posts?_limit=5&q=<키워드>`

## ✅ 요구사항

### 기능

- 초기 렌더 뒤 한 번만 게시글 5개를 가져와 목록을 표시한다.
- `loading` / `error` / `data` 3상태를 UI로 명확히 표시한다.
- 네트워크 실패 시 에러 메시지를 보여준다(콘솔에도 원인 출력).
- 언마운트 또는 의존성 변경 시 진행 중인 요청을 취소한다.

### 구현 원칙

- `useEffect(async () => {})` 금지. → effect 내부에 별도 `async` 함수 또는 IIFE 사용.
- cleanup 함수에서 `AbortController.abort()` 호출.
- effect 안에서 참조하는 값은 의존성 배열에 포함(누락 금지).
- `fetch` & `axios` 두 버전 모두 구현(동작 동일).

### 테스트/검증

- DevTools Network에서 요청 1회만 발생(초기 마운트).
- 페이지 이탈/컴포넌트 언마운트 시 `(canceled)`로 요청 중단 확인.
- 에러 유도(잘못된 URL) 시 에러 UI가 노출되는지 확인.
- (선택) 검색어 변경 시 매 변경마다 이전 요청이 취소되는지 로그로 확인.

## 📌 미션의 의의 (DevTools 관찰 포인트·에지 케이스)

- **왜 effect 내부 async?**
  `useEffect`는 동기로 cleanup(함수)을 기대합니다. effect 자체에 `async`를 붙이면 Promise를 반환해 React가 cleanup과 혼동 → 경고·예기치 않은 동작.
- **요청 취소 = 레이스 컨디션 예방**
  빠르게 화면 전환/의존성 변경 시 이전 응답이 뒤늦게 도착하여 최신 UI를 덮는 버그를 없앱니다. Network 탭에서 `(canceled)` 또는 `Status: (blocked:other)` 등으로 확인.
- **Strict Mode(개발)**
  마운트/언마운트가 추가로 일어나는 것처럼 보이는 개발 특수 규칙 때문에 로그가 2번 보일 수 있음. 하지만 cleanup이 정확하면 중복 등록/누수 없이 안전.
- **확장 여지**
  디바운스(검색), 페이지네이션/무한 스크롤, 전역 에러 토스트, 재시도(backoff), SWR/React Query로의 리팩터링 등으로 자연스럽게 이어짐.
