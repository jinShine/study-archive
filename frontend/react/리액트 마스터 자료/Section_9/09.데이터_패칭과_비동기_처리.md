# 데이터 패칭과 비동기 처리

---

## HTTP 기초 -- 요청/응답 구조

### 클라이언트-서버 통신 흐름

```
[브라우저(클라이언트)]  ──  HTTP 요청  ──▶  [서버]
         ◀── HTTP 응답 ──
```

- **요청(Request):** 메서드(GET/POST/PUT/DELETE) + URL + 헤더 + 바디
- **응답(Response):** 상태 코드 + 헤더 + 바디(JSON 등)

### HTTP 메서드

| 메서드 | 용도 | 바디 |
|--------|------|------|
| **GET** | 데이터 조회 | 없음 |
| **POST** | 데이터 생성 | 있음 |
| **PUT** | 전체 수정 | 있음 |
| **PATCH** | 부분 수정 | 있음 |
| **DELETE** | 삭제 | 선택적 |

### 주요 상태 코드

| 코드 | 의미 | 설명 |
|------|------|------|
| **200** | OK | 요청 성공 |
| **201** | Created | 리소스 생성 성공 |
| **204** | No Content | 성공했으나 응답 바디 없음 |
| **400** | Bad Request | 잘못된 요청 (클라이언트 오류) |
| **401** | Unauthorized | 인증 필요 |
| **403** | Forbidden | 권한 없음 |
| **404** | Not Found | 리소스 없음 |
| **500** | Internal Server Error | 서버 내부 오류 |
| **503** | Service Unavailable | 서버 일시 불가 |

> React에서 데이터를 가져오려면 브라우저가 서버에 HTTP 요청을 보내고, 응답으로 받은 JSON을 파싱하여 State에 저장한 뒤 렌더링한다.

## fetch vs axios 비교

### 핵심 차이 한눈에 보기

| | **fetch** | **axios** |
|--|-----------|-----------|
| **설치** | 내장 API (별도 설치 없음) | `npm install axios` |
| **JSON 파싱** | `res.json()` 수동 호출 필요 | `res.data`에 자동 파싱 |
| **에러 처리** | HTTP 에러(404, 500)도 resolve됨 | HTTP 에러 시 reject (catch로 이동) |
| **요청 취소** | `AbortController` 직접 사용 | `AbortController` signal 옵션 |
| **인터셉터** | 없음 (직접 구현) | 요청/응답 인터셉터 내장 |
| **인스턴스** | 없음 | `axios.create()`로 baseURL 등 공통 설정 |
| **타임아웃** | 직접 구현 | `timeout` 옵션 내장 |

### 코드 비교

```jsx
// fetch -- res.json() 수동 파싱 필수
fetch("https://fakestoreapi.com/products")
  .then((res) => res.json())       // 수동 파싱
  .then((json) => setData(json))
  .catch((err) => console.error(err));

// axios -- res.data에 파싱된 JSON이 바로 담김
axios.get("https://fakestoreapi.com/products")
  .then((res) => setData(res.data)) // 자동 파싱
  .catch((err) => console.error(err));
```

## 데이터 패칭 패턴 -- loading/error/data 3종 상태 관리

### useEffect 안에서 async를 다루는 규칙

```jsx
// !! useEffect 자체를 async로 선언하면 안 된다
useEffect(async () => { ... }, []);  // cleanup이 Promise가 되어 버림

// 올바른 패턴: 내부에서 async 함수를 정의하고 호출
useEffect(() => {
  async function fetchData() { ... }
  fetchData();
}, []);
```

**이유:** useEffect의 콜백은 cleanup 함수(또는 undefined)를 반환해야 한다. async 함수는 항상 Promise를 반환하므로 cleanup이 망가진다.

### 통합 예제 -- 3종 상태 관리

```jsx
import { useEffect, useState } from "react";
import axios from "axios";

function ProductList() {
  const [products, setProducts] = useState([]);   // 데이터
  const [loading, setLoading] = useState(true);    // 로딩 여부
  const [error, setError] = useState(null);        // 에러 메시지

  useEffect(() => {
    async function fetchProducts() {
      try {
        const res = await axios.get("https://fakestoreapi.com/products");
        setProducts(res.data);
      } catch (err) {
        setError("상품을 불러오는 데 실패했습니다.");
      } finally {
        setLoading(false);
      }
    }
    fetchProducts();
  }, []);  // 마운트 시 1회 실행

  // 상태별 렌더링 분기
  if (loading) return <p>로딩 중...</p>;
  if (error)   return <p style={{ color: "red" }}>{error}</p>;

  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>
          <img src={p.image} alt={p.title} width="50" />
          <span>{p.title} - ${p.price}</span>
        </li>
      ))}
    </ul>
  );
}
```

**3종 상태의 역할:**

| 상태 | 초기값 | 역할 |
|------|--------|------|
| `data` | `[]` 또는 `null` | 서버 응답 데이터 저장 |
| `loading` | `true` | 요청 진행 중 UI 표시 (스피너, 스켈레톤) |
| `error` | `null` | 실패 시 사용자에게 에러 메시지 노출 |

> 이 세 상태의 조합으로 "로딩 중 / 에러 발생 / 정상 데이터" 세 화면을 분기한다.

## 요청 취소와 에러 처리 -- AbortController + 에러 분류

### AbortController가 필요한 이유

```
1. 컴포넌트 마운트 → API 요청 시작
2. 사용자가 빠르게 다른 페이지로 이동 → 컴포넌트 언마운트
3. 응답이 뒤늦게 도착 → 이미 사라진 컴포넌트의 setState 호출
   → "Can't perform a React state update on an unmounted component" 경고
```

**해결:** useEffect의 cleanup에서 `controller.abort()`를 호출하여 진행 중인 요청을 취소한다.

### AbortController + axios 기본 패턴

```jsx
useEffect(() => {
  const controller = new AbortController(); // 1. 컨트롤러 생성

  async function fetchProducts() {
    try {
      const res = await axios.get("/products", {
        signal: controller.signal,           // 2. signal 연결
      });
      setProducts(res.data);
    } catch (err) {
      if (err?.code === "ERR_CANCELED") {
        console.log("요청이 취소되었습니다.");
        return;                              // 3. 취소는 에러 아님
      }
      setError("데이터를 불러오지 못했습니다.");
    } finally {
      setLoading(false);
    }
  }

  fetchProducts();

  return () => controller.abort();           // 4. cleanup에서 취소
}, []);
```

### 에러 세분화 패턴

```jsx
catch (err) {
  // 요청 취소 (AbortController.abort())
  if (err?.code === "ERR_CANCELED") {
    console.log("요청이 취소되었습니다.");
    return;
  }
  // 타임아웃 (axios timeout 초과)
  if (err?.code === "ECONNABORTED") {
    setError("요청 시간이 초과되었습니다. 다시 시도해 주세요.");
  }
  // 404 Not Found
  else if (err?.response?.status === 404) {
    setError("데이터를 찾을 수 없습니다.");
  }
  // 5xx 서버 오류
  else if (err?.response?.status >= 500) {
    setError("서버에 문제가 발생했습니다. 잠시 후 다시 시도해 주세요.");
  }
  // 기타
  else {
    setError("알 수 없는 오류가 발생했습니다.");
  }
}
```

### 에러 분류표

| 구분 | 코드/조건 | 사용자 메시지 | 처리 |
|------|-----------|-------------|------|
| **취소** | `ERR_CANCELED` | 노출 안 함 | `return` (정상 흐름) |
| **타임아웃** | `ECONNABORTED` | 시간 초과 안내 | 재시도 유도 |
| **404** | `err.response.status === 404` | 리소스 없음 | URL 확인 유도 |
| **5xx** | `err.response.status >= 500` | 서버 오류 안내 | 잠시 후 재시도 |
| **기타** | 위에 해당 없음 | 일반 오류 안내 | 로그 기록 |

## axios 인스턴스 -- baseURL, interceptors, 에러 정규화

### axios.create()로 인스턴스 만들기

```jsx
// src/services/apiClient.js
import axios from "axios";

export const apiClient = axios.create({
  baseURL: "https://fakestoreapi.com", // 모든 요청의 기본 주소
  timeout: 8000,                       // 8초 초과 시 ECONNABORTED
  headers: {
    "Content-Type": "application/json",
  },
});
```

**장점:**
- 모든 API 호출에서 baseURL 반복 불필요 (`apiClient.get("/products")`)
- timeout, headers 등 공통 설정을 한 곳에서 관리
- 인터셉터로 인증 토큰 자동 첨부, 에러 로깅 등 확장 가능

### 에러 정규화 함수 분리

```jsx
// src/utils/normalizeAxiosError.js
export function normalizeAxiosError(err) {
  if (err?.code === "ERR_CANCELED") {
    return "요청이 취소되었습니다.";
  }
  if (err?.code === "ECONNABORTED") {
    return "요청 시간이 초과되었습니다. 다시 시도해 주세요.";
  }
  if (err?.response?.status === 404) {
    return "데이터를 찾을 수 없습니다.";
  }
  if (err?.response?.status >= 500) {
    return "서버에 문제가 발생했습니다. 잠시 후 다시 시도해 주세요.";
  }
  return "알 수 없는 오류가 발생했습니다.";
}
```

**에러 정규화를 분리하는 이유:**
- 여러 컴포넌트에서 동일한 에러 변환 로직을 재사용
- 에러 메시지 변경 시 한 곳만 수정
- 컴포넌트는 UI 렌더링에만 집중

### 인터셉터 활용 예시

```jsx
// 요청 인터셉터 -- 매 요청에 토큰 자동 첨부
apiClient.interceptors.request.use((config) => {
  const token = localStorage.getItem("token");
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 응답 인터셉터 -- 401 시 로그인 페이지로 리다이렉트
apiClient.interceptors.response.use(
  (res) => res,
  (err) => {
    if (err?.response?.status === 401) {
      window.location.href = "/login";
    }
    return Promise.reject(err);
  }
);
```

## 커스텀 Hook -- useFetch 구현과 활용

### useFetch 구현

```jsx
import { useEffect, useState } from "react";
import { apiClient } from "../services/apiClient";

function useFetch(url) {
  const [data, setData] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    const controller = new AbortController();

    // URL이 바뀔 때마다 상태 초기화
    setLoading(true);
    setError(null);

    apiClient
      .get(url, { signal: controller.signal })
      .then((res) => {
        setData(res.data);
        setError(null);
      })
      .catch((err) => {
        const isCanceled =
          err?.code === "ERR_CANCELED" || err?.name === "CanceledError";
        if (!isCanceled) {
          if (err?.code === "ECONNABORTED") {
            setError("요청 시간이 초과되었습니다.");
          } else if (err?.response?.status === 404) {
            setError("데이터를 찾을 수 없습니다.");
          } else if (err?.response?.status >= 500) {
            setError("서버 오류가 발생했습니다.");
          } else {
            setError("알 수 없는 오류가 발생했습니다.");
          }
        }
      })
      .finally(() => {
        setLoading(false);
      });

    return () => controller.abort(); // cleanup: 요청 취소
  }, [url]); // URL이 바뀌면 새로 요청

  return { data, error, loading };
}
```

### 사용하는 컴포넌트

```jsx
function ProductList() {
  const { data: products, error, loading } = useFetch("/products");

  if (loading) return <p>로딩 중...</p>;
  if (error)   return <p style={{ color: "red" }}>에러: {error}</p>;

  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>
          <img src={p.image} alt={p.title} width="50" />
          <span>{p.title} - ${p.price}</span>
        </li>
      ))}
    </ul>
  );
}
```

### useFetch가 해결하는 문제

| 문제 | useFetch 이전 | useFetch 이후 |
|------|--------------|--------------|
| 상태 선언 반복 | 매 컴포넌트마다 data/error/loading 3개 선언 | Hook 내부에 캡슐화 |
| AbortController 반복 | 매번 생성/cleanup 작성 | Hook이 자동 처리 |
| 에러 분류 반복 | catch 블록 복붙 | Hook 내부에서 통합 |
| 관심사 분리 | 컴포넌트에 패칭 로직 혼재 | 컴포넌트는 UI만 담당 |

### 전체 데이터 흐름 요약

```
useFetch("/products")
  │
  ├─ useEffect 실행
  │   ├─ AbortController 생성
  │   ├─ apiClient.get(url, { signal }) 호출
  │   │   ├─ 성공 → setData(res.data)
  │   │   ├─ 취소 → 무시 (정상 흐름)
  │   │   └─ 실패 → setError(정규화된 메시지)
  │   └─ finally → setLoading(false)
  │
  ├─ cleanup: controller.abort()
  │
  └─ return { data, error, loading }
       │
       └─ 컴포넌트가 3종 상태로 UI 분기
```

## 면접 핵심 Q&A

**Q. fetch와 axios의 차이?**
> fetch는 브라우저 내장 API로 별도 설치가 필요 없지만, 응답을 `res.json()`으로 수동 파싱해야 하고 HTTP 에러(404, 500)도 resolve된다. axios는 설치가 필요하지만 `res.data`에 자동 파싱되고, HTTP 에러 시 reject되어 catch로 잡힌다. 또한 axios는 인스턴스, 인터셉터, 타임아웃 등 실무에 필요한 기능을 내장하고 있다.

**Q. useEffect에서 데이터를 불러오는 기본 패턴?**
> useEffect 자체를 async로 선언하면 안 된다 (cleanup이 Promise가 됨). 내부에서 async 함수를 정의하고 호출하는 패턴을 쓴다:
> ```jsx
> useEffect(() => {
>   async function fetchData() {
>     const res = await axios.get(url);
>     setData(res.data);
>   }
>   fetchData();
> }, []);
> ```

**Q. loading/error 상태는 왜 필요한가?**
> 네트워크 요청은 비동기이므로 데이터가 즉시 도착하지 않는다. loading 상태가 없으면 빈 화면이 보이고, error 상태가 없으면 실패해도 사용자가 원인을 알 수 없다. 이 두 상태로 "로딩 중 / 에러 / 정상" 세 화면을 분기하여 UX를 보장한다.

**Q. AbortController는 왜 필요한가?**
> 컴포넌트가 언마운트되거나 의존성이 변경되면 이전 요청이 무의미해진다. 이때 응답이 뒤늦게 도착하면 사라진 컴포넌트의 setState가 호출되어 메모리 누수나 경고가 발생한다. cleanup에서 `controller.abort()`를 호출하면 진행 중인 요청을 취소하고, catch에서 `ERR_CANCELED`를 확인하여 정상 종료로 처리한다.

**Q. 커스텀 Hook의 장점?**
> 여러 컴포넌트에서 반복되는 데이터 패칭 로직(상태 선언, AbortController, 에러 처리)을 하나의 Hook으로 추상화한다. 컴포넌트는 `const { data, error, loading } = useFetch(url)` 한 줄로 데이터를 가져오고 UI 렌더링에만 집중할 수 있다. 로직 변경 시 Hook만 수정하면 모든 사용처에 반영된다.

**Q. axios 인스턴스를 만드는 이유?**
> `axios.create()`로 baseURL, timeout, headers 등 공통 설정을 한 곳에서 관리한다. 매 요청마다 전체 URL을 반복하지 않아도 되고, 인터셉터로 인증 토큰 첨부나 공통 에러 처리를 일괄 적용할 수 있다.

## 용어 글로서리

| 용어 | 정의 |
|------|------|
| **HTTP** | HyperText Transfer Protocol. 브라우저와 서버 간 데이터를 주고받는 통신 규약 |
| **fetch** | 브라우저 내장 HTTP 요청 API. Promise 기반으로 동작 |
| **axios** | Promise 기반 HTTP 클라이언트 라이브러리. 자동 JSON 파싱, 인터셉터 등 제공 |
| **AbortController** | 진행 중인 비동기 요청을 취소하기 위한 Web API |
| **signal** | AbortController가 생성하는 취소 신호 객체. 요청에 연결하면 abort() 시 취소됨 |
| **인터셉터** | 요청/응답이 처리되기 전에 가로채서 공통 로직을 실행하는 미들웨어 패턴 |
| **baseURL** | axios 인스턴스의 기본 주소. 모든 요청 URL 앞에 자동으로 붙음 |
| **에러 정규화** | 다양한 에러 타입을 사용자가 읽기 쉬운 일관된 메시지로 변환하는 패턴 |
| **커스텀 Hook** | `use`로 시작하는 사용자 정의 함수. 상태 로직을 재사용 가능하게 추상화 |
| **useFetch** | data/error/loading 3종 상태와 요청 취소를 캡슐화한 데이터 패칭 커스텀 Hook |
| **ERR_CANCELED** | axios에서 AbortController로 요청이 취소되었을 때 발생하는 에러 코드 |
| **ECONNABORTED** | axios에서 timeout이 초과되었을 때 발생하는 에러 코드 |
