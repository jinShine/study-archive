## 41강. 스타일링 파트 개관 — React에서 CSS의 역할 이해하기

### 도입

React로 기능은 다 완성했는데, 화면은 여전히 밋밋하고 투박합니다.
실제 서비스에서 사용자 경험의 절반은 화면의 시각적 완성도, 즉 **스타일링**에 달려 있습니다.
이번 강의는 CSS와 React의 관계를 이해하는 첫 시간입니다.

### 개념 설명

CSS는 **Cascading Style Sheets**, HTML로 만들어진 구조에 색, 간격, 폰트, 크기, 정렬, 배경 등 시각적 속성을 입히는 언어입니다.

CSS를 건축에 비유하면:
- **HTML/JSX** → 건물의 뼈대
- **CSS** → 페인트, 조명, 가구, 인테리어

React에서는 컴포넌트가 집의 각 방이라면, CSS는 그 방을 꾸미는 설계도입니다.

### 코드 예제

```css
/* App.css */
h1 {
  color: blue;
  text-align: center;
}
```

```jsx
// App.js
import "./App.css";

export default function App() {
  return <h1>안녕하세요, React!</h1>;
}
```

### 코드 해설

- **`import "./App.css"`**: 번들러가 CSS 파일을 의존성 그래프에 포함시키고, 빌드 시 CSS를 하나로 묶어 `<style>` 태그 등으로 자동 삽입한다.
- **CSS 규칙 `h1 { ... }`**: DOM 상의 모든 `h1` 요소에 일괄 적용된다(전역 선택자). 특정 컴포넌트만 적용하려면 클래스 기반(`.title { ... }`)이나 CSS Modules가 필요하다.
- **실제 결과**: 브라우저 렌더 트리 생성 시 `h1`에 계산된 스타일(Computed Style)로 파란색/중앙 정렬이 반영된다. DevTools Styles/Computed 패널에서 확인 가능.

### 실무 비유

프로젝트 초기에 CSS는 간단해 보이지만, 컴포넌트가 수십 개, 팀원이 여러 명이 되면 "같은 이름의 클래스가 덮어쓰는" 문제가 발생한다.
이는 **공용 창고에 라벨이 헷갈리는 상자를 마구 쌓아둔 것**과 같다.

### 실무 팁

- 전역 선택자(`h1`, `button`)는 의도한 범위를 정확히 생각하고 사용한다.
- 클래스 충돌 위험을 낮추려면 네이밍 컨벤션(BEM 등) 혹은 CSS Modules을 고려하라.
- DevTools에서 **우선순위(Specificity)** 와 소스 순서를 항상 점검하라.

### 핵심 요약

React에서 CSS는 단순한 꾸밈이 아니라 **유지보수 전략의 출발점**이다.
이번 강의는 이후 배울 다양한 스타일링 방식(Modules, CSS-in-JS, 라이브러리 등)의 지도(overview) 역할을 한다.

## 42강. Vanilla CSS 적용하기 — React에서 가장 기본적인 스타일링 방식

### 도입

Vanilla는 "기본, 순수"라는 뜻이다. 즉 Vanilla CSS는 별도의 도구 없이 `.css` 파일을 만들어 React 컴포넌트에 연결하는 방법이다.

### 개념 설명

React 프로젝트에서는 CSS 파일을 `<link>`로 연결하지 않고, **자바스크립트에서 `import` 구문으로 가져온다.** 이 차이는 번들러가 CSS를 모듈로 취급하기 때문이다.

- **장점**: 파일마다 필요한 CSS를 명시적으로 의존하게 되어 관리가 쉽다.
- **단점**: **전역 범위(Global)** 로 적용되기 때문에 규모가 커지면 충돌이 발생한다.

### 코드 예제

```css
/* App.css */
h1 {
  color: blue;
  text-align: center;
}

button {
  background-color: green;
  color: white;
  padding: 10px;
  border-radius: 5px;
}
```

```jsx
// App.js
import "./App.css";

export default function App() {
  return (
    <div>
      <h1>안녕하세요, React!</h1>
      <button>클릭</button>
    </div>
  );
}
```

### 코드 해설

- **`import "./App.css"`**: 이 파일을 import하는 시점부터 그 안의 CSS 규칙은 **전역적으로 적용**된다. 다른 컴포넌트의 `h1`/`button`에도 영향이 갈 수 있다.
- **전역 스타일 문제**: 같은 프로젝트에서 `button { color: red; }`가 있다면 소스 순서에 따라 마지막 로드된 규칙이 버튼 글자색을 바꾼다. 이를 "덮어쓰기"라고 하며 디버깅 난이도가 올라간다.
- **번들링 파이프라인**: `App.css` → JS로 import → 번들러가 CSS를 수집/최적화 → 최종 번들에 인라인 `<style>` 또는 별도 CSS 파일로 주입.

### 실무 비유

Vanilla CSS는 **회사 전체에서 하나의 공용 인테리어 규칙을 공유하는 것**과 같다.
빠르게 일관성을 줄 수 있으나, 특정 방만 다르게 꾸미고 싶을 때 어려움이 생긴다.

### 실무 팁

- 작은 개인/실험 프로젝트에서는 간결하고 빠르다.
- 컴포넌트 스코프가 필요한 중대형 프로젝트는 CSS Modules나 CSS-in-JS를 선택하라.
- 전역 규칙은 Reset/Normalize 등 "의도된 공통 스타일"에만 제한하라.

### 핵심 요약

- React에서 CSS는 `import`로 연결한다.
- 간단하지만 **전역 적용**이라는 단점이 있다.
- 규모가 커질수록 관리가 어려워진다 → 모듈화/스코프 필요.

## 43강. CSS Modules — 컴포넌트 단위로 안전한 스타일 관리하기

### 도입

Vanilla CSS의 가장 큰 문제는 **전역 충돌**이다.
CSS Modules는 이를 해결하기 위해 파일 단위로 스타일을 자동으로 고유 이름으로 변환한다.

### 개념 설명

CSS Modules는 파일 이름에 `.module.css` 확장자를 붙인다.
빌드 시 내부적으로 `.button___3Gh7x`처럼 **랜덤 해시를 결합해 고유화**한다.
동일한 `.button` 이름이라도 파일마다 서로 다른 실제 클래스명이 되어 충돌하지 않는다.

### 코드 예제

```css
/* Button.module.css */
.button {
  background-color: green;
  color: white;
  padding: 10px;
  border-radius: 5px;
}
```

```jsx
// Button.js
import styles from "./Button.module.css";

export default function Button() {
  return <button className={styles.button}>클릭</button>;
}
```

### 코드 해설

- **`import styles from "./Button.module.css"`**: CSS를 객체 형태로 불러온다. 키는 원래 클래스명(`button`), 값은 빌드 결과의 고유 클래스명이다. 즉 `styles.button === "button___3Gh7x"`(예시).
- **`className={styles.button}`**: 실제 DOM에는 해시가 합쳐진 고유 클래스가 렌더되어, 다른 파일의 `.button`과 **절대 충돌하지 않는다.**
- **장점**: 전역 범위 오염 없이 컴포넌트 스코프가 자동 보장된다. BEM 같은 CSS 설계 원칙을 강제하지 않아도 충돌 방지가 가능하다.

### 실무 비유

Vanilla CSS가 **공용 회의실 규칙**이라면, CSS Modules는 **각 부서마다 독립된 사무실**을 갖는 것과 같다.
각 부서(컴포넌트)의 규칙은 다른 부서에 절대 간섭하지 않는다.

### 실무 팁

- 파일명 오타 주의: 반드시 `.module.css`.
- 클래스 접근은 `styles.className`으로만 — 문자열 하드코딩을 피하면 리팩터링 시 안정적이다.
- DevTools에서 실제 적용된 고유 클래스명을 확인해 디버깅할 수 있다.

### 핵심 요약

CSS Modules는 클래스 이름 충돌을 자동으로 해결하는 **가장 간단하고 안전한 컴포넌트 단위 스타일링 방법**이다.
`styles.className` 형태로 사용하며, 확장자는 반드시 `.module.css`여야 한다.

## 44강. CSS-in-JS — 자바스크립트 안에서 직접 스타일을 관리하기

### 도입

CSS Modules도 좋지만, 때로는 CSS 파일을 관리하는 것조차 번거롭다.
특히 props나 state 값에 따라 스타일이 바뀌어야 할 때, 자바스크립트 안에서 바로 제어하는 것이 더 직관적이다.
이때 사용하는 패턴이 **CSS-in-JS**이다.

### 개념 설명

CSS-in-JS는 말 그대로 CSS를 JS 내부에서 정의하는 방식이다. 별도의 `.css` 파일 없이, 컴포넌트 안에서 스타일 객체를 만들거나 styled-components/Emotion 같은 라이브러리를 쓴다.

- **인라인 스타일 객체**: 간단/직관적이나 `:hover`, `@media` 등 고급 기능이 제한적.
- **styled-components/Emotion**: CSS 문법을 그대로 쓰면서 props/state/테마를 쉽게 연동.

### 코드 예제 1 — 인라인 스타일 객체

```jsx
export default function Button({ primary }) {
  const buttonStyle = {
    backgroundColor: primary ? "blue" : "gray",
    color: "white",
    padding: "10px 15px",
    borderRadius: "5px",
    border: "none",
  };
  return <button style={buttonStyle}>클릭</button>;
}
```

**해설**

- 스타일이 JS 객체이며, React가 렌더링 때 인라인 CSS 문자열로 변환한다.
- 간단한 조건부 스타일(삼항 연산자)엔 매우 편리하다.
- 하지만 의사 클래스(`:hover`)나 미디어쿼리 같은 선언적 CSS 기능을 직접 쓰기 어렵다.
- 렌더링 때마다 새 객체가 만들어지면 자식 최적화에 영향을 줄 수 있으니, 필요시 `useMemo`로 메모이제이션을 고려한다.

### 코드 예제 2 — styled-components

```bash
npm install styled-components
```

```jsx
import styled from "styled-components";

const Title = styled.h1`
  color: purple;
  text-align: center;
`;

export default function App() {
  return <Title>안녕하세요, React!</Title>;
}
```

**해설**

- `styled.h1`\`...\`: 태그드 템플릿 문법으로 CSS를 그대로 작성한다.
- 렌더링 시 styled-components가 고유 클래스를 생성/주입하고 DOM에 연결한다.
- props 연동: 템플릿 안에서 `${({ prop }) => ...}`로 조건부 스타일을 자연스럽게 구현한다.
- 별도 CSS 파일 없이 로직/스타일/변형(variant)을 한 파일에서 응집력 있게 관리할 수 있다.

### 실무 비유

CSS-in-JS는 **부품(컴포넌트)과 그 부품의 스티커(스타일)를 한 상자에 담아두는 것**과 같다.
상자 안에서 라벨을 바꿔 붙이듯(=props, state) 즉시 스타일을 바꿀 수 있다.

### 실무 팁

- 동적 스타일 비중이 높을수록 CSS-in-JS의 장점이 커진다.
- 성능 이슈(런타임 클래스 생성)가 민감한 환경에선 빌드타임 추출(예: babel plugin)도 고려한다.
- 팀에서 디자이너 협업이 잦다면 디자인 토큰 + ThemeProvider 구성이 유용하다.

### 핵심 요약

CSS-in-JS는 로직과 스타일을 완전히 통합하는 방식으로, 특히 **props/state 기반의 조건부 스타일링**에 강력하다.
대표 라이브러리는 `styled-components`와 `@emotion/styled`이다.

## 45강. CSS-in-JS 실전 패턴 — 조건부 스타일링으로 살아 있는 UI 만들기

### 도입

styled-components를 이용해 실제 조건부 스타일링을 구현한다.
props, state, 사용자 상호작용, 반응형, 테마(Theme)까지 현실 서비스에서 가장 자주 쓰이는 패턴을 모두 익혀본다.

### 코드 예제 1 — props에 따라 변형되는 버튼

```jsx
import styled from "styled-components";

const Button = styled.button`
  background: ${({ variant }) =>
    variant === "primary" ? "royalblue" :
    variant === "danger"  ? "crimson"   :
                            "gray"};
  color: white;
  border: none;
  border-radius: 8px;
  padding: ${({ size }) => (size === "lg" ? "14px 18px" : "10px 14px")};
`;

export default function App() {
  return (
    <>
      <Button variant="primary" size="lg">저장</Button>
      <Button variant="danger">삭제</Button>
      <Button>기본</Button>
    </>
  );
}
```

**해설**

- `variant`/`size` props를 기반으로 CSS 값을 계산한다.
- styled-components는 렌더 시점에 현재 props를 받아 최종 CSS 문자열을 생성하고, 고유 클래스를 만들어 DOM에 주입한다.
- 같은 컴포넌트라도 props 조합에 따라 서로 다른 비주얼을 즉시 제공한다(디자인 시스템의 variant 개념).

### 코드 예제 2 — 상태(state)에 따라 토글되는 버튼

```jsx
import { useState } from "react";
import styled from "styled-components";

const Toggle = styled.button`
  background: ${({ active }) => (active ? "seagreen" : "gray")};
  color: white;
  border-radius: 6px;
`;

export default function ToggleDemo() {
  const [on, setOn] = useState(false);
  return (
    <Toggle active={on} onClick={() => setOn(v => !v)}>
      {on ? "ON" : "OFF"}
    </Toggle>
  );
}
```

**해설**

- UI 텍스트와 스타일이 동일한 상태값(`on`)에 의존한다.
- 클릭 시 상태가 토글 → `active` prop 변경 → 스타일 재계산 → 즉각적 시각 피드백.
- 로직/뷰가 분리되지 않고 한 파일에서 응집되므로 유지보수가 쉽다.

### 코드 예제 3 — 마우스 상호작용 (hover, active)

```jsx
const Action = styled.button`
  background: steelblue;
  color: white;
  &:hover { background: dodgerblue; }
  &:active { background: navy; transform: scale(0.97); }
`;
```

**해설**

- `&`는 "현재 컴포넌트의 고유 클래스"를 가리키는 자리표시자이다. 최종 CSS는 `.sc-xxx:hover`, `.sc-xxx:active` 형태로 변환된다.
- `:hover`는 포인터가 올려진 상태, `:active`는 클릭 중(마우스 다운~업) 상태에 적용된다.
- `transform: scale(0.97)`는 압력감을 주어 클릭감을 강화한다(미세한 모션이 UX 품질을 올린다).

### 코드 예제 4 — 반응형 미디어 쿼리

```jsx
const Card = styled.div`
  width: 100%;
  max-width: 720px;
  padding: 16px;
  background: #fff;

  @media (max-width: 480px) {
    padding: 12px;
    border-radius: 8px;
  }
`;
```

**해설**

- styled-components 내부에서도 표준 CSS `@media`를 그대로 쓴다.
- 화면 너비가 480px 이하로 줄면 CSS가 다시 계산되어 모바일 친화적 여백/모서리로 전환된다.
- 실제 서비스에서는 브레이크포인트를 테마 토큰(`theme.bp.sm`)으로 관리해 일관성을 확보한다.

### 코드 예제 5 — ThemeProvider로 다크모드 구현

```jsx
import styled, { ThemeProvider } from "styled-components";

const light = { bg: "#fff", fg: "#111" };
const dark  = { bg: "#111", fg: "#fff" };

const Page = styled.div`
  background: ${({ theme }) => theme.bg};
  color: ${({ theme }) => theme.fg};
`;

export default function ThemedApp({ darkMode }) {
  return (
    <ThemeProvider theme={darkMode ? dark : light}>
      <Page>테마에 따라 색상이 바뀝니다.</Page>
    </ThemeProvider>
  );
}
```

**해설**

- `ThemeProvider`는 React 컨텍스트를 사용해 하위 styled 컴포넌트에 `theme` 객체를 자동 주입한다.
- `darkMode` 값 변경 → ThemeProvider가 새로운 theme 전달 → 하위 CSS가 즉시 재계산되어 컬러 토큰이 전환된다.
- 색상뿐 아니라 간격, 폰트, 그림자, 브레이크포인트 등도 토큰화해 일관된 디자인 시스템을 구축할 수 있다.

### 실무 팁

- 버튼/배지/알림 등 공통 컴포넌트에는 `variant`/`size` prop을 표준화하여 조합 가능한 디자인 시스템을 만들라.
- `:hover`/`:active`만 잘 써도 상호작용 품질이 크게 좋아진다(포커스 상태에 대한 접근성도 함께 고려).
- `ThemeProvider`로 라이트/다크/브랜드 테마를 일관 관리하라. 디자인 토큰은 별도 파일로 분리하면 유지보수가 수월하다.
- 성능 민감 구간에서는 스타일 연산을 최소화하고, 조건부 로직이 복잡하면 class 토글(data-attr) 전략도 검토하라.

### 핵심 요약

- styled-components는 props/state/theme에 반응해 **데이터 기반 스타일**을 생성한다.
- `&`는 자기 자신을 의미하며, `:hover`, `:active` 등 의사 클래스를 안전하게 적용한다.
- 한 파일 안에서 반응형, 테마, 조건부 스타일을 모두 처리 → **현대 React 실무의 기본 도구**.

## 46강. Separation of Concerns — 스타일과 로직을 나누어 생각하기

### 도입

- **관심사의 분리(Separation of Concerns)** 는 "서로 다른 역할은 뒤섞지 말고, 나누어 관리하자"는 오래된 원칙이다.
- 전통 웹: HTML(구조) / CSS(디자인) / JS(동작)가 명확히 분리되었다.
- React 컴포넌트 시대: JSX(구조)+상태/이벤트(로직)+스타일이 한 파일에 공존할 수 있다.
- 핵심 질문: **얼마나 분리할 것인가? 얼마나 묶을 것인가?** (규모/협업/유지보수에 따라 달라짐)

### 개념 설명

- **작은 앱**: 한 파일에 구조/로직/스타일을 함께 둬도 관리 가능.
- **큰 앱/다인 협업**: 토큰/로직/표현을 단계적으로 분리하면 유지보수 유리.
- **균형 전략**(예):
  - 디자인 토큰(색/여백 등)은 `theme.js`에 모으기
  - 데이터/이벤트 로직은 커스텀 훅으로 분리
  - 컴포넌트에는 JSX 구조 + props 기반 조건부 스타일만 남기기

### 코드 예제

**theme.js — 디자인 토큰 집중 관리**

```js
export const theme = {
  colors: { primary: "#007BFF", danger: "#E74C3C", text: "#212529" },
  spacing: { sm: "8px", md: "12px", lg: "16px" },
  radius: { md: "8px" },
};
```

> theme 객체에 색상/여백/모서리 반경 등 재사용 가능한 값을 한곳에 모은다.
> 하드코딩을 피하고 토큰 참조로 통일하면, 테마 교체/브랜딩 대응이 쉬워진다.

**useSubmit.js — 로직 분리 (폼 제출)**

```js
import { useState } from "react";

export function useSubmit() {
  const [pending, setPending] = useState(false);
  const submit = async (fn) => {
    setPending(true);
    try { await fn(); } finally { setPending(false); }
  };
  return { pending, submit };
}
```

> 커스텀 훅으로 "대기 상태 관리+공통 비동기 래핑"을 캡슐화한다.
> 컴포넌트는 `pending`만 구독하면 되므로 표현과 로직이 느슨하게 결합된다.

**Button.jsx — 표현(뷰) 담당**

```jsx
import styled, { ThemeProvider } from "styled-components";
import { theme } from "./theme";
import { useSubmit } from "./useSubmit";

const Btn = styled.button`
  background: ${({ theme, variant }) =>
    variant === "danger" ? theme.colors.danger : theme.colors.primary};
  color: #fff;
  border: 0;
  border-radius: ${({ theme }) => theme.radius.md};
  padding: ${({ theme }) => theme.spacing.md} ${({ theme }) => theme.spacing.lg};
`;

export default function Button({ variant = "primary", onClick, children }) {
  const { pending, submit } = useSubmit();
  return (
    <ThemeProvider theme={theme}>
      <Btn variant={variant} disabled={pending} onClick={() => submit(onClick)}>
        {pending ? "처리 중..." : children}
      </Btn>
    </ThemeProvider>
  );
}
```

> - `styled.button` 내부에서 theme 토큰을 직접 사용해 색/간격/모서리 값을 일관 적용한다.
> - `variant`에 따라 primary/danger 분기를 한 줄로 표현해 표현 로직만 남긴다.
> - 클릭 시 `submit(onClick)`으로 감싸 비동기 대기 상태를 공통 처리하고, `disabled`로 UX를 보호한다.

### 실무 비유

- **전통 분리**: 주방/계산대/홀을 따로 둔 대형 식당 — 역할은 명확하지만 동선이 길 수 있음.
- **컴포넌트 결합**: 작은 가게에 주방/계산대/응대가 한 공간 — 빠르고 직관적이지만 복잡도 관리 필요.

### 실무 팁

- "완전 분리 vs 완전 결합"이 정답이 아님: 프로젝트 규모/팀 구성에 맞춰 **점진적 분리**를 택하라.
- 토큰/로직/뷰를 나누면 테마 교체/리팩토링/테스트가 쉬워진다.
- PR 리뷰 기준을 문서화("토큰은 theme, 로직은 훅, 컴포넌트는 뷰")하면 협업 품질이 올라간다.

### 핵심 요약

- SoC의 목적은 **유지보수성/협업성/일관성 향상**.
- **토큰/로직/뷰의 3단 분리**가 React 실무에서 가장 안전한 균형점이다.

## 47강. 스타일링 방식 비교 — Vanilla CSS, CSS Modules, CSS-in-JS

### 도입

- 동일 UI를 다양한 방식으로 스타일링할 수 있다.
- 선택 기준은 **규모, 충돌 방지, 동적 스타일링 난이도, 팀 선호, 성능**이다.

### 개념 설명

- **Vanilla CSS**
  - 장점: 단순, 러닝커브 낮음, 빌드 의존 적음
  - 단점: 전역 네임스페이스로 충돌/덮어쓰기 추적 어려움
- **CSS Modules**
  - 장점: `.module.css` → 클래스 자동 고유화(해시)로 충돌 차단, 전통 CSS 그대로
  - 단점: 파일 분리 유지 필요, 동적 스타일링은 클래스 토글 위주
- **CSS-in-JS** (styled-components/Emotion)
  - 장점: 한 파일 내 응집도, props/state 기반 동적 스타일 탁월, 클래스 자동 고유화
  - 단점: 코드/스타일 혼재로 가독성 이슈, 런타임 주입에 따른 성능 주의

### 코드 예제

```css
/* Vanilla CSS */
.button { background: green; color: #fff; }
```

> 전역 `.button` 클래스: 적용은 쉽지만 다른 파일의 `.button`과 충돌 위험이 크다.

```jsx
// CSS Modules
import styles from "./Button.module.css";
<button className={styles.button}>클릭</button>
```

> `styles.button`은 빌드 시 `button___해시`로 변환되어 파일 간 클래스 충돌을 원천 차단한다.

```jsx
// CSS-in-JS
import styled from "styled-components";
const Button = styled.button`
  background: ${({ primary }) => (primary ? "royalblue" : "gray")};
  color: #fff;
`;
```

> `primary` props 값으로 배경을 즉시 분기한다. 내부적으로 고유 클래스 생성 + DOM 주입이 이루어져 전역 충돌이 없다.

### 실무 비유

- **Vanilla**: 공유 옷장 — 꺼내기 쉽지만 섞이면 누구 옷인지 헷갈림
- **Modules**: 라벨 붙은 개인 사물함 — 충돌 없음
- **CSS-in-JS**: 옷+코디 메모를 한 옷걸이에 — 상황(날씨/약속)에 따라 즉시 코디 변경

### 실무 팁

- 프로토타입/소규모: Vanilla 빠름
- 중~대규모 협업: CSS Modules가 안전
- 테마/동적 스타일 풍부: CSS-in-JS가 생산적
- 혼합 사용도 현실적(예: 전역 리셋은 Vanilla, 컴포넌트는 Modules/CSS-in-JS)

### 핵심 요약

- **"하나의 정답"은 없다.** 팀/제품 맥락으로 선택하라.
- 충돌 리스크 vs 동적 표현력 vs 단순성의 트레이드오프를 이해하면 된다.

## 48강. Inline Styles — 가장 직접적인 스타일링 방법

### 도입

- Inline Styles는 JSX `style` 속성에 JS 객체를 넣는 가장 즉각적인 방식이다.
- 빠른 실험/국소 수정에 유리하지만, 범용 스타일링 주력으로는 한계가 있다.

### 개념 설명

- 속성명은 **camelCase** (`backgroundColor`, `fontSize`)
- 값은 보통 문자열(단위 포함)
- 의사 클래스(`:hover`)/미디어쿼리는 **직접 지원되지 않음**

### 코드 예제

```jsx
export default function Welcome() {
  const titleStyle = {
    color: "royalblue",
    fontSize: "24px",
    fontWeight: "bold",
    padding: "10px"
  };

  return <h1 style={titleStyle}>안녕하세요, React!</h1>;
}
```

**해설**

- `titleStyle`은 JS 객체이고 JSX의 `style`에 그대로 전달된다.
- 브라우저에는 `style="color: royalblue; font-size: 24px; ..."`로 인라인 스타일 문자열이 렌더된다.
- 빠른 실험에는 유리하지만, 반복 사용이 보이면 공용 클래스로 승격하는 것이 좋다.

### 실무 비유

- **급한 포스트잇 메모**: 지금 당장 표시가 필요할 때는 최고.
- 다만 문서 전체 규정(디자인 시스템)에는 맞지 않을 수 있음.

### 실무 팁

- 작은 예외 상황/프로토타입/실험용에 한정하라.
- 재사용 필요 시 `className` 또는 styled 컴포넌트로 승격하라.
- 객체 리터럴을 매 렌더마다 생성하면 리렌더를 유발할 수 있으니, 빈번히 전달하는 경우 `useMemo` 고려.

### 핵심 요약

- 인라인은 "직접 주입"이라 빠르지만 **확장성/표현력이 제한**된다.
- 작은 곳에만 쓰고, 반복되면 공용 스타일로 승격하라.

## 49강. Popular UI Libraries — Tailwind, Material UI, Ant Design 등

### 도입

- 실무에선 바퀴를 다시 만들지 않기 위해 **UI 라이브러리**를 적극 활용한다.
- 버튼/카드/모달/폼 등 표준 컴포넌트를 빠르게 조립하고, 접근성과 일관성을 확보한다.

### 개념 설명

- **Tailwind CSS**: Utility-First. 짧은 클래스 조합으로 빠르게 스타일 구성. 최신 트렌드.
- **MUI(Material UI)**: 구글 Material Design 구현. 접근성/반응형/테마 강력. 대규모 서비스 적합.
- **Ant Design**: 데이터 중심 엔터프라이즈 UI에 강점(표, 폼, 대시보드). 내부 시스템/관리도구에 적합.
- **shadcn/ui**: Tailwind 기반 고품질 컴포넌트 세트(유틸리티+컴포넌트 하이브리드).

### 코드 예제

```jsx
// Tailwind (클래스 조합)
<button className="bg-blue-500 text-white px-3 py-2 rounded">저장</button>
```

> `bg-blue-500 text-white px-3 py-2 rounded`는 배경/글자/패딩/모서리를 즉시 조합한다. CSS 파일 없이 조합 규칙만으로 일관된 스타일을 얻는다.

```jsx
// MUI (설치 후)
import Button from "@mui/material/Button";
<Button variant="contained" color="primary">저장</Button>
```

> 구글 Material Design을 구현한 컴포넌트. `variant="contained"`는 채움형 버튼, `color="primary"`는 테마의 기본 색을 사용한다.

```jsx
// Ant Design (설치 후)
import { Button } from "antd";
<Button type="primary">저장</Button>
```

> `type="primary"`는 브랜드 포인트 컬러 버튼을 의미한다. 폼/테이블 등 업무용 UI와 조합이 뛰어나다.

### 실무 비유

**완성된 레고 키트**: 가이드에 따라 빠르게 조립 → 출시 속도/일관성 향상.

### 실무 팁

- **Tailwind**: 규칙(컨벤션) 없으면 클래스가 난잡해질 수 있음 → 팀 규범 문서화.
- **MUI**: 구글 룩앤필이 브랜드와 맞는지 사전 검토. Theme 커스터마이즈로 이질감 완화.
- **Ant**: 데이터/폼/테이블이 많을수록 효율 증가. 글로벌 대상이면 시각 톤 조정 필요.
- "도구 채택 표준"을 레포 문서에 명문화(선택 이유/커스터마이즈 가이드).

### 핵심 요약

- UI 라이브러리는 **속도/일관성/접근성**을 동시에 끌어올린다.
- 프로젝트/브랜드 적합성을 먼저 평가하고 채택하라.

## 50강. 아이콘 활용하기 — React 프로젝트에서 가장 많이 쓰이는 시각 요소 추가하기

### 도입

- 아이콘은 의미를 즉시 전달하고 공간 효율을 높인다. (삭제=휴지통, 검색=돋보기)
- 실무에서는 보통 **React Icons** 같은 라이브러리를 사용한다.

### 개념 설명

- **react-icons**: Font Awesome/Material/Feather 등 여러 세트를 통합.
- 아이콘은 React 컴포넌트(SVG) 형태로 import → 크기/색을 props나 CSS로 제어.

### 코드 예제

```bash
npm install react-icons
```

```jsx
// 기본 사용
import { FaHeart } from "react-icons/fa";

export default function App() {
  return <button><FaHeart /> 좋아요</button>;
}
```

> `<FaHeart />`는 SVG 컴포넌트이므로 텍스트와 자연스럽게 인라인 배치된다. 크기/색은 상위 `font-size`/`color` 또는 스타일로 제어된다.

```jsx
// 좋아요 토글 (인라인 스타일로 동적 색상)
import { useState } from "react";
import { FaHeart } from "react-icons/fa";

export default function LikeButton() {
  const [liked, setLiked] = useState(false);
  const baseStyle = { background: "none", border: "none", cursor: "pointer", fontSize: "20px" };
  const dynamicStyle = { color: liked ? "crimson" : "gray" };

  return (
    <button onClick={() => setLiked(v => !v)} style={{ ...baseStyle, ...dynamicStyle }}>
      <FaHeart /> {liked ? "좋아요 취소" : "좋아요"}
    </button>
  );
}
```

**해설**

- 버튼 기본 모양은 `baseStyle`로 통일하고, 상태에 의한 변경점만 `dynamicStyle`로 분리했다.
- `liked` 토글 시 색이 gray ↔ crimson로 바뀌고, 텍스트도 함께 변경되어 **상태-표현 동기화**가 명확하다.
- 실제 서비스에서는 인라인 대신 클래스/스타일드 컴포넌트로 승격해 재사용성을 높인다.

### 실무 비유

- **텍스트는 설명서, 아이콘은 표지판.**
- 짧고 빠르게 의미를 전달해 사용자 인지 부하를 줄인다.

### 실무 팁

- **접근성**: 아이콘만 있는 버튼에는 `aria-label` 제공
  ```jsx
  <button aria-label="검색"><MdSearch /></button>
  ```
- **일관성**: 한 세트(Font Awesome 등)로 통일해 스타일 혼선을 줄이라.
- **퍼포먼스**: 트리셰이킹을 위해 개별 아이콘만 import.

### 핵심 요약

- 아이콘은 작지만 **UX 임팩트가 크다.**
- 라이브러리로 빠르게 적용하고, **상태/접근성/일관성**을 함께 챙기면 완성도가 올라간다.

## 면접 핵심 Q&A

### Q. CSS Modules와 일반 CSS의 차이?

> **A.** 일반 CSS는 `import`하면 전역으로 적용되어 같은 클래스명이 충돌할 수 있다.
> CSS Modules는 `*.module.css`로 지정하면 빌드 시 클래스명에 **고유 해시가 추가**되어
> (`button` → `button___3Gh7x`) 파일 단위로 스코프가 격리된다.

### Q. CSS-in-JS의 장단점?

> **A.**
> - **장점**: props/state 기반 동적 스타일링이 자연스럽고, 고유 클래스명으로 완전한 캡슐화가 가능하다. ThemeProvider로 전역 테마 관리가 편리하다.
> - **단점**: 런타임 CSS 생성으로 **성능 오버헤드**가 있고, JS 번들 크기가 증가한다.

### Q. 인라인 스타일의 한계?

> **A.** `:hover`, `:focus` pseudo-class 사용 불가, `@media` 미디어쿼리와 `@keyframes` 애니메이션도 적용할 수 없다. 스타일 재사용이 어렵고 유지보수성이 떨어진다.

### Q. Tailwind CSS의 특징?

> **A.** 유틸리티 클래스를 조합하여 스타일을 적용하는 방식이다.
> PurgeCSS로 미사용 클래스를 제거하여 번들을 최소화하고,
> `hover:`, `md:` 접두사로 상태/반응형 스타일을 지원한다.
> 단, 클래스가 길어지면 가독성이 떨어질 수 있다.

### Q. 프로젝트에서 스타일링 전략을 선택하는 기준?

> **A.**
> - **소규모 / 빠른 프로토타이핑** → Inline Style 또는 Vanilla CSS
> - **중규모 / 클래스 충돌 방지** → CSS Modules
> - **동적 스타일이 많은 SPA** → styled-components (CSS-in-JS)
> - **빠른 개발 + 커스텀 디자인** → Tailwind CSS
> - **표준 디자인 시스템 필요** → MUI 또는 Ant Design
>
> 핵심 판단 기준: **프로젝트 규모**, **동적 스타일 빈도**, **팀 기술 스택**, **성능 요구사항**

### Q. Separation of Concerns란?

> **A.** "서로 다른 역할은 나누어 관리하자"는 원칙이다. React 실무에서는 디자인 토큰(`theme.js`), 로직(커스텀 훅), 뷰(컴포넌트)를 3단 분리하는 것이 가장 안전한 균형점이다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **스코프 (Scope)** | 스타일이 영향을 미치는 범위. 전역(Global) vs 지역(Local) |
| **CSS Modules** | `.module.css` 확장자로 빌드 시 고유 클래스명을 생성하는 방식 |
| **CSS-in-JS** | JS 코드 안에서 CSS를 작성하는 패턴 (styled-components, Emotion) |
| **ThemeProvider** | styled-components의 Context 기반 테마 전달 컴포넌트 |
| **유틸리티 CSS** | 하나의 클래스가 하나의 CSS 속성을 담당하는 설계 방식 |
| **PurgeCSS** | 빌드 시 미사용 CSS 클래스를 제거하여 번들 최적화 |
| **camelCase** | JS에서 CSS 속성명 표기 방식 (`fontSize`, `backgroundColor`) |
| **Pseudo-class** | `:hover`, `:focus`, `:active` 등 요소 상태 선택자 |
| **Tagged Template Literal** | `` styled.div`...` `` 형태의 JS 문법 |
| **SoC** | Separation of Concerns. 관심사의 분리 원칙 |
| **디자인 토큰** | 색상/간격/폰트 등 재사용 가능한 디자인 값의 집합 |
