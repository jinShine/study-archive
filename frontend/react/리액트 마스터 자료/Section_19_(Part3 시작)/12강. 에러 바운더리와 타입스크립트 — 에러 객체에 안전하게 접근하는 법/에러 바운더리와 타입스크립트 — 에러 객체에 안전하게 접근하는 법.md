📖 상세 개념 가이드 (The Core Manual)
우리가 아무리 정밀한 설계도를 그리고 런타임 검문소를 세워 외부의 침입을 막아낸다고 해도, 실제 프로그램이 돌아가는 복잡한 환경에서는 언제나 예상치 못한 변수가 발생하기 마련입니다.

마치 아무리 안전 운전을 해도 예기치 못한 사고에 대비해 에어백이 필요한 것처럼, 리액트 앱에서도 특정 부분에서 발생한 에러가 앱 전체를 마비시키지 않도록 보호해 주는 장치가 필요합니다. 오늘은 우리 앱의 안전벨트 역할을 하는 에러 바운더리(Error Boundary)를 타입스크립트와 결합하여, 발생한 에러의 정체를 안전하게 파악하고 대응하는 법을 상세히 살펴보겠습니다.

1. React.ReactNode: 리액트 세상의 '만능 보자기'
에러 바운더리를 설계하기 전, 우리는 React.ReactNode라는 타입을 명확히 이해해야 합니다. 이것은 리액트에서 가장 포괄적인 타입 중 하나입니다.

/* [Example]: React.ReactNode가 담을 수 있는 것들 */
const example1: React.ReactNode = <h1>안녕하세요</h1>; // JSX 요소
const example2: React.ReactNode = "텍스트 데이터";      // 문자열
const example3: React.ReactNode = 12345;               // 숫자
const example4: React.ReactNode = [example1, example2]; // 배열
const example5: React.ReactNode = null;                // 빈 값

[개념 상세 설명]ReactNode는 한마디로 "리액트가 화면에 그릴 수 있는 모든 것"을 의미합니다. 단순히 JSX 태그뿐만 아니라 문자열, 숫자, 심지어 아무것도 없는 null이나 배열 형태의 자식들도 모두 포함합니다. 에러 바운더리에서 children(평소 화면)과 fallback(대체 화면)의 타입을 ReactNode로 지정하는 이유는, 그 어떤 복잡한 UI 구조가 들어오더라도 안전하게 감싸서 처리하기 위함입니다.

2. 왜 2026년에도 '클래스 컴포넌트'가 필요한가?
현재 리액트 개발의 99%는 훅(Hooks)을 사용하는 함수형 컴포넌트가 주도합니다. 하지만 에러 바운더리만큼은 반드시 클래스 컴포넌트로 만들어야 합니다.

기술적 이유: 에러를 포착하는 특수한 생명주기 메서드인 getDerivedStateFromError와 componentDidCatch는 현재 리액트 훅 버전(useError 같은 것)이 존재하지 않습니다. 오직 클래스에서만 작동하도록 설계되어 있습니다.
실무적 가치: 클래스 컴포넌트는 이제 '기능 구현'의 도구가 아니라 '인프라 및 아키텍처'의 도구로 활용됩니다. 앱 전체의 에러를 받아내는 '최후의 보루' 역할을 수행하기 위해 숙련된 엔지니어는 클래스 컴포넌트의 특수한 능력을 정확히 다룰 줄 알아야 합니다.
3. unknown 타입과 타입 좁히기(Type Narrowing)
타입스크립트에서 발생한 에러(error)는 기본적으로 unknown 상태입니다. 자바스크립트는 에러 발생 시 객체뿐만 아니라 단순한 문자열이나 숫자를 던질 수도 있기 때문입니다.

/* [Example]: instanceof를 활용한 신분 확인 */
function handleError(error: unknown) {
  // errorMessage = error.message; // ❌ 에러: unknown 타입에는 message가 없을 수도 있음

  if (error instanceof Error) {
    console.log(error.message); // ✅ 통과: 이제 확실히 'Error' 객체임이 증명됨
  }
}

[개념 상세 설명]unknown은 "무엇인지 모르니 함부로 다루지 마라"는 경고입니다. 우리는 **instanceof Error**라는 연산자를 사용하여 정체불명의 에러가 우리가 아는 표준 에러 객체인지 검사합니다. 이 관문을 통과하는 순간 타입스크립트는 비로소 error.message에 접근하는 것을 허용해 줍니다. 이것이 불확실한 런타임 환경을 안전하게 통제하는 시니어의 기술입니다.

💻 실습 1단계: 에러 바운더리 설계 (GlobalErrorBoundary.tsx)
앱의 안전을 책임질 '두꺼비집' 역할을 하는 클래스 컴포넌트를 구현합니다.

/* [File Path]: src/components/GlobalErrorBoundary.tsx
   [Copyright]: © nhcodingstudio 소유
   [Test Process]:
   1. Props와 State 인터페이스가 정확히 정의되었는지 확인합니다.
   2. instanceof 연산자가 에러의 정체를 제대로 밝히는지 검토합니다.
*/

import React, { ErrorInfo, ReactNode } from 'react';

// 1. Props 설계: 무엇을 감싸고, 실패 시 무엇을 보여줄 것인가?
interface Props {
  children: ReactNode; // 무엇이든 담을 수 있는 보자기
  fallback: ReactNode; // 에러 발생 시 보여줄 대피소 UI
}

// 2. State 설계: 에러 발생 여부와 에러 객체를 보관함
interface State {
  hasError: boolean;
  error: unknown; // 정체를 알 수 없는 에러를 위해 unknown 사용
}

export class GlobalErrorBoundary extends React.Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  // [메서드]: 에러 발생 시 상태를 '에러 모드'로 즉시 전환합니다.
  static getDerivedStateFromError(error: unknown): State {
    return { hasError: true, error };
  }

  // [메서드]: 에러의 상세 정보를 외부 로그 서비스에 기록할 때 사용합니다.
  componentDidCatch(error: unknown, errorInfo: ErrorInfo) {
    console.error("🚨 에러 감지 지점 정보:", errorInfo);
  }

  render() {
    if (this.state.hasError) {
      let errorMessage = "알 수 없는 시스템 오류가 발생했습니다.";

      // [핵심 절차]: unknown 에러의 신분을 확인하여 타입을 좁힙니다.
      if (this.state.error instanceof Error) {
        errorMessage = this.state.error.message;
      }

      return (
        <div style={{ padding: '30px', border: '2px solid red', borderRadius: '12px', backgroundColor: '#fff5f5' }}>
          <h2 style={{ color: '#e53e3e' }}>시스템 보호 모드 작동 중</h2>
          <p style={{ fontSize: '1.1rem' }}>에러 내용: <strong>{errorMessage}</strong></p>
          <div style={{ marginTop: '20px' }}>
            {this.props.fallback}
          </div>
        </div>
      );
    }

    // 에러가 없다면 평소처럼 자식 컴포넌트들을 렌더링합니다.
    return this.props.children;
  }
}

🔍 상세 코드 분석
React.Component<Props, State>: 클래스 컴포넌트에 우리가 만든 설계도를 주입합니다. 첫 번째는 props, 두 번째는 state의 규격입니다.
getDerivedStateFromError: 하위 컴포넌트에서 에러가 터지면 리액트가 가장 먼저 호출하는 메서드입니다. 여기서 hasError를 true로 리턴하는 순간, 아래 render 함수는 에러 UI를 그리게 됩니다.
instanceof Error: unknown 타입인 error 객체에 함부로 접근하지 않고, 표준 규격인 Error 클래스의 자식인지 확인하는 '신분증 검사' 단계입니다.
💻 실습 2단계: 에러 상황 재현 및 통합 (App.tsx)
에러 바운더리가 실제로 작동하는지 확인하기 위해 의도적으로 에러를 일으키는 컴포넌트를 만듭니다.

/* [File Path]: src/App.tsx
   [Copyright]: © nhcodingstudio 소유
   [Test Process]:
   1. Bomb 컴포넌트가 에러를 던질 때, App 전체가 죽지 않는지 확인합니다.
   2. '시스템 재시작' 버튼이 나타나는지 관찰합니다.
*/

import React from 'react';
import { GlobalErrorBoundary } from './components/GlobalErrorBoundary';

// 💣 의도적으로 에러를 던지는 위험한 컴포넌트
function Bomb() {
  // 실제 상황에서는 API 실패나 잘못된 데이터 접근 시 발생합니다.
  throw new Error("💥 엔진 임계점 도달! 냉각 장치 가동 필요");
  return <div>작동 중...</div>;
}

function App() {
  return (
    <div style={{ padding: '40px', maxWidth: '800px', margin: '0 auto', fontFamily: 'sans-serif' }}>
      <h1>12강. 에러 바운더리 & 클래스 컴포넌트 실습</h1>
      <p>훅이 주류인 시대에도 클래스 기반 인프라는 앱을 지탱하는 핵심입니다.</p>

      <hr style={{ margin: '30px 0', opacity: 0.1 }} />

      {/* [실무 활용]: 위험 구역을 에러 바운더리로 격리합니다. */}
      <GlobalErrorBoundary
        fallback={
          <button onClick={() => window.location.reload()} style={{ padding: '10px 20px', cursor: 'pointer' }}>
            시스템 재시작
          </button>
        }
      >
        <section>
          <h3>🛡️ 안전 보호 구역 내부</h3>
          <Bomb />
        </section>
      </GlobalErrorBoundary>

      <section style={{ marginTop: '40px', color: '#38a169' }}>
        <p>✅ 이 텍스트가 보인다면, 에러 바운더리가 에러 전염을 막아낸 것입니다.</p>
      </section>
    </div>
  );
}

export default App;

🔍 상세 코드 분석
fallback={<button>...</button>}: 에러 발생 시 보여줄 대피소 UI를 전달합니다. ReactNode 타입이므로 단순한 글자부터 복잡한 컴포넌트까지 무엇이든 넣을 수 있습니다.
회복 탄력성(Resilience): Bomb이 터졌음에도 하단의 "✅ 이 텍스트가 보인다면..." 영역이 정상 출력되는 것에 주목하세요. 이것이 대규모 서비스에서 특정 기능 장애가 서비스 전체 중단으로 이어지지 않게 하는 '격리 방어' 전략의 정수입니다.
