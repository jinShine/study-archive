# 19. 타입스크립트 실무 패턴 완전 정복

## 개요

Section 19는 리액트 프로젝트에서 타입스크립트를 실무 수준으로 활용하기 위한 핵심 패턴을 체계적으로 다루는 파트다. 단순히 타입을 붙이는 수준을 넘어, Interface와 import type을 통한 데이터 규격화에서 시작하여 제네릭(Generic), 유틸리티 타입(Pick, Omit, Partial), 템플릿 리터럴 타입까지 점진적으로 난이도를 높여간다. 특히 "기초 타입 정의 -> 컴포넌트 적용 -> 시스템 통합"이라는 일관된 흐름을 통해 각 개념이 실제 리액트 아키텍처에서 어떤 역할을 하는지 체감할 수 있도록 구성되어 있다.

이 섹션의 또 다른 축은 **방어적 프로그래밍**이다. 09강에서 API 명세 변경으로 인한 런타임 붕괴(White Screen of Death)를 직접 재현한 뒤, 10강과 14강에서 `is` 키워드 기반 타입 가드와 스키마 검증으로 해결하는 Pain-Solution 구조를 취한다. 또한 12강에서는 에러 바운더리를 클래스 컴포넌트로 구현하여, 예측 불가능한 런타임 에러로부터 앱 전체를 보호하는 인프라 아키텍처를 다룬다.

최종적으로 이 섹션을 마치면 "컴파일 타임에 잡을 수 있는 버그는 컴파일 타임에 잡고, 런타임에서만 확인 가능한 데이터는 타입 가드로 검증하며, 그래도 터지는 에러는 에러 바운더리로 격리한다"는 3단 방어 체계를 갖출 수 있게 된다.

## 강의별 핵심 요약

### 03강. 타입스크립트 -- 리액트 엔지니어가 반드시 갖춰야 할 정밀 설계도

- **Interface**를 통해 객체의 구조(속성명, 타입)를 정의하고, 이를 컴포넌트 Props의 계약서로 활용한다.
- **`import type`** 문법으로 타입 정보만 가져오면 최종 번들 크기가 감소하고 순환 참조 에러를 방지할 수 있다.
- `as HTMLElement` 같은 **타입 단언(Type Assertion)** 은 개발자가 확신할 때만 제한적으로 사용해야 한다.
- 데이터 규격을 `types/` 폴더에 분리하면, 속성명 변경 시 단 한 곳만 수정해도 프로젝트 전체가 동기화된다.
- `displayName.toUpperCase()` 같은 메서드 호출이 타입 추론 덕분에 안전하게 보장된다.

### 04강. 타입스크립트의 기초와 리액트의 만남 -- 이름표의 마법과 정밀 설계도

- **기본 타입(Primitive)**: `string`, `number`, `boolean`과 선택적 속성(`?`)을 Props에 적용하는 법을 다룬다.
- **구별된 공용체(Discriminated Unions)**: `state: 'loading' | 'success' | 'error'` 패턴으로 논리적으로 모순된 상태를 원천 차단한다.
- **ReactNode**: 리액트가 렌더링 가능한 모든 것(JSX, 문자열, 숫자, null 등)을 담는 포괄 타입이다.
- **ComponentPropsWithRef + Omit**: HTML 표준 속성을 상속받으면서 충돌하는 속성만 교체하는 고급 패턴이다.
- **React.ChangeEvent\<HTMLInputElement\>** 등 이벤트 타입을 명시하면 `e.target.value` 자동 완성이 완벽히 지원된다.

### 05강. 제네릭(Generic)의 심화 -- 추상화를 정복하는 숙련된 엔지니어의 무기

- **제네릭(`<T>`)** 은 타입을 변수화하여, 호출 시점에 구체적 타입을 주입받는 "마법의 거푸집"이다.
- **`any` vs 제네릭**: `any`는 타입 정보를 소멸시키지만, 제네릭은 데이터의 정체성을 끝까지 추적한다.
- **`extends` 제약 조건**: `T extends { id: string | number }`로 최소 자격 요건을 부여하면 안전하게 `item.id`에 접근 가능하다.
- **Render Props 패턴**: `DataList<T>` 컴포넌트가 레이아웃만 담당하고, 렌더링 방식은 `renderRow` 콜백으로 위임하여 재사용성을 극대화한다.
- 유틸리티 함수(`wrapWithMetadata<T>`)로 원본 타입을 보존하면서 메타데이터를 추가하는 범용 래퍼를 설계할 수 있다.

### 07강. [Solution] TS 마이그레이션 -- 오타와 잘못된 데이터 흐름을 컴파일 단계에서 차단하기

- **Interface**로 상태(State)의 필수 속성과 타입을 정의하여 데이터 표준(도량형)을 세운다.
- **Union Type**으로 Reducer의 액션 메뉴판을 한정하면 존재하지 않는 액션 타입에 대해 컴파일 에러가 발생한다.
- **Discriminated Unions**: `type` 필드를 기준으로 각 액션의 `payload` 타입이 자동 매칭된다.
- 산술 연산(`productId + 100`)에서 타입 추론 덕분에 "숫자 + 문자열 = 기괴한 결합" 버그가 원천 봉쇄된다.
- JS에서 TS로 마이그레이션 시 핵심은 "확장자 변경"이 아니라 "시스템 구성 요소 간 엄격한 계약 체결"이다.

### 08강. 인터페이스 확장과 유틸리티 타입 -- Pick, Omit, Partial 실무 적용

- **`React.ComponentPropsWithoutRef<'button'>`**: HTML 표준 속성을 한 번에 추출하여 커스텀 컴포넌트에 상속시킨다.
- **`extends`를 통한 인터페이스 확장**: 공통 속성을 부모에 정의하고 자식이 상속받아 중복을 제거한다.
- **`Pick<T, K>`**: 원본에서 필요한 속성만 화이트리스트 방식으로 추출한다.
- **`Omit<T, K>`**: 보안상 노출 금지인 속성(adminNote, secretToken)을 블랙리스트 방식으로 제거한다.
- **`Partial<T>`**: 모든 속성을 선택 사항으로 변환하여 부분 업데이트(Patch 패턴)를 안전하게 구현한다.

### 09강. [Pain] API 명세가 바뀌는 현장 -- 우리 성벽이 무너지는 유일한 통로

- Interface는 **컴파일 타임 전용** 설계도일 뿐, 런타임에서 실제 데이터를 물리적으로 강제하지 못한다.
- **`as` 타입 단언**은 타입스크립트의 검사를 포기시키는 위험한 행위이며, "트로이의 목마"가 될 수 있다.
- 서버가 `price`를 `amount`로 바꾸면 `product.price`는 `undefined`가 되어 `.toLocaleString()` 호출 시 앱이 사망한다.
- 컴파일 타임과 런타임의 **간극(Gap)** 을 인지하는 것이 방어적 아키텍처의 출발점이다.
- 이 강의는 의도적으로 White Screen of Death를 재현하여 문제의 심각성을 체감하게 하는 Pain 포인트 강의다.

### 10강. [Solution] 스키마(Schema) 기반 타입 설계 -- 프론트와 백엔드의 완벽한 싱크

- **스키마(Schema)**: 인터페이스가 "포스트잇"이라면 스키마는 데이터를 전수 조사하는 "정밀 검열기"다.
- **`is` 키워드(Type Predicate)**: `data is ProductDetail`을 반환 타입으로 사용하면 `true` 반환 시 타입이 확정(Narrowing)된다.
- 일반 `boolean` 반환과 `is` 반환의 결정적 차이: `is`는 함수 밖에서도 타입을 확정짓지만, `boolean`은 여전히 `any`로 남는다.
- **Fail Fast 전략**: 검증 실패 시 앱을 터뜨리는 대신 에러 상태를 업데이트하여 사용자에게 안내 UI를 보여준다.
- `typeof` 연산자로 각 속성의 실제 자료형을 검사하는 타입 가드 함수를 컴포넌트 진입 전에 배치한다.

### 11강. 고급 타입 기법 -- Template Literal Types로 디자인 시스템 토큰 강제하기

- **`type` 별칭(Type Alias)**: 복잡한 타입에 의미 있는 이름을 부여하여 가독성과 재사용성을 높인다.
- **템플릿 리터럴 타입**: `` `${Color}-${Level}` `` 구문으로 유니온 타입의 모든 조합을 자동 생성한다.
- Color 3종 x Level 5종 = 총 15개의 디자인 토큰이 한 줄의 타입 정의로 만들어진다.
- 정의되지 않은 토큰(예: `red-100`, `primary-600`)을 사용하면 **즉시 컴파일 에러**가 발생하여 디자인 일관성을 보장한다.
- 에디터에서 `Ctrl+Space` 자동 완성으로 유효한 토큰 목록이 제시되어 협업 시 문서 확인 비용이 절감된다.

### 12강. 에러 바운더리와 타입스크립트 -- 에러 객체에 안전하게 접근하는 법

- **에러 바운더리(Error Boundary)** 는 2026년 현재에도 **반드시 클래스 컴포넌트**로 구현해야 한다 (Hooks 미지원).
- `getDerivedStateFromError`로 에러 발생 시 상태를 전환하고, `componentDidCatch`로 로그를 기록한다.
- 에러 객체는 기본적으로 **`unknown`** 타입이므로, `instanceof Error`로 신분을 확인한 뒤에만 `.message`에 접근 가능하다.
- **격리 방어 전략**: 에러 바운더리로 감싼 영역 밖의 컴포넌트는 내부 에러의 영향을 받지 않는다.
- `fallback` prop을 `ReactNode`로 선언하면 재시작 버튼, 안내 문구 등 어떤 UI도 대체 화면으로 주입할 수 있다.

### 14강. [Solution] 엄격한 타입 가드(Type Guard)로 런타임 안정성 극대화하기

- **타입 가드 함수**를 `guards/` 폴더에 분리하여 검증 로직의 재사용성과 테스트 용이성을 확보한다.
- `data is UserProfile` 타입 서술어를 사용하면 `if` 블록 내부에서 `unknown` -> `UserProfile`로 타입이 좁혀진다.
- 닉네임 길이(`length >= 2`) 같은 **비즈니스 규칙까지 타입 가드에 포함**하여 규격 미달 데이터를 차단한다.
- `useRef<HTMLInputElement>`와 `useId`를 결합한 실전 폼에서 검증 결과에 따라 UI(테두리 색상, 포커스)를 동적으로 제어한다.
- 통과/미달/Ref 안정성 3가지 테스트 시나리오를 통해 타입 가드의 실효성을 확인한다.

## 면접 대비 Q&A

### Q1. `interface`와 `type`의 차이는 무엇이며, 언제 어떤 것을 사용하나요?

**A.** `interface`는 객체의 구조를 정의하는 데 특화되어 있고 `extends`로 상속이 가능하며, 동일 이름의 선언을 자동 병합(Declaration Merging)한다. `type`은 유니온(`|`), 인터섹션(`&`), 템플릿 리터럴 등 더 유연한 타입 조합이 가능하다. 실무에서는 **Props/State 같은 객체 구조는 `interface`**, 유니온 타입이나 조합형 타입은 `type`을 쓰는 것이 관례다.

### Q2. `import type`을 사용하는 이유는 무엇인가요?

**A.** `import type`은 해당 임포트가 순수하게 타입 정보만을 위한 것임을 명시한다. 컴파일 시 자바스크립트 번들에서 완전히 제거되므로 **번들 크기가 감소**하고, 값과 타입의 혼동으로 인한 **순환 참조 에러를 방지**할 수 있다.

### Q3. 제네릭의 `extends` 제약 조건이 필요한 이유를 설명해주세요.

**A.** `<T>` 단독으로는 T가 어떤 구조인지 알 수 없어서 `item.id` 같은 속성 접근이 불가능하다. `T extends { id: string | number }`처럼 **최소 자격 요건**을 부여하면 T가 반드시 `id`를 가진 객체임이 보장되어 안전하게 속성에 접근할 수 있다. 이는 유연성과 안전성을 동시에 확보하는 핵심 기법이다.

### Q4. 타입스크립트의 `as` 키워드가 위험한 이유는 무엇인가요?

**A.** `as`는 타입스크립트의 검증을 우회하여 개발자가 "이 데이터는 이 타입이다"라고 강제 선언하는 것이다. 실제 런타임 데이터가 해당 타입과 다를 경우에도 컴파일 에러가 발생하지 않아, 결국 **런타임에서 `undefined` 접근 에러 등 White Screen of Death**를 유발한다. 외부 데이터(API 응답)에 `as`를 사용하는 것은 특히 위험하다.

### Q5. 타입 가드에서 `boolean` 반환과 `is` 반환의 차이는 무엇인가요?

**A.** `boolean`을 반환하면 true/false 판별만 되고, 함수 밖에서 데이터 타입은 여전히 `any`/`unknown`으로 남는다. 반면 `data is ProductDetail`을 사용하면 함수가 `true`를 반환하는 순간 타입스크립트가 해당 변수의 타입을 **함수 호출부에서도 확정(Narrowing)** 시켜준다. 이를 통해 `if` 블록 내부에서 안전하게 속성 접근이 가능해진다.

### Q6. 에러 바운더리를 함수형 컴포넌트(Hooks)로 구현할 수 없는 이유는 무엇인가요?

**A.** 에러를 포착하는 생명주기 메서드인 `getDerivedStateFromError`와 `componentDidCatch`는 현재 React Hooks 버전이 존재하지 않는다. 따라서 에러 바운더리는 **반드시 클래스 컴포넌트**로 구현해야 하며, 이는 클래스 컴포넌트가 2026년에도 '인프라 아키텍처'의 도구로서 여전히 유효한 이유다.

### Q7. `Pick`, `Omit`, `Partial`의 실무 사용 사례를 각각 하나씩 들어주세요.

**A.**
- **`Pick<User, 'name' | 'email'>`**: 전체 유저 정보에서 연락처 목록에 필요한 속성만 추출할 때 사용한다.
- **`Omit<Product, 'adminNote' | 'secretToken'>`**: 보안상 민감한 필드를 제거하여 사용자 화면에 노출하지 않을 때 사용한다.
- **`Partial<UserProfile>`**: 프로필 수정 시 변경된 필드만 전송하는 PATCH 패턴에서, 모든 필드를 선택 사항으로 만들 때 사용한다.

### Q8. 구별된 공용체(Discriminated Union)란 무엇이며, 왜 `isLoading: boolean` 패턴보다 나은가요?

**A.** 구별된 공용체는 공통된 리터럴 필드(예: `state`)를 기준으로 각 분기의 데이터 구조를 다르게 정의하는 기법이다. `isLoading: boolean`은 "로딩 중이면서 동시에 에러" 같은 **논리적 모순 상태**를 허용하지만, `state: 'loading' | 'success' | 'error'`는 한 시점에 오직 하나의 상태만 존재하도록 강제하여 불가능한 상태 조합을 원천 차단한다.

### Q9. 템플릿 리터럴 타입의 실무 장점은 무엇인가요?

**A.** 유니온 타입을 조합하여 `${Color}-${Level}` 형태로 수십~수백 개의 유효한 문자열 타입을 자동 생성할 수 있다. 디자인 시스템에서 색상, 간격, 크기 토큰을 관리할 때 **오타로 인한 디자인 붕괴를 컴파일 단계에서 차단**하고, 에디터 자동 완성으로 유효한 토큰 목록을 실시간으로 제시하여 **대규모 협업 시 문서 확인 비용을 절감**한다.

### Q10. 컴파일 타임과 런타임의 간극을 메우기 위한 전략을 설명해주세요.

**A.** 3단 방어 체계를 구축한다. **(1) 컴파일 타임**: Interface, 유틸리티 타입, 제네릭으로 코드 작성 시점의 오류를 잡는다. **(2) 런타임 검증**: `is` 키워드 기반 타입 가드 함수로 외부 데이터(API 응답)의 실제 구조를 검사하고, 불일치 시 Fail Fast로 안전한 대체 UI를 보여준다. **(3) 최후 방어**: 에러 바운더리로 예측 불가능한 런타임 에러를 격리하여 앱 전체 붕괴를 방지한다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **Interface** | 객체가 가져야 할 속성과 타입을 정의하는 계약서. `extends`로 상속 가능 |
| **Type Alias** | `type` 키워드로 복합 타입에 별명을 부여하는 것 |
| **Generic (`<T>`)** | 타입을 매개변수처럼 다루어 호출 시점에 구체 타입을 주입받는 기법 |
| **`extends` (제네릭)** | 제네릭 타입 매개변수에 최소 자격 요건(구조적 제약)을 부여하는 키워드 |
| **Union Type (`\|`)** | "A 또는 B" 중 하나를 허용하는 타입 조합 |
| **Discriminated Union** | 공통 리터럴 필드를 판별자로 두어 분기별 데이터 구조를 안전하게 관리하는 패턴 |
| **Type Narrowing** | `if`, `typeof`, `instanceof` 등으로 넓은 타입을 좁은 타입으로 확정하는 과정 |
| **Type Predicate (`is`)** | 함수 반환 타입에 `data is T`를 명시하여 런타임 검증 결과를 타입 시스템에 반영하는 문법 |
| **Type Assertion (`as`)** | 개발자가 타입을 강제 확정하는 구문. 검증을 우회하므로 위험할 수 있음 |
| **`import type`** | 타입 정보만 가져오는 임포트. 번들에서 제거되어 최적화에 기여 |
| **`Pick<T, K>`** | 원본 타입 T에서 지정한 키 K만 추출하는 유틸리티 타입 |
| **`Omit<T, K>`** | 원본 타입 T에서 지정한 키 K만 제거하는 유틸리티 타입 |
| **`Partial<T>`** | 원본 타입 T의 모든 속성을 선택 사항(`?`)으로 변환하는 유틸리티 타입 |
| **Template Literal Type** | 백틱과 `${}`를 사용해 문자열 조합 타입을 자동 생성하는 고급 타입 기법 |
| **`ComponentPropsWithoutRef`** | HTML 태그의 표준 Props 타입을 ref 제외하고 추출하는 리액트 유틸리티 타입 |
| **`ReactNode`** | 리액트가 렌더링할 수 있는 모든 것(JSX, 문자열, 숫자, null 등)을 포괄하는 타입 |
| **Error Boundary** | 하위 컴포넌트의 런타임 에러를 포착하여 앱 전체 붕괴를 방지하는 클래스 컴포넌트 |
| **`getDerivedStateFromError`** | 에러 발생 시 호출되어 상태를 에러 모드로 전환하는 정적 메서드 |
| **`componentDidCatch`** | 에러 상세 정보를 외부 로그 서비스에 전송할 때 사용하는 생명주기 메서드 |
| **`unknown`** | "정체를 알 수 없으니 함부로 다루지 마라"는 안전한 최상위 타입. `any`보다 안전 |
| **Fail Fast** | 오류 발견 즉시 빠르게 실패 처리하여 연쇄 피해를 막는 전략 |
| **White Screen of Death** | 런타임 에러로 인해 앱 전체가 렌더링되지 않는 치명적 상태 |
| **Render Props** | 컴포넌트의 렌더링 방식을 콜백 함수로 위임하여 재사용성을 극대화하는 패턴 |

## 자가 점검 체크리스트

- [ ] `interface`를 사용해 Props의 구조를 정의하고, 필수/선택(`?`) 속성을 구분할 수 있다
- [ ] `import type`과 일반 `import`의 차이를 설명하고, 언제 사용해야 하는지 안다
- [ ] 구별된 공용체(Discriminated Union)를 활용하여 로딩/성공/에러 상태를 안전하게 분기할 수 있다
- [ ] 제네릭(`<T>`)의 동작 원리를 이해하고, `extends`로 제약 조건을 설정할 수 있다
- [ ] `Pick`, `Omit`, `Partial` 유틸리티 타입을 실무 시나리오에 맞게 활용할 수 있다
- [ ] `ComponentPropsWithoutRef`를 사용해 HTML 표준 속성을 상속받는 커스텀 컴포넌트를 만들 수 있다
- [ ] `as` 타입 단언의 위험성을 이해하고, 외부 데이터에 사용하지 않아야 하는 이유를 설명할 수 있다
- [ ] `is` 키워드(Type Predicate)를 사용한 타입 가드 함수를 작성하고, `boolean` 반환과의 차이를 설명할 수 있다
- [ ] 스키마 기반 검증으로 API 응답 데이터를 런타임에서 검사하는 Fail Fast 패턴을 구현할 수 있다
- [ ] 템플릿 리터럴 타입으로 디자인 토큰 조합을 자동 생성하고, 오타를 컴파일 단계에서 차단할 수 있다
- [ ] 에러 바운더리를 클래스 컴포넌트로 구현하고, `unknown` 에러에 `instanceof`로 안전 접근할 수 있다
- [ ] 컴파일 타임 -> 런타임 검증 -> 에러 격리의 3단 방어 체계를 설명할 수 있다
