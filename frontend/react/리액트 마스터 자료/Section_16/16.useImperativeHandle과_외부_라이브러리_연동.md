## 39강. useImperativeHandle 기초 -- 부모에게만 허락된 비밀 버튼

### 도입

리액트는 "상태가 이러하니 화면을 이렇게 그려라"는 **선언적** 방식을 지향한다. 하지만 실무에서는 부모가 자식에게 "지금 당장 포커스를 줘!", "애니메이션을 실행해!"라고 **직접 명령**해야 하는 상황이 발생한다. `useImperativeHandle`은 리액트 원칙을 지키면서도 자식의 캡슐화를 훼손하지 않고 부모에게 **선별적 조종기**를 건네주는 훅이다.

### 개념 설명

#### forwardRef -- ref 전달 통로 개설

리액트에서 `ref`는 일반적인 props처럼 자식에게 전달되지 않는다. 자식을 `forwardRef`로 감싸야 외부에서 보낸 ref를 받을 수 있다.

```
[부모]                    [자식 (forwardRef)]
ref={inputControlRef} -> (props, ref) => { ... }
```

#### useImperativeHandle -- 노출 범위 제한

| 방식 | ref에 담기는 것 | 위험도 |
| ---- | --------------- | ------ |
| ref를 input에 직접 연결 | DOM 노드 전체 (style, value, type 등 모든 속성) | 높음 -- 부모가 자식 내부를 마음대로 조작 가능 |
| useImperativeHandle 사용 | 허락된 함수만 담긴 객체 (focus, clear 등) | 낮음 -- 명확한 API 경계 |

#### 핵심 설계 원칙: 최소 노출 원칙 (Principle of Least Exposure)

자식은 내부 DOM이나 상태를 통째로 넘기지 않고, "나를 제어하려면 **이 버튼들만** 써!"라고 명확한 인터페이스(API)만 노출한다. 이것이 컴포넌트 간 경계를 지키는 **캡슐화의 정석**이다.

### 코드 예제

#### 자식: 비밀 조종기 정의 (MyInput.jsx)

```jsx
import { useImperativeHandle, forwardRef, useRef } from 'react';

const MyInput = forwardRef((props, ref) => {
  // 자식 내부에서 실제 DOM을 붙잡을 내부용 ref
  const inputRef = useRef();

  // 부모가 ref.current.xxx()로 호출할 수 있는 함수들만 선별 노출
  useImperativeHandle(ref, () => ({
    focus: () => {
      inputRef.current.focus();
    },
    clear: () => {
      inputRef.current.value = "";
    }
  }), []); // 마운트 시 한 번 고정

  return <input ref={inputRef} {...props} style={{ padding: '10px', border: '2px solid #3b82f6' }} />;
});

export default MyInput;
```

#### 부모: 명령 내리기 (App.jsx)

```jsx
import { useRef } from 'react';
import MyInput from './components/MyInput';

export default function App() {
  const inputControlRef = useRef();

  return (
    <div style={{ padding: '50px' }}>
      <MyInput ref={inputControlRef} placeholder="부모가 제어하는 입력창" />

      <div style={{ marginTop: '20px' }}>
        <button onClick={() => inputControlRef.current.focus()}>
          강제 포커스 명령
        </button>
        <button onClick={() => inputControlRef.current.clear()}>
          입력창 비우기 명령
        </button>
      </div>
    </div>
  );
}
```

### 코드 해설

- **`forwardRef((props, ref) => { ... })`** -- 자식 컴포넌트를 감싸서 부모의 ref를 두 번째 인자로 받을 수 있게 한다.
- **`useImperativeHandle(ref, createHandle, deps)`** -- 첫 번째 인자에 부모 ref, 두 번째에 노출할 메서드 객체를 반환하는 팩토리 함수, 세 번째에 의존성 배열을 전달한다.
- **내부 `inputRef` vs 외부 `ref`** -- 자식은 `inputRef`로 실제 DOM을 직접 제어하되, 부모에게는 `useImperativeHandle`을 통해 허락된 함수만 건넨다.
- 부모는 `inputControlRef.current.focus()`처럼 자식이 정의한 API만 호출할 수 있고, `inputControlRef.current.style`처럼 DOM에 직접 접근하는 것은 불가하다.

### 실무 비유

> 자동차의 **핸들, 페달, 기어**만 운전자에게 노출하고 엔진 내부 배선은 감추는 것과 같다. 운전자(부모)가 엔진을 직접 만지면 위험하지만, 잘 설계된 인터페이스를 통해 안전하게 조종할 수 있다.

### 핵심 포인트

| 항목 | 설명 |
| ---- | ---- |
| forwardRef | ref라는 특수 소품을 자식에게 전달하기 위한 래퍼 |
| useImperativeHandle | 부모에게 노출할 기능을 객체 형태로 **선별** 정의 |
| 캡슐화 | 자식의 DOM 노드를 직접 넘기지 않고 함수로 감싸서 노출 |
| 사용 시점 | 선언적 방식으로 해결 불가한 포커스, 스크롤, 애니메이션 등에 한정 |

## 40강. useImperativeHandle 실무 -- 폼 검증과 애니메이션 트리거

### 도입

기초에서 배운 `focus`/`clear` 수준을 넘어, 실무에서 자주 마주하는 두 가지 시나리오에 적용한다: (1) 부모가 "가입하기" 버튼을 누르면 자식 입력창이 스스로 검증하고 에러를 표시하는 **폼 검증**, (2) 명령 즉시 프레임 지연 없이 상자를 흔드는 **애니메이션 트리거**.

### 개념 설명

#### 명령적 방식이 실무의 무기가 되는 이유

일반적인 리액트 방식(Props 전달)으로 "흔들어라" 신호를 주면:
- 리액트 렌더링 스케줄링으로 인해 애니메이션이 미세하게 지연
- 연속 클릭 시 반응하지 않을 수 있음

`useImperativeHandle`은 리액트 렌더링 과정을 **우회**하여 부모의 명령에서 자식의 DOM 조작으로 곧장 이어지는 **지름길**을 만든다.

#### 강제 리플로우(Forced Reflow) 트릭

```
1. 클래스 제거 (애니메이션 초기화)
2. void element.offsetWidth  (브라우저에 "지금 당장 스타일 재계산!" 명령)
3. 클래스 추가 (애니메이션 재시작)
```

`void offsetWidth`가 없으면 브라우저는 1번과 3번을 하나로 묶어 최적화해버려 애니메이션이 재시작되지 않는다.

### 코드 예제

#### 검증 기능을 품은 입력창 (ValidatedInput.jsx)

```jsx
import { useState, useImperativeHandle, forwardRef, useRef } from 'react';

const ValidatedInput = forwardRef((props, ref) => {
  const inputRef = useRef();
  const [error, setError] = useState(false);

  useImperativeHandle(ref, () => ({
    // 부모가 호출하면 자신의 값을 검사하고 결과(boolean)를 반환
    validate: () => {
      if (!inputRef.current.value) {
        setError(true);
        return false; // 검증 실패
      }
      setError(false);
      return true; // 검증 성공
    },
    // 검증 실패 시 사용자의 시선을 이 입력창으로 이동
    focus: () => {
      inputRef.current.focus();
    }
  }), []);

  return (
    <div style={{ marginBottom: '15px' }}>
      <input
        ref={inputRef}
        placeholder={props.placeholder}
        style={{
          padding: '10px',
          border: `2px solid ${error ? 'red' : '#ccc'}`,
          borderRadius: '4px', width: '200px'
        }}
      />
      {error && <p style={{ color: 'red', fontSize: '12px' }}>필수 입력 항목입니다.</p>}
    </div>
  );
});

export default ValidatedInput;
```

#### 강제 흔들기 기능을 품은 박스 (AnimatedBox.jsx)

```jsx
import { useImperativeHandle, forwardRef, useRef } from 'react';

const AnimatedBox = forwardRef((props, ref) => {
  const boxRef = useRef();

  useImperativeHandle(ref, () => ({
    startShake: () => {
      if (boxRef.current) {
        boxRef.current.classList.remove('shake-animation');
        // 강제 리플로우: 브라우저에 즉시 스타일 재계산 요청
        void boxRef.current.offsetWidth;
        boxRef.current.classList.add('shake-animation');
      }
    }
  }), []);

  return (
    <div ref={boxRef} style={{
      width: '100px', height: '100px', background: 'coral',
      display: 'flex', alignItems: 'center', justifyContent: 'center',
      color: 'white', fontWeight: 'bold'
    }}>
      Box
    </div>
  );
});

export default AnimatedBox;
```

#### 부모에서 두 명령 조합 (App.jsx)

```jsx
import { useRef } from 'react';
import ValidatedInput from './components/ValidatedInput';
import AnimatedBox from './components/AnimatedBox';

export default function App() {
  const inputRef = useRef();
  const boxRef = useRef();

  const handleSubmit = () => {
    if (!inputRef.current.validate()) {
      inputRef.current.focus();      // 실패한 입력창에 포커스
      boxRef.current.startShake();   // 경고 애니메이션 실행
    } else {
      alert("가입 성공!");
    }
  };

  return (
    <div style={{ padding: '20px' }}>
      <ValidatedInput ref={inputRef} placeholder="이름을 입력하세요" />
      <AnimatedBox ref={boxRef} />
      <button onClick={handleSubmit}>가입하기</button>
    </div>
  );
}
```

### 코드 해설

- **`validate()` 반환값** -- boolean을 반환하므로 부모가 `if (!ref.current.validate())` 형태로 조건 분기를 작성할 수 있다. 자식이 에러 상태를 내부에서 관리하되, 검증 결과만 부모에게 보고하는 구조다.
- **깔끔한 분업** -- 부모는 자식의 CSS 클래스명이나 내부 `error` 상태를 알 필요가 없다. 오직 `validate()`, `focus()`, `startShake()`라는 이름표(API)만 보고 명령을 내린다.
- **`void boxRef.current.offsetWidth`** -- `offsetWidth`를 읽으면 브라우저가 현재까지의 스타일 변경을 즉시 반영(리플로우)한다. `void`는 반환값을 버리겠다는 명시적 표현이다. 이 한 줄이 없으면 클래스 제거와 추가가 하나의 프레임에 합쳐져 애니메이션이 재시작되지 않는다.

### 실무 비유

> 병원 접수대(부모)가 각 진료실(자식)에게 "환자 검사해!" 라는 명령만 내리고, 진료실이 검사를 수행한 뒤 "이상 없음/이상 있음"만 보고하는 구조다. 접수대는 진료실 안의 장비나 절차를 알 필요 없이 결과만 받아 다음 행동을 결정한다.

### 핵심 포인트

| 항목 | 설명 |
| ---- | ---- |
| 폼 검증 패턴 | `validate()` -> boolean 반환, 실패 시 `focus()`로 시선 유도 |
| 강제 리플로우 | `void element.offsetWidth`로 애니메이션 즉시 재시작 보장 |
| 명령 조합 | 여러 자식의 명령적 API를 부모에서 조합하여 복합 동작 구현 |
| 렌더링 우회 | 리액트 스케줄링을 건너뛰고 DOM 직접 조작으로 즉각 반응 |

## 41강. 외부 라이브러리 연동 -- 명령적 API 포장 전략

### 도입

실무에서 가장 까다로운 작업 중 하나는 **비리액트 라이브러리**(날짜 선택기, 차트, 지도 등)를 리액트 생태계 안으로 편입시키는 것이다. 이 강의에서는 Flatpickr(날짜 선택 라이브러리)를 예시로, 라이브러리의 복잡함을 자식 내부에 숨기고 부모에게 안전한 조종기만 건네는 **캡슐화 전략**을 구현한다.

### 개념 설명

#### 외부 라이브러리 연동의 3단계 아키텍처

```
[부모 App.jsx]           [래퍼 MyDatePicker.jsx]         [외부 라이브러리 Flatpickr]
                              |                                |
openPicker()  ---------->  fp.current.open()  ---------->  달력 UI 열기
closePicker() ---------->  fp.current.close() ---------->  달력 UI 닫기
clearDate()   ---------->  fp.current.clear() ---------->  선택 초기화
                              |
                    useImperativeHandle
                    (안전한 API만 노출)
```

#### 핵심 설계 원칙

| 원칙 | 설명 |
| ---- | ---- |
| 결합도 하락 | 라이브러리를 교체해도 부모 코드 수정 불필요 (래퍼 내부만 변경) |
| 보안 강화 | 부모가 input 속성을 직접 건드려 라이브러리를 망가뜨리는 것을 차단 |
| 의도 명확화 | `fp.current.clear()` 대신 `clearDate()`처럼 동료에게 친절한 이름 제공 |
| 메모리 누수 방지 | cleanup 함수에서 `fp.current.destroy()` 호출 필수 |

### 코드 예제

#### 자식: 라이브러리 래핑 (MyDatePicker.jsx)

```jsx
import { useEffect, useImperativeHandle, forwardRef, useRef } from 'react';
import flatpickr from 'flatpickr';
import 'flatpickr/dist/flatpickr.min.css';

const MyDatePicker = forwardRef((props, ref) => {
  const inputRef = useRef();   // 실제 DOM input (라이브러리가 달라붙을 지점)
  const fp = useRef();         // 라이브러리 인스턴스 보관

  // 라이브러리 초기화 및 정리
  useEffect(() => {
    fp.current = flatpickr(inputRef.current, {
      onChange: (selectedDates, dateStr) => {
        if (props.onChange) props.onChange(dateStr);
      },
    });

    // 컴포넌트 제거 시 메모리 누수 방지
    return () => {
      if (fp.current) {
        fp.current.destroy();
      }
    };
  }, [props]);

  // 부모에게 노출할 안전한 API
  useImperativeHandle(ref, () => ({
    openPicker: () => {
      if (fp.current) fp.current.open();
    },
    closePicker: () => {
      if (fp.current) fp.current.close();
    },
    clearDate: () => {
      if (fp.current) fp.current.clear();
    }
  }), []);

  return (
    <input
      ref={inputRef}
      type="text"
      placeholder="날짜를 선택하세요"
      style={{ padding: '10px', borderRadius: '8px', border: '2px solid #ddd', width: '250px' }}
    />
  );
});

export default MyDatePicker;
```

#### 부모: 라이브러리 문법 모르고 제어 (App.jsx)

```jsx
import { useRef, useState } from 'react';
import MyDatePicker from './components/MyDatePicker';

export default function App() {
  const datePickerRef = useRef();
  const [selectedDate, setSelectedDate] = useState("");

  return (
    <div style={{ padding: '50px' }}>
      <h1>데이트 피커 원격 제어</h1>
      <p>선택된 날짜: <strong>{selectedDate || "없음"}</strong></p>

      <div style={{ display: 'flex', gap: '10px', marginBottom: '20px' }}>
        <button onClick={() => datePickerRef.current.openPicker()}>창 열기</button>
        <button onClick={() => datePickerRef.current.closePicker()}>창 닫기</button>
        <button onClick={() => datePickerRef.current.clearDate()}>날짜 초기화</button>
      </div>

      <MyDatePicker
        ref={datePickerRef}
        onChange={(date) => setSelectedDate(date)}
      />
    </div>
  );
}
```

### 코드 해설

- **이중 ref 패턴** -- `inputRef`는 실제 DOM input을 가리키고, `fp`는 flatpickr 인스턴스를 보관한다. 라이브러리가 DOM에 직접 달라붙으므로 두 개의 ref가 각각 다른 역할을 수행한다.
- **useEffect로 초기화** -- 외부 라이브러리는 실제 DOM이 존재해야 초기화 가능하므로 `useEffect` 내에서 `flatpickr(inputRef.current, options)`를 호출한다.
- **cleanup의 `destroy()`** -- flatpickr 같은 외부 라이브러리는 DOM 이벤트 리스너와 추가 DOM 노드를 생성한다. 컴포넌트 언마운트 시 `destroy()`를 호출하지 않으면 **메모리 누수**가 발생한다.
- **부모의 무관심** -- `App.jsx`에는 `flatpickr` import가 전혀 없다. 부모는 `openPicker()`, `closePicker()`, `clearDate()`라는 우리가 정의한 인터페이스만 알면 된다. 나중에 flatpickr를 다른 라이브러리로 교체해도 `MyDatePicker` 내부만 수정하면 된다.

### 실무 비유

> 택시 승객(부모)은 "서울역으로 가주세요"라고만 말하고, 택시 기사(래퍼 컴포넌트)가 내비게이션(외부 라이브러리)의 복잡한 API를 다룬다. 내비게이션을 구글맵에서 카카오맵으로 바꿔도 승객에게는 아무 영향이 없다.

## 면접 핵심 Q&A

**Q1. useImperativeHandle은 무엇이고 왜 사용하나요?**

> 부모가 자식의 ref를 통해 호출할 수 있는 메서드를 **선별적으로 정의**하는 훅이다. 자식의 내부 DOM 노드를 통째로 노출하면 부모가 예상치 못한 조작을 할 수 있으므로, **최소 노출 원칙**에 따라 허락된 기능(focus, validate 등)만 API로 제공하여 캡슐화를 유지한다.

**Q2. forwardRef 없이 useImperativeHandle을 사용할 수 있나요?**

> 아니다. 리액트에서 `ref`는 일반 props로 전달되지 않는 특수 속성이다. 자식이 부모의 ref를 받으려면 반드시 `forwardRef`로 감싸야 하며, 그 안에서 `useImperativeHandle`로 노출할 인터페이스를 정의한다. (React 19부터는 props로 ref를 직접 전달 가능하지만, forwardRef 패턴이 여전히 표준적이다.)

**Q3. void element.offsetWidth가 하는 역할은 무엇인가요?**

> 브라우저에게 **강제 리플로우(Forced Reflow)** 를 유도하는 트릭이다. CSS 클래스를 제거하고 다시 추가할 때, 브라우저는 최적화를 위해 두 작업을 하나로 합쳐 애니메이션이 재시작되지 않을 수 있다. `offsetWidth`를 읽으면 브라우저가 현재까지의 스타일 변경을 즉시 계산하도록 강제하여, 클래스 재추가 시 애니메이션이 처음부터 다시 시작된다.

**Q4. 외부 라이브러리를 리액트에 연동할 때 래퍼 컴포넌트를 만드는 이유는?**

> (1) **결합도 제거**: 부모가 라이브러리 API를 직접 호출하지 않으므로 라이브러리 교체 시 래퍼 내부만 수정하면 된다. (2) **안전성**: 부모가 라이브러리가 달라붙은 DOM을 직접 조작하여 망가뜨리는 것을 방지한다. (3) **메모리 관리**: cleanup 함수에서 `destroy()`를 호출하여 메모리 누수를 체계적으로 방지한다.

**Q5. useImperativeHandle의 의존성 배열은 어떤 역할을 하나요?**

> `useImperativeHandle(ref, createHandle, deps)`의 세 번째 인자이다. 빈 배열 `[]`이면 마운트 시 한 번만 핸들 객체를 생성하고, 특정 값을 넣으면 해당 값이 변할 때마다 핸들 객체를 재생성한다. 핸들 내부에서 참조하는 상태나 props가 변할 수 있다면 의존성 배열에 포함해야 최신 값을 반영할 수 있다.

## 용어 글로서리

| 용어 | 설명 |
| ---- | ---- |
| Imperative (명령적) | "이것을 해라"처럼 구체적인 동작을 직접 지시하는 프로그래밍 방식 |
| Declarative (선언적) | "상태가 이러하니 화면은 이렇게 되어야 한다"고 결과를 선언하는 방식 (리액트 기본) |
| forwardRef | 부모의 ref를 자식 컴포넌트 내부로 전달하기 위한 리액트 래퍼 함수 |
| useImperativeHandle | 부모에게 노출할 ref 핸들(메서드 집합)을 선별적으로 정의하는 훅 |
| 캡슐화 (Encapsulation) | 내부 구현을 외부로부터 숨기고 공개 인터페이스만 제공하는 설계 원칙 |
| 강제 리플로우 (Forced Reflow) | `offsetWidth` 등을 읽어 브라우저에 즉시 레이아웃 재계산을 강제하는 기법 |
| 래퍼 컴포넌트 (Wrapper) | 외부 라이브러리를 리액트 인터페이스로 감싸는 중간 컴포넌트 |
| 이중 ref 패턴 | DOM용 ref와 라이브러리 인스턴스용 ref를 별도로 관리하는 패턴 |
| destroy() | 외부 라이브러리가 생성한 이벤트 리스너와 DOM 노드를 제거하는 정리 메서드 |
| 최소 노출 원칙 | 외부에 꼭 필요한 인터페이스만 공개하고 나머지는 숨기는 설계 원칙 |

## 자가 점검 체크리스트

- [ ] forwardRef가 ref를 자식에게 전달하기 위한 통로임을 이해했다
- [ ] useImperativeHandle로 부모에게 노출할 메서드를 선별적으로 정의할 수 있다
- [ ] DOM 노드를 직접 넘기지 않고 함수로 감싸서 넘기는 이유(캡슐화)를 설명할 수 있다
- [ ] validate() 반환값을 활용한 폼 검증 패턴을 구현할 수 있다
- [ ] 강제 리플로우(void offsetWidth)가 애니메이션 재시작에 필요한 이유를 이해했다
- [ ] 여러 자식의 명령적 API를 부모에서 조합하여 복합 동작을 구성할 수 있다
- [ ] 외부 라이브러리를 forwardRef + useImperativeHandle로 래핑하는 패턴을 적용할 수 있다
- [ ] 이중 ref 패턴(DOM ref + 인스턴스 ref)의 역할 분담을 이해했다
- [ ] cleanup에서 라이브러리 destroy()를 호출하여 메모리 누수를 방지하는 방법을 알고 있다
- [ ] 래퍼 컴포넌트 덕분에 라이브러리 교체 시 부모 코드를 수정할 필요가 없는 이유를 설명할 수 있다
