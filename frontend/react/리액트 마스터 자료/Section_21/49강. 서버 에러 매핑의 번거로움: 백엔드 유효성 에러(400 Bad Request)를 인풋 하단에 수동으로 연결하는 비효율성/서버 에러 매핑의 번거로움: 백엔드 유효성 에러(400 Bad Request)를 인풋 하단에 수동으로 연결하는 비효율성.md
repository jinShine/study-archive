[실습] 49강. [Pain] 서버 에러 매핑의 번거로움: 백엔드 유효성 에러(400 Bad Request)를 인풋 하단에 수동으로 연결하는 비효율성
지난 시간 우리는 다단계 폼의 데이터를 안전하게 지키는 법을 배웠습니다. 이제 버튼만 누르면 모든 데이터가 서버로 날아가 성공할 일만 남은 것 같죠?

하지만 현실은 그리 만만치 않습니다. 우리가 정성껏 만든 폼이 클라이언트 측의 유효성 검사(Zod)는 통과했을지 몰라도, 서버의 비즈니스 로직이라는 더 높은 문턱에 걸려 넘어질 때가 많기 때문입니다. 오늘은 서버로부터 "거절"당했을 때, 그 사유를 다시 화면에 보여주는 과정에서 발생하는 끔찍한 번거로움에 대해 함께 알아보겠습니다.

📖 상세 개념 가이드 1: 클라이언트 검증의 한계와 서버의 '거절'
우리가 버튼을 누르고 서버에 데이터를 전송하면, 서버는 그 데이터를 꼼꼼히 살펴본 뒤 문제가 있다면 보통 400 Bad Request 응답을 돌려줍니다. 이는 서버가 고장 난 것이 아니라 "네가 보낸 데이터 중에 이런 부분이 우리 규칙에 맞지 않아"라고 답장하는 것입니다.

🔍 왜 Zod만으로는 부족할까요?
클라이언트 검증 (Zod): "이메일 형식이 맞는가?", "비밀번호가 8자 이상인가?"와 같은 형식적 결함을 잡아냅니다.
서버 검증 (Server-side): "이미 가입된 이메일인가?", "추천인 코드가 실제로 존재하는가?"와 같은 데이터베이스 대조가 필요한 영역을 담당합니다.
이러한 에러들은 DB를 직접 뒤져봐야만 알 수 있기 때문에, 클라이언트 단에서는 아무리 완벽한 스키마를 짰어도 미리 알 수 없는 '미지의 영역'입니다.

📖 상세 개념 가이드 2: 수동 에러 매핑의 늪 (If-Else Hell)
서버는 보통 에러 내용을 아주 복잡한 객체 형태로 보내줍니다. 문제는 이 서버의 명단을 우리가 만든 리액트 화면의 각 입력창 하단에 다시 매칭시키는 과정이 매우 비효율적이라는 점입니다.

/* [Pain Point]: 우리가 지양해야 할 수동 에러 매핑의 전형적인 안 좋은 예시 */

// 서버에서 받아온 에러 객체 예시: { field: "email", message: "이미 사용 중인 이메일입니다." }
const handleServerErrors = (serverError: any) => {
  // 1. [Pain] 필드마다 일일이 조건을 걸어 에러 상태를 수동 업데이트해야 합니다.
  if (serverError.field === 'email') {
    setEmailError(serverError.message);
  } else if (serverError.field === 'password') {
    setPasswordError(serverError.message);
  } else if (serverError.field === 'address') {
    setAddressError(serverError.message);
  }
  // 2. [Risk] 필드가 50개라면? if-else 문이 50줄 이상 늘어나는 대참사가 발생합니다.
};

🔍 코드 상세 분석 및 문제점

로직의 비대화: 실제 비즈니스 로직보다 에러를 매핑하는 코드가 훨씬 더 길고 지저분해지는 기현상이 발생합니다.
중복 상태 관리: React Hook Form이 이미 errors 객체를 관리하고 있음에도 불구하고, 서버 에러를 위해 별도의 useState를 또 만들거나 수동으로 상태를 주무르는 중복 투자가 발생합니다.
📖 상세 개념 가이드 3: [Deep Dive] 비자 대사관의 비유와 UX의 붕괴
이해를 돕기 위해 현실적인 예시를 들어보겠습니다. 여러분이 비자를 발급받기 위해 대사관에 100페이지짜리 서류를 제출했다고 상상해 보세요.

💡 심사관의 결과지 비유 여러분은 모든 칸을 채웠고 날짜도 정확히 썼습니다(클라이언트 검증 통과). 그런데 심사관(서버)이 한참 보더니 "3페이지의 여권 번호가 유효하지 않고, 5페이지의 주소가 우리 기록에 없습니다"라고 적힌 결과지를 툭 건네줍니다. 여러분은 다시 그 결과지를 들고 3페이지와 5페이지를 직접 찾아가서 빨간 펜으로 에러 내용을 적어 넣어야 합니다. 페이지가 많을수록 이 '찾아가기' 과정 자체가 엄청난 중노동이 됩니다.

🔍 기술적 위험 요소: 오타와 사용자 불만
필드 이름 오타: 개발자가 수동 매핑 중 "emall"이라고 오타를 내면, 서버 에러는 분명히 발생했는데 화면에는 아무 표시도 되지 않습니다.
UX 최악의 상황: 사용자는 무엇이 틀렸는지 모르니 버튼이 고장 난 줄 알고 수십 번씩 다시 누르게 됩니다. 이는 서비스 신뢰도를 깎아먹는 치명적인 결함입니다.

오늘 우리는 서버 에러를 화면에 매핑하는 과정이 왜 그토록 번거로운지, 그리고 그 비효율성이 우리 코드의 품질을 어떻게 갉아먹는지 확인했습니다. 서버는 "어디가 틀렸다"라고 말만 해줄 뿐, 그 내용을 해당 인풋 하단에 자동으로 그려주지는 않습니다. 그 '마지막 배달'을 지금까지는 우리가 수동으로 해왔던 것이죠.
