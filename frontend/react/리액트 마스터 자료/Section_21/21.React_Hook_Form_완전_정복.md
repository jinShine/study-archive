# 21. React Hook Form 완전 정복

## 개요

이 섹션은 React 폼 관리의 근본적인 성능 문제부터 출발하여, **React Hook Form(RHF)**이라는 비제어 컴포넌트 기반 라이브러리를 통해 고성능 폼 아키텍처를 설계하는 전 과정을 다룬다. 제어 컴포넌트의 리렌더링 지옥에서 탈출하여, 타입 안전성 확보(TS 제네릭), 선언적 검증(Zod 스키마), 동적 필드 관리(useFieldArray), 서버 에러 동기화(setError), 접근성 자동화(A11y), 그리고 데이터 정제(Transformer)까지 실무에서 마주치는 폼 관련 문제를 체계적으로 해결한다.

### 핵심 철학
- **비제어 컴포넌트 전략**: 브라우저 DOM이 값을 직접 관리하고, React는 필요할 때만 관여한다.
- **이벤트 중심 폼 설계**: 상태 변경이 아닌 이벤트(제출, 블러, 변경)를 기준으로 폼 흐름을 제어한다.
- **관심사 분리**: 검증 로직(Zod 스키마)은 컴포넌트 밖으로, UI는 오직 화면 출력에만 집중한다.

## 강의별 핵심 요약

### 37강. 비제어 컴포넌트와 React Hook Form의 렌더링 차단 원리

**문제**: 제어 컴포넌트 방식에서 글자 하나를 입력할 때마다 `setState` -> 리렌더링이 발생하여 대규모 폼에서 심각한 성능 지연(Lag)이 발생한다.

**해결**: RHF는 **비제어 컴포넌트 전략**을 채택하여 `ref`로 DOM에 직접 접근하고, React의 상태 사이클을 우회한다.

- `useRef`를 이용한 수동 비제어 방식은 성능은 좋지만, 필드가 늘어날수록 관리가 불가능해진다.
- RHF는 비제어의 성능을 유지하면서 **선언적 인터페이스**를 제공하는 구세주다.
- `register` 함수가 `ref`, `onChange`, `onBlur`, `name`을 자동으로 인풋에 주입한다.
- `handleSubmit`은 브라우저 기본 동작 차단 + 유효성 검사 선행 + 통과 시에만 콜백 실행하는 고차 함수다.

```tsx
const { register, handleSubmit } = useForm<FormInputs>();
<input {...register("firstName")} />
```

### 38강. useForm과 TS 제네릭

**핵심**: 타입스크립트 제네릭으로 폼 데이터의 설계도를 정의하면, 필드 이름 오타가 **컴파일 타임에 원천 차단**된다.

- `useForm<T>()`: 제네릭 `T`를 받는 순간 내부 모든 함수(`register`, `watch` 등)가 해당 타입에 종속된다.
- `SubmitHandler<T>`: `onSubmit` 함수의 `data` 매개변수에 자동 타입 추론을 제공한다.
- **점 표기법(Dot Notation)**: `register("preferences.theme")`처럼 중첩 객체 경로도 타입 추론이 동작한다.
- `defaultValues`: 엔진에게 초기 상태를 알려주는 기준점(Anchor) 역할을 한다.

```ts
interface UserProfileForm {
  userName: string;
  preferences: { theme: 'light' | 'dark'; };
}
const { register } = useForm<UserProfileForm>({ defaultValues: { ... } });
```

### 39강. [Pain] useEffect와 정규표현식 검증 로직

**고통 포인트**: React 기본 기능만으로 유효성 검사를 구현하면 발생하는 3대 문제.

1. **상태의 시차(State Lag)**: `useEffect`는 화면을 먼저 그린 후 실행되므로, 에러 메시지가 찰나의 지연 후 나타난다.
2. **의존성 지옥(Dependency Hell)**: 비밀번호/비밀번호 확인처럼 교차 검증이 필요한 필드에서 `useEffect` 의존성 배열이 얽히고설킨다.
3. **폭포수 리렌더링**: 타이핑 -> `email` 상태 변경(1차 리렌더링) -> `useEffect` 실행 -> `emailError` 변경(2차 리렌더링). 글자 하나당 최소 2회 리렌더링.

```tsx
// 안티패턴: 필드가 100개면 200개의 상태 + 100개의 useEffect
const [email, setEmail] = useState("");
const [emailError, setEmailError] = useState("");
useEffect(() => { /* regex 검증 로직 */ }, [email]);
```

### 40강. [Solution] 선언적 검증

**해결**: `register`의 두 번째 인자에 **규칙을 선언**하면, "어떻게 검사할 것인가"가 아니라 "어떤 규칙이 있는가"에 집중할 수 있다.

- `required`: 필수값 검증 + 에러 메시지를 문자열로 직접 선언.
- `pattern`: 정규표현식을 `{ value, message }` 구조로 내부 자동 `test()` 수행.
- `minLength` / `maxLength`: 길이 제한을 속성 하나로 해결.
- `validate`: 커스텀 검증 함수. `true` 반환 시 합격, 문자열 반환 시 해당 문자열이 에러 메시지가 된다.
- `watch('password')`: 다른 필드의 실시간 값을 구독하여 교차 검증(비밀번호 확인)에 활용.
- `mode`: `"onChange"` | `"onTouched"` | `"onSubmit"` 등 검증 타이밍 전략을 설정.
- **Proxy 기술**: `errors.email`에 접근하는 컴포넌트만 해당 에러 변화 시 정밀 리렌더링.

```tsx
<input {...register("email", {
  required: "이메일은 필수입니다.",
  pattern: { value: /^...@...\..*$/, message: "형식 오류" }
})} />
{errors.email && <p>{errors.email.message}</p>}
```

### 41강. Zod + Schema 검증 설계

**핵심**: 검증 로직을 컴포넌트 **밖으로** 완전히 분리하여, 데이터 설계도 하나로 타입 정의와 검증을 동시에 끝내는 **단일 진실 공급원(Single Source of Truth)**.

- **zod**: 데이터 규정집 역할. `z.string().email()`, `z.number().min()` 등 체이닝 방식.
- **@hookform/resolvers**: RHF와 Zod 사이를 연결하는 통역사(Adapter).
- `z.infer<typeof schema>`: 스키마로부터 TypeScript 타입을 **자동 추출**. 스키마가 바뀌면 타입도 자동 변경.
- `refine`: 객체 전체를 인자로 받아 필드 간 교차 검증 수행. `path` 옵션으로 에러를 특정 필드에 귀속.
- `zodResolver(schema)`: `useForm`의 `resolver` 옵션에 장착하여 검증 엔진을 이식.

```ts
// schemas/authSchema.ts
export const signupSchema = z.object({
  email: z.string().min(1).email(),
  password: z.string().min(8).regex(/[A-Z]/),
  passwordConfirm: z.string()
}).refine(data => data.password === data.passwordConfirm, {
  message: "비밀번호 불일치", path: ["passwordConfirm"]
});
export type SignupInput = z.infer<typeof signupSchema>;

// 컴포넌트
const { register } = useForm<SignupInput>({ resolver: zodResolver(signupSchema) });
<input {...register("email")} />  // register 내부가 깨끗해짐
```

### 42강. register 함수의 내부 전파 원리

**핵심**: `...register("email")`이 반환하는 **4대 부품**의 정체와 동작 원리.

| 부품 | 역할 | 상세 |
|------|------|------|
| `name` | 이름표 | 내부 저장소의 키(Key), Zod 스키마가 데이터를 찾는 주소 |
| `ref` | 전용 통로 | DOM에 직접 빨대를 꽂아 가상 DOM 사이클을 우회 (비제어 핵심) |
| `onChange` | 수신기 | `useState`를 호출하지 않고 내부 JS 객체(Store)만 업데이트 -> 리렌더링 0회 |
| `onBlur` | 종료 감지 | 포커스 이탈 시 `touched`/`dirty` 마킹 및 검증 트리거 |

- **스프레드 연산자(`...`)**: 4대 부품이 담긴 가방을 열어 인풋 속성으로 한꺼번에 주입하는 멀티탭.
- **Proxy + Subscription**: `errors.email`을 사용하는 컴포넌트만 email 에러 변화 시 리렌더링.
- **리액트 엔진 우회**: `onChange` 시 내부 `Map` 객체만 수정하고 `setState` 호출 없음 -> 압도적 성능.

### 43강. Controller와 커스텀 UI

**문제**: Tailwind로 만든 커스텀 토글, MUI/Ant Design 컴포넌트 등은 `e.target.value` 대신 순수 값(boolean, number)을 던지므로 `register`의 표준 이벤트 형식과 규격이 맞지 않는다.

**해결**: `Controller`는 비표준 컴포넌트를 RHF 엔진의 제어권 안으로 끌어들이는 **만능 어댑터**.

- `name`: 엔진 저장소 주소.
- `control`: 엔진과 통신하는 무전기 채널.
- `render`: `field`(데이터 통로)와 `fieldState`(상태 신호등)를 제공하는 렌더 함수.
- `field.onChange`는 인자가 이벤트 객체인지 순수 값인지 **자동 판단**하여 저장소에 반영.
- **defaultValues 설정 필수**: 커스텀 컴포넌트는 초기값이 없으면 `undefined` 에러 발생.

```tsx
<Controller name="pushNotification" control={control}
  render={({ field }) => (
    <button onClick={() => field.onChange(!field.value)}>
      {field.value ? "ON" : "OFF"}
    </button>
  )}
/>
```

### 44강. [Pain] 동적 입력 필드 관리 복잡성

**고통 포인트**: `useState + map`으로 동적 배열(경력 사항 등)을 관리할 때 발생하는 3대 문제.

1. **상태 불일치(State Desync)**: `useState` 배열과 RHF 내부 저장소가 따로 노는 "두 명의 비서" 문제.
2. **인덱스 버그**: `key={index}` 사용 시 중간 항목 삭제하면 리액트가 "마지막 항목이 삭제됨"으로 오해 -> 포커스 유실, 상태 꼬임.
3. **문자열 조립의 함정**: `` `career.${index}.company` ``는 타입 추론이 깨져 오타 시 데이터 증발.

---

### 45강. [Solution] useFieldArray

**해결**: RHF 전용 배열 관제 시스템으로 **고유 ID 기반** 리스트 관리.

- `fields`: 실시간 데이터 스냅샷. 각 요소에 자동 생성된 `field.id` 포함.
- `append(obj)`: 맨 뒤 추가 + 고유 ID 자동 부여.
- `remove(index)`: 해당 항목 + 연결된 에러/dirty 상태까지 함께 청소.
- `move(from, to)`: 순서 변경. ID 유지되어 포커스 안 풀림. 드래그앤드롭에 필수.
- `insert(index, obj)`: 특정 위치에 끼워넣기.
- `prepend(obj)`: 맨 앞에 추가.
- `replace(arr)`: 전체 배열 교체.
- **`key={field.id}`**: 인덱스가 아닌 고유 ID를 키로 사용하여 리액트의 재조정(Reconciliation) 정확도 보장.
- **`as const` 단언**: 동적 경로의 타입 안전성 유지.

```tsx
const { fields, append, remove } = useFieldArray({ control, name: "careers" });
{fields.map((field, index) => (
  <div key={field.id}>
    <input {...register(`careers.${index}.company` as const)} />
    <button onClick={() => remove(index)}>삭제</button>
  </div>
))}
```

### 46강. useWatch vs getValues 성능 최적화

**핵심**: 폼 데이터를 읽는 두 가지 전략의 차이.

| 항목 | `useWatch` | `getValues` |
|------|-----------|-------------|
| 방식 | **구독(Subscription)** - 값 변경 시 자동 리렌더링 | **일회성 스냅샷** - 호출 시점의 값만 반환 |
| 리렌더링 | 감시 대상 변경 시마다 발생 | 발생하지 않음 |
| 적합한 상황 | 실시간 미리보기, 조건부 UI 표시 | 제출 직전 데이터 수집, 이벤트 핸들러 내부 |
| 성능 비용 | 상대적으로 높음 (구독 유지) | 낮음 (일회성) |

- `watch`는 `useWatch`의 간편 버전이지만 폼 전체를 구독하여 성능에 불리할 수 있다.
- 실시간 반영이 필요한 곳만 `useWatch`, 나머지는 `getValues`로 성능을 최적화한다.

### 47강. [Pain] 다단계 폼 데이터 증발

**고통 포인트**: 다단계(Multi-step) 폼에서 **페이지(스텝)를 이동하면 이전 단계의 컴포넌트가 언마운트**되면서 입력값이 사라지는 라이프사이클 이슈.

- RHF의 기본 동작(`shouldUnregister: true`)은 컴포넌트가 언마운트되면 해당 필드를 내부 저장소에서 제거한다.
- 스텝 1에서 입력한 데이터가 스텝 2로 이동하는 순간 증발하는 현상 발생.
- 조건부 렌더링(`{step === 1 && <Step1 />}`)에서도 동일한 문제.

---

### 48강. [Solution] shouldUnregister: false

**해결**: `useForm`의 `shouldUnregister: false` 옵션으로 **언마운트되어도 데이터를 보존**.

- 컴포넌트가 화면에서 사라져도 RHF 내부 저장소에 데이터가 남아 있다.
- 다단계 폼, 탭 전환, 조건부 렌더링 등 복잡한 폼의 상태 보존 핵심 기법.
- `defaultValues`와 함께 사용하면 이전 단계로 돌아갔을 때 입력값이 복원된다.

```tsx
const { register } = useForm({
  shouldUnregister: false,  // 언마운트 시에도 데이터 유지
  defaultValues: { step1Name: "", step2Email: "" }
});
```

### 49강. [Pain] 서버 에러 매핑 번거로움

**고통 포인트**: 백엔드 API가 `400 Bad Request`와 함께 필드별 유효성 에러를 반환할 때, 이를 각 인풋 하단에 **수동으로 매핑**하는 비효율.

- 서버 응답 구조(`{ errors: { email: "이미 사용 중", password: "너무 짧음" } }`)를 파싱하여 개별 `useState`에 일일이 넣어야 한다.
- RHF의 `errors` 객체와 서버 에러 상태가 이원화되어 관리가 복잡해진다.
- 서버 에러를 표시한 후 사용자가 수정하면 해당 에러를 수동으로 지워야 하는 번거로움.

### 50강. [Solution] setError 활용 전략

**해결**: `setError` API로 서버 유효성 에러를 RHF의 `errors` 객체에 **직접 주입**.

- `setError("email", { type: "server", message: "이미 사용 중" })`: 서버 에러를 클라이언트 검증 에러와 동일한 채널로 통합.
- API 응답 스키마를 순회하며 자동 매핑하는 유틸리티 패턴 구현 가능.
- `clearErrors("email")` 또는 사용자 재입력 시 자동 해제.
- `setError("root", ...)`: 특정 필드가 아닌 폼 전체 에러(로그인 실패 등) 표시.

```tsx
const onSubmit = async (data) => {
  try {
    await api.signup(data);
  } catch (err) {
    // 서버 에러를 RHF errors에 직접 주입
    Object.entries(err.fieldErrors).forEach(([field, msg]) => {
      setError(field, { type: "server", message: msg });
    });
  }
};
```

### 51강. 비동기 유효성 검사

**핵심**: 아이디 중복 확인, 이메일 존재 여부 등 **네트워크 요청이 필요한 검증**을 폼 흐름에 자연스럽게 통합.

- `validate` 옵션에 `async` 함수를 전달하여 비동기 검증 구현.
- 디바운스(Debounce)를 적용하여 매 타이핑마다 API 호출이 발생하지 않도록 최적화.
- `isValidating` 상태를 활용한 로딩 인디케이터 UX 처리.
- Zod의 `refine`에서도 비동기 검증 가능: `.refine(async (val) => await checkDuplicate(val))`.

```tsx
register("username", {
  validate: async (value) => {
    const exists = await checkUsername(value);
    return !exists || "이미 사용 중인 아이디입니다.";
  }
})
```

### 52강. [Pain] 웹 접근성(A11y) 레이블 문제

**고통 포인트**: `useId`만으로는 복잡한 에러 메시지와 인풋의 관계를 완벽하게 설정할 수 없다.

- 스크린 리더가 에러 메시지를 인풋과 연결하여 읽으려면 `aria-describedby`, `aria-invalid` 등의 ARIA 속성이 필요하다.
- 동적 에러 메시지의 ID를 일일이 생성하고 인풋에 바인딩하는 것은 반복적이고 실수하기 쉽다.
- 접근성 미준수는 법적 리스크(ADA, WCAG)와 사용자 이탈로 직결된다.

---

### 53강. [Solution] A11y 설계 자동화

**해결**: RHF의 `formState`와 `useId`를 조합하여 `aria-invalid`, `aria-describedby`를 **자동 바인딩**하는 재사용 가능한 패턴.

- 에러가 있을 때만 `aria-invalid="true"` 동적 토글.
- 에러 메시지 `<span>`에 고유 ID를 부여하고 인풋의 `aria-describedby`에 연결.
- Tailwind CSS의 `aria-invalid:` 변형자를 활용한 시각적 피드백 자동화.
- 재사용 가능한 `FormField` 래퍼 컴포넌트로 접근성 보일러플레이트 제거.

```tsx
<input
  {...register("email")}
  aria-invalid={!!errors.email}
  aria-describedby={errors.email ? `${id}-error` : undefined}
/>
{errors.email && (
  <span id={`${id}-error`} role="alert">{errors.email.message}</span>
)}
```

### 54강. dirtyFields 유틸리티 패턴

**핵심**: 수정 페이지(Edit Form)에서 **변경된 값만 골라내어** 효율적인 `PATCH` 요청을 보내는 실무 기법.

- `formState.dirtyFields`: 어떤 필드가 `defaultValues`로부터 변경되었는지 추적하는 객체.
- `getDirtyValues(dirtyFields, allValues)`: 변경된 필드만 필터링하는 유틸리티 함수 구현.
- 전체 데이터를 `PUT`으로 보내는 대신, 변경분만 `PATCH`로 보내 네트워크 효율 극대화.
- 중첩 객체와 배열에서도 재귀적으로 dirty 필드를 추출하는 고급 패턴.

```tsx
const onSubmit = (data) => {
  const changedData = getDirtyValues(formState.dirtyFields, data);
  api.patch("/profile", changedData);  // 변경된 필드만 전송
};
```

### 55강. reset vs resetField

**핵심**: 서버에서 데이터를 비동기로 가져온 후 폼의 기본값을 **안전하게 업데이트**하는 방법.

| 메서드 | 대상 | 용도 |
|--------|------|------|
| `reset(values)` | 폼 전체 | 서버 데이터 로딩 후 전체 기본값 재설정. `isDirty`, `errors` 등 모든 상태 초기화 |
| `resetField("name")` | 특정 필드 | 하나의 필드만 초기값으로 되돌리기 |

- **비동기 로딩의 함정**: `useEffect`에서 API 데이터를 가져온 뒤 `reset(serverData)`을 호출해야 `defaultValues`가 갱신된다.
- `reset` 없이 `setValue`만 사용하면 `isDirty`가 `true`로 남아 "변경사항 있음" 오판 발생.
- `reset`의 옵션: `keepDirty`, `keepErrors`, `keepValues` 등으로 세밀한 초기화 전략 수립 가능.

```tsx
useEffect(() => {
  fetchProfile().then(data => reset(data));  // 서버 데이터로 기본값 재설정
}, [reset]);
```

### 56강. 데이터 정제(Transformer)와 결산

**핵심**: 폼 데이터를 서버 API 스펙에 맞게 **직렬화(Serialize)**하는 Transformer 패턴과, 전체 섹션의 아키텍처적 결산.

- **Transformer**: `onSubmit` 내부에서 폼 데이터를 서버가 기대하는 구조로 변환하는 계층.
  - 날짜 형식 변환 (`Date` -> ISO 문자열)
  - 숫자 문자열 변환 (`"25"` -> `25`)
  - 중첩 구조 평탄화(Flatten) 또는 재구조화
- **이벤트 중심 폼 설계(Event-Driven Form Architecture)**: RHF의 핵심 가치.
  - 상태 변경이 아닌 이벤트(submit, blur, change)를 기점으로 데이터를 처리.
  - React의 렌더링 사이클과 분리된 독자적인 데이터 흐름.
- **섹션 결산**: 비제어 성능 + 선언적 DX + 타입 안전성 + 서버 통합 = 엔터프라이즈급 폼 아키텍처.

## 핵심 개념 비교 테이블

### 제어 vs 비제어 컴포넌트

| 항목 | 제어 컴포넌트 | 비제어 컴포넌트 (RHF) |
|------|-------------|---------------------|
| 값 관리 주체 | React (`useState`) | 브라우저 DOM (`ref`) |
| 타이핑 시 리렌더링 | 매 글자마다 발생 | 0회 (에러 변화 시에만) |
| 데이터 접근 | `state` 변수로 즉시 | `getValues()` 또는 제출 시 |
| 실시간 검증 | `useEffect` 필요 | `register` 옵션으로 선언 |
| 100개 필드 성능 | 치명적 지연 | 거의 영향 없음 |

### 검증 방식 비교

| 항목 | useState + useEffect | register 옵션 | Zod 스키마 |
|------|---------------------|---------------|-----------|
| 위치 | 컴포넌트 내부 | 컴포넌트 내부 (인라인) | 별도 파일 |
| 관심사 분리 | 불가 | 부분적 | 완전 분리 |
| 타입 추출 | 수동 인터페이스 | 수동 인터페이스 | `z.infer` 자동 추출 |
| 재사용성 | 없음 | 낮음 | 높음 (여러 폼 공유) |
| 교차 검증 | 의존성 지옥 | `validate + watch` | `refine + path` |

### register vs Controller

| 항목 | `register` | `Controller` |
|------|-----------|-------------|
| 대상 | 표준 HTML 엘리먼트 (`input`, `select`) | 커스텀 UI, 외부 라이브러리 컴포넌트 |
| 연결 방식 | 스프레드 (`...register("name")`) | `render` prop에서 `field` 객체 전달 |
| 이벤트 형식 | `e.target.value` (표준) | 순수 값 (boolean, number 등 자유) |
| 초기값 | 선택사항 | **필수** (`defaultValues` 미설정 시 에러) |

### useWatch vs getValues vs watch

| 항목 | `useWatch` | `getValues` | `watch` |
|------|-----------|-------------|---------|
| 동작 | 구독 (변경 시 리렌더링) | 일회성 스냅샷 | 구독 (폼 전체) |
| 리렌더링 | 감시 필드만 | 없음 | 폼 전체 |
| 사용처 | 실시간 미리보기 | 이벤트 핸들러, 제출 시 | 간단한 조건부 표시 |
| 성능 | 중간 | 최적 | 주의 필요 |

### reset vs resetField vs setValue

| 항목 | `reset(values)` | `resetField("name")` | `setValue("name", val)` |
|------|----------------|----------------------|------------------------|
| 대상 | 폼 전체 | 특정 필드 1개 | 특정 필드 1개 |
| isDirty 초기화 | O (전부 false) | O (해당 필드만) | X (dirty 유지) |
| errors 초기화 | O | O (해당 필드만) | X |
| 주 용도 | 서버 데이터 로딩 후 | 개별 필드 리셋 | 프로그래밍 방식 값 변경 |

## 면접 대비 Q&A

### Q1. React Hook Form이 높은 성능을 유지하는 원리는 무엇인가요?

**A**: RHF는 **비제어 컴포넌트 전략**을 채택합니다. `register`가 주입하는 `ref`를 통해 DOM에 직접 접근하여 값을 관리하므로, 타이핑 시 `setState`가 호출되지 않아 리렌더링이 발생하지 않습니다. 또한 **Proxy 기반 구독 모델**을 사용하여 `errors.email`처럼 특정 상태를 참조하는 컴포넌트만 해당 상태 변화 시 정밀 리렌더링합니다. 내부적으로 JavaScript 객체(Map/Store)만 업데이트하고 React의 Render -> Commit 사이클을 우회하는 것이 핵심입니다.

### Q2. register와 Controller의 차이점과 각각의 사용 시점을 설명해주세요.

**A**: `register`는 표준 HTML 엘리먼트(`input`, `select`, `textarea`)에 `ref`, `onChange`, `onBlur`, `name`을 스프레드 연산자로 자동 주입합니다. `e.target.value`라는 정형화된 이벤트 객체를 기대하므로, 이를 지원하는 네이티브 태그에 적합합니다. 반면 `Controller`는 커스텀 UI 컴포넌트(토글 스위치, MUI DatePicker 등)처럼 표준 이벤트 형식을 따르지 않는 컴포넌트를 RHF에 연결하는 어댑터입니다. `render` prop의 `field.onChange`에 순수 값을 직접 전달할 수 있어 어떤 형태의 컴포넌트든 제어할 수 있습니다.

### Q3. Zod와 RHF를 함께 사용할 때의 이점은 무엇인가요?

**A**: 세 가지 핵심 이점이 있습니다. 첫째, **단일 진실 공급원(SSOT)**: `z.infer`로 스키마에서 타입을 자동 추출하므로 타입 정의와 검증 규칙이 항상 일치합니다. 둘째, **관심사 분리**: 검증 로직이 컴포넌트 밖의 별도 파일로 분리되어 컴포넌트는 UI 렌더링에만 집중합니다. `register` 내부가 깨끗해집니다. 셋째, **재사용성**: 동일한 스키마를 여러 폼에서 공유하거나 서버 사이드에서도 재사용할 수 있습니다.

### Q4. 동적 배열 폼에서 key={index} 대신 field.id를 써야 하는 이유는?

**A**: 리액트의 재조정(Reconciliation) 알고리즘은 `key`를 보고 요소의 정체성을 판단합니다. 중간 항목을 삭제하면 인덱스가 재정렬되기 때문에, `key={index}` 사용 시 리액트가 "마지막 항목이 삭제됨"으로 오해하여 엉뚱한 DOM을 제거하거나 상태를 꼬이게 만듭니다. `useFieldArray`가 제공하는 `field.id`는 항목 생성 시 발급되는 고유한 식별자로, 배열 순서가 바뀌어도 해당 데이터를 끝까지 추적합니다. 이를 통해 삭제/이동 후에도 포커스 유지와 데이터 일관성이 보장됩니다.

### Q5. 다단계 폼에서 데이터 증발을 방지하는 방법은?

**A**: `useForm`의 `shouldUnregister: false` 옵션을 설정합니다. RHF의 기본 동작은 컴포넌트가 언마운트되면 해당 필드를 내부 저장소에서 제거하는 것인데, 이 옵션을 `false`로 두면 컴포넌트가 화면에서 사라져도 데이터가 저장소에 보존됩니다. 스텝 전환이나 탭 이동, 조건부 렌더링 등에서 필수적인 설정입니다.

### Q6. 서버에서 반환한 유효성 에러를 폼에 표시하는 가장 좋은 방법은?

**A**: `setError` API를 사용합니다. `setError("email", { type: "server", message: "이미 사용 중" })`처럼 호출하면 서버 에러가 RHF의 `errors` 객체에 직접 주입되어, 클라이언트 검증 에러와 동일한 채널로 인풋 하단에 표시됩니다. API 응답의 필드별 에러를 순회하며 자동 매핑하는 유틸리티를 만들면 보일러플레이트를 제거할 수 있습니다. 폼 전체 에러(로그인 실패 등)는 `setError("root", ...)`로 처리합니다.

### Q7. useWatch와 getValues의 차이를 성능 관점에서 설명해주세요.

**A**: `useWatch`는 **구독(Subscription)** 방식으로, 감시 대상 필드의 값이 변할 때마다 해당 컴포넌트를 리렌더링합니다. 실시간 미리보기나 조건부 UI 표시에 적합하지만 성능 비용이 있습니다. `getValues`는 **일회성 스냅샷**으로, 호출 시점의 값만 반환하고 리렌더링을 유발하지 않습니다. 이벤트 핸들러 내부나 제출 직전 데이터 수집에 적합합니다. 실시간 반영이 필요한 곳만 `useWatch`를 쓰고 나머지는 `getValues`를 사용하는 것이 성능 최적화의 핵심입니다.

### Q8. dirtyFields를 활용한 효율적인 PATCH 요청 패턴을 설명해주세요.

**A**: `formState.dirtyFields`는 `defaultValues`로부터 변경된 필드를 `{ fieldName: true }` 형태로 추적하는 객체입니다. 이를 활용해 `getDirtyValues(dirtyFields, allValues)` 유틸리티를 만들면, 전체 폼 데이터에서 변경된 필드만 필터링하여 `PATCH` 요청으로 보낼 수 있습니다. 서버에 불필요한 데이터를 보내지 않아 네트워크 효율이 극대화되고, 수정 충돌(Conflict) 가능성도 줄어듭니다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **비제어 컴포넌트 (Uncontrolled Component)** | React state가 아닌 DOM 자체가 입력값을 관리하는 방식. `ref`로 필요 시 값을 읽음 |
| **제어 컴포넌트 (Controlled Component)** | `value`와 `onChange`로 React state가 입력값을 완전히 관리하는 방식 |
| **register** | 인풋 엘리먼트를 RHF 엔진에 등록하고 `name`, `ref`, `onChange`, `onBlur`를 자동 주입하는 함수 |
| **Controller** | 표준 HTML 이벤트를 지원하지 않는 커스텀 UI를 RHF에 연결하는 어댑터 컴포넌트 |
| **handleSubmit** | 브라우저 기본 동작 차단 + 유효성 검사 선행 + 통과 시에만 콜백을 실행하는 고차 함수 |
| **useFieldArray** | 동적 배열 필드를 고유 ID 기반으로 관리하는 RHF 전용 훅 |
| **Zod** | TypeScript-first 스키마 검증 라이브러리. 데이터 모양 정의와 검증을 동시에 수행 |
| **zodResolver** | Zod 스키마를 RHF의 `resolver` 옵션에 장착하는 @hookform/resolvers의 어댑터 함수 |
| **z.infer** | Zod 스키마로부터 TypeScript 타입을 자동 추출하는 유틸리티 타입 |
| **refine** | Zod에서 필드 간 교차 검증(비밀번호 확인 등)을 수행하는 메서드. `path`로 에러 귀속 필드 지정 |
| **shouldUnregister** | `false`로 설정 시 컴포넌트 언마운트 후에도 데이터를 내부 저장소에 보존하는 옵션 |
| **setError** | 서버 유효성 에러 등을 RHF의 `errors` 객체에 프로그래밍 방식으로 직접 주입하는 메서드 |
| **clearErrors** | 특정 필드 또는 전체 에러를 프로그래밍 방식으로 제거하는 메서드 |
| **useWatch** | 특정 필드 값의 변화를 구독하여 리렌더링을 트리거하는 훅 (실시간 반영) |
| **getValues** | 호출 시점의 폼 데이터를 일회성 스냅샷으로 반환. 리렌더링 없음 |
| **dirtyFields** | `defaultValues`로부터 변경된 필드를 추적하는 `formState` 속성 |
| **reset** | 폼 전체의 값과 상태(isDirty, errors 등)를 초기화하는 메서드. 서버 데이터 로딩 후 필수 |
| **resetField** | 특정 필드 하나만 초기값으로 되돌리는 메서드 |
| **Transformer** | `onSubmit` 내부에서 폼 데이터를 서버 API 스펙에 맞게 직렬화하는 변환 계층 |
| **Proxy** | RHF가 `formState` 접근을 감지하여 해당 상태가 변할 때만 정밀 리렌더링하는 JavaScript 기술 |
| **mode** | 검증 타이밍 전략. `onChange`(실시간), `onTouched`(포커스 이탈 후), `onSubmit`(제출 시) 등 |
| **SSOT (Single Source of Truth)** | 데이터의 정의가 단 하나의 출처에만 존재하는 설계 원칙. Zod 스키마가 타입과 검증의 SSOT |
| **A11y (Accessibility)** | 웹 접근성. `aria-invalid`, `aria-describedby` 등 ARIA 속성으로 스크린 리더 지원 |

## 자가 점검 체크리스트

### 기초 개념
- [ ] 제어 컴포넌트와 비제어 컴포넌트의 차이를 성능 관점에서 설명할 수 있다.
- [ ] RHF가 `ref`를 통해 React 렌더링 사이클을 우회하는 원리를 이해한다.
- [ ] `register` 함수가 반환하는 4대 부품(`name`, `ref`, `onChange`, `onBlur`)의 역할을 안다.
- [ ] `handleSubmit`이 고차 함수인 이유와 동작 순서(가로채기 -> 검문 -> 승인)를 설명할 수 있다.

### 타입 안전성
- [ ] `useForm<T>()`에 제네릭을 주입하여 필드 이름 오타를 컴파일 타임에 차단할 수 있다.
- [ ] `SubmitHandler<T>` 타입으로 `onSubmit` 함수의 데이터 타입을 자동 추론할 수 있다.
- [ ] 중첩 객체 경로(`preferences.theme`)에서도 타입 추론이 동작하는 점 표기법을 이해한다.

### 검증 시스템
- [ ] `required`, `pattern`, `minLength`, `validate` 등 선언적 검증 옵션을 사용할 수 있다.
- [ ] `validate`에서 `watch`를 활용한 교차 검증(비밀번호 확인)을 구현할 수 있다.
- [ ] Zod 스키마를 설계하고 `zodResolver`로 RHF에 연결할 수 있다.
- [ ] `z.infer`로 스키마에서 타입을 자동 추출하는 SSOT 패턴을 이해한다.
- [ ] `refine`과 `path`를 사용한 필드 간 교차 검증을 구현할 수 있다.

### 고급 컴포넌트 제어
- [ ] `Controller`가 필요한 상황(커스텀 UI, 외부 라이브러리)을 판단할 수 있다.
- [ ] `Controller`의 `render` prop에서 `field` 객체를 커스텀 컴포넌트에 바인딩할 수 있다.
- [ ] `useFieldArray`로 동적 배열 폼의 추가/삭제/순서변경을 구현할 수 있다.
- [ ] `key={field.id}`를 사용해야 하는 이유와 `key={index}`의 위험성을 설명할 수 있다.

### 성능 최적화
- [ ] `useWatch`(구독)와 `getValues`(일회성)의 차이를 성능 관점에서 설명할 수 있다.
- [ ] 실시간 반영이 필요한 곳만 `useWatch`, 나머지는 `getValues`로 최적화하는 전략을 안다.
- [ ] RHF의 Proxy 기반 구독 모델이 정밀 리렌더링을 가능하게 하는 원리를 이해한다.

### 실무 패턴

- [ ] `reset`과 `resetField`의 차이를 이해하고, 서버 데이터 로딩 후 적절히 사용할 수 있다.
