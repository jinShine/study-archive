[실습] 50강. [Solution] setError 활용 전략: 서버 유효성 에러의 실시간 동기화 및 API 응답 스키마 자동 매핑
지난 시간 우리는 서버가 보내준 거절 사유를 다시 화면의 입력창 하단에 매핑하는 과정이 얼마나 고통스럽고 비효율적인지 함께 나누었습니다. 심사관의 지적 사항을 들고 100페이지가 넘는 서류를 일일이 찾아다니며 빨간 펜으로 에러를 적어 넣는 그 중노동은 개발자의 생산성뿐만 아니라 코드의 품질까지 갉아먹는 주범이었죠.

오늘은 이 번거로운 수동 작업을 끝내고, 서버의 목소리를 폼 엔진에 즉시 동기화시키는 마법 같은 해결책인 setError 활용 전략에 대해 알아보겠습니다.

📖 상세 개념 가이드 1: setError — 폼의 자동 스탬프 기계
우리가 해결해야 할 핵심 과제는 서버에서 온 에러 명단을 React Hook Form(RHF) 엔진에게 통째로 던져주어, 엔진이 알아서 각 이름표를 찾아가 "여기가 틀렸대!"라고 뿌려주게 만드는 것입니다. 여기서 주인공인 setError는 우리가 수동으로 폼의 에러 상태를 조작할 수 있게 해주는 정교한 도구입니다.

💡 시니어의 비유: 자동 스탬프 기계

지난 시간의 비자 발급 예시로 돌아가 본다면, setError는 일종의 '자동 스탬프 기계'와 같습니다. 심사관(서버)이 넘겨준 에러 리스트를 이 기계에 넣기만 하면, 기계가 빛의 속도로 서류를 넘기며 정확한 위치에 "여권 번호 오류", "주소 불일치" 같은 도장을 쾅쾅 찍어주는 것이죠.

💻 기초 구문: setError 신택스 이해하기
처음 이 라이브러리를 접하시는 분들을 위해 가장 기본적인 사용법부터 뜯어보겠습니다.

/* [Basic Syntax]: 특정 필드에 강제로 에러 주입하기 */
setError("email", {
  type: "manual", // 에러의 성격 (사용자 정의 에러)
  message: "이미 사용 중인 이메일입니다." // 화면에 보여줄 메시지
});

🔍 구문 상세 설명 및 배경
첫 번째 인자 (name): 에러를 표시할 필드의 이름입니다. (예: "email", "password") RHF는 이 이름을 보고 어느 input 하단에 메시지를 띄울지 결정합니다.
두 번째 인자 (object): 에러의 상세 정보를 담은 객체입니다.
type: 에러의 종류를 정의합니다. 클라이언트 규칙(Required 등) 위반이 아닌 경우 보통 "manual"이나 "server"를 사용합니다.
message: errors.email.message를 통해 화면에 출력될 실제 텍스트입니다.
배경: 브라우저가 알 수 없는 '중복 아이디' 같은 에러는 서버 응답을 받은 후 개발자가 수동(Manual)으로 엔진에 알려줘야 하기에 이 함수가 필요합니다.
📖 상세 개념 가이드 2: 서버 에러 자동 매핑 패턴 (Object.entries)
기술적으로 가장 우아한 방식은 서버로부터 받은 에러 객체를 반복문으로 돌리며 setError를 호출하는 것입니다. 대부분의 현대적인 API는 에러가 발생한 필드 이름과 메시지를 쌍으로 묶어서 보내주는데, 이를 이용해 자동화 라인을 구축할 수 있습니다.

💻 실전 코드: API 응답 동기화 로직
/* [Solution]: 서버 에러 자동 매핑 패턴 */
const onSubmit = async (data: LoginFormInputs) => {
  try {
    // 1. 서버에 데이터를 전송합니다.
    await loginApi(data);
    alert("로그인 성공!");
  } catch (error: any) {
    // 2. 서버가 400 에러와 함께 상세 에러 명단(errors)을 보내왔는지 확인합니다.
    // 예: { email: "이미 가입됨", password: "보안 취약" }
    const serverErrors = error.response?.data?.errors;

    if (serverErrors) {
      // 3. Object.entries를 통해 [필드명, 메시지] 쌍을 추출하여 순회합니다.
      Object.entries(serverErrors).forEach(([key, message]) => {
        // 4. setError를 호출하여 각 필드에 자동 도장을 찍습니다.
        setError(key as keyof LoginFormInputs, {
          type: "server", // 서버로부터 온 에러임을 명시
          message: message as string
        });
      });
    }
  }
};

🔍 기술적 디테일 분석
Object.entries(): 객체를 [key, value] 형태의 배열로 변환해 줍니다. "어느 칸이 틀렸는지(Key)"와 "왜 틀렸는지(Value)"를 한 번에 추출하여 순회하기에 최적입니다.
as keyof LoginFormInputs: 타입스크립트에게 "서버가 말한 이 이름이 우리 폼 설계도에 있는 칸이 맞다"라고 확신을 주는 장치입니다. 만약 서버가 설계도에 없는 엉뚱한 키를 보내면 런타임 에러를 방지할 수 있습니다.
type: "server": 이 에러는 클라이언트 검사를 통과했지만, 서버의 비즈니스 검증에서 탈락했다는 '꼬리표'입니다. 나중에 스타일을 다르게 주거나 로깅할 때 유용합니다.
📖 상세 개념 가이드 3: 단일 진실 공급원 (Single Source of Truth)
이 전략의 진정한 가치는 유지보수의 효율성에 있습니다. 에러 메시지의 구체적인 내용은 서버가 결정하고, 프론트엔드는 그 메시지를 전달받아 '뿌려주는 역할'에만 집중합니다.

/* [Architect Tip]: UI는 메시지를 '선택'하지 않고 '노출'만 합니다. */
// 서버가 "이미 가입된 메일입니다"를 "탈퇴한 계정입니다"로 바꿔도
// 프론트엔드 코드는 수정할 필요가 없습니다.
<span>{errors.email?.message}</span>
📖 상세 개념 가이드 4: 지능적인 에러 해제 (Smart Clearing)
setError로 주입된 에러는 단순히 문구만 띄우는 게 아닙니다. 사용자가 에러가 난 칸을 다시 클릭해 내용을 수정하기 시작하면, 엔진은 "아, 사용자가 고치고 있구나"라고 인지하여 해당 에러를 자동으로 해제하게 설정할 수 있습니다.

/* [UX Optimization]: 수정 시 에러 자동 제거 */
const { setError, clearErrors } = useForm({
  // reValidateMode를 onChange로 두면 사용자가 타이핑을 시작할 때
  // 기존 서버 에러가 부드럽게 사라집니다.
  reValidateMode: "onChange"
});
