[실습] 44강. [Pain] 동적 입력 필드 관리의 복잡성: 배열 형태의 데이터 관리 시 발생하는 인덱스 버그와 키 관리
우리는 지난 시간까지 register와 Controller를 통해 단일 필드부터 화려한 커스텀 UI까지 React Hook Form(RHF)의 체계 안으로 완벽하게 편입시키는 방법을 배웠습니다. 이제 웬만한 정적 폼은 여러분의 손끝에서 자유자재로 요리될 수 있는 수준에 도달했죠.

하지만 실무에서 우리를 기다리고 있는 진짜 난관은 따로 있습니다. 바로 데이터의 개수가 정해져 있지 않고 사용자가 버튼을 누를 때마다 늘어나거나 줄어드는 동적 배열 데이터입니다. 예를 들어 사용자의 경력 사항(Career History)이나 자격증 목록을 입력받는 폼을 생각해 보세요. 오늘은 이 동적 리스트를 리액트의 기본 방식(useState + map)으로 관리할 때 발생하는 인덱스 버그와 키 관리의 늪을 파헤쳐 보겠습니다.

📖 상세 개념 가이드 1: 동적 리스트 관리의 본질적 어려움
동적 배열 데이터는 단순히 필드가 늘어나는 것이 아닙니다. 각 항목(Item) 안에 회사명, 직무, 기간 등 여러 데이터가 하나의 객체 묶음으로 움직여야 합니다. 이를 리액트의 기본 기능인 useState로 관리하려고 시도하는 순간, 우리는 데이터 이중 관리라는 벽에 부딪힙니다.

/* [Concept]: 동적 리스트의 데이터 구조 */
// 우리가 관리해야 할 데이터의 모양 (배열 내 객체)
const data = {
  career: [
    { id: 1, company: "A사", role: "개발자" },
    { id: 2, company: "B사", role: "아키텍트" }
  ]
};

1. 상태 불일치 (State Desync)
React Hook Form은 성능을 위해 비제어 방식(Uncontrolled)을 지향합니다. 그런데 리스트를 그리기 위해 useState 배열을 별도로 운영하면, RHF 엔진 내부의 데이터와 우리가 만든 배열 상태가 서로 따로 노는 현상이 발생합니다.

2. 두 명의 비서 비유
이는 마치 두 명의 비서가 서로 다른 장부를 들고 일하는 것과 같습니다.

비서 A (useState): "사용자가 두 번째 경력을 지웠으니 배열 길이를 2로 줄여야지."
비서 B (RHF 엔진): "내 장부에는 여전히 3개의 데이터가 남아있는데? 인덱스 1번 자리에 2번 내용을 덮어써야 하나?"
📖 상세 개념 가이드 2: 인덱스 버그와 리액트의 오해
동적 리스트에서 가장 흔하게 발생하는 버그는 삭제 로직에서 나타납니다. 우리가 흔히 사용하는 key={index} 방식이 리액트의 가상 DOM 알고리즘을 어떻게 기만하는지 이해하는 것이 중요합니다.

🔍 기술적 배경: 왜 index를 key로 쓰면 안 되는가?
/* [Bug Case]: key={index} 사용 시 발생하는 논리적 오류 */

// 1. 초기 상태
// Index 0: 경력 A (Value: "A사")
// Index 1: 경력 B (Value: "B사") <- 삭제 대상
// Index 2: 경력 C (Value: "C사")

// 2. Index 1 삭제 후 리액트의 판단
// Index 0: 경력 A (기존과 동일하네? 유지!)
// Index 1: 경력 C (어? 원래 1번 자리에 있던 놈이 바뀌었네? 근데 Key가 1이니까 너는 B가 변한 거니?)
// 리액트: "2번 인덱스가 사라졌네! 2번 DOM을 제거하자."
리액트는 key를 보고 요소의 정체성을 파악합니다. 중간 항목을 지워도 인덱스는 다시 0, 1, 2 순서로 정렬되기 때문에, 리액트는 가장 마지막 항목이 지워졌다고 오해하고 엉뚱한 DOM 요소를 파괴하거나 상태를 꼬이게 만듭니다.

💻 실습: 고통의 시뮬레이션 (MessyDynamicForm.tsx)
실제로 수동으로 동적 배열을 관리할 때 코드가 얼마나 지저분해지고 취약해지는지 직접 목격해 보겠습니다.

/* [File Path]: src/components/MessyDynamicForm.tsx */
import React, { useState } from 'react';
import { useForm } from 'react-hook-form';

export default function MessyDynamicForm() {
  const { register, handleSubmit } = useForm();

  // 1. [Pain] RHF 엔진과 별개로 돌아가는 수동 상태 배열
  // 이 배열이 변할 때마다 컴포넌트 전체가 다시 그려지는 비효율이 발생합니다.
  const [items, setItems] = useState([{ id: Date.now() }]);

  const addItem = () => setItems([...items, { id: Date.now() }]);

  const removeItem = (index: number) => {
    // 2. [Bug] 리액트 상태(화면)는 필터링되지만,
    // RHF 엔진 내부의 실제 데이터 인덱스와는 자동으로 동기화되지 않습니다.
    const newItems = items.filter((_, i) => i !== index);
    setItems(newItems);
  };

  return (
    <form onSubmit={handleSubmit(data => console.log("최종 제출:", data))} style={{ padding: '20px' }}>
      <h2>경력 사항 입력 (Manual Management)</h2>

      {items.map((item, index) => (
        <div key={index} className="flex gap-2 mb-2" style={{ borderBottom: '1px solid #eee', paddingBottom: '10px' }}>

          {/* 3. [Vulnerability] 문자열 조립 방식의 등록
              타입스크립트의 보호를 전혀 받을 수 없으며, 오타 시 데이터가 증발합니다. */}
          <input
            {...register(`career.${index}.company`)}
            placeholder="회사명"
            style={{ border: '1px solid #ccc', padding: '5px' }}
          />

          <button type="button" onClick={() => removeItem(index)} style={{ color: 'red' }}>
            삭제
          </button>
        </div>
      ))}

      <div style={{ marginTop: '10px' }}>
        <button type="button" onClick={addItem} style={{ padding: '5px 10px', background: '#007aff', color: '#fff' }}>
          경력 추가
        </button>
        <button type="submit" style={{ marginLeft: '10px', padding: '5px 10px', background: '#000', color: '#fff' }}>
          데이터 확인
        </button>
      </div>
    </form>
  );
}
🔍 상세 기술 분석: 왜 이 코드는 실패하는가?
문자열 조립의 함정 (career.${index}.company): 우리는 지난 시간 타입스크립트 제네릭으로 오타를 막는 법을 배웠습니다. 하지만 저렇게 백틱(``)으로 경로를 조립하는 순간 타입 추론은 깨집니다. 필드가 복잡해질수록(예: projects.0.tasks.1.title) 오타 하나로 데이터가 증발할 위험이 O(n)으로 커집니다.
포커스 유실 (Focus Lost): 항목을 삭제할 때 인덱스가 앞당겨지면 리액트는 기존 엘리먼트를 재사용하려다 실패하여 포커스를 잃어버리거나, 엉뚱한 인풋에 포커스를 두는 현상이 발생합니다.
성능 저하: 리스트가 50개일 때, 1번 인풋에서 글자 하나만 쳐도 useState 때문에 50개의 인풋이 모두 리렌더링됩니다. 이는 사용자가 느끼는 타이핑 지연(Jank)의 주범입니다.
아키텍트로서 우리는 이런 불안정한 구조를 용납해서는 안 됩니다. 배열의 추가와 삭제를 마치 단일 필드를 다루듯 쉽고 안전하게 처리해 줄 전용 도구가 필요합니다. 인덱스 계산이나 키 관리 고민을 엔진에게 맡기고, 우리는 데이터의 모양에만 집중해야 합니다.
