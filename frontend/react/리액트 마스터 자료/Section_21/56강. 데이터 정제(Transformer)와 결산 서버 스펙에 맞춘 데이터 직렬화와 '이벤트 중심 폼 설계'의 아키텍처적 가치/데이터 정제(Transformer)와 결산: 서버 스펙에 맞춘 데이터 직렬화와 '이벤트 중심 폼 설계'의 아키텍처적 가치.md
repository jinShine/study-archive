[실습] 56강. 데이터 정제(Transformer)와 결산: 서버 스펙에 맞춘 데이터 직렬화와 '이벤트 중심 폼 설계'의 아키텍처적 가치
지난 시간 우리는 다단계 폼의 데이터를 안전하게 지키는 법을 배웠습니다. 이제 버튼만 누르면 성공할 것 같지만, 실무는 그리 만만치 않죠. 클라이언트 검증은 통과했을지 몰라도, 서버의 비즈니스 로직이라는 더 높은 문턱이 우리를 기다리고 있기 때문입니다.

오늘은 서버의 거절 사유를 UI에 자동으로 매핑하는 법과, 서버 스펙에 맞춰 데이터를 예쁘게 깎는 기술을 총정리해 보겠습니다.

📖 1. 서버 에러 매핑 — "말 없는 심사관"에게 목소리 입히기
우리가 정성껏 보낸 데이터가 서버로부터 400 Bad Request를 받았을 때, 그 거절 사유를 다시 화면의 입력창 하단에 매핑하는 과정은 프론트엔드 개발자의 숙명입니다.

💡 시니어의 비유: 자동 스탬프 기계

이를 수동으로 if문을 써서 매핑하는 것은 마치 100페이지 서류의 오타를 빨간 펜으로 일일이 찾아 적는 중노동과 같습니다. 우리는 이 과정을 setError라는 '자동 스탬프 기계'를 통해 자동화할 것입니다. 심사관(서버)이 넘겨준 에러 리스트를 기계에 넣기만 하면, 정확한 위치에 도장을 쾅쾅 찍어주는 것이죠.

💻 실전 코드: 서버 에러 자동 동기화 패턴
/* [Solution]: 서버 에러를 폼 엔진에 즉시 동기화하는 로직 */
const onSubmit = async (data: LoginFormInputs) => {
  try {
    // 1. 비동기 통신을 통해 서버에 로그인을 시도합니다.
    await loginApi(data);
    alert("로그인 성공!");
  } catch (error: any) {
    // 2. 서버가 400 에러와 함께 상세 에러 명단(errors)을 보내왔는지 확인합니다.
    const serverErrors = error.response?.data?.errors;

    if (serverErrors) {
      // 3. Object.entries를 통해 [키, 값] 쌍을 순회하며 꺼냅니다.
      Object.entries(serverErrors).forEach(([key, message]) => {
        // 4. [핵심] setError를 호출하여 엔진에게 에러를 강제 주입합니다.
        setError(key as keyof LoginFormInputs, {
          type: "server", // 에러 출처가 서버임을 명시
          message: message as string // 서버가 보내준 구체적인 사유
        });
      });
    }
  }
};

🔍 상세 코드 분석 및 디테일 해설
Object.entries(): 서버가 보낸 { email: "중복됨" } 객체를 ["email", "중복됨"] 형태의 배열로 변환합니다. 이를 통해 수십 개의 에러도 단 몇 줄의 반복문으로 처리할 수 있습니다.
as keyof LoginFormInputs: 타입스크립트에게 "서버가 말한 필드 이름이 우리 설계도에 있는 칸이 맞다"라고 확신시켜주는 안전장치입니다. 오타로 인한 런타임 에러를 방지합니다.
type: "server": 클라이언트의 형식 검사와 서버의 비즈니스 검증 결과를 구분하는 꼬리표입니다. 이 설정 덕분에 사용자가 입력을 수정하기 시작하면 엔진이 이 서버 에러를 지능적으로 제거(Clear)할 수 있습니다.
📖 2. 데이터 정제(Transformer) — 서버 규격으로 '환전'하기
데이터 정제의 필요성은 '환전'에 비유할 수 있습니다. 우리 주머니에는 '한국 돈(프론트엔드 데이터)'이 있지만, 방문할 국가인 '서버'에서는 그들의 화폐인 '달러(서버 규격)'만 받는 상황입니다.

💻 실전 코드: 서버 스펙 맞춤형 데이터 변환기
/* [Implementation]: 서버로 전송하기 직전의 데이터 가공(Serialization) */
const onSubmit = async (formData: RawInputs) => {
  // 1. 스프레드 연산자(...)를 사용해 원본 데이터를 안전하게 복사합니다.
  // 원본(formData)을 훼손하지 않는 '불변성' 유지는 아키텍처의 기본입니다.
  const refinedData = {
    ...formData,

    // 2. [Type Conversion]: 문자열 입력을 실제 숫자로 강제 변환합니다.
    // 인풋 값은 기본적으로 문자열이므로, 서버 DB 타입을 맞추기 위해 정제합니다.
    price: Number(formData.price),

    // 3. [Structure Compression]: 배열 형식을 문자열로 압축합니다.
    // ["React", "TypeScript"] -> "React,TypeScript" (서버 저장 스펙 최적화)
    tags: formData.tags.join(","),

    // 4. [Standardization]: 날짜를 서버 표준 규격(ISO 8601)으로 생성합니다.
    // 시차 문제를 방지하기 위해 전 세계 공통 규격인 0000-00-00T00:00:00Z 형식을 따릅니다.
    updatedAt: new Date().toISOString()
  };

  try {
    // 5. 모든 가공이 끝난 깨끗한 refinedData만 API로 전송합니다.
    await submitApi(refinedData);
    alert("데이터가 서버 규격에 맞춰 성공적으로 전송되었습니다.");
  } catch (error) {
    console.error("전송 중 오류 발생:", error);
  }
};

🔍 상세 코드 분석 및 디테일 해설
데이터 분리: refinedData를 생성하는 과정은 UI 전용 데이터와 API 전용 데이터를 분리하는 중요한 관문입니다. 폼 안에서는 자유롭게 다루되, 밖으로 나갈 때는 엄격한 규격을 갖춥니다.
불변성 유지: 원본 formData를 직접 수정하지 않고 새로운 객체를 만듦으로써, 전송 도중에 발생할 수 있는 부수 효과(Side Effect)를 차단합니다.
무결성 보장: 이 과정을 통해 서버는 별도의 가공 없이 데이터를 즉시 저장할 수 있게 되며, 시스템 전체의 데이터 안정성이 확보됩니다.
🏛️ 3. 총결산: '이벤트 중심 폼 설계'의 아키텍처적 가치
지난 42강부터 56강까지 우리는 매 순간 모든 상태를 쥐고 흔드는 방식이 아니라, 특정 이벤트가 발생했을 때만 폼 엔진이 기민하게 반응하도록 설계했습니다.

스마트 시스템: 집안의 모든 전등을 수동으로 끄는 방식에서, 사람이 움직일 때만 센서가 작동하는 스마트 홈 시스템으로 진화한 것과 같습니다.
제로-렉(Zero-Lag): 수백 개의 필드가 있는 복잡한 화면에서도 쾌적한 환경을 제공할 수 있었던 비결은 바로 비제어 컴포넌트(Uncontrolled) 기반의 이벤트 처리였습니다.
마스터의 실력: React Hook Form을 마스터한다는 것은 브라우저 자원을 효율적으로 사용하고, 서버 통신을 최적화하며, 모든 사용자를 배려하는 고도의 소프트웨어 아키텍처를 구축하는 능력을 갖추었음을 의미합니다.
🛠️ 실무 마무리를 위한 마스터 생성 스크립트
🧪 RHF 아키텍처 최종 테스트 체크리스트
1. 렌더링 고립 (Performance)
확인 방법: React DevTools의 Profiler 혹은 각 컴포넌트 내 RenderCounter 확인.
체크포인트: 특정 인풋에 타이핑할 때 부모 컴포넌트가 리렌더링되지 않고, 해당 필드만 정밀하게 업데이트되는지 확인하세요. (O(1) 성능 증명)
2. 데이터 정제 결과 (Integrity)
확인 방법: 제출(onSubmit) 시 콘솔에 찍히는 최종 JSON 객체 확인.
체크포인트:Transformer를 거친 후 문자열이 숫자로(Number), 배열이 문자열로(join) 서버 규격에 맞게 완벽히 직렬화(Serialization)되었는지 확인하세요.
3. 서버 에러 동기화 (Server Sync)
확인 방법: 서버 에러(400 Bad Request) 시나리오 강제 발생.
체크포인트: 백엔드에서 온 에러 메시지가 setError를 통해 해당 인풋 바로 아래에 정확히 매핑되는지, 그리고 사용자가 수정을 시작하면 그 메시지가 즉시 사라지는지 확인하세요.
4. 생명주기 및 영속성 (Lifecycle)
확인 방법: 다단계 폼 이동 혹은 비동기 데이터 로드 테스트.
체크포인트: 페이지를 넘겼다 돌아와도 값이 유지되는지(shouldUnregister: false), 그리고 서버 데이터를 불러온 직후 isDirty 상태가 False로 깔끔하게 초기화되는지 확인하세요.
5. 웹 접근성 연결 (A11y)
확인 방법: 브라우저 '검사(Inspect)' 도구의 Accessibility 탭 확인.
체크포인트: 인풋의 aria-describedby가 에러 메시지의 ID를 정확히 가리키고 있는지, 에러 발생 시 aria-invalid가 true로 변하는지 확인하세요.
