[실습] 42강. register 함수의 마법: onChange, onBlur, ref 노가다에서 해방되는 내부 전파 원리
우리는 지금까지 성능 최적화부터 타입 안전성 확보, 그리고 Zod를 활용한 비즈니스 로직의 완벽한 분리까지 아주 숨 가쁘게 달려왔습니다.

이제 우리 폼은 튼튼한 설계도와 강력한 엔진을 모두 갖춘 셈이지만, 여기서 한 가지 근본적인 의문이 생길 수 있습니다. 우리가 그동안 무심코 사용하던 ...register("email")이라는 짧은 코드 한 줄이 어떻게 그 복잡했던 value 설정과 onChange 이벤트 핸들러, 그리고 수동으로 관리하던 ref까지 한 번에 대신할 수 있는 것일까요? 오늘은 프론트엔드 개발자를 단순 반복적인 연결 작업에서 영원히 해방시킨 register 함수의 내부 전파 원리를 낱낱이 파헤쳐 보겠습니다.

📖 상세 개념 가이드 1: 제어 컴포넌트의 한계와 비서 시스템
우리가 React Hook Form(RHF)을 사용하기 전인 제어 컴포넌트(Controlled Components) 방식으로 폼을 만들 때는 모든 입력창마다 '전담 비서'를 한 명씩 붙여주는 것과 같았습니다.

💻 배경 지식: 수동 배선 방식 (Manual Wiring)
제어 컴포넌트 방식에서는 개발자가 데이터의 흐름을 1부터 10까지 직접 연결해야 합니다.

/* [Pain Point]: 제어 컴포넌트 - 비서에게 일일이 보고하고 결재받는 방식 */
const [name, setName] = useState("");

<input
  name="userName"                     // 1. 이름표 달기
  value={name}                        // 2. 사장님(State) 결재 받기
  onChange={(e) => setName(e.target.value)} // 3. 보고하기
  onBlur={() => validate(name)}       // 4. 나갈 때 검사하기
  ref={nameRef}                       // 5. 실제 주소 확보하기
/>
🔍 코드 상세 분석 및 문제점

기하급수적 복잡도: 필드가 1개일 때는 괜찮지만, 10개, 100개가 되면 위와 같은 5단 배선 작업을 필드 수만큼 반복해야 합니다. 코드가 비대해지고 실수가 잦아집니다.
불필요한 리렌더링: 사용자가 글자 하나를 칠 때마다 사장님(State)이 바뀌고, 사장님이 바뀌면 사무실(Component) 전체가 다시 그려집니다. 인풋이 많아지면 타이핑이 버벅거리는 현상이 발생합니다.
보일러플레이트 지옥: 코드의 절반 이상이 비즈니스 로직이 아닌 단순 데이터 연결 로직으로 채워져 가독성을 심하게 해칩니다.
동작 원리의 비효율: 리액트가 굳이 알 필요 없는 중간 입력 과정까지 모두 가상 DOM을 거쳐 처리하게 됩니다.
📖 상세 개념 가이드 2: register 함수의 4대 핵심 부품
register 함수는 이 모든 수동 과정을 자동화된 시스템으로 바꿔줍니다. 우리가 ...register("userName")이라고 적는 순간, RHF 엔진은 해당 인풋에 필요한 네 가지 핵심 부품을 가방에 담아 전달합니다.

💻 기술적 실체: register가 반환하는 객체의 내부 구조
실제로 register("email")을 콘솔에 찍어보면 아래와 같은 자바스크립트 객체가 튀어나옵니다.

/* [Anatomy]: register 함수의 실제 반환값 */
{
  name: "email",                                // 1. 이름표 (Name)
  onChange: async (event) => { /* 내부 동기화 */ }, // 2. 수신기 (Listener)
  onBlur: async (event) => { /* 검증 트리거 */ },    // 3. 종료 감지 (Blur)
  ref: (instance) => { /* DOM 주소 탈취 */ }       // 4. 전용 통로 (Ref)
}

🔍 부품별 정밀 분석 및 내부 전파 디테일

name: 입력창의 고유 식별자입니다. RHF 내부 저장소(Store)에 데이터가 저장될 때 '키(Key)' 역할을 수행하며, Zod 스키마가 특정 데이터를 찾아낼 때 사용하는 주소지가 됩니다.
ref (전용 통로):비제어 방식의 핵심입니다. 리액트의 무거운 가상 DOM 사이클을 타지 않고, 실제 브라우저의 DOM 요소에 직접 빨대를 꽂는 것과 같습니다. RHF는 이 ref를 통해 필요할 때만 인풋의 value를 직접 읽어옵니다.
onChange (수신기): 사용자가 입력할 때 실행되는 함수입니다. 하지만 리액트의 useState를 호출하지 않습니다. 대신 RHF 내부의 자바스크립트 객체(Store)의 값만 조용히 업데이트합니다. 이 과정에서 리렌더링이 발생하지 않으므로 압도적인 성능이 유지됩니다.
onBlur (종료 감지): 사용자가 인풋창 밖을 클릭해 포커스를 잃었을 때 실행됩니다. 이때 RHF는 "이제 입력이 끝났구나!"라고 판단하고, 해당 필드에 대해 dirty 혹은 touched 상태를 마킹하며 정밀한 유효성 검사를 수행합니다.
📖 상세 개념 가이드 3: 스프레드 연산자(...)의 마법
우리가 인풋 태그 안에 적어주는 점 세 개(...), 즉 스프레드 연산자는 이 네 가지 부품이 담긴 가방을 열어서 인풋 엘리먼트에게 한꺼번에 쏟아부어 주는 역할을 합니다.

/* [Mechanism]: 스프레드 연산자를 통한 속성 전파 */

// 1. register가 부품 가방을 준비합니다.
const bag = register("email");

// 2. 가방 안에는 { name, ref, onChange, onBlur }가 들어있습니다.
// 3. 아래 코드는 가방을 열어 인풋에 모든 속성을 개별적으로 꽂아주는 것과 같습니다.
<input
  name={bag.name}
  ref={bag.ref}
  onChange={bag.onChange}
  onBlur={bag.onBlur}
/>

// 4. 이 과정을 한 줄로 줄인 것이 바로 우리가 아는 마법입니다.
<input {...register("email")} />

💡 비유: 멀티탭 복잡한 가전제품 수십 개를 벽 콘센트에 일일이 꽂으려 하면 선이 꼬입니다. register는 멀티탭과 같습니다. 여러분은 기기를 멀티탭에 꽂기만 하세요. 전원 공급(Value), 접지(Ref), 과부하 차단(Validation)은 멀티탭이 알아서 처리합니다. 개발자는 벽 뒤의 복잡한 배선(이벤트 로직)을 신경 쓸 필요가 없어집니다.

📖 상세 개념 가이드 4: 성능 최적화의 기술적 배경 (Proxy & Subscription)
왜 수백 개의 필드가 있어도 register 방식은 렉이 없을까요? 내부 전파의 디테일을 살펴보면 그 이유를 알 수 있습니다.

/* [Technical Insight]: RHF의 내부 성능 메커니즘 의사 코드 */

// 1. Proxy를 통한 구독 최적화
// 사용자가 'errors.email'을 호출하는 순간, RHF는 "아, 이 컴포넌트는 email 에러에 관심이 있구나"라고 메모합니다.
const formState = new Proxy(internalState, {
  get(target, prop) {
    if (prop === 'errors') {
      trackSubscription(prop); // 구독 시작
      return target.errors;
    }
  }
});

// 2. 비제어 방식의 직접 접근 (React Bypass)
// onChange가 발생해도 리액트에게 알리지 않고 내부 Map 객체만 수정합니다.
const internalStore = new Map();
const onChange = (e) => {
  internalStore.set(e.target.name, e.target.value);
  // 렌더링 요청(setState) 없음! -> 성능 최적화의 핵심
};

🔍 성능 최적화 디테일 분석

리액트 엔진 우회:ref를 통해 값을 직접 읽어오기 때문에 타이핑 시 리액트의 Render -> Commit 사이클을 건드리지 않습니다. 즉, 인풋에 글자를 쓸 때 컴포넌트 함수 자체가 다시 실행되지 않습니다.
구독(Subscription) 모델: RHF는 폼의 전체 데이터를 중앙에서 관리하되, Proxy(프록시) 기술을 사용합니다. errors.email을 사용하는 컴포넌트가 있다면, 오직 email 필드에 에러가 생기거나 사라질 때만 그 부분에 "화면을 그려라!"라고 신호를 보냅니다.
이벤트 위임과 포착:onChange 이벤트가 발생하면 RHF는 내부적으로 해당 이벤트를 가로채서 값만 쏙 빼내어 저장소에 넣고 이벤트를 종료합니다. 리액트에게 "값이 변했으니 다시 그려!"라고 말하지 않는 것이 성능의 핵심입니다.
