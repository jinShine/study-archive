[실습] 52강. [Pain] 웹 접근성(A11y) 레이블 문제: useId만으로 해결되지 않는 복잡한 에러 메시지와 인풋의 관계 설정
지난 시간 우리는 제출 버튼을 누르기 전 실시간으로 서버와 통신하며 아이디 중복 등을 확인하는 비동기 유효성 검사 기술을 알아보았습니다. 이제 우리 폼은 성능과 기능 면에서 매우 높은 완성도를 갖추게 되었습니다.

하지만 아키텍트로서 우리가 반드시 스스로에게 던져야 할 질문이 하나 남아있습니다. "우리가 만든 이 멋진 폼이 모든 사용자에게 공평하게 정보를 전달하고 있는가?" 오늘은 소리로 화면을 읽어주는 스크린 리더(Screen Reader) 사용자들에게 우리의 에러 메시지가 얼마나 불친절할 수 있는지, 그 기술적 난관을 파헤쳐 보겠습니다.

📖 상세 개념 가이드 1: 시각적 정보의 함정과 '말 없는 직원'
웹 접근성(A11y)이 막연하다면, 관공서의 상황에 비유해 보겠습니다.

💡 관공서 서류 제출 비유

신청서를 제출했는데 담당 직원이 아무 말 없이 서류를 돌려줍니다. 눈이 보이는 여러분은 서류의 특정 칸에 쳐진 빨간 동그라미를 보고 "아, 여기가 틀렸구나"라고 금방 알 수 있습니다. 하지만 앞을 볼 수 없는 사용자라면 어떨까요? 직원이 "이름 칸이 틀렸습니다"라고 말해주지 않는다면 그 사용자는 어디를 고쳐야 할지 영원히 알 길이 없습니다.

우리가 웹에서 단순히 인풋 테두리를 빨간색으로 바꾸고 그 밑에 에러 메시지를 띄우는 것이 바로 이 '말 없는 직원'과 같은 행동입니다. 스크린 리더는 화면에 갑자기 나타난 빨간 글씨를 자동으로 읽어주지 않습니다. 시각적인 정보는 기계에게 아무런 의미가 없는 단순한 데이터 조각일 뿐이기 때문입니다.

📖 상세 개념 가이드 2: 기술적 배경 — useId의 한계와 관계 설정
많은 개발자가 리액트의 useId 훅만 쓰면 접근성 문제가 해결된다고 생각합니다. 물론 useId는 인풋과 레이블을 연결하여 "이 칸은 이름 입력창입니다"라고 알려주는 데는 완벽합니다. 하지만 에러 메시지와의 관계는 이야기가 다릅니다.

🔍 핵심 접근성 속성 (A11y Attributes)
스크린 리더 사용자에게 인풋에 포커스가 갔을 때 다음 두 가지 정보가 전달되어야 합니다.

aria-invalid: "현재 이 칸의 값이 잘못되었습니다"라는 상태 경고.
aria-describedby: "왜 잘못되었는지"에 대한 구체적인 설명(에러 메시지)과의 연결.
기술적으로 이를 해결하려면 인풋과 에러 메시지 엘리먼트의 ID를 실로 묶듯 서로 연결해주어야 합니다. 스크린 리더가 인풋에 도착했을 때 그 실을 타고 넘어가 에러 메시지까지 한꺼번에 읽어주게 만드는 것이 핵심입니다.

📖 상세 개념 가이드 3: 수동 접근성 설계의 고통 (Manual ID Management)
이 연결 작업을 수동으로 하려다 보면 개발자는 엄청난 관리의 늪에 빠지게 됩니다. 모든 인풋마다 고유한 ID를 최소 두 개씩 생성해야 하고, 상태에 따라 속성을 넣었다 뺐다 하는 동적 바인딩을 수행해야 하죠.

💻 실습: 수동으로 구현한 고통스러운 A11y 코드
/* [File Path]: src/components/ManualA11yInput.tsx */
import { useId } from 'react';

export function ManualA11yInput({ name, error }: { name: string, error?: string }) {
  // 1. [Pain] 인풋과 에러 메시지만을 위한 별도의 고유 ID가 각각 필요합니다.
  const inputId = useId();
  const errorId = useId();

  return (
    <div className="flex flex-col gap-1">
      {/* 2. 레이블 연결 (기본적인 접근성) */}
      <label htmlFor={inputId} className="font-bold">이메일 주소</label>

      <input
        id={inputId}
        type="email"
        // 3. [Pain] 에러 유무에 따라 "true"/"false"를 수동으로 토글해야 합니다.
        aria-invalid={error ? "true" : "false"}
        // 4. [Pain] 에러가 있을 때만 해당 에러 ID를 연결하고, 없을 땐 undefined 처리하는 로직이 필수입니다.
        aria-describedby={error ? errorId : undefined}
        className={`border p-2 ${error ? "border-red-500" : "border-gray-300"}`}
      />

      {/* 5. [Critical] 인풋의 aria-describedby와 아래 p태그의 id가 1글자라도 틀리면 연결은 끊어집니다. */}
      {error && (
        <p id={errorId} role="alert" className="text-red-500 text-sm">
          {error}
        </p>
      )}
    </div>
  );
}

💡 코드 상세 분석 및 문제점

ID 관리의 지옥: 필드가 20개라면 40개의 ID를 수동으로 관리해야 합니다. 중복되지 않도록 이름을 짓는 것부터가 일입니다.
조건부 바인딩의 복잡도: aria-describedby={error ? errorId : undefined}와 같은 삼항 연산자가 모든 인풋에 중복 작성됩니다. 하나라도 빼먹으면 접근성은 무너집니다.
취약한 연결: p 태그의 id와 input의 aria-describedby는 문자열로 연결됩니다. 오타가 발생해도 타입스크립트는 이를 잡아낼 수 없으며, 결과적으로 스크린 리더는 에러 메시지를 무시하게 됩니다.
📖 상세 개념 가이드 4: 아키텍트의 관점 — 확장성과 유지보수
이 방식의 진짜 문제는 확장성입니다. 필드가 수십 개가 되는 회원가입 폼이나 복잡한 설문조사에서 이 모든 ID를 오타 없이 생성하고 동적으로 바인딩하는 것은 사실상 불가능에 가까운 중노동입니다.

생산성 측면
모든 필드에 대해 useId를 중복 호출하고 레이블, 인풋, 에러 메시지 간의 ID를 수동으로 바인딩하는 데 엄청난 시간이 소모됩니다.
초기 설정은 물론, 필드가 하나 추가될 때마다 개발자가 신경 써야 할 '접근성 배선 작업'이 기하급수적으로 늘어납니다.
무결성 측면
id와 aria-describedby는 문자열로 연결되기 때문에 오타 하나로 연결이 쉽게 끊어집니다.
리액트 환경에서 동적으로 렌더링되는 에러 메시지의 ID를 추적하는 과정에서 논리적 오류가 발생할 가능성이 매우 높습니다.
유지보수 측면
접근성 관련 로직이 각 컴포넌트에 파편화되어 있어, 웹 표준이 변경되거나 정책이 바뀔 때 모든 파일을 일일이 수정해야 하는 코드 비대화 현상이 발생합니다.
결국 개발자는 생산성을 위해 접근성을 포기하거나, 코드의 복잡도가 감당할 수 없을 정도로 높아지는 기로에 서게 됩니다. 단순히 보기에 예쁜 폼을 만드는 것을 넘어, 기계가 이해할 수 있는 의미론적 연결(Semantic Connection)을 구축하는 것이 얼마나 까다로운 일인지 우리는 실감하게 되었습니다.
