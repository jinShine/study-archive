## 42강. useTransition 기초 -- 멈추지 않는 UI 구현

### 도입

사용자가 검색어를 입력할 때 수천~수만 건의 데이터를 필터링해야 하는 상황에서, 일반적인 setState만 사용하면 입력 창 자체가 버벅이는 **블로킹 현상**이 발생한다. useTransition은 상태 업데이트에 **우선순위**를 부여하여, 사용자의 타이핑은 즉시 반영하면서 무거운 연산은 백그라운드에서 처리하도록 분리하는 훅이다.

### 개념 설명

- **긴급 업데이트(Urgent)**: 사용자가 직접 상호작용하는 입력값(input value) 등, 즉시 화면에 반영되어야 하는 상태
- **비긴급 업데이트(Non-urgent)**: 무거운 리스트 필터링, 차트 리렌더링 등 약간 늦어도 사용자 경험에 큰 영향이 없는 상태
- `useTransition()`은 `[isPending, startTransition]` 튜플을 반환한다
  - `isPending`: 비긴급 작업이 진행 중인지 여부 (boolean)
  - `startTransition(callback)`: callback 내부의 setState를 비긴급으로 마킹

### 코드 예제

```jsx
import { useState, useTransition } from 'react';
import HeavyList from './components/HeavyList';

const bigData = Array.from({ length: 10000 }, (_, i) => `데이터 아이템 ${i + 1}`);

export default function App() {
  const [filterText, setFilterText] = useState('');
  const [filteredList, setFilteredList] = useState(bigData);
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) => {
    const value = e.target.value;

    // 긴급 업데이트: 입력값 즉시 반영
    setFilterText(value);

    // 비긴급 업데이트: 무거운 필터링은 나중에
    startTransition(() => {
      const filtered = bigData.filter(item => item.includes(value));
      setFilteredList(filtered);
    });
  };

  return (
    <div>
      <input value={filterText} onChange={handleChange} placeholder="검색어 입력" />
      {isPending && <span>데이터 분석 중...</span>}
      <div style={{ opacity: isPending ? 0.3 : 1 }}>
        <HeavyList items={filteredList} />
      </div>
    </div>
  );
}
```

```jsx
// HeavyList.jsx
import { memo } from 'react';

const HeavyList = memo(({ items }) => (
  <ul>
    {items.map((item, index) => (
      <li key={index}>{item}</li>
    ))}
  </ul>
));

export default HeavyList;
```

### 코드 해설

1. `setFilterText(value)` -- startTransition **밖**에 위치하므로 긴급 업데이트로 처리된다. 10,000건 필터링 중에도 타이핑이 씹히지 않는다.
2. `startTransition(() => { ... })` -- 내부의 `setFilteredList`는 비긴급으로 분류되어, 브라우저가 한가할 때 실행된다.
3. **중단 가능 렌더링(Discarding old work)**: 사용자가 "123"을 빠르게 입력하면, "1"에 대한 필터링을 하다가 "2"가 입력되면 즉시 이전 연산을 폐기하고 "12"에 대한 연산으로 전환한다.
4. `isPending`을 활용하여 투명도 조절, 로딩 텍스트 등 시각적 피드백을 제공한다.

### 실무 비유

커피숍에서 바리스타가 주문을 받는 상황에 비유할 수 있다. 손님의 주문(입력)은 즉시 메모하되, 커피 제조(무거운 연산)는 뒤에서 처리한다. 새 주문이 들어오면 아직 시작도 안 한 이전 커피는 취소하고 최신 주문부터 만든다.

### 핵심 포인트

- `startTransition` 밖의 setState는 긴급, 안의 setState는 비긴급이다
- `isPending`으로 백그라운드 작업 상태를 사용자에게 시각적으로 알린다
- 비긴급 렌더링은 더 급한 업데이트가 들어오면 **즉시 중단**된다

## 43강. useTransition 탭 전환 -- 중단 가능한 렌더링

### 도입

탭 UI에서 가벼운 페이지와 무거운 대시보드(10,000개 로우)가 공존할 때, 무거운 탭을 클릭하면 모든 버튼이 일시적으로 먹통이 되는 현상이 발생한다. useTransition을 탭 전환에 적용하면, 무거운 탭을 렌더링하는 도중에도 다른 탭 버튼이 즉시 반응한다.

### 개념 설명

- **중단 가능성(Interruptibility)**: startTransition으로 감싼 상태 변경은 렌더링 도중이라도 더 급한 입력이 들어오면 즉시 폐기(Interrupt)될 수 있다
- 무거운 탭을 그리는 중에 사용자가 다른 탭을 클릭하면, 리액트는 진행 중이던 렌더링을 버리고 새 탭으로 즉시 전환한다
- 이 과정에서 브라우저 메인 스레드는 멈추지 않으므로 호버 효과, 스크롤 등이 부드럽게 유지된다

### 코드 예제

```jsx
import { useState, useTransition } from 'react';
import HomeTab from './components/HomeTab';
import PostsTab from './components/PostsTab';
import AdminDashboard from './components/AdminDashboard';

export default function App() {
  const [tab, setTab] = useState('home');
  const [isPending, startTransition] = useTransition();

  const handleTabSelect = (nextTab) => {
    // 탭 전환을 비긴급 작업으로 마킹
    startTransition(() => {
      setTab(nextTab);
    });
  };

  return (
    <div>
      <nav>
        <button onClick={() => handleTabSelect('home')}>홈</button>
        <button onClick={() => handleTabSelect('posts')}>게시판</button>
        <button onClick={() => handleTabSelect('admin')}>대시보드 (무거움)</button>
      </nav>

      {isPending && <p>백그라운드에서 화면을 구성 중입니다...</p>}

      <main style={{ opacity: isPending ? 0.4 : 1 }}>
        {tab === 'home' && <HomeTab />}
        {tab === 'posts' && <PostsTab />}
        {tab === 'admin' && <AdminDashboard />}
      </main>
    </div>
  );
}
```

```jsx
// AdminDashboard.jsx
import { memo } from 'react';

const AdminDashboard = memo(() => {
  const items = Array.from({ length: 10000 }, (_, i) => `대시보드 리포트 #${i + 1}`);
  return (
    <div>
      <h3>관리자 상세 리포트</h3>
      <ul style={{ height: '400px', overflowY: 'auto' }}>
        {items.map((item, index) => (
          <li key={index}>{item}</li>
        ))}
      </ul>
    </div>
  );
});

export default AdminDashboard;
```

### 코드 해설

1. `handleTabSelect` 내부에서 `setTab`을 startTransition으로 감싸면, 탭 전환 자체가 비긴급 작업이 된다.
2. 대시보드 버튼을 누르면 `isPending`이 즉시 true가 되고, 리액트는 백그라운드에서 10,000개 리스트를 그리기 시작한다.
3. 렌더링 도중 사용자가 "홈"을 다시 누르면, 대시보드 연산을 **즉시 폐기**하고 홈 화면을 최우선으로 보여준다.
4. `pointerEvents: isPending ? 'none' : 'auto'` 옵션으로 계산 중 불필요한 클릭을 방지할 수도 있다.

### 실무 비유

자동차 내비게이션이 경로를 재탐색하는 중에도 화면 터치가 멈추지 않는 것과 같다. 무거운 경로 계산은 백그라운드에서 진행하되, 사용자의 새로운 조작은 항상 최우선으로 처리한다.

### 핵심 포인트

- 탭 전환 시 `setTab`을 startTransition으로 감싸면 무거운 탭 렌더링이 중단 가능해진다
- 무거운 컴포넌트에는 `memo`를 적용하여 불필요한 재연산을 방어한다
- isPending 상태로 투명도, 로딩 메시지 등 시각적 피드백을 반드시 제공한다

## 44강. useDeferredValue 기초 -- 데이터 흐름의 완급 조절

### 도입

useTransition이 상태를 **변경하는 행위(Action)**를 제어했다면, useDeferredValue는 이미 변경된 **값(Value)**의 화면 반영 타이밍을 늦추는 훅이다. 외부에서 전달받은 props나 이미 변경된 state 값을 자식 컴포넌트에 전달할 때, 무거운 렌더링을 지연시켜 입력 반응성을 보호한다.

### 개념 설명

- `useDeferredValue(value)` -- 전달된 value의 "지연된 복사본"을 반환한다
- 브라우저가 입력 처리로 바쁜 동안에는 **이전 값**을 유지하고, 한가해지면 **최신 값**을 반영한다
- `isStale = query !== deferredQuery` -- 원본과 지연 값의 비교로 "현재 데이터가 최신인지" 판단할 수 있다

### 코드 예제

```jsx
import { useState, useDeferredValue } from 'react';
import HeavyResultList from './components/HeavyResultList';

export default function App() {
  const [query, setQuery] = useState("");

  // 지연된 값 생성: 브라우저가 한가해질 때까지 이전 값을 유지
  const deferredQuery = useDeferredValue(query);

  // 원본과 지연 값이 다르면 = 아직 백그라운드 처리 중
  const isStale = query !== deferredQuery;

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="빠르게 타이핑해보세요!"
      />
      <div style={{ opacity: isStale ? 0.3 : 1 }}>
        <p>{isStale ? "최신 데이터 반영 중..." : "업데이트 완료"}</p>
        <HeavyResultList deferredQuery={deferredQuery} />
      </div>
    </div>
  );
}
```

```jsx
// HeavyResultList.jsx
import { memo } from 'react';

const HeavyResultList = memo(({ deferredQuery }) => {
  const items = [];
  for (let i = 0; i < 5000; i++) {
    if (`아이템 ${i}`.includes(deferredQuery)) {
      items.push(<li key={i}>검색 결과: 아이템 {i}</li>);
    }
  }
  return (
    <ul style={{ height: '300px', overflowY: 'auto' }}>
      {items.length > 0 ? items : <li>결과가 없습니다.</li>}
    </ul>
  );
});

export default HeavyResultList;
```

### 코드 해설

1. input의 `value`는 지연되지 않은 원본 `query`를 사용한다. 따라서 타이핑이 절대 밀리지 않는다.
2. 무거운 리스트는 `deferredQuery`를 사용한다. 타이핑이 멈추는 찰나에 리스트가 한꺼번에 갱신된다.
3. `isStale` 상태를 통해 투명도를 조절하면, 사용자는 "입력은 잘 들어갔고, 결과는 곧 나온다"는 심리적 안정감을 얻는다.

### 실무 비유

TV 생방송의 **아나운서와 자막**에 비유할 수 있다. 아나운서의 말(입력)은 실시간으로 나오지만, 자막(리스트)은 약간의 딜레이가 있어도 전체 경험에는 문제가 없다.

### 핵심 포인트

- `useDeferredValue`는 값의 화면 반영 시점을 늦추는 훅이다
- input에는 원본 값, 무거운 자식에는 지연된 값을 전달하여 반응성을 분리한다
- `isStale` 패턴으로 "데이터 동기화 중" 상태를 시각적으로 표현한다

## 45강. useDeferredValue 심화 -- React.memo와의 조합

### 도입

useDeferredValue로 값을 늦추더라도, 부모 컴포넌트가 리렌더링되면 자식 컴포넌트 함수 자체는 다시 호출된다. 지연된 값이 같아도 함수가 실행되면 성능 이득이 절반으로 깎인다. `React.memo`와의 조합이 **필수**인 이유가 여기에 있다.

### 개념 설명

- **베일아웃(Bail-out)**: React.memo가 프롭의 참조 비교(O(1))를 수행하여, 지연된 값이 이전과 동일하면 컴포넌트 실행 자체를 **건너뛰는** 최적화 메커니즘
- memo 없이 useDeferredValue만 사용하면, 부모(App)의 text가 바뀔 때마다 자식은 일단 다시 호출된다
- memo와 함께 사용하면, `deferredText`가 실제로 변할 때만 자식의 무거운 연산이 수행된다

### 코드 예제

```jsx
// App.jsx
import { useState, useDeferredValue } from 'react';
import OptimizedHeavyList from './components/OptimizedHeavyList';

export default function App() {
  const [text, setText] = useState("");
  const deferredText = useDeferredValue(text);
  const isStale = text !== deferredText;

  return (
    <div>
      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="검색어를 입력해보세요"
      />
      {isStale && <div>계산 중...</div>}
      <section style={{ opacity: isStale ? 0.3 : 1 }}>
        {/* 핵심: 지연된 값을 전달 */}
        <OptimizedHeavyList query={deferredText} />
      </section>
    </div>
  );
}
```

```jsx
// OptimizedHeavyList.jsx
import { memo } from 'react';

// memo가 프롭 비교 -> deferredText가 안 바뀌면 실행 자체를 건너뜀
const OptimizedHeavyList = memo(({ query }) => {
  console.log(`[HeavyList] "${query}" 데이터로 무거운 연산 시작...`);

  const items = Array.from({ length: 10000 }, (_, i) => (
    <div key={i}>검색 결과 아이템 #{i} (키워드: {query})</div>
  ));

  return (
    <div style={{ height: '400px', overflowY: 'auto' }}>
      {items}
    </div>
  );
});

export default OptimizedHeavyList;
```

### 코드 해설

1. 사용자가 타이핑하면 `text`가 즉시 바뀌고 App이 리렌더링된다.
2. 이때 `deferredText`는 아직 이전 값을 유지하고 있으므로, memo로 감싼 자식은 프롭이 동일하다고 판단하여 **실행 자체를 건너뛴다**.
3. 타이핑이 멈추면 deferredText가 최신 값으로 갱신되고, 그때 비로소 10,000개 루프가 실행된다.
4. CPU는 오직 "input에 글자 그리기"에만 집중하므로 극강의 부드러움이 유지된다.

### 실무 비유

고급 레스토랑의 주방 오더 시스템과 같다. 웨이터(App)가 주문을 수시로 바꿔도, 주방(HeavyList)은 최종 확정된 주문만 조리한다. 확정 전까지 주방의 자원을 낭비하지 않는 것이 핵심이다.

### 핵심 포인트

- `useDeferredValue` + `React.memo` 조합이 완전한 최적화를 달성한다
- memo 없이 지연 값만 사용하면 함수 호출은 여전히 발생한다 (성능 반감)
- memo가 있으면 지연된 값이 변하지 않는 한 **O(1) 참조 비교**로 실행을 건너뛴다

## 46강. 동시성 훅 총정리 -- 상황별 완벽 가이드

### 도입

useTransition과 useDeferredValue라는 두 가지 무기를 모두 확보한 상태에서, 실무에서 마주하는 핵심 질문은 **"어떤 상황에 무엇을 쓸 것인가?"**이다. 이 강에서는 두 훅의 결정적인 차이점과 최종 판단 기준을 정리한다.

### 개념 설명

#### useTransition -- 액션(Action) 중심의 능동적 제어

- **상태를 변경하는 코드(setState)에 직접 접근할 수 있을 때** 사용
- `startTransition`으로 setState 호출을 직접 감싸서 비긴급으로 분류
- `isPending`으로 로딩 스피너, 시각적 피드백 구현이 매우 수월
- 비유: 자동차의 핸들을 직접 꺾는 운전자

#### useDeferredValue -- 값(Value) 중심의 수동적 방어

- **상태 업데이트 함수를 제어할 수 없을 때** 사용 (외부 라이브러리, Props 등)
- 이미 변경된 값의 화면 반영 타이밍만 늦추는 방어막
- 여러 상태가 얽혀 있어도 최종 결과값 하나만 지연시키면 되므로 구조가 깔끔
- 비유: 노면의 충격을 흡수하는 자동차 서스펜션

### 코드 예제

```jsx
import { useState, useTransition, useDeferredValue } from 'react';
import HeavyView from './components/HeavyView';

export default function App() {
  const [text, setText] = useState("");
  const [list, setList] = useState("");

  // 전략 A: useTransition
  const [isPending, startTransition] = useTransition();

  // 전략 B: useDeferredValue
  const deferredText = useDeferredValue(text);

  const handleTransitionChange = (e) => {
    const val = e.target.value;
    setText(val);                          // 긴급: 타이핑
    startTransition(() => { setList(val); }); // 비긴급: 리스트
  };

  return (
    <div>
      <h2>동시성 훅 최종 비교</h2>

      <section>
        <h3>전략 A: useTransition (액션 중심)</h3>
        <input value={text} onChange={handleTransitionChange} />
        {isPending && <p>트랜지션 로딩 중...</p>}
        <HeavyView value={list} />
      </section>

      <section>
        <h3>전략 B: useDeferredValue (값 중심)</h3>
        <input value={text} onChange={(e) => setText(e.target.value)} />
        {text !== deferredText && <p>값 지연 중...</p>}
        <HeavyView value={deferredText} />
      </section>
    </div>
  );
}
```

### 코드 해설

두 전략을 한 화면에서 비교하면, 동일한 입력에 대해 동일한 결과를 보여주지만 **접근 방식**이 다르다.
- 전략 A: 개발자가 setState를 직접 감싸서 타이밍을 제어한다
- 전략 B: 최종 값을 useDeferredValue에 넘기고, 리액트가 자동으로 타이밍을 조절한다

### 최종 선택 가이드

| 판단 기준 | useTransition | useDeferredValue |
|-----------|:---:|:---:|
| setState를 직접 감쌀 수 있는가? | O | X |
| 로딩 스피너 등 중간 상태가 필요한가? | O (isPending) | 제한적 (isStale 패턴) |
| 외부 라이브러리/Props에서 오는 값인가? | X | O |
| 여러 상태 중 결과값 하나만 지연시키면 되는가? | X | O |

### 실무 비유

- **useTransition** = 교통 경찰이 직접 수신호로 차량 흐름을 제어하는 것
- **useDeferredValue** = 자동 감속 방지턱이 차량 속도를 알아서 조절하는 것

### 핵심 포인트

- 상태 변경 코드를 제어할 수 있으면 useTransition, 값만 받으면 useDeferredValue
- **이중 지연 금지**: useTransition으로 바꾼 상태를 다시 useDeferredValue로 감싸지 말 것 (렌더링 흐름 꼬임)
- 상황에 맞는 **단 하나의 무기**만 정확하게 사용한다

## 전체 섹션 요약

| 항목 | useTransition | useDeferredValue |
|------|---------------|-------------------|
| 제어 대상 | 상태 변경 **행위** (setState) | 이미 변경된 **값** (value) |
| 반환값 | `[isPending, startTransition]` | 지연된 값 (deferred copy) |
| 로딩 상태 | `isPending` 직접 제공 | `query !== deferredQuery` 패턴으로 계산 |
| memo 필요성 | 권장 | **필수** (없으면 성능 반감) |
| 사용 시점 | 내가 setState를 직접 호출할 때 | 외부에서 값을 받을 때 |
| 중단 가능성 | O (새 입력 시 이전 렌더링 폐기) | O (리액트가 자동 판단) |
| 주의사항 | 두 훅을 동시에 중첩하지 말 것 | 반드시 memo와 함께 사용 |

## 면접 핵심 Q&A

**Q1. useTransition과 useDeferredValue의 차이점은?**
> useTransition은 상태를 변경하는 코드(setState)를 직접 startTransition으로 감싸서 비긴급으로 마킹하는 **액션 중심** 훅이다. useDeferredValue는 이미 변경된 값의 화면 반영 타이밍을 늦추는 **값 중심** 훅이다. setState에 접근할 수 있으면 useTransition, 외부 props/라이브러리에서 값만 받으면 useDeferredValue를 사용한다.

**Q2. startTransition 내부의 렌더링은 왜 "중단 가능"한가?**
> React 18의 동시성 렌더링 모델에서, startTransition으로 마킹된 업데이트는 낮은 우선순위(Lane)를 부여받는다. 렌더링 도중 더 높은 우선순위의 업데이트(사용자 입력 등)가 발생하면, 리액트는 진행 중이던 저우선순위 렌더링을 즉시 폐기(Discard)하고 고우선순위 작업을 먼저 처리한다.

**Q3. useDeferredValue 사용 시 React.memo가 필수인 이유는?**
> useDeferredValue만 사용하면 부모 리렌더링 시 자식 컴포넌트 함수가 여전히 호출된다. 지연된 값이 이전과 같더라도 함수 내부의 무거운 연산이 실행되어 성능 이득이 반감된다. React.memo를 적용하면 프롭의 참조 비교를 통해 값이 동일할 때 컴포넌트 실행 자체를 건너뛰는(Bail-out) 최적화가 작동한다.

**Q4. isPending과 isStale 패턴의 차이는?**
> isPending은 useTransition이 직접 제공하는 boolean 상태값이다. isStale은 useDeferredValue에서 `원본값 !== 지연값`으로 직접 계산하는 패턴이다. 둘 다 "백그라운드 작업 진행 중"을 의미하지만, isPending은 명시적이고, isStale은 파생값이라는 차이가 있다.

**Q5. 두 훅을 동시에 중첩 사용하면 안 되는 이유는?**
> useTransition으로 비긴급 처리한 상태를 다시 useDeferredValue로 감싸면, 리액트 엔진에 이중으로 복잡한 지연 명령을 내리는 것이 된다. 스케줄러의 우선순위 판단이 꼬여 렌더링 흐름이 예측 불가능해지고, 오히려 성능이 저하될 수 있다.

**Q6. 동시성 훅은 React 18 이전에서도 사용할 수 있는가?**
> 아니다. useTransition과 useDeferredValue는 React 18에서 도입된 동시성 렌더링 모델에 기반한다. `ReactDOM.createRoot()`로 앱을 마운트해야 동시성 모드가 활성화되며, 레거시 `ReactDOM.render()`를 사용하면 이 훅들이 동기적으로 작동하여 최적화 효과가 없다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| 동시성 렌더링 (Concurrent Rendering) | React 18에서 도입된, 렌더링 작업을 분할하고 우선순위에 따라 중단/재개할 수 있는 렌더링 모델 |
| 긴급 업데이트 (Urgent Update) | 사용자 입력처럼 즉시 화면에 반영되어야 하는 높은 우선순위의 상태 업데이트 |
| 비긴급 업데이트 (Non-urgent Update) | startTransition으로 감싼, 약간 늦어도 되는 낮은 우선순위의 상태 업데이트 |
| isPending | useTransition이 반환하는 boolean. 비긴급 작업이 진행 중이면 true |
| startTransition | 콜백 내부의 setState를 비긴급으로 마킹하는 함수 |
| useDeferredValue | 값의 화면 반영 타이밍을 늦추어 지연된 복사본을 반환하는 훅 |
| 베일아웃 (Bail-out) | React.memo가 프롭 비교를 통해 불필요한 리렌더링을 건너뛰는 최적화 메커니즘 |
| isStale 패턴 | `원본값 !== 지연값`으로 현재 데이터가 최신인지 판단하는 useDeferredValue 활용 패턴 |
| 중단 가능성 (Interruptibility) | 낮은 우선순위 렌더링이 높은 우선순위 입력에 의해 즉시 폐기될 수 있는 특성 |
| 블로킹 (Blocking) | 무거운 연산이 메인 스레드를 점유하여 UI 상호작용이 멈추는 현상 |

## 자가 점검 체크리스트

- [ ] useTransition의 반환값 `[isPending, startTransition]`의 역할을 각각 설명할 수 있다
- [ ] startTransition 안과 밖의 setState가 왜 다른 우선순위를 갖는지 이해한다
- [ ] 무거운 탭 전환 시 startTransition을 제거하면 어떤 블로킹 현상이 발생하는지 체감했다
- [ ] useDeferredValue가 "값의 지연된 복사본"을 반환한다는 개념을 이해한다
- [ ] isStale 패턴을 활용하여 "데이터 동기화 중" 시각적 피드백을 구현할 수 있다
- [ ] useDeferredValue에 React.memo가 필수인 이유를 논리적으로 설명할 수 있다
- [ ] 두 훅의 선택 기준(setState 접근 가능 여부)을 즉시 판단할 수 있다
- [ ] 이중 지연(useTransition + useDeferredValue 중첩)을 피해야 하는 이유를 안다
- [ ] createRoot 기반에서만 동시성 모드가 활성화됨을 인지하고 있다
