# 46강. 동시성 훅 총정리 -- 상황별 완벽 가이드

## 도입

리액트 성능 최적화의 정점, 동시성(Concurrency) 렌더링의 대장정을 마무리하는 시간이다. 우리는 `useTransition`과 `useDeferredValue`라는 강력한 무기를 손에 넣었지만, 실무에서 "어떤 상황에 무엇을 쓸 것인가?"라는 최후의 질문에 답할 수 있어야 진정한 전문가라고 할 수 있다.

이 두 훅의 결정적인 차이점을 명확히 구분하고, 어떤 상황에서도 막힘없이 최적의 설계를 할 수 있는 최종 판단 기준을 정리해 본다.

## 개념 설명

### useTransition: 액션(Action) 중심의 능동적 제어

`useTransition`은 내가 상태를 변경하는 코드(`setState`)에 직접 접근할 수 있을 때 사용하는 도구이다. 리액트에게 "지금 내가 호출하는 이 함수는 나중에 처리해도 돼"라고 직접 명령을 내리는 방식이다.

- **핵심 포인트**: `startTransition`으로 상태 업데이트 함수를 직접 감싼다.
- **장점**: `isPending` 상태를 통해 로딩 스피너나 시각적 피드백을 구현하기가 매우 수월하다.

### useDeferredValue: 값(Value) 중심의 수동적 방어

`useDeferredValue`는 내가 상태 업데이트 함수를 제어할 수 없을 때, 즉 외부로부터 전달받은 프롭(Props)이나 이미 변경된 값의 화면 반영 타이밍만 늦추고 싶을 때 사용한다.

- **핵심 포인트**: 상태를 변경하는 함수가 내 손에 없고, 값만 내려오는 상황에서 방어막을 친다.
- **장점**: 복잡한 상태 변화가 얽혀 있어도 최종적인 '값' 하나만 지연시키면 되므로 구조가 깔끔해진다.

### 최종 선택 가이드

| 판단 기준 | useTransition | useDeferredValue |
|---|---|---|
| 상태 업데이트 함수를 감쌀 수 있는가? | Yes | No (외부 라이브러리, Props) |
| 로딩 스피너 등 '중간 상태'가 필요한가? | 유리 (isPending 제공) | 불리 (직접 비교 필요) |
| 여러 상태가 섞여서 결과값 하나만 지연시키면 되는가? | 불리 | 유리 |

### 이중 지연 금지

성능을 극한으로 높이겠다고 `useTransition`으로 바꾼 상태를 다시 `useDeferredValue`로 감싸는 행위는 피해야 한다. 이는 리액트 엔진에게 이중으로 복잡한 지연 명령을 내리는 꼴이 되어, 오히려 렌더링 흐름을 꼬이게 만들 수 있다. 상황에 맞는 단 하나의 무기만 날카롭게 사용해야 한다.

## 코드 예제

### 무거운 뷰 컴포넌트 (HeavyView.jsx)

```jsx
import { memo } from 'react';

const HeavyView = memo(({ value }) => {
  const items = Array.from({ length: 5000 }, (_, i) => (
    <div key={i}>데이터 #{i} (검색어: {value})</div>
  ));
  return <div style={{ height: '300px', overflowY: 'auto' }}>{items}</div>;
});

export default HeavyView;
```

### 두 전략 비교 메인 관제탑 (App.jsx)

```jsx
import { useState, useTransition, useDeferredValue } from 'react';
import HeavyView from './components/HeavyView';

export default function App() {
  const [text, setText] = useState("");
  const [list, setList] = useState("");

  // 1. useTransition 전략
  const [isPending, startTransition] = useTransition();

  // 2. useDeferredValue 전략
  const deferredText = useDeferredValue(text);

  const handleTransitionChange = (e) => {
    const val = e.target.value;
    setText(val); // 긴급: 타이핑
    startTransition(() => {
      setList(val); // 비긴급: 리스트
    });
  };

  return (
    <div style={{ padding: '40px' }}>
      <h1>동시성 훅 최종 비교</h1>

      <section>
        <h3>전략 A: useTransition (액션 중심)</h3>
        <input value={text} onChange={handleTransitionChange} placeholder="Transition 테스트..." />
        {isPending && <p>트랜지션 로딩 중...</p>}
        <HeavyView value={list} />
      </section>

      <hr />

      <section>
        <h3>전략 B: useDeferredValue (값 중심)</h3>
        <input value={text} onChange={(e) => setText(e.target.value)} placeholder="Deferred 테스트..." />
        {text !== deferredText && <p>값 지연 중...</p>}
        <HeavyView value={deferredText} />
      </section>
    </div>
  );
}
```

## 코드 해설

- **전략 A**: `handleTransitionChange`에서 `setFilterText`는 즉시 반영하고, `setList`만 `startTransition`으로 감싸서 비긴급 처리한다. `isPending`으로 로딩 상태를 쉽게 표현할 수 있다.
- **전략 B**: `setText`만 호출하고, `useDeferredValue`가 알아서 값의 반영 타이밍을 조절한다. 별도의 상태 분리 없이 값 하나만 지연시키므로 코드가 더 간결하다.

## 실무 비유

- **useTransition (자동차 핸들)**: 운전자가 직접 핸들을 꺾어 방향을 능동적으로 제어한다.
- **useDeferredValue (서스펜션)**: 노면의 충격을 흡수하여 차체에 전달하지 않는 수동적인 방어 장치이다.

## 핵심 포인트

- `useTransition`은 상태 업데이트 코드에 직접 접근 가능할 때, `useDeferredValue`는 값만 제어 가능할 때 사용한다.
- 두 훅을 동시에 같은 데이터에 적용하는 '이중 지연'은 피해야 한다.
- `memo`와의 조합은 두 훅 모두에서 성능 극대화의 핵심이다.
