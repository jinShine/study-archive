## 20강. useRef -- 렌더링을 일으키지 않고 값을 기억하는 법

### 도입

리액트에서 값을 기억하는 방법은 세 가지가 있다: `useState`, `useRef`, 일반 `let` 변수. 이 중 `useRef`는 **리렌더링 없이** 값을 컴포넌트 생애 내내 유지할 수 있는 특별한 도구다. 리액트가 제공하는 "비밀 금고" 역할을 한다.

### 개념 설명

- `useRef(초기값)`을 호출하면 `{ current: 초기값 }` 형태의 **일반 자바스크립트 객체**를 반환한다.
- 리액트는 이 객체의 `.current` 변경을 **감지하지 않도록** 설계되어 있어 렌더링을 유발하지 않는다.
- 컴포넌트가 리렌더링되어도 `useRef`가 만든 객체는 별도의 메모리 공간에 보관되므로 값이 초기화되지 않는다.

**세 가지 값 저장 방식 비교:**

| 구분 | `useState` | `useRef` | `let` 변수 |
|------|-----------|----------|-----------|
| 값 변경 시 리렌더링 | O (발생) | X (미발생) | X (미발생) |
| 리렌더링 후 값 유지 | O (유지) | O (유지) | X (초기화) |
| 화면에 반영 | O (즉시) | X (다음 렌더링 시) | X |
| 용도 | UI에 보이는 값 | 내부 데이터 보관 | 임시 계산 |

### 코드 예제

```jsx
import { useRef, useState } from "react";

export default function SecretCounter() {
  // [전광판] 값이 바뀌면 화면을 다시 그린다
  const [renderCount, setRenderCount] = useState(0);

  // [비밀 수첩] 값이 바뀌어도 화면은 그대로다
  const secretRef = useRef(0);

  // [일반 변수] 리렌더링마다 매번 0으로 초기화된다
  let localVariable = 0;

  const increaseSecret = () => {
    secretRef.current = secretRef.current + 1;
    localVariable = localVariable + 1;
    console.log("수첩(Ref):", secretRef.current, " | 일반 변수:", localVariable);
  };

  return (
    <div>
      <p>전광판(useState): {renderCount}</p>
      <p>수첩(useRef): {secretRef.current}</p>
      <p>일반 변수(let): {localVariable}</p>

      <button onClick={increaseSecret}>비밀 숫자 올리기</button>
      <button onClick={() => setRenderCount(prev => prev + 1)}>화면 새로고침</button>
    </div>
  );
}
```

### 코드 해설

- `useRef(0)`: 리액트에게 "0이 들어있는 객체를 만들되, 리렌더링해도 이 객체를 새로 만들지 말고 재사용해줘"라고 요청하는 것이다.
- `secretRef.current`: 값을 변경해도 리액트는 화면을 다시 그리지 않는다. 콘솔에서만 변경을 확인 가능하다.
- `let localVariable = 0`: 컴포넌트 함수가 다시 실행(리렌더링)될 때마다 선언부터 다시 실행되므로 항상 0으로 돌아간다.
- '화면 새로고침' 버튼을 누르면 `useState`가 리렌더링을 유발하고, 그때서야 `secretRef.current`에 누적된 값이 화면에 반영된다. 반면 `localVariable`은 다시 0이 된다.

### 실무 비유

`useState`는 전광판(바뀌면 모두가 본다), `useRef`는 비밀 수첩(조용히 적어두고 필요할 때만 꺼낸다), `let` 변수는 포스트잇(바람 불면 날아간다).

### 핵심 포인트

- `useRef`는 `{ current: 값 }` 객체를 반환하며, `.current`로만 접근/수정한다.
- `.current`를 변경해도 **리렌더링이 발생하지 않는다** -- 이것이 `useState`와의 가장 큰 차이다.
- 리렌더링 간에 값이 유지되므로 일반 `let` 변수와도 다르다.
- UI에 보여야 하는 값은 `useState`, 내부 로직용 값은 `useRef`를 사용한다.

## 21강. useRef -- DOM에 직접 접근하여 물리적인 요소 제어하기

### 도입

리액트는 선언적(declarative) 패러다임을 따르지만, 포커스 이동, 스크롤 제어, 미디어 재생 같은 **물리적 동작**은 선언적 방식만으로 해결하기 어렵다. `useRef`로 실제 DOM 노드를 참조하면 브라우저 표준 API를 직접 호출하여 이러한 동작을 수행할 수 있다.

### 개념 설명

- `useRef(null)`로 빈 참조 객체를 만든 뒤, JSX의 `ref` 속성에 연결하면 리액트가 렌더링 완료 후 해당 DOM 노드를 `.current`에 자동으로 넣어준다.
- `inputRef.current`는 실제 HTML 엘리먼트 그 자체이므로, `focus()`, `scrollIntoView()`, `play()` 등 브라우저 표준 API를 그대로 사용할 수 있다.
- `document.querySelector` 없이도 리액트 방식으로 안전하게 DOM에 접근할 수 있다.

**State vs Ref 역할 비교:**

| 구분 | State | Ref |
|------|-------|-----|
| 철학 | "무엇"을 그릴지 선언 | 그려진 "그것"을 직접 조작 |
| 변경 시 | 컴포넌트가 리렌더링 | 컴포넌트가 가만히 있음 |
| 주요 용도 | 화면에 보이는 값 관리 | 포커스, 스크롤, 미디어 재생, 외부 라이브러리 연동 |
| 접근 방식 | 변수 그 자체 사용 | `.current` 속성을 통해 접근 |

### 코드 예제

```jsx
import { useRef } from "react";

export default function FocusManager() {
  const inputRef = useRef(null);

  const handleFocus = () => {
    // 방어적 null 체크 -- 요소가 화면에서 사라졌을 수도 있다
    if (inputRef.current) {
      inputRef.current.focus();
      inputRef.current.style.backgroundColor = "#fff9db";
      inputRef.current.style.border = "2px solid #fab005";
      inputRef.current.placeholder = "입력을 시작하세요!";
    }
  };

  return (
    <div>
      <input
        ref={inputRef}
        type="text"
        placeholder="버튼을 눌러보세요"
      />
      <button onClick={handleFocus}>입력창으로 강제 포커스 이동</button>
    </div>
  );
}
```

### 코드 해설

- `useRef(null)`: 아직 가리킬 대상이 없으므로 초기값을 `null`로 설정한다.
- `ref={inputRef}`: 리액트가 이 `<input>`을 렌더링할 때 실제 DOM 객체를 `inputRef.current`에 넣어준다.
- `inputRef.current.focus()`: 브라우저에게 "이 요소를 활성화해!"라고 **명령**을 내리는 것이다. 리액트의 상태를 바꾸는 것이 아니다.
- `if (inputRef.current)`: 조건부 렌더링 등으로 요소가 DOM에서 제거되었을 수 있으므로, null 체크는 필수 방어 코드다.

### 실무 비유

`useRef`는 TV 리모컨과 같다. TV(DOM 요소)가 있어야 리모컨이 작동하고, 리모컨 버튼(`.focus()`, `.play()`)을 눌러 물리적 동작을 실행한다.

### 핵심 포인트

- DOM 제어용 ref는 초기값을 `null`로 설정하고, JSX의 `ref` 속성으로 연결한다.
- `.current`는 실제 HTML 엘리먼트이므로 `focus()`, `scrollIntoView()`, `play()` 등을 직접 호출할 수 있다.
- **값은 State로, 동작(포커스/스크롤)은 Ref로** 관리하는 것이 정석이다.
- `inputRef.current.value = "강제값"` 같은 직접 값 주입은 `useState`와 싱크가 깨져 버그를 유발하므로 금지한다.

## 22강. useRef 실전 활용 1부 -- 이전 상태 추적과 로직 최적화

### 도입

리액트는 기본적으로 "지금 현재의 상태"만 관리한다. "직전 값이 무엇이었는지"는 제공하지 않는다. 또한 `useEffect`는 초기 렌더링에도 실행되기 때문에 "첫 렌더링을 무시하고 싶다"는 요구에도 기본 기능만으로는 대응할 수 없다. `useRef`가 이 두 가지 빈틈을 메운다.

### 개념 설명

**패턴 1: 이전 상태(prevState) 추적**
- `useEffect`는 렌더링 **후**에 실행된다.
- `useEffect` 안에서 현재 값을 ref에 저장하면, 다음 렌더링 시점에 ref에는 **직전 값**이 남아있다.
- 이 타이밍 차이를 이용해 "이전 값 vs 현재 값" 비교가 가능해진다.

**패턴 2: 첫 렌더링(초기 실행) 스킵**
- `useRef(true)`로 "첫 방문" 플래그를 만든다.
- `useEffect` 안에서 플래그가 `true`이면 `false`로 바꾸고 즉시 return한다.
- 두 번째 렌더링부터 실제 로직이 실행된다.

**왜 반드시 useRef인가?**

| 상황 | `let` 변수 | `useState` | `useRef` (정답) |
|------|-----------|-----------|----------------|
| 이전 값 기억 | 리렌더링 때마다 초기화 | 변경 시 리렌더링 유발 (무한 루프 위험) | 데이터만 안전하게 유지 |
| 첫 실행 방지 플래그 | 리렌더링 때마다 다시 `true` | 스위치 끄는 것만으로 화면이 또 깜빡임 | 로직 흐름만 깔끔하게 제어 |

### 코드 예제

**이전 상태 추적 -- PriceTracker:**

```jsx
import { useState, useEffect, useRef } from "react";

export default function PriceTracker() {
  const [price, setPrice] = useState(1000);
  const prevPriceRef = useRef();

  useEffect(() => {
    // 렌더링 완료 "후"에 현재 값을 금고에 저장한다
    // 다음 렌더링 때 이 금고에는 "직전 값"이 들어있게 된다
    prevPriceRef.current = price;
  }, [price]);

  // 렌더링 시점에는 아직 useEffect가 실행되기 전이므로 직전 값이 남아있다
  const prevPrice = prevPriceRef.current;

  return (
    <div>
      <p>현재가: {price}원</p>
      <p>직전가: {prevPrice !== undefined ? `${prevPrice}원` : "데이터 수집 중..."}</p>
      <p>{price > prevPrice ? "상승" : price < prevPrice ? "하락" : "변동 없음"}</p>
      <button onClick={() => setPrice(p => p + 100)}>가격 올리기</button>
      <button onClick={() => setPrice(p => p - 100)}>가격 내리기</button>
    </div>
  );
}
```

**첫 렌더링 스킵 -- SkipFirstRender:**

```jsx
import { useState, useEffect, useRef } from "react";

export default function SkipFirstRender() {
  const [count, setCount] = useState(0);
  const isFirstRender = useRef(true);

  useEffect(() => {
    if (isFirstRender.current) {
      isFirstRender.current = false;
      return; // 첫 실행 시 조기 종료
    }
    // 두 번째 렌더링부터만 실행되는 영역
    alert(`수량이 ${count}개로 변경되었습니다.`);
  }, [count]);

  return (
    <div>
      <p>수량: {count}</p>
      <button onClick={() => setCount(prev => prev + 1)}>수량 추가</button>
    </div>
  );
}
```

### 코드 해설

- **PriceTracker**: 리액트는 렌더링을 먼저 하고 `useEffect`를 실행한다. 렌더링 시점에 `prevPriceRef.current`를 읽으면 아직 이전 값이 들어있고, 렌더링 후 `useEffect`에서 현재 값으로 덮어쓴다. 이 타이밍 차이가 "이전 값 추적"의 핵심이다.
- **SkipFirstRender**: `isFirstRender.current = false`로 바뀌어도 리렌더링이 발생하지 않으므로, 플래그 전환 자체가 화면에 영향을 주지 않는다. `useState`로 같은 로직을 구현하면 `setState` 호출이 리렌더링을 유발하여 무한 루프 위험이 있다.

### 실무 비유

- **이전 값 추적**: 주식 차트에서 "전일 종가 대비 오늘 현재가"를 표시하는 것과 같다.
- **첫 렌더링 스킵**: 서버에 수량을 저장하는 API가 있을 때, 페이지 로드 시 불필요한 API 호출을 차단하는 최적화 기법이다.

### 핵심 포인트

- `useEffect` 안에서 ref를 업데이트하면 "렌더 시점에는 이전 값, Effect 실행 후에는 현재 값"이라는 타이밍 차이를 활용할 수 있다.
- 첫 렌더링 스킵은 `useRef(true)` + Early Return 패턴으로 구현한다.
- 이 두 패턴 모두 `useState`나 `let` 변수로는 안전하게 구현할 수 없다 -- `useRef`만의 고유 영역이다.

## 23강. useRef 실전 프로젝트 -- 스마트 타이머와 비디오 최적화 시스템

### 도입

앞서 배운 모든 useRef 패턴(값 보관, DOM 접근, 이전 값 추적, 첫 렌더링 스킵)을 하나의 프로젝트에 통합한다. 추가로, 초당 수십 번 발생하는 이벤트에서 `useState` 대신 `useRef`를 사용하여 불필요한 리렌더링을 제거하는 **성능 최적화** 패턴을 배운다.

### 개념 설명

**타이머 ID 관리:**
- `setInterval`이 반환하는 ID는 화면에 보여줄 필요가 없다.
- `useState`에 저장하면 ID가 바뀔 때마다 불필요한 리렌더링이 발생한다.
- `useRef`에 저장하면 리렌더링 없이 ID를 보관/해제할 수 있다.

**고빈도 이벤트 최적화:**
- 비디오 `onTimeUpdate`는 초당 수십 번 발생한다.
- 이 값을 `useState`로 관리하면 컴포넌트 전체가 초당 수십 번 리렌더링된다.
- `useRef`로 관리하면 값만 조용히 갱신하고, 필요한 순간(버튼 클릭)에만 가져다 쓴다.

### 코드 예제

**스마트 타이머 -- 4가지 useRef 통합:**

```jsx
import { useState, useEffect, useRef } from "react";

export default function SmartTimer() {
  const [count, setCount] = useState(0);

  const inputRef = useRef(null);       // DOM 접근: 자동 포커스
  const timerIdRef = useRef(null);     // 값 보관: setInterval ID
  const isFirstRender = useRef(true);  // 첫 렌더링 스킵 플래그
  const prevCountRef = useRef();       // 이전 값 추적

  // 마운트 시 자동 포커스
  useEffect(() => {
    if (inputRef.current) inputRef.current.focus();
  }, []);

  // 데이터 추적 (첫 렌더링 스킵 + 이전 값 비교)
  useEffect(() => {
    if (isFirstRender.current) {
      isFirstRender.current = false;
      return;
    }
    console.log(`현재: ${count}초, 이전: ${prevCountRef.current}초`);
    prevCountRef.current = count;
  }, [count]);

  const startTimer = () => {
    if (timerIdRef.current) return; // 중복 실행 방지
    timerIdRef.current = setInterval(() => {
      setCount(prev => prev + 1);
    }, 1000);
  };

  const stopTimer = () => {
    clearInterval(timerIdRef.current);
    timerIdRef.current = null;
  };

  return (
    <div>
      <input ref={inputRef} placeholder="자동 포커스 메모장" />
      <h1>{count}s</h1>
      <button onClick={startTimer}>시작</button>
      <button onClick={stopTimer}>정지</button>
    </div>
  );
}
```

**비디오 이어보기 -- 고빈도 이벤트 최적화:**

```jsx
import { useRef } from "react";

export default function VideoOptimizer() {
  const videoRef = useRef(null);
  const lastTimeRef = useRef(0); // 재생 시간을 렌더링 없이 저장

  const onTimeUpdate = () => {
    if (videoRef.current) {
      // 초당 수십 번 실행되지만, Ref 수정이므로 리렌더링 없음 (CPU 절약)
      lastTimeRef.current = videoRef.current.currentTime;
    }
  };

  const saveProgress = () => {
    // 결정적인 순간에만 값을 추출한다
    alert(`${Math.floor(lastTimeRef.current)}초 지점을 저장했습니다.`);
  };

  return (
    <div>
      <video ref={videoRef} onTimeUpdate={onTimeUpdate} controls width="100%">
        <source src="https://www.w3schools.com/html/mov_bbb.mp4" type="video/mp4" />
      </video>
      <button onClick={saveProgress}>시청 위치 기록하기</button>
    </div>
  );
}
```

### 코드 해설

- `timerIdRef`: `setInterval`의 반환 ID를 저장한다. `useState`에 담았다면 카운트가 올라갈 때마다 ID 저장 상태 변경으로 불필요한 리렌더링이 추가 발생한다.
- `if (timerIdRef.current) return`: 이미 타이머가 돌고 있으면 중복 생성을 막는 방어 코드다. ref가 없으면 여러 개의 좀비 타이머가 생길 수 있다.
- `lastTimeRef.current = videoRef.current.currentTime`: 이 줄이 `useState`였다면, 비디오 재생 중 댓글창, 추천 리스트 등 모든 형제 컴포넌트가 버벅이며 리렌더링된다.

**데이터 흐름 요약:**
1. **입력**: 브라우저가 `onTimeUpdate` 이벤트를 발생시킨다.
2. **저장**: `lastTimeRef.current`에 조용히 기록한다 (리액트는 모른다).
3. **추출**: 사용자가 '저장' 버튼을 눌러야 비로소 값을 꺼내 사용한다.

### 실무 비유

- 타이머 ID는 "창고 열쇠"와 같다. 화면에 보여줄 필요는 없지만 잃어버리면 타이머를 멈출 수 없다.
- 비디오 재생 시간은 "속기사의 메모장"과 같다. 회의 중(재생 중)에는 빠르게 적어두기만 하고, 회의 끝(버튼 클릭)에 정리된 내용을 공유한다.

### 핵심 포인트

- `setInterval`/`setTimeout`의 ID는 반드시 `useRef`에 보관한다. `useState`에 넣으면 불필요한 리렌더링이 발생하고, 일반 변수에 넣으면 리렌더링 시 유실된다.
- 고빈도 이벤트(`onTimeUpdate`, `onScroll`, `onMouseMove`)의 실시간 값은 `useRef`로 수집하고, 필요한 순간에만 꺼내 쓰는 것이 성능 최적화의 핵심이다.
- 하나의 컴포넌트에서 useRef를 **여러 용도**(DOM 접근, 값 보관, 플래그, 이전값)로 동시에 사용할 수 있다.

## 24강. useRef 실무 패턴 3부 -- 영역 밖 클릭 감지와 외부 라이브러리 통합

### 도입

드롭다운 메뉴, 모달, 툴팁 등에서 "사용자가 영역 밖을 클릭하면 닫기" 기능은 실무 UI의 필수 요소다. 리액트의 `onClick`은 해당 요소 **내부**에서만 동작하므로, **외부** 클릭을 감지하려면 브라우저 전역 이벤트(`document`)와 `useRef`를 결합해야 한다.

### 개념 설명

**Outside Click Detection 원리:**
1. 드롭다운 전체 영역을 `useRef`로 참조한다.
2. `document`에 `mousedown` 이벤트 리스너를 등록한다.
3. 클릭된 위치(`event.target`)가 ref 영역 안에 포함되지 않으면 → 메뉴를 닫는다.
4. 컴포넌트가 언마운트될 때 리스너를 제거하여 메모리 누수를 방지한다.

**State 중심 접근 vs Ref 중심 접근:**

| 비교 항목 | State 중심 접근 | Ref 중심 접근 (정답) |
|-----------|----------------|---------------------|
| 판단 기준 | "값이 바뀌었는가?" | "사용자가 누른 DOM 노드가 내 안에 있는가?" |
| 동작 원리 | 논리적 데이터 비교 | 물리적 DOM 구조 비교 (`contains`) |
| 외부 연동 | 불가능 (리액트 내부만 감시) | 가능 (브라우저 전역 이벤트와 결합) |
| 결과 | 배경에 투명 막을 깔아야 함 | 깔끔한 논리만으로 외부 클릭 제어 가능 |

### 코드 예제

```jsx
import { useState, useEffect, useRef } from "react";

export default function AdvancedDropdown() {
  const [isOpen, setIsOpen] = useState(false);
  const dropdownRef = useRef(null);

  useEffect(() => {
    const handleClickOutside = (event) => {
      // 메뉴 상자가 존재하고, 클릭 지점이 메뉴 상자 안에 포함되지 않으면 닫기
      if (dropdownRef.current && !dropdownRef.current.contains(event.target)) {
        setIsOpen(false);
      }
    };

    // 브라우저 전체에 클릭 감시 장치 설치
    document.addEventListener("mousedown", handleClickOutside);

    // 컴포넌트 언마운트 시 감시 장치 제거 (메모리 누수 방지)
    return () => {
      document.removeEventListener("mousedown", handleClickOutside);
    };
  }, []);

  return (
    <div ref={dropdownRef} style={{ position: "relative", display: "inline-block" }}>
      <button onClick={() => setIsOpen(!isOpen)}>계정 설정</button>

      {isOpen && (
        <div style={{
          position: "absolute", top: "45px", left: "0",
          width: "200px", backgroundColor: "white",
          border: "1px solid #ddd", borderRadius: "8px",
          boxShadow: "0 4px 12px rgba(0,0,0,0.1)", padding: "10px"
        }}>
          <div style={{ padding: "10px", cursor: "pointer" }}>내 프로필</div>
          <div style={{ padding: "10px", cursor: "pointer" }}>결제 수단</div>
          <div style={{ padding: "10px", cursor: "pointer", color: "red" }}>로그아웃</div>
        </div>
      )}
    </div>
  );
}
```

### 코드 해설

- `document.addEventListener("mousedown", ...)`: 리액트의 `onClick`은 컴포넌트 내부에서만 작동한다. "영역 밖"을 알려면 리액트의 통제권을 벗어나 브라우저 전체 이벤트를 가로채야 한다.
- `dropdownRef.current.contains(event.target)`: DOM 표준 API로, 부모 요소가 특정 자식 요소를 포함하고 있는지 확인하는 가장 빠르고 정확한 방법이다.
- `return () => { document.removeEventListener(...) }`: 이 정리(cleanup) 로직이 없으면, 컴포넌트가 화면에서 사라져도 브라우저는 계속 클릭을 감시하며 에러를 발생시킨다.
- 의존성 배열이 `[]`이므로 리스너는 마운트 시 한 번만 등록되고, 언마운트 시 한 번만 제거된다.

### 실무 비유

드롭다운 ref는 "가게의 출입문 센서"와 같다. 센서 범위(ref 영역) 안에서 움직이면 반응하지 않지만, 범위를 벗어나면 자동으로 문을 닫는다(메뉴 닫기).

### 핵심 포인트

- `contains(event.target)`은 클릭 지점이 ref 영역 내부인지 판별하는 핵심 DOM API다.
- 전역 이벤트 리스너는 반드시 `useEffect`의 cleanup 함수에서 제거해야 메모리 누수를 막을 수 있다.
- 이 패턴은 드롭다운, 모달, 팝오버, 검색 자동완성 등 대부분의 오버레이 UI에 범용적으로 적용된다.
- 차트 라이브러리(D3, Chart.js) 등을 사용할 때도 ref에 인스턴스를 보관하면 리렌더링 시 인스턴스가 파괴되지 않고 유지된다.

## 25강. forwardRef -- 내가 만든 컴포넌트에 Ref 날개 달아주기

### 도입

지금까지 ref는 `<input>`, `<video>`, `<div>` 같은 HTML 기본 태그에만 연결했다. 하지만 실무에서는 우리가 직접 만든 **커스텀 컴포넌트**의 내부 DOM을 부모가 제어해야 할 일이 많다. 리액트는 기본적으로 함수형 컴포넌트에 ref 전달을 차단하는데, 이 문을 공식적으로 열어주는 것이 `forwardRef`다.

### 개념 설명

- 리액트 함수형 컴포넌트는 기본적으로 내부를 보여주지 않는 **캡슐화** 상태다.
- `<MyComponent ref={myRef} />`를 그냥 쓰면 "함수형 컴포넌트에는 ref를 줄 수 없습니다" 경고가 발생한다.
- `forwardRef`로 컴포넌트를 감싸면 두 번째 인자로 `ref`를 받을 수 있고, 이를 내부의 원하는 DOM 요소에 연결할 수 있다.

**일반 Props vs forwardRef 비교:**

| 구분 | 일반 Props | forwardRef |
|------|-----------|------------|
| 목적 | 부모가 자식에게 데이터 전달 | 부모가 자식의 물리적 DOM 요소 직접 조작 |
| 흐름 | 단방향 (부모 -> 자식) | 직접 연결 (부모 <-> 자식 내부 DOM) |
| 비유 | 배달 음식을 문 앞에 두고 가기 | 자식이 부모에게 방 열쇠를 복사해주기 |

### 코드 예제

**자식 컴포넌트 -- forwardRef로 통로 개방:**

```jsx
import { forwardRef } from "react";

// forwardRef()로 감싸면 (props, ref) 두 개의 인자를 받는다
const MyInput = forwardRef((props, ref) => {
  return (
    <div>
      <label style={{ display: "block", fontWeight: "bold" }}>{props.label}</label>
      {/* 부모에게 받은 ref를 실제 조종 대상인 <input>에 연결한다 */}
      <input
        ref={ref}
        {...props}
        style={{ padding: "10px", border: "2px solid #3b82f6", borderRadius: "8px" }}
      />
    </div>
  );
});

export default MyInput;
```

**부모 컴포넌트 -- 자식 내부 원격 제어:**

```jsx
import { useRef } from "react";
import MyInput from "./components/MyInput";

export default function App() {
  const idRef = useRef(null);

  const handleFocus = () => {
    if (idRef.current) {
      idRef.current.focus();
      idRef.current.style.backgroundColor = "#eff6ff";
    }
  };

  return (
    <div>
      {/* 커스텀 컴포넌트에 ref를 당당하게 전달한다 */}
      <MyInput ref={idRef} label="사용자 ID" placeholder="아이디를 입력하세요" />
      <button onClick={handleFocus}>자식 컴포넌트 포커스 잡기</button>
    </div>
  );
}
```

### 코드 해설

- `forwardRef((props, ref) => ...)`: 일반 컴포넌트의 `(props) => ...`과 달리 두 번째 인자로 `ref`를 명시적으로 받는다. 이것이 부모와 자식을 연결하는 전용 통로가 된다.
- `<input ref={ref} />`: 부모가 밖에서 `idRef.current.focus()`를 호출하면, 이 통로를 타고 들어온 신호가 실제 `<input>` 요소에 전달되어 포커스가 잡힌다.
- `<MyInput ref={idRef} />`: `forwardRef`로 감싸져 있으므로 에러 없이 ref가 전달된다. 감싸지 않은 컴포넌트에 이렇게 쓰면 경고가 발생한다.

### 실무 비유

`forwardRef`는 호텔의 마스터키 시스템과 같다. 일반 투숙객(props)은 문 앞에 물건을 놓고 가지만, 마스터키(forwardRef)가 있으면 방 안의 에어컨(DOM)을 직접 제어할 수 있다. 다만 모든 방에 마스터키를 발급하면 보안에 문제가 생기므로, 꼭 필요한 경우에만 사용한다.

### 핵심 포인트

- 함수형 컴포넌트에 `ref`를 전달하려면 반드시 `forwardRef`로 감싸야 한다.
- `forwardRef`는 `(props, ref)` 두 개의 인자를 받으며, `ref`를 내부의 원하는 DOM 요소에 연결한다.
- `forwardRef`는 강력하지만 **남발하지 않는다**. 리액트의 기본 원칙은 "상태로 제어하기"이며, 포커스/스크롤 같은 물리적 동작이 꼭 필요한 경우에만 사용한다.
- 디자인 시스템의 공용 입력 컴포넌트, 모달 포커스 트랩 등에서 실무적으로 자주 활용된다.

## 면접 핵심 Q&A

**Q1. `useRef`와 `useState`의 차이를 설명해주세요.**
> `useState`는 값이 변경되면 컴포넌트를 리렌더링하고 변경된 값을 화면에 반영합니다. `useRef`는 `.current` 값이 변경되어도 리렌더링을 유발하지 않으며, 값은 컴포넌트 생애 동안 유지됩니다. UI에 표시해야 하는 값은 `useState`, 내부 로직에만 필요한 값(타이머 ID, 이전 값, 플래그 등)은 `useRef`를 사용합니다.

**Q2. `useRef`로 DOM에 접근하는 과정을 설명해주세요.**
> `useRef(null)`로 참조 객체를 생성하고, JSX 요소의 `ref` 속성에 연결합니다. 리액트가 해당 요소를 렌더링하면 실제 DOM 노드가 `.current`에 자동으로 할당됩니다. 이후 `.current.focus()`, `.current.scrollIntoView()` 등 브라우저 표준 API를 직접 호출할 수 있습니다. 접근 전 반드시 null 체크를 해야 합니다.

**Q3. `useRef`로 이전 상태(previous state)를 추적하는 원리는 무엇인가요?**
> 리액트의 실행 순서를 이용합니다. 렌더링이 먼저 실행되고, `useEffect`는 렌더링 후에 실행됩니다. `useEffect` 안에서 현재 값을 ref에 저장하면, 다음 렌더링의 렌더 단계에서 ref를 읽을 때는 아직 이전 값이 남아있습니다. 이 타이밍 차이를 이용해 "이전 값 vs 현재 값" 비교가 가능해집니다.

**Q4. 영역 밖 클릭(outside click) 감지는 어떻게 구현하나요?**
> 드롭다운 전체 영역을 `useRef`로 참조하고, `useEffect`에서 `document`에 `mousedown` 이벤트 리스너를 등록합니다. 핸들러 안에서 `dropdownRef.current.contains(event.target)`으로 클릭 지점이 ref 영역 내부인지 판별하고, 외부이면 메뉴를 닫습니다. 반드시 cleanup 함수에서 `removeEventListener`로 리스너를 제거해야 메모리 누수를 방지할 수 있습니다.

**Q5. `forwardRef`는 왜 필요하며, 어떻게 사용하나요?**
> 리액트 함수형 컴포넌트는 기본적으로 캡슐화되어 있어 외부에서 ref를 전달할 수 없습니다. `forwardRef`로 컴포넌트를 감싸면 두 번째 인자로 `ref`를 받을 수 있고, 이를 내부의 원하는 DOM 요소에 연결하여 부모가 자식 내부의 DOM을 제어할 수 있게 됩니다. 포커스 관리, 디자인 시스템의 공용 컴포넌트 등에서 사용합니다.

**Q6. `setInterval`의 ID를 `useState` 대신 `useRef`에 저장하는 이유는 무엇인가요?**
> 타이머 ID는 화면에 표시할 값이 아닙니다. `useState`에 저장하면 ID가 할당/해제될 때마다 불필요한 리렌더링이 발생합니다. `useRef`에 저장하면 리렌더링 없이 ID를 안전하게 보관하고, `clearInterval`로 정확하게 해제할 수 있습니다. 일반 `let` 변수는 리렌더링 시 초기화되어 ID를 잃어버리므로 사용할 수 없습니다.

**Q7. `useRef`를 사용할 때 주의해야 할 점은 무엇인가요?**
> 첫째, ref로 DOM의 값(예: `input.value`)을 직접 변경하면 `useState`와 동기화가 깨져 예측 불가능한 버그가 발생합니다. 값은 State로, 동작(포커스/스크롤)은 Ref로 관리해야 합니다. 둘째, `forwardRef`와 직접 DOM 조작은 리액트의 선언적 철학을 해칠 수 있으므로 꼭 필요한 경우에만 사용합니다. 셋째, 전역 이벤트 리스너를 등록했다면 반드시 cleanup에서 제거해야 합니다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| `useRef` | 리렌더링 없이 값을 컴포넌트 생애 동안 유지하는 리액트 Hook. `{ current: 값 }` 객체를 반환한다. |
| `.current` | `useRef`가 반환하는 객체의 유일한 속성. 이 값을 읽거나 수정하여 ref를 사용한다. |
| `ref` 속성 | JSX 요소에 부여하여 해당 DOM 노드를 `useRef` 객체의 `.current`에 연결하는 속성. |
| `forwardRef` | 함수형 컴포넌트가 외부에서 전달받은 `ref`를 내부 DOM 요소에 전달할 수 있게 해주는 리액트 API. |
| `contains()` | DOM 표준 API. 부모 노드가 특정 자식 노드를 포함하고 있는지 boolean으로 반환한다. |
| Outside Click Detection | 특정 영역 밖을 클릭했을 때를 감지하는 UI 패턴. `useRef` + `document` 이벤트 + `contains()`로 구현한다. |
| Early Return | 조건을 만족하지 않으면 함수를 즉시 종료하는 패턴. 첫 렌더링 스킵에서 사용된다. |
| 캡슐화 (Encapsulation) | 컴포넌트 내부 구현을 외부에 노출하지 않는 원칙. `forwardRef`는 이 원칙에 공식적인 예외를 만든다. |
| cleanup 함수 | `useEffect`가 반환하는 함수. 컴포넌트 언마운트 시 또는 다음 Effect 실행 전에 호출되어 정리 작업을 수행한다. |
| 좀비 타이머 | `clearInterval`을 하지 못해 영원히 멈추지 않는 타이머. 타이머 ID를 안전하게 보관하지 않으면 발생한다. |

## 자가 점검 체크리스트

- [ ] `useRef`가 반환하는 객체의 구조(`{ current: 값 }`)를 설명할 수 있다
- [ ] `useState`, `useRef`, `let` 변수의 차이를 리렌더링 관점에서 비교할 수 있다
- [ ] `useRef(null)` + `ref` 속성으로 DOM 요소에 접근하는 과정을 작성할 수 있다
- [ ] ref로 DOM을 조작하기 전 null 체크를 하는 이유를 설명할 수 있다
- [ ] `useEffect`의 실행 타이밍을 이용해 이전 상태를 추적하는 코드를 작성할 수 있다
- [ ] `useRef(true)` 플래그로 첫 렌더링을 스킵하는 패턴을 구현할 수 있다
- [ ] `setInterval` ID를 `useRef`에 저장해야 하는 이유를 설명할 수 있다
- [ ] 고빈도 이벤트에서 `useRef`로 성능을 최적화하는 방법을 이해하고 있다
- [ ] `document` 이벤트 + `contains()`로 영역 밖 클릭을 감지하는 코드를 작성할 수 있다
- [ ] `useEffect` cleanup에서 이벤트 리스너를 제거해야 하는 이유를 설명할 수 있다
- [ ] `forwardRef`가 필요한 상황과 사용 방법을 설명할 수 있다
- [ ] ref로 DOM 값을 직접 변경하면 안 되는 이유(State와 동기화 문제)를 알고 있다
