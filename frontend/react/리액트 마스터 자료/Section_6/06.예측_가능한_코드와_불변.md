# 예측 가능한 코드와 불변

---

## 순수 함수와 순수 컴포넌트

### 순수 함수란

**같은 입력이면 항상 같은 출력**을 반환하고, **외부 상태를 변경하지 않는** 함수다.

```js
// 순수 함수 — 입력이 같으면 출력이 항상 같다
function add(x, y) {
  return x + y; // add(2, 3) → 항상 5
}

// 비순수 함수 — 실행 시점마다 결과가 다르다
function getTime() {
  return new Date().getTime();
}
```

### 순수 컴포넌트란

React 컴포넌트도 함수다. **같은 props와 state가 들어오면 항상 같은 JSX를 반환**해야 한다. React는 순수 컴포넌트를 전제로 Virtual DOM 비교와 렌더링 최적화를 수행한다.

```
(props, state) → 항상 동일한 JSX
```

### 비순수 컴포넌트의 문제

```jsx
// ❌ 비순수 — props를 직접 수정
const products = ["Shoes", "Bag"];

function ProductList({ items }) {
  items.push("Hat"); // 원본 배열 변경 → 부작용 발생
  return <ul>{items.map((i) => <li key={i}>{i}</li>)}</ul>;
}

function Summary({ items }) {
  return <p>총 상품 수: {items.length}</p>;
  // ProductList가 push한 "Hat" 때문에 3이 표시됨
}
```

`ProductList`가 `items`를 직접 수정했기 때문에, 같은 `products`를 받는 `Summary`에서 예기치 못한 결과가 발생한다. 원인을 추적하기 매우 어렵다.

### 순수하게 리팩토링

```jsx
// ✅ 순수 — 외부 데이터를 변경하지 않고 새 값을 계산만 한다
function ProductList({ products, discountRate }) {
  return (
    <ul>
      {products.map((p) => (
        <li key={p.id}>
          {p.name}: {Math.floor(p.price * (1 - discountRate))}원
        </li>
      ))}
    </ul>
  );
}
```

### 전역 변수 사용 vs useState

```jsx
// ❌ 비순수 — 전역 변수를 직접 변경
let formData = {};

function SignUpForm({ initialName }) {
  formData.name = initialName; // 외부 상태 변경
  return <p>이름: {formData.name}</p>;
}
```

```jsx
// ✅ 순수 — React의 지역 상태로 관리
import { useState } from "react";

function SignUpForm({ initialName }) {
  const [name] = useState(initialName);
  return <p>이름: {name}</p>;
}
```

| 패턴 | 문제점 | 해결 |
|------|--------|------|
| `items.push("Hat")` | props 원본 변경 | `[...items, "Hat"]` 새 배열 생성 |
| `formData.name = value` | 전역 변수 변경 | `useState`로 지역 상태 관리 |
| `외부 변수 += 1` | 렌더링마다 누적 | `useState`로 렌더링 간 상태 보존 |

## React.StrictMode

### 역할

**개발 모드에서 컴포넌트를 의도적으로 두 번 렌더링**하여 순수하지 않은 코드(부작용)를 조기에 탐지한다. 프로덕션 빌드에서는 한 번만 렌더링되므로 성능에 영향이 없다.

```jsx
// main.jsx
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App.jsx";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

### StrictMode가 잡아내는 비순수 코드

```jsx
// ❌ 외부 변수를 수정하는 컴포넌트
let external = 0;

function BadComponent() {
  external += 1;
  console.log("external:", external);
  return <p>{external}</p>;
}
// StrictMode 출력:
//   external: 1
//   external: 2   ← 두 번 렌더링으로 값이 누적됨
```

```jsx
// ✅ useState로 안전하게 관리
function GoodComponent() {
  const [name] = useState("홍길동");
  console.log("폼 렌더링:", name);
  return <p>{name}</p>;
}
// StrictMode 출력:
//   폼 렌더링: 홍길동
//   폼 렌더링: 홍길동   ← 두 번 실행해도 값이 동일
```

**핵심 원리:** 컴포넌트를 두 번 호출했을 때 결과가 달라지면, 그 컴포넌트는 순수하지 않다. StrictMode는 이 사실을 개발 중에 드러나게 해 준다.

## 객체 State 불변 업데이트

### 왜 직접 수정하면 안 되는가

React는 State의 **참조(reference)가 바뀌었는지**를 비교하여 리렌더링을 결정한다. 객체를 직접 수정하면 참조가 그대로이므로 React가 변경을 감지하지 못한다.

```jsx
// ❌ 직접 수정 — 같은 참조이므로 렌더링 안 됨
function handleBirthday() {
  user.age += 1;
  setUser(user); // 동일 참조 → 변경 감지 실패
}

// ✅ 새 객체 생성 — 참조가 달라지므로 정상 렌더링
function handleBirthday() {
  setUser({ ...user, age: user.age + 1 });
}
```

| 방식 | 코드 | 참조 변경 | 렌더링 |
|------|------|-----------|--------|
| 직접 수정 (mutation) | `user.age += 1; setUser(user)` | 동일 | 실패 |
| 불변 업데이트 (immutable) | `setUser({ ...user, age: user.age + 1 })` | 새 참조 | 정상 |

### 플랫 객체 — spread operator

```jsx
const [user, setUser] = useState({ name: "철수", age: 20 });

// 특정 필드만 교체
setUser({ ...user, age: user.age + 1 });
// 결과: { name: "철수", age: 21 }
```

### 중첩 객체 — 단계별 spread

중첩 구조에서는 **변경하려는 경로의 모든 단계를 새로 복사**해야 한다.

```jsx
const [user, setUser] = useState({
  name: "영희",
  age: 25,
  address: {
    city: "서울",
    zipcode: "12345",
  },
});

// ❌ 직접 수정 — 참조가 같아 렌더링 안 됨
user.address.city = "부산";
setUser(user);

// ✅ 단계별 spread
setUser({
  ...user,                    // 1단계: user 복사
  address: {
    ...user.address,          // 2단계: address 복사
    city: "부산",              // 바꿀 값만 덮어쓰기
  },
});
```

### 깊은 중첩 — 3단계 이상

```jsx
const [user, setUser] = useState({
  id: 1,
  profile: {
    name: "Alice",
    contact: { phone: "010-1111-2222" },
  },
});

// profile.contact.phone 변경
setUser({
  ...user,
  profile: {
    ...user.profile,
    contact: {
      ...user.profile.contact,
      phone: "010-9999-8888",
    },
  },
});
```

### 여러 중첩 필드를 동시에 변경

```jsx
const [company, setCompany] = useState({
  name: "우리회사",
  contact: { phone: "02-1234-5678", email: "info@wooricorp.com" },
  address: { city: "서울", detail: "강남구 역삼동" },
});

setCompany({
  ...company,
  address: {
    ...company.address,
    city: "부산",
    detail: "해운대구 센텀동",
  },
  contact: {
    ...company.contact,
    phone: "051-987-6543",
  },
});
```

> **규칙:** 변경 경로의 **모든 단계**를 spread로 복사하고, 마지막 단계에서만 값을 덮어쓴다. 건드리지 않는 형제 속성은 그대로 참조를 공유해도 된다.

## 배열 State 불변 업데이트

### 배열에서 금지되는 직접 수정 메서드

| 직접 수정 (금지) | 불변 대안 |
|------------------|-----------|
| `push`, `unshift` | `[...arr, newItem]` |
| `pop`, `shift`, `splice` | `arr.filter()` |
| `arr[i] = x`, `forEach` 수정 | `arr.map()` |
| `sort`, `reverse` | `[...arr].sort()`, `[...arr].reverse()` |

### 추가 — spread operator

```jsx
const [items, setItems] = useState(["공부하기", "운동하기"]);

// ❌ 직접 수정
items.push("청소하기");
setItems(items); // 같은 참조 → 렌더링 안 됨

// ✅ 새 배열 생성
setItems([...items, "청소하기"]);
```

### 삭제 — filter

```jsx
const [todos, setTodos] = useState([
  { id: 1, text: "공부하기" },
  { id: 2, text: "운동하기" },
  { id: 3, text: "청소하기" },
]);

function removeTodo(id) {
  setTodos(todos.filter((todo) => todo.id !== id));
}
```

### 수정 — map

```jsx
const [todos, setTodos] = useState([
  { id: 1, text: "공부하기", done: false },
  { id: 2, text: "운동하기", done: false },
]);

function toggleTodo(id) {
  setTodos(
    todos.map((todo) =>
      todo.id === id ? { ...todo, done: !todo.done } : todo
    )
  );
}
```

### 배열 안 객체의 중첩 필드 수정 — map + 단계별 spread

배열 요소가 중첩 객체를 포함할 때, map으로 대상을 찾고 단계별로 spread한다.

```jsx
const [posts, setPosts] = useState([
  { id: 1, title: "첫 번째 글", author: { name: "철수", email: "chul@example.com" } },
  { id: 2, title: "두 번째 글", author: { name: "영희", email: "young@example.com" } },
]);

function changeAuthorName(id, newName) {
  setPosts(
    posts.map((post) =>
      post.id === id
        ? {
            ...post,              // post 복사
            author: {
              ...post.author,     // author 복사
              name: newName,      // 필요한 값만 덮어쓰기
            },
          }
        : post
    )
  );
}
```

### 조건부 다중 업데이트 — map + 조건 분기

여러 항목을 한 번에 조건에 따라 수정할 때도 map을 사용한다.

```jsx
const [todos, setTodos] = useState([
  { id: 1, text: "공부하기", done: false, tag: "urgent" },
  { id: 2, text: "청소하기", done: false, tag: "normal" },
  { id: 3, text: "운동하기", done: false, tag: "urgent" },
]);

// tag가 "urgent"인 항목만 일괄 완료 처리
function completeUrgent() {
  setTodos(
    todos.map((todo) =>
      todo.tag === "urgent" ? { ...todo, done: true } : todo
    )
  );
}
```

```jsx
const [items, setItems] = useState([
  { id: 1, name: "노트북", price: 1200000 },
  { id: 2, name: "키보드", price: 50000 },
  { id: 3, name: "마우스", price: 30000 },
]);

// 모든 항목에 10% 할인 적용
function applyDiscount() {
  setItems(
    items.map((item) => ({
      ...item,
      price: Math.floor(item.price * 0.9),
    }))
  );
}
```

### 배열 불변 업데이트 패턴 요약

| 작업 | 패턴 | 예시 |
|------|------|------|
| **추가** | `[...arr, new]` | `setItems([...items, newItem])` |
| **삭제** | `arr.filter()` | `setItems(items.filter(i => i.id !== id))` |
| **단일 수정** | `arr.map()` | `setItems(items.map(i => i.id === id ? {...i, done: true} : i))` |
| **조건부 다중 수정** | `arr.map()` + 조건 | `setItems(items.map(i => i.tag === "urgent" ? {...i, done: true} : i))` |
| **중첩 객체 수정** | `arr.map()` + 단계별 spread | `setItems(items.map(i => i.id === id ? {...i, author: {...i.author, name}} : i))` |

## 면접 핵심 Q&A

**Q. React에서 State를 직접 수정하면 안 되는 이유?**
> React는 State의 **참조(reference) 동일성**을 비교하여 리렌더링 여부를 결정한다. `user.age = 21`처럼 직접 수정하면 객체 참조가 바뀌지 않으므로 React가 변경을 감지하지 못해 화면이 갱신되지 않는다. 반드시 `setState`에 **새 객체/배열**을 전달해야 한다.

**Q. 순수 컴포넌트란?**
> 같은 props와 state가 주어지면 **항상 동일한 JSX를 반환**하고, 렌더링 과정에서 **외부 상태를 변경하지 않는** 컴포넌트다. React는 순수 컴포넌트를 전제로 Virtual DOM 비교와 최적화를 수행하므로, props나 전역 변수를 직접 수정하면 예측 불가능한 버그가 발생한다.

**Q. StrictMode의 역할?**
> 개발 모드에서 컴포넌트를 **의도적으로 두 번 렌더링**하여 순수하지 않은 코드를 조기에 탐지한다. 외부 변수를 수정하는 컴포넌트는 두 번 호출 시 결과가 달라지므로 문제가 드러난다. 프로덕션 빌드에서는 비활성화되어 성능에 영향을 주지 않는다.

**Q. 중첩 객체 State를 업데이트하는 방법?**
> 변경 경로의 **모든 단계를 spread로 복사**하고, 마지막 단계에서만 값을 덮어쓴다.
> ```jsx
> setUser({
>   ...user,
>   address: { ...user.address, city: "부산" },
> });
> ```
> 3단계 이상 깊어지면 가독성이 떨어지므로 Immer 라이브러리 도입을 고려할 수 있다.

**Q. 배열 State에서 특정 항목을 수정하는 패턴?**
> `map`으로 새 배열을 만들면서, 대상 항목만 spread로 복사 후 값을 교체한다.
> ```jsx
> setTodos(
>   todos.map((todo) =>
>     todo.id === id ? { ...todo, done: !todo.done } : todo
>   )
> );
> ```
> 추가는 `[...arr, new]`, 삭제는 `filter`, 수정은 `map`이 기본 패턴이다.

## 용어 글로서리

| 용어 | 정의 |
|------|------|
| **순수 함수** | 같은 입력에 항상 같은 출력을 반환하고 외부 상태를 변경하지 않는 함수 |
| **순수 컴포넌트** | 같은 props/state에 항상 같은 JSX를 반환하는 React 컴포넌트 |
| **부작용 (Side Effect)** | 렌더링 과정에서 외부 변수, DOM, 네트워크 등을 변경하는 행위 |
| **StrictMode** | 개발 모드에서 컴포넌트를 두 번 렌더링하여 비순수 코드를 탐지하는 도구 |
| **불변성 (Immutability)** | 기존 데이터를 직접 수정하지 않고 새 복사본을 만들어 교체하는 원칙 |
| **참조 동일성** | `prevObj === nextObj` 비교. React가 렌더링 여부를 판단하는 기준 |
| **얕은 복사 (Shallow Copy)** | 최상위 레벨만 새로 복사하고, 중첩 객체는 참조를 공유하는 복사 방식 |
| **spread operator** | `{...obj}`, `[...arr]` 문법. 얕은 복사를 수행하여 불변 업데이트에 사용 |
| **mutation** | 기존 객체/배열을 직접 변경하는 행위. React State에서는 금지 |
| **Immer** | 직접 수정 문법으로 작성하면 내부적으로 불변 업데이트를 수행해 주는 라이브러리 |

## 자가 점검 체크리스트

- [ ] 순수 함수의 두 가지 조건(같은 입출력, 외부 상태 미변경)을 설명할 수 있다
- [ ] 순수 컴포넌트가 왜 React에서 중요한지 이유를 말할 수 있다
- [ ] props를 직접 수정했을 때 생기는 문제를 구체적으로 설명할 수 있다
- [ ] StrictMode가 비순수 코드를 어떻게 탐지하는지 알고 있다
- [ ] 객체 State를 `{ ...obj, key: newValue }` 패턴으로 업데이트할 수 있다
- [ ] 중첩 객체를 단계별 spread로 불변 업데이트할 수 있다
- [ ] 배열 추가(`spread`), 삭제(`filter`), 수정(`map`) 패턴을 모두 쓸 수 있다
- [ ] 배열 안 객체의 중첩 필드를 `map + spread`로 수정할 수 있다
- [ ] 조건부로 여러 항목을 한 번에 수정하는 다중 업데이트 패턴을 쓸 수 있다
- [ ] 직접 수정(mutation)과 불변 업데이트(immutable)의 차이를 테이블로 비교할 수 있다
