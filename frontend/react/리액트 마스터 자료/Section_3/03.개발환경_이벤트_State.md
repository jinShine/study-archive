## Vite 개발 환경

### 왜 Vite인가

| | CDN + Babel | Vite |
|--|-------------|------|
| **용도** | 학습/원리 체험 | 실무 개발 |
| **HMR** | 없음 (수동 새로고침) | 코드 저장 즉시 반영 |
| **모듈 시스템** | 없음 (전역 스크립트) | ES Modules (import/export) |
| **빌드** | 없음 | 프로덕션 번들링, 트리쉐이킹 |
| **TS 지원** | 수동 설정 | 기본 내장 |

### 프로젝트 생성과 구조

```bash
npm create vite@latest my-app -- --template react
cd my-app && npm install && npm run dev
```

```
my-app/
├─ src/
│  ├─ main.jsx      # 엔트리 포인트 (createRoot + render)
│  ├─ App.jsx        # 루트 컴포넌트
│  └─ components/    # 컴포넌트 파일들
├─ index.html        # Vite가 서빙하는 HTML
├─ vite.config.js    # Vite 설정
└─ package.json
```

### 컴포넌트 파일 생성과 import/export

```jsx
// src/components/Test.jsx — 컴포넌트 정의 + export
const Test = () => {
  return <h2>Test Component</h2>;
};
export default Test;

// src/App.jsx — import해서 사용
import Test from "./components/Test";

function App() {
  return (
    <div>
      <Test />
    </div>
  );
}
export default App;
```

**export 방식 두 가지:**

```jsx
// default export — import 시 이름 자유
export default Test;
import Test from "./Test";
import MyTest from "./Test";  // 다른 이름도 OK

// named export — import 시 이름 일치 필수
export { Test };
import { Test } from "./Test";
```

## 이벤트 핸들링

### HTML vs React 이벤트

```jsx
// HTML — 문자열로 코드 전달
<button onclick="alert('hi')">클릭</button>

// React — 함수 참조를 전달
<button onClick={() => alert('hi')}>클릭</button>
```

| HTML | React | 차이점 |
|------|-------|--------|
| `onclick` | `onClick` | camelCase |
| `"alert('hi')"` | `{handleClick}` | 문자열이 아닌 **함수 참조** |
| 전역 스코프 | 컴포넌트 스코프 | 스코프 격리 |

### onClick 기본 패턴

```jsx
function App() {
  function handleClick() {
    console.log("클릭!");
  }

  // ✅ 함수 참조 전달
  return <button onClick={handleClick}>클릭</button>;

  // ❌ 즉시 실행 — 렌더 시점에 호출되어 버림
  // return <button onClick={handleClick()}>클릭</button>;
}
```

**핵심:** `onClick={handleClick}`은 함수를 **넘기는 것**이고, `onClick={handleClick()}`은 함수를 **호출하는 것**이다. 괄호 유무가 동작을 완전히 바꾼다.

### onChange와 이벤트 객체

```jsx
function App() {
  function handleChange(event) {
    console.log("입력값:", event.target.value);
  }

  return <input type="text" onChange={handleChange} />;
}
```

- React는 브라우저 이벤트를 감싸는 **SyntheticEvent**를 제공
- `event.target` — 이벤트가 발생한 DOM 요소
- `event.target.value` — input의 현재 값
- 모든 이벤트 이름은 **camelCase**: `onChange`, `onClick`, `onSubmit`, `onKeyDown` 등

### 리스트에서 이벤트 + 매개변수 전달

```jsx
const items = ["사과", "바나나", "체리"];

function ItemList() {
  function handleClick(item, index, event) {
    console.log("항목:", item);
    console.log("인덱스:", index);
    console.log("이벤트:", event);
  }

  return (
    <ul>
      {items.map((item, index) => (
        <li
          key={item}
          onClick={(event) => handleClick(item, index, event)}
        >
          {item}
        </li>
      ))}
    </ul>
  );
}
```

**왜 화살표 함수로 감싸야 하는가?**

```jsx
// ❌ 즉시 실행 — 렌더 시점에 handleClick이 호출됨
onClick={handleClick(item, index)}

// ✅ 지연 실행 — 클릭 시점에 handleClick이 호출됨
onClick={(event) => handleClick(item, index, event)}
```

매개변수를 전달하면서 **클릭 시점에만** 실행되게 하려면, 화살표 함수로 감싸서 **새 함수를 만들어 전달**해야 한다.

## State — 변하는 값을 다루는 법

### 일반 변수 vs State

```jsx
// ❌ 일반 변수 — 값이 바뀌어도 화면이 갱신되지 않음
function Counter() {
  let count = 0;
  return (
    <div>
      <p>{count}</p>
      <button onClick={() => { count++; console.log(count); }}>
        +1
      </button>
    </div>
  );
}
// 콘솔에는 1, 2, 3... 찍히지만 화면은 계속 0
```

```jsx
// ✅ useState — 값이 바뀌면 컴포넌트가 다시 렌더링됨
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
```

**왜 일반 변수는 안 되는가?**
React는 `setState`(여기서는 `setCount`)가 호출될 때만 "다시 렌더링해야 한다"는 걸 안다. 일반 변수 변경은 React가 감지하지 못하므로 화면이 갱신되지 않는다.

### useState 구조

```jsx
const [state, setState] = useState(initialValue);
//     현재값  변경함수              초기값
```

- `useState(0)` → 숫자 상태, 초기값 0
- `useState("")` → 문자열 상태, 초기값 빈 문자열
- `useState(false)` → 불리언 상태
- `useState([])` → 배열 상태
- `useState({})` → 객체 상태

### State 변경 → 리렌더링 사이클

```
1. setCount(count + 1) 호출
2. React가 재렌더링을 예약 (배치)
3. Counter 함수가 다시 호출됨
4. useState(0)이 저장된 최신 값(1)을 반환
5. 새 JSX로 VDOM 생성 → diff → DOM 패치
```

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  console.log("렌더링! count:", count); // 매 렌더마다 출력

  return (
    <div>
      <p>{count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
// 클릭할 때마다 "렌더링! count: 1", "렌더링! count: 2"... 출력
```

### Props vs State 비교

| 구분 | Props | State |
|------|-------|-------|
| **관리 주체** | 부모 컴포넌트 | 해당 컴포넌트 자신 |
| **수정 가능?** | 읽기 전용 | `setState`로 변경 가능 |
| **역할** | 외부에서 전달된 설정값 | 내부에서 변화하는 데이터 |
| **렌더링 트리거** | 부모가 새 값을 내려줄 때 | `setState` 호출 시 |

```jsx
// Props — 부모가 값을 정해서 내려줌
function Greeting({ name }) {
  return <h1>안녕하세요, {name}!</h1>;
}

// State — 컴포넌트 스스로 값을 바꿈
function Clicker() {
  const [clicked, setClicked] = useState(false);
  return (
    <div>
      <p>{clicked ? "눌렀습니다!" : "아직 안 눌렀습니다."}</p>
      <button onClick={() => setClicked(true)}>눌러보세요</button>
    </div>
  );
}
```

> Props는 **"밖에서 들어온 값"**, State는 **"안에서 변하는 값"**.
> 둘 다 변경되면 컴포넌트가 다시 렌더링된다.

## 면접 핵심 Q&A

### 이벤트

**Q. React 이벤트와 HTML 이벤트의 차이?**
> React는 **SyntheticEvent**라는 래퍼 객체를 사용한다. 브라우저 간 이벤트 차이를 추상화하고, 이벤트 이름은 camelCase(`onClick`, `onChange`)를 쓴다. HTML은 문자열로 코드를 전달하지만, React는 **함수 참조**를 전달한다.

**Q. `onClick={handleClick}`과 `onClick={handleClick()}`의 차이?**
> `handleClick`은 함수를 **전달**하는 것이고, `handleClick()`은 **즉시 호출**하는 것이다. 괄호를 붙이면 렌더 시점에 함수가 실행되어 의도와 다른 동작이 된다. 매개변수를 넘기고 싶으면 `() => handleClick(arg)`처럼 화살표 함수로 감싸야 한다.

**Q. SyntheticEvent란?**
> React가 브라우저 네이티브 이벤트를 감싸는 **크로스 브라우저 래퍼 객체**다. 모든 브라우저에서 동일한 인터페이스를 제공한다. `event.target`, `event.preventDefault()`, `event.stopPropagation()` 등을 동일하게 사용할 수 있다.

**Q. 리스트 항목 클릭 시 해당 항목 데이터를 핸들러에 전달하는 방법?**
> map 안에서 화살표 함수로 감싸서 전달한다:
> ```jsx
> {items.map((item, i) => (
>   <li key={item.id} onClick={(e) => handleClick(item, i, e)}>
>     {item.name}
>   </li>
> ))}
> ```

### State

**Q. useState가 뭔가요?**
> 함수 컴포넌트에서 **상태를 선언하고 관리하는 Hook**이다. `const [value, setValue] = useState(초기값)` 형태로 사용하며, `setValue`를 호출하면 React가 컴포넌트를 다시 렌더링한다.

**Q. 일반 변수 대신 useState를 써야 하는 이유?**
> 일반 변수(`let count = 0`)를 바꿔도 React는 **변경을 감지하지 못해 화면이 갱신되지 않는다**. `useState`의 setter 함수를 호출해야 React가 "상태가 바뀌었다"는 걸 인지하고 리렌더링을 트리거한다.

**Q. Props와 State의 차이?**
> Props는 **부모가 전달하는 읽기 전용 값**이고, State는 **컴포넌트 자신이 소유하고 변경할 수 있는 값**이다. 둘 다 변경되면 리렌더링이 일어나지만, Props는 자식이 수정할 수 없고 State는 `setState`로 직접 변경한다.

**Q. setState 호출 후 바로 상태를 읽으면?**
> **이전 값**이 출력된다. `setState`는 비동기적으로 처리되며, 호출 직후가 아니라 **다음 렌더링 시점**에 새 값이 반영된다.
> ```jsx
> setCount(count + 1);
> console.log(count); // 아직 이전 값!
> ```

**Q. 함수형 컴포넌트와 클래스 컴포넌트에서 상태 관리 차이?**
> 클래스 컴포넌트는 `this.state`와 `this.setState()`를 사용하고, 함수형 컴포넌트는 `useState` Hook을 사용한다. React 팀은 함수형 + Hooks를 권장하며, 새 코드에서 클래스를 쓸 이유는 거의 없다.

### 개발 환경

**Q. Vite는 뭔가요? CRA와의 차이는?**
> Vite는 **ES Modules 기반 빌드 도구**로, 개발 서버 시작이 매우 빠르다. CRA(Create React App)는 Webpack 기반으로 프로젝트가 커질수록 느려진다. 현재 React 공식 문서도 Vite를 권장한다.

**Q. HMR(Hot Module Replacement)이 뭔가요?**
> 코드를 수정하면 **전체 새로고침 없이 변경된 모듈만 교체**하는 기능이다. 상태를 유지한 채 UI 변경을 즉시 반영하므로 개발 생산성이 높아진다.

**Q. default export와 named export의 차이?**
> `export default`는 파일당 하나, import 시 이름을 자유롭게 지정할 수 있다. `export { Name }`은 여러 개 가능하지만, import 시 **정확한 이름**(또는 `as` 별칭)을 써야 한다.

## 용어 글로서리

| 용어 | 정의 |
|------|------|
| **Vite** | ES Modules 기반 빌드 도구. 빠른 개발 서버와 HMR 제공 |
| **HMR** | Hot Module Replacement. 코드 수정 시 변경된 모듈만 교체 |
| **import/export** | ES Modules 문법. 컴포넌트를 파일 단위로 분리·조합 |
| **SyntheticEvent** | React의 크로스 브라우저 이벤트 래퍼 객체 |
| **event.target** | 이벤트가 발생한 DOM 요소. `event.target.value`로 input 값 접근 |
| **useState** | 함수 컴포넌트에서 상태를 선언하는 Hook. `[value, setValue]` 반환 |
| **State** | 컴포넌트 내부의 변할 수 있는 데이터. 변경 시 리렌더링 트리거 |
| **리렌더링** | State/Props 변경 시 컴포넌트 함수가 다시 호출되어 새 UI를 생성하는 과정 |
