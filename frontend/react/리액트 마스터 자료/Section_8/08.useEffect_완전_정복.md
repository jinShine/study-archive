# useEffect 완전 정복

> React에서 "화면 밖의 세계"와 소통하는 유일한 공식 창구.
> 데이터 패칭, 구독, 타이머, DOM 직접 조작 등 **사이드 이펙트**를 안전하게 다루는 방법을 정리한다.

---

## 1. useEffect 기본 구조 -- 왜 필요한가

### Side Effect(부수효과)란?

React 컴포넌트의 본업은 **props/state -> JSX 반환** 이다.
이 "순수한 렌더링" 바깥에서 일어나는 모든 작업이 사이드 이펙트다.

| 순수 렌더링 (OK)          | 사이드 이펙트 (useEffect 필요)       |
| ------------------------- | ------------------------------------ |
| props/state 읽기          | 서버 데이터 요청 (fetch/axios)       |
| JSX 반환                  | DOM 직접 조작 (document.title 등)    |
| 파생 값 계산              | 타이머 등록 (setTimeout/setInterval) |
|                           | 이벤트 리스너 등록/해제              |
|                           | 외부 라이브러리 초기화               |

### 기본 문법

```jsx
import { useEffect } from "react";

useEffect(() => {
  // --- setup: 렌더링 후 실행할 코드 ---

  return () => {
    // --- cleanup: 다음 effect 직전 또는 unmount 시 실행 ---
  };
}, [의존성1, 의존성2]);
```

**핵심 실행 타이밍**: 렌더링 -> DOM 커밋(화면 반영) -> useEffect 콜백 실행.
즉, 화면이 먼저 그려진 "뒤"에 effect가 돌아간다.

## 2. 의존성 배열 3가지 패턴

의존성 배열(두 번째 인자)이 useEffect의 **재실행 조건**을 결정한다.

| 패턴 | 문법 | 실행 시점 | 대표 용도 |
| ---- | ---- | --------- | --------- |
| 배열 생략 | `useEffect(() => { ... })` | 매 렌더링 직후(mount + 모든 re-render) | 디버그 로깅, 렌더 측정 |
| 빈 배열 `[]` | `useEffect(() => { ... }, [])` | 마운트 시 1회, cleanup은 unmount 시 1회 | 초기 API 요청, 이벤트 등록 |
| 특정 값 `[dep]` | `useEffect(() => { ... }, [dep])` | dep이 변경될 때마다 | 검색어 변경 시 재요청, 필터 변경 |

### 2-1. 배열 생략 -- 매 렌더마다 실행

```jsx
function RenderLogger() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    console.log("렌더링 발생 -- count:", count);
  }); // deps 없음 -> 매 렌더마다

  return <button onClick={() => setCount(c => c + 1)}>+1 ({count})</button>;
}
```

- setState를 effect 안에서 무조건 호출하면 **무한 루프** 위험.
- 네트워크 요청 등 비용이 큰 작업에는 부적합. 로깅/측정 용도로만 사용 권장.

### 2-2. 빈 배열 `[]` -- 마운트 시 1회

```jsx
function Modal({ onClose }) {
  useEffect(() => {
    const handleEsc = (e) => {
      if (e.key === "Escape") onClose();
    };
    window.addEventListener("keydown", handleEsc);

    return () => window.removeEventListener("keydown", handleEsc);
  }, []); // 마운트 1회 등록, unmount 시 해제

  return <div className="modal">ESC로 닫기</div>;
}
```

- Class 컴포넌트의 `componentDidMount` / `componentWillUnmount` 를 대체.
- 빈 배열이므로 React가 "의존성 변화 없음"으로 판단 -> 재실행하지 않는다.

### 2-3. 특정 값 `[dep]` -- 해당 값 변경 시

```jsx
function SearchBox() {
  const [keyword, setKeyword] = useState("");

  useEffect(() => {
    if (keyword === "") return;
    console.log(`"${keyword}" 검색 실행`);
    // fetch(`/api/search?q=${keyword}`) ...
  }, [keyword]); // keyword가 바뀔 때만 실행

  return (
    <input
      value={keyword}
      placeholder="검색어 입력"
      onChange={(e) => setKeyword(e.target.value)}
    />
  );
}
```

- 여러 값을 넣을 수도 있다: `[keyword, category, sort]` -> 셋 중 하나라도 바뀌면 재실행.
- React는 `Object.is`로 이전/현재 값을 비교한다. 동일하면 effect를 건너뛴다.

## 3. cleanup 함수 -- 왜 필요하고 언제 실행되는가

### cleanup이 없으면?

| 문제 상황 | 결과 |
| --------- | ---- |
| setInterval을 등록만 하고 해제 안 함 | 컴포넌트가 사라져도 타이머가 계속 실행 -> **메모리 누수** |
| addEventListener만 하고 remove 안 함 | 이벤트 핸들러가 쌓임 -> **중복 실행** |
| fetch 요청을 취소하지 않음 | 이미 언마운트된 컴포넌트에 setState 시도 -> **경고/버그** |

### cleanup 실행 시점 (2가지)

1. **다음 effect가 실행되기 직전** -- 이전 effect의 리소스를 정리한 뒤 새 effect 실행.
2. **컴포넌트 unmount 시** -- 화면에서 완전히 사라질 때 마지막 정리.

```
[mount]
  -> effect 실행 (setup)
[re-render]
  -> cleanup 실행 (이전 effect 정리)
  -> effect 실행 (새 setup)
[unmount]
  -> cleanup 실행 (마지막 정리)
```

> StrictMode(개발 모드)에서는 mount -> unmount -> 재mount가 한 번 더 일어나
> effect/cleanup 로그가 두 번 보일 수 있다. 프로덕션 빌드에서는 1회만 실행된다.

## 4. 실전 패턴

### 4-1. 타이머 (setInterval)

```jsx
function Timer() {
  const [seconds, setSeconds] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setSeconds(s => s + 1);
    }, 1000);

    return () => clearInterval(id); // 반드시 해제
  }, []); // 마운트 시 1회 등록

  return <p>경과: {seconds}초</p>;
}
```

**포인트**: cleanup에서 `clearInterval`을 빠뜨리면 컴포넌트 재렌더링마다 타이머가 중복 생성된다.

### 4-2. 이벤트 리스너 (window event)

```jsx
function ScrollTracker() {
  const [scrollY, setScrollY] = useState(0);

  useEffect(() => {
    const handleScroll = () => setScrollY(window.scrollY);
    window.addEventListener("scroll", handleScroll);

    return () => window.removeEventListener("scroll", handleScroll);
  }, []); // 마운트 시 등록, unmount 시 해제

  return <p>스크롤 위치: {scrollY}px</p>;
}
```

**포인트**: 등록(addEventListener)과 해제(removeEventListener)에 **동일한 함수 참조**를 사용해야 한다.

### 4-3. AbortController (fetch 요청 취소)

```jsx
function ProductList() {
  const [products, setProducts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    async function fetchData() {
      try {
        const res = await fetch("https://fakestoreapi.com/products", {
          signal: controller.signal,
        });
        const data = await res.json();
        setProducts(data);
      } catch (err) {
        if (err.name === "AbortError") {
          console.log("요청이 정상 취소됨");
          return; // 취소는 에러가 아님
        }
        setError("데이터 로딩 실패");
      } finally {
        setLoading(false);
      }
    }

    fetchData();

    return () => controller.abort(); // cleanup에서 요청 취소
  }, []);

  if (loading) return <p>로딩 중...</p>;
  if (error) return <p>{error}</p>;
  return <ul>{products.map(p => <li key={p.id}>{p.title}</li>)}</ul>;
}
```

**포인트**:
- `useEffect` 콜백 자체를 `async`로 만들 수 없다. 내부에 별도 async 함수를 선언하고 호출한다.
- cleanup에서 `controller.abort()`를 호출하면, 진행 중인 fetch가 `AbortError`를 던진다.
- 이를 catch에서 분기하여 "정상 취소"와 "진짜 에러"를 구분한다.

## 5. useEffect 실행 순서 시각화

```
컴포넌트 라이프사이클과 useEffect 흐름
========================================

[최초 마운트]
  |
  v
 render()  -->  가상 DOM 생성
  |
  v
 DOM 커밋  -->  실제 화면 반영
  |
  v
 useEffect(setup) 실행          <-- "화면이 그려진 뒤" 실행
  |
  |
[state/props 변경 -> 리렌더링]
  |
  v
 render()  -->  가상 DOM 재생성
  |
  v
 DOM 커밋  -->  화면 업데이트
  |
  v
 cleanup() 실행                 <-- 이전 effect 정리
  |
  v
 useEffect(setup) 실행          <-- 새 effect 실행
  |
  |
[컴포넌트 언마운트]
  |
  v
 cleanup() 실행                 <-- 마지막 정리
  |
  v
 컴포넌트 제거 완료


의존성 배열별 실행 타이밍 비교
========================================

 deps 없음        []              [count]
 (매 렌더)        (마운트 1회)     (count 변경 시)
    |                |                |
 mount -> run     mount -> run     mount -> run
    |                |                |
 render -> run    render -> skip   count 변경 -> run
    |                |                |
 render -> run    render -> skip   다른값 변경 -> skip
    |                |                |
 unmount->clean   unmount->clean   unmount -> clean
```

## 면접 핵심 Q&A

### Q. useEffect는 언제 실행되는가?

**A.** 컴포넌트가 렌더링되고 **DOM 커밋(실제 화면 반영)이 완료된 직후** 비동기적으로 실행된다.
브라우저가 paint를 마친 뒤에 호출되므로, 렌더링을 블로킹하지 않는다.
(화면에 즉시 반영이 필요한 경우는 `useLayoutEffect`를 사용한다.)

### Q. 의존성 배열의 역할과 3가지 패턴?

**A.** 의존성 배열은 "이 effect를 언제 다시 실행할지"를 React에 알려주는 장치이다.
- **생략**: 매 렌더링 직후 실행. 디버그 로깅 용도.
- **빈 배열 `[]`**: 마운트 시 1회 실행, unmount 시 cleanup 1회. 초기 설정/구독 용도.
- **`[dep1, dep2]`**: 지정된 값이 `Object.is` 비교로 변경되었을 때만 재실행. 가장 일반적인 패턴.

### Q. cleanup 함수는 왜 필요한가?

**A.** 이전 effect가 등록한 리소스(타이머, 이벤트 리스너, 네트워크 요청 등)를 해제하기 위함이다.
cleanup이 없으면 리소스가 누적되어 **메모리 누수**, **중복 실행**, **언마운트 후 state 업데이트 시도** 같은 버그가 발생한다.
cleanup은 (1) 다음 effect 실행 직전, (2) 컴포넌트 unmount 시 호출된다.

### Q. useEffect에서 async/await를 직접 쓸 수 없는 이유?

**A.** useEffect의 콜백은 `undefined` 또는 **cleanup 함수**를 반환해야 한다.
`async` 함수는 항상 `Promise`를 반환하므로, React가 이를 cleanup 함수로 인식하지 못한다.
해결법: effect 내부에 별도의 async 함수를 선언하고 즉시 호출한다.

```jsx
// 잘못된 방식
useEffect(async () => { ... }, []);       // Promise 반환 -> cleanup 동작 불가

// 올바른 방식
useEffect(() => {
  async function fetchData() { ... }
  fetchData();
}, []);
```

### Q. AbortController의 역할?

**A.** 진행 중인 `fetch` 요청을 **프로그래밍 방식으로 취소**하는 Web API이다.
useEffect의 cleanup에서 `controller.abort()`를 호출하면:
1. 컴포넌트가 언마운트되거나 의존성이 바뀌어 재요청할 때, 이전 요청을 즉시 취소한다.
2. **경쟁 상태(race condition)** 를 방지한다 -- 느린 이전 응답이 최신 응답을 덮어쓰는 문제.
3. 취소된 요청은 `AbortError`를 던지며, 이를 catch에서 분기 처리한다.

## 용어 글로서리

| 용어 | 설명 |
| ---- | ---- |
| Side Effect | 렌더링(JSX 반환) 외의 모든 외부 작업. API 호출, DOM 조작, 구독 등 |
| 의존성 배열 (deps) | useEffect의 두 번째 인자. effect 재실행 조건을 결정하는 값의 배열 |
| cleanup 함수 | useEffect 콜백이 반환하는 함수. 리소스 정리 담당 |
| mount | 컴포넌트가 DOM에 최초로 삽입되는 시점 |
| unmount | 컴포넌트가 DOM에서 제거되는 시점 |
| re-render | state/props 변경으로 컴포넌트가 다시 렌더링되는 것 |
| commit | React가 가상 DOM 비교 후 실제 DOM에 변경을 반영하는 단계 |
| AbortController | 진행 중인 fetch 요청을 프로그래밍 방식으로 취소하는 Web API |
| race condition | 여러 비동기 요청의 응답 순서가 꼬여 잘못된 데이터가 반영되는 문제 |
| StrictMode | 개발 모드에서 effect를 두 번 실행하여 cleanup 누락을 감지하는 React 기능 |

## 자가 점검 체크리스트

- [ ] useEffect가 렌더링 "전"이 아니라 "후"에 실행된다는 것을 설명할 수 있다
- [ ] 의존성 배열 3가지 패턴(생략 / 빈 배열 / 특정 값)의 차이를 코드 없이 설명할 수 있다
- [ ] cleanup 함수가 실행되는 2가지 시점을 말할 수 있다
- [ ] setInterval을 useEffect에서 등록하고 cleanup에서 해제하는 코드를 작성할 수 있다
- [ ] addEventListener/removeEventListener 쌍을 useEffect로 관리할 수 있다
- [ ] useEffect 콜백에 async를 직접 붙이면 안 되는 이유를 설명할 수 있다
- [ ] AbortController로 fetch 요청을 취소하는 패턴을 작성할 수 있다
- [ ] AbortError를 catch에서 분기 처리하는 이유를 설명할 수 있다
- [ ] StrictMode에서 effect가 두 번 실행되는 이유를 설명할 수 있다
- [ ] 무한 루프가 발생하는 useEffect 안티패턴을 식별하고 수정할 수 있다
