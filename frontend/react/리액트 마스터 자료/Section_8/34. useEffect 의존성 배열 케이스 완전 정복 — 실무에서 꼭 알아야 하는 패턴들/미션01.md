# 🟦 useEffect 의존성 배열 케이스 완전 정복 실습

## 📝 설명

이번 미션에서는 `useEffect`의 세 가지 케이스를 직접 체험하며 차이를 눈으로 확인합니다.
이 세 가지는 React 개발에서 가장 자주 쓰이는 패턴으로, 정확히 이해해야 실무에서 불필요한 렌더링, 중복 요청, 이벤트 누수 같은 버그를 막을 수 있습니다.

## 🎯 미션 목표

아래 세 개의 버튼을 눌러 각 케이스별 차이를 직접 관찰하세요.

- Case 1: 두 번째 인자를 생략 → 모든 렌더링마다 실행
- Case 2: 빈 배열 `[]` → 마운트 시 한 번만 실행
- Case 3: `[count]` → `count`가 바뀔 때만 실행

## 🧱 Base Code

```jsx
import { useState, useEffect } from "react";

export default function EffectCases() {
  const [count, setCount] = useState(0);
  const [mode, setMode] = useState(1);

  useEffect(() => {
    // TODO: mode에 따라 실행 패턴을 다르게 만들기
  }, []);

  return (
    <div>
      <h2>useEffect 의존성 배열 실습</h2>
      <p>현재 count: {count}</p>
      <div>
        <button onClick={() => setCount(count + 1)}>+1</button>
      </div>
      <div>
        <button onClick={() => setMode(1)}>🌀 Case 1 (모든 렌더링)</button>
        <button onClick={() => setMode(2)}>🎯 Case 2 (처음 한 번)</button>
        <button onClick={() => setMode(3)}>🔁 Case 3 (count 변화 시)</button>
      </div>
      <p>현재 선택된 케이스: {mode}</p>
    </div>
  );
}
```

## 📌 미션의 의의

- Case 1에서는 렌더링이 발생할 때마다 effect가 실행됩니다.
- Case 2에서는 컴포넌트가 화면에 처음 붙을 때 한 번만 실행됩니다.
- Case 3에서는 `count`가 바뀔 때마다 effect가 실행되고, 실행 전 cleanup이 작동합니다.

이 차이를 눈으로 확인하는 것이 목표입니다.
React의 렌더링 타이밍, cleanup 순서, 의존성 감시 메커니즘을 완벽히 이해하게 됩니다.
