# 04강. 명확한 행동 설계 -- 실무처럼 액션 흐름을 구성하는 사고방식

## 도입

단순한 INCREASE를 넘어, **"얼마만큼(amount)"** 혹은 **"무엇으로(newSide)"** 바꿀지 데이터를 포함한 액션을 처리하는 방법을 학습한다. 상태를 객체(Object)로 관리하고, 데이터 화물인 **Payload**를 실어 나르는 실무 패턴을 익힌다.

## 개념 설명

### Payload (데이터 화물)

dispatch가 택배 기사님이라면, `type`은 송장에 적힌 "배송 요망"이라는 요청 사항이고, `payload`는 박스 안에 담긴 실제 물건이다. 단순히 "가져다줘"가 아니라 **"사과 10개를 가져다줘"**라고 구체적으로 명령하는 것이다.

### 불변성 유지 (`{ ...state }`)

리듀서는 기존 상태를 직접 수정하면 안 된다. 반드시 기존 상태를 쫙 펼쳐서(`...`) 복사본을 만든 뒤, 바뀌어야 할 부분만 덮어쓰기 해서 반환해야 한다. 그래야 리액트가 "어? 값이 바뀌었네!" 하고 화면을 다시 그려준다.

### switch 문과 default

많은 액션을 처리할 때 `if-else`보다 가독성이 좋다. 마지막 `default: return state;`는 일종의 안전장치로, 정의되지 않은 명령이 들어왔을 때 앱이 멈추지 않고 현재 상태를 유지하게 한다.

## 코드 예제

```jsx
import { useReducer } from 'react';

/**
 * [운영 매뉴얼: Reducer]
 * 실무에서는 조건이 많아지므로 if 대신 switch 문을 표준으로 사용한다.
 */
function cafeteriaReducer(state, action) {
  switch (action.type) {
    case 'ADD_RICE': {
      // 중괄호 블록: 이 안에서 변수를 선언해도 다른 case와 이름이 충돌하지 않는다.
      const addedAmount = action.payload.amount; // 액션에 실려온 화물(Payload) 확인
      return {
        ...state,
        rice: state.rice + addedAmount // 기존 상태를 복사하고 rice만 업데이트
      };
    }

    case 'REFILL_SOUP':
      return {
        ...state,
        soup: state.soup + action.payload.amount
      };

    case 'CHANGE_SIDE':
      return {
        ...state,
        side: action.payload.newSide // 문자열 데이터 화물을 받아 반환
      };

    default:
      // 매뉴얼에 없는 엉뚱한 요청이 오면 '현재 상태'를 그대로 반환하여 에러를 방지
      return state;
  }
}

export default function AdvancedCafeteria() {
  // 초기 상태를 객체로 정의 (밥, 국, 반찬 재고 관리)
  const initialState = {
    rice: 20,
    soup: 30,
    side: "김치",
  };

  const [state, dispatch] = useReducer(cafeteriaReducer, initialState);

  return (
    <div>
      <h1>중앙 관리 급식실 (Advanced)</h1>

      <section>
        <p>밥 재고: <strong>{state.rice}</strong>인분</p>
        {/* payload라는 이름의 객체에 데이터를 실어서 보낸다 */}
        <button onClick={() => dispatch({ type: 'ADD_RICE', payload: { amount: 10 } })}>
          밥 10인분 추가 조리
        </button>
      </section>

      <section>
        <p>국 재고: <strong>{state.soup}</strong>인분</p>
        <button onClick={() => dispatch({ type: 'REFILL_SOUP', payload: { amount: 5 } })}>
          국 5인분 추가
        </button>
      </section>

      <section>
        <p>오늘의 반찬: <strong>{state.side}</strong></p>
        <button onClick={() => dispatch({ type: 'CHANGE_SIDE', payload: { newSide: "메추리알장조림" } })}>
          반찬 교체 (장조림)
        </button>
      </section>
    </div>
  );
}
```

## 코드 해설

| 요소 | 설명 |
|------|------|
| `action.payload.amount` | 액션에 실려온 데이터 화물. "밥 10인분"처럼 구체적인 수치를 전달 |
| `case 'ADD_RICE': { }` | switch case 안에서 중괄호 블록을 사용하면 변수 선언 시 다른 case와 이름 충돌 방지 |
| `{ ...state, rice: ... }` | 불변성 유지: 기존 상태를 복사하고 변경할 부분만 덮어쓰기 |
| `CHANGE_SIDE` | payload에 숫자뿐 아니라 문자열도 담을 수 있음 |
| `default: return state` | 안전장치: 정의되지 않은 액션이 들어와도 앱이 멈추지 않음 |

## 실무 비유

dispatch가 **택배 기사님**이라면:
- `type`은 송장에 적힌 "배송 요망"이라는 요청 사항
- `payload`는 박스 안에 담긴 **실제 물건**

단순히 "가져다줘"가 아니라 "사과 10개를 가져다줘"라고 구체적으로 명령하는 것이다. 리듀서(물류센터 직원)는 송장(type)을 보고 어떤 처리를 할지 결정하고, 박스(payload)를 열어 구체적인 데이터를 확인한다.

## 핵심 포인트

- **Payload**는 액션에 실려 가는 구체적인 데이터이다 (수량, 이름 등)
- switch 문의 case 안에서 **중괄호 블록 `{ }`**을 사용하면 변수 선언이 다른 case와 충돌하지 않는다
- `{ ...state }`로 **불변성**을 유지해야 리액트가 변화를 감지한다
- `default: return state`는 알 수 없는 액션에 대한 **안전장치**
- 상태를 객체로 관리하면 밥, 국, 반찬 등 여러 데이터를 하나의 리듀서로 통합할 수 있다
