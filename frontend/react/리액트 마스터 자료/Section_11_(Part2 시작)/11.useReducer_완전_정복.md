## 01강. 왜 useReducer가 필요한가

### 도입

useState로 장바구니를 만들면 `items`, `totalPrice`, `isLoading` 등 상태가 흩어진다.
한 곳을 수정하면서 다른 곳을 깜빡하면 **상태 불일치 버그**가 발생한다.

### 문제 코드 — useState의 파편화

```jsx
const [items, setItems] = useState([...]);
const [totalPrice, setTotalPrice] = useState(230000);

const handleRemoveItem = (id, price) => {
  setItems((prev) => prev.filter((item) => item.id !== id));
  setTotalPrice((prev) => prev - price);
  // ⚠️ setTotalPrice를 깜빡하면? → 리스트는 줄었는데 금액은 그대로!
};
```

### 핵심 포인트

- **로직의 파편화**: 여러 setState를 동기화해야 하는 부담.
- 상태가 늘어날수록(할인율, 배송비, 포인트 등) 핸들러가 괴물 코드가 된다.

### 실무 비유

> useState 방식은 **각자도생 팀 프로젝트**와 같다. 공유 문서 없이 각자 일하다가 한 명이 전화를 못 받으면(코드를 깜빡하면) 프로젝트는 산으로 간다.

## 02강. useReducer 첫걸음 — 행동 중심 상태 변화를 처음 경험하기

### 도입

흩어져 있던 setState들을 하나의 **reducer 함수(중앙 규칙서)** 로 통합한다.
사용자의 행동을 `dispatch`로 전달하는 구조를 처음 경험한다.

### 코드 예제

```jsx
import { useReducer } from 'react';

const initialState = {
  items: [
    { id: 1, name: "고성능 키보드", price: 150000, qty: 1 },
    { id: 2, name: "무선 마우스", price: 80000, qty: 1 }
  ],
  totalPrice: 230000,
  discount: 10000
};

function cartReducer(state, action) {
  switch (action.type) {
    case 'INCREASE_QTY':
      return {
        ...state,
        items: state.items.map((item) =>
          item.id === action.payload.id
            ? { ...item, qty: item.qty + 1 }
            : item
        ),
        totalPrice: state.totalPrice + action.payload.price,
        discount: Math.max(0, state.discount - 500)
      };
    default:
      return state;
  }
}

export default function ReducerCartPage() {
  const [state, dispatch] = useReducer(cartReducer, initialState);

  return (
    <ul>
      {state.items.map(item => (
        <li key={item.id}>
          {item.name} (수량: {item.qty})
          <button onClick={() => dispatch({
            type: 'INCREASE_QTY',
            payload: { id: item.id, price: item.price }
          })}>+ 수량 추가</button>
        </li>
      ))}
    </ul>
  );
}
```

### 코드 해설

- **reducer 함수**: 컴포넌트 외부에 위치해도 OK. 오직 (현재 상태, 액션)만 보고 새 상태를 계산하는 **순수 함수**.
- **dispatch**: 컴포넌트 내부에 계산 로직이 없다. "어떤 일이 일어났는지(Action)"만 보고한다.
- **dispatch의 안정성**: 리렌더링되어도 dispatch의 참조가 유지되어 성능 최적화에 유리하다.

### 실무 비유

> 집 안 곳곳에 흩어진 전등 스위치를 **거실 중앙 제어 패널**로 옮긴 것. 버튼(Action) 하나만 누르면 리듀서가 알아서 조절한다.

## 03강. useReducer 구조 완전 이해 — state, action, reducer를 생활 비유로

### 개념 구조

```
const [state, dispatch] = useReducer(reducer, initialState);
```

| 요소 | 역할 | 비유 |
|------|------|------|
| **State** | 현재 상태값 | 화이트보드에 적힌 재고 수량 |
| **Action** | 무엇을 할지 담은 객체 `{ type: '...' }` | "밥 1인분 추가!"라고 적힌 포스트잇 |
| **Dispatch** | 액션을 리듀서에게 전달하는 함수 | 포스트잇을 관리자에게 전달하는 팔 |
| **Reducer** | (state, action) → 새 state 반환 | 포스트잇을 보고 화이트보드를 업데이트하는 관리자 |

### 코드 예제 — 급식실 카운터

```jsx
function cafeteriaReducer(state, action) {
  switch (action.type) {
    case 'INCREASE': return state + 1;
    case 'DECREASE': return state > 0 ? state - 1 : 0;
    default: return state;
  }
}

export default function CafeteriaPage() {
  const [count, dispatch] = useReducer(cafeteriaReducer, 0);

  return (
    <div>
      <p>현재 남은 밥: {count}인분</p>
      <button onClick={() => dispatch({ type: 'INCREASE' })}>밥 추가</button>
      <button onClick={() => dispatch({ type: 'DECREASE' })}>밥 배식</button>
    </div>
  );
}
```

### 핵심 포인트

- reducer는 **순수 함수**: 기존 state를 직접 수정하지 않고 새 값을 반환한다.
- `default: return state`는 안전장치 — 정의되지 않은 액션이 와도 앱이 멈추지 않는다.

## 04강. 명확한 행동 설계 — 실무처럼 액션 흐름을 구성하는 사고방식

### 도입

단순한 `INCREASE`를 넘어, **"얼마만큼(amount)"** 또는 **"무엇으로(newSide)"** 바꿀지 데이터를 포함한 액션을 처리한다.

### 코드 예제 — payload 패턴

```jsx
function cafeteriaReducer(state, action) {
  switch (action.type) {
    case 'ADD_RICE':
      return { ...state, rice: state.rice + action.payload.amount };
    case 'REFILL_SOUP':
      return { ...state, soup: state.soup + action.payload.amount };
    case 'CHANGE_SIDE':
      return { ...state, side: action.payload.newSide };
    default:
      return state;
  }
}

// 사용
dispatch({ type: 'ADD_RICE', payload: { amount: 10 } });
dispatch({ type: 'CHANGE_SIDE', payload: { newSide: "메추리알장조림" } });
```

### 코드 해설

- **payload(데이터 화물)**: `type`은 송장에 적힌 요청 사항, `payload`는 박스 안에 담긴 실제 물건.
- **`{ ...state }`(불변성 유지)**: 복사본을 만들고 바뀌어야 할 부분만 덮어써야 React가 변화를 감지한다.
- **switch 문 + default**: if-else보다 가독성이 좋고, default는 안전장치 역할.

## 05강. 거대해진 reducer 분리하기 — 기능별 모듈화의 첫 실전

### 도입

리듀서가 커지면 한 파일에 두지 않고 **상수(Constants) · 액션 생성 함수(Actions) · 리듀서(Reducer)** 를 각각 분리한다.

### 폴더 구조

```
src/
├── store/
│   ├── cafeteriaConstants.js   ← 액션 이름 상수
│   ├── cafeteriaActions.js     ← 액션 크리에이터
│   └── cafeteriaReducer.js     ← 리듀서 함수
├── pages/
│   └── ModularCafeteria.jsx    ← UI 컴포넌트
```

### 코드 예제

```js
// cafeteriaConstants.js
export const ACTIONS = {
  ADD_RICE: "ADD_RICE",
  REFILL_SOUP: "REFILL_SOUP",
  CHANGE_SIDE: "CHANGE_SIDE"
};
```

```js
// cafeteriaActions.js — 요청서 자동 작성 함수
import { ACTIONS } from './cafeteriaConstants';

export const addRice = (amount) => ({ type: ACTIONS.ADD_RICE, amount });
export const refillSoup = (amount) => ({ type: ACTIONS.REFILL_SOUP, amount });
export const changeSide = (newSide) => ({ type: ACTIONS.CHANGE_SIDE, newSide });
```

```jsx
// 컴포넌트에서 사용
dispatch(addRice(10));        // dispatch({ type: 'ADD_RICE', amount: 10 })
dispatch(changeSide("장조림")); // dispatch({ type: 'CHANGE_SIDE', newSide: '장조림' })
```

### 코드 해설

- **Action Creator**: 함수 호출 한 번으로 규격화된 요청서를 생성. 내부 구조를 몰라도 된다.
- **Constants**: `"ADD_RICE"` 직접 타이핑 → 오타 시 에러 안 남. 상수 사용 → 오타 시 즉시 감지.
- **모듈화**: 주방(컴포넌트)은 넓고 깨끗해지고, 운영 지침서(로직)는 서재(store 폴더)에.

## 06강. 유지보수 가능한 Reducer 아키텍처

### 도입

실무 수준의 대규모 프로젝트에서도 흔들리지 않는 **기능 중심 폴더 구조**를 설계한다.

### 폴더 구조

```
src/reducers/
├── rice/
│   ├── riceTypes.js        ← 액션 타입 상수
│   ├── riceActions.js      ← 액션 생성 함수
│   └── riceReducer.js      ← 밥 전용 리듀서
├── soup/
│   ├── soupTypes.js
│   ├── soupActions.js
│   └── soupReducer.js
└── rootReducer.js           ← 중앙 본부
```

### 코드 예제 — rootReducer

```js
import { riceReducer } from './rice/riceReducer';
// import { soupReducer } from './soup/soupReducer';

export function rootReducer(state, action) {
  return {
    rice: riceReducer(state.rice, action),
    // soup: soupReducer(state.soup, action),
  };
}
```

### 코드 해설

- **rootReducer**: 직접 계산하지 않는다. "밥 관련이네? 밥 리듀서 너가 처리해"라고 **배분**만 한다.
- **관심사 분리**: 리듀서가 아무리 커져도 각 기능 폴더만 찾아가면 된다.
- **네이밍 규칙**: `riceReducer.js`, `riceActions.js` — 파일 이름만 보고 역할을 알 수 있다.

## 07강. 장바구니 로직 구현 — 상태 전이 흐름 설계

### 도입

장바구니의 복잡한 상태 변화를 상수-액션-리듀서 분리 패턴으로 설계한다.

### 4가지 핵심 액션

```js
export const ADD_ITEM = "ADD_ITEM";       // 상품 추가
export const INCREMENT = "INCREMENT";     // 수량 +1
export const DECREMENT = "DECREMENT";     // 수량 -1
export const REMOVE_ITEM = "REMOVE_ITEM"; // 상품 삭제
```

### 리듀서 핵심 로직

```js
export function cartReducer(state, action) {
  switch (action.type) {
    case types.ADD_ITEM:
      return { ...state, items: [...state.items, action.payload] };

    case types.INCREMENT:
      return {
        ...state,
        items: state.items.map(item =>
          item.id === action.payload
            ? { ...item, quantity: item.quantity + 1 }
            : item
        )
      };

    case types.DECREMENT:
      return {
        ...state,
        items: state.items.map(item =>
          item.id === action.payload
            ? { ...item, quantity: Math.max(1, item.quantity - 1) }
            : item
        )
      };

    case types.REMOVE_ITEM:
      return {
        ...state,
        items: state.items.filter(item => item.id !== action.payload)
      };

    default:
      return state;
  }
}
```

### 코드 해설

- **`items.map(...)`**: 기존 배열을 직접 수정하지 않고, 조건에 맞는 항목만 교체하여 새 배열 반환.
- **`Math.max(1, ...)`**: 수량이 0이나 음수가 되지 않도록 방어하는 비즈니스 로직.
- **`filter`**: 해당 ID가 아닌 상품들만 남겨 삭제를 구현.

## 08강. useReducer 장바구니 직접 구현하기 — 설계를 코드로 옮기는 시간

### 도입

데이터(`initialCart.js`), 로직(`cartReducer.js`), UI(`CartItem.jsx`)를 명확히 분리하여 실전 장바구니를 구현한다.

### 폴더 구조

```
src/
├── data/initialCart.js          ← 초기 데이터
├── reducers/cartReducer.js      ← 상태 변화 규칙
├── components/CartItem.jsx      ← 개별 상품 UI
└── CartApp.jsx                  ← 전체 조립
```

### 코드 예제 — 자식 컴포넌트

```jsx
export function CartItem({ item, onIncrement, onDecrement, onRemove }) {
  return (
    <div>
      <p>{item.name} - {item.price.toLocaleString()}원 (수량: {item.quantity})</p>
      <button onClick={onIncrement}>+</button>
      <button onClick={onDecrement}>-</button>
      <button onClick={onRemove}>삭제</button>
    </div>
  );
}
```

### 코드 해설

- **CartItem은 상태를 직접 바꾸지 않는다.** 부모가 준 함수(`onIncrement` 등)를 실행만 한다.
- **dispatch는 부모(CartApp)에서만** 호출 → 상태 흐름이 단방향으로 명확하다.

### 실무 비유

> dispatch는 식당의 벨(Bell). 손님(버튼 클릭)이 벨을 누르면서 "추가요!"라고 외치면, 주방장(reducer)이 레시피에 따라 목록(state)을 업데이트하고, 홀 서빙(UI)에게 넘긴다.

## 09강. useReducer 마무리 — 구조를 다듬고 실전 패턴으로 정리하기

### 도입

if 문 → **switch-case**로, 직접 객체 생성 → **Action Creator**로 정돈하여 실무 표준 패턴을 완성한다.

### 실무 표준 패턴

```js
// actions/cartActions.js
export const ADD = "add";
export const REMOVE = "remove";
export const INCREMENT = "increment";
export const DECREMENT = "decrement";

export const addItem = (item) => ({ type: ADD, item });
export const removeItem = (id) => ({ type: REMOVE, id });
export const increment = (id) => ({ type: INCREMENT, id });
export const decrement = (id) => ({ type: DECREMENT, id });
```

```jsx
// 컴포넌트에서 사용 — 읽기 좋은 코드
<CartItem
  key={p.id}
  item={p}
  onIncrement={() => dispatch(increment(p.id))}
  onDecrement={() => dispatch(decrement(p.id))}
  onRemove={() => dispatch(removeItem(p.id))}
/>
```

### 핵심 포인트

- **컴포넌트 파일에 복잡한 로직이 사라졌다.** 화면 구성만 남는다.
- **Action Creator 패턴**: `dispatch(increment(id))`는 키오스크 버튼을 누르는 것과 같다.
- **switch-case**: if-else 체인보다 가독성이 좋고, 각 case를 독립적으로 관리할 수 있다.

## 면접 핵심 Q&A

### Q. useState와 useReducer의 차이는?

> **A.** useState는 단일 값을 관리하기 적합하고, useReducer는 **여러 상태가 연관되어 복합적으로 변해야 할 때** 적합하다. useReducer는 상태 변화 로직을 reducer 함수에 집중시켜 **예측 가능성**과 **유지보수성**을 높인다.

### Q. reducer가 순수 함수여야 하는 이유는?

> **A.** reducer는 (현재 상태, 액션) → 새 상태를 반환하는 **순수 함수**여야 한다. 같은 입력에 항상 같은 결과를 보장해야 React가 상태 변화를 **정확히 감지**하고, 디버깅과 테스트가 용이해진다. 부수 효과(API 호출 등)는 reducer 밖에서 처리한다.

### Q. Action Creator를 사용하는 이유는?

> **A.** 컴포넌트가 액션 객체의 구체적인 구조(`{ type, payload }`)를 알 필요 없이, 함수 호출만으로 규격화된 액션을 생성한다. 오타 방지, 코드 일관성, 리팩터링 용이성이 향상된다.

### Q. rootReducer 패턴이란?

> **A.** 여러 기능별 리듀서(riceReducer, soupReducer 등)를 하나로 합치는 중앙 본부 역할의 함수다. 각 리듀서에게 상태의 일부를 위임하고 결과를 취합하여 **관심사 분리**를 실현한다.

### Q. 불변성(Immutability)을 지켜야 하는 이유는?

> **A.** React는 상태 객체의 **참조(reference)** 가 바뀌었는지로 변화를 감지한다. `...state`로 복사본을 만들어 새 객체를 반환해야 React가 "값이 바뀌었구나"라고 인식하고 화면을 다시 그린다. 원본을 직접 수정하면 참조가 같아서 리렌더링이 일어나지 않는다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **useReducer** | 복잡한 상태 로직을 reducer 함수로 통합 관리하는 React Hook |
| **Reducer** | (현재 상태, 액션) → 새 상태를 반환하는 순수 함수 |
| **Action** | 어떤 일이 일어났는지 기술하는 객체 `{ type, payload? }` |
| **Dispatch** | 액션을 리듀서에게 전달하는 함수 |
| **Payload** | 액션에 실려 보내는 데이터 화물 |
| **Action Creator** | 액션 객체를 생성하는 헬퍼 함수 |
| **Action Types (Constants)** | 문자열 오타 방지를 위해 상수로 정의한 액션 이름 |
| **rootReducer** | 여러 기능별 리듀서를 합치는 중앙 본부 함수 |
| **불변성 (Immutability)** | 기존 상태를 직접 수정하지 않고 복사본을 만들어 반환하는 원칙 |
| **순수 함수** | 같은 입력에 항상 같은 출력, 부수 효과 없는 함수 |
