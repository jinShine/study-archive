비즈니스 로직을 가로막는 '상태의 노이즈'
안녕하세요! 지난 시간 우리는 TanStack Query가 데이터의 생애주기를 얼마나 우아하게 관리하는지 살펴보았습니다. 하지만 이 편리함이 당연하게 느껴질 때쯤, 우리는 반드시 과거로 돌아가 우리가 왜 이 도구를 선택했어야만 했는지 그 근본적인 고통을 되짚어봐야 합니다.

오늘 이 시간에는 라이브러리의 도움 없이 우리가 수동으로 로딩과 에러 상태를 관리했을 때, 우리의 소중한 코드가 어떻게 서서히 오염되고 망가져 가는지 그 처참한 실상을 마주해 보겠습니다.

🏛️ 아키텍처 원칙: '용접된 코드'의 문제점
명령형 오염: 데이터가 '무엇'인지 선언하는 대신, 로딩 불을 켜고(true) 끄는(false) '어떻게'에 매몰됩니다.
전역 저장소의 정체성 상실: 성스러운 비즈니스 금고(Redux, Zustand)에 isUserLoading 같은 UI용 전등 스위치가 가득 차게 됩니다. 귀중품 금고에 일회용 젓가락을 쑤셔 넣는 것과 같습니다.
강한 결합(Tight Coupling): 비동기 통신과 UI 업데이트 로직이 서로 용접되어 있어, 순수한 기능만 떼어내어 테스트하는 것이 불가능해집니다.
🚀 스텝 바이 스텝 가이드: 수동 관리 랩(Lab) 구축
🧪 Step 1. 가짜 API 정의 (src/api/mockApi.ts)
실습을 위해 의도적으로 지연시간과 에러 가능성을 포함한 API를 설계합니다.

export interface UserData {
  id: number;
  name: string;
}

// 수동 관리의 고통을 체감하기 위한 지연 API
export const api = {
  get: async <T>(url: string): Promise<{ data: T }> => {
    console.log(`📡 [Network] ${url} 요청 시작...`);
    return new Promise((resolve, reject) => {
      setTimeout(() => {
        // 10% 확률로 에러 발생 시뮬레이션
        if (Math.random() < 0.1) {
          reject(new Error("서버 연결이 원활하지 않습니다."));
        }
        resolve({ data: { id: 1, name: "과거의 나 (수동 관리)" } as T });
      }, 1500);
    });
  }
};

💡 코드 상세 설명:

인터페이스 정의:UserData를 통해 데이터 규격을 정의합니다.
지연 및 에러 시뮬레이션: 1.5초의 지연과 10%의 에러 확률을 주어, isLoading과 error 상태가 UI에서 어떻게 처리되는지 관찰할 수 있게 합니다.

🌋 Step 4. 지옥의 보일러플레이트 (src/components/ManualUserFetcher.tsx)
비즈니스 로직보다 '청소 로직'이 더 비대해진 모습입니다.

import { useState, useEffect } from 'react';
import { api } from '../api/mockApi';
import type { UserData } from '../api/mockApi'; // ✅ Type-only import 적용

export default function ManualUserFetcher() {
  // 1. 비즈니스 데이터와 UI 상태를 위해 각각 타입을 지정한 3개의 상태가 필요합니다.
  const [data, setData] = useState<UserData | null>(null);
  const [isLoading, setIsLoading] = useState<boolean>(false);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    const fetchData = async () => {
      // 2. 요청 시작 시 로딩 스피너를 켜고 이전 에러를 초기화하는 수동 제어 단계입니다.
      setIsLoading(true);
      setError(null);

      try {
        const result = await api.get<UserData>('/user');
        setData(result.data);
      } catch (err) {
        // 3. 에러 발생 시 타입을 확인하여 안전하게 상태에 저장합니다.
        setError(err instanceof Error ? err : new Error('Unknown Error'));
      } finally {
        // 4. 통신이 마무리되면 결과와 상관없이 로딩 스위치를 수동으로 내려줍니다.
        setIsLoading(false);
      }
    };
    fetchData();
  }, []);

  if (isLoading) return <div>⌛ 수동으로 로딩 불을 켜두었습니다... (isLoading: true)</div>;
  if (error) return <div style={{ color: 'red' }}>❌ 에러 발생: {error.message}</div>;

  return (
    <div style={{ border: '1px solid #ccc', padding: '1rem' }}>
      <h4>유저 정보</h4>
      <p>이름: {data?.name}</p>
    </div>
  );
}

💡 코드 상세 설명:

import type: 런타임에 필요 없는 인터페이스는 타입 전용으로 가져와 번들 크기를 줄이고 구조를 명확히 합니다.
명령형 상태 관리:setIsLoading(true/false)를 직접 호출하는 과정에서 누락이 발생하면 UI 버그로 직결됩니다.
🏎️ Step 5. 경합 조건 방어 로직 (src/components/SafeUserDetail.tsx)
배달 사고를 막기 위해 덕지덕지 추가된 '무시 플래그'입니다.

import { useState, useEffect } from 'react';
import { api } from '../api/mockApi';
import type { UserData } from '../api/mockApi'; // ✅ Type-only import 적용

export default function SafeUserDetail({ id }: { id: number }) {
  const [data, setData] = useState<UserData | null>(null);

  useEffect(() => {
    // 1. 현재 요청이 여전히 유효한지 추적하는 boolean 타입의 '무시 플래그'입니다.
    let isCurrentRequest = true;

    const fetchDetail = async () => {
      const result = await api.get<UserData>(`/detail/${id}`);

      // 2. 배달 사고 방지: 오직 클린업 함수가 실행되기 전인 '유효한' 요청일 때만 업데이트합니다.
      // 늦게 온 이전 응답이 최신 데이터를 덮어쓰지 못하게 막습니다.
      if (isCurrentRequest) {
        setData(result.data);
      }
    };

    fetchDetail();

    // 3. 클린업 함수: 다음 요청이 시작되거나 컴포넌트가 사라질 때 "방금 요청은 이제 무효야"라고 선언합니다.
    return () => { isCurrentRequest = false; };
  }, [id]);

  return (
    <div style={{ background: '#f0f0f0', padding: '1rem' }}>
      <h4>상세 정보 (Race Condition 방어 중)</h4>
      <p>{data ? data.name : '데이터를 가져오는 중...'}</p>
    </div>
  );
}

💡 코드 상세 설명:

isCurrentRequest: 비동기 응답의 순서가 뒤바뀌는 것을 막기 위한 수동 장치입니다. 비즈니스 로직의 순수성을 해치는 대표적인 '노이즈'입니다.
🏁 최종 테스트 케이스 (Test Cases)
수동 관리 방식이 얼마나 깨지기 쉬운지(Fragile) 직접 체험해 보세요.

로딩 스위치 누락 테스트
수행:ManualUserFetcher의 finally 블록 내 setIsLoading(false)를 주석 처리해 봅니다.
결과: 데이터가 도착했음에도 화면은 영원히 로딩 중인 '좀비 상태'가 됩니다.
분석: 상태 제어의 책임이 온전히 개발자의 손끝에 달려 있을 때 발생하는 전형적인 인적 오류입니다.
경합 조건(Race Condition) 테스트
수행:SafeUserDetail에서 isCurrentRequest 로직을 제거한 뒤, ID를 아주 빠르게 연속해서 바꿔봅니다.
결과: 네트워크 속도에 따라 내가 마지막으로 선택한 ID가 아닌, 이전 ID의 데이터가 화면에 남는 역행 현상을 목격합니다.
분석: 비동기 응답 순서가 보장되지 않는 환경에서 발생하는 치명적인 UX 버그입니다.
전역 저장소 오염 확인
수행: (상상 테스트) 만약 위 상태들을 Redux나 Zustand로 옮겼다고 가정해 보세요.
결과:isUserLoading, isUserError, isPostLoading 등 수십 개의 boolean 스위치가 금고(Store)를 가득 채우게 됩니다.
분석: 도메인 데이터와 UI 상태가 뒤섞여 유지보수가 불가능해지는 지점입니다.
