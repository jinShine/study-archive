# 22. TanStack Query 완전 정복

## 개요

Section_22는 리액트 애플리케이션에서 **서버 상태(Server State)** 관리의 패러다임 전환을 다룹니다. 기존 `useEffect` 기반 데이터 페칭의 한계(Race Condition, 중복 요청, 로딩/에러 상태 수동 관리)에서 출발하여, **TanStack Query(React Query)**를 통해 "데이터를 가져오는 것이 아니라 서버와 동기화하는 것"이라는 새로운 사고 모델을 정립합니다.

58강부터 80강까지 총 23개 강의를 통해, Query Key 설계, Stale-While-Revalidate 전략, Suspense/Error Boundary 연동, Mutation과 Invalidation, 낙관적 업데이트, 무한 스크롤, Prefetching, 오프라인 지원까지 서버 상태 관리의 A-Z를 실무 수준으로 학습합니다.

## 강의별 핵심 요약

### 58강. useEffect 페칭의 재앙 — Race Condition, 중복 요청, 무분별한 로딩

- `useEffect` + `fetch` 조합의 구조적 문제: Race Condition, 중복 네트워크 요청, 무분별한 로딩 상태
- AbortController로 수동 취소하더라도 보일러플레이트가 지나치게 증가
- 캐싱, 재시도, 에러 핸들링을 모두 수동 관리하는 것은 비현실적
- 이 문제들이 TanStack Query 도입의 필연적 이유가 됨

### 59강. TanStack Query — 데이터를 가져오는 게 아니라 서버와 동기화하는 것이다

- **핵심 철학**: 데이터 페칭이 아닌 **서버 상태 동기화**
- `useQuery`의 3가지 필수 요소: `queryKey`(바코드), `queryFn`(지침서), 옵션(정책)
- 자동 캐싱, 자동 재시도, 자동 중복 요청 제거가 내장
- `isPending`, `isError`, `data` — 선언적 상태 관리

### 60강. Query Key Factory 설계 전략 — 오타 방지와 체계적인 캐시 주소 관리

- Query Key는 캐시의 **고유 주소(바코드)**
- Factory 패턴으로 키를 계층적으로 설계: `userKeys.all → details() → detail(id)`
- 오타 방지, 자동완성 지원, 일관된 무효화(Invalidation) 가능
- `as const`로 타입 안전성 확보

### 61강. [Pain] 동일 데이터를 5개 컴포넌트에서 호출할 때 발생하는 5번의 네트워크 요청

- 동일 API를 여러 컴포넌트에서 호출하면 **중복 네트워크 요청** 발생
- props drilling이나 전역 상태로 해결하려면 복잡도가 급증
- TanStack Query 없이는 "동일 데이터 = 1번만 요청"을 보장하기 어려움

### 62강. [Solution] Stale-While-Revalidate — 사용자에게는 캐시를, 뒤에서는 갱신을

- **SWR 전략**: 캐시 데이터 즉시 제공 → 백그라운드 갱신
- 동일 `queryKey`로 동시 요청해도 **네트워크 요청은 1번만** 발생
- `staleTime`(신선도) vs `gcTime`(보관 기간) 구분
- Query Key Factory 패턴으로 캐시 키 체계적 관리

### 63강. useQuery의 생애주기 — Fresh, Stale, Inactive 상태의 완벽 이해

- **Fresh**: staleTime 이내, 추가 요청 없음
- **Stale**: staleTime 초과, 백그라운드 갱신 트리거
- **Inactive**: 구독 컴포넌트 없음, gcTime 후 가비지 컬렉션
- 상태 전이 이해가 최적화의 핵심

### 64강. [Pain] 수동으로 관리하던 전역 로딩/에러 상태가 로직을 오염시킬 때

- `isLoading`, `error` 상태를 전역으로 관리하면 비즈니스 로직이 상태 관리 코드에 묻힘
- 여러 API 호출의 로딩/에러를 조합하면 복잡도가 기하급수적으로 증가
- 선언적 해결이 필요하다는 동기 부여

### 65강. [Solution] Suspense & Error Boundary 연동 — 선언적 데이터 페칭

- `useSuspenseQuery`로 Suspense와 자연스럽게 연동
- **"어떻게(how)"가 아니라 "무엇(what)"에 집중**하는 선언적 패턴
- ErrorBoundary로 에러 처리를 컴포넌트 트리 수준에서 분리
- 로딩/에러 UI를 비즈니스 로직과 완전히 분리

### 66강. useSuspenseQueries로 Waterfall 현상 격파하기

- 순차적 Suspense 호출은 **Waterfall(폭포)** 현상 유발
- `useSuspenseQueries`로 여러 쿼리를 **병렬 실행**
- "스테이크와 파스타를 동시에 조리하라" — 병렬 페칭의 핵심 비유
- 총 로딩 시간 = max(쿼리A, 쿼리B)로 단축

### 67강. useMutation — 서버 데이터를 수정하고 즉시 반영하기

- `useMutation`: 서버 데이터 **변경(POST/PUT/DELETE)** 전용 훅
- `mutate()` / `mutateAsync()` 사용법
- `onSuccess`, `onError`, `onSettled` 콜백으로 후처리
- 관찰자(useQuery)에서 행동대장(useMutation)으로 역할 전환

### 68강. [Pain] 수정 후 화면을 바꾸기 위해 reload()를 부르는 허무함

- 데이터 수정 후 `window.location.reload()`로 새로고침하는 안티패턴
- 사용자 경험 파괴, 상태 초기화, 불필요한 네트워크 비용
- "서버의 시간은 흘렀지만, 브라우저의 시간은 멈춰있다"

### 69강. [Solution] Invalidation — 명령 한 줄로 특정 데이터만 정밀하게 새로고침하기

- `queryClient.invalidateQueries()`로 특정 캐시만 무효화
- Query Key의 계층 구조를 활용한 정밀 타겟팅
- "상한 데이터는 버리고, 신선한 데이터를 즉시 배달하라"
- Mutation의 `onSuccess`에서 자동 무효화 패턴

### 70강. 낙관적 업데이트(Optimistic Updates) — 서버 응답 전 화면을 미리 바꾸는 UX

- **서버 응답을 기다리지 않고** UI를 먼저 업데이트
- `onMutate`에서 캐시 직접 수정 → 실패 시 `onError`에서 롤백
- "성공할 것을 믿고, 화면부터 지른다"
- 좋아요, 장바구니 추가 등 즉각적 피드백이 필요한 UX에 필수

### 71강. [Pain] 무한 스크롤 구현 시 꼬여버리는 페이지 번호와 데이터 배열 관리

- 페이지 번호 관리, 데이터 병합, 중복 방지를 수동으로 처리하면 버그 다발
- "데이터는 쌓여가고, 내 머릿속은 복잡해진다"
- 스크롤 위치 복원, 에러 시 재시도 등 엣지케이스 처리 어려움

### 72강. [Solution] useInfiniteQuery — 1만 개의 리스트를 부드럽게 이어 붙이는 기술

- `useInfiniteQuery`로 페이지네이션 자동 관리
- `getNextPageParam`으로 다음 페이지 URL/번호 자동 추출
- `fetchNextPage()` + `hasNextPage`로 무한 스크롤 구현
- "배열 노가다는 끝났다, 이제는 펼치기만 하라"

### 73강. Prefetching 전략 — 사용자의 발걸음보다 한발 앞서 데이터를 준비하는 기술

- `queryClient.prefetchQuery()`로 사용자 액션 전에 데이터 미리 로드
- hover, 마우스 진입 등의 이벤트에서 prefetch 트리거
- "최고의 로딩 UI는 로딩 UI가 없는 것이다"
- 페이지 이동 전 데이터를 미리 캐시에 적재

### 74강. [Pain] 서버 검색 결과의 깜빡임 현상 — 레이아웃 이동(Layout Shift)의 고통

- 필터/검색 변경 시 이전 결과 사라짐 → 로딩 → 새 결과 표시 = **깜빡임**
- Layout Shift로 UX 품질 저하
- "데이터가 바뀌는 찰나, UI는 요동친다"

### 75강. [Solution] placeholderData — 끊김 없는 필터링 UI 완성

- `placeholderData: keepPreviousData`로 이전 데이터를 유지하며 새 데이터 로드
- 깜빡임 없이 자연스러운 전환 UX 구현
- "새 데이터가 올 때까지, 이전 데이터는 자리를 비우지 않는다"
- `isPlaceholderData` 플래그로 현재 데이터가 임시인지 구분 가능

### 76강. 데이터 변환의 정석 — select 옵션으로 컴포넌트 맞춤형 데이터 가공

- `select` 옵션으로 서버 응답을 컴포넌트에 맞게 변환
- "서버는 재료를 보내고, 프론트엔드는 요리를 한다"
- API 레이어와 UI 레이어의 관심사 분리
- 동일 쿼리에서 컴포넌트별로 다른 형태의 데이터 추출

### 77강. 의존적 쿼리(Dependent Queries) — 순차적 비동기 호출의 우아한 처리

- `enabled` 옵션으로 쿼리 실행 조건 제어
- A 쿼리 결과가 있어야 B 쿼리 실행 — 의존성 체이닝
- "비동기의 연쇄 고리를 끊어내고 질서를 세우다"
- `enabled: !!userId`로 값이 있을 때만 쿼리 활성화

### 78강. [Pain] 인터넷이 끊겼을 때 앱이 멈추는 현상 — 오프라인 데이터 관리의 한계

- 네트워크 끊김 → fetch 실패 → 에러 → 앱 사용 불가
- "코드는 완벽해도 네트워크는 완벽할 수 없다"
- 오프라인 환경에서도 캐시된 데이터로 동작해야 하는 요구사항

### 79강. [Solution] Offline Support & Persist — 오프라인에서도 돌아가는 견고한 앱

- `persistQueryClient`로 캐시를 localStorage/IndexedDB에 영속화
- 앱 재시작 시 캐시 복원으로 즉시 데이터 표시
- `onlineManager`로 네트워크 상태 감지 및 자동 재시도
- "알맹이는 박제했다, 이제 껍데기만 살리면 완벽하다"

### 80강. React Compiler

- React Compiler는 빌드 타임에 자동 메모이제이션을 삽입하는 차세대 도구
- `useMemo`, `useCallback` 없이도 컴파일러가 최적화 자동 처리
- 정적 분석 → 자동 메모이제이션 → 세밀한 리렌더링 제어
- 아직 실험 단계이지만, 수동 최적화의 종말을 예고

## 핵심 개념 비교 테이블

### useEffect 페칭 vs TanStack Query

| 항목 | useEffect + fetch | TanStack Query |
|------|-------------------|----------------|
| 캐싱 | 수동 구현 필요 | 자동 (queryKey 기반) |
| 중복 요청 제거 | 불가능 | 자동 (동일 key 병합) |
| Race Condition | AbortController 수동 관리 | 내장 처리 |
| 재시도 | 수동 구현 | 자동 (retry 옵션) |
| 로딩/에러 상태 | useState로 수동 관리 | isPending, isError 자동 제공 |
| 데이터 갱신 | 수동 refetch / reload | invalidateQueries 한 줄 |
| 오프라인 지원 | 없음 | persistQueryClient |

### Query 상태 생애주기

| 상태 | 조건 | 동작 |
|------|------|------|
| **Fresh** | staleTime 이내 | 캐시 즉시 반환, 네트워크 요청 없음 |
| **Stale** | staleTime 초과 | 캐시 반환 + 백그라운드 갱신 트리거 |
| **Fetching** | 네트워크 요청 중 | 로딩 표시 또는 이전 데이터 유지 |
| **Inactive** | 구독 컴포넌트 없음 | gcTime 후 캐시 삭제 |

### 주요 훅 비교

| 훅 | 용도 | 특징 |
|----|------|------|
| `useQuery` | 데이터 **조회** (GET) | 자동 캐싱, 자동 갱신 |
| `useSuspenseQuery` | 조회 + Suspense 연동 | 로딩 상태 분리, 선언적 |
| `useSuspenseQueries` | 여러 쿼리 병렬 조회 | Waterfall 방지 |
| `useMutation` | 데이터 **변경** (POST/PUT/DELETE) | onSuccess에서 Invalidation |
| `useInfiniteQuery` | 무한 스크롤 / 페이지네이션 | 자동 페이지 관리 |

## 면접 대비 Q&A

### Q1. TanStack Query의 핵심 철학은 무엇인가요?
**A:** "데이터를 가져오는(fetching) 것이 아니라 서버와 동기화(synchronizing)하는 것"입니다. TanStack Query는 서버 상태를 클라이언트에 캐싱하고, staleTime/gcTime을 기반으로 자동으로 신선도를 관리하며, 필요한 시점에 백그라운드에서 갱신합니다.

### Q2. staleTime과 gcTime의 차이를 설명해주세요.
**A:** `staleTime`은 캐시된 데이터가 "신선함"을 유지하는 시간으로, 이 기간 동안은 동일 쿼리에 대해 네트워크 요청을 하지 않습니다. `gcTime`(Garbage Collection Time)은 모든 구독 컴포넌트가 언마운트된 후 캐시를 메모리에 보관하는 시간입니다. gcTime이 지나면 캐시가 완전히 삭제됩니다.

### Q3. Query Key Factory 패턴을 왜 사용하나요?
**A:** 캐시 키를 문자열로 직접 작성하면 오타, 불일치, 유지보수 문제가 발생합니다. Factory 패턴으로 계층적 키 구조(`userKeys.all → details() → detail(id)`)를 만들면 타입 안전성, IDE 자동완성, 체계적 Invalidation이 가능해집니다.

### Q4. 낙관적 업데이트(Optimistic Update)의 구현 흐름을 설명해주세요.
**A:** ① `onMutate`에서 현재 캐시를 스냅샷으로 저장 → ② 캐시를 즉시 수정하여 UI에 반영 → ③ 서버 요청 실행 → ④ 성공 시 `onSuccess`에서 Invalidation → ⑤ 실패 시 `onError`에서 스냅샷으로 롤백합니다.

### Q5. useSuspenseQuery와 useQuery의 차이는?
**A:** `useSuspenseQuery`는 로딩 상태를 컴포넌트 내부에서 처리하지 않고 상위 `<Suspense>` 경계에 위임합니다. 따라서 컴포넌트 안에서는 항상 데이터가 존재한다고 가정하고 비즈니스 로직에 집중할 수 있습니다. `isPending` 체크가 불필요해지는 것이 핵심 차이입니다.

### Q6. useInfiniteQuery에서 getNextPageParam의 역할은?
**A:** 서버 응답에서 **다음 페이지 정보**를 추출하는 함수입니다. 반환값이 있으면 `hasNextPage`가 `true`가 되어 `fetchNextPage()`로 다음 페이지를 불러올 수 있고, `undefined`를 반환하면 마지막 페이지로 판단합니다.

### Q7. Invalidation과 직접 캐시 수정의 차이는?
**A:** `invalidateQueries()`는 캐시를 "stale" 상태로 표시하여 다음 접근 시 자동 갱신을 트리거합니다. 직접 캐시 수정(`setQueryData`)은 서버 응답 없이 캐시를 즉시 변경합니다. 낙관적 업데이트에서는 직접 수정을 사용하고, 일반적인 Mutation 후에는 Invalidation이 안전합니다.

### Q8. placeholderData: keepPreviousData는 어떤 문제를 해결하나요?
**A:** 필터/검색 조건 변경 시 이전 데이터가 사라지고 로딩 UI가 나타나는 **Layout Shift(깜빡임)** 문제를 해결합니다. 새 데이터가 도착할 때까지 이전 데이터를 유지하여 끊김 없는 UX를 제공합니다.

### Q9. 의존적 쿼리(Dependent Queries)를 어떻게 구현하나요?
**A:** `enabled` 옵션을 사용합니다. 예: `enabled: !!userId`로 설정하면 `userId`가 존재할 때만 쿼리가 실행됩니다. A 쿼리의 결과를 B 쿼리의 `enabled` 조건으로 사용하면 순차적 비동기 호출을 선언적으로 표현할 수 있습니다.

### Q10. TanStack Query의 오프라인 지원은 어떻게 동작하나요?
**A:** `persistQueryClient`를 사용하면 캐시를 localStorage나 IndexedDB에 영속화합니다. 앱 재시작 시 저장된 캐시를 복원하여 네트워크 없이도 즉시 데이터를 표시할 수 있습니다. `onlineManager`를 통해 네트워크 복구 시 자동으로 stale 쿼리를 갱신합니다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **Server State** | 서버에 원본이 존재하는 데이터. 클라이언트는 사본(캐시)을 관리 |
| **Query Key** | 캐시의 고유 주소. 배열 형태로 계층적 구성 가능 |
| **queryFn** | 실제 데이터를 가져오는 비동기 함수 |
| **staleTime** | 캐시 데이터의 신선도 유지 시간 (기본값: 0) |
| **gcTime** | 비활성 캐시의 메모리 보관 시간 (기본값: 5분) |
| **Fresh** | staleTime 이내의 신선한 캐시 상태 |
| **Stale** | staleTime이 지나 갱신이 필요한 캐시 상태 |
| **Inactive** | 구독 컴포넌트가 없는 캐시 상태 |
| **SWR** | Stale-While-Revalidate. 캐시 우선 제공 + 백그라운드 갱신 전략 |
| **Invalidation** | 캐시를 stale로 표시하여 자동 갱신을 트리거하는 동작 |
| **Mutation** | 서버 데이터를 변경(생성/수정/삭제)하는 동작 |
| **Optimistic Update** | 서버 응답 전에 UI를 먼저 업데이트하는 패턴 |
| **Prefetching** | 사용자 액션 전에 데이터를 미리 캐시에 로드하는 기법 |
| **placeholderData** | 실제 데이터 로드 전 임시로 표시하는 대체 데이터 |
| **keepPreviousData** | 쿼리 키 변경 시 이전 데이터를 유지하는 옵션 |
| **select** | 서버 응답을 컴포넌트에 맞게 변환하는 옵션 |
| **enabled** | 쿼리 실행 조건을 제어하는 옵션 (의존적 쿼리에 활용) |
| **Dependent Query** | 다른 쿼리의 결과에 의존하여 순차 실행되는 쿼리 |
| **useInfiniteQuery** | 무한 스크롤/페이지네이션 전용 훅 |
| **getNextPageParam** | 서버 응답에서 다음 페이지 정보를 추출하는 함수 |
| **Waterfall** | 쿼리가 순차적으로 실행되어 총 로딩 시간이 합산되는 현상 |
| **Suspense** | 로딩 상태를 컴포넌트 트리 수준에서 선언적으로 처리하는 React 기능 |
| **Error Boundary** | 에러를 컴포넌트 트리 수준에서 포착하여 처리하는 React 패턴 |
| **persistQueryClient** | 캐시를 영속 저장소에 저장/복원하는 유틸리티 |
| **React Compiler** | 빌드 타임 자동 메모이제이션 도구 (실험적) |
