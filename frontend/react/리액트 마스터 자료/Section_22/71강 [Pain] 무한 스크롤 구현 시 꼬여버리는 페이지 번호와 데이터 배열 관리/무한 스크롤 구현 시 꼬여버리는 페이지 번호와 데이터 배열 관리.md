"데이터는 쌓여가고, 내 머릿속은 복잡해진다"
무한 스크롤은 사용자에게는 끊김 없는 부드러운 경험을 제공하지만, 개발자에게는 '상태 관리의 늪'을 선물합니다. 단순히 현재 페이지 데이터만 갈아 끼우면 되었던 일반 페이지네이션과 달리, 무한 스크롤은 과거의 데이터를 보존하면서 새로운 데이터를 이어 붙여야 하는 누적형 구조이기 때문입니다.

🏛️ 아키텍처 원칙: 상태의 비대화와 인지 부하
수동으로 무한 스크롤을 구현할 때 우리가 마주하게 되는 본질적인 문제는 '관리해야 할 변수가 기하급수적으로 늘어난다'는 것입니다. 비즈니스 로직보다 상태를 관리하는 코드가 더 많아지는 주객전도의 상황이 발생하죠.

📚 주요 용어 상세 설명
무한 스크롤 (Infinite Scroll): 사용자가 페이지 하단에 도달할 때마다 새로운 데이터를 자동으로 불러와 리스트 끝에 추가하는 기법입니다.
상태의 비대화 (State Bloating): 하나의 기능을 구현하기 위해 너무 많은 useState를 선언하여 컴포넌트가 무거워지는 현상입니다.
페이지 파라미터 (pageParam): 현재 몇 번째 페이지를 요청해야 하는지 알려주는 '포인터' 혹은 '커서' 역할을 합니다.
불변성 (Immutability): 리액트에서 상태를 바꿀 때 기존 값을 직접 수정하지 않고 새로운 배열을 만들어 교체해야 하는 원칙입니다.
🚀 스텝 바이 스텝 가이드: 수동 무한 스크롤의 고통 체험
Step 1. 가짜 API 및 데이터 규격 정의 (src/api/mockApi.ts)
가장 먼저 우리가 무한 스크롤로 불러올 데이터의 규격부터 정의합니다. 인터페이스 앞에 export를 붙여야 다른 파일에서 참조할 수 있습니다.

/**
 * Post 인터페이스는 무한 스크롤로 불러올 개별 게시글의 표준 규격입니다.
 * ✅ export 키워드가 있어야 외부 파일에서 임포트가 가능합니다.
 */
export interface Post {
  id: number;
  title: string;
}

/**
 * 수동 구현 테스트를 위한 가짜 페이지네이션 API
 */
export const fetchPostsManual = async (page: number): Promise<Post[]> => {
  return new Promise((resolve) => {
    setTimeout(() => {
      // 5페이지가 마지막이라고 가정합니다.
      if (page > 5) return resolve([]);
      const data = Array.from({ length: 10 }, (_, i) => ({
        id: (page - 1) * 10 + i + 1,
        title: `${page}페이지의 ${i + 1}번째 게시글`
      }));
      resolve(data);
    }, 1000);
  });
};

💡 코드 상세 해설:

인터페이스 수출:export interface Post를 통해 데이터의 형태를 확정 짓습니다. 이 작업이 누락되면 질문하신 SyntaxError가 발생하게 됩니다.
페이지네이션 시뮬레이션: 호출할 때마다 10개씩 데이터를 생성하며, 5페이지를 넘어가면 빈 배열을 반환하도록 설계하여 '종료 조건'을 테스트할 수 있게 했습니다.
Step 2. 수동 상태 관리의 늪 (src/components/ManualScroll.tsx)
본격적으로 수동 구현에 들어가면 우리는 곧바로 상태의 늪에 빠지게 됩니다.

import { useState } from 'react';
import { fetchPostsManual } from '../api/mockApi';
import type { Post } from '../api/mockApi'; // ✅ Type-only import

export default function ManualScroll() {
  // 🚩 [Pain 1] 관리해야 할 4가지 핵심 상태값들
  const [allPosts, setAllPosts] = useState<Post[]>([]);
  const [pageParam, setPageParam] = useState<number>(1);
  const [hasNextPage, setHasNextPage] = useState<boolean>(true);
  const [isFetching, setIsFetching] = useState<boolean>(false);

  const fetchMorePosts = async () => {
    // 🚩 [Pain 2] 수동 가드 로직 (중복 호출 방지)
    if (!hasNextPage || isFetching) return;

    setIsFetching(true);
    try {
      const newPosts = await fetchPostsManual(pageParam);

      if (newPosts.length === 0) {
        setHasNextPage(false);
      } else {
        /**
         * 🚩 [Pain 3] 데이터 이어 붙이기 (Manual Concatenation)
         * 스프레드 연산자를 사용하여 기존 배열 뒤에 새 배열을 강제로 합칩니다.
         */
        setAllPosts((prev) => [...prev, ...newPosts]);

        // 🚩 [Pain 4] 페이지 번호 수동 계산
        setPageParam((prev) => prev + 1);
      }
    } finally {
      setIsFetching(false);
    }
  };

  return (
    <div style={{ padding: '1rem', border: '1px solid #ddd', borderRadius: '8px' }}>
      <ul style={{ listStyle: 'none', padding: 0 }}>
        {allPosts.map(post => (
          <li key={post.id} style={{ padding: '0.8rem', borderBottom: '1px solid #eee' }}>
            {post.title}
          </li>
        ))}
      </ul>
      {hasNextPage && (
        <button
          onClick={fetchMorePosts}
          disabled={isFetching}
          style={{ width: '100%', padding: '1rem', cursor: 'pointer' }}
        >
          {isFetching ? '⏳ 데이터 로딩 중...' : '➕ 게시글 더 보기'}
        </button>
      )}
    </div>
  );
}

💡 코드 상세 해설:

누적 배열 관리:setAllPosts에서 스프레드 연산자를 사용해 데이터를 합칩니다. 데이터가 수천 개가 쌓이면 이 연산은 점차 무거워지고 가독성도 나빠집니다.
명령형 로직: 페이지를 1씩 더하고, 배열 길이를 보고 hasNextPage를 false로 바꾸는 모든 과정을 개발자가 직접 지시해야 합니다.
경합 조건 (Race Condition): 사용자가 버튼을 광클할 경우, 동일한 페이지 번호 요청이 여러 번 날아가 중복 데이터가 쌓이는 사고가 발생하기 쉽습니다.

🔍 심층 해설: 왜 수동 관리는 '지옥'인가?
성능과 가독성의 충돌: 데이터가 많아질수록 [...prev, ...newPosts] 코드는 매번 거대한 배열을 새로 생성해야 하므로 비효율적입니다.
부수 효과의 관리: 스크롤 위치를 유지하거나, 특정 아이템만 삭제했을 때 전체 페이지 인덱스가 꼬이는 현상을 수동으로 해결하려면 수백 줄의 코드가 추가로 필요합니다.
스크롤 이벤트 결합의 어려움: 현재는 버튼 클릭이지만, 실제 서비스에서는 Intersection Observer와 결합해야 합니다. 이때 데이터 로직과 DOM 감시 로직이 한데 뒤섞여 컴포넌트의 순수성이 파괴됩니다.

상태 추적의 어려움: 만약 중간에 5번째 게시글을 수정해야 한다면, 수천 개의 게시글이 담긴 배열을 돌며 일일이 수정해야 하는 노가다가 시작됩니다.
