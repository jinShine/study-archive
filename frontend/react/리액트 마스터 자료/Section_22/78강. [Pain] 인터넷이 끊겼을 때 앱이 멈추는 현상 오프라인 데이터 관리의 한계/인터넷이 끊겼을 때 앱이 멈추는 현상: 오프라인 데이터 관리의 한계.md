"코드는 완벽해도 네트워크는 완벽할 수 없다"
지난 77강에서 우리는 비동기 호출의 연쇄 고리를 우아하게 설계했습니다. 하지만 개발자가 통제할 수 없는 거대한 변수가 남아있습니다. 바로 사용자의 인터넷 연결 상태입니다. 아무리 코드가 완벽해도 사용자가 지하철 터널을 지나거나 와이파이가 불안정해지는 찰나, 앱이 먹통이 된다면 우리가 공들여 쌓아온 모든 UX는 순식간에 무너집니다. 오늘은 실제 GitHub API와 인터랙티브한 검색 인풋을 통해 오프라인 상황에서 마주하는 처절한 고통(Pain Points)을 파헤쳐 봅니다.

🏛️ 핵심 용어 사전: 오프라인이 두려운 당신을 위해
Paused (일시 정지): 네트워크가 끊기면 엔진은 재시도를 멈추고 끈기 있게 기다립니다. 엔진이 '동면'에 들어간 상태를 의미합니다.
Volatile Memory (휘발성 메모리): 기본 캐시가 저장되는 RAM 영역입니다. 새로고침 시 모든 데이터가 흔적도 없이 증발하는 이유입니다.
Persistence (지속성): 오프라인에서도 데이터를 보존하기 위해 로컬 저장소에 데이터를 '박제'하는 전략입니다.
fetchStatus: 현재 쿼리가 실제로 데이터를 가져오는 중인지(fetching), 대기 중인지(idle), 아니면 네트워크 문제로 멈췄는지(paused)를 나타내는 물리적 지표입니다.
⚙️ 오프라인의 동작 방식: 엔진이 멈추는 순간
네트워크가 단절되면 탄스택 쿼리 엔진 내부에서는 다음과 같은 연쇄 반응이 일어납니다.

단절 감지: 브라우저의 online/offline 이벤트를 감지하여 현재 상태를 파악합니다.
재시도 보류: 실행 중이던 실제 fetch 요청이 실패하면, 무한 재시도를 하는 대신 fetchStatus를 paused로 전환합니다.
무한 로딩 유발: 엔진은 "아직 시도 중"이라고 판단하기에 isLoading은 여전히 true로 남습니다. 사용자에게는 멈춰버린 스피너만 보이게 됩니다.
🦖 [Pain Point] 왜 '공룡 게임'이 나올까요? (76강의 핵심 통찰)
실습 도중 브라우저에서 공룡 게임(No Internet)을 보셨다면, 우리가 해결해야 할 두 가지 층위의 고통을 마주하신 겁니다.

현상: 오프라인 상태에서 '새로고침'을 누르는 순간, RAM에 있던 앱의 모든 코드(JS/HTML)와 데이터 캐시가 삭제됩니다.
이유: 브라우저는 앱을 다시 실행하기 위해 서버에 파일을 요청하지만 실패하고, 결국 우리 앱 대신 공룡을 소환합니다.
냉정한 현실: 다음 강의에서 배울 77강 '지속성 전략'을 써도 공룡 게임은 여전히 나옵니다. 77강은 데이터(알맹이)를 살리는 기술이지, 앱 파일(껍데기)을 살리는 기술이 아니기 때문입니다. (껍데기까지 살리려면 PWA/Service Worker가 필요합니다.)
76강의 교훈: "지금 우리 앱은 인터넷 연결선 하나에 앱의 실행 파일과 데이터가 모두 저당 잡혀 있는 유약한 존재구나!"를 깨닫는 것입니다.
🚀 실전 랩(Lab) 구축: 인터랙티브 오프라인 테스트
1. 실제 GitHub API 연동 (src/api/userApi.ts)
export interface GitHubUser {
  id: number;
  name: string;
  bio: string;
  avatar_url: string;
}

export const fetchGitHubUser = async (username: string): Promise<GitHubUser> => {
  if (!username) throw new Error("아이디를 입력해주세요.");

  console.log(`🛰️ GitHub 서버에 '${username}'의 데이터를 요청합니다...`);
  const response = await fetch(`https://api.github.com/users/${username}`);

  if (!response.ok) throw new Error('유저를 찾을 수 없거나 네트워크 에러입니다.');

  return response.json();
};

2. 프로필 검색 컴포넌트 구현 (src/components/ProfileSearch.tsx)
import { useState } from 'react';
import { useQuery } from '@tanstack/react-query';
import { fetchGitHubUser } from '../api/userApi';

export const ProfileSearch = () => {
  const [username, setUsername] = useState('');
  const [searchQuery, setSearchQuery] = useState('');

  const { data, isFetching, fetchStatus, error } = useQuery({
    queryKey: ['github-user', searchQuery],
    queryFn: () => fetchGitHubUser(searchQuery),
    enabled: !!searchQuery, // 검색어가 있을 때만 실행
    staleTime: 0, // 테스트를 위해 항상 최신 데이터를 시도하도록 설정
  });

  return (
    <div style={{ padding: '20px', border: '1px solid #ddd', borderRadius: '12px' }}>
      <div style={{ marginBottom: '20px' }}>
        <input
          type="text"
          value={username}
          onChange={(e) => setUsername(e.target.value)}
          placeholder="GitHub ID 입력 (예: octocat)"
          style={{ padding: '10px', width: '200px', marginRight: '10px' }}
        />
        <button
          onClick={() => setSearchQuery(username)}
          style={{ padding: '10px 20px', cursor: 'pointer' }}
        >
          검색
        </button>
      </div>

      <div style={{ padding: '15px', backgroundColor: '#f8f9fa', borderRadius: '8px' }}>
        <p><strong>엔진 상태 (fetchStatus):</strong>
          <span style={{ color: fetchStatus === 'paused' ? 'red' : 'blue' }}> {fetchStatus}</span>
        </p>
        <p><strong>통신 중 여부 (isFetching):</strong> {isFetching ? '🔄 요청 중...' : '✅ 대기 중'}</p>

        {fetchStatus === 'paused' && (
          <p style={{ color: 'red', fontWeight: 'bold' }}>
            🚫 오프라인 상태라 요청이 멈췄습니다! (무한 로딩 발생)
          </p>
        )}
      </div>

      {data && !isFetching && (
        <div style={{ marginTop: '20px' }}>
          <img src={data.avatar_url} alt="avatar" style={{ width: '80px', borderRadius: '50%' }} />
          <h3>{data.name}</h3>
          <p>{data.bio}</p>
        </div>
      )}
    </div>
  );
};

🧐 실습 관전 포인트: 시니어의 시선
1. 'Loading'과 'Paused'의 교묘한 간극
오프라인 상황에서는 isFetching이 true임에도 실제 네트워크 요청은 paused 되어 아무런 일도 일어나지 않는 '죽은 시간'이 발생합니다. 이 간극을 사용자에게 어떻게 설명할 것인가가 UX 설계의 핵심입니다.

2. 휘발성 데이터의 허무함 체감
성공적으로 가져온 데이터도 새로고침 한 번이면 증발합니다. 우리가 짠 똑똑한 리액트 로직들은 브라우저가 앱 파일(main.js)을 불러오지 못하는 순간 아무런 힘을 쓰지 못합니다.

한 줄 결론: 공룡 게임을 만났다는 것은 우리 앱이 아직 인터넷 연결선 하나에 앱의 존재 유무를 구걸하는 유약한 상태임을 증명합니다.

이제 이 절망을 희망으로 바꿀 차례입니다. 다음 시간, 비록 새로고침 시 공룡은 나올지언정(PWA 전까지는!), 앱이 다시 실행되는 순간만큼은 이전 데이터를 즉시 복구해내는 Persistence(지속성) 전략을 배워보겠습니다.
