네트워크 자원의 비극과 수동 관리의 한계
가족 5명이 똑같은 피자집의 똑같은 메뉴를 먹고 싶은데, 서로 대화하지 않고 각자 주문을 넣어 집 앞에 오토바이 5대가 줄을 서는 상황을 상상해 보세요. 배달비도 5번, 음식값도 5번입니다. 리액트 앱에서도 이와 똑같은 비극이 벌어집니다.

🏛️ 이론 배경: 왜 '각자도생'은 재앙인가?
2026년의 웹 서비스는 파편화된 수많은 컴포넌트로 구성됩니다. 상단 내비게이션 바, 왼쪽 사이드바, 우측 하단 퀵 메뉴, 마이페이지 본문까지... 이 모든 곳이 '현재 로그인한 유저 정보'를 동시에 필요로 합니다.

독립적 useEffect의 가동: 각 컴포넌트가 화면에 나타날 때마다 각자의 useEffect가 실행됩니다. 브라우저는 서버를 향해 똑같은 요청을 5번 동시에 쏘아 올립니다.
네트워크 병목 현상: 브라우저가 동시에 처리할 수 있는 네트워크 연결 수는 제한적입니다. 중복 요청이 이 연결을 점유하면 다른 중요한 자원(이미지, 스크립트)의 로딩이 밀리게 됩니다.
서버 비용 및 부하: 서버 개발자 입장에서는 한 번만 줘도 될 데이터를 위해 5번의 DB 조회와 연산을 수행해야 합니다. 사용자가 늘어날수록 서버 비용은 기하급수적으로 상승합니다.
🚀 스텝 바이 스텝 가이드: 재앙 재현 실습
Step 1. 가짜 API 및 인터페이스 정의 (src/api/mockApi.ts)
실제 네트워크 통신이 발생하는 것처럼 로그를 남기고 타입을 정의합니다.

// 2026년 표준: User 데이터의 구조를 명확히 정의합니다.
export interface User {
  id: number;
  name: string;
  email: string;
  bio: string;
}

// 호출 횟수를 추적하기 위한 카운터
let callCount = 0;

export const fetchUser = async (): Promise<User> => {
  callCount++;
  console.log(`📡 [Network Log] 서버 요청 발생! (총 호출 횟수: ${callCount})`);

  // 실제 네트워크 지연을 시뮬레이션합니다.
  return new Promise((resolve) => {
    setTimeout(() => {
      resolve({
        id: 1,
        name: "프론트엔드 시니어",
        email: "senior@dev.com",
        bio: "리액트 아키텍처를 설계하는 개발자입니다."
      });
    }, 1000);
  });
};

🧐 주목할 점 (Deep Dive):

callCount 변수: 이 변수를 통해 우리가 의도치 않게 서버를 얼마나 괴롭히고 있는지 눈으로 확인할 수 있습니다.
Promise와 setTimeout: 비동기 통신의 지연 시간을 줌으로써, 요청이 완료되기 전까지 컴포넌트들이 어떤 상태에 놓이는지 관찰할 수 있습니다.
Step 2. '각자도생' 컴포넌트들 (src/components/UserComponents.tsx)
동일한 유저 데이터를 필요로 하는 파편화된 컴포넌트들을 만듭니다.

import { useState, useEffect } from 'react';
import { fetchUser } from '../api/mockApi';
import type { User } from '../api/mockApi';

// 1. 네비게이션 바: 상단에서 유저 이름을 보여줍니다.
export const Navbar = () => {
  const [user, setUser] = useState<User | null>(null);
  useEffect(() => { fetchUser().then(setUser); }, []);
  return <nav style={{ borderBottom: '1px solid #ccc' }}>👤 {user?.name}님 환영합니다</nav>;
};

// 2. 사이드바: 왼쪽에서 유저의 소개글을 보여줍니다.
export const Sidebar = () => {
  const [user, setUser] = useState<User | null>(null);
  useEffect(() => { fetchUser().then(setUser); }, []);
  return <aside style={{ width: '200px', background: '#f9f9f9' }}>📝 소개: {user?.bio}</aside>;
};

// 3. 퀵메뉴: 하단에서 유저의 이메일을 보여줍니다.
export const QuickMenu = () => {
  const [user, setUser] = useState<User | null>(null);
  useEffect(() => { fetchUser().then(setUser); }, []);
  return <div style={{ position: 'fixed', bottom: 10, right: 10 }}>📧 {user?.email}</div>;
};

🧐 주목할 점 (Deep Dive):

상태의 파편화:useState<User | null>(null) 코드가 세 번이나 중복되었습니다. 이는 데이터가 서버에서 수정되어도, 세 컴포넌트가 서로 다른 시점에 데이터를 받아 화면이 일치하지 않는 문제를 야기할 수 있습니다.
중복 로직:fetchUser().then(setUser)라는 명령형 코드가 모든 곳에 박혀 있어 유지보수가 어렵습니다.
