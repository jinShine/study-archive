## 1. State 끌어올리기 (Lifting State Up)

### 문제 상황

두 형제 컴포넌트가 **동일한 데이터**를 필요로 할 때, 각각 독립된 state를 갖고 있으면 값이 동기화되지 않는다.

### 해결 원칙

> **상태의 단일 소유자(Single Source of Truth)를 공통 부모로 올린다.**

- 부모가 state와 setter를 소유한다.
- 자식은 **props**로 값과 업데이터 함수를 전달받아 "제어 컴포넌트"가 된다.

```
           App (state 소유)
          /        \
    [title]      [title]
    [setTitle]
      |              |
  TitleInput      Preview
  (입력 담당)     (표시 담당)
```

### 핵심 예제

```jsx
function TitleInput({ title, setTitle }) {
  return (
    <input
      type="text"
      value={title}
      onChange={(e) => setTitle(e.target.value)}
      placeholder="제목을 입력하세요"
    />
  );
}

function Preview({ title }) {
  return <p>{title || "제목이 없습니다."}</p>;
}

function App() {
  const [title, setTitle] = useState("");
  return (
    <>
      <TitleInput title={title} setTitle={setTitle} />
      <Preview title={title} />
    </>
  );
}
```

| 항목 | 설명 |
|------|------|
| **왜 올리는가** | 형제 컴포넌트가 같은 데이터를 공유해야 할 때 |
| **어디로 올리는가** | 두 컴포넌트의 **가장 가까운 공통 부모** |
| **자식의 역할** | props로 받은 값을 표시하고, setter를 호출해 변경을 요청 |
| **주의사항** | 너무 먼 조상으로 올리면 prop drilling 발생 -- Context 고려 |

## 2. 제어 컴포넌트 (Controlled Components)

### 개념

React state가 폼 요소의 **유일한 데이터 출처**가 되는 패턴이다.

```
사용자 입력 --onChange--> setState() --리렌더링--> value={state}
```

### 제어 vs 비제어 비교

| 구분 | 제어 컴포넌트 | 비제어 컴포넌트 |
|------|--------------|----------------|
| 값 관리 | `useState` + `value` | DOM 자체가 관리 |
| 값 접근 | state 변수로 즉시 접근 | `useRef` + `ref.current.value` |
| 실시간 검증 | 가능 (onChange 시점) | 제출 시점에서만 가능 |
| React 권장 | **권장** | 특수한 경우(`type="file"` 등)에만 사용 |

### 기본 패턴

```jsx
function ControlledForm() {
  const [name, setName] = useState("");

  function handleSubmit(e) {
    e.preventDefault();
    alert(`제출된 이름: ${name}`);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="text"
        value={name}                              // state 바인딩
        onChange={(e) => setName(e.target.value)}  // state 업데이트
      />
      <button type="submit">제출</button>
    </form>
  );
}
```

### 응용 -- 입력 제한 (최대 글자수)

제어 컴포넌트의 강점은 **onChange 시점에 로직을 끼워 넣을 수 있다**는 점이다.

```jsx
<input
  value={name}
  onChange={(e) => {
    if (e.target.value.length <= 10) setName(e.target.value); // 10자 초과 무시
  }}
/>
<p>현재 길이: {name.length} / 10</p>
```

> **포인트**: `setState`를 호출하지 않으면 입력이 "씹힌다" -- 이것이 제어 컴포넌트의 핵심 메커니즘이다.

## 3. 다중 입력 관리 -- computed property names

### 문제

입력 필드가 여러 개일 때 각각 `useState`를 선언하면 핸들러가 반복된다.

### 해결 -- 객체 state + 공통 핸들러

**하나의 객체 state**로 묶고, HTML `name` 속성과 ES6 computed property names `[name]: value`를 활용한다.

```jsx
const initialForm = { name: "", email: "", age: "" };

function SimpleForm() {
  const [user, setUser] = useState(initialForm);

  function handleChange(e) {
    const { name, value } = e.target;
    setUser((prev) => ({
      ...prev,           // 기존 값 유지 (불변성)
      [name]: value,     // 해당 키만 덮어쓰기 (computed property)
    }));
  }

  function handleSubmit(e) {
    e.preventDefault();
    alert(JSON.stringify(user, null, 2));
  }

  return (
    <form onSubmit={handleSubmit}>
      <input name="name"  value={user.name}  onChange={handleChange} />
      <input name="email" value={user.email} onChange={handleChange} />
      <input name="age"   value={user.age}   onChange={handleChange} />
      <button type="submit">제출</button>
    </form>
  );
}
```

### computed property names란?

```js
const key = "email";
const obj = { [key]: "test@test.com" };
// 결과: { email: "test@test.com" }
```

| 장점 | 설명 |
|------|------|
| 핸들러 하나로 통합 | 필드가 늘어나도 `handleChange` 하나로 충분 |
| 제출/리셋 간편 | `setUser(initialForm)` 한 줄로 리셋 |
| 검증 로직 집중 | `handleSubmit`에서 `user` 객체 하나만 검사 |

## 4. 다양한 폼 요소 제어 -- Select, Checkbox, Radio

모든 폼 요소는 동일한 원칙: **state <-> 속성 바인딩 + onChange -> setState**.
차이는 바인딩 속성(`value` vs `checked`)과 이벤트 값(`e.target.value` vs `e.target.checked`)에 있다.

### 폼 요소별 비교 표

| 요소 | state 타입 | 바인딩 속성 | onChange 값 |
|------|-----------|------------|-------------|
| **text input** | `string` | `value={state}` | `e.target.value` |
| **select** | `string` | `value={state}` | `e.target.value` |
| **checkbox (단일)** | `boolean` | `checked={state}` | `e.target.checked` |
| **checkbox (다중)** | `string[]` | `checked={arr.includes(v)}` | 체크 시 추가, 해제 시 제거 |
| **radio** | `string` | `checked={state === v}` | `e.target.value` |

### Select

```jsx
const [fruit, setFruit] = useState("apple");

<select value={fruit} onChange={(e) => setFruit(e.target.value)}>
  <option value="apple">사과</option>
  <option value="banana">바나나</option>
</select>
```

> React에서는 `<option selected>` 대신 **`<select value>`** 로 선택을 제어한다.

### Checkbox -- 단일(boolean) vs 다중(배열)

```jsx
// 단일: boolean state
const [agree, setAgree] = useState(false);
<input type="checkbox" checked={agree} onChange={(e) => setAgree(e.target.checked)} />

// 다중: 배열 state
const [skills, setSkills] = useState([]);

function handleSkillChange(e) {
  const { value, checked } = e.target;
  setSkills((prev) =>
    checked ? [...prev, value] : prev.filter((v) => v !== value)
  );
}

<input type="checkbox" value="react" checked={skills.includes("react")} onChange={handleSkillChange} />
<input type="checkbox" value="vue"   checked={skills.includes("vue")}   onChange={handleSkillChange} />
```

### Radio

같은 `name`을 공유하고, `checked={state === value}`로 선택 상태를 결정한다.

```jsx
const [gender, setGender] = useState("male");

<input type="radio" name="gender" value="male"
  checked={gender === "male"} onChange={(e) => setGender(e.target.value)} />
<input type="radio" name="gender" value="female"
  checked={gender === "female"} onChange={(e) => setGender(e.target.value)} />
```

## 5. 폼 제출과 유효성 검사

### 기본 제출 흐름

```
<form onSubmit> --> handleSubmit(e) --> e.preventDefault() --> state 기반 처리
```

`preventDefault()`를 호출하지 않으면 브라우저가 페이지를 새로고침하여 React state가 모두 초기화된다.

### 유효성 검사 패턴 -- error state + 조건부 렌더링

검증 실패 시 `setError()`로 메시지를 표시하고 **즉시 `return`** 하여 제출을 중단한다.

```jsx
function ValidatedSignupForm() {
  const [username, setUsername] = useState("");
  const [email, setEmail] = useState("");
  const [agree, setAgree] = useState(false);
  const [error, setError] = useState("");

  function handleSubmit(e) {
    e.preventDefault();

    if (username.trim() === "") { setError("사용자 이름을 입력하세요."); return; }
    if (!email.includes("@"))  { setError("올바른 이메일을 입력하세요."); return; }
    if (!agree)                { setError("약관에 동의해야 합니다.");     return; }

    setError("");        // 에러 초기화
    alert("제출 성공!"); // 또는 API 호출
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={username} onChange={(e) => setUsername(e.target.value)} />
      <input value={email}    onChange={(e) => setEmail(e.target.value)} />
      <label>
        <input type="checkbox" checked={agree} onChange={(e) => setAgree(e.target.checked)} />
        개인정보 수집 이용에 동의합니다.
      </label>

      {error && <p style={{ color: "crimson" }} role="alert">{error}</p>}
      <button type="submit">가입하기</button>
    </form>
  );
}
```

### 검사 흐름도

```
handleSubmit(e)
  |-- e.preventDefault()
  |-- [검증 1] username 비었음?  --YES--> setError + return
  |-- [검증 2] email에 @ 없음?  --YES--> setError + return
  |-- [검증 3] agree === false?  --YES--> setError + return
  |-- setError("") --> 제출 처리 (alert / API 호출)
```

### UX 보강 -- 입력 시 에러 자동 해제

```jsx
onChange={(e) => {
  setUsername(e.target.value);
  if (error && e.target.value.trim() !== "") setError("");
}}
```

### 전체 패턴 정리

| 단계 | 코드 | 설명 |
|------|------|------|
| 1. 이벤트 가로채기 | `e.preventDefault()` | 브라우저 기본 제출 방지 |
| 2. 유효성 검사 | `if (...) { setError(); return; }` | 실패 시 에러 표시 + 중단 |
| 3. 에러 초기화 | `setError("")` | 검증 통과 후 이전 에러 제거 |
| 4. 제출 처리 | API 호출 등 | 비즈니스 로직 실행 |
| 5. (선택) 리셋 | `setUser(initialForm)` | 초기값으로 되돌리기 |

## 면접 핵심 Q&A

### Q1. State 끌어올리기란? 왜 필요한가?

**A.** 형제 컴포넌트가 동일한 데이터를 공유해야 할 때, 해당 state를 **공통 부모 컴포넌트로 올려서** 단일 소유자(Single Source of Truth)로 만드는 패턴이다. React의 데이터 흐름은 **단방향(top-down)** 이므로 형제끼리 직접 통신할 수 없고, 공통 부모에서 관리해야 일관된 값을 보장할 수 있다.

---

### Q2. 제어 컴포넌트와 비제어 컴포넌트의 차이?

**A.**

| 구분 | 제어 컴포넌트 | 비제어 컴포넌트 |
|------|--------------|----------------|
| 데이터 소유 | React state | DOM 자체 |
| 값 설정 | `value={state}` | `defaultValue` |
| 값 읽기 | state 변수 참조 | `ref.current.value` |
| 실시간 제어 | 가능 (onChange 검증/변환) | 불가 |
| 사용 시점 | 대부분의 폼 (권장) | `type="file"` 등 특수 경우 |

제어 컴포넌트는 입력값을 실시간으로 검증/변환/제한할 수 있다. 비제어 컴포넌트는 제출 시점에만 `ref`로 값을 읽는다.

---

### Q3. computed property names를 폼에서 활용하는 방법?

**A.** 각 `<input>`에 `name` 속성을 부여하고, 공통 핸들러에서 `[e.target.name]: e.target.value`로 객체 state를 업데이트한다. 필드가 추가되어도 핸들러 수정이 불필요하여 확장성이 뛰어나다.

```jsx
const { name, value } = e.target;
setForm(prev => ({ ...prev, [name]: value }));
```

---

### Q4. preventDefault()의 역할?

**A.** **브라우저의 기본 동작을 차단**한다. `<form onSubmit>`에서 호출하지 않으면 브라우저가 페이지를 새로고침하며 React state가 모두 초기화된다. SPA에서는 `preventDefault()`로 기본 제출을 막고 JavaScript에서 직접 처리하는 것이 표준 방식이다.

---

### Q5. React에서 폼 유효성 검사 패턴?

**A.** 기본 패턴은 **error state + 조건부 렌더링**이다.

```jsx
const [error, setError] = useState("");

function handleSubmit(e) {
  e.preventDefault();
  if (username.trim() === "") { setError("이름 필수"); return; }
  setError("");
  // 제출 로직
}

{error && <p role="alert">{error}</p>}
```

- **제출 시점(onSubmit)**: 기본 검증 위치
- **입력 시점(onChange)**: 글자 수 제한 등 실시간 제어
- **에러 해제**: 사용자가 필드 수정 시 에러 자동 제거로 UX 개선
- **접근성**: `role="alert"`, `aria-invalid` 속성 활용

## 용어 글로서리

| 용어 | 영문 | 설명 |
|------|------|------|
| State 끌어올리기 | Lifting State Up | 공유 state를 공통 부모로 올리는 패턴 |
| 제어 컴포넌트 | Controlled Component | state가 입력값의 유일한 출처인 폼 요소 |
| 비제어 컴포넌트 | Uncontrolled Component | DOM이 값을 관리하고 ref로 접근하는 폼 요소 |
| 단일 진실 공급원 | Single Source of Truth | 데이터의 유일한 소유자/출처 원칙 |
| computed property names | Computed Property Names | `{ [변수]: 값 }` -- ES6 객체 키 동적 지정 문법 |
| preventDefault | preventDefault | 브라우저 기본 동작(폼 제출, 링크 이동 등) 차단 |
| 에러 State | Error State | 유효성 검사 결과를 담는 상태 변수 |
| 조건부 렌더링 | Conditional Rendering | `{조건 && <JSX>}` 패턴으로 UI를 선택적 표시 |
| prop drilling | Prop Drilling | props를 여러 단계 하위까지 전달하는 현상 |
| 불변성 | Immutability | 기존 객체를 수정하지 않고 새로 만드는 원칙 |

## 자가 점검 체크리스트

### State 끌어올리기
- [ ] 형제 컴포넌트가 같은 데이터를 공유할 때 state를 어디에 둬야 하는지 설명할 수 있다
- [ ] 부모에서 자식으로 state와 setter를 props로 전달하는 코드를 작성할 수 있다
- [ ] prop drilling 발생 상황과 해결 방안(Context 등)을 알고 있다

### 제어 컴포넌트
- [ ] `value` + `onChange` 패턴의 작동 원리를 설명할 수 있다
- [ ] 제어 컴포넌트와 비제어 컴포넌트의 차이를 비교할 수 있다
- [ ] onChange에서 입력을 제한(최대 글자수 등)하는 코드를 작성할 수 있다

### 다중 입력 관리
- [ ] 객체 state + computed property names 패턴을 작성할 수 있다
- [ ] `{ ...prev, [name]: value }`에서 spread와 computed property 역할을 설명할 수 있다

### 다양한 폼 요소
- [ ] Select, Checkbox(단일/다중), Radio의 바인딩 속성 차이를 설명할 수 있다
- [ ] Checkbox 다중 선택에서 배열 추가/제거 로직을 작성할 수 있다
- [ ] Radio에서 `checked={state === value}` 패턴을 이해하고 있다

### 폼 제출과 유효성 검사
- [ ] `e.preventDefault()`를 호출하지 않으면 어떤 일이 벌어지는지 설명할 수 있다
- [ ] error state + 조건부 렌더링으로 에러 메시지를 표시하는 코드를 작성할 수 있다
- [ ] 검증 실패 시 `return`으로 제출을 중단하는 패턴을 이해하고 있다
