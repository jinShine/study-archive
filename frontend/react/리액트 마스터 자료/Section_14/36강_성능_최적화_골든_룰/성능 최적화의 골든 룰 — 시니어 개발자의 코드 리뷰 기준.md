# 36강. 성능 최적화의 골든 룰 — 시니어 개발자의 코드 리뷰 기준

## 도입

성능 최적화라는 거대한 대장정을 완주한 시점에서, 단순히 기술을 사용하는 단계를 넘어 **언제 이 기술을 꺼내 들고 언제는 칼집에 넣어두어야 하는지 판단할 줄 아는 '설계자'의 시각**이 필요하다.

이번 강에서는 지금까지 배운 모든 지식을 하나로 꿰어, 실제 현업 시니어 개발자들이 코드 리뷰 시 절대 타협하지 않는 **최종 골든 룰**을 정리한다.

## 개념 설명

### 조기 최적화(Premature Optimization)의 함정

작동하기도 전에 성능부터 걱정하여 코드를 복잡하게 만드는 습관은 위험하다. 최적화 훅(`useMemo`, `useCallback`)은 공짜가 아니며, 리렌더링마다 의존성 배열을 대조하는 **O(n)의 비용**을 지불한다.

### 최적화의 트레이드오프(Trade-off)

- **비교의 비용**: `useMemo`나 `useCallback`은 리렌더링마다 의존성 배열을 대조하는 O(n)의 비용을 지불한다.
- **얕은 비교 O(1)**: `React.memo`가 주소값만 비교하는 속도는 O(1)로 매우 빠르지만, 부모가 주소값을 지켜주지 않으면(`useCallback` 미사용) 이 파수꾼은 매번 문을 열어주게 된다.

## 코드 예제

### 나쁜 사례: 조기 최적화 — BadReview.jsx

```jsx
import { useState, useCallback, useMemo } from 'react';

export default function BadReview() {
  const [count, setCount] = useState(0);

  // 나쁜 사례: 단순 로그 출력에 useCallback을 쓰는 것은
  // 함수 재생성 비용보다 의존성 배열 비교 비용이 더 커질 수 있다.
  const log = useCallback(() => console.log(count), [count]);

  // 나쁜 사례: 1 + 1 같은 단순 산수에 useMemo를 쓰는 것은 메모리 낭비다.
  const value = useMemo(() => 1 + 1, []);

  return <div onClick={log} style={{ cursor: 'pointer' }}>단순 연산 결과: {value}</div>;
}
```

### 모범 사례: 전략적 최적화 — SeniorReview.jsx

```jsx
import { useState, useCallback } from 'react';
import ExpensiveList from './ExpensiveList';

export default function SeniorReview() {
  const [items, setItems] = useState([
    { id: 1, text: "시니어의 설계 철학 공부" },
    { id: 2, text: "불필요한 훅 제거하기" }
  ]);
  const [otherState, setOtherState] = useState(0);

  // [정밀 타격] 리스트 아이템은 개수가 많아질 수 있으므로
  // 삭제 함수의 주소값을 useCallback으로 고정하여 자식의 자물쇠(memo)를 보호한다.
  const handleDelete = useCallback((id) => {
    setItems(prev => prev.filter(item => item.id !== id));
  }, []);

  return (
    <div style={{ padding: '30px', border: '2px solid #4f46e5', borderRadius: '20px' }}>
      {/* [구조적 최적화] 자주 변하는 상태는 별도 컴포넌트로 밀어내어
           부모의 렌더링 폭발 반경을 최소화한다. */}
      <h3>단순 카운터: {otherState}</h3>
      <button onClick={() => setOtherState(prev => prev + 1)}>부모 리렌더링 유발</button>

      <hr />

      {/* [참조 무결성] 진짜 무거운 리스트에만 최적화를 투입한다. */}
      <ExpensiveList items={items} onItemClick={handleDelete} />
    </div>
  );
}
```

### ExpensiveList.jsx — 최적화된 리스트 아이템

```jsx
import { memo } from 'react';

const ExpensiveList = memo(({ items, onItemClick }) => {
  console.log("리스트 파수꾼: 주소 검사 완료. 베일아웃!");
  return (
    <ul>
      {items.map(item => (
        <li key={item.id} style={{ marginBottom: '5px' }}>
          {item.text} <button onClick={() => onItemClick(item.id)}>삭제</button>
        </li>
      ))}
    </ul>
  );
});

export default ExpensiveList;
```

## 코드 해설

- **BadReview**: 단순 로그 출력에 `useCallback`, 단순 덧셈에 `useMemo`를 사용하고 있다. 함수 재생성 비용보다 의존성 배열 비교 비용이 더 크고, 메모리만 낭비된다. 이것이 조기 최적화의 전형적인 함정이다.
- **SeniorReview**: 데이터가 증명된 곳에만 정교하게 최적화 무기를 투입한다. 리스트처럼 항목 수가 많아질 수 있는 곳에만 `useCallback` + `React.memo` 조합을 사용하고, 나머지는 리액트 엔진의 기본 성능을 신뢰한다.
- **ExpensiveList**: `memo`로 감싸서 `items`와 `onItemClick`의 참조가 변하지 않으면 리렌더링을 건너뛴다. 부모가 `useCallback`으로 함수 주소를 고정해주기 때문에 이 자물쇠가 제대로 작동한다.

## 실무 비유

- **조기 최적화**는 "적이 오지도 않았는데 온 집 안에 부비트랩을 설치하는 것"과 같다. 가족(동료 개발자)이 다치기만 할 뿐이다.
- **전략적 최적화**는 "적(병목)이 어디서 오는지 정찰(Profiler)한 후, 그 진입로에만 정밀 타격하는 것"이다.
- `React.memo`는 집 앞의 **파수꾼**이다. 부모가 매번 새 택배(새 함수 참조)를 보내면 파수꾼은 매번 문을 열어줄 수밖에 없다. `useCallback`으로 같은 택배를 보내야 파수꾼이 "아, 아까 그거네"하고 문을 닫아둘 수 있다.

## 핵심 포인트

### 골든 룰 3단계

1. **측정 우선**: 프로파일러로 16ms를 초과하는 병목을 먼저 찾는다
2. **구조 우선**: 상태를 아래로 밀어내어(Push-down) 렌더링 범위를 격리한다
3. **정직한 의존성**: 성능을 위해 의존성 배열을 조작하지 않는다 — 화면에 틀린 값이 나오는 '빠른 앱'은 가치가 없다

### 최적화 판단 기준

- `useMemo`/`useCallback`은 공짜가 아니다. 의존성 배열 비교 비용 O(n)이 발생한다
- `React.memo`의 얕은 비교는 O(1)이지만, 부모가 참조 무결성을 지켜줘야만 의미가 있다
- 단순 연산이나 단순 함수에는 최적화 훅을 쓰지 않는다
- 리스트처럼 항목이 많고 자식이 무거운 곳에 정밀하게 투입한다
