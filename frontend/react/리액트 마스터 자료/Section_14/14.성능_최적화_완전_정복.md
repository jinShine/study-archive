## 26강. 렌더링의 비용 -- 리액트는 왜, 언제 화면을 다시 그리는가?

### 핵심 개념

- **리렌더링 폭포수(Waterfall)**: 부모 컴포넌트의 상태가 하나만 바뀌어도 함수 전체가 재실행되며, 자식 컴포넌트도 무조건 다시 그려진다.
- **렌더링 = 계산 비용**: 렌더링은 단순한 화면 그리기가 아니라, 함수 내부의 모든 변수/연산이 새로 만들어지는 CPU 작업이다.
- **참조 동일성(Referential Equality)**: 리액트가 "데이터가 바뀌었나?"를 판단하는 핵심 기준이다.
  - 원시 타입(숫자, 문자): **값**이 같으면 동일하다고 판단
  - 객체 타입(객체, 배열, 함수): **메모리 주소**가 같아야 동일하다고 판단. `[] === []`는 `false`이다.

### 핵심 코드

```jsx
function RenderingProcess() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState("");

  // 리렌더링마다 이 루프가 다시 돌아간다
  let heavyValue = 0;
  for (let i = 0; i < 10000000; i++) { heavyValue++; }

  return (
    <div>
      <button onClick={() => setCount(count + 1)}>숫자 올리기</button>
      <input value={text} onChange={(e) => setText(e.target.value)} />
      {/* 부모가 리렌더링되면 자식도 무조건 다시 그려진다 */}
      <ChildComponent />
    </div>
  );
}
```

### 시니어 팁

- `text`만 바꿔도 `count`와 무관한 무거운 연산이 다시 실행된다. 상태 하나만 바뀌어도 컴포넌트 함수 **전체**가 재실행되기 때문이다.
- React.memo 같은 최적화 도구 없이는 앱이 커질수록 최상위에서 상태 하나만 바뀌어도 수천 개의 하위 자식이 동시에 재실행된다.

## 27강. 메모이제이션(Memoization) 원리 이해

### 핵심 개념

- **메모이제이션**: "이미 계산한 것은 기억해두고 재활용한다"는 컴퓨터 공학의 근본적인 접근법이다.
- **시간-공간 트레이드오프(Trade-off)**: 메모리(공간)를 써서 시간을 사는 기술이다.

### 핵심 코드: 순수 재귀 vs 메모이제이션

```js
// 비효율: O(2^n) -- n=40이면 브라우저가 멈출 수 있음
function pureFibonacci(n) {
  if (n <= 1) return n;
  return pureFibonacci(n - 1) + pureFibonacci(n - 2);
}

// 메모이제이션: O(n) -- 이미 계산한 값은 즉시 반환
const memo = {};
function memoFibonacci(n) {
  if (n in memo) return memo[n];  // 장부에서 먼저 검색
  if (n <= 1) return n;
  const result = memoFibonacci(n - 1) + memoFibonacci(n - 2);
  memo[n] = result;               // 결과를 장부에 기록
  return result;
}
```

### 시니어 팁

- 모든 함수에 메모이제이션을 쓰는 것은 낭비이다. "다시 계산하는 비용"이 "메모리에 저장하는 비용"보다 압도적으로 클 때만 사용해야 한다.
- 리액트에서 메모이제이션이 중요한 이유는 연산 속도뿐 아니라, **참조 동일성을 증명**하여 불필요한 리렌더링 폭포수를 막아주는 방패 역할을 하기 때문이다.

## 28강. useMemo 기초 -- 무거운 연산 결과값을 금고에 보관하는 법

### 기본 문법

```jsx
import { useMemo } from 'react';

const memoizedValue = useMemo(() => {
  return heavyProcess(a, b); // 무거운 계산
}, [a, b]); // 이 값이 바뀔 때만 재계산
```

- **첫 번째 인자**: 리액트에게 계산을 위임하는 콜백 함수. 반드시 결과를 `return`해야 한다.
- **두 번째 인자 (의존성 배열)**: 이 안의 값이 바뀌었을 때만 콜백을 재실행한다. 동일하면 이전 결과를 그대로 반환.

### 핵심 코드

```jsx
function CalculationConsole() {
  const [number, setNumber] = useState(0);
  const [isDark, setIsDark] = useState(false);

  // number가 바뀔 때만 10억 번 루프 실행. isDark가 바뀔 때는 건너뜀
  const result = useMemo(() => {
    for (let i = 0; i < 1000000000; i++) {}
    return number * 100;
  }, [number]);

  return (
    <div style={{ backgroundColor: isDark ? '#333' : '#fff' }}>
      <input type="number" value={number} onChange={(e) => setNumber(Number(e.target.value))} />
      <p>결과: {result}</p>
      <button onClick={() => setIsDark(!isDark)}>테마 변경</button>
    </div>
  );
}
```

### 사용 황금률

| 사용해야 할 때 | 사용하지 말아야 할 때 |
|---|---|
| 10,000개 이상 데이터 정렬/필터링 | 단순 산수 (1 + 1) |
| CPU 사용량이 눈에 띄는 복잡한 로직 | 짧은 문자열 합치기 |
| **연산 비용(Time) > 저장 비용(Space)** | 금고를 만드는 비용 > 직접 계산 비용 |

## 29강. useMemo 실전 -- 복잡한 데이터 필터링 프로젝트

### 핵심 코드

```jsx
const users = generateUsers(); // 1만 명의 데이터

function OptimizedUserList() {
  const [query, setQuery] = useState("");
  const [isDark, setIsDark] = useState(false);

  // 테마 변경(isDark)은 이 금고를 건드리지 못한다
  const filteredUsers = useMemo(() => {
    return users.filter((user) =>
      user.name.toLowerCase().includes(query.toLowerCase())
    );
  }, [query]); // 오직 검색어가 바뀔 때만 필터링 재실행

  return (
    <div style={{ backgroundColor: isDark ? '#1e293b' : '#fff' }}>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <button onClick={() => setIsDark(!isDark)}>테마 전환</button>
      <p>검색 결과: {filteredUsers.length}명</p>
    </div>
  );
}
```

### 시니어 분석

- **관심사 분리(Isolation)**: useMemo로 '검색 로직'을 '테마 로직'으로부터 완벽하게 고립시켰다.
- **UI 반응성**: 1만 번의 연산이 생략된 테마 전환은 1ms 이하로 즉시 일어난다.

## 30강. useCallback 기초 -- 함수를 메모리에 박제하여 주소값 지키기

### 핵심 개념

- `useMemo`가 **계산 결과(값)**를 보관하는 금고라면, `useCallback`은 **함수의 메모리 주소 자체**를 박제하는 도구이다.
- 함수 생성 자체는 가벼운 작업이다. **진짜 목적은 자식 컴포넌트의 불필요한 리렌더링을 차단**하는 데 있다.

### 기본 문법

```jsx
import { useCallback } from 'react';

const memoizedFunction = useCallback(() => {
  console.log("함수가 실행되었습니다!");
}, []); // 의존성 배열이 비어있으면 주소값 영구 고정
```

### 최적화의 짝꿍

- `useCallback`은 자식 컴포넌트가 **React.memo로 감싸져 있을 때만** 진정한 위력을 발휘한다.
- 부모가 주소를 고정해줘도 자식이 이를 대조해보지 않으면 소용없다.

### 의존성의 함정

함수 내부에서 `count` 같은 상태를 쓴다면 의존성 배열에 `count`를 꼭 넣어야 한다. 안 그러면 함수는 영원히 `count`가 0이었던 시절의 데이터만 기억하는 **'냉동 상태'**가 된다.

## 31강. useCallback 실전 -- 함수 재생성 방지로 유령 렌더링 막기

### 유령 렌더링(Ghost Rendering)이란?

부모가 배경색만 바꿨을 뿐인데, 아무 잘못 없는 자식 컴포넌트가 억울하게 다시 그려지며 CPU 자원을 낭비하는 현상이다.

### 최적화의 2중 잠금 장치

| 상황 | 결과 |
|---|---|
| `useCallback`만 사용 | 부모가 함수 주소를 고정해도, 리액트는 "부모가 그려졌으니 자식도 그려야지"라는 기본 원칙에 따라 자식을 다시 호출 |
| `React.memo`만 사용 | 자식이 주소를 검사하려 해도, 부모가 매번 새로운 주소를 보내주니 자식은 "데이터가 바뀌었네?"라고 착각 |
| **`useCallback` + `React.memo` 결합** | 부모가 주소 고정(열쇠) + 자식이 주소 대조(자물쇠) = **유령 렌더링 완벽 차단** |

### 핵심 코드

```jsx
// 부모: useCallback으로 함수 주소 고정 (열쇠)
const createBoxStyle = useCallback(() => {
  return { backgroundColor: 'pink', width: `${size}px`, height: `${size}px` };
}, [size]); // 오직 size가 변할 때만 금고를 새로 고침

// 자식: React.memo로 Props 검사 (자물쇠)
const ExpensiveBox = memo(({ createBoxStyle }) => {
  const style = createBoxStyle();
  return <div style={style}>최적화된 박스</div>;
});
```

## 32강. React.memo 기초 -- 컴포넌트의 입구를 지키는 최후의 파수꾼

### 핵심 개념

- **React.memo**: 고차 컴포넌트(HOC)로 컴포넌트를 감싸서, Props가 변하지 않으면 렌더링을 거부(베일아웃)하게 만드는 도구.
- **얕은 비교(Shallow Comparison)**: 메모리 주소라는 '이름표'만 대조한다. `O(1)` 속도로 매우 빠르다.
- **베일아웃(Bailout)**: Props가 동일하다고 판단되면 함수 호출 생략 + 내부 연산 무시 + 가상 DOM 재조정 건너뛰기.

### 핵심 코드

```jsx
import { memo } from 'react';

const ChildComponent = memo(({ onAction }) => {
  console.log("파수꾼: Props 주소가 똑같으면 이 로그는 찍히지 않습니다.");
  return (
    <div>
      <button onClick={onAction}>동작</button>
    </div>
  );
});
```

### 시너지 공식

```
useCallback (부모가 변하지 않는 열쇠 생성) + React.memo (자식이 열쇠 확인하는 자물쇠) = 최적화 완성
```

## 33강. 리스트 아이템 최적화 -- 렌더링 폭주 막기

### 핵심 패턴: 함수형 업데이트로 의존성 제거

```jsx
// 삭제 함수: 함수형 업데이트(prev => ...)로 의존성 배열을 비운다
const handleDelete = useCallback((id) => {
  setUsers((prev) => prev.filter(user => user.id !== id));
}, []); // 빈 배열 -> 주소 영구 고정
```

- `setUsers(prev => ...)` 형태를 쓰면 `users` 상태에 직접 접근하지 않으므로 의존성 배열에서 `users`를 제거할 수 있다.
- 의존성 배열이 비어있으므로 함수 주소가 영원히 고정되어, 자식의 `React.memo` 자물쇠가 완벽하게 작동한다.

### 자식 컴포넌트

```jsx
const UserItem = memo(({ user, onDelete }) => {
  return (
    <li>
      {user.name} <button onClick={() => onDelete(user.id)}>삭제</button>
    </li>
  );
});
```

## 34강. 성능 측정 기술 -- 짐작하지 말고 프로파일러로 증명하라

### React Profiler 사용법

1. 크롬 개발자 도구(F12)의 **Profiler** 탭으로 이동
2. 톱니바퀴 아이콘 클릭 -> **"Record why each component rendered"** 옵션 활성화
3. 파란색 녹화 버튼 클릭 -> 조작 수행 -> 정지

### 플레임 차트(Flame Chart) 읽기

| 색상/형태 | 의미 |
|---|---|
| 노란색/긴 막대 | 연산 비용이 높음 (범인!) |
| 회색/차트 제외 | 렌더링이 생략됨 (베일아웃 성공) |

### 16ms의 법칙

- 60FPS의 부드러운 화면을 제공하려면 한 프레임의 연산이 **16ms 안에** 끝나야 한다.
- 100ms가 소모된다면 초당 10프레임도 안 나오는 '뚝뚝 끊기는' 앱이라는 증거이다.

### 전문가의 태도

"느린 것 같아요" 대신 **"HeavyComponent가 100ms를 점유하며 메인 스레드를 방해하고 있습니다"** 라고 데이터로 말하는 것이 전문가의 태도이다.

## 35강. 구조적 최적화 -- 훅보다 강력한 컴포넌트 설계의 힘

### Part 1: 최적화의 역설 -- 왜 모든 곳에 useMemo를 쓰면 안 되는가?

#### 최적화가 지불하는 세 가지 대가

1. **메모리 점유와 GC 부담**: useMemo로 데이터를 금고에 넣으면 가비지 컬렉터가 치우지 못하고 메모리가 부족해질 수 있다.
2. **의존성 배열 비교 연산 비용 O(n)**: 단순 산수를 지키겠다고 훅을 쓰면, 체크리스트 검토 시간이 계산 시간보다 길어지는 배보다 배꼽이 큰 상황이 발생한다.
3. **원시 타입에 대한 무의미한 최적화**: 숫자/문자열은 값 자체로 비교되므로 주소를 고정할 필요가 없다.

#### 과잉 최적화 사례

```jsx
function OverOptimization({ count, name }) {
  // 비효율: 0.000001ms면 끝날 단순 산수
  const doubled = useMemo(() => count * 2, [count]);

  // 의미 없음: 단순 로그 출력에 useCallback
  const showWelcome = useCallback(() => {
    console.log(`반가워요, ${name}님!`);
  }, [name]);

  return <div onClick={showWelcome}>점수: {doubled}</div>;
}
```

#### 오래된 클로저(Stale Closure) 함정

- `useCallback`으로 함수를 박제했는데 의존성 배열에 최신 데이터를 누락하면, 함수는 '과거의 데이터'만 기억하는 타임캡슐이 된다.
- 세상은 100이 되었는데, 금고 속 함수는 여전히 "세상은 0이야!"라고 외치는 치명적 버그 발생.

### Part 2: 구조적 최적화 -- 훅 없이 성능을 끌어올리는 기법

#### 기법 1: 상태 밀어내기 (State Push-down)

상태가 너무 높은 부모에게 있으면, 그 아래 모든 자식이 **렌더링 폭발 반경(Render Blast Radius)**에 노출된다. 상태를 실제로 사용하는 가장 낮은 곳으로 격리한다.

```jsx
// Before: App에 text 상태 -> 모든 자식이 리렌더링
// After: InputBox로 text 상태를 내림 -> HeavyComponent는 영향 없음
function InputBox() {
  const [text, setText] = useState("");
  return <input value={text} onChange={(e) => setText(e.target.value)} />;
}
```

#### 기법 2: 컴포넌트 합성 (Composition with children)

`props.children`을 활용하면 이미 외부에서 만들어져 들어온 자식은 부모가 변해도 다시 렌더링되지 않는다.

```jsx
function ColorPickerWrapper({ children }) {
  const [color, setColor] = useState("#ffffff");
  return (
    <div style={{ backgroundColor: color }}>
      <button onClick={() => setColor("#fff9db")}>노란색</button>
      {/* children으로 들어온 무거운 컴포넌트는 배경색이 바뀌어도 리렌더링되지 않음 */}
      {children}
    </div>
  );
}

// 사용: <ColorPickerWrapper><HeavyComponent /></ColorPickerWrapper>
```

## 36강. 성능 최적화 골든 룰 -- 시니어 개발자의 코드 리뷰 기준

### 조기 최적화(Premature Optimization)의 함정

```jsx
// 나쁜 사례
const log = useCallback(() => console.log(count), [count]); // 단순 로그에 useCallback
const value = useMemo(() => 1 + 1, []);                     // 1+1에 useMemo
```

### 시니어의 전략적 최적화

```jsx
function SeniorReview() {
  const [items, setItems] = useState([...]);
  const [otherState, setOtherState] = useState(0);

  // 정밀 타격: 리스트 아이템은 개수가 많아질 수 있으므로 함수 주소를 고정
  const handleDelete = useCallback((id) => {
    setItems(prev => prev.filter(item => item.id !== id));
  }, []);

  return (
    <div>
      <button onClick={() => setOtherState(prev => prev + 1)}>부모 리렌더링</button>
      {/* 진짜 무거운 리스트에만 최적화를 투입 */}
      <ExpensiveList items={items} onItemClick={handleDelete} />
    </div>
  );
}
```

### 골든 룰 3가지

1. **측정 우선**: 프로파일러로 16ms를 초과하는 병목을 먼저 찾아라.
2. **구조 우선**: 상태를 아래로 밀어내어(Push-down) 렌더링 범위를 격리하라.
3. **정직한 의존성**: 성능을 위해 의존성 배열을 조작하지 마라. 화면에 틀린 값이 나오는 '빠른 앱'은 가치가 없다.

| 도구 | 역할 | 비유 | 사용 시점 |
|---|---|---|---|
| `useMemo` | **연산 결과(값)**를 캐싱 | 금고에 결과물 보관 | 무거운 계산이 의존성과 무관하게 반복될 때 |
| `useCallback` | **함수의 메모리 주소**를 고정 | 함수를 박제 | React.memo로 보호된 자식에게 함수를 전달할 때 |

| 컴포넌트 합성 | `children`으로 **엘리먼트 재사용** | 완성된 도시락 | 부모의 상태와 무관한 자식을 보호할 때 |

## 면접 핵심 Q&A

### Q1. useMemo와 useCallback의 차이점은?

**A.** `useMemo`는 콜백 함수의 **반환값(계산 결과)**을 메모이제이션하고, `useCallback`은 **함수 자체(메모리 주소)**를 메모이제이션한다. 실제로 `useCallback(fn, deps)`는 `useMemo(() => fn, deps)`와 동일하다. useMemo는 무거운 연산의 결과를 캐싱할 때, useCallback은 자식 컴포넌트에 전달하는 함수의 참조 동일성을 유지할 때 사용한다.

### Q2. React.memo는 어떻게 동작하며, 단독으로 완벽한 최적화가 되지 않는 이유는?

**A.** React.memo는 고차 컴포넌트(HOC)로 컴포넌트를 감싸서, 이전 Props와 현재 Props를 **얕은 비교(Shallow Comparison, O(1))**한다. 주소가 동일하면 렌더링을 건너뛴다(Bailout). 그러나 부모가 리렌더링될 때마다 함수/객체는 새로운 메모리 주소를 가지므로, 부모에서 `useCallback`/`useMemo`로 주소를 고정하지 않으면 React.memo의 비교가 매번 "다르다"고 판단하여 최적화가 무력화된다.

### Q3. 참조 동일성(Referential Equality)이란 무엇이며, 왜 성능 최적화에서 중요한가?

**A.** 자바스크립트에서 객체, 배열, 함수는 값이 아닌 **메모리 주소(참조)**로 비교된다. `{} === {}`는 `false`이다. 리액트는 Props의 변경 여부를 이 참조값으로 판단하므로, 리렌더링 때마다 새로 생성되는 객체/함수는 내용이 같더라도 "변경됨"으로 취급되어 자식 컴포넌트의 불필요한 리렌더링을 유발한다. 이것이 useMemo, useCallback, React.memo가 필요한 근본 이유이다.

### Q4. 모든 변수에 useMemo, 모든 함수에 useCallback을 쓰면 좋은가?

**A.** 아니다. 메모이제이션에는 대가가 따른다: (1) 메모리 점유로 GC가 회수하지 못하는 데이터 증가, (2) 의존성 배열 비교 연산 O(n) 비용, (3) 코드 복잡성 증가와 오래된 클로저(Stale Closure) 버그 위험. 단순 산수(`count * 2`)나 원시 타입에는 불필요하다. **"연산 비용 > 비교 비용"**일 때만, 그리고 **React.memo와 결합하여 자식 리렌더링을 막아야 할 때만** 사용해야 한다.

### Q5. 프로파일러 없이 최적화를 하는 것은 왜 위험한가?

**A.** "느린 것 같다"는 추측에 기반한 최적화는 실제 병목과 다른 곳에 불필요한 복잡성을 추가할 수 있다. React Profiler를 사용하면 어떤 컴포넌트가 몇 ms를 소모하는지, 왜 리렌더링되었는지를 데이터로 확인할 수 있다. 60FPS를 유지하려면 한 프레임이 **16ms** 안에 끝나야 하며, 이를 초과하는 컴포넌트를 프로파일러로 식별한 후 정밀하게 최적화하는 것이 전문가의 방법이다.

### Q6. 구조적 최적화(State Push-down, Composition)가 훅보다 우선되어야 하는 이유는?

**A.** 상태 밀어내기(State Push-down)와 컴포넌트 합성(Composition with children)은 추가 메모리 비용이 없고, 의존성 관리 부담이 없으며, 코드가 더 깔끔해진다. 상태를 실제 사용처로 격리하면 렌더링 폭발 반경 자체가 줄어들어 훅 없이도 성능 문제가 해결된다. **훅 기반 최적화는 구조적 최적화로 해결이 안 될 때의 최후 수단**이어야 한다.

### Q7. useCallback의 의존성 배열을 빈 배열로 두면 어떤 위험이 있는가?

**A.** 함수 내부에서 참조하는 상태나 props가 변해도 함수는 초기값만 기억하는 **'냉동 상태(Stale Closure)'**가 된다. 예를 들어 `size` 상태를 사용하는 함수의 의존성 배열에서 `[size]`를 `[]`로 바꾸면, `size`를 100으로 바꿔도 함수는 영원히 초기 `size` 값으로만 동작한다. 성능을 위해 의존성을 조작하면 화면에 틀린 값이 나오는 치명적 버그가 발생한다.

## 용어 글로서리

| 용어 | 설명 |
|---|---|
| 리렌더링(Re-rendering) | 컴포넌트 함수가 다시 실행되어 내부의 모든 변수와 연산이 새로 만들어지는 과정 |
| 참조 동일성(Referential Equality) | 두 값이 같은 메모리 주소를 가리키는지 비교하는 방식. 객체/배열/함수에 적용 |
| 메모이제이션(Memoization) | 이전 계산 결과를 저장해두고 동일한 입력에 대해 즉시 반환하는 최적화 기법 |
| 의존성 배열(Dependency Array) | useMemo, useCallback, useEffect의 두 번째 인자. 값이 변할 때만 콜백을 재실행 |
| 얕은 비교(Shallow Comparison) | Props의 최상위 속성의 메모리 주소만 비교하는 O(1) 연산 |
| 베일아웃(Bailout) | Props가 동일하다고 판단되어 컴포넌트의 렌더링 사이클을 건너뛰는 리액트 내부 동작 |
| 고차 컴포넌트(HOC) | 컴포넌트를 인자로 받아 새로운 컴포넌트를 반환하는 함수. React.memo가 대표적 |
| 오래된 클로저(Stale Closure) | 메모이제이션된 함수가 과거 시점의 변수를 참조하여 최신 값을 반영하지 못하는 버그 |
| 렌더링 폭발 반경(Render Blast Radius) | 상태 변경 시 리렌더링이 전파되는 컴포넌트 트리의 범위 |
| 상태 밀어내기(State Push-down) | 상태를 실제 사용하는 가장 하위 컴포넌트로 격리하는 구조적 최적화 기법 |
| 컴포넌트 합성(Composition) | children Props를 활용해 외부에서 주입된 컴포넌트가 부모 리렌더링의 영향을 받지 않게 하는 패턴 |
| 프로파일러(Profiler) | React DevTools에서 컴포넌트별 렌더링 시간과 원인을 측정하는 진단 도구 |
| 플레임 차트(Flame Chart) | 프로파일러에서 컴포넌트별 렌더링 비용을 막대 길이와 색상으로 시각화한 차트 |
