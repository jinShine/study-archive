# 35-2강. 구조적 최적화 — 훅보다 강력한 컴포넌트 설계의 힘

## 도입

지금까지 우리는 `useMemo`나 `useCallback` 같은 '성능 진통제'들을 배웠다. 하지만 진정한 고수는 약을 쓰기 전에 먼저 '체질'부터 개선한다.

최적화 훅을 단 한 줄도 쓰지 않고도, 컴포넌트의 위치를 바꾸거나 구조를 재설계하는 것만으로 성능을 극적으로 끌어올리는 **구조적 최적화(Structural Optimization)** 기법을 다룬다.

## 개념 설명

### 상태 밀어내기 (State Push-down)

상태가 너무 높은 부모에게 있으면, 그 아래 모든 자식이 **리렌더링 폭발 반경(Render Blast Radius)**에 노출된다. 상태를 실제로 사용하는 가장 낮은 곳으로 격리하면, 불필요한 리렌더링을 훅 없이 차단할 수 있다.

### 컴포넌트 합성 (Composition)

`props.children`을 활용하면 리액트 엔진의 '엘리먼트 재사용' 원리를 이용할 수 있다. 이미 외부에서 만들어져 들어온 자식은 부모가 변해도 다시 요리되지 않는 **'완성된 도시락'**과 같다.

`props.children`으로 전달된 엘리먼트는 부모 컴포넌트가 실행되기 전인 외부에서 이미 객체 형태로 생성된 상태다. 따라서 부모가 리렌더링되어도 리액트는 "이 도시락은 밖에서 온 거네?"라고 판단하며 내부를 다시 뒤지지 않는 **자동 베일아웃(Bail-out)** 효과를 준다.

## 코드 예제

### 1단계: 상태 밀어내기 — InputBox.jsx

```jsx
import { useState } from 'react';

// 해결책: 부모(App)에 있던 text 상태를 실제 사용하는 이 컴포넌트로 내린다.
export default function InputBox() {
  const [text, setText] = useState("");

  return (
    <div style={{ padding: '10px', border: '1px solid #3b82f6', borderRadius: '8px' }}>
      <h4>텍스트 입력 (상태 격리)</h4>
      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="여기서 글자를 쳐도 옆집 차트는 가만히 있습니다."
        style={{ width: '100%', padding: '8px' }}
      />
    </div>
  );
}
```

### 2단계: 컴포넌트 합성 — ColorPickerWrapper.jsx

```jsx
import { useState } from 'react';

// Wrapper가 리렌더링되어도, 외부에서 주입된 'children'은 주소가 변하지 않는다.
export default function ColorPickerWrapper({ children }) {
  const [color, setColor] = useState("#ffffff");

  return (
    <div style={{ backgroundColor: color, padding: '30px', transition: '0.3s' }}>
      <h3>배경색 변경 (컴포넌트 합성)</h3>
      <button onClick={() => setColor("#fff9db")}>노란색</button>
      <button onClick={() => setColor("#e7f5ff")} style={{ marginLeft: '10px' }}>파란색</button>

      {/* 핵심: children으로 들어온 무거운 컴포넌트는 배경색이 바뀌어도 리렌더링되지 않는다. */}
      <div style={{ marginTop: '20px' }}>
        {children}
      </div>
    </div>
  );
}
```

## 코드 해설

- **InputBox**: 사용자가 글자를 입력해도 폭발은 `InputBox` 내부에서만 일어난다. 부모 컴포넌트가 다시 그려지지 않으니, 옆에 있던 `HeavyComponent`는 훅 없이도 완벽하게 보호된다.
- **ColorPickerWrapper**: `children`으로 받은 엘리먼트는 부모 밖에서 이미 생성된 객체이므로, 부모가 리렌더링되어도 리액트가 자동으로 베일아웃한다. `<HeavyComponent />`를 직접 내부에서 호출하면 배경색을 바꿀 때마다 리렌더링이 발생하지만, `children`으로 받으면 발생하지 않는다.

### 설계의 우선순위 전략

| 순서 | 전략 | 설명 |
|------|------|------|
| 1단계 | 상태 밀어내기 | 상태를 실제로 사용하는 가장 낮은 곳으로 격리 |
| 2단계 | 컴포넌트 합성 | `children`을 활용해 렌더링 폭발 범위를 물리적으로 분리 |
| 3단계 | 최적화 훅 | 위 구조로도 해결되지 않는 거대한 데이터나 연산에만 `useMemo`, `React.memo` 투입 |

## 실무 비유

성능 최적화의 골든 룰은 **'설계 우선, 훅은 최후'**다.

- **상태 밀어내기**는 "불이 난 방만 격리하는 방화벽"과 같다. 불(상태 변경)이 온 건물(전체 컴포넌트 트리)로 번지지 않도록 해당 방(컴포넌트)에만 가둔다.
- **컴포넌트 합성**은 "외부에서 배달된 도시락"과 같다. 주방(부모)이 다시 요리를 시작해도 이미 완성되어 배달된 도시락(children)은 영향을 받지 않는다.

## 핵심 포인트

- 구조적 최적화는 훅을 쓰기 전에 먼저 시도해야 하는 1순위 전략이다
- **상태 밀어내기(State Push-down)**: 상태를 실제로 사용하는 가장 낮은 컴포넌트로 이동시켜 리렌더링 폭발 반경을 줄인다
- **컴포넌트 합성(Composition)**: `props.children`으로 전달된 엘리먼트는 부모 리렌더링 시 자동으로 베일아웃된다
- `children`이 보호되는 원리: 외부에서 이미 생성된 객체이므로 부모 실행과 무관하게 참조가 유지된다
