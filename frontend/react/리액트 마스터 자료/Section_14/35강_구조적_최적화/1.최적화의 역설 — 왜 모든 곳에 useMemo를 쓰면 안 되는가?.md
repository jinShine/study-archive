# 35-1강. 최적화의 역설 — 왜 모든 곳에 useMemo를 쓰면 안 되는가?

## 도입

지금까지 성능을 올리는 다양한 무기들을 배웠다. 하지만 공학의 세계에는 '공짜 점심은 없다'는 유명한 격언이 있다. 우리가 사용하는 모든 최적화 도구는 리액트 엔진에게 상당한 대가를 요구한다.

무분별한 최적화가 왜 오히려 앱을 더 느리게 만들고 코드를 지저분하게 만드는지, 그 이면에 숨겨진 비용과 최적화의 역설을 파헤쳐 본다.

## 개념 설명

**최적화가 지불하는 세 가지 대가:**

최적화 훅을 호출하는 순간, 리액트 내부에서는 보이지 않는 지출이 발생한다.

**1. 메모리 점유와 가비지 컬렉션(GC)의 부담**

브라우저에는 쓰레기 데이터를 치워주는 '청소 로봇'인 가비지 컬렉터가 있다. 하지만 `useMemo`로 데이터를 금고에 넣는 순간, 청소 로봇은 그 데이터를 치우지 못하고 계속 쌓아둔다. 결국 메모리가 부족해져 전반적인 속도가 굼떠질 수 있다.

**2. 의존성 배열의 비교 연산 비용 O(n)**

리액트는 렌더링마다 의존성 배열의 값들을 이전과 대조한다. 만약 단순한 산수 연산을 지키겠다고 이 훅을 쓰면, "다시 계산하는 찰나의 시간"보다 "지난번과 같은지 체크리스트를 검토하는 시간"이 더 길어지는 배보다 배꼽이 더 큰 상황이 벌어진다.

**3. 원시 타입에 대한 무의미한 최적화**

숫자(1,000)나 문자열은 주소값이 아니라 '값 자체'로 비교된다. 주소를 고정할 필요가 전혀 없다. 여기에 최적화를 거는 것은 1,000원짜리 지폐를 지키겠다고 수백만 원짜리 금고를 사는 것과 같은 자원 낭비다.

## 코드 예제

### 과잉 최적화의 전형적인 사례

```jsx
// 최적화가 오히려 독이 되는 '과잉 최적화'의 전형적인 사례
function OverOptimization({ count, name }) {
  // 1. [비효율]: 0.000001ms면 끝날 단순 산수
  // 리액트 엔진은 매번 [count] 체크리스트를 확인하느라 에너지를 쓴다
  const doubled = useMemo(() => count * 2, [count]);

  // 2. [의미 없음]: 단순히 로그만 찍는 가벼운 함수
  // GC는 이 함수가 차지한 메모리를 회수하지 못하게 된다
  const showWelcome = useCallback(() => {
    console.log(`반가워요, ${name}님!`);
  }, [name]);

  return (
    <div onClick={showWelcome}>
      현재 점수의 두 배는 {doubled}점입니다.
    </div>
  );
}
```

## 코드 해설

- `useMemo(() => count * 2, [count])`: `count * 2`는 0.000001ms면 끝나는 단순 곱셈이다. 리액트가 매 렌더링마다 의존성 배열 `[count]`를 이전 값과 비교하는 비용이 곱셈 자체보다 더 크다. 최적화가 아니라 오히려 오버헤드를 추가한 셈이다.
- `useCallback(() => { console.log(...) }, [name])`: 단순히 로그만 찍는 가벼운 함수를 메모이제이션하면, 가비지 컬렉터가 이 함수의 메모리를 회수하지 못한다. 이 함수가 `React.memo`로 감싼 자식에게 전달되는 것이 아니라면 참조 동일성을 유지할 이유가 없다.

**오래된 클로저(Stale Closure)의 함정:**

성능을 0.1% 올리려다 화면에 틀린 값이 나오는 치명적인 버그를 만들 수 있다.

- **타임캡슐 현상**: `useCallback`으로 함수를 박제했는데, 의존성 배열에 최신 데이터를 누락하면 그 함수는 '과거의 데이터'만 기억한다.
- **결과**: 세상은 변해서 숫자가 100이 되었는데, 금고 속 함수는 여전히 "세상은 0이야!"라고 외치는 꼴이 되어버린다. 이것이 바로 유지보수의 재앙이라 불리는 오래된 클로저 문제다.

## 실무 비유

- **금고와 지폐**: 1,000원짜리 지폐를 지키겠다고 수백만 원짜리 금고를 사는 것이 원시 타입에 `useMemo`를 쓰는 것과 같다.
- **청소 로봇**: 가비지 컬렉터는 메모리의 청소 로봇이다. `useMemo`/`useCallback`으로 값을 붙잡아 두면 청소 로봇이 치울 수 없어 메모리가 쌓인다.
- **타임캡슐**: `useCallback`의 의존성 배열 누락은 과거 데이터를 타임캡슐에 가두는 것과 같다. 세상은 변했는데 함수만 과거에 머물러 있다.

## 핵심 포인트

1. `useMemo`와 `useCallback`은 공짜가 아니다. 메모리 점유, 의존성 비교 연산, GC 방해라는 세 가지 대가를 치른다.
2. 원시 타입(숫자, 문자열)은 값 자체로 비교되므로 주소를 고정할 필요가 없다. `useMemo(() => count * 2, [count])`는 낭비다.
3. 의존성 배열의 비교 비용이 연산 비용보다 클 수 있다. 단순 연산에 `useMemo`를 쓰면 배보다 배꼽이 더 커진다.
4. **최적화의 골든 룰**: (1) 연산 비용 > 비교 비용일 때, (2) `React.memo`로 보호된 자식에게 참조 동일성을 유지해야 할 때만 사용한다.
5. 최적화는 많이 할수록 좋은 것이 아니라, 필요한 지점에 정밀하게 타격하는 외과 수술과 같아야 한다. 리액트의 기본 성능을 신뢰하는 것이 때로는 가장 빠르고 깨끗한 코드를 만드는 지름길이다.
