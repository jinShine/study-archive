# 26강. 렌더링의 비용 — 리액트는 왜, 언제 화면을 다시 그리는가?

## 도입

리액트가 화면을 다시 그리는 과정(Rendering) 뒤에는 CPU와 메모리가 열심히 계산기를 두드리는 "비용"이 숨어 있다. 이번 강의에서는 어떤 상황에서 리액트가 불필요한 계산을 하는지, 리렌더링의 폭포수 현상(Waterfall)을 코드로 정밀하게 해부한다.

이번 26강부터는 리액트 개발자의 실력을 가르는 가장 중요한 잣대인 "성능 최적화"의 세계로 들어간다.

## 개념 설명

**리렌더링이 발생하는 조건:**
1. 컴포넌트의 상태(State)가 변경될 때
2. 부모 컴포넌트가 리렌더링될 때 (자식은 변화가 없어도 연쇄 리렌더링)
3. 전달받은 props가 변경될 때

**참조 동일성 (Referential Equality):**
리액트가 "데이터가 바뀌었나?"를 판단할 때 사용하는 핵심 기준이다.

| 타입 | 비교 방식 | 리렌더링 시 변화 |
|------|----------|----------------|
| 원시 타입 (숫자, 문자) | 값이 같으면 동일 | 값이 같으면 렌더링 하지 않으려 노력 |
| 객체 타입 (객체, 배열, 함수) | 주소(참조)로 비교 | 리렌더링마다 새 주소 생성 → "다른 데이터"로 판단 |

핵심: `[] === []`는 `false`다. 리렌더링이 일어나면 함수 안의 모든 객체와 함수는 새로운 주소값을 가진다. 리액트는 이를 "완전히 새로운 데이터"로 오해하고 자식에게 렌더링 신호를 보낸다.

## 코드 예제

### 리렌더링 폭포수 확인 (RenderingProcess)

```jsx
import { useState } from "react";
import { ChildComponent } from "./ChildComponent";

export default function RenderingProcess() {
  // 상태 선언: 이 값이 바뀌면 함수 전체가 '재실행'된다
  const [count, setCount] = useState(0);
  const [text, setText] = useState("");

  // 무거운 연산 모사: 리렌더링 때마다 CPU는 이 루프를 다시 돌아야 한다
  console.log("부모 컴포넌트(RenderingProcess) 렌더링 시작!");

  const startTime = performance.now();
  let heavyValue = 0;
  for (let i = 0; i < 10000000; i++) { heavyValue++; }
  const endTime = performance.now();

  console.log(`연산 소요 시간: ${(endTime - startTime).toFixed(2)}ms`);

  return (
    <div style={{ padding: '20px', border: '2px solid #ff4757', borderRadius: '15px' }}>
      <h2>부모 컴포넌트</h2>
      <p>연산 결과: {heavyValue}</p>

      <button onClick={() => setCount(count + 1)}>숫자 올리기 ({count})</button>
      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="글자를 입력해도 리렌더링이 일어납니다."
        style={{ width: '100%', marginTop: '10px', padding: '8px' }}
      />

      {/* 부모가 리렌더링되면 자식은 아무 이유 없이 다시 그려진다 */}
      <ChildComponent />
    </div>
  );
}
```

### 억울하게 다시 그려지는 자식 (ChildComponent)

```jsx
export function ChildComponent() {
  // 부모가 리렌더링될 때마다 이 콘솔이 찍힌다
  console.log("자식 컴포넌트(ChildComponent)도 다시 그려짐...");

  return (
    <div style={{ marginTop: '20px', padding: '15px', border: '1px dashed #ffa502' }}>
      <h4>자식 컴포넌트</h4>
      <p>나는 바뀐 게 없는데 부모님 때문에 다시 불려왔어요.</p>
    </div>
  );
}
```

## 코드 해설

- `const [text, setText]`: `count`와 상관없는 `text`만 바꿔도 함수 전체가 다시 실행된다. 리액트는 상태 하나만 바뀌어도 해당 컴포넌트의 모든 변수를 새로 만든다.
- `for (let i = 0; i < 10000000; ...)`: 렌더링이 단순한 "그리기"가 아니라 "계산 비용"임을 체감하기 위한 장치다. 리렌더링이 잦아지면 이 계산 때문에 화면이 버벅인다.
- `<ChildComponent />`: 자식 컴포넌트의 console.log가 찍히는 것은 눈에 보이는 화면 변화는 없지만, 리액트가 이 함수를 "다시 실행"했다는 증거다. 이것이 쌓이면 전체 성능 저하로 이어진다.

**만약 React.memo 같은 최적화 도구가 없다면?**
앱이 커질수록 최상위 App에서 상태 하나만 바뀌어도 수천 개의 하위 자식 컴포넌트가 동시에 재실행된다. 전구 하나를 갈기 위해 온 동네 집집마다 방문해서 전등 상태를 확인하는 것과 같은 비효율이다.

## 실무 비유

- **리렌더링 폭포수**: 아파트 꼭대기에서 물을 쏟으면 모든 층이 젖는 것과 같다. 부모의 상태 변경이 자식 전체로 폭포수처럼 전파된다.
- **참조 동일성**: 쌍둥이가 같은 얼굴이어도 다른 사람인 것처럼, `[]`와 `[]`는 내용이 같아도 메모리 주소가 다르면 리액트는 "다른 데이터"로 판단한다.
- **무거운 연산**: 매번 문을 열 때마다 집 전체를 청소하는 것과 같다. 필요한 방만 청소하는 것이 효율적이다.

## 핵심 포인트

1. 리액트에서 상태가 바뀌면 해당 컴포넌트 함수 전체가 재실행되며, 모든 변수와 연산이 새로 만들어진다.
2. 부모가 리렌더링되면 자식 컴포넌트도 자동으로 리렌더링된다 (변화가 없어도).
3. `[] === []`는 `false`다. 리렌더링 시 객체/배열/함수는 새 주소값을 가지므로 "다른 데이터"로 인식된다.
4. 이 참조 동일성 문제가 불필요한 자식 리렌더링의 근본 원인이다.
5. 이 문제를 해결하기 위해 `React.memo`, `useMemo`, `useCallback` 같은 최적화 도구가 존재한다.
