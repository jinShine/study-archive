# 27강. 메모이제이션(Memoization) — 컴퓨터 공학으로 이해하는 '기억하며 달리기'의 원리

## 도입

리액트 성능 최적화의 심장부인 **메모이제이션(Memoization)**의 원리를 파헤쳐 봅니다. 단순히 훅의 사용법을 외우는 것이 아니라, "이미 계산한 것은 기억해두고 재활용한다"는 컴퓨터 공학의 근본적인 접근법을 이해하는 것이 목표입니다. 이 원리를 깨우치면 `useMemo`와 `useCallback`이 왜 필요한지 스스로 납득하게 될 것입니다.

## 개념 설명

### 메모이제이션이 없는 비효율적인 연산

기억 장치가 없을 때 컴퓨터가 얼마나 고생하는지 피보나치 수열 코드로 확인해 봅니다.

### 메모이제이션을 적용한 '비밀 장부' 연산

이미 계산한 값을 '기억'하는 로직을 추가하여 연산 속도를 기적적으로 끌어올립니다.

### 시간과 공간의 거래 (Trade-off)

메모이제이션은 **세상에 공짜가 없다**는 것을 보여주는 가장 명확한 사례입니다.

- **이득 (Time Complexity):** O(2^n)에서 O(n)으로 연산 횟수를 획기적으로 줄여 실행 속도를 높입니다.
- **대가 (Space Complexity):** 이전 결과값들을 보관해야 하므로 메모리(RAM) 공간을 추가로 사용합니다.
- **결론:** 모든 함수에 메모이제이션을 쓰는 것은 낭비입니다. 연산 과정이 매우 복잡하여 '다시 계산하는 비용'이 '메모리에 저장하는 비용'보다 압도적으로 클 때만 사용해야 합니다.

## 코드 예제

### 1. 순수 재귀 방식 (비효율)

```js
// src/utils/fibonacci.js

// [비효율] 순수 재귀 방식
// 숫자가 조금만 커져도 같은 값을 구하기 위해 이미 했던 계산을 수만 번 반복합니다.
export function pureFibonacci(n) {
  if (n <= 1) return n;
  // n이 40만 되어도 연산량이 약 O(2^n)으로 폭발하며 브라우저가 멈출 수 있습니다.
  return pureFibonacci(n - 1) + pureFibonacci(n - 2);
}
```

### 2. 메모이제이션 방식 (효율)

```js
// src/utils/fibonacci.js 하단에 추가

// [효율] 메모이제이션 방식
// 한 번 계산한 결과는 'memo'라는 비밀 장부에 기록해둡니다.
const memo = {};

export function memoFibonacci(n) {
  // A. 장부를 먼저 확인합니다. 이미 계산한 적이 있다면 즉시 반환합니다.
  if (n in memo) {
    return memo[n];
  }

  // B. 장부에 없다면 처음으로 계산을 수행합니다.
  if (n <= 1) return n;

  // C. 계산 결과를 그냥 내보내지 않고, 반드시 장부에 적어둡니다.
  const result = memoFibonacci(n - 1) + memoFibonacci(n - 2);
  memo[n] = result;

  return result;
}
```

## 코드 해설

| 코드 | 역할 |
|------|------|
| `const memo = {}` | 함수 밖에 선언하여 함수가 종료되어도 계산 기록이 사라지지 않게 보호하는 **'데이터 금고'** |
| `if (n in memo)` | CPU를 쓰기 전에 먼저 **'검색'**을 수행 — 계산보다 검색이 훨씬 빠름 |
| `memo[n] = result` | 미래의 나를 위해 오늘의 고생을 기록하는 **'최적화의 핵심'** — 이 덕분에 시간 복잡도가 O(n)으로 개선 |

### 이 코드, 왜 문제가 될까요?

순수 재귀 방식은 이전에 계산했던 결과를 전혀 기억하지 못합니다. `Fibonacci(5)`를 구하기 위해 `Fibonacci(3)`을 호출하고, 또 그 안에서 수많은 중복 호출이 발생합니다. 구해야 하는 숫자 `n`이 커질수록 시간 복잡도가 O(2^n)이 되어 기하급수적으로 느려집니다.

## 실무 비유

> **리액트 최적화의 함정:** 리액트의 `useMemo`를 단순히 "성능이 좋아진다니까 모든 변수에 다 감싸자"라고 생각하면 오산입니다. 메모이제이션을 위한 장부를 만들고 값을 대조하는 과정 자체가 비용이기 때문에, 가벼운 연산에 남발하면 오히려 앱이 더 느려지고 메모리만 많이 먹게 됩니다.

> **참조 동일성과의 연결:** 리액트에서 메모이제이션이 특히 중요한 이유는 단순히 연산 속도 때문만이 아닙니다. "이전의 그 데이터 주소와 지금 주소가 같은가?"를 증명하여 **불필요한 리렌더링 폭포수를 막아주는 방패 역할**을 하기 때문입니다.

## 핵심 포인트

- 메모이제이션은 **'메모리(공간)'를 써서 '시간'을 사는 기술**이다
- 순수 재귀 O(2^n) → 메모이제이션 O(n)으로 시간 복잡도가 획기적으로 개선된다
- 모든 곳에 메모이제이션을 남발하면 오히려 성능이 떨어질 수 있다
- 리액트에서는 연산 최적화뿐 아니라 **참조 동일성 보존**이 핵심 목적이다
