# 34강. 성능 측정의 기술 — 짐작하지 말고 프로파일러로 증명하라

## 도입

성능을 올리는 다양한 도구를 배웠지만, 진정한 고수는 무기를 휘두르기 전에 먼저 적이 어디에 있는지 정확히 파악한다. "여기가 좀 느린 것 같은데?"라는 막연한 추측은 위험하다. 리액트 개발자 도구의 가장 강력한 진단 장비인 **프로파일러(Profiler)**를 사용하여, 실제 성능 범인을 데이터로 검거하고 최적화의 근거를 마련하는 과정을 다룬다.

## 개념 설명

### 16ms의 법칙

사용자에게 60FPS의 부드러운 화면을 제공하려면, 한 프레임의 연산이 **16ms 안에** 끝나야 한다. 100ms가 소모된다는 것은 초당 10프레임도 안 나오는 '뚝뚝 끊기는' 앱이라는 공학적 증거다.

### 플레임 차트(Flame Chart) 읽기

- **노란색/긴 막대**: 연산 비용이 높음 (범인)
- **회색/차트 제외**: 렌더링이 생략됨 (베일아웃 성공)

### 측정의 가치

"느린 것 같아요"라는 말 대신 "HeavyComponent가 100ms를 점유하며 메인 스레드를 방해하고 있습니다"라고 **데이터로 말하는 것**이 전문가의 태도다.

## 코드 예제

### 성능 병목 현상 재현 — 무거운 컴포넌트

```jsx
// HeavyComponent.jsx
import { useState } from 'react';

// [함정 컴포넌트]: 렌더링 시마다 강제로 100ms를 소모한다.
export default function HeavyComponent() {
  const start = performance.now();

  // 의도적인 부하: 100ms가 지날 때까지 CPU 메인 스레드를 붙잡아둔다.
  while (performance.now() - start < 100) {
    // CPU가 열심히 일하는 중...
  }

  return (
    <div style={{ padding: '20px', border: '2px solid #ff4757', marginTop: '20px', borderRadius: '10px' }}>
      <h3>매우 무거운 컴포넌트</h3>
      <p>한 번 그릴 때마다 <b>100ms</b>가 걸린다.</p>
    </div>
  );
}
```

### 프로파일러 진단 대시보드

```jsx
// PerformanceDashboard.jsx
import { useState } from 'react';
import HeavyComponent from './HeavyComponent';

export default function PerformanceDashboard() {
  const [text, setText] = useState("");
  const [count, setCount] = useState(0);

  return (
    <div style={{ padding: '30px' }}>
      <h1>성능 진단 대시보드</h1>
      <p>인풋에 글자를 치면 부모가 리렌더링되면서 자식도 매번 100ms씩 고생한다.</p>

      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="글자를 입력하면 버벅임을 느낄 수 있습니다..."
        style={{ padding: '10px', width: '350px' }}
      />
      <button onClick={() => setCount(prev => prev + 1)} style={{ marginLeft: '10px' }}>
        단순 클릭: {count}
      </button>

      <HeavyComponent />
    </div>
  );
}
```

### 데이터 기반의 해결 — React.memo 적용

```jsx
// HeavyComponent.jsx (최적화 후)
import { memo } from 'react';

// 해결책: 부모의 text가 바뀌어도 자신(Props)이 바뀌지 않으면 렌더링을 생략한다.
const HeavyComponent = memo(() => {
  const start = performance.now();
  while (performance.now() - start < 100) {}

  return (
    <div style={{ padding: '20px', border: '2px solid #10b981', marginTop: '20px', borderRadius: '10px' }}>
      <h3>최적화가 완료된 컴포넌트</h3>
      <p>이제 부모가 변해도 다시 그려지지 않는다.</p>
    </div>
  );
});

export default HeavyComponent;
```

### 프로파일러 사용법

1. 크롬 개발자 도구(F12)의 **Profiler** 탭으로 이동한다.
2. 톱니바퀴 아이콘을 눌러 **"Record why each component rendered"** 옵션을 켠다.
3. 파란색 녹화 버튼을 누르고 인풋창에 글자를 몇 개 입력한 뒤 정지한다.
4. 플레임 차트에서 노란색/긴 막대가 범인이다. 막대를 클릭하면 "Why did this render?" 정보를 확인할 수 있다.

## 코드 해설

| 요소 | 역할 |
|------|------|
| `performance.now()` + `while` 루프 | CPU 메인 스레드를 의도적으로 100ms 동안 점유하여 병목 현상을 재현한다. |
| `setText(e.target.value)` | 글자를 입력할 때마다 부모가 리렌더링되고, 자식도 함께 리렌더링되어 버벅임이 발생한다. |
| `memo(() => ...)` | Props가 없는(또는 변하지 않는) 컴포넌트를 감싸면, 부모의 리렌더링과 무관하게 렌더링을 생략한다. |
| Profiler 탭 | 각 컴포넌트가 렌더링에 소비한 시간을 시각적으로 보여주는 진단 도구다. |
| "Why did this render?" | 프로파일러가 제공하는 정보로, 해당 컴포넌트가 왜 다시 그려졌는지 원인을 알려준다. |

## 실무 비유

병원 진료를 생각해 보자. 환자가 "어딘가 아파요"라고 하면, 의사는 짐작으로 수술하지 않는다. 먼저 CT나 MRI(프로파일러)를 찍어 정확한 병변(병목 컴포넌트)을 찾고, 그 부위만 정밀하게 치료(React.memo 적용)한다. 프로파일러 없이 최적화하는 것은 진단 없이 수술하는 것과 같다.

## 핵심 포인트

- 60FPS를 유지하려면 한 프레임당 연산이 **16ms 이내**에 끝나야 한다.
- 리액트 개발자 도구의 **Profiler** 탭에서 "Record why each component rendered" 옵션을 켜면 각 컴포넌트의 렌더링 원인과 소요 시간을 확인할 수 있다.
- 플레임 차트에서 **노란색/긴 막대**가 성능 범인이고, **회색**은 베일아웃이 성공한 컴포넌트다.
- 측정 -> 진단 -> 최적화 -> 재측정의 순서를 지키는 것이 데이터 기반 성능 개선의 정석이다.
