# 18강. 커스텀 훅을 통한 추상화 전략 — 전역 상태를 우아하게 다루는 법

## 도입

지금까지 Context를 분리하고 다중 엔진을 구축했다. 하지만 소비자 컴포넌트에서 매번 `useContext(AuthStateContext)`를 직접 호출하면, 내부 구현(Context 객체명)에 의존하게 된다. 이번 강에서는 **커스텀 훅으로 Context 접근을 추상화**하여, 컴포넌트가 내부 구현을 몰라도 직관적으로 전역 상태를 사용할 수 있게 만드는 전략을 배운다.

## 개념 설명

커스텀 훅 추상화의 핵심은 두 가지다:

1. **편의성** — `useContext(AuthStateContext)` 대신 `useAuthState()`만 호출하면 된다. 임포트할 것이 줄고, 비즈니스 의도가 코드에 명확히 드러난다.
2. **안전장치** — Provider 밖에서 훅을 사용하면 React는 에러 없이 `undefined`를 반환한다. 커스텀 훅 안에서 이를 감지하여 **명시적인 에러 메시지**를 던져주면, 디버깅 시간을 극적으로 줄일 수 있다.

이 패턴을 적용하면 나중에 Context를 다른 상태 관리 라이브러리로 교체하더라도, **커스텀 훅 내부만 수정**하면 소비자 컴포넌트는 전혀 건드릴 필요가 없다.

## 코드 예제

### 안전장치가 포함된 커스텀 훅 (AuthContext에 추가)

```jsx
import { createContext, useReducer, useMemo, useContext } from "react";

const AuthStateContext = createContext();
const AuthDispatchContext = createContext();

const authReducer = (state, action) => {
  if (action.type === "LOGIN") return { user: { name: "정용수" }, isLoading: false };
  if (action.type === "LOGOUT") return { user: null, isLoading: false };
  return state;
};

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, { user: null, isLoading: false });
  return (
    <AuthStateContext.Provider value={state}>
      <AuthDispatchContext.Provider value={dispatch}>
        {children}
      </AuthDispatchContext.Provider>
    </AuthStateContext.Provider>
  );
}

// 커스텀 훅: 상태 전용 수신기
export function useAuthState() {
  const context = useContext(AuthStateContext);

  // 안전장치: Provider 밖에서 이 훅을 사용하면 명시적 에러를 던진다.
  if (context === undefined) {
    throw new Error("useAuthState는 반드시 AuthProvider 안에서 사용해야 합니다!");
  }
  return context;
}

// 커스텀 훅: 명령 전용 수신기
export function useAuthDispatch() {
  const context = useContext(AuthDispatchContext);

  if (context === undefined) {
    throw new Error("useAuthDispatch는 반드시 AuthProvider 안에서 사용해야 합니다!");
  }
  return context;
}
```

### 컴포넌트 코드 다이어트 (UserProfile)

```jsx
// Before: import { useContext } from 'react'; import { AuthStateContext... } from '...';
// After: 우리가 만든 전용 훅 하나면 충분하다.
import { useAuthState, useAuthDispatch } from "../contexts/AuthContext";

export default function UserProfile() {
  // 내부 구현(Context 객체명 등)을 몰라도 직관적으로 사용 가능하다.
  const { user, isLoading } = useAuthState();
  const dispatch = useAuthDispatch();

  if (isLoading) return <p>로딩 중...</p>;
  if (!user) return <p>로그인이 필요합니다.</p>;

  return (
    <div style={{ padding: '20px', border: '1px solid #ddd' }}>
      <h2>{user.name}님의 프로필</h2>
      <button onClick={() => dispatch({ type: "LOGOUT" })}>로그아웃</button>
    </div>
  );
}
```

## 코드 해설

- **`if (context === undefined)`:** 초보 개발자가 실수로 `AuthProvider`로 감싸지 않은 곳에서 이 훅을 쓰면, 원인 모를 버그 때문에 밤을 새우게 된다. 이 조건문은 "여기가 아니에요!"라고 정확한 위치를 짚어주는 친절한 이정표다.
- **`export function useAuth...`:** 이제 외부 컴포넌트에서는 `useContext`를 몰라도 이 함수만 불러오면 끝이다.
- **`useAuthState()`:** 컴포넌트는 "데이터가 Context에서 오는지, Redux에서 오는지" 알 필요가 없다. 그저 "유저 상태가 필요해"라는 의도만 코드로 표현할 뿐이다. 이것이 **유지보수성이 극대화**되는 시점이다.

## 실무 비유

커스텀 훅은 **자동차의 핸들**과 같다:
- 운전자(컴포넌트)는 핸들만 돌리면 된다. 내부에서 파워 스티어링 펌프가 어떻게 작동하는지(Context 구조) 알 필요가 없다.
- 핸들 없이 직접 유압 장치를 조작하면(useContext를 직접 호출) 작동은 하지만, 실수할 확률이 높고 불편하다.
- 나중에 전기 스티어링으로 교체해도(상태 관리 라이브러리 변경) 핸들 인터페이스는 그대로다.

## 핵심 포인트

| 비교 항목 | 직접 호출 (기존) | 커스텀 훅 (추상화) |
|-----------|----------------|-------------------|
| 임포트 개수 | `useContext`, Context 객체 등 다수 | 전용 훅 하나 |
| 코드 가독성 | `useContext(AuthContext)` (기술 중심) | `useAuthState()` (비즈니스 중심) |
| 에러 대응 | 런타임에 조용히 실패 (찾기 힘듦) | 명확한 에러 메시지로 즉시 파악 |
| 유지보수 | Context 이름 변경 시 모든 파일 수정 | 훅 내부 로직만 수정하면 끝 |
