## 10강. Props Drilling의 문제점

### 도입

React의 데이터 흐름은 단방향(부모 -> 자식)이다. 트리가 깊어질수록 데이터를 최종 소비자에게 전달하기 위해 중간 컴포넌트들이 **사용하지도 않는 props를 계속 전달**해야 하는 상황이 발생한다. 이것이 **Props Drilling**이다.

### 코드 예제

```
App (데이터 소유자)
  └── Layout        ← user를 전달만 함
        └── Sidebar      ← user를 전달만 함
              └── UserPanel    ← user를 전달만 함
                    └── UserName     ← 실제로 user.name을 사용
```

```jsx
// App.jsx — 데이터 소유자
export default function App() {
  const user = { name: "철수", id: "user-001", isAdmin: true };
  return <Layout user={user} />;
}
```

```jsx
// Layout.jsx — 중간 전달자 (본인은 user 불필요)
export function Layout({ user }) {
  return <Sidebar user={user} />;
}
```

```jsx
// Sidebar.jsx — 중간 전달자
export function Sidebar({ user }) {
  return <UserPanel user={user} />;
}
```

```jsx
// UserPanel.jsx — 중간 전달자
export function UserPanel({ user }) {
  return <UserName user={user} />;
}
```

```jsx
// UserName.jsx — 최종 소비자 (실제로 데이터를 사용하는 유일한 곳)
export function UserName({ user }) {
  return <p>반갑습니다, <strong>{user.name}</strong>님!</p>;
}
```

### 코드 해설

- `Layout`, `Sidebar`, `UserPanel`은 `user` 데이터에 관심이 없지만, 자식을 위해 props 인터페이스에 포함시켜야 한다.
- 만약 `user` 객체의 속성명이 `name`에서 `userName`으로 바뀌면 **App 포함 5개 파일 모두 수정**해야 한다.
- 중간 컴포넌트들이 특정 props에 의존하게 되면서 **재사용성이 떨어지고 유지보수 비용이 증가**한다.

### 실무 비유

택배를 최종 수령인에게 전달하기 위해 위층의 모든 이웃을 거쳐야 하는 상황이다. 중간 배달원들은 택배 내용물에 관심도 없는데 계속 들고 있어야 한다.

### 핵심 포인트

- Props Drilling은 **트리 깊이에 비례해 유지보수 비용이 증가**한다.
- 중간 컴포넌트가 불필요한 데이터에 의존하게 되어 **관심사가 오염**된다.
- 데이터 구조가 변경되면 경로상의 **모든 파일을 수정**해야 하는 연쇄 비용이 발생한다.

## 11강. Context가 필요한 이유 — 전역 상태란 무엇인가

### 도입

하나의 컴포넌트가 아닌 **앱 전체가 알아야 하는 데이터**가 있다. 로그인한 사용자 정보, 테마 설정, 언어 선택 등이 대표적이다. 이를 **전역 상태(Global State)** 라 하며, Props Drilling 없이 전달하는 방법이 필요하다.

### 개념 설명

| 구분 | Props Drilling (기존) | Context API (새로운 방식) |
|------|----------------------|-------------------------|
| 전달 방식 | 유선 (줄줄이 전달) | 무선 (브로드캐스팅) |
| 중간 컴포넌트 | 데이터 파이프로 전락 | 순수 UI 역할에 집중 |
| 유지보수 | 데이터 변경 시 모든 경로 수정 | 소유자와 소비자만 수정 |
| 재사용성 | 특정 props 의존으로 낮음 | 독립적 구조로 높음 |

### 실무 비유

- **기존 방식**: 사장님이 신입 사원에게 전달할 로그인 토큰을 중간 관리자(부장, 과장, 대리)를 모두 불러서 한 명 한 명 손에 쥐여주는 방식. 중간 관리자들은 자기 일 하기도 바쁜데, 하루 종일 종이 한 장을 들고 있어야 한다.
- **Context API**: 사장님이 회사 게시판(Context)에 정보를 딱 붙여두는 것. 중간 관리자들은 손이 자유로워지고, 신입 사원은 필요할 때 게시판에서 직접 확인하면 된다.

### 핵심 포인트

- **전역 상태**는 특정 컴포넌트가 아닌 앱 전체가 공유하는 데이터이다.
- Context API는 중간 컴포넌트를 거치지 않고 **데이터를 필요한 곳에 직접 전달**한다.
- 중간 컴포넌트의 인터페이스(매개변수)를 오염시키지 않아 **관심사의 분리가 유지**된다.

## 12강. Context 내부 구조 해부

### 도입

Context API의 핵심 구성 요소인 `createContext`, `Provider`, `value`, `useContext`의 역할과 동작 원리를 파헤친다. 단순한 값 전달을 넘어 **데이터와 함수를 한데 묶은 서비스 시스템**을 구축한다.

### 코드 예제

```jsx
// contexts/CenterContext.jsx
import { createContext, useState } from "react";

// createContext의 인자는 '보험값' — Provider 없이 사용 시 이 값이 반환된다
export const NoticeContext = createContext("현재 등록된 공지사항이 없습니다.");
export const CenterContext = createContext();

// 전용 Provider 컴포넌트로 로직을 은닉한다
export function CenterProvider({ children }) {
  const [lostItems, setLostItems] = useState(["지갑", "에어팟"]);

  const reportLost = (item) => setLostItems((prev) => [...prev, item]);
  const claimItem = (item) => setLostItems((prev) => prev.filter((i) => i !== item));

  // 데이터와 기능을 하나의 객체로 묶어 전달한다
  const systemValue = { lostItems, reportLost, claimItem };

  return (
    <CenterContext.Provider value={systemValue}>
      {children}
    </CenterContext.Provider>
  );
}
```

```jsx
// components/NoticeBoard.jsx — Provider 없이 '보험값' 수신
import { useContext } from "react";
import { NoticeContext } from "../contexts/CenterContext";

export function NoticeBoard() {
  const notice = useContext(NoticeContext);
  return <div>공지: {notice}</div>;
  // Provider로 감싸지 않으면 → "현재 등록된 공지사항이 없습니다." 출력
}
```

```jsx
// components/LostAndFoundDesk.jsx — 전역 시스템에서 데이터와 함수를 수신
import { useContext, useState } from "react";
import { CenterContext } from "../contexts/CenterContext";

export function LostAndFoundDesk() {
  const { lostItems, reportLost, claimItem } = useContext(CenterContext);
  const [inputText, setInputText] = useState("");

  return (
    <div>
      <ul>
        {lostItems.map(item => (
          <li key={item}>
            {item} <button onClick={() => claimItem(item)}>찾아감</button>
          </li>
        ))}
      </ul>
      <input value={inputText} onChange={(e) => setInputText(e.target.value)} />
      <button onClick={() => { reportLost(inputText); setInputText(""); }}>
        신고하기
      </button>
    </div>
  );
}
```

```jsx
// App.jsx
import { CenterProvider } from "./contexts/CenterContext";
import { NoticeBoard } from "./components/NoticeBoard";
import { LostAndFoundDesk } from "./components/LostAndFoundDesk";

export default function App() {
  return (
    <div>
      {/* Provider 없이 사용 → 보험값 출력 */}
      <NoticeBoard />

      {/* Provider로 감싸서 하위에 기능 방송 */}
      <CenterProvider>
        <LostAndFoundDesk />
      </CenterProvider>
    </div>
  );
}
```

### 코드 해설

- **`createContext("기본값")`**: `useContext`를 썼는데 상위에 `Provider`가 없다면 앱이 죽지 않고 이 기본값을 반환한다. 에러 방지용 **보험**.
- **`value` 속성**: 실무에서는 값 하나만 보내지 않고, **데이터(`lostItems`)와 함수(`reportLost`, `claimItem`)를 세트로 묶어** 보내야 진정한 전역 관리 시스템이 된다.
- **`useContext(CenterContext)`**: 부모가 전달해주길 기다리지 않고, 컴포넌트가 **능동적으로 전역 시스템에 접속**해 필요한 것만 구조 분해 할당으로 꺼내온다.
- 자식 컴포넌트에서 버튼을 눌렀지만, 실제 데이터 수정은 `CenterProvider` 내부에서 일어난다.

### 실무 비유

- **Provider** = 아파트 관리사무소의 **방송 기지국**. `value`에 담긴 정보와 기능을 전파(Broadcasting)로 쏘아 올린다.
- **useContext** = 각 세대(컴포넌트)에 설치된 **무선 라디오**. 기지국 주소(Context 객체)만 맞추면, 중간 벽(중간 컴포넌트)이 아무리 두꺼워도 정보를 수신할 수 있다.

### 핵심 포인트

- `createContext(기본값)` → Provider가 없을 때의 안전망(보험)
- `Provider`의 `value`에 **데이터 + 수정 함수**를 함께 묶어 전달하는 것이 실무 패턴
- `useContext`로 props 없이 **능동적으로 전역 데이터에 접근**

## 13강. Context 재구조화

### 도입

10강에서 만들었던 Props Drilling 기반 코드를 Context 구조로 **리팩토링**한다. 중간 전달자들의 짐을 덜어내고, Context 관련 파일을 별도의 `contexts/` 폴더에 분리하는 실무 패턴을 적용한다.

### 코드 예제

**1단계: 통로 정의 (Context 생성)**

```js
// contexts/UserContext.js
import { createContext } from "react";
export const UserContext = createContext();
```

**2단계: 전용 공급자(Provider) 구축**

```jsx
// contexts/UserProvider.jsx
import { UserContext } from "./UserContext";

export function UserProvider({ children }) {
  const userName = "chulsoo";
  return (
    <UserContext.Provider value={userName}>
      {children}
    </UserContext.Provider>
  );
}
```

**3단계: 중간 전달자의 짐 덜어내기**

```jsx
// Before: function Layout({ userName }) { ... }
// After: props를 전혀 받지 않는다!

export function Layout() {
  return <Sidebar />;  // 전달 로직 삭제
}

export function Sidebar() {
  return <UserPanel />; // 전달 로직 삭제
}

export function UserPanel() {
  return <UserName />; // 전달 로직 삭제
}
```

**4단계: 최종 소비자에서 직접 수신**

```jsx
// components/UserName.jsx
import { useContext } from "react";
import { UserContext } from "../contexts/UserContext";

export function UserName() {
  const userName = useContext(UserContext);
  return <div>Welcome, {userName}!</div>;
}
```

**5단계: 최상단에서 시스템 가동**

```jsx
// App.jsx
import { UserProvider } from "./contexts/UserProvider";
import { Layout } from "./components/Layout";

export default function App() {
  return (
    <UserProvider>
      <Layout />
    </UserProvider>
  );
}
```

### 코드 해설

- **파라미터 삭제**: 중간 컴포넌트의 `{ userName }`이 사라졌다. 이들은 **데이터 배달원에서 은퇴**하고 **순수 UI 배치**라는 본연의 임무에만 집중한다. 이것이 **관심사의 분리(Separation of Concerns)**.
- **`{ children }` 패턴**: UserProvider 태그 사이에 들어오는 모든 컴포넌트를 그대로 렌더링하면서 데이터만 입혀주는 **합성(Composition)** 기법.
- **`value`가 바뀌면** 이 Context를 사용하는 모든 하위 소비자들이 **즉시 반응(리렌더링)** 한다.
- UserProvider를 빼먹으면 `useContext`는 `undefined`를 반환하고 에러가 발생한다. 반드시 상위에 Provider가 있어야 한다.

### 핵심 포인트

- Context 정의(`createContext`)와 공급(`Provider`)을 **별도 파일로 분리**하면 App이 깔끔해진다.
- 리팩토링 후 데이터 변경이 필요하면 **소유자(Provider)와 소비자(useContext)만 수정**하면 된다.
- `contexts/` 폴더에 Context 관련 파일을 모아 관리하는 것이 실무 표준 구조이다.

## 14강. 전역 상태 엔진의 사고방식

### 도입

Context(통로)와 Reducer(두뇌)를 결합해 앱 전체의 상태를 제어하는 **전역 상태 엔진**을 만든다. Context는 데이터가 흐르는 파이프라인이고, Reducer는 어떤 요청에 대해 상태를 어떻게 바꿀지 결정하는 운영 매뉴얼이다.

### 코드 예제

```js
// contexts/CounterContext.js — 통로 정의
import { createContext } from "react";
export const CounterContext = createContext();
```

```js
// contexts/counterReducer.js — 두뇌 설계
export function counterReducer(state, action) {
  switch (action.type) {
    case "increment":
      return state + 1;
    case "decrement":
      return state - 1;
    default:
      return state;
  }
}
```

```jsx
// contexts/CounterProvider.jsx — 엔진 결합
import { useReducer } from "react";
import { CounterContext } from "./CounterContext";
import { counterReducer } from "./counterReducer";

export function CounterProvider({ children }) {
  const [state, dispatch] = useReducer(counterReducer, 0);

  return (
    <CounterContext.Provider value={{ state, dispatch }}>
      {children}
    </CounterContext.Provider>
  );
}
```

```jsx
// components/CounterScreen.jsx — 전역 상태 소비
import { useContext } from "react";
import { CounterContext } from "../contexts/CounterContext";

export default function CounterScreen() {
  const { state, dispatch } = useContext(CounterContext);

  return (
    <div>
      <h1>전역 카운트: {state}</h1>
      <button onClick={() => dispatch({ type: "increment" })}>증가 (+)</button>
      <button onClick={() => dispatch({ type: "decrement" })}>감소 (-)</button>
    </div>
  );
}
```

### 코드 해설

- **`counterReducer`**: 컴포넌트 안에서 `count + 1`을 하지 않는다. 모든 계산을 여기서 몰아서 처리하므로, 버그가 생기면 **이 파일 하나만 고치면 된다**.
- **`value={{ state, dispatch }}`**: 객체로 묶어 보내면 하위에서 `{ state, dispatch }`로 구조 분해 할당하여 필요한 것만 편하게 꺼내 쓸 수 있다.
- **데이터의 여정**: 버튼 클릭 -> `dispatch`가 요청서를 Provider로 전달 -> `counterReducer`가 계산 -> 새 값이 Context 통로를 타고 소비자에게 전달 -> 리렌더링.

### 실무 비유

- **Context** = 데이터가 흐르는 **통로(파이프라인)**
- **Reducer** = 어떤 요청을 받으면 어떻게 바꿀지 결정하는 **두뇌(운영 매뉴얼)**
- **Provider** = 통로와 두뇌를 결합한 **엔진**

### 핵심 포인트

- `Context + useReducer` 조합이 React 내장 전역 상태 관리의 **표준 패턴**이다.
- `dispatch`는 "어떻게 바꿀지" 모르고, 단지 **요청서(action)만 전달**한다.
- 상태 변경 로직이 Reducer에 집중되어 **예측 가능하고 테스트하기 쉽다**.

## 15강. 실전 인증 시스템 설계

### 도입

실무에서 가장 흔한 전역 상태인 **인증(Authentication)** 시스템을 설계한다. 사용자 정보(`user`), 로딩 여부(`isLoading`), 에러 메시지(`error`)를 한 세트로 관리하는 리듀서와 프로바이더를 구축한다.

### 코드 예제

```jsx
// contexts/AuthContext.jsx
import { createContext, useReducer } from "react";

export const AuthContext = createContext();

const authReducer = (state, action) => {
  switch (action.type) {
    case "LOGIN_START":
      return { user: null, isLoading: true, error: null };
    case "LOGIN_SUCCESS":
      return { user: action.payload, isLoading: false, error: null };
    case "LOGIN_FAILURE":
      return { user: null, isLoading: false, error: action.payload };
    case "LOGOUT":
      return { user: null, isLoading: false, error: null };
    default:
      return state;
  }
};

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, {
    user: null,
    isLoading: false,
    error: null,
  });

  return (
    <AuthContext.Provider value={{ ...state, dispatch }}>
      {children}
    </AuthContext.Provider>
  );
}
```

```jsx
// components/LoginButton.jsx
import { useContext } from "react";
import { AuthContext } from "../contexts/AuthContext";

export default function LoginButton() {
  const { isLoading, dispatch, user } = useContext(AuthContext);

  const handleLogin = async () => {
    dispatch({ type: "LOGIN_START" });

    try {
      // 실제로는 fetch/axios로 서버 통신
      await new Promise((resolve) => setTimeout(resolve, 2000));
      const mockUser = { id: 1, name: "정용수", email: "yong@example.com" };
      dispatch({ type: "LOGIN_SUCCESS", payload: mockUser });
    } catch (err) {
      dispatch({ type: "LOGIN_FAILURE", payload: "아이디나 비밀번호가 틀렸습니다." });
    }
  };

  if (user) {
    return <button onClick={() => dispatch({ type: "LOGOUT" })}>로그아웃</button>;
  }

  return (
    <button onClick={handleLogin} disabled={isLoading}>
      {isLoading ? "인증 확인 중..." : "간편 로그인하기"}
    </button>
  );
}
```

### 코드 해설

- **`LOGIN_START`**: 버튼을 누르자마자 "로딩 중"을 알려 **중복 클릭을 방지**한다.
- **`{ ...state, dispatch }`**: 스프레드 연산자로 `user`, `isLoading`, `error`를 개별적으로 꺼내기 편하게 풀어서 전달한다.
- **`disabled={isLoading}`**: 전역 상태 `isLoading` 덕분에 UX 보호 로직이 간단해진다.
- **버튼 컴포넌트의 역할**: "어떻게 상태를 바꿀지" 모른다. 단지 "로그인 시작됐어!"라고 **보고만** 할 뿐이다.

### 인증 흐름 (Data Journey)

1. 사용자가 버튼 클릭 -> `LOGIN_START` 신호 발생
2. Reducer가 `isLoading`을 `true`로 변경 -> 앱 전체에 "작업 중" 방송
3. 통신 완료 -> `LOGIN_SUCCESS` 신호 + 유저 데이터 전송
4. Reducer가 유저 정보 보관, `isLoading` 종료 -> 구독 중인 컴포넌트 자동 업데이트

### 핵심 포인트

- 인증 상태는 `{ user, isLoading, error }`의 **3요소 세트**로 관리하는 것이 실무 표준이다.
- action type을 `LOGIN_START` / `LOGIN_SUCCESS` / `LOGIN_FAILURE` / `LOGOUT`으로 분리하면 **각 단계별 UI 제어가 명확**해진다.
- 비동기 처리(서버 통신)와 전역 상태를 결합하는 가장 기본적인 패턴이다.

## 16강. Context 분리와 최적화

### 도입

하나의 Context에 `{ user, isLoading, dispatch }`를 모두 담으면, `isLoading`만 바뀌어도 `dispatch`만 필요한 컴포넌트까지 리렌더링된다. **State Context와 Dispatch Context를 분리**하여 불필요한 리렌더링을 차단하는 고급 전략을 익힌다.

### 코드 예제

```jsx
// contexts/AuthContext.jsx — 채널 분리형 Provider
import { createContext, useReducer, useMemo } from "react";

// 두 개의 독립적인 채널 개설
export const AuthStateContext = createContext();
export const AuthDispatchContext = createContext();

const authReducer = (state, action) => {
  switch (action.type) {
    case "LOGIN_START":
      return { ...state, isLoading: true };
    case "LOGIN_SUCCESS":
      return { user: action.payload, isLoading: false, error: null };
    case "LOGOUT":
      return { user: null, isLoading: false, error: null };
    default:
      return state;
  }
};

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, {
    user: null,
    isLoading: false,
    error: null,
  });

  // state의 세부 값이 바뀔 때만 새로운 참조 생성
  const memoizedState = useMemo(() => state, [state]);

  return (
    <AuthStateContext.Provider value={memoizedState}>
      <AuthDispatchContext.Provider value={dispatch}>
        {children}
      </AuthDispatchContext.Provider>
    </AuthStateContext.Provider>
  );
}
```

```jsx
// components/LogoutButton.jsx — 명령 채널만 구독
import { useContext } from "react";
import { AuthDispatchContext } from "../contexts/AuthContext";

export default function LogoutButton() {
  // 상태(state)는 무시하고 오직 명령(dispatch) 채널만 수신
  const dispatch = useContext(AuthDispatchContext);

  console.log("LogoutButton이 렌더링되었습니다.");

  return (
    <button onClick={() => dispatch({ type: "LOGOUT" })}>로그아웃</button>
  );
}
```

### 코드 해설

- **`createContext()`를 두 번 호출**: 상태용과 명령용을 **아예 다른 문(Door)** 으로 분리한다.
- **`useMemo`**: 객체는 내용이 같아도 참조 주소가 다르면 "바뀌었다"고 판단한다. `useMemo`는 상태가 정말 변했을 때만 새 주소를 부여해 **불필요한 리렌더링을 방지**한다.
- **LogoutButton**: `AuthDispatchContext`만 구독하므로 `isLoading`이 바뀌어도 **리렌더링되지 않는다**.

### 최적화 전후 비교

| 비교 항목 | 단일 Context | Context 분리 |
|-----------|-------------|-------------|
| 구조 | `{ user, isLoading, dispatch }` 한 묶음 | State와 Dispatch를 개별 통로로 전송 |
| 영향 범위 | `isLoading`만 바뀌어도 LogoutButton 리렌더링 | `isLoading`이 바뀌어도 LogoutButton 평온 |
| 성능 | 컴포넌트 많아질수록 렌더링 부하 증가 | 필요한 컴포넌트만 정밀하게 렌더링 |
| 유지보수 | 데이터와 기능이 엉켜있음 | 데이터 흐름과 행위 흐름이 분리 |

### 핵심 포인트

- **읽기 전용 데이터(State)**와 **수정용 함수(Dispatch)**를 분리하는 것이 대규모 프로젝트에서 거의 **필수적인 패턴**이다.
- `dispatch` 함수는 `useReducer`가 반환하는 값으로 **참조가 불변**이므로, 별도 Context로 분리하면 해당 채널은 절대 리렌더링을 유발하지 않는다.
- `useMemo`로 state 객체의 불필요한 참조 재생성을 방지한다.

## 17강. 다중 전역 상태 엔진 구축

### 도입

실제 앱에는 인증, 업무 관리, 알림 등 **여러 도메인의 전역 상태**가 공존한다. 각 도메인별로 독립된 엔진을 설계하고, Provider 중첩(Nesting)으로 조립하는 방법을 배운다.

### 코드 예제

**Todo 엔진 설계**

```js
// contexts/todoReducer.js
export function todoReducer(state, action) {
  switch (action.type) {
    case "ADD_TODO":
      return { ...state, todos: [...state.todos, action.payload] };
    default:
      return state;
  }
}
```

```jsx
// contexts/TodoProvider.jsx
import { useReducer, useMemo, createContext } from "react";
import { todoReducer } from "./todoReducer";

export const TodoStateContext = createContext();
export const TodoDispatchContext = createContext();

export function TodoProvider({ children }) {
  const [state, dispatch] = useReducer(todoReducer, { todos: [] });
  const memoizedState = useMemo(() => state, [state]);

  return (
    <TodoStateContext.Provider value={memoizedState}>
      <TodoDispatchContext.Provider value={dispatch}>
        {children}
      </TodoDispatchContext.Provider>
    </TodoStateContext.Provider>
  );

```

```jsx
export default function App() {
```

**여러 엔진의 정보 융합**

```jsx
// components/TodoInput.jsx
import { useContext, useState } from "react";
import { AuthStateContext } from "../contexts/AuthContext";
import { TodoDispatchContext } from "../contexts/TodoProvider";

export default function TodoInput() {
  const [text, setText] = useState("");
  const { user } = useContext(AuthStateContext);       // 인증 엔진
  const todoDispatch = useContext(TodoDispatchContext); // 업무 엔진

  const onAdd = () => {
    if (!user) {
      alert("로그인이 필요한 서비스입니다.");
      return;
    }
    todoDispatch({ type: "ADD_TODO", payload: text });
    setText("");
  };

  return (
    <div>
      <input value={text} onChange={(e) => setText(e.target.value)} />
      <button onClick={onAdd}>업무 추가</button>
      {!user && <p style={{ color: 'red' }}>먼저 로그인을 해주세요.</p>}
    </div>
  );
}
```

### 코드 해설

- **독립적인 엔진**: 업무(Todo)를 추가해도 `AuthStateContext` 채널은 조용하다. 헤더나 유저 프로필 컴포넌트는 리렌더링되지 않는다.
- **의존성 순서**: `AuthProvider`가 바깥에 있으면, 안쪽의 `TodoProvider`와 그 하위 자식들은 언제든 Auth 정보를 참조할 수 있다. **반대는 불가능**하다.
- **다중 `useContext`**: 하나의 컴포넌트가 여러 통로에 빨대를 꽂아, Auth에서 데이터를, Todo에서 행위를 가져와 **유기적으로 결합**한다.
- **확장성**: 알림(Notification) 엔진이 필요하면 `TodoProvider` 안쪽에 `NotificationProvider`를 하나 더 끼워 넣으면 된다.

### 핵심 포인트

- 도메인별로 **독립된 Context + Reducer 엔진**을 구축한다.
- Provider의 **중첩 순서가 의존성 방향**을 결정한다 (상위가 하위를 참조 가능).
- 16강의 **State/Dispatch 분리 패턴을 그대로 재활용**하는 것이 패턴화된 설계이다.

## 18강. 커스텀 훅 추상화 전략

### 도입

컴포넌트에서 `useContext(AuthStateContext)`를 직접 호출하면 Context 객체명에 의존하게 되고, Provider 밖에서 호출하면 조용히 `undefined`를 반환해 원인 파악이 어렵다. **커스텀 훅으로 감싸서** 안전장치와 추상화를 동시에 제공한다.

### 코드 예제

```jsx
// contexts/AuthContext.jsx (하단에 커스텀 훅 추가)
import { createContext, useReducer, useMemo, useContext } from "react";

const AuthStateContext = createContext();
const AuthDispatchContext = createContext();

// ... authReducer, AuthProvider 정의 ...

// 커스텀 훅: 상태 전용 수신기
export function useAuthState() {
  const context = useContext(AuthStateContext);

  // 안전장치: Provider 밖에서 호출하면 명시적 에러
  if (context === undefined) {
    throw new Error("useAuthState는 반드시 AuthProvider 안에서 사용해야 합니다!");
  }
  return context;
}

// 커스텀 훅: 명령 전용 수신기
export function useAuthDispatch() {
  const context = useContext(AuthDispatchContext);

  if (context === undefined) {
    throw new Error("useAuthDispatch는 반드시 AuthProvider 안에서 사용해야 합니다!");
  }
  return context;
}
```

```jsx
// components/UserProfile.jsx — 추상화된 훅으로 코드 다이어트
import { useAuthState, useAuthDispatch } from "../contexts/AuthContext";

export default function UserProfile() {
  const { user, isLoading } = useAuthState();
  const dispatch = useAuthDispatch();

  if (isLoading) return <p>로딩 중...</p>;
  if (!user) return <p>로그인이 필요합니다.</p>;

  return (
    <div>
      <h2>{user.name}님의 프로필</h2>
      <button onClick={() => dispatch({ type: "LOGOUT" })}>로그아웃</button>
    </div>
  );
}
```

### 코드 해설

- **`if (context === undefined)`**: Provider 밖에서 훅을 사용하면 원인 모를 버그 대신 **정확한 에러 메시지**가 출력된다. 디버깅 시간을 획기적으로 줄이는 안전장치.
- **`useAuthState()`**: 컴포넌트는 "데이터가 Context에서 오는지, Redux에서 오는지" 알 필요가 없다. 그저 "유저 상태가 필요해"라는 **의도만 표현**한다.
- 나중에 내부 구현을 바꿔도(예: Context -> Zustand) 커스텀 훅의 **인터페이스만 유지하면** 소비자 코드를 수정할 필요가 없다.

### 추상화 전후 비교

| 비교 항목 | 직접 호출 | 커스텀 훅 |
|-----------|----------|----------|
| import 개수 | `useContext`, Context 객체 등 다수 | 전용 훅 하나 |
| 코드 가독성 | `useContext(AuthContext)` (기술 중심) | `useAuthState()` (비즈니스 중심) |
| 에러 대응 | 런타임에 조용히 실패 | 명확한 에러 메시지로 즉시 파악 |
| 유지보수 | Context 이름 변경 시 모든 파일 수정 | 훅 내부 로직만 수정 |

### 핵심 포인트

- 커스텀 훅은 **안전장치(Provider 체크)** + **추상화(구현 은닉)** + **코드 다이어트(import 간소화)** 를 동시에 제공한다.
- `useAuthState`, `useAuthDispatch` 패턴은 **Context를 사용하는 모든 도메인에 동일하게 적용**할 수 있다.
- 컴포넌트가 **기술적 세부사항이 아닌 비즈니스 의도**를 표현하게 되어 가독성이 극대화된다.

## 19강. 전역 상태 관리의 완성

### 도입

10~18강에서 배운 모든 개념을 하나의 통합 프로젝트로 완성한다. Auth 엔진과 Todo 엔진이 공존하고, 커스텀 훅으로 추상화되며, 채널 분리로 최적화된 **실전 수준의 전역 상태 관리 아키텍처**를 구현한다.

### 코드 예제

**완성된 Auth 엔진**

```jsx
// contexts/AuthContext.jsx
import { createContext, useReducer, useContext, useMemo } from "react";

const AuthStateContext = createContext();
const AuthDispatchContext = createContext();

const authReducer = (state, action) => {
  if (action.type === "LOGIN") return { user: { name: "정용수" }, isLoading: false };
  if (action.type === "LOGOUT") return { user: null, isLoading: false };
  return state;
};

export function AuthProvider({ children }) {
  const [state, dispatch] = useReducer(authReducer, { user: null, isLoading: false });
  const memoizedState = useMemo(() => state, [state]);
  return (
    <AuthStateContext.Provider value={memoizedState}>
      <AuthDispatchContext.Provider value={dispatch}>
        {children}
      </AuthDispatchContext.Provider>
    </AuthStateContext.Provider>
  );
}

export const useAuthState = () => useContext(AuthStateContext);
export const useAuthDispatch = () => useContext(AuthDispatchContext);
```

**완성된 Todo 엔진**

```jsx
// contexts/TodoContext.jsx
import { createContext, useReducer, useContext, useMemo } from "react";

const TodoStateContext = createContext();
const TodoDispatchContext = createContext();

const todoReducer = (state, action) => {
  switch (action.type) {
    case "ADD_TODO":
      return {
        ...state,
        todos: [...state.todos, { id: Date.now(), text: action.payload, completed: false }],
      };
    case "TOGGLE_TODO":
      return {
        ...state,
        todos: state.todos.map(t =>
          t.id === action.payload ? { ...t, completed: !t.completed } : t
        ),
      };
    default:
      return state;
  }
};

export function TodoProvider({ children }) {
  const [state, dispatch] = useReducer(todoReducer, { todos: [] });
  const memoizedState = useMemo(() => state, [state]);
  return (
    <TodoStateContext.Provider value={memoizedState}>
      <TodoDispatchContext.Provider value={dispatch}>
        {children}
      </TodoDispatchContext.Provider>
    </TodoStateContext.Provider>
  );
}

export const useTodoState = () => useContext(TodoStateContext);
export const useTodoDispatch = () => useContext(TodoDispatchContext);
```

```jsx
        ))}
      </ul>
      <button onClick={() => dispatch({ type: "ADD_TODO", payload: "새로운 업무" })}>

```

### 코드 해설

- **`useAuthState()`, `useTodoState()`**: `useContext(UserContext)` 방식보다 직관적이다. 코드를 읽는 사람이 "이 컴포넌트는 인증 정보와 업무 정보를 쓰는구나"라고 **단번에 파악**할 수 있다.
- **가드 로직**: `if (!user)` 구문으로 서로 다른 엔진의 데이터를 조합하여 **UX를 제어**한다.
- **채널 분리 효과**: 업무 목록이 업데이트되어도, `AuthDispatchContext`를 사용 중인 로그아웃 버튼이나 Header는 **절대 리렌더링되지 않는다**.

### 통합 시스템의 데이터 순환

1. **의도**: 사용자가 '할 일 추가' 버튼을 누른다
2. **보안 체크**: 컴포넌트에서 인증 엔진의 `user` 상태를 확인한다
3. **명령**: `TodoDispatchContext`를 통해 `ADD_TODO` 액션을 발송한다
4. **최적화**: 업무 목록이 업데이트되어도 Auth 채널 구독자는 리렌더링되지 않는다
5. **반영**: 업무 목록을 구독 중인 `TodoDashboard`만 화면을 다시 그린다

### 핵심 포인트

- App은 `useState`/`useEffect` 없이 **엔진 배치에만 집중**하는 순수한 관제탑이 된다.
- 모든 엔진이 **동일한 설계 패턴**(Context 분리 + Reducer + 커스텀 훅)을 따르므로 확장이 쉽다.
- 새 도메인(알림, 장바구니 등)이 추가되어도 **Provider 하나만 끼워 넣으면 된다**.

## 면접 핵심 Q&A

### Q. Props Drilling이란 무엇이고, 어떤 문제가 있는가?

> **A.** 상위 컴포넌트의 데이터를 최종 소비자에게 전달하기 위해 중간 컴포넌트들이 사용하지 않는 props를 계속 전달해야 하는 현상이다. 중간 컴포넌트의 인터페이스가 오염되어 재사용성이 떨어지고, 데이터 구조 변경 시 경로상의 모든 파일을 수정해야 하는 유지보수 비용이 발생한다.

### Q. Context API의 핵심 구성 요소와 각각의 역할은?

> **A.**
> - **`createContext(기본값)`**: 데이터가 흐를 통로를 생성한다. 인자는 Provider 없이 사용될 때의 안전망(기본값)이다.
> - **`Provider`**: `value` 속성에 데이터를 담아 하위 트리에 브로드캐스팅하는 기지국이다.
> - **`useContext(Context)`**: 컴포넌트가 능동적으로 전역 통로에 접속해 데이터를 수신하는 훅이다.

### Q. Context + useReducer를 결합하는 이유는?

> **A.** Context는 데이터가 흐르는 **통로**이고, useReducer는 상태 변경 로직을 집중 관리하는 **두뇌**이다. 결합하면 전역 범위에서 예측 가능한 상태 관리가 가능해진다. 컴포넌트는 dispatch로 요청서만 보내고, 실제 상태 변경은 reducer에서 일어나므로 **관심사가 분리**되고 **테스트가 쉬워진다**.

### Q. Context의 State와 Dispatch를 분리하는 이유는?

> **A.** 단일 Context에 `{ state, dispatch }`를 함께 담으면, state가 변할 때마다 dispatch만 구독하는 컴포넌트도 리렌더링된다. `dispatch`는 `useReducer`가 반환하는 안정적 참조(불변)이므로 별도 Context로 분리하면 **명령만 필요한 컴포넌트는 state 변화에 영향받지 않는다**. 이는 대규모 앱에서 불필요한 렌더링을 줄이는 핵심 최적화 전략이다.

### Q. 커스텀 훅으로 Context를 감싸는 이점은?

> **A.**
> 1. **안전장치**: Provider 밖에서 호출 시 명시적 에러를 던져 디버깅이 쉬워진다.
> 2. **추상화**: 소비자 컴포넌트가 Context 객체명이나 내부 구현을 알 필요가 없다.
> 3. **유지보수**: 나중에 Context를 다른 상태 관리 도구로 교체해도 훅 인터페이스만 유지하면 소비자 코드를 수정할 필요가 없다.

### Q. 다중 Provider를 중첩할 때 순서가 중요한 이유는?

> **A.** React의 데이터 흐름은 위에서 아래로 흐른다. 바깥쪽 Provider의 데이터는 안쪽에서 참조할 수 있지만 **반대는 불가능**하다. 예를 들어 Todo 로직에서 Auth 유저 정보를 참조해야 한다면, AuthProvider가 TodoProvider의 **상위(바깥)** 에 위치해야 한다.

### Q. createContext의 기본값(Default Value)은 언제 사용되는가?

> **A.** 컴포넌트가 `useContext`로 데이터를 요청했지만 **상위 트리에 해당 Provider가 없을 때** 기본값이 반환된다. 이는 앱이 죽지 않도록 하는 일종의 보험이다. 실무에서는 커스텀 훅 내부에서 `undefined` 체크 후 에러를 던지는 방식으로 보완한다.

## 용어 글로서리

| 용어 | 설명 |
|------|------|
| **Props Drilling** | 상위 데이터를 하위에 전달하기 위해 중간 컴포넌트들이 불필요하게 props를 중계하는 현상 |
| **전역 상태 (Global State)** | 특정 컴포넌트가 아닌 앱 전체가 공유해야 하는 데이터 (인증, 테마, 언어 등) |
| **Context** | React에서 컴포넌트 트리 전체에 데이터를 전달하는 메커니즘 |
| **createContext** | Context 객체를 생성하는 함수. 인자는 Provider 없을 때의 기본값(보험) |
| **Provider** | Context의 `value`를 하위 트리에 방송(Broadcasting)하는 컴포넌트 |
| **useContext** | 함수 컴포넌트에서 Context의 value를 구독하는 훅 |
| **Reducer** | 현재 상태(state)와 요청서(action)를 받아 새로운 상태를 반환하는 순수 함수 |
| **dispatch** | Reducer에게 action(요청서)을 전달하는 함수 |
| **action** | `{ type: "ACTION_NAME", payload: data }` 형태의 상태 변경 요청 객체 |
| **채널 분리** | StateContext와 DispatchContext를 별도로 생성하여 리렌더링을 최적화하는 패턴 |
| **useMemo** | 값의 메모이제이션. 의존성이 변할 때만 재계산하여 불필요한 참조 재생성 방지 |
| **커스텀 훅** | `use` 접두사를 가진 재사용 가능한 함수. 내부 구현을 추상화하고 안전장치를 제공 |
| **Provider 중첩** | 여러 Provider를 계단식으로 쌓아 다중 전역 상태를 동시에 공급하는 패턴 |
| **Composition** | `{ children }` 패턴으로 컴포넌트를 합성하는 React의 핵심 기법 |
