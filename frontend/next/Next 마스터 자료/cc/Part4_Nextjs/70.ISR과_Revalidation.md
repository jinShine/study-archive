## 1. ISR이란? — 두 마리 토끼! 🔴

> 💡 **비유: 편의점 도시락**
> Static = 공장에서 한 번 만들어 영원히 판매 (싸고 빠르지만 옛날 도시락)
> Dynamic = 주문마다 즉석 조리 (항상 신선하지만 느리고 비쌈)
> ISR = 유통기한이 있는 도시락! 유통기한 안에는 캐시, 지나면 새로 만듦!

```
ISR (Incremental Static Regeneration):
  "정적의 속도 + 동적의 신선도"

  평소: 캐시에서 즉시 반환! (0.001초, 서버 부하 0)
  유통기한 만료 후: 백그라운드에서 새 데이터로 갱신!
  사용자는 지연을 전혀 느끼지 못함!
```

## 2. revalidate 옵션 🔴

### fetch에 직접 설정

```tsx
export default async function ProductsPage() {
  const res = await fetch("https://api.example.com/products", {
    next: { revalidate: 60 },  // ⭐ 60초마다 재검증!
  });
  const products = await res.json();
  return <ProductList products={products} />;
}
```

### Segment Config로 설정 (fetch 없을 때!)

```tsx
// Prisma, ORM 등 fetch가 아닌 데이터 조회 시:
export const revalidate = 60;  // 이 페이지 전체를 60초마다 재검증!

export default async function ProductsPage() {
  const products = await db.product.findMany();  // fetch가 아님!
  return <ProductList products={products} />;
}
```

> ⚠️ **단위는 초(seconds)!** `revalidate: 3000`이면 3000초 = 50분이다!
> `setTimeout`의 밀리초(ms)와 헷갈리지 마라!

## 3. SWR 패턴 — ISR의 동작 원리 🔴

> SWR = Stale-While-Revalidate
> "낡은(Stale) 캐시를 먼저 보여주는 동안(While), 뒤에서 새 데이터로 갱신(Revalidate)"

### 시간 타임라인 (revalidate: 60)

```
T=0초:   최초 접속 → 서버가 API 호출 → 데이터를 캐시에 저장 (타이머 60초 시작!)
         사용자 A: "상품 목록" 표시 (약간의 서버 렌더링 시간)

T=30초:  사용자 B 접속 → 캐시에서 즉시 반환! (API 호출 안 함!)
         → 0.001초! 서버 부하 0!

T=50초:  사용자 C 접속 → 캐시에서 즉시 반환!
         → 아직 60초 안이니까 신선한 데이터!

T=65초:  ⭐ 유통기한(60초) 만료 후 사용자 D 접속!
         → ① 먼저: 낡은 캐시(T=0초의 데이터)를 즉시 반환! (사용자는 기다리지 않음!)
         → ② 동시에: 백그라운드에서 서버가 API를 새로 호출!
         → ③ 완료: 새 데이터로 캐시를 교체! 타이머 리셋!

T=66초:  사용자 E 접속 → 새로 갱신된 최신 캐시를 즉시 반환!

핵심: 사용자 D는 낡은 데이터를 봤지만, 대기 시간은 0초!
      사용자 E부터는 최신 데이터를 0초만에!
      → 누구도 로딩 화면을 보지 않는다!
```

```
┌──────────────────────────────────────────────────────────┐
│  0초        30초       60초       65초       125초       │
│  ├──── 신선(fresh) ─────┤── stale ──┤── 신선 ───┤── stale│
│  │                      │            │           │       │
│  ▼                      ▼            ▼           ▼       │
│  API 호출              캐시 반환   캐시+갱신    캐시 반환 │
│  (1번)                 (0번!)     (1번)        (0번!)    │
│                                                          │
│  revalidate: 60초      유통기한    SWR!        다시 60초 │
└──────────────────────────────────────────────────────────┘
```

## 4. ISR vs Dynamic vs Static 최종 비교 🔴

```
Static (force-cache):
  ┌─ 빌드 ─┐
  │ 1번    │ → 영원히 캐시 → 데이터 영구 고정 ❌
  └────────┘

Dynamic (no-store):
  매 요청마다 API 호출 → 항상 최신 ✅ but 서버 부하 ↑↑

ISR (revalidate: 60):
  ┌─ 신선 ──────────┐ ┌─ stale ─┐ ┌─ 신선 ──────────┐
  │ 캐시 반환 (빠름!) │ │ SWR!   │ │ 캐시 반환 (빠름!) │
  └─────────── 60초 ─┘ └────────┘ └─────────── 60초 ─┘
  → 속도 + 적당한 최신성 = 가장 많이 쓰는 전략! ⭐
```

## 5. 실습

```
과제: ISR 타임라인 직접 체험

1. 현재 시간 API를 revalidate: 10으로 설정
2. npm run build → npm run start (프로덕션 필수!)
3. F5 연타 → 10초 동안 시간이 멈춰있는지 확인!
4. 10초 후 F5 → 시간이 안 바뀜! (아직 낡은 캐시)
5. 한 번 더 F5 → 시간이 "훅" 바뀜! (SWR 갱신 완료!)
6. revalidate를 3초로 줄여서 더 빠르게 체험!
7. Segment Config 방식: export const revalidate = 10; 도 테스트
```

## 면접 대비

### 🔴 필수 질문

**Q1. ISR(Incremental Static Regeneration)이란?**

> 정적 페이지의 속도 이점을 유지하면서 일정 시간마다 데이터를 갱신하는 렌더링 전략이다. `next: { revalidate: 60 }` 옵션으로 60초 동안은 캐시를 반환하고, 60초가 지난 후 요청이 오면 일단 낡은 캐시를 즉시 반환한 뒤 백그라운드에서 새 데이터를 가져와 캐시를 교체한다(SWR 패턴). 사용자는 대기 시간 없이 항상 즉시 화면을 보며, 쇼핑몰 상품 목록이나 뉴스처럼 트래픽이 많으면서 적당한 최신성이 필요한 페이지에 최적이다.

**Q2. SWR(Stale-While-Revalidate) 패턴을 설명?**

> 캐시 유효기간이 지난 후 요청이 오면, 사용자에게 낡은(Stale) 캐시 데이터를 즉시 반환하여 대기 시간을 0으로 만들고, 동시에 백그라운드에서 서버를 호출하여 새 데이터를 가져와 캐시를 갱신(Revalidate)한다. 다음 요청자부터는 갱신된 최신 데이터를 받는다. HTTP의 `stale-while-revalidate` 헤더와 같은 원리를 프레임워크 레벨에서 구현한 것이다.
