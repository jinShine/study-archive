## 1. 렌더링 방식의 진화 — 3세대 🔴

> 💡 **비유: 가구 배달의 진화**

### 1세대: CSR (Client-Side Rendering)

> 💡 **비유: 이케아 가구 박스**
> 서버가 텅 빈 HTML + 거대한 JS 설명서를 던져줌.
> 브라우저가 JS를 다운받아 실행하고, API로 데이터를 가져와서, 그제서야 화면을 그림.

```
사용자 접속 → 빈 HTML 수신 → JS 다운로드(2MB) → JS 실행 → API 호출 → 데이터 수신 → 화면 렌더링
             └── 하얀 화면 ──────────────────────────────────────────┘
             (사용자는 이 시간 동안 아무것도 못 봄! 3~5초!)
```

```tsx
// CSR 패턴 (순수 React/SPA)
"use client";
import { useState, useEffect } from "react";

export default function ProductPage() {
  const [data, setData] = useState(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    fetch("/api/products")
      .then(res => res.json())
      .then(data => { setData(data); setIsLoading(false); });
  }, []);

  if (isLoading) return <div>빙글빙글🌀</div>;  // 사용자가 처음 보는 것: 스피너!
  return <div>{data.name}</div>;
}
```

```
장점: 한 번 로드되면 페이지 전환이 빠름 (SPA)
단점: 초기 로딩 느림 (하얀 화면), SEO 불가 (빈 HTML), 번들 크기 큼
실무 용도: 지도 앱, 에디터 등 고도의 인터랙션 (로그인 후 대시보드)
```

### 2세대: SSR (Server-Side Rendering)

> 💡 **비유: 완성된 가구 + 랩으로 감싸진 상태**
> 서버가 HTML을 완성해서 보내줌. 사용자는 즉시 화면을 볼 수 있음!
> 하지만 버튼은 아직 안 눌림 (JS가 도착해야 하이드레이션 후 동작)

```
사용자 접속 → 서버가 HTML 완성 → 완성된 HTML 수신 → 화면 즉시 표시!
                                                 → JS 다운로드 → 하이드레이션 → 인터랙션 가능
```

```
장점: 초기 로딩 빠름 (HTML 즉시 표시), SEO 가능 (완성된 HTML)
단점: 서버 부하 (매 요청마다 렌더링), 거대한 JS 번들은 여전히 전송
실무 용도: 검색 결과, 뉴스, 상품 상세 (SEO 필요 + 최신 데이터)
```

### 3세대: RSC (React Server Components) — Next.js의 혁명! 🔴

> 💡 **비유: 붙박이 가구**
> 서버에서 완벽하게 완성하고, 브라우저에 JS를 보내지도 않음!
> 브라우저는 결과물(HTML)만 받아서 즉시 표시.

```tsx
// RSC 패턴 (Next.js App Router 기본!)
// 'use client'가 없으면 → 서버 컴포넌트!
import db from "@/lib/db";

export default async function ProductPage() {
  // 서버에서 직접 DB 조회! 브라우저로 이 코드가 전송되지 않음!
  const products = await db.product.findMany();

  return (
    <div>
      {products.map(p => <div key={p.id}>{p.name}</div>)}
    </div>
  );
}
// useState 없음! useEffect 없음! fetch 없음!
// → 브라우저 JS 번들에 이 코드가 포함되지 않음 = Zero Bundle Size!
```

```
장점:
  ✅ Zero Bundle Size: 서버 코드가 브라우저로 안 감 → 앱이 가벼움!
  ✅ 직접 DB 접근: 서버에서 바로 데이터베이스 조회 (API 불필요!)
  ✅ 보안: DB 비밀번호, API 키가 브라우저에 노출 안 됨
  ✅ SEO: 완성된 HTML
  ✅ 스트리밍: 준비된 부분부터 순차적으로 전송

실무 용도: 대부분의 페이지! (Next.js의 기본값)
```

### RSC의 4단계 동작 시뮬레이션 🟡

> 서버에서 브라우저까지 데이터가 어떻게 전달되는가?

```
Step 1: 요청 (Request)
  └─ 사용자가 /products에 접속

Step 2: 서버 실행 (Server Execution)
  └─ Next.js 서버가 컴포넌트를 실행
  └─ 서버 내부에서 DB를 즉시 조회 (네트워크 지연 거의 0!)
  └─ 무거운 라이브러리(마크다운 파서 등)도 서버에서 실행 후 폐기!

Step 3: 직렬화 (Serialization → RSC Payload)
  └─ 서버가 결과물을 JS 코드가 아니라 가벼운 JSON 지시서로 압축!
  └─ "1번 구역에 '아이폰', 2번 구역에 '갤럭시' 텍스트를 넣어라"
  └─ 이것이 RSC Payload! (무거운 JS 번들이 아닌 가벼운 텍스트!)

Step 4: 전송 & 렌더링 (Streaming)
  └─ 가벼운 RSC Payload가 브라우저로 날아감
  └─ 브라우저는 지시서를 보고 즉시 화면 완성!
  └─ 서버 컴포넌트의 JS 코드는 단 1바이트도 브라우저에 안 감!
```

```
CSR vs SSR vs RSC — 브라우저 JS 번들 비교:

CSR:  서버 → [빈 HTML + 2MB JS 번들] → 브라우저(JS 실행 → API 호출 → 렌더링)
      → 초기 로딩 3~5초 😱

SSR:  서버 → [완성 HTML + 2MB JS 번들] → 브라우저(즉시 표시! + JS로 하이드레이션)
      → 화면은 빠름! 하지만 여전히 2MB JS 다운로드...

RSC:  서버 → [완성 HTML + RSC Payload(수 KB)] → 브라우저(즉시 표시! JS 최소!)
      → 화면 빠름 + JS 거의 없음! 🚀
      → 'use client' 컴포넌트의 JS만 전송 (나뭇잎만!)
```

## 2. 하이브리드 모델 — 서버 + 클라이언트 조합 🔴

> 💡 **비유: 레스토랑의 주방(서버)과 홀(클라이언트)**
> 주방에서 요리를 완성(서버 컴포넌트)하고,
> 홀 직원이 손님과 대화(클라이언트 컴포넌트: 클릭, 입력)를 담당.

```
서버 컴포넌트 (기본값 — 주방):
  ✅ async/await로 데이터 직접 조회
  ✅ DB, 파일 시스템 직접 접근
  ✅ 무거운 라이브러리 사용해도 번들에 안 포함
  ❌ useState, useEffect, onClick 사용 불가!
  ❌ 브라우저 API (window, localStorage) 사용 불가!

클라이언트 컴포넌트 ('use client' 선언 — 홀):
  ✅ useState, useEffect, onClick 등 모든 훅/이벤트 사용
  ✅ 브라우저 API 사용
  ❌ async 컴포넌트 불가 (서버 전용)
  ❌ 이 컴포넌트의 코드가 브라우저로 전송됨 (번들 포함)
```

### 'use client'의 정확한 의미 🔴

```tsx
"use client";
// ⚠️ 이것은 "서버에서 렌더링하지 않겠다"는 뜻이 아니다!
// ✅ 정확한 의미: "이 컴포넌트부터 아래의 JS 코드를 브라우저로도 보내겠다"

// 서버에서도 HTML은 먼저 생성됨! (SSR)
// 그 후 브라우저에서 JS가 하이드레이션으로 기능을 입힘
```

```
'use client'가 없으면 (서버 컴포넌트):
  서버에서만 실행 → HTML만 브라우저로 → JS 전송 0바이트!

'use client'가 있으면 (클라이언트 컴포넌트):
  서버에서 HTML 먼저 생성 (SSR) → 브라우저로 HTML + JS 전송
  → 브라우저에서 하이드레이션 → onClick 등 활성화!
```

## 3. 서버/클라이언트 선택 기준 🔴

```
✅ 서버 컴포넌트 (기본값 — 대부분 이것!)
  - 데이터 표시만 하는 컴포넌트 (목록, 상세, 텍스트)
  - DB나 API에서 데이터를 가져와야 할 때
  - 보안이 필요한 정보 (API 키, DB 쿼리)
  - 무거운 라이브러리 (마크다운 파서 등)
  - SEO가 중요한 콘텐츠

✅ 클라이언트 컴포넌트 ('use client' 필요)
  - onClick, onChange 등 이벤트 핸들러
  - useState, useEffect 등 훅 사용
  - 브라우저 API (window, localStorage, geolocation)
  - 실시간 상호작용 (폼 입력, 토글, 드래그)
  - 외부 클라이언트 라이브러리 (차트, 에디터, 지도)
```

### 나뭇잎 패턴 (Leaf Component Pattern) — 가장 중요! 🔴

```tsx
// ❌ 나쁜 패턴: 페이지 전체를 클라이언트로!
"use client";  // 💀 서버 컴포넌트의 모든 이점을 포기!
export default function ProductPage() {
  const [liked, setLiked] = useState(false);
  // ... 데이터 가져오기도 여기서 (CSR로 전락!)
}

// ✅ 좋은 패턴: 상호작용이 필요한 부분만 떼어내기!
// page.tsx (서버 컴포넌트 — 데이터 조회)
export default async function ProductPage() {
  const product = await db.product.findUnique({ where: { id: 1 } });

  return (
    <div>
      <h1>{product.name}</h1>        {/* 서버: HTML만 전송 */}
      <p>{product.description}</p>    {/* 서버: HTML만 전송 */}
      <p>{product.price}원</p>        {/* 서버: HTML만 전송 */}
      <LikeButton productId={product.id} />  {/* 이것만 클라이언트! */}
    </div>
  );
}

// LikeButton.tsx (클라이언트 — 인터랙션만!)
"use client";
export default function LikeButton({ productId }: { productId: number }) {
  const [liked, setLiked] = useState(false);
  return <button onClick={() => setLiked(!liked)}>{liked ? "❤️" : "🤍"}</button>;
}
```

```
결과:
  페이지의 90% = 서버 컴포넌트 → JS 0바이트!
  페이지의 10% = 클라이언트 컴포넌트 → 최소한의 JS만!
  → 앱이 극도로 가벼움! 🚀
```

## 4. Composition 패턴 — 클라이언트 안에 서버를 넣는 법 🔴

> ⚠️ 클라이언트 컴포넌트에서 서버 컴포넌트를 직접 import하면 오염된다!
> → children으로 주입하는 것이 해결법!

```tsx
// ❌ 클라이언트가 서버를 직접 import → 서버 컴포넌트가 클라이언트로 전락!
"use client";
import HeavyServerComponent from "./HeavyServerComponent";
export default function Bad() {
  return <HeavyServerComponent />;  // 💀 서버 코드가 브라우저로 전송됨!
}

// ✅ children으로 주입 → 서버 컴포넌트의 성질이 유지!
// ClientWrapper.tsx
"use client";
export default function ClientWrapper({ children }: { children: React.ReactNode }) {
  const [isOpen, setIsOpen] = useState(true);
  return isOpen ? <div>{children}</div> : null;
}

// page.tsx (서버 컴포넌트)
import ClientWrapper from "./ClientWrapper";
import HeavyServerComponent from "./HeavyServerComponent";

export default function Page() {
  return (
    <ClientWrapper>
      <HeavyServerComponent />  {/* 서버에서 렌더링됨! 브라우저 JS에 안 포함! */}
    </ClientWrapper>
  );
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. CSR, SSR, RSC(React Server Components)를 설명하고 적합한 상황?**

> CSR은 브라우저에서 JS로 화면을 그리는 방식으로 초기 로딩이 느리고 SEO가 불가하지만 인터랙션이 많은 대시보드에 적합하다. SSR은 서버에서 HTML을 완성해 보내므로 초기 로딩이 빠르고 SEO가 가능하지만 매 요청마다 서버 부하가 발생한다. RSC는 서버에서 렌더링하되 해당 컴포넌트의 JS를 브라우저로 전송하지 않아(Zero Bundle Size) 앱이 극도로 가벼워진다. Next.js App Router에서는 RSC가 기본이고 인터랙션이 필요한 부분만 'use client'로 클라이언트 컴포넌트를 사용한다.

**Q2. 'use client'의 정확한 의미?**

> "서버에서 렌더링하지 않겠다"가 아니라 "이 컴포넌트의 JS 코드를 브라우저로도 보내겠다"는 선언이다. 서버에서 HTML은 먼저 생성(SSR)되고, 브라우저에서 JS로 하이드레이션하여 이벤트 핸들러와 훅이 활성화된다. 페이지 전체에 'use client'를 거는 것은 서버 컴포넌트의 이점을 모두 포기하는 것이므로, 상호작용이 필요한 최소 단위(나뭇잎 패턴)에만 적용해야 한다.

**Q3. 서버 컴포넌트와 클라이언트 컴포넌트를 어떻게 조합하나요?**

> 기본적으로 모든 컴포넌트는 서버 컴포넌트로 유지하고, useState/onClick 등이 필요한 최소한의 부분만 별도 파일로 분리하여 'use client'를 선언한다(나뭇잎 패턴). 클라이언트 컴포넌트 안에 서버 컴포넌트를 넣어야 할 때는 직접 import 대신 children Props로 주입하면 서버 컴포넌트의 성질이 유지된다(Composition 패턴).

```
│                                                │
│  나뭇잎 패턴:                                    │

│    → JS 번들 최소화!                             │

│                                                │
└────────────────────────────────────────────────┘
```
