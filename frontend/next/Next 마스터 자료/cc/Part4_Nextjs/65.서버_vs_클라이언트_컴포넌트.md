# 65. 서버 vs 클라이언트 컴포넌트

> **키워드**: `기본값=서버` `'use client'` `console.log 위치` `선택 기준` `나뭇잎 패턴`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 서버/클라이언트 구분, 선택 기준, 'use client' 경계, Composition 패턴 | Next.js의 핵심 |
| 🟡 이해 | 번들 최적화, Provider 래퍼, 'use client' 전염성 | 실무 설계 |
| 🟢 참고 | 직렬화(Serialization) 제약, React Flight Protocol | 내부 동작 |

## 1. 기본값 = 서버 컴포넌트 🔴

> Next.js App Router에서 **'use client' 선언이 없으면 전부 서버 컴포넌트!**

```tsx
// 서버 컴포넌트 (기본값 — 'use client' 없음!)
export default async function ProductPage() {
  // ✅ 할 수 있는 것:
  const data = await db.product.findMany();     // DB 직접 접근!
  const secret = process.env.API_SECRET;        // 서버 전용 환경 변수!
  const parsed = await parseMarkdown(content);  // 무거운 라이브러리! (번들에 안 포함!)

  console.log("서버에서 실행!");  // → VS Code 터미널에 찍힘!

  // ❌ 할 수 없는 것:
  // const [count, setCount] = useState(0);    // 에러!
  // useEffect(() => { }, []);                 // 에러!
  // onClick={() => { }}                       // 에러!
  // window.localStorage                      // 에러!

  return <div>{data.map(p => <p key={p.id}>{p.name}</p>)}</div>;
}

// 클라이언트 컴포넌트 ('use client' 선언!)
"use client";
export default function LikeButton() {
  const [liked, setLiked] = useState(false);      // ✅ 훅 사용!
  console.log("클라이언트에서 실행!");  // → 브라우저 콘솔에 찍힘!

  return <button onClick={() => setLiked(!liked)}>  // ✅ 이벤트!
    {liked ? "❤️" : "🤍"}
  </button>;
}
```

### console.log로 실행 위치 확인! (면접 단골!)

```
서버 컴포넌트의 console.log → VS Code 터미널 (서버)
클라이언트 컴포넌트의 console.log → 브라우저 F12 콘솔
```

## 2. 선택 기준 — 어디서 실행해야 하나? 🔴

```
┌──────────────────────────┬──────────┬──────────┐
│         기능              │  서버 ✅  │ 클라이언트│
├──────────────────────────┼──────────┼──────────┤
│ DB 직접 접근              │    ✅     │    ❌    │
│ 서버 전용 환경 변수       │    ✅     │    ❌    │
│ 무거운 라이브러리         │    ✅     │   ⚠️    │
│ SEO 중요한 콘텐츠         │    ✅     │    ❌    │
│ 보안 민감 정보            │    ✅     │    ❌    │
│ async/await 컴포넌트      │    ✅     │    ❌    │
├──────────────────────────┼──────────┼──────────┤
│ useState / useEffect     │    ❌     │    ✅    │
│ onClick / onChange        │    ❌     │    ✅    │
│ 브라우저 API (window 등)  │    ❌     │    ✅    │
│ 실시간 상호작용           │    ❌     │    ✅    │

└──────────────────────────┴──────────┴──────────┘

결론: 기본은 서버! 상호작용이 필요한 최소 부분만 클라이언트!
```

## 3. 나뭇잎 패턴 (Leaf Component) — 가장 중요한 설계 원칙! 🔴

```
❌ 나쁜 패턴: 페이지 전체를 'use client'
  → 서버 컴포넌트의 모든 이점(Zero Bundle, DB 접근, SEO) 포기!
  → CSR(순수 React)과 다를 바 없음!

✅ 좋은 패턴: 나뭇잎(인터랙티브 부분)만 'use client'
  → 페이지의 90%는 서버 (JS 0바이트)
  → 10%의 버튼/입력만 클라이언트 (최소 JS)
```

```tsx
// ✅ 실전 패턴: 상품 상세 페이지
// page.tsx — 서버 컴포넌트 (데이터 조회!)
import { db } from "@/lib/db";
import LikeButton from "./LikeButton";
import AddToCartButton from "./AddToCartButton";

export default async function ProductPage({ params }: Props) {
  const { id } = await params;
  const product = await db.product.findUnique({ where: { id: Number(id) } });

  return (
    <div>
      {/* 서버: HTML만 전송, JS 0바이트! */}
      <h1>{product.name}</h1>
      <p className="text-2xl">{product.price.toLocaleString()}원</p>
      <p>{product.description}</p>

      {/* 클라이언트: 이 두 버튼만 JS 필요! */}
      <div className="flex gap-4 mt-8">
        <LikeButton productId={product.id} />
        <AddToCartButton product={product} />
      </div>
    </div>
  );
}
```

## 4. 'use client' 전염성 — 가장 흔한 실수! 🔴

```tsx
// ⚠️ 'use client' 파일에서 import한 모든 모듈도 클라이언트 번들에 포함!

// ❌ 위험: 클라이언트 컴포넌트가 서버 컴포넌트를 직접 import!
"use client";
import HeavyServerComponent from "./HeavyServerComponent";
// → HeavyServerComponent의 코드(+ 모든 import)가 브라우저 번들에 포함! 💀
// → 서버 전용 코드(DB, 환경변수)가 노출될 위험!

export default function BadClient() {
  return <HeavyServerComponent />;  // 서버 컴포넌트의 성질을 잃음!
}
```

### 해결: children 주입 (Composition 패턴) 🔴

```tsx
// ✅ 클라이언트는 "구멍"만 뚫어두고, 서버가 조립!
// ClientWrapper.tsx ('use client')
"use client";
export default function ClientWrapper({ children }: { children: React.ReactNode }) {
  const [isOpen, setIsOpen] = useState(true);
  return (
    <div>
      <button onClick={() => setIsOpen(!isOpen)}>토글</button>
      {isOpen && children}  {/* 무엇이 들어올지 모름! 그냥 렌더링만! */}
    </div>
  );
}

// page.tsx (서버 컴포넌트)
import ClientWrapper from "./ClientWrapper";
import HeavyChart from "./HeavyChart";  // 서버 컴포넌트!

export default async function Page() {
  return (
    <ClientWrapper>
      <HeavyChart />  {/* 서버에서 렌더링! 브라우저 번들에 안 포함! */}
    </ClientWrapper>
  );
}
```

## 5. Provider 래퍼 패턴 (Context, TanStack Query, Zustand) 🟡

> Provider는 'use client'가 필요한데, layout.tsx는 서버 컴포넌트인 경우가 많음!
> → Provider만 클라이언트 래퍼로 분리!

```tsx
// components/Providers.tsx ('use client')
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { ThemeProvider } from "@/contexts/ThemeContext";

const queryClient = new QueryClient();

export default function Providers({ children }: { children: React.ReactNode }) {
  return (
    <QueryClientProvider client={queryClient}>
      <ThemeProvider>
        {children}
      </ThemeProvider>
    </QueryClientProvider>
  );
}

// app/layout.tsx (서버 컴포넌트!)
import Providers from "@/components/Providers";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko">
      <body>
        <Providers>  {/* 클라이언트 래퍼! */}
          {children}  {/* 서버 컴포넌트 children은 서버에서 렌더링! */}
        </Providers>
      </body>
    </html>
  );
}
```

## 6. 실습

```
과제: 서버/클라이언트 분리 실전

1. 상품 상세 페이지 (서버 컴포넌트)
   - 데이터 하드코딩
   - async 컴포넌트로 1초 지연
   - console.log가 터미널에만 찍히는지 확인!

2. 좋아요 버튼 + 장바구니 버튼 (클라이언트 컴포넌트)
   - 각각 별도 파일 (나뭇잎!)
   - console.log가 브라우저에만 찍히는지 확인!

3. ClientWrapper + children 패턴
   - 토글 가능한 클라이언트 래퍼
   - 서버 컴포넌트를 children으로 주입
   - 서버 컴포넌트의 console.log가 터미널에 찍히는지 확인!

4. Provider 래퍼
   - Providers.tsx ('use client')에 ThemeProvider 구현
   - layout.tsx (서버)에서 Providers로 감싸기
```

## 면접 대비

### 🔴 필수 질문

**Q1. Next.js에서 서버 컴포넌트와 클라이언트 컴포넌트의 차이?**

> 서버 컴포넌트(기본값)는 서버에서만 실행되어 HTML을 생성하고, JS를 브라우저로 전송하지 않는다(Zero Bundle Size). DB 직접 접근, 서버 환경변수, async/await가 가능하다. 클라이언트 컴포넌트('use client')는 서버에서 HTML을 먼저 생성(SSR)한 후 브라우저에서 JS로 하이드레이션하여 상호작용(useState, onClick)을 활성화한다.

**Q2. 'use client'의 전염성이란?**

> 클라이언트 컴포넌트에서 import한 모든 모듈은 브라우저 번들에 포함된다. 서버 컴포넌트를 클라이언트에서 직접 import하면 서버의 성질을 잃고 클라이언트로 전락한다. 이를 방지하기 위해 children 주입 패턴(Composition)을 사용하여, 클라이언트 컴포넌트는 "구멍"만 뚫어두고 부모(서버)가 서버 컴포넌트를 조립하여 전달한다.

**Q3. Provider를 layout.tsx에서 어떻게 사용하나요?**

> layout.tsx가 서버 컴포넌트인 경우, Provider(QueryClientProvider, ThemeProvider 등)는 'use client'가 필요하므로 별도의 `Providers.tsx` 래퍼 컴포넌트로 분리한다. layout.tsx에서 `<Providers>{children}</Providers>`로 감싸면 children은 여전히 서버 컴포넌트의 이점을 유지한다.
