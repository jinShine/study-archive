## 1. Request Memoization — 렌더링 내 중복 요청 제거 🟡

```
한 번의 페이지 렌더링에서 같은 fetch를 여러 번 호출해도 → 실제 요청 1번!

generateMetadata: fetch("/api/product/1")  → 실제 호출 ✅
ProductPage:      fetch("/api/product/1")  → 캐시에서 반환! (네트워크 0!)
ReviewSection:    fetch("/api/product/1")  → 캐시에서 반환! (네트워크 0!)

→ 3번 호출했지만 실제 네트워크 요청은 1번!
→ Props Drilling 없이 각자 독립적으로 fetch 가능!

⚠️ 렌더링이 끝나면 이 메모리 캐시는 폐기됨! (다음 요청에는 다시 fetch)
```

---

## 2. 캐시 로깅 — 디버깅의 눈 🟡

```ts
// next.config.ts
const nextConfig: NextConfig = {
  logging: {
    fetches: {
      fullUrl: true,  // 서버 터미널에 fetch URL + 캐시 상태 표시!
    },
  },
};

// 터미널 출력 예시:
// GET https://api.example.com/products 200 in 45ms (cache: HIT)
// GET https://api.example.com/stock 200 in 120ms (cache: SKIP)
// GET https://api.example.com/products 200 in 1ms (cache: HIT) ← Memoization!
```

## 3. HTTP Cache-Control 헤더 (Route Handler에서) 🟢

```tsx
// Route Handler에서 CDN 캐시 제어
export async function GET() {
  const data = await fetchData();

  return Response.json(data, {
    headers: {
      "Cache-Control": "s-maxage=60, stale-while-revalidate=30",
      // s-maxage=60: CDN이 60초 동안 캐시!
      // stale-while-revalidate=30: 60초 후 30초 동안은 낡은 캐시 반환 + 백그라운드 갱신!
    },
  });
}
```

```
타임라인:
  T=0초:   첫 요청 → CDN 캐시 저장 (60초 유효)
  T=30초:  요청 → CDN 캐시 즉시 반환! (서버 안 감!)
  T=65초:  요청 → 낡은 캐시 즉시 반환 + 백그라운드 갱신! (SWR!)
  T=90초:  유예 기간(30초) 만료 → 캐시 폐기 → 다시 서버 요청
```
