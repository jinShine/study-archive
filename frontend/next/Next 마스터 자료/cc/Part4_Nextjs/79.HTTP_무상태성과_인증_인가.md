## 1. HTTP 무상태성(Stateless) — 서버는 기억하지 않는다 🔴

> 💡 **비유: 건망증 은행원**
> 방금 신분증을 보여줬는데, 1초 후에 다시 물어본다.
> "고객님, 신분증 보여주시겠습니까?"
> HTTP 서버는 **매 요청이 완전히 독립적**이다. 이전 요청을 기억하지 않는다!

```
요청 1: POST /login (로그인 성공! ✅)
요청 2: GET /mypage → "누구세요?" 😱

서버가 기억 안 하니까 매번 "나 로그인한 사람이야!"를 증명해야 한다!
→ 이 증명서가 "세션 ID" 또는 "JWT 토큰"!
```

### 왜 Stateless인가? — Scale-out의 마법

```
Stateful (상태 유지) 서버:
  사용자 100만 명의 로그인 상태를 RAM에 저장
  → 메모리 폭발! 서버 1대로 감당 불가!
  → 서버를 늘려도 "1번 서버에 로그인한 사용자가 2번 서버로 가면?" 문제!

Stateless (무상태) 서버:
  서버가 아무것도 기억하지 않음!
  사용자가 매 요청마다 증명서(토큰)를 들고 옴
  → 어떤 서버가 받아도 토큰만 확인하면 됨!
  → 서버를 10대, 100대 무한 확장 가능! (Scale-out!)
```

## 2. 인증(Authentication) vs 인가(Authorization) 🔴

```
인증 (Authentication — AuthN):
  "당신이 누구인지" 확인하는 과정
  = 로그인! (아이디/비밀번호, 소셜 로그인, 생체 인식)
  실패 시: 401 Unauthorized ← ⚠️ 이름이 혼란스럽지만 "인증" 실패!

인가 (Authorization — AuthZ):
  "당신이 무엇을 할 수 있는지" 확인하는 과정
  = 권한 검사! (일반 유저 vs 관리자)
  실패 시: 403 Forbidden

비유:
  인증 = 회사 로비에서 사원증 검사 (넌 누구야?)
  인가 = 서버실 앞에서 권한 확인 (넌 들어갈 수 있어?)
```

> ⚠️ **면접 킬러**: "401 Unauthorized는 이름과 달리 **인증(Authentication)** 실패입니다.
> **인가(Authorization)** 실패는 403 Forbidden입니다."

## 3. Session vs JWT 🔴

### Session — 서버가 장부를 관리

```
1. 로그인 성공 → 서버가 세션 ID 생성 → DB/Redis에 저장
2. 세션 ID를 쿠키로 클라이언트에 전달
3. 매 요청마다 쿠키의 세션 ID → 서버가 DB에서 조회 → 유저 확인

장점: 서버가 주도권 → 강제 로그아웃, 동시 접속 제한 가능!
단점: 서버 확장 시 세션 공유 필요 (Redis)
```

### JWT — 토큰 자체에 정보 담기

```
1. 로그인 성공 → 서버가 JWT 생성 (유저 ID + 권한 + 만료시간 + 서명)
2. JWT를 클라이언트에 전달 (쿠키 또는 헤더)
3. 매 요청마다 JWT 전송 → 서버가 서명만 검증 (DB 조회 안 함!)

장점: 서버 확장 쉬움! (DB 조회 없이 검증!)
단점: 한 번 발급하면 만료 전까지 무효화 불가 → Refresh Token 필요!

JWT 구조: header.payload.signature
  header: 알고리즘 정보
  payload: 유저 ID, 권한, 만료시간 (⚠️ 암호화 아님! Base64 인코딩!)
  signature: 위조 방지 서명 (서버만 아는 비밀 키로 생성)

⚠️ Payload는 누구나 열어볼 수 있다! 비밀번호 절대 넣지 마!
```

### 비교표

```
┌──────────┬──────────────┬──────────────┐
│          │ Session       │ JWT          │
├──────────┼──────────────┼──────────────┤
│ 저장 위치 │ 서버 (DB/Redis)│ 클라이언트   │
│ 서버 부하 │ DB 조회 필요   │ 서명 검증만!  │
│ 확장성    │ 세션 공유 필요 │ 무한 확장 ✅  │
│ 강제 로그아웃│ ✅ 쉬움      │ ❌ 어려움    │
│ 토큰 크기 │ 작음 (세션 ID)│ 큼 (페이로드) │
│ 실무 선택 │ 전통적 웹     │ API/모바일 ✅ │
└──────────┴──────────────┴──────────────┘
```

## 4. Cookie 보안 속성 — 토큰 저장의 방패! 🔴

```
Set-Cookie: token=eyJhbG...; HttpOnly; Secure; SameSite=Strict; Max-Age=3600

HttpOnly   → JS(document.cookie)로 접근 불가! XSS 방어!
Secure     → HTTPS에서만 전송! 도청 방어!
SameSite   → 다른 사이트에서 요청 시 쿠키 안 보냄! CSRF 방어!
  Strict: 완전 차단 (같은 사이트만)
  Lax: 링크 이동은 허용 (기본값)
  None: 제한 없음 (반드시 Secure와 함께!)
Max-Age    → 쿠키 수명 (초 단위)
```

### JWT 저장 위치: localStorage ❌ → HttpOnly Cookie ✅

```
❌ localStorage:
  장점: 간편
  단점: XSS 한 줄(localStorage.getItem("token"))에 탈취! 💀

✅ HttpOnly Cookie:
  장점: JS 접근 원천 차단! XSS로 못 훔쳐감!
  단점: CSRF 방어 필요 (SameSite로 해결)
  → 실무 보안 정석!
```

## 5. Next.js에서의 인증 흐름 🔴

```
로그인:
  1. 사용자가 이메일/비밀번호 입력 → Server Action 호출
  2. 서버에서 DB 확인 → JWT 생성
  3. HttpOnly Cookie에 JWT 저장 (Set-Cookie 헤더)
  4. redirect("/dashboard")

인증 확인 (매 요청):
  1. 서버 컴포넌트/미들웨어에서 Cookie 읽기
  2. JWT 검증 (서명 확인 + 만료 확인)
  3. 유효 → 페이지 렌더링 / 무효 → redirect("/login")

로그아웃:
  1. Server Action에서 Cookie 삭제
  2. redirect("/login")
```

## 면접 대비

### 🔴 필수 질문

**Q1. HTTP가 Stateless인 이유?**

> 서버가 클라이언트의 상태를 기억하지 않으므로, 서버를 수십 대로 확장(Scale-out)할 때 어떤 서버가 요청을 받아도 동일하게 처리할 수 있다. 상태를 저장하면 서버 간 동기화가 필요하고 메모리 부담이 커진다. 대신 클라이언트가 매 요청마다 인증 토큰(JWT/세션 ID)을 전송하여 신원을 증명한다.

**Q2. 401과 403의 차이?**

> 401 Unauthorized는 인증(Authentication) 실패로 유효한 인증 정보가 없는 상태다(로그인 안 함/토큰 만료). 403 Forbidden은 인가(Authorization) 실패로 인증은 되었지만 해당 자원에 접근할 권한이 없는 상태다(일반 유저가 관리자 페이지 접근).

**Q3. JWT를 localStorage가 아닌 HttpOnly Cookie에 저장해야 하는 이유?**

> localStorage는 JavaScript(`window.localStorage`)로 자유롭게 접근 가능하여 XSS 공격 시 토큰이 탈취된다. HttpOnly Cookie는 JavaScript에서 접근 자체가 차단되어 XSS로부터 안전하다. Secure 속성으로 HTTPS 전용 전송, SameSite로 CSRF 방어를 추가하면 3중 보안이 완성된다.
