## 1. 매크로 캐싱의 비극 — "빈대 잡으려다 초가삼간" 🟡

> 💡 **비유: 블랙프라이데이 한정판 스니커즈 매장**
> 자정에 반값 할인 시작! 수백만 명이 새로고침을 연타한다!

### 극단적 선택 1: 전체 동적 (데이터 신선도 100%)

```
모든 접속자에게 매번 DB 조회 → 항상 최신 재고!
수백만 요청 × DB 쿼리 = 💥 서버 폭발!
```

### 극단적 선택 2: 전체 정적 (서버 부하 0%)

```
페이지를 10분 캐시로 얼려둠 → 0.001초 반환! 서버 부하 0!
하지만 1초 만에 품절된 스니커즈가 10분간 "구매 가능"으로 표시 😱
→ 결제 시 전부 튕김 → 클레임 폭주!
```

### 매크로의 한계 — All or Nothing

```
블랙프라이데이 상품 상세 페이지를 분석하면:

🔵 90%의 정적 영역 (거의 안 변함):
  - 상품 이미지, 브랜드 설명, 리뷰 (1년째 같은 내용)

🔴 10%의 동적 영역 (1초마다 변함):
  - 장바구니 아이콘 (사용자마다 다름!)
  - "현재 재고: 3개" (1초마다 변함!)

과거의 투박한 해결:
  ❌ 전체 캐시 → 재고까지 얼어붙음!
  ❌ 전체 동적 → 90% 정적 콘텐츠도 매번 DB 조회!
  ❌ cookies() 한 줄이면 → 페이지 전체 캐시가 박살남!

→ 이것이 "페이지 단위(매크로) 캐싱"의 치명적 한계!
→ 빈대(10%) 잡으려다 초가삼간(90%) 태우는 꼴!
```

## 2. 마이크로 캐싱 혁명 — 컴포넌트 단위 핀셋 통제! 🟡

> 💡 **비유: 똑똑한 스마트 스프링클러**
> 물(서버 자원)을 줄 때 모든 곳에 뿌리지 않고,
> 꽃(동적 데이터)에만 주고 돌멩이(정적 콘텐츠)에는 안 준다!

```
마이크로 캐싱 = 컴포넌트 단위로 캐싱 전략을 다르게!

같은 상품 상세 페이지 안에서:
  ┌─────────────────────────────────────────────┐
  │  공지사항 헤더    → cacheLife("weeks")  1주!  │
  │  상품 이미지/설명  → cacheLife("hours")  1시간│
  │  실시간 리뷰      → cacheLife("minutes") 1분 │
  │  실시간 재고      → 캐시 없음! 매번 조회!     │
  │  장바구니 아이콘   → 클라이언트 컴포넌트!      │
  └─────────────────────────────────────────────┘

수백만 접속 시:
  공지사항: 1주간 캐시 반환 (DB 조회 0!)
  상품 설명: 1시간간 캐시 반환 (DB 조회 0!)
  리뷰: 1분마다 1번만 조회 (수백만 → 1번!)
  재고: 매번 조회 (이건 실시간이어야 하니까!)

→ 서버 부하: 극도로 낮아짐!
→ 재고 신선도: 100% 정확!
→ 이것이 "스위트스팟(Sweet Spot)"!
```

## 3. 'use cache' — 마법의 지시어 🟡

```tsx
// 이 컴포넌트의 렌더링 결과물을 캐시에 보관!
"use cache";
import { cacheLife } from "next/cache";

export default async function ProductDescription({ id }: { id: string }) {
  cacheLife("hours");  // 1시간 TTL!

  // 이 무거운 DB 쿼리는 1시간에 1번만 실행!
  // 나머지 접속자는 캐시에서 0.001초 반환!
  const description = await db.product.getDescription(id);

  return <article className="prose">{description}</article>;
}
```

### cacheLife 프로필 상세

```tsx
cacheLife("seconds");   // stale: 0,   revalidate: 1초   → 거의 실시간
cacheLife("minutes");   // stale: 5분, revalidate: 1분   → 리뷰, 댓글
cacheLife("hours");     // stale: 5분, revalidate: 1시간 → 상품 설명
cacheLife("days");      // stale: 5분, revalidate: 1일   → 공지사항
cacheLife("weeks");     // stale: 5분, revalidate: 1주   → 이용약관
cacheLife("max");       // 최대한 오래!                   → 변하지 않는 콘텐츠

stale: 클라이언트 라우터 캐시 유지 시간
  → 이 시간 동안 서버에 묻지도 않고 캐시 반환!

revalidate: 서버 백그라운드 갱신 주기
  → 이 시간이 지나면 SWR 패턴으로 백그라운드 갱신!
```

> 💡 **TTL 스위트스팟**: "이 데이터가 몇 분 옛날 걸 보여줘도 치명적이지 않은가?"
> 재고 → 0분 (실시간!) / 리뷰 → 1분 OK / 설명 → 1시간 OK / 약관 → 1주 OK

## 4. TTFB 병목과 스트리밍 — PPR의 존재 이유 🟡

### 문제: 하나가 느리면 전체가 멈춤!

```
리뷰 컴포넌트의 cacheLife가 만료되어 DB를 3초간 조회하는 상황:

❌ 스트리밍 없이:
  서버가 리뷰를 완성할 때까지 → 3초 동안 → 화면 전체가 백지! 🤍
  이미 캐시에 있는 상품 설명(0.001초)까지 3초를 기다림!
  TTFB(Time To First Byte) = 3초 → 사용자: "사이트 죽었나?"

✅ 스트리밍(PPR):
  0.001초: 캐시된 상품 설명이 먼저 브라우저에 도착! 화면 표시!
  0.001초: 리뷰 자리에 <Suspense> fallback(스피너) 표시!
  3초 후: 리뷰 데이터가 완성되면 스트리밍으로 스피너 자리에 쏙!
  → 사용자는 0.001초만에 화면을 봄! 리뷰만 나중에 채워짐!
```

```
시간선:

❌ 스트리밍 없이:
  T=0                                    T=3초
  |──── 하얀 화면 (TTFB 3초) ──────────|──── 전체 표시!

✅ 스트리밍(PPR):
  T=0.001초   T=0.1초                    T=3초
  |── 껍데기! ──|── 설명 표시! ──────────|── 리뷰 스트리밍!
   (즉시!)      (캐시!)      (스피너)     (완성!)
```

> 💡 **PPR = Partial Prerendering**
> 정적 껍데기는 빌드 시 미리 만들어두고 (0초!)
> 동적 부분은 `<Suspense>` 구멍으로 남겨두어 스트리밍!

## 5. PPR 구현 — <Suspense> 방호벽 아키텍처 🟡

### 절대 규칙: 부모(껍데기)에서 await params 금지!

```
⚠️ PPR 모드에서 껍데기가 0초로 로드되려면
   껍데기 컴포넌트는 순수 정적이어야 함!
   await params 같은 동적 데이터를 읽으면 → 빌드 에러!

해결: params를 Promise 그대로 자식에게 넘기고
      <Suspense> 안쪽에서 await!
```

```tsx
// page.tsx — 정적 껍데기 (0초에 로드!)
import { Suspense } from "react";
import CachedDescription from "./CachedDescription";
import LiveStock from "./LiveStock";
import LiveReviews from "./LiveReviews";

export default function ProductPage({ params }: { params: Promise<{ id: string }> }) {
  // ⚠️ 여기서 await params 하지 않는다! 정적 유지!
  return (
    <div className="max-w-4xl mx-auto p-8">
      {/* 정적 영역: 0초에 즉시 표시! */}
      <h1 className="text-4xl font-bold">프리미엄 무선 헤드폰</h1>
      <p className="text-2xl text-gray-700 mt-2">299,000원</p>

      {/* 캐시된 영역: 1시간 캐시! (Suspense 방호벽!) */}
      <Suspense fallback={<div className="h-32 bg-gray-100 animate-pulse rounded mt-6" />}>
        <CachedDescription paramsPromise={params} />
      </Suspense>

      {/* 실시간 영역: 매번 서버 조회! */}
      <Suspense fallback={<p className="text-blue-500 mt-4 animate-pulse">재고 확인 중...</p>}>
        <LiveStock paramsPromise={params} />
      </Suspense>

      {/* 1분 캐시 영역: */}
      <Suspense fallback={<div className="h-48 bg-gray-50 animate-pulse rounded mt-6" />}>
        <LiveReviews paramsPromise={params} />
      </Suspense>
    </div>
  );
}
```

```tsx
// CachedDescription.tsx — 1시간 캐시 (무거운 DB 쿼리!)
"use cache";
import { cacheLife } from "next/cache";

export default async function CachedDescription({
  paramsPromise,
}: {
  paramsPromise: Promise<{ id: string }>;
}) {
  cacheLife("hours");  // 1시간!

  const { id } = await paramsPromise;

  // 2초 걸리는 무거운 쿼리! 하지만 1시간에 1번만!
  await new Promise(res => setTimeout(res, 2000));
  const desc = "이 헤드폰은 최고급 드라이버를 탑재하여...";

  return (
    <div className="mt-6 p-6 bg-gray-50 rounded-xl">
      <h2 className="font-bold text-xl mb-2">상품 설명</h2>
      <p className="text-gray-700 leading-relaxed">{desc}</p>
      <p className="text-xs text-gray-400 mt-2">
        캐시 시간: {new Date().toLocaleTimeString()} (1시간 동안 고정!)
      </p>
    </div>
  );
}
```

```tsx
// LiveStock.tsx — 실시간! 캐시 없음!
export default async function LiveStock({
  paramsPromise,
}: {
  paramsPromise: Promise<{ id: string }>;
}) {
  const { id } = await paramsPromise;
  const currentTime = new Date().toLocaleTimeString();

  return (
    <div className="mt-4 p-4 bg-green-50 border border-green-200 rounded-lg">
      <p className="font-bold text-green-700">
        재고: 7개 남음 (실시간: {currentTime})
      </p>
    </div>
  );
}
```

```tsx
// LiveReviews.tsx — 1분 캐시
"use cache";
import { cacheLife, cacheTag } from "next/cache";

export default async function LiveReviews({
  paramsPromise,
}: {
  paramsPromise: Promise<{ id: string }>;
}) {
  const { id } = await paramsPromise;

  cacheLife("minutes");  // 1분!
  cacheTag(`reviews-${id}`);  // 태그! (정밀 무효화용!)

  // 3초 걸리는 무거운 쿼리! 하지만 1분에 1번!
  await new Promise(res => setTimeout(res, 3000));

  return (
    <div className="mt-6">
      <h2 className="font-bold text-xl mb-4">고객 리뷰</h2>
      <div className="space-y-3">
        <div className="p-4 bg-white border rounded-lg">
          <p className="font-bold">⭐⭐⭐⭐⭐ 최고!</p>
          <p className="text-gray-600">음질이 정말 좋습니다.</p>
        </div>
        <div className="p-4 bg-white border rounded-lg">
          <p className="font-bold">⭐⭐⭐⭐ 만족</p>
          <p className="text-gray-600">배송이 빨랐어요.</p>
        </div>
      </div>
      <p className="text-xs text-gray-400 mt-2">
        리뷰 캐시 시간: {new Date().toLocaleTimeString()} (1분마다 갱신!)
      </p>
    </div>
  );
}
```

## 6. 온디맨드 캐시 무효화 — 3가지 저격총 🟡

### 상황: 악성 리뷰를 삭제해야 한다!

```
문제: 리뷰가 cacheLife("minutes")로 1분간 캐시되어 있음
     DB에서 삭제해도 화면에 1분간 남아있음!
     → 온디맨드(즉시) 무효화가 필요!
```

```tsx
import { revalidatePath, revalidateTag } from "next/cache";

export async function deleteReview(reviewId: string, productId: string) {
  // DB에서 삭제
  await db.review.delete({ where: { id: reviewId } });

  // 🎯 무기 1: revalidatePath — URL 기반 광역 타격
  revalidatePath(`/products/${productId}`);
  // → 이 페이지의 모든 캐시를 날림! (간단하지만 넓은 범위)

  // 🎯 무기 2: revalidateTag — 태그 기반 정밀 타격
  revalidateTag(`reviews-${productId}`);
  // → 리뷰 컴포넌트의 캐시만 콕 집어서 무효화!
  // → 상품 설명 캐시는 건드리지 않음! (효율적!)

  // 🎯 무기 3: updateTag — 즉시 클라이언트까지! (Next.js 16)
  // updateTag(`reviews-${productId}`);
  // → 서버 캐시 파괴 + 현재 화면을 보고 있는 사용자에게도 즉시 반영!
}
```

### 3가지 도구 비교표

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│              │ revalidatePath│ revalidateTag│ updateTag    │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 기준         │ URL 경로      │ 데이터 태그   │ 데이터 태그   │
│ 범위         │ 페이지 전체   │ 태그 달린 것만│ 태그 달린 것만│
│ 속도         │ 다음 요청 시  │ 다음 요청 시  │ 즉시! ⚡      │
│ 클라이언트   │ 다음 방문 시  │ 다음 방문 시  │ 현재 화면도!  │
│ 실무 용도    │ 단순 CRUD    │ 다중 경로     │ 실시간 반응    │
│ 추천 상황    │ 게시글 수정   │ 공지사항 변경  │ 좋아요/삭제   │
└──────────────┴──────────────┴──────────────┴──────────────┘

선택 기준:
  단순 → revalidatePath (충분!)
  같은 데이터가 여러 페이지 → revalidateTag (정밀!)
  즉각 반응 필요 → updateTag (즉시!)
```

## 8. 실습

```
과제: PPR + 마이크로 캐싱 상품 페이지 (완전판!)

1. next.config.ts: cacheComponents: true

2. 4개의 독립 컴포넌트:
   - 정적 껍데기: 상품명 + 가격 (0초!)
   - CachedDescription: 'use cache' + cacheLife("hours") + 2초 지연
   - LiveStock: 캐시 없음! 매번 새 시간!
   - CachedReviews: 'use cache' + cacheLife("minutes") + cacheTag + 3초 지연

3. 각각 <Suspense>로 감싸기 (스켈레톤 UI!)

4. npm run build → ◐ (PPR) 기호 확인!

5. npm run start:
   - 첫 접속: 껍데기 0초 → 설명 2초 → 리뷰 3초 (스트리밍!)
   - F5 연타: 설명은 시간 고정(캐시!) / 재고는 매번 변경(실시간!)
   - 1분 후: 리뷰 시간이 갱신되는 순간 확인! (SWR!)

6. 리뷰 삭제 Server Action → revalidateTag로 리뷰만 무효화!
   - 설명 캐시는 그대로인지 확인!
```

## 면접 대비

### 🟡 이해 질문

**Q1. 매크로 캐싱과 마이크로 캐싱의 차이?**

> 매크로 캐싱은 페이지 전체를 통째로 캐싱하거나 동적으로 처리하는 방식이다. 실시간 재고(10%) 때문에 변하지 않는 상품 설명(90%)까지 매번 렌더링하는 비효율이 발생한다. 마이크로 캐싱은 `'use cache'` 지시어로 컴포넌트 단위로 독립적인 TTL을 부여하여, 같은 페이지 안에서도 상품 설명은 1시간, 리뷰는 1분, 재고는 실시간으로 각각 다른 캐싱 전략을 적용할 수 있다.

**Q2. PPR(Partial Prerendering)이란?**

> 하나의 페이지에서 정적 껍데기(상품명, 가격)는 빌드 시 미리 생성하여 0초에 서빙하고, 동적 부분(재고, 리뷰)은 `<Suspense>` 구멍으로 남겨두어 서버에서 데이터가 준비되는 대로 스트리밍으로 채워넣는다. 느린 컴포넌트(3초짜리 리뷰) 하나 때문에 전체 페이지가 3초간 멈추는 TTFB 병목을 해결한다. 빌드 로그에서 ◐(반달) 기호로 표시된다.

**Q3. revalidatePath, revalidateTag, updateTag의 차이?**

> `revalidatePath`는 URL 경로의 모든 캐시를 무효화하는 광역 도구다. `revalidateTag`는 특정 태그가 붙은 캐시만 정밀 무효화하여, 리뷰를 지울 때 상품 설명 캐시는 건드리지 않을 수 있다. `updateTag`(Next.js 16)는 서버 캐시를 파괴하면서 현재 화면을 보고 있는 클라이언트까지 즉시 갱신한다.
