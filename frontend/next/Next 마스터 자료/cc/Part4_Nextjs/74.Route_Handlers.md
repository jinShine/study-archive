## 1. Route Handlers — Next.js의 API 서버 🔴

> 💡 Express/Koa 같은 별도 백엔드 서버 없이, Next.js 안에서 바로 API를 만든다!

```
규칙:
  1. 파일 이름: route.ts (소문자!)
  2. 함수 이름: 대문자 HTTP 메서드 (GET, POST, PATCH, DELETE)
  3. ⚠️ 같은 폴더에 page.tsx와 공존 불가!

  app/api/products/route.ts      → GET /api/products
  app/api/products/[id]/route.ts → GET /api/products/123
```

## 2. GET 핸들러 🔴

```tsx
// app/api/products/route.ts
import { NextResponse } from "next/server";

const products = [
  { id: 1, name: "노트북", price: 1200000 },
  { id: 2, name: "키보드", price: 85000 },
];

// 함수 이름이 반드시 대문자 GET! (소문자 get은 무시됨!)
export async function GET() {
  return NextResponse.json(products);
  // → { "Content-Type": "application/json" } 자동 설정!
}
```

### 동적 Route Handler ([id])

```tsx
// app/api/products/[id]/route.ts
import { NextResponse } from "next/server";

export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }  // Next.js 15: Promise!
) {
  const { id } = await params;

  const product = products.find(p => p.id === parseInt(id));

  if (!product) {
    return NextResponse.json(
      { error: "상품을 찾을 수 없습니다" },
      { status: 404 }
    );
  }

  return NextResponse.json(product);
}
```

## 3. POST 핸들러 + Validation 🔴

```tsx
// app/api/products/route.ts
export async function POST(request: Request) {
  try {
    // 1. 요청 본문 파싱
    const body = await request.json();

    // 2. 유효성 검사 (1차 방어선!)
    if (!body.name || !body.price) {
      return NextResponse.json(
        { error: "상품명과 가격은 필수입니다" },
        { status: 400 }
      );
    }

    if (typeof body.price !== "number" || body.price <= 0) {
      return NextResponse.json(
        { error: "가격은 양수여야 합니다" },
        { status: 400 }
      );
    }

    // 3. 데이터 생성
    const newProduct = {
      id: Date.now(),
      name: body.name,
      price: body.price,
    };

    // 4. 성공 응답 (201 Created!)
    return NextResponse.json(newProduct, { status: 201 });

  } catch (error) {
    // 5. 서버 에러
    return NextResponse.json(
      { error: "서버 내부 오류" },
      { status: 500 }
    );
  }
}
```

## 4. NextRequest — 요청 상세 정보 🟡

```tsx
import { NextRequest, NextResponse } from "next/server";

export async function GET(request: NextRequest) {
  // URL 정보
  const url = request.nextUrl;
  const searchParams = url.searchParams;
  const sort = searchParams.get("sort");     // ?sort=price
  const page = searchParams.get("page");     // ?page=2

  // 헤더 정보
  const authHeader = request.headers.get("Authorization");
  const contentType = request.headers.get("Content-Type");

  // 쿠키 정보
  const token = request.cookies.get("token")?.value;

  return NextResponse.json({ sort, page });
}
```

## 5. Route Handlers vs Server Actions — 언제 뭘 쓸까? 🔴

```
Route Handlers (route.ts):
  ✅ 외부 클라이언트가 호출할 API (모바일 앱, 서드파티)
  ✅ 웹훅(Webhook) 수신
  ✅ 파일 다운로드/업로드
  ✅ RESTful API가 필요한 경우
  → "다양한 클라이언트에게 데이터를 제공"

Server Actions ('use server'):
  ✅ 내 웹 앱의 폼 전송/데이터 변경
  ✅ 내부 CRUD 처리
  ✅ 캐시 무효화 + 리다이렉트
  → "내 프론트엔드에서 내 백엔드 호출" (RPC!)
  → fetch/URL 설계 불필요! 함수만 호출!

실무 결론:
  내부 폼/변경 → Server Actions (75번!)
  외부 API 제공 → Route Handlers
```

## 면접 대비

### 🔴 필수 질문

**Q1. Route Handlers와 Server Actions의 차이?**

> Route Handlers(`route.ts`)는 RESTful API 엔드포인트로, URL과 HTTP 메서드를 직접 설계하며 외부 클라이언트(모바일 앱, 서드파티)에 데이터를 제공할 때 사용한다. Server Actions(`'use server'`)는 프레임워크가 자동으로 숨겨진 엔드포인트를 생성하여 함수 호출처럼 서버 로직을 실행하며, 내 웹 앱의 폼 전송과 데이터 변경에 사용한다. 내부 CRUD는 Server Actions가 DX와 타입 안전성에서 우월하다.
