## 1. Next.js 15의 가장 큰 변화 — Opt-in 캐싱! 🔴

> ⚠️ **이것을 모르면 Next.js 면접에서 탈락!**

```
❌ Next.js 14 (과거 — Opt-out):
  "기본적으로 모든 fetch를 캐싱한다."
  "캐싱을 원하지 않으면 개발자가 no-store를 명시해야 한다."
  → 문제: "왜 내 데이터가 안 바뀌지?" 유령 버그의 온상!

✅ Next.js 15 (현재 — Opt-in):
  "기본적으로 아무것도 캐싱하지 않는다."
  "캐싱을 원하면 개발자가 명시적으로 force-cache를 선언한다."
  → 안전: 기본이 최신 데이터! 캐싱은 의도적으로!
```

> 💡 **면접 킬러 답변**: "Next.js 15부터 캐싱 기본 정책이 Opt-out에서 Opt-in으로 바뀌었습니다.
> 기본적으로 모든 요청이 동적(no-store)이며, 성능 최적화가 필요한 곳에만
> 개발자가 명시적으로 force-cache나 revalidate를 선언합니다."

## 2. Dynamic Rendering — 매번 새롭게! 🔴

```tsx
// Next.js 15 기본 동작: 매 요청마다 서버에서 새로 렌더링!
export default async function ProductsPage() {
  // 옵션 없음 = 기본 = 동적 렌더링 (매 요청마다 새로!)
  const res = await fetch("https://api.example.com/products");
  const products = await res.json();

  return <ProductList products={products} />;
}

// 명시적으로 no-store (같은 효과, 의도를 명확히!)
const res = await fetch("https://api.example.com/products", {
  cache: "no-store",  // "캐싱하지 마! 항상 최신!"
});
```

```
Dynamic Rendering:
  사용자 A 접속 → 서버가 API 호출 → HTML 생성 → 응답
  사용자 B 접속 → 서버가 API 호출 → HTML 생성 → 응답
  사용자 C 접속 → 서버가 API 호출 → HTML 생성 → 응답
  → 매번 최신 데이터! 하지만 서버 부하 ↑

적합한 용도:
  - 개인화된 데이터 (마이페이지, 장바구니)
  - 실시간 데이터 (주식, 재고)
  - 검색 결과 (쿼리마다 다름)
```

## 3. Static Rendering — 한 번 굽고 재사용! 🔴

```tsx
// force-cache: 한 번 가져와서 캐시에 영구 저장!
export default async function AboutPage() {
  const res = await fetch("https://api.example.com/company-info", {
    cache: "force-cache",  // "한 번만 가져와서 영원히 캐시!"
  });
  const info = await res.json();

  return <CompanyInfo data={info} />;
}
```

```
Static Rendering:
  빌드 시 → API 호출 1번 → HTML 생성 → 캐시에 저장
  사용자 A 접속 → 캐시에서 즉시 반환 (API 호출 안 함!)
  사용자 B 접속 → 캐시에서 즉시 반환
  100만 명 접속 → 캐시에서 즉시 반환 (서버 부하 0!)
  → 빛의 속도! 하지만 데이터가 과거에 멈춤

적합한 용도:
  - 회사 소개, 이용약관 (거의 안 변함)
  - 블로그 포스트 (작성 후 고정)
  - 마케팅 랜딩 페이지
```

## 4. 두 캐시 계층 이해하기 🔴

```
Next.js에는 2가지 독립적인 캐시가 있다:

┌── Data Cache ──────────────────────────────────┐
│ fetch() 결과(JSON 데이터)를 캐싱                  │
│ 제어: cache: "force-cache" / "no-store"          │
│       next: { revalidate: 60 }                   │
│ Next.js 15 기본: 캐싱 안 함 (no-store)            │
└────────────────────────────────────────────────┘

┌── Full Route Cache ────────────────────────────┐
│ 페이지의 완성된 HTML 껍데기를 캐싱                 │
│ 빌드 시 정적 페이지는 자동으로 HTML 생성           │
│ 제어: 페이지 내 모든 fetch가 정적이면 → 자동 정적!  │
│       하나라도 동적이면 → 전체 동적!               │
└────────────────────────────────────────────────┘
```

### 컴파일러의 정적/동적 판단 기준

```
빌드(npm run build) 시 Next.js가 각 페이지를 분석:

동적으로 만드는 것들 (하나라도 있으면 동적!):
  ✅ cookies() 또는 headers() 사용
  ✅ searchParams를 읽는 경우
  ✅ fetch에 cache: "no-store" 있는 경우
  ✅ export const dynamic = "force-dynamic" 선언

위 조건이 하나도 없으면:
  → 컴파일러가 "정적 페이지구나!" 판단
  → 빌드 시 HTML로 영구 박제 (SSG)!
  → ⚠️ fetch 기본이 no-store여도, Full Route Cache가 HTML을 박제할 수 있음!
```

## 5. 빌드 로그 읽기 🔴

```bash
npm run build

# 빌드 완료 후 출력되는 로그:

Route (app)                    Size     First Load JS
─ ○ /                          5.2 kB   85 kB
─ ○ /about                     2.1 kB   82 kB
─ ƒ /products                  3.5 kB   83 kB
─ ƒ /products/[id]             4.2 kB   84 kB
─ ● /blog/[slug]               3.8 kB   83 kB

기호 의미:
  ○ (Static)   → 빌드 시 HTML 생성 완료. 데이터 페칭 없음 또는 전부 캐싱됨
  ƒ (Dynamic)  → 매 요청마다 서버에서 렌더링 (no-store, cookies 등)
  ● (SSG)      → generateStaticParams로 빌드 시 데이터까지 포함하여 HTML 생성

확인: 내가 의도한 대로 ○/ƒ/●가 나오는지 반드시 체크!
  "이 페이지는 동적이어야 하는데 ○가 뜬다?" → 캐싱 설정 확인!
  "이 페이지는 정적이어야 하는데 ƒ가 뜬다?" → 동적 함수 사용 확인!
```

## 6. ⚠️ 캐싱 테스트는 반드시 프로덕션에서! 🔴

```bash
# ❌ 개발 모드에서 캐싱 테스트 → 부정확!
npm run dev
# dev 모드에서는 캐싱이 느슨하게 동작하여 실제와 다름!

# ✅ 프로덕션 모드에서 캐싱 테스트 → 정확!
npm run build    # 빌드
npm run start    # 프로덕션 서버 실행
# 이제 새로고침하면서 데이터가 바뀌는지/안 바뀌는지 확인!
```

```
┌──────────┬──────────────────┬──────────────────┬──────────────────┐
│          │ Dynamic (SSR)    │ Static (SSG)     │ ISR (70번!)      │
├──────────┼──────────────────┼──────────────────┼──────────────────┤
│ 코드      │ 기본 fetch       │ force-cache      │ revalidate: 60   │
│ 시점      │ 매 요청마다      │ 빌드 시 1번      │ 주기적 갱신      │
│ 속도      │ 보통             │ 매우 빠름 ⚡      │ 빠름 (캐시)      │
│ 데이터    │ 항상 최신 ✅     │ 빌드 시점 고정 ❌ │ 주기적 최신 🟡   │
│ 서버 부하  │ 높음 ❌          │ 없음 ✅          │ 낮음 ✅          │
│ 용도      │ 실시간/개인화    │ 약관/블로그      │ 쇼핑몰/뉴스      │
│ 빌드 로그  │ ƒ               │ ○               │ ○ (revalidate)   │
└──────────┴──────────────────┴──────────────────┴──────────────────┘
```

## 면접 대비

### 🔴 필수 질문

**Q1. Next.js 15에서 캐싱 기본 정책이 어떻게 바뀌었나요?**

> Next.js 14까지는 Opt-out 방식으로 모든 fetch가 기본적으로 캐싱되어 개발자가 `no-store`를 명시해야 최신 데이터를 얻었다. Next.js 15부터는 Opt-in 방식으로 기본이 캐싱 없음(동적 렌더링)이며, 캐싱이 필요한 곳에만 `force-cache`나 `revalidate`를 명시적으로 선언한다. 이로써 "데이터가 안 바뀌는" 유령 버그가 방지되고, 기본 상태가 가장 안전(항상 최신)하게 되었다.

**Q2. SSR, SSG, ISR을 각각 설명하고 적합한 상황?**

> SSR(Dynamic)은 매 요청마다 서버에서 렌더링하여 항상 최신 데이터를 제공하며 개인화/실시간 데이터에 적합하다. SSG(Static)는 빌드 시 HTML을 생성하여 CDN에서 즉시 서빙하므로 가장 빠르지만 데이터가 빌드 시점에 고정되어 약관/블로그에 적합하다. ISR은 SSG처럼 캐시를 제공하되 설정한 시간마다 백그라운드에서 갱신하여 속도와 최신성을 모두 잡으며 쇼핑몰/뉴스에 적합하다.

**Q3. Next.js 빌드 로그의 ○, ƒ, ● 기호 의미?**

> ○(Static)은 빌드 시 정적 HTML이 생성된 페이지, ƒ(Dynamic)은 매 요청마다 서버에서 렌더링되는 동적 페이지, ●(SSG)는 `generateStaticParams`로 빌드 시 데이터까지 포함하여 정적 생성된 페이지를 의미한다. 의도한 렌더링 방식과 빌드 로그가 일치하는지 반드시 확인해야 한다.
