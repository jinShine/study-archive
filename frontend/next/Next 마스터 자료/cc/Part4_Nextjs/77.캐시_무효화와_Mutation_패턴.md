## 1. 캐싱의 역설 — DB는 바뀌었는데 화면이 안 바뀜! 🔴

```
문제 상황:
  1. 상품 목록 페이지가 force-cache 또는 revalidate: 60으로 캐싱됨
  2. Server Action으로 새 상품을 DB에 추가!
  3. 상품 목록 페이지로 이동 → 새 상품이 안 보임! 😱
  4. 이유: 캐시에 저장된 옛날 HTML을 보여주고 있으니까!

해결: 데이터가 바뀌면 → 관련 캐시를 강제로 파괴!
```

## 2. revalidatePath — 경로 캐시 파괴! 🔴

```tsx
"use server";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

export async function createProduct(prevState: FormState, formData: FormData) {
  const name = formData.get("name") as string;
  const price = Number(formData.get("price"));

  // 유효성 검사
  if (!name || name.length < 2) {
    return { success: false, message: "상품명 2자 이상!" };
  }

  // DB 저장
  await db.product.create({ data: { name, price } });

  // ⭐ 캐시 파괴! "/products" 경로의 캐시를 날려버림!
  revalidatePath("/products");

  // ⭐ 사용자를 목록 페이지로 이동!
  redirect("/products");
  // ⚠️ redirect는 내부적으로 Error를 던져서 함수를 종료하므로
  // 반드시 함수의 마지막에! (이후 코드 실행 안 됨!)
}
```

### revalidatePath 옵션

```tsx
// 1. 특정 페이지만 파괴
revalidatePath("/products");         // /products 페이지 캐시 파괴

// 2. 동적 경로도 가능
revalidatePath(`/products/${id}`);   // /products/123 캐시 파괴

// 3. 하위 경로 전체 파괴 (layout 타입)
revalidatePath("/products", "layout");
// → /products, /products/1, /products/1/reviews 전부!

// 4. 현재 페이지만 (page 타입)
revalidatePath("/products", "page");
```

## 3. Mutation 완전체 패턴 — 생성 + 캐시 파괴 + 리다이렉트 🔴

```tsx
// ⭐ 실무에서 가장 많이 쓰는 완전한 Server Action 패턴!
"use server";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";

export async function createPost(prevState: FormState, formData: FormData) {
  const title = formData.get("title") as string;
  const content = formData.get("content") as string;

  // 1. 유효성 검사 (실패하면 에러 반환 → 화면에 표시)
  if (!title || title.length < 2) {
    return { success: false, message: "제목 2자 이상!" };
  }

  // 2. DB 저장
  await db.post.create({ data: { title, content } });

  // 3. 캐시 파괴 (목록 페이지가 최신 데이터를 보여주게!)
  revalidatePath("/posts");

  // 4. 리다이렉트 (생성 후 목록으로 이동!)
  redirect("/posts");

  // ⚠️ redirect 이후의 코드는 실행 안 됨!
  // return은 유효성 검사 실패 시에만 도달!
}

// 삭제 패턴
export async function deletePost(id: number) {
  await db.post.delete({ where: { id } });
  revalidatePath("/posts");
  // 삭제는 redirect 없이 현재 페이지에서 목록 갱신!
}

// 수정 패턴
export async function updatePost(id: number, prevState: FormState, formData: FormData) {
  const title = formData.get("title") as string;

  if (!title) return { success: false, message: "제목 필수!" };

  await db.post.update({ where: { id }, data: { title } });

  revalidatePath(`/posts/${id}`);  // 상세 페이지 캐시 파괴
  revalidatePath("/posts");         // 목록 페이지도 캐시 파괴
  redirect(`/posts/${id}`);
}
```

## 4. revalidateTag — 태그 기반 다중 무효화 🟡

```tsx
// 데이터를 가져올 때 태그를 부여!
const res = await fetch("/api/products", {
  next: { tags: ["products"] },  // "products" 태그 부착!
});

// 데이터가 변경되면 태그로 무효화!
import { revalidateTag } from "next/cache";

export async function createProduct(formData: FormData) {
  await db.product.create({ data: { ... } });

  revalidateTag("products");
  // → "products" 태그가 붙은 모든 fetch 캐시가 무효화!
  // → /products 페이지뿐 아니라 /home의 추천 상품 등도 갱신!
}
```

### revalidatePath vs revalidateTag

```
revalidatePath: URL 경로 기반
  "이 주소의 캐시를 날려!"
  간단하지만, 같은 데이터가 여러 페이지에 있으면 일일이 지정해야 함

revalidateTag: 데이터 태그 기반
  "이 태그가 붙은 모든 캐시를 날려!"
  하나의 태그로 여러 페이지의 캐시를 한 번에 무효화!

실무:
  단순 CRUD → revalidatePath (충분!)
  같은 데이터가 여러 곳 → revalidateTag (정밀!)
```

## 5. redirect 주의사항 🔴

```tsx
// ⚠️ redirect는 내부적으로 Error를 throw하여 함수를 즉시 종료!
// → try-catch 안에서 쓰면 잡혀버림!

// ❌ 잘못된 패턴
try {
  await db.create({ data });
  redirect("/posts");  // Error throw → catch에 잡힘!
} catch (error) {
  // redirect의 Error까지 잡아버림! 😱
  return { success: false, message: "실패!" };
}

// ✅ 올바른 패턴: redirect를 try-catch 밖에!
try {
  await db.create({ data });
} catch (error) {
  return { success: false, message: "DB 저장 실패!" };
}
// try 밖에서 redirect!
revalidatePath("/posts");
redirect("/posts");
```

## 6. 실습

```
과제: 게시판 CRUD 완전체

1. 생성: createPost → DB 저장 → revalidatePath → redirect
2. 삭제: deletePost → DB 삭제 → revalidatePath (현재 페이지)
3. 수정: updatePost → DB 수정 → revalidatePath(목록+상세) → redirect
4. 모든 Server Action에 유효성 검사 + 에러 반환
5. useActionState로 에러/성공 피드백
6. SubmitButton으로 pending 상태 관리
7. 캐시된 목록 페이지에서 생성/삭제 후 즉시 반영되는지 확인!
```

## 면접 대비

### 🔴 필수 질문

**Q1. Server Action에서 데이터 변경 후 캐시를 갱신하는 방법?**

> `revalidatePath("/경로")`로 해당 URL의 Data Cache와 Full Route Cache를 무효화하여 다음 요청 시 서버가 새 데이터를 가져오게 한다. 같은 데이터가 여러 페이지에 노출되면 `revalidateTag("태그")`로 태그 기반 다중 무효화가 가능하다. 데이터 변경 후 사용자를 다른 페이지로 보내려면 `redirect("/경로")`를 사용하며, redirect는 내부적으로 Error를 throw하므로 반드시 함수의 마지막에 호출해야 한다.
