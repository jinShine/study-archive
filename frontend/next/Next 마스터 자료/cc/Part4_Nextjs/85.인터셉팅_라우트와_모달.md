## 1. 왜 이 기술이 필요한가? — State vs URL 딜레마 🟢

> 💡 **비유: 모달을 띄울 때 겪는 3대 재앙**

```
❌ useState로만 모달을 관리하면:

  1. 공유 불가: 모달이 열린 상태의 URL을 복사해서 친구에게 보내면?
     → 친구는 모달이 닫힌 기본 화면만 봄! (URL이 안 바뀌니까)

  2. 새로고침 증발: 모달을 열고 F5를 누르면?
     → isOpen = false로 초기화 → 모달 사라짐!

  3. 뒤로 가기 탈주: 모달을 닫으려고 뒤로 가기를 누르면?
     → 모달이 닫히는 대신 사이트 밖으로 나가버림!

이유: 모달의 상태가 URL에 반영되지 않았기 때문!
해결: 모달을 띄울 때 URL도 함께 변경!
     → 하지만 URL을 바꾸면 페이지 전체가 바뀜 (Hard Navigation)!
     → 뒤에 깔린 피드가 사라져버림! 😱
```

### 아키텍트의 딜레마

```
목표: "피드를 유지한 채 URL만 /photo/1로 바꾸고, 모달을 그 위에 띄우자!"

이것을 해결하는 것이 바로:
  병렬 라우트(@modal) + 인터셉팅 라우트((..)photo) 융합!
```

## 2. 인터셉팅 라우트란? 🟢

> 사용자가 `<Link href="/photo/1">`를 클릭했을 때,
> 원래 목적지(/photo/1)로 가지 않고 → 현재 화면 위에 가로챈 UI를 띄움!
> 하지만 URL은 /photo/1로 정확히 변경!

### 문법: 라우트 세그먼트 기준으로 이동

```
(.)folder    → 같은 레벨의 세그먼트를 가로챔
(..)folder   → 한 단계 위 세그먼트를 가로챔
(..)(..)folder → 두 단계 위
(...)folder  → 루트(app) 수준을 가로챔

⚠️ 파일 시스템 경로가 아니라 "라우트 세그먼트" 기준!
```

### 두 가지 얼굴 — Soft vs Hard Navigation

```
같은 URL(/photo/1)인데 접근 방법에 따라 다른 화면!

Soft Navigation (내부 클릭):
  피드에서 사진 클릭 → (..) 함정이 발동!
  → 피드는 그대로 + 위에 모달 팝업!
  → URL: /photo/1 (동기화 ✅)

Hard Navigation (새로고침/외부 접속):
  주소창에 /photo/1 직접 입력 또는 F5
  → 함정 무시! 원본 photo/[id]/page.tsx 전체 화면!
  → 피드 없이 사진만 꽉 찬 페이지!
```

## 3. 완전한 폴더 구조 — 3D 입체 설계도 🟢

```
src/app/
├── photo/                       ← 원본 (Hard Nav 시 사용)
│   └── [id]/
│       ├── page.tsx             ← 🧱 전체 화면 사진 페이지
│       └── PhotoDetail.tsx      ← (Suspense 부품)
│
└── feed/                        ← 피드 영역
    ├── layout.tsx               ← 🏗️ 조립 공장 (1층 + 2층 포개기!)
    ├── page.tsx                 ← 📱 1층: 사진 목록 피드 (바닥)
    └── @modal/                  ← 🛡️ 2층: 모달 전용 슬롯
        ├── default.tsx          ← 🛑 평소엔 투명(null)! 404 방어!
        └── (..)photo/           ← 🪤 함정! /photo를 가로챔!
            └── [id]/
                ├── page.tsx     ← ✨ 모달 UI (반투명 오버레이)
                └── ModalDetail.tsx ← (Suspense 부품)
```

### 왜 이런 구조인가?

```
photo 원본이 feed 밖에 있는 이유:
  → 외부 공유(Hard Nav)시 피드에 종속되지 않는 독립 페이지!
  → SEO: 검색엔진이 사진 페이지를 독립적으로 인덱싱!

@modal이 feed 안에 있는 이유:
  → 모달은 "피드 화면 위"에 떠야 하므로
  → feed/layout.tsx가 @modal 슬롯을 통제!

(..)photo인 이유:
  → 현재 위치: feed/@modal 안
  → 가로챌 대상: photo (feed와 같은 레벨 = app 직하위)
  → feed에서 한 단계 위(..)로 올라가면 photo를 만남!
```

## 4. 핵심 코드 — 완전한 구현 🟢

### Step 1: 피드 레이아웃 (1층 + 2층 조립)

```tsx
// feed/layout.tsx
export default function FeedLayout({
  children,   // 1층: 피드 리스트 (바닥)
  modal,      // 2층: 모달 슬롯 (@modal 폴더)

  children: React.ReactNode;
  modal: React.ReactNode;

  return (
    <div className="relative min-h-screen">
      <main>{children}</main>

    </div>
  );
}
```

### Step 5: 모달 닫기 버튼 (History Stack 제어!)

```tsx
// feed/@modal/CloseModalButton.tsx
"use client";
import { useRouter } from "next/navigation";

export default function CloseModalButton() {
  const router = useRouter();

  return (
    <button
      onClick={() => router.back()}
      className="absolute top-4 right-4 p-2 px-4 bg-gray-100 hover:bg-gray-200
                 text-gray-800 rounded-full font-bold transition-colors"
      aria-label="모달 닫기"
    >
      ✕
    </button>
  );
}
```

> 💡 **왜 `router.back()`인가?**
> 이 모달은 **Soft Navigation**으로만 열리는 컴포넌트다.
> Soft Nav = 히스토리 스택에 [피드] → [모달] 순서로 쌓임.
> `router.back()` = 맨 위 [모달]을 빼내서 → [피드]로 돌아감!
> ✅ push("/feed")를 쓰면 [피드]→[모달]→[피드] 3개가 되어 스택이 꼬임!

### Step 6: 원본 전체 화면 (Hard Navigation용)

## 5. History Stack (LIFO) 완벽 해부 🟢

> 💡 **비유: 접시 무더기(Stack)**
> 새 접시는 맨 위에 올리고(push), 꺼낼 때도 맨 위부터(pop).
> 브라우저의 히스토리도 이 LIFO(Last-In, First-Out) 구조!

### Soft Navigation 시나리오 (정상!)

```
1. 사용자가 /feed에 접속
   Stack: [/feed]

2. 사진을 클릭 (Soft Nav → 모달!)
   Stack: [/feed] → [/photo/1]    ← push!
   화면: 피드 + 모달 오버레이

3. "✕ 닫기" 클릭 (router.back())
   Stack: [/feed]                  ← pop! (/photo/1 제거!)
   화면: 피드만 (모달 사라짐!)
   ✅ 완벽!
```

### Hard Navigation 시나리오 (주의!)

```
1. 외부에서 /photo/1 직접 접속 (또는 모달 상태에서 F5)
   Stack: [/photo/1]              ← 스택에 피드가 없음!
   화면: 전체 화면 사진 페이지

2. 만약 여기서 router.back()을 쓰면?
   Stack: [] (비었음!)             ← pop!
   → 브라우저 밖으로 나감! (구글, 카카오톡 등) 😱

3. 그래서 <Link href="/feed">를 사용!
   Stack: [/photo/1] → [/feed]    ← push! (새 접시 추가)
   → 사이트 내에서 안전하게 이동! ✅
```

### ❌ 절대 하면 안 되는 것: push로 모달 닫기

```
모달이 열린 상태에서:
  Stack: [/feed] → [/photo/1]

❌ router.push("/feed")로 닫으면:
  Stack: [/feed] → [/photo/1] → [/feed]  ← push! (3개!)

사용자가 뒤로 가기를 누르면:
  [/feed] → [/photo/1]  ← 모달이 좀비처럼 다시 뜸! 💀

✅ router.back()으로 닫아야:
  Stack: [/feed]  ← pop! (깔끔!)
```

## 7. 테스트 시나리오 3가지 🟢

### 테스트 1: 모달 열기 (Soft Nav)

```
1. http://localhost:3000/feed 접속
2. "상세보기" 클릭
3. 확인:
   ✅ 피드가 뒤에 유지된 채 모달이 떠오르는가?
   ✅ 주소창이 /photo/1로 바뀌었는가?
   ✅ 스크롤 위치가 유지되는가?
```

### 테스트 2: 새로고침 (Hard Nav)

```
1. 모달이 뜬 상태에서 F5 (또는 주소창에서 Enter)
2. 확인:
   ✅ 모달이 사라지고 전체 화면 사진 페이지가 뜨는가?
   ✅ "피드로 돌아가기" 링크가 있는가?
```

### 테스트 3: 닫기 + 뒤로 가기

```
1. 피드 → 사진 클릭 (모달)
2. "✕ 닫기" 클릭
3. 확인:
   ✅ 모달이 닫히고 피드로 돌아가는가?
   ✅ 브라우저 "뒤로 가기" 버튼과 동일하게 동작하는가?
   ✅ 피드의 스크롤/입력 상태가 유지되는가?
```

**Q1. 인터셉팅 라우트란?**

> 사용자가 내부 링크를 클릭할 때(Soft Navigation) 원래 목적지 대신 현재 화면 위에 가로챈 UI를 보여주는 기법이다. URL은 목적지로 정확히 변경되지만 화면 전환은 일어나지 않는다. 새로고침이나 외부 접속(Hard Navigation) 시에는 가로채기가 무시되고 원본 페이지가 전체 화면으로 렌더링된다. 병렬 라우트(@modal)와 결합하면 인스타그램 스타일의 모달 UX를 구현할 수 있다.

**Q2. 모달에서 router.back()과 router.push("/feed")의 차이?**

> `router.back()`은 히스토리 스택에서 현재 항목을 제거(pop)하여 이전 페이지로 돌아간다. Soft Navigation으로 모달을 열었을 때 스택에 [피드]→[모달]이 있으므로 back()으로 피드로 깔끔하게 돌아갈 수 있다. `router.push("/feed")`는 스택에 새 항목을 추가(push)하여 [피드]→[모달]→[피드]가 되어, 뒤로 가기 시 모달이 다시 나타나는 무한 루프 버그가 발생한다.

│  History Stack:                                  │

│                                                │
