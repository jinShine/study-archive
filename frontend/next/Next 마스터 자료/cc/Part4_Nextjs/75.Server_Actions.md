## 0. 웹 폼 전송의 3세대 진화사 — Server Actions의 뿌리 🟡

> 💡 기술의 진정한 가치를 이해하려면, **왜 이 기술이 탄생했는지** 역사를 알아야 한다.

### 1세대: 1990년대 — 직관적이지만 투박한 HTML Form

```html
<!-- 자바스크립트 0줄! 브라우저가 알아서 처리! -->
<form action="/api/save-user" method="POST">
  <input type="text" name="username" />
  <button type="submit">저장</button>
</form>

<!-- 버튼 클릭 → 브라우저가 input 값을 모아서 → /api/save-user로 POST 전송! -->
```

```
장점: 너무나 직관적! action에 목적지, name에 데이터. 끝!
치명적 단점: 제출할 때마다 화면이 하얗게 깜빡! (Full Page Reload!)
            입력 중이던 다른 상태(스크롤, 입력값)가 전부 사라짐!
```

### 2세대: 2010년대 — SPA & AJAX (깜빡임 해결, 복잡성 폭발)

```tsx
// UX는 좋아졌지만... 코드량이 폭발!
async function handleSubmit(event) {
  event.preventDefault();  // 1. 브라우저 기본 동작 막기!

  const formData = {       // 2. 데이터를 수동으로 수집!
    username: event.target.username.value,
  };

  const res = await fetch('/api/save-user', {   // 3. API URL 기억!
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },  // 4. 헤더 설정!
    body: JSON.stringify(formData),              // 5. JSON 변환!
  });

  if (!res.ok) throw new Error('실패');          // 6. 에러 처리!
  const data = await res.json();                 // 7. 응답 파싱!
  setResult(data);                               // 8. 상태 업데이트!
}
```

```
장점: 깜빡임 없는 부드러운 UX!
단점: "이름 하나 저장"하려는데 코드가 8줄!
     → 본질적 비즈니스 로직(DB 저장)보다 통신 보일러플레이트가 더 많음!
     → 주객전도 현상!
```

### 3세대: 2024년~ — Server Actions (RPC의 부활!)

```tsx
// 📄 actions.ts (서버)
'use server';
export async function saveUser(formData: FormData) {
  const username = formData.get('username') as string;
  await db.user.create({ name: username });
}

// 📄 page.tsx (클라이언트)
import { saveUser } from './actions';

export default function Page() {
  return (
    <form action={saveUser}>
      <input type="text" name="username" />
      <button type="submit">저장</button>
    </form>
  );
}
// fetch 코드 0줄! URL 설계 0줄! JSON 변환 0줄!
// 1990년대의 직관성 + 2010년대의 부드러움 + 2024년의 타입 안전성!
```

```
진화의 핵심:
  1990년대: 직관적이지만 깜빡임 (HTML Form)
  2010년대: 부드럽지만 복잡함 (SPA + fetch)
  2024년대: 직관적이면서 부드러움! (Server Actions = RPC의 부활!)

가장 첨단의 기술이 가장 근본적인 형태(HTML Form)로 돌아왔다!
복잡한 네트워크 통신을 프레임워크가 완벽히 추상화!
```

### RPC(Remote Procedure Call)란?

```
RPC = "원격 서버의 함수를 마치 로컬 함수처럼 호출하는 기술"

  클라이언트: saveUser("Alice")  ← 그냥 함수 호출처럼 보임!
       ↓
  [숨겨진 네트워크 통신] ← 프레임워크가 알아서!
       ↓
  서버: 실제 saveUser("Alice") 실행!
       ↓
  [결과 반환]
       ↓
  클라이언트: 화면 업데이트!

Server Actions = 웹 버전의 RPC!
  'use server' 함수를 클라이언트에서 import하면
  → 프레임워크가 숨겨진 API 엔드포인트를 자동 생성!
  → 개발자는 함수 호출만! 나머지는 프레임워크가 전부!
```

### API Route vs Server Actions 비교표

```
┌──────────────────┬──────────────────┬──────────────────┐
│                  │ Route Handler     │ Server Actions   │
├──────────────────┼──────────────────┼──────────────────┤
│ 엔드포인트 설계  │ 수동 (/api/users) │ 자동 (해시 ID)    │
│ 호출 방식        │ fetch(url)        │ 함수 import!      │
│ 데이터 변환      │ JSON.stringify    │ 자동 직렬화!       │
│ 타입 안전성      │ 수동 공유         │ End-to-End 자동!  │
│ 캐시 연동        │ 수동              │ revalidate 내장!   │
│ 코드량          │ 많음              │ 극소!              │
├──────────────────┼──────────────────┼──────────────────┤
│ 적합한 상황      │ 외부 API 제공     │ 내부 폼/CRUD ⭐   │
│                  │ (모바일, 서드파티) │ (내 앱 전용)      │
└──────────────────┴──────────────────┴──────────────────┘
```

## 1. Server Actions — API Route의 종말! 🔴

> 💡 **비유: 전화 주문 → 직접 방문**
> Route Handler = 전화(fetch)로 주문 (URL 외우고, JSON 포장하고...)
> Server Actions = 주방에 직접 가서 "이거 만들어주세요!" (함수 호출 한 줄!)

```tsx
// ❌ 과거: Route Handler + fetch (보일러플레이트!)
async function handleSubmit(formData) {
  const res = await fetch("/api/products", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ name: formData.name, price: formData.price }),
  });
  if (!res.ok) throw new Error("실패");
  const data = await res.json();
}

// ✅ 현재: Server Actions (함수 호출 한 줄!)
// actions.ts
"use server";
export async function createProduct(formData: FormData) {
  const name = formData.get("name") as string;
  const price = Number(formData.get("price"));
  await db.product.create({ data: { name, price } });
}

// page.tsx
import { createProduct } from "@/actions";
<form action={createProduct}>
  <input name="name" />
  <input name="price" type="number" />
  <button type="submit">등록</button>
</form>
// fetch 코드 0줄! URL 설계 0줄! JSON 변환 0줄!
```

## 2. 'use server' 지시어 🔴

```tsx
// 방법 1: 파일 전체를 서버 전용으로 (추천!)
// actions.ts
"use server";  // 파일 최상단! 모든 export 함수가 서버 액션!

export async function createProduct(formData: FormData) { }
export async function deleteProduct(id: number) { }
export async function updateProduct(id: number, data: Partial<Product>) { }

// 방법 2: 함수 내부에 선언 (서버 컴포넌트 안에서만!)
export default async function Page() {
  async function handleSubmit(formData: FormData) {
    "use server";  // 이 함수만 서버 액션!
    await db.create({ data: { name: formData.get("name") } });
  }
  return <form action={handleSubmit}>...</form>;
}
```

### 'use server'가 하는 일 (Under the Hood)

```
빌드 타임:
  1. 'use server' 함수 발견!
  2. 고유 해시 ID 부여: $action_abc123
  3. 숨겨진 API 엔드포인트 자동 생성!

런타임:
  1. 사용자가 폼 전송 버튼 클릭
  2. React가 브라우저의 기본 POST 요청을 가로챔
  3. 내부적으로 fetch → POST 요청 + 헤더에 Next-Action: $action_abc123
  4. 서버가 해당 ID의 함수를 찾아서 실행!
  5. 결과(RSC Payload)를 브라우저로 반환

→ 개발자가 작성하는 코드: <form action={함수}> 한 줄!
→ 프레임워크가 하는 일: URL 생성, fetch, 직렬화, 통신 전부!
```

## 3. FormData에서 데이터 추출 🔴

```tsx
"use server";

export async function createProduct(formData: FormData) {
  // FormData.get(): input의 name 속성으로 값 추출!
  const name = formData.get("name") as string;
  const price = Number(formData.get("price"));
  const description = formData.get("description") as string;

  // 유효성 검사
  if (!name || name.length < 2) {
    throw new Error("상품명은 2자 이상!");
  }
  if (!price || price <= 0) {
    throw new Error("가격은 양수!");
  }

  // DB 저장
  await db.product.create({
    data: { name, price, description },
  });
}
```

```html
<!-- input의 name이 FormData의 키가 됨! -->
<input name="name" />         → formData.get("name")
<input name="price" />        → formData.get("price")
<textarea name="description" /> → formData.get("description")
```

## 4. 클라이언트 컴포넌트에서 Server Action 호출 🔴

```tsx
// actions.ts
"use server";
export async function toggleLike(productId: number) {
  await db.like.toggle({ productId, userId: getCurrentUserId() });
  revalidatePath(`/products/${productId}`);
}

// LikeButton.tsx ('use client')
"use client";
import { toggleLike } from "@/actions";

export default function LikeButton({ productId }: { productId: number }) {
  return (
    // form action으로도 가능!
    <form action={toggleLike.bind(null, productId)}>
      <button type="submit">좋아요</button>
    </form>

    // 또는 onClick에서 직접 호출!
    // <button onClick={() => toggleLike(productId)}>좋아요</button>
  );
}
```

> 💡 **bind 패턴**: `action={fn.bind(null, 추가인자)}`로 FormData 외의 인자를 전달!

## 5. Server Actions의 5대 장점 🔴

```
1. 보일러플레이트 제거: fetch/URL/JSON 코드 0줄!
2. 타입 안전성: 서버 함수의 타입을 클라이언트가 End-to-End 공유!
3. 보안: 서버 코드가 브라우저에 노출 안 됨!
4. 캐시 연동: revalidatePath/Tag로 UI 자동 갱신!
5. Progressive Enhancement: JS 로딩 전에도 폼 전송 작동!
```


## 면접 대비

### 🔴 필수 질문

**Q1. Server Actions이란?**

> Next.js에서 `'use server'` 지시어로 선언한 서버 전용 비동기 함수다. `<form action={서버함수}>` 형태로 사용하면, 프레임워크가 빌드 타임에 숨겨진 API 엔드포인트를 자동 생성하고, 폼 전송 시 내부적으로 POST 요청을 보내 서버 함수를 실행한다. 개발자는 fetch/URL/JSON 코드 없이 함수 호출만으로 서버 로직을 실행할 수 있으며, 이는 RPC(Remote Procedure Call) 패턴의 현대적 구현이다.

**Q2. Server Actions의 장점?**

> 첫째, API 엔드포인트(route.ts)를 만들 필요가 없어 보일러플레이트가 제거된다. 둘째, TypeScript로 서버 함수의 매개변수/반환 타입을 클라이언트가 End-to-End로 공유하여 타입 안전성이 보장된다. 셋째, `revalidatePath`/`revalidateTag`로 mutation 후 UI 캐시를 자동 갱신한다. 넷째, Progressive Enhancement로 JS 로딩 전에도 HTML Form 기본 동작으로 전송이 가능하다.
