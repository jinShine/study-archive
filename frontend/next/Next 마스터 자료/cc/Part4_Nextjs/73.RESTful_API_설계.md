## 1. REST란? — 시스템 간의 대화 규칙 🔴

> 💡 **비유: 관공서 신청서**
> URL = 어떤 자원(resource) → 명사! `/products`, `/users`
> HTTP Method = 무엇을 할지 → 동사! GET, POST, PATCH, DELETE

```
RESTful API의 완벽한 문장 구조:

GET    /products        → "상품 목록을 조회하라"
GET    /products/1      → "1번 상품을 조회하라"
POST   /products        → "새 상품을 생성하라"
PATCH  /products/1      → "1번 상품을 부분 수정하라"
PUT    /products/1      → "1번 상품을 전체 교체하라"
DELETE /products/1      → "1번 상품을 삭제하라"

❌ 안티 패턴:
  GET /getProducts       → URL에 동사 넣지 마!
  POST /createProduct    → Method가 동사 역할!
  GET /deleteProduct/1   → GET으로 삭제? 위험!
```

## 2. HTTP 상태 코드 🔴

```
✅ 2xx 성공
  200 OK           → 조회/수정 성공
  201 Created      → 생성 성공! (POST 응답)
  204 No Content   → 삭제 성공! (본문 없음)

🔀 3xx 리다이렉트
  301 Moved        → 영구 이동
  302 Found        → 임시 이동

❌ 4xx 클라이언트 오류
  400 Bad Request    → 잘못된 요청 (유효성 검사 실패)
  401 Unauthorized   → 인증 안 됨 (로그인 필요!) ⚠️ 이름 주의!
  403 Forbidden      → 인가 안 됨 (권한 없음!)
  404 Not Found      → 자원 없음
  409 Conflict       → 충돌 (이미 존재하는 데이터)
  422 Unprocessable  → 문법은 맞지만 처리 불가 (비즈니스 규칙 위반)

💥 5xx 서버 오류
  500 Internal       → 서버 에러 (코드 버그!)
  502 Bad Gateway    → 중간 서버 에러
  503 Unavailable    → 서버 점검 중
```

> ⚠️ **면접 필수**: 401(인증=Authentication 실패)과 403(인가=Authorization 실패)의 차이!
> 401: "누구세요? 로그인부터!" / 403: "누군진 아는데, 권한이 없어!"

## 3. JSON — API의 공용어 🔴

```json
{
  "id": 1,
  "name": "초경량 노트북",
  "price": 1200000,
  "inStock": true,
  "tags": ["electronics", "laptop"],
  "specs": {
    "weight": "1.2kg",
    "screen": "14inch"
  }
}
```

> ⚠️ JSON 문법: 키는 큰따옴표 필수! / 마지막 쉼표 금지! / 주석 불가!

## 4. PATCH vs PUT 🟡

```
PATCH: 일부만 수정 (실무에서 더 많이 사용!)
  PATCH /products/1
  Body: { "price": 1100000 }  ← 가격만 변경!

PUT: 전체를 교체
  PUT /products/1
  Body: { "name": "노트북", "price": 1100000, "inStock": true, ... }
  ← 보내지 않은 필드는 null/삭제될 수 있음!

실무: PATCH를 기본으로! PUT은 "전체 덮어쓰기"가 명확할 때만.
```

## 5. 실무 API 응답 설계 🟡

```ts
// 성공 응답 표준
{
  "success": true,
  "data": { "id": 1, "name": "노트북" },
  "message": "상품이 생성되었습니다"
}

// 에러 응답 표준
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "상품명은 2자 이상이어야 합니다",
    "details": {
      "name": ["2자 이상 입력하세요"],
      "price": ["0보다 커야 합니다"]
    }
  }
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. RESTful API의 핵심 원칙?**

> URL은 자원을 나타내는 명사(`/products`, `/users/1`)로, HTTP 메서드(GET/POST/PATCH/DELETE)가 행위를 나타내는 동사 역할을 한다. URL에 동사를 넣지 않고(`/getProducts` X), 적절한 HTTP 상태 코드(200/201/400/404/500)를 반환하며, JSON 형식으로 데이터를 주고받는다.

**Q2. 401과 403의 차이?**

> 401 Unauthorized는 인증(Authentication) 실패로, 유효한 인증 정보(로그인/토큰)가 없는 상태다. 403 Forbidden은 인가(Authorization) 실패로, 인증은 되었지만 해당 자원에 접근할 권한이 없는 상태다.
