## 1. Search Params란? 🔴

> 💡 **비유: 택시에게 추가 요구사항**
> Path(경로) = "강남역으로 가주세요" (목적지)
> Search Params = "고속도로 타주시고, 빨리 가주세요" (옵션)

```
URL 해부:
/products?category=laptop&sort=price&page=2

/products          → 경로 (Path) — "어디에"
?category=laptop   → 검색 조건 — "노트북만"
&sort=price        → 정렬 — "가격순"
&page=2            → 페이지 — "2페이지"
```

### Next.js 15에서 searchParams는 Promise! 🔴

```tsx
// app/products/page.tsx
type Props = {
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>;
};

export default async function ProductsPage({ searchParams }: Props) {
  const params = await searchParams;  // ← await 필수!

  // 안전한 추출 (undefined 대비)
  const query = typeof params.q === "string" ? params.q : "";
  const sort = typeof params.sort === "string" ? params.sort : "";
  const page = typeof params.page === "string" ? parseInt(params.page) : 1;

  // 서버에서 바로 필터링! (클라이언트 JS 없이!)
  const products = allProducts
    .filter(p => p.name.includes(query))
    .sort((a, b) => sort === "asc" ? a.price - b.price : b.price - a.price);

  return (
    <div>
      <h1>상품 ({products.length}개)</h1>
      {/* 검색 결과 렌더링 */}
    </div>
  );
}
```

## 2. `<form>` action으로 서버 사이드 검색 — JS 0줄! 🔴

> 💡 **Web Platform First**: JavaScript 없이 HTML `<form>`의 기본 동작으로 검색!
> 브라우저가 알아서 input의 `name` 값을 URL의 쿼리 스트링으로 변환!

```tsx
// components/Search.tsx (서버 컴포넌트! 'use client' 없음!)
export default function Search({ initialQuery }: { initialQuery?: string }) {
  return (
    <form action="/products" className="flex gap-2">
      {/* name="q" → URL에 ?q=입력값 으로 자동 변환! */}
      <input
        name="q"
        defaultValue={initialQuery}
        placeholder="검색어 입력..."
        className="border rounded px-3 py-2 flex-1"
      />
      <button type="submit" className="bg-blue-600 text-white px-4 py-2 rounded">
        검색
      </button>
    </form>
  );
}

// 동작 흐름:
// 1. 사용자가 "노트북" 입력 후 엔터 또는 검색 버튼
// 2. 브라우저가 /products?q=노트북 으로 GET 요청 (JS 없이!)
// 3. 서버 컴포넌트가 searchParams에서 q="노트북" 추출
// 4. 서버에서 필터링 → 완성된 HTML 반환
// 5. 새로고침해도 검색어 유지! (URL에 있으니까!)
// 6. 공유 가능! (링크에 검색어 포함!)
```

### 검색어가 URL에 있으므로:

```
✅ 새로고침해도 검색 결과 유지!
✅ 뒤로 가기/앞으로 가기 작동!
✅ 링크 공유 시 같은 검색 결과!
✅ JavaScript가 로드되기 전에도 검색 작동!
✅ SEO: 검색 결과 페이지도 크롤링 가능!
```

## 3. 정렬 + 검색 조건 누적 🟡

```tsx
// 검색어를 유지하면서 정렬 버튼 추가
import Link from "next/link";

export default async function ProductsPage({ searchParams }: Props) {
  const params = await searchParams;
  const query = typeof params.q === "string" ? params.q : "";
  const sort = typeof params.sort === "string" ? params.sort : "";

  return (
    <div>
      <Search initialQuery={query} />

      {/* 정렬 버튼: 기존 검색어(q)를 유지하면서 sort만 변경! */}
      <div className="flex gap-2 my-4">
        <Link
          href={{ pathname: "/products", query: { q: query, sort: "asc" } }}
          className="px-3 py-1 bg-gray-200 rounded"
        >
          가격 낮은순
        </Link>
        <Link
          href={{ pathname: "/products", query: { q: query, sort: "desc" } }}
          className="px-3 py-1 bg-gray-200 rounded"
        >
          가격 높은순
        </Link>
        <Link href="/products" className="px-3 py-1 bg-gray-100 rounded">
          초기화
        </Link>
      </div>

      {/* 상품 목록 */}
    </div>
  );
}
```

> ⚠️ **핵심**: `href={{ query: { q: query, sort: "asc" } }}` 패턴!
> `href="/products?sort=asc"`라고 쓰면 검색어(q)가 사라진다!
> 기존 조건을 **객체에 함께** 넣어야 누적됨!

## 4. defaultValue의 중요성 🔴

```tsx
// ❌ value + onChange (제어 컴포넌트) → 서버 컴포넌트에서 사용 불가!
<input value={query} onChange={(e) => setQuery(e.target.value)} />
// useState는 서버 컴포넌트에서 못 씀!

// ✅ defaultValue (비제어) → 서버 컴포넌트에서 OK!
<input name="q" defaultValue={query} />
// → 서버가 URL에서 파악한 검색어를 초기값으로 채워줌
// → 사용자가 수정 가능 (React가 통제하지 않음)
// → 페이지 새로고침 후에도 검색창에 검색어가 남아있음!
```

## 5. 실습

```
과제: 서버 사이드 검색 + 정렬 + 필터링

6. 조건 변경 시 다른 조건 유지!
7. "3개 상품 표시 중 (전체 10개)" 카운트
8. 결과 없으면 "검색 결과가 없습니다" + 필터 초기화 링크
```

## 면접 대비

### 🔴 필수 질문

**Q1. Path Parameter와 Search Parameter의 차이? 언제 각각 쓰나요?**

> Path Parameter(`/products/123`)는 특정 자원을 식별할 때 사용한다. "정확히 123번 상품"처럼 대상 자체를 지정한다. Search Parameter(`?sort=price&page=2`)는 자원을 필터링/정렬/페이징 등 옵션으로 가공할 때 사용한다. RESTful 설계에서 Path는 "무엇을(What)", Query는 "어떻게(How)"에 해당한다.

**Q2. 검색 기능을 URL 기반으로 구현해야 하는 이유?**

> 검색어/필터를 useState로만 관리하면 새로고침 시 초기화, 뒤로 가기 미작동, 링크 공유 불가라는 3대 UX 문제가 발생한다. URL searchParams에 반영하면 새로고침해도 유지, 브라우저 히스토리 작동, 링크 공유 시 동일한 결과를 보장한다. Next.js의 서버 컴포넌트에서는 HTML `<form>`의 기본 동작으로 JS 없이도 검색이 작동하여 접근성과 성능도 향상된다.
