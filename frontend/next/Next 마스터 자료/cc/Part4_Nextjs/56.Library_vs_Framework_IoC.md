## 1. React만으로는 왜 부족한가? 🔴

> 💡 **비유: 레고 vs 풀옵션 모델하우스**
> React(Library) = 레고 블록만 줌. 설계도도, 시공도 직접 해야 함
> Next.js(Framework) = 벽, 배관, 전기까지 다 갖춘 모델하우스. 인테리어만 하면 됨!

### React만으로 프로젝트를 시작하면 직접 해야 하는 것들

```
React만 있을 때 개발자가 직접 선택하고 설치해야 하는 것:
  ❌ 라우팅 → react-router 설치 + 설정
  ❌ 서버 사이드 렌더링(SSR) → 직접 구현 (매우 복잡!)
  ❌ API 라우트 → Express/Koa 별도 서버 구축
  ❌ 코드 스플리팅 → Webpack/Vite 설정
  ❌ 이미지 최적화 → 별도 라이브러리
  ❌ SEO → Helmet 라이브러리 + SSR
  ❌ 환경 변수 → dotenv 설정
  ❌ TypeScript → 수동 설정
  ❌ CSS → 별도 도구 설정
  ❌ 빌드/배포 → CI/CD 직접 구축

Next.js는 이 모든 것을 "관습(Convention)"으로 해결:
  ✅ 라우팅 → 폴더 = 주소 (파일 시스템 라우팅)
  ✅ SSR/SSG/ISR → 프레임워크가 자동
  ✅ API → route.ts 파일 하나로
  ✅ 코드 스플리팅 → 페이지 단위 자동
  ✅ 이미지 → next/image 자동 최적화
  ✅ SEO → generateMetadata
  ✅ 환경 변수 → .env 자동 로드
  ✅ TypeScript → create-next-app에서 선택
  ✅ CSS → Tailwind 기본 지원
  ✅ 배포 → Vercel 원클릭
```

## 2. 제어의 역전(IoC) — 프레임워크의 핵심 원리 🔴

> 💡 **비유: 셀프 건축 vs 아파트 입주**
> Library: 내가 도구를 사서 내가 집을 짓는다 (내가 주도권)
> Framework: 이미 지어진 건물에 들어가서 인테리어만 한다 (프레임워크가 주도권)

```
Library (React):
  "내 코드"가 React를 호출한다
  → 나: "React야, 이 컴포넌트를 렌더링해줘"
  → 주도권: 나

Framework (Next.js):
  "Next.js"가 내 코드를 호출한다
  → Next.js: "page.tsx 파일 있네? 이걸 렌더링해야겠다!"
  → 주도권: Next.js

이것이 제어의 역전(Inversion of Control)!
```

```
React (Library):
  개발자 → "BrowserRouter 설정해야지"
          → "Route 컴포넌트 만들어야지"
          → "이 URL에 이 컴포넌트 연결해야지"

Next.js (Framework):
  개발자 → app/about/page.tsx 파일을 만든다
  Next.js → "아, /about 경로에 이 파일을 보여줘야 하는구나!" (자동!)
```

### Convention over Configuration (설정보다 관습) 🔴

```
파일 이름이 곧 규칙이다:

page.tsx     → 이 경로의 화면 (필수!)
layout.tsx   → 공통 레이아웃 (감싸는 껍데기)
loading.tsx  → 로딩 UI (자동 Suspense)
error.tsx    → 에러 UI (자동 Error Boundary)
not-found.tsx → 404 UI
route.ts     → API 엔드포인트
middleware.ts → 요청 가로채기

폴더 이름이 곧 URL이다:
  app/about/page.tsx → /about
  app/products/[id]/page.tsx → /products/123
  app/(marketing)/page.tsx → / (그룹핑, URL에 안 나옴)
```

## 3. Next.js 프로젝트 시작 🔴

```bash
npx create-next-app@latest my-app

# 설정 마법사 (2026년 기준 추천 답변):
# ✅ TypeScript
# ✅ ESLint
# ✅ Tailwind CSS
# ✅ src/ directory
# ✅ App Router (중요!)
# ✅ Turbopack
# ❌ customize import alias (기본 @/* 사용)

cd my-app
npm run dev
# → http://localhost:3000
```

### 프로젝트 구조

```
my-app/
├── src/
│   └── app/                 ← App Router의 심장!
│       ├── layout.tsx       ← 루트 레이아웃 (html, body 필수!)
│       ├── page.tsx         ← 메인 페이지 (/)
│       ├── globals.css      ← 전역 CSS
│       └── favicon.ico
├── public/                  ← 정적 파일 (이미지, 폰트)
├── next.config.ts           ← Next.js 설정
├── tailwind.config.ts       ← Tailwind 설정
├── tsconfig.json            ← TypeScript 설정
└── package.json
```

### 루트 레이아웃 — 모든 페이지의 골격

```tsx
// src/app/layout.tsx — ⚠️ 필수! html과 body 태그는 여기에만!
import type { Metadata } from "next";
import "./globals.css";

export const metadata: Metadata = {
  title: "나의 웹사이트",
  description: "Next.js로 만든 멋진 사이트",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="ko">
      <body>
        <header>공통 네비게이션</header>
        <main>{children}</main>
        {/* children = 현재 경로의 page.tsx가 들어옴! */}
        <footer>공통 푸터</footer>
      </body>
    </html>
  );
}
```

### 첫 페이지

```tsx
// src/app/page.tsx — / 경로
export default function HomePage() {
  return (
    <div>
      <h1>안녕, Next.js!</h1>
      <p>이것이 메인 페이지입니다.</p>
    </div>
  );
}
```

## 4. App Router vs Pages Router 🟡

```
Pages Router (옛날 방식 — 레거시):
  pages/index.tsx → /
  pages/about.tsx → /about
  getServerSideProps, getStaticProps 등 특수 함수 사용
  ❌ 2026년 기준 새 프로젝트에서는 사용하지 않음!

App Router (현재 표준 — 2023년~):
  app/page.tsx → /
  app/about/page.tsx → /about
  서버 컴포넌트가 기본!
  layout.tsx, loading.tsx, error.tsx 등 Convention 파일
  ✅ 이 커리큘럼에서 다루는 방식!
```

> ⚠️ 면접에서 "Pages Router도 아시나요?"라고 물어볼 수 있다.
> "App Router를 주로 사용하지만, Pages Router의 getServerSideProps/getStaticProps도 이해하고 있습니다"
> 정도면 충분하다. 새 프로젝트는 100% App Router!

## 5. 개발 서버 명령어

```bash
npm run dev      # 개발 서버 (Turbopack, Hot Reload)
npm run build    # 프로덕션 빌드 (최적화 + 정적 생성)
npm run start    # 프로덕션 서버 실행
npm run lint     # ESLint 검사
```

> 💡 **캐싱 테스트는 반드시 `build` → `start`로!**
> 개발 모드(`dev`)에서는 캐싱이 다르게 동작한다.

## 면접 대비

### 🔴 필수 질문

**Q1. Library(React)와 Framework(Next.js)의 차이?**

> Library는 개발자가 필요할 때 호출하는 도구이고, Framework은 개발자의 코드를 프레임워크가 호출하는 구조다. 이를 제어의 역전(IoC)이라 한다. React는 UI 렌더링만 제공하므로 라우팅, SSR, 최적화 등을 직접 구성해야 한다. Next.js는 파일 시스템 라우팅, 서버 컴포넌트, 이미지 최적화, SEO 등을 관습(Convention)으로 제공하여 설정 없이 즉시 사용할 수 있다.

**Q2. Next.js의 App Router에서 Convention 파일들의 역할?**

> `page.tsx`는 해당 경로의 UI, `layout.tsx`는 공통 레이아웃(상태 유지), `loading.tsx`는 로딩 UI(자동 Suspense), `error.tsx`는 에러 UI(자동 Error Boundary), `not-found.tsx`는 404 UI, `route.ts`는 API 엔드포인트다. 파일 이름이 곧 역할을 정의하는 것이 "설정보다 관습" 원칙이다.

**Q3. 왜 Next.js를 선택하나요?**

> React만으로는 SSR, 라우팅, SEO, 이미지 최적화 등을 직접 구성해야 하는 부담이 크다. Next.js는 이를 프레임워크 레벨에서 제공하며, 특히 서버 컴포넌트(RSC)로 Zero Bundle Size, Server Actions로 API 없는 데이터 변경, ISR로 성능과 최신성의 균형 등 React만으로는 불가능한 기능들을 지원한다. 2026년 기준 React 프로젝트의 사실상 표준 프레임워크다.
