## 1. App Router의 절대 규칙 🔴

### page.tsx — "이 경로에 화면이 있다!"

```
규칙: 폴더 안에 page.tsx가 있어야만 브라우저에서 접근 가능!
     page.tsx가 없으면 → 404!
     대소문자 중요! Page.tsx ❌ page.tsx ✅

app/
├── page.tsx              → / (메인)
├── about/
│   └── page.tsx          → /about
├── products/
│   ├── page.tsx          → /products (목록)
│   └── [id]/
│       └── page.tsx      → /products/123 (상세)
└── dashboard/
    ├── page.tsx          → /dashboard
    └── settings/
        └── page.tsx      → /dashboard/settings
```

### layout.tsx — "이 경로의 공통 껍데기"

```tsx
// app/layout.tsx — Root Layout (필수!)
// ⚠️ 반드시 <html>과 <body> 태그를 포함해야 함!
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko">
      <body>
        <nav>공통 네비게이션</nav>
        <main>{children}</main>  {/* 하위 page.tsx가 여기에! */}
        <footer>공통 푸터</footer>
      </body>
    </html>
  );
}

// app/dashboard/layout.tsx — 대시보드 전용 레이아웃
// ⚠️ 하위 레이아웃에는 <html>, <body> 넣지 마!
export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="flex">
      <aside className="w-64">사이드바</aside>
      <div className="flex-1">{children}</div>
    </div>
  );
}
```

### 레이아웃 중첩 (샌드위치 구조) 🔴

```
/dashboard/settings 접속 시 렌더링 순서:

1. Root Layout (app/layout.tsx)        ← 가장 바깥 (html, body)
   └── 2. Dashboard Layout (app/dashboard/layout.tsx)  ← 사이드바
        └── 3. Settings Page (app/dashboard/settings/page.tsx)  ← 실제 내용

실제 HTML:
<html>
  <body>
    <nav>공통 네비</nav>
    <main>
      <div class="flex">
        <aside>사이드바</aside>
        <div>설정 페이지 내용</div>  ← page.tsx
      </div>
    </main>
    <footer>공통 푸터</footer>
  </body>
</html>
```

### 레이아웃의 핵심 특성

```
✅ 페이지 전환 시 레이아웃은 리렌더링되지 않는다!
  → 사이드바의 검색어, 스크롤 위치, 토글 상태가 유지됨!
  → "레이아웃은 불변이다" = Next.js의 핵심 UX

✅ children: 자리 → 하위 layout.tsx 또는 page.tsx가 들어옴
```

## 2. Special Files (약속된 파일들) 🔴

```
app/
├── page.tsx         → 화면 UI (필수)
├── layout.tsx       → 공통 레이아웃 (children 필수)
├── loading.tsx      → 로딩 UI (자동 Suspense fallback!)
├── error.tsx        → 에러 UI (자동 Error Boundary! 'use client' 필수!)
├── not-found.tsx    → 404 UI
├── template.tsx     → layout과 비슷하지만 매 네비게이션마다 재생성
├── default.tsx      → 병렬 라우트의 기본 UI (Hard Navigation 방어)
└── route.ts         → API 엔드포인트 (page.tsx와 같은 폴더 금지!)
```

## 3. Colocation — 관련 파일을 함께 두기 🔴

> page.tsx가 아닌 파일은 **라우트로 인식되지 않는다!**
> 같은 폴더에 컴포넌트, 스타일, 유틸을 두어도 안전!

```
app/
└── dashboard/
    ├── page.tsx           → /dashboard (유일한 진입점)
    ├── layout.tsx         → 레이아웃
    ├── Chart.tsx          → ❌ 라우트 아님! (부품 컴포넌트)
    ├── StatsCard.tsx      → ❌ 라우트 아님! (부품 컴포넌트)
    ├── dashboard.css      → ❌ 라우트 아님! (전용 스타일)
    └── utils.ts           → ❌ 라우트 아님! (유틸 함수)

/dashboard/Chart 접속 → 404! (page.tsx가 없으므로)
```

> 💡 **Colocation의 이점**: 관련 파일이 한 폴더에 모여 있으니 찾기 쉽고 관리 편함!
> 과거처럼 `components/`, `styles/` 폴더에 흩어놓지 않아도 됨!

## 면접 대비

### 🔴 필수 질문

**Q1. Next.js App Router에서 page.tsx와 layout.tsx의 역할?**

> `page.tsx`는 해당 경로의 유일한 UI로, 이 파일이 없으면 그 경로는 404다. 반드시 소문자 `page.tsx`여야 한다. `layout.tsx`는 해당 폴더와 하위 모든 페이지를 감싸는 공통 UI 껍데기다. 페이지 전환 시 레이아웃은 리렌더링되지 않아 사이드바의 입력값이나 스크롤 위치가 유지된다. Root Layout만 `<html>`과 `<body>` 태그를 포함해야 한다.

**Q2. Colocation이란?**

> App Router에서는 page.tsx가 아닌 파일은 라우트로 인식되지 않으므로, 관련 컴포넌트/스타일/유틸을 해당 페이지와 같은 폴더에 안전하게 둘 수 있다. 이를 통해 관련 코드를 한 곳에 모아 유지보수성을 높이고, 과거처럼 `components/` 폴더에 모든 컴포넌트를 흩어놓는 것을 방지한다.
