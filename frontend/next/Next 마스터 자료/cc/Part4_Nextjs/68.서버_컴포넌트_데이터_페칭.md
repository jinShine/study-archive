## 1. 서버 컴포넌트 페칭의 혁명 🔴

> 💡 **비유: 주방에서 완성된 요리**
> CSR = 손님이 식재료를 받아서 직접 조리 (useEffect + fetch)
> SSR = 주방에서 요리 완성 후 배달 (서버 컴포넌트 페칭!)

```tsx
// ❌ CSR 방식 (순수 React): 복잡한 보일러플레이트!
"use client";
export default function OldWay() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    fetch("/api/products", { signal: controller.signal })
      .then(res => res.json())
      .then(data => setData(data))
      .catch(err => setError(err))
      .finally(() => setLoading(false));
    return () => controller.abort();
  }, []);

  if (loading) return <Spinner />;
  if (error) return <Error />;
  return <div>{data.name}</div>;
}

// ✅ 서버 컴포넌트 방식: async/await 한 줄!
export default async function NewWay() {
  const res = await fetch("https://api.example.com/products");
  if (!res.ok) throw new Error("상품 조회 실패");
  const products = await res.json();

  return (
    <ul>
      {products.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
  // useState 없음! useEffect 없음! loading 상태 없음!
  // → loading.tsx가 자동으로 처리!
  // → error.tsx가 자동으로 에러 잡아줌!
  // → 이 fetch 코드는 브라우저로 전송되지 않음! (보안!)
}
```

## 2. fetch 에러 처리 — error.tsx 연동! 🔴

```tsx
export default async function ProductsPage() {
  const res = await fetch("https://api.example.com/products");

  // ⚠️ fetch는 404/500에서 reject 안 함! res.ok로 체크 필수!
  if (!res.ok) {
    throw new Error(`상품 데이터를 가져올 수 없습니다. (HTTP ${res.status})`);
    // → throw하면 가장 가까운 error.tsx가 잡아서 에러 UI 표시!
    // → 앱 전체가 죽지 않고, 해당 세그먼트만 에러 UI로 교체!
  }

  const products = await res.json();
  return <ProductList products={products} />;
}
```

```
에러 흐름:
  fetch 실패 → throw Error → error.tsx가 잡아서 에러 UI 표시!
  레이아웃은 살아있음! 사용자가 네비게이션으로 다른 페이지 이동 가능!
  "다시 시도" 버튼(reset) → Page를 다시 렌더링 시도!
```

## 3. 서버 컴포넌트 페칭의 3대 장점 🔴

### 장점 1: Zero Bundle Size

```
서버 컴포넌트에서 import한 모든 것은 브라우저 번들에 포함 안 됨!

import { parse } from "marked";       // 마크다운 파서 (50KB)
import { format } from "date-fns";    // 날짜 포맷 (30KB)
import Prism from "prismjs";          // 코드 하이라이팅 (80KB)

// 이 모든 라이브러리 → 서버에서만 실행 → 브라우저 번들 0KB!
// CSR이었다면 50+30+80 = 160KB가 사용자에게 전송됐을 것!
```

### 장점 2: 보안

```
서버에서 직접 DB 접근 → API 키, DB URL이 브라우저에 노출 안 됨!

const data = await db.product.findMany();  // Prisma 쿼리
// → 이 코드가 브라우저 Network 탭에 안 찍힘!
// → 해커가 개발자 도구를 열어도 DB 쿼리를 볼 수 없음!

const res = await fetch("https://api.stripe.com/...", {
  headers: { Authorization: `Bearer ${process.env.STRIPE_SECRET}` },
});
// → STRIPE_SECRET이 브라우저에 노출 안 됨!
```

### 장점 3: 네트워크 효율

```
서버 → 서버 통신:
  Next.js 서버와 API 서버가 같은 데이터센터에 있으면
  지연 시간이 1ms 수준! (사용자 브라우저→서버는 100ms+)

Request Memoization:
  같은 렌더링 사이클에서 동일한 fetch를 여러 번 호출해도
  실제 네트워크 요청은 1번만! (자동 중복 제거!)
```

## 4. 직접 DB 접근 vs fetch API 🟡

```tsx
// 방법 1: 외부 API를 fetch로 호출
export default async function Page() {
  const res = await fetch("https://api.example.com/products");
  const products = await res.json();
  // → Next.js가 fetch를 확장하여 캐싱 옵션 제공!
}

// 방법 2: DB를 직접 접근 (Prisma 등 ORM)
export default async function Page() {
  const products = await db.product.findMany();
  // → fetch가 아니므로 Next.js의 fetch 캐싱이 적용 안 됨!
  // → Segment Config(export const revalidate)로 캐싱 제어!
}

// 방법 3: 데이터 접근 함수 분리 (실무 추천!)
// lib/data.ts
export async function getProducts() {
  return db.product.findMany({ orderBy: { createdAt: "desc" } });
}

// page.tsx
import { getProducts } from "@/lib/data";
export default async function Page() {
  const products = await getProducts();
  return <ProductList products={products} />;
}
```

## 5. Request Memoization — 중복 요청 자동 제거 🟡

```tsx
// generateMetadata와 Page 컴포넌트에서 같은 fetch를 2번 호출해도:

export async function generateMetadata({ params }: Props) {
  const product = await fetch(`/api/products/${id}`).then(r => r.json());
  return { title: product.name };  // 1번째 호출
}

export default async function Page({ params }: Props) {
  const product = await fetch(`/api/products/${id}`).then(r => r.json());
  return <h1>{product.name}</h1>;  // 2번째 호출
}

// 실제 네트워크 요청: 1번만! (2번째는 메모리에서 반환!)
// → 코드의 독립성을 유지하면서 성능도 확보!
// → Props Drilling 없이 각자 필요한 곳에서 fetch!
```

> 💡 **핵심**: 같은 URL + 같은 옵션의 fetch는 한 렌더링 사이클 내에서 자동으로 중복 제거!
> 이것은 `React.cache()`와 Next.js의 확장된 fetch가 함께 동작하는 결과.

5. console.log 확인
   - 서버 컴포넌트 내 console.log → 터미널에만 찍히는지 확인!

## 면접 대비

### 🔴 필수 질문

**Q1. Next.js 서버 컴포넌트에서 데이터를 가져오는 방법?**

> 서버 컴포넌트는 async 함수이므로 `await fetch()` 또는 Prisma 같은 ORM으로 직접 DB를 조회할 수 있다. useEffect/useState 없이 컴포넌트 본문에서 바로 비동기 작업이 가능하며, 로딩은 loading.tsx가, 에러는 error.tsx가 자동으로 처리한다. fetch 코드는 브라우저로 전송되지 않아 API 키 등의 보안이 보장된다.

**Q2. 서버 컴포넌트 페칭의 장점 3가지?**

> 첫째, Zero Bundle Size — 서버에서만 실행되는 코드와 라이브러리가 브라우저 번들에 포함되지 않아 앱이 가볍다. 둘째, 보안 — DB 쿼리, API 키, 환경변수가 브라우저에 노출되지 않는다. 셋째, 네트워크 효율 — 서버 간 통신은 지연이 극히 적고, Request Memoization으로 같은 요청의 중복이 자동 제거된다.
