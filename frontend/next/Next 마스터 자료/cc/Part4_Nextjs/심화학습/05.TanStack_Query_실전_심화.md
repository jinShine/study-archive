## 1. 캐시 데이터의 3대 상태 — Fresh, Stale, Inactive 🔴

> 💡 **비유: 식당의 음식 상태!**
> Fresh = 갓 만든 김이 모락모락 나는 요리 (서버에 안 물어봄!)
> Stale = 식어서 신선하지 않은 요리 (일단 주고 → 뒤에서 새로 만듦!)
> Inactive = 손님이 떠난 빈 테이블의 요리 (타이머 후 폐기!)

### Fresh (신선!) — 서버에 절대 안 물어봄!

```
상태: 서버 원본과 캐시가 100% 일치한다고 보증!
효과: 같은 데이터를 요청해도 → 네트워크 요청 0번! 캐시에서 즉시 반환!
기간: staleTime 동안 유지!

⚠️ staleTime 기본값 = 0 ← 이것 때문에 초보자가 혼란!
   데이터가 나오자마자 0초 만에 "상했다"고 판정!
   → 화면 이동할 때마다 백그라운드 리패칭 발생!
   → "캐싱 적용했는데 왜 계속 API 호출이 되지?" 의 원인!
```

### Stale (식었음!) — 일단 주고 → 뒤에서 새로 만듦!

```
상태: 데이터가 있지만 서버 원본이 변했을 수도 있음!
핵심: 사용자에게 빈 화면 대신 이전 데이터를 즉시 보여줌!
       동시에 백그라운드에서 서버에 새 데이터 요청! (Background Refetching!)
       새 데이터가 오면 → 화면을 슬쩍 교체! (사용자 눈치 못 챔!)

이것이 TanStack Query의 진가!
  → 로딩 스피너 0초!
  → 항상 최신 데이터!
```

### Inactive (비활성!) — 타이머 후 폐기!

```
상태: 이 데이터를 보던 모든 컴포넌트가 언마운트됨!
동작: gcTime 타이머 시작! (기본 5분)
      타이머 안에 다시 돌아오면 → 캐시에서 즉시! (뒤로 가기 즉시!)
      타이머 끝나면 → 메모리에서 완전 삭제! (메모리 누수 방지!)

실무 팁: gcTime을 길게 잡으면 (예: 24시간)
  → 게시판에서 나갔다 돌아와도 즉시 표시!
  → 하지만 메모리 사용량 주의!
```

### staleTime vs gcTime 설정

```tsx
// providers.tsx
const [queryClient] = useState(() => new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60,      // 1분간 Fresh! (서버 안 물어봄!)
      gcTime: 1000 * 60 * 10,    // 비활성 후 10분간 메모리 보관!
    },
  },
}));

// 개별 쿼리에서 덮어쓰기도 가능!
const { data } = useQuery({
  queryKey: ["product", id],
  queryFn: fetchProduct,
  staleTime: 1000 * 60 * 5,  // 상품 정보는 5분간 Fresh! (전역 1분 무시!)
});
```

### isPending vs isFetching 차이 🔴

```
isPending:
  캐시에 데이터가 전혀 없고 + 최초 요청 중!
  → 스켈레톤 UI 표시! (첫 방문!)

isFetching:
  백그라운드에서 데이터를 가져오는 중!
  → 이미 캐시된 데이터가 있을 수도 있음!
  → 화면에 이미 뭔가 보이는 상태에서 조용히 갱신!

실무 패턴:
  if (isPending) return <SkeletonUI />;     // 첫 로딩!
  if (isError) return <ErrorMessage />;      // 에러!
  return <div>{data.name}</div>;             // 성공! (isFetching 중이어도 표시!)
```

## 2. useMutation + invalidateQueries — 데이터 변경 후 즉시 갱신! 🔴

> 💡 **비유: 메뉴판 가격을 고친 뒤 → 모든 테이블에 새 메뉴판 배포!**

### useMutation 기본 구조

```tsx
"use client";
import { useMutation, useQueryClient } from "@tanstack/react-query";

export default function ProfileEditForm({ userId }: { userId: string }) {
  const queryClient = useQueryClient();  // 중앙 캐시 매니저 호출!

  const { mutate, isPending, isError, error, isSuccess } = useMutation({
    // mutationFn: 서버에 데이터를 보내는 함수!
    mutationFn: async (newName: string) => {
      const res = await fetch(`/api/users/${userId}`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ name: newName }),
      });
      if (!res.ok) throw new Error("수정 실패!");
      return res.json();
    },

    // 성공 시: 캐시 무효화! → 관련 데이터 자동 갱신!
    onSuccess: () => {
      queryClient.invalidateQueries({
        queryKey: ["user", userId],  // 이 키의 캐시를 "상했다"로 표시!
      });
      // → ["user", userId]를 보고 있는 모든 컴포넌트가 자동 리패치!
      // → 새로고침 없이 화면이 최신화!
    },
  });

  return (
    <form onSubmit={(e) => { e.preventDefault(); mutate("김철수"); }}>
      <button disabled={isPending}>
        {isPending ? "저장 중..." : "이름 변경"}
      </button>
      {isError && <p className="text-red-500">{error.message}</p>}
      {isSuccess && <p className="text-green-500">성공!</p>}
    </form>
  );
}
```

### invalidateQueries의 계층적 무효화!

```
쿼리 키를 배열로 계층화하면 → 상위 키 하나로 하위 전부 무효화!

설정:
  ['posts']                    ← 전체 게시글
  ['posts', 'list', 1]         ← 1페이지 목록
  ['posts', 'detail', 15]      ← 15번 상세

무효화:
  queryClient.invalidateQueries({ queryKey: ['posts'] });
  → 위의 3개 캐시가 전부 "상했다"로 표시!
  → 도서관의 "역사 코너" 전체를 새 책으로 교체하는 것!

더 정밀하게:
  queryClient.invalidateQueries({ queryKey: ['posts', 'list'] });
  → 목록 캐시만 무효화! 상세 페이지 캐시는 그대로!
```

## 3. Optimistic Update — 서버 응답 전에 화면부터 바꾸기! 🔴

> 💡 **비유: 주문이 성공할 거라고 낙관적으로 믿고, 테이블에 요리를 미리 올려두기!**
> 실패하면? → 다시 치운다! (롤백!)

### 3단계 라이프사이클: onMutate → onError → onSettled

```tsx
"use client";
import { useMutation, useQueryClient } from "@tanstack/react-query";

interface Todo {
  id: number;
  text: string;
  completed: boolean;
}

export default function TodoItem({ todo }: { todo: Todo }) {
  const queryClient = useQueryClient();

  const { mutate } = useMutation<Todo, Error, Todo, { previousTodos: Todo[] | undefined }>({
    // 제네릭 4개: <반환타입, 에러타입, 변수타입, 컨텍스트타입>

    mutationFn: async (updatedTodo: Todo) => {
      const res = await fetch(`/api/todos/${updatedTodo.id}`, {
        method: "PATCH",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(updatedTodo),
      });
      if (!res.ok) throw new Error("업데이트 실패!");
      return res.json();
    },

    // ─────────────────────────────────────────
    // Step 1: onMutate — 서버 요청 직전! (스냅샷 + 낙관적 업데이트!)
    // ─────────────────────────────────────────
    onMutate: async (newTodo: Todo) => {
      // 1-1. 진행 중인 리패칭을 취소! (낙관적 데이터와 충돌 방지!)
      await queryClient.cancelQueries({ queryKey: ["todos"] });

      // 1-2. 현재 상태를 타임캡슐에 저장! (롤백용!)
      const previousTodos = queryClient.getQueryData<Todo[]>(["todos"]);

      // 1-3. 서버 응답 전에 캐시를 즉시 변경! (화면이 바로 바뀜!)
      queryClient.setQueryData<Todo[]>(["todos"], (old) => {
        if (!old) return [newTodo];
        return old.map((t) => (t.id === newTodo.id ? newTodo : t));
      });

      // 1-4. 타임캡슐을 다음 단계로 전달!
      return { previousTodos };
    },

    // ─────────────────────────────────────────
    // Step 2: onError — 서버 에러 시! (롤백!)
    // ─────────────────────────────────────────
    onError: (err, newTodo, context) => {
      // 타임캡슐에서 이전 데이터를 꺼내 복원!
      if (context?.previousTodos) {
        queryClient.setQueryData(["todos"], context.previousTodos);
      }
      alert("실패! 원래 상태로 복구됩니다.");
    },

    // ─────────────────────────────────────────
    // Step 3: onSettled — 성공이든 실패든 항상 실행! (최종 정산!)
    // ─────────────────────────────────────────
    onSettled: () => {
      // 서버와 클라이언트를 100% 동기화!
      queryClient.invalidateQueries({ queryKey: ["todos"] });
    },
  });

  return (
    <li>
      <input
        type="checkbox"
        checked={todo.completed}
        onChange={() => mutate({ ...todo, completed: !todo.completed })}
      />
      <span className={todo.completed ? "line-through" : ""}>{todo.text}</span>
    </li>
  );
}
```

### 동작 흐름 시각화

```
사용자 클릭! (체크박스 토글)
  │
  ▼
onMutate 실행! (서버 요청 전!)
  ├─ 1. cancelQueries: 진행 중인 리패칭 취소!
  ├─ 2. getQueryData: 현재 상태 스냅샷 저장! (타임캡슐!)
  ├─ 3. setQueryData: 캐시를 즉시 변경! → 화면 0ms에 반영!
  └─ 4. return { previousTodos }: 타임캡슐 전달!
  │
  ▼
서버에 PATCH 요청 전송... (1~3초 걸릴 수 있음!)
  │
  ├── 🟢 성공! → onSettled 실행!
  │     └─ invalidateQueries: 서버와 최종 동기화!
  │
  └── 🔴 실패! → onError 실행!
        ├─ setQueryData(previousTodos): 원래 상태로 롤백!
        └─ alert("실패!")
        └─ onSettled도 실행!
              └─ invalidateQueries: 혹시 모를 불일치 정리!

핵심: 사용자는 1~3초의 서버 지연을 전혀 느끼지 못함!
      화면이 0ms에 반영되니까!
```

## 4. 서버 프리페치 + HydrationBoundary — SSR 데이터 동기화! 🟡

> 💡 **비유: 서버 주방에서 미리 요리를 완성해서 → 건조시켜 배달 → 브라우저에서 물 부어 복원!**

### 왜 필요한가?

```
일반 useQuery만 쓰면:
  1. 서버가 빈 HTML 전송
  2. 브라우저에서 JS 로드
  3. useQuery가 API 호출 (이때 로딩 스피너!)
  4. 데이터 도착 → 화면 렌더링
  → 사용자가 로딩 스피너를 봐야 함! 느림!

프리페치 + 하이드레이션:
  1. 서버에서 데이터를 미리 가져옴! (prefetchQuery)
  2. 데이터를 건조(dehydrate)해서 HTML에 포함!
  3. 브라우저가 HTML 수신 → 즉시 데이터 표시! (로딩 0초!)
  4. 하이드레이션: 건조 데이터를 useQuery 캐시에 주입!
  5. useQuery가 실행되지만 → 캐시에 이미 있으므로 API 호출 안 함!
  → 사용자에게 즉시 완성된 화면! + SEO 완벽!
```

### 구현 패턴

```tsx
// lib/get-query-client.ts — 서버용 안전한 QueryClient
import { QueryClient } from "@tanstack/react-query";
import { cache } from "react";

// cache()로 감싸서 한 요청 내에서 하나의 인스턴스만 사용!
// (다른 사용자의 데이터가 섞이는 "상태 누수" 방지!)
export const getQueryClient = cache(() => new QueryClient({
  defaultOptions: {
    queries: { staleTime: 60 * 1000 },  // 1분간 Fresh!
  },
}));
```

```tsx
// app/posts/page.tsx — 서버 컴포넌트 (데이터를 미리 가져옴!)
import { Suspense } from "react";
import { dehydrate, HydrationBoundary } from "@tanstack/react-query";
import { getQueryClient } from "@/lib/get-query-client";
import PostList from "./PostList";

async function PrefetchedPostList() {
  const queryClient = getQueryClient();

  // 서버에서 데이터를 미리 가져와 캐시에 저장!
  await queryClient.prefetchQuery({
    queryKey: ["posts"],
    queryFn: async () => {
      const res = await fetch("https://jsonplaceholder.typicode.com/posts");
      return res.json();
    },
  });

  return (
    // dehydrate: 캐시를 건조(직렬화)! → HTML에 포함!
    // HydrationBoundary: 브라우저에서 건조 데이터를 캐시에 복원!
    <HydrationBoundary state={dehydrate(queryClient)}>
      <PostList />
    </HydrationBoundary>
  );
}

export default function PostsPage() {
  return (
    <main>
      <h1>게시글</h1>
      {/* Suspense로 감싸서 하이드레이션 에러 방지! */}
      <Suspense fallback={<p>로딩 중...</p>}>
        <PrefetchedPostList />
      </Suspense>
    </main>
  );
}
```

```tsx
// app/posts/PostList.tsx — 클라이언트 컴포넌트 (캐시에서 즉시!)
"use client";
import { useQuery } from "@tanstack/react-query";

export default function PostList() {
  const { data: posts } = useQuery({
    queryKey: ["posts"],  // 서버에서 프리페치한 것과 같은 키!

      return res.json();
    },
    // → 서버에서 이미 데이터를 채워놨으므로 API 호출 안 함!
    // → 로딩 스피너 0초!
  });

  return (
    <ul>
      {posts?.slice(0, 5).map((post: any) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

### initialData + initialDataUpdatedAt 패턴 (대안)

```tsx
// 서버 컴포넌트에서 데이터를 가져와 Props로 전달!
export default async function Page() {
  const res = await fetch("https://api.example.com/posts");
  const posts = await res.json();
  const serverTime = Date.now();  // 서버 시간 기록!

  return (
    <Suspense fallback={<p>로딩...</p>}>
      <PostList initialData={posts} updatedAt={serverTime} />
    </Suspense>
  );
}

// 클라이언트 컴포넌트
"use client";
export default function PostList({ initialData, updatedAt }) {
  const { data } = useQuery({
    queryKey: ["posts"],
    queryFn: fetchPosts,
    initialData: initialData,           // 서버 데이터를 초기값으로!
    initialDataUpdatedAt: updatedAt,     // 서버 시간 동기화!
    staleTime: 1000 * 60 * 5,          // 5분간 Fresh!
  });
  // → initialDataUpdatedAt이 없으면
  //   "이 데이터가 언제 만들어졌는지 모르겠어!"
  //   → 즉시 백그라운드 리패칭! (서버 왕복 낭비!)
  // → initialDataUpdatedAt이 있으면
  //   "5분 전에 만들어졌구나! staleTime 남았으니 리패칭 안 해!"
}
```

## 5. useInfiniteQuery — 무한 스크롤 심화! 🟡

> 💡 **비유: 끝없이 이어지는 무한 리필 뷔페!**
> 접시가 비어갈 때 → 매니저가 새 접시를 자동으로 채워줌!

### 핵심 구조

```tsx
"use client";
import { useInfiniteQuery } from "@tanstack/react-query";
import { useEffect } from "react";
import { useInView } from "react-intersection-observer";

export default function InfinitePhotoGallery() {
  // 바닥 감지 센서! (화면 맨 밑에 부착!)
  const { ref, inView } = useInView({ delay: 300 });

  const {
    data,
    fetchNextPage,    // 다음 페이지 가져오기!
    hasNextPage,      // 다음 페이지가 있는지?
    isFetchingNextPage, // 다음 페이지를 가져오는 중인지?
  } = useInfiniteQuery({
    queryKey: ["photos", "infinite"],
    queryFn: ({ pageParam }) =>
      fetch(`https://picsum.photos/v2/list?page=${pageParam}&limit=9`)
        .then((res) => res.json()),

    // v5 필수! 시작 페이지 번호!
    initialPageParam: 1,

    // 다음 페이지 번호 계산!
    getNextPageParam: (lastPage, allPages, lastPageParam) => {
      // 9개가 꽉 찼으면 → 다음 페이지가 있다!
      return lastPage.length === 9 ? lastPageParam + 1 : undefined;
      // undefined를 반환하면 → "더 이상 없다!" → hasNextPage = false!
    },

    staleTime: 1000 * 60 * 5,  // 5분 캐시!

    // ⭐ 메모리 최적화! 최대 4페이지만 메모리에 유지!
    maxPages: 4,
    // → 오래전 스크롤한 위쪽 데이터는 메모리에서 해제!
    // → 스마트폰 메모리 폭발 방지!
  });

  // 바닥에 도달하면 → 다음 페이지 자동 요청!
  useEffect(() => {
    if (inView && hasNextPage && !isFetchingNextPage) {
      fetchNextPage();
    }
  }, [inView, hasNextPage, isFetchingNextPage, fetchNextPage]);

  // 2중 배열을 1차원으로 평탄화!
  const allPhotos = data?.pages.flatMap((page) => page) || [];

  return (
    <div>
      <div className="grid grid-cols-3 gap-4">
        {allPhotos.map((photo, index) => (
          <img
            key={`${photo.id}-${index}`}
            src={photo.download_url}
            alt={photo.author}
            className="w-full aspect-square object-cover rounded"
          />
        ))}
      </div>

      {/* 바닥 센서 */}
      <div ref={ref} className="h-20 flex items-center justify-center">
        {isFetchingNextPage
          ? <p>새로운 사진 로딩 중...</p>
          : hasNextPage
            ? <p>스크롤 내려서 더 보기</p>
            : <p>모든 사진을 봤습니다!</p>}
      </div>
    </div>
  );
}
```

### maxPages — 메모리 폭발 방지!

```
maxPages가 없으면:
  스크롤을 50번 하면 → 50페이지 × 9장 = 450장의 이미지가 DOM에!
  → 스마트폰 메모리 폭발! → 앱 뻗음!

maxPages: 4로 설정하면:
  항상 최근 4페이지(36장)만 메모리에 유지!
  → 오래전 위쪽 사진은 자동 해제!
  → 다시 위로 스크롤하면 → 다시 가져옴!
  → 양방향 무한 스크롤!
```

### 서버 프리페치 + 무한 스크롤

```tsx
// app/gallery/page.tsx — 서버에서 첫 페이지를 미리!
import { dehydrate, HydrationBoundary, QueryClient } from "@tanstack/react-query";
import { cache } from "react";
import InfinitePhotoGallery from "./InfinitePhotoGallery";

const getQueryClient = cache(() => new QueryClient());

export default async function GalleryPage() {
  const queryClient = getQueryClient();

  // 무한 스크롤의 첫 페이지를 서버에서 프리페치!
  await queryClient.prefetchInfiniteQuery({
    queryKey: ["photos", "infinite"],
    queryFn: ({ pageParam = 1 }) =>
      fetch(`https://picsum.photos/v2/list?page=${pageParam}&limit=9`)
        .then((res) => res.json()),
    initialPageParam: 1,
  });

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <InfinitePhotoGallery />
      {/* → 첫 9장은 로딩 0초! → 스크롤하면 나머지 자동 로드! */}
    </HydrationBoundary>
  );
}
```

### 🔴 필수 질문

**Q1. TanStack Query의 staleTime과 gcTime 차이?**

> `staleTime`은 데이터가 Fresh에서 Stale로 전환되는 시간으로, Fresh 동안은 동일 요청에 서버 호출을 하지 않는다. 기본값은 0이므로 명시적 설정이 필요하다. `gcTime`은 데이터가 Inactive(구독자 0명) 상태가 된 후 메모리에서 삭제되기까지의 유예 시간이다(기본 5분). 뒤로 가기 시 즉시 표시를 위해 gcTime을 길게 잡되, 메모리 누수를 고려해야 한다.

**Q2. Optimistic Update의 3단계를 설명하세요.**

> `onMutate`에서 서버 요청 전에 캐시를 즉시 변경하여 화면을 0ms에 반영하고, 롤백용 스냅샷을 저장한다. `onError`에서 서버 실패 시 스냅샷으로 캐시를 복원(롤백)한다. `onSettled`에서 성공/실패 무관하게 `invalidateQueries`로 서버와 최종 동기화한다. 이를 통해 네트워크 지연을 사용자가 인지하지 못하게 하면서도 데이터 정합성을 보장한다.

**Q3. 서버에서 prefetchQuery한 데이터를 클라이언트가 어떻게 받나요?**

> 서버 컴포넌트에서 `prefetchQuery`로 데이터를 캐시에 채운 후, `dehydrate(queryClient)`로 직렬화하여 `HydrationBoundary`의 state에 전달한다. 브라우저에서 `useQuery`가 동일한 queryKey로 실행될 때 이미 캐시에 데이터가 있으므로 API 호출 없이 즉시 렌더링된다. `initialDataUpdatedAt`으로 서버 시간을 전달하면 불필요한 백그라운드 리패칭도 방지한다.
