## 1. GitHub Actions에서 Playwright 실행 🔴

> 코드를 push할 때마다 자동으로 E2E 테스트 실행!

### 워크플로우 파일

```yaml
# .github/workflows/playwright.yml
name: Playwright Tests

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest

    steps:
      # 1. 코드 체크아웃
      - uses: actions/checkout@v4

      # 2. Node.js 설치
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      # 3. 의존성 설치
      - name: Install dependencies
        run: npm ci

      # 4. Playwright 브라우저 설치 (+ 시스템 의존성!)
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps

      # 5. Next.js 빌드 (프로덕션 환경 테스트!)
      - name: Build Next.js
        run: npm run build

      # 6. E2E 테스트 실행!
      - name: Run Playwright tests
        run: npx playwright test
        env:
          CI: true

      # 7. 실패 시 HTML 리포트를 아티팩트로 업로드!
      - uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

### 핵심 포인트 해설

```
npm ci (vs npm install):
  CI 환경에서는 npm ci를 사용!
  → package-lock.json 기준으로 정확한 버전 설치!
  → npm install보다 빠르고 재현성 보장!

install --with-deps:
  Ubuntu에 Chromium/Firefox/WebKit이 필요로 하는
  시스템 라이브러리(libgbm, libnss3 등)를 함께 설치!
  → 이거 빠뜨리면 CI에서 브라우저가 안 열림!

npm run build:
  dev 모드 대신 프로덕션 빌드로 테스트!
  → 실제 배포 환경과 동일한 조건! (캐싱 동작 등)
  → playwright.config.ts의 webServer를 "npm start"로 변경:

  webServer: {
    command: process.env.CI ? "npm start" : "npm run dev",
    url: "http://localhost:3000",
  }

upload-artifact:
  테스트가 실패하면 HTML 리포트를 GitHub에 업로드!
  → PR에서 리포트 다운로드해서 실패 원인 분석 가능!
  → retention-days: 30일간 보관!
```

## 2. 병렬 실행과 Sharding 🟡

### workers: 로컬 병렬 실행

```ts
// playwright.config.ts
export default defineConfig({
  // 로컬: CPU 코어 수에 맞게 자동! / CI: 안정성 위해 1개!
  workers: process.env.CI ? 1 : undefined,

  // 또는 명시적으로 지정
  workers: 4,  // 4개 테스트를 동시에!
});
```

### Sharding: CI에서 여러 머신으로 분산!

```yaml
# 테스트를 4개 머신에 분산 실행! (대규모 프로젝트!)
jobs:
  test:
    strategy:
      matrix:
        shardIndex: [1, 2, 3, 4]
        shardTotal: [4]
    steps:
      - name: Run Playwright tests
        run: npx playwright test --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}
```

```
100개 테스트를 4개 머신에 분산:
  머신 1: 테스트 1~25
  머신 2: 테스트 26~50
  머신 3: 테스트 51~75
  머신 4: 테스트 76~100
  → 총 시간: 25개 분량! (4배 빠름!) ⚡
```

## 3. 디버깅 도구 3종 세트 🔴

### 도구 1: UI Mode (가장 많이 사용!) ⭐

```bash
npx playwright test --ui
```

```
UI Mode 기능:
  ✅ 테스트 목록을 GUI로 보고 원하는 것만 클릭 실행!
  ✅ 각 스텝별 스크린샷 자동 캡처!
  ✅ DOM 스냅샷 확인 (그 시점의 HTML 구조!)
  ✅ 시간 여행: 슬라이더로 테스트 과정을 앞뒤로 탐색!
  ✅ 실시간 Watch 모드: 코드 수정하면 자동 재실행!

언제 사용?
  → 테스트를 새로 작성할 때!
  → 실패 원인을 빠르게 파악할 때!
  → 가장 자주 쓰는 도구! ⭐
```

### 도구 2: Codegen (테스트 자동 생성!)

```bash
npx playwright codegen http://localhost:3000
```

```
Codegen 기능:
  ✅ 실제 브라우저가 열리고, 마우스로 클릭하면 코드가 자동 생성!
  ✅ 입력, 클릭, 이동을 그대로 테스트 코드로 변환!
  ✅ 최적의 Locator를 자동 추천! (getByRole 우선!)
  ✅ 생성된 코드를 복사해서 .spec.ts에 붙여넣기만 하면 끝!

언제 사용?
  → 테스트 코드 작성이 처음일 때!
  → 복잡한 UI 흐름의 Locator를 빨리 찾고 싶을 때!
  → "이 요소를 어떻게 선택하지?" 고민될 때!
```

### 도구 3: Trace Viewer (실패 분석!)

```bash
# 먼저 trace가 켜진 상태에서 테스트 실행
npx playwright test --trace on

# trace 파일 열기
npx playwright show-trace test-results/trace.zip
```

```
Trace Viewer 기능:
  ✅ 실패한 테스트의 모든 단계를 타임라인으로!
  ✅ 각 액션 시점의 스크린샷!
  ✅ 네트워크 요청/응답 전체 로그!
  ✅ 콘솔 로그!
  ✅ DOM 스냅샷!

언제 사용?
  → CI에서 테스트가 실패했는데 로컬에서는 재현이 안 될 때!
  → "왜 이 클릭이 안 됐지?" 원인 분석!
  → playwright.config.ts에서 trace: "on-first-retry" 설정 권장!
```

## 4. 리포트 🟡

### HTML 리포트

```ts
// playwright.config.ts
export default defineConfig({
  reporter: "html",  // 기본! 가장 보기 좋음!
});
```

```bash
# 테스트 후 리포트 열기
npx playwright show-report

# 결과:
#   ✅ 통과: 초록색
#   ❌ 실패: 빨간색 + 스크린샷 + 에러 메시지
#   ⏭️ 건너뜀: 회색
#   🔄 재시도: 노란색
```

### 다중 리포터 (CI용)

```ts
reporter: [
  ["html"],                           // 시각적 리포트
  ["list"],                           // 터미널 출력
  ["github"],                         // PR에 인라인 코멘트!
  ["json", { outputFile: "results.json" }],  // JSON 데이터
],
```

## 5. 실무 베스트 프랙티스 🔴

### DO (해야 할 것) ✅

```
1. 선택자: getByRole 우선!
   ✅ page.getByRole("button", { name: "저장" })
   ❌ page.locator(".btn-primary")

2. 격리: 테스트끼리 독립적으로!
   ✅ 각 테스트가 자체 데이터를 생성/정리
   ❌ 테스트 A가 만든 데이터를 테스트 B가 의존

3. 대기: Auto-waiting 신뢰!
   ✅ await expect(el).toBeVisible()
   ❌ await page.waitForTimeout(3000)

4. 네이밍: 행동 + 결과로!
   ✅ test("빈 폼 제출 시 에러 메시지가 표시된다")
   ❌ test("test 1")

5. 인증: storageState로 재사용!
   ✅ 한 번 로그인 → 상태 저장 → 모든 테스트에서 재사용
   ❌ 매 테스트마다 로그인 페이지 방문

6. CI: 프로덕션 빌드로 테스트!
   ✅ npm run build → npm start
   ❌ npm run dev (캐싱 동작이 다름!)
```

### DON'T (하면 안 되는 것) ❌

```
1. ❌ waitForTimeout 하드코딩
   → await page.waitForTimeout(5000)
   → 느리고 불안정! Auto-waiting이나 waitForResponse 사용!

2. ❌ CSS 선택자 의존
   → page.locator(".MuiButton-root > span:nth-child(2)")
   → 스타일 변경하면 바로 깨짐! getByRole 사용!

3. ❌ 테스트 간 순서 의존
   → "테스트 A에서 만든 데이터를 테스트 B에서 삭제"
   → 병렬 실행 시 순서 보장 안 됨! 각자 독립!

4. ❌ 모든 것을 E2E로
   → "함수 하나의 로직도 E2E로 검증"
   → 유닛 테스트가 적합! E2E는 핵심 흐름만!

5. ❌ 스크린샷을 Git에 커밋
   → 이미지 파일이 repo를 비대하게 만듦!
   → .gitignore에 test-results/ 추가!
```

### 테스트 피라미드 균형

```
실무 권장 비율:

  E2E (Playwright):   10~20%  ← 핵심 비즈니스 흐름만!
    "회원가입 → 로그인 → 상품 구매 → 결제"
    "검색 → 필터 → 상세 → 장바구니"

  통합 테스트:        20~30%  ← API + DB 연동
    "Server Action이 DB에 정확히 저장하는가"

  유닛 테스트 (Vitest): 50~70%  ← 비즈니스 로직
    "가격 계산 함수", "유효성 검사 함수"

E2E 테스트가 많을수록:
  → 실행 시간 ⬆️ (브라우저 자동화는 느림!)
  → 유지보수 비용 ⬆️ (UI 바뀔 때마다 수정!)
  → CI 비용 ⬆️ (러닝 타임 = 과금!)
```

## 7. 전체 명령어 치트시트

```bash
# === 테스트 실행 ===
npx playwright test                    # 전체 실행 (헤드리스)
npx playwright test --headed           # 브라우저 보면서!
npx playwright test --project=chromium # 특정 브라우저만
npx playwright test tests/auth.spec.ts # 특정 파일만
npx playwright test -g "로그인"         # 이름에 "로그인" 포함된 것만

# === 디버깅 ===
npx playwright test --ui              # UI 모드 ⭐
npx playwright test --debug           # 디버거 모드 (breakpoint!)
npx playwright codegen localhost:3000  # 코드 자동 생성!
npx playwright test --trace on        # Trace 수집

# === 리포트 ===
npx playwright show-report            # HTML 리포트 열기
npx playwright show-trace trace.zip   # Trace Viewer 열기

# === 관리 ===
npx playwright install                # 브라우저 설치/업데이트
npx playwright test --update-snapshots # 스크린샷 기준 갱신
```

## 8. 실습

```
과제: CI/CD 파이프라인 완성

1. .github/workflows/playwright.yml 작성
   - push/PR 시 자동 실행
   - Playwright 브라우저 설치
   - Next.js 빌드 → E2E 테스트
   - 실패 시 리포트 업로드

2. playwright.config.ts 최적화
   - CI: workers 1, retries 2
   - 로컬: workers auto, retries 0
   - trace: "on-first-retry"

3. 디버깅 도구 체험
   - npx playwright test --ui 로 UI 모드 체험!
   - npx playwright codegen 으로 자동 생성 체험!
   - 일부러 테스트 실패시키고 HTML 리포트에서 원인 분석!

4. .gitignore에 Playwright 관련 폴더 추가
```

## 면접 대비

### 🔴 필수 질문

**Q1. E2E 테스트를 CI에서 어떻게 운영하나요?**

> GitHub Actions에서 PR/push 시 자동 실행한다. `npm ci`로 의존성을 설치하고 `npx playwright install --with-deps`로 브라우저 엔진을 설치한 뒤, Next.js를 프로덕션 빌드(`npm run build`)하여 실제 배포 환경과 동일하게 테스트한다. 실패 시 HTML 리포트와 Trace를 아티팩트로 업로드하여 원인을 분석한다. 대규모 프로젝트에서는 Sharding으로 여러 머신에 분산 실행하여 속도를 높인다.

**Q2. 테스트 피라미드에서 E2E의 비중은?**

> E2E는 전체 테스트의 10~20%만 차지해야 한다. 브라우저 자동화는 느리고 유지보수 비용이 높으므로 "회원가입→로그인→구매" 같은 핵심 비즈니스 흐름만 커버한다. 나머지는 유닛 테스트(50~70%, Vitest)와 통합 테스트(20~30%)로 채운다. E2E가 너무 많으면 CI 시간과 비용이 폭증한다.
