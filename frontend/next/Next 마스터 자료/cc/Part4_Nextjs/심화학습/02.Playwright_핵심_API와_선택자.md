# Playwright 심화 02. 핵심 API와 선택자(Locator)

> **키워드**: `Locator` `getByRole` `getByTestId` `Auto-waiting` `Assertion` `page.fill` `page.click`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | Locator 전략, 핵심 액션, Assertion | 매일 쓰는 API |
| 🟡 이해 | Auto-waiting, 선택자 우선순위, 다중 요소 | 안정적 테스트 |

## 1. 선택자(Locator) — 요소를 찾는 법 🔴

> 💡 **비유: 식당에서 사람 찾기**
> ❌ CSS 선택자: "3번 테이블 왼쪽에 앉은 사람" → 자리 바뀌면 못 찾음!
> ✅ Role 선택자: "웨이터 역할을 하는 사람" → 어디 있든 찾음!

### 선택자 우선순위 (가장 중요!)

```
1순위: getByRole()      ← 접근성 역할! (가장 안정적, 가장 권장!) ⭐
2순위: getByText()      ← 보이는 텍스트!
3순위: getByLabel()     ← <label> 연결!
4순위: getByPlaceholder() ← placeholder 텍스트!
5순위: getByTestId()    ← data-testid 속성! (최후의 수단!)
❌ 최악: page.locator(".class") / page.locator("#id") ← 깨지기 쉬움!

원칙: 사용자가 보는 것으로 찾아라!
  사용자는 CSS 클래스를 모른다!
  사용자는 "버튼"이나 "입력칸"을 본다!
  → getByRole이 가장 사용자에 가까운 선택자!
```

### getByRole — 접근성 기반 (1순위) ⭐

```ts
// 버튼 찾기
page.getByRole("button", { name: "저장" });
// → <button>저장</button>
// → <button aria-label="저장">💾</button>

// 링크 찾기
page.getByRole("link", { name: "About" });
// → <a href="/about">About</a>
// → <Link href="/about">About</Link>

// 제목(heading) 찾기
page.getByRole("heading", { name: "상품 목록" });
// → <h1>상품 목록</h1> ~ <h6>상품 목록</h6>

// 특정 레벨의 heading
page.getByRole("heading", { name: "소개", level: 2 });
// → <h2>소개</h2> (h1은 안 잡힘!)

// 입력 필드 찾기
page.getByRole("textbox", { name: "이메일" });
// → <input type="text" aria-label="이메일" />
// → <label>이메일<input type="text" /></label>

// 체크박스
page.getByRole("checkbox", { name: "동의합니다" });

// 주요 Role 목록:
//   button, link, heading, textbox, checkbox, radio,
//   combobox(select), navigation, dialog, tab, list, listitem
```

### getByText — 텍스트 기반 (2순위)

```ts
// 정확한 텍스트
page.getByText("환영합니다!");

// 부분 일치 (기본!)
page.getByText("환영");  // "환영합니다!"도 잡힘!

// 정확히 일치하게 하려면
page.getByText("환영합니다!", { exact: true });

// 정규식으로 유연하게
page.getByText(/상품 \d+개/);  // "상품 5개", "상품 123개" 등
```

### getByLabel — 라벨 기반 (3순위)

```ts
// <label>과 연결된 input 찾기
page.getByLabel("이메일");
// → <label for="email">이메일</label>
//   <input id="email" type="email" />

page.getByLabel("비밀번호");
```

### getByPlaceholder — placeholder 기반 (4순위)

```ts
page.getByPlaceholder("검색어를 입력하세요");
// → <input placeholder="검색어를 입력하세요" />
```

### getByTestId — 최후의 수단 (5순위)

```tsx
// 컴포넌트에 data-testid 속성 추가
<div data-testid="product-card">상품 카드</div>

// 테스트에서 찾기
page.getByTestId("product-card");

// ⚠️ 사용자가 볼 수 없는 속성이므로 최후의 수단!
// Role이나 Text로 찾을 수 없을 때만 사용!
```

## 2. 핵심 액션(Actions) — 사용자처럼 행동하기 🔴

### 클릭

```ts
// 단순 클릭
await page.getByRole("button", { name: "저장" }).click();

// 더블 클릭
await page.getByText("항목").dblclick();

// 우클릭
await page.getByText("항목").click({ button: "right" });
```

### 입력

```ts
// 텍스트 입력 (기존 내용 지우고 새로 입력!)
await page.getByLabel("이메일").fill("test@example.com");

// 한 글자씩 타이핑 (자동완성 테스트 등)
await page.getByLabel("검색").pressSequentially("React", { delay: 100 });

// 기존 값 지우기
await page.getByLabel("이메일").clear();
```

### 선택 (드롭다운)

```ts
// <select> 값 선택
await page.getByLabel("카테고리").selectOption("electronics");

// value가 아닌 텍스트로 선택
await page.getByLabel("카테고리").selectOption({ label: "전자제품" });
```

### 체크박스 / 라디오

```ts
// 체크
await page.getByRole("checkbox", { name: "동의" }).check();

// 체크 해제
await page.getByRole("checkbox", { name: "동의" }).uncheck();

// 체크 상태 확인
await expect(page.getByRole("checkbox", { name: "동의" })).toBeChecked();
```

### 키보드

```ts
// Enter 키
await page.getByLabel("검색").press("Enter");

// 단축키
await page.keyboard.press("Control+A");
await page.keyboard.press("Meta+C");  // Mac Cmd+C
```

### 페이지 이동

```ts
// URL로 이동
await page.goto("/products");

// 뒤로 가기
await page.goBack();

// 앞으로 가기
await page.goForward();

// 새로고침
await page.reload();
```

## 3. Assertion(검증) — 결과 확인 🔴

### 페이지 전체 검증

```ts
// URL 확인
await expect(page).toHaveURL("/products");
await expect(page).toHaveURL(/\/products\/\d+/);  // 정규식!

// 타이틀 확인
await expect(page).toHaveTitle("상품 목록 | My Shop");
await expect(page).toHaveTitle(/상품/);
```

### 요소 검증

```ts
const button = page.getByRole("button", { name: "저장" });

// 보이는지
await expect(button).toBeVisible();

// 안 보이는지
await expect(button).not.toBeVisible();
// 또는
await expect(button).toBeHidden();

// 활성화 상태
await expect(button).toBeEnabled();
await expect(button).toBeDisabled();

// 텍스트 포함
await expect(button).toHaveText("저장");
await expect(button).toContainText("저");  // 부분 일치

// 속성 확인
await expect(button).toHaveAttribute("type", "submit");

// CSS 클래스 확인
await expect(button).toHaveClass(/bg-blue/);

// 개수 확인
await expect(page.getByRole("listitem")).toHaveCount(5);
```

### 입력 필드 검증

```ts
// 입력된 값 확인
await expect(page.getByLabel("이메일")).toHaveValue("test@example.com");

// placeholder 확인
await expect(page.getByRole("textbox")).toHaveAttribute("placeholder", "검색...");
```

## 4. Auto-waiting — Playwright의 자동 대기 🟡

> 💡 **Playwright의 가장 강력한 무기!**
> 다른 테스트 도구(Cypress, Selenium)에서 `sleep(3000)` 같은 하드코딩이 필요했던 것을
> Playwright는 알아서 기다려줌!

```
page.click("button"):
  1. 요소가 DOM에 존재하는지 확인 → 없으면 대기!
  2. 요소가 화면에 보이는지 확인 (display: none이면 대기!)
  3. 요소가 안정적인지 확인 (애니메이션 중이면 대기!)
  4. 요소가 클릭 가능한지 확인 (다른 요소에 가려져 있으면 대기!)
  5. 모든 조건 충족 → 클릭!

→ setTimeout이나 waitFor 없이도 안정적인 테스트!
→ 서버 컴포넌트의 스트리밍이 끝날 때까지도 자동 대기!
```

### 수동 대기가 필요한 경우 (드물지만!)

```ts
// 특정 요소가 나타날 때까지 대기
await page.waitForSelector("[data-testid='loaded']");

// 네트워크 요청이 끝날 때까지 대기
await page.waitForResponse("**/api/products");

// URL이 바뀔 때까지 대기
await page.waitForURL("/dashboard");

// 로딩 상태가 사라질 때까지 대기
await page.waitForLoadState("networkidle");

// ⚠️ 절대 쓰지 말 것:
// await page.waitForTimeout(3000);  ← 하드코딩 대기! 불안정!
```

## 5. 다중 요소 다루기 🟡

```ts
// 모든 상품 카드 가져오기
const cards = page.getByTestId("product-card");

// 개수 확인
await expect(cards).toHaveCount(6);

// 첫 번째 카드
await cards.first().click();

// 마지막 카드
await cards.last().click();

// n번째 카드 (0부터!)
await cards.nth(2).click();  // 3번째

// 특정 조건으로 필터링
await cards.filter({ hasText: "노트북" }).click();

// 자식 요소에서 찾기
await cards.first().getByRole("button", { name: "장바구니" }).click();
```

## 6. 실습

```
과제: Locator + Action + Assertion 종합 테스트

1. tests/products.spec.ts 생성

2. 테스트 케이스 작성:
   - 상품 목록 페이지 접속 → heading "상품 목록" 확인 (getByRole)
   - 상품 카드가 3개 이상인지 확인 (toHaveCount)
   - 검색창에 "노트북" 입력 후 Enter (fill + press)
   - URL에 ?q=노트북이 포함되는지 확인 (toHaveURL)
   - 첫 번째 상품의 "상세보기" 클릭 (filter + click)
   - 상세 페이지 URL 확인 (toHaveURL(/products\/\d+/))
   - "장바구니 담기" 버튼이 활성화 상태인지 확인 (toBeEnabled)

3. npx playwright test --headed 로 브라우저에서 확인!
```

## 면접 대비

### 🔴 필수 질문

**Q1. Playwright의 Locator 전략 우선순위?**

> `getByRole`이 1순위로 접근성(a11y) 역할 기반이라 UI 변경에 가장 안정적이다. 그 다음 `getByText`, `getByLabel`, `getByPlaceholder` 순이며, `getByTestId`는 다른 방법으로 찾을 수 없을 때의 최후 수단이다. CSS 선택자(`.class`, `#id`)는 스타일 변경에 깨지기 쉬워 권장하지 않는다.

**Q2. Playwright의 Auto-waiting이란?**

> 클릭, 입력 등 액션을 수행할 때 요소가 DOM에 존재하고, 화면에 보이고, 안정적이고, 클릭 가능한 상태가 될 때까지 자동으로 대기하는 기능이다. `setTimeout`이나 수동 `waitFor` 없이도 비동기 렌더링(서버 컴포넌트 스트리밍 등)을 안정적으로 테스트할 수 있어, 테스트의 신뢰성(Flaky 방지)이 높아진다.

## 핵심 요약

```
+------------------------------------------------+
|  Playwright 핵심 API 요약                        |
+------------------------------------------------+
|                                                |
|  선택자 우선순위:                                 |
|    getByRole ⭐ > getByText > getByLabel        |
|    > getByPlaceholder > getByTestId             |
|    ❌ .class, #id 지양!                         |
|                                                |
|  핵심 액션:                                      |
|    click(), fill(), press(), selectOption()      |
|    check(), goto(), goBack()                    |
|                                                |
|  검증(Assertion):                                |
|    expect(page).toHaveURL() / toHaveTitle()     |
|    expect(el).toBeVisible() / toHaveText()      |
|    expect(el).toBeEnabled() / toHaveCount()     |
|                                                |
|  Auto-waiting:                                   |
|    모든 액션이 자동으로 요소를 기다림!              |
|    sleep/setTimeout 불필요!                      |
|                                                |
+------------------------------------------------+
```
