## 1. 상태 누수(State Bleeding) — 왜 전역 Store가 위험한가? 🔴

> 💡 **비유: 24시간 공용 주방에서 모든 손님이 같은 냄비를 쓰는 참사!**
> A 손님이 소금을 넣었는데 → B 손님의 국에도 짜짐!

```
일반 Zustand (create):
  const useStore = create((set) => ({ count: 0, ... }));
  → 이 Store는 모듈 레벨에서 딱 하나만 존재! (싱글톤!)

문제: Next.js 서버는 24시간 켜져 있는 하나의 프로세스!
  → A 사용자가 접속 → count가 5로 변경됨
  → B 사용자가 접속 → count가 5인 상태에서 시작! 😱
  → A 사용자의 장바구니가 B에게 보이는 대참사!

이것이 "상태 누수(State Bleeding)"!
  → 서버 환경에서 전역 싱글톤 Store를 쓰면 발생!
  → 해결: 사용자마다 독립된 Store 인스턴스를 만들어야 함!
```

## 2. 해결: vanilla Store + Context Provider 패턴 🔴

> 💡 **비유: 손님마다 전용 비서를 한 명씩 배정!**
> 공용 냄비 대신 → 개인 전용 냄비를 하나씩 지급!

### Step 1: Store 공장(Factory) 만들기

```tsx
// store/cartStore.ts
import { createStore } from "zustand/vanilla";
// ⚠️ 일반 create가 아니라 zustand/vanilla의 createStore!
// → React 훅에 묶이지 않는 순수한 Store를 만드는 공장!
// → 호출할 때마다 완전히 새로운 독립 인스턴스가 생성!

export interface CartState {
  items: number;
  add: () => void;
  remove: () => void;
  reset: () => void;
}

// Store를 "바로 만드는" 게 아니라 "만드는 공장"을 만듦!
// → 손님이 올 때마다 이 공장을 가동해서 새 Store를 찍어냄!
export const createCartStore = (initState: number = 0) => {
  return createStore<CartState>((set) => ({
    items: initState,
    add: () => set((state) => ({ items: state.items + 1 })),
    remove: () => set((state) => ({ items: Math.max(0, state.items - 1) })),
    reset: () => set({ items: 0 }),
  }));
};
```

```
왜 zustand/vanilla인가?

  일반 create: React 훅(useState, useEffect)과 묶여있음
    → 서버에서 사용하면 "훅은 서버에서 못 씀" 에러!

  createStore (vanilla): React와 무관한 순수 JS Store
    → 서버에서도 안전하게 인스턴스를 만들 수 있음!
    → 나중에 useStore 훅으로 React와 연결!
```

### Step 2: Context Provider로 명함 배급소 만들기

```tsx
// components/CartProvider.tsx
"use client";

import { createContext, useState, use } from "react";
import { createCartStore, type CartState } from "@/store/cartStore";
import { useStore } from "zustand";

// Store 인스턴스의 타입
type CartStore = ReturnType<typeof createCartStore>;

// 명함을 담을 빈 통(Context) 생성
export const CartContext = createContext<CartStore | null>(null);

export function CartProvider({
  children,
  initialItems = 0,
}: {
  children: React.ReactNode;
  initialItems?: number;
}) {
  // ⭐ useState의 지연 초기화(Lazy Initialization)!
  // → 리렌더링되어도 Store가 새로 만들어지지 않음!
  // → 컴포넌트 생명주기 동안 딱 1개의 인스턴스만 유지!
  const [store] = useState(() => createCartStore(initialItems));

  // React 19의 최신 Context 문법 (.Provider 생략 가능!)
  return <CartContext value={store}>{children}</CartContext>;
}

// 커스텀 훅: 셀렉터로 필요한 것만 골라 쓰기!
export function useCartStore<T>(selector: (state: CartState) => T): T {
  const store = use(CartContext); // React 19의 use 훅!
  if (!store) throw new Error("CartProvider missing!");
  return useStore(store, selector);
  // → store와 React를 연결!
  // → selector가 반환하는 값이 바뀔 때만 리렌더링!
}
```

```
왜 useState(() => createCartStore())인가?

  ❌ const store = createCartStore();
    → 리렌더링 때마다 새 Store가 만들어짐!
    → 사용자가 담은 물건이 사라지는 대참사!

  ✅ const [store] = useState(() => createCartStore());
    → 최초 마운트 때 딱 1번만 실행!
    → 리렌더링되어도 같은 Store 유지! ⭐
```

```
왜 Context.Provider에 Store "인스턴스"를 넣는가?

  ❌ 숫자(count: 5)를 직접 넣으면:
    → 숫자가 바뀔 때마다 Provider가 리렌더링!
    → 하위 모든 컴포넌트도 리렌더링! (폭포수!)

  ✅ Store 인스턴스(명함)를 넣으면:
    → 인스턴스의 주소는 절대 안 바뀜!
    → Provider 리렌더링 0회!
    → 자식들은 셀렉터로 필요한 것만 구독! (어깨 톡톡!)
```

### Step 3: 서버 컴포넌트에서 초기값 주입

```tsx
// app/page.tsx (서버 컴포넌트)
import { CartProvider } from "@/components/CartProvider";
import CartCounter from "@/components/CartCounter";

export default async function Page() {
  // 서버에서 DB 조회로 초기 장바구니 수량을 가져옴!
  const initialItems = 7; // 실무: await db.cart.count({ where: { userId } })

  return (
    <CartProvider initialItems={initialItems}>
      {/* CartCounter는 클라이언트 컴포넌트! */}
      {/* → 서버에서 7을 물고 가서 → 브라우저에서 즉시 표시! */}
      <CartCounter />
    </CartProvider>
  );
}
```

```tsx
// components/CartCounter.tsx (클라이언트 컴포넌트)
"use client";
import { useCartStore } from "./CartProvider";

export default function CartCounter() {
  // 셀렉터: items 값만 골라서 구독!
  // → items가 변할 때만 이 컴포넌트 리렌더링!
  // → Store의 다른 값(reset 함수 등)이 변해도 무반응!
  const items = useCartStore((state) => state.items);
  const add = useCartStore((state) => state.add);

  return (
    <div>
      <p className="text-6xl font-bold">{items}</p>
      <button onClick={add}>상품 담기</button>
    </div>
  );
}
```

### 데이터의 여정 요약

```
1. 서버 컴포넌트: DB에서 initialItems = 7을 가져옴
2. CartProvider: 7이 채워진 새 Store 인스턴스를 생성
3. Context: Store의 명함(주소)을 하위에 배포
4. CartCounter: 명함으로 Store를 찾아 items = 7을 즉시 표시!
5. 사용자 클릭: add() → items = 8 → CartCounter만 리렌더링!
```

## 3. Persist 미들웨어 — 새로고침에도 데이터 생존! 🟡

> 💡 **비유: 브라우저의 영구 금고(localStorage)에 데이터 보관!**
> RAM은 새로고침하면 날아가지만 → 금고는 브라우저를 꺼도 유지!

### skipHydration이 필수인 이유!

```
문제: PPR(Partial Prerendering) 환경에서...

  서버: "장바구니 = 0개" (서버는 브라우저 금고를 못 열어봄!)
  브라우저: "금고에 5개 있는데?"

  → 서버가 보낸 HTML(0개)과 브라우저의 상태(5개)가 불일치!
  → Hydration Mismatch 에러! 💥

해결: skipHydration + 수동 rehydrate!
  1. 처음에는 서버와 동일하게 0개로 시작! (에러 방지!)
  2. 하이드레이션 완료 후 → 금고를 열어 5개로 업데이트!
```

### 구현 코드

```tsx
// store/cartStore.ts — Persist 미들웨어 적용!
import { createStore } from "zustand/vanilla";
import { persist, createJSONStorage } from "zustand/middleware";

export interface CartState {
  items: number;
  add: () => void;
}

export const createCartStore = (initState: number = 0) => {
  return createStore<CartState>()(
    persist(
      (set) => ({
        items: initState,
        add: () => set((state) => ({ items: state.items + 1 })),
      }),
      {
        name: "cart-storage",  // localStorage의 키 이름!
        storage: createJSONStorage(() => localStorage),
        skipHydration: true,   // ⭐ 자동 복구 중단! 서버와 불일치 방지!
      }
    )
  );
};
```

```tsx
// components/CartProvider.tsx — 수동 rehydrate!
"use client";

import { createContext, useState, useEffect, use } from "react";
import { createCartStore, type CartState } from "@/store/cartStore";
import { useStore, Mutate, StoreApi } from "zustand";

export const CartContext = createContext<ReturnType<typeof createCartStore> | null>(null);

export function CartProvider({
  children,
  initialItems = 0,
}: {
  children: React.ReactNode;
  initialItems?: number;
}) {
  const [store] = useState(() => createCartStore(initialItems));

  useEffect(() => {
    // ⭐ 하이드레이션이 끝난 안전한 시점에서만 금고를 열어!
    // Zustand의 내부 타입을 명시해야 rehydrate()에 접근 가능!
    type PersistedStore = Mutate<
      StoreApi<CartState>,
      [["zustand/persist", unknown]]
    >;
    (store as unknown as PersistedStore).persist.rehydrate();
    // → localStorage에서 실제 데이터를 꺼내와 Store에 주입!
    // → 화면이 0 → 5로 부드럽게 업데이트!
  }, [store]);

  return <CartContext value={store}>{children}</CartContext>;
}

export function useCartStore<T>(selector: (state: CartState) => T): T {
  const store = use(CartContext);
  if (!store) throw new Error("CartProvider missing!");
  return useStore(store, selector);
}
```

### isMounted 패턴 — 시각적 깜빡임 방지!

```tsx
// components/CartCounter.tsx
"use client";
import { useState, useEffect } from "react";
import { useCartStore } from "./CartProvider";

export default function CartCounter() {
  const [isMounted, setIsMounted] = useState(false);
  const items = useCartStore((state) => state.items);
  const add = useCartStore((state) => state.add);

  useEffect(() => {
    setIsMounted(true); // 브라우저에 안착 완료!
  }, []);

  // 서버 HTML과 동일한 "복구 중..." 표시 → 하이드레이션 에러 방지!
  if (!isMounted) {
    return <div className="animate-pulse bg-gray-200 h-20 rounded" />;
  }

  // 금고에서 복구된 진짜 데이터!
  return (
    <div>
      <p className="text-6xl font-bold">{items}</p>
      <button onClick={add}>상품 담기</button>
    </div>
  );
}
```

```
isMounted 전략이 하이드레이션 에러를 막는 원리:

  서버 렌더링: isMounted = false → "복구 중..." 스켈레톤!
  브라우저 첫 렌더링: isMounted = false → "복구 중..." 스켈레톤!
  → 서버와 브라우저가 동일! → 하이드레이션 성공! ✅

  useEffect 실행: isMounted = true → 리렌더링!
  → 스켈레톤이 사라지고 → localStorage에서 복구된 진짜 숫자 표시!

  suppressHydrationWarning은 html/body 태그에만!
  데이터 컴포넌트에는 isMounted 전략이 정석!
```

        <CartProvider>
          {/* children(메인)과 modal(@modal) 모두 같은 Store! */}

## 면접 대비

### 🔴 필수 질문

**Q1. Next.js에서 Zustand의 상태 누수 문제와 해결법?**

> Next.js 서버는 하나의 장기 실행 프로세스이므로, `create()`로 만든 전역 싱글톤 Store는 여러 사용자 간에 상태가 공유되는 상태 누수(State Bleeding)가 발생한다. 해결법은 `zustand/vanilla`의 `createStore`로 Store 팩토리 함수를 만들고, React Context Provider 내부에서 `useState`의 지연 초기화로 사용자마다 독립된 인스턴스를 생성하여 전달하는 것이다. Provider에는 변하는 값이 아닌 Store 인스턴스(주소)를 넣어 불필요한 리렌더링을 방지한다.

**Q2. Zustand Persist에서 skipHydration이 왜 필요한가?**

> Persist 미들웨어는 기본적으로 컴포넌트 마운트 시 자동으로 localStorage에서 데이터를 복구한다. 하지만 Next.js의 SSR/PPR 환경에서는 서버가 보낸 HTML(items=0)과 브라우저가 복구한 상태(items=5)가 불일치하여 Hydration Mismatch 에러가 발생한다. `skipHydration: true`로 자동 복구를 중단하고, `useEffect` 안에서 `rehydrate()`를 수동 호출하면 하이드레이션이 완료된 안전한 시점에서만 localStorage 데이터를 주입하여 에러를 방지한다.

**Q3. Context API 대신 Zustand를 쓰는 이유?**

> Context API는 값이 변경되면 구독한 모든 하위 컴포넌트가 리렌더링되는 "확성기 효과"가 있어 성능이 떨어진다. Zustand는 셀렉터를 통해 컴포넌트가 필요한 값만 구독하므로, 해당 값이 변할 때만 해당 컴포넌트만 리렌더링하는 "어깨 톡톡" 방식이다. 또한 Provider 래핑이 필수가 아니라 서버 컴포넌트의 이점을 살리기 쉽고, 보일러플레이트가 극히 적다.
