## 1. Security Headers — 웹 서버의 방탄유리! 🔴

> 💡 **비유: 건물 유리를 방탄유리로 교체!**
> 내부 경비원(프록시, DAL)이 아무리 강해도
> 건물 유리가 얇으면 돌을 던져 깨고 들어올 수 있다!
> Security Headers = HTTP 응답에 보안 지시서를 첨부!

### next.config.ts에서 설정

```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  async headers() {
    return [
      {
        source: "/(.*)", // 모든 경로에 적용!
        headers: [
          // ─── 1. 클릭재킹 방어! ───
          {
            key: "X-Frame-Options",
            value: "DENY",
            // 다른 사이트가 우리 사이트를 <iframe>으로 띄우는 것을 차단!
            // 해커가 "아이폰 당첨!" 페이지 위에 투명한 iframe으로
            // 우리의 "전재산 송금" 버튼을 겹쳐놓는 클릭재킹 공격 방어!
          },

          // ─── 2. MIME 스니핑 방어! ───
          {
            key: "X-Content-Type-Options",
            value: "nosniff",
            // 해커가 악성 JS를 image.jpg로 위장해서 올렸을 때
            // 브라우저가 "이거 스크립트네?" 하고 실행하는 것을 차단!
            // "서버가 이미지라고 했으면 이미지로만 취급해!" 강제!
          },

          // ─── 3. 정보 유출 방어! ───
          {
            key: "Referrer-Policy",
            value: "strict-origin-when-cross-origin",
            // 외부 링크로 나갈 때 주소창의 상세 경로가 꼬리표로 따라가는 것 방지!
            // /reset?token=abc123 → 도메인(example.com)만 전달!
          },

          // ─── 4. HTTPS 강제! ───
          {
            key: "Strict-Transport-Security",
            value: "max-age=31536000; includeSubDomains; preload",
            // 1년간 이 사이트는 무조건 HTTPS만 사용!
            // http://로 접속해도 브라우저가 알아서 https://로 변환!
            // 패킷 스니핑(네트워크 도청) 방어!
          },

          // ─── 5. XSS 최후의 보루! (CSP) ───
          {
            key: "Content-Security-Policy",
            value: [
              "default-src 'self'",           // 기본: 우리 서버만!
              "script-src 'self' 'unsafe-inline'", // 스크립트: 우리 서버 + 인라인
              "style-src 'self' 'unsafe-inline'",  // 스타일: 우리 서버 + 인라인
              "img-src 'self' https://trusted-cdn.com", // 이미지: 허용된 CDN만!
            ].join("; "),
            // "허락한 출처에서만 리소스를 로드해!" 브라우저에 강제!
            // 해커가 XSS로 악성 스크립트를 심어도 → 실행 자체를 거부!
          },
        ],
      },
    ];
  },
};

export default nextConfig;
```

### 5대 Security Headers 요약

```
┌──────────────────────────┬──────────────────────────────────┐
│ 헤더                     │ 방어 대상                          │
├──────────────────────────┼──────────────────────────────────┤
│ X-Frame-Options: DENY    │ 클릭재킹 (iframe으로 버튼 겹치기!)  │
│ X-Content-Type-Options   │ MIME 스니핑 (이미지인 척 JS 실행!)  │
│ Referrer-Policy          │ 정보 유출 (URL 꼬리표 제거!)        │
│ Strict-Transport-Security│ HTTP 접속 차단 (HTTPS 강제!)       │
│ Content-Security-Policy  │ XSS (허락한 출처만 실행!)          │
└──────────────────────────┴──────────────────────────────────┘

실무 주의:
  ⚠️ CSP가 너무 엄격하면 자체 JS/CSS도 차단!
     → 개발 중에는 'unsafe-inline' 허용!
     → 프로덕션에서는 nonce 또는 hash로 정밀 제어!

  ⚠️ 설정 변경 후 반드시 서버 재시작! (Ctrl+C → npm run dev)
  ⚠️ HSTS는 localhost(HTTP)에서 문제 발생 가능!
     → 로컬 테스트 시 HSTS 줄을 주석 처리!
```

## 2. proxy.ts — 전역 프록시 (네트워크 경계 방어!) 🟡

> 💡 **비유: 대문 안내 직원!**
> 모든 방문객이 서버에 닿기 전에 반드시 거치는 관문!
> 신분증(토큰) 확인 → 비정상 접근 즉시 차단!

### proxy.ts(또는 middleware.ts) 핵심 구조

```ts
// proxy.ts 또는 middleware.ts (프로젝트 루트!)
import { NextResponse } from "next/server";
import type { NextRequest, NextFetchEvent } from "next/server";

export function middleware(request: NextRequest, event: NextFetchEvent) {
  // 1. 쿠키에서 세션 토큰 확인!
  const sessionToken = request.cookies.get("session_token")?.value;

  // 2. 보호 구역 접근 제어!
  if (request.nextUrl.pathname.startsWith("/dashboard") && !sessionToken) {
    return NextResponse.redirect(new URL("/login", request.url));
    // → 서버를 깨우기도 전에 에지에서 즉시 리다이렉트!
    // → 서버 부하 0!
  }

  // 3. 백그라운드 로깅 (응답 지연 없음!)
  event.waitUntil(
    fetch("https://analytics.example.com/log", {
      method: "POST",
      body: JSON.stringify({ path: request.nextUrl.pathname }),
    }).catch(() => {})
  );

  // 4. 커스텀 헤더 주입!
  const headers = new Headers(request.headers);
  headers.set("x-verified", "true");

  const response = NextResponse.next({
    request: { headers },
  });

  // 5. 응답 쿠키 세팅!
  response.cookies.set("last_visit", Date.now().toString());
  return response;
}

// 정적 파일은 프록시 제외! (성능!)
export const config = {
  matcher: [
    "/((?!api|_next/static|_next/image|favicon.ico|sitemap.xml|robots.txt).*)",
  ],
};
```

```
proxy.ts의 핵심 메서드:

  NextResponse.redirect(url):
    → 메인 서버를 깨우지 않고 에지에서 즉시 리다이렉트!
    → 비로그인 사용자를 /login으로 쫓아내기!

  NextResponse.next():
    → "통과!" 메인 서버로 요청 전달!
    → 커스텀 헤더를 덧붙일 수 있음!

  event.waitUntil(promise):
    → 응답을 먼저 보내고 → 백그라운드에서 로깅!
    → 사용자 체감 속도에 영향 0!

  config.matcher:
    → _next/static, 이미지 등은 프록시 안 거침!
    → 에지 서버 부하 최소화!
```

### BFF (Backend For Frontend) — 내부 프록시 (route.ts)

```tsx
// app/api/proxy/ai-analyze/route.ts
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const userData = await request.json();

  // ⭐ 서버 전용 환경 변수! (NEXT_PUBLIC_ 없음!)
  // → 브라우저 번들에 절대 포함 안 됨!
  const secretKey = process.env.OPENAI_SECRET_KEY;

  const result = await fetch("https://api.openai.com/v1/chat", {
    headers: { Authorization: `Bearer ${secretKey}` },
    method: "POST",
    body: JSON.stringify(userData),
  });

  return NextResponse.json(await result.json());
}
```

```
BFF 패턴의 3대 이점:

  1. API 키 은닉: OPENAI_SECRET_KEY가 브라우저에 노출 안 됨!
  2. CORS 우회: 같은 도메인이므로 브라우저 CORS 에러 없음!
  3. 데이터 가공: 외부 API 응답을 프론트엔드에 맞게 정리해서 전달!
```

## 3. 환경 변수 — 서버와 클라이언트의 두꺼운 유리벽! 🔴

### 핵심 규칙: NEXT_PUBLIC_ 접두사

```
서버 전용 (접두사 없음):
  DATABASE_URL=postgresql://...
  OPENAI_SECRET_KEY=sk-...
  SESSION_SECRET=abc...
  → process.env.DATABASE_URL → 서버에서만 접근 가능!
  → 브라우저에서 접근하면 undefined! (보안!)

브라우저 공개 (NEXT_PUBLIC_ 접두사):
  NEXT_PUBLIC_API_URL=https://api.myapp.com
  NEXT_PUBLIC_GA_ID=G-12345
  → process.env.NEXT_PUBLIC_API_URL → 브라우저에서도 접근 가능!
  → ⚠️ 여기에 비밀번호/API키를 넣으면 전 세계에 공개됨!
```

### 빌드 타임 프리징(Freezing) — 치명적 함정!

```
NEXT_PUBLIC_ 변수의 동작 원리:

  빌드 시(npm run build):
    빌드 시스템이 코드를 스캔하다가
    process.env.NEXT_PUBLIC_API_URL을 발견하면
    → 그 텍스트를 지우고 → 실제 값으로 교체! (인라인!)
    → 코드에 "https://api.myapp.com"이 박혀버림!

  문제:
    빌드 후에 .env를 수정해도 → 클라이언트 코드는 안 바뀜!
    → "왜 값이 안 바뀌지?" → 빌드 다시 해야 함!

  서버 전용 변수는?
    → 빌드 시점에 박히지 않음!
    → 런타임(요청 시점)에 매번 새로 읽음!
    → connection() 호출로 동적 렌더링하면 항상 최신값!
```

```tsx
// 클라이언트 컴포넌트에서의 환경 변수
"use client";

// ✅ 성공! (빌드 시 실제 값으로 교체됨!)
console.log(process.env.NEXT_PUBLIC_API_URL);

// ❌ 실패! (서버 전용이라 undefined!)
console.log(process.env.DATABASE_URL);

// ❌ 실패! (동적 조회는 빌드 시스템이 못 찾음!)
const key = "NEXT_PUBLIC_API_URL";
console.log(process.env[key]); // undefined!

// ❌ 실패! (변수에 담으면 못 찾음!)
const env = process.env;
console.log(env.NEXT_PUBLIC_API_URL); // undefined!
```

```tsx
// 서버 컴포넌트에서의 환경 변수
import { connection } from "next/server";

export default async function Page() {
  // connection()으로 동적 렌더링 모드로 전환!
  // → 빌드 시점에 굳지 않고 요청마다 최신값!
  await connection();

  // ✅ 서버에서는 모든 변수에 접근 가능!
  const dbUrl = process.env.DATABASE_URL;
  const apiUrl = process.env.NEXT_PUBLIC_API_URL;

  return <div>{apiUrl}</div>;
}
```

## 4. 기기 감지 + 동적 리다이렉트 (보너스!) 🟡

```ts
// middleware.ts — 모바일 사용자를 /m 경로로 리다이렉트!
import { NextRequest, NextResponse, userAgent } from "next/server";

export function middleware(req: NextRequest) {
  const { device } = userAgent(req);
  const url = req.nextUrl.clone();

  if (device.type === "mobile") {
    // 이미 /m 경로면 → 무한 루프 방지!
    if (url.pathname.startsWith("/m")) {
      return NextResponse.next();
    }
    url.pathname = `/m${url.pathname}`;
    return NextResponse.redirect(url);
  }

  return NextResponse.next();
}
```

```
CSS display:none과의 차이:
  CSS: 화면에서 숨기지만 무거운 JS는 다운로드됨! (느림!)
  프록시: 물리적으로 다른 페이지로 보냄! (가벼운 모바일 전용 페이지!)
  → 진짜 모바일 최적화!
```

## 면접 대비

### 🔴 필수 질문

**Q1. Security Headers에서 가장 중요한 것과 각각의 역할?**

> 가장 중요한 것은 CSP(Content-Security-Policy)로, 허락한 출처에서만 스크립트/이미지를 로드하도록 브라우저를 강제하여 XSS 공격의 최후 방어선 역할을 한다. `X-Frame-Options: DENY`는 클릭재킹(iframe으로 버튼 겹치기)을 방어하고, `X-Content-Type-Options: nosniff`는 MIME 스니핑(이미지인 척 JS 실행)을 차단한다. `HSTS`는 HTTP 접속을 HTTPS로 강제하여 도청을 방지한다. Next.js에서는 `next.config.ts`의 `headers()` 함수로 모든 응답에 자동 주입한다.

**Q2. NEXT_PUBLIC_ 접두사의 의미와 빌드 타임 프리징?**

> `NEXT_PUBLIC_` 접두사가 있는 환경 변수만 브라우저 번들에 포함(인라인)된다. 빌드 시점에 `process.env.NEXT_PUBLIC_X`라는 텍스트를 실제 값으로 치환하므로, 빌드 후 `.env`를 수정해도 클라이언트 코드에 반영되지 않는다(빌드 타임 프리징). 접두사가 없는 변수는 서버에서만 접근 가능하며 브라우저에서는 undefined가 된다. 서버 전용 변수를 런타임에 동적으로 읽으려면 `connection()`으로 동적 렌더링 모드를 활성화해야 한다.

**Q3. middleware.ts(proxy.ts)는 어디서 실행되며 무엇을 할 수 있나요?**

> Edge Runtime에서 실행되어 모든 요청이 메인 서버에 도달하기 전에 가로챈다. 인증 체크(토큰 확인 후 리다이렉트), 국제화(언어 감지), 기기 감지(모바일 리다이렉트), 요청 헤더 수정, 백그라운드 로깅(`event.waitUntil`)이 가능하다. `config.matcher`로 정적 파일을 제외하여 성능을 최적화하고, `NextResponse.redirect`로 서버 부하 없이 에지에서 즉시 응답한다.
