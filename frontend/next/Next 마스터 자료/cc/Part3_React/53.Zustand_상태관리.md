## 1. 왜 Zustand인가? — Redux와의 비교 🔴

```
Redux (2015~):
  - 보일러플레이트 폭발: action → actionType → reducer → dispatch → selector
  - 파일 5개 만들어야 State 하나 추가 😱
  - Provider 필수

Zustand (2021~):
  - 파일 1개에 State + Action 전부! ⚡
  - 보일러플레이트 거의 없음
  - Provider 불필요 (어디서든 import해서 사용!)
  - 번들 크기 1.2KB (Redux Toolkit: ~33KB)
  - TypeScript 기본 지원
```

```tsx
// Redux Toolkit: 상태 하나 추가에 이만큼...
// slice.ts, store.ts, Provider 설정, useSelector, useDispatch...

// Zustand: 이게 전부!
import { create } from "zustand";

const useCounterStore = create<CounterState>((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  reset: () => set({ count: 0 }),
}));

// 컴포넌트에서 사용: Provider 없이!
function Counter() {
  const count = useCounterStore((state) => state.count);
  const increment = useCounterStore((state) => state.increment);

  return <button onClick={increment}>{count}</button>;
}
```

### 스토어 생성

```tsx
// stores/useAuthStore.ts
import { create } from "zustand";

type User = { id: number; name: string; email: string; role: "admin" | "user" };

type AuthState = {
  // State (데이터)
  user: User | null;
  isAuthenticated: boolean;

  // Actions (변경 함수)
  login: (user: User) => void;
  logout: () => void;
  updateProfile: (data: Partial<User>) => void;
};

export const useAuthStore = create<AuthState>((set) => ({
  // 초기 State
  user: null,
  isAuthenticated: false,

  // Actions
  login: (user) => set({ user, isAuthenticated: true }),
  logout: () => set({ user: null, isAuthenticated: false }),
  updateProfile: (data) => set((state) => ({
    user: state.user ? { ...state.user, ...data } : null,
  })),
}));
```

### 컴포넌트에서 사용

```tsx
// ⭐ 핵심: selector로 필요한 것만 구독!
function Header() {
  // 이 컴포넌트는 user가 바뀔 때만 리렌더링!
  const user = useAuthStore((state) => state.user);
  const logout = useAuthStore((state) => state.logout);

  if (!user) return <a href="/login">로그인</a>;

  return (
    <div>
      <span>{user.name}</span>
      <button onClick={logout}>로그아웃</button>
    </div>
  );
}

// 다른 컴포넌트: isAuthenticated만 구독
function ProtectedRoute({ children }: { children: React.ReactNode }) {
  const isAuth = useAuthStore((state) => state.isAuthenticated);
  if (!isAuth) return <Navigate to="/login" />;
  return <>{children}</>;
}
```

## 3. Selector 패턴 — 리렌더링 최적화의 핵심 🔴

```tsx
// ❌ 나쁜 패턴: 스토어 전체를 구독 → 어떤 State가 바뀌어도 리렌더링!
const { user, cart, theme } = useStore();
// user만 쓰는데 cart가 바뀌어도 리렌더링 😱

// ✅ 좋은 패턴: 필요한 것만 selector로 구독!
const user = useStore((state) => state.user);
// user가 바뀔 때만 리렌더링! cart/theme 변경은 무시! ✅

// 여러 값이 필요하면 얕은 비교(shallow)
import { shallow } from "zustand/shallow";

const { user, isAuthenticated } = useStore(
  (state) => ({ user: state.user, isAuthenticated: state.isAuthenticated }),
  shallow  // 객체의 1단계 속성만 비교 (참조 비교 대신)
);
```

> 💡 **Zustand의 선택적 구독이 Context보다 나은 이유**:
> Context는 value가 바뀌면 **모든 소비자**가 리렌더링.
> Zustand는 selector로 구독한 **특정 값이 바뀔 때만** 리렌더링!

## 4. persist 미들웨어 — 새로고침해도 상태 유지! 🔴

```tsx
import { create } from "zustand";
import { persist } from "zustand/middleware";

type CartItem = { id: number; name: string; price: number; quantity: number };

type CartState = {
  items: CartItem[];
  addItem: (product: Omit<CartItem, "quantity">) => void;
  removeItem: (id: number) => void;
  updateQuantity: (id: number, quantity: number) => void;
  clearCart: () => void;
  total: () => number;
};

export const useCartStore = create<CartState>()(
  persist(
    (set, get) => ({
      items: [],

      addItem: (product) => set((state) => {
        const existing = state.items.find(item => item.id === product.id);
        if (existing) {
          return {
            items: state.items.map(item =>
              item.id === product.id
                ? { ...item, quantity: item.quantity + 1 }
                : item
            ),
          };
        }
        return { items: [...state.items, { ...product, quantity: 1 }] };
      }),

      removeItem: (id) => set((state) => ({
        items: state.items.filter(item => item.id !== id),
      })),

      updateQuantity: (id, quantity) => set((state) => ({
        items: quantity <= 0
          ? state.items.filter(item => item.id !== id)
          : state.items.map(item =>
              item.id === id ? { ...item, quantity } : item
            ),
      })),

      clearCart: () => set({ items: [] }),

      // get()으로 현재 상태 읽기 (파생 값)
      total: () => get().items.reduce((sum, item) => sum + item.price * item.quantity, 0),
    }),
    {
      name: "cart-storage",  // localStorage 키 이름
      // 기본: localStorage에 JSON으로 저장
      // 새로고침해도 장바구니 유지! ⭐
    }
  )
);
```

## 5. 스토어 설계 원칙 🟡

```
1. 관심사별로 스토어를 분리하라!
   ❌ useAppStore (user + cart + theme + notification + ...)
   ✅ useAuthStore / useCartStore / useThemeStore / useNotificationStore

2. 서버 상태와 클라이언트 상태를 섞지 마라!
   ❌ Zustand에 API 데이터 저장 (캐싱/동기화 직접 관리해야 함)
   ✅ TanStack Query로 서버 데이터 / Zustand로 UI 상태

3. Action 이름은 동사로!
   ✅ addItem, removeItem, toggleTheme, setFilter
   ❌ item, theme, filter (상태명과 혼동)

4. 파생 값(계산 값)은 get()으로!
   total은 items에서 계산 가능 → 별도 State 아닌 get()으로 계산
```

### 실무 아키텍처 (2026년 표준)

```
서버 데이터 → TanStack Query (자동 캐싱/동기화)
  useQuery, useMutation, invalidateQueries

UI 상태 → Zustand (선택적 구독/미들웨어)
  useCartStore, useThemeStore, useFilterStore

폼 상태 → React Hook Form + Zod (54번!)
  useForm, zodResolver

테마/인증 → Context (드물게 변하는 전역)
  ThemeProvider, AuthProvider
```

## 면접 대비

### 🔴 필수 질문

**Q1. Zustand를 선택한 이유? Redux와의 차이?**

> Zustand는 Redux 대비 보일러플레이트가 극도로 적고(파일 1개에 State + Action), Provider가 필요 없어 설정이 간단하다. 번들 크기가 1.2KB로 매우 가볍고, selector 패턴으로 필요한 State만 구독하여 불필요한 리렌더링을 방지한다. persist/devtools 미들웨어로 localStorage 동기화와 상태 추적도 간편하다.

**Q2. Zustand에서 리렌더링을 최적화하는 방법?**

> selector 함수를 사용하여 스토어에서 필요한 값만 구독한다. `useStore(state => state.count)`처럼 작성하면 count가 변할 때만 해당 컴포넌트가 리렌더링된다. 여러 값을 구독할 때는 `shallow` 비교 함수를 두 번째 인자로 전달하여 객체의 얕은 비교로 불필요한 리렌더링을 방지한다.
