## 1. Props Drilling — 데이터의 긴 여행 🔴

> 💡 **비유: 전화 돌리기**
> 사장님(최상위)이 인턴(최하위)에게 메시지를 전달하려면
> 사장 → 부장 → 과장 → 대리 → 인턴으로 순서대로 전달해야 한다.
> 중간의 부장과 과장은 메시지를 쓰지도 않으면서 그냥 넘겨주기만 한다!

```tsx
// ❌ Props Drilling: 중간 컴포넌트가 불필요하게 Props를 전달만 함
function App() {
  const [user, setUser] = useState({ name: "철수", theme: "dark" });
  return <Layout user={user} onThemeChange={(t) => setUser({ ...user, theme: t })} />;
}

function Layout({ user, onThemeChange }: Props) {
  // Layout은 user를 직접 안 쓰는데 아래로 전달만 함!
  return (
    <div>
      <Header user={user} onThemeChange={onThemeChange} />
      <Main user={user} />
    </div>
  );
}

function Header({ user, onThemeChange }: Props) {
  // Header도 theme만 쓰는데 user 전체를 받음!
  return <ThemeToggle theme={user.theme} onThemeChange={onThemeChange} />;
}

function ThemeToggle({ theme, onThemeChange }: Props) {
  // 드디어 여기서 사용!
  return <button onClick={() => onThemeChange(theme === "dark" ? "light" : "dark")}>{theme}</button>;
}
```

### Props Drilling의 문제점

```
1. 코드 복잡도 ↑: 중간 컴포넌트가 불필요한 Props를 전달
2. 유지보수 ↓: Props가 추가/삭제되면 경로의 모든 컴포넌트를 수정
3. 재사용성 ↓: 중간 컴포넌트가 특정 Props에 의존
```

### 해결 방법

```
깊이 2~3단계: Props Drilling OK (자연스러움)
깊이 4단계+: 해결책 필요!
  → Context API (48번에서 상세히!)
  → Zustand 같은 상태 관리 (53번에서!)
  → children 합성 패턴 (아래에서!)
```

## 2. State 끌어올리기 (Lifting State Up) 🔴

> 💡 **비유: 공유 화이트보드**
> 두 팀이 같은 데이터를 공유해야 하면, 각자 메모하는 대신
> 공통 상위에 화이트보드를 두고 거기서 읽고 쓴다.

```
❌ 문제: 형제 컴포넌트끼리 직접 통신 불가!

  ┌── Parent ──┐
  │            │
  ChildA    ChildB
  (온도 입력)  (변환 결과 표시)

  ChildA가 입력한 온도를 ChildB에서 보려면?
  → State를 Parent로 끌어올려야 한다!
```

```tsx
// ✅ 해결: 공통 부모가 State를 관리
function TemperatureConverter() {
  const [celsius, setCelsius] = useState(0);  // 부모가 State 관리!

  return (
    <div>
      {/* 자식 A: 입력 (콜백으로 부모에게 알림) */}
      <CelsiusInput value={celsius} onChange={setCelsius} />

      {/* 자식 B: 결과 표시 (부모에게서 Props로 받음) */}
      <FahrenheitDisplay celsius={celsius} />
    </div>
  );
}

function CelsiusInput({ value, onChange }: { value: number; onChange: (v: number) => void }) {
  return (
    <label>
      섭씨(°C):
      <input
        type="number"
        value={value}
        onChange={(e) => onChange(Number(e.target.value))}
      />
    </label>
  );
}

function FahrenheitDisplay({ celsius }: { celsius: number }) {
  const fahrenheit = (celsius * 9) / 5 + 32;
  return <p>화씨(°F): {fahrenheit.toFixed(1)}</p>;
}
```

### 끌어올리기 판단 기준

```
"이 State를 여러 컴포넌트에서 공유해야 하는가?"

YES → 가장 가까운 공통 부모로 끌어올리기
NO  → 해당 컴포넌트 내부에서 관리 (불필요하게 올리지 마라!)

원칙: State는 가능한 한 사용하는 곳에 가깝게!
```

## 3. 역방향 데이터 흐름 (콜백 Props) 🔴

> 자식 → 부모로 데이터를 전달하는 유일한 방법: **콜백 함수를 Props로 내려보내기**

```tsx
// 부모: 콜백 함수를 자식에게 내려줌
function TodoApp() {
  const [todos, setTodos] = useState<string[]>([]);

  // 자식이 호출할 콜백 함수를 정의
  const handleAddTodo = (text: string) => {
    setTodos(prev => [...prev, text]);
  };

  const handleDeleteTodo = (index: number) => {
    setTodos(prev => prev.filter((_, i) => i !== index));
  };

  return (
    <div>
      <AddTodoForm onAdd={handleAddTodo} />
      <TodoList todos={todos} onDelete={handleDeleteTodo} />
    </div>
  );
}

// 자식: Props로 받은 콜백을 호출하여 부모에게 데이터 전달
function AddTodoForm({ onAdd }: { onAdd: (text: string) => void }) {
  const [input, setInput] = useState("");

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    if (!input.trim()) return;
    onAdd(input.trim());   // 부모에게 "이 텍스트를 추가해줘!"
    setInput("");
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={input} onChange={(e) => setInput(e.target.value)} />
      <button type="submit">추가</button>
    </form>
  );
}

function TodoList({ todos, onDelete }: { todos: string[]; onDelete: (i: number) => void }) {
  return (
    <ul>
      {todos.map((todo, i) => (
        <li key={i}>
          {todo}
          <button onClick={() => onDelete(i)}>삭제</button>
        </li>
      ))}
    </ul>
  );
}
```

```
데이터 흐름 시각화:

  TodoApp (State: todos)
    │                     ▲
    │ todos (Props ↓)     │ onAdd("새 할일") (콜백 ↑)
    │ onDelete (콜백 ↓)   │
    ▼                     │
  TodoList             AddTodoForm
```

## 4. children 합성으로 Props Drilling 줄이기 🟡

```tsx
// ❌ Props Drilling
function App() {
  const [user, setUser] = useState(currentUser);
  return <Layout user={user} />;  // Layout이 user를 Header에게 전달만 함
}

function Layout({ user }: { user: User }) {
  return (
    <div>
      <Header user={user} />  {/* Layout은 user를 안 쓰는데 전달만! */}
      <main>콘텐츠</main>
    </div>
  );
}

// ✅ children 합성: Layout은 user에 대해 모름!
function App() {
  const [user, setUser] = useState(currentUser);
  return (
    <Layout
      header={<Header user={user} />}  {/* App에서 직접 주입! */}
    >
      <main>콘텐츠</main>
    </Layout>
  );
}

function Layout({ header, children }: { header: React.ReactNode; children: React.ReactNode }) {
  return (
    <div>
      {header}      {/* user에 대해 전혀 모름! 그냥 렌더링만 */}
      {children}
    </div>
  );
}
```

> 💡 **핵심**: 컴포넌트가 "무엇이 들어올지" 모르게 하면 재사용성이 극대화된다.
> Layout은 header에 UserMenu가 오든, Logo가 오든 상관없다!

## 5. 컴포넌트 분리 기준 🟡

```
컴포넌트를 쪼개야 할 때:
  1. 한 컴포넌트가 200줄 이상 → 역할별로 분리
  2. 같은 UI가 2곳 이상 반복 → 재사용 컴포넌트로 추출
  3. 서로 다른 변경 이유 → 단일 책임 원칙(SRP)
  4. 한 부분만 자주 리렌더링 → 분리하여 리렌더링 격리

컴포넌트를 합쳐야 할 때:
  1. Props가 너무 많이 오가며 복잡 → 과도한 분리
  2. 항상 함께 사용되고 독립적으로 의미 없음
  3. 상태를 공유하는 게 자연스러운 단위
```

### Props 설계 원칙

```tsx
// ❌ 나쁜 Props: 너무 많은 개별 값
<UserCard
  userName="철수"
  userEmail="chul@test.com"
  userAge={25}
  userAvatar="/img/chul.jpg"
  userRole="admin"
/>

// ✅ 좋은 Props: 관련 데이터를 객체로 묶기
<UserCard user={user} />

// ❌ 나쁜 Props: boolean 지옥
<Button isPrimary isLarge isDisabled isLoading hasIcon />

// ✅ 좋은 Props: 유니온 리터럴로 의도 명확히
<Button variant="primary" size="lg" disabled loading />
```

## 6. 실습: 쇼핑 카트 시스템

```
과제: State 끌어올리기 + 콜백 Props 실전

1. App: 전체 State 관리 (products, cart)
2. ProductList: 상품 목록 + "장바구니 담기" 버튼
3. Cart: 장바구니 목록 + 수량 변경 + 삭제 + 총액 표시
4. 데이터 흐름:
   - ProductList → App: onAddToCart(productId)
   - Cart → App: onUpdateQuantity(productId, qty), onRemove(productId)
5. 같은 상품을 다시 담으면 수량만 +1
```

<details>
<summary>핵심 타입과 App 정답</summary>

```tsx
type Product = { id: number; name: string; price: number };
type CartItem = { product: Product; quantity: number };

function App() {
  const [cart, setCart] = useState<CartItem[]>([]);

  const products: Product[] = [
    { id: 1, name: "노트북", price: 1200000 },
    { id: 2, name: "키보드", price: 85000 },
    { id: 3, name: "마우스", price: 45000 },
  ];

  const handleAddToCart = (productId: number) => {
    setCart(prev => {
      const existing = prev.find(item => item.product.id === productId);
      if (existing) {
        return prev.map(item =>
          item.product.id === productId
            ? { ...item, quantity: item.quantity + 1 }
            : item
        );
      }
      const product = products.find(p => p.id === productId)!;
      return [...prev, { product, quantity: 1 }];
    });
  };

  const handleUpdateQuantity = (productId: number, quantity: number) => {
    if (quantity <= 0) {
      setCart(prev => prev.filter(item => item.product.id !== productId));
    } else {
      setCart(prev => prev.map(item =>
        item.product.id === productId ? { ...item, quantity } : item
      ));
    }
  };

  const handleRemove = (productId: number) => {
    setCart(prev => prev.filter(item => item.product.id !== productId));
  };

  const total = cart.reduce((sum, item) => sum + item.product.price * item.quantity, 0);

  return (
    <div style={{ display: "flex", gap: 32 }}>
      <ProductList products={products} onAddToCart={handleAddToCart} />
      <Cart
        items={cart}
        total={total}
        onUpdateQuantity={handleUpdateQuantity}
        onRemove={handleRemove}
      />
    </div>
  );
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. Props Drilling이란? 해결 방법은?**

> 깊이 중첩된 컴포넌트에 데이터를 전달하기 위해 중간 컴포넌트들이 자신은 사용하지 않는 Props를 전달만 하는 현상이다. 해결 방법으로는 Context API(전역 상태 공유), 상태 관리 라이브러리(Zustand), children 합성 패턴(중간 컴포넌트를 데이터에서 분리)이 있다. 2~3단계 정도의 Drilling은 자연스러우며, 4단계 이상일 때 해결을 고려한다.

**Q2. State를 끌어올린다(Lifting State Up)는 것은?**

> 두 컴포넌트가 같은 데이터를 공유해야 할 때, 해당 State를 가장 가까운 공통 부모 컴포넌트로 이동시키는 것이다. 부모가 State를 관리하고, 자식들에게는 Props(데이터)와 콜백 함수(변경 요청)를 내려보낸다. 이를 통해 형제 컴포넌트 간 데이터 동기화가 가능해진다.
