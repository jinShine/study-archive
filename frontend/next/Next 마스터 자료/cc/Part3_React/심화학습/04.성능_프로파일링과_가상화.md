## 1. React DevTools Profiler — 성능의 엑스레이 🔴

```
설치: 크롬 웹스토어에서 "React Developer Tools" 설치

사용법:
  1. F12 → Profiler 탭
  2. 설정(⚙️) → "Record why each component rendered" 체크!
  3. 녹화(●) 시작 → UI 조작 → 정지(■)
  4. 결과 분석:
     - 각 컴포넌트의 렌더링 횟수
     - 렌더링에 걸린 시간
     - "왜 렌더링되었는지" (Why did this render?)
     - 불필요한 리렌더링 발견!

Highlight Updates:
  Components 탭 → 설정(⚙️) → "Highlight updates when components render" 체크
  → 리렌더링되는 컴포넌트에 파란색 테두리가 깜빡!
  → 관련 없는 컴포넌트까지 깜빡이면 → 최적화 필요!
```

### 성능 최적화 판단 플로우

```
1. "눈에 보이는 렉(jank)이 있는가?"
   ├─ NO → 최적화 불필요! (과도한 최적화는 독)
   └─ YES → 2단계로

2. DevTools Profiler로 원인 찾기
   ├─ 특정 컴포넌트가 너무 자주 리렌더링?
   │   → State 위치 조정 / memo 적용
   ├─ 렌더링 자체가 오래 걸림?

   ├─ 리스트가 수백~수천 개?
   │   → 가상화(Virtualization) 적용!
   └─ 번들이 너무 큼?
       → 코드 스플리팅 (03번 참고)
```

## 2. 가상화(Virtualization) — 대량 리스트의 해결사 🟡

> 💡 **비유: 고속도로 표지판**
> 서울에서 부산까지 **1000개의 표지판이 있지만**, 운전자는 눈앞의 5~6개만 본다.
> 가상화 = 화면에 보이는 항목만 렌더링하고, 나머지는 "있는 척"만 한다!

```
❌ 가상화 없이 (1만 개 리스트):
  <ul>
    <li>1</li>    ← DOM에 존재, 화면에 안 보임
    <li>2</li>    ← DOM에 존재, 화면에 안 보임
    ...
    <li>50</li>   ← DOM에 존재, 화면에 보임! 👀
    <li>51</li>   ← DOM에 존재, 화면에 보임! 👀
    ...
    <li>10000</li> ← DOM에 존재, 화면에 안 보임
  </ul>
  → DOM 노드 10,000개 → 느리고 메모리 과다 사용!

✅ 가상화:
  <ul style="height: 1000000px"> ← 전체 높이만 유지 (스크롤바 정상)
    <li style="top: 4900px">50</li>  ← 보이는 것만 렌더링!
    <li style="top: 4950px">51</li>
    <li style="top: 5000px">52</li>
    ...
    <li style="top: 5500px">62</li>
  </ul>
  → DOM 노드 ~15개 → 빠르고 가벼움!
```

### @tanstack/react-virtual (2026년 추천)

```bash
npm install @tanstack/react-virtual
```

```tsx
import { useVirtualizer } from "@tanstack/react-virtual";

function VirtualList({ items }: { items: string[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const virtualizer = useVirtualizer({
    count: items.length,      // 전체 아이템 수
    getScrollElement: () => parentRef.current,  // 스크롤 컨테이너
    estimateSize: () => 50,   // 각 아이템의 예상 높이 (px)
    overscan: 5,              // 화면 밖으로 추가 렌더링할 개수 (스크롤 부드러움)
  });

  return (
    <div
      ref={parentRef}
      style={{ height: 400, overflow: "auto" }}  // 스크롤 컨테이너
    >
      <div style={{ height: virtualizer.getTotalSize(), position: "relative" }}>
        {virtualizer.getVirtualItems().map(virtualItem => (
          <div
            key={virtualItem.key}
            style={{
              position: "absolute",
              top: virtualItem.start,
              height: virtualItem.size,
              width: "100%",
            }}
          >
            {items[virtualItem.index]}
          </div>
        ))}
      </div>
    </div>
  );
}

// 사용: 10만 개도 60fps!
const items = Array.from({ length: 100000 }, (_, i) => `아이템 ${i + 1}`);
<VirtualList items={items} />
```

### 가상화를 적용해야 할 때

```
리스트 아이템이 50개 이하 → 가상화 불필요 (오버엔지니어링!)
리스트 아이템이 100~500개 → 상황 봐서 결정
리스트 아이템이 500개 이상 → 가상화 강력 추천!
리스트 아이템이 1만 개 이상 → 가상화 필수!!

가상화 대안:
  - 페이지네이션 (한 번에 20개씩)
  - 무한 스크롤 (TanStack Query useInfiniteQuery)
  - "더 보기" 버튼
```

## 3. Intersection Observer — 뷰포트 감지 🟡

> "이 요소가 화면에 보이면 실행해!" (lazy loading, 무한 스크롤에 활용)

```tsx
function useInView(options?: IntersectionObserverInit) {
  const ref = useRef<HTMLDivElement>(null);
  const [isInView, setIsInView] = useState(false);

  useEffect(() => {
    if (!ref.current) return;

    const observer = new IntersectionObserver(([entry]) => {
      setIsInView(entry.isIntersecting);
    }, options);

    observer.observe(ref.current);
    return () => observer.disconnect();
  }, [options]);

  return { ref, isInView };
}

// 사용: 무한 스크롤 트리거
function ProductList() {
  const { ref: bottomRef, isInView } = useInView();
  const { fetchNextPage, hasNextPage } = useInfiniteQuery({ /* ... */ });

  useEffect(() => {
    if (isInView && hasNextPage) {
      fetchNextPage();  // 화면 바닥에 도달하면 다음 페이지 로드!
    }
  }, [isInView, hasNextPage, fetchNextPage]);

  return (
    <div>
      {/* 상품 리스트 */}
      <div ref={bottomRef} />  {/* 이 요소가 화면에 보이면 → 다음 페이지 로드 */}
    </div>
  );
}
```

> 💡 **실무에서는** `react-intersection-observer` 라이브러리의 `useInView` 사용:
> ```tsx
> import { useInView } from "react-intersection-observer";
> const { ref, inView } = useInView({ threshold: 0 });
> ```

## 4. Web Vitals 핵심 지표 🔴

```
Core Web Vitals (2026년 기준):

LCP (Largest Contentful Paint) — "화면이 뜨기까지"
  → 가장 큰 콘텐츠가 화면에 나타나는 시간
  → 좋음: 2.5초 이하 / 나쁨: 4초 이상
  → 개선: 이미지 최적화, 코드 스플리팅, SSR

INP (Interaction to Next Paint) — "반응 속도"
  → 사용자 클릭 → 화면 변화까지 시간
  → 좋음: 200ms 이하 / 나쁨: 500ms 이상
  → 개선: 무거운 계산 분리, useMemo, useTransition

CLS (Cumulative Layout Shift) — "레이아웃 안정성"
  → 화면 요소가 갑자기 밀리는 정도
  → 좋음: 0.1 이하 / 나쁨: 0.25 이상
  → 개선: 이미지 width/height 명시, 폰트 font-display: swap
```

### Lighthouse로 측정

```
1. 크롬 DevTools → Lighthouse 탭
2. Categories: Performance, Accessibility, SEO 체크
3. "Analyze page load" 클릭
4. 점수 확인: 90점 이상이면 양호!
5. 개선 제안 따라 수정
```

**Q1. React 앱의 성능을 최적화하는 방법?**

> 첫째, State를 사용하는 곳에 가깝게 배치하여 리렌더링 범위를 최소화한다. 둘째, React.memo + useMemo/useCallback으로 불필요한 리렌더링을 방지한다. 셋째, 코드 스플리팅(React.lazy/next/dynamic)으로 초기 번들 크기를 줄인다. 넷째, 대량 리스트는 가상화(@tanstack/react-virtual)를 적용한다. React DevTools Profiler로 실제 병목을 측정한 후 최적화해야 하며, 성능 문제가 없는데 미리 최적화하는 것은 지양한다.

**Q2. Core Web Vitals 3가지를 설명해주세요.**

> LCP(Largest Contentful Paint)는 가장 큰 콘텐츠가 화면에 나타나는 시간으로 2.5초 이하가 목표다. INP(Interaction to Next Paint)는 사용자 상호작용 후 화면이 변하는 속도로 200ms 이하가 목표다. CLS(Cumulative Layout Shift)는 페이지 로드 중 레이아웃이 밀리는 정도로 0.1 이하가 목표다. Lighthouse로 측정할 수 있다.
