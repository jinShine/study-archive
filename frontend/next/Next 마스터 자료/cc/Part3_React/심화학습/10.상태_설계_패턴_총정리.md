## 1. 4가지 상태의 분류 🔴

> 💡 **핵심**: 모든 상태를 같은 도구(useState/Redux)로 관리하면 안 된다!
> 상태의 **성격**에 따라 적절한 도구를 선택해야 한다.

```
┌─────────────────────────────────────────────────┐
│              React 상태 4대 분류                  │
├────────────┬────────────────────────────────────┤
│ Server     │ API에서 가져온 데이터               │
│ State      │ 다른 사용자가 변경 가능             │
│            │ → TanStack Query ⭐                │
├────────────┼────────────────────────────────────┤
│ Client     │ UI 상태 (모달, 사이드바, 선택)      │
│ State      │ 이 사용자만의 상태                  │
│            │ → Zustand / useState ⭐            │
├────────────┼────────────────────────────────────┤
│ URL        │ 검색어, 필터, 페이지, 정렬           │
│ State      │ 공유 가능해야 하는 상태             │
│            │ → URL searchParams / Next.js ⭐    │
├────────────┼────────────────────────────────────┤
│ Form       │ 폼 입력값, 유효성 상태              │
│ State      │ 제출 전까지만 유지되는 임시 상태     │
│            │ → React Hook Form + Zod ⭐         │
└────────────┴────────────────────────────────────┘
```

## 2. 각 상태별 도구 선택 상세 🔴

### Server State → TanStack Query

```
특징:
  - 서버 DB에 원본이 있는 데이터
  - 다른 사용자가 언제든 변경 가능
  - 캐싱, 동기화, 백그라운드 갱신이 필요

예시: 사용자 목록, 상품 정보, 게시글, 주문 내역, 알림

도구: TanStack Query (useQuery, useMutation)
  ✅ 자동 캐싱 + 중복 요청 제거
  ✅ staleTime으로 신선도 제어
  ✅ invalidateQueries로 mutation 후 자동 갱신
  ✅ DevTools로 캐시 상태 시각화
```

### Client State → Zustand (전역) / useState (로컬)

```
특징:
  - 이 사용자, 이 세션에서만 존재하는 상태
  - 서버에 저장할 필요 없음
  - UI의 열림/닫힘, 선택 상태 등

로컬 (한 컴포넌트):
  → useState
  예: 모달 열림 토글, 아코디언 펼침, 드롭다운 열림

전역 (여러 컴포넌트 공유):
  → Zustand
  예: 장바구니, 테마, 사이드바 상태, 알림 설정

영구 저장 필요:
  → Zustand + persist 미들웨어
  예: 다크 모드 설정, 언어 설정, 장바구니

드물게 변하는 전역:
  → Context API
  예: 인증(로그인) 상태, 테마 Provider
```

### URL State → URL searchParams

```
특징:
  - 링크를 공유하면 같은 상태를 볼 수 있어야 함
  - 브라우저 뒤로 가기/앞으로 가기에 반응해야 함
  - 새로고침해도 유지되어야 함

예시: 검색어, 카테고리 필터, 정렬 순서, 페이지 번호, 선택된 탭

도구:
  순수 React: useSearchParams (react-router)
  Next.js: searchParams + <Link> / router.push

  ❌ 나쁜 패턴: 검색어를 useState로만 관리
    → URL이 안 바뀜 → 공유 불가! 새로고침하면 초기화!

  ✅ 좋은 패턴: URL에 ?q=react&sort=latest로 반영
    → 공유 가능! 새로고침해도 유지! 뒤로 가기 작동!
```

### Form State → React Hook Form + Zod

```
특징:
  - 사용자가 입력 중인 임시 데이터
  - 유효성 검사가 필요
  - 제출(submit) 후에는 서버로 전송되고 사라짐

도구:
  간단한 폼 (필드 2~3개): useState
  복잡한 폼 (필드 5개+): React Hook Form + Zod
    ✅ 비제어 방식 → 성능 좋음
    ✅ z.infer → 타입 자동 추출
    ✅ 에러 메시지 자동 관리
```

## 3. 상태 최소화 원칙 — "이것이 진짜 State인가?" 🔴

### 판단 플로우

```
질문 1: "이 값이 다른 State에서 계산 가능한가?"
  YES → State가 아니다! 렌더링 중 계산하라! (파생 상태)
  예: filteredItems = items.filter(...)
     totalPrice = cart.reduce(...)
     isValid = !errors.length

질문 2: "이 값이 시간이 지나도 변하지 않는가?"
  YES → State가 아니다! const로 충분하다!

질문 3: "이 값이 Props로 받은 것인가?"
  YES → 보통 State가 아니다! Props를 그대로 사용하라!
  예외: Props를 초기값으로만 쓸 때 (비제어)

질문 4: 위 3개에 해당 안 함
  → 진짜 State다! useState 또는 적절한 도구 사용
```

### 실전 예시

```tsx
// ❌ 불필요한 State들
const [fullName, setFullName] = useState("");  // firstName + lastName에서 계산 가능!
const [filteredItems, setFilteredItems] = useState([]);  // items + filter에서 계산 가능!
const [isFormValid, setIsFormValid] = useState(false);  // errors에서 계산 가능!

// ✅ 파생 값으로 계산
const fullName = `${firstName} ${lastName}`;
const filteredItems = items.filter(item => item.category === filter);
const isFormValid = Object.keys(errors).length === 0;
```

## 4. 상태 위치 결정 — "어디에 둘까?" 🔴

```
1. "이 State를 사용하는 컴포넌트가 1개?"
   → 그 컴포넌트 내부 useState

2. "형제 컴포넌트가 공유해야?"
   → 공통 부모로 끌어올리기 (Lifting State Up)

3. "먼 거리의 컴포넌트가 공유해야?"
   → 3~4단계 이내: Props Drilling OK
   → 4단계+: Context 또는 Zustand

4. "서버 데이터?"
   → TanStack Query

5. "URL에 반영되어야?"
   → URL searchParams

6. "폼 입력?"
   → React Hook Form
```

## 7. 안티 패턴 — 절대 하지 마라! 🔴

```
❌ 1. 모든 것을 하나의 거대한 스토어에!
  → 관심사별 분리! (auth / cart / filter / theme)

❌ 2. 서버 데이터를 Zustand에 저장!
  → TanStack Query! (캐싱/동기화가 자동!)

❌ 3. URL에 있어야 할 상태를 useState로!
  → 검색어/필터/페이지는 URL searchParams!

❌ 4. 파생 값을 State로!
  → filteredItems, totalPrice는 렌더링 중 계산!

❌ 5. useEffect로 State 동기화!
  → 이벤트 핸들러에서 직접 처리하거나, 파생 값으로 계산!

❌ 6. 모든 컴포넌트에 memo!
  → 문제가 있을 때만! 측정 먼저!

❌ 7. Props 10개 이상 전달!
  → 객체로 묶거나, Context/Zustand로 전환!
```

## 면접 대비

### 🔴 필수 질문

**Q1. React에서 상태를 어떻게 분류하고 관리하나요?**

> 4가지로 분류한다. Server State(API 데이터)는 TanStack Query로 캐싱/동기화를 자동 관리한다. Client State(UI 상태)는 로컬이면 useState, 전역이면 Zustand로 관리한다. URL State(검색/필터/페이지)는 URL searchParams로 관리하여 공유와 새로고침을 보장한다. Form State(폼 입력)는 React Hook Form + Zod로 관리하여 성능과 타입 안전성을 확보한다.

**Q2. "이것이 진짜 State인가?" 판단 기준?**

> 다른 State에서 계산 가능하면 파생 값이므로 State가 아니다. 시간이 지나도 변하지 않으면 상수이므로 State가 아니다. Props로 받은 것은 보통 State가 아니다. 이 3가지에 해당하지 않을 때만 진짜 State이다. 불필요한 State는 useEffect 동기화 지옥과 불일치 버그의 원인이 된다.
