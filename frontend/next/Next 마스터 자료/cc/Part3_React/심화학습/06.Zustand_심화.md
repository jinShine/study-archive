## 1. Immer 미들웨어 — 중첩 객체를 직관적으로 수정 🔴

```bash
npm install immer
```

```tsx
import { create } from "zustand";
import { immer } from "zustand/middleware/immer";

type State = {
  user: {
    name: string;
    address: {
      city: string;
      zip: string;
    };
    settings: {
      theme: "light" | "dark";
      notifications: boolean;
    };
  };
  updateCity: (city: string) => void;
  toggleTheme: () => void;
};

// ❌ Immer 없이: 스프레드 지옥!
const useStore = create<State>((set) => ({
  user: { name: "철수", address: { city: "서울", zip: "06000" }, settings: { theme: "light", notifications: true } },
  updateCity: (city) => set((state) => ({
    user: {
      ...state.user,
      address: {
        ...state.user.address,
        city,  // 3단계 중첩 스프레드... 😱
      },
    },
  })),
}));

// ✅ Immer로: 직접 변경처럼 쓰면 내부적으로 불변 처리!
const useStore = create<State>()(
  immer((set) => ({
    user: { name: "철수", address: { city: "서울", zip: "06000" }, settings: { theme: "light", notifications: true } },
    updateCity: (city) => set((state) => {
      state.user.address.city = city;  // 직접 변경처럼! 하지만 불변! ✅
    }),
    toggleTheme: () => set((state) => {
      state.user.settings.theme = state.user.settings.theme === "light" ? "dark" : "light";
    }),
  }))
);
```

## 2. 비동기 액션 🔴

```tsx
type AuthState = {
  user: User | null;
  isLoading: boolean;
  error: string | null;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
};

const useAuthStore = create<AuthState>()(
  immer((set) => ({
    user: null,
    isLoading: false,
    error: null,

    login: async (email, password) => {
      set((state) => { state.isLoading = true; state.error = null; });

      try {
        const res = await fetch("/api/auth/login", {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify({ email, password }),
        });

        if (!res.ok) throw new Error("로그인 실패");
        const user = await res.json();

        set((state) => { state.user = user; state.isLoading = false; });
      } catch (error) {
        set((state) => {
          state.error = error instanceof Error ? error.message : "알 수 없는 에러";
          state.isLoading = false;
        });
      }
    },

    logout: () => set((state) => { state.user = null; }),
  }))
);
```

## 3. 미들웨어 스택 — 여러 미들웨어 조합 🟡

```tsx
import { create } from "zustand";
import { devtools, persist } from "zustand/middleware";
import { immer } from "zustand/middleware/immer";

// 미들웨어 스택: devtools > persist > immer
const useCartStore = create<CartState>()(
  devtools(                          // 3. Redux DevTools 연동
    persist(                         // 2. localStorage 자동 동기화
      immer((set, get) => ({         // 1. 불변 업데이트 간소화
        items: [],
        addItem: (product) => set((state) => {
          const existing = state.items.find(i => i.id === product.id);
          if (existing) {
            existing.quantity += 1;  // Immer 덕분에 직접 수정 가능!
          } else {
            state.items.push({ ...product, quantity: 1 });
          }
        }),
        get total() {  // 파생 상태 (getter)
          return get().items.reduce((sum, i) => sum + i.price * i.quantity, 0);
        },
      })),
      { name: "cart-storage" }       // persist 설정
    ),
    { name: "CartStore" }            // devtools 설정
  )
);
```

### 미들웨어 적용 순서 (안에서 바깥으로)

```
immer → persist → devtools → create

코드 작성은 바깥에서 안으로:
  create(devtools(persist(immer((set) => ({ ... })))))
```

## 4. 스토어 분리 전략 🔴

```
❌ 나쁜 패턴: 하나의 거대한 스토어
  useAppStore → user + cart + theme + notifications + filters + ...
  → 하나가 바뀌면 전체 영향, 테스트 어려움

✅ 좋은 패턴: 관심사별 분리
  stores/
  ├── useAuthStore.ts      → 인증 (user, login, logout)
  ├── useCartStore.ts      → 장바구니 (items, addItem, removeItem)
  ├── useThemeStore.ts     → 테마 (theme, toggleTheme)
  ├── useFilterStore.ts    → 필터 (search, category, sort)
  └── useNotificationStore.ts → 알림 (notifications, markAsRead)
```

### 스토어 간 통신이 필요하면?

```tsx
// 방법 1: 컴포넌트에서 조합
function CheckoutButton() {
  const items = useCartStore(s => s.items);
  const user = useAuthStore(s => s.user);

  const handleCheckout = () => {
    if (!user) return alert("로그인 필요!");
    processPayment(user, items);
  };
}

// 방법 2: 한 스토어에서 다른 스토어 접근 (드물게)

  checkout: () => {
    const user = useAuthStore.getState().user;  // 다른 스토어의 현재 상태
    if (!user) throw new Error("로그인 필요");
    // ...
  },
}));
```

## 5. 실무 패턴: 필터 스토어 🔴

```tsx
type FilterState = {
  search: string;
  category: string;
  sort: "latest" | "price-asc" | "price-desc" | "popular";
  page: number;
  perPage: number;

  setSearch: (search: string) => void;
  setCategory: (category: string) => void;
  setSort: (sort: FilterState["sort"]) => void;
  setPage: (page: number) => void;
  resetFilters: () => void;
};

const DEFAULT_FILTERS = {
  search: "",
  category: "all",
  sort: "latest" as const,
  page: 1,
  perPage: 20,
};

const useFilterStore = create<FilterState>((set) => ({
  ...DEFAULT_FILTERS,

  setSearch: (search) => set({ search, page: 1 }),  // 검색 시 페이지 리셋!
  setCategory: (category) => set({ category, page: 1 }),
  setSort: (sort) => set({ sort, page: 1 }),
  setPage: (page) => set({ page }),
  resetFilters: () => set(DEFAULT_FILTERS),
}));

// TanStack Query와 결합!
function ProductList() {
  const { search, category, sort, page, perPage } = useFilterStore();

  const { data, isLoading } = useQuery({
    queryKey: ["products", { search, category, sort, page, perPage }],
    queryFn: () => fetchProducts({ search, category, sort, page, perPage }),
  });
  // 필터가 바뀌면 queryKey가 바뀜 → 자동으로 새 데이터 로드!
}
```

## 6. 실습

```
과제: Zustand 심화 실전

1. useCartStore (Immer + persist + devtools)

3. useAuthStore (비동기 액션)
   - login/logout 비동기 처리
   - isLoading, error 상태 관리
```

## 면접 대비

### 🔴 필수 질문

**Q1. Zustand에서 Immer 미들웨어를 쓰는 이유?**

> 중첩된 객체를 업데이트할 때 스프레드 연산자를 여러 단계로 사용하는 것은 가독성이 떨어지고 실수하기 쉽다. Immer를 사용하면 `state.user.address.city = "부산"`처럼 직접 변경하는 것처럼 작성하되, 내부적으로는 불변 업데이트가 보장된다.

**Q2. Zustand 스토어를 어떻게 분리하나요?**

> 관심사(도메인)별로 스토어를 분리한다. 인증(useAuthStore), 장바구니(useCartStore), 테마(useThemeStore), 필터(useFilterStore) 등으로 나누면 각 스토어가 독립적으로 관리되어 유지보수가 용이하고, 한 스토어의 변경이 다른 스토어의 소비자에게 영향을 주지 않는다.
