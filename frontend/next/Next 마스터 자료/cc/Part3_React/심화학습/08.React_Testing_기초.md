# React 심화 08. React Testing 기초

> **키워드**: `Vitest` `React Testing Library` `render` `screen` `userEvent` `getByRole` `테스트 종류`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 테스트의 종류, RTL 기본 (render/screen/userEvent), 접근성 쿼리 | 채용 시 가산점 |
| 🟡 이해 | 비동기 테스트, MSW 모킹, 커스텀 훅 테스트 | 실무 CI/CD |

## 1. 테스트의 종류 🔴

```
단위 테스트 (Unit Test) — 가장 작은 단위
  함수 하나, 훅 하나, 컴포넌트 하나를 격리하여 테스트
  도구: Vitest, Jest
  "이 함수에 3을 넣으면 6이 나오는가?"

통합 테스트 (Integration Test) — 여러 단위의 조합
  컴포넌트 간 상호작용, API 연동을 테스트
  도구: Vitest + React Testing Library (RTL)
  "버튼을 클릭하면 모달이 열리고 데이터가 표시되는가?"

E2E 테스트 (End-to-End) — 사용자 관점
  실제 브라우저에서 전체 시나리오를 테스트
  도구: Playwright, Cypress
  "회원가입 → 로그인 → 상품 구매 전체 과정이 되는가?"

┌───────────────────┐
│    E2E (느림)      │  ← 가장 적게 (핵심 시나리오만)
├───────────────────┤
│  통합 (보통)       │  ← React 테스트의 메인!
├───────────────────┤
│ 단위 (빠름)        │  ← 유틸 함수, 커스텀 훅
└───────────────────┘
  테스트 피라미드: 아래로 갈수록 많이, 위로 갈수록 적게
```

## 3. 첫 번째 테스트 🔴

```tsx
// Counter.tsx
function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>카운트: {count}</p>
      <button onClick={() => setCount(c => c + 1)}>증가</button>
      <button onClick={() => setCount(0)}>리셋</button>
    </div>
  );
}

// Counter.test.tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import Counter from "./Counter";

describe("Counter", () => {
  it("초기값은 0이다", () => {
    render(<Counter />);
    expect(screen.getByText("카운트: 0")).toBeInTheDocument();
  });

  it("증가 버튼을 클릭하면 카운트가 올라간다", async () => {
    const user = userEvent.setup();
    render(<Counter />);

    await user.click(screen.getByRole("button", { name: "증가" }));
    expect(screen.getByText("카운트: 1")).toBeInTheDocument();

    await user.click(screen.getByRole("button", { name: "증가" }));
    expect(screen.getByText("카운트: 2")).toBeInTheDocument();
  });

  it("리셋 버튼을 클릭하면 0으로 돌아간다", async () => {
    const user = userEvent.setup();
    render(<Counter />);

    await user.click(screen.getByRole("button", { name: "증가" }));
    await user.click(screen.getByRole("button", { name: "증가" }));
    await user.click(screen.getByRole("button", { name: "리셋" }));

    expect(screen.getByText("카운트: 0")).toBeInTheDocument();
  });
});
```

### 실행

```bash
npx vitest        # 워치 모드 (파일 변경 시 자동 재실행)
npx vitest run    # 1회 실행
```

## 4. RTL 핵심 원칙 — "사용자처럼 테스트하라!" 🔴

```
RTL의 철학:
  "사용자가 보고 상호작용하는 것을 기준으로 테스트하라!"
  → DOM 구현 세부사항(className, state 값)이 아닌
  → 사용자가 보는 것(텍스트, 역할, 라벨)으로 요소를 찾아라!
```

### 쿼리 우선순위 (공식 권장 순서!)

```tsx
// ⭐ 1순위: 접근성 기반 (가장 추천!)
screen.getByRole("button", { name: "제출" });    // 역할 + 이름
screen.getByLabelText("이메일");                   // label 연결된 input
screen.getByPlaceholderText("검색어 입력...");      // placeholder
screen.getByText("안녕하세요");                     // 보이는 텍스트
screen.getByDisplayValue("현재 입력값");             // input의 현재 value

// 2순위: 시맨틱 기반
screen.getByAltText("프로필 사진");                 // img의 alt
screen.getByTitle("닫기");                          // title 속성

// 3순위: 테스트 전용 (다른 방법이 없을 때만!)
screen.getByTestId("custom-element");              // data-testid 속성
// ⚠️ testid는 최후의 수단! 사용자는 testid를 모른다!
```

### get / query / find 차이

```tsx
// getByX: 요소가 반드시 있어야 함 (없으면 에러!)
screen.getByText("제목");

// queryByX: 요소가 없을 수 있음 (없으면 null 반환)
expect(screen.queryByText("에러")).not.toBeInTheDocument();  // "없음"을 테스트!

// findByX: 비동기! 요소가 나타날 때까지 기다림 (Promise 반환)
const element = await screen.findByText("로딩 완료!");  // 비동기 렌더링 후 확인
```

## 5. 폼 테스트 🟡

```tsx
// LoginForm.test.tsx
describe("LoginForm", () => {
  it("유효한 데이터로 제출하면 onSubmit이 호출된다", async () => {
    const user = userEvent.setup();
    const handleSubmit = vi.fn();  // Mock 함수!

    render(<LoginForm onSubmit={handleSubmit} />);

    // 입력
    await user.type(screen.getByLabelText("이메일"), "test@test.com");
    await user.type(screen.getByLabelText("비밀번호"), "password123");

    // 제출
    await user.click(screen.getByRole("button", { name: "로그인" }));

    // 검증
    expect(handleSubmit).toHaveBeenCalledWith({
      email: "test@test.com",
      password: "password123",
    });
    expect(handleSubmit).toHaveBeenCalledTimes(1);
  });

  it("이메일이 비어있으면 에러 메시지를 표시한다", async () => {
    const user = userEvent.setup();
    render(<LoginForm onSubmit={vi.fn()} />);

    // 비밀번호만 입력하고 제출
    await user.type(screen.getByLabelText("비밀번호"), "password123");
    await user.click(screen.getByRole("button", { name: "로그인" }));

    // 에러 메시지 확인
    expect(screen.getByText("이메일을 입력하세요")).toBeInTheDocument();
  });
});
```

## 6. 비동기 테스트 (API 데이터) 🟡

```tsx
// MSW (Mock Service Worker)로 API 모킹
import { http, HttpResponse } from "msw";
import { setupServer } from "msw/node";

const server = setupServer(
  http.get("/api/users", () => {
    return HttpResponse.json([
      { id: 1, name: "철수" },
      { id: 2, name: "영희" },
    ]);
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

describe("UserList", () => {
  it("사용자 목록을 불러와서 표시한다", async () => {
    render(<UserList />);

    // 로딩 표시 확인
    expect(screen.getByText("로딩 중...")).toBeInTheDocument();

    // 데이터 로드 후 확인 (findBy = 비동기 대기!)
    expect(await screen.findByText("철수")).toBeInTheDocument();
    expect(await screen.findByText("영희")).toBeInTheDocument();

    // 로딩이 사라졌는지
    expect(screen.queryByText("로딩 중...")).not.toBeInTheDocument();
  });

  it("API 에러 시 에러 메시지를 표시한다", async () => {
    // 이 테스트에서만 에러 응답으로 오버라이드!
    server.use(
      http.get("/api/users", () => {
        return new HttpResponse(null, { status: 500 });
      })
    );

    render(<UserList />);

    expect(await screen.findByText(/에러/)).toBeInTheDocument();
  });
});
```

## 7. 실전 테스트 패턴 🔴

### 무엇을 테스트해야 하는가?

```
✅ 테스트해야 하는 것:
  - 사용자가 보는 것 (텍스트, 버튼, 입력 필드)
  - 사용자의 행동 결과 (클릭 → 모달 열림, 입력 → 에러 표시)
  - 조건부 렌더링 (로그인 시 프로필, 미로그인 시 로그인 버튼)
  - 에러 상황 (API 실패 시 에러 메시지)

❌ 테스트하지 않아도 되는 것:
  - 구현 세부사항 (state 값, 내부 함수 호출)
  - 스타일 (className이 뭔지)
  - 외부 라이브러리의 동작 (TanStack Query의 캐싱이 잘 되는지)
```

### 🟡 이해 질문

**Q1. 프론트엔드에서 테스트를 작성하는 이유?**

> 첫째, 리팩토링 안전망 — 코드를 변경해도 기존 기능이 깨지지 않았음을 확인할 수 있다. 둘째, 문서화 — 테스트 코드가 컴포넌트의 기대 동작을 설명한다. 셋째, CI/CD — 배포 전 자동으로 테스트를 실행하여 버그가 프로덕션에 나가는 것을 방지한다.

**Q2. React Testing Library의 철학?**

> "사용자가 보고 상호작용하는 것을 기준으로 테스트하라." DOM 구현 세부사항(state 값, className)이 아닌 사용자가 보는 것(텍스트, 역할, 라벨)으로 요소를 찾고, 사용자가 하는 행동(클릭, 타이핑)으로 상호작용하며, 사용자가 보는 결과(화면 변화)를 검증한다.
