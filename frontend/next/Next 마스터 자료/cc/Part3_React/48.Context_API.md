## 1. Context가 해결하는 문제 🔴

> 💡 44번에서 배운 Props Drilling의 해결책!
> 중간 컴포넌트를 거치지 않고 **어디서든** 데이터에 접근할 수 있게 한다.

```
Props Drilling:                   Context:
App ─── user ──▶ Layout          App (Provider: user)
                 │                    │
Layout ── user ──▶ Header        Header ← useContext(직접 접근!)
                   │
Header ── user ──▶ Avatar        Avatar ← useContext(직접 접근!)

중간 Layout이 user를 전달만 함!    Layout은 user를 모름! (깔끔!)
```

## 2. Context 기본 사용법 🔴

### 3단계: 생성 → 제공 → 소비

```tsx
import { createContext, useContext, useState, useCallback } from "react";

// ========== 1단계: Context 생성 ==========
type ThemeContextType = {
  theme: "light" | "dark";
  toggleTheme: () => void;
};

const ThemeContext = createContext<ThemeContextType | null>(null);

// ========== 2단계: Provider 컴포넌트 ==========
function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<"light" | "dark">("light");

  const toggleTheme = useCallback(() => {
    setTheme(prev => prev === "light" ? "dark" : "light");
  }, []);

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// ========== 3단계: 커스텀 훅으로 소비 (정석!) ==========
function useTheme() {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error("useTheme은 ThemeProvider 안에서만 사용할 수 있습니다");
  }
  return context;
}
```

### 사용

```tsx
// App: Provider로 감싸기
function App() {
  return (
    <ThemeProvider>
      <Header />
      <Main />
      <Footer />
    </ThemeProvider>
  );
}

// 어떤 깊이의 컴포넌트에서든 직접 접근!
function Header() {
  const { theme, toggleTheme } = useTheme();  // Props Drilling 없이!
  return (
    <header className={theme === "dark" ? "bg-gray-900" : "bg-white"}>
      <button onClick={toggleTheme}>테마 변경</button>
    </header>
  );
}
```

## 3. Context의 리렌더링 문제 🔴

> ⚠️ Context의 가장 큰 단점: **value가 바뀌면 모든 소비자가 리렌더링!**

```tsx
// ❌ 문제: value 객체가 매 렌더링마다 새로 생성
function BadProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState({ name: "철수" });
  const [theme, setTheme] = useState("light");

  return (
    // 이 객체가 매 렌더링마다 새로 만들어짐 → 모든 소비자 리렌더링!
    <MyContext.Provider value={{ user, theme, setUser, setTheme }}>
      {children}
    </MyContext.Provider>
  );
}

// ✅ 해결 1: useMemo로 value 안정화
function BetterProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState({ name: "철수" });
  const [theme, setTheme] = useState("light");

  const value = useMemo(() => ({ user, theme, setUser, setTheme }), [user, theme]);

  return (
    <MyContext.Provider value={value}>
      {children}
    </MyContext.Provider>
  );
}

// ✅ 해결 2: Context를 분리! (가장 추천!)
// 자주 바뀌는 것과 안 바뀌는 것을 별도 Context로!
const UserContext = createContext<...>(null);     // 사용자 정보 (드물게 변경)
const ThemeContext = createContext<...>(null);     // 테마 (드물게 변경)
const CartContext = createContext<...>(null);      // 장바구니 (자주 변경)
// → 장바구니가 바뀌어도 User/Theme 소비자는 리렌더링 안 됨!
```

## 4. Context vs Zustand — 언제 뭘 쓸까? 🟡

```
Context가 적합한 경우:
  ✅ 테마, 로케일(언어), 인증 상태 등 "드물게 변하는" 전역 데이터
  ✅ Provider로 특정 트리 범위만 감싸야 할 때
  ✅ 외부 라이브러리 없이 해결하고 싶을 때

Zustand가 적합한 경우 (53번에서 상세!):
  ✅ 자주 변하는 상태 (장바구니, 폼, 필터 등)
  ✅ 여러 컴포넌트가 같은 상태의 "일부"만 구독해야 할 때 (선택적 리렌더링)
  ✅ Provider 없이 어디서든 접근하고 싶을 때
  ✅ 미들웨어(persist, devtools) 필요할 때

실무 정석 (2026년):
  인증/테마 → Context
  서버 상태 → TanStack Query
  클라이언트 상태 → Zustand
```

## 5. 실전 패턴: 인증 Context 🔴

```tsx
type User = { id: number; name: string; email: string; role: "admin" | "user" };

type AuthContextType = {
  user: User | null;
  isAuthenticated: boolean;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
};

const AuthContext = createContext<AuthContextType | null>(null);

function AuthProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null);

  const login = useCallback(async (email: string, password: string) => {
    const res = await fetch("/api/auth/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ email, password }),
    });
    if (!res.ok) throw new Error("로그인 실패");
    const userData = await res.json();
    setUser(userData);
  }, []);

  const logout = useCallback(() => {
    setUser(null);
    // 쿠키/토큰 삭제 로직
  }, []);

  const value = useMemo(() => ({
    user,
    isAuthenticated: !!user,
    login,
    logout,
  }), [user, login, logout]);

  return (
    <AuthContext.Provider value={value}>
      {children}
    </AuthContext.Provider>
  );
}

function useAuth() {
  const context = useContext(AuthContext);
  if (!context) throw new Error("useAuth는 AuthProvider 안에서만!");
  return context;
}

// 사용
function LoginButton() {
  const { isAuthenticated, user, logout } = useAuth();

  if (isAuthenticated) {
    return (
      <div>
        <span>{user!.name}</span>
        <button onClick={logout}>로그아웃</button>
      </div>
    );
  }

  return <a href="/login">로그인</a>;
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. Context API의 리렌더링 문제와 해결법?**

> Context의 value가 변경되면 해당 Context를 소비하는 모든 컴포넌트가 리렌더링된다. 해결 방법으로 첫째, `useMemo`로 value 객체를 안정화하여 불필요한 참조 변경을 방지한다. 둘째, 자주 변하는 상태와 드물게 변하는 상태를 별도 Context로 분리한다. 셋째, 자주 변하는 전역 상태에는 Context 대신 Zustand 같은 상태 관리 라이브러리를 사용하여 선택적 구독(selector)으로 필요한 컴포넌트만 리렌더링한다.

**Q2. Context를 사용할 때 커스텀 훅을 만드는 이유?**

> `createContext(null)`로 생성 시 소비자에서 null 체크를 매번 해야 한다. 커스텀 훅(`useTheme`, `useAuth`)에서 null 체크를 한 번만 수행하고 에러를 던지면, 소비자 컴포넌트에서는 null 체크 없이 안전하게 값에 접근할 수 있다. Provider 밖에서 사용하는 실수도 즉시 에러로 잡아낸다.
