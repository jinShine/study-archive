## 1. State란? — 컴포넌트의 기억 🔴

> 💡 **비유: 화이트보드**
> 컴포넌트가 "기억해야 할 정보"를 적어두는 화이트보드.
> 화이트보드의 내용이 바뀌면(setState) → 화면이 자동으로 갱신(리렌더링)!

```tsx
import { useState } from "react";

function Counter() {
  // 구조 분해: [현재값, 변경함수] = useState(초기값)
  const [count, setCount] = useState(0);
  //      ↑ 읽기      ↑ 쓰기         ↑ 초기값

  return (
    <div>
      <p>카운트: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
      <button onClick={() => setCount(0)}>리셋</button>
    </div>
  );
}
```

### State의 핵심 규칙 🔴

```
1. State가 바뀌면 → 컴포넌트가 리렌더링된다!
2. State는 직접 변경하면 안 된다! → setState 함수만 사용
3. State는 비동기적으로 업데이트된다 (배치)
4. 렌더링 중에 State를 변경하면 안 된다 (무한 루프!)
```

```tsx
// ❌ 직접 변경: 리렌더링이 발생하지 않음!
count = count + 1;           // React가 변화를 모름!

// ✅ setState 함수: React에게 "상태가 바뀌었어!" 알림
setCount(count + 1);          // React가 리렌더링 스케줄링!

// ❌ 렌더링 중 State 변경: 무한 루프!
function Bad() {
  const [count, setCount] = useState(0);
  setCount(count + 1);  // 💀 렌더링 → 상태 변경 → 리렌더링 → 상태 변경 → 무한!
  return <p>{count}</p>;
}
```

## 2. 다양한 타입의 State 🔴

```tsx
// 숫자
const [count, setCount] = useState(0);

// 문자열
const [name, setName] = useState("");

// 불린
const [isOpen, setIsOpen] = useState(false);

// 배열 (불변 업데이트 필수!)
const [items, setItems] = useState<string[]>([]);
setItems([...items, "새 항목"]);           // 추가
setItems(items.filter(i => i !== "삭제")); // 삭제

// 객체 (스프레드로 복사 후 수정!)
const [user, setUser] = useState({ name: "", email: "" });
setUser({ ...user, name: "새이름" });      // 일부만 수정

// null 허용 (제네릭 명시!)
const [selected, setSelected] = useState<Product | null>(null);

// 복잡한 객체
const [form, setForm] = useState({
  name: "",
  email: "",
  address: {
    city: "",
    zip: "",
  },
});
// 중첩 객체 수정
setForm({
  ...form,
  address: { ...form.address, city: "서울" },
});
```

## 3. 불변성 — React State의 절대 규칙 🔴

> Part 2의 22번(불변성)에서 배운 내용이 여기서 실전으로!

```tsx
const [todos, setTodos] = useState([
  { id: 1, text: "공부", done: false },
  { id: 2, text: "운동", done: false },
]);

// ❌ 원본 직접 변경: React가 변화를 감지 못함!
todos[0].done = true;
setTodos(todos);  // 같은 참조 → React: "안 바뀌었네?" → 리렌더링 안 됨!

// ✅ 새 배열 생성: 다른 참조 → React: "바뀌었다!" → 리렌더링!
setTodos(todos.map(todo =>
  todo.id === 1 ? { ...todo, done: true } : todo
));

// 불변 업데이트 패턴 총정리:
// 추가: setItems(prev => [...prev, newItem])
// 삭제: setItems(prev => prev.filter(item => item.id !== id))
// 수정: setItems(prev => prev.map(item => item.id === id ? {...item, ...changes} : item))
// 정렬: setItems(prev => [...prev].sort((a, b) => a.name.localeCompare(b.name)))
```

## 4. 함수형 업데이트 (이전 값 기반) 🔴

> ⚠️ 같은 이벤트 핸들러에서 setState를 여러 번 호출할 때 중요!

```tsx
// ❌ 문제: 배치 업데이트 때문에 3이 아니라 1만 증가!
function handleClick() {
  setCount(count + 1);  // 0 + 1 = 1 (count는 아직 0!)
  setCount(count + 1);  // 0 + 1 = 1 (count는 여전히 0!)
  setCount(count + 1);  // 0 + 1 = 1 (count는 여전히 0!)
  // 결과: 1 (3이 아님!)
}

// ✅ 해결: 함수형 업데이트 (이전 값을 콜백으로 받음)
function handleClick() {
  setCount(prev => prev + 1);  // 0 → 1
  setCount(prev => prev + 1);  // 1 → 2
  setCount(prev => prev + 1);  // 2 → 3
  // 결과: 3 ✅
}
```

> 💡 **실무 원칙**: 이전 상태를 기반으로 업데이트할 때는 **항상 콜백 형태**를 써라!
> `setCount(prev => prev + 1)` > `setCount(count + 1)`

## 5. 배치 업데이트 (Batching) 🟡

> React는 성능을 위해 여러 setState를 **모아서 한 번에** 리렌더링한다.

```tsx
function handleClick() {
  setName("철수");     // 리렌더링 안 함 (아직!)
  setAge(25);          // 리렌더링 안 함 (아직!)
  setEmail("a@b.com"); // 리렌더링 안 함 (아직!)
  // 핸들러 끝 → 3개의 변경을 모아서 리렌더링 1번만!
}
```

> 💡 React 18+에서는 setTimeout, fetch 콜백 등에서도 자동 배칭이 된다.
> (React 17 이전에는 이벤트 핸들러 안에서만 배칭됐다)

## 6. 이벤트 핸들링 🔴

### 기본 패턴

```tsx
function App() {
  // 패턴 1: 인라인 화살표 함수 (간단한 로직)
  return <button onClick={() => console.log("클릭!")}>클릭</button>;

  // 패턴 2: 핸들러 함수 분리 (복잡한 로직)
  const handleClick = () => {
    console.log("클릭!");
    // 복잡한 로직...
  };
  return <button onClick={handleClick}>클릭</button>;

  // ❌ 즉시 실행 주의! 함수를 "전달"해야지 "호출"하면 안 됨!
  return <button onClick={handleClick()}>클릭</button>;  // 💀 렌더링 시 즉시 실행!
  return <button onClick={handleClick}>클릭</button>;    // ✅ 클릭 시 실행
}
```

### 이벤트 객체와 매개변수

```tsx
// 이벤트 객체 받기
const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  console.log(e.target.value);    // 입력값
  console.log(e.target.name);     // input의 name 속성
};

// 추가 매개변수가 필요할 때
const handleDelete = (id: number) => {
  setItems(prev => prev.filter(item => item.id !== id));
};

// 사용: 화살표 함수로 감싸서 인자 전달
{items.map(item => (
  <button key={item.id} onClick={() => handleDelete(item.id)}>
    삭제
  </button>
))}
```

### 이벤트 핸들러 네이밍 관례

```tsx
// 컴포넌트 내부 핸들러: handle + 동사
const handleClick = () => { };
const handleSubmit = () => { };
const handleChange = () => { };
const handleDelete = (id: number) => { };

// Props로 받는 콜백: on + 동사
type Props = {
  onClick: () => void;
  onSubmit: (data: FormData) => void;
  onChange: (value: string) => void;
  onDelete: (id: number) => void;
};

// 조합 예시
function SearchBar({ onSearch }: { onSearch: (query: string) => void }) {
  const [query, setQuery] = useState("");

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault();
    onSearch(query);  // 부모에게 검색어 전달
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <button type="submit">검색</button>
    </form>
  );
}
```

## 7. 합성 이벤트 (SyntheticEvent) 🟡

```tsx
// React는 브라우저의 네이티브 이벤트를 감싸는 "합성 이벤트"를 사용
// → 모든 브라우저에서 동일하게 동작 (크로스 브라우저 호환)

const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => {
  e.preventDefault();   // 기본 동작 방지 (a 태그 이동, form 제출 등)
  e.stopPropagation();  // 이벤트 버블링 중단
  e.target;             // 실제 클릭된 요소
  e.currentTarget;      // 핸들러가 붙은 요소
  e.nativeEvent;        // 원본 브라우저 이벤트에 접근 (드물게 필요)
};
```

## 8. 실습: 할 일 관리 앱 (CRUD)

```
과제: useState만으로 할 일(Todo) 앱 구현

기능:
1. 새 할 일 추가 (input + 엔터 또는 추가 버튼)
2. 완료 토글 (클릭하면 취소선)
3. 삭제
4. 전체 개수 / 완료 개수 표시
5. 모든 상태 업데이트는 불변성 지켜서!

타입:
type Todo = { id: number; text: string; done: boolean };
```

<details>
<summary>정답 보기</summary>

```tsx
type Todo = { id: number; text: string; done: boolean };

function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [input, setInput] = useState("");

  const handleAdd = () => {
    if (!input.trim()) return;
    setTodos(prev => [...prev, { id: Date.now(), text: input.trim(), done: false }]);
    setInput("");
  };

  const handleToggle = (id: number) => {
    setTodos(prev => prev.map(todo =>
      todo.id === id ? { ...todo, done: !todo.done } : todo
    ));
  };

  const handleDelete = (id: number) => {
    setTodos(prev => prev.filter(todo => todo.id !== id));
  };

  const handleKeyDown = (e: React.KeyboardEvent) => {
    if (e.key === "Enter") handleAdd();
  };

  const doneCount = todos.filter(t => t.done).length;

  return (
    <div>
      <h1>할 일 ({doneCount}/{todos.length} 완료)</h1>

      <div>
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          onKeyDown={handleKeyDown}
          placeholder="할 일 입력"
        />
        <button onClick={handleAdd}>추가</button>
      </div>

      <ul>
        {todos.map(todo => (
          <li key={todo.id}>
            <span
              onClick={() => handleToggle(todo.id)}
              style={{
                textDecoration: todo.done ? "line-through" : "none",
                color: todo.done ? "#999" : "#000",
                cursor: "pointer",
              }}
            >
              {todo.text}
            </span>
            <button onClick={() => handleDelete(todo.id)}>삭제</button>
          </li>
        ))}
      </ul>

      {todos.length === 0 && <p>할 일이 없습니다!</p>}
    </div>
  );
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. React에서 상태(State)를 직접 변경하면 안 되는 이유?**

> React는 상태 변화를 참조 비교(`===`)로 감지한다. 원본 객체/배열을 직접 변경하면 참조가 동일하므로 React가 변화를 인식하지 못해 리렌더링이 발생하지 않는다. `setState`로 새로운 참조(스프레드 연산자 등)를 생성해야 React가 변화를 감지하고 UI를 업데이트한다.

**Q2. `setCount(count + 1)`과 `setCount(prev => prev + 1)`의 차이?**

> `setCount(count + 1)`은 현재 렌더링 시점의 `count` 값(클로저)을 사용한다. 같은 이벤트에서 여러 번 호출해도 같은 `count` 값을 참조하므로 한 번만 증가한다. `setCount(prev => prev + 1)`은 React가 보장하는 최신 상태값(`prev`)을 기반으로 계산하므로, 여러 번 호출하면 정확히 그만큼 증가한다. 이전 상태에 의존하는 업데이트에는 반드시 콜백 형태를 사용해야 한다.

**Q3. React의 배치 업데이트(Batching)란?**

> React가 여러 `setState` 호출을 하나로 모아서 한 번의 리렌더링만 수행하는 최적화 기법이다. React 18부터는 이벤트 핸들러뿐 아니라 setTimeout, fetch 콜백 등 모든 곳에서 자동 배칭이 적용된다. 이를 통해 불필요한 리렌더링을 방지하여 성능을 향상시킨다.
