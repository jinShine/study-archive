## 1. 합성(Composition) > 상속(Inheritance) 🔴

> React 공식 문서: "React는 강력한 합성 모델을 가지고 있으며, 상속 대신 합성을 사용하여 컴포넌트 간에 코드를 재사용하는 것을 권장합니다."

```tsx
// ❌ 상속 사고방식: "특수한 Card"를 만들려면 Card를 상속?
// React에서 클래스 상속은 쓰지 않는다!

// ✅ 합성 사고방식: Card에 다양한 내용을 "끼워 넣기"
function Card({ title, children, footer }: {
  title: string;
  children: React.ReactNode;
  footer?: React.ReactNode;
}) {
  return (
    <div className="border rounded-lg overflow-hidden">
      <div className="p-4 border-b bg-gray-50">
        <h3 className="font-bold">{title}</h3>
      </div>
      <div className="p-4">{children}</div>
      {footer && <div className="p-4 border-t bg-gray-50">{footer}</div>}
    </div>
  );
}

// 사용: 같은 Card인데 내용이 전혀 다르다!
<Card title="프로필">
  <UserProfile user={user} />
</Card>

<Card title="통계" footer={<Button>새로고침</Button>}>
  <Chart data={salesData} />
</Card>

<Card title="알림">
  <NotificationList items={notifications} />
</Card>
```

### 합성의 3가지 패턴

```
1. children: 가장 기본. 태그 사이에 무엇이든 넣기
2. 명시적 slot: header, footer, sidebar 등 이름 붙은 영역
3. render prop: 함수를 넘겨서 렌더링 위임
```

## 2. Compound Component 패턴 🟡

> 💡 **비유: HTML의 `<select>` + `<option>`**
> `<select>`와 `<option>`은 함께 써야 의미가 있고, 상태를 암묵적으로 공유한다.
> 이처럼 여러 컴포넌트가 하나의 기능을 위해 함께 동작하는 패턴.

```tsx
// 사용 모습 (선언적이고 직관적!)
<Accordion>
  <Accordion.Item>
    <Accordion.Trigger>섹션 1</Accordion.Trigger>
    <Accordion.Content>섹션 1의 내용...</Accordion.Content>
  </Accordion.Item>
  <Accordion.Item>
    <Accordion.Trigger>섹션 2</Accordion.Trigger>
    <Accordion.Content>섹션 2의 내용...</Accordion.Content>
  </Accordion.Item>
</Accordion>
```

### 간단한 구현 (Tabs 컴포넌트)

```tsx
type TabsContextType = {
  activeTab: string;
  setActiveTab: (tab: string) => void;
};

const TabsContext = createContext<TabsContextType | null>(null);

function useTabs() {
  const ctx = useContext(TabsContext);
  if (!ctx) throw new Error("Tabs 컴포넌트 안에서만 사용하세요");
  return ctx;
}

// 부모: 상태 관리
function Tabs({ defaultTab, children }: { defaultTab: string; children: React.ReactNode }) {
  const [activeTab, setActiveTab] = useState(defaultTab);

  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab }}>
      <div>{children}</div>
    </TabsContext.Provider>
  );
}

// 탭 버튼 목록
function TabList({ children }: { children: React.ReactNode }) {
  return <div className="flex border-b">{children}</div>;
}

// 개별 탭 버튼
function Tab({ value, children }: { value: string; children: React.ReactNode }) {
  const { activeTab, setActiveTab } = useTabs();
  const isActive = activeTab === value;

  return (
    <button
      onClick={() => setActiveTab(value)}
      className={`px-4 py-2 ${isActive ? "border-b-2 border-blue-500 font-bold" : ""}`}
    >
      {children}
    </button>
  );
}

// 탭 내용
function TabPanel({ value, children }: { value: string; children: React.ReactNode }) {
  const { activeTab } = useTabs();
  if (activeTab !== value) return null;
  return <div className="p-4">{children}</div>;
}

// 합체!
Tabs.List = TabList;
Tabs.Tab = Tab;
Tabs.Panel = TabPanel;

// 사용
<Tabs defaultTab="profile">
  <Tabs.List>
    <Tabs.Tab value="profile">프로필</Tabs.Tab>
    <Tabs.Tab value="settings">설정</Tabs.Tab>
    <Tabs.Tab value="billing">결제</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panel value="profile">프로필 내용...</Tabs.Panel>
  <Tabs.Panel value="settings">설정 내용...</Tabs.Panel>
  <Tabs.Panel value="billing">결제 내용...</Tabs.Panel>
</Tabs>
```

> 💡 **실무에서**: shadcn/ui, Radix UI, Headless UI가 이 패턴으로 만들어져 있다!

## 3. Render Props 패턴 🟡

> "무엇을 렌더링할지"를 함수(Props)로 받아서 호출 시점에 결정하는 패턴

```tsx
// 마우스 위치를 추적하는 범용 컴포넌트
function MouseTracker({ render }: { render: (pos: { x: number; y: number }) => React.ReactNode }) {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handler = (e: MouseEvent) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener("mousemove", handler);
    return () => window.removeEventListener("mousemove", handler);
  }, []);

  return <>{render(position)}</>;
}

// 사용: 같은 로직으로 다른 UI를 렌더링!
<MouseTracker render={({ x, y }) => (
  <p>마우스 위치: ({x}, {y})</p>
)} />

<MouseTracker render={({ x, y }) => (
  <div style={{ position: "fixed", left: x, top: y }}>👆 커서</div>
)} />
```

> 💡 **현재는 커스텀 훅이 대부분 대체!**
> `useMousePosition()` 훅으로 만드는 것이 더 깔끔하다.
> Render Props는 레거시 코드에서 만나거나, children에 데이터를 전달할 때 가끔 사용.

## 4. 컴포넌트 설계 원칙 정리 🔴

### 원칙 1: 단일 책임 (Single Responsibility)

```
하나의 컴포넌트는 하나의 역할만.
"이 컴포넌트가 하는 일을 한 문장으로 설명할 수 있는가?"

✅ UserAvatar: 사용자 프로필 이미지를 표시한다
✅ SearchInput: 검색어를 입력받는다
✅ ProductCard: 상품 정보를 카드 형태로 표시한다

❌ DashboardWithSidebarAndUserProfileAndChart: ???
```

### 원칙 2: Props 인터페이스를 최소화

```tsx
// ❌ 너무 많은 Props → 컴포넌트가 너무 많은 것을 안다
<Button
  text="저장"
  textColor="white"
  bgColor="blue"
  fontSize={16}
  fontWeight="bold"
  borderRadius={8}
  paddingX={16}
  paddingY={8}
  onClick={handleSave}
  isLoading={loading}
  loadingText="저장 중..."
  disabled={!isValid}
  icon="save"
  iconPosition="left"
/>

// ✅ 핵심 Props만 + 나머지는 variant/children으로
<Button variant="primary" onClick={handleSave} loading={loading} disabled={!isValid}>
  <SaveIcon /> 저장
</Button>
```

### 원칙 3: 구현이 아니라 의도를 표현

```tsx
// ❌ 구현 세부사항이 이름에 드러남
<div className="flex-container">
<RedBoldText>

// ✅ 의도가 이름에 드러남
<CardGrid>
<ErrorMessage>
<NavigationMenu>
<PriceDisplay>
```

### 원칙 4: 재사용 가능한 것과 특수한 것을 분리

```
범용 컴포넌트 (UI 라이브러리):
  Button, Input, Card, Modal, Badge, Avatar, Tooltip
  → 비즈니스 로직 없음! 어디서든 재사용 가능

도메인 컴포넌트 (이 프로젝트 전용):
  ProductCard, UserProfile, OrderTable, CartSidebar
  → 비즈니스 데이터와 로직 포함

페이지 컴포넌트 (최상위):
  HomePage, ProductDetailPage, CheckoutPage
  → 데이터 페칭 + 하위 컴포넌트 조립
```

## 5. Headless UI 패턴 🟢

> "로직은 제공하되, UI(스타일)는 완전히 사용자에게 위임"

```
대표 라이브러리:
  - Headless UI (@headlessui/react)
  - Radix UI (radix-ui/primitives)
  - React Aria (react-aria)
  - shadcn/ui (Radix + Tailwind CSS) ← 2026년 가장 인기!

원리:
  라이브러리가 접근성(a11y), 키보드 네비게이션, 상태 관리를 담당
  개발자가 Tailwind CSS 등으로 원하는 디자인을 입힘
```

```tsx
// shadcn/ui 스타일 (Headless + Tailwind)
import { Dialog, DialogContent, DialogTitle, DialogTrigger } from "@/components/ui/dialog";

<Dialog>
  <DialogTrigger asChild>
    <Button>모달 열기</Button>
  </DialogTrigger>
  <DialogContent>
    <DialogTitle>확인</DialogTitle>
    <p>정말 삭제하시겠습니까?</p>
  </DialogContent>
</Dialog>
// → 접근성, 키보드, 포커스 트랩 등은 라이브러리가 처리
// → 디자인은 Tailwind로 원하는대로!
```

## 6. 실습

```
과제: 재사용 가능한 Modal 시스템 설계

1. Modal (기본 껍데기): overlay + 중앙 컨테이너 + ESC 닫기
2. Modal.Header: 제목 + 닫기 버튼
3. Modal.Body: 본문 영역 (children)
4. Modal.Footer: 버튼 영역 (children)
5. useModal 커스텀 훅: isOpen, open, close
6. Compound Component 패턴 적용
7. 키보드(ESC) + 오버레이 클릭으로 닫기
```

## 면접 대비

### 🔴 필수 질문

**Q1. 합성(Composition)이란?**

> 작은 컴포넌트를 조합하여 복잡한 UI를 구축하는 React의 핵심 설계 방식이다. children Props로 부모가 자식의 내용을 결정하고, 명시적 slot Props(header, footer)로 여러 영역을 주입할 수 있다. React에서는 클래스 상속 대신 합성을 사용하여 코드를 재사용한다.

**Q2. Compound Component 패턴이란?**

> 여러 컴포넌트가 하나의 기능을 위해 암묵적으로 상태를 공유하며 동작하는 패턴이다. HTML의 `<select>` + `<option>` 관계처럼, 내부적으로 Context를 사용하여 부모가 상태를 관리하고 자식들이 접근한다. 선언적이고 유연한 API를 제공하며, shadcn/ui, Radix UI 같은 현대 UI 라이브러리가 이 패턴을 사용한다.
