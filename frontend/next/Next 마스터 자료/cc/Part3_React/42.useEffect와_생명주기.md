## 1. useEffect란? — 렌더링 이후의 부수효과 🔴

> 💡 **비유: 식당의 뒷정리**
> 요리(렌더링)가 끝나고 손님에게 나간 후에,
> 주방 청소, 재료 주문 같은 **뒷정리(부수효과)**를 하는 것.
> 요리 도중에 하면 안 된다! (렌더링 중 부수효과 = 버그)

```
렌더링 = "UI가 어떻게 보여야 하는지" 계산 (순수!)
부수효과 = 그 외 모든 것:
  - API 데이터 가져오기 (fetch)
  - 타이머 설정 (setTimeout/setInterval)
  - DOM 직접 조작
  - 이벤트 리스너 등록
  - 로깅
  - 외부 라이브러리 연동
```

```tsx
useEffect(() => {
  // 이 코드는 렌더링이 끝난 후에 실행된다!
  // (부수효과를 여기에 작성)

  return () => {
    // 클린업: 다음 effect 실행 전 또는 언마운트 시 실행
  };
}, [의존성]);  // 이 값이 바뀔 때만 effect 재실행
```

## 2. 의존성 배열 3가지 패턴 🔴

### 패턴 1: 빈 배열 `[]` — 마운트 시 1번만 실행

```tsx
// 컴포넌트가 처음 화면에 나타날 때 딱 1번만!
useEffect(() => {
  console.log("마운트됨! (1번만 실행)");

  return () => {
    console.log("언마운트됨! (화면에서 사라질 때)");
  };
}, []);  // 빈 배열 = 의존성 없음 = 변화 감지 대상 없음 = 1번만!

// 실전 활용:
// - 초기 데이터 로드 (API 호출)
// - 이벤트 리스너 등록/해제
// - 외부 라이브러리 초기화
```

### 패턴 2: 의존성 배열에 값 `[dep]` — 값이 바뀔 때마다 실행

```tsx
// userId가 바뀔 때마다 새로운 사용자 데이터를 가져옴
useEffect(() => {
  console.log(`userId가 ${userId}로 바뀜! 데이터 다시 로드`);
  fetchUser(userId);

  return () => {
    console.log(`이전 userId(${userId})의 effect 정리`);
  };
}, [userId]);  // userId가 바뀔 때만 재실행

// 실전 활용:
// - 검색어 변경 시 API 재호출
// - 라우트 파라미터 변경 시 데이터 갱신
// - Props 변경에 따른 외부 동기화
```

### 패턴 3: 의존성 배열 없음 — 매 렌더링마다 실행

```tsx
// ⚠️ 거의 사용하지 않음! 매 렌더링마다 실행되어 성능 저하 위험
useEffect(() => {
  console.log("매번 실행됨 (모든 렌더링 후)");
});  // 의존성 배열 자체가 없음!

// 드물게 사용하는 경우: 디버깅용 로그
```

### 한눈에 비교

```
useEffect(fn)           → 매 렌더링 후 실행 (거의 안 씀)
useEffect(fn, [])       → 마운트 시 1번만 (초기 로드)
useEffect(fn, [a, b])   → a 또는 b가 바뀔 때 (반응형)
```

## 3. 클린업 함수 — 뒷정리의 기술 🔴

> 💡 **비유: 체크아웃**
> 호텔 방(컴포넌트)을 나갈 때 불 끄고, 수건 정리하는 것.
> 안 하면 다음 손님(새 렌더링)에게 문제가 발생!

### 클린업이 필요한 경우

```tsx
// 1. 이벤트 리스너 등록/해제
useEffect(() => {
  const handleResize = () => setWidth(window.innerWidth);
  window.addEventListener("resize", handleResize);

  return () => {
    window.removeEventListener("resize", handleResize);  // 반드시 해제!
    // 안 하면: 언마운트 후에도 리스너가 남아서 메모리 누수!
  };
}, []);

// 2. 타이머 설정/해제
useEffect(() => {
  const timerId = setInterval(() => {
    setSeconds(prev => prev + 1);
  }, 1000);

  return () => {
    clearInterval(timerId);  // 반드시 해제!
    // 안 하면: 컴포넌트가 사라져도 타이머가 계속 돌아감!
  };
}, []);

// 3. API 요청 취소 (AbortController)
useEffect(() => {
  const controller = new AbortController();

  fetch(`/api/users/${userId}`, { signal: controller.signal })
    .then(res => res.json())
    .then(data => setUser(data))
    .catch(err => {
      if (err.name !== "AbortError") console.error(err);
    });

  return () => {
    controller.abort();  // 이전 요청 취소 (Race Condition 방지!)
  };
}, [userId]);

// 4. 구독(Subscription) 해제
useEffect(() => {
  const unsubscribe = store.subscribe(() => {
    setData(store.getState());
  });

  return () => unsubscribe();
}, []);
```

### 클린업 실행 시점

```
컴포넌트 마운트:
  1. 렌더링
  2. Effect 실행 ← (처음에는 클린업 없음)

의존성 변경 (userId: 1 → 2):
  1. 렌더링
  2. 이전 Effect의 클린업 실행 ← (userId=1의 뒷정리)
  3. 새 Effect 실행 ← (userId=2의 새 Effect)

컴포넌트 언마운트:
  1. 마지막 Effect의 클린업 실행 ← (최종 뒷정리)
```

## 4. StrictMode의 더블 호출 — 의도된 것! 🟡

```tsx
// 개발 모드에서 StrictMode는 모든 Effect를 2번 실행한다!
// 목적: 클린업을 제대로 안 했을 때 버그를 빨리 발견하게 하려고

// 실행 순서 (개발 모드):
// 1. 마운트 → Effect 실행
// 2. 언마운트 시뮬레이션 → 클린업 실행
// 3. 다시 마운트 → Effect 다시 실행

// 프로덕션에서는 1번만 실행! 걱정하지 마라!
// 단, 클린업을 제대로 안 했으면 개발 중에 이상한 동작이 보인다
// → 그것이 바로 StrictMode의 존재 이유!
```

> 💡 **면접 포인트**: "StrictMode에서 useEffect가 2번 실행되는 건 의도된 동작입니다.
> 클린업 함수가 제대로 작동하는지 검증하기 위한 React의 개발 도구이며,
> 프로덕션에서는 1번만 실행됩니다."

## 5. useEffect 흔한 실수 🔴

### 실수 1: 무한 루프

```tsx
// ❌ 의존성에 매 렌더링마다 바뀌는 값
useEffect(() => {
  setCount(count + 1);  // 상태 변경 → 리렌더링 → Effect 재실행 → 상태 변경 → ♾️
}, [count]);  // count가 바뀔 때마다 실행... 그런데 안에서 count를 바꿈!

// ❌ 객체/배열을 의존성에 직접 넣기
const options = { page: 1, limit: 10 };  // 매 렌더링마다 새 객체!
useEffect(() => {
  fetchData(options);
}, [options]);  // 참조가 매번 달라서 무한 루프!

// ✅ 해결: 원시값을 의존성으로
useEffect(() => {
  fetchData({ page, limit });
}, [page, limit]);  // 원시값은 변하지 않으면 같은 값
```

### 실수 2: 의존성 누락

```tsx
// ❌ ESLint 경고 무시: 의존성 배열에 count가 없음!
useEffect(() => {
  const id = setInterval(() => {
    console.log(count);  // 항상 초기값(0)만 출력! (클로저!)
  }, 1000);
  return () => clearInterval(id);
}, []);  // count를 안 넣었으므로 최초의 count(0)만 기억함

// ✅ 해결 1: 함수형 업데이트
useEffect(() => {
  const id = setInterval(() => {
    setCount(prev => prev + 1);  // 최신 값 사용!
  }, 1000);
  return () => clearInterval(id);
}, []);  // setCount는 안정적이므로 의존성 불필요

// ✅ 해결 2: 의존성에 포함
useEffect(() => {
  console.log(count);
}, [count]);  // count가 바뀔 때마다 실행
```

### 실수 3: useEffect 안에서 async 직접 사용

```tsx
// ❌ useEffect 콜백 자체를 async로 만들면 안 됨!
useEffect(async () => {  // 💥 에러! (Promise를 반환하므로 클린업으로 인식)
  const data = await fetchData();
  setData(data);
}, []);

// ✅ 내부에 async 함수를 선언하고 호출
useEffect(() => {
  const loadData = async () => {
    try {
      const data = await fetchData();
      setData(data);
    } catch (err) {
      setError(err);
    } finally {
      setLoading(false);
    }
  };
  loadData();
}, []);
```

## 6. "useEffect가 필요하지 않을 수도 있다" 🔴

> ⚠️ React 공식 문서의 핵심 조언: **"You Might Not Need an Effect"**

```tsx
// ❌ 파생 상태를 useEffect로 동기화 (흔한 실수!)
const [items, setItems] = useState<Product[]>([]);
const [filteredItems, setFilteredItems] = useState<Product[]>([]);

useEffect(() => {
  setFilteredItems(items.filter(i => i.price > 1000));
}, [items]);
// 문제: 불필요한 렌더링 2번! (items 변경 → 리렌더 → effect → filteredItems 변경 → 리렌더)

// ✅ 렌더링 중에 직접 계산 (파생 상태!)
const [items, setItems] = useState<Product[]>([]);
const filteredItems = items.filter(i => i.price > 1000);  // 렌더링 때 바로 계산!

// ❌ 이벤트에 대한 반응을 useEffect로 (또 다른 흔한 실수!)
useEffect(() => {
  if (submitted) {
    sendAnalytics("form_submitted");
    navigate("/success");
  }
}, [submitted]);

// ✅ 이벤트 핸들러에서 직접 처리!
const handleSubmit = () => {
  setSubmitted(true);
  sendAnalytics("form_submitted");
  navigate("/success");
};
```

### useEffect가 필요한 경우 vs 불필요한 경우

```
✅ useEffect가 필요한 경우:
  - 외부 시스템과 동기화 (API, WebSocket, DOM, 타이머)
  - 이벤트 리스너 등록/해제
  - 외부 라이브러리 초기화
  - 컴포넌트 마운트 시 초기 데이터 로드

❌ useEffect가 불필요한 경우:
  - 파생 상태 계산 → 렌더링 중 직접 계산!
  - 이벤트에 대한 반응 → 이벤트 핸들러에서 직접!
  - Props 변경에 따른 State 리셋 → key를 바꿔서 리셋!
  - 사용자 이벤트 처리 → onClick 등 핸들러에서!
```

## 7. 실전: 데이터 페칭 패턴 (useEffect)

```tsx
// ⭐ 실무에서 가장 많이 쓰는 useEffect 패턴
function UserProfile({ userId }: { userId: number }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    const controller = new AbortController();
    let isCancelled = false;

    const loadUser = async () => {
      setLoading(true);
      setError(null);

      try {
        const res = await fetch(`/api/users/${userId}`, {
          signal: controller.signal,
        });
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        const data = await res.json();

        if (!isCancelled) {
          setUser(data);
        }
      } catch (err) {
        if (!isCancelled && err instanceof Error && err.name !== "AbortError") {
          setError(err.message);
        }
      } finally {
        if (!isCancelled) {
          setLoading(false);
        }
      }
    };

    loadUser();

    return () => {
      isCancelled = true;
      controller.abort();
    };
  }, [userId]);

  if (loading) return <p>로딩 중...</p>;
  if (error) return <p>에러: {error}</p>;
  if (!user) return <p>사용자를 찾을 수 없습니다</p>;

  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  );
}
```

> 💡 **이 패턴이 복잡하다고 느껴진다면 정상이다!**
> 실무에서는 이 보일러플레이트를 **TanStack Query**(52번)가 한 줄로 해결해준다.
> 하지만 원리를 이해해야 라이브러리도 제대로 쓸 수 있다!

function Stopwatch() {
  const [time, setTime] = useState(0);       // 밀리초
  const [isRunning, setIsRunning] = useState(false);

  useEffect(() => {
    if (!isRunning) return;  // 일시정지면 interval 안 만듦

    const id = setInterval(() => {
      setTime(prev => prev + 10);
    }, 10);

    return () => clearInterval(id);  // 클린업!
  }, [isRunning]);  // isRunning이 바뀔 때마다 interval 재설정

  const formatTime = (ms: number) => {
    const seconds = Math.floor(ms / 1000);
    const milliseconds = Math.floor((ms % 1000) / 10);
    return `${seconds}.${String(milliseconds).padStart(2, "0")}`;
  };

  return (
    <div>
      <p style={{ fontSize: "3rem", fontFamily: "monospace" }}>
        {formatTime(time)}
      </p>
      <button onClick={() => setIsRunning(prev => !prev)}>
        {isRunning ? "일시정지" : "시작"}
      </button>
      <button onClick={() => { setIsRunning(false); setTime(0); }}>
        리셋
      </button>
    </div>
  );
}

## 면접 대비

### 🔴 필수 질문

**Q1. useEffect의 의존성 배열을 설명해주세요.**

> 의존성 배열은 useEffect가 "어떤 값이 변했을 때 재실행할지"를 지정한다. 빈 배열(`[]`)이면 마운트 시 1번만 실행, 값이 들어있으면(`[a, b]`) a 또는 b가 변경될 때마다 재실행, 배열 자체가 없으면 매 렌더링마다 실행된다. React는 이전 의존성과 현재 의존성을 `Object.is`로 비교한다.

**Q2. useEffect의 클린업 함수는 언제 실행되나요?**

> 두 가지 시점에 실행된다. 첫째, 의존성이 변경되어 Effect가 재실행되기 직전에 이전 Effect의 클린업이 먼저 실행된다. 둘째, 컴포넌트가 언마운트될 때 마지막 Effect의 클린업이 실행된다. 이를 통해 이벤트 리스너 해제, 타이머 정리, API 요청 취소 등의 뒷정리를 수행한다.

**Q3. useEffect에서 async를 직접 사용할 수 없는 이유?**

> useEffect의 콜백은 클린업 함수(또는 undefined)를 반환해야 하는데, async 함수는 항상 Promise를 반환하므로 타입이 맞지 않는다. 내부에 async 함수를 별도로 선언하고 호출하는 패턴(`const loadData = async () => { }; loadData();`)을 사용한다.

**Q4. StrictMode에서 useEffect가 2번 실행되는 이유?**

> 개발 모드에서 클린업 함수가 제대로 작동하는지 검증하기 위해 의도적으로 마운트 → 언마운트 → 재마운트를 시뮬레이션한다. 클린업이 없거나 불완전하면 이 과정에서 버그가 조기에 드러난다. 프로덕션 빌드에서는 1번만 실행된다.
