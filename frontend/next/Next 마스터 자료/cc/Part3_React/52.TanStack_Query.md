## 1. 왜 TanStack Query인가? — Server State vs Client State 🔴

> 💡 **비유: 도서관 책 vs 내 메모장**
> Server State = 도서관의 책 (다른 사람도 빌려가고, 반납하고, 내용이 바뀔 수 있음)
> Client State = 내 메모장 (나만 쓰고, 나만 바꿈)
> 이 두 가지를 **같은 방식으로 관리하면 안 된다!**

```
Client State (클라이언트 상태):
  - UI 상태: 모달 열림/닫힘, 사이드바 토글
  - 폼 입력값, 선택된 탭
  - 테마, 언어 설정
  → useState, Zustand로 관리

Server State (서버 상태):          ⭐ TanStack Query의 영역!
  - API에서 가져온 데이터 (사용자 목록, 상품, 게시글)
  - 다른 사용자가 언제든 변경할 수 있는 데이터
  - 캐싱, 동기화, 백그라운드 갱신이 필요한 데이터
  → TanStack Query로 관리
```

### useEffect + fetch의 한계 vs TanStack Query

```tsx
// ❌ useEffect + fetch: 46번에서 배운 고통스러운 패턴
function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    let cancelled = false;
    setLoading(true);
    fetch("/api/users", { signal: controller.signal })
      .then(res => { if (!res.ok) throw new Error(); return res.json(); })
      .then(data => { if (!cancelled) setUsers(data); })
      .catch(err => { if (!cancelled) setError(err.message); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; controller.abort(); };
  }, []);

  // 문제: 캐싱 없음, 중복 요청, 백그라운드 갱신 없음, 재시도 없음...
}

// ✅ TanStack Query: 위의 모든 것을 한 줄로!
function UserList() {
  const { data: users, isLoading, error } = useQuery({
    queryKey: ["users"],
    queryFn: () => fetch("/api/users").then(res => res.json()),
  });
  // 캐싱 ✅ / 중복 요청 제거 ✅ / 백그라운드 갱신 ✅ / 에러 재시도 ✅ / DevTools ✅
}
```

## 3. useQuery — 데이터 조회 (GET) 🔴

```tsx
import { useQuery } from "@tanstack/react-query";

// API 함수 (순수한 fetch 함수)
async function fetchUsers(): Promise<User[]> {
  const res = await fetch("/api/users");
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

function UserList() {
  const {
    data,         // 성공 시 데이터 (User[] | undefined)
    isLoading,    // 최초 로딩 중 (캐시 없을 때)
    isFetching,   // 백그라운드 갱신 중 (캐시는 있지만 갱신 중)
    error,        // 에러 객체
    isError,      // 에러 여부
    refetch,      // 수동 갱신 함수
  } = useQuery({
    queryKey: ["users"],           // 캐시 키 (이 키로 데이터를 식별/캐싱)
    queryFn: fetchUsers,           // 데이터를 가져올 함수
    staleTime: 5 * 60 * 1000,     // 5분 동안 캐시 데이터를 "신선"하다고 판단
  });

  if (isLoading) return <Spinner />;
  if (isError) return <p>에러: {error.message}</p>;

  return (
    <ul>
      {data?.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
```

### queryKey — 캐시의 열쇠 🔴

```tsx
// 같은 queryKey → 같은 캐시를 공유!
// 다른 queryKey → 별도의 캐시!

// 기본
queryKey: ["users"]                    // 전체 사용자 목록

// 파라미터별 캐시
queryKey: ["users", userId]            // userId별 별도 캐시
queryKey: ["users", { role: "admin" }] // 필터별 별도 캐시

// 계층적 키 (무효화 시 유용!)
queryKey: ["users"]                    // invalidate → 전체 users 캐시 무효화
queryKey: ["users", 1]                 // invalidate → 1번 유저만 무효화
queryKey: ["users", 1, "posts"]        // invalidate → 1번 유저의 게시글만 무효화

// 실전: 검색 + 페이징
queryKey: ["products", { search: query, page, category }]
```

### staleTime vs gcTime 🟡

```
staleTime (신선 유지 시간):
  데이터를 가져온 후 이 시간 동안은 "신선하다"고 판단.
  신선한 동안은 같은 queryKey로 요청해도 네트워크 호출 안 함! (캐시 반환)

  staleTime: 0 (기본값) → 항상 stale. 컴포넌트 마운트마다 백그라운드 갱신
  staleTime: 60000     → 1분간 신선. 1분 내 같은 요청은 캐시에서 즉시 반환
  staleTime: Infinity  → 영원히 신선. 수동 invalidate 전까지 갱신 안 함

gcTime (가비지 컬렉션 시간):
  캐시 데이터가 사용되지 않은 후 이 시간이 지나면 메모리에서 삭제.
  기본값: 5분

  컴포넌트 언마운트 → gcTime 카운트다운 시작
  gcTime 만료 전 다시 마운트 → 캐시 데이터 즉시 사용! (네트워크 없이)
  gcTime 만료 후 다시 마운트 → 캐시 없음. 새로 fetch
```

```
시간선 예시 (staleTime: 1분, gcTime: 5분):

T=0   : fetch → 캐시 저장 (fresh!)
T=30초: 같은 쿼리 → 캐시 즉시 반환 (아직 fresh. 네트워크 안 탐!)
T=1분 : 캐시가 stale 됨
T=1분+: 같은 쿼리 → 캐시 즉시 반환 + 백그라운드에서 갱신 (stale-while-revalidate!)
T=3분 : 컴포넌트 언마운트
T=8분 : gcTime(5분) 만료 → 캐시 메모리에서 삭제
T=9분 : 다시 마운트 → 캐시 없음 → 새로 fetch + 로딩 표시
```

## 4. useMutation — 데이터 변경 (POST/PATCH/DELETE) 🔴

```tsx
import { useMutation, useQueryClient } from "@tanstack/react-query";

function CreateUserForm() {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: async (newUser: { name: string; email: string }) => {
      const res = await fetch("/api/users", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(newUser),
      });
      if (!res.ok) throw new Error("생성 실패");
      return res.json();
    },
    onSuccess: () => {
      // ⭐ 핵심: 생성 성공 후 사용자 목록 캐시를 무효화!
      // → UserList 컴포넌트가 자동으로 새 데이터를 가져옴!
      queryClient.invalidateQueries({ queryKey: ["users"] });
    },
    onError: (error) => {
      alert(`에러: ${error.message}`);
    },
  });

  const handleSubmit = (data: { name: string; email: string }) => {
    mutation.mutate(data);
  };

  return (
    <form onSubmit={/* ... */}>
      {/* ... input fields ... */}
      <button type="submit" disabled={mutation.isPending}>
        {mutation.isPending ? "생성 중..." : "사용자 생성"}
      </button>
      {mutation.isError && <p className="text-red-500">{mutation.error.message}</p>}
    </form>
  );
}
```

### invalidateQueries — 캐시 무효화 🔴

```tsx
const queryClient = useQueryClient();

// 특정 키의 캐시 무효화
queryClient.invalidateQueries({ queryKey: ["users"] });

// 계층적 무효화: "users"로 시작하는 모든 캐시!
queryClient.invalidateQueries({ queryKey: ["users"] });
// → ["users"], ["users", 1], ["users", 1, "posts"] 모두 무효화!

// 특정 사용자만 무효화
queryClient.invalidateQueries({ queryKey: ["users", userId] });

// 모든 쿼리 무효화 (로그아웃 시)
queryClient.invalidateQueries();
```

## 5. 실전 패턴: CRUD 완전체 🔴

```tsx
// API 함수들
const api = {
  getUsers: async (): Promise<User[]> => {
    const res = await fetch("/api/users");
    if (!res.ok) throw new Error("조회 실패");
    return res.json();
  },
  createUser: async (data: CreateUserInput): Promise<User> => {
    const res = await fetch("/api/users", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(data),
    });
    if (!res.ok) throw new Error("생성 실패");
    return res.json();
  },
  updateUser: async ({ id, ...data }: UpdateUserInput): Promise<User> => {
    const res = await fetch(`/api/users/${id}`, {
      method: "PATCH",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(data),
    });
    if (!res.ok) throw new Error("수정 실패");
    return res.json();
  },
  deleteUser: async (id: number): Promise<void> => {
    const res = await fetch(`/api/users/${id}`, { method: "DELETE" });
    if (!res.ok) throw new Error("삭제 실패");
  },
};

// 커스텀 훅들 (재사용!)
function useUsers() {
  return useQuery({
    queryKey: ["users"],
    queryFn: api.getUsers,
  });
}

function useCreateUser() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: api.createUser,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ["users"] }),
  });
}

function useUpdateUser() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: api.updateUser,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ["users"] }),
  });
}

function useDeleteUser() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: api.deleteUser,
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ["users"] }),
  });
}

// 컴포넌트에서 사용: 깔끔!
function UserManagement() {
  const { data: users, isLoading } = useUsers();
  const createUser = useCreateUser();
  const deleteUser = useDeleteUser();

  if (isLoading) return <Spinner />;

  return (
    <div>
      <button onClick={() => createUser.mutate({ name: "새 유저", email: "new@test.com" })}>
        추가
      </button>
      {users?.map(user => (
        <div key={user.id}>
          {user.name}
          <button onClick={() => deleteUser.mutate(user.id)}>삭제</button>
        </div>
      ))}
    </div>
  );
}
```

## 6. 고급 패턴 🟡

### Optimistic Update (낙관적 업데이트)

```tsx
const deleteUser = useMutation({
  mutationFn: api.deleteUser,
  onMutate: async (userId) => {
    // 1. 진행 중인 refetch 취소 (충돌 방지)
    await queryClient.cancelQueries({ queryKey: ["users"] });

    // 2. 이전 데이터 백업 (롤백용)
    const previousUsers = queryClient.getQueryData<User[]>(["users"]);

    // 3. 캐시에서 먼저 삭제! (UI 즉시 반영)
    queryClient.setQueryData<User[]>(["users"], old =>
      old?.filter(user => user.id !== userId) ?? []
    );

    return { previousUsers };  // 롤백 데이터
  },
  onError: (err, userId, context) => {
    // 4. 에러 시 롤백!
    queryClient.setQueryData(["users"], context?.previousUsers);
  },
  onSettled: () => {
    // 5. 성공이든 실패든 캐시 갱신
    queryClient.invalidateQueries({ queryKey: ["users"] });
  },
});
```

### Infinite Query (무한 스크롤)

```tsx
const {
  data,
  fetchNextPage,
  hasNextPage,
  isFetchingNextPage,
} = useInfiniteQuery({
  queryKey: ["products"],
  queryFn: ({ pageParam }) =>
    fetch(`/api/products?cursor=${pageParam}`).then(r => r.json()),
  initialPageParam: 0,
  getNextPageParam: (lastPage) => lastPage.nextCursor ?? undefined,
});

// 모든 페이지의 아이템을 평탄화
const allProducts = data?.pages.flatMap(page => page.items) ?? [];

// 더 보기 버튼 또는 Intersection Observer
<button onClick={() => fetchNextPage()} disabled={!hasNextPage || isFetchingNextPage}>
  {isFetchingNextPage ? "로딩 중..." : hasNextPage ? "더 보기" : "끝!"}
</button>
```

## 면접 대비

### 🔴 필수 질문

**Q1. TanStack Query(React Query)란? 왜 사용하나요?**

> 서버 상태(Server State)를 관리하는 라이브러리다. useEffect + fetch로 직접 관리할 때 발생하는 문제들 — 캐싱 없음, 중복 요청, 백그라운드 갱신 없음, 로딩/에러 보일러플레이트 — 을 자동으로 해결한다. queryKey로 캐시를 식별하고, staleTime으로 신선도를 제어하며, invalidateQueries로 mutation 후 자동 갱신하는 것이 핵심이다.

**Q2. Server State와 Client State의 차이? 왜 분리해야 하나요?**

> Server State는 서버에서 오는 데이터로, 다른 사용자에 의해 변경될 수 있고 캐싱/동기화/백그라운드 갱신이 필요하다. Client State는 UI 상태(모달, 폼 입력)로 현재 사용자에게만 존재한다. 둘을 같은 도구(useState/Redux)로 관리하면 캐싱/동기화 로직을 수동으로 구현해야 하므로 TanStack Query(서버)와 Zustand(클라이언트)로 분리하는 것이 2026년 실무 표준이다.

**Q3. staleTime과 gcTime의 차이?**

> staleTime은 데이터를 "신선하다"고 판단하는 시간이다. 신선한 동안은 같은 쿼리를 요청해도 네트워크 호출 없이 캐시를 반환한다. gcTime은 캐시 데이터가 사용되지 않은 후 메모리에서 삭제되기까지의 시간이다. 기본값은 staleTime: 0(항상 stale), gcTime: 5분이다.

**Q4. mutation 후 목록을 갱신하는 방법?**

> `useMutation`의 `onSuccess` 콜백에서 `queryClient.invalidateQueries({ queryKey: ["리소스"] })`를 호출하면, 해당 queryKey의 캐시가 무효화되어 관련 useQuery가 자동으로 새 데이터를 가져온다. 즉각적인 UI 반영이 필요하면 Optimistic Update 패턴으로 캐시를 먼저 수정하고 서버 응답을 기다린다.
