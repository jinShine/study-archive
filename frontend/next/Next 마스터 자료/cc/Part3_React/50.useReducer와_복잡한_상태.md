# 50. useReducer와 복잡한 상태

> **키워드**: `useReducer` `reducer` `dispatch` `action` `판별 유니온` `useState vs useReducer`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | useReducer 기본, reducer 패턴, useState와 선택 기준 | 복잡한 상태 관리 |
| 🟡 이해 | 판별 유니온 Action, Context + useReducer 조합 | 대규모 상태 |
| 🟢 참고 | useReducer + Immer, 상태 머신 패턴 | 고급 |

---

## 1. useReducer란? 🔴

> 💡 **비유: 은행 창구**
> `useState` = ATM (간단한 입출금만)
> `useReducer` = 은행 창구 (전표(action)를 작성해서 제출하면 직원(reducer)이 처리)
> 복잡한 거래(여러 State가 연관되어 바뀌는 경우)는 은행 창구가 더 안전하다.

```tsx
// useReducer(리듀서함수, 초기상태) → [현재상태, 디스패치함수]
const [state, dispatch] = useReducer(reducer, initialState);

// dispatch로 "무엇을 할지"(action) 전달
dispatch({ type: "INCREMENT" });
dispatch({ type: "SET_VALUE", payload: 42 });
```

## 2. 기본 예제: 카운터 🔴

```tsx
// 1. State 타입
type CounterState = {
  count: number;
  step: number;
};

// 2. Action 타입 (판별 유니온!)
type CounterAction =
  | { type: "INCREMENT" }
  | { type: "DECREMENT" }
  | { type: "RESET" }
  | { type: "SET_STEP"; payload: number };

// 3. Reducer 함수: (현재상태, 액션) => 새 상태
function counterReducer(state: CounterState, action: CounterAction): CounterState {
  switch (action.type) {
    case "INCREMENT":
      return { ...state, count: state.count + state.step };
    case "DECREMENT":
      return { ...state, count: state.count - state.step };
    case "RESET":
      return { count: 0, step: 1 };
    case "SET_STEP":
      return { ...state, step: action.payload };
    // default가 없어도 TS가 모든 케이스를 처리했는지 검증!
  }
}

// 4. 컴포넌트에서 사용
function Counter() {
  const [state, dispatch] = useReducer(counterReducer, { count: 0, step: 1 });

  return (
    <div>
      <p>카운트: {state.count} (step: {state.step})</p>
      <button onClick={() => dispatch({ type: "INCREMENT" })}>+</button>
      <button onClick={() => dispatch({ type: "DECREMENT" })}>-</button>
      <button onClick={() => dispatch({ type: "RESET" })}>리셋</button>
      <input
        type="number"
        value={state.step}
        onChange={(e) => dispatch({ type: "SET_STEP", payload: Number(e.target.value) })}
      />
    </div>
  );
}
```

> 💡 **핵심 패턴**: Action 타입을 판별 유니온으로 정의하면
> `dispatch({ type: "INCREMENT", payload: 5 })` 같은 잘못된 호출을 **컴파일 타임에** 차단!

## 3. useState vs useReducer 선택 기준 🔴

```
useState를 쓸 때:
  ✅ 단순한 값 (boolean, number, string)
  ✅ 독립적인 State들 (서로 영향 없음)
  ✅ 업데이트 로직이 간단 (직접 값 설정)

useReducer를 쓸 때:
  ✅ 여러 State가 서로 연관되어 함께 바뀜
  ✅ 업데이트 로직이 복잡 (조건 분기, 계산)
  ✅ action의 종류가 여러 가지
  ✅ 다음 상태가 이전 상태에 의존
  ✅ 테스트가 필요한 비즈니스 로직

실무 판단법:
  "setState 호출이 3개 이상이고 서로 연관되면 useReducer를 고려하라"
```

## 4. 실전: Todo 앱 (useReducer) 🟡

```tsx
type Todo = { id: number; text: string; done: boolean };

type TodoState = {
  todos: Todo[];
  filter: "all" | "active" | "completed";
};

type TodoAction =
  | { type: "ADD"; text: string }
  | { type: "TOGGLE"; id: number }
  | { type: "DELETE"; id: number }
  | { type: "EDIT"; id: number; text: string }
  | { type: "CLEAR_COMPLETED" }
  | { type: "SET_FILTER"; filter: TodoState["filter"] };

function todoReducer(state: TodoState, action: TodoAction): TodoState {
  switch (action.type) {
    case "ADD":
      return {
        ...state,
        todos: [...state.todos, { id: Date.now(), text: action.text, done: false }],
      };
    case "TOGGLE":
      return {
        ...state,
        todos: state.todos.map(todo =>
          todo.id === action.id ? { ...todo, done: !todo.done } : todo
        ),
      };
    case "DELETE":
      return {
        ...state,
        todos: state.todos.filter(todo => todo.id !== action.id),
      };
    case "EDIT":
      return {
        ...state,
        todos: state.todos.map(todo =>
          todo.id === action.id ? { ...todo, text: action.text } : todo
        ),
      };
    case "CLEAR_COMPLETED":
      return {
        ...state,
        todos: state.todos.filter(todo => !todo.done),
      };
    case "SET_FILTER":
      return { ...state, filter: action.filter };
  }
}

function TodoApp() {
  const [state, dispatch] = useReducer(todoReducer, {
    todos: [],
    filter: "all",
  });

  // 파생 상태
  const filteredTodos = state.todos.filter(todo => {
    if (state.filter === "active") return !todo.done;
    if (state.filter === "completed") return todo.done;
    return true;
  });

  const activeCount = state.todos.filter(t => !t.done).length;

  return (
    <div>
      <h1>할 일 ({activeCount}개 남음)</h1>
      <AddTodo onAdd={(text) => dispatch({ type: "ADD", text })} />

      <div>
        {(["all", "active", "completed"] as const).map(f => (
          <button
            key={f}
            onClick={() => dispatch({ type: "SET_FILTER", filter: f })}
            style={{ fontWeight: state.filter === f ? "bold" : "normal" }}
          >
            {f}
          </button>
        ))}
      </div>

      <ul>
        {filteredTodos.map(todo => (
          <li key={todo.id}>
            <span
              style={{ textDecoration: todo.done ? "line-through" : "none" }}
              onClick={() => dispatch({ type: "TOGGLE", id: todo.id })}
            >
              {todo.text}
            </span>
            <button onClick={() => dispatch({ type: "DELETE", id: todo.id })}>삭제</button>
          </li>
        ))}
      </ul>

      <button onClick={() => dispatch({ type: "CLEAR_COMPLETED" })}>
        완료된 항목 삭제
      </button>
    </div>
  );
}
```

> 💡 **useReducer의 장점이 빛나는 순간**:
> - 6가지 action이 하나의 reducer에서 체계적으로 관리
> - TypeScript 판별 유니온으로 잘못된 dispatch 컴파일 타임 차단
> - reducer 함수만 따로 테스트 가능 (순수 함수!)
> - Context와 결합하면 전역 상태 관리 가능

## 면접 대비

### 🔴 필수 질문

**Q1. useState와 useReducer의 차이? 언제 각각 사용하나요?**

> `useState`는 단순한 값(boolean, string, 독립적 상태)에 적합하다. `useReducer`는 여러 상태가 서로 연관되어 함께 변하거나, 업데이트 로직이 복잡하거나, action의 종류가 많을 때 적합하다. reducer 함수는 순수 함수이므로 로직을 컴포넌트 밖으로 분리하여 독립적으로 테스트할 수 있다. TypeScript의 판별 유니온과 결합하면 잘못된 dispatch를 컴파일 타임에 차단할 수 있다.
