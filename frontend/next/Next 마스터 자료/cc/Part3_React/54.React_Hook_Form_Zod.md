## 1. 왜 React Hook Form + Zod인가? 🔴

```
45번에서 배운 제어 컴포넌트 폼의 문제점:
  1. 필드 10개면 useState 10개 + onChange 10개 😱
  2. 타이핑할 때마다 전체 폼이 리렌더링 → 성능 저하
  3. 유효성 검사 로직을 직접 작성 → 반복적이고 에러 프론
  4. 에러 상태, 터치 상태, 서밋 상태... 관리할 것이 너무 많음

React Hook Form이 해결:
  ✅ 비제어(ref) 기반 → 리렌더링 최소화 (성능!)
  ✅ register 한 줄로 입력 필드 연결
  ✅ handleSubmit으로 검증 + 제출 자동화
  ✅ errors 객체로 필드별 에러 메시지 자동 관리

+ Zod 연동:
  ✅ 스키마 하나로 검증 규칙 + TypeScript 타입 동시 생성!
  ✅ z.infer로 타입 자동 추출 → 타입 중복 0%
```

## 3. 기본 사용법 — 회원가입 폼 🔴

```tsx
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

// 1. Zod 스키마 (검증 규칙 + 에러 메시지)
const signupSchema = z.object({
  name: z.string()
    .min(2, "이름은 2자 이상이어야 합니다")
    .max(20, "이름은 20자 이하여야 합니다"),
  email: z.string()
    .email("올바른 이메일 형식이 아닙니다"),
  password: z.string()
    .min(8, "비밀번호는 8자 이상이어야 합니다")
    .regex(/[A-Z]/, "대문자를 1개 이상 포함해야 합니다")
    .regex(/[0-9]/, "숫자를 1개 이상 포함해야 합니다"),
  confirmPassword: z.string(),
  age: z.coerce.number()  // 문자열 input 값을 자동으로 숫자 변환!
    .min(1, "나이를 입력하세요")
    .max(150, "올바른 나이를 입력하세요")
    .optional(),
  terms: z.literal(true, {
    errorMap: () => ({ message: "약관에 동의해주세요" }),
  }),
}).refine((data) => data.password === data.confirmPassword, {
  message: "비밀번호가 일치하지 않습니다",
  path: ["confirmPassword"],  // 이 에러를 confirmPassword 필드에 표시
});

// 2. 스키마에서 타입 자동 추출! (z.infer)
type SignupForm = z.infer<typeof signupSchema>;

// 3. 컴포넌트
function SignupPage() {
  const {
    register,      // input을 RHF에 연결하는 함수
    handleSubmit,  // 폼 제출 처리 (검증 통과 시만 콜백 실행)
    formState: {
      errors,       // 필드별 에러 객체
      isSubmitting,  // 제출 중 여부
      isValid,       // 모든 필드 유효 여부
    },
  } = useForm<SignupForm>({
    resolver: zodResolver(signupSchema),  // Zod와 연결!
    mode: "onBlur",  // blur 시 검증 시작 (타이핑 중엔 안 함)
  });

  // 4. 검증 통과된 데이터만 여기에 도달!
  const onSubmit = async (data: SignupForm) => {
    console.log("검증 통과! 데이터:", data);
    // API 호출...
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)} className="space-y-4">
      {/* 이름 */}
      <div>
        <label htmlFor="name">이름</label>
        <input
          id="name"
          {...register("name")}
          className={errors.name ? "border-red-500" : "border-gray-300"}
        />
        {errors.name && (
          <p className="text-red-500 text-sm mt-1">{errors.name.message}</p>
        )}
      </div>

      {/* 이메일 */}
      <div>
        <label htmlFor="email">이메일</label>
        <input id="email" type="email" {...register("email")} />
        {errors.email && <p className="text-red-500 text-sm">{errors.email.message}</p>}
      </div>

      {/* 비밀번호 */}
      <div>
        <label htmlFor="password">비밀번호</label>
        <input id="password" type="password" {...register("password")} />
        {errors.password && <p className="text-red-500 text-sm">{errors.password.message}</p>}
      </div>

      {/* 비밀번호 확인 */}
      <div>
        <label htmlFor="confirmPassword">비밀번호 확인</label>
        <input id="confirmPassword" type="password" {...register("confirmPassword")} />
        {errors.confirmPassword && (
          <p className="text-red-500 text-sm">{errors.confirmPassword.message}</p>
        )}
      </div>

      {/* 나이 (선택) */}
      <div>
        <label htmlFor="age">나이 (선택)</label>
        <input id="age" type="number" {...register("age")} />
        {errors.age && <p className="text-red-500 text-sm">{errors.age.message}</p>}
      </div>

      {/* 약관 동의 */}
      <div>
        <label>
          <input type="checkbox" {...register("terms")} />
          이용약관에 동의합니다
        </label>
        {errors.terms && <p className="text-red-500 text-sm">{errors.terms.message}</p>}
      </div>

      {/* 제출 */}
      <button
        type="submit"
        disabled={isSubmitting}
        className="bg-blue-600 text-white px-4 py-2 rounded disabled:opacity-50"
      >
        {isSubmitting ? "가입 중..." : "회원가입"}
      </button>
    </form>
  );
}
```

### 동작 흐름

```
1. 사용자가 input에 타이핑
   → register가 ref로 DOM 값을 추적 (리렌더링 없음! ⚡)

2. 포커스를 떠남 (blur)
   → Zod 스키마가 해당 필드를 검증
   → 실패하면 errors 객체에 에러 메시지 추가 → 해당 필드만 리렌더링!

3. 제출 버튼 클릭
   → handleSubmit이 전체 폼을 Zod로 검증
   → 통과하면 onSubmit 콜백 실행 (data는 타입 안전!)
   → 실패하면 errors에 모든 에러 표시

제어 컴포넌트 대비 리렌더링 횟수:
  제어: 필드 10개 × 글자 50개 = 500번 리렌더링 😱
  RHF:  blur 시 + 제출 시 = ~10번 리렌더링 ✅
```

## 4. 핵심 API 정리 🔴

### register — input 연결

```tsx
<input {...register("fieldName")} />
// register가 반환하는 것: { name, ref, onChange, onBlur }
// → input의 값을 ref로 추적하므로 State 없이 동작!
```

### handleSubmit — 폼 제출

```tsx
<form onSubmit={handleSubmit(onSubmit, onError)}>
// onSubmit: 검증 통과 시 호출 (data: FormType)
// onError: 검증 실패 시 호출 (errors) — 선택적
```

### errors — 에러 객체

```tsx
errors.name?.message     // "이름은 2자 이상이어야 합니다"
errors.email?.message    // "올바른 이메일 형식이 아닙니다"
errors.password?.message // "비밀번호는 8자 이상이어야 합니다"
```

### watch — 실시간 값 관찰 (필요할 때만!)

```tsx
const password = watch("password");  // 비밀번호 필드의 실시간 값

// 비밀번호 강도 표시에 활용
const strength = getPasswordStrength(password);

// ⚠️ watch는 리렌더링을 발생시키므로 꼭 필요한 곳에만!
```

### setValue — 프로그래밍적 값 설정

```tsx
// 외부에서 폼 값을 변경할 때
setValue("name", "자동 채움", { shouldValidate: true });

// 서버에서 받은 데이터로 폼 초기화
const { reset } = useForm<FormType>({ resolver: zodResolver(schema) });
useEffect(() => {
  if (userData) reset(userData);  // 서버 데이터로 폼 채우기
}, [userData]);
```

### mode — 검증 타이밍

```tsx
useForm({
  mode: "onBlur",      // ⭐ 추천! 포커스 떠날 때 검증
  mode: "onChange",     // 타이핑할 때마다 검증 (성능 주의)
  mode: "onSubmit",     // 제출할 때만 검증 (기본값)
  mode: "onTouched",    // 첫 blur 후부터 onChange 검증
  mode: "all",          // onBlur + onChange 동시
});
```

## 5. 서버 에러 통합 🟡

```tsx
const onSubmit = async (data: SignupForm) => {
  try {
    await api.signup(data);
    router.push("/welcome");
  } catch (error) {
    if (error instanceof ApiError && error.code === "EMAIL_EXISTS") {
      // 서버 에러를 특정 필드에 표시!
      setError("email", {
        type: "server",
        message: "이미 등록된 이메일입니다",
      });
    } else {
      // 전체 폼 에러
      setError("root", {
        type: "server",
        message: "서버 오류가 발생했습니다. 잠시 후 다시 시도해주세요.",
      });
    }
  }
};

// root 에러 표시
{errors.root && (
  <div className="bg-red-50 text-red-600 p-3 rounded">{errors.root.message}</div>
)}
```

## 6. 재사용 가능한 FormField 컴포넌트 🟡

```tsx
// 매번 label + input + error를 반복하지 않도록 추출!
type FormFieldProps = {
  label: string;
  name: string;
  register: any;
  error?: { message?: string };
  type?: string;
  placeholder?: string;
};

function FormField({ label, name, register, error, type = "text", placeholder }: FormFieldProps) {
  return (
    <div className="space-y-1">
      <label htmlFor={name} className="block text-sm font-medium">
        {label}
      </label>
      <input
        id={name}
        type={type}
        placeholder={placeholder}
        {...register(name)}
        className={`w-full p-2 border rounded ${
          error ? "border-red-500 focus:ring-red-500" : "border-gray-300 focus:ring-blue-500"
        }`}
      />
      {error?.message && (
        <p className="text-red-500 text-sm">{error.message}</p>
      )}
    </div>
  );
}

// 사용: 깔끔!
<FormField label="이름" name="name" register={register} error={errors.name} />
<FormField label="이메일" name="email" register={register} error={errors.email} type="email" />
<FormField label="비밀번호" name="password" register={register} error={errors.password} type="password" />
```

## 면접 대비

### 🔴 필수 질문

**Q1. React Hook Form이 제어 컴포넌트보다 나은 이유?**

> RHF는 비제어(ref) 기반이므로 타이핑할 때마다 리렌더링이 발생하지 않아 대규모 폼에서 성능이 뛰어나다. 제어 컴포넌트는 필드 N개마다 useState N개 + onChange N개가 필요하지만, RHF는 `register` 한 줄로 연결된다. Zod와 결합하면 스키마 하나로 검증 규칙과 TypeScript 타입을 동시에 관리할 수 있어 유지보수성이 크게 향상된다.

**Q2. Zod의 `z.infer`가 중요한 이유?**

> 스키마에서 TypeScript 타입을 자동으로 추출하여, 검증 규칙과 타입 정의를 하나의 소스(Single Source of Truth)에서 관리한다. 스키마를 수정하면 타입이 자동으로 바뀌므로 불일치가 원천적으로 방지되고, 폼 데이터가 onSubmit 콜백에 도달했을 때 타입 안전성이 100% 보장된다.
