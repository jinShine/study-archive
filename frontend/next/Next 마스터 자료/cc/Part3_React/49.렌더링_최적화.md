## 1. 리렌더링이 발생하는 3가지 경우 🔴

```
1. State가 변경될 때     → setState() 호출
2. Props가 변경될 때     → 부모에서 다른 Props 전달
3. 부모가 리렌더링될 때  → 자식도 무조건 리렌더링! ← 가장 중요!
```

> ⚠️ **핵심**: 부모가 리렌더링되면 **Props가 안 바뀌어도** 자식이 리렌더링된다!
> 이것을 방지하는 것이 `React.memo`의 역할.

---

## 2. React.memo — 컴포넌트 메모이제이션 🔴

> "Props가 바뀌지 않았으면 이전 결과를 재사용해라"

```tsx
// ❌ memo 없이: 부모가 리렌더링되면 항상 함께 리렌더링
function ExpensiveList({ items }: { items: string[] }) {
  console.log("ExpensiveList 렌더링!"); // 부모가 바뀔 때마다 출력
  return <ul>{items.map(item => <li key={item}>{item}</li>)}</ul>;
}

// ✅ memo: Props가 같으면 리렌더링 건너뛰기!
const ExpensiveList = memo(function ExpensiveList({ items }: { items: string[] }) {
  console.log("ExpensiveList 렌더링!"); // items가 바뀔 때만 출력!
  return <ul>{items.map(item => <li key={item}>{item}</li>)}</ul>;
});
```

### memo가 작동하지 않는 경우 (가장 흔한 실수!) 🔴

```tsx
function Parent() {
  const [count, setCount] = useState(0);

  // ❌ 매 렌더링마다 새 배열/객체 생성 → memo가 무용지물!
  const items = ["a", "b", "c"];  // 매번 새 참조!
  const style = { color: "red" }; // 매번 새 참조!
  const handleClick = () => { };  // 매번 새 참조!

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <MemoChild items={items} style={style} onClick={handleClick} />
      {/* memo를 썼는데도 매번 리렌더링! items/style/onClick이 매번 새 참조니까! */}
    </div>
  );
}

// ✅ 해결: useMemo + useCallback으로 참조 안정화!
function Parent() {
  const [count, setCount] = useState(0);

  const items = useMemo(() => ["a", "b", "c"], []);     // 참조 고정!
  const style = useMemo(() => ({ color: "red" }), []);    // 참조 고정!
  const handleClick = useCallback(() => { }, []);         // 참조 고정!

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <MemoChild items={items} style={style} onClick={handleClick} />
      {/* ✅ 이제 count가 바뀌어도 MemoChild는 리렌더링 안 됨! */}
    </div>
  );
}
```

## 3. useMemo — 계산 결과 메모이제이션 🔴

> "이 계산이 무거우니까, 의존성이 안 바뀌면 이전 결과를 재사용해라"

```tsx
// useMemo(계산함수, [의존성])
const expensiveValue = useMemo(() => {
  return items.filter(i => i.price > 1000).sort((a, b) => b.price - a.price);
}, [items]);  // items가 바뀔 때만 재계산!
```

### useMemo를 써야 할 때 vs 안 써도 될 때

```tsx
// ✅ 써야 할 때: 무거운 계산
const filtered = useMemo(() => {
  return bigArray.filter(item => complexFilter(item, criteria));
}, [bigArray, criteria]);

// ✅ 써야 할 때: memo된 자식에 전달하는 객체/배열
const config = useMemo(() => ({ theme, locale }), [theme, locale]);
return <MemoChild config={config} />;

// ❌ 안 써도 될 때: 간단한 계산
const fullName = useMemo(() => `${first} ${last}`, [first, last]);
// → 그냥 const fullName = `${first} ${last}`; 이면 충분!
// 문자열 연결은 useMemo 오버헤드보다 빠르다!
```

## 4. useCallback — 함수 메모이제이션 🔴

> "이 함수가 매번 새로 만들어지지 않게, 의존성이 안 바뀌면 같은 참조를 유지해라"
> `useMemo(() => fn, deps)`의 축약형!

```tsx
// useCallback(함수, [의존성])
const handleDelete = useCallback((id: number) => {
  setItems(prev => prev.filter(item => item.id !== id));
}, []);  // setItems는 안정적이므로 의존성 불필요
```

### useCallback이 필요한 경우

```tsx
// ✅ 필요: memo된 자식에 콜백을 전달할 때
const MemoChild = memo(function Child({ onDelete }: { onDelete: (id: number) => void }) {
  // onDelete 참조가 바뀌면 리렌더링됨!
});

function Parent() {
  // useCallback으로 참조 고정!
  const handleDelete = useCallback((id: number) => {
    setItems(prev => prev.filter(item => item.id !== id));
  }, []);

  return <MemoChild onDelete={handleDelete} />;
}

// ✅ 필요: useEffect의 의존성으로 함수를 넘길 때
const fetchData = useCallback(async () => {
  const res = await fetch(`/api/items?page=${page}`);
  return res.json();
}, [page]);

useEffect(() => {
  fetchData();
}, [fetchData]);

// ❌ 불필요: memo된 자식이 없고 의존성으로도 안 쓸 때
const handleClick = () => { console.log("클릭!"); };
// → useCallback 안 써도 됨. 과도한 최적화!
```

## 5. "최적화하지 않는 것이 최선일 때" 🔴

> ⚠️ **가장 중요한 원칙: 성능 문제가 있을 때만 최적화하라!**

```
React는 충분히 빠르다. 대부분의 리렌더링은 문제가 되지 않는다.

memo/useMemo/useCallback의 비용:
  - 메모리 사용 ↑ (이전 값 저장)
  - 의존성 비교 비용
  - 코드 복잡도 ↑
  - 가독성 ↓

최적화가 필요한 신호:
  1. 눈에 보이는 렉(jank)이 발생
  2. React DevTools Profiler에서 특정 컴포넌트가 빈번하게 리렌더링
  3. 리스트가 수백~수천 개 (→ 가상화 고려)
  4. 무거운 계산이 매 렌더링마다 반복

최적화 순서 (효과 큰 것부터):
  1. State를 사용하는 곳에 가깝게 배치 (끌어내리기!)
  2. 컴포넌트 분리 (자주 바뀌는 부분을 격리)
  3. 자식 합성(children) 패턴
  4. 그래도 느리면 → memo + useMemo + useCallback
  5. 리스트가 길면 → react-window 가상화
```

### React Compiler (React 19) 🟢

```
React 19의 React Compiler는 memo/useMemo/useCallback을
자동으로 추가해준다! 개발자가 수동으로 최적화할 필요가 줄어듦.

하지만 "어떤 문제를 해결하는 것인지" 이해는 여전히 필수!
면접에서 원리를 물어보기 때문.
```

## 6. React DevTools로 리렌더링 추적 🟡

```
1. 크롬 React DevTools 설치
2. Components 탭 → 설정(⚙️) → "Highlight updates" 체크
   → 리렌더링 되는 컴포넌트가 파란색 테두리로 반짝!

3. Profiler 탭 → 녹화(●) → UI 조작 → 정지(■)
   → 어떤 컴포넌트가 몇 번, 왜 리렌더링됐는지 확인!
   → "Why did this render?" 확인 가능

실전 사용법:
  - 버튼 클릭 시 관련 없는 컴포넌트까지 깜빡이면? → memo 필요
  - 특정 컴포넌트가 비정상적으로 오래 걸리면? → useMemo 필요
  - 리렌더링 횟수가 예상보다 많으면? → State 위치 확인
```

## 7. 실습

```

1. 부모 컴포넌트: count State + 자식 3개 렌더링
2. 자식 A: count 표시 (리렌더링 필요)
3. 자식 B: 무거운 리스트 (1000개) → count와 무관! (리렌더링 불필요)

- 최적화 전: console.log로 B, C도 리렌더링되는 것 확인
- 최적화 후: memo + useCallback 적용하여 B, C 리렌더링 방지
- React DevTools Profiler로 차이 확인
```

## 면접 대비

### 🔴 필수 질문

**Q1. `useMemo`와 `useCallback`의 차이?**

> `useMemo`는 **계산 결과(값)**를 메모이제이션한다. 의존성이 변하지 않으면 이전에 계산한 값을 반환하여 무거운 연산의 반복을 방지한다. `useCallback`은 **함수 자체(참조)**를 메모이제이션한다. 의존성이 변하지 않으면 같은 함수 참조를 유지하여 `memo`된 자식 컴포넌트의 불필요한 리렌더링을 방지한다. `useCallback(fn, deps)`는 `useMemo(() => fn, deps)`와 동일하다.

**Q2. `React.memo`는 언제 사용하나요?**

> 부모가 리렌더링될 때 Props가 변하지 않았음에도 불필요하게 리렌더링되는 자식 컴포넌트에 사용한다. 단, memo만으로는 부족하고 부모에서 전달하는 객체/배열/함수 Props도 `useMemo`/`useCallback`으로 참조를 안정화해야 한다. 성능 문제가 실제로 있을 때만 적용하고, 모든 컴포넌트에 memo를 거는 것은 오히려 비효율적이다.

**Q3. 불필요한 리렌더링을 줄이는 방법은?**

> 첫째, State를 사용하는 곳에 가깝게 배치하여 영향 범위를 최소화한다. 둘째, 자주 변하는 부분을 별도 컴포넌트로 분리한다. 셋째, children 합성 패턴으로 리렌더링 범위를 격리한다. 그래도 성능 문제가 있으면 `React.memo` + `useMemo`/`useCallback`을 적용한다. React DevTools Profiler로 실제 병목을 측정한 후 최적화하는 것이 올바른 접근이다.
