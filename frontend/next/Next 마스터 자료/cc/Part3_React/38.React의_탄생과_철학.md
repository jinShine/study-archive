## 1. React가 해결한 문제 — "왜 React가 필요한가?" 🔴

> 💡 **비유: 레고 vs 점토**
> jQuery 시대 = 점토로 조각하기 (원하는 형태는 되지만, 하나 바꾸면 전체를 다시 빚어야 함)
> React = 레고 블록 (블록 단위로 조립하고, 바꿀 블록만 교체하면 됨)

### 명령형(Imperative) vs 선언적(Declarative) 🔴

```js
// ❌ 명령형 (jQuery 시대): "어떻게" 할지 하나하나 지시
const button = document.getElementById("btn");
const counter = document.getElementById("counter");
let count = 0;

button.addEventListener("click", () => {
  count++;
  counter.textContent = count;           // 직접 DOM 조작
  if (count > 10) {
    counter.style.color = "red";         // 직접 스타일 조작
    button.setAttribute("disabled", "");  // 직접 속성 조작
  }
});
// 문제: 상태(count)와 UI(DOM)가 따로 놀 수 있다! → 버그의 온상

// ✅ 선언적 (React): "무엇을" 보여줄지만 선언
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <span style={{ color: count > 10 ? "red" : "black" }}>
        {count}
      </span>
      <button
        onClick={() => setCount(count + 1)}
        disabled={count > 10}
      >
        증가
      </button>
    </div>
  );
}
// React가 알아서 DOM을 업데이트한다!
// 상태(count)가 바뀌면 → UI가 자동으로 동기화 → 버그 원천 차단!
```

```
명령형: "1. 냄비를 꺼내라 2. 물을 넣어라 3. 불을 켜라 4. 면을 넣어라 5. 3분 기다려라"
선언적: "라면 한 그릇 주세요" ← React는 이것!
       (어떻게 만들지는 React가 알아서 처리)
```

> 💡 **핵심**: React에서 개발자는 "현재 상태에서 UI가 어떻게 보여야 하는지"만 선언한다.
> 상태가 바뀌면 React가 알아서 DOM을 최소한으로 업데이트한다.

## 2. Virtual DOM — React의 비밀 무기 🔴

> 💡 **비유: 건축 설계도**
> 건물(실제 DOM)을 매번 허물고 다시 짓는 대신,
> 설계도(Virtual DOM)를 2장 그려서 비교한 뒤, 달라진 부분만 시공한다.

```
State 변경 발생!
     │
     ▼
[1단계] 새로운 Virtual DOM 트리 생성 (메모리에서 계산 — 빠름!)
     │
     ▼
[2단계] 이전 Virtual DOM과 비교 (Diffing — 뭐가 바뀌었지?)
     │
     ▼
[3단계] 바뀐 부분만 실제 DOM에 적용 (Reconciliation — 최소 업데이트!)
```

```
예시: 리스트에서 3번째 항목의 텍스트가 바뀜

실제 DOM (느림):
  ❌ 리스트 전체를 지우고 다시 그림 → 깜빡임, 성능 저하

Virtual DOM (빠름):
  ✅ 3번째 <li>의 텍스트만 교체 → 나머지는 건드리지 않음

이전 VDOM:        새 VDOM:           실제 DOM 변경:
<ul>              <ul>
  <li>A</li>        <li>A</li>       (변경 없음)
  <li>B</li>        <li>B</li>       (변경 없음)
  <li>C</li>  →     <li>D</li>  →    <li>의 텍스트만 C→D로 교체!
</ul>             </ul>
```

### 재조정(Reconciliation) 알고리즘의 2가지 규칙 🔴

```
규칙 1: 타입이 다르면 → 통째로 교체 (트리 파괴 후 재생성)
  <div> → <span> 으로 바뀌면 → div 안의 모든 자식도 다 버리고 새로 만듦

규칙 2: 같은 타입이면 → 속성만 비교하여 변경된 것만 업데이트
  <div className="old"> → <div className="new"> → className만 교체!

규칙 3: 리스트는 key로 식별
  key가 같으면 → "같은 요소"로 인식하여 재사용
  key가 없으면 → 순서로 비교 → 삽입/삭제 시 비효율적!
```

> ⚠️ **면접 필수**: "Virtual DOM이 무조건 빠른 건 아닙니다. 비교(Diffing) 과정 자체도 비용입니다.
> 하지만 **불필요한 DOM 조작을 최소화**하여 대부분의 상황에서 충분히 빠릅니다."

## 3. 컴포넌트 — UI의 레고 블록 🔴

> 💡 **비유: 레고 블록**
> 하나의 거대한 HTML 파일 대신, 작은 블록(컴포넌트)으로 쪼개서 조립한다.
> 같은 블록을 여러 곳에서 재사용할 수 있다.

```
전체 페이지를 컴포넌트로 분해:

┌─── App ──────────────────────────┐
│ ┌─── Header ───────────────────┐ │
│ │  Logo    NavMenu    UserIcon │ │
│ └──────────────────────────────┘ │
│ ┌─── Main ─────┐ ┌─── Sidebar ┐ │
│ │ ProductList   │ │ FilterPanel│ │
│ │  ┌─────────┐ │ │ SearchBar  │ │
│ │  │ Product │ │ │ CategoryList│ │
│ │  │  Card   │ │ └────────────┘ │
│ │  └─────────┘ │                 │
│ └──────────────┘                 │
│ ┌─── Footer ───────────────────┐ │
│ └──────────────────────────────┘ │
└──────────────────────────────────┘
```

### 컴포넌트의 3대 특성

```
1. 재사용성: ProductCard를 100곳에서 사용 가능
2. 캡슐화: 각 컴포넌트가 자체 로직과 스타일을 품음
3. 합성: 작은 컴포넌트를 조립하여 복잡한 UI 구축
```

## 4. 단방향 데이터 흐름 (One-way Data Flow) 🔴

> 💡 **비유: 폭포수**
> 데이터는 항상 부모 → 자식으로 한 방향으로 흐른다.
> 자식이 부모의 데이터를 직접 수정할 수 없다.

```
    ┌──── App (상태: user, cart) ────┐
    │                                │
    │   user ↓           cart ↓      │
    │                                │
    ├── Header           ├── Cart    │
    │   (user.name)      │ (cart)    │
    │                    │           │
    │                    ├── CartItem│
    │                    │ (item)    │
    └────────────────────────────────┘

    데이터는 위 → 아래로만 흐른다 (Props)
    아래 → 위로 보내려면? 콜백 함수를 Props로 내려보낸다!
```

```tsx
// 부모 → 자식: Props로 데이터 전달 (아래로)
function Parent() {
  const [name, setName] = useState("철수");
  return <Child name={name} onNameChange={setName} />;
}

// 자식 → 부모: 콜백 함수 호출 (위로 이벤트 전달)
function Child({ name, onNameChange }: { name: string; onNameChange: (n: string) => void }) {
  return (
    <input
      value={name}
      onChange={(e) => onNameChange(e.target.value)}  // 부모의 상태를 "요청"
    />
  );
}
```

> 💡 **왜 단방향인가?**
> 양방향이면 "누가 데이터를 바꿨지?" 추적이 불가능 → 디버깅 지옥
> 단방향이면 데이터 변경의 출처가 항상 명확 → 예측 가능한 코드!

## 5. 순수 함수로서의 컴포넌트 🟡

```
React 컴포넌트 = 순수 함수

입력(Props, State) → 출력(JSX)

같은 입력 → 항상 같은 출력 (부수효과 없음!)
```

```tsx
// ✅ 순수 컴포넌트: 같은 props면 항상 같은 JSX 반환
function Greeting({ name }: { name: string }) {
  return <h1>안녕, {name}!</h1>;
}

// ❌ 순수하지 않은 컴포넌트: 렌더링할 때마다 결과가 다름
function BadComponent() {
  const time = new Date().toLocaleTimeString(); // 렌더링 중 부수효과!
  return <p>{time}</p>;
}

// ✅ 부수효과는 useEffect에서 처리
function GoodComponent() {
  const [time, setTime] = useState("");
  useEffect(() => {
    setTime(new Date().toLocaleTimeString()); // 렌더링 후 부수효과!
  }, []);
  return <p>{time}</p>;
}
```

> 💡 **React의 약속**: "컴포넌트는 순수 함수다. 부수효과는 useEffect에서."
> 이 규칙을 지키면 컴포넌트가 예측 가능하고 테스트 가능해진다.

```
my-app/
├── src/
│   ├── main.tsx       ← 진입점 (ReactDOM.createRoot)
│   ├── App.tsx        ← 루트 컴포넌트
```

```tsx
// src/main.tsx — React 앱의 시작점
import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App";
import "./index.css";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

> 💡 **StrictMode**: 개발 환경에서 컴포넌트를 **두 번** 렌더링하여 부수효과를 감지한다.

## 면접 대비

### 🔴 필수 질문

**Q1. Virtual DOM이란 무엇이고, 왜 사용하나요?**

> Virtual DOM은 실제 DOM의 가벼운 복사본으로, 메모리에 존재하는 JavaScript 객체 트리다. 상태가 변경되면 새로운 Virtual DOM을 생성하고, 이전 것과 비교(Diffing)하여 달라진 부분만 실제 DOM에 반영(Reconciliation)한다. 실제 DOM 조작은 비용이 크므로, 변경을 최소화하여 성능을 최적화한다. 다만 Virtual DOM 자체가 무조건 빠른 것은 아니며, 불필요한 DOM 조작을 줄이는 것이 핵심 가치다.

**Q2. React의 "선언적(Declarative)" 프로그래밍이란?**

> 개발자가 "UI가 어떻게 보여야 하는지(What)"만 선언하고, "어떻게 DOM을 조작할지(How)"는 React가 처리하는 방식이다. 상태에 따른 UI를 JSX로 선언하면, 상태가 변경될 때 React가 자동으로 DOM을 최소한으로 업데이트한다. 명령형(jQuery)과 달리 상태와 UI의 불일치 버그가 원천적으로 방지된다.

**Q3. React의 단방향 데이터 흐름이란?**

> 데이터(Props)가 부모 컴포넌트에서 자식 컴포넌트로 한 방향으로만 흐르는 설계 원칙이다. 자식이 부모의 상태를 직접 변경할 수 없고, 부모가 내려준 콜백 함수를 호출하여 상태 변경을 "요청"한다. 이로써 데이터 변경의 출처가 항상 명확하여 디버깅과 유지보수가 용이해진다.

**Q4. React의 재조정(Reconciliation) 알고리즘?**

> Virtual DOM 비교 시 3가지 규칙을 따른다. 첫째, 요소 타입이 다르면(div→span) 트리를 통째로 교체한다. 둘째, 같은 타입이면 변경된 속성만 업데이트한다. 셋째, 리스트 요소는 `key` prop으로 식별하여 삽입/삭제/이동을 효율적으로 처리한다. 이 때문에 리스트 렌더링 시 고유한 key를 부여하는 것이 중요하다.
