## 1. Git이 뭔데? 🔴

> 💡 **비유: 게임의 세이브 파일!**
> 코드를 작성하다 망하면 → 이전 세이브 포인트로 돌아감!
> 여러 명이 동시에 같은 파일을 수정해도 → 합칠 수 있음!

```
Git = "코드의 시간 여행 머신" + "팀 협업 도구"

핵심 개념:
  Repository(저장소): 프로젝트 폴더 = Git이 관리하는 공간
  Commit(커밋): 세이브 포인트! "이 시점의 코드를 기억해!"
  Branch(브랜치): 평행 세계! "원본은 건드리지 않고 실험!"
  Merge(병합): 평행 세계를 원본에 합치기!
  Remote(원격): GitHub = 클라우드 저장소 (팀원과 공유!)
```

## 2. 매일 쓰는 Git 명령어 🔴

### 기본 흐름 (혼자 작업)

```bash
# 1. 현재 상태 확인 (가장 많이 치는 명령어!)
git status
# 빨간색 = 수정했지만 아직 스테이징 안 된 파일
# 초록색 = 스테이징 완료! 커밋 준비 OK!

# 2. 변경된 파일을 스테이징 (세이브할 파일 선택!)
git add .               # 모든 변경 파일!
git add src/app/page.tsx # 특정 파일만!

# 3. 커밋 (세이브!)
git commit -m "feat: 상품 목록 페이지 구현"
# -m = 메시지! 뭘 했는지 적기!

# 4. GitHub에 올리기!
git push origin main
# origin = GitHub 원격 저장소
# main = 메인 브랜치
```

### 팀 작업 흐름 (브랜치!)

```bash
# 1. 새 기능 브랜치 만들기
git checkout -b feature/product-list
# -b = 브랜치를 만들면서(branch) 동시에 이동(checkout)!
# "product-list 기능을 개발할 평행 세계를 만들었다!"

# 2. 코드 작성 → 커밋 → 푸시
git add .
git commit -m "feat: 상품 목록 컴포넌트 구현"
git push origin feature/product-list

# 3. GitHub에서 PR(Pull Request) 만들기!
# → 브라우저에서 GitHub 접속 → "Compare & pull request" 클릭!

# 4. 코드 리뷰 후 main에 병합!
# → GitHub에서 "Merge pull request" 클릭!

# 5. 로컬로 돌아와서 최신 코드 받기
git checkout main
git pull origin main

# 6. 사용 끝난 브랜치 삭제
git branch -d feature/product-list
```

### 자주 쓰는 기타 명령어

```bash
# 변경 내용 확인
git diff                   # 수정한 내용 보기
git log --oneline          # 커밋 히스토리 한 줄씩

# 되돌리기
git checkout -- file.tsx   # 수정한 파일을 마지막 커밋 상태로!
git reset HEAD~1           # 마지막 커밋 취소 (코드는 유지!)
git reset --hard HEAD~1    # 마지막 커밋 취소 + 코드도 삭제! ⚠️

# 임시 저장 (다른 브랜치로 급히 갈 때!)
git stash                  # 작업 중인 것 임시 보관!
git stash pop              # 보관한 것 꺼내기!

# 원격 저장소 연결
git remote add origin https://github.com/user/repo.git
git clone https://github.com/user/repo.git  # 처음 받기
```

## 3. 커밋 컨벤션 (Conventional Commits) 🔴

> 💡 커밋 메시지를 아무렇게나 쓰면 → 나중에 "이게 뭐였지?" 😱
> 규칙을 정해두면 → 히스토리만 봐도 뭘 했는지 한눈에!

```bash
# 형식: type: 설명
# type은 아래 중 하나!

feat:     새 기능 추가!
fix:      버그 수정!
style:    스타일만 변경 (CSS, 포맷팅, 세미콜론 등)
refactor: 리팩토링 (기능 변경 없이 코드 개선)
docs:     문서 변경 (README 등)
test:     테스트 추가/수정
chore:    빌드 설정, 패키지 등 (코드와 무관한 작업)

# 예시:
git commit -m "feat: 상품 검색 기능 구현"
git commit -m "fix: 장바구니 수량 버그 수정"
git commit -m "style: 헤더 반응형 레이아웃 적용"
git commit -m "refactor: API 호출 로직 커스텀 훅으로 분리"
git commit -m "docs: README에 설치 가이드 추가"
git commit -m "chore: eslint 설정 업데이트"
```

### 좋은 커밋 vs 나쁜 커밋

```bash
# ❌ 나쁜 커밋 메시지
git commit -m "수정"
git commit -m "ㅋㅋ"
git commit -m "fix"
git commit -m "여러 가지 수정"

# ✅ 좋은 커밋 메시지
git commit -m "feat: 상품 상세 페이지 동적 라우팅 구현"
git commit -m "fix: 검색 결과 0건일 때 에러 메시지 미표시 수정"
git commit -m "refactor: useProducts 커스텀 훅 추출"

# 규칙:
#   1. 한글 OK! (팀 컨벤션에 따라)
#   2. "무엇을" 했는지 명확하게!
#   3. 하나의 커밋 = 하나의 논리적 변경!
#      ❌ 검색 기능 + 로그인 버그 수정을 한 커밋에!
#      ✅ 검색 기능 1커밋 + 버그 수정 1커밋!
```

## 4. 브랜치 전략 (Git Flow) 🟡

```
실무에서 가장 많이 쓰는 브랜치 구조:

main (또는 master)
  │  ← 배포되는 코드! 항상 안정적!
  │
  ├── develop
  │     │  ← 개발 중인 코드가 모이는 곳!
  │     │
  │     ├── feature/product-list    ← 기능 개발!
  │     ├── feature/auth-login      ← 기능 개발!
  │     └── feature/cart            ← 기능 개발!
  │
  └── hotfix/critical-bug          ← 긴급 버그 수정!

흐름:
  1. develop에서 feature 브랜치를 만들어 기능 개발
  2. 완성되면 develop으로 PR → 코드 리뷰 → 머지
  3. develop이 안정적이면 main으로 머지 → 배포!
  4. 긴급 버그는 main에서 hotfix 브랜치 → 수정 → main으로!
```

### 소규모 팀 (간단 버전!)

```
main
  │
  ├── feature/product-list    ← 기능 개발!
  ├── feature/auth-login
  └── fix/cart-bug            ← 버그 수정!

소규모 팀에서는 develop 없이
feature/fix 브랜치 → main으로 직접 PR!
```

## 5. PR (Pull Request) & 코드 리뷰 🔴

> 💡 **PR = "내 코드를 main에 넣어줘!" 요청**
> 코드 리뷰 = 팀원이 내 코드를 검토하는 것!

### 좋은 PR 작성법

```markdown
## PR 제목
feat: 상품 검색 기능 구현

## 변경 사항
- 검색 컴포넌트 추가 (SearchBar.tsx)
- Server Action으로 서버 사이드 검색 구현
- URL 쿼리 스트링으로 검색 상태 관리

## 스크린샷
(변경 전/후 스크린샷!)

## 테스트
- [x] 검색어 입력 시 URL 변경 확인
- [x] 검색 결과 필터링 확인
- [x] 빈 검색어 시 전체 목록 표시

## 관련 이슈
Closes #42
```

### 코드 리뷰할 때 보는 것

```
✅ 코드 스타일: 일관성 있는가? (ESLint/Prettier)
✅ 네이밍: 변수/함수 이름이 명확한가?
✅ 보안: API 키나 비밀번호가 노출되지 않는가?
✅ 성능: 불필요한 리렌더링이나 무한 루프는 없는가?
✅ 에러 처리: 예외 상황을 처리했는가?
✅ 테스트: 테스트를 추가했는가?
```

## 6. Merge vs Rebase 🟡

```
Merge (병합):
  기록이 보존됨! "여기서 합쳤어"라는 커밋이 생김.
  초보에게 안전! 충돌 시 한 번만 해결!

  main: ─A──B──C──────M── (M = merge commit!)
                      ↗
  feat: ───D──E──F──┘

Rebase (리베이스):
  히스토리가 깔끔한 직선! merge 커밋 없음!
  하지만 실수하면 위험! 공유 브랜치에서는 절대 하지 마!

  main: ─A──B──C──D'──E'──F'── (깔끔한 직선!)

초보 추천: Merge만 쓰자!
실무 추천: main에 합칠 때 Squash Merge! (여러 커밋 → 1개로 합침!)
```

## 7. 충돌(Conflict) 해결 🔴

> 💡 충돌 = 두 사람이 같은 파일의 같은 줄을 수정했을 때!

```
충돌 발생 시 파일에 이런 표시가 생김:

<<<<<<< HEAD
const title = "내가 수정한 제목";
=======
const title = "팀원이 수정한 제목";
>>>>>>> feature/other-branch

해결법:
  1. 둘 중 하나를 선택하거나 합침!
  2. <<<, ===, >>> 표시를 모두 지움!
  3. git add . → git commit!

VS Code에서는:
  "Accept Current" (내 것!) / "Accept Incoming" (상대 것!)
  / "Accept Both" (둘 다!) 버튼이 뜸! 클릭 한 번! ⭐
```

## 면접 대비

### 🔴 필수 질문

**Q1. Git에서 merge와 rebase의 차이?**

> merge는 두 브랜치의 히스토리를 유지하면서 합치며 merge commit이 생긴다. rebase는 커밋을 다른 브랜치 위에 재배치하여 직선형 히스토리를 만든다. 팀에서는 안전한 merge를 기본으로 사용하고, 깔끔한 히스토리가 필요하면 squash merge를 활용한다. rebase는 이미 공유된 브랜치에서는 사용하면 안 된다.

**Q2. 좋은 커밋 메시지의 조건?**

> Conventional Commits 규칙에 따라 `type: 설명` 형식으로 작성한다. type은 feat(기능), fix(버그), refactor(리팩토링) 등이며, 하나의 커밋에는 하나의 논리적 변경만 포함한다. "수정", "fix" 같은 모호한 메시지 대신 "feat: 상품 검색 기능 구현"처럼 무엇을 했는지 명확히 작성해야 나중에 히스토리 추적이 가능하다.
