## 1. ESLint와 Prettier가 뭔데? 🔴

> 💡 **비유: 맞춤법 검사기 vs 서식 정리기!**
>
> **ESLint = 맞춤법 검사기 (문법 경찰!)** 👮
> "이 변수를 선언했는데 안 쓰고 있어!" (사용 안 하는 변수!)
> "이 함수는 undefined를 반환할 수 있어!" (잠재적 버그!)
> "useEffect에 의존성 배열이 빠졌어!" (React 규칙 위반!)
> → **코드의 "품질"을 검사!** (버그를 미리 잡아줌!)
>
> **Prettier = 서식 정리기 (인테리어 디자이너!)** 🎨
> "따옴표는 큰따옴표로 통일!" (일관성!)
> "들여쓰기는 2칸으로!" (가독성!)
> "줄 끝에 세미콜론 붙여!" (스타일!)
> → **코드의 "외관"을 정리!** (보기 좋게!)

```
ESLint: "이 코드가 맞아?" (품질! 버그!)
  → 잘못된 코드를 빨간 줄로 경고!
  → 잠재적 버그를 미리 방지!

Prettier: "이 코드가 예뻐?" (스타일! 포맷!)
  → 저장만 누르면 코드가 자동으로 예뻐짐!
  → 팀원 모두가 동일한 스타일!

둘 다 필요한 이유:
  ESLint만: 버그는 잡지만 코드가 지저분할 수 있음
  Prettier만: 예쁘지만 버그는 못 잡음
  둘 다: 버그도 잡고 + 예쁘기도 하고! ✅
```

## 2. Next.js에서 ESLint 설정 (이미 포함!) 🔴

> create-next-app으로 프로젝트를 만들면 ESLint가 이미 설치되어 있다!

### 기본 설정 확인

```js
// eslint.config.mjs (Next.js 15+ Flat Config!)
import { dirname } from "path";
import { fileURLToPath } from "url";
import { FlatCompat } from "@eslint/eslintrc";

const __dirname = dirname(fileURLToPath(import.meta.url));
const compat = new FlatCompat({ baseDirectory: __dirname });

const eslintConfig = [
  ...compat.extends("next/core-web-vitals", "next/typescript"),
  // next/core-web-vitals = Next.js 권장 규칙!
  // next/typescript = TypeScript 규칙!
];

export default eslintConfig;
```

### ESLint 실행

```bash
# 프로젝트 전체 검사!
npm run lint
# 또는
npx next lint

# 결과:
# ✅ No ESLint warnings or errors (깨끗!)
# ⚠️ Warning: 'x' is defined but never used (경고!)
# ❌ Error: React Hook useEffect has a missing dependency (에러!)
```

### 자주 나오는 ESLint 경고/에러

```tsx
// ⚠️ 경고: 사용하지 않는 변수!
const unusedVariable = "안 쓰는데 왜 만들었어?";
// → 해결: 지우거나, 앞에 _ 붙이기! (_unusedVariable)

// ❌ 에러: useEffect 의존성 누락!
useEffect(() => {
  fetchData(userId);
}, []); // userId가 변해도 다시 안 실행됨!
// → 해결: 의존성 배열에 추가!
// }, [userId]);

// ❌ 에러: img 대신 next/image 사용!
<img src="/photo.jpg" />  // Next.js에서 금지!
// → 해결:
import Image from "next/image";
<Image src="/photo.jpg" alt="사진" width={300} height={200} />

// ❌ 에러: a 태그 대신 next/link 사용!
<a href="/about">About</a>  // Next.js에서 비권장!
// → 해결:
import Link from "next/link";
<Link href="/about">About</Link>
```

## 3. Prettier 설치 & 설정 🔴

### 설치

```bash
# Prettier 본체 + ESLint 충돌 방지 플러그인!
npm install -D prettier eslint-config-prettier
```

```
eslint-config-prettier가 하는 일:
  ESLint에도 코드 스타일 규칙이 있고,
  Prettier에도 코드 스타일 규칙이 있음.
  → 둘이 충돌할 수 있음! (ESLint: 세미콜론 필수! / Prettier: 세미콜론 불필요!)
  → eslint-config-prettier = "코드 스타일은 Prettier에게 맡겨!"
  → ESLint의 스타일 규칙을 끔 → 충돌 방지! ✅
```

### .prettierrc 파일 만들기

```json
// 프로젝트 루트에 .prettierrc 파일 생성!
{
  "semi": true,
  "singleQuote": false,
  "tabWidth": 2,
  "trailingComma": "all",
  "printWidth": 100,
  "bracketSpacing": true,
  "arrowParens": "always",
  "endOfLine": "lf"
}
```

### 각 옵션 설명

```
semi: true
  → 줄 끝에 세미콜론(;) 붙이기!
  → const x = 1;  (O)
  → const x = 1   (X)

singleQuote: false
  → 문자열에 큰따옴표("") 사용!
  → "hello"  (O)
  → 'hello'  (X)
  → ⚠️ JSX에서는 항상 큰따옴표! (HTML 표준!)

tabWidth: 2
  → 들여쓰기 2칸! (프론트엔드 표준!)
  → 백엔드(Java/Python)는 보통 4칸이지만,
    프론트엔드는 2칸이 글로벌 표준!

trailingComma: "all"
  → 마지막 항목 뒤에도 쉼표!
  → { a: 1, b: 2, }  ← 마지막 2 뒤에 쉼표!
  → Git diff가 깔끔해짐! (추가할 때 윗줄 수정 안 해도 됨!)

printWidth: 100
  → 한 줄이 100자를 넘으면 자동 줄바꿈!
  → 80은 너무 좁고, 120은 너무 넓고, 100이 적당! ⭐

bracketSpacing: true
  → 객체 중괄호 안에 공백!
  → { name: "Kim" }  (O)
  → {name: "Kim"}    (X)

arrowParens: "always"
  → 화살표 함수 인자에 항상 괄호!
  → (x) => x + 1  (O)
  → x => x + 1    (X)

endOfLine: "lf"
  → 줄바꿈 문자를 LF로 통일! (Mac/Linux 표준!)
  → Windows(CRLF)와 Mac(LF) 혼재 방지!
```

### ESLint에 Prettier 연동

```js
// eslint.config.mjs 수정!
import { dirname } from "path";
import { fileURLToPath } from "url";
import { FlatCompat } from "@eslint/eslintrc";

const __dirname = dirname(fileURLToPath(import.meta.url));
const compat = new FlatCompat({ baseDirectory: __dirname });

const eslintConfig = [
  ...compat.extends(
    "next/core-web-vitals",
    "next/typescript",
    "prettier",          // ⭐ 추가! ESLint의 스타일 규칙을 Prettier에 위임!
  ),
];

export default eslintConfig;
```

## 4. VS Code 자동 포맷팅 설정 🔴

> 저장(Cmd+S)만 누르면 코드가 자동으로 예뻐진다!

### 1단계: VS Code 확장 설치

```
필수 설치:
  1. ESLint (dbaeumer.vscode-eslint)
  2. Prettier (esbenp.prettier-vscode)
```

### 2단계: settings.json 설정

```json
// Cmd + Shift + P → "Settings JSON" → Open User Settings (JSON)
{
  // 저장할 때 자동으로 Prettier 포맷팅! ⭐
  "editor.formatOnSave": true,

  // 기본 포맷터를 Prettier로!
  "editor.defaultFormatter": "esbenp.prettier-vscode",

  // 저장할 때 ESLint 자동 수정!
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  }
}
```

### 동작 흐름

```
Cmd + S (저장!) 누르면:

  1. Prettier가 먼저 실행!
     → 들여쓰기, 따옴표, 세미콜론, 줄바꿈 자동 정리!

  2. ESLint가 자동 수정!
     → 사용 안 하는 import 제거!
     → 자동으로 고칠 수 있는 경고 수정!

  3. 파일 저장!

→ 저장만 누르면 코드가 자동으로 깨끗해짐! ⭐
→ 수동으로 포맷팅할 필요 0!
```

## 6. 커스텀 ESLint 규칙 추가 🟡

```js
// eslint.config.mjs
const eslintConfig = [
  ...compat.extends("next/core-web-vitals", "next/typescript", "prettier"),

  // 커스텀 규칙 추가!
  {
    rules: {
      // console.log를 경고로! (배포 전에 지우라고!)
      "no-console": "warn",

      // 사용 안 하는 변수 → 에러! (_로 시작하는 건 OK!)
      "@typescript-eslint/no-unused-vars": [
        "error",
        { argsIgnorePattern: "^_" },
      ],

      // any 타입 사용 → 경고!
      "@typescript-eslint/no-explicit-any": "warn",
    },
  },
];
```

### 규칙 레벨

```
"off"   또는 0  → 규칙 끄기 (무시!)
"warn"  또는 1  → 경고 (노란 줄! 빌드는 됨!)
"error" 또는 2  → 에러 (빨간 줄! 빌드 실패!)
```

## 면접 대비

### 🔴 필수 질문

**Q1. ESLint와 Prettier의 역할 차이?**

> ESLint는 코드의 "품질"을 검사하여 잠재적 버그를 잡아낸다(사용 안 하는 변수, useEffect 의존성 누락 등). Prettier는 코드의 "스타일"을 통일한다(들여쓰기, 따옴표, 세미콜론 등). ESLint에도 스타일 규칙이 있어서 충돌할 수 있으므로 `eslint-config-prettier`로 ESLint의 스타일 규칙을 비활성화하고 스타일은 Prettier에게 위임한다.

**Q2. 팀 프로젝트에서 코드 스타일을 어떻게 통일하나요?**

> `.prettierrc` 파일로 코드 스타일 규칙(들여쓰기 2칸, 큰따옴표 등)을 정의하고 Git에 커밋하여 팀원 모두가 공유한다. VS Code의 `formatOnSave` 설정으로 저장 시 자동 포맷팅되게 하고, CI에서 `npm run lint`를 실행하여 규칙 위반 코드가 병합되지 않도록 한다. 이로써 수동 코드 리뷰에서 스타일 논쟁이 사라지고 로직 리뷰에 집중할 수 있다.
