## 1. 패키지 매니저가 뭔데? 🔴

> 💡 **비유: 앱 스토어!**
> iPhone에서 앱을 설치할 때 "App Store"를 쓰듯이,
> 자바스크립트에서 남이 만든 코드(라이브러리)를 설치할 때 "npm"을 쓴다!

```
패키지 = 다른 개발자가 만들어둔 코드 묶음 (라이브러리!)
매니저 = 이걸 설치/삭제/업데이트하는 도구!

예시: "날짜를 예쁘게 표시하고 싶다"
  ❌ 직접 만들기: 3일 걸림...
  ✅ npm install date-fns → 남이 만든 걸 1초 만에 설치! → 바로 사용!

npmjs.com에 200만 개 이상의 패키지가 등록되어 있음!
React, Next.js, Tailwind CSS도 전부 npm 패키지!
```

## 2. npm 핵심 명령어 🔴

### 프로젝트 시작

```bash
# 새 프로젝트 초기화 (package.json 생성!)
npm init -y
# -y = 질문 없이 기본값으로! (Yes to all!)

# Next.js 프로젝트 생성 (이걸 더 많이 씀!)
npx create-next-app@latest my-app
```

### 패키지 설치

```bash
# 일반 의존성 설치 (프로덕션에 필요한 것!)
npm install react
npm install zustand
npm i axios        # i = install의 약자!

# 여러 개 동시에!
npm install zustand @tanstack/react-query zod

# 개발 의존성 설치 (개발할 때만 필요한 것!)
npm install -D typescript @types/react
npm install --save-dev eslint prettier
# -D = --save-dev의 약자!
# 프로덕션 배포에는 포함되지 않음!

# 전역 설치 (시스템 어디서든 사용!)
npm install -g vercel
# -g = global! 프로젝트가 아니라 내 컴퓨터에 설치!
```

### 패키지 삭제

```bash
npm uninstall axios        # 삭제!
npm uninstall -D eslint    # 개발 의존성에서 삭제!
```

### 실행

```bash
npm run dev        # package.json의 scripts.dev 실행!
npm run build      # 빌드!
npm run start      # 프로덕션 서버 실행!
npm test           # 테스트! (test는 run 생략 가능!)

# 패키지를 설치하지 않고 임시 실행!
npx create-next-app@latest  # create-next-app을 설치 안 하고 바로 실행!
npx playwright test          # Playwright를 전역 설치 안 해도 OK!
```

## 3. package.json 완벽 해부 🔴

> 프로젝트의 "신분증"! 이 파일 하나에 프로젝트의 모든 정보가!

```json
{
  "name": "my-next-app",
  "version": "0.1.0",
  "private": true,

  "scripts": {
    "dev": "next dev --turbopack",
    "build": "next build",
    "start": "next start",
    "lint": "next lint",
    "test:e2e": "npx playwright test"
  },

  "dependencies": {
    "next": "^15.0.0",
    "react": "^19.0.0",
    "react-dom": "^19.0.0",
    "zustand": "^5.0.0",
    "@tanstack/react-query": "^5.0.0"
  },

  "devDependencies": {
    "typescript": "^5.0.0",
    "@types/react": "^19.0.0",
    "eslint": "^9.0.0",
    "prettier": "^3.0.0",
    "@playwright/test": "^1.45.0"
  }
}
```

### 각 필드 설명

```
name: 프로젝트 이름
version: 버전
private: true → npm에 실수로 공개 발행 방지!

scripts: 단축 명령어!
  "dev": "next dev" → npm run dev를 치면 next dev가 실행!
  원하는 스크립트를 자유롭게 추가 가능!

dependencies: 프로덕션 의존성!
  → 실제 서비스 운영에 필요한 패키지!
  → npm install 할 때 이 목록을 보고 설치!

devDependencies: 개발 의존성!
  → 개발할 때만 필요! (타입, 린터, 테스트 등)
  → 프로덕션 빌드에는 포함 안 됨!
```

### 버전 번호의 비밀 (SemVer)

```
"next": "^15.2.3"

15   .   2   .   3
Major   Minor   Patch

Major (15): 호환성이 깨지는 대변화! (기존 코드가 안 될 수 있음!)
Minor (2):  새 기능 추가! (기존 코드는 그대로 동작!)
Patch (3):  버그 수정! (안전한 업데이트!)

기호의 의미:
  ^15.2.3 → "15.x.x까지 OK!" (Minor + Patch 자동 업데이트!)
             (가장 많이 사용! ⭐)
  ~15.2.3 → "15.2.x까지 OK!" (Patch만 자동 업데이트!)
  15.2.3  → "정확히 이 버전만!" (기호 없음!)
```

## 4. Lock 파일 — 왜 중요한가? 🟡

> 💡 **비유: 레시피의 정확한 계량!**
> package.json = "밀가루 적당히, 설탕 조금" (범위만!)
> lock 파일 = "밀가루 200g, 설탕 30g" (정확한 양!)

```
package-lock.json (npm)
pnpm-lock.yaml (pnpm)
yarn.lock (yarn)

이 파일이 하는 일:
  → 모든 패키지의 "정확한" 버전을 기록!
  → 팀원 A와 팀원 B가 동일한 환경을 보장!

⚠️ 절대 규칙:
  ✅ lock 파일을 Git에 커밋! (팀원과 공유!)
  ❌ lock 파일을 .gitignore에 넣지 마!
  ❌ lock 파일을 수동으로 수정하지 마!

npm ci (vs npm install):
  npm install → package.json 기준 (범위 내 최신 설치)
  npm ci      → lock 파일 기준! (정확한 버전 설치!)
  CI/CD에서는 반드시 npm ci! ⭐
```

## 5. npm vs pnpm vs yarn 비교 🟡

```
┌──────────┬──────────────┬──────────────┬──────────────┐
│          │ npm          │ pnpm ⭐       │ yarn         │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 기본 제공 │ Node.js 포함 │ 별도 설치     │ 별도 설치     │
│ 속도     │ 보통          │ 가장 빠름!    │ 빠름          │
│ 디스크   │ 많이 차지     │ 적게 차지! ⭐ │ 보통          │
│ node_modules │ 중복 설치 │ 심볼릭 링크!  │ 중복 설치     │
│ Lock 파일 │ package-lock │ pnpm-lock    │ yarn.lock    │
│ 시장 점유 │ 1위 (기본!)  │ 급성장 중!    │ 2위          │
│ 추천     │ 초보에게!     │ 실무에서! ⭐  │ 레거시 프로젝트│
└──────────┴──────────────┴──────────────┴──────────────┘

초보: npm으로 시작! (Node.js에 기본 포함!)
실무: pnpm 추천! (빠르고 디스크 절약!)
```

### pnpm 설치 & 사용

```bash
# 설치
npm install -g pnpm

# 사용법 (npm과 거의 동일!)
pnpm install          # = npm install
pnpm add zustand      # = npm install zustand
pnpm add -D eslint    # = npm install -D eslint
pnpm remove axios     # = npm uninstall axios
pnpm dev              # = npm run dev (run 생략 가능!)
pnpm build            # = npm run build
```

### pnpm이 빠른 이유

```
npm: 프로젝트마다 node_modules에 전부 복사!
  프로젝트 A: node_modules/react (50MB)
  프로젝트 B: node_modules/react (50MB 또!)
  → 디스크 100MB 사용!

pnpm: 글로벌 저장소에 1번만 저장 + 심볼릭 링크!
  글로벌 저장소: react (50MB) ← 여기에 1번만!
  프로젝트 A: node_modules/react → (링크!) → 글로벌
  프로젝트 B: node_modules/react → (링크!) → 글로벌
  → 디스크 50MB만 사용! (절반!)
```

## 6. npx — 설치 없이 실행! 🔴

```bash
# npx = "이 패키지를 설치하지 않고 임시로 실행!"

npx create-next-app@latest my-app
# create-next-app을 내 컴퓨터에 영구 설치하지 않고
# 최신 버전을 한 번만 다운받아서 실행 후 삭제!

npx playwright test
# @playwright/test가 devDependencies에 있으면 그걸 사용!
# 없으면 임시 다운로드 후 실행!

npx tsc --noEmit
# TypeScript 컴파일러를 전역 설치 안 해도 실행!

# 언제 npx를 쓰나?
#   1. 프로젝트 생성 도구 (create-next-app, create-vite 등)
#   2. 가끔 쓰는 CLI 도구 (playwright, prisma 등)
#   3. 특정 버전으로 실행하고 싶을 때 (@latest!)
```

## 7. 자주 겪는 문제 & 해결법 🔴

```bash
# 문제 1: "모듈을 찾을 수 없습니다" (Module not found)
# 해결: node_modules 삭제 후 재설치!
rm -rf node_modules
npm install  # (또는 pnpm install)

# 문제 2: lock 파일 충돌 (Git merge conflict)
# 해결: lock 파일을 삭제하고 재생성!
rm package-lock.json  # (또는 pnpm-lock.yaml)
npm install           # lock 파일이 새로 생김!

# 문제 3: "permission denied" (권한 에러)
# 해결: sudo를 쓰지 말고 nvm으로 Node.js 관리!
# (sudo npm install은 보안 위험!)

# 문제 4: 버전 호환 문제
# 해결: package.json에서 버전 확인 후
npm install next@15.2.3  # 특정 버전 지정!
```

```
과제: 패키지 매니저 실전 체험!

1. npm init -y로 빈 프로젝트 생성
2. package.json의 구조 확인
3. npm install zustand → dependencies에 추가되는지!
4. npm install -D prettier → devDependencies에!
5. package-lock.json이 생기는지 확인!
6. npm uninstall zustand → 삭제 확인!
7. npx create-next-app@latest로 Next.js 프로젝트 생성!
8. (도전!) pnpm 설치 후 pnpm install → 속도 비교!
```

## 면접 대비

### 🔴 필수 질문

**Q1. dependencies와 devDependencies의 차이?**

> `dependencies`는 프로덕션 런타임에 필요한 패키지(React, Next.js, Zustand 등)이고, `devDependencies`는 개발 시에만 필요한 패키지(TypeScript, ESLint, Prettier, 테스트 도구 등)이다. 프로덕션 빌드 시 devDependencies는 번들에 포함되지 않아 최종 빌드 크기를 줄인다.

**Q2. package-lock.json(lock 파일)의 역할?**

> 모든 패키지의 정확한 버전을 기록하여 팀원 간 동일한 환경을 보장한다. package.json은 `^15.0.0`처럼 범위를 명시하지만, lock 파일은 `15.2.3`으로 정확히 고정한다. Git에 반드시 커밋해야 하며, CI 환경에서는 `npm ci`로 lock 파일 기준 설치를 해야 재현성이 보장된다.

## 핵심 요약

```
┌────────────────────────────────────────────────┐
│  패키지 매니저 핵심 요약                          │
├────────────────────────────────────────────────┤
│                                                │
│  npm install [패키지]     → 설치!               │
│  npm install -D [패키지]  → 개발용 설치!         │
│  npm uninstall [패키지]   → 삭제!               │
│  npm run dev             → 스크립트 실행!        │
│  npx [도구]              → 설치 없이 실행!       │
│                                                │
│  package.json: 프로젝트 신분증!                  │
│    dependencies: 프로덕션용 (React, Next 등)     │
│    devDependencies: 개발용 (TS, ESLint 등)      │
│    scripts: 단축 명령어!                         │
│                                                │
│  버전: ^15.2.3 = 15.x.x까지 OK!                │
│  lock 파일: 정확한 버전 고정! Git에 커밋!        │
│  CI에서: npm ci! (lock 파일 기준!)               │
│                                                │
│  추천: 초보 npm → 실무 pnpm! ⭐                 │
│                                                │
└────────────────────────────────────────────────┘
```
