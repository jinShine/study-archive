## 1. Core Web Vitals — 구글이 정한 성능 기준! 🔴

> 💡 **이 점수가 나쁘면 → 구글 검색 순위가 떨어진다!**

```
구글이 정한 "사용자 경험" 핵심 지표 3가지:

1. LCP (Largest Contentful Paint): 2.5초 이내! ⭐
   "가장 큰 콘텐츠(이미지, 텍스트)가 화면에 뜨는 시간!"

   좋음: 0~2.5초 🟢
   보통: 2.5~4초 🟡
   나쁨: 4초 이상 🔴

2. INP (Interaction to Next Paint): 200ms 이내!
   "버튼 클릭 후 화면이 반응하는 시간!"

   좋음: 0~200ms 🟢
   보통: 200~500ms 🟡
   나쁨: 500ms 이상 🔴

3. CLS (Cumulative Layout Shift): 0.1 이하!
   "화면이 갑자기 밀리는 정도!"
   (광고가 로드되면서 버튼이 아래로 밀리는 현상!)

   좋음: 0~0.1 🟢
   보통: 0.1~0.25 🟡
   나쁨: 0.25 이상 🔴
```

## 2. Lighthouse로 측정하기 🔴

```
Chrome DevTools → Lighthouse 탭!

1. F12 → Lighthouse 탭 클릭
2. Categories: "Performance" 체크!
3. "Analyze page load" 클릭!
4. 30초~1분 후 결과!

점수:
  90~100: 🟢 훌륭!
  50~89:  🟡 개선 필요!
  0~49:   🔴 심각!

주요 지표:
  FCP (First Contentful Paint): 첫 콘텐츠가 보이는 시간
  LCP: 가장 큰 콘텐츠가 보이는 시간 ⭐
  TBT (Total Blocking Time): JS가 메인 스레드를 막는 시간
  CLS: 레이아웃 이동 점수 ⭐
  Speed Index: 콘텐츠가 채워지는 속도

온라인 도구:
  → https://pagespeed.web.dev (구글 공식!)
  → 배포된 URL을 넣으면 실제 환경에서 측정!
```

## 3. 이미지 최적화 — next/image 🔴

> 이미지는 웹 페이지에서 가장 무거운 리소스! LCP에 직결!

```tsx
// ❌ 나쁜 패턴: 일반 <img> 태그
<img src="/hero.jpg" />
// → 원본 크기(5MB!) 그대로 다운로드!
// → WebP 변환 안 됨!
// → width/height 없어서 CLS 발생!
// → 스크롤 안 한 이미지도 전부 다운로드!

// ✅ 좋은 패턴: Next.js Image 컴포넌트!
import Image from "next/image";

<Image
  src="/hero.jpg"
  alt="히어로 이미지"    // ⚠️ alt 필수! (접근성!)
  width={1200}          // ⚠️ 필수! (CLS 방지!)
  height={600}          // ⚠️ 필수! (CLS 방지!)
  priority              // LCP 이미지에만! (첫 화면!)
/>
```

### next/image가 해주는 것

```
1. 자동 리사이징: 기기 크기에 맞게! (모바일: 작게, 데스크톱: 크게!)
2. 포맷 변환: JPG → WebP/AVIF 자동! (용량 30~50% 절감!)
3. Lazy Loading: 스크롤해서 보이는 시점에 다운로드! (기본!)
4. CLS 방지: width/height로 미리 공간 확보!
5. 캐싱: 한 번 변환한 이미지를 CDN에 캐시!

priority 속성:
  → 첫 화면에 보이는 큰 이미지(LCP)에만 사용!
  → Lazy Loading을 끄고 즉시 로드!
  → 남용하면 오히려 느려짐! (첫 화면 1~2개만!)
```

### 외부 이미지 설정

```ts
// next.config.ts
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: "https",
        hostname: "images.unsplash.com",
      },
      {
        protocol: "https",
        hostname: "avatars.githubusercontent.com",
      },
    ],
  },
};
```

## 4. 폰트 최적화 — next/font 🔴

```tsx
// ❌ 나쁜 패턴: Google Fonts CDN 직접 로드
// → FOUT (Flash of Unstyled Text): 기본 폰트 → 깜빡 → 커스텀 폰트!
// → 외부 CDN 요청으로 LCP 지연!
// → CLS 발생! (폰트 크기가 달라서 레이아웃 밀림!)

// ✅ 좋은 패턴: next/font (빌드 시 폰트 최적화!)
import { Inter } from "next/font/google";

const inter = Inter({
  subsets: ["latin"],  // 필요한 문자셋만! (용량 절감!)
  display: "swap",     // 기본 폰트 → 커스텀 폰트로 부드럽게 전환!
});

// layout.tsx에 적용!
export default function RootLayout({ children }) {
  return (
    <html className={inter.className}>
      <body>{children}</body>
    </html>
  );
}
```

```
next/font가 해주는 것:
  1. 빌드 시 폰트 파일을 프로젝트에 포함! (외부 요청 0!)
  2. 자동 서브셋: 사용하는 문자만! (용량 절감!)
  3. CSS font-display: swap 자동 적용!
  4. CLS 방지: 폰트 변경 시 레이아웃 밀림 최소화!
```

## 5. 코드 스플리팅 & Lazy Loading 🟡

```tsx
// Next.js는 자동으로 페이지별 코드 스플리팅!
// /products 페이지 코드는 /about 페이지에서 로드 안 됨!

// 무거운 컴포넌트를 수동으로 Lazy Loading!
import dynamic from "next/dynamic";

// 에디터 같은 무거운 컴포넌트:
const HeavyEditor = dynamic(() => import("@/components/Editor"), {
  loading: () => <p>에디터 로딩 중...</p>,
  ssr: false,  // 서버에서 렌더링 안 함! (브라우저 전용 라이브러리!)
});

// 모달처럼 처음에 안 보이는 컴포넌트:
const Modal = dynamic(() => import("@/components/Modal"));

export default function Page() {
  return (
    <div>
      <h1>페이지</h1>
      {/* 에디터가 실제로 필요할 때만 JS를 다운로드! */}
      <HeavyEditor />
    </div>
  );
}
```

```
자동 코드 스플리팅 (Next.js 기본!):
  → 각 page.tsx가 별도의 JS 번들!
  → /products에 접속하면 products.js만 다운로드!
  → /about의 JS는 다운로드 안 함!

수동 Lazy Loading (dynamic import):
  → 무거운 라이브러리(에디터, 차트, 지도 등)에 사용!
  → 첫 페이지 로드에 포함 안 됨! → 초기 로딩 빨라짐!
  → ssr: false → 브라우저 전용 라이브러리에 필수!
```

## 6. 번들 사이즈 분석 🟡

```bash
# 번들 분석기 설치
npm install -D @next/bundle-analyzer

# next.config.ts에 추가
import withBundleAnalyzer from "@next/bundle-analyzer";

const nextConfig = {};
export default process.env.ANALYZE === "true"
  ? withBundleAnalyzer({ enabled: true })(nextConfig)
  : nextConfig;

# 분석 실행!
ANALYZE=true npm run build
# → 브라우저에 번들 시각화가 열림!
# → "이 패키지가 500KB?!" 발견 가능!
```

```
번들 줄이기 팁:
  1. lodash 전체 → lodash-es로! (Tree Shaking!)
  2. moment.js → date-fns로! (경량!)
  3. 사용 안 하는 import 제거! (ESLint가 잡아줌!)
  4. dynamic import로 무거운 것 분리!
  5. 서버 컴포넌트에서 무거운 라이브러리 사용! (브라우저 번들 0!)
```

## 면접 대비

### 🔴 필수 질문

**Q1. Core Web Vitals 3가지를 설명해주세요.**

> LCP(Largest Contentful Paint)는 가장 큰 콘텐츠가 화면에 표시되는 시간으로 2.5초 이내가 좋다. INP(Interaction to Next Paint)는 사용자 인터랙션 후 화면이 반응하는 시간으로 200ms 이내가 좋다. CLS(Cumulative Layout Shift)는 콘텐츠가 로드되면서 레이아웃이 밀리는 정도로 0.1 이하가 좋다. 구글은 이 지표를 검색 순위에 반영하므로 SEO에 직접적인 영향을 미친다.

**Q2. Next.js에서 이미지 최적화는 어떻게 하나요?**

> `next/image`의 `Image` 컴포넌트를 사용한다. 자동으로 WebP/AVIF 포맷 변환(용량 30~50% 절감), 기기 크기에 맞는 리사이징, 스크롤 시점에 다운로드하는 Lazy Loading을 제공한다. width/height 속성으로 미리 공간을 확보하여 CLS를 방지하고, 첫 화면의 LCP 이미지에는 `priority` 속성을 추가하여 즉시 로드한다.
