## 1. Docker가 뭔데? 🟡

> 💡 **비유: 이사할 때 짐을 컨테이너 박스에 싣는 것!**
> 내 컴퓨터에서 잘 되던 프로젝트가 → 서버에서 안 될 때!
> "내 컴퓨터에서는 되는데요..." 문제를 해결!
>
> Docker = 앱을 실행하는 데 필요한 모든 것(코드 + 환경 + 설정)을
>          하나의 "컨테이너 박스"에 담아서 어디서든 동일하게 실행!

```
왜 Docker가 필요한가?

  문제:
    내 Mac에서: Node.js 20, npm 10, 잘 됨! ✅
    팀원 Windows: Node.js 18, npm 9, 에러! ❌
    배포 서버 Linux: Node.js 16, npm 8, 에러! ❌

  해결:
    Docker 컨테이너에 "Node.js 20 + 내 코드 + 설정"을 담아!
    → Mac에서 실행해도 동일!
    → Windows에서 실행해도 동일!
    → Linux 서버에서 실행해도 동일!
    → "어디서든 같은 결과!" ⭐

컨테이너 vs 가상 머신(VM):
  VM: 운영체제 전체를 복제! (무겁고 느림! 수 GB!)
  Docker: 앱 실행에 필요한 것만! (가볍고 빠름! 수 MB~GB!)
```

## 2. 핵심 용어 3가지 🟡

```
1. 이미지 (Image) = 설계도! 📋
   → "이 앱을 실행하려면 이것들이 필요해!" 라는 레시피!
   → Dockerfile로 만듦!
   → 한 번 만들면 변하지 않음! (불변!)

2. 컨테이너 (Container) = 실행 중인 인스턴스! 📦
   → 이미지를 바탕으로 실제 실행한 것!
   → 하나의 이미지에서 여러 컨테이너를 만들 수 있음!
   → 비유: 이미지 = 붕어빵 틀 / 컨테이너 = 실제 붕어빵!

3. Dockerfile = 이미지를 만드는 레시피! 📝
   → "1단계: Node.js 설치, 2단계: 코드 복사, 3단계: 빌드..."
   → 이 파일만 있으면 누구든 동일한 이미지를 만들 수 있음!
```

## 3. Next.js Dockerfile 작성 🟡

```dockerfile
# 프로젝트 루트에 Dockerfile 생성!

# ──────────────────────────────────
# Stage 1: 의존성 설치
# ──────────────────────────────────
FROM node:20-alpine AS deps
# node:20-alpine = Node.js 20이 설치된 가벼운 Linux!
# alpine = 최소 크기 Linux! (보통 50MB vs 일반 900MB!)

WORKDIR /app
# 컨테이너 안에서 작업할 폴더!

COPY package.json package-lock.json ./
# package.json만 먼저 복사! (캐시 활용!)

RUN npm ci
# npm ci = lock 파일 기준으로 정확한 버전 설치!

# ──────────────────────────────────
# Stage 2: 빌드
# ──────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

COPY --from=deps /app/node_modules ./node_modules
# Stage 1에서 설치한 node_modules를 가져옴!

COPY . .
# 전체 소스 코드 복사!

RUN npm run build
# Next.js 프로덕션 빌드!

# ──────────────────────────────────
# Stage 3: 실행 (프로덕션!)
# ──────────────────────────────────
FROM node:20-alpine AS runner

WORKDIR /app

ENV NODE_ENV=production

# 빌드 결과물만 복사! (소스 코드는 안 가져옴! → 이미지 크기 최소!)
COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static

EXPOSE 3000

CMD ["node", "server.js"]
# Next.js standalone 서버 실행!
```

### next.config.ts에 standalone 설정

```ts
// next.config.ts
const nextConfig = {
  output: "standalone",
  // → .next/standalone 폴더에 독립 실행 가능한 서버 생성!
  // → node_modules 전체가 아니라 필요한 것만! (이미지 크기 최소!)
};
```

## 4. Docker 명령어 기초 🟡

```bash
# 이미지 빌드! (Dockerfile을 읽어서 이미지 생성!)
docker build -t my-next-app .
# -t my-next-app = 이미지 이름을 my-next-app으로!
# . = 현재 폴더의 Dockerfile을 사용!

# 컨테이너 실행!
docker run -p 3000:3000 my-next-app
# -p 3000:3000 = 내 컴퓨터의 3000번 포트와 컨테이너의 3000번 포트를 연결!
# → http://localhost:3000에서 접속 가능!

# 백그라운드 실행!
docker run -d -p 3000:3000 my-next-app
# -d = detach! (터미널 안 잡고 백그라운드에서!)

# 실행 중인 컨테이너 확인
docker ps

# 컨테이너 중지
docker stop [컨테이너ID]

# 이미지 목록
docker images

# 이미지 삭제
docker rmi my-next-app
```

## 5. docker-compose — 여러 서비스 한 번에! 🟢

```yaml
# docker-compose.yml
# Next.js + PostgreSQL을 한 번에 실행!

version: "3.8"

services:
  # Next.js 앱!
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgresql://user:password@db:5432/mydb
    depends_on:
      - db
    # db가 먼저 실행된 후에 web이 실행!

  # PostgreSQL 데이터베이스!
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      # 컨테이너가 꺼져도 데이터 유지!

volumes:
  postgres_data:
```

```bash
# 한 번에 실행!
docker-compose up -d
# → Next.js + PostgreSQL이 동시에 뜸!

# 중지!
docker-compose down

# 로그 확인!
docker-compose logs web
```

## 7. 프론트엔드 개발자에게 Docker가 필요한 순간 🟡

```
1. 팀 프로젝트에서 환경 통일!
   "Node.js 몇 버전이야?" 질문 끝!
   → docker-compose up 한 번이면 모든 팀원이 동일한 환경!

2. 백엔드 API + DB를 로컬에서 돌릴 때!
   → 백엔드 개발자가 Docker 이미지를 주면
   → docker-compose up으로 API + DB 한 번에!
   → 백엔드 설치/설정 필요 없음!

3. AWS/GCP 배포할 때!
   → Vercel을 못 쓰는 상황 (사내 보안 등)
   → Docker 이미지를 AWS ECS, GCP Cloud Run에 올리면 끝!

4. CI/CD 파이프라인에서!
   → GitHub Actions에서 Docker 이미지 빌드
   → → 레지스트리(ECR)에 푸시
   → → 서버에서 Pull해서 실행!

솔직한 현실:
  → Vercel이 있으면 Docker 없이도 배포 가능! ⭐
  → Docker는 "알면 좋고, 필요할 때 배우면 되는" 기술!
  → 면접에서 "Docker 사용 경험?" → 기본 개념만 알면 OK!
```

## 면접 대비

### 🟡 이해 질문

**Q1. Docker를 왜 사용하나요?**

> "내 컴퓨터에서는 되는데 서버에서 안 돼요" 문제를 해결한다. Docker는 앱 실행에 필요한 코드, 런타임(Node.js), 의존성(node_modules), 환경 설정을 하나의 컨테이너에 패키징하여 어떤 환경에서든 동일하게 실행되도록 보장한다. Dockerfile로 이미지(설계도)를 만들고, 이 이미지로 컨테이너(실행 인스턴스)를 생성한다.
