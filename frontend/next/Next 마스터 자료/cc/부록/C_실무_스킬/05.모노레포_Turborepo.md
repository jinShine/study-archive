## 1. 모노레포가 뭔데? 🟢

> 💡 **비유: 아파트 단지 vs 단독 주택!**
> 폴리레포(기존) = 프로젝트마다 별도의 GitHub 저장소! (단독 주택!)
> 모노레포 = 하나의 저장소에 여러 프로젝트! (아파트 단지!)

```
폴리레포 (Poly-repo) — 기존 방식:
  my-frontend/          ← GitHub 저장소 1
  my-backend/           ← GitHub 저장소 2
  my-design-system/     ← GitHub 저장소 3
  my-shared-utils/      ← GitHub 저장소 4

  문제:
    → 공유 코드를 npm에 배포해야 다른 프로젝트에서 사용!
    → 버전 관리 지옥! ("이 패키지 3.2.1에서 3.3.0으로 올려야 해!")
    → 한 곳을 수정하면 다른 저장소도 일일이 업데이트!

모노레포 (Monorepo) — 현대 방식:
  my-company/           ← GitHub 저장소 1개에 전부!
  ├── apps/
  │   ├── web/          ← Next.js 프론트엔드!
  │   ├── admin/        ← 관리자 대시보드!
  │   └── mobile/       ← React Native 앱!
  ├── packages/
  │   ├── ui/           ← 공유 UI 컴포넌트!
  │   ├── utils/        ← 공유 유틸리티!
  │   └── config/       ← 공유 ESLint/TS 설정!
  └── package.json

  장점:
    → 공유 코드를 npm 배포 없이 바로 import!
    → 한 곳만 수정하면 전체에 반영!
    → 코드 리뷰가 한 PR에서!
    → 일관된 설정(ESLint, TS, Prettier)!
```

### 누가 모노레포를 쓰나?

```
구글: 모든 코드가 하나의 저장소! (수십억 줄!)
메타(페이스북): 모노레포!
버셀(Vercel): Turborepo 제작사!
마이크로소프트: Rush 모노레포 도구!

언제 모노레포가 필요한가?
  ✅ 프론트엔드 + 백엔드가 같은 타입을 공유할 때!
  ✅ 여러 앱이 같은 UI 컴포넌트를 쓸 때!
  ✅ 디자인 시스템을 만들 때!
  ✅ 팀이 3명 이상이고 프로젝트가 2개 이상일 때!

언제 불필요한가?
  ❌ 혼자 하는 개인 프로젝트!
  ❌ 프로젝트가 1개뿐일 때!
  → 복잡성만 늘어남! 단순하게!
```

## 2. Turborepo — Vercel이 만든 모노레포 도구! 🟢

```bash
# Turborepo로 모노레포 생성!
npx create-turbo@latest my-company

# 생성되는 구조:
my-company/
├── apps/
│   ├── web/            ← Next.js 앱!
│   └── docs/           ← 문서 사이트!
├── packages/
│   ├── ui/             ← 공유 UI 컴포넌트!
│   ├── eslint-config/  ← 공유 ESLint 설정!
│   └── typescript-config/ ← 공유 TS 설정!
├── turbo.json          ← Turborepo 설정!
├── package.json        ← 루트 (워크스페이스 정의!)
└── pnpm-workspace.yaml ← pnpm 워크스페이스!
```

### turbo.json — 빌드 파이프라인!

```json
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "dev": {
      "persistent": true,
      "cache": false
    },
    "lint": {},
    "test": {}
  }
}
```

```
"dependsOn": ["^build"]
  → "내가 의존하는 패키지를 먼저 빌드!"
  → web이 ui를 import → ui를 먼저 빌드!

"outputs": [".next/**"]
  → 빌드 결과물을 캐시! (다음에 변경 없으면 스킵!)

"persistent": true
  → dev 서버처럼 계속 실행되는 태스크!

"cache": false
  → dev는 캐시하지 않음! (매번 새로!)
```

### 핵심 명령어

```bash
# 모든 앱/패키지의 build 실행!
turbo build

# 모든 앱의 dev 서버 동시에!
turbo dev

# 특정 앱만!
turbo dev --filter=web

# 린트!
turbo lint

# 💡 Turborepo의 마법: 캐싱!
# 코드가 안 바뀐 패키지는 빌드를 건너뜀! ⚡
# "packages/ui가 안 바뀌었으니 빌드 스킵! 0.1초!"
```

## 3. 공유 패키지 만들기 🟢

```tsx
// packages/ui/src/Button.tsx — 공유 버튼!
export function Button({
  children,
  variant = "primary",
}: {
  children: React.ReactNode;
  variant?: "primary" | "secondary";
}) {
  return (
    <button className={variant === "primary" ? "bg-blue-500" : "bg-gray-500"}>
      {children}
    </button>
  );
}
```

```tsx
// apps/web/app/page.tsx — web 앱에서 사용!
import { Button } from "@company/ui";  // npm 배포 없이 바로 import! ⭐

export default function Page() {
  return <Button variant="primary">클릭!</Button>;
}
```

```tsx
// apps/admin/app/page.tsx — admin 앱에서도 같은 버튼!
import { Button } from "@company/ui";  // 동일한 버튼!

export default function AdminPage() {
  return <Button variant="secondary">관리자 버튼!</Button>;
}
```

```
핵심:
  packages/ui에서 버튼을 수정하면
  → web과 admin에서 동시에 반영! ⭐
  → npm 배포/업데이트 필요 없음!
  → 일관된 디자인 시스템!
```

## 4. 프론트엔드 개발자에게 모노레포가 필요한 순간 🟢

```
실무 시나리오:

  1. 디자인 시스템 구축!
     → Button, Input, Modal 등을 공유 패키지로!
     → 웹, 모바일, 관리자 앱에서 동일한 UI!

  2. 타입 공유!
     → 프론트엔드와 백엔드가 같은 타입을 사용!
     → packages/types에 User, Product 타입 정의!
     → 프론트와 백엔드 모두 import!

  3. 설정 공유!
     → ESLint, Prettier, TypeScript 설정을 한 곳에!
     → 모든 프로젝트가 동일한 코드 스타일!

현실적 조언:
  → 취업 면접에서 "모노레포 경험?" → 개념만 알면 OK!
  → 실무에서 필요할 때 도입! (처음부터 모노레포 ❌)
  → 프로젝트가 2개 이상 + 공유 코드 필요할 때 시작!
```

## 면접 대비

### 🟢 참고 질문

**Q1. 모노레포란? 장단점?**

> 모노레포는 여러 프로젝트(앱, 패키지)를 하나의 Git 저장소에서 관리하는 방식이다. 장점은 코드 공유가 쉽고(npm 배포 없이 import), 일관된 설정(ESLint, TS), 원자적 변경(한 PR에서 여러 프로젝트 수정)이 가능하다. 단점은 저장소가 커지고, 빌드 시간이 길어질 수 있으며(Turborepo 캐싱으로 해결), 초기 설정이 복잡하다. Turborepo는 Vercel이 만든 모노레포 도구로, 빌드 캐싱과 병렬 실행으로 빌드 속도를 최적화한다.
