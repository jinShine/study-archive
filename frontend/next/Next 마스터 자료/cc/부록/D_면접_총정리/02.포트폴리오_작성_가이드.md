## 🔥 트러블 슈팅 (가장 중요!)
### 1. Hydration Mismatch 에러 해결
- **문제**: 서버/클라이언트 시간 불일치로 에러 발생
- **원인**: Date() 결과가 서버와 브라우저에서 다름
- **해결**: useEffect에서 클라이언트 전용 로직으로 분리
- **배운 점**: SSR과 CSR의 렌더링 차이를 이해하게 됨

### 2. 무한 리렌더링 최적화
- **문제**: 상품 목록 페이지에서 불필요한 리렌더링 100회+
- **원인**: useEffect 의존성 배열에 객체 참조가 매번 변경
- **해결**: useMemo로 객체 참조 안정화 + React DevTools Profiler 분석
- **결과**: 리렌더링 100회 → 3회로 감소

## 5. 트러블 슈팅 — 면접에서 가장 높은 점수!

> "프로젝트에서 겪은 어려운 문제와 해결 방법"은 면접 단골 질문!

```
트러블 슈팅 작성 공식:

  1. 문제 (Problem): "어떤 현상이 발생했는가?"
  2. 원인 (Cause): "왜 발생했는가?" (디버깅 과정!)
  3. 해결 (Solution): "어떻게 고쳤는가?" (코드/설정 변경!)
  4. 결과 (Result): "수치적 개선!" (성능 50% 향상 등!)
  5. 배운 점 (Lesson): "이 경험에서 무엇을 배웠는가?"

좋은 트러블 슈팅 주제:
  ✅ 성능 최적화 (Lighthouse 60 → 95)
  ✅ Hydration Mismatch 해결
  ✅ 무한 리렌더링 디버깅
  ✅ CORS 에러 해결
  ✅ 캐싱 전략 (ISR + revalidate)
  ✅ 모바일 반응형 이슈
  ✅ 배포 시 환경 변수 문제
```

## 6. GitHub 프로필 관리

```
면접관이 GitHub에서 보는 것:

  1. 잔디 (기여 그래프)
     → 꾸준히 커밋하는 사람인가?
     → 매일 1커밋이라도! (TIL, 학습 기록도 OK!)

  2. README 프로필
     → github.com/username 에 표시되는 프로필!
     → 자기소개, 기술 스택, 대표 프로젝트 링크!

  3. 커밋 메시지
     → "fix" ❌ → "fix: 장바구니 수량 음수 버그 수정" ✅
     → Conventional Commits 사용!

  4. 코드 품질
     → 면접관이 실제로 코드를 읽음!
     → 타입 정의, 폴더 구조, 네이밍, 에러 처리 확인!

  5. PR/이슈 기록
     → 혼자 하는 프로젝트라도 이슈+PR 형식으로!
     → "이 사람은 팀 워크플로우를 아는구나!" 인상!
```

  핵심 기술:
    - React/Next.js (App Router, Server Components)

    - Tailwind CSS (반응형, 디자인 시스템)
    - Playwright E2E 테스트

  사용 경험:
    - Prisma + PostgreSQL (풀스택 프로젝트)
    - Docker (컨테이너 배포)
    - GitHub Actions (CI/CD)

→ 핵심 기술은 상세하게! 보조 기술은 간략하게!
→ "사용 경험"과 "핵심 기술"을 구분!
