## HTML/CSS (10문)

### 1. 시맨틱 HTML이란?

> `<div>` 대신 `<header>`, `<nav>`, `<main>`, `<article>`, `<footer>` 등 **의미가 있는 태그**를 사용하는 것. 검색 엔진이 페이지 구조를 이해하고(SEO), 스크린 리더가 콘텐츠를 올바르게 읽을 수 있다(접근성).

### 2. CSS에서 position의 종류와 차이?

> `static`(기본, 문서 흐름대로), `relative`(자기 자신 기준 이동, 원래 공간 유지), `absolute`(가장 가까운 positioned 조상 기준, 문서 흐름에서 빠짐), `fixed`(뷰포트 기준, 스크롤해도 고정), `sticky`(스크롤 시 특정 위치에 고정).

### 3. Flexbox와 Grid의 차이?

> Flexbox는 **1차원**(한 줄/한 열)의 정렬에 최적화되어 네비게이션, 카드 행렬에 적합하다. Grid는 **2차원**(행+열)의 복잡한 레이아웃에 적합하며, 전체 페이지 레이아웃이나 대시보드에 사용한다. 실무에서는 대부분 Flexbox로 충분하고, 복잡한 그리드가 필요할 때 Grid를 쓴다.

### 4. box-sizing: border-box란?

> 기본값 `content-box`는 width에 padding과 border가 **추가**되어 총 크기가 커지지만, `border-box`는 width 안에 padding과 border가 **포함**되어 직관적으로 크기를 계산할 수 있다. 실무에서는 `* { box-sizing: border-box; }`를 전역 설정하는 것이 표준이다.

### 5. CSS specificity(특이성/우선순위)는 어떻게 계산?

> `!important` > 인라인 스타일(1000) > ID 선택자(100) > 클래스/속성 선택자(10) > 태그 선택자(1) 순이다. 같은 특이성이면 나중에 선언된 것이 우선한다. `!important`는 유지보수를 어렵게 만들므로 사용을 지양한다.

### 6. 반응형 웹이란? 어떻게 구현?

> 하나의 HTML로 모바일/태블릿/데스크톱 등 다양한 화면 크기에 대응하는 웹이다. CSS 미디어 쿼리(`@media`), 유동 단위(`%`, `vw`, `rem`), Flexbox/Grid를 사용하며, `<meta name="viewport">` 태그가 필수다. Tailwind CSS에서는 `md:`, `lg:` 접두사로 간편하게 구현한다.

### 7. CSS-in-JS(Styled Components)와 Tailwind CSS 비교?

> CSS-in-JS는 JS 파일 안에 CSS를 작성하여 컴포넌트와 스타일을 결합하지만 런타임 오버헤드가 있다. Tailwind CSS는 유틸리티 클래스를 HTML에 직접 적용하며 빌드 시 미사용 CSS를 제거(Purge)하여 번들이 가볍다. Next.js에서는 Tailwind CSS가 공식 권장이며, 서버 컴포넌트와 호환성이 좋다.

### 8. z-index가 작동하지 않는 이유?

> `z-index`는 `position`이 `static`이 아닌 요소(`relative`, `absolute`, `fixed`, `sticky`)에서만 작동한다. 또한 **Stacking Context**(쌓임 맥락)가 부모에 의해 생성되면, 자식의 z-index는 그 맥락 안에서만 비교된다.

### 9. display: none vs visibility: hidden vs opacity: 0 차이?

> `display: none`은 DOM에서 공간도 차지하지 않고 완전히 사라진다. `visibility: hidden`은 보이지 않지만 공간은 차지한다. `opacity: 0`은 투명하지만 공간을 차지하고 **클릭 이벤트도 동작**한다. 접근성 측면에서 `display: none`은 스크린 리더도 무시한다.

### 10. Tailwind CSS의 장점?

> 클래스명을 고민할 필요 없이 유틸리티 클래스(`flex`, `p-4`, `bg-blue-500`)를 조합하여 빠르게 스타일링한다. 미사용 CSS 자동 제거로 번들이 작고, 반응형(`md:`)과 다크모드(`dark:`)를 접두사 하나로 처리한다. 디자인 시스템의 일관성(spacing, color)이 자동으로 유지된다.

## JavaScript (15문)

### 11. var, let, const의 차이?

> `var`는 함수 스코프이고 호이스팅 시 undefined로 초기화된다. `let`은 블록 스코프이고 재할당 가능하며, `const`는 블록 스코프이고 재할당 불가(객체 내부 변경은 가능)하다. 실무에서는 `const`를 기본으로 쓰고, 재할당이 필요할 때만 `let`을 쓴다.

### 12. 호이스팅(Hoisting)이란?

> 변수와 함수 **선언**이 코드 실행 전에 해당 스코프의 최상단으로 끌어올려지는 현상이다. `var`는 선언만 호이스팅(값은 undefined), `let`/`const`는 호이스팅되지만 TDZ(Temporal Dead Zone)로 인해 선언 전 접근 시 ReferenceError가 발생한다. 함수 선언문은 전체가 호이스팅된다.

### 13. 클로저(Closure)란?

> 함수가 선언된 **렉시컬 환경(스코프)**을 기억하여, 외부 함수가 종료된 후에도 외부 변수에 접근할 수 있는 현상이다. React의 `useState`가 클로저를 활용하며, 데이터 은닉과 상태 유지에 사용된다.

### 14. this 바인딩 규칙?

> 일반 함수에서 `this`는 **호출 방식**에 따라 결정된다. 전역: `window`, 메서드: 호출한 객체, `new`: 새로 생성된 인스턴스, `call`/`apply`/`bind`: 명시적 지정. 화살표 함수는 자신만의 `this`가 없고 **선언 시점의 외부 this**를 상속한다. React에서는 화살표 함수를 주로 사용하여 this 문제를 회피한다.

### 15. == vs === 차이?

> `==`(동등)은 타입 변환(Type Coercion) 후 비교하여 `"1" == 1`이 `true`가 된다. `===`(일치)은 타입 변환 없이 타입과 값 모두 비교하여 `"1" === 1`이 `false`다. 실무에서는 **항상 `===`을 사용**한다.

### 16. 이벤트 루프(Event Loop)를 설명하세요.

> JavaScript는 싱글 스레드로, Call Stack에서 동기 코드를 실행하고 비동기 작업(setTimeout, fetch)은 Web API에 위임한다. 작업 완료 시 콜백을 Task Queue(매크로)나 Microtask Queue(Promise)에 넣고, Call Stack이 비면 Event Loop가 큐에서 콜백을 꺼내 실행한다. Microtask(Promise)가 Macrotask(setTimeout)보다 우선 실행된다.

### 17. Promise란? async/await와의 관계?

> Promise는 비동기 작업의 **미래 결과**를 나타내는 객체로, pending→fulfilled(성공) 또는 rejected(실패) 상태를 가진다. `async/await`는 Promise의 **문법적 설탕(Syntactic Sugar)**으로, `.then()` 체이닝 대신 동기 코드처럼 읽기 쉽게 작성할 수 있다. `await`는 Promise가 resolve될 때까지 해당 함수의 실행을 일시 정지한다.

### 18. 구조 분해 할당(Destructuring)이란?

> 배열이나 객체에서 값을 추출하여 변수에 할당하는 문법이다. `const { name, age } = user;`(객체), `const [first, ...rest] = arr;`(배열). 기본값 설정(`= default`)과 중첩 구조 분해도 가능하며, 함수 매개변수에서 Props를 받을 때 필수적으로 사용된다.

### 19. 스프레드 연산자(...)의 용도?

> 배열/객체를 **얕은 복사**하거나 합칠 때 사용한다. `[...arr1, ...arr2]`(배열 합치기), `{ ...obj, name: "new" }`(객체 복사+수정). React에서 상태 불변성을 유지할 때 필수적이며, `setState(prev => ({ ...prev, name: "new" }))`처럼 사용한다.

### 20. map, filter, reduce의 차이?

> `map`은 배열의 각 요소를 **변환**하여 새 배열 반환, `filter`는 조건에 맞는 요소만 **걸러서** 새 배열 반환, `reduce`는 배열을 **하나의 값으로 축소**(합계, 그룹화 등) 한다. 셋 다 원본 배열을 변경하지 않으며, React에서 리스트 렌더링에 `map`이 필수이다.

### 21. 얕은 복사(Shallow Copy)와 깊은 복사(Deep Copy)?

> 얕은 복사는 객체의 1단계만 복사하여 중첩 객체는 참조를 공유한다(`{ ...obj }`, `Object.assign`). 깊은 복사는 중첩 객체까지 모두 새로 복사한다(`structuredClone()`, `JSON.parse(JSON.stringify())`). React 상태 업데이트에서 불변성을 지키려면 변경하는 깊이까지 모두 복사해야 한다.

### 22. 이벤트 버블링과 캡처링?

> 이벤트가 발생하면 캡처링(최상위→타겟)으로 내려갔다가 버블링(타겟→최상위)으로 올라간다. 기본적으로 핸들러는 버블링 단계에서 실행된다. `event.stopPropagation()`으로 전파를 막고, 이벤트 위임(Event Delegation)은 버블링을 활용해 부모에 하나의 핸들러를 두는 패턴이다.

### 23. debounce와 throttle의 차이?

> `debounce`는 마지막 호출 후 일정 시간이 지나야 실행(검색 자동완성: 타이핑 멈추면 API 호출). `throttle`은 일정 시간 간격으로 최대 1번 실행(스크롤 이벤트: 100ms마다 한 번). 둘 다 과도한 함수 호출을 방지하여 성능을 최적화한다.

### 24. 옵셔널 체이닝(?.)과 널 병합 연산자(??)란?

> `user?.address?.city`는 중간에 null/undefined가 있으면 에러 대신 undefined를 반환한다. `value ?? "기본값"`은 value가 null 또는 undefined일 때만 기본값을 사용한다(`||`와 달리 `0`, `""`는 유효한 값으로 취급). 둘 다 안전한 데이터 접근에 필수적이다.

### 25. ES Modules (import/export)란?

> JavaScript의 공식 모듈 시스템으로, `export`로 내보내고 `import`로 가져온다. `export default`는 파일당 1개, Named Export는 여러 개 가능하다. Tree Shaking(사용 안 하는 코드 제거)이 가능하여 번들 크기를 줄인다. CommonJS(`require`)는 Node.js 전용이며, 브라우저와 Next.js는 ES Modules를 사용한다.

## TypeScript (10문)

### 26. TypeScript를 왜 사용하나요?

> 정적 타입 검사로 **컴파일 단계에서 버그를 잡아**내어 런타임 에러를 방지한다. IDE의 자동완성과 리팩토링이 강력해지고, 코드 자체가 문서 역할을 하여 팀 협업이 쉬워진다. Next.js, React 생태계가 TypeScript를 기본으로 지원하며, 2026년 현재 프론트엔드 실무 표준이다.

### 27. interface와 type의 차이?

> `interface`는 `extends`로 확장하고 같은 이름으로 선언하면 자동 병합(Declaration Merging)된다. `type`은 유니온(`|`), 인터섹션(`&`), 유틸리티 타입 등 더 유연한 조합이 가능하다. 객체 타입은 `interface`, 유니온/조건부 타입은 `type`을 사용하는 것이 관례이다.

### 28. 제네릭(Generics)이란?

> 타입을 **매개변수처럼** 받아서 재사용 가능한 컴포넌트/함수를 만드는 문법이다. `function identity<T>(arg: T): T`에서 `T`는 호출 시 결정된다. React에서 `useState<number>(0)`, `Promise<User>` 등에 사용되며, 타입 안전성을 유지하면서 코드 재사용성을 높인다.

### 29. any, unknown, never의 차이?

> `any`는 모든 타입을 허용하고 타입 검사를 포기(사용 지양!). `unknown`은 모든 타입을 허용하지만 사용 전 **타입 좁히기(Narrowing)가 필수**여서 안전하다. `never`는 절대 발생할 수 없는 타입(에러를 throw하는 함수, 빈 유니온)이다. `any` 대신 `unknown`을 사용하자.

### 30. 유틸리티 타입을 아는 대로 설명하세요.

> `Partial<T>`(모든 속성 선택적), `Required<T>`(모든 속성 필수), `Pick<T, K>`(특정 속성만 선택), `Omit<T, K>`(특정 속성 제외), `Record<K, V>`(키-값 매핑), `Readonly<T>`(읽기 전용). 실무에서 `Partial`(폼 수정 시), `Pick`/`Omit`(API 응답 가공 시)을 가장 많이 사용한다.

### 31. 타입 좁히기(Narrowing)란?

> 넓은 타입을 좁은 타입으로 확정하는 기법이다. `typeof`(원시 타입), `instanceof`(클래스), `in`(속성 존재), 사용자 정의 타입 가드(`is` 키워드)를 사용한다. `if (typeof value === "string") { value.toUpperCase() }`처럼 조건문 내에서 TypeScript가 자동으로 타입을 추론한다.

### 32. React에서 이벤트 타입은?

> `React.ChangeEvent<HTMLInputElement>`(input 변경), `React.FormEvent<HTMLFormElement>`(폼 제출), `React.MouseEvent<HTMLButtonElement>`(클릭). `e.target.value`의 타입이 자동으로 `string`이 된다. Props 타입은 `interface Props { onClick: (id: number) => void; }`처럼 명시한다.

### 33. as와 타입 단언(Type Assertion)이란?

> TypeScript에게 "나는 이 값의 타입을 안다!"고 알려주는 것이다. `const input = document.getElementById("name") as HTMLInputElement;` 처럼 사용한다. 타입 시스템을 우회하는 것이므로 **남용하면 안 된다**. 대부분의 경우 타입 좁히기(Narrowing)로 해결하는 것이 안전하다.

### 34. tsconfig.json의 strict 옵션이란?

> `strict: true`는 `strictNullChecks`, `noImplicitAny`, `strictFunctionTypes` 등 7개 엄격한 타입 체크를 한 번에 켜는 옵션이다. null/undefined 가능성을 강제로 체크하고, any 타입 사용을 금지한다. 새 프로젝트에서는 반드시 켜야 하며, Next.js `create-next-app`은 기본으로 활성화되어 있다.

### 35. Zod란? TypeScript와 어떻게 연동?

> Zod는 **런타임 데이터 검증** 라이브러리로, 스키마를 정의하면 TypeScript 타입이 자동 추론된다. `const schema = z.object({ name: z.string() }); type User = z.infer<typeof schema>;`로 스키마와 타입을 하나의 소스에서 관리한다. Server Actions의 FormData 검증, API 응답 검증에 사용한다.

## React (20문)

### 36. React가 뭔가요? 왜 사용하나요?

> UI를 **컴포넌트** 단위로 쪼개어 재사용하는 JavaScript 라이브러리다. Virtual DOM으로 효율적으로 화면을 업데이트하고, 선언적(Declarative) 코드로 "어떻게(How)"가 아닌 "무엇을(What)" 보여줄지 기술한다. 거대한 생태계(Next.js, React Native)와 커뮤니티가 강점이다.

### 37. Virtual DOM이란?

> 실제 DOM의 가벼운 복사본을 메모리에 유지하고, 상태 변경 시 **이전 Virtual DOM과 새 Virtual DOM을 비교(Diffing)**하여 실제로 변경된 부분만 실제 DOM에 반영(Reconciliation)한다. 전체 DOM을 다시 그리는 것보다 효율적이며, React의 성능 최적화 핵심 메커니즘이다.

### 38. 리렌더링은 언제 발생하나요?

> **State 변경**, **Props 변경**, **부모 리렌더링** 3가지 경우에 발생한다. Context 값 변경도 해당 Context를 구독한 컴포넌트를 리렌더링한다. `React.memo`로 Props가 같으면 리렌더링을 건너뛸 수 있고, `useMemo`/`useCallback`으로 불필요한 계산/함수 재생성을 방지한다.

### 39. useState와 useReducer의 차이?

> `useState`는 단순한 상태(토글, 카운터, 입력값)에 적합하고, `useReducer`는 여러 관련 상태를 하나의 로직으로 관리하거나 상태 전환이 복잡할 때(폼의 다단계 상태, 장바구니 CRUD) 적합하다. `useReducer`는 액션 타입으로 의도가 명확해지고 테스트하기 쉽다.

### 40. useEffect란? 언제 사용하나요?

> 컴포넌트가 렌더링된 **이후** 부수 효과(Side Effect)를 실행하는 훅이다. API 호출, 이벤트 리스너 등록, 타이머 설정 등에 사용한다. 의존성 배열로 실행 시점을 제어하며(`[]`=마운트 시 1번, `[dep]`=dep 변경 시), 클린업 함수로 메모리 누수를 방지한다. React 19/Next.js에서는 서버 컴포넌트의 async/await가 useEffect의 데이터 패칭을 대체한다.

### 41. useRef란? 언제 사용하나요?

> 렌더링 사이에 값을 유지하지만 **값이 변해도 리렌더링을 일으키지 않는** 훅이다. DOM 요소 직접 접근(`ref.current.focus()`), 이전 값 기억, 타이머 ID 저장 등에 사용한다. `useState`와 달리 `.current` 값 변경 시 화면이 다시 그려지지 않는 것이 핵심 차이다.

### 42. 커스텀 훅이란?

> `use` 접두사로 시작하는 함수에 **훅 로직을 추출**하여 재사용하는 패턴이다. `useLocalStorage`, `useDebounce`, `useIntersectionObserver` 등 반복되는 로직을 한 곳에서 관리한다. 상태 로직(관심사)을 UI에서 분리하여 테스트가 쉽고 코드가 깔끔해진다.

### 43. Context API란? 언제 사용하나요?

> Props를 중간 컴포넌트를 거치지 않고 **깊은 자식에게 직접 전달**하는 메커니즘이다. 테마, 인증 상태, 언어 설정 등 앱 전역에서 필요한 데이터에 사용한다. 단, Context 값이 변경되면 구독한 **모든** 컴포넌트가 리렌더링되므로, 자주 변하는 데이터에는 Zustand 같은 외부 상태 관리 도구가 적합하다.

### 44. React.memo, useMemo, useCallback의 차이?

> `React.memo`는 **컴포넌트**를 감싸서 Props가 같으면 리렌더링을 건너뛴다. `useMemo`는 **계산된 값**을 메모이제이션하여 비용이 큰 연산의 반복을 방지한다. `useCallback`은 **함수**를 메모이제이션하여 자식에게 전달하는 콜백의 참조 동일성을 유지한다. React 19의 React Compiler는 이들을 자동으로 처리한다.

### 45. key Props가 왜 중요한가요?

> React가 리스트의 각 항목을 **고유하게 식별**하여 효율적으로 업데이트하기 위함이다. key가 없거나 index를 key로 쓰면, 항목이 추가/삭제/재정렬될 때 잘못된 DOM을 재사용하여 **버그**가 발생한다. DB의 고유 ID를 key로 사용하는 것이 정석이다.

### 46. 제어 컴포넌트 vs 비제어 컴포넌트?

> 제어 컴포넌트는 **React state**가 폼 값을 관리(`value={state}` + `onChange`)하여 실시간 유효성 검사가 가능하다. 비제어 컴포넌트는 **DOM**이 값을 관리(`ref`로 접근)하여 간단한 폼에 적합하다. React Hook Form은 비제어 방식으로 리렌더링을 최소화하면서 제어 컴포넌트의 기능을 제공한다.

### 47. TanStack Query(React Query)를 왜 사용하나요?

> 서버 상태(API 데이터)를 전문적으로 관리하는 라이브러리다. 자동 캐싱, 백그라운드 리패칭, 로딩/에러 상태 자동 관리, 무한 스크롤(Infinite Query), 옵티미스틱 업데이트를 제공한다. `useEffect` + `useState`로 직접 구현하면 코드가 복잡해지고 캐시/재시도/에러 처리를 반복해야 하는 문제를 해결한다.

### 48. Zustand를 왜 사용하나요?

> React의 클라이언트 전역 상태를 간결하게 관리하는 라이브러리다. Redux 대비 보일러플레이트가 극히 적고(`create()` 한 줄), Provider 래핑이 불필요하며, TypeScript 지원이 뛰어나다. 장바구니, 사이드바 상태, 모달 상태 등 **서버 상태가 아닌 UI 상태**에 사용한다.

### 49. React Hook Form + Zod 조합의 장점?

> React Hook Form은 비제어 방식으로 리렌더링을 최소화하여 폼 성능이 뛰어나다. Zod resolver를 연동하면 스키마 기반 유효성 검사와 TypeScript 타입 추론이 자동으로 이루어져, 검증 로직과 타입 정의를 한 곳에서 관리할 수 있다. Server Actions에서도 동일한 Zod 스키마를 재사용할 수 있다.

### 50. React 19의 주요 변경점?

> `use()` 훅으로 Promise와 Context를 직접 읽고, `useActionState`로 서버 액션의 상태를 관리하며, `useOptimistic`으로 낙관적 업데이트를 구현한다. `useFormStatus`로 폼 전송 상태를 추적하고, React Compiler가 `useMemo`/`useCallback`을 자동 처리한다. `ref`가 Props로 전달 가능해져 `forwardRef`가 불필요해졌다.

### 51. Props Drilling이란? 해결 방법?

> 데이터를 중간 컴포넌트들을 거쳐 깊은 자식까지 전달하는 것으로, 중간 컴포넌트가 불필요하게 Props를 전달만 한다. 해결 방법은 **Context API**(전역 데이터), **Zustand**(클라이언트 상태), **컴포넌트 합성(Composition)**(children으로 직접 전달), **서버 컴포넌트**(각자 데이터 패칭)이다.

### 52. 에러 바운더리(Error Boundary)란?

> 자식 컴포넌트에서 발생한 렌더링 에러를 **잡아서 대체 UI**를 보여주는 컴포넌트다. 에러가 발생해도 앱 전체가 크래시되지 않고 해당 영역만 에러 UI로 교체된다. 클래스 컴포넌트의 `getDerivedStateFromError`로 구현하며, Next.js에서는 `error.tsx` 파일이 자동으로 Error Boundary 역할을 한다.

### 53. Suspense란?

> 비동기 데이터가 준비될 때까지 **로딩 UI(fallback)**를 보여주는 선언적 방법이다. Next.js에서 서버 컴포넌트의 데이터 패칭을 감싸면 스트리밍으로 준비된 부분부터 보여준다. `loading.tsx`는 `<Suspense>`의 약속된 파일 버전이다. React 19에서는 `use()` 훅과 함께 클라이언트에서도 활용한다.

### 54. 합성(Composition) 패턴이란?

> 컴포넌트를 결합하여 복잡한 UI를 만드는 패턴이다. `children` Props로 부모가 자식의 내용을 결정하고, Render Props나 Slot 패턴으로 유연한 구조를 만든다. Next.js에서는 **클라이언트 컴포넌트 안에 서버 컴포넌트를 넣을 때 children으로 주입**하는 것이 핵심 패턴이다.

### 55. React에서 상태를 어디에 두어야 하나요?

> **로컬 상태**(useState): 해당 컴포넌트에서만 사용하는 UI 상태. **서버 상태**(TanStack Query): API에서 오는 데이터. **전역 클라이언트 상태**(Zustand): 여러 컴포넌트에서 공유하는 UI 상태(장바구니, 사이드바). **URL 상태**(searchParams): 검색, 필터, 정렬. Next.js에서는 서버 컴포넌트가 직접 데이터를 가져오므로 클라이언트 상태의 필요성이 크게 줄어든다.

## Next.js (25문)

### 56. Next.js를 왜 사용하나요? React만으로는 안 되나요?

> React는 UI 라이브러리일 뿐, 라우팅/서버 렌더링/데이터 패칭/빌드 최적화를 직접 구성해야 한다. Next.js는 이 모든 것을 프레임워크로 제공하여 **파일 시스템 라우팅, SSR/SSG/ISR, 서버 컴포넌트, 이미지/폰트 최적화, API Routes**가 기본 내장된다. "설정보다 관습(Convention over Configuration)" 철학으로 생산성이 높다.

### 57. 서버 컴포넌트(RSC)란?

> 서버에서만 실행되어 HTML로 완성된 채 클라이언트에 전달되는 컴포넌트다. 해당 컴포넌트의 **JavaScript가 브라우저로 전송되지 않아(Zero Bundle Size)** 앱이 극도로 가벼워진다. DB 직접 접근, API 키 은닉, 무거운 라이브러리 사용이 안전하게 가능하다. Next.js App Router에서는 **모든 컴포넌트가 기본적으로 서버 컴포넌트**이다.

### 58. 'use client'의 정확한 의미?

> "서버에서 렌더링하지 않겠다"가 아니라 "이 컴포넌트의 **JS 코드를 브라우저로도 보내겠다**"는 선언이다. 서버에서 HTML은 먼저 생성(SSR)되고, 브라우저에서 하이드레이션으로 이벤트가 활성화된다. 상호작용(onClick, useState)이 필요한 **최소한의 컴포넌트**에만 적용해야 한다(나뭇잎 패턴).

### 59. 하이드레이션(Hydration)이란?

> 서버에서 생성된 정적 HTML에 **JavaScript를 결합**하여 인터랙티브한 앱으로 만드는 과정이다. 서버 HTML과 브라우저의 렌더링 결과가 다르면 Hydration Mismatch 에러가 발생한다. 대표적 원인은 `Date`, `Math.random()`, `window` 분기이며, `useEffect`에서 클라이언트 전용 로직을 처리하거나 `suppressHydrationWarning`으로 해결한다.

### 60. SSR, SSG, ISR의 차이와 사용 시나리오?

> **SSR**(매 요청마다 서버 렌더링): 실시간 데이터, 개인화 페이지. **SSG**(빌드 시 미리 생성): 변하지 않는 페이지(블로그, 약관). **ISR**(주기적 재생성): 가끔 변하는 페이지(상품 목록, 뉴스). Next.js 15에서는 기본이 동적(SSR)이고, `cache: 'force-cache'`로 SSG, `next: { revalidate: 60 }`으로 ISR을 명시적으로 설정한다.

### 61. Next.js 15에서 캐싱이 어떻게 바뀌었나요?

> **Opt-out → Opt-in으로 전환**되었다. 14 이전에는 기본적으로 모든 fetch를 캐싱했지만(Opt-out), 15부터는 기본적으로 캐싱하지 않는다(Opt-in). 개발자가 `cache: 'force-cache'`나 `next: { revalidate }` 등을 **명시적으로 선언**해야만 캐싱된다. 데이터 신선도가 기본 보장되어 "왜 데이터가 안 바뀌지?" 버그가 사라졌다.

### 62. Server Actions란?

> `'use server'` 지시어로 선언한 서버 전용 비동기 함수로, `<form action={fn}>`에 직접 바인딩한다. 프레임워크가 **숨겨진 API 엔드포인트를 자동 생성**하여 fetch/URL/JSON 코드 없이 서버 함수를 호출한다. RPC의 현대적 구현이며, `revalidatePath`/`redirect`로 캐시 무효화와 페이지 이동을 처리한다.

### 63. Route Handler(route.ts)와 Server Actions의 차이?

> Route Handler는 RESTful API 엔드포인트(`GET`, `POST` 등)를 수동으로 설계하며, **외부 시스템(모바일 앱, 서드파티)**에 데이터를 제공할 때 적합하다. Server Actions는 프레임워크가 엔드포인트를 자동 생성하며, **내부 폼/CRUD**에 최적화되어 타입 안전성과 캐시 연동이 자동이다.

### 64. revalidatePath, revalidateTag, updateTag 차이?

> `revalidatePath`는 URL 경로의 모든 캐시를 무효화하는 광역 도구다. `revalidateTag`는 특정 태그가 붙은 캐시만 정밀 무효화하여 다중 경로의 동일 데이터를 한 번에 처리한다. `updateTag`(Next.js 16)는 서버 캐시를 파괴하면서 현재 화면을 보고 있는 클라이언트까지 즉시 갱신한다.

### 65. useFormStatus와 useActionState 차이?

> `useFormStatus`는 **부모 `<form>`의 전송 상태(pending)**를 추적하여 버튼 비활성화/로딩 UI에 사용한다. 반드시 form 내부의 자식 컴포넌트에서 호출해야 한다. `useActionState`는 **서버 액션의 반환값(state)**을 클라이언트 상태로 동기화하여, 성공/에러 메시지를 화면에 렌더링한다. prevState로 이전 상태를 서버가 기억한다.

### 66. useOptimistic이란?

> 서버 응답을 기다리지 않고 **UI를 즉시 업데이트**하는 훅이다. 사용자가 "좋아요"를 누르면 0ms에 하트가 빨갛게 변하고, 백그라운드에서 서버 통신이 이루어진다. 서버에서 에러가 발생하면 **자동으로 원래 상태로 롤백**된다. 네트워크 지연을 사용자가 인지하지 못하게 하여 UX를 극대화한다.

### 67. Middleware란? 어떤 용도?

> 요청이 서버에 도달하기 **전에** 실행되는 코드로, 인증 체크(비로그인→로그인 리다이렉트), 국제화(언어 감지), A/B 테스트, Rate Limiting에 사용한다. Edge Runtime에서 실행되어 전 세계 CDN에서 빠르게 동작한다. `middleware.ts` 파일을 프로젝트 루트에 생성한다.

### 68. 병렬 라우트(@folder)란?

> 폴더 이름에 `@`를 붙여 하나의 URL에서 **여러 독립적인 UI 슬롯**을 동시에 렌더링하는 기법이다. `layout.tsx`에서 `children`, `analytics`, `feed` 등의 Props로 받아 배치한다. 한 슬롯이 느려도 다른 슬롯은 즉시 표시되며, 에러도 해당 슬롯 내에서 격리된다.

### 69. 인터셉팅 라우트란?

> 내부 링크 클릭(Soft Navigation) 시 **원래 목적지 대신 현재 화면 위에 모달**로 표시하면서 URL은 정확히 변경하는 기법이다. `(..)` 문법으로 라우팅 흐름을 가로채며, 새로고침(Hard Navigation)시에는 전체 화면 페이지가 렌더링된다. 병렬 라우트(`@modal`)와 결합하여 인스타그램 스타일 모달을 구현한다.

### 70. PPR(Partial Prerendering)이란?

> 하나의 페이지에서 정적 껍데기(Static Shell)는 빌드 시 미리 생성하고, 동적 부분은 `<Suspense>` 구멍으로 남겨두어 **스트리밍으로 채워넣는** 기술이다. 사용자는 0초에 껍데기를 보고, 느린 데이터는 로딩 후 자연스럽게 채워진다. `'use cache'`와 `cacheLife`로 컴포넌트 단위 캐싱(마이크로 캐싱)을 구현한다.

### 71. generateStaticParams란?

> 동적 라우트(`[id]`)의 페이지를 **빌드 시 미리 생성**하는 함수다. DB에서 모든 ID를 가져와 `[{ id: "1" }, { id: "2" }]` 형태로 반환하면, 해당 페이지들이 정적 HTML로 구워져 CDN에서 0초에 서빙된다. `dynamicParams: false`로 명단 외 접근을 404 처리할 수 있다.

### 72. generateMetadata란?

> URL 파라미터에 따라 **동적으로 메타데이터**(title, description, OG 태그)를 생성하는 함수다. 상품 상세 페이지마다 다른 제목과 OG 이미지를 설정하여 SEO와 소셜 공유를 최적화한다. 같은 URL의 fetch는 Request Memoization으로 중복 호출되지 않는다.

### 73. loading.tsx와 error.tsx의 역할?

> `loading.tsx`는 데이터 로딩 중 **자동으로 Suspense fallback**으로 사용되어 스켈레톤 UI를 보여준다. `error.tsx`는 해당 라우트에서 에러 발생 시 **Error Boundary**로 작동하여 에러 UI와 `reset()` 재시도 버튼을 제공한다. `error.tsx`는 반드시 `'use client'`로 선언해야 한다.

### 74. not-found.tsx와 notFound() 함수?

> `not-found.tsx`는 존재하지 않는 경로 접근 시 보여줄 커스텀 404 페이지다. `notFound()` 함수는 URL은 존재하지만 **데이터가 없을 때**(DB에 해당 ID 없음) 프로그래밍 방식으로 404를 발생시킨다. HTTP 404 상태 코드도 정확히 반환하여 SEO에 올바른 신호를 보낸다.

### 75. Next.js에서 환경 변수는 어떻게 관리?

> `.env.local` 파일에 설정하며, `NEXT_PUBLIC_` 접두사가 있으면 **브라우저에서 접근 가능**(공개), 없으면 **서버에서만 접근**(비밀)한다. DB 비밀번호, API 키는 절대 `NEXT_PUBLIC_`을 붙이면 안 된다. Vercel 배포 시에는 대시보드의 Environment Variables에 별도 설정한다.

### 76. Soft Navigation과 Hard Navigation 차이?

> Soft Navigation은 `<Link>`를 통한 클라이언트 사이드 이동으로, 전체 새로고침 없이 변경된 부분만 교체하며 **레이아웃 상태가 유지**된다. Hard Navigation은 브라우저 새로고침이나 URL 직접 입력으로, 서버에 완전히 새로운 HTML을 요청하여 **모든 상태가 초기화**된다. `<Link>`와 `<a>` 태그의 핵심 차이점이다.

### 77. Prefetching이란?

> `<Link>` 컴포넌트가 **뷰포트에 보이는 순간** 해당 페이지의 데이터를 백그라운드에서 미리 다운로드하는 기술이다. 사용자가 실제로 클릭했을 때 이미 데이터가 준비되어 있어 **0ms에 가까운 페이지 전환**이 가능하다. 프로덕션에서만 작동하며, `prefetch={false}`로 비활성화할 수 있다.

### 78. Route Groups ()란?

> 폴더 이름을 `(marketing)`, `(app)` 처럼 소괄호로 감싸면 **URL에 영향을 주지 않으면서** 레이아웃을 분리할 수 있다. 마케팅 랜딩 페이지와 앱 대시보드에 각각 다른 Root Layout을 적용할 때 사용한다. Private Folder(`_folder`)는 라우팅에서 제외되어 유틸리티 함수를 안전하게 배치할 수 있다.

### 79. Colocation이란?

> 관련 파일(컴포넌트, 스타일, 테스트)을 **같은 폴더에 배치**하는 원칙이다. Next.js App Router에서는 `page.tsx`가 아닌 파일은 라우트로 인식되지 않으므로, 같은 폴더에 컴포넌트를 안전하게 배치할 수 있다. 별도의 `components/` 폴더를 찾아 헤매지 않아도 된다.

### 80. next/image와 일반 img 태그 차이?

> `next/image`는 자동 WebP/AVIF 변환(30~50% 용량 절감), 기기별 리사이징, Lazy Loading(기본), CLS 방지(width/height 필수)를 제공한다. 일반 `<img>`는 이 모든 것을 수동으로 처리해야 하며, Next.js의 ESLint 규칙이 `<img>` 사용을 경고한다. LCP 이미지에는 `priority` 속성을 추가한다.

## 웹 기초/CS (10문)

### 81. HTTP와 HTTPS 차이?

> HTTP는 평문 전송, HTTPS는 TLS/SSL로 **암호화** 전송이다. HTTPS는 도청 방지(암호화), 서버 인증(피싱 방지), 데이터 무결성(변조 방지)을 보장한다. 모든 현대 웹사이트에 필수이며, 구글은 HTTPS 사이트에 SEO 가산점을 준다.

### 82. REST API란?

> URL은 **자원(명사)**을, HTTP 메서드는 **행동(동사)**을 나타내는 아키텍처 스타일이다. `GET /products`(조회), `POST /products`(생성), `PATCH /products/1`(수정), `DELETE /products/1`(삭제). URL에 동사를 넣지 않고(`/getProducts` ❌) 메서드로 행동을 구분한다.

### 83. 쿠키, 세션, JWT 차이?

> **쿠키**는 브라우저에 저장되어 매 요청에 자동 전송되는 데이터(4KB). **세션**은 서버가 사용자 상태를 저장하고 클라이언트에는 Session ID만 쿠키로 전달. **JWT**는 서버가 상태를 저장하지 않고(Stateless) 토큰 자체에 정보를 담아 클라이언트가 보관. JWT는 Scale-out에 유리하고 Session은 보안이 강하다.

### 84. XSS와 CSRF 차이?

> **XSS**는 해커가 사이트에 악성 스크립트를 삽입하여 다른 사용자의 데이터를 탈취하는 공격이다(방어: 이스케이프, HttpOnly 쿠키). **CSRF**는 사용자가 로그인한 상태에서 해커 사이트를 방문하면, 쿠키가 자동 전송되어 위조 요청이 실행되는 공격이다(방어: SameSite 쿠키, CSRF 토큰).

### 85. CORS란?

> 브라우저의 Same-Origin Policy에 의해 다른 출처(프로토콜+도메인+포트)의 리소스 접근이 차단될 때, 서버가 `Access-Control-Allow-Origin` 헤더로 특정 출처를 **허가**하는 메커니즘이다. CORS 에러는 서버에서 해결해야 하며, Next.js의 Server Actions는 같은 출처이므로 CORS 문제가 없다.

### 86. 브라우저 렌더링 파이프라인?

> HTML→**DOM**, CSS→**CSSOM**, DOM+CSSOM→**Render Tree**, **Layout**(위치/크기 계산), **Paint**(픽셀 그리기), **Composite**(레이어 합성). Reflow(Layout 재계산)가 가장 비싸고, transform/opacity 변경은 Composite만 실행되어 가장 효율적이다. 애니메이션에 left/top 대신 transform을 사용하는 이유다.

### 87. 이벤트 루프란?

> JavaScript는 싱글 스레드로, Call Stack에서 동기 코드를 실행하고 비동기 작업(setTimeout, fetch)은 Web API에 위임한다. 작업 완료 시 콜백을 Queue에 넣고, **Call Stack이 비면** Event Loop가 Queue에서 콜백을 꺼내 실행한다. Microtask(Promise)가 Macrotask(setTimeout)보다 우선한다.

### 88. localStorage vs sessionStorage vs Cookie?

> **localStorage**: 영구 저장, 5~10MB, JS로 접근. **sessionStorage**: 탭 닫으면 삭제, 5~10MB, JS로 접근. **Cookie**: 서버 자동 전송, 4KB, HttpOnly로 JS 차단 가능. 인증 토큰은 HttpOnly Cookie에, UI 설정(다크모드)은 localStorage에 저장한다.

### 89. Web Accessibility(접근성)란?

> 장애인, 고령자 등 모든 사용자가 웹을 동등하게 이용할 수 있도록 하는 것이다. 시맨틱 HTML, alt 속성, aria 속성, 키보드 내비게이션, 충분한 색상 대비를 제공한다. 법적 의무(한국 장애인차별금지법)이기도 하며, SEO와 UX에도 긍정적이다.

### 90. 웹 성능 최적화 방법?

> **로딩 최적화**: 코드 스플리팅, 이미지 최적화(next/image), 폰트 최적화(next/font), Lazy Loading. **렌더링 최적화**: 불필요한 리렌더링 방지(memo, useMemo), Reflow 최소화(transform 사용). **네트워크 최적화**: 캐싱(ISR, CDN), Gzip/Brotli 압축, HTTP/2. **측정**: Lighthouse, Core Web Vitals.

## 기타 실무/소프트 스킬 (10문)

### 91. Git에서 merge와 rebase 차이?

> merge는 두 브랜치의 히스토리를 유지하며 merge commit이 생긴다. rebase는 커밋을 다른 브랜치 위에 재배치하여 직선 히스토리를 만든다. 공유된 브랜치에서는 rebase 사용을 지양하며, 팀에서는 squash merge(여러 커밋→1개)를 주로 사용한다.

### 92. CI/CD란?

> **CI**(Continuous Integration): 코드 변경을 자주 병합하고 자동으로 빌드/테스트를 실행. **CD**(Continuous Deployment): 테스트 통과 시 자동으로 배포. GitHub Actions에서 PR마다 lint, 타입 체크, E2E 테스트를 실행하고, main 병합 시 Vercel이 자동 배포하는 것이 전형적인 파이프라인이다.

### 93. 코드 리뷰에서 무엇을 보나요?

> 코드 스타일 일관성, 네이밍 명확성, 보안 취약점(API 키 노출 등), 성능 이슈(불필요한 리렌더링), 에러 처리 여부, 테스트 추가 여부를 확인한다. "왜 이렇게 구현했는가?"에 대한 근거를 물어보고, 더 나은 방법을 제안한다.

### 94. 기술 부채(Technical Debt)란?

> 빠른 출시를 위해 최적이 아닌 코드를 작성하여 **나중에 갚아야 할 비용**이 발생하는 것이다. 리팩토링, 테스트 부재, 레거시 의존성 등이 누적되면 새 기능 개발 속도가 점점 느려진다. 스프린트마다 일정 비율(20% 등)을 기술 부채 해소에 할당하는 것이 실무 관행이다.

### 95. Agile/Scrum이란?

> **Agile**은 짧은 주기(Sprint, 1~2주)로 개발→피드백→개선을 반복하는 방법론이다. **Scrum**은 Agile의 구체적 프레임워크로, Sprint Planning(계획), Daily Standup(매일 15분 공유), Sprint Review(데모), Retrospective(회고)로 구성된다. JIRA/Linear로 이슈를 관리한다.

### 96. 프론트엔드와 백엔드 협업 시 주의할 점?

> API 명세(Swagger/OpenAPI)를 먼저 합의하고, Request/Response 타입을 공유한다. 프론트엔드가 API 개발 전에 Mock 데이터로 작업할 수 있도록 하고, 에러 응답 형식(status code, message)을 표준화한다. 타입을 공유 패키지(모노레포)로 관리하면 불일치를 방지한다.

### 97. 테스트를 왜 작성해야 하나요?

> 코드 변경 시 **기존 기능이 깨지지 않음을 보장**한다. 유닛 테스트로 비즈니스 로직을, E2E 테스트로 핵심 사용자 흐름을 검증한다. 테스트가 있으면 리팩토링이 자유로워지고, 코드 리뷰 시 동작 검증이 자동화된다. 테스트 피라미드(유닛 70% > 통합 20% > E2E 10%)를 지킨다.

### 98. 최근 관심 있는 기술/트렌드는?

> (예시) React 19의 Server Components와 React Compiler, Next.js의 PPR과 마이크로 캐싱, 타입 안전한 풀스택 개발(tRPC, Zod), AI 코드 어시스턴트(Copilot) 활용, Edge Computing 등. **본인이 실제로 관심 있고 해봤던 것**을 구체적으로 말해야 한다.

### 99. 새로운 기술을 어떻게 학습하나요?

> 공식 문서를 먼저 읽고, 간단한 토이 프로젝트로 직접 사용해본다. 핵심 개념을 블로그나 노션에 정리하고, 실무 프로젝트에 점진적으로 적용한다. 커뮤니티(GitHub Discussions, Discord)에서 질문하고, 기술 컨퍼런스(FEConf 등) 발표를 시청한다. **"공식 문서 → 직접 구현 → 정리 → 적용"** 사이클이 핵심이다.

### 100. 프론트엔드 개발자로서 본인의 강점은?

> (자신만의 이야기를 준비해야 한다!) 기술적 강점(TS 타입 설계, 성능 최적화, 접근성 등), 소프트 스킬(커뮤니케이션, 문서화, 코드 리뷰), 학습 태도(새 기술 빠른 습득, 커뮤니티 참여) 중에서 **실제 경험과 결과**를 들어 구체적으로 답변해야 한다.
