## 1. 테스트가 뭔데? 왜 해야 하는데? 🔴

> 💡 **비유: 자동차 출고 전 검사**
> 우리가 만든 웹사이트 = 공장에서 갓 조립된 자동차
> 테스트 = 도로에 나가기 전에 "브레이크 작동하나? 핸들 돌아가나?" 확인하는 것!
>
> 테스트 없이 배포하면?
> = 브레이크 확인 안 하고 고속도로에 올라가는 것! 🚗💥

```
"코드를 수정했더니 다른 기능이 깨졌다!" — 이게 현실.

예시: 로그인 기능을 수정했는데
     → 장바구니 페이지가 갑자기 안 열림!
     → 상품 검색이 작동 안 함!
     → 결제 버튼이 사라짐!

사람이 일일이 확인하면?
  → 100개 페이지를 하나하나 클릭하며 확인해야 함
  → 시간: 3시간
  → 실수: 10개 페이지 깜빡 잊음
  → 결과: 버그가 고객에게 도달! 😱

자동화 테스트가 확인하면?
  → 100개 테스트를 컴퓨터가 30초 만에 전부 확인!
  → 실수: 0개 (기계는 까먹지 않음!)
  → 결과: 버그를 배포 전에 잡음! ✅
```

## 2. 테스트의 3가지 종류 — 테스트 피라미드 🔴

> 💡 **비유: 자동차 검사의 3단계**

```
테스트 피라미드:

        /  E2E 테스트  \         ← 가장 느리지만 가장 현실적!
       / 통합 테스트     \        ← 부품들이 잘 맞물리는지
      / 유닛 테스트       \       ← 나사 하나하나 점검
     ────────────────────
```

### 유닛 테스트 (Unit Test)

```
뭐야?  함수 하나, 컴포넌트 하나를 따로 떼어내서 테스트
도구:  Vitest, Jest
비유:  "이 나사가 규격에 맞는지 확인" (부품 단위!)

예시:
  ✅ 가격 계산 함수에 10000을 넣으면 "10,000원"이 나오는가?
  ✅ formatDate 함수에 날짜를 넣으면 "2026-03-25"가 나오는가?

특징:
  ⚡ 매우 빠름 (1초에 수백 개!)
  😐 하지만 "전체 흐름"은 못 봄 (나사는 멀쩡한데 자동차가 안 달릴 수 있음)
```

### 통합 테스트 (Integration Test)

```
뭐야?  여러 부품이 합쳐졌을 때 잘 작동하는지 테스트
도구:  Vitest + Testing Library
비유:  "엔진 + 변속기를 연결했을 때 잘 맞물리는가?"

예시:
  ✅ Server Action을 호출하면 DB에 정확히 저장되는가?
  ✅ API에 요청을 보내면 올바른 JSON이 돌아오는가?

특징:
  🔄 보통 속도
  👍 부품 간 연결을 확인! 하지만 "사용자 화면"은 못 봄
```

### E2E 테스트 (End-to-End Test) ⭐ 오늘 배울 것!

```
뭐야?  실제 브라우저를 열어서, 실제 사용자처럼 클릭/입력/이동!
도구:  Playwright ⭐ (우리가 배울 것!)
비유:  "실제로 도로에서 출발 → 주행 → 주차까지 전체 시운전!"

예시:
  ✅ 브라우저를 열고 → 로그인 페이지에 가서 → 아이디/비밀번호 입력 →
     로그인 버튼 클릭 → 대시보드가 뜨는가?
  ✅ 상품 검색 → 클릭 → 장바구니 담기 → 결제 → 완료 화면!

특징:
  🐢 가장 느림 (실제 브라우저를 열어야 하니까!)
  🏆 하지만 가장 현실적! (사용자가 실제로 겪는 것과 동일!)
  🎯 "진짜 되는지"를 확인하는 최종 방어선!
```

### 실무에서의 비율

```
E2E (Playwright):      10~20%  ← 핵심 흐름만! (회원가입→로그인→구매)
통합 테스트:           20~30%  ← API + DB 연동
유닛 테스트 (Vitest):  50~70%  ← 함수, 유틸리티

왜 E2E를 적게?
  → 느리고 비용이 많이 들어서 (CI 실행 시간 = 돈!)
  → 핵심 비즈니스 흐름만 E2E로 보장하고
  → 세부 로직은 유닛/통합으로 촘촘하게!
```

## 3. Playwright가 뭔데? 🔴

> 💡 **비유: 로봇 비서**
> Playwright = 컴퓨터가 조종하는 투명한 로봇 비서!
> 이 로봇이 실제 브라우저를 열고, 마우스를 움직이고, 키보드를 치고,
> 화면에 뭐가 뜨는지 확인까지 해줌!

```
Playwright vs 다른 도구:

  Selenium: 2004년 탄생. 원조! 하지만 느리고 불안정함 😢
  Cypress:  인기 많았음. 하지만 하나의 브라우저만 지원.
  Playwright: 2020년 Microsoft가 만듦. 현재 최강! ⭐

Playwright가 최강인 이유:
  ✅ 크로스 브라우저: Chrome, Firefox, Safari를 동시에 테스트!
  ✅ Auto-waiting: 요소가 준비될 때까지 알아서 기다려줌!
     → 다른 도구: sleep(3000)으로 강제 대기 필요 (불안정!)
     → Playwright: "버튼이 클릭 가능해질 때까지" 자동 대기! (안정적!)
  ✅ Codegen: 마우스 클릭만 하면 테스트 코드를 자동 생성!
  ✅ Trace Viewer: 실패 시 단계별 스크린샷 + 네트워크 로그!
  ✅ Next.js 공식 지원: webServer 연동으로 개발 서버 자동 실행!
```

## 4. 설치하기 🔴

### Step 1: Next.js 프로젝트에서 설치

```bash
# ⚠️ 반드시 Next.js 프로젝트 루트 폴더에서 실행!
# (package.json이 있는 폴더!)

npm init playwright@latest
```

### Step 2: 설치 마법사 질문에 답하기

```
질문이 하나씩 나올 거야. 이렇게 답하면 돼:

❓ Where to put your end-to-end tests?
   → tests    (Enter!)
   (테스트 파일을 넣을 폴더 이름. tests가 가장 일반적!)

❓ Add a GitHub Actions workflow?
   → Yes      (Y + Enter!)
   (나중에 GitHub에 코드를 올리면 자동으로 테스트 실행!)

❓ Install Playwright browsers?
   → Yes      (Y + Enter!)
   (Chromium, Firefox, WebKit 3개의 브라우저 엔진을 다운로드!)
   (⚠️ 용량이 좀 큼! 약 400MB. 인내심을 가지고 기다리자!)
```

### Step 3: 설치 완료 후 확인

```
설치가 끝나면 프로젝트에 이런 파일/폴더가 생김:

my-next-app/
├── src/app/                    ← 기존 Next.js 코드 (그대로!)
│
├── tests/                      ← 🧪 [새로 생김!] E2E 테스트 폴더!
│   └── example.spec.ts         ← 예제 테스트 파일 (나중에 수정할 거야)
│
├── playwright.config.ts        ← ⚙️ [새로 생김!] Playwright 설정 파일!
│
├── .github/
│   └── workflows/
│       └── playwright.yml      ← 🤖 [새로 생김!] CI/CD 자동화 파일!
│
└── package.json                ← (scripts에 테스트 명령어가 추가됨)

이 3개만 기억하면 돼:
  1. tests/ 폴더       → 테스트 코드를 작성하는 곳
  2. playwright.config.ts → 설정을 변경하는 곳
  3. playwright.yml    → GitHub에서 자동 실행하는 설정
```

## 5. playwright.config.ts — 설정 파일 이해하기 🟡

> 💡 **비유: 테스트 로봇의 작전 지시서**
> "어디를 테스트할지, 어떤 브라우저를 쓸지, 실패하면 어떻게 할지"를 적어놓은 문서!

```ts
// playwright.config.ts
// 이 파일이 Playwright의 두뇌! 하나하나 설명할게.

import { defineConfig, devices } from "@playwright/test";

export default defineConfig({

  // ─────────────────────────────────────────────
  // 📁 테스트 파일 위치
  // ─────────────────────────────────────────────
  testDir: "./tests",
  // "tests 폴더 안에 있는 .spec.ts 파일들을 테스트해라!"

  // ─────────────────────────────────────────────
  // ⏱️ 타임아웃 (테스트 하나당 최대 시간)
  // ─────────────────────────────────────────────
  timeout: 30 * 1000,
  // 30초 안에 끝나지 않으면 → "이 테스트 실패!" 처리
  // 왜? 무한 루프에 빠져서 영원히 안 끝나는 걸 방지!

  // ─────────────────────────────────────────────
  // 🔄 재시도 횟수
  // ─────────────────────────────────────────────
  retries: process.env.CI ? 2 : 0,
  // CI 환경(GitHub Actions): 실패하면 2번 더 시도!
  //   (네트워크 불안정 등 일시적 실패를 방지)
  // 로컬(내 컴퓨터): 재시도 안 함! (빠른 피드백!)
  //
  // process.env.CI가 뭐야?
  //   GitHub Actions에서 실행하면 자동으로 CI=true가 설정됨!
  //   내 컴퓨터에서는 undefined → 0번 재시도!

  // ─────────────────────────────────────────────
  // 👷 병렬 실행 워커 수
  // ─────────────────────────────────────────────
  workers: process.env.CI ? 1 : undefined,
  // CI: 1개씩 순서대로 (안정적!)
  // 로컬: 자동 (CPU 코어 수만큼 병렬 실행! 빠름!)

  // ─────────────────────────────────────────────
  // 📊 리포트 형식
  // ─────────────────────────────────────────────
  reporter: "html",
  // 테스트 끝나면 예쁜 HTML 웹 페이지 리포트 생성!
  // npx playwright show-report 로 열어볼 수 있음!

  // ─────────────────────────────────────────────
  // 🌐 모든 테스트에 적용되는 공통 설정
  // ─────────────────────────────────────────────
  use: {
    // 기본 URL 설정!
    baseURL: "http://localhost:3000",
    // 이걸 설정하면:
    //   page.goto("/about")  ← 이렇게만 써도 됨! (편함!)
    //   page.goto("http://localhost:3000/about")  ← 이렇게 안 써도 됨!

    // 실패 시 자동 스크린샷 찍기
    screenshot: "only-on-failure",
    // 테스트가 실패하면 → 그 순간의 화면을 사진으로 저장!
    // "뭘 보고 있었길래 실패했지?" 확인할 때 유용!

    // 실패 시 액션 추적 (Trace)
    trace: "on-first-retry",
    // 첫 번째 재시도 때 모든 동작을 녹화!
    // 클릭, 입력, 네트워크 요청을 타임라인으로 확인 가능!
  },

  // ─────────────────────────────────────────────
  // 🖥️ 테스트할 브라우저 목록
  // ─────────────────────────────────────────────
  projects: [
    {
      name: "chromium",
      use: { ...devices["Desktop Chrome"] },
      // Chrome 브라우저에서 테스트!
    },
    {
      name: "firefox",
      use: { ...devices["Desktop Firefox"] },
      // Firefox에서도 테스트! (크로스 브라우저!)
    },
    {
      name: "webkit",
      use: { ...devices["Desktop Safari"] },
      // Safari에서도! (Mac/iOS 사용자를 위해!)
    },

    // 📱 모바일 테스트도 가능!
    // {
    //   name: "mobile-chrome",
    //   use: { ...devices["Pixel 5"] },
    // },
  ],

  // ─────────────────────────────────────────────
  // ⭐ 가장 중요! Next.js 연동!
  // ─────────────────────────────────────────────
  webServer: {
    command: "npm run dev",
    // 테스트 시작 전에 자동으로 Next.js 개발 서버를 켜줌!
    // 수동으로 npm run dev를 먼저 실행할 필요 없음!

    url: "http://localhost:3000",
    // 이 주소가 응답할 때까지 기다린 후 테스트 시작!

    reuseExistingServer: !process.env.CI,
    // 로컬: 이미 npm run dev가 돌고 있으면 → 그걸 재사용!
    // CI: 항상 새로 실행!

    timeout: 120 * 1000,
    // 서버가 뜨는데 최대 2분까지 기다림!
    // Next.js 빌드가 좀 오래 걸릴 수 있으니까!
  },
});
```

### 💡 가장 중요한 설정 3가지

```
1. baseURL: "http://localhost:3000"
   → page.goto("/about") 이렇게 짧게 쓸 수 있게!

2. webServer: { command: "npm run dev" }
   → 테스트 실행하면 Next.js 서버가 자동으로 켜짐!
   → 따로 터미널에서 npm run dev 안 해도 됨!

3. projects: [chromium, firefox, webkit]
   → 3가지 브라우저에서 동시에 테스트!
```

## 6. 첫 번째 테스트 작성하기! 🔴

> 이제 직접 테스트를 써보자! 긴장하지 마. 아주 쉬워.

### Step 1: 테스트 파일 만들기

```
tests/ 폴더 안에 있는 example.spec.ts를 지우고,
home.spec.ts를 새로 만들자!

tests/
└── home.spec.ts   ← 이 파일을 만들 거야!
```

### Step 2: 코드 작성

```ts
// tests/home.spec.ts

// 1. Playwright에서 test와 expect를 가져옴
//    test = "이런 테스트를 할 거야!"라고 선언하는 함수
//    expect = "이게 맞는지 확인해봐!"라고 검증하는 함수
import { test, expect } from "@playwright/test";

// ──────────────────────────────────────
// 테스트 1: 홈페이지가 제대로 뜨는지 확인!
// ──────────────────────────────────────
test("홈페이지가 정상적으로 렌더링된다", async ({ page }) => {
  // page = Playwright가 열어주는 실제 브라우저 탭!
  // 이 안에서 진짜 사용자처럼 행동할 수 있어!

  // 1단계: 홈페이지로 이동!
  //   (baseURL이 설정되어 있으니까 "/"만 써도 됨!)
  await page.goto("/");

  // 2단계: 페이지 제목(브라우저 탭 이름)을 확인!
  //   <title>Create Next App</title> 이 있는지!
  await expect(page).toHaveTitle(/Next/);
  //   /Next/ = 정규식! "Next"라는 글자가 포함되어 있으면 OK!
  //   정확히 일치 안 해도 됨! "Create Next App"도 통과!
});

// ──────────────────────────────────────
// 테스트 2: 다른 페이지로 이동할 수 있는지 확인!
// ──────────────────────────────────────
test("About 페이지로 이동할 수 있다", async ({ page }) => {
  // 1단계: 홈에서 시작
  await page.goto("/");

  // 2단계: "About"이라는 링크를 찾아서 클릭!
  //   getByRole("link") = <a> 태그를 찾아라!
  //   { name: "About" } = 텍스트가 "About"인 것!
  await page.getByRole("link", { name: "About" }).click();

  // 3단계: 주소창이 /about으로 바뀌었는지 확인!
  await expect(page).toHaveURL("/about");

  // 4단계: About 페이지에 특정 텍스트가 보이는지 확인!
  await expect(page.getByText("소개")).toBeVisible();
});
```

### 코드 한 줄씩 뜯어보기

```
test("이름", async ({ page }) => { ... })
│      │         │        │
│      │         │        └─ page = 실제 브라우저 탭!
│      │         └─ 비동기! (브라우저 조작은 시간이 걸리니까)
│      └─ 테스트 이름 (한글 OK! 무엇을 테스트하는지 명확하게!)
└─ "테스트를 정의한다!"

await page.goto("/")
│      │      │
│      │      └─ "/" = 홈페이지 (baseURL + "/")
│      └─ goto = "이 주소로 가라!" (주소창에 URL 입력하는 것!)
└─ await = "페이지가 완전히 뜰 때까지 기다려!" (비동기!)

await expect(page).toHaveTitle(/Next/)
│       │      │          │
│       │      │          └─ 정규식으로 "Next" 포함 확인!
│       │      └─ toHaveTitle = <title> 태그 확인!
│       └─ expect = "이것을 검증해봐!"
└─ await = "검증이 끝날 때까지 기다려!"

await page.getByRole("link", { name: "About" }).click()
│       │        │        │                        │
│       │        │        │                        └─ 클릭!
│       │        │        └─ 텍스트가 "About"인 것!
│       │        └─ getByRole = 접근성 역할로 요소를 찾아라!
│       │           "link" = <a> 태그!
│       └─ page = 현재 브라우저 탭
└─ await = "요소를 찾고 클릭할 때까지 기다려!"
```

## 면접 대비

### 🔴 필수 질문

**Q1. E2E 테스트란? 유닛 테스트와 어떻게 다른가요?**

> E2E(End-to-End) 테스트는 실제 브라우저를 자동화하여 사용자의 전체 사용 흐름을 검증하는 테스트이다. 유닛 테스트가 함수 하나를 격리하여 검증한다면, E2E는 프론트엔드 UI부터 백엔드 API, 데이터베이스까지 전체 시스템이 연동되는 것을 확인한다. 테스트 피라미드에서 가장 느리지만 가장 현실적인 검증 방법이며, 핵심 비즈니스 흐름(로그인, 결제 등)에 적용한다.

**Q2. Playwright를 선택한 이유?**

> Playwright는 Microsoft가 만든 E2E 테스트 도구로, Chromium/Firefox/WebKit 세 가지 브라우저를 동시에 테스트할 수 있다. 가장 큰 장점은 Auto-waiting으로, 요소가 DOM에 존재하고 화면에 보이고 클릭 가능해질 때까지 자동으로 기다려줘서 Flaky(불안정한) 테스트가 거의 없다. 또한 Codegen으로 마우스 클릭만으로 테스트 코드를 자동 생성할 수 있고, Next.js의 webServer 설정으로 개발 서버를 자동 실행할 수 있어 DX가 뛰어나다.
