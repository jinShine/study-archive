# 부록 Playwright 02. 핵심 API와 선택자(Locator) 완전 정복

> **키워드**: `Locator` `getByRole` `getByText` `getByTestId` `Auto-waiting` `Assertion` `click` `fill`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | Locator 우선순위, 핵심 액션(click/fill), Assertion(검증) | 매일 쓰는 API |
| 🟡 이해 | Auto-waiting, 다중 요소, 체이닝, 필터링 | 안정적 테스트 |

---

## 1. 선택자(Locator)란? — "화면에서 요소를 찾는 법" 🔴

> 💡 **비유: 식당에서 특정 사람을 찾는 법**
>
> 너는 식당에 들어갔어. 친구를 찾아야 해. 어떻게 찾을까?
>
> ❌ 나쁜 방법: "3번 테이블 왼쪽 의자에 앉은 사람!" → 자리가 바뀌면 못 찾음!
> ✅ 좋은 방법: "파란 모자 쓴 사람!" → 어디 앉았든 찾을 수 있음!
>
> 웹에서도 마찬가지야:
> ❌ CSS 선택자: `.btn-primary > span:nth-child(2)` → 스타일 바꾸면 못 찾음!
> ✅ Role 선택자: "버튼 역할을 하는 것 중 '저장'이라고 적힌 것" → 항상 찾음!

### 핵심: "사용자가 보는 것으로 찾아라!"

```
사용자는 CSS 클래스(.btn-primary)를 모른다!
사용자는 id(#submit-btn)를 모른다!

사용자가 아는 것:
  "저장 버튼이 어디있지?" → 버튼(role) + 저장(이름) = getByRole!
  "이메일 입력칸이 어디있지?" → 라벨이 "이메일"인 입력칸 = getByLabel!
  "검색어를 입력하세요... 라고 적힌 칸" → placeholder = getByPlaceholder!
```

## 2. 선택자 우선순위 — 이 순서대로 써! 🔴

```
가장 좋음 ──────────────────────────────────── 최후의 수단

1순위      2순위      3순위          4순위           5순위
getByRole  getByText  getByLabel    getByPlaceholder getByTestId
  ⭐         👍         👍             👌              🆘

❌ 절대 쓰지 마:
  page.locator(".class-name")   ← CSS 클래스는 언제든 바뀜!
  page.locator("#id")           ← id도 리팩토링 시 바뀔 수 있음!
  page.locator("div > span:nth-child(2)") ← 최악! HTML 구조가 바뀌면 끝!
```

## 3. getByRole — 1순위! 접근성 기반! ⭐ 🔴

> 💡 **HTML 태그에는 "역할(Role)"이 있다!**
> `<button>` = 역할: "button"
> `<a>` = 역할: "link"
> `<h1>` ~ `<h6>` = 역할: "heading"
> `<input type="text">` = 역할: "textbox"
> `<input type="checkbox">` = 역할: "checkbox"

### 기본 사용법

```ts
// 🔘 버튼 찾기
await page.getByRole("button", { name: "저장" }).click();
// HTML: <button>저장</button>
// 설명: "button" 역할 중에서 텍스트가 "저장"인 것!

// 🔗 링크 찾기
await page.getByRole("link", { name: "About" }).click();
// HTML: <a href="/about">About</a>
// 또는: <Link href="/about">About</Link>  (Next.js의 Link도 잡힘!)

// 📝 제목(heading) 찾기
await page.getByRole("heading", { name: "상품 목록" });
// HTML: <h1>상품 목록</h1> (h1~h6 전부 잡힘!)

// 특정 레벨의 heading만!
await page.getByRole("heading", { name: "소개", level: 2 });
// HTML: <h2>소개</h2>만! (h1은 안 잡힘!)

// 📋 입력 필드 찾기
await page.getByRole("textbox", { name: "이메일" });
// HTML: <label>이메일 <input type="text" /></label>
// 또는: <input type="text" aria-label="이메일" />

// ☑️ 체크박스 찾기
await page.getByRole("checkbox", { name: "동의합니다" });
// HTML: <label><input type="checkbox" /> 동의합니다</label>

// 📻 라디오 버튼
await page.getByRole("radio", { name: "남성" });

// 📂 네비게이션 영역
await page.getByRole("navigation");
// HTML: <nav>...</nav>
```

### 자주 쓰는 Role 모음

```
Role          HTML 태그          예시
──────────────────────────────────────────
button       <button>            "로그인", "저장"
link         <a>, <Link>         "About", "홈으로"
heading      <h1>~<h6>           "상품 목록", "회원가입"
textbox      <input type="text"> "이름", "이메일"
checkbox     <input type="checkbox"> "동의합니다"
radio        <input type="radio">    "남성", "여성"
combobox     <select>            "카테고리 선택"
navigation   <nav>               네비게이션 바
dialog       <dialog>            모달/팝업
list         <ul>, <ol>          목록
listitem     <li>                목록 항목
img          <img>               이미지
```

## 4. getByText — 2순위! 보이는 텍스트로 찾기 🔴

```ts
// 기본: 부분 일치! ("환영합니다!"에서 "환영"만 써도 찾음!)
await page.getByText("환영");
// HTML: <p>환영합니다!</p>  ← 찾음! ✅
// HTML: <span>환영해요~</span>  ← 이것도 찾음! ✅

// 정확히 일치하게 하려면:
await page.getByText("환영합니다!", { exact: true });
// HTML: <p>환영합니다!</p>  ← 찾음! ✅
// HTML: <p>환영합니다!!</p>  ← 안 찾음! ❌ (느낌표 2개니까!)

// 정규식으로 유연하게:
await page.getByText(/상품 \d+개/);
// HTML: <p>상품 5개</p>  ← 찾음! ✅
// HTML: <p>상품 123개</p>  ← 찾음! ✅
// HTML: <p>상품 없음</p>  ← 안 찾음! ❌

// 대소문자 무시:
await page.getByText(/welcome/i);
// HTML: <h1>Welcome</h1>  ← 찾음! ✅
// HTML: <h1>WELCOME</h1>  ← 찾음! ✅
```

## 5. getByLabel — 3순위! 라벨로 입력칸 찾기 🔴

> 💡 HTML `<label>` 태그와 연결된 `<input>`을 찾는 방법!

```ts
await page.getByLabel("이메일").fill("test@example.com");
// 이게 찾는 HTML:
//
//   방법 1: for + id 연결
//   <label for="email">이메일</label>
//   <input id="email" type="email" />
//
//   방법 2: label이 input을 감싸기
//   <label>이메일 <input type="email" /></label>
//
//   방법 3: aria-label
//   <input type="email" aria-label="이메일" />

await page.getByLabel("비밀번호").fill("1234");
await page.getByLabel("전화번호").fill("010-1234-5678");
```

## 6. getByPlaceholder — 4순위! 힌트 텍스트로 찾기 🟡

```ts
await page.getByPlaceholder("검색어를 입력하세요").fill("노트북");
// HTML: <input placeholder="검색어를 입력하세요" />

await page.getByPlaceholder("이메일을 입력해주세요").fill("a@b.com");
```

> ⚠️ placeholder가 없는 input에는 사용 불가!
> → getByLabel이나 getByRole을 우선 사용하자!

## 7. getByTestId — 5순위! 최후의 수단! 🟡

> 다른 방법으로 도저히 찾을 수 없을 때만 사용!

```tsx
// 컴포넌트에 data-testid 속성을 추가해야 함!
<div data-testid="product-card">
  <h3>초경량 노트북</h3>
  <p>1,200,000원</p>
</div>
```

```ts
// 테스트에서 찾기
await page.getByTestId("product-card").click();
```

```
왜 최후의 수단인가?
  → data-testid는 사용자가 볼 수 없는 속성!
  → "사용자가 보는 것으로 찾아라" 원칙에 위배!
  → 하지만 복잡한 UI에서 다른 방법이 없을 때는 유용!

예: 동일한 텍스트가 여러 개 있을 때
  <div data-testid="header-cart">장바구니</div>
  <div data-testid="sidebar-cart">장바구니</div>
  → getByText("장바구니")는 2개가 잡혀서 에러!
  → getByTestId("header-cart")로 정확히 하나만!
```

## 8. 핵심 액션(Actions) — 사용자처럼 행동하기 🔴

> 요소를 찾았으면, 이제 뭘 할 건지! 클릭? 입력? 선택?

### 클릭하기

```ts
// 단순 클릭 ⭐ (가장 많이 사용!)
await page.getByRole("button", { name: "저장" }).click();

// 더블 클릭
await page.getByText("항목").dblclick();

// 우클릭 (컨텍스트 메뉴)
await page.getByText("항목").click({ button: "right" });

// 특정 위치 클릭 (요소의 좌상단에서 x:10, y:20 지점)
await page.getByTestId("canvas").click({ position: { x: 10, y: 20 } });
```

### 입력하기 (텍스트)

```ts
// fill() = 기존 내용을 지우고 새로 입력! ⭐
await page.getByLabel("이메일").fill("test@example.com");
// 기존에 뭐가 적혀있든 전부 지우고 → "test@example.com" 입력!

// clear() = 내용만 지우기
await page.getByLabel("이메일").clear();

// pressSequentially() = 한 글자씩 타이핑! (자동완성 테스트용)
await page.getByLabel("검색").pressSequentially("React", { delay: 100 });
// R...e...a...c...t 한 글자씩! 100ms 간격으로!
// 검색 자동완성 드롭다운을 테스트할 때 유용!
```

### 선택하기 (드롭다운)

```ts
// <select> 태그에서 값 선택
await page.getByLabel("카테고리").selectOption("electronics");
// HTML: <select>
//         <option value="electronics">전자제품</option>
//         <option value="clothing">의류</option>
//       </select>

// value가 아닌 보이는 텍스트로 선택
await page.getByLabel("카테고리").selectOption({ label: "전자제품" });
```

### 체크박스 / 라디오

```ts
// 체크하기 ✅
await page.getByRole("checkbox", { name: "동의합니다" }).check();

// 체크 해제하기 ☐
await page.getByRole("checkbox", { name: "동의합니다" }).uncheck();

// 이미 체크되어 있어도 check()는 에러 안 남! (멱등성!)
```

### 키보드 입력

```ts
// Enter 키 누르기 ⭐
await page.getByLabel("검색").press("Enter");
// 검색창에 입력 후 Enter로 제출할 때!

// Tab 키
await page.keyboard.press("Tab");

// 단축키
await page.keyboard.press("Control+A");  // 전체 선택
await page.keyboard.press("Meta+C");     // Mac에서 Cmd+C (복사)
```

### 페이지 이동

```ts
// URL로 이동 ⭐
await page.goto("/products");
// baseURL + "/products" = "http://localhost:3000/products"

// 뒤로 가기
await page.goBack();

// 앞으로 가기
await page.goForward();

// 새로고침
await page.reload();
```

## 9. Assertion(검증) — "이게 맞는지 확인!" 🔴

> 💡 **비유: 시험지 채점!**
> 액션(클릭, 입력) = 시험 문제를 푸는 것
> Assertion(검증) = 정답을 채점하는 것!
> 채점 없이 문제만 풀면 → 맞았는지 모름!

### 페이지 전체 검증

```ts
// URL 확인 ⭐ (가장 많이 사용!)
await expect(page).toHaveURL("/products");
await expect(page).toHaveURL(/\/products\/\d+/);  // 정규식!
// /products/1, /products/42 등 전부 통과!

// 페이지 제목(탭 이름) 확인
await expect(page).toHaveTitle("상품 목록 | My Shop");
await expect(page).toHaveTitle(/상품/);  // "상품" 포함이면 OK!
```

### 요소 보이는지 확인

```ts
const button = page.getByRole("button", { name: "저장" });

// 보이는지! ⭐
await expect(button).toBeVisible();

// 안 보이는지!
await expect(button).not.toBeVisible();
// 또는:
await expect(button).toBeHidden();

// 💡 toBeVisible vs toBeHidden:
//   toBeVisible: DOM에 존재하고 + 화면에 보여야 함!
//   toBeHidden: DOM에 없거나, display:none이거나, visibility:hidden
```

### 텍스트 확인

```ts
// 정확한 텍스트 ⭐
await expect(button).toHaveText("저장하기");
// <button>저장하기</button>  ← 통과!
// <button>저장</button>      ← 실패! (정확히 일치해야!)

// 텍스트 포함 (부분 일치!) ⭐
await expect(button).toContainText("저장");
// <button>저장하기</button>  ← 통과! ("저장"이 포함되어 있으니까!)
// <button>임시 저장 중...</button>  ← 통과!
```

### 활성화/비활성화 확인

```ts
// 버튼이 클릭 가능한 상태인지!
await expect(button).toBeEnabled();

// 버튼이 비활성화(disabled)인지!
await expect(button).toBeDisabled();
// <button disabled>저장</button>  ← 통과!
```

### 체크 상태 확인

```ts
// 체크되어 있는지!
await expect(page.getByRole("checkbox", { name: "동의" })).toBeChecked();

// 체크 안 되어 있는지!
await expect(page.getByRole("checkbox", { name: "동의" })).not.toBeChecked();
```

### 입력값 확인

```ts
// input에 입력된 값 확인!
await expect(page.getByLabel("이메일")).toHaveValue("test@example.com");

// 빈 값인지!
await expect(page.getByLabel("이메일")).toHaveValue("");
// 또는:
await expect(page.getByLabel("이메일")).toBeEmpty();
```

### 개수 확인

```ts
// 상품 카드가 몇 개인지!
await expect(page.getByTestId("product-card")).toHaveCount(6);

// 0개인지! (검색 결과 없음)
await expect(page.getByTestId("product-card")).toHaveCount(0);
```

### 속성(Attribute) 확인

```ts
// href 확인
await expect(page.getByRole("link", { name: "About" })).toHaveAttribute("href", "/about");

// type 확인
await expect(page.getByLabel("비밀번호")).toHaveAttribute("type", "password");
```

### 부정(NOT) 검증

```ts
// 모든 검증에 .not을 붙이면 반대!
await expect(button).not.toBeVisible();     // 안 보여야 함!
await expect(button).not.toBeDisabled();    // 비활성화가 아니어야 함!
await expect(page).not.toHaveURL("/login"); // 로그인 페이지가 아니어야 함!
```

## 10. Auto-waiting — "알아서 기다려주는 마법!" 🟡

> 💡 **이것이 Playwright의 가장 강력한 무기!**

```
다른 도구 (Selenium, Cypress 등):
  "버튼을 찾을 수 없습니다!" (아직 로딩 중인데 너무 빨리 찾으려 해서!)
  → 해결: await sleep(3000);  ← 3초 강제 대기 (불안정!)
  → 3초가 충분하지 않으면? 또 실패!
  → 3초가 너무 길면? 테스트가 느려짐!

Playwright:
  "버튼이 아직 안 보여? 그럼 보일 때까지 기다릴게!"
  → 자동으로 기다림! (최대 30초, 설정 변경 가능!)
  → 0.1초 만에 보이면? 0.1초 후 즉시 실행!
  → 3초 걸리면? 3초 후 실행!
  → sleep(3000) 같은 하드코딩 필요 없음! ⭐
```

### Auto-waiting이 작동하는 과정

```
page.getByRole("button", { name: "저장" }).click() 호출 시:

  1. ⏳ 요소가 DOM에 존재하는지 확인 → 없으면 기다림!
  2. ⏳ 요소가 화면에 보이는지 확인 → display:none이면 기다림!
  3. ⏳ 요소가 안정적인지 확인 → 애니메이션 중이면 기다림!
  4. ⏳ 요소가 클릭 가능한지 확인 → 다른 요소에 가려져 있으면 기다림!
  5. ✅ 모든 조건 충족! → 클릭!

이 전체 과정이 자동! 코드 한 줄이면 됨!
```

### 그래도 수동 대기가 필요할 때 (드물지만!)

```ts
// 특정 요소가 나타날 때까지 대기
await page.waitForSelector("[data-loaded='true']");

// 네트워크 요청이 끝날 때까지 대기
await page.waitForResponse("**/api/products");
// ** = 아무 도메인이나! (와일드카드!)

// URL이 바뀔 때까지 대기
await page.waitForURL("/dashboard");

// ⚠️ 절대 쓰지 말 것!
await page.waitForTimeout(3000);  // ← 하드코딩 대기! 불안정!
// 왜 안 되나? 3초가 부족하면 실패, 충분하면 시간 낭비!
```

## 11. 여러 개의 요소 다루기 🟡

> 상품 카드가 6개 있을 때, 특정 것만 골라내는 법!

```ts
// 모든 상품 카드 선택
const cards = page.getByTestId("product-card");

// 개수 확인
await expect(cards).toHaveCount(6);

// 첫 번째 것!
await cards.first().click();

// 마지막 것!
await cards.last().click();

// n번째 (0부터 시작!)
await cards.nth(2).click();  // 3번째!

// 텍스트로 필터링!
await cards.filter({ hasText: "노트북" }).click();
// 6개 카드 중 "노트북"이 포함된 것만!

// 자식 요소에서 찾기!
await cards.first().getByRole("button", { name: "장바구니" }).click();
// 첫 번째 카드 안에 있는 "장바구니" 버튼을 클릭!
```

## 면접 대비

### 🔴 필수 질문

**Q1. Playwright에서 요소를 선택하는 우선순위는?**

> `getByRole`이 1순위로 HTML 접근성 역할(button, link, heading 등) 기반이라 UI가 변경되어도 가장 안정적이다. 그 다음 보이는 텍스트(`getByText`), 라벨(`getByLabel`), 플레이스홀더(`getByPlaceholder`) 순이며, `getByTestId`는 다른 방법으로 찾을 수 없을 때의 최후 수단이다. CSS 선택자(`.class`, `#id`)는 스타일이나 구조 변경에 깨지기 쉬워 사용하지 않는다.

**Q2. Playwright의 Auto-waiting이 왜 중요한가?**

> 클릭, 입력 등 액션 수행 시 요소가 DOM에 존재하고, 화면에 보이고, 안정적이고, 클릭 가능한 상태가 될 때까지 자동으로 대기하는 기능이다. `setTimeout(3000)` 같은 하드코딩 없이도 서버 컴포넌트의 스트리밍이나 비동기 렌더링을 안정적으로 테스트할 수 있어 Flaky(불안정한) 테스트를 방지한다.
