## 1. 디버깅 도구 3종 세트 🔴

> 💡 테스트가 실패했을 때 "왜 실패했지?" 원인을 찾는 도구들!
> 이 3개만 알면 어떤 테스트 실패든 해결할 수 있어!

### 도구 1: UI Mode — 가장 많이 사용! ⭐⭐⭐

```bash
npx playwright test --ui
```

> 💡 **비유: CCTV 관제실!**
> 로봇이 테스트를 실행하는 과정을 CCTV로 지켜보는 것!
> 테스트의 매 단계를 스크린샷으로 보여주고,
> 시간을 앞뒤로 돌리며 "이 시점에 화면이 어떻게 생겼는지" 확인 가능!

```
UI Mode가 제공하는 기능:

  📋 테스트 목록
     왼쪽에 모든 테스트가 리스트로!
     원하는 것만 골라서 클릭 한 번으로 실행!

  📸 단계별 스크린샷
     "goto → 스크린샷 → click → 스크린샷 → 검증 → 스크린샷"
     매 단계가 캡처되어 있어서 "이 시점에 화면이 어땠는지?" 바로 확인!

  ⏪ 시간 여행
     슬라이더를 앞뒤로 움직이면 테스트 과정을 탐색!
     "아, 여기서 클릭이 안 된 건 아직 버튼이 안 떴기 때문이구나!" 발견!

  🔍 DOM 스냅샷
     각 시점의 HTML 구조를 그대로 보여줌!
     "이 버튼의 클래스가 뭐였지?" "이 요소가 숨겨져 있었나?" 확인!

  🔄 Watch 모드
     코드를 수정하고 저장하면 → 자동으로 해당 테스트만 다시 실행!
     일일이 명령어를 안 쳐도 됨!

언제 사용?
  ✅ 테스트를 새로 작성할 때 (제일 많이!)
  ✅ 실패 원인을 빠르게 파악할 때
  ✅ 일상적인 개발 중 항상! ⭐
```

### 도구 2: Codegen — 테스트 코드 자동 생성! ⭐⭐

```bash
# 마법의 명령어!
npx playwright codegen http://localhost:3000
```

> 💡 **비유: 비서가 내 행동을 기록해서 보고서를 써주는 것!**
> 내가 마우스로 클릭하고 키보드로 입력하면 →
> 비서(Codegen)가 그 행동을 코드로 자동 변환!

```
사용법:

  1. 명령어를 치면 브라우저 2개가 열림!
     - 왼쪽: 실제 사이트 (여기서 마우스로 조작!)
     - 오른쪽: 코드가 실시간으로 써지는 창!

  2. 왼쪽에서 버튼을 클릭하면:
     → 오른쪽에 await page.getByRole("button", { name: "저장" }).click();
     자동으로 써짐!

  3. 입력 필드에 텍스트를 치면:
     → await page.getByLabel("이메일").fill("test@example.com");
     자동으로 써짐!

  4. 최적의 선택자를 자동 추천!
     → getByRole을 우선 사용! (가장 안정적인 선택자!)

  5. 다 끝나면 코드를 복사해서 .spec.ts에 붙여넣기!

언제 사용?
  ✅ 처음 테스트를 작성할 때 (뭘 써야 할지 모르겠을 때!)
  ✅ 복잡한 UI의 선택자를 빨리 찾고 싶을 때
  ✅ "이 요소를 getByRole로 어떻게 찾지?" 고민될 때
```

### 도구 3: Trace Viewer — 실패 원인 분석! ⭐

```bash
# trace를 수집하면서 테스트 실행
npx playwright test --trace on

# 실패한 테스트의 trace 파일 열기
npx playwright show-trace test-results/*/trace.zip
```

> 💡 **비유: 비행기 블랙박스!**
> 비행기(테스트)가 추락(실패)했을 때,
> 블랙박스(Trace)를 열면 "추락 직전에 무슨 일이 있었는지" 전부 기록되어 있음!

```
Trace Viewer가 보여주는 것:

  📸 스크린샷 타임라인
     테스트의 모든 단계를 스크린샷으로!
     "실패 직전에 화면이 어떻게 생겼는지?" 바로 확인!

  🌐 네트워크 요청/응답
     어떤 API를 호출했는지, 응답이 뭐였는지 전부!
     "API가 500을 뱉었구나!" "응답이 빈 배열이었구나!" 발견!

  📝 콘솔 로그
     console.log, console.error 전부 기록!
     "여기서 에러가 났구나!" 확인!

  🗂️ DOM 스냅샷
     각 시점의 HTML 구조!

언제 사용?
  ✅ CI(GitHub Actions)에서 실패했는데 로컬에서는 재현이 안 될 때!
  ✅ "이 클릭이 왜 안 됐지?" 원인을 모를 때
  ✅ 실패한 테스트의 네트워크 상태를 확인하고 싶을 때
```

### playwright.config.ts에서 trace 설정

```ts
use: {
  // 추천 설정! 첫 번째 재시도 때만 trace 수집!
  trace: "on-first-retry",
  // → 평소엔 수집 안 함 (빠름!)
  // → 실패해서 재시도할 때만 수집! (실패 원인 분석용!)

  // 다른 옵션들:
  // trace: "on"           ← 항상 수집 (느리지만 확실!)
  // trace: "off"          ← 수집 안 함
  // trace: "retain-on-failure" ← 실패한 것만 보관!
}
```

## 2. GitHub Actions — 자동으로 테스트 실행! 🔴

> 💡 **비유: 경비원이 24시간 지켜보는 것!**
> 코드를 GitHub에 올릴 때마다(push/PR) →
> 경비원(GitHub Actions)이 자동으로 E2E 테스트를 실행!
> 실패하면 → PR에 빨간 X 표시! → 머지 차단!

### 워크플로우 파일 (전체!)

```yaml
# .github/workflows/playwright.yml
#
# 이 파일이 하는 일:
# 1. 코드를 push하거나 PR을 만들면 → 자동으로 실행!
# 2. 브라우저를 설치하고 → Next.js를 빌드하고 → 테스트 실행!
# 3. 실패하면 → HTML 리포트를 저장!

name: Playwright Tests  # 워크플로우 이름 (GitHub에 표시!)

# 언제 실행할지?
on:
  push:
    branches: [main, develop]  # main이나 develop에 push할 때!
  pull_request:
    branches: [main]           # main으로 PR을 만들 때!

jobs:
  test:
    timeout-minutes: 60  # 최대 60분! (넘으면 강제 종료)
    runs-on: ubuntu-latest  # Ubuntu Linux에서 실행!

    steps:
      # ────────────────────────────────
      # 1단계: 코드 가져오기
      # ────────────────────────────────
      - uses: actions/checkout@v4
        # GitHub에서 우리 코드를 다운로드!

      # ────────────────────────────────
      # 2단계: Node.js 설치
      # ────────────────────────────────
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      # ────────────────────────────────
      # 3단계: 패키지 설치
      # ────────────────────────────────
      - name: Install dependencies
        run: npm ci
        # npm ci = package-lock.json 기준으로 정확히 설치!
        # npm install보다 빠르고 재현성 보장!
        # ⚠️ CI에서는 반드시 npm ci를 써야 해!

      # ────────────────────────────────
      # 4단계: Playwright 브라우저 설치
      # ────────────────────────────────
      - name: Install Playwright Browsers
        run: npx playwright install --with-deps
        # Chromium, Firefox, WebKit 다운로드!
        # --with-deps: Ubuntu에 필요한 시스템 라이브러리도 함께!
        # (libgbm, libnss3 등 → 이거 없으면 브라우저가 안 열림!)

      # ────────────────────────────────
      # 5단계: Next.js 프로덕션 빌드
      # ────────────────────────────────
      - name: Build Next.js
        run: npm run build
        # ⚠️ dev 모드가 아니라 프로덕션 빌드!
        # 실제 배포 환경과 동일한 조건에서 테스트!
        # (캐싱 동작, 최적화 등이 dev와 다름!)

      # ────────────────────────────────
      # 6단계: E2E 테스트 실행!
      # ────────────────────────────────
      - name: Run Playwright tests
        run: npx playwright test
        env:
          CI: true
          # CI=true가 설정되면:
          #   playwright.config.ts의 process.env.CI가 true!
          #   → retries: 2 (실패 시 2번 재시도!)
          #   → workers: 1 (순서대로 안정적 실행!)

      # ────────────────────────────────
      # 7단계: 실패 시 리포트 저장!
      # ────────────────────────────────
      - uses: actions/upload-artifact@v4
        if: ${{ !cancelled() }}
        # 테스트가 실패해도 리포트는 저장! (cancelled만 제외!)
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
          # 30일간 보관! GitHub에서 다운로드 가능!
```

### CI에서 실패하면 어떻게 하나?

```
1. GitHub PR 페이지에 빨간 X 표시!
   "Playwright Tests: Failed" 😱

2. Actions 탭에서 실패한 워크플로우 클릭!

3. 하단의 "Artifacts" 섹션에서 "playwright-report" 다운로드!

4. 압축 풀고 → index.html을 브라우저에서 열면!
   → 어떤 테스트가 실패했는지!
   → 실패 시점의 스크린샷!
   → 에러 메시지!
   전부 확인 가능!

5. trace 파일도 함께 저장되었으면:
   npx playwright show-trace trace.zip
   → 실패 직전의 타임라인까지 확인! 🔍
```

### 프로덕션 빌드로 테스트하는 이유

```
❌ npm run dev로 테스트하면:
  → 캐싱이 느슨하게 풀려 있음!
  → 최적화가 안 되어 있음!
  → 실제 배포 환경과 다르게 동작할 수 있음!
  → "로컬에서는 되는데 배포하면 안 돼요!" 현상!

✅ npm run build + npm start로 테스트하면:
  → 실제 배포 환경과 100% 동일!
  → 캐싱, 최적화, 정적 생성 모두 동작!
  → "CI에서 통과하면 배포해도 안전!" 보장!
```

```ts
// playwright.config.ts에서 CI일 때 build + start 사용!
webServer: {
  command: process.env.CI
    ? "npm run build && npm start"  // CI: 프로덕션 빌드!
    : "npm run dev",                // 로컬: 개발 서버!
  url: "http://localhost:3000",
  reuseExistingServer: !process.env.CI,
},
```

## 3. 실무 베스트 프랙티스 🔴

### ✅ DO (해야 할 것!)

```
1. 선택자: getByRole 우선!
   ✅ page.getByRole("button", { name: "저장" })
   ❌ page.locator(".btn-primary")
   → CSS 클래스는 디자이너가 언제든 바꿀 수 있음!
   → Role은 기능이 바뀌지 않는 한 안정적!

2. 격리: 테스트끼리 독립적으로!
   ✅ 각 테스트가 필요한 데이터를 스스로 생성!
   ❌ "테스트 A가 만든 상품을 테스트 B가 삭제"
   → 병렬 실행 시 순서가 보장 안 돼서 깨짐!

3. 대기: Auto-waiting 믿기!
   ✅ await expect(button).toBeVisible()
   ❌ await page.waitForTimeout(3000)
   → 3초가 부족하면? 실패! 3초가 넘으면? 시간 낭비!

4. 테스트 이름: "행동 → 결과"로 짓기!
   ✅ test("빈 폼 제출 시 에러 메시지가 표시된다")
   ✅ test("로그인 성공 후 대시보드로 이동한다")
   ❌ test("test 1")
   ❌ test("form test")

5. 인증: storageState 재사용!
   ✅ 한 번 로그인 → 쿠키 저장 → 모든 테스트 재사용!
   ❌ 10개 테스트마다 로그인 페이지 방문!

6. CI: 프로덕션 빌드!
   ✅ npm run build → npm start
   ❌ npm run dev (캐싱 동작이 다름!)
```

### ❌ DON'T (하면 안 되는 것!)

```
1. ❌ waitForTimeout 하드코딩!
   await page.waitForTimeout(5000);
   → 느리고 불안정! auto-waiting 사용!

2. ❌ CSS 선택자 의존!
   page.locator("div.card > span:nth-child(2)")
   → HTML 구조 바꾸면 바로 깨짐!

3. ❌ 테스트 간 순서 의존!
   "테스트 1에서 만든 데이터를 테스트 2가 사용"
   → 병렬 실행하면 순서 보장 안 됨!

4. ❌ 모든 것을 E2E로!
   "함수 하나의 로직도 E2E로 테스트"
   → 유닛 테스트가 적합! E2E는 핵심 흐름만!

5. ❌ 테스트 결과 파일을 Git에 커밋!
   test-results/, playwright-report/ 등
   → .gitignore에 추가! repo가 비대해짐!
```

## 5. 전체 명령어 치트시트 🔴

```bash
# ═══════════ 테스트 실행 ═══════════
npx playwright test                     # 전체 실행! (기본!)
npx playwright test --headed            # 브라우저 보면서!
npx playwright test --project=chromium  # Chrome만!
npx playwright test tests/auth.spec.ts  # 특정 파일만!
npx playwright test -g "로그인"          # 이름에 "로그인" 포함!

# ═══════════ 디버깅 ═══════════
npx playwright test --ui               # UI 모드 ⭐⭐⭐
npx playwright test --debug            # 디버거 모드 (breakpoint!)
npx playwright codegen localhost:3000  # 코드 자동 생성! ⭐⭐
npx playwright test --trace on         # Trace 수집!

# ═══════════ 리포트 ═══════════
npx playwright show-report             # HTML 리포트!
npx playwright show-trace trace.zip    # Trace Viewer!

# ═══════════ 관리 ═══════════
npx playwright install                 # 브라우저 업데이트!
npx playwright test --update-snapshots # 스크린샷 기준 갱신!
```

### 3가지만 기억하면 되는 핵심 명령어

```
매일 개발:   npx playwright test --ui      (GUI에서 편하게!)
코드 생성:   npx playwright codegen        (마우스로 클릭 → 코드!)
커밋 전:    npx playwright test            (전체 한 방!)
```

## 6. 테스트 전략: 뭘 테스트해야 할까? 🔴

```
실무에서 E2E로 테스트해야 하는 것:

  🔴 반드시:
  ✅ 핵심 비즈니스 흐름
     "회원가입 → 로그인 → 상품 검색 → 장바구니 → 결제"
     이게 깨지면 매출이 0이 되니까!

  ✅ 인증/인가 흐름
     "로그인 → 보호 페이지 접근 → 미로그인 시 차단"
     보안이 뚫리면 대참사!

  🟡 가능하면:
  ✅ 주요 CRUD
     "상품 등록 → 목록에 표시 → 수정 → 삭제"

  ✅ 검색 & 필터링
     "검색어 입력 → 결과 표시 → 정렬 변경"

  ✅ Next.js 특수 파일
     "404 페이지, 에러 페이지, 로딩 UI"

  🟢 E2E로 안 해도 되는 것:
  ❌ 유틸리티 함수 (formatPrice, validateEmail)
     → 유닛 테스트(Vitest)로!

  ❌ 개별 컴포넌트 렌더링
     → 컴포넌트 테스트(Testing Library)로!

  ❌ CSS 스타일 (색상, 폰트 크기)
     → 스크린샷 비교 또는 수동 확인!
```

### 테스트 피라미드 비율 (추천!)

```
E2E (Playwright):      10~20%  ← 핵심 흐름만!
통합 테스트:           20~30%  ← API + DB
유닛 테스트 (Vitest):  50~70%  ← 로직, 유틸리티

E2E가 많을수록:
  → CI 실행 시간 ⬆️ (브라우저 열고 닫는 게 느림!)
  → 유지보수 비용 ⬆️ (UI 바뀌면 테스트도 수정!)
  → 클라우드 비용 ⬆️ (GitHub Actions 실행 시간 = 과금!)

그래서 핵심 흐름만 E2E로!
나머지는 빠르고 저렴한 유닛/통합으로!
```

## 면접 대비

### 🔴 필수 질문

**Q1. Playwright 테스트를 CI에서 어떻게 운영하나요?**

> GitHub Actions에서 PR/push 시 자동 실행한다. `npm ci`로 의존성을 설치하고, `npx playwright install --with-deps`로 브라우저 엔진과 시스템 의존성을 설치한다. Next.js를 프로덕션 빌드(`npm run build`)하여 실제 배포 환경과 동일하게 테스트하며, 실패 시 HTML 리포트와 Trace를 아티팩트로 업로드하여 원인을 분석한다. `CI=true` 환경 변수로 재시도 횟수와 워커 수를 자동 조절한다.

**Q2. Playwright의 디버깅 도구 3가지를 설명해주세요.**

> 첫째, UI Mode(`--ui`)는 가장 많이 사용하는 도구로 테스트를 GUI에서 선택/실행하고 단계별 스크린샷과 DOM 스냅샷을 시간 여행하며 확인한다. 둘째, Codegen은 실제 브라우저에서 마우스 클릭만으로 테스트 코드를 자동 생성하여 최적의 선택자(getByRole)를 추천한다. 셋째, Trace Viewer는 실패한 테스트의 스크린샷 타임라인, 네트워크 로그, 콘솔 출력을 분석하여 CI에서 재현이 안 되는 실패 원인을 찾을 수 있다.

**Q3. E2E 테스트를 전체의 몇 %로 유지해야 하나요?**

> 테스트 피라미드에 따라 E2E는 10~20%만 유지한다. 브라우저 자동화는 느리고 유지보수 비용이 높으므로 "회원가입→로그인→결제" 같은 핵심 비즈니스 흐름만 커버한다. 유틸리티 함수는 유닛 테스트(Vitest, 50~70%)로, API+DB 연동은 통합 테스트(20~30%)로 채워야 CI 실행 시간과 비용의 균형을 맞출 수 있다.
