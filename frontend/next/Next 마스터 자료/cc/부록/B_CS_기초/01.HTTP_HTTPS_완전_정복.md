> **키워드**: `HTTP` `HTTPS` `요청/응답` `상태 코드` `헤더` `메서드` `TLS/SSL` `무상태성`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | HTTP 구조, 메서드, 상태 코드, 무상태성 | 면접 최다 출제! |
| 🟡 이해 | 헤더, HTTPS 원리, 쿠키, 캐시 헤더 | 실무 디버깅 |

## 1. HTTP가 뭔데? 🔴

> 💡 **비유: 식당 주문 시스템!**
> 손님(브라우저) → 주문서(요청) → 주방(서버) → 음식(응답) → 손님에게 전달!
>
> HTTP = 브라우저와 서버가 데이터를 주고받는 **약속된 규칙**!

```
HTTP = HyperText Transfer Protocol (하이퍼텍스트 전송 규약)

"규약"이라는 단어가 핵심!
  → 브라우저가 서버에게 아무렇게나 말하면 서버가 이해 못 함!
  → "이런 형식으로 보내면 이런 형식으로 답해줄게!" 약속!
  → 전 세계 모든 웹사이트가 이 약속을 따름!

HTTP의 특징:
  1. 요청(Request) - 응답(Response) 구조
  2. 무상태(Stateless) — 서버가 이전 요청을 기억 안 함!
  3. 텍스트 기반 — 사람이 읽을 수 있는 형태!
```

## 2. HTTP 요청(Request)의 구조 🔴

> 브라우저가 서버에게 보내는 "주문서"!

```
HTTP 요청 = 4가지로 구성!

┌─────────────────────────────────────────┐
│  1. 요청 라인 (Request Line)             │
│     GET /products?sort=price HTTP/1.1    │
│     [메서드] [경로+쿼리] [HTTP 버전]      │
├─────────────────────────────────────────┤
│  2. 헤더 (Headers)                       │
│     Host: www.myshop.com                │
│     Content-Type: application/json       │
│     Authorization: Bearer eyJhbG...      │
│     Cookie: sessionId=abc123             │
├─────────────────────────────────────────┤
│  3. 빈 줄 (헤더와 본문의 경계!)            │
│                                         │
├─────────────────────────────────────────┤
│  4. 본문 (Body) — POST/PUT/PATCH만!      │
│     { "name": "노트북", "price": 1200000 }│
└─────────────────────────────────────────┘
```

## 3. HTTP 메서드 (동사!) 🔴

> 💡 **비유: 식당에서 하는 행동!**
> GET = 메뉴판 보기 (조회!)
> POST = 새 주문하기 (생성!)
> PUT/PATCH = 주문 변경하기 (수정!)
> DELETE = 주문 취소하기 (삭제!)

```
메서드    목적           본문(Body)  안전?  멱등?
─────────────────────────────────────────────────
GET      데이터 조회     없음!       ✅     ✅
POST     데이터 생성     있음!       ❌     ❌
PUT      데이터 전체 교체  있음!     ❌     ✅
PATCH    데이터 일부 수정  있음!     ❌     ❌
DELETE   데이터 삭제     없거나 있음  ❌     ✅

안전(Safe): 서버 데이터를 변경하지 않는가?
  → GET만 안전! (보기만 하니까!)

멱등(Idempotent): 여러 번 실행해도 결과가 같은가?
  → GET: 10번 조회해도 데이터 안 변함! ✅
  → DELETE: 1번 삭제든 10번 삭제든 결과는 "삭제됨"! ✅
  → POST: 10번 보내면 10개 생성됨! ❌ (멱등 아님!)
```

### 실무에서의 RESTful API

```
REST = 주소(URL)는 명사 + 행동은 메서드(동사)!

✅ RESTful (올바른 설계!)
  GET    /products         → 상품 목록 조회
  GET    /products/42      → 42번 상품 상세 조회
  POST   /products         → 새 상품 등록
  PATCH  /products/42      → 42번 상품 수정
  DELETE /products/42      → 42번 상품 삭제

❌ 안티패턴 (나쁜 설계!)
  GET    /getProducts      → 동사가 URL에! ❌
  POST   /createProduct    → 동사가 URL에! ❌
  POST   /deleteProduct/42 → DELETE를 써야지! ❌
```

## 4. HTTP 응답(Response)의 구조 🔴

> 서버가 브라우저에게 보내는 "음식 + 영수증"!

```
┌─────────────────────────────────────────┐
│  1. 상태 라인 (Status Line)              │
│     HTTP/1.1 200 OK                     │
│     [버전] [상태코드] [상태 메시지]        │
├─────────────────────────────────────────┤
│  2. 헤더 (Headers)                       │
│     Content-Type: application/json       │
│     Cache-Control: max-age=3600          │
│     Set-Cookie: token=eyJhbG...          │
├─────────────────────────────────────────┤
│  3. 빈 줄                                │
├─────────────────────────────────────────┤
│  4. 본문 (Body)                          │
│     { "id": 42, "name": "노트북" }       │
└─────────────────────────────────────────┘
```

## 5. HTTP 상태 코드 — 서버의 대답! 🔴

> 💡 **3자리 숫자! 첫 번째 자리가 의미를 결정!**

### 2xx: 성공! 🟢

```
200 OK           → 요청 성공! (가장 일반적!)
201 Created      → 새 데이터가 생성됨! (POST 성공!)
204 No Content   → 성공했지만 보낼 데이터가 없음! (DELETE 성공!)
```

### 3xx: 리다이렉트! 🔵

```
301 Moved Permanently → 영구 이동! "이 주소로 가!" (SEO에 중요!)
302 Found            → 임시 이동! "잠깐 이쪽으로!"
304 Not Modified     → 캐시된 데이터 그대로 써! (변경 없음!)
```

### 4xx: 클라이언트 잘못! 🟡

```
400 Bad Request      → 요청이 잘못됨! (데이터 형식 오류!)
401 Unauthorized     → 인증 안 됨! "누구세요? 로그인하세요!" ⭐
403 Forbidden        → 인가 안 됨! "로그인은 했는데 권한이 없어!" ⭐
404 Not Found        → 없는 페이지! "그런 거 없는데요?"
405 Method Not Allowed → "이 URL에 GET은 되는데 DELETE는 안 돼!"
409 Conflict         → 충돌! "이미 존재하는 데이터!"
422 Unprocessable    → 형식은 맞는데 내용이 이상함! (유효성 검사 실패!)
429 Too Many Requests → 요청 너무 많음! "잠깐 쉬었다 와!" (Rate Limit!)
```

### 5xx: 서버 잘못! 🔴

```
500 Internal Server Error → 서버가 터짐! "뭔가 잘못됐어요..." ⭐
502 Bad Gateway          → 서버 앞의 프록시가 응답을 못 받음!
503 Service Unavailable  → 서버 점검 중! "잠시 후 다시 와주세요!"
504 Gateway Timeout      → 서버가 너무 오래 걸림! (타임아웃!)
```

### 면접 필수! 401 vs 403

```
401 Unauthorized (인증 실패!):
  "당신이 누군지 모르겠어요. 로그인 먼저 하세요!"
  → 로그인 안 한 상태에서 마이페이지 접근!
  → 토큰이 만료됨!

403 Forbidden (인가 실패!):
  "누군지는 알겠는데, 이건 볼 권한이 없어요!"
  → 일반 사용자가 관리자 페이지 접근!
  → 로그인은 했지만 등급이 낮음!

⚠️ 401의 영어 이름이 "Unauthorized"라서 헷갈리지만,
   실제 의미는 "Unauthenticated(인증 안 됨)"이야!
```

## 6. HTTP 헤더 — 메타데이터! 🟡

> 💡 **비유: 택배 박스에 붙은 송장!**
> 박스 안의 물건(Body) 외에 "보내는 사람, 취급 주의, 유통기한" 같은 부가 정보!

### 요청 헤더 (브라우저 → 서버)

```
Content-Type: application/json
  → "내가 보내는 데이터는 JSON 형식이야!"

Authorization: Bearer eyJhbGciOiJ...
  → "내 신분증(JWT 토큰)이야! 확인해봐!"

Cookie: sessionId=abc123; theme=dark
  → "내 쿠키 정보야!" (브라우저가 자동 전송!)

Accept: application/json
  → "JSON 형식으로 응답해줘!"

User-Agent: Mozilla/5.0 (iPhone; ...)
  → "나는 iPhone Safari에서 접속했어!"
```

### 응답 헤더 (서버 → 브라우저)

```
Content-Type: application/json
  → "내가 보내는 데이터는 JSON이야!"

Set-Cookie: token=eyJ...; HttpOnly; Secure; SameSite=Strict
  → "이 쿠키를 저장해! 보안 설정도 지켜!"

Cache-Control: max-age=3600
  → "이 데이터를 1시간 동안 캐시해도 돼!"

Access-Control-Allow-Origin: https://mysite.com
  → "이 출처에서 오는 요청만 허용!" (CORS!)

Location: /products/42
  → "여기로 리다이렉트해!" (301/302와 함께!)
```

## 7. 무상태성(Stateless) — HTTP의 본질! 🔴

> 💡 **비유: 심각한 건망증 은행원!**
> 방금 신분증 보여줬는데, 1초 뒤에 다시 "고객님, 신분증 주세요?" 😱

```
HTTP는 무상태(Stateless)!
  → 서버가 이전 요청을 기억하지 않음!
  → 매 요청이 완전히 독립적!

왜 이렇게 만들었나?
  → 수백만 명의 사용자 정보를 서버 메모리에 저장하면 → 서버 터짐!
  → 무상태이기 때문에 서버를 10대, 100대 늘려도 OK! (Scale-out!)
  → 1번 서버에서 처리하든 100번 서버에서 처리하든 결과 동일!

그럼 로그인 상태는 어떻게 유지?
  → 쿠키(Cookie) + 세션(Session)!
  → JWT(JSON Web Token)!
  → "매 요청마다 신분증(토큰)을 같이 보내!" 방식!
  → (B-04에서 상세히!)
```

## 8. HTTPS — 보안 HTTP! 🔴

> 💡 **비유: 일반 우편 vs 등기 우편!**
> HTTP = 엽서 (누구나 내용을 볼 수 있음!) ❌
> HTTPS = 봉인된 등기 우편 (내용이 암호화!) ✅

```
HTTPS = HTTP + TLS/SSL (암호화!)

HTTP 통신:
  브라우저 ──── "비밀번호: 1234" ────→ 서버
              ↑ 해커가 중간에서 엿봄! 😱

HTTPS 통신:
  브라우저 ──── "aK3#xP9$mQ..." ────→ 서버
              ↑ 해커가 봐도 의미 없음! ✅
              (암호화되어서 해독 불가!)

HTTPS가 하는 일 3가지:
  1. 암호화: 데이터를 암호로 변환! (도청 방지!)
  2. 인증: "이 서버가 진짜 네이버인지?" 확인! (피싱 방지!)
  3. 무결성: 데이터가 중간에 변조되지 않았는지 확인!

실무에서:
  → 2026년 현재 모든 웹사이트가 HTTPS 필수!
  → Chrome은 HTTP 사이트에 "주의 요함" 경고!
  → Vercel, Netlify 등은 HTTPS 자동 적용!
  → 로컬 개발(localhost)은 HTTP여도 OK! (예외!)
```

### TLS 핸드셰이크 (간단 버전!)

```
브라우저 → 서버: "안녕! 암호화 통신하자!"
서버 → 브라우저: "좋아! 내 인증서(공개키)야!"
브라우저: "인증서 검증... 진짜 네이버 맞네! ✅"
브라우저 → 서버: (공개키로 암호화한 비밀키 전송!)
서버: (비밀키 수신! 이제 이걸로 암호화 통신!)
→ 이후 모든 데이터가 이 비밀키로 암호화! 🔒
```

  "로그인이 풀려요!" → Headers에서 쿠키/토큰 확인!
```

## 10. HTTP 버전 🟡

```
HTTP/1.1 (1997):
  → 현재도 많이 사용!
  → 한 번에 하나의 요청만 처리 (순차적!)
  → Keep-Alive로 연결 재사용

HTTP/2 (2015):
  → 하나의 연결에서 여러 요청 동시 처리! (멀티플렉싱!)
  → 헤더 압축!
  → 서버 푸시 (서버가 먼저 데이터를 보낼 수 있음!)
  → 대부분의 현대 웹사이트가 사용! ⭐

HTTP/3 (2022):
  → TCP 대신 QUIC(UDP 기반) 사용!
  → 연결 속도 더 빠름!
  → 구글, 유튜브 등이 사용 중!
```

## 면접 대비

### 🔴 필수 질문

**Q1. HTTP와 HTTPS의 차이?**

> HTTP는 데이터를 평문으로 전송하여 중간에서 도청당할 수 있지만, HTTPS는 TLS/SSL 프로토콜로 데이터를 암호화하여 전송한다. HTTPS는 암호화(도청 방지), 서버 인증(피싱 방지), 데이터 무결성(변조 방지) 3가지를 보장한다. 2026년 현재 모든 웹사이트는 HTTPS가 필수이며, 브라우저는 HTTP 사이트에 보안 경고를 표시한다.

**Q2. HTTP 상태 코드 401과 403의 차이?**

> 401 Unauthorized는 "인증(Authentication) 실패"로, 사용자가 누구인지 확인할 수 없을 때(토큰 없음/만료) 반환한다. 403 Forbidden은 "인가(Authorization) 실패"로, 사용자는 확인됐지만 해당 리소스에 접근할 권한이 없을 때 반환한다. 예를 들어 일반 사용자가 관리자 페이지에 접근하면 403이다.

**Q3. HTTP가 무상태(Stateless)인 이유?**

> 서버가 클라이언트의 상태를 기억하지 않기 때문에 수평 확장(Scale-out)이 가능하다. 수백만 사용자의 상태를 서버 메모리에 저장하면 서버가 터지지만, 무상태이면 동일한 서버를 여러 대 복제하여 로드밸런서로 분산할 수 있다. 로그인 같은 상태 유지는 쿠키+세션이나 JWT 토큰을 매 요청마다 함께 보내는 방식으로 해결한다.

**Q4. GET과 POST의 차이?**

> GET은 데이터를 조회하는 안전한(Safe) 메서드로, 요청 본문(Body)이 없고 데이터를 URL 쿼리 스트링에 포함한다. POST는 데이터를 생성하는 메서드로, 요청 본문에 JSON 등의 데이터를 담아 보낸다. GET은 멱등(같은 요청을 여러 번 보내도 결과 동일)하지만, POST는 여러 번 보내면 데이터가 중복 생성될 수 있어 멱등하지 않다. GET은 브라우저가 캐싱할 수 있지만 POST는 캐싱하지 않는다.

## 핵심 요약

```
┌────────────────────────────────────────────────┐
│  HTTP/HTTPS 핵심 요약                            │
├────────────────────────────────────────────────┤
│                                                │
│  HTTP = 브라우저↔서버 통신 규약!                 │
│  요청: [메서드] [URL] + 헤더 + 본문              │
│  응답: [상태코드] + 헤더 + 본문                  │
│                                                │
│  메서드: GET(조회) POST(생성) PATCH(수정)        │
│         PUT(교체) DELETE(삭제)                   │
│                                                │
