## 1. URL을 입력하면 무슨 일이? 🔴

> 💡 **면접 최고 인기 질문!** "브라우저에 URL을 입력하면 어떤 일이 일어나나요?"

```
사용자가 https://www.myshop.com을 입력하면:

Step 1: DNS 조회 🔍
  "www.myshop.com의 IP 주소가 뭐지?"
  → DNS 서버에 물어봄 → "93.184.216.34야!"
  → 도메인 이름을 IP 주소로 변환!
  (비유: "강남역 맛집"을 "서울시 강남구 역삼동 123번지"로 변환!)

Step 2: TCP 연결 🤝
  브라우저 ↔ 서버 사이에 안전한 통신 채널 개설!
  → 3-way handshake: SYN → SYN-ACK → ACK
  (비유: "여보세요?" → "네, 들려요!" → "좋아요, 시작합시다!")
  → HTTPS면 여기서 TLS 핸드셰이크도 추가!

Step 3: HTTP 요청 📤
  GET / HTTP/1.1
  Host: www.myshop.com
  → "이 주소의 HTML 문서를 주세요!"

Step 4: 서버 응답 📥
  HTTP/1.1 200 OK
  Content-Type: text/html
  → HTML 문서가 브라우저로 도착!

Step 5: 렌더링 파이프라인 🎨 (이것이 오늘의 핵심!)
  → HTML을 파싱하여 화면에 그리기!
  → 아래에서 상세 설명!

Step 6: 추가 리소스 다운로드 📦
  → HTML 안의 CSS, JS, 이미지 등을 추가 요청!
```

## 2. 렌더링 파이프라인 6단계 🔴

> 💡 **비유: 건축 과정!**
> HTML = 설계도 → DOM = 골조 → CSSOM = 인테리어 도면
> → Render Tree = 최종 시공 계획 → Layout = 측량 → Paint = 칠하기!

```
HTML 문서가 도착한 후:

  ┌──────┐    ┌──────┐    ┌───────────┐
  │ HTML │───▶│ DOM  │───▶│           │
  └──────┘    └──────┘    │ Render    │    ┌────────┐    ┌───────┐    ┌───────────┐
                          │ Tree     │───▶│ Layout │───▶│ Paint │───▶│ Composite │
  ┌──────┐    ┌──────┐    │ (합체!)   │    └────────┘    └───────┘    └───────────┘
  │ CSS  │───▶│CSSOM │───▶│           │
  └──────┘    └──────┘    └───────────┘

  1. Parse HTML → DOM (Document Object Model)
  2. Parse CSS → CSSOM (CSS Object Model)
  3. DOM + CSSOM = Render Tree (렌더 트리)
  4. Layout (레이아웃/리플로우) — 위치와 크기 계산!
  5. Paint (페인트) — 실제 픽셀로 그리기!
  6. Composite (합성) — 레이어 합치기!
```

### Step 1: HTML → DOM 🔴

```
HTML 텍스트를 트리 구조로 변환!

HTML:
  <html>
    <body>
      <h1>안녕</h1>
      <p>반갑습니다</p>
    </body>
  </html>

DOM Tree:
  html
   └── body
        ├── h1 → "안녕"
        └── p  → "반갑습니다"

DOM = Document Object Model
  → HTML 문서를 자바스크립트로 조작할 수 있는 객체 트리!
  → document.querySelector("h1") ← DOM에서 h1을 찾는 것!
```

### Step 2: CSS → CSSOM 🔴

```
CSS 텍스트를 트리 구조로 변환!

CSS:
  body { font-size: 16px; }
  h1 { color: blue; font-weight: bold; }
  p { display: none; }

CSSOM Tree:
  body (font-size: 16px)
   ├── h1 (color: blue, font-weight: bold, font-size: 16px 상속!)
   └── p  (display: none, font-size: 16px 상속!)

CSSOM = CSS Object Model
  → CSS도 트리 구조!
  → 상위 요소의 스타일이 하위로 상속(Cascade)!
```

### Step 3: Render Tree (합체!) 🔴

```
DOM + CSSOM = Render Tree!

⚠️ 핵심: display: none인 요소는 Render Tree에 포함되지 않음!

DOM:               CSSOM:              Render Tree:
  html               body(16px)          body(16px)
   └── body           ├── h1(blue)        └── h1(blue, bold)
        ├── h1         └── p(none)              → "안녕"
        └── p                             (p는 display:none이라 제외!)

→ "실제로 화면에 그려질 것들"만 남은 트리!
→ <head>, <script>, display:none 등은 제외!
```

### Step 4: Layout (레이아웃) 🔴

```
"각 요소가 화면의 어디에, 어떤 크기로 배치되는지 계산!"

Render Tree의 각 노드에 대해:
  → x좌표, y좌표 계산!
  → width, height 계산!
  → 부모/형제와의 관계에 따라 위치 결정!

예시:
  h1 → x:0, y:0, width:1200px, height:40px
  p  → (display:none이라 계산 안 함!)

💡 이 단계가 비용이 큼!
  → 요소의 크기나 위치가 바뀌면 다시 계산해야 함! (Reflow!)
```

### Step 5: Paint (페인트) 🔴

```
"계산된 위치에 실제 픽셀을 그리기!"

  → 텍스트 렌더링!
  → 색상 채우기!
  → 테두리, 그림자 그리기!
  → 이미지 배치!

💡 배경색이나 글자색이 바뀌면 다시 그림! (Repaint!)
  → 하지만 위치/크기가 안 바뀌면 Layout은 안 해도 됨!
```

### Step 6: Composite (합성) 🟡

```
"여러 레이어를 합쳐서 최종 화면 완성!"

  → 브라우저는 요소들을 여러 레이어로 나눠서 그림!
  → GPU(그래픽카드)가 레이어를 합성!
  → transform, opacity 변경은 이 단계만 다시 함! (가장 빠름! ⚡)
```

## 3. Reflow vs Repaint — 성능의 핵심! 🔴

> 💡 **면접 필수!** "Reflow와 Repaint의 차이를 설명하세요."

```
Reflow (= Layout 재계산):
  "위치나 크기가 변했어! 전부 다시 계산해야 해!" 😰
  → 비용이 가장 큼! (부모/형제도 영향!)
  → 트리거: width, height, padding, margin, display,
            font-size, position, top/left, 텍스트 내용 변경!

Repaint (= 다시 그리기):
  "색상만 변했어! 위치는 그대로야!" 😊
  → Reflow보다 가벼움!
  → 트리거: color, background-color, visibility, outline,
            box-shadow, border-color

Composite Only (가장 가벼움! ⚡):
  "레이어만 다시 합치면 돼!"
  → GPU가 처리! 메인 스레드 안 건드림!
  → 트리거: transform, opacity

비용 순서:
  Reflow (😰) > Repaint (😊) > Composite (⚡)
```

### 성능 최적화 팁

```tsx
// ❌ Reflow 유발! (비용 높음!)
element.style.width = "200px";     // 크기 변경 → Reflow!
element.style.marginTop = "20px";  // 위치 변경 → Reflow!

// ✅ Composite Only! (비용 낮음!)
element.style.transform = "translateX(200px)";  // GPU! ⚡
element.style.opacity = "0.5";                   // GPU! ⚡

// ❌ 나쁜 애니메이션 (Reflow 매 프레임!)
@keyframes bad {
  from { left: 0; }       ← Reflow!
  to { left: 200px; }     ← Reflow!
}

// ✅ 좋은 애니메이션 (Composite Only!)
@keyframes good {
  from { transform: translateX(0); }     ← GPU! ⚡
  to { transform: translateX(200px); }   ← GPU! ⚡
}
```

### Tailwind CSS에서의 적용

```tsx
// ❌ Reflow (left 사용)
<div className="absolute left-0 transition-all">

// ✅ Composite (transform 사용!)
<div className="translate-x-0 transition-transform">

// Tailwind의 transition 클래스:
//   transition-transform → transform만 애니메이션! (GPU! ⚡)
//   transition-opacity   → opacity만! (GPU! ⚡)
//   transition-all       → 모든 속성! (Reflow 위험! ⚠️)
```

## 4. JavaScript가 렌더링을 막는다! 🔴

```
HTML 파싱 중 <script> 태그를 만나면:
  → HTML 파싱이 멈춤! (Parser Blocking!)
  → JS를 다운로드하고 실행할 때까지 대기!
  → 그동안 화면이 안 그려짐! 😱

해결:
  <script src="app.js" defer></script>
  → defer: HTML 파싱과 동시에 JS 다운로드!
           HTML 파싱 끝난 후 JS 실행!
           (Next.js가 기본으로 사용!) ⭐

  <script src="app.js" async></script>
  → async: HTML 파싱과 동시에 JS 다운로드!
           다운로드 끝나면 즉시 실행! (파싱 잠시 멈춤!)

  비교:
    없음:  HTML 파싱 멈춤 → JS 다운 → JS 실행 → 파싱 재개 😰
    defer: HTML 파싱 계속! → JS 다운(병렬!) → 파싱 끝 → JS 실행 ⭐
    async: HTML 파싱 계속! → JS 다운(병렬!) → 즉시 실행(파싱 멈춤!)

  Next.js는?
    → next/script 컴포넌트로 자동 최적화!
    → 대부분 defer 또는 lazy 로딩!
```

## 5. CSS도 렌더링을 막는다! 🟡

```
CSSOM이 완성되기 전에는 Render Tree를 만들 수 없음!
  → CSS 다운로드가 느리면 → 화면이 안 그려짐! (Render Blocking!)

해결:
  1. Critical CSS: 첫 화면에 필요한 CSS만 <head>에 인라인!
  2. 미디어 쿼리: <link rel="stylesheet" media="print" href="print.css">
     → 프린트용 CSS는 화면 렌더링을 막지 않음!
  3. Next.js: CSS를 자동으로 코드 스플리팅!
     → 각 페이지에 필요한 CSS만 로드! ⭐
```

## 6. Core Web Vitals — 성능 지표 🟡

```
구글이 정한 "사용자 경험" 핵심 지표 3가지!

LCP (Largest Contentful Paint): 2.5초 이내! ⭐
  → "가장 큰 콘텐츠가 화면에 뜨는 시간!"
  → 이미지, 큰 텍스트 블록 등
  → 2.5초 넘으면 SEO 패널티!

FID (First Input Delay) → INP로 교체!:
INP (Interaction to Next Paint): 200ms 이내!
  → "사용자가 클릭한 후 화면이 반응하는 시간!"
  → 버튼 눌렀는데 0.5초 후에 반응하면 → 사용자 이탈!

CLS (Cumulative Layout Shift): 0.1 이하!
  → "화면이 갑자기 밀리는 정도!"
  → 광고가 로드되면서 버튼이 아래로 밀리는 현상!
  → Next.js의 Image 컴포넌트가 CLS 방지! (width/height 필수!)

측정 도구:
  → Chrome DevTools → Lighthouse 탭!
  → web.dev/measure
  → PageSpeed Insights (Google)
```

## 면접 대비

### 🔴 필수 질문

**Q1. 브라우저에 URL을 입력하면 어떤 일이 일어나나요?**

> DNS 조회로 도메인을 IP 주소로 변환하고, TCP 연결(+TLS 핸드셰이크)을 수립한 뒤 HTTP 요청을 보낸다. 서버가 HTML 응답을 보내면 브라우저는 렌더링 파이프라인을 시작한다. HTML을 파싱하여 DOM 트리를, CSS를 파싱하여 CSSOM 트리를 만들고, 이 둘을 합쳐 Render Tree를 구성한다. 이후 Layout(위치/크기 계산), Paint(픽셀 그리기), Composite(레이어 합성)를 거쳐 화면에 표시된다.

**Q2. Reflow와 Repaint의 차이?**

> Reflow는 요소의 위치나 크기가 변할 때 레이아웃을 다시 계산하는 것으로 비용이 가장 크다(width, margin, font-size 변경 등). Repaint는 색상 등 시각적 속성만 변할 때 다시 그리는 것으로 Reflow보다 가볍다. 가장 효율적인 것은 transform/opacity만 변경하는 Composite Only로, GPU가 처리하여 메인 스레드를 차단하지 않는다. 따라서 애니메이션은 left/top 대신 transform을 사용해야 한다.

**Q3. script 태그의 defer와 async 차이?**

> 기본 script는 HTML 파싱을 멈추고 JS를 다운로드/실행하여 렌더링을 차단한다. `defer`는 HTML 파싱과 병렬로 JS를 다운로드하고 파싱 완료 후에 실행하여 렌더링을 막지 않는다. `async`는 병렬 다운로드 후 즉시 실행하므로 파싱이 잠시 멈출 수 있다. Next.js는 기본적으로 defer 방식을 사용하여 최적화한다.
