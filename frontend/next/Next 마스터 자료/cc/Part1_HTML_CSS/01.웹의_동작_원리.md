> **키워드**: `HTTP` `HTTPS` `DNS` `클라이언트-서버` `URL 구조` `요청/응답` `상태 코드`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 클라이언트-서버, HTTP, URL 구조, 상태 코드 | 면접 단골 + 실무 매일 사용 |

## 1. 시작하기 전에 — 우리가 매일 쓰는 "인터넷"의 정체 🔴

> 💡 **비유: 편의점 주문**
> 너(브라우저)가 편의점(서버)에 가서 "삼각김밥 하나 주세요"라고 말하면,
> 편의점 직원이 진열대에서 삼각김밥을 꺼내 건네준다.
> 이게 인터넷의 전부다.

```
너(브라우저)  ──── "삼각김밥 주세요" ────▶  편의점(서버)
                    [요청 Request]

너(브라우저)  ◀──── 🍙 삼각김밥 ─────────  편의점(서버)
                    [응답 Response]
```

웹에서 일어나는 모든 일은 이 **요청(Request)과 응답(Response)** 두 단어로 설명된다.

### 핵심 용어 정리

- **클라이언트(Client)**: 요청을 보내는 쪽. 크롬, 사파리, 모바일 앱 등
- **서버(Server)**: 요청을 받아서 처리하고 응답을 보내는 컴퓨터
- **프로토콜(Protocol)**: 둘이 대화할 때의 약속된 규칙 (= HTTP)

## 2. URL의 해부학 — 주소창에 숨겨진 비밀 🔴

주소창에 아무 생각 없이 치는 그 글자들, 하나하나가 전부 의미가 있다.

```
https://www.example.com:443/products/123?sort=price&order=asc#reviews
|_____|  |_______________|____|___________|___________________|_______|
  ①           ②           ③       ④              ⑤             ⑥
```

| 번호 | 이름 | 역할 | 예시 |
|------|------|------|------|
| ① | **Scheme (프로토콜)** | 어떤 규칙으로 통신할지 | `https://` |
| ② | **Host (도메인)** | 어느 서버에 갈지 | `www.example.com` |
| ③ | **Port (포트)** | 서버의 몇 번 문으로 들어갈지 | `:443` (HTTPS 기본값, 생략 가능) |
| ④ | **Path (경로)** | 서버 안에서 어떤 자원을 원하는지 | `/products/123` |
| ⑤ | **Query String (쿼리)** | 추가 조건/옵션 | `?sort=price&order=asc` |
| ⑥ | **Fragment (해시)** | 페이지 내 특정 위치 | `#reviews` |

> 💡 **비유: 택배 주소**
> - Scheme = 우체국(KR우편) vs 퀵서비스(오토바이) → 배달 방법
> - Host = 서울시 강남구 → 어느 동네
> - Port = 101동 → 어느 건물 입구
> - Path = 302호 → 어느 집
> - Query = "문 앞에 놔주세요" → 추가 요청사항

## 3. "브라우저에 URL을 입력하면 무슨 일이 일어나나요?" 🔴

> ⚠️ **면접 최다 출제 질문**. 이 흐름을 자기 말로 설명할 수 있어야 한다.

### Step-by-step 실행 추적

```
Step 1: URL 입력
  └─ 사용자가 주소창에 "www.naver.com" 입력 후 엔터

Step 2: DNS 조회
  └─ 브라우저: "www.naver.com의 진짜 주소(IP)가 뭐지?"
  └─ DNS 서버: "223.130.200.107 이야"
  └─ (전화번호부에서 이름으로 번호 찾는 것과 같음)

Step 3: TCP 연결 (3-way Handshake)
  └─ 브라우저 → 서버: "통화 가능?" (SYN)
  └─ 서버 → 브라우저: "응, 가능!" (SYN-ACK)
  └─ 브라우저 → 서버: "좋아, 시작하자!" (ACK)

Step 4: HTTP 요청 전송
  └─ 브라우저가 서버에게 "GET / HTTP/1.1" 요청을 보냄
  └─ "나한테 메인 페이지(/) HTML을 줘!"

Step 5: 서버 처리 & 응답
  └─ 서버가 요청을 받아서 HTML 파일을 찾음
  └─ "HTTP/1.1 200 OK" + HTML 본문을 응답으로 보냄

Step 6: 브라우저 렌더링
  └─ HTML 파싱 → DOM 트리 생성
  └─ CSS 파싱 → CSSOM 트리 생성
  └─ DOM + CSSOM → 렌더 트리 → Layout → Paint → 화면에 표시!
  └─ (이 과정은 Part 1 - 05번에서 자세히 다룬다)
```

> 💡 **비유: 중국집 배달 주문**
> 1. 전화번호부에서 중국집 번호 찾기 (DNS)
> 2. 전화 연결 확인 "여보세요?" (TCP Handshake)
> 3. "짜장면 하나 배달해주세요" (HTTP 요청)
> 4. 중국집에서 요리 (서버 처리)
> 5. 배달원이 짜장면 가져옴 (HTTP 응답)
> 6. 그릇 열어서 먹기 시작 (브라우저 렌더링)

## 4. HTTP — 웹의 공용어 🔴

HTTP(HyperText Transfer Protocol)는 브라우저와 서버가 대화할 때 쓰는 **약속된 언어**다.

### HTTP 요청(Request)의 구조

```
GET /products/123 HTTP/1.1        ← 요청 라인 (메서드 + 경로 + 버전)
Host: www.example.com             ← 헤더 (부가 정보들)
Accept: text/html
User-Agent: Chrome/120

                                   ← 빈 줄 (헤더와 본문의 경계)
(본문은 GET 요청에서는 보통 비어있음)
```

### HTTP 응답(Response)의 구조

```
HTTP/1.1 200 OK                   ← 상태 라인 (버전 + 상태 코드 + 메시지)
Content-Type: text/html            ← 헤더
Content-Length: 1234

<!DOCTYPE html>                    ← 본문 (실제 데이터)
<html>
  <head><title>상품 상세</title></head>
  <body>...</body>
</html>
```

### HTTP 메서드 — "뭘 하고 싶은지" 말하는 동사 🔴

| 메서드 | 목적 | 비유 | 예시 |
|--------|------|------|------|
| **GET** | 데이터 **조회** (읽기) | 메뉴판 보기 | 상품 목록 불러오기 |
| **POST** | 데이터 **생성** (쓰기) | 주문하기 | 회원가입, 게시글 작성 |
| **PUT** | 데이터 **전체 수정** | 주문 전체 변경 | 프로필 전체 덮어쓰기 |
| **PATCH** | 데이터 **부분 수정** | 주문 일부 변경 | 닉네임만 변경 |
| **DELETE** | 데이터 **삭제** | 주문 취소 | 게시글 삭제 |

### HTTP 상태 코드 — 서버의 대답 🔴

> 💡 **비유: 편의점 직원의 대답**

```
✅ 2xx (성공)
  200 OK          → "여기요, 삼각김밥입니다!" (정상 응답)
  201 Created     → "새 상품 등록 완료했습니다!" (생성 성공)
  204 No Content  → "삭제 완료! 더 줄 건 없어요" (본문 없는 성공)

🔀 3xx (리다이렉트)
  301 Moved       → "그 상품은 2층으로 옮겼어요" (영구 이동)
  302 Found       → "임시로 옆 매장에서 판매 중이에요" (임시 이동)

❌ 4xx (클라이언트 잘못)
  400 Bad Request → "뭐라고요? 말씀을 못 알아듣겠어요" (잘못된 요청)
  401 Unauthorized→ "회원증 보여주세요" (인증 필요 = 로그인 안 함)
  403 Forbidden   → "회원이긴 한데 VIP만 입장 가능합니다" (권한 없음)
  404 Not Found   → "그런 상품 저희 매장에 없는데요?" (자원 없음)

💥 5xx (서버 잘못)
  500 Internal    → "죄송합니다, 저희 시스템에 문제가..." (서버 에러)
  502 Bad Gateway → "중간 배송 업체가 응답을 안 해요" (게이트웨이 에러)
  503 Unavailable → "점검 중입니다. 잠시 후 다시 오세요" (서비스 불가)
```

> ⚠️ **면접 필수**: `401 Unauthorized`는 이름과 달리 **인증(Authentication)** 실패다.
> **인가(Authorization)** 실패는 `403 Forbidden`이다. 이름에 속지 마라!

## 5. DNS — 인터넷의 전화번호부 🟡

> 💡 **비유: 전화번호부**
> 우리가 "네이버"라고 부르는 건 사람이 기억하기 쉬운 **별명(도메인)**이다.
> 컴퓨터끼리는 `223.130.200.107` 같은 **IP 주소**로만 통신한다.
> DNS는 별명 → IP 주소로 바꿔주는 전화번호부다.

```
브라우저: "www.naver.com의 IP가 뭐야?"

   ① 브라우저 캐시 확인 → 없으면
   ② OS 캐시 확인 → 없으면
   ③ 라우터 캐시 확인 → 없으면
   ④ ISP의 DNS 서버에 물어봄 → 없으면
   ⑤ Root DNS → .com DNS → naver.com DNS → IP 발견!

DNS 서버: "223.130.200.107 이야!"
```

## 6. HTTPS — 도청을 막는 암호화 통신 🟡

> 💡 **비유: 편지 vs 등기우편**
> HTTP = 엽서. 배달하는 사람이 내용을 다 볼 수 있음 (도청 가능)
> HTTPS = 자물쇠 달린 금고에 넣어서 보냄. 열쇠가 있는 사람만 열 수 있음

### HTTP와 HTTPS의 차이

```
❌ HTTP (위험)
  브라우저 ────── "비밀번호: 1234" ──────▶ 서버
                  ↑ 해커가 중간에서 훔쳐봄!

✅ HTTPS (안전)
  브라우저 ────── "dk#@!x$2f..." ──────▶ 서버
                  ↑ 암호화되어 있어서 해커가 봐도 해독 불가
```

### 핵심 포인트

- HTTPS = HTTP + **SSL/TLS 암호화**
- 주소창에 🔒 자물쇠 아이콘 = HTTPS
- 2026년 현재, HTTPS는 **선택이 아니라 필수**
  - 구글 SEO 랭킹에 영향
  - 브라우저가 HTTP 사이트에 "안전하지 않음" 경고 표시
  - 쿠키의 `Secure` 속성은 HTTPS에서만 작동

## 7. 개발자 도구로 직접 확인하기 (실습) 🔴

이론만 읽지 말고, **직접 눈으로 확인**해야 체화된다.

### 실습 1: 네트워크 탭으로 요청/응답 확인

```
1. 크롬 브라우저에서 아무 웹사이트 접속 (예: naver.com)
2. F12 (개발자 도구) → Network 탭 클릭
3. 새로고침 (F5)
4. 첫 번째 요청(문서)을 클릭하면:
   - Headers 탭: 요청 메서드, 상태 코드, 헤더 정보
   - Preview 탭: 서버가 보내준 HTML 미리보기
   - Response 탭: 실제 HTML 소스코드
```

### 실습 2: 상태 코드 직접 체험

```
1. 브라우저 주소창에 입력해보자:
   - https://www.google.com          → 200 OK (정상)
   - https://www.google.com/asdfgh   → 404 Not Found (없는 페이지)

2. 개발자 도구 Network 탭에서 Status 열 확인
```

```

- Scheme:  ?

- Path:    ?
- Query:   ?
- Fragment: ?
```

## 면접 대비

### 🔴 필수 질문

**Q1. 브라우저에 URL을 입력하면 무슨 일이 일어나나요?**

> DNS 조회로 도메인을 IP 주소로 변환하고, TCP 3-way Handshake로 서버와 연결을 맺는다. 이후 HTTP 요청을 전송하면, 서버가 요청을 처리하여 HTTP 응답(HTML)을 반환한다. 브라우저는 받은 HTML을 파싱하여 DOM 트리를 만들고, CSS와 결합한 렌더 트리를 기반으로 Layout → Paint → Composite 과정을 거쳐 화면을 그린다.

**Q2. HTTP 메서드 GET과 POST의 차이는?**

> GET은 서버에서 데이터를 조회(읽기)할 때 사용하며, 요청 본문(Body)이 없고 URL의 쿼리 스트링으로 데이터를 전달한다. POST는 서버에 데이터를 생성(쓰기)할 때 사용하며, 요청 본문에 데이터를 담아 전송한다. GET은 브라우저에 캐싱되지만 POST는 캐싱되지 않는다.

**Q3. HTTP 상태 코드 401과 403의 차이는?**

> 401 Unauthorized는 인증(Authentication) 실패로, 로그인 자체가 되지 않은 상태다. 403 Forbidden은 인가(Authorization) 실패로, 로그인은 되었지만 해당 자원에 접근할 권한이 없는 상태다.

**Q4. HTTPS는 HTTP와 무엇이 다른가요?**

> HTTPS는 HTTP에 SSL/TLS 암호화 계층을 추가한 프로토콜이다. 클라이언트와 서버 사이의 데이터를 암호화하여 중간자 공격(Man-in-the-Middle)과 도청을 방지한다. SEO와 쿠키 보안(Secure 속성) 등의 이유로 현대 웹에서는 필수다.

### 🟡 개념 질문

**Q5. DNS란 무엇인가요?**

> Domain Name System의 약자로, 사람이 읽기 쉬운 도메인 이름(예: google.com)을 컴퓨터가 이해하는 IP 주소(예: 142.250.196.78)로 변환해주는 시스템이다. 인터넷의 전화번호부라고 할 수 있다.

**Q6. URL과 URI의 차이는?**

> URI(Uniform Resource Identifier)는 자원을 식별하는 가장 넓은 개념이고, URL(Uniform Resource Locator)은 자원의 위치(Location)를 나타내는 URI의 하위 개념이다. 실무에서는 대부분 URL과 URI를 혼용하지만, 정확히는 쿼리 스트링과 프래그먼트까지 포함한 전체 주소가 URI다.
