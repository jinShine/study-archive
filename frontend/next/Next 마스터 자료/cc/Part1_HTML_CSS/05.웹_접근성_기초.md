# 5. 웹 접근성(a11y) 기초

> **키워드**: `WAI-ARIA` `role` `aria-label` `alt` `키보드 네비게이션` `색상 대비` `스크린 리더`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | alt 텍스트, 키보드 접근, 기본 ARIA | 실무 코드 리뷰에서 지적당하는 1순위 |
| 🟡 이해 | ARIA role/속성, 색상 대비, focus 관리 | 설명할 수 있으면 가산점 |
| 🟢 참고 | WCAG 가이드라인, 법적 의무 | 시니어 영역 |

## 1. 접근성(Accessibility)이 뭔데? 🔴

> 💡 **비유: 건물의 경사로(Ramp)**
> 계단만 있는 건물은 휠체어 사용자가 들어갈 수 없다.
> 경사로를 만들면 휠체어, 유모차, 캐리어 끌고 오는 사람 모두가 편해진다.
> 웹 접근성도 마찬가지. 장애인을 위해 만들면 **모든 사용자**가 편해진다.

### 접근성이 필요한 사용자

```
👁️ 시각 장애: 전맹(스크린 리더 사용), 저시력(확대/고대비 필요)
👂 청각 장애: 영상 자막, 시각적 알림 필요
🖐️ 운동 장애: 마우스 사용 불가 → 키보드/음성으로 조작
🧠 인지 장애: 단순한 UI, 명확한 안내 필요
📱 상황적 장애: 한 손으로 조작, 밝은 햇빛, 소음 환경
```

> ⚠️ **a11y**는 "accessibility"의 약어 (a + 11글자 + y)

## 2. 당장 실천하는 접근성 체크리스트 🔴

### 규칙 1: 이미지에는 반드시 alt 텍스트

```html
<!-- ❌ 스크린 리더: "이미지" (무슨 이미지인지 모름) -->
<img src="cat.jpg">

<!-- ❌ 의미 없는 alt (쓰나마나) -->
<img src="cat.jpg" alt="이미지">

<!-- ✅ 의미 있는 설명 -->
<img src="cat.jpg" alt="창가에 앉아 바깥을 바라보는 회색 고양이">

<!-- ✅ 장식용 이미지는 빈 alt (스크린 리더가 건너뜀) -->
<img src="decorative-line.png" alt="">
```

### 규칙 2: 키보드로 모든 기능 사용 가능

```
필수 키보드 조작:
  Tab     → 다음 요소로 이동
  Shift+Tab → 이전 요소로 이동
  Enter   → 버튼 클릭 / 링크 이동
  Space   → 체크박스 토글 / 버튼 클릭
  Esc     → 모달 닫기
  화살표   → 라디오 버튼, 탭 이동
```

```html
<!-- ❌ div를 버튼으로 사용 → 키보드로 선택 불가, 스크린 리더가 버튼으로 인식 안 함 -->
<div class="button" onclick="handleClick()">클릭하세요</div>

<!-- ✅ 진짜 버튼 사용 → Tab으로 포커스, Enter로 클릭, 스크린 리더가 "버튼"이라고 읽음 -->
<button onclick="handleClick()">클릭하세요</button>
```

> 💡 **황금 규칙**: 클릭할 수 있는 요소는 반드시 `<button>` 또는 `<a>`를 써라.
> `<div>` + `onClick`은 접근성의 적이다.

### 규칙 3: 색상만으로 정보를 전달하지 않기

```
❌ "빨간색 항목이 에러입니다"  → 색맹 사용자는 구분 불가

✅ "❌ 빨간색 + 아이콘 + 텍스트로 에러 표시"
   → 색상 + 아이콘 + 텍스트 3중 표시
```

### 규칙 4: focus가 보여야 한다

```css
/* ❌ 포커스 링 제거 → 키보드 사용자가 현재 위치를 모름 */
*:focus { outline: none; }

/* ✅ 포커스 링을 예쁘게 커스텀 */
*:focus-visible {
  outline: 2px solid #4F46E5;
  outline-offset: 2px;
}
```

> `:focus-visible`은 **키보드로 접근했을 때만** 포커스 링을 보여준다.
> 마우스 클릭 시에는 안 보이므로 디자인을 해치지 않는다.

## 3. WAI-ARIA 기초 🟡

시맨틱 태그로 표현이 어려운 경우, ARIA 속성으로 접근성 정보를 보강한다.

> 💡 **ARIA 제1원칙**: 네이티브 HTML 요소로 해결 가능하면 ARIA를 쓰지 마라!
> `<button>`으로 해결되는 걸 `<div role="button">`으로 만들지 마라.

### 자주 쓰는 ARIA 속성

```html
<!-- role: 요소의 역할을 알려줌 (네이티브 태그가 없을 때만!) -->
<div role="alert">비밀번호가 틀렸습니다.</div>
<div role="dialog" aria-modal="true">모달 내용</div>

<!-- aria-label: 화면에 텍스트가 없는 요소에 이름 부여 -->
<button aria-label="메뉴 열기">☰</button>
<button aria-label="닫기">✕</button>

<!-- aria-hidden: 스크린 리더에게 숨김 (장식용 아이콘 등) -->
<span aria-hidden="true">🎨</span> 디자인 섹션

<!-- aria-expanded: 열림/닫힘 상태 알림 -->
<button aria-expanded="false" aria-controls="menu">메뉴</button>
<ul id="menu" hidden>
  <li>항목 1</li>
</ul>

<!-- aria-live: 실시간으로 변경되는 영역 알림 -->
<div aria-live="polite">검색 결과 5건</div>
<!-- polite: 현재 읽는 것 끝난 후 안내 / assertive: 즉시 안내 -->

<!-- aria-required: 필수 입력 표시 -->
<input type="text" aria-required="true">

<!-- aria-describedby: 추가 설명 연결 -->
<input type="password" aria-describedby="pw-help">
<p id="pw-help">비밀번호는 8자 이상이어야 합니다.</p>
```

## 4. 실습: 접근성 감사(Audit) 해보기

### 실습 1: Lighthouse 접근성 점수 확인

```
1. 크롬에서 아무 웹사이트 접속
2. F12 → Lighthouse 탭
3. Categories에서 "Accessibility" 체크
4. "Analyze page load" 클릭
5. 접근성 점수와 개선 항목 확인
```

### 실습 2: 키보드만으로 네비게이션

```
1. 마우스를 치우고 Tab 키만으로 웹사이트 탐색
2. 모든 버튼/링크에 포커스가 가는지 확인
3. 포커스 링(outline)이 보이는지 확인
4. Enter로 버튼 클릭이 되는지 확인
```

### 실습 3: 02번 자기소개 페이지 접근성 개선

```
이전에 만든 자기소개 페이지를 열고:
1. 모든 <img>에 의미 있는 alt 텍스트 추가
2. 모든 입력 요소에 <label> 연결 확인
3. 색상만으로 정보를 전달하는 곳이 있는지 체크
4. Tab으로 모든 인터랙티브 요소에 접근 가능한지 확인
5. Lighthouse 접근성 점수 90 이상 달성
```

## 면접 대비

### 🟡 이해 질문

**Q1. 웹 접근성(a11y)이란? 왜 중요한가?**

> 장애인, 고령자, 일시적 장애(한 손 사용, 밝은 햇빛 등)를 포함한 모든 사용자가 웹 콘텐츠를 동등하게 이용할 수 있도록 보장하는 것이다. SEO 향상, 법적 의무(장애인차별금지법), 사용자층 확대라는 실질적 이점이 있다.

**Q2. ARIA란 무엇인가?**

> WAI-ARIA(Web Accessibility Initiative - Accessible Rich Internet Applications)는 HTML만으로 전달이 어려운 접근성 정보를 보충하는 속성 체계다. role, aria-label, aria-hidden, aria-expanded 등이 있다. 단, 네이티브 HTML 요소로 해결 가능한 경우 ARIA보다 시맨틱 태그를 우선 사용해야 한다.

**Q3. `<div onclick>`대신 `<button>`을 써야 하는 이유?**

> `<button>`은 기본적으로 키보드 포커스가 가능하고, Enter/Space로 클릭할 수 있으며, 스크린 리더가 "버튼"이라고 읽어준다. `<div>`는 이 모든 것이 없어서 `tabindex`, `role="button"`, `onKeyDown` 등을 수동으로 추가해야 한다. 접근성과 코드 간결성 모두에서 네이티브 요소가 우월하다.
