## 1. 왜 원본을 직접 변경하면 안 되는가? 🔴

> 💡 **비유: 공유 문서**
> 팀원 5명이 같은 Google 문서를 편집하고 있다.
> 내가 몰래 내용을 바꾸면, 다른 4명이 보고 있던 내용도 갑자기 바뀐다.
> **복사본을 만들어서 수정**하면 원본은 안전하다.

```js
// ❌ 직접 변경 (Mutation) — 참조를 공유하는 모든 곳에 영향!
const user = { name: "철수", age: 25 };
const backup = user;      // 같은 객체를 가리킴
user.age = 26;
console.log(backup.age);  // 26 😱 backup도 바뀜!

// ✅ 불변 업데이트 (Immutable Update) — 새 객체 생성
const user = { name: "철수", age: 25 };
const updated = { ...user, age: 26 };  // 새 객체!
console.log(user.age);     // 25 ✅ 원본 안전
console.log(updated.age);  // 26 ✅ 새 객체
```

### React에서 불변성이 필수인 이유

```jsx
// React는 상태가 바뀌었는지를 "참조 비교(===)"로 판단한다!

// ❌ 직접 변경 → React가 변화를 감지 못함 → 리렌더링 안 됨!
const handleClick = () => {
  user.age = 26;           // 같은 객체의 속성만 바꿈
  setUser(user);           // 참조가 같으므로 React: "아, 안 바뀌었네" → 무시!
};

// ✅ 새 객체 생성 → React가 변화를 감지 → 리렌더링!
const handleClick = () => {
  setUser({ ...user, age: 26 });  // 새 참조! React: "오, 바뀌었네!" → 리렌더링!
};
```

```
React의 변화 감지 원리:

oldState === newState ?
  ├─ true  → "안 바뀌었네" → 리렌더링 안 함
  └─ false → "바뀌었다!" → 리렌더링 실행

직접 변경: 같은 참조 → true → 화면 안 바뀜 😱
새 객체:   다른 참조 → false → 화면 업데이트 ✅
```

## 2. 불변 업데이트 패턴 총정리 🔴

### 객체 업데이트

```js
const user = { name: "철수", age: 25, email: "test@test.com" };

// 속성 수정
const updated = { ...user, age: 26 };

// 속성 추가
const withRole = { ...user, role: "admin" };

// 속성 삭제 (rest 패턴)
const { email, ...withoutEmail } = user;
// withoutEmail = { name: "철수", age: 25 }
```

### 배열 업데이트

```js
const items = ["사과", "바나나", "딸기"];

// 추가 (push 대신)
const added = [...items, "포도"];             // 뒤에 추가
const prepended = ["멜론", ...items];         // 앞에 추가

// 삭제 (splice 대신)
const removed = items.filter((_, i) => i !== 1);  // 인덱스 1 제거
// ["사과", "딸기"]

// 수정 (직접 대입 대신)
const updated = items.map((item, i) =>
  i === 1 ? "오렌지" : item
);
// ["사과", "오렌지", "딸기"]

// 정렬 (sort 대신 — sort는 원본 변경!)
const sorted = [...items].sort();
// 또는: items.toSorted() (ES2023+)
```

### 중첩 객체 업데이트 🟡

```js
const state = {
  user: {
    name: "철수",
    address: {
      city: "서울",
      zip: "06000"
    }
  },
  posts: [
    { id: 1, title: "첫 글" },
    { id: 2, title: "둘째 글" }
  ]
};

// 중첩 객체 수정: 모든 단계를 스프레드해야 함!
const newState = {
  ...state,
  user: {
    ...state.user,
    address: {
      ...state.user.address,
      city: "부산"            // 여기만 바꾸기 위해 3단계 스프레드 😰
    }
  }
};

// 배열 안의 객체 수정
const updatedPosts = {
  ...state,
  posts: state.posts.map(post =>
    post.id === 1 ? { ...post, title: "수정된 글" } : post
  )
};
```

> 💡 중첩이 깊어지면 코드가 지저분해진다 → **Immer 라이브러리**로 해결!

### Immer로 간결하게 (실무 패턴) 🟡

```js
import { produce } from 'immer';

// Immer는 "직접 변경하는 것처럼" 쓰면 내부적으로 불변 업데이트를 해준다!
const newState = produce(state, draft => {
  draft.user.address.city = "부산";  // 직접 변경처럼 보이지만 불변!
  draft.posts[0].title = "수정된 글";
});

// Zustand에서 Immer 활용:
// set(produce(state => { state.count++ }))
```

## 3. 실습: React 상태 업데이트 시뮬레이션

```js
// 아래 setState 호출이 올바른지 판단하고, 틀렸다면 수정하라

const [user, setUser] = useState({ name: "철수", age: 25, hobbies: ["코딩"] });

// 1번: 나이 변경
setUser(user.age = 26);  // 올바른가?

// 2번: 취미 추가
user.hobbies.push("독서");
setUser(user);           // 올바른가?

// 3번: 이름 변경
setUser({ ...user, name: "영희" });  // 올바른가?
```

<summary>정답 보기</summary>

```js
// 1번: ❌ 원본 직접 변경 + setUser에 26(숫자)을 전달

// 2번: ❌ push는 원본 변경 + 같은 참조를 setUser에 전달

// 3번: ✅ 올바른 불변 업데이트!
```

## 면접 대비

### 🔴 필수 질문

**Q1. React에서 불변성이 왜 중요한가?**

> React는 상태 변화를 참조 비교(`===`)로 감지한다. 원본 객체를 직접 변경하면 참조가 동일하므로 React가 변화를 인식하지 못해 리렌더링이 발생하지 않는다. 반드시 스프레드 연산자 등으로 새 객체/배열을 생성하여 다른 참조를 만들어야 React가 변화를 감지하고 화면을 업데이트한다.

**Q2. 배열에 요소를 추가할 때 `push` 대신 어떻게 하나요?**

> `push`는 원본 배열을 직접 변경하므로 React 상태에서 사용하면 안 된다. 대신 `[...arr, newItem]`으로 스프레드 연산자를 사용하여 기존 배열을 복사하고 새 요소를 추가한 새 배열을 생성한다. 삭제는 `filter`, 수정은 `map`을 사용한다.
