# 31. ES Modules

> **키워드**: `import` `export` `named/default` `CommonJS` `ESM` `트리 셰이킹` `번들러`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | import/export 문법, named vs default | 매일 쓰는 기본 문법 |
| 🟡 이해 | CommonJS vs ESM, 트리 셰이킹, 동적 import | 빌드 최적화 |
| 🟢 참고 | 번들러(Vite/Webpack) 동작 원리 | 빌드 도구 심화 |

---

## 1. import / export 기본 🔴

### Named Export (이름 내보내기) — 여러 개 가능

```js
// math.js — 내보내기
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; }

// main.js — 가져오기 (이름이 정확히 일치해야 함!)
import { PI, add, multiply } from "./math.js";
import { add as sum } from "./math.js";  // 별칭(as)으로 이름 변경
import * as math from "./math.js";        // 전체를 네임스페이스로
math.add(1, 2);
```

### Default Export (기본 내보내기) — 파일당 1개만

```js
// Button.js — 내보내기
export default function Button({ children }) {
  return <button>{children}</button>;
}

// main.js — 가져오기 (아무 이름으로 받을 수 있음!)
import Button from "./Button.js";
import MyButton from "./Button.js";  // 이름 자유!
```

### 혼합 사용

```js
// api.js
export default class ApiClient { }    // default
export const BASE_URL = "/api";        // named
export function fetcher(url) { }       // named

// main.js
import ApiClient, { BASE_URL, fetcher } from "./api.js";
```

> 💡 **실무 원칙**:
> - 컴포넌트 → `export default function Component` (파일당 1개)
> - 유틸/상수 → `export const`, `export function` (named)
> - React/Next.js에서는 `page.tsx`, `layout.tsx` 등이 `export default` 필수

## 2. CommonJS vs ESM 🟡

```js
// CommonJS (CJS) — Node.js 전통 방식
const fs = require("fs");
module.exports = { add, multiply };

// ES Modules (ESM) — 현대 표준 ✅
import fs from "fs";
export { add, multiply };
```

| 구분 | CommonJS | ES Modules |
|------|----------|-----------|
| 문법 | `require` / `module.exports` | `import` / `export` |
| 로딩 | 동기 (런타임) | 정적 분석 (빌드 타임) |
| 트리 셰이킹 | ❌ 불가 | ✅ 가능! |
| 환경 | Node.js (구버전) | 브라우저 + Node.js |
| 현재 상태 | 레거시 (유지보수용) | ✅ 표준 |

## 3. 트리 셰이킹(Tree-shaking) 🟡

> 💡 **비유: 나무 흔들기**
> 나무(라이브러리)를 흔들면 죽은 잎(사용 안 한 코드)이 떨어진다.
> 번들러가 import하지 않은 코드를 최종 빌드에서 자동 제거하는 것.

```js
// lodash 전체를 import하면 트리 셰이킹 불가!
import _ from "lodash";           // ❌ 전체 번들 포함 (~70KB)
_.debounce(fn, 300);

// 필요한 함수만 import하면 나머지는 제거!
import debounce from "lodash/debounce"; // ✅ debounce만 포함 (~2KB)

// ESM 방식의 named import도 트리 셰이킹 가능
import { debounce } from "lodash-es";   // ✅ ESM 버전
```

> 💡 `export default`보다 `named export`가 트리 셰이킹에 유리하다.
> 번들러가 "이 함수는 어디서도 import되지 않았으니 제거" 판단을 더 정확하게 할 수 있다.

## 4. 동적 import — 코드 스플리팅 🟡

```js
// 정적 import — 항상 로드 (번들에 포함)
import HeavyComponent from "./HeavyComponent";

// 동적 import — 필요할 때만 로드 (코드 스플리팅!)
const HeavyComponent = await import("./HeavyComponent");

// React에서: React.lazy + Suspense
const HeavyComponent = React.lazy(() => import("./HeavyComponent"));

// Next.js에서: next/dynamic
import dynamic from "next/dynamic";
const Chart = dynamic(() => import("./Chart"), {
  loading: () => <p>차트 로딩중...</p>,
  ssr: false,  // 서버 렌더링 제외 (브라우저 전용 라이브러리)
});
```

## 면접 대비

### 🟡 이해 질문

**Q1. 트리 셰이킹이란?**

> 번들러(Vite, Webpack)가 빌드 시 실제로 import되지 않은 코드를 최종 번들에서 자동으로 제거하는 최적화 기법이다. ES Modules의 정적 분석이 가능한 `import`/`export` 문법 덕분에 가능하며, CommonJS의 `require`에서는 동작하지 않는다. 라이브러리를 통째로 import하지 말고 필요한 함수만 named import하면 번들 크기를 크게 줄일 수 있다.
