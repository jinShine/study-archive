## 1. JavaScript는 싱글 스레드다 🔴

> 💡 **비유: 1인 셰프 레스토랑**
> JavaScript는 **요리사가 1명**인 식당이다. 한 번에 하나의 요리만 할 수 있다.
> 그런데 손님이 100명이면? 주문을 받아두고(큐), 순서대로 처리한다.
> 오래 걸리는 요리(네트워크 요청)는 보조 주방(Web API)에 맡기고 다음 주문을 처리한다.

```
싱글 스레드 = 한 번에 하나의 코드만 실행할 수 있다

그런데 웹은:
  - 서버에서 데이터를 가져오면서 (fetch)
  - 2초 뒤에 알림을 띄우면서 (setTimeout)
  - 사용자 클릭에 반응하면서 (addEventListener)
  → 동시에 여러 일이 일어나는 것처럼 보인다!

어떻게? → 이벤트 루프(Event Loop)라는 시스템 덕분!
```

## 2. 이벤트 루프의 구조 🔴

```
┌──────────────────────────────────────────────────────────┐
│                    JavaScript 엔진                       │
│  ┌─────────────┐                                        │
│  │  Call Stack  │  ← 현재 실행 중인 함수가 쌓이는 곳      │
│  │ ┌─────────┐ │                                        │
│  │ │ fn()    │ │  ← 가장 위(최근)부터 실행               │
│  │ ├─────────┤ │                                        │
│  │ │ main()  │ │                                        │
│  │ └─────────┘ │                                        │
│  └─────────────┘                                        │
└──────────────────────────────────────────────────────────┘
         │ 비동기 작업 위임                 │ 완료된 콜백 등록
         ▼                                ▼
┌─────────────────┐            ┌──────────────────────┐
│    Web APIs      │            │  Microtask Queue     │ ← ⚡ 먼저!
│  (브라우저 제공)  │  ──────▶  │  Promise.then()      │
│  setTimeout      │            │  async/await 이후    │
│  fetch           │            │  queueMicrotask()    │
│  addEventListener│            ├──────────────────────┤
└─────────────────┘            │  Task Queue (Macro)   │ ← 나중에
                                │  setTimeout 콜백     │
                                │  setInterval 콜백    │
                                │  이벤트 핸들러       │
                                └──────────────────────┘
                                         │
                                    이벤트 루프가
                                  Call Stack이 비면
                                  큐에서 꺼내서 실행!
```

### 이벤트 루프의 동작 규칙 (절대 외워야 함!)

```
1. Call Stack에서 동기 코드를 하나씩 실행
2. 비동기 작업(setTimeout, fetch 등)을 만나면 Web API에 위임
3. Web API가 작업을 완료하면 콜백을 큐에 등록
4. Call Stack이 완전히 비면(!) 이벤트 루프가:
   a. Microtask Queue를 전부 비운다 (Promise 등) ← 먼저!
   b. Task Queue(Macrotask)에서 하나를 꺼내 실행 ← 나중에
5. 다시 1번으로 돌아감 (무한 반복)
```

> ⚠️ **핵심**: Microtask(Promise)가 Macrotask(setTimeout)보다 **항상 먼저** 실행된다!

## 3. Step-by-step 실행 추적 🔴

### 면접 단골 문제: 출력 순서를 맞춰라!

```js
console.log("1");

setTimeout(() => {
  console.log("2");
}, 0);

Promise.resolve().then(() => {
  console.log("3");
});

console.log("4");
```

### 실행 추적

```
[Call Stack]           [Microtask Queue]    [Task Queue]
──────────────────────────────────────────────────────

Step 1: console.log("1") 실행
  → 출력: "1"

Step 2: setTimeout(cb, 0)
  → Web API에 위임 → 0ms 후 콜백이 Task Queue에 등록
  [Task Queue: cb("2")]

Step 3: Promise.resolve().then(cb)
  → 이미 resolved → 콜백이 Microtask Queue에 등록
  [Microtask Queue: cb("3")]

Step 4: console.log("4") 실행
  → 출력: "4"

Step 5: Call Stack이 비었다!
  → 이벤트 루프: Microtask Queue 먼저!
  → console.log("3") 실행
  → 출력: "3"

Step 6: Microtask Queue 비었다!
  → Task Queue에서 꺼내기
  → console.log("2") 실행
  → 출력: "2"

최종 출력 순서: 1 → 4 → 3 → 2
```

> 💡 `setTimeout(fn, 0)`은 "0ms 뒤에 실행"이 아니다!
> "현재 동기 코드 + Microtask가 **전부 끝난 후** 가장 빨리 실행"이다!

## 4. 심화 예제 🟡

```js
console.log("start");

setTimeout(() => console.log("timeout 1"), 0);
setTimeout(() => console.log("timeout 2"), 0);

Promise.resolve()
  .then(() => {
    console.log("promise 1");
    return Promise.resolve();
  })
  .then(() => console.log("promise 2"));

Promise.resolve().then(() => console.log("promise 3"));

console.log("end");
```

<details>
<summary>출력 순서 보기</summary>

```
start           ← 동기 코드
end             ← 동기 코드
promise 1       ← Microtask (첫 번째 Promise.then)
promise 3       ← Microtask (두 번째 Promise.then)
promise 2       ← Microtask (체이닝된 .then — promise 1이 끝나야 등록됨)
timeout 1       ← Macrotask
timeout 2       ← Macrotask
```

실행 순서: 동기 코드 전부 → Microtask 전부 → Macrotask 하나씩

</details>

## 5. setTimeout 0의 실전 활용 🟡

```js

  // 브라우저가 화면을 다시 그린 후에 실행됨
```

## 6. requestAnimationFrame 🟡

```js
// 브라우저가 다음 프레임을 그리기 직전에 실행
// 애니메이션에 최적화 (보통 60fps = 16.6ms마다)

function animate() {
  element.style.left = `${position}px`;
  position += 2;

  if (position < 500) {
    requestAnimationFrame(animate);  // 다음 프레임에 다시 호출
  }
}
requestAnimationFrame(animate);

// 실행 순서에서의 위치:
// 동기 코드 → Microtask → requestAnimationFrame → Paint → Macrotask
```

## 7. 실습: 이벤트 루프 퀴즈 5문제

각 문제의 출력 순서를 **종이에 먼저 예측**한 후 콘솔에서 확인:

```js
// 문제 1
console.log("A");
setTimeout(() => console.log("B"), 0);
console.log("C");

// 문제 2
setTimeout(() => console.log("X"), 0);
Promise.resolve().then(() => console.log("Y"));
console.log("Z");

// 문제 3
async function foo() {
  console.log("1");
  await Promise.resolve();
  console.log("2");
}
console.log("3");
foo();
console.log("4");

// 문제 4
setTimeout(() => console.log("macro"), 0);
queueMicrotask(() => console.log("micro 1"));
Promise.resolve().then(() => console.log("micro 2"));
console.log("sync");

// 문제 5 (고난이도)
async function a() {
  console.log("a1");
  await b();
  console.log("a2");
}
async function b() {
  console.log("b1");
  await Promise.resolve();
  console.log("b2");
}
console.log("start");
a();
console.log("end");
```

<details>
<summary>정답 보기</summary>

```js
// 문제 1: A → C → B
// 문제 2: Z → Y → X
// 문제 3: 3 → 1 → 4 → 2
//   (foo() 호출 → "1" 출력 → await에서 멈춤 → "4" 출력 → Microtask "2" 실행)
// 문제 4: sync → micro 1 → micro 2 → macro
// 문제 5: start → a1 → b1 → end → b2 → a2
```

</details>

## 면접 대비

### 🔴 필수 질문 (면접 3대장 중 3번!)

**Q1. JavaScript의 이벤트 루프를 설명해주세요.**

> JavaScript는 싱글 스레드로 한 번에 하나의 코드만 실행할 수 있다. 비동기 작업(setTimeout, fetch 등)은 Web API에 위임하고, 완료되면 콜백을 큐에 등록한다. 이벤트 루프는 Call Stack이 비었을 때 큐에서 콜백을 꺼내 실행하는 무한 반복 메커니즘이다. Microtask Queue(Promise.then, async/await)가 Task Queue(setTimeout, 이벤트 핸들러)보다 항상 우선 처리된다.

**Q2. Microtask와 Macrotask의 차이?**

> Microtask(Promise.then, queueMicrotask, async/await의 await 이후)는 현재 실행 중인 코드가 끝난 직후 Call Stack이 비는 순간 전부 처리된다. Macrotask(setTimeout, setInterval, 이벤트 핸들러)는 Microtask가 모두 비워진 후 하나씩 처리된다. 따라서 `setTimeout(fn, 0)`이라도 Promise.then보다 항상 늦게 실행된다.

**Q3. `setTimeout(fn, 0)`은 정말 0ms 후에 실행되나요?**

> 아니다. 0ms는 "최소 대기 시간"일 뿐이다. 실제로는 현재 실행 중인 동기 코드와 모든 Microtask가 완료된 후에야 Task Queue에서 꺼내져 실행된다. 브라우저는 최소 4ms의 지연도 강제하며, Call Stack이 바쁘면 수백ms 이상 지연될 수 있다.

**Q4. `async/await`에서 `await` 뒤의 코드는 어디서 실행되나요?**

> `await` 키워드를 만나면 해당 async 함수의 실행이 일시 중단(suspend)되고, `await` 뒤의 코드는 Microtask Queue에 등록된다. Call Stack이 비고 이전 Microtask가 처리된 후 실행된다. 즉, `await` 뒤의 코드는 사실상 `.then()` 안의 콜백과 동일하게 동작한다.
