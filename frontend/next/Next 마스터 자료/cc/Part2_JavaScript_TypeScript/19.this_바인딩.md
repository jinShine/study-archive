## 1. this란? — JavaScript에서 가장 혼란스러운 키워드 🔴

> 💡 **비유: "나"라는 대명사**
> "나"라는 단어는 **누가 말하느냐**에 따라 다른 사람을 가리킨다.
> 철수가 "나"라고 하면 철수이고, 영희가 "나"라고 하면 영희다.
> JavaScript의 `this`도 마찬가지. **누가(어떻게) 함수를 호출하느냐**에 따라 다른 것을 가리킨다.

> ⚠️ **핵심 원칙**: `this`는 함수가 **정의될 때**가 아니라 **호출될 때** 결정된다!
> (화살표 함수만 예외 — 정의 시점에 결정)

## 2. this가 결정되는 4가지 규칙 🔴

### 규칙 1: 전역 호출 — "나 혼자" (Default Binding)

```js
function showThis() {
  console.log(this);
}

showThis();
// 브라우저: window 객체
// Node.js: global 객체
// strict mode: undefined
```

> 💡 아무 객체에도 속하지 않은 함수를 그냥 호출하면, `this`는 전역 객체를 가리킨다.
> `"use strict"` 모드에서는 `undefined`가 된다 (더 안전).

### 규칙 2: 메서드 호출 — "점(.) 왼쪽" (Implicit Binding)

```js
const user = {
  name: "철수",
  greet() {
    console.log(`안녕, 나는 ${this.name}`);
  }
};

user.greet();  // "안녕, 나는 철수" ← this === user (점 왼쪽!)
```

> 💡 **핵심**: `obj.method()` 형태로 호출하면 `this`는 **점(.) 바로 왼쪽의 객체**!

### 규칙 2의 함정: 메서드를 분리하면 this가 사라진다!

```js
const user = {
  name: "철수",
  greet() {
    console.log(`안녕, 나는 ${this.name}`);
  }
};

const fn = user.greet;  // 메서드를 변수에 빼냄
fn();                   // "안녕, 나는 undefined" 😱
// fn()은 점 없이 호출 → 규칙 1(전역) 적용!

// React에서 이런 실수가 발생하는 상황:
// <button onClick={user.greet}>  ← 메서드가 분리됨! this 소실!
```

### 규칙 3: 명시적 바인딩 — call / apply / bind 🟡

```js
function greet(greeting, punctuation) {
  console.log(`${greeting}, 나는 ${this.name}${punctuation}`);
}

const user = { name: "철수" };

// call: this를 지정하고 인수를 개별 전달
greet.call(user, "안녕", "!");    // "안녕, 나는 철수!"

// apply: this를 지정하고 인수를 배열로 전달
greet.apply(user, ["하이", "~"]); // "하이, 나는 철수~"

// bind: this가 고정된 새 함수를 반환 (즉시 실행 X)
const boundGreet = greet.bind(user, "헬로");
boundGreet("?");                  // "헬로, 나는 철수?"
```

```
call  → 즉시 실행 + 인수 개별 전달   fn.call(obj, a, b)
apply → 즉시 실행 + 인수 배열 전달   fn.apply(obj, [a, b])
bind  → 새 함수 반환 (나중에 실행)   const newFn = fn.bind(obj)
```

### 규칙 4: 화살표 함수 — 상위 스코프의 this를 상속 (Lexical this) 🔴

```js
const user = {
  name: "철수",

  // 일반 함수: 호출 시점에 this 결정
  greetNormal: function() {
    setTimeout(function() {
      console.log(this.name);  // undefined 😱 (setTimeout의 콜백은 전역에서 호출)
    }, 1000);
  },

  // 화살표 함수: 정의 시점의 상위 스코프 this를 그대로 사용!
  greetArrow: function() {
    setTimeout(() => {
      console.log(this.name);  // "철수" ✅ (화살표 함수는 상위인 greetArrow의 this를 상속)
    }, 1000);
  }
};

user.greetNormal(); // undefined
user.greetArrow();  // "철수"
```

> 💡 **화살표 함수의 this 규칙**:
> - 자체 `this`를 만들지 않는다
> - 자신이 **정의된 위치**의 상위 스코프 `this`를 그대로 사용
> - `call`/`apply`/`bind`로도 `this`를 바꿀 수 없다!

```
일반 함수:  this = 호출 방법에 따라 매번 달라짐 (동적)
화살표 함수: this = 정의된 위치의 상위 this 고정 (정적, 렉시컬)
```

## 3. this 결정 우선순위 🟡

```
우선순위 (높은 → 낮은):

1. new 바인딩    → new Foo()      → this = 새로 생성된 객체
2. 명시적 바인딩  → call/apply/bind → this = 지정한 객체
3. 암묵적 바인딩  → obj.method()   → this = 점 왼쪽 객체
4. 기본 바인딩    → fn()           → this = window(전역) / undefined(strict)

⚡ 화살표 함수는 위 규칙을 전부 무시하고 상위 스코프 this를 고정!
```

## 4. React에서의 this 🔴

```jsx
// React에서 this 문제가 거의 없는 이유:
// → 함수형 컴포넌트 + 화살표 함수가 기본이기 때문!

function Counter() {
  const [count, setCount] = useState(0);

  // ✅ 화살표 함수로 이벤트 핸들러 → this 걱정 없음
  const handleClick = () => {
    setCount(count + 1);
  };

  return <button onClick={handleClick}>{count}</button>;
}

// ❌ 옛날 클래스 컴포넌트에서는 this 문제가 항상 발생했다:
// class Counter extends React.Component {
//   handleClick() {
//     this.setState(...)  // ← onClick에 넘기면 this가 사라짐!
//   }
//   // 해결: this.handleClick = this.handleClick.bind(this)
//   // 또는: handleClick = () => { ... } (화살표 함수)
// }
```

> 💡 **결론**: 현대 React(함수형 컴포넌트 + Hooks)에서는 클래스를 쓰지 않으므로
> `this` 문제가 거의 발생하지 않는다. 하지만 **면접에서는 반드시 물어본다!**

## 5. 실습: this 탐정 챌린지

아래 각 `console.log(this.name)`의 결과를 **종이에 먼저 예측**:

```js
const name = "전역";

const obj = {
  name: "객체",
  normal: function() {
    console.log("1:", this.name);
  },
  arrow: () => {
    console.log("2:", this.name);
  },
  nested: function() {
    const inner = function() {
      console.log("3:", this.name);
    };
    inner();

    const innerArrow = () => {
      console.log("4:", this.name);
    };
    innerArrow();
  }
};

obj.normal();     // 1: ?
obj.arrow();      // 2: ?
obj.nested();     // 3: ? 4: ?

const fn = obj.normal;
fn();             // 1: ?
```

<details>
<summary>정답 보기</summary>

```js
obj.normal();
// 1: "객체" — 메서드 호출. 점 왼쪽 = obj

obj.arrow();
// 2: "전역" (또는 undefined) — 화살표 함수는 상위 스코프 this.
// obj 자체는 블록이 아니라 객체 리터럴이므로 상위 = 전역!
// ⚠️ 객체의 메서드로 화살표 함수를 쓰면 안 되는 이유!

obj.nested();
// 3: "전역" (또는 undefined) — inner()는 점 없이 호출 → 전역 바인딩
// 4: "객체" — innerArrow는 화살표 함수 → 상위(nested)의 this = obj

const fn = obj.normal;
fn();
// 1: "전역" (또는 undefined) — 메서드를 변수에 분리 → 점 없이 호출 → 전역
```

**핵심 교훈**:
- 객체 메서드에 화살표 함수를 쓰면 `this`가 객체가 아니라 전역을 가리킨다!
- 메서드를 변수에 빼서 호출하면 `this`가 사라진다!
- 콜백에서 상위 this를 쓰려면 화살표 함수를 써라!

</details>

## 면접 대비

### 🔴 필수 질문 (면접 3대장 중 2번!)

**Q1. `this`가 결정되는 규칙을 설명해주세요.**

> JavaScript의 `this`는 함수가 호출되는 방식에 따라 4가지 규칙으로 결정된다. 첫째, 일반 함수 호출 시 `this`는 전역 객체(strict mode에서는 undefined). 둘째, 메서드 호출(`obj.fn()`) 시 점(.) 왼쪽의 객체. 셋째, `call`/`apply`/`bind`로 명시적으로 지정한 객체. 넷째, `new` 키워드로 호출 시 새로 생성되는 객체. 화살표 함수만 예외로, 자체 `this`를 생성하지 않고 정의된 위치의 상위 스코프 `this`를 그대로 상속한다.

**Q2. 화살표 함수의 `this`는 일반 함수와 어떻게 다른가요?**

> 일반 함수의 `this`는 호출 방식에 따라 동적으로 결정되지만, 화살표 함수의 `this`는 자신이 정의된 위치의 상위 스코프 `this`를 정적으로 상속한다(Lexical this). `call`/`apply`/`bind`로도 바꿀 수 없다. 이 때문에 `setTimeout` 콜백이나 React 이벤트 핸들러에서 상위 컨텍스트의 `this`를 안전하게 사용할 수 있다.

**Q3. `call`, `apply`, `bind`의 차이?**

> 세 메서드 모두 함수의 `this`를 명시적으로 지정한다. `call`은 인수를 개별 전달하며 즉시 실행한다. `apply`는 인수를 배열로 전달하며 즉시 실행한다. `bind`는 `this`가 영구적으로 고정된 새로운 함수를 반환하며 즉시 실행하지 않는다.
