# 34. interface vs type

> **키워드**: `interface` `type` `extends` `선언 병합` `readonly` `인덱스 시그니처`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | interface와 type의 차이, 선택 기준, extends | 면접 단골 |
| 🟡 이해 | 선언 병합, readonly, 인덱스 시그니처 | 실무 패턴 |
| 🟢 참고 | 조건부 타입, 매핑 타입 | 고급 |

---

## 1. interface — 객체의 형태를 정의 🔴

```ts
interface User {
  id: number;
  name: string;
  email: string;
  age?: number;          // 선택적 속성 (없어도 됨)
  readonly createdAt: string;  // 읽기 전용 (수정 불가)
}

const user: User = {
  id: 1,
  name: "철수",
  email: "chul@test.com",
  createdAt: "2026-03-22",
};

user.name = "영희";      // ✅ 가능
user.createdAt = "변경";  // ❌ readonly 속성은 수정 불가!
```

### 확장 (extends)

```ts
interface Animal {
  name: string;
  age: number;
}

// 확장 — 기존 속성 + 새 속성
interface Dog extends Animal {
  breed: string;
  bark(): void;
}

// 다중 확장도 가능!
interface GuideDog extends Dog {
  owner: string;
  isWorking: boolean;
}
```

## 2. type — 어떤 타입이든 별칭 🔴

```ts
// 객체 (interface와 동일하게 사용 가능)
type User = {
  id: number;
  name: string;
  email: string;
};

// 유니온 (interface로는 불가!)
type Status = "idle" | "loading" | "success" | "error";
type StringOrNumber = string | number;

// 인터섹션으로 확장
type Animal = { name: string; age: number };
type Dog = Animal & { breed: string };

// 함수 타입
type Callback = (data: string) => void;
type Fetcher = (url: string) => Promise<Response>;

// 튜플
type Point = [number, number];
type RGB = [number, number, number];
```

---

## 3. 핵심 차이점 비교 🔴

| 구분 | `interface` | `type` |
|------|-----------|--------|
| **객체 타입** | ✅ | ✅ |
| **유니온 타입** | ❌ 불가! | ✅ `string \| number` |
| **인터섹션** | `extends`로 확장 | `&`로 합성 |
| **선언 병합** | ✅ (자동 합쳐짐) | ❌ 중복 선언 에러 |
| **함수/튜플 타입** | 가능하지만 어색 | ✅ 자연스러움 |
| **계산된 속성** | ❌ 불가 | ✅ `{ [K in Keys]: V }` |

### 선언 병합 (Declaration Merging) 🟡

```ts
// interface: 같은 이름으로 선언하면 자동 합쳐짐!
interface User { name: string; }
interface User { age: number; }  // 에러 없이 합쳐짐!
// User = { name: string; age: number }

// type: 같은 이름으로 선언하면 에러!
type User = { name: string };
type User = { age: number };  // ❌ Duplicate identifier 'User'
```

> 💡 선언 병합은 외부 라이브러리의 타입을 확장할 때 유용하다.
> 예: `next-auth`의 `Session` 타입에 커스텀 필드 추가

## 4. 실무 선택 가이드 🔴

```
✅ interface를 기본으로 사용하는 경우:
  - 객체의 형태(Shape)를 정의할 때
  - 컴포넌트 Props 타입
  - API 응답 타입
  - 클래스가 구현(implements)할 계약

✅ type을 사용하는 경우:
  - 유니온 타입: type Status = "a" | "b"
  - 함수 타입: type Fn = (x: number) => void
  - 유틸리티 타입 조합: type PartialUser = Partial<User>
  - 튜플: type Pair = [string, number]
  - 매핑/조건부 타입 (고급)

💡 팀 컨벤션이 가장 중요!
  - 팀에서 "객체는 interface, 나머지는 type" 규칙을 정하면 따르기
  - React 생태계에서는 type을 더 많이 쓰는 추세 (2026년)
  - 둘 다 가능한 상황이면 취향 차이. 일관성이 핵심!
```

## 5. React Props 타입 패턴 🔴

```ts
// Props 타입 정의 (type 사용 — React 커뮤니티 주류)
type ButtonProps = {
  children: React.ReactNode;       // 자식 요소 (텍스트, JSX 등)
  variant?: "primary" | "secondary"; // 선택적 + 리터럴 유니온
  size?: "sm" | "md" | "lg";
  disabled?: boolean;
  onClick?: () => void;             // 선택적 콜백
};

function Button({ children, variant = "primary", size = "md", ...rest }: ButtonProps) {
  return <button {...rest}>{children}</button>;
}

// 또는 interface 사용
interface CardProps {
  title: string;
  description: string;
  image?: string;
  children?: React.ReactNode;
}
```

## 6. 인덱스 시그니처 🟡

```ts
// 키 이름은 모르지만, 키-값의 타입은 아는 경우
interface StringMap {
  [key: string]: string;  // 어떤 문자열 키든, 값은 string
}

const colors: StringMap = {
  primary: "#4F46E5",
  secondary: "#64748B",
  danger: "#EF4444",
  // 어떤 키든 추가 가능, 값만 string이면 됨
};

// Record 유틸리티 타입 (더 간결한 대안)
type ColorMap = Record<string, string>;
```

## 면접 대비

### 🔴 필수 질문

**Q1. `type`과 `interface`의 차이?**

> `interface`는 객체의 형태를 정의하는 데 특화되어 있고, 같은 이름으로 재선언하면 자동으로 병합(Declaration Merging)된다. `extends`로 확장한다. `type`은 유니온, 인터섹션, 함수, 튜플 등 모든 종류의 타입에 별칭을 붙일 수 있지만 재선언이 불가하다. 객체 타입은 둘 다 가능하며, 팀 컨벤션에 따라 선택한다. React 생태계에서는 `type`이 더 많이 쓰이는 추세다.

**Q2. 언제 `interface`, 언제 `type`을 쓰나요?**

> 객체의 형태(Props, API 응답)를 정의하고 확장 가능성이 있으면 `interface`. 유니온 타입, 함수 타입, 유틸리티 타입 조합 등 `interface`로 표현이 불가하거나 어색한 경우 `type`. 가장 중요한 것은 팀 내 일관성이다.
