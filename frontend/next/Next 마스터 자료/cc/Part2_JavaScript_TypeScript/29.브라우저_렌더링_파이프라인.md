## 1. 브라우저 렌더링 5단계 🔴

> 💡 **비유: 건축 과정**
> HTML = 설계도(구조) → CSS = 인테리어 도면(스타일) → 합쳐서 → 측량(Layout) → 시공(Paint) → 완성(Composite)

```
Step 1: HTML 파싱 → DOM 트리
  └─ HTML 문서를 한 줄씩 읽으며 태그를 노드로 변환

Step 2: CSS 파싱 → CSSOM 트리
  └─ CSS 파일/스타일을 파싱하여 스타일 규칙 트리 생성

Step 3: DOM + CSSOM → 렌더 트리 (Render Tree)
  └─ 화면에 실제로 보이는 요소만 포함 (display:none 제외!)

Step 4: Layout (Reflow)
  └─ 각 요소의 정확한 위치와 크기를 계산 (좌표 계산)

Step 5: Paint & Composite
  └─ 픽셀로 그리고(Paint), 레이어를 합성(Composite)하여 화면에 표시
```

```
HTML ──▶ DOM 트리     ──┐
                       ├──▶ 렌더 트리 ──▶ Layout ──▶ Paint ──▶ Composite ──▶ 🖥️
CSS  ──▶ CSSOM 트리   ──┘
```

## 2. Reflow vs Repaint 🔴

```
Reflow (Layout 재계산) — 비용 큼! 🔴
  원인: width, height, margin, padding, position, font-size 변경
  과정: Layout → Paint → Composite 전부 다시!

Repaint (다시 칠하기) — 비용 중간 🟡
  원인: color, background, visibility, box-shadow 변경
  과정: Paint → Composite만 다시 (Layout은 건너뜀)

Composite만 — 비용 최소 🟢 ← GPU 가속!
  원인: transform, opacity 변경
  과정: Composite만 다시! (Layout, Paint 모두 건너뜀)
```

```
성능 비용:
  Reflow >>>>>>> Repaint >>> Composite

  ❌ width 변경: Layout + Paint + Composite (느림)
  🟡 color 변경: Paint + Composite (보통)
  ✅ transform 변경: Composite만 (빠름!) ← 애니메이션은 여기!
```

> 💡 **CSS 애니메이션 황금 규칙 (13번 복습)**:
> 이동 → `transform: translate()`, 크기 → `scale()`, 투명도 → `opacity`
> width, left, margin으로 애니메이션하면 매 프레임 Reflow → 버벅거림!

## 3. `<script>` 태그의 defer와 async 🟡

```html
<!-- 기본: HTML 파싱을 차단하고 스크립트를 다운+실행 -->
<script src="app.js"></script>

<!-- async: 다운로드는 병렬, 실행 시 파싱 중단 (실행 순서 보장 안 됨) -->
<script async src="analytics.js"></script>

<!-- defer: 다운로드는 병렬, HTML 파싱 완료 후 순서대로 실행 ← 가장 추천! -->
<script defer src="app.js"></script>
```

```
기본 <script>:     HTML ──■■■(다운+실행)──■■■── HTML 계속  (파싱 차단!)
async:             HTML ──────────────── HTML (다운은 병렬, 실행 시 잠깐 멈춤)
defer:             HTML ──────────────── HTML 완료 → 스크립트 실행 ✅
```

> 💡 Next.js에서는 프레임워크가 `<script>` 로딩을 자동 최적화하므로 직접 관리할 일이 드물다.
> 하지만 면접에서 "defer와 async의 차이?"는 반드시 물어본다!

## 4. Critical Rendering Path 최적화 🟡

```
화면을 빠르게 띄우려면 (FCP 개선):
1. CSS는 <head>에 (렌더 블로킹 → 일찍 로드)
2. JS는 <body> 끝 또는 defer (파싱 차단 방지)
3. 불필요한 CSS/JS 제거 (번들 크기 줄이기)
4. 이미지 lazy loading (뷰포트 밖은 나중에)
5. 웹 폰트 font-display: swap (FOIT 방지)
```

## 면접 대비

### 🔴 필수 질문

**Q1. 브라우저 렌더링 과정을 설명해주세요.**

> HTML을 파싱하여 DOM 트리, CSS를 파싱하여 CSSOM 트리를 생성한다. 이 둘을 결합하여 화면에 보이는 요소만으로 렌더 트리를 만들고, Layout 단계에서 각 요소의 위치와 크기를 계산한다. 마지막으로 Paint 단계에서 픽셀로 그리고, Composite 단계에서 레이어를 합성하여 화면에 표시한다.

**Q2. Reflow와 Repaint의 차이? 성능 최적화 방법?**

> Reflow는 요소의 크기/위치가 변해 Layout부터 다시 계산하는 과정이고, Repaint는 색상 등 시각적 속성만 변해 다시 칠하는 과정이다. Reflow가 비용이 훨씬 크다. 애니메이션에는 `transform`과 `opacity`만 사용하면 Compositor 단계에서만 처리되어 GPU 가속이 적용된다.

**Q3. `defer`와 `async`의 차이?**

> 둘 다 HTML 파싱과 병렬로 스크립트를 다운로드한다. `async`는 다운로드 완료 즉시 실행하며 실행 순서가 보장되지 않는다. `defer`는 HTML 파싱이 완전히 끝난 후 문서에 작성된 순서대로 실행된다. 일반적으로 `defer`가 권장된다.
