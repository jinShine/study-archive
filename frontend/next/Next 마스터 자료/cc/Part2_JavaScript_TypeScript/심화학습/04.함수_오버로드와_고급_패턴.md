## 1. 함수 오버로드 (Function Overloads) 🟡

> 같은 함수 이름이지만 **입력 타입에 따라 반환 타입이 달라지는** 것을 정의

```ts
// ❌ 유니온만으로는 반환 타입이 정확하지 않다
function parse(input: string | number): string | number {
  if (typeof input === "string") return parseInt(input);
  return String(input);
}
const result = parse("42"); // string | number 😱 (실제로는 number인데!)

// ✅ 오버로드 시그니처로 정확한 반환 타입 지정
function parse(input: string): number;    // 시그니처 1
function parse(input: number): string;    // 시그니처 2
function parse(input: string | number): string | number {  // 구현부
  if (typeof input === "string") return parseInt(input);
  return String(input);
}

const a = parse("42");  // number ✅
const b = parse(42);    // string ✅
```

### 실전: createElement 스타일

```ts
// 태그에 따라 반환되는 HTML 요소 타입이 다른 패턴
function createElement(tag: "div"): HTMLDivElement;
function createElement(tag: "span"): HTMLSpanElement;
function createElement(tag: "input"): HTMLInputElement;
function createElement(tag: string): HTMLElement;
function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const div = createElement("div");     // HTMLDivElement ✅
const input = createElement("input"); // HTMLInputElement ✅
```

> 💡 **실무에서**: 직접 오버로드를 작성할 일은 드물지만,
> 라이브러리(React, Next.js, Zod 등)의 타입을 읽을 때 오버로드를 이해해야 한다.

## 2. 제네릭 고급 패턴 🟡

### 여러 제네릭 타입 변수

```ts
// 키-값 쌍을 다루는 함수
function createPair<K extends string, V>(key: K, value: V): { key: K; value: V } {
  return { key, value };
}

const pair = createPair("name", "철수");
// { key: "name"; value: "철수" } — 리터럴 타입까지 추론!

// 객체에서 특정 키의 값을 안전하게 추출
function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] {
  return items.map(item => item[key]);
}

const users = [
  { id: 1, name: "철수", age: 25 },
  { id: 2, name: "영희", age: 30 },
];

pluck(users, "name");  // string[] ← ["철수", "영희"]
pluck(users, "age");   // number[] ← [25, 30]
pluck(users, "email"); // ❌ 에러! "email"은 키가 아님
```

### 제네릭 기본값

```ts
// 제네릭에 기본 타입 지정
type ApiResponse<T = unknown> = {
  data: T;
  status: number;
};

// T를 명시하지 않으면 unknown
const res1: ApiResponse = { data: "something", status: 200 };
// T를 명시하면 해당 타입
const res2: ApiResponse<User> = { data: { id: 1, name: "철수" }, status: 200 };
```

### const 타입 파라미터 (TS 5.0+)

```ts
// 일반 제네릭: 배열이 넓은 타입으로 추론
function createArray<T>(items: T[]): T[] {
  return items;
}
const arr = createArray(["a", "b", "c"]);
// string[] — 넓은 타입

// const 제네릭: 리터럴 타입을 유지!
function createArray<const T>(items: T[]): T[] {
  return items;
}
const arr = createArray(["a", "b", "c"]);
// readonly ["a", "b", "c"] — 리터럴 유지! (as const와 비슷한 효과)
```

## 3. 브랜디드 타입 (Branded Types) 🟢

> 같은 원시 타입이지만 **의미가 다른** 것을 구분하는 테크닉

```ts
// ❌ 문제: userId와 productId가 둘 다 string이라 섞여도 에러 없음
function getUser(userId: string) { }
function getProduct(productId: string) { }

const userId = "user_123";
getProduct(userId);  // 🔴 논리적으로 틀렸는데 에러 없음!

// ✅ 브랜디드 타입으로 해결
type UserId = string & { __brand: "UserId" };
type ProductId = string & { __brand: "ProductId" };

function createUserId(id: string): UserId {
  return id as UserId;
}
function createProductId(id: string): ProductId {
  return id as ProductId;
}

function getUser(userId: UserId) { }
function getProduct(productId: ProductId) { }

const userId = createUserId("user_123");
const productId = createProductId("prod_456");

getUser(userId);       // ✅
getProduct(productId); // ✅
getProduct(userId);    // ❌ 에러! UserId는 ProductId가 아님!
```

> 💡 **실무 활용**: 금액(원 vs 달러), ID(사용자 vs 상품), 인코딩된 문자열(Base64 vs URL) 등
> 같은 타입이지만 의미가 다른 값을 컴파일 타임에 구분할 때 강력하다.

## 면접 대비

### 🟡 개념 질문

**Q1. 함수 오버로드란?**

> 같은 이름의 함수가 입력 타입에 따라 다른 반환 타입을 가지도록 여러 시그니처를 선언하는 것이다. 구현부는 하나지만, 호출 시그니처를 여러 개 작성하여 TS가 입력에 맞는 정확한 반환 타입을 추론하게 한다. 직접 작성보다는 라이브러리 타입을 읽을 때 이해해야 하는 개념이다.
