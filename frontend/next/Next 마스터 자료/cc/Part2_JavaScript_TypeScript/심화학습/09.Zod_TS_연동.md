## 1. Zod가 해결하는 문제 🔴

> 💡 **비유: 공항 보안 검색대**
> TypeScript = 여권 검사 (문서상으로는 OK)
> Zod = X-ray 검사 (실제 내용물을 스캔!)
> TS는 **컴파일 타임**에만 체크하고 런타임에서는 사라진다.
> 외부 데이터(API, 폼 입력)는 런타임에 진짜 검증이 필요하다!

```ts
// ❌ TS만으로는 런타임 검증 불가!
type User = { name: string; age: number };

const data: unknown = await res.json();
const user = data as User;  // 💀 as로 억지 캐스팅! 런타임에 터질 수 있음!
console.log(user.name);     // data가 실제로 { name, age } 형태인지 보장 못함!

// ✅ Zod로 런타임 + 컴파일타임 둘 다 검증!
import { z } from "zod";

const UserSchema = z.object({
  name: z.string().min(2, "이름은 2자 이상"),
  age: z.number().min(1).max(150),
});

type User = z.infer<typeof UserSchema>;  // 스키마에서 타입 자동 추출!
// { name: string; age: number } — 직접 타입을 안 만들어도 됨!

const result = UserSchema.safeParse(data);
if (result.success) {
  console.log(result.data.name);  // ✅ 런타임 검증 완료 + 타입 안전!
} else {
  console.error(result.error.issues);  // 어떤 필드가 왜 틀렸는지 상세 정보
}
```

### 핵심 이점

```
1. Single Source of Truth: 스키마 1개 = 검증 규칙 + TS 타입 동시 생성
2. 런타임 안전성: API 응답, 폼 입력, URL 파라미터 등 외부 데이터 검증
3. 상세 에러 메시지: 어떤 필드가 왜 틀렸는지 자동 생성
4. React Hook Form 통합: zodResolver로 원클릭 연동
```

## 2. Zod 스키마 기본 문법 🔴

### 원시 타입

```ts
z.string()                // string
z.number()                // number
z.boolean()               // boolean
z.null()                  // null
z.undefined()             // undefined
z.literal("admin")        // 정확히 "admin"만 허용
z.enum(["admin", "user"]) // "admin" | "user"
```

### 문자열 검증

```ts
z.string()
  .min(2, "최소 2자")
  .max(50, "최대 50자")
  .email("올바른 이메일 형식이 아닙니다")
  .url("올바른 URL이 아닙니다")
  .regex(/^[a-z]+$/, "소문자만 허용")
  .trim()                    // 앞뒤 공백 자동 제거
  .toLowerCase()             // 자동 소문자 변환
```

### 숫자 검증

```ts
z.number()
  .min(1, "1 이상이어야 합니다")
  .max(150)
  .int("정수만 허용")
  .positive("양수만 허용")
  .nonnegative("0 이상만 허용")
```

### 객체 스키마

```ts
const UserSchema = z.object({
  name: z.string().min(2),
  email: z.string().email(),
  age: z.number().min(1).optional(),  // 선택적
  role: z.enum(["admin", "user"]).default("user"),  // 기본값
  bio: z.string().nullable(),         // string | null 허용
});

type User = z.infer<typeof UserSchema>;
// { name: string; email: string; age?: number; role: "admin" | "user"; bio: string | null }
```

### 배열 스키마

```ts
z.array(z.string())              // string[]
z.array(z.number()).min(1)       // 최소 1개 이상
z.array(UserSchema).max(10)      // User[] (최대 10개)

z.string().array()               // 위와 동일한 다른 문법
```

### 유니온과 판별 유니온

```ts
// 유니온
z.union([z.string(), z.number()])  // string | number

// 판별 유니온 (discriminatedUnion — 성능 더 좋음!)
const ResultSchema = z.discriminatedUnion("status", [
  z.object({ status: z.literal("success"), data: z.array(z.string()) }),
  z.object({ status: z.literal("error"), message: z.string() }),
]);
```

## 3. parse vs safeParse 🔴

```ts
const schema = z.string().email();

// parse: 실패하면 에러를 throw (try-catch 필요)
try {
  const email = schema.parse("invalid");  // 💥 ZodError!
} catch (e) {
  if (e instanceof z.ZodError) {
    console.error(e.issues);
  }
}

// safeParse: 절대 throw하지 않음! 결과 객체 반환 ✅ (실무 추천!)
const result = schema.safeParse("invalid");
if (result.success) {
  console.log(result.data);    // 검증된 데이터
} else {
  console.log(result.error.issues);  // 에러 상세 정보
  // [{ code: "invalid_string", message: "Invalid email", path: [] }]
}
```

> 💡 **실무 원칙**: `safeParse`를 기본으로 써라. `parse`는 에러를 던져서 흐름 제어가 어렵다.

## 4. z.infer — 스키마에서 타입 자동 추출 🔴

> "스키마 하나만 정의하면 검증 규칙 + TypeScript 타입이 동시에 완성된다!"

```ts
// ❌ 타입과 검증을 별도로 관리 → 불일치 위험!
type User = { name: string; email: string; age: number };
// + 검증 로직을 또 따로 작성...

// ✅ Zod: Single Source of Truth
const UserSchema = z.object({
  name: z.string().min(2),
  email: z.string().email(),
  age: z.number().positive(),
});

type User = z.infer<typeof UserSchema>;
// 자동 추출: { name: string; email: string; age: number }

// 스키마를 수정하면 타입도 자동으로 바뀜! 불일치 불가능!
```

### 폼 입력 타입과 DB 타입 분리

```ts
// 회원가입 폼 스키마 (비밀번호 확인 포함)
const RegisterSchema = z.object({
  name: z.string().min(2, "이름은 2자 이상"),
  email: z.string().email("올바른 이메일을 입력하세요"),
  password: z.string().min(8, "비밀번호는 8자 이상"),
  confirmPassword: z.string(),
}).refine(data => data.password === data.confirmPassword, {
  message: "비밀번호가 일치하지 않습니다",
  path: ["confirmPassword"],
});

type RegisterInput = z.infer<typeof RegisterSchema>;
// { name: string; email: string; password: string; confirmPassword: string }

// DB에 저장할 때는 confirmPassword 제외
type CreateUserData = Omit<RegisterInput, "confirmPassword">;
```

## 5. React Hook Form + Zod 통합 🔴

```bash
npm install react-hook-form @hookform/resolvers zod
```

```tsx
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

// 1. 스키마 정의 (검증 규칙 + 타입 동시!)
const LoginSchema = z.object({
  email: z.string().email("올바른 이메일을 입력하세요"),
  password: z.string().min(8, "8자 이상 입력하세요"),
});

type LoginForm = z.infer<typeof LoginSchema>;

// 2. 컴포넌트
function LoginPage() {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting },
  } = useForm<LoginForm>({
    resolver: zodResolver(LoginSchema),  // Zod와 연결!
  });

  const onSubmit = async (data: LoginForm) => {
    // data는 이미 검증 완료된 타입 안전한 데이터!
    console.log(data.email, data.password);
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <div>
        <input {...register("email")} placeholder="이메일" />
        {errors.email && <p className="text-red-500">{errors.email.message}</p>}
      </div>

      <div>
        <input {...register("password")} type="password" placeholder="비밀번호" />
        {errors.password && <p className="text-red-500">{errors.password.message}</p>}
      </div>

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? "로그인 중..." : "로그인"}
      </button>
    </form>
  );
}
```

> 💡 **핵심 흐름**:
> Zod 스키마 → `z.infer`로 타입 추출 → RHF `useForm<Type>` → `zodResolver` 연결
> = 검증 + 타입 + 에러 메시지가 한 곳에서 관리됨!

## 6. Next.js Server Actions + Zod 🟡

```ts
// actions.ts
"use server";
import { z } from "zod";

const CreatePostSchema = z.object({
  title: z.string().min(2, "제목은 2자 이상"),
  content: z.string().min(10, "내용은 10자 이상"),
});

export async function createPost(prevState: any, formData: FormData) {
  const raw = {
    title: formData.get("title"),
    content: formData.get("content"),
  };

  // 서버에서도 Zod로 검증!
  const result = CreatePostSchema.safeParse(raw);

  if (!result.success) {
    return {
      success: false,
      errors: result.error.flatten().fieldErrors,
      // { title: ["제목은 2자 이상"], content: ["내용은 10자 이상"] }
    };
  }

  // result.data는 타입 안전한 검증된 데이터!
  await db.post.create({ data: result.data });

  return { success: true, errors: {} };
}
```

> 💡 **3중 방어선**: HTML 검증(required) → 클라이언트 Zod(RHF) → 서버 Zod(Server Action)

## 면접 대비

### 🔴 필수 질문

**Q1. Zod를 사용하는 이유?**

> TypeScript는 컴파일 타임에만 타입을 체크하고 런타임에서는 사라진다. API 응답, 폼 입력 등 외부 데이터는 런타임에 실제 값을 검증해야 하며, Zod는 이 런타임 검증을 스키마 기반으로 수행한다. `z.infer<typeof Schema>`로 스키마에서 TypeScript 타입을 자동 추출하여, 검증 규칙과 타입을 한 곳(Single Source of Truth)에서 관리할 수 있다.

**Q2. React Hook Form과 Zod를 어떻게 연동하나요?**

> `@hookform/resolvers`의 `zodResolver`를 사용한다. Zod 스키마를 정의하고, `z.infer`로 타입을 추출한 뒤, `useForm<Type>({ resolver: zodResolver(schema) })`로 연결한다. 폼 제출 시 Zod가 자동으로 검증하고, 실패하면 `errors` 객체에 필드별 에러 메시지가 담긴다.
