# TS 심화 02. tsconfig.json 핵심 옵션

> **키워드**: `strict` `target` `module` `paths` `baseUrl` `jsx` `moduleResolution`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | strict 모드, paths 별칭, jsx 설정 | 프로젝트 세팅 필수 |
| 🟡 이해 | target/module, include/exclude, 각 strict 하위 옵션 | 트러블슈팅 |
| 🟢 참고 | composite, incremental, references | 모노레포 |

---

## 1. tsconfig.json이란? 🔴

> 💡 **비유: 교통 규칙**
> tsconfig는 "이 프로젝트에서 TypeScript를 어떤 규칙으로 사용할지" 정하는 설정 파일이다.
> 규칙이 느슨하면 버그가 새어나오고, 엄격하면 안전하지만 불편하다.

```
Next.js는 create-next-app 시 자동 생성해준다.
하지만 각 옵션이 뭘 하는지 알아야 트러블슈팅이 가능하다!
```

## 2. 핵심 옵션 해부 🔴

### Next.js 기본 tsconfig 분석

```json
{
  "compilerOptions": {
    /* ========== 타입 체크 엄격도 ========== */
    "strict": true,
    // ↑ 이 한 줄이 아래 7개를 전부 켠다:
    // "strictNullChecks": true     → null/undefined 체크 강제
    // "noImplicitAny": true        → any 추론 금지
    // "strictFunctionTypes": true   → 함수 매개변수 타입 엄격
    // "strictBindCallApply": true   → bind/call/apply 타입 체크
    // "strictPropertyInitialization": true → 클래스 속성 초기화 강제
    // "noImplicitThis": true       → this가 any가 되는 것 금지
    // "alwaysStrict": true         → "use strict" 자동 삽입

    /* ========== 모듈 & 경로 ========== */
    "module": "esnext",           // ESM 모듈 시스템 사용
    "moduleResolution": "bundler", // Vite/Webpack 번들러 방식 해석
    "baseUrl": ".",               // 경로 별칭의 기준점
    "paths": {
      "@/*": ["./src/*"]          // @/components → ./src/components
    },

    /* ========== 컴파일 대상 ========== */
    "target": "es2017",           // 컴파일 결과물의 JS 버전
    "lib": ["dom", "dom.iterable", "esnext"],  // 사용 가능한 API 타입

    /* ========== JSX (React) ========== */
    "jsx": "preserve",            // JSX를 변환하지 않고 Next.js/Babel에게 위임

    /* ========== 기타 ========== */
    "esModuleInterop": true,      // CommonJS 모듈을 ESM처럼 import 가능
    "skipLibCheck": true,         // node_modules의 .d.ts 타입 체크 건너뛰기 (빌드 속도↑)
    "forceConsistentCasingInFileNames": true, // 파일명 대소문자 일관성 강제
    "resolveJsonModule": true,    // .json 파일 import 허용
    "isolatedModules": true,      // 각 파일을 독립 모듈로 컴파일
    "incremental": true,          // 증분 컴파일 (빌드 속도↑)
    "noEmit": true,               // JS 파일 생성 안 함 (Next.js가 대신 처리)
  },

  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx"],  // 컴파일 대상 파일
  "exclude": ["node_modules"]                            // 제외할 파일
}
```

## 3. 가장 중요한 옵션 상세 해설 🔴

### strict: true — 절대 끄지 마라!

```ts
// strict가 꺼져 있으면 이런 위험한 코드가 통과됨:

// strictNullChecks: false일 때
const user: User = getUser();
console.log(user.name);  // 🔴 user가 null일 수 있는데 에러 없음 → 런타임 폭발!

// noImplicitAny: false일 때
function add(a, b) {  // 🔴 a, b가 any로 추론 → 타입 안전성 0
  return a + b;
}
```

> ⚠️ **strict: true는 타협 불가!** 끄면 TypeScript를 쓰는 의미가 없다.
> "에러가 너무 많아서 끈다"는 것은 "안전벨트가 불편해서 뺀다"와 같다.

### paths — 경로 별칭 (절대 경로 import) 🔴

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"],
      "@/components/*": ["./src/components/*"],
      "@/lib/*": ["./src/lib/*"],
      "@/types/*": ["./src/types/*"]
    }
  }
}
```

```ts
// ❌ 상대 경로 — 파일 이동하면 전부 깨짐
import { Button } from "../../../components/ui/Button";
import { formatDate } from "../../../../lib/utils";

// ✅ 절대 경로 — 어디서든 동일한 경로
import { Button } from "@/components/ui/Button";
import { formatDate } from "@/lib/utils";
```

> 💡 Next.js는 `@/*` 별칭을 create-next-app 시 자동 설정해준다.

### target vs lib

```
target: 컴파일 결과물의 JavaScript 버전
  "es5"     → 모든 구형 브라우저 지원 (IE11)
  "es2017"  → async/await 네이티브 사용 ← Next.js 기본
  "esnext"  → 최신 JS 기능 전부 사용

lib: 코드에서 사용 가능한 API 타입 정의
  "dom"           → document, window, fetch 등 브라우저 API
  "dom.iterable"  → NodeList의 forEach 등
  "esnext"        → Promise, Map, Set 등 최신 JS API
```

## 4. 자주 마주치는 tsconfig 트러블슈팅 🟡

### "Cannot find module '@/...'"

```
원인: paths를 설정했지만 번들러에도 알려줘야 함

Next.js: 자동 처리 ✅ (별도 설정 불필요)
Vite: vite.config.ts에 resolve.alias 추가 필요
```

### "JSX element implicitly has type 'any'"

```json
// jsx 설정이 빠졌을 때 발생
{
  "compilerOptions": {
    "jsx": "preserve"  // React + Next.js
    // 또는 "react-jsx" (순수 React, Vite)
  }
}
```

### "모듈을 찾을 수 없습니다" (.css, .svg 등)

```ts
// src/types/modules.d.ts 파일 생성
declare module "*.css" {
  const content: { [className: string]: string };
  export default content;
}

declare module "*.svg" {
  const content: React.FunctionComponent<React.SVGAttributes<SVGElement>>;
  export default content;
}
```

## 5. 실습: tsconfig 해석 챌린지

```json
// 아래 tsconfig의 각 옵션이 하는 역할을 설명하라

{
  "compilerOptions": {
    "strict": true,
    "noEmit": true,
    "jsx": "preserve",
    "moduleResolution": "bundler",
    "paths": { "@/*": ["./src/*"] },
    "skipLibCheck": true
  },
  "include": ["**/*.ts", "**/*.tsx"],
  "exclude": ["node_modules", ".next"]
}
```

<details>
<summary>정답 보기</summary>

```
strict: true          → 7가지 엄격 옵션 전부 활성화 (타입 안전성 최대화)
noEmit: true          → JS 파일을 직접 생성하지 않음 (Next.js 빌드 시스템이 대신 처리)
jsx: "preserve"       → JSX 문법을 변환하지 않고 그대로 둠 (Next.js/SWC가 변환)
moduleResolution: "bundler" → import 해석을 Webpack/Turbopack 방식으로
paths: @/* → ./src/*  → @/로 시작하는 import를 src/ 폴더로 매핑 (절대 경로)
skipLibCheck: true    → node_modules 안의 .d.ts 타입 체크 건너뜀 (빌드 속도 향상)
include              → .ts, .tsx 파일만 컴파일 대상
exclude              → node_modules와 .next는 제외
```

</details>

## 핵심 요약

```
┌────────────────────────────────────────────────┐
│         tsconfig.json 핵심 요약                 │
├────────────────────────────────────────────────┤
│                                                │
│  strict: true → 절대 끄지 마라! (TS의 존재 이유)│
│  paths: @/* → 상대 경로 지옥 탈출               │
│  noEmit: true → Next.js가 빌드 담당             │
│  jsx: "preserve" → JSX 변환은 Next.js/SWC에게   │
│  skipLibCheck: true → 빌드 속도 향상            │
│                                                │
│  트러블슈팅:                                     │
│    모듈 못 찾음 → paths + 번들러 설정 확인       │
│    .css/.svg import → d.ts 선언 파일 추가       │
│                                                │
└────────────────────────────────────────────────┘
```
