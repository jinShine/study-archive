## 1. catch의 error는 unknown이다 🔴

```ts
// TS 4.4+부터 catch의 error는 unknown 타입!
try {
  await fetch("/api/data");
} catch (error) {
  // error: unknown — 뭐가 올지 모름!

  // ❌ 바로 접근 불가
  console.error(error.message);  // 에러! unknown에 message 없음

  // ✅ 타입 가드 후 접근
  if (error instanceof Error) {
    console.error(error.message);   // ✅ string
    console.error(error.stack);     // ✅ string | undefined
  } else {
    console.error("알 수 없는 에러:", error);
  }
}
```

### 실전 에러 처리 유틸 함수

```ts
// 에러를 안전하게 메시지로 변환
function getErrorMessage(error: unknown): string {
  if (error instanceof Error) return error.message;
  if (typeof error === "string") return error;
  return "알 수 없는 에러가 발생했습니다";
}

// 사용
try {
  await riskyOperation();
} catch (error) {
  const message = getErrorMessage(error);
  toast.error(message);  // UI에 표시
}
```

## 2. 커스텀 에러 클래스 🟡

> 에러의 종류를 구분하여 다르게 처리하고 싶을 때

```ts
// 기본 에러 확장
class AppError extends Error {
  constructor(
    message: string,
    public code: string,
    public statusCode: number = 500,
  ) {
    super(message);
    this.name = "AppError";
  }
}

// 구체적인 에러 클래스들
class NotFoundError extends AppError {
  constructor(resource: string) {
    super(`${resource}을(를) 찾을 수 없습니다`, "NOT_FOUND", 404);
    this.name = "NotFoundError";
  }
}

class ValidationError extends AppError {
  constructor(
    message: string,
    public fields: Record<string, string[]>,
  ) {
    super(message, "VALIDATION_ERROR", 400);
    this.name = "ValidationError";
  }
}

class UnauthorizedError extends AppError {
  constructor() {
    super("로그인이 필요합니다", "UNAUTHORIZED", 401);
    this.name = "UnauthorizedError";
  }
}

// 사용: instanceof로 분기
try {
  await updateUser(data);
} catch (error) {
  if (error instanceof ValidationError) {
    setErrors(error.fields);  // 필드별 에러 메시지 표시
  } else if (error instanceof UnauthorizedError) {
    router.push("/login");     // 로그인 페이지로
  } else if (error instanceof NotFoundError) {
    notFound();                // Next.js 404
  } else {
    throw error;               // 예상 못한 에러는 상위로 전파
  }
}
```

## 3. Result 패턴 — throw 없는 에러 처리 🟡

> 💡 **throw 대신 반환값으로 성공/실패를 표현하는 패턴**
> Go 언어, Rust에서 영감. Server Actions에서 특히 유용!

```ts
// Result 타입 정의 (판별 유니온!)
type Result<T, E = Error> =
  | { success: true; data: T }
  | { success: false; error: E };

// Result를 반환하는 함수
async function fetchUser(id: number): Promise<Result<User>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) {
      return { success: false, error: new Error(`HTTP ${res.status}`) };
    }
    const data = await res.json();
    return { success: true, data };
  } catch (error) {
    return { success: false, error: error instanceof Error ? error : new Error("Unknown") };
  }
}

// 사용: try-catch 없이 깔끔!
const result = await fetchUser(1);
if (result.success) {
  console.log(result.data.name);      // ✅ User 타입
} else {
  console.error(result.error.message); // ✅ Error 타입
}
```

### Server Actions에서의 Result 패턴

```ts
// actions.ts
"use server";

type ActionResult<T = void> =
  | { success: true; data: T; message: string }
  | { success: false; error: string; fieldErrors?: Record<string, string[]> };

export async function createPost(formData: FormData): Promise<ActionResult<{ id: number }>> {
  const raw = { title: formData.get("title"), content: formData.get("content") };

  const parsed = PostSchema.safeParse(raw);
  if (!parsed.success) {
    return {
      success: false,
      error: "입력값이 올바르지 않습니다",
      fieldErrors: parsed.error.flatten().fieldErrors,
    };
  }

  try {
    const post = await db.post.create({ data: parsed.data });
    return { success: true, data: { id: post.id }, message: "게시글이 작성되었습니다" };
  } catch {
    return { success: false, error: "서버 오류가 발생했습니다" };
  }
}
```

> 💡 **Result 패턴의 장점**:
> 1. throw가 없어서 에러 흐름이 예측 가능
> 2. 판별 유니온이라 TS가 자동으로 타입 좁힘
> 3. 성공/실패 데이터 구조가 명확

## 4. React에서의 에러 핸들링 전략 🔴

### 계층별 에러 처리

```
┌─────────────────────────────────────────┐
│  1. 컴포넌트 레벨                        │
│     try-catch (이벤트 핸들러, async 작업) │
│     → toast 알림, 에러 메시지 표시       │
├─────────────────────────────────────────┤
│  2. 라우트 레벨                          │
│     error.tsx (Next.js Error Boundary)   │
│     → 에러 UI + reset 버튼              │
├─────────────────────────────────────────┤
│  3. 전역 레벨                            │
│     global-error.tsx                     │
│     → 치명적 에러 시 전체 앱 복구        │
├─────────────────────────────────────────┤
│  4. 모니터링                             │
│     Sentry, LogRocket                    │
│     → 에러 추적, 알림, 디버깅 정보       │
└─────────────────────────────────────────┘
```

### 실전: 안전한 async 이벤트 핸들러

```tsx
// ❌ 에러를 무시하는 코드
const handleSubmit = async () => {
  const res = await fetch("/api/save", { method: "POST", body: JSON.stringify(data) });
  const result = await res.json();
  toast.success("저장 완료!");  // 실패해도 성공 토스트가 뜸! 💀
};

// ✅ 에러를 제대로 처리하는 코드
const handleSubmit = async () => {
  try {
    const res = await fetch("/api/save", { method: "POST", body: JSON.stringify(data) });
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const result = await res.json();
    toast.success("저장 완료!");
  } catch (error) {
    toast.error(getErrorMessage(error));
    // 또는: Sentry.captureException(error);
  }
};
```

## 5. Error.cause — 에러 체이닝 (ES2022) 🟢

```ts
// 원인 에러를 보존하면서 새 에러를 던지기
async function getUser(id: number): Promise<User> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res.json();
  } catch (error) {
    throw new Error("사용자를 가져올 수 없습니다", {
      cause: error,  // 원인 에러를 보존!
    });
  }
}

// 디버깅 시 원인 추적
try {
  await getUser(999);
} catch (error) {
  if (error instanceof Error) {
    console.error("에러:", error.message);
    console.error("원인:", error.cause);  // 원본 HTTP 에러
  }
}
```

```ts
// 1.
function getErrorMessage(error: unknown): string {
  if (error instanceof Error) return error.message;
  if (typeof error === "string") return error;
  return "알 수 없는 에러가 발생했습니다";
}

// 2.
type Result<T> =
  | { success: true; data: T }
  | { success: false; error: string };

// 3.
async function safeFetch<T>(url: string): Promise<Result<T>> {
  try {
    const res = await fetch(url);
    if (!res.ok) {
      return { success: false, error: `HTTP ${res.status}: ${res.statusText}` };
    }
    const data: T = await res.json();
    return { success: true, data };
  } catch (error) {
    return { success: false, error: getErrorMessage(error) };
  }
}

// 4.
function UserProfile({ userId }: { userId: number }) {
  const [state, setState] = useState<
    | { status: "idle" }
    | { status: "loading" }
    | { status: "success"; data: User }
    | { status: "error"; message: string }
  >({ status: "idle" });

  useEffect(() => {
    setState({ status: "loading" });
    safeFetch<User>(`/api/users/${userId}`).then(result => {
      if (result.success) {
        setState({ status: "success", data: result.data });
      } else {
        setState({ status: "error", message: result.error });
      }
    });
  }, [userId]);

  switch (state.status) {
    case "idle":
    case "loading":
      return <p>로딩 중...</p>;
    case "error":
      return <p className="text-red-500">{state.message}</p>;
    case "success":
      return <h1>{state.data.name}</h1>;
  }
}
```

## 면접 대비

### 🔴 필수 질문

**Q1. TypeScript에서 catch의 error는 왜 unknown인가?**

> JavaScript에서는 어떤 값이든 `throw`할 수 있다(`throw "에러"`, `throw 42`, `throw null`). 따라서 catch에서 받는 error의 타입을 `Error`로 확정할 수 없어 TypeScript 4.4부터 `unknown`으로 정의된다. 반드시 `instanceof Error` 타입 가드를 거친 후에 `message`나 `stack`에 접근해야 한다.

**Q2. 에러 처리를 어떻게 계층적으로 설계하나요?**

> 컴포넌트 레벨에서 try-catch로 예상 가능한 에러(네트워크, 유효성)를 처리하여 사용자에게 피드백한다. 라우트 레벨에서는 Next.js의 `error.tsx`가 예상치 못한 에러를 잡아 에러 UI와 복구 버튼을 제공한다. 전역 레벨에서는 `global-error.tsx`가 최후의 방어선이다. 모든 에러는 Sentry 같은 모니터링 서비스로 자동 전송하여 운영 중 추적한다.
