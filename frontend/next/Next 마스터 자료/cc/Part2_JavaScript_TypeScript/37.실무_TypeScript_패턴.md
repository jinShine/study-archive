## 1. any vs unknown vs never 🔴

### any — 타입 시스템을 끄는 스위치 ❌

```ts
let value: any = "hello";
value = 42;           // ✅ 아무거나 할당
value.toFixed(2);     // ✅ 컴파일 에러 없음
value.nonExistent();  // ✅ 컴파일 에러 없음 → 런타임에 폭발! 💥

// ❌ any는 TypeScript를 쓰는 의미를 없앤다. 절대 쓰지 마라!
// 유일한 허용: 마이그레이션 중 임시 사용, 외부 라이브러리 타입 미지원
```

### unknown — 안전한 any ✅

```ts
let value: unknown = "hello";
value = 42;           // ✅ 아무거나 할당 가능 (여기까지는 any와 같음)

value.toFixed(2);     // ❌ 에러! unknown은 직접 사용 불가!
// 반드시 타입 체크를 해야만 사용 가능:
if (typeof value === "number") {
  value.toFixed(2);   // ✅ number로 좁혀진 후에만 사용 가능
}
```

> 💡 **"뭐가 올지 모를 때"**: `any` 대신 `unknown`을 써라.
> 외부 데이터(API 응답, JSON.parse 결과, catch의 error)는 `unknown`이 안전하다.

```ts
// ✅ catch의 error는 unknown으로 처리
try {
  await fetch("/api/data");
} catch (error: unknown) {
  if (error instanceof Error) {
    console.error(error.message);
  }
}
```

### never — "절대 발생하지 않는 값"

```ts
// 함수가 절대 정상 반환하지 않을 때
function throwError(message: string): never {
  throw new Error(message);
}

function infiniteLoop(): never {
  while (true) { }
}

// 실전: switch의 완전성 검사 (Exhaustive Check)
type Shape = "circle" | "rectangle" | "triangle";

function area(shape: Shape) {
  switch (shape) {
    case "circle": return 0;
    case "rectangle": return 0;
    case "triangle": return 0;
    default:
      const _exhaustive: never = shape;
      // 새 Shape 추가했는데 case를 안 넣으면 여기서 에러! ✅
      throw new Error(`Unhandled shape: ${_exhaustive}`);
  }
}
```

### 한눈에 비교

```
any     → 뭐든 할당 ✅ / 뭐든 사용 ✅ → 위험! ❌
unknown → 뭐든 할당 ✅ / 사용 전 체크 필수 → 안전! ✅
never   → 할당 불가 / 절대 발생 안 함 → 완전성 검사용
```

## 2. as const — 불변 리터럴 타입 🟡

```ts
// 일반 객체: 타입이 넓게 추론
const config = {
  apiUrl: "https://api.example.com",  // string
  timeout: 3000,                       // number
  retries: 3,                          // number
};

// as const: 값 그 자체가 타입!
const config = {
  apiUrl: "https://api.example.com",  // "https://api.example.com" (리터럴!)
  timeout: 3000,                       // 3000 (리터럴!)
  retries: 3,                          // 3 (리터럴!)
} as const;
// + 모든 속성이 readonly가 됨 (수정 불가)

// 실전: Enum 대안 (Enum보다 가벼움!)
const STATUS = {
  IDLE: "idle",
  LOADING: "loading",
  SUCCESS: "success",
  ERROR: "error",
} as const;

type Status = typeof STATUS[keyof typeof STATUS];
// "idle" | "loading" | "success" | "error"

// 배열에도 사용
const ROLES = ["admin", "editor", "viewer"] as const;
type Role = typeof ROLES[number];  // "admin" | "editor" | "viewer"
```

> 💡 **Enum보다 `as const` 추천**: Enum은 런타임 코드를 생성하지만,
> `as const`는 컴파일 후 사라져서 번들 크기에 영향 없음!

## 3. satisfies — 타입 체크 + 추론 유지 🟡

```ts
type Colors = Record<string, string | string[]>;

// 문제: 타입을 명시하면 추론이 사라짐
const colors: Colors = {
  primary: "#4F46E5",
  gray: ["#F9FAFB", "#6B7280", "#111827"],
};
colors.primary.toUpperCase();  // ❌ 에러! string | string[]이라서

// ✅ satisfies: 타입을 체크하면서 추론은 유지!
const colors = {
  primary: "#4F46E5",
  gray: ["#F9FAFB", "#6B7280", "#111827"],
} satisfies Colors;

colors.primary.toUpperCase();  // ✅ string으로 추론 유지!
colors.gray.map(c => c);       // ✅ string[]으로 추론 유지!
```

## 4. 실무 React + TypeScript 패턴 🔴

### Props 타입 패턴

```tsx
// 기본 Props
type ButtonProps = {
  children: React.ReactNode;
  variant?: "primary" | "secondary" | "danger";
  size?: "sm" | "md" | "lg";
  disabled?: boolean;
  onClick?: () => void;
};

// HTML 속성 확장 (button의 모든 속성 + 커스텀)
type ButtonProps = {
  variant?: "primary" | "secondary";
} & React.ButtonHTMLAttributes<HTMLButtonElement>;

// children을 강제하는 패턴
type CardProps = React.PropsWithChildren<{
  title: string;
  description?: string;
}>;
```

### 이벤트 타입

```tsx
// 이벤트 핸들러 타입
const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  console.log(e.target.value);
};

const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
  e.preventDefault();
};

const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => {
  console.log("클릭!");
};
```

### API 응답 타입 설계

```ts
// 공통 API 응답 구조
type ApiResponse<T> = {
  success: true;
  data: T;
} | {
  success: false;
  error: { message: string; code: string };
};

// 엔드포인트별 타입 (판별 유니온!)
async function fetchUser(id: number): Promise<ApiResponse<User>> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();
}

const result = await fetchUser(1);
if (result.success) {
  console.log(result.data.name);    // ✅ User 타입 자동 완성!
} else {
  console.error(result.error.message); // ✅ 에러 타입 자동 완성!
}
```

### useState에 제네릭

```tsx
// 초기값으로 타입 추론이 안 될 때 제네릭 명시
const [user, setUser] = useState<User | null>(null);
const [items, setItems] = useState<Product[]>([]);
const [status, setStatus] = useState<"idle" | "loading" | "error">("idle");
```

type Product = {

  name: string;
  price: number;
  category: string;
  description?: string;
  images?: string[];
  status: "active" | "draft" | "sold_out";

// 과제 2
type ApiResponse<T> =
  | { success: true; data: T }
  | { success: false; error: { message: string; code: string } };

async function typedFetch<T>(url: string): Promise<ApiResponse<T>> {
  try {
    const res = await fetch(url);
    if (!res.ok) {
      return { success: false, error: { message: `HTTP ${res.status}`, code: "HTTP_ERROR" } };
    }
    const data: T = await res.json();
    return { success: true, data };
  } catch (e) {
    return { success: false, error: { message: "Network error", code: "NETWORK_ERROR" } };
  }
}

// 과제 3
function processInput(input: unknown) {
  if (typeof input === "string") {
    console.log(input.length);
    console.log(input.toUpperCase());
  }
}

## 면접 대비

### 🔴 필수 질문

**Q1. `any`와 `unknown`의 차이?**

> `any`는 타입 체크를 완전히 무시하여 어떤 연산이든 허용한다. 런타임 에러의 원인이 된다. `unknown`은 어떤 값이든 할당할 수 있지만, 사용하기 전에 반드시 타입 가드(`typeof`, `instanceof` 등)로 좁혀야 한다. 외부 데이터(API, JSON.parse, catch error)에는 `any` 대신 `unknown`을 사용해야 안전하다.

**Q2. `never` 타입은 언제 사용하나요?**

> 절대 발생하지 않는 값의 타입이다. 항상 에러를 던지는 함수나 무한 루프 함수의 반환 타입으로 사용된다. 실전에서는 `switch`문의 `default` 케이스에서 판별 유니온의 완전성을 검사(exhaustive check)하는 데 활용한다. 새로운 케이스를 추가했는데 핸들링을 빠뜨리면 컴파일 에러가 발생하여 누락을 방지한다.
