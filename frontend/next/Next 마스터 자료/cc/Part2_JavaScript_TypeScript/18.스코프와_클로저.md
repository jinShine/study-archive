## 1. 스코프(Scope)란? 🔴

> 💡 **비유: 시력 범위**
> "이 변수가 보이는(접근 가능한) 영역"이 스코프다.
> 방(블록) 안에서 만든 것은 방 안에서만 보인다.
> 거실(전역)에 놓은 것은 모든 방에서 보인다.

### 3가지 스코프

```js
// 1. 전역 스코프 (Global Scope)
const globalVar = "나는 어디서든 보인다";

// 2. 함수 스코프 (Function Scope) — var의 범위
function myFunction() {
  var functionVar = "나는 이 함수 안에서만 보인다";
  console.log(globalVar);    // ✅ 접근 가능 (전역은 어디서든)
}
console.log(functionVar);    // ❌ ReferenceError

// 3. 블록 스코프 (Block Scope) — let/const의 범위
if (true) {
  let blockVar = "나는 이 {} 안에서만 보인다";
  const alsoBlock = "나도!";
  var notBlock = "나는 블록을 뚫고 나간다!"; // var는 블록 무시!
}
console.log(blockVar);    // ❌ ReferenceError
console.log(notBlock);    // ✅ "나는 블록을 뚫고 나간다!" (var의 위험성!)
```

```
시각화:

┌─── 전역 스코프 ────────────────────────────────┐
│  const globalVar = "전역"                       │
│                                                │
│  ┌─── 함수 스코프 ────────────────────────┐    │
│  │  var functionVar = "함수"               │    │
│  │                                         │    │
│  │  ┌─── 블록 스코프 ────────────┐        │    │
│  │  │  let blockVar = "블록"      │        │    │
│  │  │  (여기서 globalVar 접근 ✅)  │        │    │
│  │  └─────────────────────────────┘        │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

## 2. 렉시컬 스코프(Lexical Scope) 🔴

> 💡 **비유: 출생지 결정**
> 함수의 스코프는 **어디서 호출되었는지**가 아니라 **어디서 정의(작성)되었는지**에 따라 결정된다.
> 서울에서 태어난 사람이 부산에서 일해도 주민등록 상 "서울 출생"인 것처럼.

```js
const x = 10;

function outer() {
  const x = 20;

  function inner() {
    console.log(x);  // ?
  }

  return inner;
}

const fn = outer();
fn();  // 20 ✅ (inner가 정의된 곳의 x = outer의 20)
```

```
inner()가 호출되는 곳(전역)의 x는 10이지만,
inner()가 정의된 곳(outer 내부)의 x는 20이다.

렉시컬 스코프에 의해 → 정의된 곳 기준 → 20 출력!
```

> ⚠️ **면접 핵심**: "JavaScript는 렉시컬 스코프를 따릅니다. 함수가 어디서 호출되느냐가 아니라, 어디서 정의되었느냐에 따라 상위 스코프가 결정됩니다."

## 3. 스코프 체인(Scope Chain) 🟡

> 변수를 찾을 때, 현재 스코프 → 상위 스코프 → 전역까지 타고 올라간다.

```js
const a = "전역 a";

function first() {
  const b = "first의 b";

  function second() {
    const c = "second의 c";
    console.log(c);  // ✅ 현재 스코프에서 발견
    console.log(b);  // ✅ 상위(first)에서 발견
    console.log(a);  // ✅ 전역에서 발견
    console.log(d);  // ❌ 어디에도 없음 → ReferenceError
  }

  second();
}
```

```
스코프 체인 탐색 순서:

second 스코프 → first 스코프 → 전역 스코프 → 없으면 에러!
    [c: ✅]       [b: ✅]        [a: ✅]      [d: ❌]
```

## 4. 클로저(Closure) — JavaScript의 마법 🔴

> 💡 **비유: 졸업 후에도 사물함 열쇠를 가진 졸업생**
> 학교(함수)를 졸업(실행 종료)했는데, 졸업생(내부 함수)이 사물함 열쇠(외부 변수 참조)를 아직 들고 있다.
> 학교가 문을 닫았어도, 열쇠만 있으면 사물함을 열 수 있다.

### 클로저의 정의

```
클로저 = 함수 + 그 함수가 정의된 렉시컬 환경의 조합

쉽게 말하면:
"이미 실행이 끝난 외부 함수의 변수에
 여전히 접근할 수 있는 내부 함수"
```

### 클로저의 동작

```js
function createCounter() {
  let count = 0;  // 외부 함수의 변수

  return function() {  // 내부 함수 (클로저!)
    count++;           // 외부 함수가 끝났는데도 count에 접근 가능!
    return count;
  };
}

const counter = createCounter();
// createCounter()는 실행이 끝났다. 하지만...

console.log(counter());  // 1
console.log(counter());  // 2
console.log(counter());  // 3
// count가 살아있다! 클로저가 count를 "기억"하고 있다!

const counter2 = createCounter();  // 새로운 클로저 → 새로운 count
console.log(counter2());  // 1 (counter와 독립적!)
```

```
메모리 시각화:

createCounter() 실행 종료 후:

  counter ──▶ function() { count++; return count; }
                    │
                    └──▶ 렉시컬 환경: { count: 3 } ← 가비지 컬렉션 안 됨!
                                                       (클로저가 참조하고 있으므로)

  counter2 ──▶ function() { count++; return count; }
                    │
                    └──▶ 렉시컬 환경: { count: 1 } ← 별도의 독립된 환경!
```

## 5. 클로저의 실무 활용 🔴

### 활용 1: 데이터 은닉 (Private 변수)

```js
function createBankAccount(initialBalance) {
  let balance = initialBalance;  // 외부에서 직접 접근 불가!

  return {
    deposit(amount) {
      balance += amount;
      return balance;
    },
    withdraw(amount) {
      if (amount > balance) return "잔액 부족!";
      balance -= amount;
      return balance;
    },
    getBalance() {
      return balance;
    }
  };
}

const myAccount = createBankAccount(10000);
myAccount.deposit(5000);    // 15000
myAccount.withdraw(3000);   // 12000
myAccount.getBalance();     // 12000
// myAccount.balance → undefined! (직접 접근 불가 ✅)
```

### 활용 2: 함수 팩토리

```js
function createMultiplier(factor) {
  return (number) => number * factor;  // factor를 클로저로 기억
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
const percent = createMultiplier(0.01);

double(50);   // 100
triple(50);   // 150
percent(50);  // 0.5
```

### 활용 3: React Hooks의 원리! 🔴

```js
// React의 useState가 클로저로 동작하는 원리 (극도로 단순화)
function simpleUseState(initialValue) {
  let state = initialValue;  // 클로저가 이 값을 기억!

  function getState() {
    return state;
  }

  function setState(newValue) {
    state = newValue;
    // 실제 React는 여기서 리렌더링을 트리거한다
  }

  return [getState, setState];
}

const [getName, setName] = simpleUseState("철수");
console.log(getName());  // "철수"
setName("영희");
console.log(getName());  // "영희" — 클로저가 state를 기억하고 있으므로!
```

> 💡 **React 연결**: `useState`, `useEffect`, `useCallback` 등 모든 Hooks는 클로저 원리로 동작한다.
> Hooks가 컴포넌트의 state를 "기억"할 수 있는 이유가 바로 클로저!

## 6. 클로저의 흔한 함정 🔴

### 함정: for 루프와 var

```js
// ❌ var의 클로저 함정
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
// 예상: 0, 1, 2
// 실제: 3, 3, 3 😱 (var는 함수 스코프라서 같은 i를 공유!)

// ✅ let으로 해결 (블록 스코프 → 각 반복마다 새로운 i)
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
// 결과: 0, 1, 2 ✅
```

```
var의 경우 (함수 스코프):
  모든 setTimeout 콜백이 같은 i(=3)를 바라봄

  setTimeout(() => console.log(i)) ──┐
  setTimeout(() => console.log(i)) ──┼──▶ i = 3 (루프 종료 후)
  setTimeout(() => console.log(i)) ──┘

let의 경우 (블록 스코프):
  각 반복마다 새로운 i가 생성됨

  setTimeout(() => console.log(i)) ──▶ i = 0
  setTimeout(() => console.log(i)) ──▶ i = 1
  setTimeout(() => console.log(i)) ──▶ i = 2
```

## 7. 실습: 클로저 마스터 챌린지

```js
// 과제 1: 호출 횟수를 세는 함수
// createCallCounter()를 호출하면, 실행될 때마다 "n번째 호출!" 을 출력하는 함수를 반환
const fn = createCallCounter();
fn(); // "1번째 호출!"
fn(); // "2번째 호출!"
fn(); // "3번째 호출!"

// 과제 2: 한 번만 실행되는 함수 (once)
// once(fn)은 fn을 최초 1번만 실행하고, 이후 호출은 무시하는 함수를 반환
const initialize = once(() => console.log("초기화 완료!"));
initialize(); // "초기화 완료!"
initialize(); // (아무것도 안 함)
initialize(); // (아무것도 안 함)

// 과제 3: 아래 코드의 출력을 예측하라
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// 출력: ?
```

## 면접 대비

### 🔴 필수 질문 (면접 3대장 중 1번!)

**Q1. 클로저(Closure)란 무엇인가요?**

> 클로저는 함수와 그 함수가 선언된 렉시컬 환경의 조합이다. 외부 함수의 실행이 끝나더라도, 내부 함수가 외부 함수의 변수를 참조하고 있으면 그 변수는 메모리에서 해제되지 않고 유지된다. 이를 통해 데이터 은닉(private 변수), 함수 팩토리, 상태 유지 등의 패턴을 구현할 수 있다. React의 Hooks(useState, useEffect 등)도 클로저 원리로 동작한다.

**Q2. 클로저의 실무 활용 예시를 들어주세요.**

> 첫째, 데이터 은닉 — 외부에서 직접 접근할 수 없는 private 변수를 만들 수 있다. 둘째, 함수 팩토리 — `createMultiplier(2)`처럼 설정값을 기억하는 특화된 함수를 생성할 수 있다. 셋째, React Hooks — `useState`가 컴포넌트의 상태를 렌더링 사이에 기억하는 것이 클로저 원리다.

**Q3. `var`를 `for`문에서 `setTimeout`과 함께 쓰면 왜 문제가 생기나요?**

> `var`는 블록 스코프가 아닌 함수 스코프이므로, 루프의 모든 반복에서 동일한 변수 `i`를 공유한다. `setTimeout` 콜백이 실행될 때는 루프가 이미 종료되어 `i`가 최종값이 되어 있으므로 모든 콜백이 같은 값을 출력한다. `let`은 블록 스코프이므로 각 반복마다 새로운 `i`가 생성되어 이 문제가 해결된다.

**Q4. 렉시컬 스코프란?**

> 함수의 상위 스코프가 함수가 호출된 위치가 아니라 정의(작성)된 위치에 의해 결정되는 스코프 규칙이다. JavaScript는 렉시컬 스코프를 따르므로, 내부 함수는 자신이 작성된 위치의 외부 변수에 접근할 수 있다.
