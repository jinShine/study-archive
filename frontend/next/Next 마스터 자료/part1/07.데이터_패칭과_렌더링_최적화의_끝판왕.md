## 1. 서버 컴포넌트 페칭 — async/await 한 줄의 혁명 🔴

### ❌ 과거 (CSR): useEffect + useState의 폭포수

```tsx
/* ❌ 전통적인 리액트 방식 — 브라우저가 직접 데이터 요청 */
'use client';
import { useState, useEffect } from 'react';

export default function Page() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('/api/products')           // 브라우저에서 요청 (네트워크 탭에 기록됨)
      .then((res) => res.json())
      .then((data) => {
        setData(data);
        setLoading(false);
      });
  }, []);

  if (loading) return <div>로딩 중...</div>;
  return <div>{data.name}</div>;
}
```

### ✅ 현재 (서버 컴포넌트): async/await 한 줄

```tsx
/* ✅ Next.js 서버 컴포넌트 — 서버가 직접 데이터 요청 */
// 'use client' 없음 = 서버 컴포넌트

interface Post {
  id: number;
  title: string;
  body: string;
}

export default async function ProductsPage() {
  // 서버에서 직접 fetch → 브라우저 네트워크 탭에 안 찍힘!
  const res = await fetch('https://jsonplaceholder.typicode.com/posts');

  if (!res.ok) {
    throw new Error('데이터를 가져오는 데 실패했습니다.');
    // → error.tsx가 자동으로 낚아챔!
  }

  const posts: Post[] = await res.json();

  return (
    <div>
      <h1>Best Sellers (API Data)</h1>
      {posts.slice(0, 6).map((post) => (
        <div key={post.id}>
          <h3>{post.title}</h3>
          <p>${100 + post.id * 10}</p>
        </div>
      ))}
    </div>
  );
}
```

### 서버 컴포넌트 페칭의 압도적 장점

| 장점 | 설명 |
|------|------|
| **Zero Bundle Size** | lodash, date-fns 등 무거운 라이브러리 사용해도 브라우저 JS 전송량 0 |
| **보안** | API 키, DB 비밀번호가 브라우저에 절대 노출 안 됨 |
| **속도** | 서버-대-서버 통신 (같은 네트워크망) → 폭포수 지연 제거 |
| **SEO** | 완성된 HTML이 처음부터 제공 → 검색 엔진이 모든 콘텐츠 즉시 읽음 |

### 확인 방법

```
브라우저 개발자 도구 → Network 탭 → Fetch/XHR 필터
→ API 호출 로그가 안 보이면 = 서버에서 가져온 것! ✅

console.log를 서버 컴포넌트에 추가
→ VS Code 터미널에 출력 (브라우저 콘솔 ❌) = 서버 실행 확인 ✅
```

## 2. 렌더링 3대 전략 — Dynamic vs Static vs ISR 🔴

### Next.js 15의 중요한 변화

> ⚠️ **Next.js 14**: fetch 기본값 = 캐싱 ON (혼란 유발)
> ⚠️ **Next.js 15**: fetch 기본값 = **캐싱 OFF** (항상 최신 데이터 보장)

### 2-1. Dynamic (동적 렌더링) — 매번 새로 요리

```tsx
// Next.js 15 기본 동작 (옵션 없으면 자동으로 Dynamic)
const res = await fetch('https://timeapi.io/api/Time/current/zone?timeZone=Asia/Seoul');
// 또는 명시적으로:
const res = await fetch('...', { cache: 'no-store' });
```

- 접속할 때마다 서버가 새 HTML 생성
- **용도**: 주식 시세, 장바구니, 개인화 추천 등 실시간 필수 데이터

### 2-2. Static (정적 렌더링) — 한 번 얼리기

```tsx
const res = await fetch('...', {
  cache: 'force-cache',  // 빌드 시 한 번만 가져와서 영구 캐싱
});
```

- 빌드 시점에 한 번만 데이터 가져옴 → 평생 동일한 HTML
- **용도**: 회사 소개, 이용 약관, 블로그 포스트 등 변하지 않는 콘텐츠

### 2-3. ISR (점진적 정적 재생성) — 두 마리 토끼 🔴

```tsx
const res = await fetch('...', {
  next: { revalidate: 60 },  // 60초마다 백그라운드에서 갱신
});
```

> ⚠️ **단위 주의**: `revalidate` 값은 **초(Seconds)** 단위! (밀리초 아님!)
> `revalidate: 3000` = 3초가 아니라 **50분**!

### 3대 전략 비교

```
┌──────────┬──────────────────┬──────────────────┬──────────────────┐
│          │  ⚡ Dynamic       │  🧊 Static        │  ♻️ ISR          │
├──────────┼──────────────────┼──────────────────┼──────────────────┤
│ 코드     │ 기본값 / no-store │ force-cache       │ revalidate: 60   │
│ 실행     │ 매 요청마다       │ 빌드 시 1번       │ N초마다 갱신      │
│ 속도     │ 보통 (API 의존)   │ 최고 빠름 (0.01초)│ 빠름 (캐시 제공)  │
│ 최신성   │ 항상 최신         │ 빌드 시점 고정    │ N초 단위 최신     │
│ 용도     │ 실시간/개인화     │ 변하지 않는 콘텐츠 │ 쇼핑몰 목록/뉴스  │
│ 빌드로그 │ ƒ (Dynamic)      │ ○ (Static)       │ ● (SSG)          │
└──────────┴──────────────────┴──────────────────┴──────────────────┘
```

### ISR의 SWR 패턴 동작 원리 🟡

```
revalidate: 3 (3초) 설정 시:

[0초]  최초 접속 → 서버가 API 호출 → 캐시에 저장 (10:00:00)
[0~3초] 1만 명 접속해도 → 캐시된 10:00:00 즉시 전송 (API 호출 0!)
[3초]  캐시 만료 → 누군가 접속
       → 일단 상한 데이터(10:00:00) 즉시 전송 (대기 없음!)
       → 백그라운드에서 몰래 새 데이터 fetch
[갱신 완료] 다음 접속자부터 → 새 데이터(10:00:04) 제공

핵심: 사용자에게 단 한 번의 로딩 지연도 없다!
```

### fetch를 안 쓸 때 — Route Segment Config 🟢

```tsx
// Prisma, Axios 등 fetch 외 라이브러리 사용 시
// 파일 최상단에 한 줄 추가하면 페이지 전체에 ISR 적용
export const revalidate = 60;

export default async function Page() {
  const products = await prisma.product.findMany();
  // ...
}
```

## 3. generateStaticParams — 상세 페이지 정적 빌드 🔴

### 비유: VIP 초대장 사전 인쇄

```
"행사 시작 전에 1번~100번 VIP 네임택을 미리 인쇄해놔!"
= "빌드할 때 /products/1 ~ /products/100 HTML을 미리 만들어놔!"
```

### 구현

```tsx
// src/app/products/[id]/page.tsx

// 1. 빌드 타임에 딱 한 번 실행 → ID 명단 넘기기
export async function generateStaticParams() {
  const posts = await fetch('https://jsonplaceholder.typicode.com/posts')
    .then((res) => res.json());

  // 반드시 문자열 형태의 객체 배열!
  return posts.slice(0, 10).map((post: { id: number }) => ({
    id: String(post.id),  // ⚠️ 숫자가 아니라 문자열!
  }));
}

// 2. 상세 페이지 컴포넌트
type Props = {
  params: Promise<{ id: string }>;
};

export default async function ProductDetail({ params }: Props) {
  const { id } = await params;  // Next.js 15: await 필수!

  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
  if (!res.ok) return <div>상품을 찾을 수 없습니다.</div>;

  const product = await res.json();

  return (
    <div>
      <h1>{product.title} (ID: {id})</h1>
      <p>{product.body}</p>
    </div>
  );
}
```

### 명단에 없는 ID 접속 시 — `dynamicParams` 🟡

| 옵션 | 동작 |
|------|------|
| `true` (기본값) | 명단에 없는 ID → 서버가 즉시 렌더링 후 캐싱 |
| `false` | 명단에 없는 ID → 무조건 **404** |

```tsx
// 명단 외 접속 차단
export const dynamicParams = false;
```

### 빌드 로그 확인 🟢

```
npm run build 출력:

○  (Static)    : 순수 정적 페이지
ƒ  (Dynamic)   : 실시간 렌더링 페이지
●  (SSG)       : 데이터 포함 정적 빌드 ← generateStaticParams 성공!

● /products/[id]  (10 ms)
  ├ /products/1
  ├ /products/2
  └ /products/10   ← 10개 HTML 파일 미리 생성 완료!
```

## 4. generateMetadata — SEO & 소셜 미디어 지배 🔴

### 왜 필요한가?

```
링크를 카카오톡에 공유했을 때:

❌ 메타데이터 없음: "localhost:3000/products/1" (텍스트만)
✅ 메타데이터 있음: [썸네일 이미지] "아이폰 15 특가!" (예쁜 카드)

→ 클릭률(CTR) 차이 = 매출 차이
```

### 구현

```tsx
// src/app/products/[id]/page.tsx
import { Metadata } from 'next';

type Props = {
  params: Promise<{ id: string }>;
};

// 페이지 렌더링 전, <head> 태그를 완성하기 위해 먼저 실행
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { id } = await params;

  const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
  const product = await res.json();

  return {
    title: `${product.title} | My Shop`,       // 브라우저 탭 + 구글 검색 제목
    description: product.body.slice(0, 100),     // 구글 검색 요약문

    openGraph: {                                  // 카톡/슬랙 미리보기 카드
      title: `${product.title} 특가 판매 중!`,
      description: '한정 수량 특가 상품을 확인하세요!',
      images: [`/api/og?title=${product.title}`],
      type: 'website',
    },
  };
}

export default async function ProductDetail({ params }: Props) {
  // ... 페이지 컴포넌트
}
```

### Request Memoization — fetch 중복 걱정 없음! 🟡

```
"generateMetadata에서 fetch + 페이지에서 또 fetch = 2번 호출?"

→ ❌ 아님! Next.js가 자동 중복 제거

1번째 호출 (generateMetadata): 실제 API 요청 → 결과 메모리 캐싱
2번째 호출 (ProductDetail):    API 안 감 → 메모리에서 0.001초 반환

→ 개발자는 성능 걱정 없이, 필요한 곳에서 바로 fetch 가능!
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js 서버 컴포넌트에서 데이터를 가져오는 방식과 장점은?**

> 서버 컴포넌트는 `async` 함수로 선언하여 함수 본문에서 직접 `await fetch()`로 데이터를 가져옵니다. `useEffect`/`useState` 없이 동기식 코드처럼 직관적이며, 서버-대-서버 통신으로 폭포수 지연이 없습니다. **Zero Bundle Size**(무거운 라이브러리 사용해도 브라우저 JS 전송 0), **보안**(API 키 노출 불가), **SEO**(완성된 HTML 즉시 제공)가 핵심 장점입니다.

**Q2. Next.js 15에서 fetch 캐싱 기본값이 바뀐 이유와 3가지 전략은?**

> Next.js 14는 기본 캐싱으로 "데이터가 안 변한다"는 혼란이 잦았습니다. **15부터 기본값이 캐싱 OFF**로 변경되어 항상 최신 데이터를 보장합니다. 3가지 전략: **Dynamic**(기본값, 매 요청마다 새로 렌더링), **Static**(`force-cache`, 빌드 시 1번 → 영구 캐싱), **ISR**(`revalidate: N`, 정적 속도 + N초마다 백그라운드 갱신). ISR은 SWR 패턴으로 상한 데이터를 먼저 보여주고 뒤에서 갱신하여 사용자 대기 시간이 0입니다.

**Q3. `generateStaticParams`의 역할과 반환 형식은?**

> 동적 라우팅(`[id]`) 페이지를 **빌드 시점에 미리 정적 HTML로 생성**하기 위해 ID 명단을 프레임워크에 넘기는 함수입니다. 반드시 `[{ id: '1' }, { id: '2' }]` 형태의 **문자열** 객체 배열을 반환해야 하며, 키 이름은 폴더명(`[id]`)과 일치해야 합니다. `dynamicParams` 옵션으로 명단 외 ID 접속 시 동적 렌더링(true) 또는 404(false)를 선택할 수 있습니다.

**Q4. `generateMetadata`란 무엇이고 왜 필요한가요?**

> 동적 라우팅 상세 페이지에서 URL의 ID에 따라 `<title>`, `<meta description>`, Open Graph 태그를 **동적으로 생성**하는 함수입니다. SEO(구글 검색 노출)와 소셜 미디어 공유(카카오톡 미리보기 카드)에 필수적이며, 이 함수와 페이지 컴포넌트에서 동일한 fetch를 호출해도 **Request Memoization**으로 실제 API 호출은 1번만 됩니다.

**Q5. ISR의 `revalidate` 옵션 동작 과정을 설명해주세요.**

> `revalidate: 60` 설정 시: (1) 최초 접속 시 서버가 데이터를 가져와 캐시에 저장, (2) 60초 이내 접속자에게는 캐시된 HTML 즉시 제공(API 호출 0), (3) 60초 경과 후 접속 시 **상한 데이터를 먼저 즉시 전송**하고 백그라운드에서 새 데이터 fetch, (4) 갱신 완료 후 다음 접속자부터 최신 데이터 제공. 이 **SWR(Stale-While-Revalidate)** 패턴 덕분에 사용자에게 로딩 지연이 한 번도 발생하지 않습니다.

### 🟡 개념 질문

**Q6. Request Memoization이란?**

> 하나의 페이지 렌더링 사이클 내에서 **동일한 URL의 fetch가 여러 번 호출**되면, 첫 번째만 실제 네트워크 요청을 보내고 결과를 메모리에 캐싱합니다. 이후 호출은 메모리에서 즉시 반환됩니다. 덕분에 `generateMetadata`와 페이지 컴포넌트에서 같은 API를 각각 호출해도 성능 저하가 없으며, Props Drilling 없이 필요한 곳에서 바로 fetch하는 독립적 코드 작성이 가능합니다.

**Q7. Open Graph(OG) 태그의 비즈니스적 가치는?**

> 카카오톡, 슬랙, X(트위터) 등에 링크 공유 시 **썸네일 이미지 + 제목 + 설명**이 담긴 카드 형태로 렌더링되는 국제 표준 메타데이터입니다. 텍스트 URL만 전송될 때와 화려한 OG 카드가 표시될 때의 **클릭률(CTR) 차이**가 매우 크며, 이는 유입량과 매출 상승으로 직결되는 가성비 좋은 마케팅 디테일입니다.

## 📋 핵심 요약

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                    │
│  1️⃣  서버 컴포넌트 페칭: async/await 한 줄로 데이터 로딩           │
│     → Zero Bundle Size + 보안 + SEO + 폭포수 제거                 │
│     → useEffect/useState 완전 대체                                │
│                                                                    │
│  2️⃣  렌더링 3대 전략 (Next.js 15 기본 = 캐싱 OFF):                │
│     ⚡ Dynamic: 매번 새로 (기본값)                                 │
│     🧊 Static: 빌드 시 1번 (force-cache)                          │
│     ♻️ ISR: N초마다 갱신 (revalidate: N) ← 실무 표준              │
│                                                                    │
│  3️⃣  generateStaticParams: 동적 상세 페이지를 빌드 시 미리 생성    │
│     → 0.01초 로딩 + dynamicParams로 명단 외 ID 제어               │
│                                                                    │
│  4️⃣  generateMetadata: 동적 SEO + OG 태그                        │
│     → 구글 검색 + 카톡 미리보기 카드 최적화                        │
│     → Request Memoization으로 fetch 중복 걱정 0                    │
│                                                                    │
│  5️⃣  테스트 주의: 캐싱/ISR은 반드시 npm run build → npm start     │
│     (개발 모드에서는 정확히 동작하지 않음!)                         │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```
