## 1. 동적 라우팅(Dynamic Routes) — 대괄호 `[]`의 마법 🔴

### 비유: 조커 카드 (와일드카드)

상품이 10만 개인 쇼핑몰에서 폴더를 10만 개 만들 수는 없다.
대괄호 `[]`로 폴더를 감싸면 **어떤 값이든 받아들이는 만능 폴더**가 된다.

```
정적 라우팅 (고정)              동적 라우팅 (가변)
app/products/shoes/page.tsx    app/products/[slug]/page.tsx
app/products/shirt/page.tsx    → 이 파일 하나로 무한대 페이지!
app/products/hat/page.tsx
```

### 이름 짓기 규칙

| 폴더명 | 용도 | 예시 URL |
|--------|------|----------|
| `[id]` | DB 고유 번호 | `/products/123` |
| `[slug]` | 사람이 읽기 좋은 텍스트 (SEO 유리) | `/products/iphone-15-pro` |
| `[item]`, `[name]` 등 | 자유롭게 네이밍 가능 | 폴더명이 곧 params의 key |

### 실습: 상품 상세 페이지

```tsx
// src/app/products/[slug]/page.tsx

export default async function ProductDetailPage({
  params,
}: {
  params: Promise<{ slug: string }>;  // ⚠️ Next.js 15: Promise로 감싼다!
}) {
  const { slug } = await params;      // ⚠️ 반드시 await로 꺼낸다!

  return (
    <div>
      <h1>상품 상세 페이지</h1>
      <p>요청한 상품 코드: <strong>{slug}</strong></p>
    </div>
  );
}
```

```
동작 시뮬레이션:

/products/macbook   → slug = "macbook"
/products/12345     → slug = "12345"
/products/galaxy-s25 → slug = "galaxy-s25"

모두 같은 page.tsx 하나가 처리!
```

## 2. Next.js 15의 거대한 변화: 비동기 params 🔴

> ⚠️ **Next.js 15부터 `params`와 `searchParams`는 `Promise` 객체로 변경되었다.**
> 반드시 `async` 함수 + `await`로 처리해야 한다!

```
❌ 과거 (Next.js 14 이하)          ✅ 현재 (Next.js 15)
params: { id: string }            params: Promise<{ id: string }>
바로 접근 가능                      await로 기다려야 접근 가능
```

### 왜 비동기로 바뀌었나?

> 💡 **비유**: 셰프와 손님
>
> 과거: 셰프가 손님의 모든 주문이 끝날 때까지 아무것도 안 하고 대기
> 현재: 손님이 메뉴를 고민하는 동안 셰프는 미리 그릇을 세팅하고 재료를 손질
>
> → **스트리밍(Streaming)** 기술과 연결되어 사용자가 흰 화면 보는 시간을 줄여줌

## 3. 목록-상세(List-Detail) 패턴 🟡

현대 웹 서비스의 **표준 아키텍처**: 목록에서 클릭 → 상세 페이지로 이동

### 목록 페이지

```tsx
// src/app/products/page.tsx
import Link from "next/link";

const products = [
  { id: 1, name: "초경량 노트북", price: "1,200,000원" },
  { id: 2, name: "무소음 기계식 키보드", price: "185,000원" },
  { id: 3, name: "인체공학 버티컬 마우스", price: "89,000원" },
];

export default function ProductListPage() {
  return (
    <div className="grid grid-cols-3 gap-6">
      {products.map((product) => (
        <div key={product.id} className="border rounded-lg p-6">
          <h2 className="font-bold">{product.name}</h2>
          <p>{product.price}</p>
          {/* 각 상품의 고유 ID를 주소에 심어줌 */}
          <Link href={`/products/${product.id}`}>상세보기 →</Link>
        </div>
      ))}
    </div>
  );
}
```

### 상세 페이지

```tsx
// src/app/products/[id]/page.tsx

const products = [
  { id: 1, name: "초경량 노트북", price: "1,200,000원", description: "대학생 강추!" },
  { id: 2, name: "무소음 기계식 키보드", price: "185,000원", description: "사무실에서도 OK" },
  { id: 3, name: "인체공학 버티컬 마우스", price: "89,000원", description: "손목 보호 최고" },
];

export default async function ProductDetailPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;

  // ⚠️ params에서 꺼낸 값은 항상 string → 숫자 비교 시 parseInt 필수!
  const product = products.find((p) => p.id === parseInt(id));

  if (!product) {
    return <div className="text-red-500">존재하지 않는 상품입니다.</div>;
  }

  return (
    <div>
      <h1 className="text-4xl font-bold">{product.name}</h1>
      <p className="text-2xl">{product.price}</p>
      <p>{product.description}</p>
    </div>
  );
}
```

```
데이터 흐름:

[목록 페이지]  ──Link href="/products/1"──▶  [상세 페이지]
/products                                    /products/[id]
                                             params = { id: "1" }
                                             → find로 데이터 조회
                                             → 화면에 렌더링
```

## 4. 라우팅 우선순위 — 구체적인 것이 우선한다 🟡

```
products/
├── [slug]/         ← 와일드카드 (동적)
│   └── page.tsx
├── new-arrival/    ← 고정 이름 (정적)
│   └── page.tsx
```

| 접속 URL | 매칭되는 페이지 | 이유 |
|----------|----------------|------|
| `/products/macbook` | `[slug]/page.tsx` | 고정 폴더 없음 → 와일드카드 |
| `/products/new-arrival` | `new-arrival/page.tsx` | **고정 이름이 와일드카드보다 우선!** |

> 💡 특별한 디자인이 필요한 페이지는 따로 정적 폴더를 만들면 된다.

## 5. Search Params — 주소창의 물음표(?) 🔴

### 비유: 택시 기사님께 전달하는 추가 요구사항

```
Path (경로):       "강남역으로 가주세요"          → /products
Search Params:     "빨리, 고속도로로 가주세요"    → ?sort=asc&cat=laptop
```

```
URL 구조 분해:

https://mysite.com/products?sort=asc&q=노트북
                   ├─ Path ─┤├── Search Params ──┤
                   (목적지)    (옵션/필터/검색어)
```

### Path Params vs Search Params 사용 기준

| 구분 | Path Params (`[id]`) | Search Params (`?key=value`) |
|------|---------------------|-------------------------------|
| 용도 | 리소스 식별 (이것이 뭔지) | 필터/정렬/검색 (어떻게 보여줄지) |
| 예시 | `/products/123` | `/products?sort=asc&q=노트북` |
| 필수 여부 | 필수 (없으면 다른 페이지) | 선택 (없어도 기본 화면) |

### 실습: 검색 + 정렬 구현

```tsx
// src/app/products/page.tsx
import Link from "next/link";
import Search from "@/components/Search";

const products = [
  { id: 1, name: "초경량 노트북", price: 1200000 },
  { id: 2, name: "무소음 기계식 키보드", price: 185000 },
  { id: 3, name: "인체공학 버티컬 마우스", price: 89000 },
];

interface PageProps {
  searchParams: Promise<{ sort?: string; q?: string }>;  // Promise!
}

export default async function ProductListPage({ searchParams }: PageProps) {
  const { sort, q } = await searchParams;  // await 필수!

  // 서버 사이드 필터링 (브라우저가 아닌 서버 CPU 사용)
  let filtered = [...products];  // 원본 보호: 복사본에서 작업
  if (q) filtered = filtered.filter((p) => p.name.includes(q));
  if (sort === "asc") filtered.sort((a, b) => a.price - b.price);
  if (sort === "desc") filtered.sort((a, b) => b.price - a.price);

  return (
    <div>
      <Search initialQuery={q || ""} />

      <div className="flex gap-2">
        {/* 검색어를 유지하면서 정렬 조건만 변경 */}
        <Link href={{ query: { q, sort: "asc" } }}>가격 낮은 순</Link>
        <Link href={{ query: { q, sort: "desc" } }}>가격 높은 순</Link>
        <Link href="/products">초기화</Link>
      </div>

      {filtered.map((p) => (
        <div key={p.id}>
          <h2>{p.name}</h2>
          <p>{p.price.toLocaleString()}원</p>
        </div>
      ))}
    </div>
  );
}
```

### `<form>` 기반 서버 검색 — JS 없이 동작! 🟡

```tsx
// components/Search.tsx — 순수 서버 컴포넌트 (useState 없음!)
export default function Search({ initialQuery }: { initialQuery?: string }) {
  return (
    <form action="/products">  {/* 브라우저가 직접 /products?q=검색어 생성 */}
      <input
        name="q"               {/* name 속성이 URL의 ?q=... 를 만든다 */}
        placeholder="검색어를 입력하세요..."
        defaultValue={initialQuery}  {/* 서버에서 받은 초기값 표시 */}
      />
    </form>
  );
}
```

```
동작 흐름:

1. 사용자가 "노트북" 입력 후 엔터
2. 브라우저가 JS 없이 /products?q=노트북 으로 GET 요청
3. 서버가 await searchParams로 "노트북" 획득
4. 서버에서 필터링 → 완성된 HTML 전송
5. 검색창에 "노트북"이 이미 적힌 상태로 도착 (defaultValue)
```

> 💡 **Web Platform First**: `<form>`과 `name` 속성만으로 JS 로딩 전에도 검색 동작.
> 이것이 **점진적 향상(Progressive Enhancement)**.

## 6. 중첩 동적 라우팅 🔴

카테고리와 상품 ID를 동시에 표현하고 싶을 때:

```
src/app/shop/[category]/[itemId]/page.tsx

/shop/electronics/iphone-15   → category: "electronics", itemId: "iphone-15"
/shop/fashion/nike-air-max     → category: "fashion", itemId: "nike-air-max"
```

```tsx
// src/app/shop/[category]/[itemId]/page.tsx
export default async function ShopItemPage({
  params,
}: {
  params: Promise<{ category: string; itemId: string }>;
}) {
  const { category, itemId } = await params;

  return (
    <div>
      <p>카테고리: <strong>{category}</strong></p>
      <p>상품 ID: <strong>{itemId}</strong></p>
    </div>
  );
}
```

## 7. Catch-all Segments — 무한 깊이 경로 🔴

### 문제: 카테고리 깊이가 제각각

```
/shop/electronics                          ← 1단계
/shop/electronics/laptops/gaming           ← 3단계
/shop/electronics/components/gpu/nvidia    ← 4단계
```

각 단계마다 폴더를 만들 수 없다 → **Catch-all Segments**로 해결!

### 핵심 문법: `[...slug]` vs `[[...slug]]`

```
┌────────────────────┬──────────────────────────────────────┐
│ [...slug]          │ [[...slug]]                          │
│ (기본 Catch-all)   │ (선택적 Optional Catch-all)           │
├────────────────────┼──────────────────────────────────────┤
│ /shop/a     ✅     │ /shop/a     ✅                       │
│ /shop/a/b   ✅     │ /shop/a/b   ✅                       │
│ /shop       ❌ 404 │ /shop       ✅ (빈 배열 반환)         │
└────────────────────┴──────────────────────────────────────┘
```

> 💡 실무에서는 메인 페이지까지 커버하는 **이중 대괄호 `[[...slug]]`**를 주로 사용한다.

### 실습: 아마존 스타일 카테고리 페이지

```tsx
// src/app/shop/[[...slug]]/page.tsx
import { notFound } from "next/navigation";

interface PageProps {
  params: Promise<{ slug?: string[] }>;  // Optional이므로 ?
}

export default async function CategoryPage({ params }: PageProps) {
  const { slug = [] } = await params;  // 빈 배열 기본값 = /shop 대비

  const depth = slug.length;
  const currentCategory = depth > 0 ? slug[depth - 1] : "Home";

  return (
    <div>
      {/* Breadcrumb 네비게이션 */}
      <nav>
        <span>Shop</span>
        {slug.map((segment, index) => {
          const href = `/shop/${slug.slice(0, index + 1).join("/")}`;
          return (
            <span key={index}>
              {" > "}
              <a href={href}>{segment}</a>
            </span>
          );
        })}
      </nav>

      {/* 깊이에 따른 조건부 렌더링 */}
      {depth === 0 ? (
        <h1>오늘의 핫딜을 만나보세요!</h1>
      ) : (
        <h1>{currentCategory} ({depth}단계 카테고리)</h1>
      )}

      {/* URL에 특정 키워드 포함 시 배너 표시 */}
      {slug.includes("sale") && (
        <div className="bg-red-100">마감 임박 세일!</div>
      )}
    </div>
  );
}
```

```
URL별 slug 배열 변환:

/shop                              → slug = []              (depth: 0)
/shop/electronics                  → slug = ["electronics"]  (depth: 1)
/shop/electronics/laptops/gaming   → slug = ["electronics", "laptops", "gaming"]  (depth: 3)
/shop/clothes/sale/summer          → slug = ["clothes", "sale", "summer"]
                                     → sale 포함 → 세일 배너 표시!
```

### Breadcrumb 핵심 로직 🟢

```
slug = ["electronics", "laptops", "gaming"]

index 0: slug.slice(0, 1) = ["electronics"]
         → /shop/electronics

index 1: slug.slice(0, 2) = ["electronics", "laptops"]
         → /shop/electronics/laptops

index 2: slug.slice(0, 3) = ["electronics", "laptops", "gaming"]
         → /shop/electronics/laptops/gaming (마지막 → 링크 없이 굵은 텍스트)
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js의 동적 라우팅(Dynamic Routes)이란 무엇인가요?**

> 폴더명에 대괄호 `[id]`를 사용하면 고정된 경로가 아니라 **어떤 값이든 받아들이는 와일드카드** 경로가 됩니다. 하나의 `page.tsx`로 무한대의 페이지를 처리할 수 있으며, URL에 포함된 값은 `params` 객체를 통해 접근합니다. Next.js 15부터 params는 `Promise` 타입이므로 `await`로 처리해야 합니다.

**Q2. Next.js 15에서 params와 searchParams가 Promise로 변경된 이유는?**

> **스트리밍(Streaming) 최적화**를 위해서입니다. 파라미터 해석을 비동기로 처리하면, URL 파라미터가 준비되는 동안 서버가 페이지의 다른 부분(레이아웃, 정적 콘텐츠 등)을 먼저 렌더링하여 사용자에게 보낼 수 있습니다. 사용자가 하얀 화면을 보는 시간이 줄어드는 성능 최적화의 핵심입니다.

**Q3. Path Params와 Search Params의 차이는?**

> **Path Params**(`/products/123`)는 리소스를 식별하는 데 사용하며 필수값입니다. **Search Params**(`?sort=asc&q=노트북`)는 해당 리소스를 어떻게 보여줄지(정렬, 필터, 검색 등) 결정하는 선택적 옵션입니다. Path는 "무엇을", Search는 "어떻게"를 담당합니다.

**Q4. Catch-all Segments `[...slug]`와 Optional Catch-all `[[...slug]]`의 차이는?**

> `[...slug]`는 하위 경로가 **최소 1개 이상** 있어야 하며 루트 경로 접속 시 404가 됩니다. `[[...slug]]`(이중 대괄호)는 하위 경로가 **없어도 동작**하여 루트 경로에서 빈 배열을 반환합니다. 실무에서는 메인 페이지부터 깊은 카테고리까지 하나의 파일로 처리할 수 있는 `[[...slug]]`를 주로 사용합니다.

**Q5. Next.js에서 라우팅 우선순위는 어떻게 동작하나요?**

> **구체적인 것이 우선**합니다. `products/[slug]`(동적)과 `products/new-arrival`(정적)이 동시에 있으면, `/products/new-arrival` 접속 시 정적 폴더가 먼저 매칭됩니다. 나머지 URL들은 와일드카드가 처리합니다. 이 덕분에 특별한 디자인이 필요한 페이지만 정적 폴더로 분리할 수 있습니다.

### 🟡 개념 질문

**Q6. HTML `<form>`만으로 검색 기능을 구현하는 것의 장점은?**

> `<form action="/path">`와 `<input name="q">`만으로 브라우저가 자바스크립트 없이 `/path?q=검색어` URL을 생성합니다. 이 **Web Platform First** 접근법은 JS가 로딩되기 전에도 검색이 동작하고, `useState`/`onChange` 없이 서버 컴포넌트로 처리되므로 코드가 간결하며, 점진적 향상(Progressive Enhancement)의 핵심입니다.

**Q7. params에서 꺼낸 값이 항상 string인 이유와 주의점은?**

> URL은 본질적으로 **텍스트**이므로, params에서 추출한 값은 무조건 `string` 타입입니다. 숫자 ID로 데이터를 조회할 때 `===` 비교를 하면 타입이 달라 항상 false가 되므로, `parseInt(id)` 등으로 형변환 후 비교해야 합니다.
