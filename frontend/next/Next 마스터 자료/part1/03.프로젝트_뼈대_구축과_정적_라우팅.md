| 등급 | 섹션 | 의미 |

| 🔴 필수 | page.tsx/layout.tsx 규칙, 파일 시스템 라우팅, 중첩 레이아웃, 레이아웃 상태 유지 | 면접 단골 + 실무 매일 사용 |

## 1. 실무 환경 세팅 — 1분 만에 끝내는 CLI 🟢

### 1-1. Node.js 버전 확인

```bash
node -v
# 권장: v20.x 이상 (Active LTS)
# 최소: v18.18.0 이상
```

> ⚠️ Next.js 15는 React 19 기반이므로 최신 서버 컴포넌트(RSC) API를 100% 활용하려면 **v20 이상 LTS** 필수.

### 1-2. 프로젝트 생성

```bash
npx create-next-app@latest
# npx: 항상 최신 버전을 실행
# @latest: 2026년 기준 Next.js 15 이상 보장
```

### 1-3. 설정 마법사 — 권장 선택

| 질문 | 선택 | 이유 |
|------|------|------|
| TypeScript? | **Yes** | 컴파일 단계에서 오류 잡아냄 (2026년 필수) |
| ESLint? | **Yes** | 문법 오류/스타일 자동 감시 |
| Tailwind CSS? | **Yes** | Next.js 공식 기본 스타일링 엔진 |
| `src/` directory? | **Yes** | 설정 파일과 소스 코드를 분리 (실무 표준) |
| App Router? | **Yes** | 핵심! No 선택 시 구버전(Pages Router) 설치됨 |
| Turbopack? | **Yes** | Rust 기반 초고속 번들러, 서버 시작 최대 700배 빠름 |
| Import Alias? | **No** | 기본값 `@/*` 사용 (충분함) |

### 1-4. 개발 서버 실행

```bash
cd my-next-app
npm run dev

# 출력: Ready in 1234ms (Turbopack)
# 접속: http://localhost:3000
```

## 2. 프로젝트 구조 한눈에 보기 🟡

```
my-next-app/
├── node_modules/          # 외부 라이브러리 저장소 (절대 수정 금지)
├── public/                # 정적 파일 보관소 (이미지, 폰트, favicon)
├── src/
│   └── app/               # ⭐ App Router의 심장부 (실제 코딩 위치)
│       ├── layout.tsx      # Root Layout (최상위 껍데기)
│       ├── page.tsx        # 메인 페이지 (/ 경로)
│       └── globals.css     # 전역 스타일
├── eslint.config.mjs      # 코드 품질 검사 (Flat Config)
├── next.config.ts         # Next.js 프레임워크 설정
├── tailwind.config.ts     # CSS 프레임워크 설정
├── tsconfig.json          # TypeScript 컴파일 규칙
└── package.json           # 프로젝트 정보 및 의존성 명세서
```

### 루트 설정 파일 역할 요약

| 파일 | 비유 | 역할 |
|------|------|------|
| `package.json` | 주민등록증 | 프로젝트 정보, 의존성 목록, 스크립트 명령어 |
| `tsconfig.json` | 엄격한 감독관 | TypeScript 컴파일 규칙 (함부로 수정 금지) |
| `tailwind.config.ts` | 디자인 시스템 | 색상, 폰트, 여백 등 스타일 규칙 |
| `eslint.config.mjs` | 코드 경찰관 | 세미콜론, 미사용 변수 등 감시 |
| `next.config.ts` | 중앙 관제탑 | 이미지 도메인, 리다이렉트 등 서버 제어 |

## 3. 파일 시스템 라우팅의 절대 규칙 🔴

### 절대 규칙 1: `page.tsx` — 가게 간판

> **반드시 소문자 `page.tsx`여야만 브라우저에서 화면이 렌더링된다.**
> `Page.tsx`, `index.tsx`, `home.tsx` → 전부 ❌ (404 에러)

```
src/app/
├── page.tsx               👉 내사이트.com/          ✅ 화면 표시
├── about/
│   └── page.tsx           👉 내사이트.com/about     ✅ 화면 표시
├── login/
│   └── page.tsx           👉 내사이트.com/login     ✅ 화면 표시
```

```tsx
// src/app/about/page.tsx
export default function AboutPage() {
  return (
    <div>
      <h1>안녕하세요, About 페이지입니다!</h1>
      <p>폴더가 곧 URL 주소가 되었습니다. 이것이 파일 시스템 라우팅!</p>
    </div>
  );
}
```

> 💡 함수 이름(`AboutPage`)은 자유롭게 지을 수 있지만, **`export default`는 필수**다.

### 절대 규칙 2: `layout.tsx` — 인테리어 골조

여러 페이지가 공유하는 **공통 UI(껍데기)**를 정의한다.
페이지 이동 시에도 **다시 생성되지 않고 상태가 유지(Persist)**된다.

```tsx
// src/app/layout.tsx (Root Layout)
import type { Metadata } from "next";
import "./globals.css";

export const metadata: Metadata = {
  title: "Create Next App",
  description: "Generated by create next app",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    // ⚠️ html과 body 태그는 오직 Root Layout에만!
    <html lang="ko">
      <body>
        <nav>My Navbar (공통 영역)</nav>

        {/* page.tsx가 이 children 자리에 렌더링된다 */}
        <main>{children}</main>

        <footer>My Footer (공통 영역)</footer>
      </body>
    </html>
  );
}
```

### Root Layout 핵심 규칙

| 규칙 | 설명 |
|------|------|
| `<html>`, `<body>` 필수 | Root Layout에만 포함. 하위 레이아웃에는 ❌ |
| `children` prop 필수 | 하위 page.tsx나 중첩 layout이 들어오는 슬롯(구멍) |
| `metadata` export | SEO를 위한 title, description 설정 |

### 상황별 예약 파일 (Special Files) 🟡

| 파일명 | 역할 |
|--------|------|
| `loading.tsx` | 데이터 로딩 중 보여줄 UI (React Suspense 자동 적용) |
| `error.tsx` | 에러 발생 시 보여줄 UI (Error Boundary 자동 적용) |
| `not-found.tsx` | 존재하지 않는 경로 접근 시 404 UI |

## 4. Colocation (위치 병합) — 혁신적 파일 관리 🟡

> "관련된 코드는 멀리 두지 말고 **한 방(폴더)에 같이 두자**"

### ❌ 과거 (Pages Router): 폴더 안 파일 = 전부 라우트

```
dashboard/Chart.tsx 생성 → site.com/dashboard/Chart로 접근 가능 🚨
→ 부품으로 쓰려던 컴포넌트가 의도치 않게 페이지로 노출!
```

### ✅ 현재 (App Router): `page.tsx`가 아니면 절대 라우트 아님

```
src/app/dashboard/
├── page.tsx       ✅ /dashboard (유일한 진입점)
├── layout.tsx     ✅ 약속된 파일 (레이아웃)
├── Chart.tsx      ❌ 라우트 아님 (단순 컴포넌트 부품)
└── styles.css     ❌ 라우트 아님 (전용 스타일)
```

```tsx
// src/app/dashboard/page.tsx
import Chart from './Chart';  // 같은 폴더의 부품을 안전하게 import

export default function DashboardPage() {
  return (
    <div>
      <h2>대시보드 메인 페이지</h2>
      <Chart />  {/* 부품으로 사용 — 안전! */}
    </div>
  );
}
```

```tsx
// src/app/dashboard/Chart.tsx (부품 컴포넌트)
export default function Chart() {
  return (
    <div>
      <p>이곳은 Chart.tsx 내부입니다.</p>
      <p>/dashboard/Chart 주소로는 저를 만날 수 없어요!</p>
    </div>
  );
}
```

> ⚠️ 브라우저에서 `localhost:3000/dashboard/Chart` 접속 시 → **404 Error** (정상!)

## 5. 중첩 레이아웃 (Nested Layout) 🔴

### 비유: 샌드위치 조립

레이아웃은 **폭포수처럼 상위에서 하위로** 물려 내려간다.

```
/about/company 접속 시 조립 순서:

┌──────────────────────────────────────┐  ← 1단계: Root Layout (app/layout.tsx)
│  [Global Header]                      │
│  ┌──────────────────────────────────┐│  ← 2단계: About Layout (about/layout.tsx)
│  │  [About 전용 네비게이션]           ││
│  │  ┌──────────────────────────────┐││  ← 3단계: Company Page (about/company/page.tsx)
│  │  │  우리 회사를 소개합니다.       │││
│  │  └──────────────────────────────┘││
│  └──────────────────────────────────┘│
│  [Global Footer]                      │
└──────────────────────────────────────┘
```

```tsx
// src/app/about/layout.tsx — About 섹션 전용 껍데기
export default function AboutLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <section className="border-4 border-blue-500 m-4 p-4 rounded-lg">
      <nav className="bg-blue-100 p-4 mb-4 rounded">
        <h2>About 섹션 전용 네비게이션</h2>
        <ul className="flex gap-4">
          <li>소개</li>
          <li>연혁</li>
          <li>팀원</li>
        </ul>
      </nav>
      <main>{children}</main>  {/* page.tsx가 여기에 들어온다 */}
    </section>
  );
}
```

```tsx
// src/app/about/company/page.tsx — 중첩된 하위 페이지
export default function CompanyPage() {
  return (
    <div>
      <h2>우리 회사를 소개합니다.</h2>
      <p>이곳은 /about/company 경로입니다.</p>
    </div>
  );
}
```

## 6. 레이아웃의 불변성 — 상태 유지의 비밀 🔴

### 핵심 원리

```
기존 HTML 방식 (Hard Navigation)
  메뉴 클릭 → 화면 전체 새로고침 → 검색어/스크롤 위치 초기화 😫

Next.js 방식 (Soft Navigation)
  메뉴 클릭 → layout은 그대로 → page.tsx만 교체 → 상태 유지 ✅
```

```
┌─────────────────────────────────────────┐
│  layout.tsx (사이드바) — 절대 안 바뀜     │
│  ┌───────────┐  ┌─────────────────────┐ │
│  │ 검색창     │  │  page.tsx           │ │
│  │ [Next.js]  │  │  ← 이 부분만 교체!  │ │
│  │            │  │                     │ │
│  │ 메뉴 목록  │  │  홈 → 설정 → 홈     │ │
│  │ · 홈       │  │  페이지가 바뀌어도   │ │
│  │ · 설정     │  │  검색창 텍스트 유지! │ │
│  └───────────┘  └─────────────────────┘ │
└─────────────────────────────────────────┘
```

### 실습: 관리자 대시보드

```tsx
// src/app/dashboard/layout.tsx
import Link from "next/link";

export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex h-screen text-white">
      {/* 왼쪽 사이드바 (고정 영역) */}
      <aside className="w-64 bg-gray-900 p-6 flex flex-col border-r border-gray-800">
        <h1 className="text-2xl font-bold mb-6 text-emerald-400">Dev Dashboard</h1>

        {/* 상태 유지 테스트용 검색창 */}
        <div className="mb-6">
          <input
            type="text"
            placeholder="검색어를 입력해보세요..."
            className="w-full p-2 rounded bg-gray-800 text-sm"
          />
          <p className="text-xs text-gray-500 mt-2">
            여기에 글을 쓰고 메뉴를 이동해보세요!
          </p>
        </div>

        {/* 네비게이션 — Link 컴포넌트 사용 */}
        <nav className="space-y-2 flex-1">
          <Link href="/dashboard" className="block p-3 rounded hover:bg-gray-800">
            홈 (Overview)
          </Link>
          <Link href="/dashboard/settings" className="block p-3 rounded hover:bg-gray-800">
            설정 (Settings)
          </Link>
        </nav>
      </aside>

      {/* 오른쪽 콘텐츠 영역 (가변 영역) */}
      <main className="flex-1 bg-gray-100 text-black p-8 overflow-y-auto">
        {children}
      </main>
    </div>
  );
}
```

### 상태 유지 확인 테스트

```
Step 1. localhost:3000/dashboard 접속
Step 2. 사이드바 검색창에 "Next.js 테스트" 입력
Step 3. '설정 (Settings)' 클릭
Step 4. 확인: 우측 콘텐츠는 바뀌었지만, 검색창 텍스트는 그대로!  ✅
Step 5. 본문을 끝까지 스크롤 → 사이드바는 고정 유지  ✅
```

## 7. `public` 폴더 — 정적 자산의 창고 🟢

```tsx
// public/logo.png 파일을 사용하는 코드
import Image from 'next/image';

export default function Page() {
  return (
    // '/' 경로가 곧 public 폴더를 의미한다
    <Image src="/logo.png" alt="Logo" width={100} height={100} />
  );
}
```

> 💡 `public` 폴더에 넣은 파일은 별도 import 없이 **절대 경로 `/`로 바로 접근** 가능.
> 브라우저에서 `localhost:3000/favicon.ico` 입력 시 바로 표시됨.

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js App Router의 파일 시스템 라우팅이란 무엇인가요?**

> `app/` 디렉토리 내의 **폴더 구조가 곧 URL 경로**가 되는 라우팅 방식입니다. 별도의 라우팅 설정 파일 없이, 폴더를 만들고 `page.tsx` 파일을 생성하면 자동으로 해당 경로가 등록됩니다. 파일명은 반드시 소문자 `page.tsx`여야 하며, 이는 "설정(Configuration)보다 관습(Convention)"이라는 Next.js의 철학을 반영합니다.

**Q2. `page.tsx`와 `layout.tsx`의 역할과 차이를 설명해주세요.**

> `page.tsx`는 해당 경로의 **고유한 UI 콘텐츠**를 담당하며, 이 파일이 없으면 해당 경로는 404가 됩니다. `layout.tsx`는 해당 폴더와 하위 모든 페이지를 감싸는 **공통 UI(껍데기)**를 정의합니다. 가장 큰 차이는 **상태 유지** 여부로, 페이지 이동 시 `page.tsx`는 교체되지만 `layout.tsx`는 언마운트되지 않아 입력값, 스크롤 위치 등이 보존됩니다.

**Q3. 중첩 레이아웃(Nested Layout)이란 무엇인가요?**

> 하위 폴더에 `layout.tsx`를 추가하면 상위 레이아웃 위에 **폭포수처럼 겹쳐지는** 구조입니다. 예를 들어 `/about/company` 접속 시 Root Layout → About Layout → Company Page 순으로 샌드위치처럼 조립됩니다. 각 레이아웃은 독립적으로 상태를 유지하며, 페이지 이동 시 변경된 children 부분만 교체됩니다.

**Q4. Next.js에서 레이아웃의 상태가 유지되는 원리는 무엇인가요?**

> Next.js는 페이지 전환 시 `layout.tsx`를 **언마운트하지 않고 그대로 유지**합니다. `<Link>` 컴포넌트를 통한 Soft Navigation으로 변경된 `children`(page.tsx) 영역만 서버에서 받아와 교체하는 **부분 렌더링(Partial Rendering)** 방식입니다. 덕분에 사이드바 검색어, 스크롤 위치, 비디오 재생 상태 등이 보존됩니다.

### 🟡 개념 질문

**Q5. Colocation(위치 병합)이란 무엇인가요?**

> App Router에서는 `page.tsx`라는 약속된 이름의 파일만 라우트로 인식하고, 나머지 파일은 무시합니다. 덕분에 컴포넌트, 스타일, 유틸리티 파일을 해당 페이지와 **같은 폴더에 안전하게 배치**할 수 있습니다. 과거 Pages Router에서는 폴더 내 모든 파일이 라우트가 되어 부품 컴포넌트가 의도치 않게 노출되는 문제가 있었습니다.

**Q6. Root Layout의 필수 규칙은 무엇인가요?**

> Root Layout(`app/layout.tsx`)은 반드시 `<html>`과 `<body>` 태그를 포함해야 합니다. 이 두 태그는 오직 Root Layout에만 존재해야 하며, 하위 레이아웃에서는 절대 사용하면 안 됩니다. 또한 `children` prop을 필수로 받아 하위 페이지가 렌더링될 위치를 지정해야 합니다.

**Q7. Next.js의 Special Files(loading.tsx, error.tsx, not-found.tsx)는 어떤 역할을 하나요?**

> Next.js는 파일명으로 특정 상황의 UI를 약속합니다. `loading.tsx`는 데이터 로딩 중 자동으로 React Suspense를 적용하고, `error.tsx`는 에러 발생 시 Error Boundary를 자동 적용합니다. `not-found.tsx`는 존재하지 않는 경로에 대한 404 UI를 제공합니다. 모두 해당 폴더에 파일을 생성하는 것만으로 동작합니다.
