## 1. Library의 한계 — 설정(Configuration) 지옥 🔴

### 비유: 셀프 건축 vs 풀옵션 모델하우스

순수 리액트(Library)만으로 프로젝트를 만드는 건, **빈 땅을 산 건축가**와 같다.

```
리액트가 주는 것: 벽돌(UI) 하나뿐

건물을 짓기 위해 개발자가 직접 구해야 하는 것들:
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│  기둥     │  │  시멘트   │  │  전기배선  │  │  수도관   │
│ (라우터)  │  │(상태관리) │  │ (번들러)  │  │ (데이터)  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
      ↑              ↑             ↑             ↑
  react-router   redux/zustand   webpack      react-query
  직접 설치       직접 설치       직접 설정     직접 설치
```

이 "무한한 자유"는 곧 **설정의 늪**이 된다. 페이지 이동 하나를 위해 이런 코드를 써야 했다:

```jsx
/* ❌ React(Library) — 개발자가 모든 것을 수동으로 지시하는 방식 */
import { BrowserRouter, Routes, Route } from "react-router-dom";

function App() {
  return (
    <BrowserRouter>           {/* 1. 라우터 껍데기를 씌우고 */}
      <Routes>
        {/* 2. 어떤 URL에 어떤 화면을 보여줄지 일일이 '매핑' */}
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
      </Routes>
    </BrowserRouter>
  );
}
```

### 라이브러리 방식의 3대 페인 포인트

| 문제 | 설명 |
|------|------|
| 높은 피로도 | 외부 도구를 일일이 설치하고, 그 도구의 규칙을 새로 학습해야 한다 |
| 유지보수 막노동 | 페이지가 하나 늘어날 때마다 중앙 설정 파일을 열어 라우팅 코드를 수정해야 한다 |
| 주도권의 피곤함 | 코드를 '호출'하고 '제어'하는 모든 책임이 개발자인 '나'에게 있다 |

## 2. Framework의 등장 — 제어의 역전(IoC) 🔴

### 핵심 개념: 제어의 역전 (Inversion of Control)

> 💡 **비유**: 호텔 시스템
>
> 리액트 = "101호 문을 열면 침대를 보여줘"라고 **내가** 지시서를 작성
> Next.js = 101호 문을 여는 순간 **프레임워크가 알아서** 방을 보여줌

쉽게 말해 **'주도권이 넘어갔다'**는 뜻이다.

```
[ Library (React) ]                    [ Framework (Next.js) ]

  ┌──────────┐                           ┌──────────────┐
  │  내 코드  │ ──호출──▶ 도구들          │  Next.js 뼈대 │
  │  (중심)   │          (라우터 등)       │   (중심)      │
  └──────────┘                           └──────┬───────┘
                                                │
  내가 도구를 조종한다                    프레임워크가 ──호출──▶ 내 코드
  (자유도 ⬆️, 설정 피로도 ⬆️)                              (부품처럼 끼워짐)
                                         (생산성 ⬆️, 비즈니스 로직 집중 ⬆️)
```

```jsx
/* ✅ Next.js(Framework) — 설정 대신 '관습'을 따르는 방식 */

// app/page.tsx (메인 화면)
export default function HomePage() {
  return <h1>🏨 호텔 로비 (/)</h1>;
}

// app/about/page.tsx (About 화면)
export default function AboutPage() {
  // 라우터 설정 코드 없이, 폴더와 파일 생성만으로 '/about' 주소가 연결된다!
  return <h1>ℹ️ 101호 객실 (/about)</h1>;
}
```

> ⚠️ **핵심 포인트**: 개발자는 `export default`로 완성된 요리(컴포넌트)를 주방장(Next.js)에게 건네주기만 하면 된다.
> 렌더링을 **언제**, **어떻게** 할 것인지는 전적으로 프레임워크가 제어한다.

### 한 줄 정리

```
설정(Configuration)보다 관습(Convention)을 따른다
= Next.js를 배운다는 것은 프레임워크의 '규칙'에 동의하는 과정이다
```

## 3. 파일 시스템 라우팅 — 폴더가 곧 주소다 🔴

### ❌ 과거: 수동 지도의 늪

```jsx
/* ❌ React Router — 하나하나 직접 연결 */
import Home from './Home';
import About from './About';
import Mypage from './Mypage';

<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/about" element={<About />} />
  <Route path="/mypage" element={<Mypage />} />
</Routes>
```

| 문제 | 설명 |
|------|------|
| 깜빡하면 에러 | Routes 목록에 등록 안 하면 화면이 안 뜬다 |
| 오타의 지옥 | `/about`을 `/abput`으로 쓰면 404 에러 |
| 관리의 어려움 | 페이지 100개 = 라우팅 코드 100줄 |

### ✅ 현재: 폴더를 만들면 주소가 자동으로 생긴다

```
app/
├── page.tsx               👉 내사이트.com/
├── about/
│   └── page.tsx           👉 내사이트.com/about
├── login/
│   └── page.tsx           👉 내사이트.com/login
```

> 💡 **비유**: 호텔 프런트 vs 아파트 호수
>
> **과거 (수동 라우팅)** = 호텔 프런트
> → 직원이 장부를 뒤적이며 안내. 장부에 안 적으면 손님이 방을 못 찾음
>
> **현재 (파일 시스템 라우팅)** = 아파트 호수
> → 별도 안내 불필요. '101동 202호'라는 주소(폴더 경로)만 있으면 바로 찾아감

## 4. 동적 라우팅 [id] — 만능 열쇠 🔴

상품이 1억 개인 쇼핑몰을 만든다고 생각해보자.
폴더를 1억 개 만들어야 할까? **절대 아니다.**

### 대괄호 `[]` = 변수형 폴더 (Dynamic Route)

```
app/
└── products/
    └── [id]/          ← "여기는 어떤 단어든 들어올 수 있어!"
        └── page.tsx
```

### 동작 시뮬레이션

```
사용자가 "내사이트.com/products/galaxy" 입력

Step 1: Next.js가 products/ 폴더로 간다
Step 2: galaxy라는 폴더는 없지만, [id]라는 만능 폴더가 있다!
Step 3: Next.js → "사용자가 'galaxy'라고 입력했어.
         이걸 params라는 봉투에 담아줄게!"
Step 4: 페이지가 봉투를 열어보고 → "갤럭시 상품을 보여달라는구나!"

┌─────────────────────────────────────┐
│  params = { id: "galaxy" }          │
│                                     │
│  /products/iphone  → { id: "iphone" }│
│  /products/galaxy  → { id: "galaxy" }│
│  /products/123     → { id: "123" }   │
└─────────────────────────────────────┘
```

> 💡 **비유**: **빈 명찰**
> "방문객"이라고 적힌 빈 명찰이다. 철수가 오면 '철수'가 되고, 영희가 오면 '영희'가 되는 만능 명찰.

## 5. 렌더링 패러다임의 진화 — CSR → SSR → RSC 🟡

> 💡 **대비유**: 이 3단계를 **가구 주문과 배달 과정**으로 이해하자.

### 5-1. 1세대: CSR (Client Side Rendering) 🟡

> 💡 **비유**: 조립되지 않은 이케아 가구 박스
>
> 서버(공장)가 **텅 빈 상자**(빈 HTML)와 **두꺼운 조립 설명서**(거대한 JS 파일)만 던져준다.
> 브라우저(고객)가 땀 흘리며 다 읽고 → "나사(데이터)가 없네?" → 서버에 API 요청 → 도착 후 조립 시작

```
CSR 타임라인 (폭포수 현상)

[서버] ──빈 HTML 전송──▶ [브라우저]
                         │
                         ▼  JS 다운로드 (3~5초) ⏳
                         │
                         ▼  JS 실행 → "데이터가 없네?"
                         │
                         ▼  API 요청 ──────▶ [서버]
                         │                    │
                         │  ⏳ 기다림...       │
                         │                    │
                         ◀── 데이터 응답 ──────┘
                         │
                         ▼  드디어 화면 렌더링! 😮‍💨

사용자가 보는 것: 하얀 화면 → 로딩 스피너 → (한참 뒤) 실제 콘텐츠
```

```jsx
/* ❌ CSR의 폭포수 지연 */
import { useState, useEffect } from 'react';

export default function ProductPage() {
  const [data, setData] = useState(null);      // 1. 처음엔 빈 그릇
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {                            // 2. 화면이 텅 빈 채로 뜬 직후에야
    fetch('/api/products')                     //    서버로 데이터를 가지러 떠남
      .then((res) => res.json())
      .then((data) => {
        setData(data);
        setIsLoading(false);                   // 3. 긴 통신이 끝나야 로딩 사라짐
      });
  }, []);

  if (isLoading) return <div>로딩 중... 🌀</div>;  // 사용자가 먼저 마주하는 화면
  return <div>{data.name}</div>;                     // 한참 뒤에야 보이는 실제 콘텐츠
}
```

**CSR의 치명적 문제:**
- 하얀 화면 + 로딩 스피너를 하염없이 봐야 한다
- 구글 검색 봇(SEO)이 빈 페이지만 보고 "내용 없네" 하고 떠난다

### 5-2. 2세대: SSR (Server Side Rendering) 🟡

> 💡 **비유**: 랩으로 꽁꽁 싸인 완성형 가구
>
> 서버(공장)의 전문가가 **뼈대까지 다 조립해서** 배달해준다.
> 상자를 열자마자 완성된 가구(HTML)를 **즉시** 눈으로 볼 수 있다!
> 하지만... 랩으로 싸여 있어서 서랍을 열거나 스위치를 **누를 수(상호작용)는 없다.**

```
SSR 타임라인

[서버] ──완성된 HTML 전송──▶ [브라우저]
                              │
                              ▼ 즉시 화면 표시! 👀 (하지만 클릭 안 됨)
                              │
                              ▼ 거대한 JS 다운로드 ⏳ (여전히 무거움)
                              │
                              ▼ JS 실행 → 클릭 가능해짐! 🖱️

✅ 개선: 하얀 화면 사라짐 + SEO 해결
❌ 한계: 거대한 JS를 브라우저로 또 보내야 하는 근본적 무거움은 여전
```

### 5-3. 3세대: RSC (React Server Components) 🔴

> 💡 **비유**: 최고급 장비로 시공 완료된 '붙박이 가구'
>
> 서버(공장)에서 전문가들이 최고급 장비(DB 연결)로 **벽에 아예 완벽하게 설치**해버린다.
> 브라우저(고객)는 도구(JS)나 설명서를 전달받을 필요가 **0%**.
> 그저 집에 들어와 이미 완벽하게 설치된 가구를 **즉시 사용**하기만 하면 된다.

```jsx
/* ✅ RSC — Next.js의 근본, 압도적인 간결함 */

// 'use client'가 없다 = 이 코드는 오직 서버 안에서만 실행된다
import db from '@/lib/db';  // DB에 직접 연결해도 해킹 위험 0%

// async 하나만 붙이면 끝 (useState, useEffect 완전 증발!)
export default async function ProductPage() {

  // 서버 내부 통신 → 네트워크 지연 0초에 가까움
  const data = await db.product.findMany();

  // 브라우저는 이 '결과물(HTML)'만 배달받는다
  return (
    <div>
      {data.map((item) => (
        <div key={item.id}>{item.name}</div>
      ))}
    </div>
  );
}
```

> ⚠️ **Zero Bundle Size**: DB 조회 로직이든, 거대한 외부 라이브러리든, 브라우저로 전송되는 JS 용량은 **0바이트**.
> 브라우저는 오직 서버가 연산해낸 **결과물**만 받기 때문에 앱이 깃털처럼 가벼워진다.

### 렌더링 3세대 비교 요약

```
┌──────────┬──────────────────┬────────────────┬──────────────────┐
│          │  CSR (1세대)      │  SSR (2세대)    │  RSC (3세대)      │
├──────────┼──────────────────┼────────────────┼──────────────────┤
│  비유    │ 이케아 조립가구   │ 랩 싸인 완성품  │ 붙박이 시공 완료  │
│  화면    │ 하얀 화면 → 로딩  │ 즉시 표시       │ 즉시 표시         │
│  JS 전송 │ 전부 전송 (무거움)│ 전부 전송 (무거움)│ 0바이트 (가벼움) │
│  SEO     │ ❌ 빈 페이지     │ ✅ 콘텐츠 있음   │ ✅ 콘텐츠 있음    │
│  데이터  │ 브라우저에서 fetch│ 서버에서 fetch   │ 서버에서 직접 DB  │
└──────────┴──────────────────┴────────────────┴──────────────────┘
```

## 6. RSC의 동작 원리 — 4단계 시뮬레이션 🟢

```
Step 1. 요청 (Request)
  사용자가 페이지 주소를 입력하고 접속

Step 2. 서버 실행 (Server Execution)
  Next.js 서버가 컴포넌트를 실행 → DB에서 데이터를 즉시 가져옴

Step 3. 직렬화 (Serialization)
  결과물을 'RSC Payload'라는 가벼운 텍스트 지시서(JSON)로 압축
  예: "1번 구역에는 '아이폰', 2번 구역에는 '갤럭시' 글자만 넣어라"

Step 4. 전송 및 반영 (Streaming)
  가벼운 텍스트 지시서만 브라우저로 전송 → 1초 만에 화면 완성
```

## 7. 하이브리드 모델과 'use client'의 비밀 🔴

"서버에서 다 끝내버리면 '좋아요' 버튼 클릭은 어떻게 하나요?"

**정답**: 모든 것을 서버에서만 할 수는 없다.
사용자와 상호작용(`onClick`, `useState` 등)이 필요한 요소는 **브라우저(클라이언트)의 힘을 빌려야** 한다.

### 비유: 고급 레스토랑의 주방과 홀 직원

```
┌─────────────────────────────────────────────────┐
│                고급 레스토랑                       │
│                                                   │
│  ┌──────────────┐          ┌──────────────────┐  │
│  │   🍳 주방     │          │   🤵 홀           │  │
│  │  (서버)       │──요리──▶│  (브라우저)        │  │
│  │              │          │                    │  │
│  │ · DB 조회    │          │ · 클릭 이벤트      │  │
│  │ · 무거운 연산 │          │ · 상태 변화        │  │
│  │ · 절대 밖에   │          │ · 사용자와 대화    │  │
│  │   안 나감     │          │                    │  │
│  └──────────────┘          └──────────────────┘  │
│                                                   │
│  기본값: 모든 코드는 주방(서버)에 있다              │
│  'use client' = 주방에서 홀로 나가는 '출입증'       │
└─────────────────────────────────────────────────┘
```

### 규칙 정리

| 구분 | 서버 컴포넌트 (기본값) | 클라이언트 컴포넌트 |
|------|----------------------|-------------------|
| 선언 | 아무것도 안 씀 | 파일 맨 위에 `'use client'` |
| 역할 | DB 조회, 무거운 연산 | 클릭, 입력, 상태 변화 |
| JS 전송 | ❌ 안 보냄 | ✅ 브라우저로 보냄 |
| 비유 | 주방 (보이지 않는 곳) | 홀 직원 (손님과 대화) |

> ⚠️ **Next.js가 미친 듯이 빠른 이유**:
> 무거운 짐은 서버에 다 던져두고, 클릭 같은 **가벼운 기능만** 브라우저로 보낸다.
> "보낼 필요 없는 코드는 보내지 않는다"는 철학.

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. 리액트(Library)와 Next.js(Framework)의 근본적인 차이는 무엇인가요?**

> 리액트는 UI 라이브러리로, 라우팅·번들링·상태관리 등을 개발자가 직접 선택하고 조립해야 합니다. 반면 Next.js는 프레임워크로, **제어의 역전(IoC)** 원리에 따라 프레임워크가 애플리케이션의 흐름을 제어하고, 개발자는 규칙(Convention)에 따라 컴포넌트를 제공하는 방식입니다. 이를 통해 "설정(Configuration)보다 관습(Convention)"이라는 철학을 실현합니다.

**Q2. Next.js의 파일 시스템 라우팅이란 무엇인가요?**

> Next.js App Router에서는 `app/` 디렉토리 내의 **폴더 구조가 곧 URL 경로**가 됩니다. 별도의 라우팅 설정 파일 없이, 폴더를 만들고 `page.tsx` 파일을 생성하면 자동으로 해당 경로가 라우트로 등록됩니다. 동적 라우팅은 `[id]`처럼 대괄호로 폴더명을 감싸서 구현합니다.

**Q3. RSC(React Server Components)란 무엇이며, 왜 혁명적인가요?**

> RSC는 컴포넌트를 **서버에서만 실행**하고, 그 결과(HTML)만 브라우저로 전송하는 패러다임입니다. DB 조회나 무거운 라이브러리를 사용해도 브라우저로 전송되는 JS가 **0바이트(Zero Bundle Size)**이기 때문에 앱이 극도로 가벼워집니다. CSR의 폭포수 지연과 SEO 문제를 모두 해결한 3세대 렌더링 방식입니다.

**Q4. 서버 컴포넌트와 클라이언트 컴포넌트의 차이는 무엇인가요?**

> Next.js에서 **기본값은 서버 컴포넌트**입니다. 서버에서만 실행되므로 DB 직접 접근, async/await 사용이 가능하고 JS가 브라우저로 전송되지 않습니다. 반면, `onClick`이나 `useState` 같은 브라우저 상호작용이 필요한 컴포넌트는 파일 상단에 `'use client'`를 선언하여 **클라이언트 컴포넌트**로 만들어야 합니다. 이 하이브리드 모델이 Next.js 성능의 핵심입니다.

### 🟡 개념 질문

**Q5. CSR의 폭포수(Waterfall) 현상이란 무엇인가요?**

> CSR에서는 빈 HTML → JS 다운로드 → JS 실행 → API 요청 → 데이터 수신 → 렌더링 순서로 단계가 순차적으로 진행됩니다. 각 단계가 이전 단계의 완료를 기다려야 하므로 폭포수처럼 지연이 누적되어 사용자가 하얀 화면과 로딩 스피너를 오래 봐야 하는 문제가 발생합니다.

**Q6. CSR → SSR → RSC로의 진화 과정을 설명해주세요.**

> **CSR**(1세대)은 브라우저가 모든 렌더링을 담당해 초기 로딩이 느리고 SEO가 불가했습니다. **SSR**(2세대)은 서버에서 HTML을 미리 그려 보내 초기 화면은 빨라졌지만, 상호작용을 위한 거대한 JS를 여전히 브라우저로 보내야 했습니다. **RSC**(3세대)는 서버 전용 코드를 아예 브라우저로 보내지 않아 JS 전송량을 0으로 만들고, 서버에서 DB에 직접 접근할 수 있게 한 혁명적 패러다임입니다.

│                                                           │
│  3️⃣  렌더링 진화:                                          │
│     CSR(이케아) → SSR(랩포장) → RSC(붙박이 시공)             │
│     핵심: JS 전송량 0바이트 (Zero Bundle Size)              │
│                                                           │
│  4️⃣  하이브리드 모델:                                       │
│     기본 = 서버 컴포넌트 (무거운 것 담당)                    │
│     'use client' = 클라이언트 컴포넌트 (상호작용 담당)       │
│     → "보낼 필요 없는 코드는 보내지 않는다"                  │
│                                                           │
└─────────────────────────────────────────────────────────┘
