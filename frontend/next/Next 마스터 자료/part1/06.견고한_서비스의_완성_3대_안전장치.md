## 1. 3대 안전장치 한눈에 보기 🔴

```
사용자 경험을 지키는 3대 방어선:

┌─────────────┬──────────────────────────────────┬────────────────┐
│ Special File │ 역할                              │ 대응 상황       │
├─────────────┼──────────────────────────────────┼────────────────┤
│ not-found   │ 길 잃은 사용자를 위한 나침반        │ 404 에러        │
│ loading     │ 기다림을 디자인하는 스켈레톤        │ 데이터 로딩 중   │
│ error       │ 시스템 붕괴를 막는 최후의 방어선    │ 런타임 에러      │
└─────────────┴──────────────────────────────────┴────────────────┘
```

### 공통 규칙

| 규칙 | 설명 |
|------|------|
| 파일명 필수 | 반드시 소문자 `not-found.tsx`, `loading.tsx`, `error.tsx` |
| 계층 구조 | 에러 발생 지점에서 **가장 가까운** 파일이 우선 |
| 글로벌 vs 로컬 | `app/` 최상위 = 전체 적용, 특정 폴더 = 해당 섹션만 |

## 2. not-found.tsx — 404 에러 처리 🔴

### 2-1. 자동 작동: 존재하지 않는 경로

`app/not-found.tsx`를 만들면, 아무 경로나 쳐서 404가 발생할 때 **자동으로** 이 파일이 렌더링된다.

```tsx
// src/app/not-found.tsx
import Link from 'next/link';

export default function NotFound() {
  return (
    <div className="min-h-screen flex flex-col items-center justify-center">
      <h1 className="text-9xl font-extrabold text-indigo-200">404</h1>
      <h3 className="text-2xl font-bold mt-8">길을 잃으셨나요?</h3>
      <p className="text-gray-500 mt-2">요청하신 페이지를 찾을 수 없습니다.</p>
      <Link href="/" className="mt-8 px-8 py-3 bg-indigo-600 text-white rounded-lg">
        홈으로 돌아가기
      </Link>
    </div>
  );
}
```

### 2-2. 수동 호출: `notFound()` 함수

URL은 맞지만 **DB에 데이터가 없는 경우** 개발자가 직접 404를 트리거한다.

```tsx
// src/app/products/[id]/page.tsx
import { notFound } from 'next/navigation';

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const product = await getProduct(id);

  // 데이터 없으면 → 렌더링 즉시 중단 → 가장 가까운 not-found.tsx 표시
  if (!product) {
    notFound();
  }

  return <div>{product.name}</div>;
}
```

```
notFound() 동작 원리:

Step 1. notFound() 호출
Step 2. 현재 컴포넌트 렌더링 즉시 중단 (Throw)
Step 3. 상위 계층으로 올라가며 가장 가까운 not-found.tsx 찾기
Step 4. 해당 UI로 화면 대체 + HTTP 404 상태 코드 반환

→ 검색 엔진에도 "이 페이지는 없어요" 정확히 전달 (SEO 유지)
```

### 글로벌 vs 로컬

```
app/
├── not-found.tsx              ← 글로벌 (전체 앱에서 최종 낙하산)
└── products/
    └── not-found.tsx          ← 로컬 (products 섹션 전용 404)
```

> 💡 `/products/999` 접속 시 → `products/not-found.tsx` 우선 → 없으면 → `app/not-found.tsx`

## 3. loading.tsx — 로딩 UI & Streaming SSR 🔴

### 문제: 데이터 로딩 중 하얀 화면

```
❌ 기존 (Blocking SSR):

사용자 클릭 → [3초 하얀 화면] → 갑자기 페이지 등장
→ "인터넷 끊겼나?" → 이탈 😫

✅ Streaming SSR (loading.tsx):

사용자 클릭 → [즉시 로딩 UI 표시] → 데이터 도착 → 부드럽게 교체
→ "로딩 중이구나!" → 기다림 ✅
```

### 3-1. 글로벌 로딩 스피너

```tsx
// src/app/loading.tsx
export default function Loading() {
  return (
    <div className="fixed inset-0 flex items-center justify-center bg-white/80 backdrop-blur-sm z-50">
      <div className="flex flex-col items-center gap-4">
        <div className="w-16 h-16 border-4 border-indigo-600 border-t-transparent rounded-full animate-spin"></div>
        <p className="text-indigo-600 font-semibold animate-pulse">
          Loading amazing products...
        </p>
      </div>
    </div>
  );
}
```

### 3-2. 세그먼트 전용 스켈레톤 UI 🟡

실제 콘텐츠의 **뼈대(그림자)**를 미리 보여주는 현대적 방식.

```tsx
// src/app/products/loading.tsx — 글로벌 스피너 대신 이것이 우선!
export default function ProductsLoading() {
  return (
    <div className="py-12 px-8">
      {/* 제목 뼈대 */}
      <div className="h-10 w-48 bg-gray-200 rounded-lg mb-8 animate-pulse"></div>

      {/* 실제 레이아웃과 동일한 그리드 구조 → Layout Shift 방지 */}
      <div className="grid grid-cols-1 md:grid-cols-2 gap-6">
        {[...Array(4)].map((_, i) => (
          <div key={i} className="bg-white rounded-xl p-6">
            <div className="h-5 w-20 bg-gray-200 rounded-full mb-3 animate-pulse"></div>
            <div className="h-7 w-3/4 bg-gray-200 rounded mb-2 animate-pulse"></div>
            <div className="h-5 w-1/4 bg-gray-200 rounded animate-pulse"></div>
            <div className="h-10 bg-gray-200 rounded-lg mt-6 animate-pulse"></div>
          </div>
        ))}
      </div>
    </div>
  );
}
```

### Streaming SSR + Suspense 동작 원리 🔴

```
Next.js 내부 동작 (가상 코드):

<Layout>                          ← 즉시 전송! (레이아웃은 기다릴 필요 없음)
  <Suspense fallback={<Loading />}>  ← 데이터 안 왔으면 Loading 먼저 표시
    <ProductsPage />                 ← 3초 뒤 데이터 도착 → 스트림으로 교체
  </Suspense>
</Layout>
```

```
타임라인:

[0초] 클릭 → Layout + Loading UI 즉시 전송  ← 사용자: "로딩 중이구나"
[3초] 데이터 도착 → Loading을 치우고 실제 콘텐츠 스트림 전송
                    → 사용자: 화면이 부드럽게 채워짐!
```

> 💡 **fallback** = "플랜 B", "임시 대타 UI"
> 데이터가 준비되면 대타를 치우고 진짜 콘텐츠를 스트림으로 흘려보냄.

## 4. error.tsx — 에러 처리 & 자가 복구 🔴

### 4-1. 절대 규칙 2가지

| 규칙 | 이유 |
|------|------|
| 파일명: `error.tsx` | Next.js 약속 파일 (자동 인식) |
| **`'use client'` 필수** | 사용자가 "다시 시도" 버튼을 **클릭**해야 하므로 |

### 4-2. Props: `error`와 `reset`

| Prop | 역할 |
|------|------|
| `error` | 에러 상세 정보 (`message`, `digest`) |
| `reset` | 고장 난 page.tsx를 **새로고침 없이** 재실행하는 복구 함수 |

```tsx
// src/app/products/error.tsx
'use client';  // ⚠️ 절대 규칙!
import { useEffect } from 'react';

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    console.error('[Error 낚아챔]:', error);
  }, [error]);

  return (
    <div className="flex flex-col items-center justify-center gap-6 p-8">
      <div className="text-6xl animate-bounce">💣</div>
      <h2 className="text-2xl font-bold">문제가 발생했습니다.</h2>
      <p className="text-gray-600">일시적인 오류입니다. 잠시 후 다시 시도해 주세요.</p>

      {/* 보안용 digest 코드 (고객센터에 전달용) */}
      {error.digest && (
        <p className="text-xs text-gray-500 bg-gray-100 p-2 rounded font-mono">
          Error ID: {error.digest}
        </p>
      )}

      {/* 자가 복구 버튼 */}
      <button
        onClick={() => reset()}
        className="px-6 py-2.5 bg-red-600 text-white rounded-lg hover:bg-red-700"
      >
        다시 시도하기
      </button>
    </div>
  );
}
```

### 4-3. Error Boundary 샌드위치 구조 🟡

```
Layout (빵)           ← 에러 나도 멀쩡히 유지!
  └── Error Boundary (양상추) ← error.tsx가 파편 흡수
        └── Page (패티)       ← 여기서 에러 터짐 💥

→ 패티가 폭발해도 빵(사이드바, 네비게이션)은 무사!
→ 사용자: "상품 목록만 잠깐 문제구나" (앱 전체 다운 아님)
```

```
reset() 동작:

[에러 화면]
  └── "다시 시도" 클릭
       └── 브라우저 새로고침 ❌ (주소창 안 돌아감)
       └── page.tsx만 서버에 재요청 ✅
       └── 50% 확률로 성공 → 정상 화면 복구!
```

### 4-4. `digest` 보안 메커니즘 🟡

```
서버에서 에러 발생:
  "데이터베이스 비밀번호 오류: pwd123..." ← 민감 정보!

Next.js 보안 처리:
  브라우저에 보내는 것: digest = "38920492" (무작위 해시만)
  서버 터미널 로그: 진짜 에러 메시지 + digest 코드

개발자 추적 방법:
  1. 브라우저의 Error ID: 38920492 복사
  2. 서버 터미널에서 Ctrl+F로 검색
  3. 진짜 원인 확인!
```

## 5. global-error.tsx — 지구 멸망급 에러 대응 🟢

`error.tsx`는 자신의 **부모인 `layout.tsx`에서 터진 에러는 잡지 못한다.**
최상위 Root Layout이 깨지면 → `global-error.tsx`가 최후의 보루.

```tsx
// src/app/global-error.tsx
'use client';

export default function GlobalError({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    // 레이아웃이 붕괴되었으므로 html부터 직접 새로 그린다!
    <html lang="ko">
      <body className="flex flex-col items-center justify-center min-h-screen bg-red-50">
        <h2 className="text-3xl font-bold text-red-600 mb-4">
          치명적인 시스템 에러
        </h2>
        <button onClick={() => reset()} className="px-6 py-3 bg-red-600 text-white rounded-lg">
          서비스 전체 다시 로드
        </button>
      </body>
    </html>
  );
}
```

> ⚠️ `global-error.tsx`는 `<html>`, `<body>` 태그를 **직접 포함**해야 한다!
> (기존 레이아웃이 깨진 상태에서 렌더링되므로)

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js의 3대 Special Files(not-found, loading, error)의 역할은?**

> `not-found.tsx`는 존재하지 않는 경로(404)에 대한 커스텀 UI를 제공하며, `notFound()` 함수로 프로그래밍적 트리거도 가능합니다. `loading.tsx`는 데이터 로딩 중 Suspense fallback으로 자동 적용되어 스켈레톤 UI를 보여줍니다. `error.tsx`는 런타임 에러 발생 시 Error Boundary로 동작하며, `reset()` 함수로 새로고침 없이 복구를 시도할 수 있습니다. 세 파일 모두 **약속된 파일명**을 사용해야 하며 계층 구조에서 가장 가까운 파일이 우선 적용됩니다.

**Q2. Next.js의 Streaming SSR이란 무엇인가요?**

> 기존 Blocking SSR은 서버가 모든 데이터를 다 가져와 HTML을 **완전히 완성할 때까지** 브라우저에 아무것도 보내지 않았습니다. Streaming SSR은 레이아웃과 loading.tsx(fallback)를 **먼저 전송**하고, 데이터가 준비되면 해당 부분만 **스트림으로 추가 전송**하여 교체합니다. 내부적으로 React Suspense를 활용하며, 사용자가 하얀 화면 대신 즉시 로딩 UI를 볼 수 있어 체감 속도가 크게 향상됩니다.

**Q3. `error.tsx`가 반드시 클라이언트 컴포넌트여야 하는 이유는?**

> 에러 화면에서 사용자가 "다시 시도" 버튼을 **클릭(onClick)**하여 `reset()` 함수를 호출해야 하기 때문입니다. 이벤트 리스너와 상호작용은 브라우저(클라이언트)에서만 가능하므로, 반드시 파일 최상단에 `'use client'`를 선언해야 합니다.

**Q4. `error.tsx`의 `reset()` 함수는 어떻게 동작하나요?**

> `reset()`은 에러가 발생한 `page.tsx` 영역만 **브라우저 새로고침 없이 서버에 재요청**하여 다시 렌더링을 시도합니다. 부모인 `layout.tsx`(사이드바, 네비게이션 등)는 그대로 유지됩니다. 이는 React Error Boundary의 구조 덕분으로, Layout > Error Boundary > Page 샌드위치 형태에서 가장 안쪽(Page)만 교체하는 원리입니다.

**Q5. `notFound()` 함수와 `not-found.tsx` 파일의 관계는?**

> `not-found.tsx`는 404 UI를 정의하는 **약속된 파일**이고, `notFound()`는 이를 **프로그래밍적으로 트리거**하는 함수입니다. URL 경로 자체가 없으면 자동으로 `not-found.tsx`가 렌더링되지만, URL은 맞지만 DB에 데이터가 없는 경우 `notFound()`를 호출하여 렌더링을 즉시 중단하고 가장 가까운 `not-found.tsx`를 표시합니다. HTTP 404 상태 코드도 정확히 반환되어 SEO를 유지합니다.

### 🟡 개념 질문

**Q6. `error.tsx`가 `layout.tsx`의 에러를 잡지 못하는 이유와 해결책은?**

> Error Boundary 구조상 `layout.tsx`가 `error.tsx`의 **부모**이므로, 부모에서 발생한 에러를 자식이 잡을 수 없습니다. 최상위 Root Layout 에러에 대응하려면 `app/global-error.tsx`를 만들어야 하며, 기존 레이아웃이 깨진 상태에서 렌더링되므로 `<html>`과 `<body>` 태그를 직접 포함해야 합니다.

**Q7. 스켈레톤 UI를 만들 때 가장 중요한 점은?**

> 실제 콘텐츠의 **레이아웃 구조(그리드, 카드 크기)를 그대로 모사**해야 합니다. 로딩 UI에서 실제 콘텐츠로 교체될 때 화면이 위아래로 덜컹거리는 **Layout Shift**를 방지하여 사용자 경험을 유지하는 것이 핵심입니다.
