## 1. 하이드레이션(Hydration)이란? 🔴

### 비유: 컵라면 속 건조 채소

서버에서 갓 도착한 HTML은 **바짝 말린 건조 채소**와 같다.
모양은 파, 버섯, 고추인데 — 씹어 먹을 수도 없고 맛도 나지 않는다.

```
건조 채소(정적 HTML)     +     뜨거운 물(JavaScript)     =     생생한 채소(살아있는 앱)
┌──────────────┐         ┌──────────────┐            ┌──────────────────┐
│ 버튼은 보이는데│         │ 이벤트 리스너  │            │ 클릭하면 반응하고  │
│ 눌러도 반응 ❌ │   +     │ 상태 관리      │     =      │ 입력도 되고       │
│ 입력도 안 됨 ❌ │         │ 클릭 핸들러    │            │ 드롭다운도 열림 ✅ │
└──────────────┘         └──────────────┘            └──────────────────┘
```

> 💡 **Hydration** = "수분 공급"
> 바짝 마른 정적 HTML(건조 채소)에 자바스크립트(뜨거운 물)를 부어서,
> 클릭도 되고 입력도 되는 **살아있는 앱**으로 변신시키는 과정이다.

## 2. 서버와 브라우저의 릴레이 협동 🟡

1번 문서에서 배운 렌더링 방식과 연결해서 큰 그림을 그려보자.

```
┌─────────────────────────────────────────────────────────────┐
│                    렌더링 → 하이드레이션 전체 흐름             │
│                                                               │
│  [서버]                              [브라우저]                │
│    │                                    │                     │
│    │── 완성된 HTML 전송 ──────────────▶│                     │
│    │   (스테이크가 접시에 담겨 나옴)       │ 👀 눈으로 볼 수 있다│
│    │                                    │ ❌ 클릭은 안 된다    │
│    │                                    │                     │
│    │── JS 번들 전송 ─────────────────▶│                     │
│    │   (포크와 나이프 배달)              │                     │
│    │                                    │                     │
│    │                                    ▼ 하이드레이션!       │
│    │                                    │ ✅ 이벤트 리스너 장착│
│    │                                    │ ✅ 클릭 가능해짐     │
│    │                                    │ ✅ 살아있는 앱 완성  │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **비유**: 고급 스테이크 하우스
>
> 주방장(서버)이 스테이크(HTML)를 완벽하게 구워서 내어줌
> → 눈으로 볼 수 있다! 하지만 포크와 나이프(JS)가 아직 안 옴
> → 웨이터(브라우저)가 달려와 포크와 나이프를 쥐여줌 = **하이드레이션**

## 3. Next.js의 필살기 — 부분 하이드레이션 🟡

과거에는 '좋아요' 버튼 하나를 살리기 위해, 페이지 **전체**에 거대한 JS를 들이부어야 했다.
Next.js는 **부분 하이드레이션(Partial Hydration)**으로 이걸 해결한다.

### 비유: 똑똑한 스마트 스프링클러

```
마당에 물 주기

❌ 과거: 호스로 마당 전체에 물 뿌림 (돌멩이에도 물 줌)
   ┌─────────────────────────────────┐
   │  💧💧💧💧💧💧💧💧💧💧💧💧   │
   │  🪨 돌  🌸 꽃  🪨 돌  🌸 꽃   │  ← 전부 다 물줌 (비효율)
   └─────────────────────────────────┘

✅ Next.js: 꽃에만 정확히 물 쏘기
   ┌─────────────────────────────────┐
   │       💧          💧            │
   │  🪨 돌  🌸 꽃  🪨 돌  🌸 꽃   │  ← 꽃(use client)만 물줌
   └─────────────────────────────────┘
```

| 구분 | 하이드레이션 | 이유 |
|------|------------|------|
| 서버 컴포넌트 (🪨 돌멩이) | ❌ 안 함 | 영원히 변하지 않는 정적 콘텐츠 |
| 클라이언트 컴포넌트 (🌸 꽃) | ✅ 함 | `'use client'` 출입증이 붙은 것만 |

> ⚠️ 이 덕분에 Next.js 앱은 **획기적으로 가볍고 빠르다**.

## 4. Hydration Mismatch — 전 세계 개발자의 악몽 🔴

하이드레이션에는 아주 까다로운 규칙이 하나 있다:

> **"서버가 그린 HTML과 브라우저가 그리려는 HTML이 1픽셀도 틀리지 않고 완벽히 일치해야 한다!"**

일치하지 않으면? 리액트는 혼란에 빠져 기존 HTML을 **다 부수고 처음부터 다시 그린다** → 빨간 에러 발생.

### Mismatch를 유발하는 3대 실수

```jsx
/* ❌ Mismatch를 유발하는 전형적인 실수들 */
export default function TimeDisplay() {

  // 🚨 실수 1: 시간 (Time)
  // 서버 시간(10:00:00)과 브라우저 도착 시간(10:00:01)은 영원히 다르다!
  const time = new Date().toLocaleTimeString();

  // 🚨 실수 2: 랜덤 값 (Random)
  // 서버 주사위(0.12)와 브라우저 주사위(0.99)는 일치할 수 없다!
  const randomId = Math.random();

  return (
    <div id={randomId.toString()}>
      <p>{time}</p>

      {/* 🚨 실수 3: 잘못된 HTML 구조 */}
      <p>
        <div>Description</div>  {/* ← p 안에 div 불가 (HTML 문법 위반) */}
      </p>
    </div>
  );
}
```

| 실수 유형 | 원인 | 왜 불일치? |
|----------|------|-----------|
| 시간 (`new Date()`) | 서버와 브라우저의 실행 시점이 다름 | 서버: 10:00:00 / 브라우저: 10:00:01 |
| 랜덤 (`Math.random()`) | 매번 다른 값 생성 | 서버: 0.12 / 브라우저: 0.99 |
| 잘못된 HTML 중첩 | `<p>` 안에 `<div>` | 브라우저가 몰래 교정 → 리액트가 불일치 감지 |

> 🟢 **Next.js 15 기쁜 소식**: 과거엔 에러 메시지가 불친절했지만, v15부터 서버/브라우저 차이를 **'틀린 그림 찾기'처럼** 명확하게 비교해서 보여준다.

## 5. Mismatch 해결 — 두 가지 패턴 🔴

### 패턴 1: Two-Pass Rendering (강제 일치) — 정석

초기값을 **비워둬서** 서버와 브라우저의 첫 모습을 강제로 일치시키는 방법이다.

```jsx
/* ✅ 패턴 1: Two-Pass Rendering */
'use client';
import { useState, useEffect } from "react";

export default function SafeTimeDisplay() {
  // 초기값을 null로 → 서버: "없음!" / 브라우저 첫 모습: "없음!" → 일치!
  const [time, setTime] = useState<string | null>(null);

  useEffect(() => {
    // useEffect는 하이드레이션이 완전히 끝난 후, 오직 브라우저에서만 실행
    setTime(new Date().toLocaleTimeString());
  }, []);

  return (
    <div>
      <h1>현재 시각</h1>
      {time ? <p>{time}</p> : <p>Loading...</p>}
    </div>
  );
}
```

```
동작 흐름:

Step 1. 서버 렌더링     → time = null → "Loading..." 출력
Step 2. 브라우저 첫 렌더 → time = null → "Loading..." 출력  ← ✅ 일치!
Step 3. 하이드레이션 완료
Step 4. useEffect 실행  → setTime("10:30:05") → 실제 시간 표시
```

### 패턴 2: suppressHydrationWarning (치트키)

어쩔 수 없이 텍스트 차이가 날 때 사용하는 리액트 전용 치트키.

```jsx
/* ✅ 패턴 2: suppressHydrationWarning */
<p suppressHydrationWarning>
  {new Date().toLocaleTimeString()}
</p>
```

> ⚠️ **주의**: **텍스트**가 다를 때만 사용 가능.
> 태그 구조 자체가 다를 때 쓰면 앱이 망가진다!

### 두 패턴 비교

| 구분 | Two-Pass Rendering | suppressHydrationWarning |
|------|-------------------|-------------------------|
| 안전성 | ✅ 가장 안전한 정석 | ⚠️ 제한적 사용 |
| 원리 | 초기값 일치 → useEffect로 업데이트 | 불일치를 알면서 경고만 끔 |
| 사용 조건 | 모든 Mismatch 상황 | 텍스트 차이만 있을 때 |
| 사용 빈도 | 🔴 자주 사용 | 🟢 드물게 사용 |

## 6. Hard Navigation의 한계 — 철거 후 재건축 🟡

### 비유: 이사 = 포크레인 철거

```
Hard Navigation (기존 <a> 태그)

  현재 페이지               새 페이지
  ┌──────────┐             ┌──────────┐
  │  헤더     │  💥 파괴!   │  헤더     │  ← 같은 건데 또 그림
  │  콘텐츠   │ ──────────▶│  새 콘텐츠 │
  │  푸터     │  전부 폐기   │  푸터     │  ← 같은 건데 또 그림
  └──────────┘             └──────────┘

  사용자 경험: 화면 깜빡! 🫣 → 하얀 화면 → 새 페이지 로드
```

### Hard Navigation의 3대 페인 포인트

| 문제 | 설명 |
|------|------|
| 화면 깜빡임 (White Flash) | 집이 부서지고 새 집이 지어지는 순간 하얀 공백 발생 |
| 상태 소멸 (State Loss) | 작성 중인 댓글, 스크롤 위치, 사이드바 상태 등 전부 증발 |
| 비효율 | 공통 헤더/푸터까지 전부 다시 그림 |

```jsx
/* ❌ Hard Navigation — 파괴적인 이동 */
export default function NavBar() {
  return (
    <nav>
      {/* 클릭 = 새로고침 = 모든 상태 초기화 */}
      <a href="/dashboard">Dashboard (Hard)</a>
      <a href="/settings">Settings (Hard)</a>
    </nav>
  );
}
```

## 7. Soft Navigation — Next.js `<Link>`의 마법 🔴

### 비유: 부분 리모델링

```
Soft Navigation (Next.js <Link>)

  현재 페이지               이동 후
  ┌──────────┐             ┌──────────┐
  │  헤더     │  그대로 유지  │  헤더     │  ← 안 건드림
  │  콘텐츠   │ ──교체만──▶ │  새 콘텐츠 │  ← 이 부분만 바꿈!
  │  푸터     │  그대로 유지  │  푸터     │  ← 안 건드림
  └──────────┘             └──────────┘

  사용자 경험: 앱처럼 부드럽게 슝~ ✨ (깜빡임 0)
```

### 핵심: RSC Payload로 필요한 부분만 교체

```
React Router (SPA)           vs           Next.js Router
┌──────────────────┐                 ┌──────────────────┐
│ 서버가 보내는 것:  │                 │ 서버가 보내는 것:  │
│ JSON(목재+나사)   │                 │ RSC Payload       │
│                  │                 │ (완성된 가구)      │
├──────────────────┤                 ├──────────────────┤
│ 브라우저가 할 일:  │                 │ 브라우저가 할 일:  │
│ 직접 조립 🔨      │                 │ 제자리에 놓기만 📦 │
│ (클라이언트 연산 ⬆️)│                 │ (브라우저 부담 ⬇️) │
└──────────────────┘                 └──────────────────┘
```

```jsx
/* ✅ Soft Navigation — 부드러운 전환 */
import Link from "next/link";

export default function NavBar() {
  return (
    <nav>
      {/* 클릭 시 RSC Payload만 받아와 즉시 교체 */}
      {/* 새로고침 없음 → 스크롤, 영상 재생 상태 유지 */}
      <Link href="/dashboard">Dashboard (Soft)</Link>
      <Link href="/settings">Settings (Soft)</Link>
    </nav>
  );
}
```

### `<a>` vs `<Link>` 비교

| 구분 | `<a href>` | `<Link href>` |
|------|-----------|---------------|
| 이동 방식 | Hard Navigation | Soft Navigation |
| 새로고침 | ✅ 발생 (전체 리로드) | ❌ 없음 |
| 상태 유지 | ❌ 전부 소멸 | ✅ 유지 |
| 공통 UI | 다시 그림 | 그대로 유지 |
| 체감 속도 | 느림 (깜빡) | 빠름 (앱 같은 UX) |

## 8. Prefetching — 미리 가져오기 마법 🟡

`<Link>`에는 숨겨진 무기가 하나 더 있다: **프리패칭(Prefetching)**.

### 비유: 눈치 빠른 셰프의 예측 요리

```
사용자가 메뉴판의 '파스타'를 뚫어져라 쳐다봄

Step 1. 감지 (Viewport)
  └─ 링크가 화면에 보이는 순간 Next.js가 감지

Step 2. 다운로드 (Background)
  └─ 클릭도 하기 전에, 백그라운드에서 다음 페이지 데이터를 몰래 다운로드

Step 3. 즉시 전환 (Instant Transition)
  └─ 실제로 클릭했을 때, 이미 요리는 완성!
     네트워크 대기 시간 0초 → 화면 즉시 전환
```

```
Prefetching 타임라인

[링크가 화면에 보임]  ──(자동)──▶  [백그라운드 다운로드]
                                        │
                                        ▼
[사용자가 클릭!]  ──(0ms 대기)──▶  [즉시 화면 전환!] ⚡
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. 하이드레이션(Hydration)이란 무엇인가요?**

> 서버에서 렌더링된 정적 HTML에 자바스크립트를 연결하여 이벤트 리스너를 부착하고, 상호작용이 가능한 동적 애플리케이션으로 만드는 과정입니다. 서버가 보낸 HTML 구조를 재사용하면서 React가 이벤트 핸들러와 상태를 연결(attach)합니다.

**Q2. Hydration Mismatch란 무엇이고, 왜 발생하나요?**

> 서버에서 렌더링한 HTML과 클라이언트에서 렌더링하려는 HTML이 일치하지 않을 때 발생합니다. 대표적 원인으로는 `new Date()`, `Math.random()` 같은 **비결정적 값** 사용, 그리고 `<p>` 안에 `<div>`를 넣는 등의 **잘못된 HTML 중첩**이 있습니다. 발생 시 React가 기존 HTML을 폐기하고 처음부터 다시 렌더링하므로 성능에 치명적입니다.

**Q3. Hydration Mismatch를 어떻게 해결하나요?**

> 가장 안전한 방법은 **Two-Pass Rendering** 패턴입니다. 초기 상태를 `null`로 설정해 서버와 클라이언트의 첫 렌더링을 일치시키고, `useEffect`에서 클라이언트 전용 값을 설정합니다. 부득이하게 텍스트만 차이가 나는 경우에는 `suppressHydrationWarning` prop을 사용할 수 있습니다.

**Q4. Hard Navigation과 Soft Navigation의 차이는 무엇인가요?**

> **Hard Navigation**은 `<a>` 태그를 사용한 전통적 방식으로, 페이지 이동 시 브라우저가 전체 페이지를 새로고침하며 모든 상태가 초기화됩니다. **Soft Navigation**은 Next.js `<Link>` 컴포넌트를 통해 새로고침 없이 변경된 부분만 서버로부터 RSC Payload를 받아 교체하는 방식으로, 스크롤 위치나 컴포넌트 상태가 유지되어 앱 같은 사용자 경험을 제공합니다.

**Q5. Next.js에서 `<a>` 대신 `<Link>`를 써야 하는 이유는?**

> `<Link>`는 Soft Navigation으로 새로고침 없이 부분 교체만 하므로 상태가 유지되고 UX가 우수합니다. 또한 **Prefetching** 기능으로 링크가 뷰포트에 보이는 순간 다음 페이지를 미리 로드하여 클릭 시 즉시 전환됩니다. 공통 레이아웃(헤더/푸터)을 다시 렌더링하지 않아 성능도 좋습니다.

### 🟡 개념 질문

**Q6. 부분 하이드레이션(Partial Hydration)이란?**

> Next.js에서 서버 컴포넌트는 하이드레이션 대상에서 제외되고, `'use client'`가 선언된 클라이언트 컴포넌트만 선택적으로 하이드레이션하는 최적화 기법입니다. 페이지 전체에 JS를 보내지 않고 상호작용이 필요한 부분만 JS를 전송하므로 앱이 가벼워집니다.

**Q7. Prefetching이란 무엇이고 어떻게 동작하나요?**

> Next.js `<Link>` 컴포넌트가 사용자의 뷰포트(화면)에 들어오면, 클릭하기 전에 백그라운드에서 해당 페이지의 데이터를 미리 다운로드하는 기능입니다. 사용자가 실제로 클릭했을 때 이미 데이터가 준비되어 있으므로 네트워크 대기 시간 없이 즉시 화면이 전환됩니다.

│                                                                │
│  3️⃣  Hydration Mismatch 3대 원인:                               │
│     · new Date() (시간 차이)                                     │

│                                                                │
│  5️⃣  Prefetching: 링크가 화면에 보이면 미리 다운로드              │
