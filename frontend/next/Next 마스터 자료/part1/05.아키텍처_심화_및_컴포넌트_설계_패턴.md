## 1. Route Groups — URL에서 사라지는 '투명 망토' 🟡

### 문제: 마케팅 페이지와 대시보드의 레이아웃이 완전히 다르다

하나의 `layout.tsx`에서 조건문으로 분기하면 코드가 엉망이 된다.

### 해결: 소괄호 `()` = URL에 반영 안 되는 폴더

```
src/app/
├── (marketing)/           ← URL에서 사라짐!
│   ├── layout.tsx         ← 마케팅 전용 (화려한 상단 바)
│   └── page.tsx           ← 접속 주소: /
│
└── (app)/                 ← URL에서 사라짐!
    ├── layout.tsx         ← 앱 전용 (좌측 사이드바)
    └── dashboard/
        └── page.tsx       ← 접속 주소: /dashboard
```

```
URL 매핑:

(marketing)/page.tsx        →  /              (marketing은 무시)
(app)/dashboard/page.tsx    →  /dashboard     (app은 무시)
```

### 다중 루트 레이아웃 핵심 규칙 🟢

| 규칙 | 설명 |
|------|------|
| 최상위 `layout.tsx` 삭제 | 충돌 방지 (각 그룹이 독립 레이아웃) |
| 각 그룹에 `<html>`, `<body>` 포함 | 독립적인 루트 레이아웃 역할 |
| `globals.css` 직접 import | 최상위 없으므로 각 layout에서 직접 |

```tsx
// src/app/(marketing)/layout.tsx
import "@/app/globals.css";

export default function MarketingLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko" suppressHydrationWarning>
      <body className="bg-white">
        <header className="sticky top-0 bg-white/80 backdrop-blur-md border-b">
          <span className="text-xl font-bold text-indigo-600">MyService</span>
          <a href="/dashboard" className="px-5 py-2.5 bg-indigo-600 text-white rounded-full">
            Get Started
          </a>
        </header>
        <main>{children}</main>
        <footer>© 2026 MyService Inc.</footer>
      </body>
    </html>
  );
}
```

```tsx
// src/app/(app)/layout.tsx
import "@/app/globals.css";

export default function AppLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko" suppressHydrationWarning>
      <body className="bg-slate-50">
        <div className="flex h-screen">
          <aside className="w-72 bg-white border-r">
            <span className="text-lg font-bold">Admin Pro</span>
            <nav>{/* 사이드바 메뉴 */}</nav>
          </aside>
          <main className="flex-1 overflow-y-auto p-8">{children}</main>
        </div>
      </body>
    </html>
  );
}
```

## 2. Private Folders — 비밀 금고 `_folder` 🟡

폴더명 앞에 **언더바 `_`**를 붙이면 라우터에서 완전히 무시된다.

```
src/app/
├── _utils/              ← /_utils 접속 시 404!
│   └── date-formatter.ts
├── _components/         ← /_components 접속 시 404!
│   └── LikeButton.tsx
└── page.tsx             ← / 접속 정상
```

```tsx
// src/app/_utils/date-formatter.ts
export const formatDate = (date: Date): string => {
  return new Intl.DateTimeFormat('ko-KR', {
    year: 'numeric', month: 'long', day: 'numeric',
  }).format(date);
};
```

> 💡 **Colocation(공동 배치)**: 관련 컴포넌트/유틸리티를 페이지 옆에 두면서도 URL 노출 없이 안전하게 관리.

### Route Groups vs Private Folders 비교

| 구분 | Route Groups `()` | Private Folders `_` |
|------|-------------------|---------------------|
| 문법 | 소괄호 `(name)` | 언더바 `_name` |
| 용도 | 레이아웃 분리 (논리적 그룹핑) | 라우팅 제외 (보안/정리) |
| URL 영향 | 경로에서 제외 | 경로에서 제외 |
| `page.tsx` 가능? | ✅ 가능 | ❌ 불가 (라우트 자체 무시) |

## 3. Server vs Client 컴포넌트 🔴

### 핵심 원리

> **Next.js App Router에서 모든 컴포넌트는 기본적으로 서버 컴포넌트다.**

```
┌─────────────────────────────────────────────────────────┐
│  서버 컴포넌트 (기본값)          클라이언트 컴포넌트       │
│                                                           │
│  ┌─────────────┐               ┌─────────────┐          │
│  │ 🍳 주방      │               │ 🍽️ 손님 테이블│          │
│  │ (서버)       │               │ (브라우저)    │          │
│  │             │               │              │          │
│  │ · DB 직접 접근│               │ · onClick    │          │
│  │ · API 키 안전│               │ · useState   │          │
│  │ · async/await│               │ · useEffect  │          │
│  │ · JS 전송 0  │               │ · 브라우저 API │          │
│  └─────────────┘               └─────────────┘          │
│                                                           │
│  선언: 아무것도 안 씀             선언: 'use client'        │
└─────────────────────────────────────────────────────────┘
```

### 비유: 주방에서 끝내는 요리

```
서버 컴포넌트 = 주방(서버)에서 요리를 완벽하게 끝내서 완성된 접시(HTML)만 배달
  → 손님(브라우저)은 요리 도구(JS) 다운로드 불필요 → 빠르다!

클라이언트 컴포넌트 = 손님 테이블에서 직접 불을 켜고 조리
  → JS가 브라우저로 전송되어야 함 → 상호작용 가능!
```

### ❌ 서버 컴포넌트에서 발생하는 에러

```tsx
// src/app/counter/page.tsx — 'use client' 없음 = 서버 컴포넌트
import { useState } from 'react';  // ❌ 에러!

export default function CounterPage() {
  const [count, setCount] = useState(0);  // ❌ 서버에는 상태 메모리 없음!

  return (
    <button onClick={() => setCount(count + 1)}>  {/* ❌ 서버에 마우스 없음! */}
      Count: {count}
    </button>
  );
}
// 에러: "useState only works in Client Components"
```

### ✅ `'use client'`로 해결

```tsx
'use client';  // ← 파일 최상단에 선언 (import보다 위!)
import { useState } from 'react';

export default function CounterPage() {
  const [count, setCount] = useState(0);  // ✅ 이제 동작!

  return (
    <button onClick={() => setCount(count + 1)}>  {/* ✅ 클릭 가능! */}
      Count: {count}
    </button>
  );
}
```

### 언제 무엇을 써야 하나?

| 서버 컴포넌트 (기본값) | 클라이언트 컴포넌트 (`'use client'`) |
|----------------------|--------------------------------------|
| DB 직접 쿼리 / API 호출 | `onClick`, `onChange` 등 이벤트 리스너 |
| 민감 정보 (API 키, DB 비밀번호) | `useState`, `useEffect` 등 리액트 훅 |
| SEO 중요 콘텐츠 (블로그, 상품 설명) | `window`, `localStorage` 등 브라우저 API |
| 무거운 라이브러리 (결과만 전송) | 실시간 UI 상태 변경 |

## 4. 나뭇잎 컴포넌트 패턴 (Leaf Component) 🔴

> **에러 난다고 페이지 전체에 `'use client'` 붙이지 마라!**
> 상호작용이 필요한 **아주 작은 조각만** 따로 떼어내라.

### 비유: 나무 구조

```
🌳 컴포넌트 트리

  page.tsx (서버 - 줄기/가지)           ← 무겁고 튼튼
    ├── <h1>제목</h1> (서버)            ← JS 전송 0
    ├── <ProductList /> (서버)          ← DB 조회, JS 전송 0
    └── <LikeButton /> (클라이언트 - 🍃 나뭇잎)  ← 이것만 JS 전송!
```

```tsx
// src/app/_components/LikeButton.tsx — 나뭇잎 (최소 단위)
'use client';
import { useState } from 'react';

export default function LikeButton() {
  const [isLiked, setIsLiked] = useState(false);
  return (
    <button onClick={() => setIsLiked(!isLiked)}>
      {isLiked ? '❤️' : '🤍'}
    </button>
  );
}
```

```tsx
// src/app/products/page.tsx — 줄기 (서버 컴포넌트 유지!)
import LikeButton from '@/app/_components/LikeButton';

export default function ProductsPage() {
  // 서버에서 데이터 조회 (JS 전송 0)
  const products = [
    { id: 1, name: 'Premium Headphone', price: '$299' },
    { id: 2, name: 'Office Chair', price: '$150' },
  ];

  return (
    <div>
      <h1>Best Sellers</h1>  {/* 서버 렌더링 */}
      {products.map((p) => (
        <div key={p.id}>
          <h3>{p.name}</h3>    {/* 서버 렌더링 */}
          <p>{p.price}</p>     {/* 서버 렌더링 */}
          <LikeButton />       {/* 🍃 이것만 클라이언트! */}
        </div>
      ))}
    </div>
  );
}
```

> 💡 **결과**: 브라우저가 다운로드할 JS = LikeButton 코드만! 나머지는 서버에서 완성된 HTML.

## 5. Composition 패턴 — 조립의 함정 탈출법 🔴

### ❌ 절대 금지: 클라이언트가 서버를 직접 import

```tsx
'use client';
import ServerComponent from './ServerComponent';  // ⚠️ 오염!

export default function BadClient() {
  return <ServerComponent />;  // ServerComponent가 강제로 클라이언트로 변환됨!
}
```

```
'use client'의 전염성:

  ClientComponent.tsx ('use client')
    └── import ServerComponent  ← 서버의 성질을 잃음!
        └── import HeavyLibrary ← 무거운 코드가 브라우저로 전송!
        └── DB_PASSWORD         ← 보안 위협! 🚨
```

### ✅ 해결: children 주입 패턴 (액자와 사진)

> **"클라이언트는 구멍만 뚫어놓고, 조립은 서버(부모)가 담당한다"**

```tsx
// 1. ClientFrame.tsx — 액자 (클라이언트)
'use client';

export default function ClientFrame({ children }: { children: React.ReactNode }) {
  return (
    <div className="border-4 border-blue-500 p-4">
      <button onClick={() => alert('I am Client!')}>클라이언트 버튼</button>
      <div className="mt-4">
        {children}  {/* ← 누가 들어올지 모른 채 구멍만 비워둠 */}
      </div>
    </div>
  );
}
```

```tsx
// 2. page.tsx — 조립 공장 (서버)
import ClientFrame from '@/components/ClientFrame';
import HeavyServerList from '@/components/HeavyServerList';

export default function Page() {
  return (
    <main>
      <h1>서버와 클라이언트의 완벽한 결합</h1>
      <ClientFrame>
        <HeavyServerList />  {/* ← 서버에서 렌더링 완료 후 구멍에 주입 */}
      </ClientFrame>
    </main>
  );
}
```

```
렌더링 과정:

Step 1. 서버가 HeavyServerList를 완전히 HTML로 렌더링
Step 2. 서버가 ClientFrame의 children 구멍에 완성된 HTML을 끼워 넣음
Step 3. 브라우저로 전송

결과: HeavyServerList는 클라이언트 안에 있지만
      JS 번들에 포함 안 됨! 서버 컴포넌트 유지! ✅
```

### 확인 방법

| 테스트 | 기대 결과 |
|--------|----------|
| `HeavyServerList`에 `console.log('DB Fetching')` 추가 | VS Code **터미널**에 출력 (브라우저 콘솔 ❌) |
| 클라이언트에서 직접 import 후 네트워크 탭 확인 | JS 다운로드 용량 증가 (오염 확인) |
| children 패턴 적용 후 네트워크 탭 확인 | JS 다운로드 용량 최소 유지 |

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js App Router에서 서버 컴포넌트와 클라이언트 컴포넌트의 차이는?**

> App Router에서 **모든 컴포넌트는 기본적으로 서버 컴포넌트**입니다. 서버에서 렌더링이 완료되어 HTML만 브라우저로 전송되므로 JS 번들 크기가 0이고 SEO에 유리합니다. `useState`, `onClick` 등 브라우저 상호작용이 필요한 컴포넌트는 파일 최상단에 `'use client'`를 선언하여 **클라이언트 컴포넌트**로 만들어야 하며, 이 경우 해당 코드가 JS 번들에 포함되어 브라우저로 전송됩니다.

**Q2. `'use client'` 지시어의 정확한 의미는?**

> "서버 렌더링을 포기한다"는 뜻이 **아닙니다**. 서버에서 HTML 뼈대를 먼저 렌더링하고, 이후 브라우저에서 JS가 다운로드되어 이벤트 리스너를 부착하는 **하이드레이션**이 일어납니다. `'use client'`는 "이 파일부터 브라우저로 JS를 보내라"는 **클라이언트 경계선**을 긋는 지시어이며, 이 파일이 import하는 모든 하위 모듈도 클라이언트 번들에 포함됩니다.

**Q3. 클라이언트 컴포넌트 안에 서버 컴포넌트를 넣으려면 어떻게 해야 하나요?**

> 직접 `import`하면 서버 컴포넌트가 **강제로 클라이언트로 변환**되어 성능/보안 문제가 발생합니다. 대신 **Composition(합성) 패턴**을 사용합니다. 클라이언트 컴포넌트는 `children` prop으로 구멍만 뚫어두고, 서버 컴포넌트(부모)에서 조립하여 주입합니다. 이렇게 하면 서버 컴포넌트는 서버에서 렌더링이 완료된 HTML 상태로 전달되어 JS 번들에 포함되지 않습니다.

**Q4. 나뭇잎 컴포넌트 패턴이란?**

> 페이지 전체에 `'use client'`를 붙이는 대신, **상호작용이 필요한 최소 단위(버튼, 입력창 등)만** 별도 파일로 분리하여 클라이언트 컴포넌트로 만드는 설계 전략입니다. 나무의 줄기/가지(페이지, 레이아웃)는 서버 컴포넌트로 유지하고, 바람에 흔들리는 나뭇잎(좋아요 버튼 등)만 클라이언트로 만들어 **JS 번들 크기를 최소화**합니다.

### 🟡 개념 질문

**Q5. Route Groups란 무엇이고 왜 사용하나요?**

> 폴더명을 소괄호 `()`로 감싸면 URL 경로에서 제외됩니다. 이를 통해 마케팅 페이지와 관리자 대시보드처럼 성격이 다른 영역을 **독립적인 레이아웃**으로 분리할 수 있습니다. 각 그룹에 별도의 `layout.tsx`를 두어 완전히 다른 `<html>`, `<body>` 구조를 가질 수 있으며, URL은 깔끔하게 유지됩니다.

**Q6. Private Folders(`_`)는 어떤 역할을 하나요?**

> 폴더명 앞에 언더바 `_`를 붙이면 Next.js 라우터가 해당 폴더를 **완전히 무시**합니다. 브라우저에서 `/_utils` 같은 경로로 접근하면 404가 됩니다. 이를 통해 유틸리티 함수, 내부 컴포넌트, DB 로직 등을 페이지 옆에 안전하게 배치(Colocation)하면서도 URL 노출을 방지할 수 있습니다.

**Q7. `console.log`로 서버/클라이언트 컴포넌트를 구분하는 방법은?**

> 서버 컴포넌트에 `console.log`를 넣으면 **VS Code 터미널**(서버)에 출력되고, 클라이언트 컴포넌트(`'use client'`)에 넣으면 **브라우저 개발자 도구 콘솔**(F12)에 출력됩니다. 이 차이로 코드가 어디서 실행되는지 즉시 확인할 수 있습니다.

```
│     줄기(서버) → 가지(서버) → 🍃 나뭇잎(클라이언트)               │

│                                                                    │
│  5️⃣  Composition 패턴: 클라이언트가 서버를 품어야 할 때            │
│     ❌ 직접 import → 서버가 클라이언트로 오염                      │
│     ✅ children 구멍 → 서버(부모)가 조립하여 주입                   │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```
