## 1. 캐싱의 영원한 딜레마 🔴

```
데이터 신선도 (Freshness)  ←──── 트레이드오프 ────→  서버 성능 (Performance)

매번 DB 조회           ←→            캐시된 스냅샷 제공
100% 최신 데이터        ←→            0.001초 응답
서버 과부하 위험          ←→            썩은 데이터 위험
```

> 💡 **캐시 무효화(Cache Invalidation)**: 캐시에 얼린 데이터를 **언제 버리고 새 데이터를 가져올지** 결정하는 타이밍의 예술.
> 컴퓨터 과학에서 가장 어려운 문제 중 하나.

## 2. 렌더링 4대 전략과 실무 배치 🔴

| 전략 | 설명 | 실무 배치 |
|------|------|----------|
| **CSR** | 브라우저가 JS로 화면 그림 | 지도 드래그, 모달, 좋아요 애니메이션 |
| **SSR** | 매 요청마다 서버가 새로 그림 | 마이페이지, 장바구니, 주식 호가창 |
| **SSG** | 빌드 시 1번 구워서 영구 제공 | 회사 소개, 이용약관, 정적 블로그 |
| **ISR** | 일정 시간마다 백그라운드 갱신 | 쇼핑몰 상품 목록, 뉴스 메인 |

## 3. Next.js 15의 파괴적 변화: Opt-out → Opt-in 🔴

```
Next.js 14 (Opt-out):  기본 = 캐싱 ON → 개발자가 no-store로 거부
  → "왜 데이터가 안 바뀌지?" 유령 버그 유발

Next.js 15 (Opt-in):   기본 = 캐싱 OFF → 개발자가 force-cache로 허가
  → 안전한 출발 (항상 최신) + 정밀한 핀셋 통제
```

### 두 캐시 계층 이해 🟡

| 계층 | 대상 | 설명 |
|------|------|------|
| **Data Cache** | fetch 결과 (JSON) | Next.js 15: 기본 OFF |
| **Full Route Cache** | HTML 껍데기 전체 | 조건 충족 시 여전히 SSG로 박제! |

> ⚠️ Data Cache가 OFF여도, 동적 함수(`cookies`, `searchParams`, `no-store`)를 안 쓰면
> 컴파일러가 HTML 껍데기를 SSG로 영구 박제해버림!
> → **`cache: 'no-store'`를 명시적으로 선언**해야 동적 렌더링 강제!

## 4. 캐싱 3대 전략 실전 🔴

### Dynamic (매번 새로)

```tsx
const res = await fetch('https://timeapi.io/...', {
  cache: 'no-store'  // 컴파일러의 SSG 박제를 파훼!
});
```

### Static (영구 결빙)

```tsx
const res = await fetch('https://timeapi.io/...', {
  cache: 'force-cache'  // 빌드 시 1번 → 영구 캐싱
});
```

### ISR (시간 기반 갱신)

```tsx
const res = await fetch('https://timeapi.io/...', {
  next: { revalidate: 60 }  // 60초마다 백그라운드 갱신 (초 단위!)
});
```

### ISR의 SWR 패턴 동작 🔴

```
[0초]   최초 접속 → API 호출 → 캐시에 저장 + 60초 타이머 시작
[1~59초] 수만 명 접속 → 캐시만 제공 (API 호출 0!)
[60초]  캐시 만료 → 누군가 접속
        → 일단 낡은 데이터 즉시 제공 (대기 0초!)
        → 백그라운드에서 몰래 API 호출 → 캐시 교체
[다음]  새 접속자부터 최신 데이터 제공

핵심: 사용자에게 단 한 번의 로딩 지연도 없다!
```

> ⚠️ `revalidate` 단위는 **초(Seconds)**! `3000` = 3초가 아니라 **50분**!

## 5. 매크로 → 마이크로 캐싱 혁명 🟡

### 매크로 캐싱의 비극 (과거)

```
페이지 전체를 통째로 얼리거나 녹이는 투박한 방식

90% 정적 (상품 설명) + 10% 동적 (재고, 장바구니)
→ 재고 때문에 페이지 전체 동적 렌더링 = 서버 혹사
→ 페이지 전체 캐싱 = 재고 정보 부패
→ All or Nothing 딜레마!
```

### 마이크로 캐싱 (현재: `use cache`)

```
컴포넌트 단위로 쪼개서 각각 다른 캐싱 전략 적용!

<ProductPage>
  <LiveCart />              ← 동적 (매번 새로)
  <LiveStockCount />        ← 동적 (매번 새로)
  <HeavyProductDesc />      ← 'use cache' (영구 결빙, 0.001초)
</ProductPage>
```

```tsx
// HeavyProductDescription.tsx
'use cache';  // 이 컴포넌트만 독립적으로 캐시!

export default async function HeavyProductDescription() {
  const data = await db.product.getDescription();  // 최초 1회만 실행
  return <article>{data}</article>;
}
```

> 💡 `use cache`는 fetch 옵션이 아닌 **컴포넌트/함수 단위** 캐싱
> → Prisma, Axios 등 fetch 외 라이브러리도 캐싱 가능!

## 6. PPR + `<Suspense>` — 정적 껍데기 + 동적 스트리밍 🔴

### PPR (Partial Prerendering) = 정적 + 동적의 하이브리드

```
[T=0.001초] 정적 껍데기 즉시 전송 (제목, 배경, 푸터)
[T=0.010초] 동적 부분은 <Suspense> 구멍 + fallback(로딩 UI) 표시
[T=3.000초] 데이터 완성 → 구멍에 스트리밍으로 채워 넣기
            (브라우저 새로고침 없이!)
```

### 절대 규칙: 부모 껍데기에서 `await params` 금지!

```tsx
// page.tsx — 정적 껍데기 (await params 안 함!)
export default function ProductPage({ params }) {
  return (
    <div>
      <h1>상품 제목 (Static Shell)</h1>
      <Suspense fallback={<p>로딩 중...</p>}>
        <DynamicReview paramsPromise={params} />  {/* Promise 그대로 전달 */}
      </Suspense>
    </div>
  );
}

// DynamicReview.tsx — Suspense 안에서 안전하게 해제
export default async function DynamicReview({ paramsPromise }) {
  const { id } = await paramsPromise;  // 여기서 해제!
  // ...
}
```

### `cacheLife` — 캐시 수명 제어 🟡

```tsx
'use cache';
import { cacheLife } from 'next/cache';

export default async function ProductReview({ paramsPromise }) {
  const { id } = await paramsPromise;
  cacheLife('minutes');  // stale: 5분, revalidate: 1분

  const reviews = await db.review.findMany({ productId: id });
  return <ul>...</ul>;
}
```

## 7. 온디맨드 캐시 무효화 — 3가지 정밀 타격 도구 🔴

### 문제: 시간 기반 갱신만으로는 부족하다

```
악성 리뷰 삭제 → DB에서는 지워졌지만 → 캐시에 4분 남음!
→ 수백만 명이 욕설 리뷰를 계속 봄 😱
```

### 해결: 데이터 변경 즉시 캐시 강제 파괴

```tsx
import { revalidatePath, revalidateTag } from 'next/cache';

export async function deleteReviewAction(reviewId, productId) {
  await db.review.delete({ where: { id: reviewId } });

  // 방법 1: 경로 기반 (단일 페이지)
  revalidatePath(`/product/${productId}`);

  // 방법 2: 태그 기반 (여러 페이지에 흩어진 데이터)
  revalidateTag('review-list');
}
```

### 3가지 도구 비교

| 도구 | 대상 | 동작 | 실무 배치 |
|------|------|------|----------|
| `revalidatePath` | URL 경로 | 해당 경로 캐시 즉시 파괴 | 단일 페이지 갱신 |
| `revalidateTag` | 태그 이름 | "Stale" 마킹 → 백그라운드 갱신 | 공지 배너, 네비게이션 메뉴 |
| `updateTag` | 태그 이름 | 서버+브라우저 캐시 **즉시** 파괴 | 리뷰 삭제, 장바구니 담기 |

### `cacheTag`로 태그 부착 🟢

```tsx
'use cache';
import { cacheTag } from 'next/cache';

export default async function ReviewList({ productId }) {
  cacheTag(`reviews-${productId}`);  // 이 캐시에 태그 부착!
  const reviews = await db.review.findMany({ productId });
  return <ul>...</ul>;
}
```

```tsx
// 삭제 시 태그 기반 정밀 타격
export async function deleteReview(reviewId, productId) {
  await db.review.delete({ where: { id: reviewId } });
  revalidateTag(`reviews-${productId}`);  // 이 태그가 달린 모든 캐시 파괴!
}
```

> 💡 **Surrogate Keys 원리**: URL이 아닌 **데이터 자체**에 이름표를 달아 추적
> → 같은 데이터가 10개 페이지에 흩어져도 태그 하나로 전부 무효화!

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js 15의 Opt-in 캐싱이란?**

> Next.js 14까지는 기본적으로 모든 fetch를 캐싱(Opt-out)하여 "데이터가 안 바뀐다"는 버그가 잦았습니다. **15부터 기본값이 캐싱 OFF(Opt-in)**로 변경되어 항상 최신 데이터를 보장하며, 성능이 필요한 곳에만 개발자가 `force-cache`나 `revalidate`를 명시적으로 선언합니다. 단, Full Route Cache는 동적 함수가 없으면 여전히 SSG로 박제하므로 `no-store` 명시가 중요합니다.

**Q2. ISR의 SWR 패턴을 설명해주세요.**

> `next: { revalidate: 60 }`으로 설정하면: (1) 60초간 캐시 즉시 제공(API 호출 0), (2) 만료 후 접속 시 **낡은 캐시를 먼저 즉시 제공**(대기 0초), (3) 백그라운드에서 새 데이터 fetch → 캐시 교체, (4) 다음 접속자부터 최신 데이터. **Stale-While-Revalidate** 패턴으로 사용자에게 로딩 지연이 한 번도 발생하지 않습니다.

**Q3. PPR(Partial Prerendering)이란?**

> 하나의 페이지에서 **정적 껍데기(Static Shell)**는 빌드 시 구워두고, 동적 부분만 `<Suspense>` 구멍을 뚫어 스트리밍으로 채우는 하이브리드 기술입니다. 사용자는 0초 만에 껍데기를 보고, 무거운 데이터는 준비되는 대로 구멍에 흘려 넣습니다. 정적의 속도 + 동적의 신선도 = **스위트스팟** 달성.

**Q4. `revalidatePath`와 `revalidateTag`의 차이는?**

> `revalidatePath`는 **특정 URL 경로**의 캐시를 파괴합니다. 단일 페이지 갱신에 적합하지만, 같은 데이터가 여러 페이지에 있으면 모든 경로를 하드코딩해야 합니다. `revalidateTag`는 **데이터에 부착한 태그**를 기준으로 해당 태그가 달린 모든 캐시를 한 번에 무효화합니다. 데이터가 10개 페이지에 흩어져도 태그 하나로 전부 갱신 가능합니다.

**Q5. 매크로 캐싱과 마이크로 캐싱의 차이는?**

> **매크로 캐싱**(과거)은 페이지 전체를 통째로 캐싱하여, 10% 동적 영역 때문에 90% 정적 영역까지 희생하는 All-or-Nothing 딜레마가 있었습니다. **마이크로 캐싱**(Next.js 15+)은 `'use cache'` 지시어로 **컴포넌트 단위**로 캐싱을 쪼개, 정적 부품만 독립적으로 얼리고 동적 부품은 실시간으로 유지합니다. fetch 외 Prisma 등 ORM도 캐싱 가능합니다.

### 🟡 개념 질문

**Q6. `<Suspense>`가 PPR에서 필수인 이유는?**

> PPR 컴파일러는 빌드 시 정적 껍데기를 만들려 합니다. 이때 `await params` 같은 동적 데이터를 껍데기에서 직접 읽으면 빌드 에러가 발생합니다. `<Suspense>`로 감싸면 컴파일러에게 "이 안쪽은 빈 구멍으로 남겨둬라"고 지시하여 껍데기를 안전하게 보호합니다. 동시에 로딩 UI(fallback)를 제공하여 사용자 경험도 챙깁니다.

**Q7. `cacheLife('minutes')`의 3가지 생명주기는?**

> **stale**(5분): 브라우저가 서버에 묻지 않고 자체 캐시를 보여주는 시간. **revalidate**(1분): 이 시간 이후 접속 시 낡은 캐시를 먼저 주고 백그라운드에서 갱신(SWR). **expire**: 아무도 방문하지 않으면 캐시가 완전 폐기되는 시간. 컴포넌트마다 다른 TTL을 부여하여 정밀한 수명 관리가 가능합니다.
