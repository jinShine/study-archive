## 1. RESTful API — 시스템 간의 대화 규칙 🔴

### API란?

> 💡 **비유**: 관공서 창구
>
> 창구 직원에게 "대충 떼주세요" → 쫓겨남
> **정확한 표준 양식**에 맞춰 신청서 작성해야 처리됨
>
> API = 시스템 간 데이터를 주고받기 위한 **행정 규격 창구**

### REST를 완성하는 두 가지 원칙

```
REST = 명사(Resource) + 동사(HTTP Method)

❌ 과거 (안티 패턴) — 주소에 동사가 섞임
GET  /get-products-list
POST /create-product

✅ RESTful (정석) — 주소는 명사, 동작은 메서드에 위임
GET    /products       → 상품 목록 조회
POST   /products       → 새 상품 등록
PATCH  /products/1     → 1번 상품 일부 수정
DELETE /products/1     → 1번 상품 삭제
```

### HTTP 메서드 CRUD 매핑

| 메서드 | CRUD | 설명 | 실무 주의 |
|--------|------|------|----------|
| `GET` | Read | 데이터 조회 | Body 없음 |
| `POST` | Create | 데이터 생성 | Body에 새 데이터 |
| `PATCH` | Update | **일부** 수정 | 실무에서 선호 |
| `PUT` | Update | **전체** 덮어쓰기 | 다른 필드 날아갈 위험 |
| `DELETE` | Delete | 데이터 삭제 | |

### JSON — 기계들의 공용어

```json
{
  "id": 1,
  "name": "초경량 노트북",
  "price": 1200000,
  "inStock": true,
  "tags": ["electronics", "laptop"]
}
```

> ⚠️ JSON 문법 주의: 마지막 쉼표(Trailing Comma) ❌, 큰따옴표(`""`)만 사용 (작은따옴표 ❌)

## 2. Route Handlers 기초 — `route.ts`의 절대 규칙 🔴

### 핵심 규칙

| 규칙 | 설명 |
|------|------|
| 파일명 | 반드시 `route.ts` (page.tsx 아님!) |
| 함수명 | **대문자** `GET`, `POST`, `PATCH`, `DELETE` (소문자 → 무시됨) |
| **독점 규칙** | 같은 폴더에 `page.tsx`와 `route.ts` 공존 불가! |
| 위치 | 보통 `app/api/` 하위에 배치 |

```
src/app/
├── page.tsx              ← UI (화면)
└── api/
    └── products/
        └── route.ts      ← API (데이터) — page.tsx와 동시 불가!
```

### GET & POST 구현

```ts
// src/app/api/products/route.ts
import { NextResponse } from "next/server";

interface Product {
  id: number;
  name: string;
  price: number;
}

// 인메모리 DB
const products: Product[] = [
  { id: 1, name: "초경량 노트북", price: 1200000 },
  { id: 2, name: "무소음 키보드", price: 185000 },
];

// GET: 전체 조회
export async function GET() {
  return NextResponse.json(products);
}

// POST: 새 데이터 등록 + Validation
export async function POST(request: Request) {
  try {
    const body: Partial<Product> = await request.json();

    // 1차 방어선 (Validation)
    if (!body.name || !body.price) {
      return NextResponse.json(
        { error: "상품명 또는 가격이 누락되었습니다." },
        { status: 400 }
      );
    }

    const newProduct: Product = {
      id: products.length + 1,
      name: body.name,
      price: Number(body.price),
    };
    products.push(newProduct);

    return NextResponse.json(newProduct, { status: 201 });
  } catch (error) {
    return NextResponse.json(
      { error: "서버 내부 오류" },
      { status: 500 }
    );
  }
}
```

### HTTP 상태 코드 핵심

| 코드 | 의미 | 사용 |
|------|------|------|
| `200` | OK | 조회 성공 |
| `201` | Created | 생성 성공 |
| `400` | Bad Request | 클라이언트 측 잘못 (필수값 누락 등) |
| `404` | Not Found | 자원 없음 |
| `500` | Internal Server Error | 서버 측 오류 |

## 3. URI vs URL, Path Variable vs Query String 🟡

### 주소창 해부도

```
https://www.mystore.com/api/products/1?category=laptop&sort=price
|___|   |______________|  |___________|  |________________________|
Scheme     Host             Path           Query String
(통신규약) (물리적 위치)    (자원 식별)      (부가 조건)

└─────────────────────────────────────────────────────────────────┘
                    URI (통합 식별자 = 전체)
└──────────────────────┘
URL (위치 지정자 = 앞부분)
```

### Path Variable vs Query String

| 구분 | Path Variable | Query String |
|------|--------------|--------------|
| 문법 | `/products/[id]` | `/products?sort=price` |
| 용도 | **대상 확정** (What) | **조건 부여** (How) |
| 예시 | "1번 상품 가져와" | "가격순 정렬해" |
| 필수성 | 필수 (없으면 다른 리소스) | 선택 (없어도 기본 동작) |

### NextRequest & nextUrl로 Query String 추출

```ts
import { NextRequest } from "next/server";

export async function GET(request: NextRequest) {
  // nextUrl: 프레임워크가 URL을 미리 분해해둔 객체
  const searchParams = request.nextUrl.searchParams;
  const sortOption = searchParams.get("sort");  // "price"
  const category = searchParams.get("category"); // "laptop"
  // ...
}
```

## 4. 동적 라우트 핸들러 — `[id]`로 특정 데이터 타겟팅 🔴

```
src/app/api/products/[id]/route.ts

GET /api/products/1   → id = "1"
GET /api/products/999 → id = "999"
```

```ts
// src/app/api/products/[id]/route.ts
import { NextResponse } from "next/server";

export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }  // Next.js 15: Promise!
) {
  const { id } = await params;  // await 필수!

  const product = {
    id,
    name: `${id}번 상품`,
    price: 50000,
    description: "시스템에서 발행한 데이터입니다.",
  };

  return NextResponse.json(product);
}
```

### 풀스택 파이프라인: API → 서버 컴포넌트 → UI

```tsx
// src/app/products/[id]/page.tsx — 서버 컴포넌트가 API 호출
export default async function ProductDetailPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;

  // 서버에서 직접 자기 자신의 API 호출
  const response = await fetch(`http://localhost:3000/api/products/${id}`, {
    cache: "no-store",  // 항상 최신 데이터
  });
  const product = await response.json();

  return (
    <div>
      <h1>{product.name}</h1>
      <p>{product.price.toLocaleString()}원</p>
    </div>
  );
}
```

## 6. Memoization과 HTTP 캐시 제어 🟡

### 6-1. Request Memoization — 렌더링 내 중복 제거

```
하나의 페이지 렌더링 사이클 안에서:

Call 1: fetch("https://api.github.com/repos/vercel/next.js")
  → 🔴 Cache Miss → 실제 네트워크 요청 → 결과 메모리 저장

Call 2: fetch("https://api.github.com/repos/vercel/next.js")
  → 🟢 Cache Hit → 네트워크 안 감 → 메모리에서 0.001초 반환

렌더링 끝나면 메모리 폐기! (다음 사용자는 다시 처음부터)
```

### 6-2. HTTP Cache-Control — 글로벌 캐시

```ts
export async function GET() {
  const data = { message: "캐시 통제 데이터" };

  return Response.json(data, {
    headers: {
      "Cache-Control": "s-maxage=60, stale-while-revalidate=30",
    },
  });
}
```

```
Stale-While-Revalidate 타임라인:

[0초]   최초 접속 → 원본 서버에서 데이터 생성 → CDN에 60초간 보관
[30초]  재접속 → CDN이 캐시에서 즉시 제공 (원본 서버 안 감)
[70초]  유통기한 지남 → 일단 낡은 데이터 즉시 제공 + 백그라운드 갱신
[100초] 유통기한 + 유예기간 모두 끝남 → 완전 폐기 → 다시 원본 요청
```

### 6-3. Segment Config — fetch 없이 캐싱 🟢

```ts
// src/app/api/static-github/route.ts
export const revalidate = 60;  // 파일 전체 60초 캐싱

export async function GET() {
  // Prisma, Axios 등 fetch 외 라이브러리도 이 설정에 의해 캐싱됨
  const res = await fetch("https://api.github.com/repos/vercel/next.js");
  const data = await res.json();

  return NextResponse.json({
    stars: data.stargazers_count,
    generatedAt: new Date().toISOString(),
  });
}
```

### Next.js 15 캐싱 변화 (중요!)

```
Next.js 14: fetch 기본값 = 캐싱 ON (혼란 유발)
Next.js 15: fetch 기본값 = 캐싱 OFF (항상 최신)

→ 캐싱이 필요하면 개발자가 명시적으로 설정해야 함!
```

## 7. Webhook 수신 & 풀스택 파이프라인 실전 🟡

### Webhook = 외부 이벤트 수신소

```ts
// src/app/api/webhooks/stars/route.ts
import { NextRequest, NextResponse } from "next/server";

interface StarEvent {
  sender: { login: string; avatar_url: string };
  repository: { full_name: string };
  starred_at?: string;
}

let starEvents: StarEvent[] = [];

// POST: 외부에서 이벤트 수신
export async function POST(request: NextRequest) {
  try {
    const body: StarEvent = await request.json();

    if (!body.sender?.login || !body.repository?.full_name) {
      return NextResponse.json({ error: "유효하지 않은 데이터" }, { status: 400 });
    }

    body.starred_at = new Date().toISOString();
    starEvents = [body, ...starEvents].slice(0, 5);  // 메모리 방어 (최대 5개)

    return NextResponse.json({ message: "기록 완료" }, { status: 201 });
  } catch (error) {
    return NextResponse.json({ error: "서버 내부 오류" }, { status: 500 });
  }
}

// GET: 저장된 이벤트 조회
export async function GET() {
  return NextResponse.json({
    events: starEvents,
    updatedAt: new Date().toISOString(),
  });
}
```

### 서버 컴포넌트에서 실시간 렌더링

```tsx
// src/app/notifications/page.tsx
export default async function NotificationsPage() {
  const response = await fetch("http://localhost:3000/api/webhooks/stars", {
    cache: "no-store",  // 항상 최신 데이터
  });
  const data = await response.json();

  return (
    <div>
      <h1>실시간 알림 피드</h1>
      {data.events.length === 0 ? (
        <p>아직 수신된 알림이 없습니다.</p>
      ) : (
        <ul>
          {data.events.map((event, i) => (
            <li key={i}>
              <strong>{event.sender.login}</strong>님이
              <span> {event.repository.full_name}</span>에 별을 눌렀습니다!
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

```
풀스택 파이프라인 흐름:

[Postman/외부] ──POST──▶ [Route Handler] ──저장──▶ [인메모리 DB]
                                                        │
[브라우저] ──새로고침──▶ [서버 컴포넌트] ──GET──▶ [Route Handler] ──조회──▶ [인메모리 DB]
                              │
                              ▼
                        [완성된 HTML 전송]
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. RESTful API란 무엇인가요?**

> REST(Representational State Transfer)는 API를 설계하는 아키텍처 스타일입니다. 핵심 원칙은 두 가지입니다: URL은 제어하려는 **자원(Resource)을 명사로** 표현하고(`/products`), 조회/생성/수정/삭제 등의 동작은 **HTTP 메서드(GET/POST/PATCH/DELETE)**에 위임합니다. 주소에 동사를 포함하는 `/get-products` 같은 방식은 안티 패턴이며, `/products`라는 명사에 메서드로 의미를 부여하는 것이 표준입니다.

**Q2. Next.js의 Route Handlers란 무엇이고, 어떤 규칙이 있나요?**

> `route.ts` 파일을 통해 Next.js 프로젝트 내에 **독립 API 서버**를 구축하는 기능입니다. 핵심 규칙: (1) 함수명은 반드시 **대문자** `GET`, `POST` 등이어야 하며, (2) 같은 폴더에 `page.tsx`와 `route.ts`가 **공존할 수 없습니다**(독점 규칙). (3) Next.js 15에서 동적 라우트의 `params`는 `Promise` 타입이므로 `await`로 처리해야 합니다.

**Q3. NextRequest와 NextResponse의 역할은?**

> **NextRequest**는 웹 표준 `Request`를 Next.js에 맞게 확장한 객체로, `nextUrl.searchParams`를 통해 쿼리 스트링을 우아하게 추출할 수 있습니다. **NextResponse**는 응답 객체로, `.json()` 메서드로 JSON 응답과 상태 코드를 한 번에 제어합니다. 쿠키, 리다이렉트, 커스텀 헤더 주입까지 가능한 통합 응답 도구입니다.

**Q4. Path Variable과 Query String의 차이는?**

> **Path Variable**(`/products/[id]`)은 특정 리소스를 **확정(What)**할 때 사용하며 필수값입니다. **Query String**(`?sort=price&category=laptop`)은 리소스를 **어떻게 가공할지(How)** 조건을 부여하는 선택적 옵션입니다. RESTful 설계에서 "1번 상품을 가져와"는 Path Variable, "가격 낮은 순으로 정렬해"는 Query String입니다.

**Q5. API에서 Validation(검증)이 중요한 이유는?**

> 클라이언트가 보내는 데이터는 항상 신뢰할 수 없습니다. 필수 필드 누락, 잘못된 타입, 악의적 데이터 주입 등을 **서버 진입점에서 즉시 차단**해야 시스템 무결성이 보장됩니다. 검증 실패 시 `400 Bad Request`를 반환하여 클라이언트 측 잘못임을 명확히 하고, 불량 데이터가 DB에 오염되는 것을 방지합니다.

### 🟡 개념 질문

**Q6. Request Memoization과 HTTP 캐시의 차이는?**

> **Request Memoization**은 하나의 페이지 렌더링 사이클 내에서 동일 URL의 `fetch` 중복을 제거하는 **짧은 임시 메모리**입니다. 렌더링이 끝나면 폐기됩니다. **HTTP 캐시(Cache-Control)**는 CDN 등 글로벌 네트워크에 데이터를 보관하는 **영구적 기억 장치**로, `s-maxage`와 `stale-while-revalidate`로 유통기한을 제어합니다. Memoization은 서버 내부 최적화, HTTP 캐시는 네트워크 전체 최적화입니다.

**Q7. Next.js 15에서 fetch 캐싱 기본값이 변경된 이유는?**

> Next.js 14까지는 모든 fetch를 자동 캐싱하여 "데이터가 안 바뀐다"는 혼란이 잦았습니다. **15부터 기본값이 `no-store`(캐싱 OFF)**로 변경되어 항상 최신 데이터를 보장합니다. 캐싱이 필요하면 `force-cache`, `next: { revalidate: N }`, 또는 `export const revalidate`로 개발자가 **명시적으로** 설정해야 합니다.
