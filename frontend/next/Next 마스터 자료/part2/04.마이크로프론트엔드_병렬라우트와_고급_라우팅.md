## 1. 모놀리식의 비극 → 모듈러 아키텍처 🟡

### 과거: 모든 UI가 하나의 덩어리

```
모놀리식(Monolithic) 아키텍처의 2대 재앙:

재앙 1. 렌더링 병목 (White Screen)
  → 그래프 로딩 5초 → 관련 없는 공지사항까지 5초 대기

재앙 2. 단일 장애점 (SPOF)
  → 피드 영역 에러 1개 → 그래프 + 공지사항까지 전체 붕괴
```

### 현재: 독립된 슬롯으로 분할

```
모듈러 아키텍처 = 화면을 독립 구역(Slot)으로 쪼개기

┌────────────────┬──────────────┐
│                │  @feed       │
│  @graph        │  (에러 발생!) │  ← 격리됨! 다른 곳 무사
│  (5초 지연...) │──────────────│
│                │  @notice     │
│                │  (0.001초!)  │  ← 먼저 준비된 것부터 표시
└────────────────┴──────────────┘

+ Streaming + PPR 엔진 결합 → 준비된 부분부터 즉시 렌더링
```

## 2. 병렬 라우트 (Parallel Routes) — `@folder` 🔴

### 핵심: 폴더명 앞에 `@` 붙이면 독립 슬롯

```
src/app/dashboard/
├── @analytics/        ← 📊 슬롯 A (URL에 반영 안 됨!)
│   ├── page.tsx
│   └── default.tsx    ← 필수! 새로고침 시 안전장치
├── @team/             ← 👥 슬롯 B
│   ├── page.tsx
│   └── default.tsx
├── layout.tsx         ← 조립 공장
└── page.tsx           ← children 슬롯
```

### layout.tsx에서 슬롯 조립

```tsx
// src/app/dashboard/layout.tsx
export default function DashboardLayout({
  children,   // page.tsx
  analytics,  // @analytics 폴더
  team,       // @team 폴더 (골뱅이 뺀 이름이 prop!)
}: {
  children: React.ReactNode;
  analytics: React.ReactNode;
  team: React.ReactNode;
}) {
  return (
    <div className="grid grid-cols-12 gap-6">
      <div className="col-span-8">{children}</div>
      <div className="col-span-4">
        <div>{analytics}</div>
        <div>{team}</div>
      </div>
    </div>
  );
}
```

### `default.tsx` — 새로고침 시 최후의 보루

```tsx
// src/app/dashboard/@analytics/default.tsx
export default function AnalyticsDefault() {
  return <p>기본 통계 (default.tsx 가동)</p>;
}
```

> ⚠️ **Hard Navigation(새로고침) 시** 프레임워크가 슬롯에 매칭할 페이지를 못 찾으면 404!
> → `default.tsx`가 플랜 B로 렌더링되어 붕괴 방지.
> → **병렬 라우트 설계 시 default.tsx는 필수 의무!**

## 3. State vs URL 딜레마 🔴

### 모달을 `useState`로만 관리하면?

```
❌ 3대 UX 재앙:

1. 공유 단절: URL이 안 바뀌므로 친구에게 공유 불가
2. 새로고침 증발: F5 누르면 모달 사라짐
3. 뒤로가기 탈주: 모달 닫으려고 뒤로가기 → 사이트 밖으로 튕김!
```

### URL로 해결하려면?

```
URL을 /photo/123으로 변경하면 → 공유 가능! ✅
하지만 → Hard Navigation 발생 → 배경(피드) 전체 파괴 😱
```

> 💡 **아키텍트의 목표**: 배경 맥락 유지 + URL 동기화 = 동시 달성

## 4. 인터셉팅 라우트 (Intercepting Routes) — `(..)` 🔴

### 핵심: 라우팅 흐름을 중간에서 낚아채기

```
사용자가 /photo/1로 이동하려고 클릭
→ (..)photo 함정 발견!
→ 원래 목적지 대신, 현재 화면 위에 가로챈 UI만 덮어씌움
→ URL은 /photo/1로 변경 (웹의 진리 유지!)
→ 배경(피드)은 그대로 유지!
```

### 문법: 상대 경로처럼 동작

| 문법 | 의미 |
|------|------|
| `(.)folder` | 같은 레벨 가로채기 |
| `(..)folder` | **한 단계 위** 레벨 가로채기 |
| `(..)(..)folder` | 두 단계 위 |
| `(...)folder` | 루트(app) 레벨 |

### 두 개의 얼굴 (렌더링 분기)

```
Soft Navigation (내부 클릭):
  → (..)photo 함정 발동 → 피드 위에 모달 팝업

Hard Navigation (새로고침/외부 접속):
  → 함정 무시 → 원본 photo/[id]/page.tsx 전체 화면
```

## 5. 인스타그램 모달 아키텍처 — @ + (..) 융합 🔴

### 최종 설계도

```
src/app/
├── photo/
│   └── [id]/
│       └── page.tsx          ← 🧱 Hard Nav: 단독 전체 화면
└── feed/
    ├── layout.tsx            ← 🏗️ 1층(피드) + 2층(모달) 조립
    ├── page.tsx              ← 📄 1층: 피드 목록
    └── @modal/               ← 🛡️ 2층: 모달 전용 슬롯
        ├── default.tsx       ← 🛑 평소엔 null (투명)
        └── (..)photo/        ← 🪤 Soft Nav: 가로채기 함정!
            └── [id]/
                └── page.tsx  ← ✨ 팝업 모달 UI
```

### 동작 흐름

```
[Soft Nav] 피드에서 사진 클릭
  → (..)photo 함정 발동
  → @modal 슬롯에 팝업 모달 주입
  → 1층(피드) 유지 + 2층(모달) 떠오름
  → URL: /photo/1 (동기화!)

[Hard Nav] F5 또는 외부 링크 접속
  → 함정 무시
  → photo/[id]/page.tsx 전체 화면 렌더링
  → 피드 없이 독립 페이지
```

### 핵심 코드

```tsx
// feed/layout.tsx — 3D 조립 공장
export default function FeedLayout({ children, modal }) {
  return (
    <div className="relative">
      <main>{children}</main>  {/* 1층: 피드 */}
      {modal}                   {/* 2층: 평소 null, 클릭 시 모달 */}
    </div>
  );
}

// feed/@modal/default.tsx — 투명 방어막
export default function ModalDefault() {
  return null;  // 평소엔 2층이 보이지 않음
}

// feed/@modal/(..)photo/[id]/page.tsx — 가로채기 모달
export default function PhotoModal({ params }) {
  return (
    <div className="fixed inset-0 bg-black/80 flex items-center justify-center">
      <div className="bg-white rounded-2xl p-8">
        <CloseModalButton />
        <Suspense fallback={<p>로딩 중...</p>}>
          <ModalDetail paramsPromise={params} />
        </Suspense>
      </div>
    </div>
  );
}
```

## 6. History Stack과 모달 닫기 전략 🔴

### 브라우저 히스토리 = 접시 쌓기 (LIFO)

```
피드 접속      → 스택: [/feed]
사진 클릭      → 스택: [/feed, /photo/1]   (pushState)
```

### ❌ 잘못된 닫기: `router.push('/feed')`

```
스택: [/feed, /photo/1, /feed]  ← 접시 3개!
→ 뒤로가기 누르면 모달이 좀비처럼 부활! (무한 루프)
```

### ✅ 올바른 닫기: `router.back()`

```
스택: [/feed]  ← 맨 위 접시만 제거 (Pop)
→ 깔끔하게 피드로 복귀!
```

### 두 갈래 종료 전략

| 상황 | 스택 상태 | 닫기 전략 |
|------|----------|----------|
| **Soft Nav** (모달) | `[/feed, /photo/1]` | `router.back()` — 접시 제거 |
| **Hard Nav** (단독 페이지) | `[/photo/1]` 뿐 | `<Link href="/feed">` — 새 접시 추가 |

```tsx
// 모달 닫기 버튼 (Soft Nav 전용)
'use client';
import { useRouter } from 'next/navigation';

export default function CloseModalButton() {
  const router = useRouter();
  return (
    <button onClick={() => router.back()}>✕ 닫기</button>
  );
}

// 단독 페이지 귀환 버튼 (Hard Nav 전용)
import Link from 'next/link';

export default function StandardPhotoPage() {
  return (
    <Link href="/feed">피드로 돌아가기</Link>
    // router.back() 쓰면 사이트 밖으로 튕김!
  );
}
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Next.js의 병렬 라우트(Parallel Routes)란?**

> 폴더명 앞에 `@`를 붙여 하나의 URL에서 여러 개의 **독립 슬롯(Slot)**을 동시에 렌더링하는 기술입니다. `layout.tsx`에서 각 슬롯을 props로 받아 화면의 원하는 위치에 배치합니다. 슬롯 간 완전히 독립적이므로, 한 슬롯이 지연되거나 에러가 나도 다른 슬롯에 영향을 주지 않는 **장애 격리(Fault Isolation)**가 가능합니다. 새로고침 시 매칭 실패를 방지하기 위해 `default.tsx`가 필수입니다.

**Q2. 인터셉팅 라우트(Intercepting Routes)란?**

> `(..)` 문법으로 사용자의 라우팅 흐름을 **중간에서 가로채어** 원래 목적지 대신 현재 화면 위에 다른 UI를 덮어씌우는 기술입니다. URL은 목적지로 정확히 변경되지만, 배경 화면의 맥락은 유지됩니다. 내부 클릭(Soft Navigation)에서만 작동하고, 새로고침(Hard Navigation)에서는 무시되어 원본 페이지가 렌더링됩니다. 이 **렌더링 분기** 덕분에 공유 가능한 URL + 맥락 유지를 동시에 달성합니다.

**Q3. 인스타그램 모달 패턴에서 `@`와 `(..)`를 함께 쓰는 이유는?**

> `@modal` 병렬 라우트로 모달이 뜰 **전용 슬롯**을 확보하고, 그 안에 `(..)photo` 인터셉팅 라우트를 설치하여 클릭 시 **피드 배경을 유지한 채 모달만 팝업**시킵니다. `default.tsx`로 평소엔 슬롯을 투명하게 유지하고, 새로고침 시에는 `photo/[id]/page.tsx` 원본이 전체 화면으로 렌더링되는 **완벽한 렌더링 분기**가 달성됩니다.

**Q4. 모달 닫기에서 `router.push`가 아닌 `router.back()`을 써야 하는 이유는?**

> 히스토리 스택은 **LIFO(후입선출)** 구조입니다. `router.push('/feed')`를 쓰면 스택에 새 접시가 추가되어 `[/feed, /photo/1, /feed]`가 됩니다. 이후 뒤로가기를 누르면 모달이 다시 나타나는 **무한 루프**가 발생합니다. `router.back()`은 맨 위 접시만 제거(Pop)하여 안전하게 이전 화면으로 복귀합니다. 단, Hard Navigation으로 접속한 단독 페이지에서는 스택 바닥이 보장되지 않으므로 `<Link href="/feed">`로 **명시적 귀환 경로**를 제공해야 합니다.

### 🟡 개념 질문

**Q5. `default.tsx`의 역할과 필수인 이유는?**

> 병렬 라우트에서 Hard Navigation(새로고침) 시, 프레임워크가 각 `@슬롯` 내부에서 현재 URL에 매칭되는 `page.tsx`를 찾지 못하면 **404 에러로 전체 화면이 붕괴**합니다. `default.tsx`는 이때 **플랜 B**로 렌더링되어 시스템을 안전하게 유지합니다. 모달 슬롯의 경우 `return null`로 투명 상태를 유지하여 평소에는 보이지 않게 합니다.

**Q6. Soft Navigation과 Hard Navigation의 차이는?**

> **Soft Navigation**은 `<Link>` 클릭 시 브라우저가 서버에 새 HTML을 요청하지 않고, 프론트엔드 라우터가 필요한 컴포넌트만 교체하는 방식입니다. 기존 화면의 State와 DOM이 메모리에 보존됩니다. **Hard Navigation**은 새로고침이나 URL 직접 입력으로 브라우저가 모든 기억을 초기화하고 서버에 완전히 새로운 HTML을 요청하는 방식입니다. 인터셉팅 라우트는 Soft에서만 작동하고 Hard에서는 원본 페이지가 렌더링됩니다.
