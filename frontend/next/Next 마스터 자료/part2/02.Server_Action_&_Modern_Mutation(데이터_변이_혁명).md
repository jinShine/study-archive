## 1. RPC의 부활 — Server Actions의 뿌리 🟡

### 비유: 마법의 전화

> 서버에 있는 함수를 **내 컴퓨터의 함수처럼** 직접 호출하는 기술 = RPC

```
[ 클라이언트 ]                          [ 서버 ]
     │                                     ▲
     │ saveUser("Alice") 호출              │ 실제 saveUser 실행
     ▼                                     │
[ Client Stub ]                      [ Server Skeleton ]
(숨겨진 대리자)                      (숨겨진 수신자)
     │                                     ▲
     │ 데이터 포장 & 직렬화                 │ 데이터 해체
     ▼                                     │
═══════════════ [ Network ] ═══════════════
```

### 웹 Mutation의 진화 🟢

```
Phase 1 (1990s): HTML <form> 직접 전송
  ✅ 직관적    ❌ 페이지 전체 새로고침 (깜빡임)

Phase 2 (2010s): SPA + AJAX (fetch + preventDefault)
  ✅ 깜빡임 없음  ❌ 보일러플레이트 폭발 (주객전도)

Phase 3 (2024~): Server Actions (RPC 부활)
  ✅ 직관적 + 깜빡임 없음 + 보일러플레이트 0
```

## 2. Server Actions — `'use server'`의 마법 🔴

### 핵심: API Route 없이 서버 함수를 직접 호출

```tsx
// actions.ts (서버)
'use server';  // 이 파일의 모든 코드는 서버에서만 실행됨 (브라우저 번들 제외!)

export async function createProductAction(formData: FormData) {
  const title = formData.get("title") as string;
  const price = Number(formData.get("price"));

  if (!title || price <= 0) {
    throw new Error("상품명과 올바른 가격을 입력해주세요.");
  }

  console.log(`💾 [DB 저장] ${title}, ${price}원`);
  return { success: true, message: "등록 완료" };
}
```

```tsx
// page.tsx (클라이언트)
import { createProductAction } from "./actions";

export default function Page() {
  return (
    <form action={createProductAction}>  {/* 서버 함수를 form에 직접 연결! */}
      <input type="text" name="title" />
      <input type="number" name="price" />
      <button type="submit">등록</button>
    </form>
  );
}
```

### Under the Hood — 프레임워크가 하는 일

```
빌드 시: 'use server' 함수 → 고유 해시 ID 부여 → 숨겨진 엔드포인트 자동 생성
실행 시: 버튼 클릭 → React가 브라우저 POST를 가로챔
        → Next-Action: $action_xyz123 헤더로 은밀한 RPC 통신
        → 서버에서 함수 실행 → RSC Payload로 UI 업데이트
```

### API Route vs Server Actions 비교

| 구분 | `route.ts` (API Route) | Server Actions |
|------|----------------------|----------------|
| 엔드포인트 | 개발자가 URL 수동 설계 | 프레임워크가 자동 생성 |
| 호출 방식 | `fetch('/api/...')` | `import { fn } from './actions'` |
| 데이터 변환 | `JSON.stringify` / `request.json()` 수동 | 프레임워크가 자동 직렬화 |
| 타입 안전성 | 프/백 인터페이스 별도 공유 | End-to-End 100% 타입 공유 |
| 캐시 연동 | 별도 처리 필요 | `revalidatePath` 한 줄로 끝 |
| 적합한 상황 | 외부 시스템/모바일 앱 연동 | **내 앱의 프론트↔백 통신** |

## 3. useFormStatus — 다중 클릭 방어 🟡

### 절대 규칙: `<form>` 내부의 **자식 컴포넌트**에서만 호출!

```tsx
// components/SubmitButton.tsx
'use client';
import { useFormStatus } from 'react-dom';

export default function SubmitButton() {
  const { pending } = useFormStatus();  // 부모 폼의 전송 상태 추적

  return (
    <button type="submit" disabled={pending}>
      {pending ? "저장 중..." : "등록"}
    </button>
  );
}
```

```
useFormStatus 4가지 프로퍼티:

pending  : Boolean  — 폼이 전송 중인지 (핵심!)
data     : FormData — 현재 전송 중인 데이터
method   : String   — HTTP 메서드 ('post')
action   : Function — 연결된 서버 액션 참조
```

## 4. useActionState — 서버 결과를 화면에 동기화 🔴

### 핵심: 서버 액션의 반환값을 자동으로 state에 주입

```tsx
const [state, formAction, isPending] = useActionState(actionFunction, initialState);
```

### 절대 규칙: 서버 함수의 첫 번째 인자 = `prevState`

```tsx
// actions.ts
'use server';

export interface ProductFormState {
  success: boolean;
  message: string;
  attemptCount: number;
}

export async function createProductAction(
  prevState: ProductFormState,  // 프레임워크가 자동 주입!
  formData: FormData
): Promise<ProductFormState> {
  const currentAttempt = prevState.attemptCount + 1;
  const title = formData.get("title") as string;

  if (!title || title.length < 2) {
    return { success: false, message: "2글자 이상!", attemptCount: currentAttempt };
  }

  return { success: true, message: `등록 완료: ${title}`, attemptCount: currentAttempt };
}
```

```tsx
// page.tsx
'use client';
import { useActionState } from 'react';
import { createProductAction, ProductFormState } from './actions';

export default function Page() {
  const initialState: ProductFormState = { success: false, message: '', attemptCount: 0 };
  const [state, formAction] = useActionState(createProductAction, initialState);

  return (
    <form action={formAction}>  {/* formAction 사용 (원본 아님!) */}
      <input name="title" />
      <button type="submit">등록</button>
      {state.message && (
        <div className={state.success ? "text-green-600" : "text-red-600"}>
          시도: {state.attemptCount}회 — {state.message}
        </div>
      )}
    </form>
  );
}
```

## 5. revalidatePath + redirect — Mutation 패턴 🔴

### 캐싱의 역설

```
DB에 새 데이터 추가 → 목록 페이지로 이동 → 옛날 데이터만 보임!
→ force-cache가 낡은 스냅샷을 꽉 쥐고 안 놓음
```

### 해결: 캐시 파괴 → 동선 이동

```tsx
// actions.ts
'use server';
import { revalidatePath } from 'next/cache';
import { redirect } from 'next/navigation';

export async function createProductAction(prevState, formData) {
  // ... 유효성 검사 + DB 저장 ...

  revalidatePath('/products');  // /products 캐시 즉시 폐기!
  redirect('/products');         // 사용자를 목록으로 강제 이동
  // ⚠️ redirect는 내부적으로 Error를 던짐 → 반드시 맨 마지막에!
}
```

```
revalidatePath 동작:

Step 1. Purge: 해당 경로의 Data Cache + Router Cache에 "Stale" 낙인
Step 2. Invalidation: 기존 HTML/JSON 스냅샷 즉시 폐기
Step 3. 다음 접속 시: 캐시 비어있음 → DB에서 최신 데이터 fetch
Step 4. 새 데이터로 화면 렌더링 + 다시 캐싱
```

## 6. useOptimistic — 낙관적 UI 업데이트 🟡

### 비유: 서버 응답을 기다리지 않고 UI를 먼저 바꾸기

```
클릭 → UI 즉시 변경 (가짜 상태) → 서버 호출 (1초 지연)
                                    ├── 🟢 성공: 가짜 → 진짜로 승격
                                    └── 🔴 실패: 자동 롤백!
```

```tsx
// LikeButton.tsx
'use client';
import { useOptimistic } from 'react';
import { toggleLikeAction } from '../actions';

export default function LikeButton({ productId, initialLiked }) {
  const [optimisticLiked, addOptimistic] = useOptimistic(
    initialLiked,
    (state, value: boolean) => value
  );

  const handleLike = async () => {
    const newStatus = !optimisticLiked;
    addOptimistic(newStatus);  // 0초 만에 UI 변경!

    try {
      await toggleLikeAction(productId, newStatus);  // 1초 걸려도 사용자 모름
    } catch {
      alert("실패! 원래 상태로 복구됩니다.");
      // 프레임워크가 자동 롤백!
    }
  };

  return (
    <form action={handleLike}>
      <button type="submit">
        {optimisticLiked ? '❤️ 좋아요 취소' : '🤍 좋아요'}
      </button>
    </form>
  );
}
```

## 7. HTTP 무상태성과 인증/인가 아키텍처 🔴

### 7-1. HTTP Stateless — 서버는 기억하지 않는다

> 💡 **비유**: 건망증 은행원
> 방금 신분증 보여줬는데 1초 뒤 "신분증 주세요" → 매 요청이 완전히 독립적

```
왜 이렇게 설계했나?

서버가 상태를 안 기억 → 서버 100대로 복제 가능 (Scale-out)
→ 어떤 서버가 요청 받아도 동일한 결과
→ 수천만 트래픽 처리 가능!
```

### 7-2. 인증(AuthN) vs 인가(AuthZ)

| 구분 | 인증 (Authentication) | 인가 (Authorization) |
|------|---------------------|---------------------|
| 질문 | "당신은 **누구**인가?" | "당신은 **무엇을 할 수** 있는가?" |
| 비유 | 사원증으로 로비 통과 | 1급 기밀실 출입 권한 확인 |
| 실패 코드 | **401** Unauthorized | **403** Forbidden |
| 의미 | 로그인 안 됨/토큰 만료 | 로그인은 됐지만 권한 없음 |

### 7-3. 상태 유지 4대 저장소

| 저장소 | 비유 | 위치 | 보안 | 용량 |
|--------|------|------|------|------|
| **Session** | VIP 장부 | 서버 (Redis) | 최고 | 무제한 |
| **JWT** | 전자 여권 | 클라이언트 | 높음 (단, Payload 평문!) | 작음 |
| **LocalStorage** | 개인 배낭 | 브라우저 | 낮음 (XSS 취약!) | ~5MB |
| **Cookie** | 마법의 지갑 | 브라우저 (자동전송) | 설정 따라 | ~4KB |

### 7-4. Cookie 3대 보안 자물쇠 🔴

```
Set-Cookie: accessToken=eyJhbG...; Secure; HttpOnly; SameSite=Strict
```

| 속성 | 방어 대상 | 설명 |
|------|----------|------|
| `Secure` | 도청 (MITM) | HTTPS에서만 전송 |
| `HttpOnly` | XSS 공격 | JS(`document.cookie`)로 접근 불가 |
| `SameSite` | CSRF 공격 | 동일 출처 요청에만 전송 |

> ⚠️ **실무 주의**: `HttpOnly` 때문에 클라이언트 컴포넌트에서 쿠키 직접 읽기 불가
> → 반드시 서버 컴포넌트/미들웨어에서 검증 후 props로 전달

### JWT + HttpOnly Cookie 인증 흐름

```
[로그인]
1. 클라이언트 → 서버: ID/PW 전송
2. 서버: DB 검증 → JWT 생성 → Set-Cookie (HttpOnly, Secure)
3. 브라우저: 쿠키 자동 저장 (JS 접근 불가)

[이후 요청]
4. 브라우저 → 서버: 쿠키 자동 첨부
5. 서버: JWT 서명 검증 (DB 조회 불필요!) → 인가 처리
```

## 🎯 면접 대비

### 🔴 필수 질문

**Q1. Server Actions란 무엇이고, API Route와 어떻게 다른가요?**

> Server Actions는 `'use server'` 지시어로 선언된 서버 전용 함수를 클라이언트에서 직접 호출하는 **RPC 기반 데이터 변이 패턴**입니다. API Route(`route.ts`)처럼 URL을 설계하고 fetch로 호출하는 대신, 서버 함수를 import하여 `<form action={fn}>`으로 연결하면 프레임워크가 내부적으로 숨겨진 엔드포인트를 자동 생성하고 통신합니다. End-to-End 타입 안전성과 `revalidatePath`를 통한 캐시 연동이 한 번의 왕복으로 끝나는 것이 핵심 장점입니다.

**Q2. `useActionState`의 역할과 `prevState` 규칙을 설명해주세요.**

> `useActionState`는 서버 액션의 반환값을 클라이언트 상태에 **자동 동기화**하는 React 19 훅입니다. `const [state, formAction] = useActionState(action, initialState)` 형태로 사용하며, 서버 함수는 반드시 **첫 번째 인자로 `prevState`(이전 상태)**를 받아야 합니다. 프레임워크가 이전 상태를 자동 주입하므로, 클라이언트 `useState` 없이도 제출 시도 횟수 등의 누적 상태를 서버-클라이언트 간에 공유할 수 있습니다.

**Q3. `revalidatePath`는 무엇이고 왜 필요한가요?**

> 데이터 변이(Mutation) 후 캐시에 남아있는 **낡은 스냅샷을 강제로 폐기**하는 함수입니다. `force-cache`로 얼려둔 페이지는 DB가 변경되어도 과거 데이터를 보여주는 "캐싱의 역설"이 발생합니다. `revalidatePath('/products')`를 호출하면 해당 경로의 Data Cache와 Router Cache를 즉시 무효화하고, 다음 접속 시 DB에서 최신 데이터를 가져와 렌더링합니다. `redirect`와 함께 사용하면 완벽한 Mutation 패턴이 완성됩니다.

**Q4. HTTP 무상태성(Stateless)이란 무엇이고, 왜 이렇게 설계했나요?**

> 서버가 클라이언트의 이전 요청을 전혀 기억하지 않아 매 요청이 완전히 독립적인 것입니다. 로그인 성공 직후에도 다음 요청에서 "당신이 누군지 모른다"고 합니다. 이렇게 설계한 이유는 **Scale-out(수평 확장)** 때문입니다. 서버가 상태를 안 가지므로 동일한 무상태 서버를 100대로 복제하고 로드밸런서로 트래픽을 분산할 수 있습니다. 이 문제를 해결하기 위해 JWT, Session, Cookie 등의 상태 유지 기술이 발전했습니다.

**Q5. 인증(401)과 인가(403)의 차이는?**

> **인증(Authentication, 401)**: "당신이 누구인지 모르겠다" — 토큰이 없거나 만료됨. **인가(Authorization, 403)**: "당신이 누구인지는 알지만, 이 리소스에 접근할 권한이 없다" — 로그인은 됐지만 관리자 전용 페이지 접근 등. 프론트엔드는 401이면 로그인 페이지로 리다이렉트, 403이면 "권한 없음" 메시지를 표시하는 식으로 분기 처리합니다.

**Q6. Cookie의 `HttpOnly`, `Secure`, `SameSite` 속성의 역할은?**

> **HttpOnly**: `document.cookie`로 JS 접근 차단 → XSS 공격 방어. **Secure**: HTTPS에서만 쿠키 전송 → 도청(MITM) 방어. **SameSite**: 동일 출처 요청에만 쿠키 전송 → CSRF 공격 방어. 세 속성을 모두 적용한 `Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict`가 JWT 인증의 실무 보안 정석입니다.

### 🟡 개념 질문

**Q7. `useOptimistic`의 동작 원리와 롤백 메커니즘은?**

> 서버 응답을 기다리지 않고 UI를 **즉시 변경(가짜 상태)**하여 사용자에게 0초 반응을 제공합니다. `addOptimistic(newValue)` 호출 즉시 화면이 바뀌고, 백그라운드에서 서버 액션을 실행합니다. 성공하면 캐시 갱신으로 진짜 상태로 승격되고, 에러 발생 시 가짜 상태가 자동으로 폐기되어 **개발자 코드 없이 원래 상태로 롤백**됩니다.

**Q8. `useFormStatus`의 절대 규칙은?**

> `useFormStatus`는 부모 `<form>`의 Context를 읽는 훅이므로, 반드시 **`<form>` 내부의 자식 컴포넌트**에서만 호출해야 합니다. 폼과 같은 레벨에서 호출하면 부모를 찾지 못해 에러가 발생합니다. 따라서 SubmitButton 등을 별도 파일로 분리하는 **컴포넌트 격리(Decoupling)** 패턴이 필수입니다.

│  1️⃣  Server Actions = RPC 부활                                    │
│     · 'use server' → 서버 전용 함수 (브라우저 번들 제외)            │
│     · <form action={서버함수}> → API Route 불필요                  │
│     · 프레임워크가 숨겨진 엔드포인트 자동 생성                      │
│                                                                    │
│  2️⃣  폼 상태 관리 3종 세트:                                       │
│     · useFormStatus: 전송 중 다중 클릭 방어 (자식 컴포넌트 필수!)   │
│     · useActionState: 서버 결과 → 화면 자동 동기화 (prevState!)    │
│     · useOptimistic: 0초 UI 반응 + 자동 롤백                      │
│                                                                    │
│  3️⃣  Mutation 패턴:                                               │
│     DB 저장 → revalidatePath (캐시 파괴) → redirect (동선 이동)    │
│     ⚠️ redirect는 반드시 맨 마지막에! (내부적으로 Error throw)     │
│                                                                    │
│  4️⃣  HTTP 무상태성 → 상태 유지 기술:                               │
│     · Session: 서버 장부 (Redis) — 통제력 최고                      │
│     · JWT: 자가 수용적 토큰 — Scale-out 최적                       │
│     · Cookie: 자동 전송 지갑 — HttpOnly/Secure/SameSite 필수       │
│     · LocalStorage: XSS 취약 — 민감 데이터 저장 금지!              │
│                                                                    │
│  5️⃣  인증(401) vs 인가(403):                                      │
│     · 401 = "누구세요?" (로그인 필요)                               │
│     · 403 = "알겠는데 권한 없어요" (접근 금지)                      │
