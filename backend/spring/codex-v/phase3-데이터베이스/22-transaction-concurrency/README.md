# 22. 트랜잭션과 동시성

> 오늘의 목표: 여러 작업을 안전하게 묶는 트랜잭션의 의미를 이해하고, 동시에 같은 데이터를 수정할 때 생기는 문제를 막는 기본 전략을 익힌다.

---

## 오늘 끝나면 되는 것

- 트랜잭션이 왜 필요한지 설명할 수 있다.
- ACID를 큰 흐름으로 말할 수 있다.
- `@Transactional`의 역할을 설명할 수 있다.
- 격리 수준과 전파 속성의 개념을 이해한다.
- 낙관적 락과 비관적 락 차이를 설명할 수 있다.

---

## 머릿속 그림

```text
트랜잭션
 = 여러 작업을 하나로 묶음
 = 전부 성공하거나 전부 실패
```

예를 들어 송금은 아래 두 작업이 같이 성공해야 합니다.

1. A 계좌 잔액 감소
2. B 계좌 잔액 증가

이 둘을 따로 처리하면 데이터가 쉽게 망가집니다.

---

## ACID를 쉽게 이해하기

- Atomicity: 모두 성공 or 모두 실패
- Consistency: 데이터 규칙 유지
- Isolation: 동시에 실행돼도 서로 덜 꼬이게
- Durability: 커밋된 결과는 유지

외우기보다 상황으로 이해하는 게 좋습니다.

> 송금 중 오류가 나면 전부 되돌아가야 한다 -> Atomicity

---

## `@Transactional`이 하는 일

```java
@Transactional
public void transfer(Long fromId, Long toId, int amount) {
    withdraw(fromId, amount);
    deposit(toId, amount);
}
```

이 어노테이션이 있으면 보통:

- 메서드 시작 시 트랜잭션 시작
- 메서드 끝나면 commit
- 예외가 터지면 rollback

이 흐름을 스프링이 잡아줍니다.

---

## 읽기 전용 트랜잭션

```java
@Transactional(readOnly = true)
public StudyLog getById(Long id) {
    return repository.findById(id).orElseThrow();
}
```

조회 전용 메서드에는 `readOnly = true`를 붙이는 습관이 좋습니다.

이유:

- 의도를 명확히 보여줌
- 변경 감지 부담을 줄이는 데 도움

---

## 격리 수준은 왜 필요할까

동시에 여러 트랜잭션이 같은 데이터를 보면 문제가 생길 수 있습니다.

대표 예:

- Dirty Read
- Non-repeatable Read
- Phantom Read

초급 단계에서는:

> "서로 동시에 읽고 쓰는 상황에서 어느 정도까지 간섭을 허용할지 정하는 기준"

정도로 이해하면 충분합니다.

스프링에서는 보통 DB 기본값을 많이 따릅니다.

---

## 전파 속성 감각

서비스 메서드 안에서 다른 `@Transactional` 메서드를 또 호출할 때,
기존 트랜잭션에 합칠지 새로 만들지 정하는 개념입니다.

가장 많이 보는 것:

- `REQUIRED`: 기존 트랜잭션 있으면 참여, 없으면 새로 시작
- `REQUIRES_NEW`: 무조건 새 트랜잭션 시작

초보 단계에서는 `REQUIRED`가 기본이라는 점만 잘 기억하면 됩니다.

---

## 동시성 문제 예시

재고가 1개 남았는데 두 사용자가 동시에 주문하면?

```text
사용자 A 조회 -> 재고 1
사용자 B 조회 -> 재고 1
사용자 A 감소 -> 0
사용자 B 감소 -> -1
```

이런 문제를 동시성 문제라고 합니다.

## 낙관적 락 vs 비관적 락

### 낙관적 락

보통 `@Version`으로 처리합니다.

```java
@Version
private Long version;
```

생각 방식:

- 충돌이 드물다고 가정
- 수정 시점에 버전이 바뀌었으면 실패 처리

장점:

- 락을 오래 잡지 않음

### 비관적 락

조회 시점부터 DB 락을 잡습니다.

생각 방식:

- 충돌이 자주 난다고 가정
- 먼저 잠가서 다른 트랜잭션 수정 방지

장점:

- 충돌을 더 직접적으로 막음

단점:

- 대기/병목 가능

---

## Named Lock, Redis 분산 락은 언제 보나

여러 애플리케이션 인스턴스가 같이 동작하거나,
DB 행 락만으로 처리하기 어려울 때 등장합니다.

지금 단계에서는:

- Named Lock: DB 이름 기반 락
- Redis 분산 락: 외부 저장소 기반 락

정도로 감각만 잡아도 충분합니다.

---

## 자주 하는 실수

- `@Transactional` 없이 변경 감지를 기대하는 것
- 조회 메서드와 수정 메서드에 같은 관성으로 어노테이션을 붙이는 것
- 동시성 문제를 재현하지도 않고 "가끔 생겨요"로 넘기는 것
- 낙관적 락과 비관적 락을 상황 구분 없이 섞어 말하는 것
