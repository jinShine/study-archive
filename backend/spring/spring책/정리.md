# 스프링/스프링 부트 학습 정리 (복습용)

## 1) 이번 범위 핵심 요약
1. 스프링은 엔터프라이즈 애플리케이션 개발을 쉽게 하도록 돕는 프레임워크다.
2. IoC는 제어의 역전, DI는 의존성 주입이다.
3. 스프링 부트는 스타터와 자동 구성을 통해 초기 설정을 크게 줄여준다.
4. 애노테이션은 메타데이터를 코드에 선언해서 프레임워크 동작을 유도한다.
5. @SpringBootApplication은 아래 3개를 포함한다.
	- @SpringBootConfiguration
	- @ComponentScan
	- @EnableAutoConfiguration
6. @Component 계열 애노테이션을 붙인 클래스는 스프링 빈으로 등록된다.
	- @Controller / @RestController: 웹 요청 처리
	- @Service: 비즈니스 로직
	- @Repository: 데이터 접근 계층
	- @Configuration: 설정 클래스

## 2) 내가 반드시 이해해야 하는 포인트
### 2-1. @ComponentScan
- 시작 클래스 기준 하위 패키지를 스캔해서 빈을 등록한다.
- 스캔 범위를 잘못 잡으면 빈 미등록 오류가 난다.

### 2-2. @EnableAutoConfiguration
- 클래스패스 의존성과 조건(Conditional)에 따라 스프링이 자동으로 빈을 구성한다.
- 내가 등록한 빈이 있으면 자동 구성 빈보다 우선 적용되는 경우가 많다.

### 2-3. 자동 구성을 추적하는 이유
- 개발 중 내가 직접 만든 적 없는 빈이 동작할 때 원인 파악이 필요하다.
- 이때 자동 구성 리포트와 조건 매칭 결과를 확인해야 디버깅이 빨라진다.

## 3) 현재 정리에서 보완이 필요한 부분
1. WAS 용어 수정 필요
	- Web Application Service가 아니라 Web Application Server가 정확한 표현이다.
2. 자동 구성 메타 파일 설명 보완 필요
	- spring.factories는 과거 버전 중심 설명이다.
	- 최근 스프링 부트(3.x+)는 주로 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports를 사용한다.
3. IoC/DI/AOP/PSA를 "안다" 수준이 아니라 예시로 연결 필요
	- IoC: 객체 생성 주체가 개발자 코드가 아닌 컨테이너
	- DI: 생성자 주입으로 구현체 교체 용이
	- AOP: 공통 관심사(로깅, 트랜잭션) 분리
	- PSA: 기술마다 다른 API를 스프링이 일관된 방식으로 추상화

## 4) build.gradle 복습 포인트

```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '4.0.6'          // 부트 플러그인
    id 'io.spring.dependency-management' version '1.1.7'   // 의존성 버전 자동 관리
}

group = 'com.example'   // 프로젝트 그룹
version = '0.0.1-SNAPSHOT'

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(25)        // 사용할 JDK 버전
    }
}

repositories {
    mavenCentral()   // 의존성을 받아올 원격 저장소
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'   // 웹 개발 기본 의존성
    implementation 'org.springframework.boot:spring-boot-starter'       // 핵심 스타터
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
}

tasks.named('test') {
    useJUnitPlatform()   // JUnit5 기반 테스트 실행
}
```

### 각 블록 역할
- `plugins`: 빌드 도구 기능 확장
- `repositories`: 어디서 라이브러리를 받을지 지정
- `dependencies`: 프로젝트에 필요한 외부 라이브러리 등록

### 확인/점검
1. `spring-boot-starter`는 `spring-boot-starter-web`에 이미 포함되는 경우가 많아 중복 여부 점검 필요
2. Java 25는 환경에 따라 호환성 이슈가 있을 수 있으니 프로젝트 기준 버전 정책 확인 필요

## 5) 스프링 3계층 아키텍처

```
HTTP 요청 (/test)
    ↓
[프레젠테이션 계층] TestController.java   ← @Controller / @RestController
    ↕
[비즈니스 계층]     TestService.java      ← @Service
    ↕
[퍼시스턴스 계층]   MemberRepository.java ← @Repository
    ↕
데이터베이스
```

### 각 계층 역할
| 계층 | 클래스 예시 | 애노테이션 | 책임 |
|------|------------|-----------|------|
| 프레젠테이션 | TestController | @Controller | HTTP 요청/응답 처리 |
| 비즈니스 | TestService | @Service | 핵심 비즈니스 로직 |
| 퍼시스턴스 | MemberRepository | @Repository | DB 접근 및 데이터 처리 |

### 핵심 포인트
- 각 계층은 **단방향 의존**: 컨트롤러 → 서비스 → 리포지토리 순으로만 호출
- 계층 간 데이터는 주로 DTO(요청/응답)나 도메인 객체로 주고받는다
- @Repository는 DB 예외를 스프링 DataAccessException으로 변환해주는 역할도 한다

## 7) 프로젝트 디렉토리 구조 (`src/main/resources`)

```
src/main/
├── java/                       ← 자바 소스 코드
└── resources/
    ├── static/                 ← 정적 리소스 (HTML, CSS, JS, 이미지)
    ├── templates/              ← 동적 템플릿 (Thymeleaf, Mustache 등)
    └── application.properties  ← 애플리케이션 설정 파일
```

### 각 폴더 역할
| 위치 | 용도 | 예시 |
|------|------|------|
| `static/` | 빌드 그대로 서빙되는 정적 파일 | `index.html`, `app.css` |
| `templates/` | 서버에서 렌더링되는 뷰 템플릿 | `home.html` (Thymeleaf) |
| `application.properties` | 포트, DB, 로깅 등 설정값 | `server.port=8080` |

### 핵심 포인트
- 정적 리소스 기본 경로: `/static`, `/public`, `/resources`, `/META-INF/resources`
- API 서버만 만들 거면 `templates/`는 안 써도 된다
- 설정 파일은 환경별 분리 가능: `application-dev.yml`, `application-prod.yml`

## 8) application.properties vs application.yml

### properties 형식
```properties
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/test
spring.datasource.username=root
spring.jpa.hibernate.ddl-auto=update
```

### yml 형식
```yaml
server:
  port: 8080
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
  jpa:
    hibernate:
      ddl-auto: update
```

### yml을 더 많이 쓰는 이유
1. **계층 구조 가독성**: 들여쓰기로 그룹이 한눈에 보임
2. **중복 제거**: `spring.datasource.*` 같은 prefix를 반복하지 않음
3. **다중 프로파일 한 파일 관리**: `---` 구분자로 dev/prod 설정을 한 파일에 작성 가능
4. **리스트/맵 표현이 자연스러움**: 배열, 중첩 객체 표현이 properties보다 직관적

### 주의점
- yml은 **들여쓰기에 민감**(스페이스 사용, 탭 금지)
- 둘 다 있으면 properties가 우선 적용됨

## 9) Gradle 의존성 스코프

| 스코프 | 컴파일 클래스패스 | 런타임 클래스패스 | 테스트 | 용도 |
|--------|:---:|:---:|:---:|------|
| `implementation` | O | O | O | 일반적인 의존성 (가장 많이 사용) |
| `runtimeOnly` | X | O | O | 런타임에만 필요 (예: JDBC 드라이버) |
| `compileOnly` | O | X | X | 컴파일 시에만 필요 (예: Lombok) |
| `testImplementation` | X | X | O | 테스트 코드 전용 |
| `annotationProcessor` | 컴파일 시 별도 | X | X | 애노테이션 처리기 (예: Lombok, MapStruct) |

### 각 스코프 상세 설명

**`implementation`**
- 가장 기본이 되는 의존성 스코프. 컴파일할 때도, 실행할 때도 클래스패스에 포함된다.
- 이 프로젝트를 다른 프로젝트가 의존할 때, implementation으로 선언한 라이브러리는 **외부로 노출되지 않는다** (캡슐화).
- 예: `spring-boot-starter-web`, `jackson`, `jpa` 등 대부분의 라이브러리

**`runtimeOnly`**
- 코드에서 직접 import해서 쓰지는 않지만, 실행 시 반드시 있어야 하는 라이브러리.
- 컴파일 타임에는 인터페이스(API)만 있으면 되고, 실제 구현체는 런타임에 로딩된다.
- 예: MySQL/PostgreSQL JDBC 드라이버 — JPA가 내부적으로 로드하므로 코드에서 직접 참조할 일이 없다

**`compileOnly`**
- 컴파일할 때만 필요하고, 빌드된 JAR에는 포함되지 않는다.
- 런타임에는 이미 역할이 끝난 라이브러리에 사용한다.
- 예: Lombok — 컴파일 시 getter/setter 코드를 생성하고 나면 런타임에 Lombok 자체는 불필요

**`annotationProcessor`**
- 컴파일 시점에 애노테이션을 읽어서 **코드를 자동 생성**하는 처리기를 등록한다.
- `compileOnly`와 역할이 다르다: compileOnly는 "참조용", annotationProcessor는 "코드 생성용"
- 예: Lombok(`@Getter` → getter 메서드 생성), MapStruct(`@Mapper` → 매핑 코드 생성)

**`testImplementation`**
- 테스트 코드(`src/test/`)에서만 사용 가능한 의존성. 메인 코드에서는 접근 불가.
- 빌드된 JAR에 포함되지 않으므로 배포 크기에 영향을 주지 않는다.
- 예: JUnit, Mockito, AssertJ, `spring-boot-starter-test`

### 예시
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'  // 일반
    runtimeOnly 'com.mysql:mysql-connector-j'                          // JDBC 드라이버
    compileOnly 'org.projectlombok:lombok'                             // Lombok 어노테이션
    annotationProcessor 'org.projectlombok:lombok'                     // Lombok 코드 생성
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

### 핵심 포인트
- `implementation` vs `api`: `implementation`은 의존성을 외부로 노출하지 않아 빌드 속도/캡슐화에 유리
- Lombok은 `compileOnly` + `annotationProcessor` **둘 다** 등록해야 정상 작동
- DB 드라이버는 코드에서 직접 import할 필요가 없으므로 `runtimeOnly`가 적절

## 12) JUnit 5 테스트 기초

### 테스트 생명주기 애노테이션

| 애노테이션 | 실행 시점 | 실행 횟수 | 용도 |
|-----------|----------|----------|------|
| `@BeforeAll` | 모든 테스트 **전** 1회 | 1번 (static 메서드) | DB 연결, 공통 데이터 초기화 |
| `@BeforeEach` | **각** 테스트 메서드 전 | 테스트 수만큼 | 테스트별 초기 상태 세팅 |
| `@AfterEach` | **각** 테스트 메서드 후 | 테스트 수만큼 | 데이터 정리, 리소스 해제 |
| `@AfterAll` | 모든 테스트 **후** 1회 | 1번 (static 메서드) | DB 연결 종료, 최종 정리 |

```
@BeforeAll  ← 1회
│
├─ @BeforeEach
│  └─ @Test (테스트 1)
│  └─ @AfterEach
│
├─ @BeforeEach
│  └─ @Test (테스트 2)
│  └─ @AfterEach
│
@AfterAll   ← 1회
```

### 기타 테스트 애노테이션
| 애노테이션 | 설명 |
|-----------|------|
| `@Test` | 테스트 메서드임을 선언 |
| `@DisplayName("설명")` | 테스트 이름을 가독성 좋게 지정 (한글 가능) |
| `@Disabled` | 해당 테스트를 일시적으로 비활성화 (skip) |

## 13) JUnit Assertions vs AssertJ 비교

### JUnit 기본 (Assertions)
```java
import static org.junit.jupiter.api.Assertions.*;

assertEquals(expected, actual);           // 값 비교
assertTrue(condition);                     // 조건 참
assertNotNull(object);                     // null 아님
assertThrows(Exception.class, () -> {});   // 예외 발생 검증
```

### AssertJ (가독성 우선)
```java
import static org.assertj.core.api.Assertions.assertThat;

assertThat(actual).isEqualTo(expected);                 // 값 비교
assertThat(list).hasSize(3);                            // 크기 검증
assertThat(name).isNotNull().startsWith("홍");           // 체이닝 가능
assertThat(list).extracting("name").contains("홍길동");   // 객체 리스트에서 필드 추출 비교
assertThatThrownBy(() -> service.get(0))                // 예외 검증
    .isInstanceOf(IllegalArgumentException.class)
    .hasMessage("존재하지 않는 ID");
```

### 비교 요약
| 항목 | JUnit Assertions | AssertJ |
|------|:---:|:---:|
| 기본 제공 | O (JUnit 포함) | X (별도 의존성, 단 starter-test에 포함) |
| 가독성 | `assertEquals(a, b)` — 뭐가 expected인지 헷갈림 | `assertThat(a).isEqualTo(b)` — 자연어처럼 읽힘 |
| 메서드 체이닝 | X | O |
| 컬렉션/객체 검증 | 불편 | `extracting`, `filteredOn` 등 강력 |
| 실무 선호도 | 단순 검증용 | **대부분 실무에서 AssertJ 사용** |

> 결론: 기본 동작은 JUnit Assertions로 이해하되, 실제 코드는 AssertJ로 작성하는 게 표준

## 14) 스프링 부트 테스트 의존성 비교

### `spring-boot-starter-test` (기본)
```groovy
testImplementation 'org.springframework.boot:spring-boot-starter-test'
```
- JUnit 5, Mockito, AssertJ, Spring Test 등 테스트 핵심 라이브러리 **올인원 패키지**
- 단위 테스트, 서비스 계층 테스트에 필요한 모든 것 포함
- `@SpringBootTest`, `@MockBean` 등 사용 가능

### `spring-boot-starter-webmvc-test` (웹 계층 특화)
```groovy
testImplementation 'org.springframework.boot:spring-boot-starter-webmvc-test'
```
- **웹 MVC 계층(컨트롤러) 테스트에 특화**된 자동 구성 제공
- `@WebMvcTest`, `@AutoConfigureMockMvc` 등 MVC 테스트 전용 애노테이션 포함
- MockMvc 관련 설정을 자동으로 잡아줌

### 언제 뭘 쓰나?
| 상황 | 의존성 |
|------|--------|
| 서비스/리포지토리 단위 테스트 | `starter-test`만으로 충분 |
| 컨트롤러 테스트 (MockMvc) | `starter-test` + `starter-webmvc-test` |

## 15) MockMvc 컨트롤러 테스트 코드 분석

### 전체 코드
```java
@SpringBootTest
@AutoConfigureMockMvc
public class TestControllerTest {

    @Autowired
    protected MockMvc mockMvc;

    @Autowired
    private WebApplicationContext context;

    @Autowired
    private MemberRepository memberRepository;

    @BeforeEach
    public void mockMvcSetup() {
        this.mockMvc = MockMvcBuilders.webAppContextSetup(context).build();
    }

    @AfterEach
    public void cleanUp() {
        memberRepository.deleteAll();
    }

    @DisplayName("getAllMembers(): 모든 회원 정보를 조회한다.")
    @Test
    public void getAllMembers() throws Exception {
        // given — 테스트 데이터 준비
        final String url = "/test";
        Member savedMember = memberRepository.save(new Member("홍길동"));

        // when — 테스트 실행
        final ResultActions result = mockMvc.perform(
            get(url).accept(MediaType.APPLICATION_JSON)
        );

        // then — 결과 검증
        result.andExpect(status().isOk())
              .andExpect(jsonPath("$[0].id").value(savedMember.getId()))
              .andExpect(jsonPath("$[0].name").value(savedMember.getName()));
    }
}
```

### 애노테이션별 역할
| 애노테이션 | 설명 |
|-----------|------|
| `@SpringBootTest` | @SpringBootApplication 클래스를 찾아 **전체 빈을 로드**한 테스트용 컨텍스트 생성 |
| `@AutoConfigureMockMvc` | MockMvc를 자동 생성·구성. 실제 서버 없이 HTTP 요청/응답 시뮬레이션 가능 |

### 코드 흐름 상세

**1) 테스트 준비 (given)**
```java
Member savedMember = memberRepository.save(new Member("홍길동"));
```
- 실제 DB에 테스트 데이터 저장 (H2 등 인메모리 DB 사용)

**2) 요청 수행 (when)**
```java
mockMvc.perform(get(url).accept(MediaType.APPLICATION_JSON));
```
- `perform()`: HTTP 요청을 시뮬레이션. 실제 서버를 띄우지 않고 컨트롤러 호출
- `get(url)`: GET /test 요청 생성
- `accept(MediaType.APPLICATION_JSON)`: "JSON으로 응답해줘"라고 서버에 요청

**3) 결과 검증 (then)**
```java
result.andExpect(status().isOk())                                    // HTTP 200 확인
      .andExpect(jsonPath("$[0].id").value(savedMember.getId()))     // 첫 번째 요소의 id 확인
      .andExpect(jsonPath("$[0].name").value(savedMember.getName())); // 첫 번째 요소의 name 확인
```
- `status().isOk()`: 응답 상태코드가 200인지 검증
- `jsonPath("$[0].id")`: JSON 배열의 첫 번째 객체의 id 필드 접근 (JSONPath 문법)

### Given-When-Then 패턴
테스트 코드 작성의 표준 패턴이다:
| 단계 | 역할 | 이 코드에서 |
|------|------|-----------|
| **Given** | 테스트에 필요한 데이터/상태 준비 | DB에 멤버 저장 |
| **When** | 테스트할 동작 실행 | MockMvc로 GET 요청 |
| **Then** | 결과가 기대값과 일치하는지 검증 | 상태코드, JSON 필드값 확인 |

### MockMvc 주요 메서드 정리
| 메서드 | 역할 |
|--------|------|
| `get()`, `post()`, `put()`, `delete()` | HTTP 메서드별 요청 생성 |
| `.content(body)` | 요청 본문 설정 (POST/PUT) |
| `.contentType(MediaType.APPLICATION_JSON)` | 요청 Content-Type 설정 |
| `.accept(MediaType.APPLICATION_JSON)` | 응답 형식 지정 |
| `status().isOk()` / `status().isCreated()` | 상태코드 검증 |
| `jsonPath("$.필드")` | JSON 응답 필드값 검증 |

데이터 베이스

JPA와 하이버네이트
- DBMS에도 여러 종류가 있는 것처럼 ORM에도 여러 종류가 있습니다. 자바에서 JPA를 표준으로 사용한다.
JPA는 자바에서 관계형 데이터 베이스를 사용하는 방식을 정의한 인터페이스
인터페이스이므로 실제 사용을 위해서는 ORM 프레임워크를 추가로 선택해야 한다. 대표적으로 하이버네이트!
하이버네이트는 JPA 인터페이스를 구현한 구현체이자 자바용 ORM 프레임워크입니다. 내부적으로는 JDBC API를 사용하여 DB와 통신합니다. JPA는 인터페이스, 하이버네이트는 구현체라고 생각하면 됩니다.

엔티티(Entity)
- 엔티티는 데이터베이스 테이블과 매핑되는 자바 클래스입니다.
- 엔티티 클래스는 @Entity 애노테이션으로 표시됩니다.

엔티티 매니저(Entity Manager)
- 엔티티를 관리해 데이터베이스와 애플리케이션 사이에서 객체를 생성, 수정, 삭제하는 등의 역할을 한다.
그런 엔티티 매니저를 만드는 곳이 엔티티 매니저 팩토리(Entity Manager Factory)입니다. 엔티티 매니저 팩토리는 애플리케이션이 시작될 때 생성되고, 애플리케이션이 종료될 때 폐기됩니다. 엔티티 매니저 팩토리는 데이터베이스 연결 설정과 같은 초기화 작업을 수행합니다.

영속성 컨텍스트?
- 영속성 컨텍스트는 JPA의 중요한 특징 중 하나로, 엔티티를 관리하는 가상의 공간.
- 영속성의 컨텍스트에는 1차 캐시, 쓰기지연, 변경 감지, 지연 로딩이라는 특징이 있다.

1. 1차 캐시
- @Id 애너테이션을 식별자로 사용하고 조회시 1차 캐시에서 데이터를 조회하고 있으면 반환 없으면 DB에서 조회하여 1차 캐시에 저장 후 반환한다.
- 캐시된 데이터는 데이터베이스를 거치지 않고 빠르게 조회할 수 있다.

2. 쓰기 지연
- 트랜잭션을 커밋하기 전까지는 데이터베이스에 실제로 쿼리를 보내지 않고, 모았다가 트랜잭션을 커밋하면 모았던 쿼리를 한번에 실행하는 방식입니다. 이를 통해 데이터베이스에 대한 부하를 줄이고 성능을 향상시킬 수 있습니다.

3. 변경 감지
- 트랜잭션을 커밋하면 1차 캐시에 저장되어 있는 엔티티의 값과 현재 엔티티으 ㅣ값을 비교해서 변경된 값이 있다면 변경 사항을 감지해 변경된 값을 데이터베이스에 자동으로 반영

4. 지연 로딩
- 쿼리로 요청한 데이터를 애플리케이션에 바로 로딩하는 것이 아니라 필요할 때 쿼리를 날려 데이터를 조회하는 것을 의미

REST API 규칙
1. URL에는 동사를 쓰지말고, 명사를 사용한다.
    - 예를 들어 학생 중에 id가 1인 학생의 정보를 가져오는 URL
        - /students/1 (O)
        - /get-student?student_id=1 (X)
            - 누군가는 get, show 이런 동사를 쓸수 있다. 이런걸 REST API 규칙에서는 권장하지 않는다. 왜냐하면 HTTP 메서드가 이미 동사 역할을 하고 있기 때문이다.
2. 동사는 HTTP 메서드를 활용해서 행위를 표현한다.
    - GET: 조회
        - id가 1인 블로그 글을 조회 하는 API
            - GET /articles/1
    - POST: 생성
        - 블로그 글을 생성하는 API
            - POST /articles
    - PUT: 전체 업데이트
        - id가 1인 블로그 글을 전체 업데이트 하는 API
            - PUT /articles/1
    - PATCH: 일부 업데이트
        - id가 1인 블로그 글의 제목만 업데이트 하는 API
            - PATCH /articles/1
    - DELETE: 삭제
        - id가 1인 블로그 글을 삭제하는 API
            - DELETE /articles/1
