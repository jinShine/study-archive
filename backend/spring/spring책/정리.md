# 스프링/스프링 부트 학습 정리 (복습용)

## 1) 이번 범위 핵심 요약
1. 스프링은 엔터프라이즈 애플리케이션 개발을 쉽게 하도록 돕는 프레임워크다.
2. IoC는 제어의 역전, DI는 의존성 주입이다.
3. 스프링 부트는 스타터와 자동 구성을 통해 초기 설정을 크게 줄여준다.
4. 애노테이션은 메타데이터를 코드에 선언해서 프레임워크 동작을 유도한다.
5. @SpringBootApplication은 아래 3개를 포함한다.
	- @SpringBootConfiguration
	- @ComponentScan
	- @EnableAutoConfiguration
6. @Component 계열 애노테이션을 붙인 클래스는 스프링 빈으로 등록된다.
	- @Controller / @RestController: 웹 요청 처리
	- @Service: 비즈니스 로직
	- @Repository: 데이터 접근 계층
	- @Configuration: 설정 클래스

## 2) 내가 반드시 이해해야 하는 포인트
### 2-1. @ComponentScan
- 시작 클래스 기준 하위 패키지를 스캔해서 빈을 등록한다.
- 스캔 범위를 잘못 잡으면 빈 미등록 오류가 난다.

### 2-2. @EnableAutoConfiguration
- 클래스패스 의존성과 조건(Conditional)에 따라 스프링이 자동으로 빈을 구성한다.
- 내가 등록한 빈이 있으면 자동 구성 빈보다 우선 적용되는 경우가 많다.

### 2-3. 자동 구성을 추적하는 이유
- 개발 중 내가 직접 만든 적 없는 빈이 동작할 때 원인 파악이 필요하다.
- 이때 자동 구성 리포트와 조건 매칭 결과를 확인해야 디버깅이 빨라진다.

## 3) 현재 정리에서 보완이 필요한 부분
1. WAS 용어 수정 필요
	- Web Application Service가 아니라 Web Application Server가 정확한 표현이다.
2. 자동 구성 메타 파일 설명 보완 필요
	- spring.factories는 과거 버전 중심 설명이다.
	- 최근 스프링 부트(3.x+)는 주로 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports를 사용한다.
3. IoC/DI/AOP/PSA를 "안다" 수준이 아니라 예시로 연결 필요
	- IoC: 객체 생성 주체가 개발자 코드가 아닌 컨테이너
	- DI: 생성자 주입으로 구현체 교체 용이
	- AOP: 공통 관심사(로깅, 트랜잭션) 분리
	- PSA: 기술마다 다른 API를 스프링이 일관된 방식으로 추상화

## 4) build.gradle 복습 포인트

```groovy
plugins {
    id 'java'
    id 'org.springframework.boot' version '4.0.6'          // 부트 플러그인
    id 'io.spring.dependency-management' version '1.1.7'   // 의존성 버전 자동 관리
}

group = 'com.example'   // 프로젝트 그룹
version = '0.0.1-SNAPSHOT'

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(25)        // 사용할 JDK 버전
    }
}

repositories {
    mavenCentral()   // 의존성을 받아올 원격 저장소
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'   // 웹 개발 기본 의존성
    implementation 'org.springframework.boot:spring-boot-starter'       // 핵심 스타터
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
    testRuntimeOnly 'org.junit.platform:junit-platform-launcher'
}

tasks.named('test') {
    useJUnitPlatform()   // JUnit5 기반 테스트 실행
}
```

### 각 블록 역할
- `plugins`: 빌드 도구 기능 확장
- `repositories`: 어디서 라이브러리를 받을지 지정
- `dependencies`: 프로젝트에 필요한 외부 라이브러리 등록

### 확인/점검
1. `spring-boot-starter`는 `spring-boot-starter-web`에 이미 포함되는 경우가 많아 중복 여부 점검 필요
2. Java 25는 환경에 따라 호환성 이슈가 있을 수 있으니 프로젝트 기준 버전 정책 확인 필요

## 5) 스프링 3계층 아키텍처

```
HTTP 요청 (/test)
    ↓
[프레젠테이션 계층] TestController.java   ← @Controller / @RestController
    ↕
[비즈니스 계층]     TestService.java      ← @Service
    ↕
[퍼시스턴스 계층]   MemberRepository.java ← @Repository
    ↕
데이터베이스
```

### 각 계층 역할
| 계층 | 클래스 예시 | 애노테이션 | 책임 |
|------|------------|-----------|------|
| 프레젠테이션 | TestController | @Controller | HTTP 요청/응답 처리 |
| 비즈니스 | TestService | @Service | 핵심 비즈니스 로직 |
| 퍼시스턴스 | MemberRepository | @Repository | DB 접근 및 데이터 처리 |

### 핵심 포인트
- 각 계층은 **단방향 의존**: 컨트롤러 → 서비스 → 리포지토리 순으로만 호출
- 계층 간 데이터는 주로 DTO(요청/응답)나 도메인 객체로 주고받는다
- @Repository는 DB 예외를 스프링 DataAccessException으로 변환해주는 역할도 한다

## 7) 프로젝트 디렉토리 구조 (`src/main/resources`)

```
src/main/
├── java/                       ← 자바 소스 코드
└── resources/
    ├── static/                 ← 정적 리소스 (HTML, CSS, JS, 이미지)
    ├── templates/              ← 동적 템플릿 (Thymeleaf, Mustache 등)
    └── application.properties  ← 애플리케이션 설정 파일
```

### 각 폴더 역할
| 위치 | 용도 | 예시 |
|------|------|------|
| `static/` | 빌드 그대로 서빙되는 정적 파일 | `index.html`, `app.css` |
| `templates/` | 서버에서 렌더링되는 뷰 템플릿 | `home.html` (Thymeleaf) |
| `application.properties` | 포트, DB, 로깅 등 설정값 | `server.port=8080` |

### 핵심 포인트
- 정적 리소스 기본 경로: `/static`, `/public`, `/resources`, `/META-INF/resources`
- API 서버만 만들 거면 `templates/`는 안 써도 된다
- 설정 파일은 환경별 분리 가능: `application-dev.yml`, `application-prod.yml`

## 8) application.properties vs application.yml

### properties 형식
```properties
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/test
spring.datasource.username=root
spring.jpa.hibernate.ddl-auto=update
```

### yml 형식
```yaml
server:
  port: 8080
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/test
    username: root
  jpa:
    hibernate:
      ddl-auto: update
```

### yml을 더 많이 쓰는 이유
1. **계층 구조 가독성**: 들여쓰기로 그룹이 한눈에 보임
2. **중복 제거**: `spring.datasource.*` 같은 prefix를 반복하지 않음
3. **다중 프로파일 한 파일 관리**: `---` 구분자로 dev/prod 설정을 한 파일에 작성 가능
4. **리스트/맵 표현이 자연스러움**: 배열, 중첩 객체 표현이 properties보다 직관적

### 주의점
- yml은 **들여쓰기에 민감**(스페이스 사용, 탭 금지)
- 둘 다 있으면 properties가 우선 적용됨

## 9) Gradle 의존성 스코프

| 스코프 | 컴파일 클래스패스 | 런타임 클래스패스 | 테스트 | 용도 |
|--------|:---:|:---:|:---:|------|
| `implementation` | O | O | O | 일반적인 의존성 (가장 많이 사용) |
| `runtimeOnly` | X | O | O | 런타임에만 필요 (예: JDBC 드라이버) |
| `compileOnly` | O | X | X | 컴파일 시에만 필요 (예: Lombok) |
| `testImplementation` | X | X | O | 테스트 코드 전용 |
| `annotationProcessor` | 컴파일 시 별도 | X | X | 애노테이션 처리기 (예: Lombok, MapStruct) |

### 각 스코프 상세 설명

**`implementation`**
- 가장 기본이 되는 의존성 스코프. 컴파일할 때도, 실행할 때도 클래스패스에 포함된다.
- 이 프로젝트를 다른 프로젝트가 의존할 때, implementation으로 선언한 라이브러리는 **외부로 노출되지 않는다** (캡슐화).
- 예: `spring-boot-starter-web`, `jackson`, `jpa` 등 대부분의 라이브러리

**`runtimeOnly`**
- 코드에서 직접 import해서 쓰지는 않지만, 실행 시 반드시 있어야 하는 라이브러리.
- 컴파일 타임에는 인터페이스(API)만 있으면 되고, 실제 구현체는 런타임에 로딩된다.
- 예: MySQL/PostgreSQL JDBC 드라이버 — JPA가 내부적으로 로드하므로 코드에서 직접 참조할 일이 없다

**`compileOnly`**
- 컴파일할 때만 필요하고, 빌드된 JAR에는 포함되지 않는다.
- 런타임에는 이미 역할이 끝난 라이브러리에 사용한다.
- 예: Lombok — 컴파일 시 getter/setter 코드를 생성하고 나면 런타임에 Lombok 자체는 불필요

**`annotationProcessor`**
- 컴파일 시점에 애노테이션을 읽어서 **코드를 자동 생성**하는 처리기를 등록한다.
- `compileOnly`와 역할이 다르다: compileOnly는 "참조용", annotationProcessor는 "코드 생성용"
- 예: Lombok(`@Getter` → getter 메서드 생성), MapStruct(`@Mapper` → 매핑 코드 생성)

**`testImplementation`**
- 테스트 코드(`src/test/`)에서만 사용 가능한 의존성. 메인 코드에서는 접근 불가.
- 빌드된 JAR에 포함되지 않으므로 배포 크기에 영향을 주지 않는다.
- 예: JUnit, Mockito, AssertJ, `spring-boot-starter-test`

### 예시
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'  // 일반
    runtimeOnly 'com.mysql:mysql-connector-j'                          // JDBC 드라이버
    compileOnly 'org.projectlombok:lombok'                             // Lombok 어노테이션
    annotationProcessor 'org.projectlombok:lombok'                     // Lombok 코드 생성
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
```

### 핵심 포인트
- `implementation` vs `api`: `implementation`은 의존성을 외부로 노출하지 않아 빌드 속도/캡슐화에 유리
- Lombok은 `compileOnly` + `annotationProcessor` **둘 다** 등록해야 정상 작동
- DB 드라이버는 코드에서 직접 import할 필요가 없으므로 `runtimeOnly`가 적절

아래와 같은 코드 설명 필요함

```
package com.example.springboot_developer;

import org.junit.jupiter.api.AfterEach;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.MediaType;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.webmvc.test.autoconfigure.AutoConfigureMockMvc;
import org.springframework.test.web.servlet.MockMvc;
import org.springframework.test.web.servlet.ResultActions;
import org.springframework.test.web.servlet.setup.MockMvcBuilders;
import org.springframework.web.context.WebApplicationContext;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*

@SpringBootTest // 테스트용 애플리케이션 컨텍스트 생성
@AutoConfigureMockMvc // MockMvc 생성 및 자동 구성
public class TestControllerTest {

    @Autowired
    protected MockMvc mockMvc;

    @Autowired
    private WebApplicationContext context;

    @Autowired
    private MemberRepository memberRepository;

    @BeforeEach
    public void mockMvcSetup() {
        this.mockMvc = MockMvcBuilders.webAppContextSetup(context).build();
    }

    @AfterEach
    public void cleanUp() {
        memberRepository.deleteAll();
    }

    @DisplayName("getAllMembers(): 모든 회원 정보를 조회한다.")
    @Test
    public void getAllMembers() throws Exception {
        // given
        final String url = "/test";
        Member savedMember = memberRepository.save(new Member("홍길동"));

        // when
        final ResultActions result = mockMvc.perform(get(url).accept(org.springframework.http.MediaType.APPLICATION_JSON));
        // perform() 메서드는 요청을 전송하는 역할을 하는 메서드
        // accept() 메서드는 클라이언트가 서버로부터 어떤 형식의 응답을 기대하는지를 지정하는 메서드

        // then
        result.andExpect(status().isOk())
              .andExpect(jsonPath("$[0].id").value(savedMember.getId()))
              .andExpect(jsonPath("$[0].name").value(savedMember.getName()));
    }
}

```

@SpringBootTest
- 메인 애플리케이션 클래스에 추가하는 애너테이션인 @SpringBootApplication이 있는 클래스를 찾고, 그 클래스에 포함되어 있는 빈을 찾은 다음 테스트용 애플리케이션 컨텍스트라는 것을 만듭니다.

@AutoConfigureMockMvc
- MockMvc를 생성하고 자동으로 구성하는 애너테이션
- MockMvc는 서버에 배포하지 않고도 테스트용 MVC 환경을 만들어 요청 및 전송, 응답기능을 제공하는 유틸리티 클래스, 즉, 컨트롤러를 테스트할 때 사용되는 클래스.
