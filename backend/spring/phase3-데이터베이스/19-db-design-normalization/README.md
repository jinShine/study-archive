## 1. 왜 DB 설계가 중요한가? 🔴

### 비유: 옷장 정리

옷장을 떠올려보자.

```
❌ 정리 안 된 옷장
┌─────────────────────────────────┐
│ 양말+속옷+셔츠+바지+코트        │
│ 전부 한 칸에 쑤셔넣음            │
│ 양말 한 짝 찾으려면 다 뒤져야 함  │
│ 같은 옷이 여기저기 중복으로 있음  │
└─────────────────────────────────┘

✅ 정리된 옷장
┌──────┬──────┬──────┬──────┐
│ 양말 │ 속옷 │ 상의 │ 하의 │  ← 종류별로 분리
│      │      │      │      │  ← 찾기 쉬움
│      │      │      │      │  ← 중복 없음
└──────┴──────┴──────┴──────┘
```

DB 설계도 마찬가지다:
- **정리 안 된 DB** = 데이터 중복, 수정할 때 실수, 삭제하면 다른 데이터도 날아감
- **정리된 DB** = 데이터 중복 없음, 수정이 안전, 삭제해도 다른 데이터에 영향 없음

### 나쁜 설계의 실제 피해

이런 테이블이 있다고 해보자:

```
orders (주문 테이블) — 나쁜 설계
┌────┬───────────┬────────────┬──────────┬─────────┬──────────────┐
│ id │ 고객이름   │ 고객전화    │ 상품이름  │ 상품가격 │ 주문일        │
├────┼───────────┼────────────┼──────────┼─────────┼──────────────┤
│  1 │ 홍길동    │ 010-1234   │ 키보드    │ 50000   │ 2026-03-01   │
│  2 │ 홍길동    │ 010-1234   │ 마우스    │ 30000   │ 2026-03-02   │
│  3 │ 김철수    │ 010-5678   │ 키보드    │ 50000   │ 2026-03-03   │
│  4 │ 홍길동    │ 010-9999   │ 모니터    │ 200000  │ 2026-03-05   │
└────┴───────────┴────────────┴──────────┴─────────┴──────────────┘
```

**문제가 보이나?**

| 문제 | 구체적 상황 | 피해 |
|------|------------|------|
| **데이터 중복** | "홍길동", "010-1234"가 여러 행에 반복 | 저장 공간 낭비, 불일치 위험 |
| **수정 이상** | 홍길동이 전화번호를 바꾸면? → 행 1,2는 `010-1234`, 행 4는 `010-9999` → **어떤 게 맞는 번호?** | 데이터 불일치 발생 |
| **삽입 이상** | 아직 주문 안 한 신규 고객을 등록하려면? → 주문 정보가 없어서 행을 만들 수 없다 | 데이터 등록 불가 |
| **삭제 이상** | 주문 id=3을 삭제하면? → 김철수의 전화번호 정보도 함께 사라진다 | 의도치 않은 데이터 손실 |

> 🔴 **이 4가지 이상 현상(Anomaly)은 면접에서 자주 물어본다.** "정규화가 왜 필요한지" 물으면 이 이상 현상을 설명하면 된다.

## 2. ERD — 설계의 시작 🔴

### ERD란?

**E**ntity **R**elationship **D**iagram — 테이블 간의 관계를 그림으로 표현한 **설계 도면**이다.

건축에서 집 짓기 전에 도면을 그리는 것처럼, 코딩하기 전에 ERD를 먼저 그린다.

### 2-1. ERD의 3가지 구성 요소

```
1. 엔티티(Entity) = 테이블 = 네모 상자
2. 속성(Attribute) = 컬럼 = 상자 안의 항목
3. 관계(Relationship) = 테이블 간 연결선
```

### 2-2. 엔티티(Entity) 찾기 🔴

> "명사를 찾아라"

요구사항에서 **명사**를 뽑으면 엔티티 후보가 된다.

```
요구사항: "학생이 강의를 수강하고, 강사가 강의를 개설한다"

명사 추출:
  - 학생 → student 테이블
  - 강의 → course 테이블
  - 강사 → instructor 테이블
  - 수강 → enrollment 테이블 (행위도 엔티티가 될 수 있다!)
```

**엔티티가 될 수 있는 것 vs 없는 것:**

| 엔티티 O | 엔티티 X |
|----------|----------|
| 회원, 상품, 주문, 게시글 | 이름, 가격, 날짜 (→ 이건 속성) |
| 카테고리, 태그, 댓글 | "빠른", "인기있는" (→ 이건 형용사) |
| 결제, 배송, 수강 (행위) | "조회하다", "삭제하다" (→ 이건 기능) |

### 2-3. 관계(Relationship)와 카디널리티 🔴

관계 = 엔티티 사이의 연결. **카디널리티** = 연결의 수적 관계.

#### 관계를 찾는 실전 방법: "동사 찾기" 🔴

엔티티를 찾았으면, 엔티티 사이에 어울리는 **동사**를 찾아보자.

```
1단계: 엔티티 간에 어울리는 동사 찾기
   - A(주어)가 B를 ___한다.  (A가 주어)
   - B(주어)가 A에 의해 ___된다.  (B가 주어)
   ** 서비스의 관점에서 동사를 떠올려야 한다.

2단계: "하나의" or "여러 개의" 찾기
   - 하나의 A는 (하나의 or 여러 개의) B를 ___한다.  (A의 관점)
   - 하나의 B는 (하나의 or 여러 개의) A에 의해 ___된다.  (B의 관점)
   ** 문장 처음에 "하나의"를 반드시 붙여야 헷갈리지 않는다.

3단계: 관계 판단
   - A, B 관점 전부 "하나"면 → A : B = 1:1
   - A 관점에서 "여러 개" B, B 관점에서 "하나" A → A : B = 1:N
   - A 관점에서 "하나" B, B 관점에서 "여러 개" A → A : B = N:1
   - A, B 관점 전부 "여러 개"면 → A : B = N:M
```

**예시로 연습:**

```
사용자(users) ↔ 게시글(posts)

동사: 사용자가 게시글을 "작성한다" / 게시글은 사용자에 의해 "작성된다"

하나의 사용자는 여러 개의 게시글을 작성한다 → "여러 개"
하나의 게시글은 하나의 사용자에 의해 작성된다 → "하나"

→ 사용자 : 게시글 = 1 : N ✅
```

```
학생(students) ↔ 수강 과목(courses)

동사: 학생이 과목을 "듣는다" / 과목은 학생에 의해 "들어진다"

하나의 학생은 여러 개의 과목을 듣는다 → "여러 개"
하나의 과목은 여러 명의 학생에 의해 들어진다 → "여러 개"

→ 학생 : 수강 과목 = N : M ✅ (중간 테이블 필요!)
```

> 💡 **이 방법이 익숙해지면 요구사항만 봐도 ERD가 머릿속에 그려진다.** 처음엔 어색하지만 3~4번만 해보면 자연스러워진다.

#### 1:1 관계 (One-to-One)

```
한 명의 회원은 하나의 프로필을 가진다

member                      member_profile
┌──────────┐    1 : 1    ┌───────────────┐
│ id (PK)  │─────────────│ id (PK)       │
│ email    │             │ member_id(FK) │  ← UNIQUE!
│ password │             │ bio           │
└──────────┘             │ avatar_url    │
                         └───────────────┘
```

> 💡 1:1은 왜 테이블을 나눌까?
> - **자주 사용하는 데이터**와 **가끔 사용하는 데이터**를 분리 (성능)
> - 예: 로그인할 때는 email/password만 필요, 프로필 페이지에서만 bio/avatar 필요

#### 1:N 관계 (One-to-Many) — 가장 많이 쓰인다!

```
한 명의 회원이 여러 개의 게시글을 작성한다

member                      post
┌──────────┐    1 : N    ┌──────────────┐
│ id (PK)  │────────────→│ id (PK)      │
│ name     │    (한 명이  │ title        │
└──────────┘    여러 개)   │ content      │
                         │ member_id(FK)│  ← FK는 "N"쪽에!
                         └──────────────┘
```

> 🔴 **FK(외래키)는 항상 "N"(많은 쪽) 테이블에 둔다.** 이거 면접에서 물어본다.
>
> 왜? member 테이블에 post_id를 넣으면 여러 값을 넣어야 하니까 1NF 위반이 된다.

#### N:M 관계 (Many-to-Many)

```
한 학생이 여러 강의를 수강하고, 한 강의에 여러 학생이 있다

student                enrollment (중간 테이블)        course
┌──────────┐          ┌───────────────┐          ┌──────────┐
│ id (PK)  │──┐       │ id (PK)       │      ┌──│ id (PK)  │
│ name     │  └──────→│ student_id(FK)│      │  │ title    │
└──────────┘          │ course_id(FK) │←─────┘  │ credits  │
                      │ enrolled_at   │          └──────────┘
                      └───────────────┘
                       ↑ 중간 테이블이 핵심!
```

> 🔴 **N:M 관계는 직접 구현할 수 없다.** 반드시 **중간 테이블**을 만들어서 `1:N + N:1`로 분해한다.
>
> 중간 테이블에는 추가 속성(등록일, 성적 등)을 넣을 수 있어서 오히려 더 유연하다.

### 2-4. ERD 표기법 — 까마귀 발 표기법 (Crow's Foot) 🔴

실무와 면접에서 가장 많이 쓰이는 표기법이다.

```
기호 설명:
──────  직선           = 필수 (반드시 있어야 한다)
──○───  원(circle)     = 선택 (없을 수도 있다)
──<───  까마귀 발       = 여러 개 (Many)
──|───  수직선         = 하나 (One)

조합 예시:
──||──  정확히 하나 (필수 1개)
──|○──  0 또는 1개
──<|──  하나 이상 (필수 다수)
──<○──  0개 이상 (선택 다수)
```

가장 자주 쓰는 패턴:

```
member ──||────────<○── post
         필수1개        0개이상

읽는 법: "회원 한 명은 게시글을 0개 이상 가진다"
         "게시글 하나는 반드시 회원 한 명에 속한다"
```

### 2-5. ERD 설계 순서 (실습용)

```
1단계: 요구사항에서 명사 추출 → 엔티티 후보 목록
2단계: 각 엔티티의 속성(컬럼) 결정
3단계: 엔티티 간 관계와 카디널리티 결정
4단계: PK, FK 결정
5단계: ERD 그리기 (dbdiagram.io 추천)
```

### 2-6. ERD 도구 🟡

| 도구 | 특징 | 추천 상황 |
|------|------|----------|
| **dbdiagram.io** | 코드로 ERD 작성, 무료 | 개인 학습, 빠른 설계 (추천!) |
| **ERDCloud** | 한국어 지원, 협업 | 팀 프로젝트 |
| **MySQL Workbench** | DB 연결해서 역공학 ERD | 기존 DB 분석 |
| **draw.io** | 범용 다이어그램 | 발표 자료 |

**dbdiagram.io 코드 예시:**

```
// dbdiagram.io에서 이 코드를 붙여넣으면 ERD가 그려진다
Table member {
  id bigint [pk, increment]
  name varchar(50)
  email varchar(100)
}

Table post {
  id bigint [pk, increment]
  title varchar(200)
  content text
  member_id bigint [ref: > member.id]  // N:1 관계
}

Table tag {
  id bigint [pk, increment]
  name varchar(50)
}

Table post_tag {
  post_id bigint [ref: > post.id]
  tag_id bigint [ref: > tag.id]

  indexes {
    (post_id, tag_id) [pk]  // 복합 PK
  }
}
```

## 3. 함수적 종속 — 정규화의 핵심 원리 🔴

### 비유: 주민등록번호와 이름

주민등록번호를 알면 이름이 **딱 하나** 결정된다. 이걸 **함수적 종속**이라고 한다.

```
주민등록번호 → 이름       ✅ 함수적 종속 (번호가 이름을 결정)
이름 → 주민등록번호       ❌ 함수적 종속 아님 (동명이인이 있을 수 있음)
```

### 정의

> **A → B**: A의 값을 알면 B의 값이 **하나로** 결정된다

```
학번 → 학생이름          "학번을 알면 이름이 하나로 결정된다"
(학번, 과목코드) → 성적   "학번+과목코드를 알면 성적이 하나로 결정된다"
```

### 왜 이게 중요한가?

정규화의 모든 규칙은 결국 **"각 컬럼이 PK에만 종속되도록 만들어라"**라는 원칙이다.

```
🎯 정규화의 목표를 한 문장으로:
"모든 컬럼은 PK 전체에, 오직 PK에만 종속되어야 한다"
   ↑ 2NF가 보장    ↑ 3NF가 보장
```

> 이 문장을 기억하면 1NF~3NF를 전부 관통하는 원리를 이해한 거다.

## 4. 제1정규형 (1NF) 🔴

### 규칙: 하나의 셀에는 하나의 값만!

**원자값(Atomic Value)**만 허용한다. "더 이상 쪼갤 수 없는 값"이라는 뜻이다.

### 위반 사례 1: 하나의 셀에 여러 값

```
❌ 1NF 위반 — 학생 수강 테이블
┌────────┬───────────┬──────────────────────┐
│ 학번   │ 학생이름   │ 수강과목              │
├────────┼───────────┼──────────────────────┤
│ 2001   │ 홍길동    │ 자바, 스프링, 알고리즘  │  ← 한 셀에 3개!
│ 2002   │ 김철수    │ 자바, 네트워크         │  ← 한 셀에 2개!
└────────┴───────────┴──────────────────────┘
```

**왜 문제인가?**

```sql
-- "스프링 수강생"을 찾으려면?
SELECT * FROM student WHERE 수강과목 = '스프링';      -- ❌ 안 됨! (값이 "자바, 스프링, 알고리즘")
SELECT * FROM student WHERE 수강과목 LIKE '%스프링%'; -- ⚠️ 되긴 하지만...
-- "스프링부트"도 매칭될 수 있고, 인덱스도 못 쓴다 → 느리고 위험
```

### 해결: 행을 분리하거나, 별도 테이블 생성

**방법 1: 행 분리 (간단하지만 중복 발생)**

```
┌────────┬───────────┬──────────┐
│ 학번   │ 학생이름   │ 수강과목  │
├────────┼───────────┼──────────┤
│ 2001   │ 홍길동    │ 자바      │
│ 2001   │ 홍길동    │ 스프링    │  ← 학생이름이 중복되네?
│ 2001   │ 홍길동    │ 알고리즘  │     → 이건 2NF에서 해결
│ 2002   │ 김철수    │ 자바      │
│ 2002   │ 김철수    │ 네트워크  │
└────────┴───────────┴──────────┘
```

**방법 2: 별도 테이블 (권장)**

```
student (학생)                enrollment (수강)
┌────────┬───────────┐       ┌────────┬───────────┐
│ 학번   │ 학생이름   │       │ 학번   │ 과목코드   │
├────────┼───────────┤       ├────────┼───────────┤
│ 2001   │ 홍길동    │       │ 2001   │ CS101     │
│ 2002   │ 김철수    │       │ 2001   │ CS201     │
└────────┴───────────┘       │ 2001   │ CS301     │
                             │ 2002   │ CS101     │
                             │ 2002   │ CS401     │
                             └────────┴───────────┘
```

### 위반 사례 2: 반복 그룹 (컬럼이 반복되는 형태)

```
❌ 1NF 위반 — 과목을 컬럼으로 나열
┌────────┬───────────┬────────┬────────┬────────┐
│ 학번   │ 학생이름   │ 과목1   │ 과목2   │ 과목3   │
├────────┼───────────┼────────┼────────┼────────┤
│ 2001   │ 홍길동    │ 자바    │ 스프링  │ 알고리즘│
│ 2002   │ 김철수    │ 자바    │ 네트워크│ NULL   │  ← 과목 2개면 NULL
└────────┴───────────┴────────┴────────┴────────┘

문제: 과목을 4개 수강하면? → 컬럼을 또 추가해야 한다 (과목4, 과목5...)
```

> 🔴 **1NF 정리**: "한 셀 한 값, 반복 컬럼 금지"

### 심화: "한 가지 정보"의 기준은 서비스마다 다르다 🟡

```
users (사용자)
┌────┬────────┐        ┌────┬─────┬────────┐
│ id │ 이름   │   vs   │ id │ 성  │ 이름    │
├────┼────────┤        ├────┼─────┼────────┤
│ 1  │ 박재성 │        │ 1  │ 박  │ 재성    │
│ 2  │ 김시온 │        │ 2  │ 김  │ 시온    │
└────┴────────┘        └────┴─────┴────────┘
```

"박재성"이 한 가지 정보일까, "박" + "재성" 두 가지 정보일까?

**둘 다 맞다.** 서비스에서 성과 이름을 따로 검색해야 한다면 분리하고, 통째로만 쓴다면 합쳐도 된다.

```
전화번호도 마찬가지:
  0321112345  → 한 가지 정보?
  032-111-2345 → 지역번호(032) + 국번(111) + 번호(2345) 세 가지 정보?

→ 서비스에서 지역번호별 통계가 필요하면 분리, 아니면 통째로 저장
```

> 💡 **"한 가지 정보"는 절대적이지 않다. 서비스에서 데이터의 사용 방식에 따라 결정해야 한다.** 이게 현업에서 DB 설계가 어려운 이유다.

## 5. 제2정규형 (2NF) 🔴

### 전제: 1NF를 만족해야 한다

### 규칙: 모든 컬럼이 PK **전체**에 종속되어야 한다

**부분 종속(Partial Dependency)**을 제거한다.

> 💡 PK가 하나의 컬럼이면 부분 종속이 발생할 수 없다. **복합 PK일 때만** 2NF 위반이 가능하다!

### 비유로 이해하기

```
도서관 대출 기록 — 키(key)가 (회원번호, 도서번호) 두 개라고 하자

"대출일" → 회원번호 + 도서번호 둘 다 알아야 결정됨 ✅ (전체 종속)
"회원이름" → 회원번호만 알면 결정됨 ❌ (부분 종속! 도서번호는 필요 없음)
"도서제목" → 도서번호만 알면 결정됨 ❌ (부분 종속! 회원번호는 필요 없음)
```

### 위반 사례

```
❌ 2NF 위반 — 수강 성적 테이블
PK = (학번, 과목코드) ← 복합 PK

┌────────┬──────────┬───────────┬──────────┬────────┐
│ 학번   │ 과목코드  │ 학생이름   │ 과목이름  │ 성적   │
├────────┼──────────┼───────────┼──────────┼────────┤
│ 2001   │ CS101    │ 홍길동    │ 자바      │ A      │
│ 2001   │ CS201    │ 홍길동    │ 스프링    │ B+     │
│ 2002   │ CS101    │ 김철수    │ 자바      │ A+     │
└────────┴──────────┴───────────┴──────────┴────────┘

종속 관계를 분석해보면:
  (학번, 과목코드) → 성적      ✅ PK 전체에 종속
  학번 → 학생이름              ❌ PK의 일부에만 종속 (부분 종속!)
  과목코드 → 과목이름           ❌ PK의 일부에만 종속 (부분 종속!)
```

**부분 종속이 왜 문제인가?**

```
"홍길동"이 이름을 개명하면?
→ 학번 2001이 있는 모든 행(행1, 행2)을 수정해야 한다
→ 하나라도 빠뜨리면? → 수정 이상!
```

### 해결: 부분 종속되는 컬럼을 별도 테이블로 분리

```
✅ 2NF 만족 — 3개 테이블로 분리

student (학생)                course (과목)               enrollment (수강 성적)
┌────────┬───────────┐       ┌──────────┬──────────┐     ┌────────┬──────────┬────────┐
│ 학번   │ 학생이름   │       │ 과목코드  │ 과목이름  │     │ 학번   │ 과목코드  │ 성적   │
│  (PK)  │           │       │   (PK)   │          │     │  (FK)  │   (FK)   │        │
├────────┼───────────┤       ├──────────┼──────────┤     ├────────┼──────────┼────────┤
│ 2001   │ 홍길동    │       │ CS101    │ 자바      │     │ 2001   │ CS101    │ A      │
│ 2002   │ 김철수    │       │ CS201    │ 스프링    │     │ 2001   │ CS201    │ B+     │
└────────┴───────────┘       └──────────┴──────────┘     │ 2002   │ CS101    │ A+     │
                                                         └────────┴──────────┴────────┘
                                                          PK = (학번, 과목코드) 복합PK

이제:
  학번 → 학생이름        ✅ student 테이블에서 PK 전체에 종속
  과목코드 → 과목이름     ✅ course 테이블에서 PK 전체에 종속
  (학번, 과목코드) → 성적  ✅ enrollment 테이블에서 PK 전체에 종속
```

> 🔴 **2NF 정리**: "복합 PK의 **일부**에만 종속되는 컬럼을 분리하라"

## 6. 제3정규형 (3NF) 🔴

### 전제: 2NF를 만족해야 한다

### 규칙: PK가 아닌 컬럼이 다른 PK가 아닌 컬럼에 종속되면 안 된다

**이행적 종속(Transitive Dependency)**을 제거한다.

### 비유로 이해하기

```
A → B → C 라면 (A가 B를 결정하고, B가 C를 결정하면)
A → C 가 되는데 (A가 C를 간접적으로 결정)

이걸 "이행적 종속"이라고 한다.

구체적으로:
  학번 → 학과코드 → 학과이름

  학번으로 학과코드를 알 수 있고
  학과코드로 학과이름을 알 수 있으니까
  학번으로 학과이름을 알 수 있다 → 이행적 종속!
```

### 위반 사례

```
❌ 3NF 위반 — 학생 테이블
PK = 학번

┌────────┬───────────┬──────────┬──────────┐
│ 학번   │ 학생이름   │ 학과코드  │ 학과이름  │
│  (PK)  │           │          │          │
├────────┼───────────┼──────────┼──────────┤
│ 2001   │ 홍길동    │ CS       │ 컴퓨터공학│
│ 2002   │ 김철수    │ CS       │ 컴퓨터공학│  ← "컴퓨터공학" 중복!
│ 2003   │ 이영희    │ EE       │ 전자공학  │
└────────┴───────────┴──────────┴──────────┘

종속 관계:
  학번 → 학생이름       ✅ PK에 직접 종속
  학번 → 학과코드       ✅ PK에 직접 종속
  학과코드 → 학과이름    ❌ PK가 아닌 컬럼(학과코드)에 종속! (이행적 종속)
```

**이행적 종속이 왜 문제인가?**

```
1. 학과 이름이 "컴퓨터공학" → "소프트웨어학"으로 바뀌면?
   → CS 학과인 학생 전부(행1, 행2)를 수정해야 한다 → 수정 이상

2. CS 학과 학생이 전부 자퇴하면?
   → "컴퓨터공학"이라는 학과 정보도 사라진다 → 삭제 이상

3. "경영학과(BA)"를 신설하고 싶은데 학생이 아직 없으면?
   → 학생 없이는 학과 정보를 넣을 수 없다 → 삽입 이상
```

### 해결: 이행적 종속되는 컬럼을 별도 테이블로 분리

```
✅ 3NF 만족 — 2개 테이블로 분리

student (학생)                          department (학과)
┌────────┬───────────┬──────────┐       ┌──────────┬──────────┐
│ 학번   │ 학생이름   │ 학과코드  │       │ 학과코드  │ 학과이름  │
│  (PK)  │           │   (FK)   │       │   (PK)   │          │
├────────┼───────────┼──────────┤       ├──────────┼──────────┤
│ 2001   │ 홍길동    │ CS       │       │ CS       │ 컴퓨터공학│
│ 2002   │ 김철수    │ CS       │       │ EE       │ 전자공학  │
│ 2003   │ 이영희    │ EE       │       │ BA       │ 경영학   │  ← 학생 없이도 등록 가능!
└────────┴───────────┴──────────┘       └──────────┴──────────┘

이제:
  학번 → 학생이름       ✅ student에서 PK에 직접 종속
  학번 → 학과코드       ✅ student에서 PK에 직접 종속
  학과코드 → 학과이름    ✅ department에서 PK에 직접 종속
```

> 🔴 **3NF 정리**: "PK가 아닌 컬럼끼리의 종속(이행적 종속)을 제거하라"

### 1NF~3NF 한 줄 요약 (면접 암기용)

```
1NF: 한 셀에 한 값 (원자값)
2NF: PK 전체에 종속 (부분 종속 제거)  — 복합 PK일 때만 해당
3NF: PK에만 종속 (이행적 종속 제거) — 비키 컬럼끼리 종속 금지
```

> 🔴 이 세 줄을 면접에서 자신 있게 말할 수 있으면 정규화의 80%를 아는 거다.

## 7. BCNF (Boyce-Codd Normal Form) 🟡

### 전제: 3NF를 만족해야 한다

### 3NF로도 부족한 경우가 있다?

3NF는 "PK가 아닌 컬럼 → PK가 아닌 컬럼" 종속을 막았다. 하지만 **"PK가 아닌 컬럼 → PK의 일부"** 종속은 막지 못한다.

### 규칙: 모든 결정자(→ 왼쪽)가 후보키여야 한다

> **결정자(Determinant)**: 다른 컬럼의 값을 결정하는 컬럼
> **후보키(Candidate Key)**: PK가 될 수 있는 컬럼 (유일하게 행을 식별)

### 위반 사례

```
수강 지도 테이블 — 한 과목을 여러 교수가 가르치지만,
각 교수는 하나의 과목만 담당한다고 하자.

┌────────┬──────────┬───────────┐
│ 학번   │ 과목     │ 담당교수   │
├────────┼──────────┼───────────┤
│ 2001   │ 자바     │ 김교수    │
│ 2001   │ 스프링   │ 이교수    │
│ 2002   │ 자바     │ 박교수    │
│ 2002   │ 스프링   │ 이교수    │
└────────┴──────────┴───────────┘

PK = (학번, 과목)  ← 학번+과목이면 행이 유일하게 결정된다

종속 관계:
  (학번, 과목) → 담당교수     ✅ PK → 비키 (3NF OK)
  담당교수 → 과목             ❌ 비키 → PK의 일부! (BCNF 위반!)

  "이교수"를 알면 "스프링"이 결정된다 (교수는 한 과목만 담당하니까)
  → 담당교수가 "결정자"인데, 후보키가 아니다!
```

### 해결: 결정자를 PK로 만드는 테이블 분리

```
✅ BCNF 만족

professor_course (교수-과목)           student_professor (학생-교수)
┌───────────┬──────────┐              ┌────────┬───────────┐
│ 담당교수   │ 과목     │              │ 학번   │ 담당교수   │
│   (PK)    │          │              │  (FK)  │   (FK)    │
├───────────┼──────────┤              ├────────┼───────────┤
│ 김교수    │ 자바      │              │ 2001   │ 김교수    │
│ 이교수    │ 스프링    │              │ 2001   │ 이교수    │
│ 박교수    │ 자바      │              │ 2002   │ 박교수    │
└───────────┴──────────┘              │ 2002   │ 이교수    │
                                      └────────┴───────────┘
```

> 🟡 **BCNF 정리**: "결정자는 반드시 후보키여야 한다." 실무에서 BCNF까지 직접 적용하는 경우는 드물지만, 면접에서 "BCNF가 뭔지" 물어볼 수 있다. 3NF와 차이점을 설명하면 OK.

## 8. 정규화 과정 전체 흐름 정리 🔴

### 한눈에 보기

```
비정규 테이블 (하나에 다 때려넣은 상태)
    │
    ▼  [1NF] 한 셀에 한 값, 반복 컬럼 제거
    │
  1NF 만족
    │
    ▼  [2NF] 부분 종속 제거 (복합 PK일 때)
    │
  2NF 만족
    │
    ▼  [3NF] 이행적 종속 제거
    │
  3NF 만족
    │
    ▼  [BCNF] 모든 결정자가 후보키
    │
  BCNF 만족
```

### 실전 예제: 처음부터 끝까지

**요구사항**: 온라인 서점의 주문 데이터를 관리한다.

```
비정규 테이블:
┌──────┬──────┬────────┬──────┬──────────────────────────────┬──────┐
│주문ID│고객명│고객전화 │주문일│ 주문상품                      │총금액│
├──────┼──────┼────────┼──────┼──────────────────────────────┼──────┤
│ O001 │홍길동│010-1234│03-01 │ 자바의정석(30000), 스프링(35000)│65000│
│ O002 │김철수│010-5678│03-02 │ 자바의정석(30000)              │30000│
└──────┴──────┴────────┴──────┴──────────────────────────────┴──────┘
```

**Step 1: 1NF 적용** — 주문상품 컬럼에 여러 값이 있다 → 행 분리

```
┌──────┬──────┬────────┬──────┬────────────┬──────┬────┐
│주문ID│고객명│고객전화 │주문일│ 상품명      │단가  │수량│
├──────┼──────┼────────┼──────┼────────────┼──────┼────┤
│ O001 │홍길동│010-1234│03-01 │ 자바의정석  │30000 │ 1  │
│ O001 │홍길동│010-1234│03-01 │ 스프링인액션│35000 │ 1  │
│ O002 │김철수│010-5678│03-02 │ 자바의정석  │30000 │ 1  │
└──────┴──────┴────────┴──────┴────────────┴──────┴────┘

PK = (주문ID, 상품명)  ← 복합 PK
1NF ✅ (각 셀에 원자값)
```

**Step 2: 2NF 적용** — 복합 PK이므로 부분 종속 확인

```
종속 관계:
  주문ID → 고객명, 고객전화, 주문일    ❌ PK의 일부에만 종속 (부분 종속!)
  상품명 → 단가                      ❌ PK의 일부에만 종속 (부분 종속!)
  (주문ID, 상품명) → 수량             ✅ PK 전체에 종속

→ 분리!

order (주문)                        product (상품)
┌──────┬──────┬────────┬──────┐    ┌────────────┬──────┐
│주문ID│고객명│고객전화 │주문일│    │ 상품명      │ 단가 │
│ (PK) │      │        │      │    │  (PK)      │      │
├──────┼──────┼────────┼──────┤    ├────────────┼──────┤
│ O001 │홍길동│010-1234│03-01 │    │ 자바의정석  │30000 │
│ O002 │김철수│010-5678│03-02 │    │ 스프링인액션│35000 │
└──────┴──────┴────────┴──────┘    └────────────┴──────┘

order_item (주문 상세)
┌──────┬────────────┬────┐
│주문ID│ 상품명      │수량│
│ (FK) │  (FK)      │    │
├──────┼────────────┼────┤
│ O001 │ 자바의정석  │ 1  │
│ O001 │ 스프링인액션│ 1  │
│ O002 │ 자바의정석  │ 1  │
└──────┴────────────┴────┘

2NF ✅ (부분 종속 제거 완료)
```

**Step 3: 3NF 적용** — 이행적 종속 확인

```
order 테이블을 보자:
  주문ID → 고객명       ✅
  주문ID → 고객전화     ✅... 잠깐,
  고객명 → 고객전화     ❌ 비키 컬럼끼리 종속! (이행적 종속)

→ 고객 정보를 분리!

customer (고객)                     order (주문)
┌──────┬──────┬────────┐           ┌──────┬──────┬──────┐
│고객ID│고객명│고객전화 │           │주문ID│고객ID│주문일│
│ (PK) │      │        │           │ (PK) │ (FK) │      │
├──────┼──────┼────────┤           ├──────┼──────┼──────┤
│ C001 │홍길동│010-1234│           │ O001 │ C001 │03-01 │
│ C002 │김철수│010-5678│           │ O002 │ C002 │03-02 │
└──────┴──────┴────────┘           └──────┴──────┴──────┘

product (상품)           order_item (주문 상세)
┌──────┬────────────┬──────┐  ┌──────┬──────┬────┐
│상품ID│ 상품명      │ 단가 │  │주문ID│상품ID│수량│
│ (PK) │            │      │  │ (FK) │ (FK) │    │
├──────┼────────────┼──────┤  ├──────┼──────┼────┤
│ P001 │ 자바의정석  │30000 │  │ O001 │ P001 │ 1  │
│ P002 │ 스프링인액션│35000 │  │ O001 │ P002 │ 1  │
└──────┴────────────┴──────┘  │ O002 │ P001 │ 1  │
                              └──────┴──────┴────┘

3NF ✅ (이행적 종속 제거 완료)
```

**최종 ERD:**

```
customer ──||──────<○── order ──||──────<○── order_item ──○>──────||── product
  고객         1:N      주문        1:N       주문상세        N:1        상품
```

> 🔴 이 예제가 면접에서 "정규화 과정을 설명해보세요"에 대한 완벽한 답변이 된다.

## 9. 가짜 중복 vs 진짜 중복 🔴

### DB 설계의 핵심 판단 기준

테이블을 설계하고 데이터를 넣어봤는데 중복처럼 보이는 게 있다. 그런데 **모든 중복이 나쁜 건 아니다.**

```
posts (게시글)
┌────┬──────────────────┬──────┬────────┐
│ id │ 제목              │ 내용 │ 작성자  │
├────┼──────────────────┼──────┼────────┤
│  1 │ [속보] 개발자 채용 │ 내용 │ 박재성 │
│  2 │ [속보] 개발자 채용 │ 내용 │ 박재성 │
│  3 │ [속보] 개발자 채용 │ 내용 │ 박재성 │
└────┴──────────────────┴──────┴────────┘

제목 컬럼: "박재성"이 같은 제목의 글을 3개 씀 → 중복처럼 보인다
내용 컬럼: 내용도 같아 보인다
작성자 컬럼: "박재성"이 3번 반복된다

🤔 전부 중복일까?
```

### 판단 방법: "A를 수정하면 B도 같이 수정되어야 하는가?"

```
1. 1번 게시글의 작성자 이름을 수정하면,
   2번 게시글의 작성자 이름도 같이 수정되어야 하는가?
   → Yes! (= 진짜 중복) ← 개명하면 다 바뀌어야 하니까

2. 1번 게시글의 제목을 수정하면,
   2번 게시글의 제목도 같이 수정되어야 하는가?
   → No! (= 가짜 중복) ← 각각 독립적인 게시글이니까

3. 1번 게시글의 내용을 수정하면,
   2번 게시글의 내용도 같이 수정되어야 하는가?
   → No! (= 가짜 중복) ← 각각 다른 글이니까
```

> 🔴 **진짜 중복만 테이블 분리로 해결한다.** 가짜 중복은 건드리면 안 된다!

```
→ 작성자(진짜 중복)만 분리:

posts (게시글)                    users (사용자)
┌────┬──────────────────┬──────┬──────────┐   ┌────┬────────┐
│ id │ 제목              │ 내용 │작성자 id  │   │ id │ 이름   │
│    │                  │      │  (FK)    │   ├────┼────────┤
├────┼──────────────────┼──────┼──────────┤   │  1 │ 박재성 │
│  1 │ [속보] 개발자 채용 │ 내용 │    1     │   └────┴────────┘
│  2 │ [속보] 개발자 채용 │ 내용 │    1     │
│  3 │ [속보] 개발자 채용 │ 내용 │    1     │
└────┴──────────────────┴──────┴──────────┘
** FK의 중복(1, 1, 1)은 중복이라고 판단하지 않는다.
```

### 숨어있는 중복도 찾아라 🟡

눈에 보이는 중복뿐만 아니라 **숨어있는 중복**도 있다.

```
posts (게시글)
┌────┬───────┬──────┬──────────┬──────────────┐
│ id │ 제목  │ 내용 │ 좋아요 수 │ 사용자 id(FK) │
├────┼───────┼──────┼──────────┼──────────────┤
│  1 │ 제목1 │ 내용1│    2     │      1       │
└────┴───────┴──────┴──────────┴──────────────┘

likes (좋아요)
┌────┬──────────────┬──────────────┐
│ id │ 사용자 id(FK) │ 게시글 id(FK) │
├────┼──────────────┼──────────────┤
│  1 │       1      │      1       │
│  2 │       2      │      1       │
└────┴──────────────┴──────────────┘
```

겉으로 보면 중복 컬럼이 없어 보인다. 하지만:

```
2번 사용자가 1번 게시글의 좋아요를 취소하면?
→ likes 테이블에서 데이터 삭제
→ posts 테이블의 좋아요 수도 2 → 1로 수정해야 한다!

💡 "좋아요 수"는 likes 테이블에서 COUNT로 구할 수 있는 값이다.
   즉, 다른 테이블의 데이터로 계산 가능한 값 = "숨어있는 중복"
```

> 숨어있는 중복은 주로 **집계 값(합계, 평균, 최대값, 카운트 등)**에서 많이 나타난다.
>
> 이걸 의도적으로 남겨두는 게 바로 **반정규화**다. (바로 다음 섹션에서 설명)

## 10. 반정규화 (Denormalization) 🟡

### 정규화가 항상 정답은 아니다

정규화를 하면 테이블이 많아진다. 테이블이 많아지면 **JOIN이 많아진다**. JOIN이 많아지면 **쿼리가 느려질 수 있다**.

### 비유: 도서관 vs 내 책상

```
정규화 = 도서관
  - 모든 책이 분류 기준에 따라 정리되어 있다
  - 깔끔하고 중복이 없다
  - 하지만 여러 카테고리의 책을 참고하려면 이 서가 저 서가 왔다갔다 해야 한다

반정규화 = 내 책상 위
  - 자주 보는 책 3권을 책상에 꺼내놨다
  - 도서관에도 있고 책상에도 있으니 "중복"이다
  - 하지만 바로 펼쳐볼 수 있으니 "빠르다"
```

### 반정규화란?

> 성능 향상을 위해 **의도적으로** 정규화 규칙을 어기는 것

### 반정규화 기법 🟡

#### 기법 1: 컬럼 중복 (가장 흔한 반정규화)

```
정규화된 상태 (3NF):

order                          customer
┌──────┬──────┬──────┐         ┌──────┬──────┐
│주문ID│고객ID│주문일│         │고객ID│고객명│
└──────┴──────┴──────┘         └──────┴──────┘

-- 주문 목록에 고객명을 보여주려면 항상 JOIN 필요
SELECT o.주문ID, c.고객명, o.주문일
FROM order o
JOIN customer c ON o.고객ID = c.고객ID;
```

```
반정규화 (컬럼 중복):

order
┌──────┬──────┬──────┬──────┐
│주문ID│고객ID│고객명│주문일│   ← 고객명을 중복 저장!
└──────┴──────┴──────┴──────┘

-- JOIN 없이 바로 조회 가능
SELECT 주문ID, 고객명, 주문일 FROM order;
```

> ⚠️ 고객이 이름을 바꾸면 order 테이블도 수정해야 한다. 이 **동기화 비용**이 반정규화의 대가다.

#### 기법 2: 파생 컬럼 (계산 결과 저장)

```
정규화:
-- 게시글의 댓글 수를 알려면 매번 COUNT
SELECT p.*, (SELECT COUNT(*) FROM comment c WHERE c.post_id = p.id) AS comment_count
FROM post p;

반정규화:
-- post 테이블에 comment_count 컬럼 추가
post
┌────┬───────┬───────────────┐
│ id │ title │ comment_count │  ← 댓글 달릴 때마다 +1, 삭제 시 -1
└────┴───────┴───────────────┘

SELECT id, title, comment_count FROM post;  -- COUNT 불필요!
```

#### 기법 3: 테이블 병합 (1:1 관계를 하나로)

```
정규화:
member ←→ member_profile  (1:1)

반정규화:
member (프로필 컬럼을 member에 합침)
┌────┬──────┬───────┬──────┬────────────┐
│ id │ name │ email │ bio  │ avatar_url │
└────┴──────┴───────┴──────┴────────────┘
```

#### 기법 4: 테이블 분할 (수직/수평)

```
수직 분할: 자주 쓰는 컬럼과 안 쓰는 컬럼 분리
  member_core: id, name, email (자주 조회)
  member_detail: id, bio, avatar, created_at (가끔 조회)

수평 분할: 데이터를 범위로 나눔
  order_2025: 2025년 주문
  order_2026: 2026년 주문
```

### 반정규화 판단 기준 🔴

```
반정규화를 해도 되는 경우:
  ✅ 조회가 압도적으로 많고, 수정은 거의 없는 데이터
  ✅ JOIN 비용이 성능에 심각한 영향을 주는 경우
  ✅ 실시간 집계가 필요한 카운터 (좋아요 수, 댓글 수 등)

반정규화하면 안 되는 경우:
  ❌ 데이터 정합성이 최우선인 경우 (금융, 의료)
  ❌ 수정이 빈번한 데이터 (동기화 비용 > JOIN 비용)
  ❌ "느릴 것 같다"는 감이 아니라 실제 측정 없이 반정규화하는 것
```

> 🔴 **면접 핵심**: "반정규화는 왜 하나요?" → "정규화로 인한 JOIN 비용이 성능 병목이 될 때, **측정 결과를 근거로** 의도적으로 중복을 허용하는 것입니다. 대표적으로 자주 조회하는 컬럼 중복, 파생 컬럼 추가, 테이블 병합 등이 있습니다."

## 11. 테이블 네이밍 규칙 🔴

실무에서 지키는 네이밍 규칙이다. 혼자 할 때도 습관 들여야 나중에 팀에서 안 헤맨다.

### 규칙 1: 소문자만 사용

```
firstName  // Bad! → MySQL은 OS에 따라 대소문자 구분이 다름
first_name // Good!
```

### 규칙 2: snake_case 사용

```
studyLogId   // Bad! (camelCase)
study_log_id // Good! (snake_case)
```

### 규칙 3: 축약어를 쓰지 않는다

```
fst_nm     // Bad! → 다른 사람이 못 읽음
first_name // Good!

std_id     // Bad!
student_id // Good!
```

### 규칙 4: SQL 예약어만 대문자

```sql
SELECT name, age
FROM users
WHERE age = 20
AND name = '재성';

-- SQL 키워드(SELECT, FROM, WHERE, AND)만 대문자
-- 테이블명, 컬럼명은 소문자
```

### 규칙 5: 테이블명은 복수형 (선택)

```
user → users    (복수형)
post → posts    (복수형)

💡 이건 회사마다 다르다. 중요한 건 프로젝트 내에서 일관성을 지키는 것.
   JPA의 Entity 클래스는 단수형(User, Post)을 쓰고,
   테이블명은 복수형(users, posts)으로 매핑하는 패턴이 흔하다.
```

> 🔴 **네이밍 규칙은 개발팀에서 가장 먼저 합의하는 것 중 하나다.** 면접에서 "DB 설계 경험이 있나요?" 물으면 네이밍 규칙도 언급하면 좋다.

## 12. 실전 설계 실습 🔴

### 실습 1: 학습 일지 서비스 ERD 설계

**요구사항:**
- 회원은 이메일로 가입한다 (이름, 이메일, 비밀번호)
- 회원은 학습 일지를 작성한다 (제목, 내용, 카테고리, 학습 시간, 학습 날짜)
- 학습 일지에 태그를 붙일 수 있다 (하나의 일지에 여러 태그, 하나의 태그에 여러 일지)
- 학습 일지에 댓글을 달 수 있다 (작성자, 내용)

**Step 1: 엔티티 추출**

```
명사: 회원, 학습 일지, 태그, 댓글
→ 엔티티: member, study_log, tag, comment
→ N:M 중간 테이블: study_log_tag
```

**Step 2: 속성 결정**

```
member:    id, name, email, password, created_at
study_log: id, member_id(FK), title, content, category, study_time, study_date, created_at
tag:       id, name
comment:   id, study_log_id(FK), member_id(FK), content, created_at
study_log_tag: study_log_id(FK), tag_id(FK)
```

**Step 3: 관계 결정**

```
member ──||──────<○── study_log    (1:N) 회원 → 학습 일지
member ──||──────<○── comment      (1:N) 회원 → 댓글
study_log ──||───<○── comment      (1:N) 학습 일지 → 댓글
study_log ──<○───<○── tag          (N:M) 학습 일지 ↔ 태그 (중간: study_log_tag)
```

**Step 4: SQL DDL**

```sql
CREATE TABLE member (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    name       VARCHAR(50)  NOT NULL,
    email      VARCHAR(100) NOT NULL UNIQUE,
    password   VARCHAR(255) NOT NULL,
    created_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);

CREATE TABLE study_log (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    member_id  BIGINT       NOT NULL,
    title      VARCHAR(200) NOT NULL,
    content    TEXT,
    category   VARCHAR(50)  NOT NULL,
    study_time INT          NOT NULL DEFAULT 0,  -- 분 단위
    study_date DATE         NOT NULL,
    created_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    FOREIGN KEY (member_id) REFERENCES member(id)
);

CREATE TABLE tag (
    id   BIGINT      NOT NULL AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL UNIQUE,
    PRIMARY KEY (id)
);

CREATE TABLE study_log_tag (
    study_log_id BIGINT NOT NULL,
    tag_id       BIGINT NOT NULL,
    PRIMARY KEY (study_log_id, tag_id),
    FOREIGN KEY (study_log_id) REFERENCES study_log(id),
    FOREIGN KEY (tag_id) REFERENCES tag(id)
);

CREATE TABLE comment (
    id           BIGINT   NOT NULL AUTO_INCREMENT,
    study_log_id BIGINT   NOT NULL,
    member_id    BIGINT   NOT NULL,
    content      TEXT     NOT NULL,
    created_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    FOREIGN KEY (study_log_id) REFERENCES study_log(id),
    FOREIGN KEY (member_id) REFERENCES member(id)
);
```

**정규화 검증:**

```
✅ 1NF: 모든 셀에 원자값만 존재 (태그는 중간 테이블로 분리)
✅ 2NF: 복합 PK는 study_log_tag뿐이고, 추가 속성 없으므로 부분 종속 없음
✅ 3NF: 각 테이블에서 비키 컬럼 간 종속 없음
         (고객명 같은 중복 저장 없이 FK로 참조)
```

### 실습 2: 온라인 쇼핑몰 ERD 설계

**요구사항:**
- 고객이 회원가입한다 (이름, 이메일, 전화번호, 주소)
- 상품이 있다 (상품명, 가격, 재고, 카테고리)
- 고객이 상품을 주문한다 (한 번에 여러 상품 가능)
- 각 주문 상품의 수량과 가격을 기록한다

**직접 해보자! 아래는 정답.**

<details>
<summary>정답 보기 (직접 해본 후에 펼치기)</summary>

```
엔티티: customer, product, category, order, order_item

customer ──||──────<○── order         (1:N)
order ──||─────────<○── order_item    (1:N)
product ──||───────<○── order_item    (1:N → 결과적으로 N:M 분해)
category ──||──────<○── product       (1:N)
```

```sql
CREATE TABLE category (
    id   BIGINT      NOT NULL AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL UNIQUE,
    PRIMARY KEY (id)
);

CREATE TABLE customer (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    name       VARCHAR(50)  NOT NULL,
    email      VARCHAR(100) NOT NULL UNIQUE,
    phone      VARCHAR(20),
    address    VARCHAR(255),
    created_at DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);

CREATE TABLE product (
    id          BIGINT       NOT NULL AUTO_INCREMENT,
    name        VARCHAR(200) NOT NULL,
    price       INT          NOT NULL,
    stock       INT          NOT NULL DEFAULT 0,
    category_id BIGINT,
    PRIMARY KEY (id),
    FOREIGN KEY (category_id) REFERENCES category(id)
);

CREATE TABLE `order` (
    id          BIGINT   NOT NULL AUTO_INCREMENT,
    customer_id BIGINT   NOT NULL,
    order_date  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    total_price INT      NOT NULL DEFAULT 0,  -- 반정규화! (매번 SUM 안 하려고)
    PRIMARY KEY (id),
    FOREIGN KEY (customer_id) REFERENCES customer(id)
);

CREATE TABLE order_item (
    id         BIGINT NOT NULL AUTO_INCREMENT,
    order_id   BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity   INT    NOT NULL,
    price      INT    NOT NULL,  -- 주문 당시 가격! (상품 가격이 바뀔 수 있으니까)
    PRIMARY KEY (id),
    FOREIGN KEY (order_id) REFERENCES `order`(id),
    FOREIGN KEY (product_id) REFERENCES product(id)
);
```

> 💡 **`order_item.price`는 반정규화가 아니다!** 주문 당시 가격을 기록하는 것이다. 상품 가격은 나중에 변할 수 있으므로, 주문 시점의 가격을 별도로 저장해야 한다. 이건 **비즈니스 요구사항**이다.

> 💡 **`order.total_price`는 반정규화다.** `order_item`의 `price * quantity`를 SUM하면 구할 수 있지만, 주문 목록을 조회할 때마다 JOIN + SUM하기 싫어서 미리 계산해 저장한다.

</details>

## 13. 설계할 때 자주 하는 실수들 🔴

### 실수 1: ID를 안 만든다

```
❌ 자연키를 PK로 사용
CREATE TABLE member (
    email VARCHAR(100) PRIMARY KEY,  -- 이메일이 PK
    name VARCHAR(50)
);

문제: 이메일이 변경되면? → FK로 참조하는 모든 테이블도 변경해야 한다!
```

```
✅ 대리키(Surrogate Key) 사용
CREATE TABLE member (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,  -- 의미 없는 숫자 PK
    email VARCHAR(100) NOT NULL UNIQUE,     -- 이메일은 UNIQUE로
    name VARCHAR(50)
);
```

> 🔴 **JPA에서도 `@Id` + `@GeneratedValue`로 대리키를 기본으로 사용한다.** 이유: PK는 절대 변하면 안 되는데, 비즈니스 데이터(이메일, 주민번호 등)는 변할 수 있다.

### 실수 2: 모든 것을 한 테이블에

```
❌ 하나의 거대한 테이블
member (id, name, email, password, address, phone, bio, avatar,
        last_login, login_count, point, grade, ...)

→ 컬럼이 20개 이상이면 테이블 분리를 의심하자
```

### 실수 3: N:M을 중간 테이블 없이 구현

```
❌ 컬럼에 여러 값 저장
post (id, title, tag_ids = "1,2,3")

→ 1NF 위반! 검색도 안 되고, 정합성도 보장 안 됨
```

### 실수 4: created_at, updated_at 빼먹기

```
실무에서 거의 모든 테이블에 필요한 컬럼:
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

→ "이 데이터 언제 만들어졌지?" "마지막 수정이 언제지?"
→ 디버깅과 감사(Audit)에 필수
```

### 실수 5: 소프트 삭제 미고려

```
하드 삭제: DELETE FROM member WHERE id = 1;  → 진짜 삭제, 복구 불가

소프트 삭제: UPDATE member SET deleted_at = NOW() WHERE id = 1;
  → deleted_at이 NULL이면 활성, NOT NULL이면 삭제 처리
  → 필요하면 복구 가능

💡 JPA에서는 @SQLDelete + @Where로 구현한다 (20번 이후에 배움)
```

## 14. 면접 대비 🔴🟡

### 🔴 필수 — 이것만은 반드시 답할 수 있어야 한다

**Q1: 정규화가 뭔가요? 왜 하나요?**

> 데이터의 중복을 최소화하고 무결성을 보장하기 위해 테이블을 분리하는 과정입니다. 정규화를 하지 않으면 삽입/수정/삭제 이상(Anomaly)이 발생합니다.

**Q2: 1NF, 2NF, 3NF를 각각 설명해주세요.**

> - **1NF**: 모든 컬럼의 값이 원자값이어야 합니다. 한 셀에 여러 값을 넣거나, 반복 컬럼을 만들면 안 됩니다.
> - **2NF**: 1NF를 만족하면서, 모든 비키 속성이 기본키 **전체**에 함수적 종속되어야 합니다. 복합 키의 일부에만 종속되는 부분 종속을 제거합니다.
> - **3NF**: 2NF를 만족하면서, 비키 속성이 다른 비키 속성에 종속되는 이행적 종속을 제거합니다.

**Q3: 반정규화는 언제 하나요?**

> 정규화로 인한 과도한 JOIN이 성능 병목이 되는 경우, **측정 결과를 근거로** 의도적으로 중복을 허용합니다. 대표적으로 자주 조회하는 컬럼 중복, 집계 결과 컬럼 추가 등이 있습니다.

**Q4: 이상 현상(Anomaly)의 종류를 설명해주세요.**

> - **삽입 이상**: 불필요한 데이터를 함께 입력해야만 원하는 데이터를 저장할 수 있는 문제
> - **수정 이상**: 중복된 데이터 중 일부만 수정되어 데이터 불일치가 발생하는 문제
> - **삭제 이상**: 데이터를 삭제할 때 의도치 않은 다른 정보까지 함께 삭제되는 문제

**Q5: FK(외래키)는 어느 쪽 테이블에 두나요?**

> 1:N 관계에서 FK는 **N(다)** 쪽에 둡니다. 1 쪽에 두면 여러 값을 저장해야 하므로 1NF를 위반합니다.

**Q6: N:M 관계는 어떻게 구현하나요?**

> 직접 구현할 수 없으므로 **중간 테이블(연결 테이블)**을 만들어 1:N + N:1로 분해합니다. 중간 테이블에는 양쪽의 FK를 가지며, 필요시 추가 속성(등록일, 수량 등)을 넣을 수 있습니다.

**Q7: 진짜 중복과 가짜 중복의 차이는?**

> 진짜 중복은 **한 곳을 수정하면 다른 곳도 반드시 수정해야 하는** 데이터입니다 (예: 작성자명). 가짜 중복은 값이 우연히 같아 보이지만 각각 독립적인 데이터입니다 (예: 같은 제목의 서로 다른 게시글). 테이블 분리는 진짜 중복에 대해서만 수행합니다.

### 🟡 개념만 — 물어보면 설명할 수 있으면 OK

**Q8: BCNF와 3NF의 차이는?**

> 3NF는 비키→비키 종속을 금지하지만, 비키→키의 일부 종속은 허용합니다. BCNF는 더 엄격하게 **모든 결정자가 후보키**여야 한다고 요구합니다.

**Q9: 자연키(Natural Key)와 대리키(Surrogate Key)의 차이는?**

> - **자연키**: 비즈니스 의미가 있는 값을 PK로 사용 (이메일, 주민번호)
> - **대리키**: 비즈니스 의미 없는 자동 생성 값을 PK로 사용 (AUTO_INCREMENT)
> - 실무에서는 대리키를 권장합니다. 자연키는 비즈니스 변경(이메일 변경 등)에 취약하기 때문입니다.

**Q10: 소프트 삭제(Soft Delete)란?**

> 실제로 DELETE하지 않고 `deleted_at` 컬럼에 삭제 시각을 기록하는 방식입니다. 데이터 복구가 가능하고, 감사(Audit) 추적에 유리합니다. 단점은 모든 쿼리에 `WHERE deleted_at IS NULL` 조건이 필요하다는 것입니다.
