# Harness Engineering

**AI가 잘 일하는 환경을 설계하는 기술**

> 출처: 이호연 (Team Attention), 2026.04.07 발표 정리·재구성

---

## 들어가며

AI에게 같은 일을 시켰는데 어떤 날은 한 번에 끝나고, 어떤 날은 열 번을 다시 시켜도 결과가 엉망인 경험이 있을 겁니다. 대부분의 사람들은 이 차이를 **"오늘은 AI가 좀 안 좋네"** 또는 **"프롬프트를 더 잘 써야겠다"**라고 해석합니다.

그런데 정말 그럴까요?

LangChain은 올해 초 흥미로운 실험 결과를 공개했습니다. Terminal-Bench 벤치마크에서 **같은 모델(GPT-5.2-Codex)을 고정해놓고**, AI가 일하는 방식만 바꿨더니 순위가 30위에서 Top 5로 올라갔습니다. 점수는 14%p 상승. 모델은 그대로였습니다.

OpenAI 엔지니어 3~7명은 5개월 동안 100만 줄의 코드를 만들었습니다. 그 중 인간이 직접 작성한 코드는 **0줄**이었습니다. 그들이 5개월 동안 했던 일은 "코딩"이 아니라, **AI가 코딩할 환경을 설계하는 것**이었습니다.

이 글은 바로 그 "환경"에 관한 이야기입니다.

### 이 글은 이런 분들을 위한 것입니다

- Claude Code, Cursor, Copilot 같은 AI 코딩 도구를 쓰고 있지만 결과물이 들쭉날쭉한 분
- 프롬프트 엔지니어링은 어느 정도 익숙해졌는데, 그 다음이 궁금한 분
- 팀 단위로 AI 워크플로우를 표준화하고 싶은 분
- "AI가 시키면 시키는 대로만 하는 도구"에서 "스스로 일하는 파트너"로 격상시키고 싶은 분

### 오늘 당신이 얻어갈 것

1. **관점의 전환** — 왜 프롬프트 튜닝의 시대가 저물고 있는지
2. **6개의 축** — AI 작업 환경을 구성하는 구체적인 틀
3. **실전 패턴** — 지금 당장 적용할 수 있는 스킬·규칙·오케스트레이션 설계법
4. **검증된 도구** — Team Attention이 공개한 실제 저장소와 사용법
5. **자가 진단 도구** — 내가 잘 가고 있는지 판단하는 기준

## Chapter 1. 왜 지금 Harness인가

### "더 좋은 프롬프트를 쓰면 되지 않나요?"

많은 사람들이 AI의 결과물이 만족스럽지 않을 때 가장 먼저 하는 일은 **프롬프트를 다시 쓰는 것**입니다. 더 자세히, 더 구체적으로, 더 친절하게. 이 접근은 효과가 있습니다 — 어느 지점까지는요.

문제는 프롬프트 엔지니어링의 수확 체감이 생각보다 빨리 찾아온다는 겁니다. 한 번의 대화에서 아무리 프롬프트를 잘 써도, 다음 세션에서 AI는 아무것도 기억하지 못합니다. 같은 프로젝트에서 일하는데도 매번 처음부터 설명해야 합니다. "우리 회사는 snake_case를 써" "테스트는 pytest로 짜" "이 모듈은 JWT 인증이야" — 같은 말을 백 번 반복합니다.

더 근본적인 문제는 **프롬프트로는 해결할 수 없는 영역이 존재한다는 것**입니다. 예를 들어 이런 것들:

- AI가 커밋 전에 반드시 테스트를 돌리게 만드는 것
- AI가 `git push --force`를 절대 실행하지 못하게 막는 것
- AI가 실수하더라도 메인 브랜치에 영향이 가지 않도록 격리하는 것
- 여러 AI 에이전트가 각자 다른 관점에서 작업을 병렬로 검증하는 것

이런 것들은 프롬프트로 "하지 마"라고 말한다고 해결되지 않습니다. **환경**이 필요합니다.

### 업계의 네 가지 신호

2025년 후반부터 2026년 초에 공개된 주요 테크 회사들의 사례를 보면 공통된 방향이 보입니다.

**LangChain의 실험**
동일한 모델로 단 4주간 작업해 Terminal-Bench 순위를 30위에서 Top 5로 끌어올렸습니다. 52.8 → 66.5%. 이들이 바꾼 것은 Self Verification Loop, Context 자동 수집, Doom Loop 탐지 같은 **에이전트가 일하는 방식 자체**였습니다.

**OpenAI의 현실**
엔지니어 3~7명이 5개월간 100만 줄의 코드를 생산했고 인간 코드는 0줄이었습니다. 그들이 내놓은 다섯 가지 교훈에 "더 좋은 모델을 쓰라"는 없었습니다. 모두 **환경 설계**에 관한 것이었습니다.

**Anthropic의 비용 실험**
같은 일을 싱글 에이전트에게 시키면 20분에 $9가 들고 실패했습니다. 3개의 에이전트로 구성된 하네스에 6시간에 $200을 쓰자 완전히 동작했습니다. 비용을 줄이고 싶을 때 해결책은 "더 싼 모델"이 아니라 "더 나은 하네스"였고, 간소화한 버전은 $124로도 품질을 유지했습니다.

**Stripe의 규모**
주당 1,000개의 PR이 완전 무인으로 자동 머지되고 있습니다. 이게 가능한 이유는 전문화된 소형 에이전트 다수 + 매 스텝 검증 게이트 + 정밀 컨텍스트 관리, 즉 **촘촘한 하네스** 때문입니다. 하네스 없이는 불가능한 규모입니다.

네 회사의 공통된 메시지는 하나입니다.

> **모델 교체로 5% 개선하는 것보다, 하네스 설계로 15% 개선하는 것이 현실적이다.**

GPT-5가 GPT-6가 되는 데 쓰이는 수백억 달러는 우리가 어쩔 수 없지만, 하네스 설계는 개인 개발자도 오늘 당장 시작할 수 있는 일입니다.

### 프롬프트 → 컨텍스트 → 하네스

이 흐름은 세 단계의 진화로 이해하면 명확합니다.

**Level 1: Prompt Engineering — "이렇게 말해봐"**
질문을 잘 하면 답이 좋아진다는 깨달음. 2023~2024년의 주요 화두였습니다. Chain-of-Thought, Few-shot 예시, 역할 부여. 지금도 유효하지만, 이것만으로는 한계가 명확합니다.

**Level 2: Context Engineering — "이런 배경지식을 줘봐"**
배경을 알려주면 AI가 맥락을 이해한다는 인식. 시스템 프롬프트, RAG, 프로젝트 문서를 주입하는 단계입니다. 많은 팀이 현재 이 단계에 있습니다.

**Level 3: Harness Engineering — "환경 자체를 설계하자"**
작업 환경을 만들어주면 AI가 혼자서도 잘 일한다는 깨달음. 맥락만이 아니라 **도구, 규칙, 경계, 검증, 개선 루프**까지 모두 설계합니다. 이게 지금 우리가 있어야 할 지점입니다.

세 단계는 대체 관계가 아니라 **축적 관계**입니다. Harness Engineering은 Prompt와 Context를 버리는 게 아니라, 그 위에 쌓아 올리는 것입니다.

## Chapter 2. Harness란 무엇인가

### 정의: 환경 자체를 설계한다

Harness Engineering을 한 문장으로 정의하면 이렇습니다.

> **AI가 혼자서도 잘 일할 수 있는 작업 환경을 만들어주는 것.**

이 정의를 이해하는 가장 쉬운 방법은 **신입사원을 처음 받는 상황**에 비유해보는 것입니다.

신입이 들어오면 우리는 이런 것들을 준비합니다. 책상과 노트북(**도구**), 회사 위키(**맥락**), 코딩 컨벤션 문서(**규칙**), "이건 반드시 선임 승인받고 해야 해"라는 체크리스트(**경계**), 코드 리뷰 프로세스(**검증**), 매주 회고(**개선**). 신입이 아무리 똑똑해도 이 환경이 없으면 헤맵니다. 있으면 처음 보는 일도 스스로 처리합니다.

AI도 똑같습니다. 모델이 좋아도 환경이 없으면 매번 같은 실수를 반복합니다. 환경이 갖춰지면 자율성이 폭발적으로 증가합니다.

OpenAI와 Anthropic은 이 개념을 각자 이렇게 표현합니다.

- **OpenAI**: "사람이 방향을 잡고, 에이전트가 실행한다."
- **Anthropic**: "규율은 코드가 아니라 스캐폴딩에서 드러난다."

말은 달라도 가리키는 것은 같습니다. 사람의 역할이 **직접 하는 것**에서 **잘 하게 만드는 것**으로 이동한다는 것.

### Harness의 6개 축

그럼 구체적으로 무엇을 설계해야 할까요? 여섯 개의 축이 있습니다.

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  구조    │ ──▶ │  맥락    │ ──▶ │  계획    │
│Scaffold  │     │ Context  │     │ Planning │
│뭘 깔아두나│     │뭘 아는가 │     │뭘 정하나 │
└──────────┘     └──────────┘     └──────────┘
                                        │
                                        ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│  개선    │ ◀── │  검증    │ ◀── │  실행    │
│Compound  │     │ Verify   │     │ Execute  │
│어떻게    │     │어떻게    │     │어떻게    │
│나아지나  │     │믿는가   │     │시키는가  │
└──────────┘     └──────────┘     └──────────┘
     ▲
     └─────────────── 다시 구조로 ────────
```

각 축의 역할은 이렇습니다.

1. **구조(Scaffolding)** — 프로젝트 폴더링, 도구 배치, 경계 설정. 한 번 깔아두면 오래 쓰는 기반.
2. **맥락(Context)** — AI가 작업에 들어갈 때 무엇을 알고 있어야 하는가. CLAUDE.md, rules, docs.
3. **계획(Planning)** — AI에게 일을 시키기 전에 무엇을 할지 함께 정하는 단계.
4. **실행(Execution)** — 혼자 할지, 부하 파견할지, 팀을 꾸릴지 결정하는 오케스트레이션.
5. **검증(Verification)** — 결과물을 어떻게 믿을 것인가. 기준 설정, 크로스 체크, 눈 달아주기.
6. **개선(Compound)** — 관측하고 패턴을 뽑아 스킬·규칙으로 만드는 지속적 개선.

> 💡 **참고:** Team Attention이 공개한 공식 체크리스트(`team-attention/harness`)는 이 6개 축을 **5개 축 35개 항목**으로 재구성합니다. 계획(Planning)이 실행(Execution)에 흡수되어 "실행 설계"라는 이름으로 묶입니다. 어느 쪽을 쓰든 본질은 같습니다.

### 왜 순환 구조인가

이 여섯 단계는 일직선이 아니라 **순환**입니다. 개선 단계에서 얻은 인사이트가 다시 구조를 다듬는 데 반영되고, 새 구조는 새 맥락을 낳고, 새 맥락은 새 계획을 낳습니다.

이 순환이 돌면 돌수록 AI는 점점 더 잘 일하게 됩니다. 이걸 **Compound Effect**라고 부릅니다. 처음에는 뭘 해야 할지 매번 설명해야 했지만, 한 달 뒤에는 그냥 "처리해줘"만 해도 됩니다. 하네스가 쌓였기 때문입니다.

지금부터 여섯 축을 하나씩 들여다보겠습니다.

## Chapter 3. 구조(Scaffolding) — 한 번 해두면 계속 쓰는 것

### 왜 구조가 품질을 결정하는가

AI에게 "사용자 인증 기능 추가해줘"라고 시켰다고 해봅시다. 두 개의 프로젝트가 있습니다.

**프로젝트 A**: `src/` 폴더 하나에 파일 100개가 평면적으로 나열되어 있습니다. 비즈니스 로직과 유틸리티가 뒤섞여 있고, 테스트는 소스 옆에 산재합니다.

**프로젝트 B**: `src/domain`, `src/services`, `src/infrastructure` 식으로 레이어가 분리되어 있고, `tests/`에 테스트가, `docs/architecture.md`에 아키텍처 문서가 있습니다.

프로젝트 A에서 AI는 추측에 의존합니다. "이 파일에 인증 로직을 넣어도 될까? 아니면 새 파일을 만들까? 테스트는 어디에 쓰지?" 추측은 일관성을 깹니다. 시간이 지날수록 코드베이스는 더 엉망이 됩니다.

프로젝트 B에서 AI는 바로 압니다. "도메인에 User 엔티티, 서비스에 AuthService, 인프라에 JwtProvider, 테스트는 `tests/auth.spec.ts`." 기존 패턴을 복제하면 그만입니다. 새 기능 = 기존 패턴 복제가 되는 순간, AI는 일관된 코드를 생산합니다.

발표자의 표현을 빌리자면:

> **3개월 뒤 AI 품질 — 구조 없음: 30%, 클린 아키텍처: 85%.**

55%p 차이가 어디서 오는가? 모델 차이가 아닙니다. **사람이 한 번 설계해둔 구조** 차이입니다.

### 표준 폴더 구조 — 실전 예시

이호연님이 제안하는 기본 구조는 이렇습니다.

```
my-project/
├── src/           # 비즈니스 로직
├── docs/          # AI의 참고 문서 (사람이 관리)
├── tests/         # 검증 인프라
├── .dev/          # 개발 도구/스크립트 (AI가 남기는 기록)
├── .claude/       # AI 설정
│   ├── rules/     # 상황별 규칙
│   ├── skills/    # 반복 작업 레시피
│   ├── agents/    # 전문가 에이전트
│   └── hooks/     # 안전장치
├── out/           # 빌드 산출물
└── CLAUDE.md      # 프로젝트 지도
```

각 폴더가 명확한 책임을 가집니다. AI에게 "테스트 작성해줘"라고 하면 `tests/`로 갑니다. "문서 업데이트해줘"라고 하면 `docs/`로 갑니다. 설명하지 않아도 됩니다 — 구조가 이미 말하고 있기 때문입니다.

### 사람의 문서 vs AI의 문서 — 반드시 분리하라

여기서 많이 실수하는 지점이 있습니다. `docs/`에 모든 걸 다 넣는 것입니다. 비즈니스 요구사항, API 스펙, 디버깅 기록, 실험 노트가 한 폴더에 뒤섞입니다. 시간이 지나면 무엇이 "진실"이고 무엇이 "흔적"인지 구분이 안 갑니다.

분리하세요. **두 개의 폴더로.**

**`docs/` — 사람이 관리, 비즈니스의 진실**
- 비즈니스 룰과 도메인 정의
- 체크리스트, 온보딩 가이드
- ADR (Architecture Decision Record)
- API 스펙, 외부 연동 규격

**`.dev/` — AI가 남기는 기록, 작업의 흔적**
- learnings (이번 세션에서 새로 알게 된 것)
- troubleshooting 기록
- 작업 로그, 디버깅 히스토리
- 실험 결과, 스크래치패드

왜 분리해야 하나요? **사람이 docs를 관리하는 것을 멈추는 순간, AI도 엉뚱한 맥락으로 일하기 때문입니다.** docs는 사람의 책임입니다. 여기가 오염되면 AI의 결과물 전체가 오염됩니다. .dev는 AI가 자유롭게 쓰게 놔두되, docs는 사람이 지킵니다.

실제 예시로 `team-attention/hoyeon` 저장소를 보면, 이 분리가 엄격하게 지켜집니다. `docs/learnings/lessons-learned.md`에는 팀이 공식화한 교훈이 들어가고, `.dev/specs/{feature}/context/`에는 작업 중 AI가 남긴 learnings, decisions, issues 파일이 개별 저장됩니다.

### AI의 5가지 도구 — 각자의 역할

`.claude/` 하위에 들어가는 다섯 가지 도구 타입이 있습니다. 헷갈리기 쉬우니 명확히 구분해둡시다.

**Skills — 반복 작업의 레시피**
`/commit`, `/review`, `/deploy` 같은 슬래시 명령. "이런 상황에서는 이 순서로 작업하라"는 매뉴얼. 예를 들어 `/bugfix` 스킬은 "진단 → 분석 → 수정 → 검증"의 4단계를 자동으로 밟게 합니다. **한 번 짜두면 팀 전체가 같은 방식으로 일하게 됩니다.**

**Agents — 전문가 팀원**
서브에이전트 파견이나 팀 구성을 위한 전문화된 AI. 예를 들어 `team-attention/hoyeon`에는 역할이 명확히 분리된 9개 에이전트가 있습니다 — `worker`(구현), `gap-analyzer`(누락 탐지), `tradeoff-analyzer`(위험도 평가), `verification-planner`(검증 전략), `reviewer`(검토), `git-master`(커밋) 등. 메인 에이전트가 이들을 불러서 특정 관점의 작업을 위임합니다.

**Plugins — 패키지로 묶어 배포**
위의 스킬/에이전트/훅을 하나로 묶어서 팀이나 커뮤니티에 배포하는 단위. `team-attention/plugins-for-claude-natives`는 9개의 플러그인을 모아둔 마켓플레이스입니다. Claude Code에서 `/plugin marketplace add team-attention/plugins-for-claude-natives`로 추가하고 필요한 것만 설치할 수 있습니다.

**Hooks — 자동 안전장치**
코드 레벨에서 AI의 행동을 가로채는 장치. 네 가지 타입이 있습니다.
- **Pre**: 실행 전 차단 (예: `git push --force` 감지 시 막기)
- **Post**: 실행 후 검사 (예: 커밋 후 테스트 자동 실행)
- **Stop**: 일지 기록 (세션 종료 시 자동 로그)
- **Notification**: 알림 (중요한 작업 완료 시 Slack 전송)

**MCP — 외부 시스템 연결**
DB, Slack, Linear, Jira, Gmail 같은 외부 시스템에 AI가 직접 접근하게 해주는 프로토콜. "티켓 생성해줘"라고 하면 Linear MCP를 통해 실제로 티켓이 만들어집니다.

### 경계 설정 — 이중 안전장치

AI에게 자율성을 주면서도 선을 넘지 않게 하려면 경계가 필요합니다. 세 가지 질문으로 설계하세요.

**1. 뭘 알려줄까 (What should AI know?)**
→ `CLAUDE.md`, `.claude/rules/`에 프로젝트 규칙, 코딩 스타일, 금기사항을 작성.

**2. 어디까지 허용할까 (How much autonomy?)**
→ `settings.json`의 Permission Mode로 설정:
- `plan`: 계획만 세우고 실행은 승인받기
- `auto`: 일정 범위 내 자동 실행
- `bypass`: 모두 허용 (위험, 신뢰 구간에서만)

**3. 뭘 막을까 (What to block?)**
→ `.claude/hooks/`에서 위험한 명령을 실행 전 자동 차단.

핵심은 **이중 안전장치** 원칙입니다. 말로만 경계 설정하지 말고, 코드로도 경계 설정하세요.

예를 들어 이런 식:

```markdown
# CLAUDE.md 일부
## 절대 하지 말 것
- main 브랜치에 직접 push 금지
- .env 파일 수정 금지
- 프로덕션 DB에 직접 쿼리 금지
```

```javascript
// .claude/hooks/pre-bash.js
// git push --force 감지 시 실행 전 차단
if (command.includes('git push') && command.includes('--force')) {
  throw new Error('Force push is not allowed in this project');
}
```

CLAUDE.md의 규칙을 AI가 "이해해서" 안 하는 것이 아니라, Hook이 **강제로** 못 하게 만듭니다. 말로는 뚫려도 코드로는 안 뚫립니다.

### 실전: 어디서부터 시작할까

`team-attention/harness` 플러그인을 설치하면 구조 설계를 도와주는 두 개의 스킬을 바로 쓸 수 있습니다.

```bash
# Claude Code에서 플러그인 마켓플레이스 추가
/plugin marketplace add team-attention/harness

# harness 플러그인 설치
/plugin install harness
```

설치하면 사용 가능한 스킬들:

- **`/scaffold`** — Greenfield 프로젝트에 AI-optimized 하네스 구조를 자동으로 깔아줍니다. CLAUDE.md, rules, 폴더 구조까지 한 번에 생성.
- **`/check-harness`** — 현재 프로젝트의 Harness 성숙도를 **5축 35개 체크리스트**로 진단합니다. L1(시작하기) → L2(내 것으로 만들기) → L3(자율 운영) 3단계 성숙도로 평가.

빈 프로젝트에 `/scaffold` 한 번 돌려보면 왜 구조가 중요한지 5분 안에 체감됩니다. 이미 있는 프로젝트라면 `/check-harness`로 진단해보세요 — 구멍이 보일 겁니다.

스킬을 직접 만들고 싶다면 Anthropic이 제공하는 공식 `/skill-creator`를 쓰세요.

## Chapter 4. 맥락(Context) — AI가 무엇을 알고 일하는가

### 설정 파일의 3단 상속 구조

AI가 작업을 시작할 때 어떤 정보를 들고 들어가느냐가 결과물의 방향을 결정합니다. 이 "들고 가는 정보"를 관리하는 핵심 메커니즘이 **CLAUDE.md의 3단 상속 구조**입니다.

```
USER     ~/.claude/CLAUDE.md         나만 적용 · 모든 프로젝트 공통
   ↓ 상속
PROJECT  my-app/CLAUDE.md            팀 공유 · Git 커밋, 이 프로젝트 전용
   ↓ 상속 + 오버라이드
FOLDER   src/auth/CLAUDE.md          이 폴더 작업 시만 (특수 규칙)
```

핵심 원리: **하위가 상위를 덮어쓴다. 가장 좁은 범위가 우선.**

구체적인 예로 보겠습니다.

- User 레벨: "나는 camelCase를 선호한다"
- Project 레벨: "이 프로젝트는 snake_case를 쓴다"
- Folder 레벨 (`src/auth/`): "auth 모듈은 PascalCase를 쓴다 (기존 코드 일관성)"

AI가 `src/auth/`에서 작업하면: **PascalCase**.
`src/payment/`에서 작업하면: **snake_case** (프로젝트 기본).
개인 사이드 프로젝트에서 작업하면: **camelCase** (User 기본).

이 구조가 우아한 이유는, 규칙을 **가장 적절한 위치에 둘 수 있기 때문**입니다. 모두에게 적용해야 할 규칙은 User에, 팀 전체 규칙은 Project에, 특정 모듈의 예외는 Folder에. 중복도 없고 충돌도 없습니다.

### CLAUDE.md 실전 가이드

각 레벨에 무엇을 쓸지 감을 잡기 어려울 수 있습니다. 원칙은 이렇습니다.

**User 레벨 (`~/.claude/CLAUDE.md`)** — 내 작업 습관

```markdown
# 개인 작업 스타일
- 응답은 한국어로
- 코드 주석은 영어로
- 불필요한 설명 최소화, 핵심만
- 실수가 있으면 먼저 인정하고 고치기
```

**Project 레벨 (`my-app/CLAUDE.md`)** — 이 프로젝트의 기술 스택, 컨벤션, 중요한 제약

```markdown
# Puree Platform

## 스택
- Next.js 15 (App Router), TypeScript, Shadcn/ui
- 상태: Zustand, 서버 상태: React Query
- 배포: Firebase App Hosting

## 컨벤션
- 컴포넌트: PascalCase, 훅: useCamelCase, 유틸: camelCase
- API는 반드시 React Query로 래핑
- 스타일링: Tailwind only (CSS 파일 금지)

## 절대 하지 말 것
- main 브랜치 직접 push 금지
- 테스트 없는 API 함수 추가 금지
- Staging API 직접 호출 금지 (반드시 프록시 경유)

## 참고 문서
- 아키텍처: docs/architecture.md
- 디자인 시스템: docs/design-system.md
```

**Folder 레벨 (`src/auth/CLAUDE.md`)** — 특정 모듈의 특수한 규칙

```markdown
# auth 모듈

- 모든 토큰 검증은 JwtTokenProvider 경유
- 세션은 Redis에 저장 (로컬 메모리 금지)
- 에러는 AuthError 클래스로 감싸서 반환
```

여기서 가장 중요한 원칙 하나:

> **최대 200줄 정도를 유지하며 계속 업데이트하라. 너무 길어지면 AI 성능이 급격히 저하된다.**

CLAUDE.md가 500줄이 넘기 시작하면 AI가 핵심을 놓치기 시작합니다. 길어지는 순간 Progressive Disclosure로 분리해야 합니다.

### Progressive Disclosure — 필요할 때 필요한 것만

인간 엔지니어가 회사의 모든 문서를 머리에 담고 일하지 않듯이, AI도 **지금 이 작업에 필요한 정보**만 필요합니다. 테스트 작성하는데 배포 체크리스트까지 들고 올 필요 없습니다.

해결 방법은 의외로 단순합니다. **SKILL/CLAUDE.md에 모든 내용을 넣지 말고, "이 상황에서는 이걸 참고해"라고 포인터만 두세요.**

```markdown
# CLAUDE.md (핵심만 30줄)

프로젝트 개요와 절대 규칙만 여기.

## 상황별 참고 문서
- 코드 작성 시 → references/code-style.md
- 테스트 작성 시 → references/testing-guide.md
- API 설계 시 → references/api-convention.md
- 배포 준비 시 → references/deploy-checklist.md
```

폴더 구조로 이렇게 됩니다:

```
my-skill/
├── SKILL.md                    # 30~50줄
└── references/
    ├── code-style.md          # 필요할 때만 로드
    ├── testing-guide.md
    ├── api-convention.md
    └── deploy-checklist.md
```

AI는 코드 작성 작업에서는 `code-style.md`만 읽습니다. 테스트 작업에서는 `testing-guide.md`만 읽습니다. 컨텍스트를 절약하면서 동시에 필요한 정보는 놓치지 않는 구조입니다.

이게 **Progressive Disclosure**입니다. 정보를 점진적으로 드러냅니다. 한꺼번에 다 주지 않습니다.

> **"한꺼번에 다 주면 AI도 헷갈린다."**

### glob 패턴으로 조건부 로드

`.claude/rules/` 폴더는 Progressive Disclosure를 한 단계 자동화해줍니다. 규칙 파일 상단에 **glob 패턴**을 지정하면, 해당 패턴에 매칭되는 파일을 작업할 때만 자동으로 로드됩니다.

```
.claude/rules/
├── code-style.md     # glob 없음 → 항상 로드
├── testing.md        # glob 없음 → 항상 로드
├── react-*.md        # glob: **/*.tsx → tsx 작업 시만
├── api-design.md     # glob: **/routes/** → API 라우트 작업 시만
└── security.md       # glob: **/*.sql, **/auth/** → 보안 민감 영역만
```

실제 규칙 파일의 상단은 이렇게 생겼습니다.

```markdown
---
glob: ["**/*.sql", "**/auth/**"]
---

# 보안 규칙

- .env 직접 수정 금지
- SQL은 반드시 parameterized query 사용
- 사용자 입력 검증 없이 DB에 전달 금지
- auth 관련 에러 메시지에 내부 정보 노출 금지
```

AI가 `src/utils/helper.ts`를 편집할 때는 이 파일이 로드되지 않습니다. `src/auth/login.ts`를 편집할 때는 자동으로 로드됩니다. 규칙은 세밀해지는데 컨텍스트는 오히려 가벼워집니다.

### 세션 관리 — 쌓이면 비워라

아무리 잘 설계해도 한 세션을 오래 쓰면 컨텍스트는 쌓입니다. 그리고 쌓이면 AI가 둔해집니다.

실전 기준은 이렇습니다.

| 컨텍스트 사용량 | 상태 | 권장 액션 |
|---|---|---|
| ~20% | 🟢 쾌적 | 계속 진행 |
| ~50% | 🟡 중간 | `/compact` 고려 |
| ~80% | 🔴 과부하 | `/clear` 또는 새 세션 |

세 가지 도구를 상황에 맞게 씁니다.

**`/clear`** — 컨텍스트 완전 초기화. 다른 주제로 전환할 때.
오전에 인증 로직 작업하다가 오후에 디자인 시스템 작업으로 넘어간다? `/clear`. 서로 다른 맥락이 섞이면 AI가 엉뚱한 판단을 합니다.

**`/compact`** — 오래된 대화를 요약·압축. 같은 주제를 이어갈 때.
같은 리팩토링 작업을 계속 하는데 대화가 길어졌다? `/compact`. 앞부분의 세부 대화는 버리고 핵심 결정과 현재 상태만 남깁니다.

**`handoff`** — 현재 세션의 맥락을 파일로 저장 → 새 세션에서 이어받기.
내일 이어서 할 복잡한 작업이다? 현재 맥락을 `handoff.md` 파일로 떨어뜨리고, 내일 새 세션에서 그 파일을 읽게 합니다. 세션 간 맥락 손실 없이 전환.

이호연님의 실전 기준: **20~30% 쯤에서 새로 시작**. 아예 다른 맥락이면 `/clear`, 이어가야 하면 `handoff`. 50%, 80%까지 참지 않습니다. 빨리 비울수록 좋습니다.

### 주기적 점검 — 쌓기만 하지 말고 비우기도

마지막 주제는 정리입니다. Skill과 MCP는 의외로 컨텍스트를 많이 먹습니다. 안 쓰는 Skill, 쓸 일 없는 MCP가 주렁주렁 달려있으면 **context pollution**이 생깁니다.

주기적으로 `/context` 명령으로 현재 사용량을 카테고리별로 확인하세요.

```
Context Usage                            Opus 4.6 (1M context)
                                         claude-opus-4-6
                                         61.2k/1m tokens (6%)

Estimated usage by category
 System prompt: 6.3k tokens (0.6%)
 System tools: 7.2k tokens (0.7%)
 Custom agents: 3.6k tokens (0.4%)
 Memory files: 473 tokens (0.0%)
 Skills: 7.3k tokens (0.7%)
 Messages: 41.2k tokens (4.1%)
 Free space: 900.8k tokens (90.1%)
 Autocompact buffer: 33k tokens (3.3%)
```

Skills가 비정상적으로 큰데 거의 안 쓴다? 정리 대상입니다. Custom agents가 쌓여있는데 이번 달에 한 번도 안 불렀다? 정리 대상입니다.

CLAUDE.md가 비대해졌다고 느끼면 AI 본인에게 진단을 시키는 방법이 있습니다.

> "CLAUDE.md, .claude/rules를 분석해서 논리적으로 문제가 없는지 + 중복되거나 대치되는 내용은 없는지 분석해줘"

의외로 많이 잡힙니다. **쌓는 것만큼 점검하고 비우는 것도 Context Engineering입니다.**

## Chapter 5. 계획(Planning) — 일을 시키기 전에 먼저 정하기

### "해줘"의 함정

구조와 맥락을 잘 깔았다고 해도, 일을 시킬 때 실수하기 쉬운 지점이 하나 더 있습니다. 바로 **"해줘"로 시작하는 것**입니다.

"결제 시스템 리팩토링해줘." 이 한 문장으로 AI에게 작업을 시키면 무슨 일이 벌어질까요?

1. AI가 알아서 만듭니다. (자기 나름대로 해석해서)
2. 결과를 보니 엉뚱한 방향입니다. "아닌데..."
3. 다시 시킵니다. "PG사 연동은 그대로 두고 내부 구조만 개선해줘."
4. 또 엉뚱합니다. "DB 스키마는 건드리지 마."
5. 반복... 시간만 날립니다.

왜 이럴까요? **사람은 자기 머릿속의 전제를 프롬프트에 다 담지 못하기 때문입니다.**

"결제 시스템 리팩토링"이라는 말 뒤에는 수많은 암묵적 가정이 있습니다. 현재 PG사 연동은 유지한다, DB 스키마는 건드리지 않는다, 기존 테스트는 통과해야 한다, 성능을 해치면 안 된다... 이걸 다 쓰자니 번거롭고, 안 쓰자니 AI는 모릅니다. 그 결과 "엉뚱한 결과 → 수정 → 다시" 사이클이 반복됩니다.

해결책은 단순합니다. **계획과 실행을 분리하세요.**

### 같이 계획부터 세우기

"해줘" 대신 이렇게 말해보세요.

> "결제 시스템 리팩토링할 건데, 같이 계획 세워보자."

그러면 흐름이 바뀝니다.

1. AI가 계획 초안을 작성합니다. "현재 구조 분석 → 개선 포인트 → 단계별 리팩토링 순서 → 검증 방법."
2. 사람이 검토합니다. "여기는 PG사 연동 유지한다고 명시해줘."
3. AI가 수정합니다.
4. 사람이 승인합니다.
5. 승인된 계획대로 AI가 실행합니다.
6. 검수할 때 기준이 명확하니 빠르게 통과합니다.

검수 횟수가 줄어들고, 결과의 예측 가능성이 올라갑니다. **계획 단계에서의 10분이 실행 단계에서의 1시간을 절약합니다.**

### AskUserQuestion 패턴

계획을 세울 때 가장 강력한 기법 하나를 소개합니다. **AI가 사람에게 질문하게 만드는 것**입니다.

사람은 자기 전제를 다 쏟아내지 못하지만, AI는 적절한 질문을 던지면 그 전제를 끌어낼 수 있습니다. 이걸 활용하는 프롬프트 패턴입니다.

```
나 결제 시스템을 리팩토링하려고 해.
너가 지금 이해한 게 뭔지 정리해주고,
모호한 점이 없도록 내게 계속 질문해서 명확하게 해줘.
```

이 한 문장이 대화의 방향을 완전히 바꿉니다.

```
AI: 이해한 것: 결제 모듈의 기존 구조를 개선하려는 것.

    질문 1: 현재 PG사 연동이 단일인가요, 멀티인가요?
    질문 2: 구독 결제도 포함되나요?
    질문 3: DB 스키마 변경도 허용 범위인가요?

사용자: 멀티 PG, 구독 포함, DB 스키마는 건드리지 마.

AI: 명확해졌습니다. 그러면 계획을 세우겠습니다...
```

이 패턴의 진짜 가치는 **unknown-unknown을 발굴한다는 것**입니다. 내가 놓치고 있는지조차 몰랐던 고려사항이 AI의 질문을 통해 드러납니다.

### 커스텀 Plan 스킬 — `/specify`와 `/deep-interview`

Claude Code에는 Plan Mode가 기본 내장되어 있습니다. 하지만 프로젝트가 복잡해지면 기본 Plan Mode로 부족한 지점이 옵니다. 이때 **커스텀 Plan 스킬**을 만듭니다.

Team Attention의 `harness` 플러그인에 포함된 `/specify` 스킬은 이런 흐름을 구현합니다.

```
1. 목표 확인     — 사용자의 의도를 미러링해서 정리
       ↓
2. 인터뷰        — AskUserQuestion으로 모호한 지점 해소
       ↓
3. 요구사항 도출 — 기능 요구사항 + 비기능 요구사항 분리
       ↓
4. 태스크 분해   — 작업 단위로 쪼개기, 의존성 명시
       ↓
5. spec.md 파일  — 구조화된 구현 계획으로 저장
       ↓
   → /execute 명령으로 플랜 기반 실행
```

핵심 전환은 **implicit(머릿속) → explicit(플랜 파일)**입니다. 머릿속에만 있던 계획이 파일로 떨어지는 순간, 여러 이점이 생깁니다.

- 나중에 다시 볼 수 있다
- 팀원과 공유할 수 있다
- AI 세션이 바뀌어도 이어받을 수 있다
- 검증 기준이 된다

### 계획 단계를 돕는 도구들 — 실제 저장소 기준

세 가지 스킬·플러그인이 각기 다른 단계를 담당합니다.

**`/specify` (harness 플러그인)**
목표를 구조화된 구현 계획(`spec.md`)으로 변환. 인터뷰 기반 계획 수립. 단순한 초안 단계부터 시작하기 좋습니다.

```bash
/specify "사용자 인증 시스템 구현"
```

**`/deep-interview` (harness 플러그인)**
Socratic 방식의 깊이 있는 요구사항 인터뷰. **Ambiguity Score 기반**으로 모호함을 정량적으로 측정하며 질문을 이어갑니다. unknown-unknown을 줄이는 데 특화되어 있습니다.

```bash
/deep-interview "주제"
```

**`/clarify` (plugins-for-claude-natives 플러그인)**
모호한 요구사항을 정확한 스펙으로 변환. 객관식 질문으로 애매함을 해소한 뒤 Before/After 비교를 보여주는 게 특징입니다.

```
Before: "Add a login feature"
After:  Goal: Add username/password login with self-registration.
        Scope: Login, logout, registration, password reset.
        Constraints: 24h session, bcrypt, rate limit 5 attempts.
```

공통점은 모두 **모호함을 줄이는 것**입니다.

> **계획 단계에서 모호함을 줄이는 것이 실행 품질을 결정한다.**

## Chapter 6. 실행(Orchestration) — 혼자 vs 파견 vs 팀

### 3가지 실행 패턴

계획이 서면 이제 실행입니다. 여기서의 핵심 질문은 하나입니다. **"이 작업을 누가 할 것인가?"**

선택지는 세 가지입니다.

**1. 혼자 (Single)** — AI 하나가 처음부터 끝까지
구조: `AI → 결과물`
용도: 단순 작업, 대부분의 일상 코딩
비용: 1x

**2. 부하 파견 (Subagent)** — 메인이 서브에이전트에게 위임
구조: `메인 → (조사 / 작업 / 검증) → 메인이 종합`
용도: 병렬 처리, 전문화된 관점이 필요할 때
비용: ~2x

**3. 팀 협업 (Team Mode)** — 여러 에이전트가 서로 소통
구조: `PM ↔ 개발 ↔ 디자인 ↔ QA` (상호 소통)
용도: 다관점이 필요한 복잡한 작업
비용: **~7x**

비용 차이를 보세요. Team Mode는 토큰이 약 7배 듭니다. **그래서 90%의 경우는 Single이나 Subagent로 충분합니다.** Team Mode는 정말 다관점 소통이 필요한 경우에만 꺼내는 카드입니다.

### 상황별 오케스트레이션 — 실전 예시

**순차 파이프라인 (Single + TaskCreate)**
> 상황: 블로그 글 작성. 조사 → 초안 → 퇴고 순서로 가야 한다.
> 프롬프트: "AI 에이전트 트렌드 조사해서 → 초안 작성하고 → 퇴고까지 순서대로 진행해줘"
> 동작: TaskCreate로 체크박스 3개 생성, 하나씩 직렬 수행. 이전 단계 결과물을 다음 단계 입력으로.

**병렬 Subagent**
> 상황: 경쟁사 3곳의 랜딩 페이지를 동시 분석. 서로 독립적.
> 프롬프트: "A사, B사, C사 랜딩 페이지를 각각 에이전트 파견해서 동시에 분석해줘. 끝나면 비교표 만들어."
> 동작: Agent 3개 spawn, 병렬 분석, 메인이 결과 취합.

**Team Mode (Orchestrator-Worker 패턴)**
> 상황: 새 기능을 설계하면서 동시에 구현하고 리뷰까지 받아야 한다. 에이전트끼리 소통이 필요.
> 프롬프트: "Team 켜서 설계자, 구현자, 리뷰어 3명으로 팀 꾸려줘. 설계 나오면 구현자가 바로 시작해."

`team-attention/hoyeon`의 `/execute` 스킬이 이 Orchestrator-Worker 패턴을 정교하게 구현한 예입니다.

```
Orchestrator (reads PLAN.md)
  ├── Parse TODOs → Create Tasks with dependencies
  ├── Parallelize non-blocked Tasks
  └── For each TODO:
      ├── Worker agent (implementation)
      ├── Verify (3 checks: functional, static, runtime)
      ├── Context save (learnings, decisions, issues)
      └── git-master (atomic commit)
```

핵심 규칙 세 가지를 주목하세요.
- **Orchestrator는 코드를 쓰지 않는다** — 위임하고 검증만 한다
- **플랜 체크박스(`### [x] TODO N:`)가 단일 진실 공급원** — AI가 마음대로 해석하지 않는다
- **실패한 태스크는 최대 3번 재시도** — 반드시 성공해야 함. 3번 내에 안 되면 reconciliation.

핵심은 **작업의 성격에 맞는 패턴을 고르는 것**입니다. 독립적 병렬 작업을 Team Mode로 돌리면 토큰 낭비입니다. 복잡한 크로스 피드백이 필요한 작업을 Single로 돌리면 품질이 떨어집니다.

### Ralph Loop — 될 때까지 반복

실행 단계의 가장 강력한 패턴을 하나 소개합니다. **Ralph Loop**입니다.

아이디어는 간단합니다. "뭐가 되면 끝인지"만 정해주면, AI가 알아서 될 때까지 계속 돕니다.

구조:
```
1. 완료 기준 합의 (사람이 한 번만)
       ↓
2. AI가 작업
       ↓
3. 기준 충족?
      ├─ NO → 2번으로 돌아가기
      └─ YES → PASS
```

예시를 보겠습니다. "랜딩 페이지 만들어줘"를 Ralph Loop로 돌리면:

```
완료 기준: 모바일 반응형 + Lighthouse 90점 이상 + 카피 3번 이상 퇴고

시도 1: 페이지 완성, Lighthouse 72점 → 기준 미달 → 다시
시도 2: 92점 달성, 그러나 모바일 레이아웃 깨짐 → 기준 미달 → 다시
시도 3: 레이아웃 수정, 카피는 2회만 퇴고 → 기준 미달 → 다시
시도 4: 전 항목 충족 → PASS
```

사람이 한 일은? **기준 정하기 1번.** 나머지는 AI가 알아서 돌립니다.

이 패턴이 강력한 이유는 **비동기성**입니다. 기준을 정하고 Ralph Loop를 시작하면, 커피 마시러 갔다 와도 됩니다. AI가 혼자 4번 시도하고 기준을 충족한 결과물을 내놓습니다. 사람은 "뭐가 되면 끝인가"에만 집중하면 됩니다.

> **"뭐가 되면 끝인지"만 정해주면 AI가 될 때까지 계속 돈다.**

### Auto Research — 자율 실험 루프

Ralph Loop의 확장판이 **Auto Research**입니다. Andrej Karpathy가 공개한 `karpathy/autoresearch` 저장소가 대표적 예시입니다.

아이디어: ML 연구 같은 실험 중심 작업에서, 사람은 방향만 정하고 AI가 밤새 실험을 돌린다.

흐름:
```
코드 수정(train.py) → 실행(5분 학습) → 평가(성능 비교) → 판단(유지/폐기)
         └──────────────────── 자동 반복 ────────────────────┘
```

세팅:
1. `program.md`에 연구 방향 작성 (예: "이 모델의 validation loss를 낮추는 실험을 진행한다")
2. AI가 `train.py`만 수정 (아키텍처나 하이퍼파라미터 변경)
3. 학습 후 성능 비교
4. 개선되면 유지, 아니면 폐기 → 다음 실험

결과: 시간당 ~12개의 실험을 자율적으로 수행. 밤새 무인 운영.

이게 왜 혁명적인가요? 연구자가 실험을 하나씩 수동으로 돌리면 하루 10개가 한계입니다. Auto Research는 하룻밤에 100개 이상을 돌릴 수 있습니다. **탐색 공간이 10배 넓어집니다.**

핵심 패턴은 `수정 → 실행 → 평가 → 반복`입니다. 이 루프에 자율성을 부여하면 됩니다. 사람은 `program.md` 하나만 잘 쓰면 됩니다. 이게 스킬이 하는 역할과 정확히 같습니다. **방향을 코드화하는 것.**

### `/ultrawork` — 전체 파이프라인 자동화

`team-attention/hoyeon`이 제공하는 궁극의 실행 패턴은 `/ultrawork`입니다. Plan → PR 생성 → 구현까지 전체 파이프라인을 한 줄로 돌립니다.

```bash
/ultrawork feature-name
  → /specify (interview + plan)
  → /open (create Draft PR)
  → /execute (implement all TODOs)
```

Stop hook이 각 단계 종료를 감지해서 자동으로 다음 단계를 트리거합니다. 사람은 `/ultrawork feature-name` 한 줄만 입력하고, 중간 HITL(Human-in-the-Loop) 체크포인트 9개에서만 개입하면 됩니다.

이런 도구의 핵심 통찰은 이것입니다. **자동화할 수 있는 전환을 수동으로 돌리는 것이 가장 큰 낭비다.** `/specify` 끝나면 `/open`을 자동으로 호출하면 되는데, 왜 사람이 매번 타이핑하나요? Hook이 해결합니다.

## Chapter 7. 검증(Verification) — 어떻게 믿을 것인가

### 원칙 1: 기준이 있어야 검증이 가능하다

"AI한테 맡겼는데 대충 끝냈다고 하는데, 실제로 보니 엉성해요."

이 호소를 자주 듣습니다. 해결책은 간단합니다. **작업 전에 "뭘 만들고 어떻게 검증할지"를 합의하세요.** 이걸 **Sprint Contract**라고 부릅니다.

기준 없이 시키면 두 가지 중 하나가 벌어집니다.
1. AI가 끝없이 작업합니다 (어디까지 가야 할지 모름)
2. 대충 끝내고 끝났다고 합니다 (기준이 없으니까)

Sprint Contract의 세 가지 요소:

**1. 완료 조건을 먼저 정한다**
"이 3가지가 되면 끝" — 시작 전에 합의. 예:
- 테스트 커버리지 80% 이상
- Lighthouse 성능 점수 90 이상
- 피드백 3회 이상 반영

**2. 조건을 측정 가능하게 쓴다**
❌ "잘 되게 해줘"
✅ "테스트 통과 + 빌드 성공 + ESLint 에러 0개"

측정 가능한 조건이어야 AI가 스스로 판단할 수 있습니다. 주관적 조건은 사람이 매번 개입해야 합니다.

**3. 미달이면 다시 돌린다**
기준이 있으니 자동 반복이 가능해집니다. 바로 앞 챕터의 Ralph Loop와 연결되는 지점입니다. **명확한 기준 = 자동화의 조건**입니다.

`team-attention/hoyeon`은 이 원칙을 **A-items / H-items**로 구체화합니다.

- **A-items** (Agent 자동 검증): AI가 자동으로 확인할 수 있는 항목 — 테스트 통과, 빌드 성공, typecheck, lint
- **H-items** (Human 확인 필요): 사람이 눈으로 봐야 하는 항목 — UX 흐름, 디자인 일관성, 비즈니스 판단

Plan 최상단의 Verification Summary 섹션에 이 두 카테고리를 명시합니다. 이렇게 하면 A-items는 Ralph Loop로 자동 검증되고, H-items만 사람이 개입하면 됩니다.

### 원칙 2: Generator와 Evaluator를 분리하라

여기가 검증의 핵심 중 핵심입니다. Anthropic이 공식 문서에서 반복해서 강조하는 원칙입니다.

> "자기 작업을 평가하면, quality가 mediocre해도 자신있게 칭찬한다."

AI에게 자기가 쓴 코드를 리뷰시키면, 웬만하면 "훌륭합니다"라고 답합니다. 왜? **같은 컨텍스트 안에서 만들고 평가하기 때문입니다.** 자기가 내린 결정에 대한 합리화 바이어스가 발동합니다.

해결책은 **Generator와 Evaluator를 물리적으로 분리하는 것**입니다.

- **Generator**: 결과물을 만드는 AI. 자기 일을 열심히 한다.
- **Evaluator**: 결과물을 평가하는 AI. 비판적으로 본다.

두 에이전트가 **다른 세션, 다른 컨텍스트**에서 돕니다. Evaluator는 Generator가 어떤 고민을 했는지, 어떤 제약 하에서 결정했는지 모릅니다. 결과물만 보고 판단합니다. 그래서 냉정하게 지적할 수 있습니다.

Anthropic은 이걸 이렇게 표현합니다:

> "만든 AI와 확인하는 AI를 분리하는 것이 **가장 강력한 레버**다."
> "Evaluator를 회의적으로 튜닝하는 게, Generator를 자기비판적으로 만드는 것보다 **훨씬 쉽다**."

두 번째 문장이 특히 중요합니다. Generator에게 "자기 비판적으로 봐"라고 시키는 것은 효과가 약합니다. Evaluator를 따로 두고 "회의적으로 판단해"라고 시키는 게 훨씬 쉽고 강력합니다.

실제로 `team-attention/hoyeon`은 이 원칙을 에이전트 설계로 구현합니다. `worker`(Sonnet)가 구현을 담당하고, `reviewer`(Opus)가 최종 검토를 맡습니다. 두 에이전트는 서로 다른 모델을 쓸 뿐만 아니라, 서로 다른 컨텍스트에서 작동합니다.

### 원칙 3: 모델도 역할도 나눈다

한 걸음 더 나아가 **모델 자체를 분리**할 수 있습니다.

| 역할 | 추천 모델 | 이유 |
|---|---|---|
| 코드 리뷰 | Codex | 로직 오류, 보안 취약점, 테스트 누락 검출에 강함 |
| 문서 리뷰 | Gemini | 일관성, 정확성, 구조 검증에 강함 |
| 복잡한 판단 | Opus | 아키텍처 검토, 긴 맥락 통합 |
| 빠른 확인 | Sonnet | 반복 검증에 비용 효율적 |

서로 다른 모델은 서로 다른 맹점을 가집니다. 한 모델이 놓친 것을 다른 모델이 잡아냅니다. **크로스 체크**입니다.

이 원칙을 극단까지 밀어붙인 플러그인이 `plugins-for-claude-natives`의 **`agent-council`**입니다. "summon the council"이라고 말하면 Gemini CLI, GPT, Codex를 병렬로 호출해서 각자의 의견을 받은 뒤 Claude가 합의안을 합성해줍니다.

```
# 사용 예
User: "summon the council - should I use TypeScript or JavaScript for my new project?"
```

한 가지 냉정한 사실을 덧붙이면:

> "Out of the box, Claude is a poor QA agent."

박스에서 꺼낸 그대로의 Claude는 형편없는 QA 에이전트입니다. Generator로는 훌륭하지만 Evaluator로는 느슨합니다. 그래서 **검증 에이전트도 튜닝이 필요**합니다. 기준을 구체적으로, 회의적으로 설정해야 쓸 수 있습니다.

예를 들어 이런 식으로 Evaluator 프롬프트를 세팅합니다.

```markdown
# evaluator role

너는 코드 리뷰어다. 다음 원칙을 따른다:

1. **기본 자세는 의심이다.** 코드가 일단 작동하는 것처럼 보여도 엣지 케이스를 먼저 생각한다.
2. **"잘 된 것 같다"는 답변은 금지.** 구체적인 근거 없이 긍정적인 평가를 내리지 않는다.
3. **보안/성능/가독성/테스트 4가지 관점**에서 각각 지적한다. 지적할 게 없으면 그 관점은 건너뛴다.
4. **판단 근거는 항상 코드의 구체적인 라인**에서 온다.
```

### 원칙 4: 에이전트에게 눈을 달아주기

코드 리뷰만으로는 부족한 경우가 많습니다. 특히 UI 작업. 코드가 아무리 깨끗해도 실제로 렌더링했을 때 레이아웃이 깨져 있을 수 있습니다.

AI에게 **눈**을 달아줘야 합니다. 세 가지 방법이 있습니다.

**Browser Agent (chrome-cdp, agent-browser)**
실제 Chrome 브라우저를 제어합니다. DOM 탐색, 클릭, 스크린샷, 네비게이션. 웹앱의 UX를 직접 검증할 수 있습니다. "로그인 버튼을 눌러서 에러가 안 나는지 확인해줘" 같은 지시가 가능해집니다.

**Computer Use (built-in MCP)**
스크린샷 + 마우스/키보드로 모든 앱을 제어합니다. 웹이 아닌 네이티브 앱, Figma 같은 디자인 도구까지 검증 가능. 범위가 Browser Agent보다 넓지만 정확도는 조금 떨어집니다.

**시각 검증 루프**
패턴 자체를 만듭니다.
```
만든다 → 스크린샷 찍는다 → 보고 판단한다 → 수정한다
         (generate → screenshot → evaluate)
```

이 루프는 사람이 눈으로 하는 일을 AI가 대신하는 것입니다. 디자인 검수, 레이아웃 확인, 시각적 회귀 테스트 같은 작업이 자동화됩니다.

한발 더 나아가, `plugins-for-claude-natives`의 **`interactive-review`** 플러그인을 쓰면 Claude가 만든 계획·문서를 웹 UI에서 체크박스와 코멘트로 검토할 수 있습니다. 터미널에서 긴 마크다운을 읽는 대신 브라우저에서 시각적으로 리뷰하고, 결과를 구조화된 피드백으로 Claude에게 되돌려줍니다.

> **검증 범위를 코드에서 화면까지 확장하면 사람이 끼어들 일이 줄어든다.**

### 안전장치 3종 — 실수해도 괜찮은 구조

마지막으로, 검증이 실패했을 때를 대비한 안전장치입니다. 이건 **실수를 막는 게 아니라, 실수해도 괜찮은 구조를 만드는 것**입니다.

**1. 되돌릴 수 있는 환경**
브랜치나 Worktree로 격리된 환경에서 작업하세요. AI가 아무리 실수해도 메인은 안전합니다.

```bash
git worktree add ../my-project-ai-branch feature/ai-task
# AI는 이 워크트리에서만 작업
# 메인은 건드리지 못함
```

`team-attention/hoyeon`은 이를 `hy` CLI로 자동화합니다.

```bash
hy create auth         # 워크트리 생성 + spec 이동
hy go auth             # 워크트리로 이동 + Claude 실행
hy status              # 모든 워크트리 진행률 + PR 상태
hy cleanup auth        # 워크트리 제거 + 브랜치 정리
```

여러 기능을 병렬로 작업할 때 워크트리별로 독립된 Claude 세션을 띄울 수 있어서, 한 세션의 실패가 다른 세션에 영향을 주지 않습니다.

**2. 위험한 건 사람이 확인**
삭제, 배포, 외부 발송 같은 비가역 작업은 **반드시 사람 승인 후 실행**. Runtime Gate 패턴입니다. AI가 "rm -rf"를 치려고 하면 Hook이 가로채서 사람 승인을 요청합니다.

hoyeon의 계획 단계는 9개의 HITL 체크포인트로 이를 강제합니다. HIGH risk 결정(DB 스키마, 인증, breaking API)은 반드시 사용자 승인 + rollback 전략을 포함해야 합니다.

**3. Dry-run 먼저**
"이렇게 할 건데 맞나?" 미리보기 후 실행. DB 마이그레이션, 대량 리팩토링 같은 작업은 실행 전에 영향 범위를 시뮬레이션합니다. 확인 후 실제 실행.

> **"실수해도 괜찮은 구조"를 만드는 것이 핵심.**

실수를 0으로 만들려는 시도는 실패합니다. AI도, 사람도 실수합니다. 중요한 건 **실수의 비용을 낮추는 구조**입니다. 되돌릴 수 있으면 실수는 학습 기회가 됩니다.

## Chapter 8. 개선(Compound) — 어떻게 계속 나아질 것인가

### 작업 → 관측 → 패턴 발견 → 개선

이제 마지막 축입니다. 개선. 여기서 마법이 일어납니다.

지금까지 잘 설계된 하네스가 있다고 해도, 시간이 지나면 자연스럽게 마모됩니다. 비즈니스 요구사항이 바뀌고, 코드베이스가 진화하고, AI 모델이 업그레이드됩니다. 한 번 설계하고 끝이 아닙니다. **돌려야 합니다.**

기본 사이클:
```
작업 → 관측(세션+사용 패턴) → 패턴 발견 → Skill 또는 Rule → 더 나은 작업
```

그리고 이 사이클을 돌리는 두 가지 간단한 법칙이 있습니다.

### 3번 법칙

**법칙 1: 같은 작업을 3번 반복하면 → Skill로 만든다**

"어? 이거 저번 주에도 했던 작업인데" 싶은 순간이 있습니다. 한 번은 우연이고, 두 번은 징조이고, **세 번은 패턴**입니다. 세 번째에 멈추세요. 스킬로 만들 타이밍입니다.

예를 들어 PR 만들 때마다 이런 체크리스트를 돌린다면:
- 코드 변경사항 요약
- 테스트 결과 첨부
- 관련 이슈 링크
- 리뷰어 태그

세 번째에 `/pr-checklist` 스킬로 만드세요. 이후로는 한 줄이면 됩니다.

**법칙 2: 같은 실수를 3번 반복하면 → Rule로 만든다**

AI가 자꾸 같은 실수를 반복한다면, 그건 **AI의 문제가 아니라 규칙이 없는 것**입니다. 세 번째에 `.claude/rules/`에 규칙을 추가하거나 CLAUDE.md에 명시하세요.

예를 들어 AI가 자꾸 `any` 타입을 남발한다면:

```markdown
# .claude/rules/typescript.md
---
glob: ["**/*.ts", "**/*.tsx"]
---

## 타입 규칙
- `any` 타입 사용 금지. 정 모르면 `unknown` 사용 후 타입 가드로 좁히기.
- 외부 라이브러리 타입이 부족하면 별도 `.d.ts` 파일로 정의
```

이후로는 같은 실수를 안 합니다. **실수가 규칙이 되는 순간, 개선 루프가 돌기 시작합니다.**

### Skill 개선 루프

스킬도 한 번 만들고 끝이 아닙니다. 쓰다 보면 부족한 점이 보입니다. 루프:

```
만들기 → 사용 → 세션 분석 → 병목 발견 → 개선 → 다시 사용
```

스킬 자체를 계속 다듬습니다. 3개월 전에 만든 스킬과 지금 스킬이 같다면, 그건 제대로 쓰고 있지 않다는 신호일 수 있습니다.

### 단순화 — 좋은 하네스는 작아진다

여기서 반직관적인 메시지가 하나 나옵니다.

> **좋은 Harness는 점점 단순해진다.**

대부분의 시스템은 시간이 지날수록 복잡해집니다. 이것저것 추가되니까요. 하지만 Harness는 반대여야 합니다.

세 가지 신호에 주의하세요.

**신호 1: 안 쓰는 건 치운다**
한 달 동안 한 번도 안 쓴 Skill, MCP, Rule은 바로 삭제하세요. 쌓이면 AI slop이 됩니다. 컨텍스트 공간을 차지하고, AI의 판단을 흐립니다.

**신호 2: 모델이 좋아지면 Harness를 재평가한다**
예전에는 반드시 필요했던 가드레일이, 새 모델에서는 불필요할 수 있습니다. GPT-4 시절에 필요했던 장황한 예시가 Claude 4.6에서는 오히려 방해가 될 수 있습니다. **모델이 업그레이드되면 하네스도 다이어트시키세요.**

**신호 3: 과설계 신호를 인식한다**
Skill이 30개가 넘어간다? 규칙 파일이 50개를 넘는다? 뭔가 잘못된 것입니다. **과설계**입니다. 일상 작업에 쓰는 스킬은 사실 5~10개면 충분합니다. 나머지는 쌓인 쓰레기일 가능성이 높습니다.

Anthropic의 표현을 빌리자면:

> **"Harness의 공간은 모델이 좋아져도 줄어들지 않는다. 이동할 뿐이다."**

GPT-6가 나와도 Harness가 사라지지는 않습니다. 하지만 "무엇을 설계해야 하는가"는 바뀝니다. 예전에 하던 세세한 제약은 사라지고, 대신 더 큰 자율성을 주는 구조가 중요해집니다.

### 자가 진단 — 잘 가고 있나?

내가 지금 잘 하고 있는지 판단하는 체크리스트입니다.

**✅ 잘 가고 있다는 신호**

- **같은 말을 두 번 하지 않는다** — 맥락 전달이 잘 되고 있다. "우리는 snake_case 쓴다"를 또 설명하지 않는다.
- **실수가 규칙이 된다** — 개선 루프가 돌고 있다. AI가 같은 실수를 반복하기 전에 규칙이 추가된다.
- **차단 장치가 뭔가를 막고 있다** — 사고가 구조적으로 예방되고 있다. Hook이 정기적으로 발동된다.
- **불필요한 것이 줄어든다** — 복잡해지는 게 아니라 단순해지고 있다.

**❌ 실패하고 있다는 징후**

- **검수에 시간이 더 오래 걸린다** — 검증 자동화가 부족하다. Evaluator를 분리하거나 Sprint Contract를 더 명확히 해야 한다.
- **시켰는데 원하는 결과가 안 나온다** — 맥락 전달 또는 계획 단계가 부족하다. CLAUDE.md나 Plan 스킬을 점검하라.
- **스킬·에이전트가 많은데 잘 안 쓴다** — Context pollution 발생 중. 제대로 동작도 안 하고 있다. 정리하라.
- **가이드 파일이 길어지고 관리 안 된다** — CLAUDE.md·docs가 분리/정리 없이 쌓이고 있다. Progressive Disclosure 구조로 재편하라.

정기적으로 이 체크리스트로 셀프 리뷰를 하세요. 월 1회가 적당합니다. 좀 더 체계적으로 하고 싶다면 `/check-harness` 스킬이 5축 35개 항목으로 자동 진단해줍니다.

### `session-wrap` — 개선 루프를 자동화하기

개선 루프를 수동으로 돌리기는 번거롭습니다. 그래서 개선 자체를 자동화하는 도구가 있습니다. `plugins-for-claude-natives`의 `session-wrap` 플러그인입니다.

이 플러그인은 **2단계 멀티 에이전트 파이프라인**으로 동작합니다.

```
Phase 1: 병렬 분석
┌──────────────┬──────────────┬──────────────┬──────────────┐
│ doc-updater  │ automation-  │ learning-    │ followup-    │
│ 문서 갱신 필요│ scout        │ extractor    │ suggester    │
│ 항목 감지    │ 자동화 기회  │ 인사이트 추출│ 후속 작업    │
└──────┬───────┴──────┬───────┴──────┬───────┴──────┬───────┘
       └──────────────┴──────────────┴──────────────┘
                            ▼
Phase 2: 검증
┌──────────────────────────────────────────────────┐
│ duplicate-checker (중복 제안 필터링)              │
└──────────────────────────────────────────────────┘
                            ▼
                    사용자가 최종 선택
```

구성된 3개의 스킬:

- **`/wrap`** — 세션 종료 시 전체 파이프라인 실행. `/wrap [commit message]`로 빠른 커밋도 가능.
- **`/history-insight`** — Claude Code 세션 히스토리를 분석해서 반복 패턴, 결정, 테마 추출.
- **`/session-analyzer`** — 세션 행동이 SKILL.md 명세대로 실행됐는지 사후 검증.

사람이 해야 할 일은 **"제안을 받아들일지 말지 결정하는 것"** 하나뿐입니다. 관측과 추출은 자동화됩니다.

세션 하나하나가 다음 세션을 더 잘 만드는 연료가 됩니다. 이게 **Compound Effect**의 진짜 의미입니다. 하네스가 스스로 진화합니다.

### 9.1. team-attention/harness

**역할**: Harness Engineering 세션을 위한 **실습용 플러그인**. 강의자료 + 체크리스트 + 기초 스킬 4종을 묶어둔 가벼운 패키지.

**링크**: https://github.com/team-attention/harness

**포함된 스킬 4종**:

| 스킬 | 설명 | 사용법 |
|---|---|---|
| `/check-harness` | 현재 프로젝트의 Harness 성숙도를 5축 35개 체크리스트로 진단 | `/check-harness` |
| `/scaffold` | Greenfield 프로젝트에 AI-optimized 하네스 구조를 스캐폴딩 | `/scaffold` |
| `/specify` | 목표를 구조화된 구현 계획(spec.md)으로 변환 | `/specify "목표"` |
| `/deep-interview` | Socratic 방식의 요구사항 인터뷰 (Ambiguity Score 기반) | `/deep-interview "주제"` |

**성숙도 모델 (5축 35개)**:

1. **준비 (Scaffolding)** — AI가 프로젝트를 스스로 파악할 수 있는가
2. **맥락 (Context)** — CLAUDE.md, 규칙, 점진적 노출
3. **실행 설계 (Execution)** — 계획, 위임, 오케스트레이션
4. **검증 (Verification)** — 테스트, 리뷰, 품질 관리
5. **개선 (Improvement)** — 학습, 피드백 루프

각 축은 **L1 시작하기 → L2 내 것으로 만들기 → L3 자율 운영** 세 단계로 평가됩니다.

**설치 & 시작**:

```bash
# 1. 레포 클론
git clone https://github.com/team-attention/harness.git

# 2. Claude Code 실행
cd harness
claude

# 3. 현재 프로젝트 진단
/check-harness

# 4. 새 프로젝트에 스캐폴딩
/scaffold

# 5. 모호한 요구사항 인터뷰
/deep-interview 뭘 만들어야 할지 모르겠어

# 6. 구현 계획 생성
/specify "사용자 인증 시스템 구현"
```

**추가 자료**:
- `materials/harness-checklist.md` — 전체 체크리스트 마크다운
- `materials/slides/` — 발표 슬라이드 50장 (HTML), `viewer.html`로 로컬에서 열람 가능

### 9.2. team-attention/hoyeon

**역할**: Spec-Driven Development(SDD)를 자동화하는 **멀티 에이전트 워크플로우 툴킷**. 이호연님 본인의 개인 Harness 전체 구성을 공개한 저장소. 하네스의 정교한 완성형을 보고 싶다면 여기를 참고하세요.

**링크**: https://github.com/team-attention/hoyeon  
**스타**: 51+  
**라이선스**: 공개 (fork/학습 가능)

**핵심 워크플로우**:

```
/specify → /open → /execute → /publish → /compound
```

| 단계 | 스킬 | 역할 |
|---|---|---|
| 1 | `/specify` | 인터뷰 기반 계획 수립. gap/tradeoff/verification-planner 병렬 분석 후 reviewer 승인을 거쳐 `PLAN.md` 생성 |
| 2 | `/open` | 승인된 스펙으로 Draft PR 생성 (`feat/{name}` 브랜치) |
| 3 | `/execute` | Orchestrator가 PLAN.md를 읽고 TODO별로 Worker 에이전트에 위임, 검증 후 원자적 커밋 |
| 4 | `/publish` | Draft PR을 Ready for Review로 전환 |
| 5 | `/compound` | 완료된 PR에서 learnings를 추출해 `docs/learnings/`에 저장 |

**원샷 자동화**: `/ultrawork feature-name`으로 `/specify → /open → /execute` 전체를 한 번에 실행.

**전문화된 에이전트 9종**:

| 에이전트 | 모델 | 역할 |
|---|---|---|
| `worker` | Sonnet | 위임된 TODO 구현 |
| `gap-analyzer` | Haiku | 계획 전 누락 요구사항/함정 탐지 |
| `tradeoff-analyzer` | Sonnet | 위험도(LOW/MED/HIGH) 평가, 단순 대안 제시 |
| `verification-planner` | Sonnet | 4-Tier 테스트 모델(Unit/Integration/E2E/Agent Sandbox) 기반 검증 전략 |
| `docs-researcher` | Sonnet | 내부 문서(ADR, README) 탐색 |
| `external-researcher` | Sonnet | 외부 라이브러리, 공식 문서 리서치 |
| `ux-reviewer` | Sonnet | UX 관점 평가, regression 방지 |
| `reviewer` | Opus | 계획의 명료성, 검증 가능성, 완결성 평가 |
| `git-master` | Sonnet | 원자적 커밋 자동 생성 |

**9개 Human-in-the-Loop 체크포인트**: 잘못된 전제 방지, 비즈니스 판단, 기술 선택, HIGH risk 승인 등 핵심 의사결정 지점마다 사용자 확인을 요구합니다.

**Verification Summary 패턴**: PLAN.md 최상단에 자동 생성.
- **A-items**: Agent 자동 검증 (테스트, 빌드, typecheck)
- **H-items**: Human 확인 필요 (UX, 디자인, 비즈니스)
- **External Dependencies Strategy**: Pre-work / During / Post-work

이 분류 덕분에 AI는 A-items까지만 자동으로 돌리고, H-items에서 사람 개입을 요청합니다.

**워크트리 기반 병렬 개발 — `hy` CLI**:

```bash
# 초기 설정 (한 번만)
/init  # .dev/config.yml 생성, hy CLI 설치

# 워크플로우
/specify auth                 # 계획 승인
/worktree create auth         # 워크트리 생성 + spec 이동
hy go auth                    # cd + claude 실행
/execute                      # 워크트리에서 구현

# 상태 확인
hy status
# #   NAME     PROGRESS    CHANGES  BEHIND  SESSIONS  PR
# 1   auth     3/5 ███░░   2        0       2         #42
# 2   payment  5/5 █████   0        3       0         -
```

여러 기능을 병렬로 작업할 때 각 워크트리가 독립된 Claude 세션을 가집니다.

**이 저장소를 참고할 가치**: 개별 스킬·에이전트·훅이 **실제로 어떻게 조합되는지**를 볼 수 있는 가장 상세한 예시. 직접 이대로 쓰지 않더라도, 내 하네스를 설계할 때 참고할 패턴이 많습니다.

## Chapter 10. 마무리 — 사람의 역할은 어떻게 바뀌는가

여기까지 왔으면 한 가지는 분명해졌을 겁니다. Harness Engineering은 **AI를 잘 쓰는 기술**이 아닙니다. **사람의 역할을 재정의하는 기술**입니다.

예전에는 개발자의 가치가 "얼마나 코드를 잘 쓰느냐"에 있었습니다. 지금은 바뀌고 있습니다. 코드는 AI가 씁니다. 사람의 가치는 다른 곳에 있습니다.

| 축 | 사람의 새로운 역할 |
|---|---|
| **구조 (Scaffold)** | 비즈니스 요구사항이 바뀔 때마다 코드베이스 구조를 발전시키기. 새 도구나 서비스가 들어오면 폴더링과 경계를 재설계하기. AI가 따라올 수 있는 아키텍처를 유지하기. |
| **맥락 (Context)** | CLAUDE.md와 docs/를 코드와 함께 갱신하기. 새 컨벤션이 생기면 rules/에 추가하기. 오래된 규칙은 정리하기 — **가비지 컬렉션**. |
| **개선 (Compound)** | AI 산출물의 품질을 지속적으로 모니터링. 반복되는 실수 → 규칙으로, 반복 작업 → 스킬로. 안 쓰는 건 치우고, 쓰는 건 날카롭게. |

> **AI가 코드를 쓰는 시대, 사람의 역할은 "잘 짜기"에서 "잘 일하는 환경을 만들기"로 바뀐다.**

### 월요일 아침에 할 수 있는 것

이 글을 다 읽고 "좋은 내용이네"로 끝나면 아무것도 바뀌지 않습니다. 월요일 아침에 할 수 있는 세 가지를 정리합니다.

**1. `team-attention/harness` 설치하고 `/check-harness` 돌리기**

5분이면 끝납니다. 내 프로젝트가 5축 35개 중 어디가 비었는지 즉시 알 수 있습니다. 안 채워진 축 중 **가장 쉬워 보이는 것 하나만** 먼저 보완하세요.

**2. `/wrap` 스킬로 세션 회고 시작하기**

`plugins-for-claude-natives`의 `session-wrap`을 설치하고, 오늘 세션이 끝날 때 `/wrap`을 돌려보세요. AI가 "이 패턴은 Skill로 만들어볼 만 하다", "이 실수는 Rule로 잡자"는 제안을 합니다. 그중 **하나만 받아들이세요.** 그게 당신의 첫 번째 하네스 개선입니다.

**3. `hoyeon` 저장소 한 번 훑어보기**

직접 설치할 필요도 없습니다. README만 읽어도 "이런 식으로 설계할 수 있구나"라는 감이 옵니다. 특히 **Verification Summary 섹션**(A-items / H-items 분리)과 **9개 HITL 체크포인트**는 어떤 하네스에도 이식 가능한 패턴입니다.

### ⚠️ 조심할 점

마지막으로 한 가지 경고를 덧붙입니다.

**Harness를 설계할 때 너무 옥죄려고 하지 마세요.**

많은 사람들이 Harness를 처음 배우면 "이것도 막아야지", "저것도 규칙으로 만들어야지" 하면서 감옥을 짓기 시작합니다. 그 결과 AI는 아무것도 할 수 없는 환경이 됩니다.

원칙은 반대입니다.

> **자유도를 높게 주고, 안 되는 것만 제한하기.**

**생각보다 Agent는 예측 가능하게 잘 움직입니다.** 좋은 환경을 만들어주면 지시를 초과하는 자율성으로 일합니다. 당신의 역할은 감옥 설계자가 아니라 **좋은 작업 환경 설계자**입니다.

### 핵심 용어 정리

| 용어 | 정의 |
|---|---|
| **Harness Engineering** | AI가 혼자서도 잘 일할 수 있는 작업 환경을 설계하는 기술 |
| **Scaffolding** | 프로젝트 폴더링, 도구 배치, 경계 설정 등 기반 구조 |
| **CLAUDE.md** | AI에게 전달되는 프로젝트 지도. 3단 상속(User/Project/Folder) |
| **Progressive Disclosure** | 필요한 정보만 필요할 때 보여주는 컨텍스트 관리 원칙 |
| **Sprint Contract** | 작업 전에 "뭘 만들고 어떻게 검증할지" 합의하는 것 |
| **Ralph Loop** | 완료 기준을 정하고 충족할 때까지 AI가 자동 반복하는 패턴 |
| **Generator/Evaluator 분리** | 만드는 AI와 확인하는 AI를 물리적으로 분리하는 검증 원칙 |
| **Compound Effect** | 하네스가 시간이 지날수록 스스로 진화하며 품질이 누적되는 효과 |
| **3번 법칙** | 같은 작업 3번 → Skill로, 같은 실수 3번 → Rule로 |
| **A-items / H-items** | 검증 자동화 가능(Agent)과 인간 확인 필요(Human)의 분리 |
| **HITL Checkpoint** | Human-in-the-Loop, 자동 파이프라인 중 사람 확인이 필요한 지점 |

### 기억해야 할 수치

- CLAUDE.md: **200줄 이하** 유지
- 컨텍스트 **20~30%** 차면 새로 시작
- 같은 작업 **3번** 반복 → Skill로
- 같은 실수 **3번** 반복 → Rule로
- Team Mode는 토큰 **~7배**
- 모델 교체 **5%** < 하네스 설계 **15%** 개선
- `/check-harness`는 **5축 35개** 항목, **L1/L2/L3** 3단계 성숙도
- `hoyeon`은 **9개 HITL 체크포인트**, **9개 전문 에이전트**
- 실패 태스크 **최대 3번** 재시도 (hoyeon reconciliation)

### 다섯 가지 금과옥조

1. **"프롬프트만으로는 부족하다"** — 환경 자체를 설계하라
2. **"한꺼번에 다 주면 AI도 헷갈린다"** — Progressive Disclosure
3. **"해줘가 아니라 물어봐"** — AI가 스스로 맥락을 채우게
4. **"뭐가 되면 끝인지만 정해주면 AI가 계속 돈다"** — Sprint Contract
5. **"좋은 Harness는 점점 단순해진다"** — 복잡해지면 뭔가 잘못된 것
