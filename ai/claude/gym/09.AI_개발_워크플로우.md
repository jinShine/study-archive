# 09. AI 개발 워크플로우 — 탐색 → 계획 → 코딩 → 커밋

> **키워드**: `탐색-계획-코딩-커밋` `think 키워드` `ultrathink` `UI 반복` `스크린샷 반복` `서브에이전트 활용`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 4단계 워크플로우, think 키워드, 반복의 중요성 | 모든 작업의 기본 |
| 🔴 필수 | UI 개발 워크플로우 | 프론트엔드 개발자 필수 |
| 🟡 이해 | 서브에이전트 활용 타이밍 | 효율 극대화 |

---

## 1. 왜 워크플로우가 필요한가? 🔴

```
❌ 초보자의 방식:
   "로그인 기능 만들어줘" → Claude가 바로 코딩 시작 → 엉성한 결과

✅ 프로의 방식:
   탐색 → 계획 → 코딩 → 커밋
   → 각 단계에서 Claude가 생각하고 검증
   → 훨씬 좋은 결과
```

### 만능 워크플로우 공식

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 1. 탐색  │───▶│ 2. 계획  │───▶│ 3. 코딩  │───▶│ 4. 커밋  │
│ Explore │    │  Plan   │    │  Code   │    │ Commit  │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
   파일 읽기     전략 수립       실제 구현       PR 생성
   정보 수집     "think"        검증 포함       문서화

⚠️ 1~2단계가 가장 중요! 빠지면 Claude가 바로 코딩에 뛰어들어 품질 떨어짐
```

---

## 2. 1단계: 탐색 (Explore) 🔴

```
비유: 요리하기 전에 냉장고 먼저 열어보는 것

"김치찌개 만들어줘" → 바로 냄비 꺼내면 재료가 있는지도 모름
✅ "먼저 냉장고에 뭐가 있는지 봐줘" → 재료 확인 → 그 후 요리
```

### 실전 명령 패턴

```bash
# ❌ 바로 코딩 요청 (Claude가 추측으로 짬)
"로그인 기능 만들어줘"

# ✅ 탐색부터 시작 (현재 코드 파악)
"src/ 폴더 구조를 먼저 훑어봐.
그리고 로그인 관련 파일이 있으면 읽어봐.
아직 코드는 작성하지 마."

# ✅ 구체적 파일 지정
"@src/auth/login.ts 이 파일을 읽고,
지금 어떤 방식으로 인증하는지 요약해줘.
아직 수정하지 말고."
```

### 💡 서브에이전트 활용 팁

```
복잡한 문제일수록 서브에이전트로 탐색하자:

"코드 검토 전문 에이전트로 @src/auth/ 폴더를 분석해줘"
"파일 검색 전문 에이전트로 결제 관련 코드를 찾아줘"

→ 메인 컨텍스트는 깨끗하게 유지
→ 탐색 결과 요약만 받음
→ 토큰 절약 + 더 정확한 분석
```

> 💡 **핵심 포인트**: "아직 코드 작성하지 마" 라는 명시적 지시가 중요!
> 안 그러면 Claude는 탐색 중에도 수정하려 든다.

## 3. 2단계: 계획 (Plan) — `think` 키워드 🔴

### think 키워드 레벨

```
사고 예산이 점점 늘어난다:

"think"        → 기본 사고       (간단한 작업)
"think hard"   → 더 깊은 사고     (중간 복잡도)
"think harder" → 매우 깊은 사고   (복잡한 문제)
"ultrathink"   → 최대 깊이 사고   (아키텍처 결정)
```

### 실전 예시

```bash
# 간단한 기능
"로그인 버튼 UI 개선 방법을 think 해보고 계획을 세워줘"

# 중간 난이도
"JWT 인증 리팩토링 방안을 think hard 한 후,
단계별 계획을 제시해줘. 아직 구현은 하지 마."

# 복잡한 설계
"마이크로서비스 아키텍처로 전환하는 방안을
ultrathink 해서 3가지 옵션으로 비교 분석해줘."
```

### 계획 저장하기 — 중요!

```
💡 계획이 마음에 들면 → 문서나 GitHub 이슈로 저장

이유:
- 3단계(구현)가 틀어지면 계획 단계로 돌아갈 수 있음
- 팀원과 공유 가능
- 나중에 "왜 이렇게 설계했지?" 추적 가능
```

**저장 예시:**

```bash
# Claude에게:
"위 계획을 docs/plans/auth-refactor.md 로 저장해줘"

# 또는:
"위 계획을 GitHub 이슈로 만들어줘. 제목은 'Auth 리팩토링 계획'"
```

## 4. 3단계: 코딩 (Code) — 검증하면서 🔴

```
❌ 그냥 코딩:
"위 계획대로 구현해줘"
→ Claude가 쭉 코드 쏟아냄
→ 중간에 이상한 부분 있어도 모르고 진행

✅ 검증하면서 코딩:
"위 계획대로 구현하되, 각 단계마다 멈춰서
'이 부분이 제대로 작동할까?',
'빠뜨린 건 없을까?'
확인하면서 진행해줘."
```

### 검증 프롬프트 패턴 모음

```bash
# 패턴 1: 단계별 검증
"코드 작성할 때마다 한 번씩
'이 로직이 맞나?', '사용자가 헷갈리지 않을까?'
체크해가면서 해줘."

# 패턴 2: 엣지 케이스 검증
"각 함수 만들 때마다:
1. 정상 케이스
2. 엣지 케이스 (빈 값, null, undefined)
3. 에러 케이스
세 가지를 고려하며 진행해줘."

# 패턴 3: 테스트 동시 작성
"기능 하나 만들 때마다 테스트도 함께 짜줘.
테스트 통과 확인한 후 다음 기능으로 넘어가."

# 패턴 4: 리뷰어 관점
"코드 작성할 때 '시니어 리뷰어라면 어떻게 지적할까?'
생각하면서 해줘."
```

## 5. 4단계: 커밋 & PR 🔴

```bash
"작업 내용을 커밋하고 PR을 생성해줘.
커밋 메시지는 한국어로,
PR 설명에는 변경 이유, 테스트 방법, 스크린샷을 포함해줘.
그리고 README나 CHANGELOG에도 반영해야 할 내용이 있으면 업데이트해줘."
```

### 좋은 커밋의 체크리스트

```
☐ 커밋 메시지: 무엇을 왜 바꿨는지
☐ 한 커밋 = 하나의 논리적 변경
☐ 빌드/테스트 통과 확인
☐ PR 설명: 변경 이유 + 테스트 방법
☐ UI 변경이면 → 스크린샷 첨부
☐ 문서 업데이트 (README, CHANGELOG 등)
```

## 6. UI 개발 전용 워크플로우 🔴

### 비유

```
글만으로 디자인 요청 = 전화로 그림 그려달라는 것
스크린샷 반복      = 그림 보여주며 "이렇게, 이렇게 고쳐주세요"
                     → 훨씬 정확!
```

### 전체 흐름

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  1. 디자인 목표 제공 (이미지)                          │
│        ↓                                             │
│  2. Claude가 코드로 구현                              │
│        ↓                                             │
│  3. Claude가 스크린샷 찍어서 확인 ←─┐                  │
│        ↓                          │                  │
│  4. 디자인과 비교, 차이 파악         │  반복!         │
│        ↓                          │                  │
│  5. 코드 수정 ────────────────────┘                  │
│        ↓                                             │
│  6. 목표 달성하면 커밋                                │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 준비물: 스크린샷 도구

```
Claude가 자기 결과를 "볼 수 있게" 해주는 MCP 서버:

- Playwright MCP     → 웹 브라우저 자동 스크린샷
- Puppeteer MCP      → 브라우저 자동화 (Playwright 대안)
- iOS Simulator MCP  → iOS 앱 화면 스크린샷

또는 수동:
- 직접 스크린샷 찍어서 드래그앤드롭
- 복사-붙여넣기
```

### 실전 프롬프트 예시

```bash
# Step 1: 디자인 제공
"@design/landing-page.png
이 디자인대로 React 컴포넌트를 만들어줘."

# Step 2: 구현 + 확인
"구현한 결과를 Playwright로 스크린샷 찍어서,
원본 디자인과 비교해줘. 다른 부분을 리스트로 알려줘."

# Step 3: 반복
"리스트의 1, 3, 5번 문제를 수정해줘.
수정 후 다시 스크린샷 찍어서 확인해."

# Step 4: 완성
"디자인과 거의 일치하면 커밋해줘."
```

## 7. 핵심: 반복의 힘 🔴

```
💡 인간과 똑같다. 첫 시도에 완벽한 결과는 드물다.

1차 시도: 기본 구조 완성 (60점)
2차 시도: 세부 디자인 개선 (80점)
3차 시도: 거의 완벽 (95점)

Claude가 자신의 결과를 볼 수 있어야 개선할 수 있다.
→ 스크린샷, 테스트 결과, 로그 등을 제공하자!
```

### 반복을 잘 이끌어내는 프롬프트

```bash
# ❌ 한 번에 끝내려는 프롬프트
"완벽한 로그인 UI 만들어줘"

# ✅ 반복을 유도하는 프롬프트
"로그인 UI를 만들고, 스크린샷 찍어서 보고,
개선할 점 3가지를 찾아서 수정하고,
다시 스크린샷 찍어서 확인해. 2번 반복해줘."
```

## 8. 팀 리더를 위한 워크플로우 전략 👔

### 표준 워크플로우를 팀 규칙으로

```
.claude/rules/workflow.md 에 추가:

---
# 개발 워크플로우 규칙

모든 기능 개발은 다음 순서로 진행:

1. 탐색: 관련 파일 읽기, "아직 코드 작성 금지"
2. 계획: think/think hard 키워드로 전략 수립, docs/plans/에 저장
3. 코딩: 단계별 검증 포함
4. 커밋: 한국어 메시지, PR에 이유/테스트 방법/스크린샷 포함

예외 없이 모든 작업에 적용한다.
---

→ git에 커밋하면 팀 전체에 표준 워크플로우 적용!
```

### 커스텀 슬래시 명령어로 자동화

```
.claude/commands/feature.md 파일 생성:

---
다음 기능을 4단계 워크플로우로 진행해줘:
$ARGUMENTS

1. 탐색: 관련 파일을 찾아서 읽어줘 (아직 코드 작성 금지)
2. 계획: think hard로 계획을 세워서 보여줘
3. 내 승인을 받은 후 코딩 (단계별 검증 포함)
4. 커밋 + PR 생성
---

→ /feature 로그인 기능 → 자동으로 4단계 진행!
```

> 💡 커스텀 슬래시 명령어는 [10.커스텀_슬래시_명령어.md](10.커스텀_슬래시_명령어.md)에서 자세히!

```
   [붙여넣기]"
```

## 실습: 4단계 워크플로우 직접 해보기

```bash
# 아무 프로젝트 폴더에서 Claude Code 실행

# Step 1: 탐색
@src/ 폴더 구조를 훑어보고, 주요 파일을 리스트업해줘.
아직 코드 수정하지 마.

# Step 2: 계획
사용자 프로필 페이지를 추가하는 방법을 think hard 해보고,
어떤 파일을 만들고 수정해야 할지 계획을 세워줘.
docs/plans/profile-page.md 에 저장해줘.

# Step 3: 코딩 (계획 확인 후)
위 계획대로 구현해줘. 각 단계마다:
- 이게 맞는지 확인
- 엣지 케이스 고려
- 테스트 코드도 함께 작성

# Step 4: 커밋
작업 내용을 커밋해줘. 한국어 메시지로,
PR 설명에는 변경 이유와 테스트 방법을 포함해줘.
```

## 면접 대비

### 🔴 필수 질문

**Q: Claude Code의 기본 워크플로우를 설명하라.**
> 탐색(Explore) → 계획(Plan) → 코딩(Code) → 커밋(Commit) 4단계입니다. 특히 1~2단계가 중요한데, 이 없이는 Claude가 바로 코딩에 뛰어들어 품질이 떨어집니다. 계획 단계에서는 `think`, `think hard`, `ultrathink` 같은 키워드로 사고 예산을 조절할 수 있습니다.

**Q: UI 개발 시 효과적인 워크플로우는?**
> 디자인 이미지 제공 → 코드로 구현 → 스크린샷으로 확인 → 차이점 찾아 수정의 반복 패턴입니다. Playwright나 Puppeteer MCP 서버로 Claude가 자기 결과물을 직접 볼 수 있게 해야 효과적입니다.

### 🟡 개념 질문

**Q: `think` 키워드의 역할은?**
> Claude의 사고 예산(토큰)을 늘려 더 철저한 분석을 유도하는 키워드입니다. 레벨은 `think` < `think hard` < `think harder` < `ultrathink` 순이며, 복잡한 문제일수록 높은 레벨을 사용합니다.
