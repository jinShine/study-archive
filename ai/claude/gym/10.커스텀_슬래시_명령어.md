# 10. 커스텀 슬래시 명령어 만들기

> **키워드**: `슬래시 명령어` `.claude/commands/` `$ARGUMENTS` `팀 공유 명령어`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 슬래시 명령어란, 만드는 법, 실전 예시 | 반복 작업 자동화 |
| 🔴 필수 | 팀 공유 명령어 전략 | 팀 리더 필수 |
| 🟡 이해 | $ARGUMENTS, 범위(scope), 조합 패턴 | 고급 활용 |

---

## 1. 슬래시 명령어란? 🔴

```
비유: 키보드 "매크로"

매번 "테스트 먼저 짜고, 구현하고, 리뷰해줘"라고 긴 설명 쓰기 귀찮잖아?
→ /tdd 라는 단어 하나로 대체할 수 있다.

Claude Code의 /init, /help, /model 같은 것들도 슬래시 명령어.
이걸 내가 직접 만들 수 있다!
```

### 이미 쓰고 있는 슬래시 명령어들

```
/init       ← 프로젝트 초기화
/help       ← 도움말
/model      ← 모델 변경
/compact    ← 대화 압축
/doctor     ← 진단

⬇ 이런 것들을 내가 직접 만들 수 있다! ⬇

/review     ← 내가 만든 코드 리뷰 명령어
/feature    ← 4단계 워크플로우 자동 실행
/commit-kr  ← 한국어 커밋 메시지 자동 생성
```

---

## 2. 어떻게 만드나? 🔴

### 저장 위치 = 범위(scope)

```
범위가 다르면 저장 위치가 다르다:

전역 (모든 프로젝트에서 사용):
  ~/.claude/commands/my-command.md

프로젝트 (이 프로젝트에서만, 팀 공유):
  프로젝트/.claude/commands/my-command.md
```

### 기본 구조

```
파일 이름      = 명령어 이름
파일 내용      = Claude에게 전달할 프롬프트
```

**간단 예시:**

```bash
# 파일 생성
mkdir -p .claude/commands
echo "이 코드의 성능 문제를 분석하고 최적화 방안을 제안해줘" \
  > .claude/commands/optimize.md

# 사용
/optimize
```

그게 전부다! 파일 하나 만들면 바로 슬래시 명령어가 된다.

## 3. 실전 예시 — 가장 많이 쓰는 커맨드 5가지 🔴

### 예시 1: `/review` — 코드 리뷰 자동화

**파일**: `.claude/commands/review.md`

```markdown
방금 작성한 코드를 시니어 개발자 관점에서 리뷰해줘.

체크 항목:
1. 버그/잠재적 오류
2. 성능 이슈
3. 보안 취약점
4. 가독성 (네이밍, 구조)
5. 테스트 커버리지

각 항목에 대해:
- 발견한 문제 (없으면 "없음")
- 개선 제안 (구체적 코드 예시와 함께)

마지막에 종합 점수 (10점 만점)와 한 줄 요약.
```

**사용:** `/review`

---

### 예시 2: `/commit-kr` — 한국어 커밋 자동 생성

**파일**: `~/.claude/commands/commit-kr.md`

```markdown
현재 스테이징된 변경사항을 기반으로 한국어 커밋 메시지를 만들어줘.

형식:
- 타입: feat / fix / refactor / docs / test / chore
- 예: `feat: 사용자 로그인 기능 추가`

규칙:
- 제목은 50자 이내
- 본문은 "왜" 이 변경이 필요한지 설명
- 이모지 사용 금지

커밋 메시지를 보여주고, 내가 OK 하면 커밋 실행해줘.
```

**사용:** `/commit-kr`

---

### 예시 3: `/feature` — 4단계 워크플로우 자동화

**파일**: `.claude/commands/feature.md`

```markdown
다음 기능을 개발해줘: $ARGUMENTS

반드시 4단계 워크플로우로 진행:

## 1단계: 탐색
- 관련 파일을 찾아서 읽기
- 기존 코드 패턴 파악
- 이 단계에서는 코드 작성 금지

## 2단계: 계획
- think hard 해서 구현 전략 수립
- 수정/생성할 파일 리스트
- docs/plans/ 에 계획 저장

내 승인을 기다린다. "OK"라고 하면 3단계 진행.

## 3단계: 코딩
- 계획대로 구현
- 각 단계마다 검증 (엣지 케이스 고려)
- 테스트도 함께 작성

## 4단계: 커밋
- 한국어 커밋 메시지
- PR 생성 (이유/테스트 방법 포함)
```

**사용:** `/feature 사용자 프로필 페이지 추가`

> 💡 **`$ARGUMENTS`**: 명령어 뒤에 쓴 내용이 이 자리에 들어간다!

### 예시 4: `/explain` — 코드 설명해주는 선생님

**파일**: `~/.claude/commands/explain.md`

```markdown
다음 파일/코드를 비전공자도 이해할 수 있게 설명해줘: $ARGUMENTS

설명 규칙:
1. 먼저 일상 비유로 이 코드가 "뭘 하는지" 설명
2. 주요 부분을 Step 1, 2, 3... 으로 분해
3. 각 라인이 왜 필요한지 설명
4. 주의사항이나 함정이 있으면 언급
5. 이 코드를 응용할 만한 예시 한 가지 제시

한국어로, 친근한 톤(반말)으로.
```

**사용:** `/explain @src/auth/login.ts`

---

### 예시 5: `/debug` — 버그 체계적으로 찾기

**파일**: `.claude/commands/debug.md`

```markdown
다음 문제를 체계적으로 디버깅해줘: $ARGUMENTS

진행 순서:
1. **증상 파악**: 정확히 어떤 현상이 발생하는가?
2. **재현 방법**: 어떻게 하면 이 버그를 재현할 수 있는가?
3. **로그/에러 확인**: 관련 로그나 에러 메시지 수집
4. **가설 3개 제시**: 원인 가능성 3가지와 각각의 근거
5. **검증**: 각 가설을 어떻게 검증할지 제시
6. **수정안**: 가장 유력한 원인의 수정 코드

절대 추측만으로 수정하지 말 것.
로그나 테스트로 원인 확인 후 수정.
```

**사용:** `/debug 로그인 후 간헐적으로 401 에러 발생`

## 4. `$ARGUMENTS` 변수 이해하기 🟡

```
/명령어 뒤에 쓴 내용이 → $ARGUMENTS 자리에 대입된다

예시:
---
파일: .claude/commands/greet.md
---
"$ARGUMENTS 님, 안녕하세요!"

사용: /greet 철수
결과: "철수 님, 안녕하세요!"

사용: /greet 김 팀장
결과: "김 팀장 님, 안녕하세요!"
```

### 실전 활용

```markdown
# .claude/commands/pr.md
$ARGUMENTS 관련 PR을 생성해줘.

제목: "$ARGUMENTS" 기반으로 작성
본문:
- 변경 이유
- 테스트 방법
- 관련 이슈 (있다면)
```

```bash
# 사용
/pr 로그인 에러 핸들링 개선
```

## 5. 명령어 조합 패턴 🟡

### 패턴 1: 명령어가 다른 명령어를 호출

**`.claude/commands/safe-commit.md`**

```markdown
안전한 커밋 절차:

1. 먼저 `/review`를 실행해서 코드 리뷰
2. 리뷰에서 CRITICAL 이슈가 있으면 → 중단
3. 없으면 `/commit-kr` 실행
4. git push
```

### 패턴 2: 체크리스트 명령어

**`.claude/commands/ship-check.md`**

```markdown
배포 전 체크리스트를 실행해줘:

☐ 빌드 성공 여부: `npm run build`
☐ 테스트 통과 여부: `npm test`
☐ 린트 통과 여부: `npm run lint`
☐ 타입 체크: `npx tsc --noEmit`
☐ 번들 크기 확인
☐ 환경변수 설정 확인

각 항목을 순서대로 실행하고, 실패하면 즉시 중단.
모두 통과하면 "배포 가능 ✅" 출력.
```

## 6. 팀 리더 전략 — 팀 명령어 배포 👔

### 전략: 팀 공통 슬래시 명령어 라이브러리

```
프로젝트/.claude/commands/
├── review.md          ← 팀 코드 리뷰 기준
├── feature.md         ← 새 기능 개발 워크플로우
├── bugfix.md          ← 버그 수정 워크플로우
├── commit-kr.md       ← 한국어 커밋 규칙
├── pr.md              ← PR 생성 템플릿
├── deploy-check.md    ← 배포 전 체크리스트
└── onboard.md         ← 새 팀원 온보딩
```

**git에 커밋하면:**
- 팀원이 clone → `/feature`, `/review` 등 바로 사용 가능
- 팀 규칙이 명령어로 "실행 가능한" 형태가 됨
- 새 팀원도 금방 적응

### 예시: 새 팀원 온보딩 명령어

**`.claude/commands/onboard.md`**

```markdown
새 팀원이시네요! 프로젝트 온보딩을 도와드릴게요.

다음을 순서대로 진행:

1. @README.md 를 읽고 프로젝트 개요 요약
2. @CLAUDE.md 를 읽고 프로젝트 규칙 요약
3. @.claude/rules/ 폴더의 규칙 파일들 요약
4. src/ 폴더 구조 설명 (핵심 폴더 중심)
5. 주요 기술 스택과 각각의 역할 설명
6. 빌드/테스트 명령어 정리

마지막에:
- "첫 번째로 해볼만한 간단한 작업" 제안
- "주의해야 할 점" 정리
```

**사용법 안내 (README에 추가):**

```markdown
## 첫 날에 할 일
1. git clone
2. Claude Code 실행
3. /onboard 입력
→ 프로젝트 이해 완료!
```

## 7. 주의사항 🟡

```
❌ 슬래시 명령어에 하면 안 되는 것:
- 비밀번호/토큰 하드코딩 (git에 올라감!)
- 너무 긴 프롬프트 (100줄 넘으면 유지보수 어려움)
- 너무 구체적인 파일 경로 (다른 프로젝트에서 재사용 불가)

✅ 좋은 슬래시 명령어:
- 한 가지 명확한 목적
- $ARGUMENTS로 유연하게
- 결과가 예측 가능
- 50줄 이내 권장
```

## 실습: 나만의 슬래시 명령어 3개 만들기

```bash
# 1. 전역 명령어 폴더 생성
mkdir -p ~/.claude/commands

# 2. /explain 명령어
cat > ~/.claude/commands/explain.md << 'EOF'
다음을 비전공자도 이해할 수 있게 한국어로 설명해줘: $ARGUMENTS

규칙:
- 일상 비유 먼저
- Step 1, 2, 3... 으로 분해
- 각 단계가 왜 필요한지 설명
- 반말 톤
EOF

# 3. /tdd 명령어
cat > ~/.claude/commands/tdd.md << 'EOF'
TDD 방식으로 구현해줘: $ARGUMENTS

순서:
1. 실패하는 테스트 먼저 작성
2. 테스트가 통과하는 최소 코드 작성
3. 리팩토링

각 단계마다 테스트 결과를 보여주고 진행해.
EOF

# 4. /commit-kr 명령어
cat > ~/.claude/commands/commit-kr.md << 'EOF'
스테이징된 변경사항으로 한국어 커밋 메시지 생성:
- 타입: feat/fix/refactor/docs/test/chore
- 제목 50자 이내
- 본문은 "왜"를 설명
- 이모지 금지

메시지 보여주고, OK 하면 커밋 실행.
EOF

# 5. Claude Code에서 테스트
/explain @src/app.ts
/tdd 장바구니 기능
/commit-kr
```

## 면접 대비

### 🔴 필수 질문

**Q: 커스텀 슬래시 명령어를 만드는 방법은?**
> `.claude/commands/이름.md` 파일에 Claude에게 전달할 프롬프트를 작성하면 됩니다. 전역(`~/.claude/commands/`)과 프로젝트 레벨(`./.claude/commands/`) 두 범위가 있습니다. `$ARGUMENTS` 변수로 명령어 뒤의 인자를 받을 수 있습니다.

**Q: 팀에서 슬래시 명령어를 공유하려면?**
> `프로젝트/.claude/commands/` 폴더에 파일을 만들고 git에 커밋하면 됩니다. 팀원이 clone하면 모든 커스텀 명령어를 바로 사용할 수 있어 팀 워크플로우가 표준화됩니다.

## 핵심 요약

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  슬래시 명령어 = 자주 쓰는 프롬프트를 /명령어로 단축            │
│                                                              │
│  만들기:                                                      │
│  .claude/commands/이름.md 파일 생성 → /이름 으로 실행          │
│                                                              │
│  범위:                                                        │
│  전역: ~/.claude/commands/ (모든 프로젝트)                    │
│  프로젝트: .claude/commands/ (팀 공유, git 커밋)              │
│                                                              │
│  변수:                                                        │
│  $ARGUMENTS = 명령어 뒤에 쓴 내용                             │
│                                                              │
│  필수 명령어 5선:                                             │
│  /review  /commit-kr  /feature  /explain  /debug             │
│                                                              │
│  팀 리더 팁:                                                  │
│  팀 워크플로우를 슬래시 명령어로 만들어 배포                   │
│  → 새 팀원도 금방 적응                                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
