## 1. CLAUDE.md란? 🔴

```
비유: 새로 온 팀원에게 주는 "우리 팀 업무 매뉴얼"

매번 새 대화마다:
  "이 프로젝트는 TypeScript이고, Prettier 쓰고,
   테스트는 Jest이고, 커밋은 한국어로..."
  → 귀찮잖아!

CLAUDE.md = 이걸 한 번만 써두면 매번 자동으로 Claude가 읽는 파일
```

### CLAUDE.md vs settings.json 차이

```
CLAUDE.md      = "어떻게 일하라" (행동 규칙, 코딩 스타일)
settings.json  = "뭘 할 수 있나" (권한, MCP, 모델 설정)

비유:
CLAUDE.md      = 업무 매뉴얼 ("커밋 메시지는 한국어로")
settings.json  = 출입 카드 ("서버실은 출입 금지")
```

## 2. CLAUDE.md 계층 구조 🔴

CLAUDE.md는 **3단계**로 쌓인다. 위에서 아래로 우선순위가 높아진다.

```
┌─────────── 1단계: 전역 (모든 프로젝트 공통) ───────────┐
│  ~/.claude/CLAUDE.md                                   │
│  "나는 한국어로 대화해. 커밋은 영어로."                    │
│  → 내가 하는 모든 프로젝트에 적용                         │
└────────────────────────────────────────────────────────┘
         ↓ 위에 겹쳐짐

┌─────────── 2단계: 프로젝트 (팀 공유) ──────────────────┐
│  프로젝트/CLAUDE.md                                    │
│  "TypeScript 프로젝트, Prettier 사용, Jest 테스트"      │
│  → 이 프로젝트를 clone한 모든 사람에게 적용 (git 커밋)   │
└────────────────────────────────────────────────────────┘
         ↓ 위에 겹쳐짐

┌─────────── 3단계: 하위 폴더 (모듈별 규칙) ─────────────┐
│  프로젝트/src/api/CLAUDE.md                            │
│  "API 응답은 항상 { success, data, error } 형태"        │
│  → 이 폴더 안에서 작업할 때만 적용                       │
└────────────────────────────────────────────────────────┘

최종 적용: 1단계 + 2단계 + 3단계 = 합쳐서 적용
충돌 시: 하위가 상위를 덮어씀 (가까운 게 우선)
```

### 어디에 쓰면 좋을까?

| 위치 | 용도 | 예시 |
|------|------|------|
| `~/.claude/CLAUDE.md` | 내 개인 습관/스타일 | 한국어 대화, 커밋 스타일 |
| `프로젝트/CLAUDE.md` | 팀 규칙 (git 공유) | 기술 스택, 코딩 컨벤션 |
| `프로젝트/폴더/CLAUDE.md` | 모듈별 규칙 | API 응답 형식, DB 규칙 |

## 3. CLAUDE.md 작성 원칙 🟡

### ⚠️ 가장 중요한 원칙: 60줄 이하!

```
❌ CLAUDE.md가 200줄 → Claude가 규칙을 무시하기 시작
✅ CLAUDE.md가 60줄  → 매번 정확히 따름

이유:
1. CLAUDE.md는 매 요청마다 컨텍스트에 포함됨
2. 길면 토큰 낭비 (= 돈 낭비)
3. 길면 Claude가 중요한 규칙을 놓침
```

### 작성 체크리스트

```
CLAUDE.md에 넣어야 할 것:
☐ 기술 스택 (언어, 프레임워크, 도구)
☐ 코딩 컨벤션 (네이밍, 포맷팅)
☐ 빌드/테스트 명령어
☐ 금지사항
☐ 언어 설정

CLAUDE.md에 넣지 말아야 할 것:
☐ 긴 설명이나 튜토리얼
☐ 코드 예시 (간단한 것만 OK)
☐ 이미 도구가 강제하는 것 (Prettier가 있으면 포맷 규칙 불필요)
☐ 임시 작업 내용 (이건 대화로)
```

## 4. 실전 템플릿 — 가장 많이 쓰는 5가지 🔴

### 템플릿 1: 🧑‍💻 1인 개발자 (전역 설정)

> `~/.claude/CLAUDE.md` — 모든 프로젝트에 적용되는 나의 기본 규칙

```markdown
# 전역 규칙

## 언어
- 대화: 한국어
- 코드/변수명: 영어
- 커밋 메시지: 한국어
- 주석: 한국어

## 작업 스타일
- 간결하게 답변, 불필요한 요약 생략
- 코드 변경 시 "왜" 이렇게 했는지 한 줄 설명
- 에러 발생 시: 원인 → 해결 → 확인 순서

## 금지
- console.log 디버깅 남기지 않기
- any 타입 사용 금지 (TypeScript)
- 요청하지 않은 리팩토링 하지 않기
```

## 6. 팀 리더를 위한 CLAUDE.md 운영 전략 👔

### 전략 1: 계층 분리

```
팀 규칙 (git 커밋):
  프로젝트/CLAUDE.md           → 코딩 컨벤션, 빌드 명령, 금지사항
  프로젝트/src/api/CLAUDE.md   → API 전용 규칙
  프로젝트/src/web/CLAUDE.md   → 프론트엔드 전용 규칙

개인 설정 (git 미포함):
  ~/.claude/CLAUDE.md          → 각자의 언어, 스타일
```

### 전략 2: 온보딩 자동화

```
새 팀원이 왔을 때:

1. git clone 프로젝트
2. claude 실행
   → CLAUDE.md가 자동으로 로드됨
   → 팀 규칙을 자동으로 따름!

❌ 옛날: "이 문서 읽고, 이 규칙 따르고, 이 도구 설치하고..."
✅ 지금: "git clone 하고 claude 치면 끝"
```

### 전략 3: 점진적 개선

```
Week 1: 기본만 (기술 스택 + 빌드 명령)
Week 2: 팀에서 반복되는 실수 → 금지사항에 추가
Week 3: 좋은 패턴 발견 → 컨벤션에 추가
Week 4: 필요 없는 규칙 정리 → 60줄 이하 유지

CLAUDE.md는 "살아있는 문서"다.
코드 리뷰에서 반복되는 피드백 → CLAUDE.md에 규칙으로 추가하면
Claude가 알아서 그 실수를 방지해준다!
```

## 7. `.claude/rules/` — 규칙을 파일로 분리하기 🔴

### CLAUDE.md가 60줄 넘을 것 같을 때

```
비유: 회사 매뉴얼이 너무 두꺼우면?
      → 목차는 한 장, 세부 규칙은 별도 문서로 분리!

CLAUDE.md = 목차 + 핵심 규칙 (60줄 이하)
.claude/rules/ = 세부 규칙 파일들 (각각 독립 문서)
```

### 폴더 구조

```
your-project/
├── CLAUDE.md                  ← 핵심 규칙 (간결하게)
├── .claude/
│   └── rules/
│       ├── code-style.md      ← 코드 스타일 가이드라인
│       ├── testing.md         ← 테스트 규칙
│       └── security.md        ← 보안 요구사항
```

> 💡 `.claude/rules/` 안의 파일들은 Claude가 **자동으로 읽는다.**
> CLAUDE.md에 "rules 폴더를 참조해"라고 쓸 필요 없음!

### 실전 예시: 규칙 분리하기

**CLAUDE.md (간결한 핵심만)**

```markdown
# 프로젝트 규칙

## 기술 스택
- Next.js 15, TypeScript, Tailwind CSS, Prisma

## 빌드 & 테스트
- 빌드: `npm run build`
- 테스트: `npm test`
- 린트: `npm run lint`

## 금지
- any 타입, console.log, 하드코딩 API 키 금지
```

**.claude/rules/code-style.md**

```markdown
# 코드 스타일

## 네이밍
- 컴포넌트: PascalCase (LoginForm.tsx)
- 함수/변수: camelCase (getUserById)
- 상수: UPPER_SNAKE_CASE (MAX_RETRY_COUNT)
- 타입/인터페이스: PascalCase + 접두사 없음 (User, not IUser)

## 파일 구조
- 한 파일에 한 컴포넌트
- 배럴 파일(index.ts)로 re-export
- 파일 크기 200줄 이하 권장

## 임포트 순서
1. React/Next.js
2. 외부 라이브러리
3. 내부 모듈 (@/)
4. 상대 경로 (./)
5. 타입 (type imports)
```

**.claude/rules/testing.md**

```markdown
# 테스트 규칙

## 필수 테스트
- 모든 API 엔드포인트: 성공/실패 케이스
- 유틸 함수: 엣지 케이스 포함
- 컴포넌트: 렌더링 + 인터랙션

## 테스트 네이밍
- describe: 테스트 대상 이름
- it/test: "should + 기대 동작"
- 예: `it('should return 404 when user not found')`

## 금지
- 테스트 스킵(skip/only) 커밋 금지
- 하드코딩된 테스트 데이터 금지 (팩토리 사용)
```

**.claude/rules/security.md**

```markdown
# 보안 규칙

## 입력 검증
- 모든 사용자 입력: zod 스키마로 검증
- API 파라미터: 타입 + 범위 체크
- SQL: 반드시 파라미터 바인딩 (raw query 금지)

## 인증/인가
- API 라우트: 미들웨어로 인증 체크
- 민감 작업: 추가 권한 확인
- 세션 토큰: httpOnly + secure 쿠키

## 금지
- 비밀번호 평문 저장 금지
- API 키 코드 하드코딩 금지
- CORS: * 허용 금지 (도메인 지정)
```

### 언제 분리하나?

```
규칙이 적을 때 (프로젝트 초기):
  → CLAUDE.md 하나로 충분

규칙이 많아질 때 (팀 성장, 프로젝트 복잡):
  → CLAUDE.md = 핵심만
  → .claude/rules/ = 세부 규칙 분리

판단 기준:
  CLAUDE.md가 60줄 넘을 것 같으면 → 분리!
```

> 💡 **팀 리더 팁 👔**: rules/ 폴더도 git에 커밋된다.
> 팀원이 clone하면 규칙 파일도 같이 받아지므로,
> 모든 팀원이 동일한 규칙으로 Claude를 쓸 수 있다.

```
CLAUDE.md:
  "테스트 통과 전 커밋하지 않기"     ← 행동 규칙 (Claude가 따름)

settings.json:
  deny: ["Bash(git push:*)"]        ← 시스템 차단 (물리적으로 못 함)

합치면:
  Claude는 규칙을 따르려 하고 + 실수해도 시스템이 막아줌 = 이중 안전장치!
```

> 💡 **핵심 원칙: "메커니즘으로 품질 강제, 프롬프트로 하지 말 것"**
> - CLAUDE.md에 "좋은 코드 짜줘" → ❌ 희망사항
> - Hook으로 린트 자동 실행 + CLAUDE.md에 구체적 규칙 → ✅ 시스템

## 실습: 내 CLAUDE.md 만들기

```bash
# 1. 전역 CLAUDE.md 만들기
cat > ~/.claude/CLAUDE.md << 'EOF'
# 전역 규칙

## 언어
- 대화: 한국어
- 코드/변수명: 영어
- 커밋 메시지: 한국어

## 스타일
- 간결 답변, 불필요한 요약 생략
- 코드 변경 시 이유 한 줄 설명

## 금지
- console.log 남기지 않기
- 요청하지 않은 리팩토링 하지 않기
EOF

# 2. 프로젝트 CLAUDE.md 만들기 (아무 프로젝트 폴더에서)
# claude 실행 후
/init 프로젝트 초기화를 한국어로 진행해주세요.

# 3. 생성된 CLAUDE.md 확인
@CLAUDE.md 현재 규칙을 보여줘

# 4. 위 템플릿을 참고해서 수정
@CLAUDE.md 기술 스택과 빌드 명령, 금지사항을 추가해줘
```

## 면접 대비

### 🔴 필수 질문

**Q: CLAUDE.md의 역할은?**
> Claude Code가 매 세션마다 자동으로 읽는 프로젝트 규칙 파일입니다. 기술 스택, 코딩 컨벤션, 빌드 명령, 금지사항 등을 정의하여 매번 설명하지 않아도 Claude가 프로젝트 규칙을 따르게 합니다.

**Q: CLAUDE.md의 계층 구조를 설명하라.**
> 전역(~/.claude/CLAUDE.md), 프로젝트(프로젝트/CLAUDE.md), 하위 폴더(프로젝트/폴더/CLAUDE.md) 3단계입니다. 모두 합쳐서 적용되며, 충돌 시 하위(가까운) 규칙이 우선합니다.

### 🟡 개념 질문

**Q: CLAUDE.md를 60줄 이하로 유지해야 하는 이유는?**
> CLAUDE.md는 매 요청마다 컨텍스트에 포함되므로, 길면 토큰 낭비이고 Claude가 중요 규칙을 놓칠 수 있습니다. 또한 "메커니즘으로 품질 강제"가 원칙이라, 긴 설명보다 Hooks와 에이전트로 자동화하는 것이 효과적입니다.
