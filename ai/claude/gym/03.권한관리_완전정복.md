# 03. 권한 관리 완전정복

> **키워드**: `permissions` `allow` `deny` `ask` `settings.json` `패턴 매칭` `권한 모드`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | allow/deny/ask 구조, 우선순위, 실전 예시 | 보안의 기본 |
| 🟡 이해 | 패턴 매칭, 권한 모드, additionalDirectories | 세밀한 제어 |
| 🟢 참고 | 보안 강화 설정 | 특수 상황용 |

---

## 1. 권한이 왜 필요한가? 🔴

```
비유: 집에 가사도우미를 부른 상황

❌ 권한 없이:
   가사도우미가 내 방, 금고, 서재 어디든 들어갈 수 있음
   → 위험!

✅ 권한 설정:
   "거실과 주방은 자유롭게 (allow)"
   "서재는 물어보고 (ask)"
   "금고는 절대 안 됨 (deny)"
   → 안전!

Claude Code도 똑같다.
내 컴퓨터의 파일을 읽고, 쓰고, 명령어를 실행할 수 있으니까
→ 어디까지 허용할지 정해줘야 한다.
```

---

## 2. 권한의 3가지 레벨 🔴

```json
{
  "permissions": {
    "allow": [...],   // ✅ 자동 허용 (물어보지도 않고 실행)
    "ask": [...],     // 🤔 확인 후 실행 (매번 물어봄)
    "deny": [...]     // ❌ 무조건 차단 (allow에 있어도 차단)
  }
}
```

### 우선순위 (이것만 기억하자!)

```
deny (절대 차단)  >  allow (자동 허용)  >  ask (물어봄)  >  기본값 (물어봄)

예시:
  allow에 "Bash" 추가         → 모든 bash 명령 자동 실행
  deny에 "Bash(rm:*)" 추가    → rm 명령은 차단됨!
  → deny가 이기므로 rm은 실행 안 됨 ✅
```

---

## 3. 무엇을 허용/차단할 수 있나? 🔴

### allow 규칙 — 가장 많이 쓰는 것들

| 규칙 | 설명 | 실전 예시 |
|------|------|-----------|
| `Bash(명령어)` | 특정 bash 명령만 허용 | `"Bash(npm run lint)"` |
| `Bash(패턴:*)` | 패턴으로 여러 명령 허용 | `"Bash(npm run test:*)"` |
| `Read(경로)` | 특정 파일/폴더 읽기 허용 | `"Read(./src/**)"` |
| `Write(경로)` | 특정 파일/폴더 쓰기 허용 | `"Write(./src/**)"` |
| `Bash` | 모든 bash 명령 허용 | ⚠️ 위험! 신중하게 |

### deny 규칙 — 반드시 넣어야 하는 것들

```json
"deny": [
  "Bash(curl:*)",           // 외부 네트워크 요청 차단
  "Bash(rm:*)",             // 파일 삭제 명령 차단
  "Read(./.env)",           // 환경변수 파일 보호
  "Read(./.env.*)",         // .env.local, .env.production 등 보호
  "Read(./secrets/**)",     // secrets 폴더 전체 보호
  "Write(./config/prod.*)"  // 프로덕션 설정 보호
]
```

> 💡 **팀 리더 팁 👔**: `.env`, `secrets/`는 deny에 **무조건** 넣어라.
> 팀원이 실수로 Claude에게 시크릿을 노출시키는 걸 방지할 수 있다.

## 4. 패턴 매칭 이해하기 🟡

```
*   = "이 레벨에서 아무거나" (단일 레벨)
**  = "하위 폴더 전부 포함" (다중 레벨)
:*  = "뒤에 뭐가 오든" (명령 인수)
```

### 예시로 이해하기

```
"Bash(npm run test:*)"
├── ✅ npm run test:unit    (매칭)
├── ✅ npm run test:e2e     (매칭)
└── ❌ npm run build        (매칭 안 됨)

"Read(./src/**)"
├── ✅ ./src/app.ts              (매칭)
├── ✅ ./src/components/Button.tsx (매칭 - 하위 폴더도!)
└── ❌ ./config/db.ts             (매칭 안 됨)

"Read(./.env*)"
├── ✅ ./.env            (매칭)
├── ✅ ./.env.local      (매칭)
├── ✅ ./.env.production (매칭)
└── ❌ ./src/.env.test   (매칭 안 됨 - 경로가 다름)
```

## 5. 실전 설정 예시 — 가장 많이 쓰는 것들 🔴

### 🧑‍💻 1인 개발자 — 편하게 쓰기

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git *)",
      "Bash(npx *)",
      "Read",
      "Write"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Read(./.env*)",
      "Read(./secrets/**)"
    ]
  }
}
```

> 💡 개인 프로젝트라 대부분 허용하되, 삭제와 시크릿만 막음

### 👔 팀 프로젝트 — 안전하게 쓰기

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test)",
      "Bash(npm run test:*)",
      "Bash(git status)",
      "Bash(git diff)",
      "Bash(git log *)",
      "Read(./src/**)",
      "Write(./src/**)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Bash(git commit:*)",
      "Write(./package.json)",
      "Write(./tsconfig.json)"
    ],
    "deny": [
      "Bash(curl:*)",
      "Bash(rm:*)",
      "Bash(npm publish:*)",
      "Read(./.env*)",
      "Read(./secrets/**)",
      "Write(./.env*)",
      "Write(./config/prod.*)"
    ]
  }
}
```

> 💡 **팀 리더 포인트 👔**:
> - `git push`는 ask → 실수로 푸시 방지
> - `npm publish`는 deny → 실수로 배포 방지
> - `package.json` 수정은 ask → 의존성 변경 확인

### 🔒 보안 강화 — 최소 권한

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run test)",
      "Read(./README.md)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Write(./src/**)"
    ],
    "deny": [
      "Bash",
      "Read(./.env*)",
      "WebFetch"
    ]
  }
}
```

> 💡 금융, 의료 등 민감한 프로젝트에서 사용

## 6. 설정 파일 위치와 우선순위 🟡

```
설정 파일이 여러 곳에 있을 수 있다:

1. 프로젝트 레벨:  .claude/settings.json      ← git에 포함 (팀 공유)
2. 프로젝트 로컬:  .claude/settings.local.json ← git에 미포함 (개인용)
3. 유저 레벨:      ~/.claude/settings.json     ← 모든 프로젝트 공통

우선순위: 프로젝트 로컬 > 프로젝트 > 유저
```

> 💡 **팀 리더 팁 👔**: 
> - `.claude/settings.json`에 팀 공통 deny 규칙을 넣고 git에 커밋
> - 팀원 개인 설정은 `settings.local.json`에 (gitignore됨)

## 7. additionalDirectories 🟡

기본적으로 Claude는 **실행한 폴더만** 접근 가능하다. 다른 폴더도 보게 하려면:

```json
{
  "permissions": {
    "additionalDirectories": [
      "../shared-libs/",             // 공유 라이브러리 폴더
      "~/company/design-tokens/",    // 디자인 토큰
      "/usr/local/templates/"        // 시스템 템플릿
    ]
  }
}
```

**쓰는 상황:**
- 모노레포에서 다른 패키지 참조할 때
- 공유 설정 파일이 상위 폴더에 있을 때
- 회사 공통 템플릿을 참조할 때

## 8. 권한 모드 (defaultMode) 🟡

```json
{
  "permissions": {
    "defaultMode": "default"  // 여기에 모드 이름
  }
}
```

| 모드 | 설명 | 쓰는 상황 |
|------|------|-----------|
| `default` | 도구 처음 쓸 때 물어봄 | 일반 개발 (기본값) |
| `acceptEdits` | 파일 편집은 자동 승인 | 신뢰할 수 있는 작업 |
| `plan` | 분석만 가능, 수정 불가 | 코드 리뷰, 계획 단계 |
| `bypassPermissions` | 모든 권한 건너뜀 | ⚠️ 안전한 환경에서만! |

```
❌ 절대 하면 안 되는 것:
   프로덕션 서버에서 bypassPermissions 사용
   → Claude가 rm -rf / 를 실행할 수도 있다!

✅ bypassPermissions가 괜찮은 경우:
   Docker 컨테이너 안, CI/CD 파이프라인 등 격리된 환경
```

## 9. `/permissions` 명령어로 확인하기 🔴

```bash
# Claude Code 안에서
/permissions

# → 현재 설정된 allow, deny, ask 목록이 나온다
# → 여기서 바로 수정도 가능
```

## 주의사항 체크리스트

```
⚠️ 권한 설정 시 반드시 확인할 것:

☐ .env, secrets/ 등 민감 파일 → deny에 추가했는가?
☐ rm, curl 등 위험한 명령 → deny에 추가했는가?
☐ deny가 allow보다 우선이라는 것을 이해했는가?
☐ Bash 패턴은 접두사 매칭 → 우회 가능성 인지했는가?
☐ 최소 권한 원칙 → 필요한 것만 allow에 추가했는가?
```

## 핵심 요약

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  권한 = Claude가 내 컴퓨터에서 할 수 있는 일의 범위            │
│                                                              │
│  3단계:  deny (차단)  >  allow (허용)  >  ask (물어봄)        │
│                                                              │
│  반드시 deny:  .env, secrets/, rm, curl                      │
│  편하게 allow: npm run *, git status, Read/Write(src/**)     │

│                                                              │
│  설정 위치:                                                   │
│  팀 공유 → .claude/settings.json (git 커밋)                  │
│  개인용 → .claude/settings.local.json (gitignore)            │
│  전역   → ~/.claude/settings.json                            │
│                                                              │
│  확인: /permissions  진단: /doctor                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
