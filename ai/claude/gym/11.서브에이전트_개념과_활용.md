# 11. 서브에이전트(Sub-agent) 개념과 활용

> **키워드**: `서브에이전트` `/agents` `.claude/agents/` `역할 분리` `컨텍스트 격리` `위임`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 서브에이전트란, 왜 필요한가, 실전 예시 | 1인 개발자의 팀 구성 |
| 🔴 필수 | 코드리뷰/테스트/보안 에이전트 | 매일 쓰는 핵심 |
| 🟡 이해 | 에이전트 설계 원칙, 조합 패턴 | 고급 활용 |

---

## 1. 서브에이전트란? 🔴

```
비유: 전문가 팀원을 "고용"하는 것

나 (메인 Claude):        "로그인 기능 만들어줘"
                         → 기획, 개발, 리뷰, 테스트 다 혼자 함
                         → 컨텍스트가 금방 꽉 참

메인 Claude + 서브에이전트:
                         "로그인 만들고, @code-reviewer 한테 리뷰 시켜"
                         메인: 개발만
                         code-reviewer: 리뷰만
                         → 각자 자기 일만 함
                         → 메인 컨텍스트 깨끗하게 유지!
```

### 왜 쓰나?

```
┌─────────── 서브에이전트의 3가지 장점 ───────────┐
│                                                │
│  1. 컨텍스트 분리                               │
│     서브에이전트는 독립된 컨텍스트에서 작업       │
│     → 메인 대화가 길어져도 깨끗                 │
│                                                │
│  2. 전문화                                      │
│     역할별 최적화된 프롬프트                     │
│     → 더 정확한 결과                            │
│                                                │
│  3. 병렬 처리                                   │
│     여러 에이전트를 동시에 실행 가능             │
│     → 더 빠른 작업                              │
│                                                │
└────────────────────────────────────────────────┘
```

---

## 2. 언제 서브에이전트를 써야 하나? 🔴

```
✅ 서브에이전트 쓰면 좋은 때:

┌── 탐색 작업 ─────────────────────────────┐
│   "파일 찾아줘", "구조 분석해줘"            │
│   → 결과만 요약받고 메인 컨텍스트 유지     │
└──────────────────────────────────────────┘

┌── 리뷰/검증 작업 ─────────────────────────┐
│   "내 코드 리뷰해줘", "보안 체크해줘"      │
│   → 독립적 시각으로 검토                   │
└──────────────────────────────────────────┘

┌── 반복적 작업 ───────────────────────────┐
│   "이 파일들 100개 각각 처리해줘"          │
│   → 병렬로 돌려서 속도 ↑                   │
└──────────────────────────────────────────┘

❌ 서브에이전트 안 쓰는 게 좋은 때:
- 간단한 작업 (오버헤드만 생김)
- 메인과 긴밀하게 협업해야 하는 작업
- 서브에이전트가 작업 맥락을 모르면 안 되는 경우
```

## 5. 실전 에이전트 예시 5가지 🔴

### 에이전트 1: 📝 code-reviewer — 코드 리뷰 전문

**파일**: `.claude/agents/code-reviewer.md`

```markdown
---
name: code-reviewer
description: 코드 작성 완료 후 품질/버그/보안 리뷰. 시니어 개발자 관점.
tools: Read, Grep, Glob
---

# 코드 리뷰어

너는 10년차 시니어 개발자다. 방금 작성된 코드를 리뷰한다.

## 체크 항목

1. **버그 & 잠재적 오류**
   - null/undefined 처리
   - 엣지 케이스
   - 비동기 처리 누락

2. **성능**
   - 불필요한 반복
   - 메모리 누수 가능성
   - DB N+1 쿼리

3. **보안**
   - SQL 인젝션
   - XSS 취약점
   - 하드코딩된 시크릿

4. **가독성**
   - 네이밍
   - 함수 크기 (30줄 초과 시 지적)
   - 중복 코드

## 출력 형식

각 항목마다:
- 🔴 CRITICAL: 즉시 수정 필요
- 🟡 MEDIUM: 개선 권장
- 🟢 LOW: 참고 사항

각 이슈는 구체적 코드 예시와 개선안 포함.
마지막에 종합 평가 (10점 만점).

## 금지
- 애매한 피드백 ("좀 더 좋게 해보세요")
- 직접 코드 수정 (제안만)
```

```markdown
name: test-writer

tools: Read, Write, Bash
---

# 테스트 작성자

너는 테스트 전문가다. 주어진 코드에 대한 테스트를 작성한다.

## 테스트 체크리스트

모든 함수/컴포넌트에 대해:

1. **정상 케이스** (happy path)
   - 가장 일반적인 사용 시나리오

2. **엣지 케이스**
   - 빈 값 (empty string, empty array)
   - null / undefined
   - 최대값 / 최소값
   - 경계 조건 (0, -1, Infinity)

3. **에러 케이스**
   - 잘못된 입력
   - 예외 상황
   - 권한 없음

## 규칙

- describe: "함수/컴포넌트 이름"
- it/test: "should [기대 동작] when [조건]"
- Arrange → Act → Assert 패턴
- 하드코딩 대신 팩토리/픽스처 사용
- 목(mock)은 최소화

## 검증

테스트 작성 후 반드시:
1. `npm test` 실행
2. 전부 통과 확인
3. 실패 시 원인 분석 후 수정

## 출력

생성된 테스트 파일 경로 + 커버리지 요약.
```

```markdown
---
name: security-reviewer

tools: Read, Grep, Glob
---

# 보안 검토자

너는 보안 전문가다. OWASP Top 10을 기반으로 취약점을 찾는다.

## 검사 항목

1. **Injection** (SQL, NoSQL, Command)
   - 사용자 입력이 직접 쿼리에 들어가는가?
   - ORM을 올바르게 사용하는가?

2. **인증/인가**
   - 패스워드 평문 저장 여부
   - JWT 만료 시간 설정
   - 세션 관리

3. **민감 정보 노출**
   - API 키 하드코딩
   - 에러 메시지에 내부 정보 노출
   - 로그에 비밀번호/토큰

4. **XSS / CSRF**
   - 입력 검증
   - 출력 이스케이프
   - CSRF 토큰

5. **의존성 취약점**
   - `npm audit` 결과 확인

## 출력 형식

🔴 CRITICAL: 즉시 수정 (예: SQL 인젝션)
🟡 HIGH: 빠른 수정 권장 (예: 약한 암호화)
🟢 INFO: 참고 (예: 강화 제안)

각 이슈:
- 발견 위치 (파일:라인)
- 공격 시나리오
- 수정 방법 (코드 예시)
```

---

```markdown
# 계획 수립 전문가

너는 시스템 설계 전문가다. 주어진 요구사항을 단계별 계획으로 분해한다.

## 계획 수립 프로세스

1. **요구사항 분석**
   - 기능 요구사항
   - 비기능 요구사항 (성능, 보안 등)
   - 제약 조건

2. **현재 코드 조사**
   - 관련 파일 탐색
   - 기존 패턴 파악
   - 재사용 가능한 컴포넌트

3. **대안 제시** (최소 2개)
   - 각 방안의 장단점
   - 트레이드오프

4. **권장 방안 선택**
   - 선택 이유
   - 위험 요소

5. **단계별 구현 계획**
   - Step 1: ...
   - Step 2: ...
   - 각 단계의 파일/함수 명시
   - 예상 소요 시간

## 금지

- 코드 직접 작성 금지 (계획만!)
- 추측으로 현재 코드 판단 금지 (반드시 파일 읽기)
- 단일 방안 제시 금지 (항상 대안 2개 이상)

## 출력

마크다운 문서 형식. docs/plans/에 저장 가능한 수준으로.
```

```markdown
# 코드 탐색 전문가

너는 코드베이스 탐색 전문가다. 주어진 주제와 관련된 파일을 찾고 요약한다.

## 작업 순서

1. **키워드 추출**
   - 질문에서 핵심 키워드 추출

2. **파일 검색**
   - Glob으로 관련 파일 찾기
   - Grep으로 키워드 포함 파일 찾기

3. **핵심 파일 선별**
   - 상위 5~10개 파일만 선택
   - 각 파일의 역할 요약

4. **의존 관계 파악**
   - 파일 간 import 관계
   - 핵심 함수의 호출 흐름

## 출력 형식

### 관련 파일 (중요도 순)
1. `경로/파일.ts` - 한 줄 설명
2. ...

### 핵심 흐름
간단한 다이어그램 또는 순서도

### 주의할 점
- 복잡한 부분
- 팁

## 금지

- 전체 파일 내용 덤프 금지 (요약만)
- 코드 수정 금지 (탐색만)
```

## 6. 조합 패턴 — 에이전트 파이프라인 🟡

### 패턴 1: 순차 파이프라인

```
explorer → planner → 메인 Claude (구현) → code-reviewer → test-writer

1. explorer: 관련 파일 탐색
2. planner: 계획 수립
3. 메인: 계획대로 구현
4. code-reviewer: 리뷰
5. test-writer: 테스트 작성
```

**실행 방법:**

```bash
# 메인 대화에서:
"로그인 기능을 추가해야 해.

1. 먼저 @explorer 로 관련 파일 찾아줘
2. 그 다음 @planner 로 계획 세워줘
3. 계획 승인 후, 메인에서 구현
4. 구현 완료되면 @code-reviewer 로 리뷰
5. 리뷰 통과하면 @test-writer 로 테스트 작성"
```

### 패턴 2: 병렬 검증

```
                ┌─ code-reviewer ─┐
구현 완료 ─────┤                 ├─ 종합 판단
                └─ security-reviewer ┘

동시에 실행 → 두 결과를 종합 → 수정
```

**실행 방법:**

```bash
"방금 구현한 결제 기능에 대해:
@code-reviewer 와 @security-reviewer 를 동시에 실행해서
각각의 리뷰를 받아줘."
```

## 7. 팀 리더를 위한 에이전트 전략 👔

### 전략 1: 팀 전용 에이전트 라이브러리

```
프로젝트/.claude/agents/
├── code-reviewer.md       ← 우리 팀 코딩 규칙 반영
├── security-reviewer.md   ← 회사 보안 정책 반영
├── test-writer.md         ← 팀 테스트 규칙 반영
├── db-reviewer.md         ← DB 스키마 규칙 반영
└── pr-reviewer.md         ← PR 생성 자동화

→ git 커밋 → 팀원 누구나 같은 기준으로 리뷰/테스트
```

### 전략 2: 리뷰 자동화

```bash
# PR 생성 전에 자동 리뷰
.claude/commands/pre-pr.md:
---
PR 생성 전 검증을 진행해줘:

1. @code-reviewer 로 코드 품질 체크
2. @security-reviewer 로 보안 체크
3. @test-writer 로 테스트 커버리지 체크

셋 다 통과해야 PR 생성.
하나라도 CRITICAL 이슈면 수정 후 재검증.
---

사용:
/pre-pr
```

### 전략 3: 전문 영역별 에이전트

```
DB 팀:    database-reviewer.md (스키마 변경 검토)
프론트:   ui-reviewer.md (접근성, 반응형)
백엔드:   api-reviewer.md (RESTful 규칙)
DevOps:  infra-reviewer.md (배포 스크립트)

→ 각 영역 전문가가 팀에 없어도 에이전트로 커버
```

## 8. 시스템 에이전트 활용하기 🟡

Claude Code에는 **내장 에이전트**가 있다:

| 에이전트 | 용도 |
|---------|------|
| `general-purpose` | 다양한 탐색/리서치 작업 |
| `Explore` | 빠른 코드베이스 탐색 (전용 도구) |
| `Plan` | 구현 계획 수립 (Plan 모드) |
| `code-reviewer` | 기본 코드 리뷰 (superpowers 기반) |

**사용:**

```bash
# 바로 Agent 도구를 호출
"Explore 에이전트로 이 프로젝트의 인증 관련 코드를 찾아줘"
"Plan 에이전트로 리팩토링 전략을 세워줘"
```

## 면접 대비

### 🔴 필수 질문

**Q: 서브에이전트를 왜 쓰나?**
> 컨텍스트 격리, 전문화, 병렬 처리 3가지 이유입니다. 메인 Claude의 컨텍스트를 깨끗하게 유지하면서, 각 에이전트가 자기 역할에 최적화된 프롬프트로 더 정확한 결과를 내고, 여러 에이전트를 동시에 실행해 속도도 높일 수 있습니다.

**Q: 커스텀 에이전트를 만드는 법은?**
> `.claude/agents/이름.md` 파일에 YAML 프론트매터(name, description, tools)와 시스템 프롬프트를 작성합니다. 전역(`~/.claude/agents/`)과 프로젝트(`./.claude/agents/`) 범위가 있으며, 프로젝트 에이전트는 git에 커밋하여 팀 전체가 공유할 수 있습니다.

## 핵심 요약

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  서브에이전트 = 가상 팀원 (각자 전문 역할)                     │
│                                                              │
│  3대 장점:                                                    │
│  컨텍스트 격리 + 전문화 + 병렬 처리                           │
│                                                              │
│  만들기:                                                      │
│  .claude/agents/이름.md                                      │
│  + 프론트매터 (name, description, tools)                      │
│  + 시스템 프롬프트 (역할, 규칙, 행동)                          │
│                                                              │
│  필수 에이전트 5선:                                           │
│  code-reviewer  test-writer  security-reviewer               │
│  planner  explorer                                           │
│                                                              │
│  조합 패턴:                                                   │
│  순차: explorer → planner → 구현 → reviewer → tester         │
│  병렬: code-reviewer + security-reviewer (동시)               │
│                                                              │
│  팀 리더: git 커밋된 에이전트로 리뷰 기준 통일                 │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
