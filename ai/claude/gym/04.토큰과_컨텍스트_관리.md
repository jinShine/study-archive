# 04. 토큰과 컨텍스트 관리

> **키워드**: `토큰` `컨텍스트 윈도우` `/compact` `/clear` `/context` `auto-compact` `비용 최적화`

| 등급 | 섹션 | 의미 |
|------|------|------|
| 🔴 필수 | 토큰 개념, /compact, /clear, 비용 절약법 | 돈과 직결 |

## 1. 토큰이란? 🔴

```
비유: 토큰 = "단어 조각"

사람:    "안녕하세요"       → 1개의 인사말
Claude:  "안녕하세요"       → 약 3~5개의 토큰으로 분해

영어는 효율적:   "Hello"    → 1 토큰
한국어는 비효율:  "안녕하세요" → 3~5 토큰
코드:           "function" → 1 토큰

💰 토큰 = 돈
   Claude를 쓸 때마다 토큰이 소비된다.
   토큰을 아끼면 = 돈을 아끼는 것!
```

### 토큰이 소비되는 곳

```
┌─────────────── 한 번의 요청에서 ───────────────┐
│                                                 │
│  입력 토큰 (내가 보내는 것):                      │
│  ├── 내 메시지                                   │
│  ├── CLAUDE.md 내용                              │
│  ├── @로 참조한 파일들                            │
│  ├── 이전 대화 내역                               │
│  └── 시스템 프롬프트                              │
│                                                 │
│  출력 토큰 (Claude가 응답하는 것):                 │
│  ├── 답변 텍스트                                  │
│  ├── 생성한 코드                                  │
│  └── 도구 호출 결과                               │
│                                                 │
│  💰 둘 다 비용이 든다!                            │
│  입력이 특히 비싸지는 이유:                        │
│  → 대화가 길어질수록 매번 이전 내용을 다시 보내니까 │
│                                                 │
└─────────────────────────────────────────────────┘
```

## 2. 컨텍스트 윈도우란? 🟡

```
비유: Claude의 "책상 크기"

작은 책상 (컨텍스트 윈도우가 작은 모델):
  → 책 3~4권만 올려놓을 수 있음
  → 새 책을 올리려면 오래된 책을 치워야 함

큰 책상 (컨텍스트 윈도우가 큰 모델):
  → 책 20~30권까지 올려놓을 수 있음
  → 하지만 결국 한계는 있음

┌──────────── 컨텍스트 윈도우 크기 ────────────┐
│  ████████████████████████░░░░░░░░░░          │
│  ↑ 사용 중                    ↑ 남은 공간     │
│                                               │
│  80% 넘으면 → Claude가 불안정해지기 시작        │
│  꽉 차면    → 오래된 대화를 자동 삭제           │
└───────────────────────────────────────────────┘
```

> ⚠️ **핵심**: AI 모델이 더 똑똑해지고, 컨텍스트 윈도우가 더 커지더라도
> 결국 우리가 해야 할 일은 **컨텍스트를 효과적으로 관리하는 것**

## 3. 토큰 관리 명령어 🔴

### `/clear` — 완전 초기화

```
비유: 책상 위의 모든 것을 쓸어버리고, 깨끗한 상태로 다시 시작

언제 쓰나?
├── 완전히 다른 작업을 시작할 때
├── Claude가 이상한 맥락에 빠졌을 때
└── "처음부터 다시!" 하고 싶을 때
```

```bash
/clear
# → 이전 대화 전부 삭제, 새 세션 시작
```

> ⚠️ 이전 대화가 **전부** 사라진다. 돌이킬 수 없음!

### `/compact` — 압축 (핵심!) 🔴

```
비유: 회의록을 "요약본"으로 바꾸는 것

원본 대화 (500 토큰):
  나: "로그인 기능에 이메일 검증을 추가해줘"
  Claude: "네, src/auth/login.ts를 수정하겠습니다..."
  나: "비밀번호 규칙도 추가해줘"
  Claude: "비밀번호에 대문자, 숫자, 특수문자를 요구하도록..."
  나: "테스트도 짜줘"
  Claude: "jest 테스트를 작성하겠습니다..."

/compact 후 (100 토큰):
  [요약: 로그인 기능에 이메일 검증 + 비밀번호 규칙 추가 완료.
   수정 파일: src/auth/login.ts, tests/auth.test.ts]
```

```bash
/compact
# → 이전 대화를 요약해서 토큰 절약
# → 중요한 정보는 유지, 불필요한 세부사항 제거
```

**언제 쓰나?**

```
✅ /compact 쓰면 좋은 때:
├── 대화가 길어져서 Claude 응답이 느려질 때
├── 한 가지 작업이 끝나고 다음 작업으로 넘어갈 때
├── "이전 대화 내용은 기억하되, 토큰은 아끼고 싶을 때"
└── 비용이 걱정될 때

❌ /compact 하면 안 되는 때:
├── 지금 작업 중인 코드의 디테일이 중요할 때
└── Claude가 방금 한 작업을 정확히 기억해야 할 때
```

> 💡 `/compact`와 `/clear`의 차이:
> - `/compact` = 요약해서 유지 (기억은 남김)
> - `/clear` = 전부 삭제 (백지 상태)

### `/context` — 현재 뭘 기억하고 있나 확인 🟢

```bash
/context
# → 현재 컨텍스트에 뭐가 들어있는지 보여줌
# → 어떤 파일, 이전 대화, 시스템 정보 등
```

## 4. auto-compact 설정 🟡

대화가 길어지면 **자동으로** 이전 내역을 압축하는 기능.

```
수동 /compact:  내가 직접 타이밍을 결정
auto-compact:   Claude가 알아서 판단해서 압축

설정 방법:
/config → auto-compact 활성화
```

**작동 방식:**

```
대화 시작 ─────────────────────────────────── 대화 끝
│                                              │
│ █████████████████████████████░░░░░░░░░░       │
│                              ↑                │
│                    여기서 자동 compact 발동     │
│                                              │
│ 중요한 정보는 유지, 불필요한 것만 정리         │
└──────────────────────────────────────────────┘
```

> 💡 **추천**: 긴 세션 작업을 자주 한다면 켜두는 게 좋다.

## 5. 토큰 절약 실전 팁 🔴

### 규칙 1: 큰 파일을 통째로 보내지 마라

```bash
# ❌ 나쁜 예: 1000줄짜리 파일 전체를 컨텍스트에 넣기
@src/giant-file.ts 이 파일 분석해줘

# ✅ 좋은 예: 필요한 부분만 지정
@src/giant-file.ts 의 50~80번째 줄에 있는 login 함수만 봐줘
```

### 규칙 2: 작업 전환 시 /compact

```bash
# 작업 1: 로그인 기능 완료
# ...긴 대화...

/compact    # ← 작업 1 요약하고 토큰 확보

# 작업 2: 회원가입 기능 시작
회원가입 API를 만들어줘
```

### 규칙 3: 모델을 상황에 맞게 선택

```
간단한 질문, 코드 생성  → Haiku  (저렴, 빠름)
일반 개발 작업           → Sonnet (균형)
복잡한 디버깅, 아키텍처  → Opus   (비싸지만 정확)

/model  로 언제든 변경 가능!
```

### 규칙 4: CLAUDE.md는 간결하게

```
❌ CLAUDE.md가 200줄   → 매 요청마다 200줄의 토큰 소비
✅ CLAUDE.md가 60줄    → 매 요청마다 60줄만 소비

CLAUDE.md는 매 요청마다 컨텍스트에 포함되므로
짧을수록 토큰이 절약된다!
```

### 규칙 5: 다른 세션에서 이어하기

```bash
# 긴 작업은 여러 세션으로 나눠서
# 세션 1: 기능 구현
# (종료)

# 세션 2: 이어서
claude --continue
# 또는
/resume

# → 이전 컨텍스트를 복원하되, 불필요한 부분은 자동 정리됨
```

## 6. 비용 최적화 전략 👔

### 팀 리더가 알아야 할 것

```
┌─────────── 월간 토큰 비용을 줄이는 법 ───────────┐
│                                                   │
│  1. CLAUDE.md 60줄 이하로 유지                     │
│     → 매 요청마다 소비되는 고정 토큰 줄이기         │
│                                                   │
│  2. 팀원들에게 /compact 습관 들이기                 │
│     → 불필요한 컨텍스트 누적 방지                   │
│                                                   │
│  3. 모델 가이드라인 정하기                          │
│     → "PR 리뷰는 Sonnet, 설계는 Opus"             │
│                                                   │
│  4. auto-compact 팀 기본 설정으로 켜기              │
│     → settings.json에 넣어서 팀 전체 적용           │
│                                                   │
│  5. 큰 파일은 라인 범위 지정해서 참조               │
│     → @file.ts:50-80 패턴 교육                    │
│                                                   │
└───────────────────────────────────────────────────┘
```
