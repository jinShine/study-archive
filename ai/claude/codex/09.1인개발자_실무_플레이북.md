# 09. 1인개발자 실무 플레이북

> 학습 목표: Claude Code를 혼자 만드는 제품의 실행 엔진으로 활용하는 루틴을 만든다.

## 학습 목표

- 혼자 개발할 때 Claude Code를 어떤 순서로 붙여야 하는지 안다.
- 기능 개발, 버그 수정, 리팩터링, 테스트, 문서화를 한 흐름으로 설계할 수 있다.
- "빠르게 만들기"와 "무너지지 않게 만들기"를 같이 잡는 방법을 익힌다.

## 왜 중요한가

1인개발자는 역할이 너무 많다. 기획자, 구현자, QA, 문서 담당자, 운영 담당자를 하루 안에 계속 오가야 한다. Claude Code의 진짜 가치는 코드를 한 줄 더 빨리 쓰는 데 있지 않다. 역할 전환 비용을 줄이고, 내가 놓치는 검증을 붙여 주는 데 있다.

즉, 솔로 개발에서 Claude Code는 "부하 직원"이 아니라 "실행 루프"다.

## 핵심 개념

### 1. 1인개발자의 기본 루프

가장 안정적인 기본 루프는 다음과 같다.

1. 문제 정의
2. 관련 코드 탐색
3. 최소 구현
4. 테스트 또는 수동 검증
5. 변경 요약과 다음 작업 연결

중요한 것은 항상 이 순서로 굴리는 것이다.

### 2. 기능 추가 루프

기능을 추가할 때는 아래 순서를 추천한다.

1. 기존 패턴 탐색
2. 구현 범위 축소
3. 필요한 파일만 수정
4. 테스트나 검증 절차 작성
5. 문서나 TODO 갱신

### 3. 버그 수정 루프

버그는 감으로 고치면 빠르지만 재발한다. Claude Code에는 아래 순서가 잘 맞는다.

1. 재현 조건 정리
2. 원인 후보 3개 이하로 축소
3. 가장 가능성 높은 원인 하나만 수정
4. 회귀 여부 확인

### 4. 리팩터링 루프

리팩터링은 "예뻐 보이게"가 아니라 "안전하게 바꾸기"다. 따라서 목적이 있어야 한다.

- 중복 제거
- 테스트 용이성 개선
- 변경 범위 축소
- 성능 문제 완화

목적 없는 리팩터링은 혼자 개발일수록 위험하다.

### 5. 문서화 루프

코드를 만든 직후가 가장 정확한 문서화 타이밍이다. 혼자 개발자는 문서를 자주 미루는데, Claude Code를 붙이면 이 비용을 낮출 수 있다.

문서화는 아래 세 가지로 나누면 좋다.

- 기능 설명
- 운영 체크리스트
- 다음 작업 메모

## 실무에서 잘 쓰는 사람의 방식

- 기능 요청 전에 기존 패턴을 먼저 찾게 한다.
- "최소 수정"을 자주 강조한다.
- 구현 뒤 테스트 또는 수동 검증 절차를 반드시 받는다.
- 끝날 때 변경 요약과 남은 위험을 짧게 남긴다.
- 잘된 루프는 skill이나 `CLAUDE.md`로 승격한다.

## 자주 망하는 방식

- 생각나는 대로 즉시 구현부터 시킨다.
- 버그 수정과 리팩터링을 한 번에 섞는다.
- 테스트가 없으면 그냥 넘어간다.
- 문서화는 나중으로 미룬다.
- Claude가 한 번 해낸 방식을 재사용하지 않고 또 처음부터 시킨다.

## 바로 써보는 예시

### 기능 추가 요청 템플릿

```text
프로필 화면에 닉네임 수정 기능을 추가하고 싶어.
먼저 기존 프로필 조회/수정 패턴을 찾아서 재사용 가능한 부분을 알려줘.
그다음 최소 구현 범위로만 수정해줘.
수정 후에는 테스트가 있으면 실행하고, 없으면 수동 검증 절차를 적어줘.
마지막에는 변경 요약과 남은 위험만 짧게 알려줘.
```

### 버그 수정 요청 템플릿

```text
회원가입 직후 환영 이메일이 중복 발송되는 문제를 조사해줘.
재현 경로와 관련 파일을 먼저 찾고,
원인 후보를 3개 이하로 줄인 뒤 가장 가능성이 높은 것 하나만 수정해줘.
회귀 가능성도 같이 점검해줘.
```

### 문서화 요청 템플릿

```text
방금 수정한 인증 흐름을 기준으로
새로운 팀원이 이해할 수 있는 짧은 운영 메모를 작성해줘.
구현 이유, 핵심 파일, 검증 방법, 주의점 순서로 정리해줘.
```

## 1인개발 적용

혼자 만들 때는 Claude Code를 아래 세 구간에 붙이면 체감이 가장 크다.

- 설계 직전: 기존 패턴과 리스크 찾기
- 구현 직후: 테스트와 회귀 점검
- 마감 직전: 문서화와 릴리즈 체크

특히 밤에 지친 상태에서 마지막 검증 루프를 Claude에게 맡기면 놓치는 것이 많이 줄어든다.

## 팀 적용

이 문서는 1인개발 중심이지만, 혼자 잘 굴리는 루프는 팀 도입의 프로토타입이 된다. 개인 루프에서 아래가 검증되면 팀 확장이 쉽다.

- 어떤 요청 템플릿이 잘 먹히는가
- 어떤 검증이 자동화 가치가 큰가
- 어떤 규칙은 `CLAUDE.md`로 올려야 하는가

즉, 개인 플레이북은 팀 운영 플레이북의 실험실이다.

## 체크리스트

- 기능, 버그, 리팩터링, 문서화 루프를 각각 설명할 수 있다.
- 혼자 개발할 때 Claude Code를 붙일 핵심 지점을 안다.
- 구현 뒤 검증과 요약을 반드시 붙이는 습관이 생겼다.
- 반복적으로 잘 되는 루프를 skill이나 규칙으로 승격할 수 있다.
- Claude Code를 "코드 생성기"보다 "실행 루프"로 보기 시작했다.
