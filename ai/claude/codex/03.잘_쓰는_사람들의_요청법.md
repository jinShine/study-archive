# 03. 잘 쓰는 사람들의 요청법

> 학습 목표: 프롬프트를 예쁘게 쓰는 법이 아니라, Claude Code가 일하기 좋은 작업 명세를 만드는 법을 익힌다.

## 학습 목표

- 작업을 한 번에 크게 던지지 않고 단계로 쪼갤 수 있다.
- 검증과 산출물 형식을 요청에 포함시키는 습관을 만든다.

## 왜 중요한가

Claude Code를 잘 쓰는 사람은 문장을 화려하게 쓰지 않는다. 대신 작업이 흘러갈 레일을 먼저 깐다. "뭘 만들지"만 말하면 Claude는 최대한 추측해서 움직인다. "어떤 순서로 탐색하고, 어떤 제약을 지키고, 무엇으로 검증할지"까지 말하면 결과 품질이 훨씬 안정적이다.

요청의 핵심은 설득이 아니라 명세다.

## 핵심 개념

### 1. 좋은 요청의 기본 구조

좋은 요청은 보통 아래 5가지를 포함한다.

1. 목표: 무엇을 바꾸려는가
2. 맥락: 어디를 먼저 봐야 하는가
3. 제약: 무엇은 건드리면 안 되는가
4. 검증: 무엇으로 끝났다고 볼 것인가
5. 출력 형식: 마지막에 무엇을 보여줄 것인가

### 2. 요청은 단계가 있어야 한다

Claude Code는 탐색, 수정, 테스트를 모두 할 수 있다. 그래서 "바로 구현"보다 "먼저 탐색 -> 방향 제시 -> 구현 -> 검증" 순서가 훨씬 안정적이다.

### 3. 질문보다 작업 지시가 낫다

`이거 가능한가?`는 대화를 만든다. `가능한지 관련 코드를 먼저 확인하고, 가능하면 최소 수정안과 위험 요소를 정리해줘`는 작업을 만든다.

### 4. 출력 형식이 품질을 만든다

결과를 요약, 체크리스트, 패치 설명, 테스트 결과 형식으로 받도록 지정하면 검토 시간이 줄어든다.

## 실무에서 잘 쓰는 사람의 방식

- 첫 문장에서 성공 조건을 적는다.
- 범위 밖 항목을 같이 적는다.
- "무조건 수정"이 아니라 "먼저 확인 후 수정"을 요청한다.
- 끝에 검증 명령이나 검증 방식까지 붙인다.
- 최종 응답 형식을 제한한다. 예를 들면 `변경 요약`, `테스트 결과`, `남은 위험` 3개만 달라고 한다.
- 한 번의 요청 안에서도 탐색 단계와 실행 단계를 구분한다.

## 자주 망하는 방식

- 모호한 표현을 낭만적으로 길게 쓴다.
- 범위를 안 적어서 Claude가 저장소를 너무 넓게 뒤진다.
- 검증 조건을 안 줘서 "수정은 했지만 확신이 없는 결과"를 받는다.
- 결과 형식을 안 정해서 설명은 길고 핵심은 없는 답을 받는다.
- 요청을 너무 잘게 쪼개서 매번 맥락을 다시 설명한다.

## 바로 써보는 예시

### 예시 1. 버그 수정 요청

```text
로그인 후 간헐적으로 대시보드로 이동하지 않는 문제를 조사해줘.
먼저 관련 라우팅과 인증 상태 관리 코드를 찾아서 원인 후보를 3개 이하로 정리해줘.
가장 가능성이 높은 원인 하나만 수정해줘.
수정 후에는 테스트가 있으면 실행하고, 없으면 수동 검증 절차를 적어줘.
마지막 답변은 변경 요약, 검증 결과, 남은 위험 순서로 짧게 정리해줘.
```

### 예시 2. 기능 추가 요청

```text
사용자 프로필 화면에 닉네임 편집 기능을 추가하고 싶어.
먼저 기존 프로필 조회/수정 흐름을 파악하고, 재사용 가능한 컴포넌트가 있는지 확인해줘.
UI를 새로 크게 바꾸지 말고 현재 패턴을 유지해줘.
구현 후에는 영향을 받은 파일과 테스트 방법만 간단히 정리해줘.
```

### 예시 3. 문서화 요청

```text
현재 결제 모듈의 동작 흐름을 새 팀원이 이해할 수 있게 문서화해줘.
코드 기준으로만 작성하고 추측은 하지 말아줘.
입력, 검증, 외부 호출, 실패 처리 순서로 설명해줘.
```

## 1인개발 적용

혼자 개발할 때는 Claude에게 두 가지 역할을 번갈아 맡기면 좋다.

- `탐색자`: 관련 파일과 리스크를 찾게 한다.
- `작업자`: 좁혀진 범위만 수정하게 한다.

한 요청에 이 역할을 분리해서 쓰면 된다.

```text
먼저 탐색자로 움직여서 관련 파일과 리스크를 찾고,
그다음 작업자로 움직여 최소 수정만 해줘.
```

이 한 줄만 추가해도 쓸데없는 과감함이 줄어든다.

## 팀 적용

팀에서는 개인의 말빨보다 요청 템플릿이 중요하다. 누구나 비슷한 품질로 Claude Code를 쓰게 하려면 공통 요청 틀을 만들어야 한다.

추천 템플릿은 아래처럼 단순해야 한다.

```text
[목표]
[먼저 볼 곳]
[건드리면 안 되는 것]
[검증 방식]
[최종 보고 형식]
```

이 정도만 통일해도 코드리뷰에서 "왜 이렇게까지 바뀌었지?" 같은 사고가 많이 줄어든다.

## 체크리스트

- 좋은 요청의 5요소를 설명할 수 있다.
- 탐색 단계와 실행 단계를 나눠서 요청할 수 있다.
- 검증 조건을 요청에 반드시 넣는 이유를 안다.
- 최종 응답 형식을 미리 제한할 수 있다.
- 내 프로젝트에 맞는 요청 템플릿 초안을 만들 수 있다.
