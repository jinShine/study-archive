## 왜 중요한가

Claude Code는 읽기만 하는 도구가 아니다. 파일을 수정하고, 셸 명령을 실행하고, 외부 도구도 만질 수 있다. 그래서 실력은 "얼마나 똑똑하게 시키느냐"만으로 결정되지 않는다. `어디까지 허용할지`, `언제 컨텍스트를 비울지`, `언제 새 세션으로 넘길지`를 아는 것이 훨씬 중요하다.

좋은 사용자는 Claude를 많이 믿는 사람이 아니라, 제대로 통제하는 사람이다.

## 핵심 개념

### 1. 권한 시스템

공식 문서 기준으로 Claude Code는 읽기, Bash 실행, 파일 수정 같은 도구 권한을 세밀하게 제어할 수 있다. 권한 규칙은 보통 `allow`, `ask`, `deny`의 사고방식으로 이해하면 된다.

중요한 원칙은 이렇다.

- 읽기는 비교적 자유롭다.
- 셸 실행과 파일 수정은 주의가 필요하다.
- 규칙은 "한 번에 전부 허용"보다 "자주 쓰는 안전한 범위만 허용"이 낫다.

### 2. 권한 모드

현재 공식 문서에서 자주 보게 되는 모드는 다음과 같다.

- `default`: 기본 승인 흐름
- `acceptEdits`: 작업 디렉터리 안의 일반적 파일 수정과 일부 파일 시스템 명령을 더 쉽게 허용
- `plan`: 분석 중심, 수정 없음
- `auto`: 자동 승인 기반의 실험적 흐름
- `dontAsk`: 사전 허용 없이는 도구를 막는 보수적 흐름
- `bypassPermissions`: 매우 강력하며 격리 환경에서만 권장

입문자는 보통 `default` 또는 필요한 범위의 `acceptEdits`에서 시작하는 편이 안전하다.

### 3. 컨텍스트는 작업 공간이다

컨텍스트가 꽉 차면 Claude의 판단 비용이 커지고, 예전 정보를 계속 끌고 다니느라 집중력이 흐려진다. 읽은 파일, 대화, 스킬, 메모리 설명이 다 여기에 들어온다.

### 4. 컨텍스트 관리 명령

- `/compact`: 핵심만 남기고 압축
- `/clear`: 대화 컨텍스트 정리
- `/cost`: 사용량과 비용 감각 확인

압축은 "잊기"가 아니라 "핵심만 들고 다시 시작하기"에 가깝다.

### 5. 비용은 길이보다 낭비에서 터진다

비용을 키우는 주범은 긴 문장 자체보다 불필요한 탐색과 중복 읽기다. 같은 파일을 여러 번 읽거나, 범위 없는 요청으로 관련 없는 파일까지 훑게 만들면 금방 비싸진다.

## 실무에서 잘 쓰는 사람의 방식

- 처음부터 강한 권한 모드를 쓰지 않는다.
- `allow`는 자주 쓰는 안전한 명령에만 좁게 건다.
- 긴 세션에서 주제가 바뀌면 압축하거나 새 세션으로 넘긴다.
- 관련 없는 파일을 읽지 않게 요청 범위를 좁힌다.
- 구현이 끝난 뒤에는 검증 결과만 남기고 대화를 정리한다.
- 권한과 비용을 생산성의 일부로 본다.

## 자주 망하는 방식

- 귀찮다는 이유로 과한 자동 승인 모드를 사용한다.
- 같은 세션에서 버그 수정, 아키텍처 토론, 문서 초안을 다 섞는다.
- 프로젝트 전체를 매번 다 읽게 한다.
- 명령 하나 허용해 놓고 사실상 모든 셸 작업이 가능한 상태를 만든다.
- 비용을 신경 쓰지 않다가 나중에만 놀란다.

## 바로 써보는 예시

### 권한을 조심스럽게 여는 요청

```text
먼저 읽기와 분석만으로 원인을 찾아줘.
수정이 필요하다고 판단되면 어떤 파일을 왜 바꿔야 하는지 먼저 설명해줘.
그다음 내가 확인한 뒤 수정해도 돼.
```

### 긴 세션을 정리하는 요청

```text
지금까지의 핵심만 남도록 압축해줘.
결정된 내용, 아직 안 끝난 작업, 다음 세션 첫 질문에 넣어야 할 문장만 정리해줘.
```

### 비용을 줄이는 요청

```text
`src/auth`와 `src/routes`만 먼저 보고 원인 후보를 좁혀줘.
관련 없으면 다른 디렉터리는 읽지 말아줘.
```

## 1인개발 적용

혼자 개발할 때는 속도를 내고 싶은 마음 때문에 권한을 빨리 넓히기 쉽다. 하지만 개인 프로젝트일수록 사고가 나면 복구도 내가 해야 한다. 그래서 아래 순서를 권장한다.

1. 탐색 중심 요청
2. 소규모 수정 허용
3. 반복 검증 명령 허용
4. 자동화는 마지막

이 순서로 가면 편함은 조금 늦게 오지만 실수 비용이 훨씬 낮다.

## 팀 적용

팀에서는 권한이 곧 거버넌스다. 누군가는 안전하게 쓰고, 누군가는 과감하게 쓰면 팀 결과가 불안정해진다. 최소한 아래는 팀 표준이 있어야 한다.

- 기본 권한 모드
- 허용 가능한 Bash 패턴
- 보호할 파일과 디렉터리
- 압축 또는 세션 종료 기준
- 비용을 모니터링하는 방식

팀 차원의 규칙은 "무조건 금지"보다 "여기까지는 자동, 그 이상은 승인" 구조가 현실적이다.

## 체크리스트

- 권한 규칙과 권한 모드의 차이를 설명할 수 있다.
- 어떤 상황에서 `default`, `acceptEdits`, `plan`을 쓸지 말할 수 있다.
- 컨텍스트가 길어질수록 왜 품질이 흔들리는지 이해한다.
- `/compact`, `/clear`, `/cost`를 언제 쓰는지 안다.
- 속도보다 통제를 우선하는 습관이 생기기 시작했다.
