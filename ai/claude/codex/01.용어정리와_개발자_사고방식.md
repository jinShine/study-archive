# 01. 용어 정리와 개발자 사고방식

> 학습 목표: Claude Code를 채팅 앱이 아니라 작업 시스템으로 이해한다.

## 학습 목표

- Claude Code에서 자주 나오는 핵심 용어를 혼동 없이 설명할 수 있다.
- "좋은 질문"보다 "좋은 작업 지시"가 왜 중요한지 이해한다.
- 하네스, 메모리, 스킬, 훅, MCP, 서브에이전트의 역할 차이를 구분할 수 있다.

## 선행 지식

- 없음

## 왜 중요한가

Claude Code를 처음 쓰는 사람이 가장 먼저 부딪히는 문제는 기능이 아니라 머릿속 모델이다. 도구를 "코드 잘 짜주는 챗봇"으로 보면 계속 대화만 길어진다. 반대로 Claude Code를 `탐색 -> 계획 -> 수정 -> 실행 -> 검증` 루프를 수행하는 작업자라고 보면 요청 방식과 설정 방식이 완전히 달라진다.

이 문서는 기능 설명서가 아니다. 이후 모든 문서를 읽을 때 기준이 되는 공통 언어를 맞추는 문서다.

## 핵심 개념

### 1. 세션

세션은 Claude Code와 한 번 연결된 작업 흐름이다. 현재 대화, 읽은 파일, 실행한 명령, 불러온 규칙이 이 세션의 컨텍스트를 만든다.

### 2. 컨텍스트

컨텍스트는 Claude가 지금 판단에 사용하고 있는 작업 재료다. 내 요청, 읽은 코드, `CLAUDE.md`, 메모리, 스킬 설명, 일부 도구 설명이 모두 여기에 들어간다.

### 3. 토큰

토큰은 컨텍스트를 채우는 비용 단위다. 말이 길고 읽는 파일이 많을수록 토큰이 늘어난다. Claude Code 실력은 결국 "적은 토큰으로 정확한 판단 재료를 주는 능력"과 연결된다.

### 4. 권한

Claude Code는 읽기, 쓰기, 셸 실행, 외부 도구 접근을 모두 다룰 수 있다. 그래서 권한은 옵션이 아니라 안전장치다. 잘 쓰는 사람은 성능보다 권한 구조를 먼저 본다.

### 5. `CLAUDE.md`

`CLAUDE.md`는 프로젝트 규칙서다. 코딩 스타일, 빌드 명령, 리뷰 기준, 폴더 구조, 팀 관습처럼 "매번 다시 말하기 싫은 것"을 적는 곳이다.

### 6. Auto memory

Auto memory는 Claude가 작업하면서 스스로 적어 두는 기억이다. 공식 문서 기준으로 `CLAUDE.md`와는 별개 시스템이며, 주로 빌드 명령, 디버깅 힌트, 반복 패턴 같은 것을 저장한다. 사람이 직접 지시하는 규칙은 `CLAUDE.md`, Claude가 쌓는 작업 기억은 auto memory라고 생각하면 된다.

### 7. 스킬

스킬은 재사용 가능한 작업 묶음이다. 현재 권장 형식은 `.claude/skills/<name>/SKILL.md`이며, 직접 `/이름`으로 부를 수도 있고 Claude가 자동으로 불러올 수도 있다. 반복 업무를 줄이는 핵심 단위다.

### 8. 커맨드

슬래시 커맨드는 세션 안에서 Claude Code를 조작하는 명령이다. `/help`, `/permissions`, `/memory`, `/model`처럼 내장 커맨드가 있고, 사용자 정의 커맨드도 만들 수 있다. 다만 현재 공식 흐름은 예전 `.claude/commands/`보다 `skills`를 더 권장한다.

### 9. 훅

훅은 특정 이벤트 전후에 자동으로 실행되는 규칙이다. 예를 들어 파일을 수정한 뒤 테스트를 돌리거나, 특정 Bash 실행 전에 검증 스크립트를 넣을 수 있다. Claude를 "설득"하는 것이 아니라 "자동 반응"을 붙이는 장치다.

### 10. MCP

MCP는 Claude Code를 외부 도구와 연결하는 표준이다. 이슈 트래커, 문서 도구, 데이터베이스, 내부 API처럼 사람이 복붙하던 맥락을 Claude가 직접 읽게 해 준다.

### 11. 서브에이전트

서브에이전트는 특정 작업을 분리해서 처리하는 보조 작업자다. 공식 문서에서는 `subagent`라는 표현을 쓴다. 코드 탐색, 리뷰, 특정 조사처럼 본 대화에 잡음을 늘리고 싶지 않은 작업을 따로 맡기기 좋다.

### 12. 하네스

`하네스`는 별도 제품명이 아니라 Claude가 일하는 실행 환경 전체를 가리키는 개념으로 이해하는 편이 좋다. 프롬프트, 도구, 권한, 메모리, 훅, 스킬, 서브에이전트, 실패 복구, 컨텍스트 관리까지 포함한 운영 레이어라고 보면 된다.

### 13. Claude Agent SDK

예전에는 `Claude Code SDK`라는 이름이 쓰였지만, 현재 공식 명칭은 `Claude Agent SDK`다. Claude Code를 움직이는 agent harness를 라이브러리 형태로 꺼내 써서, 직접 에이전트를 만드는 쪽으로 이어진다.

## 실무에서 잘 쓰는 사람의 방식

- 질문보다 작업 단위를 먼저 정의한다.
- "이 파일 수정해줘"보다 "원인 파악 -> 수정 -> 테스트 -> 변경 요약"처럼 루프를 지정한다.
- 한 세션에서 다룰 목표를 1개로 좁힌다.
- 규칙은 채팅창에 누적하지 않고 `CLAUDE.md`, 스킬, 훅으로 이동시킨다.
- 외부 정보가 자주 필요하면 복붙하지 않고 MCP 도입을 검토한다.
- 컨텍스트를 버리기 아까운 순간과 버려야 하는 순간을 구분한다.

## 자주 망하는 방식

- Claude를 검색창처럼 쓰면서 같은 설명을 계속 반복한다.
- 모든 프로젝트에 같은 요청 문장을 복붙한다.
- 권한 구조를 이해하지 않고 곧바로 강한 자동 승인 모드로 간다.
- 스킬, 훅, MCP를 한 번에 다 붙인다.
- 대화가 길어질수록 판단 품질이 떨어지는데도 정리 없이 계속 이어 간다.
- `하네스`를 "엄청난 비밀 기능"처럼 생각하고 실제로는 기본기를 건너뛴다.

## 바로 써보는 예시

아래 질문은 같은 문제를 전혀 다른 방식으로 다룬다.

### 덜 좋은 요청

```text
로그인 기능 좀 만들어줘
```

### 더 좋은 요청

```text
현재 저장소에서 로그인 흐름 관련 파일을 먼저 찾아줘.
그다음 기존 인증 방식과 충돌하지 않는 구현 방향을 3단계 이하로 요약해줘.
합의된 방향으로만 수정하고, 마지막에 테스트나 수동 검증 방법을 적어줘.
```

이 두 문장의 차이는 "무엇을 할지"가 아니라 "어떻게 일할지"를 지정했느냐에 있다.

## 1인개발 적용

혼자 개발할 때 Claude Code의 가치는 속도보다 전환 비용 감소에 있다. 오전에는 설계, 오후에는 구현, 밤에는 문서화처럼 머리를 계속 갈아끼우지 않아도 된다. 내가 정한 작업 루프를 Claude가 따라오게 만들면 된다.

추천 루프는 이렇다.

1. 문제를 한 문장으로 적는다.
2. 관련 코드만 찾게 한다.
3. 수정 범위를 한 번 축소한다.
4. 구현과 검증을 붙인다.
5. 배운 점은 `CLAUDE.md`나 메모리로 이동시킨다.

## 팀 적용

팀에서는 용어 통일이 곧 비용 절감이다. `메모리`, `규칙`, `자동화`, `검증`, `권한`을 팀이 각자 다르게 이해하면 같은 도구를 써도 결과가 들쭉날쭉해진다. 팀 도입의 첫 단계는 기능 배포가 아니라 공통 언어 확립이다.

최소한 아래 표현은 팀에서 같은 의미로 쓰는 것이 좋다.

- `CLAUDE.md`: 사람이 정의한 규칙
- `auto memory`: Claude가 축적한 작업 기억
- `skill`: 재사용 가능한 업무 절차
- `hook`: 자동 반응 규칙
- `MCP`: 외부 도구 연결
- `subagent`: 분리된 작업자
- `harness`: 이 모든 것을 묶는 운영 환경

## 체크리스트

- 세션, 컨텍스트, 토큰의 차이를 설명할 수 있다.
- `CLAUDE.md`와 auto memory의 역할 차이를 설명할 수 있다.
- 스킬, 훅, MCP, 서브에이전트가 각각 언제 필요한지 말할 수 있다.
- `하네스`를 별도 제품이 아니라 운영 개념으로 설명할 수 있다.
- Claude Code를 "대화 도구"가 아니라 "작업 루프 실행기"로 바라보기 시작했다.
