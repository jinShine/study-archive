## 왜 중요한가

Claude Code를 쓰다 보면 이상하게 같은 말을 계속 하게 된다. "이 프로젝트는 테스트 먼저", "빌드는 pnpm", "API 응답 형식은 이거", "이 폴더는 건드리지 마" 같은 말들이다. 이걸 계속 채팅으로만 전달하면 매 세션마다 다시 시작한다.

잘 쓰는 사람은 반복 설명을 대화에 쌓지 않는다. `CLAUDE.md`, path-scoped rules, auto memory, skill로 역할을 나눠서 외부화한다.

## 핵심 개념

### 1. 사람 규칙과 Claude 기억은 다르다

공식 문서 기준으로 Claude Code에는 크게 두 종류의 기억 구조가 있다.

- `CLAUDE.md`: 사람이 적는 영속 규칙
- auto memory: Claude가 작업하면서 쌓는 기억

둘 다 세션 시작 시 로드되지만, 목적은 다르다.

### 2. `CLAUDE.md`에 넣을 것

`CLAUDE.md`에는 매 세션마다 꼭 알아야 하는 사실을 넣는다.

- 빌드/테스트 명령
- 코딩 규칙
- 아키텍처 제약
- 금지 사항
- 리뷰 기준
- 폴더 구조 이해에 필요한 핵심 정보

즉, "새 팀원에게도 똑같이 알려줘야 하는 것"이 들어가면 된다.

### 3. `CLAUDE.md`에 넣지 말아야 할 것

- 일회성 작업 지시
- 너무 긴 절차 문서
- 특정 작업에서만 필요한 상세 매뉴얼
- 금방 변하는 회의 메모

이런 내용은 스킬, 별도 문서, 혹은 일회성 요청으로 분리하는 편이 낫다.

### 4. auto memory의 역할

auto memory는 사람이 매번 적지 않아도 되는 패턴을 Claude가 저장하는 기능이다. 공식 문서 기준으로 프로젝트별 메모리 디렉터리에 저장되며, 빌드 명령이나 디버깅 힌트 같은 반복 정보를 축적한다.

중요한 점은 auto memory를 `정책 저장소`로 쓰면 안 된다는 것이다. 정책은 사람이 관리해야 한다.

### 5. 어디에 무엇을 둘까

아래 기준으로 나누면 쉽다.

- 늘 지켜야 하는 규칙: `CLAUDE.md`
- Claude가 일하다 배운 실무 힌트: auto memory
- 특정 작업 절차: skill
- 자동 검증이나 강제 로직: hook

### 6. 작게 시작하는 구조

처음부터 긴 `CLAUDE.md`를 만들 필요는 없다. 공식 문서도 짧고 구체적인 규칙을 권장한다. 처음에는 아래 4개만 있어도 충분하다.

1. 실행 명령
2. 테스트 명령
3. 코드 스타일 핵심
4. 절대 건드리면 안 되는 경계

## 실무에서 잘 쓰는 사람의 방식

- Claude가 같은 실수를 두 번 하면 규칙 파일로 이동시킨다.
- 매번 말하는 빌드 명령이 있으면 바로 `CLAUDE.md`에 적는다.
- `CLAUDE.md`는 짧게 유지하고, 길어질 내용은 스킬이나 별도 문서로 뺀다.
- auto memory는 참고 자산으로 보고, 중요한 정책은 사람이 직접 관리한다.
- 프로젝트 규칙과 개인 선호를 섞지 않는다.

## 자주 망하는 방식

- 모든 걸 `CLAUDE.md`에 넣어서 거대한 잡문서를 만든다.
- 반대로 아무 규칙도 안 적고 매번 채팅으로만 설명한다.
- auto memory에 저장되겠지 하고 중요한 규칙을 방치한다.
- 프로젝트 규칙과 개인 취향을 구분하지 않는다.
- 규칙을 만들어 놓고 팀에 공유하지 않는다.

## 바로 써보는 예시

### 최소 `CLAUDE.md` 초안 요청

```text
이 저장소를 기준으로 `CLAUDE.md` 초안을 만들어줘.
처음 버전이니 아래 4가지만 담아줘.
1. 실행/테스트 명령
2. 핵심 디렉터리 설명
3. 코드 수정 시 꼭 지킬 규칙
4. 건드리면 안 되는 영역
길게 쓰지 말고 새 팀원이 첫날 읽는 문서처럼 정리해줘.
```

### 규칙과 스킬을 분리하는 요청

```text
지금 우리가 자주 반복하는 지시 중에서
`CLAUDE.md`에 넣을 것과 skill로 분리할 것을 나눠줘.
판단 기준도 같이 적어줘.
```

## 1인개발 적용

혼자 개발할 때는 `CLAUDE.md`를 내 미래의 나를 위한 문서라고 생각하면 좋다. 오늘은 기억나지만 2주 뒤에는 다시 헤맬 정보가 가장 가치 있다.

예를 들면 아래 같은 항목이 좋다.

- 로컬 실행 명령
- 배포 전 확인 순서
- 자주 깨지는 부분
- 이 프로젝트에서 선호하는 구현 방식

혼자라고 해서 규칙이 필요 없는 것이 아니라, 혼자일수록 기억을 외부화해야 속도가 난다.

## 팀 적용

팀에서는 규칙 레이어를 분리해야 한다.

- 개인 레이어: 내 말투, 개인 선호, 개인 단축 흐름
- 프로젝트 레이어: 이 저장소 공통 규칙
- 조직 레이어: 보안, 법무, 컴플라이언스 같은 상위 규칙

팀 문서가 잘 작동하려면 "누가 바꿀 수 있는지"도 명확해야 한다. `CLAUDE.md`는 위키가 아니라 작업 규칙서다. 책임자 없이 길어지면 금방 신뢰를 잃는다.

## 체크리스트

- `CLAUDE.md`와 auto memory의 차이를 설명할 수 있다.
- 규칙, 기억, 절차, 자동검증을 각각 어디에 둘지 구분할 수 있다.
- `CLAUDE.md`에 꼭 들어갈 최소 4항목을 정할 수 있다.
- 같은 실수가 반복되면 채팅이 아니라 규칙 파일로 옮겨야 함을 이해한다.
- 개인 규칙과 팀 규칙을 분리하는 감각이 생겼다.
