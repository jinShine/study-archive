## 왜 중요한가

도구 학습은 대부분 "아는 것"에서 끝난다. 하지만 Claude Code는 써보지 않으면 실력이 거의 늘지 않는다. 어떤 요청이 먹히고, 어떤 자동화가 오히려 발목을 잡는지는 실제 시나리오 안에서만 보인다.

그래서 마지막 문서는 기능 설명이 아니라 훈련 계획이다.

## 핵심 개념

### 1. 시나리오 단위로 연습해야 한다

다음 중 하나라도 실제로 해보면 실력이 빨리 는다.

- 버그 재현부터 수정까지
- 신규 기능의 최소 구현
- 테스트 공백 메우기
- 릴리즈 전 체크
- 새 팀원용 문서 작성

### 2. 수준은 기능 개수로 올라가지 않는다

성장은 아래 순서로 일어난다.

1. 정확한 요청을 한다
2. 검증을 붙인다
3. 반복 패턴을 구조화한다
4. 팀 규칙으로 승격한다

즉, skill을 많이 만든다고 고수가 되는 것은 아니다.

### 3. 좋은 성장 과제의 조건

- 실제 프로젝트에 쓸 수 있어야 한다
- 결과를 검증할 수 있어야 한다
- 다음에도 반복해서 개선할 수 있어야 한다

## 실무에서 잘 쓰는 사람의 방식

- 매주 하나의 실전 루프를 고른다.
- 성공한 요청은 템플릿화한다.
- 실패한 요청은 왜 실패했는지 기록한다.
- 자동화는 검증된 루프만 올린다.
- 월말에는 `CLAUDE.md`, skill, hook을 한 번 정리한다.

## 자주 망하는 방식

- 문서만 읽고 끝낸다.
- 멋있어 보이는 고급 기능부터 건드린다.
- 실패한 요청을 분석하지 않는다.
- 실제 저장소에 적용하지 않고 샘플 프로젝트만 만진다.
- 30일 안에 모든 걸 다 하려고 한다.

## 바로 써보는 예시

### 시나리오 1. 버그 수정 훈련

```text
현재 프로젝트에서 최근에 자주 언급된 버그 하나를 고를게.
Claude Code로 아래 순서를 연습해봐.
1. 관련 파일 찾기
2. 원인 후보 3개 이하로 줄이기
3. 수정
4. 검증
5. 회고 메모 작성
```

### 시나리오 2. 기능 추가 훈련

```text
작고 독립적인 UI 또는 API 기능 하나를 고른다.
Claude Code에게 기존 패턴 탐색 -> 최소 구현 -> 테스트 -> 변경 요약 루프로 맡긴다.
이후 내가 직접 리뷰하면서 어떤 요청이 좋았는지 기록한다.
```

### 시나리오 3. 팀 문서화 훈련

```text
새 팀원이 가장 많이 헷갈릴 기능 하나를 고른다.
Claude Code로 운영 메모 초안을 만들고,
사람이 검토해 실제 온보딩 문서로 다듬는다.
```

## 1인개발 적용

### 30일 계획

- 1주 차: 설치, 요청법, 권한 감각 익히기
- 2주 차: `CLAUDE.md` 초안 만들기
- 3주 차: 자주 쓰는 요청 2개를 skill 후보로 정리
- 4주 차: 실제 기능 추가나 버그 수정 3건을 Claude Code 루프로 처리

### 60일 계획

- 반복되는 검증 하나를 hook 또는 반자동 루프로 정착
- 문서화 루틴 붙이기
- MCP 후보 1개 검토

### 90일 계획

- 내 작업 플레이북 완성
- `CLAUDE.md`, skill, 검증 루틴 정리
- 새 협업자가 들어와도 설명 가능한 운영 문서 마련

## 팀 적용

### 30일 계획

- 파일럿 저장소 선정
- 팀 `CLAUDE.md` 초안 작성
- 공통 요청 템플릿과 리뷰 기준 수립

### 60일 계획

- 공통 skill 1~2개 도입
- 기본 권한 정책 정리
- 온보딩 체크리스트 문서화

### 90일 계획

- MCP나 고급 자동화 후보 검토
- 파일럿 회고 후 다른 저장소 확장 여부 결정
- 팀 운영 주체와 업데이트 절차 확정

## 체크리스트

- 실전 시나리오 3개 이상을 바로 실행할 수 있다.
- 30/60/90일 계획을 내 상황에 맞게 수정할 수 있다.
- 성장을 기능 수집이 아니라 운영 습관 형성으로 본다.
- 실패한 요청을 개선 자산으로 바꾸는 습관이 생겼다.
- 이제 Claude Code를 배운다는 느낌보다 함께 일한다는 감각이 생기기 시작했다.
