## 왜 중요한가

많은 사람이 Claude Code를 쓰다가 어느 순간 "여러 역할을 나눠서 동시에 일하게 할 수 없나?"라는 질문을 하게 된다. 이 지점부터 에이전트와 하네스를 이해해야 한다.

하지만 여기서 흔한 오해가 있다. 하네스를 무슨 별도 제품처럼 이해하는 것이다. 실제로는 Claude가 작업을 수행하도록 감싸는 실행 구조를 뜻하는 쪽에 가깝다. 이 구조를 이해하면 "대화를 잘하는 것"에서 "작업 시스템을 설계하는 것"으로 사고가 넘어간다.

## 핵심 개념

### 1. 서브에이전트란 무엇인가

공식 문서 기준으로 `subagent`는 특정 작업을 별도 컨텍스트에서 처리하는 보조 작업자다. 탐색 결과나 로그 덤프처럼 본 대화를 더럽히는 정보를 분리하는 데 특히 유용하다.

대표적으로 잘 맞는 작업은 이렇다.

- 코드베이스 탐색
- 대안 비교 조사
- 로그/에러 분석
- 보조 리뷰

### 2. 왜 메인 대화에서 다 하면 안 되나

모든 것을 메인 대화 하나에 넣으면 컨텍스트가 금방 비대해진다. 탐색 결과가 길어질수록 구현 단계의 집중력이 떨어진다. 서브에이전트는 이런 잡음을 격리한다.

### 3. built-in subagent 개념

공식 문서에는 읽기 중심 탐색용 에이전트 같은 기본 서브에이전트가 소개되어 있다. 핵심은 이름보다 역할이다. 읽기 전용 탐색, 계획 보조, 일반 작업 같은 성격을 분리하는 것이다.

### 4. 하네스란 무엇인가

이 커리큘럼에서는 하네스를 아래처럼 정의한다.

`Claude가 실제로 일하게 만드는 운영 구조`

여기에는 보통 아래 요소가 들어간다.

- 시스템 프롬프트와 사용자 요청
- 파일/셸/MCP 도구
- 권한 규칙
- `CLAUDE.md`와 memory
- skill과 hook
- 서브에이전트 분리
- 오류 처리와 재시도
- 컨텍스트 압축과 복구

즉, 하네스는 "성능 향상 비법"이 아니라 "Claude가 어떤 규칙 안에서 어떻게 움직일지 묶어 두는 틀"이다.

### 5. Claude Code와 Claude Agent SDK

현재 공식 명칭은 `Claude Agent SDK`다. 예전 `Claude Code SDK`에서 이름이 바뀌었다. 이 SDK는 Claude Code를 움직이는 agent harness의 핵심 능력을 라이브러리로 꺼내 쓸 수 있게 만든다.

쉽게 말하면 다음과 같다.

- Claude Code: 완성형 작업 도구
- Claude Agent SDK: 그 작동 원리를 프로그래밍 가능한 형태로 꺼낸 것

### 6. 언제 SDK까지 가야 하나

아래 상황이면 SDK를 고려할 수 있다.

- 반복되는 조직 워크플로를 코드로 묶고 싶다
- CI나 백엔드 시스템 안에서 에이전트를 실행하고 싶다
- 여러 단계의 승인과 도구 연결을 직접 통제하고 싶다

그 전까지는 Claude Code 자체만 잘 써도 충분한 경우가 많다.

## 실무에서 잘 쓰는 사람의 방식

- 메인 대화에는 결정과 결과만 남긴다.
- 조사성 작업은 서브에이전트로 분리한다.
- 하네스를 "설정 많이 하는 것"이 아니라 "작업 흐름을 제어하는 것"으로 이해한다.
- SDK는 필요가 생겼을 때 간다. 멋있어서 먼저 가지 않는다.
- 역할이 다른 작업은 역할이 다른 에이전트로 나눈다.

## 자주 망하는 방식

- 모든 걸 병렬로 돌리면 무조건 빠르다고 믿는다.
- 서브에이전트가 내 결정을 대신할 거라고 기대한다.
- 하네스를 신비한 은어처럼 쓰고 실제 운영 규칙은 없다.
- SDK를 먼저 배우느라 정작 Claude Code 기본 흐름은 약하다.
- 서브에이전트에게 범위와 산출물 형식을 안 준다.

## 바로 써보는 예시

### 조사와 구현을 분리하는 요청

```text
먼저 탐색 전용 작업자로 관련 파일과 리스크를 조사해줘.
메인 흐름에는 핵심 결론만 가져오고,
그다음 그 결론을 바탕으로 최소 수정안을 진행해줘.
```

### 하네스 관점으로 현재 설정을 점검하는 요청

```text
지금 이 프로젝트에서 Claude가 일하는 흐름을 하네스 관점으로 점검해줘.
규칙, 권한, 검증, 자동화, 외부 도구 연결 중 비어 있는 부분을 정리해줘.
```

### SDK 검토 기준을 묻는 요청

```text
우리가 Claude Code 사용을 넘어서 Claude Agent SDK까지 검토할지 판단하고 싶어.
지금 요구사항이 CLI 설정만으로 충분한지, 아니면 SDK가 필요한지 기준을 제시해줘.
```

## 1인개발 적용

혼자 개발할 때는 "미니 팀을 운영한다"는 감각이 도움이 된다. 내가 직접 여러 역할을 오갈 때 드는 정신적 비용을 Claude가 줄여주는 것이다.

추천 역할 분리는 아래처럼 단순하게 시작하면 된다.

- 탐색자
- 구현자
- 검토자

굳이 화려한 이름이나 복잡한 구조가 필요 없다. 역할만 다르면 충분하다.

## 팀 적용

팀에서는 에이전트 분리가 조직 구조를 닮기 쉽다. 하지만 사람 조직을 그대로 복제할 필요는 없다. 중요한 것은 책임과 산출물이다.

예를 들면 아래처럼 정의하는 편이 실용적이다.

- 탐색 에이전트: 관련 코드와 리스크 찾기
- 구현 에이전트: 합의된 범위 수정
- 검증 에이전트: 테스트, 체크리스트, 정책 확인

여기서 하네스는 이 셋이 어떤 규칙과 권한 안에서 움직일지 정하는 운영 층이다.

## 체크리스트

- 서브에이전트가 왜 컨텍스트 분리에 유리한지 설명할 수 있다.
- 하네스를 운영 구조로 설명할 수 있다.
- Claude Code와 Claude Agent SDK의 관계를 말할 수 있다.
- SDK가 필요한 상황과 아직 아닌 상황을 구분할 수 있다.
- 역할 분리의 기준을 `사람 흉내`가 아니라 `책임과 산출물`로 잡을 수 있다.
