# 10. 팀도입 운영 플레이북

> 학습 목표: Claude Code를 개인 장난감이 아니라 팀 시스템으로 도입할 때 필요한 운영 원칙을 잡는다.

## 학습 목표

- 팀 차원에서 무엇을 표준화하고 무엇을 개인 자율에 맡길지 구분할 수 있다.
- 온보딩, 리뷰, 권한, 검증, 문서화 관점에서 팀 도입 체크포인트를 만들 수 있다.
- 과도한 통제와 무질서한 자유 사이에서 균형을 잡을 수 있다.

## 왜 중요한가

개인이 잘 쓰는 것과 팀이 안정적으로 쓰는 것은 다르다. 팀 도입에서 중요한 것은 "다들 많이 쓰게 만들기"가 아니라 "어디까지 믿고 어디서 검증할지"를 팀이 합의하는 것이다.

Claude Code 팀 도입은 기술 프로젝트이기도 하지만 운영 프로젝트이기도 하다.

## 핵심 개념

### 1. 팀 도입의 첫 원칙

처음부터 전사 표준을 만들지 않는다. 가장 먼저 해야 할 일은 특정 저장소나 특정 워크플로에서 성공 루프를 검증하는 것이다.

예를 들면 아래처럼 시작하는 편이 좋다.

- 한 저장소
- 한 팀
- 한두 개 핵심 워크플로
- 제한된 권한 구조

### 2. 팀 표준으로 올릴 것

- 프로젝트 `CLAUDE.md`
- 기본 권한 모드와 허용 범위
- 리뷰/검증 기준
- 공통 skill
- 최소 온보딩 문서

### 3. 개인 자율에 맡길 것

- 개인 output style
- 개인 skill
- 개인 메모
- 세부 작업 문장 스타일

팀이 모든 것을 표준화하려고 하면 반발이 커지고, 아무것도 표준화하지 않으면 품질이 무너진다.

### 4. 팀 운영의 핵심 지표

도입 성패는 "사용량"보다 아래 지표에 가깝다.

- 리뷰에서 되돌아오는 변경 비율
- 반복 설명 감소 여부
- 온보딩 속도
- 배포 전 검증 누락 감소
- 위험한 권한 사용 빈도

### 5. 안전장치

팀에서는 개인보다 안전장치가 중요하다.

- 보호할 파일 지정
- 승인 필요한 작업 범위 정의
- 종료 전 체크 절차
- 외부 도구 연결 승인 정책
- 로그와 변경 이력 확인 습관

### 6. 운영 주체

팀 도입이 잘 되려면 최소 한 명은 운영 주체가 있어야 한다. 이 사람은 가장 잘 코딩하는 사람이 아니라, 규칙과 워크플로를 돌보는 사람이다.

## 실무에서 잘 쓰는 사람의 방식

- 작은 성공 사례를 표준으로 승격한다.
- 팀 `CLAUDE.md`는 짧고 명확하게 유지한다.
- 리뷰 체크리스트를 Claude용과 사람용으로 나눈다.
- 자동화는 읽기 전용 검증부터 시작한다.
- 온보딩 문서에 "첫 세션에서 하지 말아야 할 것"까지 적는다.

## 자주 망하는 방식

- 개인의 고급 설정을 팀 전체 표준으로 밀어붙인다.
- 권한 정책 없이 사용만 권장한다.
- skill과 hook을 너무 많이 만들어 아무도 기억하지 못한다.
- 운영 책임자 없이 모두의 문서로 만든다.
- 성공 기준 없이 "AI 도입" 자체를 목표로 삼는다.

## 바로 써보는 예시

### 팀 표준 초안 요청

```text
우리 팀이 Claude Code를 도입하려고 해.
프로젝트 `CLAUDE.md`, 권한 기본값, 공통 skill, 리뷰 기준, 온보딩 체크리스트로 나눠서
최소 운영 초안을 설계해줘.
처음 버전이니 과도한 자동화는 제외해줘.
```

### 리뷰 기준 정리 요청

```text
사람 리뷰와 Claude 리뷰를 섞어 쓰고 싶어.
Claude가 먼저 잡기 좋은 항목과 사람이 끝까지 봐야 하는 항목을 나눠줘.
```

### 도입 파일럿 설계 요청

```text
한 팀, 한 저장소, 두 가지 워크플로만 대상으로 Claude Code 도입 파일럿을 하고 싶어.
4주짜리 파일럿 계획을 만들어줘.
각 주차별 목표, 성공 지표, 위험 요소를 포함해줘.
```

## 1인개발 적용

혼자 일해도 이 문서가 중요한 이유가 있다. 나중에 협업자가 들어오거나 외주, 동업, 채용이 시작되면 결국 팀 운영 관점이 필요해진다. 지금 개인 프로젝트에서 정리하는 규칙은 미래 팀의 씨앗이 된다.

## 팀 적용

실제 팀 도입은 아래 순서를 추천한다.

1. 작은 파일럿 선정
2. 공통 `CLAUDE.md` 작성
3. 기본 권한 정책 설정
4. 공통 요청 템플릿 배포
5. 리뷰 기준과 온보딩 문서 정리
6. 파일럿 회고 후 범위 확장

핵심은 속도보다 재현성이다. 누가 써도 비슷한 품질이 나오기 시작해야 팀 시스템이 된다.

## 체크리스트

- 팀 표준과 개인 자율의 경계를 설명할 수 있다.
- 첫 파일럿을 어떻게 작게 시작할지 안다.
- 팀이 반드시 표준화해야 할 항목을 정할 수 있다.
- 사용량보다 운영 지표가 중요하다는 점을 이해한다.
- 운영 주체가 왜 필요한지 설명할 수 있다.
