# 02. 설치와 첫 세션 시작하기

> 학습 목표: Claude Code를 안전하게 설치하고, 첫 세션에서 무엇을 해야 하는지 몸에 익힌다.

## 학습 목표

- 현재 권장 설치 방식과 대체 설치 방식을 구분할 수 있다.
- 첫 세션에서 꼭 확인해야 할 명령과 설정을 안다.
- 시작 직후 어떤 작업을 시켜야 실패가 적은지 이해한다.

## 선행 지식

- [01.용어정리와_개발자_사고방식.md](/Users/buzz/Documents/Private/projects/study-archive/ai/claude/codex/01.용어정리와_개발자_사고방식.md)

## 왜 중요한가

설치 자체는 10분 안에 끝날 수 있다. 문제는 "설치 직후 무엇을 하느냐"다. 대부분은 로그인하고 바로 큰 작업을 시키다가 권한, 경로, 프로젝트 규칙, 모델 상태를 확인하지 못해 첫 인상이 나빠진다.

첫 세션은 생산성을 뽑는 시간이 아니라 작업 환경을 길들이는 시간이다.

## 핵심 개념

### 1. 설치 방식

현재 공식 문서 기준으로는 네이티브 설치가 기본 권장 경로다. 기존 npm 설치도 가능하지만, 호환성 이유가 없으면 최신 권장 경로를 따르는 편이 안전하다.

기본 흐름은 다음 두 가지로 이해하면 충분하다.

- 권장: 공식 설치 스크립트 또는 공식 배포 방식으로 네이티브 설치
- 대안: `npm install -g @anthropic-ai/claude-code`

### 2. 첫 실행 전에 확인할 것

- 현재 셸이 무엇인지
- 어떤 폴더에서 `claude`를 실행하는지
- 프로젝트가 git 저장소인지
- 네트워크와 로그인 상태가 정상인지
- 이미 존재하는 `.claude/` 설정이 있는지

### 3. 첫 세션에서 자주 쓰는 내장 커맨드

- `/help`: 사용 가능한 명령 확인
- `/permissions`: 권한 상태 확인
- `/memory`: 메모리 및 auto memory 상태 확인
- `/model`: 현재 모델 확인 또는 변경
- `/config`: 설정 확인
- `/doctor`: 설치 이상 여부 확인
- `/mcp`: MCP 연결 상태 확인
- `/agents`: 서브에이전트 관리

### 4. 첫 세션의 올바른 목표

첫날 해야 할 일은 대규모 구현이 아니다. Claude가 이 저장소를 어떻게 읽는지, 어디까지 접근하는지, 어떤 명령으로 검증할 수 있는지 확인하는 것이다.

## 실무에서 잘 쓰는 사람의 방식

- 프로젝트 루트에서 시작한다.
- 첫 요청은 구현보다 "탐색과 설명"으로 연다.
- `CLAUDE.md`가 없으면 곧바로 장문 규칙서를 쓰기보다, 먼저 필요한 규칙이 무엇인지 확인한다.
- 권한을 천천히 올린다.
- `/doctor`, `/permissions`, `/memory`를 초반에 확인한다.
- 첫 성공 경험을 작은 작업으로 만든다. 예를 들면 구조 설명, 특정 모듈 분석, 테스트 실행 경로 파악 같은 것들이다.

## 자주 망하는 방식

- 홈 디렉터리나 너무 상위 폴더에서 Claude를 실행한다.
- 프로젝트 구조를 보기도 전에 "회원가입 전체 만들어줘"부터 던진다.
- 권한 요청을 귀찮아해서 바로 강한 모드로 올린다.
- 기존 설정 파일을 읽지 않고 새 규칙을 덧쓴다.
- 첫 세션에서 실패한 원인을 Claude 성능 문제로만 생각한다.

## 바로 써보는 예시

### 설치 후 가장 먼저 해볼 요청

```text
이 저장소 구조를 먼저 파악해줘.
앱의 진입점, 테스트 실행 명령, 자주 수정될 핵심 디렉터리 후보를 짧게 정리해줘.
아직 파일 수정은 하지 말고, 내가 Claude Code를 안전하게 쓰기 위해 먼저 확인해야 할 점도 알려줘.
```

### 첫 세션 점검 루틴

```text
1. 이 프로젝트에서 중요한 디렉터리 5개만 추려줘.
2. 실행 또는 테스트 명령 후보가 무엇인지 package.json, Makefile, README 기준으로 찾아줘.
3. 지금 상태에서 Claude가 수정 전에 확인해야 할 위험 요소가 있으면 알려줘.
```

이렇게 시작하면 Claude가 탐색 중심으로 움직이고, 내가 저장소를 다시 이해하는 데도 도움이 된다.

## 1인개발 적용

혼자 작업할 때는 첫 세션을 "작업 컨베이어 벨트 만들기"로 생각하면 좋다. 내가 나중에 계속 반복할 명령과 규칙을 찾는 것이다.

예를 들면 이런 질문이 좋다.

- 이 프로젝트에서 기능 추가 전에 항상 돌려야 할 명령이 뭐야?
- 자주 건드리는 폴더는 어디야?
- 지금 `CLAUDE.md`에 적어둘 만한 규칙 후보가 뭐야?

이 세 가지만 뽑아도 다음 세션의 효율이 달라진다.

## 팀 적용

팀에서는 첫 세션 경험이 중요하다. 각자 제멋대로 설치하고 사용하면 "누구는 잘 되고 누구는 안 되는" 상태가 된다. 최소한 아래는 팀 공통으로 정해 두는 것이 좋다.

- 권장 설치 방식
- 프로젝트 루트 위치
- 기본 권한 모드
- 필수 확인 커맨드
- 첫 세션에서 금지할 작업

예를 들어 신입 온보딩 문서에 "첫날은 탐색만, 둘째 날부터 수정" 같은 규칙을 넣으면 혼란이 크게 줄어든다.

## 체크리스트

- 권장 설치 방식과 npm 대안 설치를 구분할 수 있다.
- 프로젝트 루트에서 세션을 시작해야 하는 이유를 안다.
- `/help`, `/permissions`, `/memory`, `/doctor`의 용도를 설명할 수 있다.
- 첫 세션에서는 탐색 중심으로 시작해야 한다는 원칙을 이해한다.
- 첫 성공 경험을 작은 작업으로 설계할 수 있다.
