# 06. 커맨드, 훅, 스킬로 반복 작업 줄이기

> 학습 목표: 자주 반복하는 지시를 매번 장문으로 쓰지 않고, 재사용 가능한 구조로 바꾼다.

## 학습 목표

- 내장 커맨드, 사용자 정의 커맨드, skill, hook의 역할 차이를 안다.

- 일관된 작업 루틴을 Claude Code 안에 심는 법을 익힌다.

## 왜 중요한가

Claude Code를 몇 주만 써도 같은 요청을 계속 하게 된다. "리뷰 모드로 봐줘", "배포 전 체크해줘", "리팩터링 후 테스트까지 해줘", "이 저장소 규칙 맞는지 확인해줘" 같은 것들이다.

이걸 계속 직접 쓰면 시간이 아깝고, 미묘하게 빠뜨리는 부분이 생긴다. 반복 패턴은 구조로 올려야 한다.

## 핵심 개념

### 1. 내장 커맨드

`/help`, `/permissions`, `/memory`, `/mcp`, `/agents` 같은 것은 Claude Code가 기본 제공하는 명령이다. 세션을 제어하거나 상태를 보는 데 쓴다.

### 2. 사용자 정의 커맨드와 skill

예전에는 `.claude/commands/` 기반 커스텀 커맨드가 널리 쓰였고 지금도 동작한다. 하지만 현재 공식 권장 흐름은 `.claude/skills/<name>/SKILL.md` 기반의 `skill`이다.

skill이 더 좋은 이유는 다음과 같다.

- `/이름`으로 직접 호출 가능
- Claude가 자동으로 불러올 수도 있음
- supporting files, 스크립트, 예시 문서를 같이 둘 수 있음
- 도구 허용 범위를 skill 단위로 설계 가능

### 3. skill의 좋은 용도

- 코드 리뷰 절차
- 배포 전 체크 절차
- 특정 프레임워크 관습
- 팀 고유 아키텍처 규칙
- 반복 문서화 포맷

### 4. hook의 좋은 용도

hook은 "부탁"이 아니라 "자동 반응"이다. 공식 문서 기준으로 `PreToolUse`, `PostToolUse`, `Stop`, `Notification` 등 다양한 이벤트에 붙일 수 있다. 최근에는 command/HTTP뿐 아니라 prompt 기반, agent 기반 hook도 지원한다.

hook이 잘 맞는 곳은 이렇다.

- 수정 전 위험 명령 차단
- 수정 후 테스트 실행
- 종료 전에 검증
- 특정 파일 변경 감시

### 5. 무엇을 skill로 만들고 무엇을 hook으로 만들까

- 사람이 의도적으로 실행할 절차: skill
- 항상 자동으로 검사할 규칙: hook
- 상태 확인용 동작: 내장 커맨드
- 과거 호환용 짧은 커맨드: custom command

이 구분만 잘해도 구조가 훨씬 깔끔해진다.

## 실무에서 잘 쓰는 사람의 방식

- 자주 쓰는 요청은 skill로 묶는다.
- "반드시 지켜야 하는 것"만 hook으로 강제한다.
- hook은 빠르고 예측 가능하게 유지한다.
- skill 설명은 짧고 명확하게 써서 Claude가 자동 선택하기 쉽게 만든다.
- side effect가 큰 skill은 자동 호출을 막고 수동 호출 전용으로 둔다.

## 자주 망하는 방식

- 모든 것을 hook으로 강제해서 Claude를 답답하게 만든다.
- 모든 것을 skill로 빼서 결국 사람이 어떤 skill을 써야 할지 더 헷갈리게 만든다.
- 길고 모호한 설명으로 skill을 만들어 자동 호출 품질을 떨어뜨린다.
- 시간이 오래 걸리는 검증을 매번 동기 hook으로 돌린다.
- 레거시 `commands`와 `skills`를 섞어 두고 관리 기준을 만들지 않는다.

## 바로 써보는 예시

### 예시 1. 리뷰용 skill

```text
이 저장소에서 자주 하는 코드 리뷰 절차를 skill로 만들고 싶어.
아래 순서를 따르는 `/review-local` 초안을 설계해줘.
1. 변경 파일 파악
2. 위험 요소 찾기
3. 테스트 또는 검증 공백 찾기
4. 최종 보고 형식 통일
```

### 예시 2. 수정 후 테스트 hook 아이디어

```text
파일 수정 뒤에 항상 전체 테스트는 비싸니까,
변경 파일 타입에 따라 가벼운 검사를 돌리는 hook 전략을 제안해줘.
어떤 건 hook으로, 어떤 건 수동 skill로 두는 게 좋은지도 설명해줘.
```

### 예시 3. 배포 전 수동 skill

```text
배포는 자동 실행되면 안 돼.
사용자가 직접 호출할 때만 실행되는 `/pre-release-check` skill 초안을 만들어줘.
```

## 1인개발 적용

혼자 작업할 때는 아래 세 가지부터 만드는 것을 권장한다.

1. `review` 계열 skill
2. `release-check` 계열 skill
3. 수정 후 가벼운 테스트 hook

이 세 가지가 있으면 장문의 재요청이 크게 줄어든다. 중요한 것은 멋진 자동화보다 "내가 진짜 자주 쓰는 패턴"을 먼저 올리는 것이다.

## 팀 적용

팀에서는 공통 skill이 사실상 작업 표준이 된다. 리뷰 형식, 문서화 형식, 점검 루틴을 skill로 공유하면 말 대신 절차가 남는다. 다만 팀 전체에 강제하는 항목은 최소화해야 한다.

추천 원칙은 이렇다.

- 팀 공통: 꼭 지켜야 하는 검증은 hook
- 팀 권장: 자주 쓰는 루틴은 skill
- 개인 최적화: 개인 skill이나 output style

이렇게 나누면 팀 표준과 개인 생산성이 충돌하지 않는다.

## 체크리스트

- 내장 커맨드, skill, custom command, hook의 차이를 설명할 수 있다.
- 반복 작업을 skill로 올릴 후보를 3개 이상 찾을 수 있다.
- 자동 강제가 필요한 항목만 hook으로 보내야 한다는 기준을 안다.
- side effect가 큰 skill은 자동 호출을 막는 것이 안전하다는 점을 이해한다.
- 내 프로젝트에 맞는 첫 skill 설계 아이디어가 생겼다.
